# Wave 1: Implement --gaps Flag in plan-set and execute-set SKILL.md

## Objective

Add gap-closure workflow support to both `plan-set` and `execute-set` skills so that merged/complete sets with remaining gaps (listed in GAPS.md) can undergo targeted gap-closure planning and execution. The implementation is entirely within SKILL.md prompt files -- no library code changes needed.

## Critical Constraints

- **Never attempt state transitions on merged/complete sets.** The `merged` status is terminal in `state-transitions.cjs` (line 9: `merged: []`). Any `state transition set` call on a merged set will throw. Gap-closure mode must skip ALL state transition steps entirely.
- **Preserve existing behavior for non-gaps invocations.** All changes must be gated behind the `--gaps` flag. Normal plan-set and execute-set flows must be completely unaffected.
- **GAPS.md is required input.** If `--gaps` is passed but `.planning/sets/${SET_ID}/GAPS.md` does not exist, fail fast with a clear error message.

## Owned Files

- `skills/plan-set/SKILL.md`
- `skills/execute-set/SKILL.md`

## Task 2: Add --gaps flag handling and status-gate bypass to execute-set/SKILL.md

**File:** `skills/execute-set/SKILL.md`

### What to change

**Step 1 (Resolve Set and Validate Preconditions):**

1. The "Check for --gaps Flag" subsection already exists (lines 61-63) but only says to execute waves without WAVE-COMPLETE.md markers. Expand it:

   ```
   ### Check for --gaps Flag

   Parse the user's input for the `--gaps` flag. If present, set `GAPS_MODE=true`. Gap-closure mode:
   - Relaxes status validation to accept `complete` and `merged` sets
   - Only executes gap-closure waves (those with `<!-- gap-closure: true -->` header or waves numbered higher than the last WAVE-COMPLETE.md marker from the original execution)
   - Skips ALL state transitions (the set remains in its current status)
   - Does NOT trigger solo auto-merge
   ```

2. In "Validate Set Status" (lines 76-93), modify the `complete` or `merged` branch:
   - If `--gaps` flag IS present AND status is `complete` or `merged`:
     - Validate that `.planning/sets/${SET_ID}/GAPS.md` exists. If missing, display error: "No GAPS.md found for set '{SET_ID}'. Run /rapid:plan-set {SET_INDEX} --gaps first to plan gap-closure waves." STOP.
     - Validate that gap-closure wave PLAN.md files exist (waves numbered higher than existing WAVE-COMPLETE markers). If none found, display: "No gap-closure wave plans found. Run /rapid:plan-set {SET_INDEX} --gaps first." STOP.
     - Display: "Gap-closure mode: executing gap-closure waves for set '{SET_ID}' (status: {status})."
     - Continue.
   - If `--gaps` flag is NOT present: Keep existing behavior (STOP with "already complete/merged").

**Step 2 (Re-Entry Detection):**

3. In gaps mode, the existing re-entry detection naturally handles gap-closure waves because it checks for WAVE-COMPLETE.md markers. Waves from the original execution will have markers and be skipped. Gap-closure waves (newly planned) will NOT have markers and will be scheduled for execution. No changes needed to the re-entry logic itself, but add a clarifying note:

   ```
   **Gap-closure mode note:** In gap-closure mode, previously completed waves (from original execution) are skipped via their WAVE-COMPLETE.md markers. Only newly planned gap-closure waves (without markers) are executed.
   ```

**Step 3 (State Transition):**

4. Wrap the state transition in a conditional:
   - If `GAPS_MODE=true`: Skip entirely. Display: "Gap-closure mode: skipping state transition (set remains in '{status}' status)."
   - If `GAPS_MODE=false`: Keep existing behavior (`planned -> executed`).

**Step 5 (Post-Execution Verification):**

5. In gaps mode, modify the verifier prompt to focus on gap-specific verification:
   - Instead of full success criteria from ROADMAP.md, pass GAPS.md as the verification input
   - Instruction: "Verify that the specific gaps listed in GAPS.md have been resolved by the gap-closure wave implementations."
   - After verification, update GAPS.md in-place: for each resolved gap, add a `**Status:** Resolved` marker. For unresolved gaps, add `**Status:** Unresolved` with details.

**Step 6 (Complete Set):**

6. Wrap the entire Step 6 in a conditional:
   - If `GAPS_MODE=true`: Skip state transition AND solo auto-merge entirely. Display: "Gap-closure mode: skipping completion transition (set remains in '{status}' status)."
     - Still commit marker files (WAVE-COMPLETE.md for gap-closure waves, updated GAPS.md).
     - Adjust the commit message: `execute-set(${SET_ID}): gap-closure execution complete`
     - Adjust the display summary: "Gap-closure for set '{SET_ID}' complete. {N} gap-closure waves executed."
     - Adjust the next-step suggestion based on verification results:
       - If all gaps resolved: `> **Next step:** /rapid:review {SET_INDEX}` (or note that set is already merged)
       - If gaps remain: `> **Next step:** /rapid:plan-set {SET_INDEX} --gaps` (plan another round)
   - If `GAPS_MODE=false`: Keep existing behavior.
