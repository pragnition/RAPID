<task type="auto">
  <name>Task 1: Create /quick skill</name>
  <files>skills/quick/SKILL.md</files>
  <action>
    Create `skills/quick/` directory and write SKILL.md.

    **Frontmatter:**
    ```yaml
    ---
    description: Ad-hoc changes without set structure -- planner, verifier, executor pipeline
    allowed-tools: Bash(rapid-tools:*), Agent, AskUserQuestion, Read, Write, Glob, Grep
    ---
    ```

    **Structure (fully autonomous after task description):**

    **Step 0: Environment + Banner**
    - Standard v3 env preamble
    - `node "${RAPID_TOOLS}" display banner quick`

    **Step 1: Gather Task Description**
    - Use AskUserQuestion (freeform): "Describe what you'd like to do. Be specific about the changes needed -- files, behavior, constraints."
    - Record the task description.

    **Step 2: Create Quick Task Directory**
    - Count existing quick task directories: `ls .planning/quick/ 2>/dev/null | wc -l`
    - Compute NEXT_ID = count + 1
    - Generate slug from task description (lowercase, hyphens, truncated to 40 chars)
    - Create directory: `.planning/quick/${NEXT_ID}-${SLUG}/`
    - Record TASK_DIR path.

    **Step 3: Spawn Planner Agent**
    - Display: "Planning quick task..."
    - Spawn the **rapid-planner** agent with:
      ```
      Plan a quick task for in-place execution (no worktree, no set lifecycle).

      ## Task Description
      {user's task description}

      ## Instructions
      1. Analyze the task and determine which files need modification
      2. Produce a single PLAN.md with 1-3 tasks
      3. Write the plan to {TASK_DIR}/{NEXT_ID}-PLAN.md
      4. Keep it focused -- this is a quick task, not a multi-wave set

      ## Working Directory
      {projectRoot}
      ```
    - Parse RAPID:RETURN. If planner fails: display error breadcrumb, STOP.
    - Verify PLAN.md was written: check `{TASK_DIR}/{NEXT_ID}-PLAN.md` exists.

    **Step 4: Spawn Plan Verifier Agent**
    - Display: "Verifying plan..."
    - Spawn the **rapid-plan-verifier** agent with:
      ```
      Verify the quick task plan.

      ## Plan
      {Full content of {NEXT_ID}-PLAN.md}

      ## Working Directory
      {projectRoot}

      ## Output
      Write VERIFICATION-REPORT.md to {TASK_DIR}/VERIFICATION-REPORT.md
      ```
    - Parse RAPID:RETURN for verdict.
    - If FAIL: Display issues. Use AskUserQuestion:
      - "Plan verification failed. Override and execute anyway, or cancel?"
      - Options: "Override" / "Cancel"
      - If Cancel: STOP.

    **Step 5: Spawn Executor Agent**
    - Display: "Executing quick task..."
    - Spawn the **rapid-executor** agent with:
      ```
      Implement the quick task plan.

      ## Your PLAN
      {Full content of {NEXT_ID}-PLAN.md}

      ## Commit Convention
      After each task, commit with: quick({SLUG}): description

      ## Working Directory
      {projectRoot}
      ```
    - IMPORTANT: No worktree path -- executor works in current directory (project root).
    - Parse RAPID:RETURN.

    **Step 6: Write Summary and Complete**
    - Write `{TASK_DIR}/{NEXT_ID}-SUMMARY.md` with:
      ```markdown
      # Quick Task {NEXT_ID}: {SLUG}

      **Description:** {task description}
      **Date:** {ISO date}
      **Status:** {COMPLETE/CHECKPOINT/BLOCKED from executor return}
      **Commits:** {commit hashes from executor return}
      **Files Modified:** {artifacts from executor return}
      ```
    - Commit the quick task directory:
      ```bash
      git add "{TASK_DIR}"
      git commit -m "quick({SLUG}): complete quick task {NEXT_ID}"
      ```
    - Display completion:
      ```
      Quick task {NEXT_ID} complete.
      Directory: {TASK_DIR}
      ```
    - Do NOT add to STATE.json sets array per research recommendation (avoids polluting /status).
    - Do NOT suggest a next action (fire-and-forget -- user is done).

    **Error Handling:**
    - RAPID_TOOLS not set: suggest /rapid:install
    - Planner failure: STOP with error breadcrumb
    - Verifier FAIL: offer Override/Cancel
    - Executor CHECKPOINT: write handoff, suggest re-running /rapid:quick
    - Executor BLOCKED: display blocker, STOP

    **Anti-Patterns:**
    - Do NOT create a worktree -- quick tasks execute in-place
    - Do NOT add full set lifecycle state to STATE.json
    - Do NOT run discuss-set flow -- user's task description IS the context
    - Do NOT prompt between pipeline steps (fully autonomous after initial task description)
    - Do NOT suggest /rapid:review after quick task (not a set, not reviewable via standard pipeline)

    **Key Principles:**
    - Fire-and-forget: user describes task, three agents handle the rest
    - In-place execution on current branch
    - 3-agent pipeline: planner -> plan-verifier -> executor
    - Artifacts stored in .planning/quick/ for auditability
    - No set lifecycle overhead

<task type="auto">
  <name>Task 2: Create /add-set skill</name>
  <files>skills/add-set/SKILL.md</files>
  <action>
    Create `skills/add-set/` directory and write SKILL.md.

    **Frontmatter:**
    ```yaml
    ---
    description: Add a new set to an existing project mid-milestone with discovery and contract generation
    allowed-tools: Bash(rapid-tools:*), AskUserQuestion, Read, Write, Glob, Grep
    ---
    ```

    Note: No `Agent` in allowed-tools -- this skill does not spawn subagents. It uses AskUserQuestion for discovery and writes artifacts directly.

    **Structure:**

    **Step 0: Environment + Banner**
    - Standard v3 env preamble
    - `node "${RAPID_TOOLS}" display banner add-set`

    **Step 1: Load State and Validate**
    - `node "${RAPID_TOOLS}" state get --all`
    - Parse to get current milestone ID and existing sets.
    - Display current milestone: "{MILESTONE_ID} ({MILESTONE_NAME})"
    - Display existing sets with their statuses.

    **Step 2: Interactive Discovery (Mini Discuss-Set)**
    - Use AskUserQuestion (freeform): "What should this new set accomplish? Describe the scope, goals, and key deliverables."
    - Use AskUserQuestion (freeform): "What files or areas of the codebase will this set modify? Are there dependencies on existing sets?"
    - Parse answers to build set definition.

    **Step 3: Generate Set ID**
    - Derive a kebab-case set ID from the description (e.g., "add payment processing" -> "payment-processing")
    - Display proposed set ID. Use AskUserQuestion:
      - "Use set ID '{proposed-id}'?"
      - Options: "Yes" / "Custom ID"
      - If Custom ID: ask freeform for the ID.
    - Validate set ID is unique (not already in STATE.json sets).

    **Step 4: Create Set Directory and Artifacts**
    - Create `.planning/sets/${SET_ID}/` directory
    - Write `.planning/sets/${SET_ID}/DEFINITION.md`:
      ```markdown
      # Set: {SET_ID}

      **Created:** {date} (via /add-set)
      **Milestone:** {MILESTONE_ID}

      ## Scope
      {User's description from Step 2}

      ## Key Deliverables
      {Extracted from user answers}

      ## Dependencies
      {Any cross-set dependencies mentioned}
      ```
    - Write `.planning/sets/${SET_ID}/CONTRACT.json`:
      ```json
      {
        "setId": "{SET_ID}",
        "milestone": "{MILESTONE_ID}",
        "created": "{ISO date}",
        "exports": { "functions": [], "types": [] },
        "imports": { "functions": [], "types": [] },
        "fileOwnership": []
      }
      ```
      Note: CONTRACT.json starts empty -- file ownership and exports/imports are populated during /plan-set.

    **Step 5: Update STATE.json**
    - Read STATE.json via `node "${RAPID_TOOLS}" state get --all`
    - Parse JSON, find current milestone's sets array
    - Add new set object: `{ "id": "{SET_ID}", "status": "pending" }`
    - Write updated STATE.json via the Write tool (same pattern as /init roadmapper writes STATE.json)
    - Commit state: `node "${RAPID_TOOLS}" state commit-state "add-set(${SET_ID}): add new set to milestone"`

    **Step 6: Update ROADMAP.md**
    - Read .planning/ROADMAP.md
    - Find the current milestone section
    - Append a new set entry with the description
    - Write updated ROADMAP.md

    **Step 7: Commit and Next Steps**
    - Commit all new artifacts:
      ```bash
      git add ".planning/sets/${SET_ID}/"
      git add ".planning/ROADMAP.md"
      git commit -m "add-set(${SET_ID}): add set to ${MILESTONE_ID}"
      ```
    - Display:
      ```
      Set '{SET_ID}' added to milestone '{MILESTONE_ID}'.
      Status: pending
      ```
    - Display next step: `> **Next step:** /rapid:start-set {SET_INDEX}`
    - Display progress breadcrumb:
      ```
      init [done] > start-set > discuss-set > plan-set > execute-set > review > merge
      ```

    **Error Handling:**
    - RAPID_TOOLS not set: suggest /rapid:install
    - STATE.json missing: suggest /rapid:init
    - Duplicate set ID: suggest alternative ID
    - STATE.json write failure: show error, suggest retry

    **Anti-Patterns:**
    - Do NOT auto-start the set -- user runs /start-set separately (locked decision)
    - Do NOT run a full discuss-set flow -- keep discovery to 2 questions
    - Do NOT spawn subagents -- this is a lightweight interactive skill
    - Do NOT create a worktree -- that happens in /start-set
    - Do NOT write wave PLAN.md files -- that happens in /plan-set

    **Key Principles:**
    - Mini discovery: 2 questions to understand scope
    - CONTRACT.json generated (initially empty, populated during planning)
    - STATE.json updated with new pending set
    - ROADMAP.md updated with set description
    - Suggests /start-set as the explicit next action
    - No subagent spawns -- lightweight interactive command
  </action>
