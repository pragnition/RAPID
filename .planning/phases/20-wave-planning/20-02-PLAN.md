<task type="auto">
  <name>Task 1: Wave researcher, wave planner, and job planner role modules</name>
  <files>src/modules/roles/role-wave-researcher.md, src/modules/roles/role-wave-planner.md, src/modules/roles/role-job-planner.md</files>
  <action>
    Create all three role modules following the established patterns from role-set-planner.md and role-research-stack.md. Each module follows the same structure: Role title, Responsibilities, Input, Output (with template), Constraints.

    **1. role-wave-researcher.md** (~70-90 lines)

    Title: "Role: Wave Research Agent"
    Description: A focused research subagent that investigates implementation specifics for a single wave. Unlike init research (5 parallel agents), this is a single targeted agent.

    Input:
    - WAVE-CONTEXT.md (from /discuss -- the developer's implementation vision)
    - CONTRACT.json (the set's interface contracts)
    - SET-OVERVIEW.md (high-level set approach)
    - Targeted codebase files (source files the wave's jobs will modify)
    - Job descriptions from the wave (IDs and titles from STATE.json)

    Responsibilities:
    - Investigate how to implement the wave's jobs given the user's vision
    - Use Context7 MCP for documentation lookups on libraries/frameworks in scope
    - Read targeted source files to understand existing patterns and integration points
    - Identify implementation risks, edge cases, and recommended approaches
    - Focus on actionable findings -- this is NOT a broad stack survey

    Output: Write `.planning/waves/{setId}/{waveId}/WAVE-RESEARCH.md` with sections:
    - Implementation Findings (per job: approach analysis, pattern recommendations)
    - Library/Framework Notes (version-specific guidance from Context7)
    - Integration Points (how jobs connect to existing code)
    - Risks and Edge Cases
    - Recommendations

    Constraints:
    - Single-pass research (do not request follow-up information)
    - Keep output under 300 lines -- focused, not encyclopedic
    - Do NOT modify any files other than WAVE-RESEARCH.md
    - Do NOT make architectural decisions -- surface information for the planner
    - End with RAPID:RETURN protocol (COMPLETE with artifacts list)

    **2. role-wave-planner.md** (~90-110 lines)

    Title: "Role: Wave Planner"
    Description: Produces a high-level WAVE-PLAN.md that breaks down the wave into per-job approach summaries with file assignments and coordination notes.

    Input:
    - WAVE-CONTEXT.md (developer vision from /discuss)
    - WAVE-RESEARCH.md (research findings)
    - CONTRACT.json (interface contracts)
    - SET-OVERVIEW.md (high-level set context)
    - OWNERSHIP.json (file ownership map)

    Responsibilities:
    - Produce high-level per-job implementation plans
    - Assign primary file ownership per job within the wave (prevent intra-wave file conflicts)
    - Identify intra-wave dependencies and coordination needs
    - Assess complexity per job (S/M/L)
    - Note risks and mitigations at the wave level

    Output: Write `.planning/waves/{setId}/{waveId}/WAVE-PLAN.md` using the template from 20-RESEARCH.md:
    ```markdown
    # WAVE-PLAN: {wave-id}
    **Set:** {set-name}  **Wave:** {wave-number}  **Generated:** {date}

    ## Wave Objective
    {2-3 sentences from WAVE-CONTEXT.md}

    ## Job Summaries
    ### Job: {job-id}
    **Objective:** {what}  **Approach:** {how}  **Key Files:** [list]
    **Dependencies:** {intra-wave}  **Estimated complexity:** {S/M/L}

    ## Intra-Wave Coordination
    {shared files, ordering, integration points}

    ## Risks
    | Risk | Impact | Mitigation |
    ```

    Constraints:
    - Keep each job summary under 15 lines -- this is high-level, not detailed
    - File assignments MUST respect OWNERSHIP.json -- do not assign files owned by other sets
    - If two jobs must touch the same file, assign one as primary owner and note the coordination
    - Do NOT produce detailed implementation steps (that is the Job Planner's job)
    - End with RAPID:RETURN protocol

    **3. role-job-planner.md** (~90-110 lines)

    Title: "Role: Job Planner"
    Description: Creates a detailed implementation plan for a single job within a wave. One instance is spawned per job, receiving the wave-level context.

    Input:
    - WAVE-PLAN.md (the job's summary section from the wave planner)
    - WAVE-RESEARCH.md (relevant research findings)
    - WAVE-CONTEXT.md (developer vision)
    - CONTRACT.json (interface contracts -- exports this job satisfies, imports it consumes)
    - Targeted source files for this specific job

    Responsibilities:
    - Produce a detailed JOB-PLAN.md with implementation steps
    - Each step should be atomic and committable
    - List acceptance criteria that can be verified
    - Map which contract exports this job satisfies and which imports it consumes
    - Note behavioral invariants the implementation must respect

    Output: Write `.planning/waves/{setId}/{waveId}/{jobId}-PLAN.md` using the template from 20-RESEARCH.md:
    ```markdown
    # JOB-PLAN: {job-id}
    **Set:** {set-name}  **Wave:** {wave-id}  **Job:** {job-name}  **Generated:** {date}

    ## Objective
    ## Approach
    ## Files to Create/Modify
    | File | Action | Purpose |
    ## Implementation Steps
    1. **{Step}** - {instruction} - Commit: `type({set}): {desc}`
    ## Acceptance Criteria
    - [ ] {criterion}
    ## Contract Compliance
    - Exports implemented: [list]
    - Imports consumed: [list]
    - Invariants honored: [list]
    ## Notes
    ```

    Constraints:
    - Implementation steps MUST be specific enough for an executor agent to follow without interpretation
    - Each step produces one atomic commit
    - Only modify files assigned to this job in WAVE-PLAN.md (respect intra-wave file ownership)
    - Do NOT ask the user questions -- this agent works autonomously from WAVE-CONTEXT.md
    - Do NOT skip Contract Compliance section -- every JOB-PLAN.md must map back to CONTRACT.json
    - End with RAPID:RETURN protocol

    **All three modules:**
    - Follow the markdown structure of role-set-planner.md (# Role title, ## sections)
    - Use the same heading hierarchy (##, ###)
    - Include behavioral constraints section matching role-research-stack.md pattern
    - Reference RAPID:RETURN protocol for structured output
  </action>
