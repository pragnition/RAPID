  <name>Task 1: Add handleReview CLI subcommands to rapid-tools.cjs</name>
  <files>src/bin/rapid-tools.cjs</files>
  <action>
Add a new `case 'review':` in the main switch statement of rapid-tools.cjs (around line 170, after the 'wave-plan' case) that routes to a new `handleReview` function.

Create `async function handleReview(cwd, subcommand, args)` following the established pattern from handleExecute/handleWavePlan. It requires `../lib/review.cjs` and implements these subcommands:

1. **`review scope <set-id> <wave-id>`** — Calls `review.scopeWaveForReview(cwd, worktreePath, baseBranch)`. Must resolve worktree path from the worktree registry (require worktree.cjs, call loadRegistry). Accepts optional `--branch <branch>` flag (default 'main'). Outputs JSON: `{ changedFiles, dependentFiles, totalFiles }`.

2. **`review log-issue <set-id> <wave-id>`** — Reads JSON issue object from stdin (fs.readFileSync(0, 'utf-8')), parses it, calls `review.logIssue(cwd, setId, waveId, issue)`. Outputs JSON: `{ logged: true, issueId }`.

3. **`review list-issues <set-id>`** — Calls `review.loadSetIssues(cwd, setId)`. Accepts optional `--status <status>` filter. Outputs JSON array of issues.

4. **`review update-issue <set-id> <wave-id> <issue-id> <status>`** — Calls `review.updateIssueStatus(cwd, setId, waveId, issueId, status)`. Outputs JSON: `{ updated: true }`.

5. **`review lean <set-id> <wave-id>`** — The lean wave-level review subcommand. This is the one called by execute SKILL.md after reconciliation. It:
   - Resolves worktree path from registry
   - Calls `review.scopeWaveForReview()` to get scoped files
   - Verifies planned artifacts exist (reads JOB-PLAN.md files in the wave dir, checks each listed file exists in worktree)
   - For missing artifacts: creates issues with type='artifact', source='lean-review', autoFixAttempted=false
   - Attempts auto-fix for missing artifacts: if the artifact is a simple file that should have been created, log as unfixable. Auto-fix is limited to verifiable issues only (per CONTEXT.md decision).
   - Outputs JSON: `{ issues: [...], autoFixed: number, needsAttention: [...] }`
   - Issues with autoFixSucceeded=false go into needsAttention array

6. **`review summary <set-id>`** — Calls `review.loadSetIssues(cwd, setId)` then `review.generateReviewSummary(setId, issues)`. Writes REVIEW-SUMMARY.md to `.planning/waves/{setId}/REVIEW-SUMMARY.md`. Outputs JSON: `{ written: true, path, issueCount }`.

Follow error handling pattern: use `error()` from core.cjs for user-facing errors, `process.exit(1)` on missing args.

### Step 3g.1: Lean wave review

After successful reconciliation (PASS or PASS_WITH_WARNINGS), run lean review automatically:

node "${RAPID_TOOLS}" review lean <set-id> <wave-id>

Parse the JSON output for `{ issues, autoFixed, needsAttention }`.

If "Log and continue": proceed to next step (issues are already logged in REVIEW-ISSUES.json).
If "Pause execution": commit state and exit with resume instructions.

**If reconciliation was FAIL:** Skip lean review entirely (wave needs re-execution first).

**2. --fix-issues flag support (new Step 0b):**

Add to Step 0 (after environment setup) a check for `--fix-issues` flag. When present, the skill enters a different flow:

```markdown
### Step 0b: Check for --fix-issues flag

If the user invoked `/rapid:execute <set-id> --fix-issues`, enter issue fix mode instead of normal execution:

1. Load all open issues:
   ```bash
   node "${RAPID_TOOLS}" review list-issues <set-id> --status open
   ```

2. If no open issues: inform user "No open issues for set '{set-id}'." and exit.

3. Present issue summary to user using AskUserQuestion:
   - **question:** "{N} open issue(s) across {W} waves. Fix all or select?"

   - Read each issue's file and description
   - Apply targeted fixes
   - Commit atomically
   - Return COMPLETE with fixed/unfixable arrays

5. After bugfix returns, update issue statuses:
   ```bash
   node "${RAPID_TOOLS}" review update-issue <set-id> <wave-id> <issue-id> fixed
   ```

6. Print summary of fixes and exit.
```
