  <action>
Create 6 agent role modules following the established pattern from existing roles (e.g., role-job-executor.md, role-reviewer.md). Each role module is a markdown file that serves as the prompt for the subagent. All roles MUST use the RAPID:RETURN protocol for structured output.

**role-unit-tester.md:**
- Purpose: Generate test plans for user approval, then write and run tests with full observability
- Input: Receives scoped file list, project test framework info, and acceptance criteria from JOB-PLAN.md
- Flow: (1) Analyze code to identify testable behaviors. (2) Generate a test plan as structured markdown (test file path, test cases with description/expected behavior). (3) Output the test plan in RAPID:RETURN with status=CHECKPOINT and data containing the plan. (4) When resumed after approval, write test files, run them via `node --test`, capture stdout/stderr. (5) Return COMPLETE with data containing { testsWritten, testsPassed, testsFailed, output (raw stdout/stderr), testFiles }.
- Constraint: Only test files in the scoped file list. Use node:test framework (the standard in this project). Generate tests that import the actual module and test real behavior.
- RAPID:RETURN data schema: { testPlan: [...], testsWritten: number, testsPassed: number, testsFailed: number, output: string, testFiles: string[] }

**role-bug-hunter.md:**
- Purpose: Perform broad static analysis on scoped files with risk/confidence scoring
- Input: Receives explicit scoped file list (ONLY these files may be reported on)
- Flow: (1) Read each scoped file. (2) Analyze for: logic errors, off-by-one errors, null/undefined risks, error handling gaps, concurrency issues, security concerns, API contract mismatches. (3) For each finding, assign risk (critical/high/medium/low) and confidence (high/medium/low). (4) Return COMPLETE with structured findings.
- Constraint: ONLY report bugs in the provided scoped file list. Pre-existing issues outside scope are not this wave's responsibility. Read-only analysis + bash for linting -- no file modifications.
- RAPID:RETURN data schema: { findings: [{ id, file, line, category, description, risk, confidence, codeSnippet }], totalFindings: number, scopeFiles: string[] }

**role-devils-advocate.md:**
- Purpose: Challenge bug hunter findings with code evidence. Attempt to disprove each finding.
- Input: Receives the hunter's findings array and access to the same scoped files
- Flow: (1) For each finding, read the relevant code and surrounding context. (2) Attempt to determine if the finding is: a false positive (code is actually correct), already handled elsewhere, impossible to trigger, or a genuine bug. (3) Provide evidence (code snippets, execution paths) supporting the challenge. (4) Return COMPLETE with per-finding assessment.
- Constraint: Read-only -- no file modifications, no bash. Only Read, Grep, Glob tools. This agent is an adversary to the hunter, not a collaborator.
- RAPID:RETURN data schema: { assessments: [{ findingId, verdict: 'agree'|'disagree'|'uncertain', evidence: string, reasoning: string }] }

**role-judge.md:**
- Purpose: Produce final ruling on each contested finding based on hunter and advocate evidence
- Input: Receives hunter findings AND advocate assessments
- Flow: (1) For each finding, weigh hunter evidence against advocate challenge. (2) Produce ruling: ACCEPTED (real bug, should fix), DISMISSED (false positive or irrelevant), DEFERRED (needs human judgment -- contested with strong evidence on both sides). (3) For ACCEPTED rulings, assign fix priority (1=critical/immediate, 2=high, 3=medium, 4=low). (4) For DEFERRED rulings, include both hunter and advocate evidence summaries for human review. (5) Write REVIEW-BUGS.md at the wave path with all rulings. (6) Return COMPLETE with rulings.
- Constraint: Can Read and Write (for REVIEW-BUGS.md). Cannot modify source files.
- RAPID:RETURN data schema: { rulings: [{ findingId, ruling: 'ACCEPTED'|'DISMISSED'|'DEFERRED', priority?: number, reasoning: string, hunterEvidence: string, advocateEvidence: string }], accepted: number, dismissed: number, deferred: number }

**role-bugfix.md:**
- Purpose: Fix accepted bugs with targeted, atomic changes
- Input: Receives array of ACCEPTED rulings with file/line/description from judge
- Flow: (1) For each accepted bug (in priority order), read the file and understand the issue. (2) Apply the minimal fix. (3) Run existing tests to verify no regressions. (4) Stage and commit the fix with message format: `fix({setId}): {brief description}`. (5) Track which files were modified (for scope narrowing in re-hunts). (6) Return COMPLETE with fix results.
- Constraint: Only modify files referenced in the accepted bugs. Atomic commits per bug or per logical group. If a fix breaks tests, revert and report as unfixable.
- RAPID:RETURN data schema: { fixed: [{ findingId, file, commitHash }], unfixable: [{ findingId, reason }], modifiedFiles: string[] }

**role-uat.md:**
- Purpose: Generate multi-step acceptance test plans and execute automated steps via browser automation
- Input: Receives JOB-PLAN.md acceptance criteria, WAVE-CONTEXT.md decisions, and browser automation config (chrome-devtools or playwright)
- Flow: (1) Derive test scenarios from acceptance criteria and user decisions. (2) For each scenario, create steps tagged as [automated] or [human]. (3) Output test plan in RAPID:RETURN with status=CHECKPOINT for user approval. (4) When resumed, execute [automated] steps using the configured browser automation tool. For [human] steps, describe what to verify and include screenshot if web-based, then pause with CHECKPOINT. (5) After all steps, produce REVIEW-UAT.md and return COMPLETE with results.
- Constraint: Automated steps use the configured MCP tool (Playwright MCP or Chrome DevTools MCP). Failed steps are logged with evidence but execution continues. Dev server readiness check before first automated step.
- RAPID:RETURN data schema: { testPlan: [{ step, type: 'automated'|'human', description, expected }], results: [{ step, type, status: 'pass'|'fail'|'skipped', evidence?: string }], passed: number, failed: number, skipped: number }
