    - Input specification (what context it receives)
    - Output specification (exact file path, format, structure)
    - Behavioral constraints

    **role-codebase-synthesizer.md** (INIT-03):
    - Purpose: Deep analysis of brownfield codebases
    - Inputs: Project directory path, scan manifest from context.cjs detectCodebase/buildScanManifest
    - Analyzes: file structure, key functions/classes, API endpoints, code style conventions, tech stack, dependency graph, testing patterns
    - Output: Structured markdown report (CODEBASE-ANALYSIS.md in .planning/research/) with sections for each analysis area
    - Uses Read tool to examine source files, Grep for pattern detection
    - Must NOT modify any source files
    - Must include specific file paths and line numbers as evidence

    **role-research-stack.md** (INIT-04):
    - Purpose: Research the project's technology stack and dependencies
    - Inputs: Project description, brownfield analysis (if available)
    - Investigates: framework versions, dependency health, compatibility issues, upgrade paths
    - Output: .planning/research/STACK.md
    - Should use Context7 MCP for documentation lookups when available, WebFetch as fallback
    - Scope boundary: Stack/dependencies ONLY, not features or architecture

    **role-research-features.md** (INIT-04):
    - Purpose: Research feature implementation approaches
    - Inputs: Project description, user's feature requirements, brownfield analysis (if available)
    - Investigates: feature decomposition, implementation strategies, third-party service options, data models
    - Output: .planning/research/FEATURES.md
    - Scope boundary: Feature planning ONLY, not stack choices or pitfalls

    **role-research-architecture.md** (INIT-04):
    - Purpose: Research architectural patterns and design decisions
    - Inputs: Project description, tech stack info, brownfield analysis (if available)
    - Investigates: project structure, API design, data flow, state management, deployment patterns
    - Output: .planning/research/ARCHITECTURE.md
    - Scope boundary: Architecture ONLY, not specific feature implementations

    **role-research-pitfalls.md** (INIT-04):
    - Purpose: Research known failure modes and common mistakes
    - Inputs: Project description, tech stack, brownfield analysis (if available)
    - Investigates: known bugs in dependencies, common anti-patterns for the stack, performance traps, security vulnerabilities
    - Output: .planning/research/PITFALLS.md
    - Scope boundary: Known failure modes ONLY, not cross-cutting concerns (those go to Oversights)

    **role-research-oversights.md** (INIT-04):
    - Purpose: Capture cross-cutting concerns and things easy to miss
    - Inputs: Project description, tech stack, brownfield analysis (if available)
    - Investigates: accessibility, internationalization, error handling patterns, logging/observability, CI/CD, environment config, testing strategy, documentation needs
    - Output: .planning/research/OVERSIGHTS.md
    - Scope boundary: Cross-cutting concerns that span multiple features/systems, NOT specific pitfalls

    **role-research-synthesizer.md** (INIT-05):
    - Purpose: Read all 5 research outputs and produce unified SUMMARY.md
    - Inputs: All 5 research files from .planning/research/ (STACK.md, FEATURES.md, ARCHITECTURE.md, PITFALLS.md, OVERSIGHTS.md)
    - Produces: .planning/research/SUMMARY.md with deduplicated insights, cross-references between research areas, priority rankings, and a recommended approach section
    - Must NOT introduce new research -- only synthesize, deduplicate, and prioritize existing findings
    - Must flag contradictions between research outputs

    **role-roadmapper.md** (INIT-06):
    - Purpose: Create project roadmap with sets/waves/jobs hierarchy and interface contracts
    - Inputs: .planning/research/SUMMARY.md, project description, team size, model selection
    - Produces: Structured JSON output containing:
      1. ROADMAP.md content (human-readable sets/waves/jobs)
      2. STATE.json milestone/set/wave/job structure
      3. CONTRACT.json content for each set (exports, imports, behavioral contracts)
    - Set boundary principle per user decision: minimize cross-set dependencies, consider number of developers, optimize for ease of merge
    - Contracts must be unified across sets (all generated together, cross-referenced)
    - Depth: sets with full contracts + wave stubs (ordering and job titles). Detailed job plans deferred to Phase 20
    - Uses propose-then-approve pattern: generates full proposal, expects feedback loop
    - Must NOT write files directly -- returns structured data for SKILL.md to write via CLI
