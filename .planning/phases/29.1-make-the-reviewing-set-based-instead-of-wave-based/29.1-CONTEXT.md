# Phase 29.1: Make the Reviewing Set-Based Instead of Wave-Based - Context

**Gathered:** 2026-03-10
**Status:** Ready for planning

<domain>
## Phase Boundary

Change the review pipeline from iterating per-wave (running the full unit test → bug hunt → UAT pipeline N times, once per wave) to reviewing at the set level (one pass across all changed files in the set branch). The `/rapid:review` skill, review.cjs library, and review CLI commands all shift from wave-scoped to set-scoped. Lean wave-level review (auto-runs during reconciliation) is unaffected.

</domain>

<decisions>
## Implementation Decisions

### Scope aggregation
- Single git diff of set branch vs main — all changed files across all waves in one scope
- Keep one-hop dependents (files that import changed files) in scope
- Drop wave gating — no longer check individual wave statuses before review. Set-level status check (executing/reviewing) is sufficient
- Remove wave argument from `/rapid:review` — review is set-only. No more `/rapid:review 1 1.1`

### Artifact organization
- Reports at set level: REVIEW-UNIT.md, REVIEW-BUGS.md, REVIEW-UAT.md all live at `.planning/waves/{setId}/`
- Findings tagged with originating wave using job plan file lists (JOB-PLAN.md per wave lists target files → map findings to waves by file overlap)
- Keep REVIEW-SUMMARY.md as a separate top-level aggregation file at `.planning/waves/{setId}/`
- Replace `scopeWaveForReview` with `scopeSetForReview` in review.cjs (existing function repurposed, not additive)

### Review agent context
- Chunked by directory when scope is large — files in the same parent directory stay together for coherent analysis
- Bug hunt: hunters run per-chunk in parallel, all findings merged, then one advocate + one judge pass on merged findings
- Unit test: chunked by directory (same as bug hunt)
- UAT: runs once on full set scope — UAT tests user workflows, not individual files
- Re-hunts (cycles 2-3): flat scope on bugfix-modified files only, no re-chunking

### Issue tracking
- One REVIEW-ISSUES.json at set level: `.planning/waves/{setId}/REVIEW-ISSUES.json`
- Each issue tagged with originating wave (from wave attribution via job plan file lists)
- Bugfix agent commits in set worktree with set-scoped messages (no wave reference)
- CLI commands (review scope, log-issue, update-issue, summary) drop wave argument entirely — set-id only

### Claude's Discretion
- Directory chunking threshold (file count that triggers chunking vs single-agent pass)
- How to handle files that appear in multiple waves' job plans (attribution ambiguity)
- Chunk merging strategy for bug hunt findings before advocate/judge pass
- Unit test plan presentation format at set level (grouped by directory chunk or flat)
- How `review scope` CLI command output changes for set-level scoping

</decisions>

<specifics>
## Specific Ideas

- The current SKILL.md Step 3 "For Each Wave (Sequential)" becomes a single set-level pass — the per-wave loop is removed entirely
- Wave attribution from JOB-PLAN.md file lists is best-effort — files not in any job plan get tagged as "unattributed"
- The chunking-by-directory approach naturally aligns with Phase 32's scoper agent (REV-01 through REV-04) — this phase simplifies the per-wave loop; Phase 32 later adds intelligent file categorization within the set scope
- Parallel chunk hunters align well with Claude Code's subagent parallelism (up to 5 concurrent)

</specifics>
