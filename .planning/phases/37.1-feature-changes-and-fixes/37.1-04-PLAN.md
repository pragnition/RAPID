  <files>skills/migrate/SKILL.md</files>
  <action>
    Create skills/migrate/SKILL.md with this structure:

    **Frontmatter:**
    ```yaml
    ---
    description: Migrate an existing project from another planning framework to RAPID conventions
    allowed-tools: Bash(rapid-tools:*), Read, Write, AskUserQuestion, Glob, Grep
    ---
    ```

    **Title:** `# /rapid:migrate -- Project Migration`

    **Intro:** "You are the RAPID migration assistant. This skill helps migrate existing projects from other planning frameworks (GSD, openspec, or custom structures) to RAPID conventions. It uses heuristic detection to identify the source framework, backs up existing files, and restructures with user confirmation at each step."

    **Step 1: Environment Setup** -- Standard env preamble.

    **Display Stage Banner** -- `node "${RAPID_TOOLS}" display banner migrate`

    **Step 2: Detect Framework** --
    ```bash
    DETECT_RESULT=$(node "${RAPID_TOOLS}" migrate detect 2>&1)
    ```
    Parse the JSON result. Display findings to the user:
    - Framework detected: {type} (confidence: {confidence})
    - Artifacts found: list each artifact with its path and description
    - If type is 'none': "No existing planning structure detected. Nothing to migrate. Consider running /rapid:init instead." Then STOP.

    **Step 3: User Confirmation for Detection** --
    Use AskUserQuestion:
    - "Proceed" -- "Looks correct, continue with migration"
    - "Cancel" -- "Don't migrate, exit"

    If Cancel: display "Migration cancelled." and STOP.

    **Step 4: Backup** --
    Display: "Creating backup of .planning/ to .planning.bak/..."
    ```bash
    BACKUP_RESULT=$(node "${RAPID_TOOLS}" migrate backup 2>&1)
    ```
    If backup fails (e.g., .planning.bak/ already exists): display error, suggest removing old backup, and STOP.
    If success: display "Backup created at .planning.bak/"

    **Step 5: User Confirmation for Transform** --
    Use AskUserQuestion:
    - "Transform" -- "Restructure files to RAPID conventions"
    - "Cancel" -- "Keep backup only, don't transform"

    If Cancel: display "Backup created but no transformation applied. Your original files are at .planning.bak/" and STOP.

    **Step 6: Transform** --
    Pass the detection result to the transform command:
    ```bash
    echo '${DETECT_RESULT}' | node "${RAPID_TOOLS}" migrate transform 2>&1
    ```
    Parse the result. Display:
    - Files transformed: list each
    - Files skipped: list each with reason
    - Errors: list each (if any)

    If errors occurred: warn but don't fail. The backup is available for manual recovery.

    **Step 7: Verification** --
    Display "Migration complete. Running /rapid:status to verify..."

    Instruct the user to run `/rapid:status` to see their project in RAPID format. Do NOT auto-run another skill.

    Display:
    ```
    > **Migration complete.**
    > Backup: .planning.bak/
    > Run `/rapid:status` to view your project in RAPID format.
    > Run `/rapid:init` if you need to set up state tracking from scratch.
    ```

    **Important Notes:**
    - Backup is mandatory before any transformation
    - User confirmation required at each step (detection, backup, transform)
    - If .planning.bak/ already exists, migration refuses to proceed (prevents double-migrate accidents)
    - Framework-agnostic: detection uses heuristics, not hard-coded framework knowledge
    - The migration restructures directory layout and file naming, but does NOT rewrite content (decisions, roadmaps, etc. are preserved as-is)

<task type="auto">
  <name>Task 2: Create /rapid:quick skill</name>
  <files>skills/quick/SKILL.md</files>
  <action>
    Create skills/quick/SKILL.md with this structure:

    **Frontmatter:**
    ```yaml
    ---
    description: Run a quick task on the current branch with automatic state tracking and commit
    allowed-tools: Bash(rapid-tools:*), Read, Write, Edit, AskUserQuestion, Glob, Grep
    ---
    ```

    **Title:** `# /rapid:quick -- Quick Task`

    **Intro:** "You are the RAPID quick task executor. This skill handles ad-hoc tasks that developers want to complete without the full set/wave/job ceremony. Quick tasks work directly on the current branch, are tracked in project state, and auto-commit when done. There are no guardrails on task size -- trust the developer."

    **Step 1: Environment Setup** -- Standard env preamble.

    **Display Stage Banner** -- `node "${RAPID_TOOLS}" display banner quick`

    **Step 2: Get Task Description** --
    The user provides the task description as an argument: `/rapid:quick Fix the login button alignment`

    If no description provided, use AskUserQuestion to ask: "What quick task would you like to complete?"

    Store the description.

    **Step 3: Get Next Task ID** --
    ```bash
    NEXT_ID=$(node "${RAPID_TOOLS}" quick next-id 2>&1)
    ```

    **Step 4: Execute the Task** --
    Actually perform the requested work. This is the core of the skill -- read the codebase, understand the request, make the changes. Use all available tools (Read, Write, Edit, Bash, Glob, Grep) to implement the task.

    This step has no rigid structure -- adapt to whatever the task requires. The skill trusts the developer's judgment on task scope.

    **Step 5: Auto-Commit** --
    After the task is complete:
    1. Stage all changed files: `git add -A`
    2. Create a descriptive commit message based on what was done
    3. Commit: `git commit -m "{descriptive message}"`
    4. Capture the commit hash: `COMMIT_HASH=$(git rev-parse HEAD)`

    **Step 6: Track in State** --
    ```bash
    node "${RAPID_TOOLS}" quick add "{description}" --commit "{COMMIT_HASH}" --dir "{quick-task-dir}"
    ```

    Where quick-task-dir is a slugified version of the task description (first 50 chars, lowercased, spaces-to-hyphens).

    Note: The quick add CLI command may need to accept the commit hash and directory as additional arguments. If the CLI from Plan 01 only accepts description, pass the full info as a structured description string instead: `"{id}-{slug}: {description}"`.

    **Step 7: Summary** --
    Display:
    ```
    --- Quick Task Complete ---
    Task #{id}: {description}
    Commit: {commit_hash}
    Branch: {current_branch}
    ---------------------------
    ```

    Display next step:
    ```
    > **Next step:** `/rapid:status` to view project state
    ```

    **Important Notes:**
    - Quick tasks work on the CURRENT branch (no worktree isolation)
    - No size guardrails -- the developer decides what constitutes a "quick" task
    - Auto-commit is mandatory -- every quick task produces a commit
    - Task is tracked in STATE.json via the quickTasks array
    - Quick tasks can be run between sets, during sets, or anytime
    - If the task fails partway, do NOT auto-commit. Inform the user of the failure and let them decide what to do.
  </action>
