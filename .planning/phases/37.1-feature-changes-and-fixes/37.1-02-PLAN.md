    Rewrite skills/discuss/SKILL.md from scratch. The new structure:

    **Frontmatter:**
    ```yaml
    ---
    description: Capture developer implementation vision for a set via structured discussion
    allowed-tools: Bash(rapid-tools:*), Read, AskUserQuestion, Glob, Grep, Agent
    ---
    ```

    **Title:** `# /rapid:discuss -- Set Discussion`

    **Intro paragraph:** "You are the RAPID set discussion facilitator. This skill captures developer implementation vision for an entire set before autonomous planning begins. It identifies product and UX gray areas across all waves, conducts a single-round discussion per area, and splits decisions into per-wave WAVE-CONTEXT.md files."

    **Step 1: Environment Setup** -- Keep existing env preamble pattern exactly.

    **Display Stage Banner** -- Keep existing pattern: `node "${RAPID_TOOLS}" display banner discuss`

    **Step 2: Resolve Set** -- Accept set reference (not wave). User invokes as:
    - `/rapid:discuss 1` (numeric index)
    - `/rapid:discuss auth-system` (set ID)

    Use `node "${RAPID_TOOLS}" resolve set "<user-input>"` to resolve. Parse JSON result for milestoneId, setId, setIndex. Fail if set not found.

    **Step 3: Load Set Context** -- Read ALL waves in the set:
    - Get set state: `node "${RAPID_TOOLS}" state get set <milestoneId> <setId>`
    - For each wave in set: read `.planning/waves/{setId}/{waveId}/` artifacts (CONTRACT.json, DEFINITION.md, WAVE-PLAN.md if exists)
    - Read the set's CONTRACT.json at `.planning/sets/{setId}/CONTRACT.json`
    - Read SET-OVERVIEW.md if exists at `.planning/sets/{setId}/SET-OVERVIEW.md`
    - Build a unified context picture of what the set needs to accomplish across all waves

    **Step 4: Status Check and State Transition** --
    - Check wave statuses. For discussion to proceed, waves should be `pending` or `discussing` (for re-discuss).
    - If any wave is already `planning` or later AND there are still `pending` waves: warn and ask to confirm partial re-discuss.
    - If ALL waves are `planning` or later: offer "Re-discuss" option (will overwrite existing WAVE-CONTEXT.md files).
    - Transition all `pending` waves to `discussing` status:
      ```bash
      node "${RAPID_TOOLS}" state transition wave <milestoneId> <setId> <waveId> discussing
      ```
      Do this in a loop for each pending wave.

    **Step 5: Identify Gray Areas** --
    - Analyze the set's scope across all waves.
    - Identify gray areas that need developer input. Focus ONLY on:
      - Product/UX decisions (what should the user experience be?)
      - High-level developer preferences (coding style choices, library preferences)
      - Cross-wave coordination decisions (how should waves interact?)
    - Do NOT surface:
      - Technical implementation details (which function to use, how to structure code)
      - Architecture decisions that follow established patterns
      - Anything Claude can decide autonomously
    - Present gray areas as a numbered list. If zero gray areas found, skip to Step 7.

    **Step 6: Single-Round Discussion** --
    For EACH gray area (in order):
    - Present ONE AskUserQuestion with:
      - Clear description of the gray area
      - 2-4 concrete options (named, with brief description)
      - An "Other" option: "Something else (describe what you'd prefer)"
    - Record the user's choice. Move immediately to the next gray area.
    - Do NOT show "Looks good" after each area.
    - Do NOT do a follow-up "Round 2" for specifics.

    After ALL gray areas are answered:

    **Step 6b: Final Confirmation** --
    Use AskUserQuestion with exactly two options:
    - "Looks good" -- "Proceed with these decisions"
    - "Revise" -- "I want to change something"

    Do NOT show a summary of decisions before asking. Just ask.

    If "Revise": Ask the user for freeform text describing what to change. Then re-run the discussion from Step 5 with the revision context in mind (re-present gray areas, incorporating the user's revision feedback).

    If "Looks good": proceed to Step 7.

    **Step 7: Write Per-Wave WAVE-CONTEXT.md Files** --
    - Ensure wave directories exist: `node "${RAPID_TOOLS}" wave-plan create-wave-dir "${SET_ID}" "${WAVE_ID}"` for each wave.
    - For each wave in the set, determine which decisions are relevant to that wave (based on the wave's scope from CONTRACT.json and DEFINITION.md).
    - Write a WAVE-CONTEXT.md file at `.planning/waves/{setId}/{waveId}/WAVE-CONTEXT.md` for each wave containing:
      - Decisions relevant to that wave
      - Developer preferences that apply
      - Any cross-wave coordination notes relevant to this wave
    - Decisions that apply to ALL waves should appear in ALL WAVE-CONTEXT.md files.

    **Step 8: Next Step** --
    Display: `> **Next step:** /rapid:plan {setIndex}`
    Where setIndex is the numeric index from the resolve step.

    **Important Notes section** at the end:
    - Set-scoped discussion: All waves discussed in a single session, decisions split to per-wave files.
    - Product/UX focus: Do not surface technical implementation details as gray areas.
    - Single round: One question per gray area, no follow-up rounds.
    - Final gate only: "Looks good" appears once at the end, not per-area.
    - Re-discuss: If all waves are already discussed/planned, offer re-discuss option.
    - No wave-level invocation: This skill accepts set references only (e.g., `/rapid:discuss 1`), not wave dot-notation.
