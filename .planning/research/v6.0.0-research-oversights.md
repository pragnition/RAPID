# Oversights Research -- Scale & Quality (v6.0.0)

## Cross-Cutting Concerns

### 1. DAG Schema Migration (v1 to v2 Coexistence)

The milestone elevates DAG.json from a secondary artifact to the central orchestration concept. Currently, two DAG schemas coexist: `createDAG` (v1, no `version` field, `sets` key in waves) and `createDAGv2` (v2, `version: 2`, `nodes` key in waves, typed nodes with `set`/`wave`/`job`). The existing `recalculateDAG` in `add-set.cjs` (line 16) calls `createDAG` (v1), while `tryLoadDAG` in `dag.cjs` returns whatever is on disk without checking version. `getExecutionOrder` reads `dag.waves[waveNum].sets` -- which only exists in v1 format. Adding DAG-driven group annotations (developer groups, cross-group dependencies) requires deciding which schema version becomes canonical.

**What gets skipped:** A migration path for existing DAG.json files. Projects created under v5 will have v1 DAGs. If v6 changes the schema to support group annotations, `tryLoadDAG` will silently load a v1 DAG and downstream code that expects group metadata will get `undefined`.

**Recommendation:** Define an explicit `dag version detect -> migrate -> validate` pipeline early. The `migrate.cjs` module already has version detection patterns for STATE.json; apply the same approach to DAG.json. Add a `dagVersion` field to the return of `tryLoadDAG`.

### 2. STATE.json Partial Write Protection (Bug Fix #3 Broader Pattern)

Bug fix #3 addresses the specific case where the roadmapper's incomplete state overwrites a valid STATE.json. However, this is a symptom of a broader pattern: Step 9 of init uses `Write tool` to write STATE.json directly, bypassing `withStateTransaction()`. The SKILL.md orchestrator writes files with the Write tool because agents are instructed not to write state directly -- but the skill itself does not validate the data against `ProjectState` Zod schema before writing.

**What gets skipped:** Schema validation at the skill layer. Currently validation only happens inside `withStateTransaction` and `readState`. The Write tool has no awareness of state schemas. Any skill step that writes STATE.json via the Write tool (init Step 9c, new-version, add-set) can produce an invalid file without detection until the next `readState` call.

**Recommendation:** Add a `state validate` CLI subcommand that accepts JSON on stdin and validates it against `ProjectState` schema without writing. Skills should call this before writing. Alternatively, route all state writes through `state set-full` CLI command that validates before writing.

### 3. Developer Group Assignment Has No Persistence Layer

Feature #2 (DAG-driven set grouping) requires assigning sets to developer groups and tracking cross-group dependencies. Currently, there is no field in `SetState` (state-schemas.cjs) for `group` or `developerGroup`. The DAG `nodes` array has `id`, `wave`, `status`, and (in v2) `type` -- but no `group` annotation. The OWNERSHIP.json tracks per-file ownership by set, but has no concept of groups of sets.

**What gets skipped:** The data model for developer groups. This is not just a display concern -- the scaffold command needs group info to generate stubs per group, the merge pipeline needs it to prioritize intra-group merges, and the DAG show command needs it for visualization. If groups are only stored in ROADMAP.md prose (not machine-readable), downstream tools cannot consume them.

**Recommendation:** Extend DAG node schema with an optional `group` field. Extend state schema with optional `group` on `SetState`. Define group semantics early: is a group just a label, or does it carry constraints (e.g., "group A cannot merge until group B's stubs are replaced")? The answer affects every downstream consumer.

### 4. Stub Lifecycle Management on Shared Branch

Feature #3 (scaffold overhaul) introduces stubs that live on a shared branch and get replaced by real implementations. The current `stub.cjs` generates stubs into `.rapid-stubs/` inside a single set's worktree -- a local, ephemeral artifact. The new model places stubs on a shared branch visible to all developers. This fundamentally changes stub lifecycle:

- **Creation:** After init/scaffold, stubs are committed to main (or a shared branch).
- **Consumption:** All set worktrees see stubs via their branch ancestry.
- **Replacement:** When a set implements the real code, the stub file must be deleted and the real file committed -- but other set worktrees still reference the stub path.
- **Merge order:** If set A's branch still has the stub when set B's real implementation is merged, the next merge of set A will have a file that was deleted by set B.

**What gets skipped:** The stub replacement protocol. Who deletes the stub? How does a worktree know whether it should use the stub or the real implementation? The current `cleanupStubFiles` removes `.rapid-stubs/` from a single worktree, but shared-branch stubs require coordinated replacement.

**Recommendation:** Define the replacement invariant before implementing: "When a set is merged, any stubs it provides are atomically replaced by real files. Dependent worktrees must rebase or merge main to pick up real implementations." This needs to be documented in the merge pipeline and enforced by a pre-merge check (e.g., "set X still imports from stub -- rebase first").

### 5. Foundational Set #0 Ordering in DAG

Feature #3 mentions an "optional foundational set #0 as shared work before groups branch." The current DAG assigns waves based on dependency edges. A foundational set with no outgoing edges but that every other set depends on would naturally land in Wave 1 by Kahn's algorithm. However, there is no concept of a "special" set in the current state model -- all sets are treated uniformly.

**What gets skipped:**
- How set #0 is distinguished in state (is it a regular set with status `pending -> merged`, or a different entity?).
- Whether set #0 is executed by a single developer or is pre-execution shared setup.
- Whether set #0's completion is a prerequisite for `start-set` on any group's sets (currently, `start-set` does not check DAG dependencies -- it only checks that the set exists and has status `pending`).

**Recommendation:** If set #0 is implemented as a regular set, enforce DAG ordering in `start-set`: a set cannot be started if its DAG dependencies are not `merged`. This is currently not checked. If set #0 is a different concept (pre-set scaffold), it needs its own lifecycle status.

### 6. PRINCIPLES.md Propagation to Agents

Feature #5 (meta principles) captures architectural/style decisions in `PRINCIPLES.md` and a summary in `CLAUDE.md`. Currently, agents receive context via `generateScopedClaudeMd` (worktree.cjs line 795), which reads the style guide from `.planning/context/STYLE_GUIDE.md` and the set's definition/contract. There is no mechanism to inject `PRINCIPLES.md` into agent context.

**What gets skipped:**
- The `assembleExecutorPrompt` function in `execute.cjs` does not read PRINCIPLES.md.
- The `buildBrandingContext` pattern (execute.cjs line 35) shows how optional context files are loaded -- PRINCIPLES.md needs a parallel `buildPrinciplesContext` function.
- Research agents spawned in Step 7 of init receive only the project brief. If principles are captured in Step 4 of init, they need to be included in research agent prompts.
- The `generateScopedClaudeMd` function in worktree.cjs loads style guide but not principles. Both discuss-set and plan-set agents also need this context.

**Recommendation:** Add PRINCIPLES.md loading to three injection points: (1) `generateScopedClaudeMd` for execution, (2) `assembleExecutorPrompt` for all phases, (3) research agent prompt templates in init SKILL.md. Follow the `buildBrandingContext` pattern -- graceful if missing, formatted as a context section.

### 7. Spec File Validation and Format Standardization

Feature #4 (/init spec file integration) allows users to provide spec files during init. The new-version skill already implements `--spec` parsing (new-version/SKILL.md Step 0.5), but its format expectation is loose: "The spec file uses structured Markdown with category headings... Use LLM understanding to match headings to categories even if the exact heading text differs."

**What gets skipped:**
- No validation of spec file format before passing to research agents. A malformed spec (e.g., a binary file, an HTML export, a PDF) will be passed as raw content and silently produce poor research.
- No standardized spec file template for users to follow. The new-version skill mentions `## Features`, `## Bug Fixes`, etc. but there is no `.planning/SPEC-TEMPLATE.md` for user reference.
- Research agents that "build on existing findings" (rather than starting fresh) need a defined protocol for how to incorporate spec content. Currently, research agents receive only a project brief. If a spec provides partial architecture decisions, the architecture research agent needs to know which decisions are settled vs. open.

**Recommendation:** Create a lightweight spec file schema (a Markdown template with required sections and optional sections). Validate that the file is readable Markdown with at least one recognized section heading before passing to the pipeline. Mark spec-provided content with a `[FROM SPEC]` tag so research agents can distinguish user-provided constraints from open questions.

### 8. Cross-Group Testing Infrastructure

Feature #3 mentions "all we need for these stubs is to be able to allow observable testing between groups of sets." The current test infrastructure (`detectTestFrameworks` in context.cjs, `testFrameworks` in config.json) is language-level: it detects frameworks like jest, pytest, node:test. There is no concept of cross-group integration tests.

**What gets skipped:**
- How stubs enable testability. A stub that throws `Error('Stub: not yet implemented')` is not testable -- it crashes the test. Stubs need to return mock data (configurable fixtures) or behave as no-ops to enable meaningful integration testing.
- Where cross-group test files live. Each worktree is isolated. A test in group A's worktree that tests integration with group B's API will fail when group B's code is still a stub. Should integration tests live in the shared branch alongside stubs?
- How test results from one group inform another group's work. Currently there is no cross-set test result sharing mechanism.

**Recommendation:** Evolve `generateStub` (stub.cjs) to produce functional mock stubs rather than throwing stubs. Each stub function should return a configurable default value (or accept a fixture file). Add an integration test directory convention (e.g., `tests/integration/`) on the shared branch where cross-group tests can validate stub contracts.

### 9. Init Step Ordering and Idempotency

The bug fixes (#1 REQUIREMENTS.md overwrite, #3 STATE.json overwrite) reveal a broader concern: init steps are not idempotent and their ordering creates race conditions. If init fails at Step 8 and the user re-runs, Steps 1-7 execute again, potentially overwriting user edits made between runs.

**What gets skipped:**
- No checkpoint/resume for init. If init fails at Step 7 (research), the user must re-run the entire pipeline. Steps 1-6 cannot be skipped even though their artifacts are already on disk.
- The `--mode upgrade` path in `scaffoldProject` (init.cjs line 278) correctly skips existing files, but the SKILL.md does not use upgrade mode for re-runs -- it defaults to fresh mode.
- The research agents in Step 7 always write fresh output, even if valid research files already exist from a previous run. This wastes tokens and API costs on re-runs.

**Recommendation:** Add a `init status` CLI command that reports which init steps have completed (based on file existence: STATE.json exists = scaffold done, research/*.md exist = research done, ROADMAP.md exists = roadmap done). The skill can then skip completed steps on re-run. This is low cost to implement (file existence checks) and high value for user experience during failure recovery.

### 10. Flag Naming Conventions and CLI Consistency

Bug fix #2 (--desc vs --description) is a specific instance of a broader consistency gap. Looking at the CLI help text:

- `init scaffold --name <n> --desc <d>` uses `--desc` (abbreviated)
- `quick log --description <d>` uses `--description` (full form)
- No flags use a `--long-form / --short-alias` pattern (e.g., `--desc` is not documented as a shorthand for `--description`)

**What gets skipped:** A flag naming standard for the CLI. As the CLI grows (DAG group commands, scaffold stubs, spec file path), inconsistent flag naming creates agent confusion. LLM agents are particularly sensitive to flag mismatches because they follow SKILL.md instructions literally.

**Recommendation:** Standardize on full-form flags as the canonical version. Add a `parseArgs` utility (the project already has `src/lib/args.cjs`) that supports aliases. Document each command's flags in a single location (tool-docs.cjs already has command descriptions; extend with flag lists). Fix the mismatch by accepting both `--desc` and `--description` as aliases.

### 11. Agent Subagent Namespace Enforcement

The tech debt item (agents must only call rapid: prefixed subagents) is currently only documented in `core-identity.md` as a prose instruction (line 41: "Only use `rapid:*` skills and commands"). This is enforced by LLM compliance, not by code.

**What gets skipped:**
- There is no runtime validation that spawned subagents match the `rapid-*` naming pattern. The Agent tool accepts any agent name.
- Skills list non-rapid skills in system context (the system-reminder shows `superpowers:*`, `gsd-merge:*`, `p-research:*`, etc.). An agent following instructions may still be confused by seeing these skills listed.
- The `allowed-tools` field in SKILL.md frontmatter restricts tool access but not agent spawning. There is no `allowed-agents` equivalent.

**Recommendation:** This is fundamentally an LLM-behavioral concern rather than a code-enforcement concern, because Claude Code's plugin system does not support agent spawn filtering. The most effective fix is: (1) strengthen the prose instruction in core-identity.md to include an explicit deny list of known non-rapid namespaces, (2) add a post-execution check in the returns parser that flags if an agent's output mentions spawning non-rapid agents, (3) file a feature request with Claude Code for `allowed-agents` in skill frontmatter.

### 12. REQUIREMENTS.md Append-vs-Overwrite Pattern

Bug fix #1 (REQUIREMENTS.md overwrite) has a broader implication: the init pipeline has multiple write-to-same-file patterns that lack conflict awareness. Step 4D writes user-approved criteria to REQUIREMENTS.md. Step 5 scaffolds, which generates a fresh REQUIREMENTS.md template. The fix (per SKILL.md lines 458-460, which already describe the append-on-existing pattern) needs to be propagated to `scaffoldProject` in init.cjs.

**What gets skipped:**
- The `scaffoldProject` function's file generators map (init.cjs line 236) generates all 6 files unconditionally in `fresh` mode. There is no per-file exclusion list. Fixing the REQUIREMENTS.md overwrite requires either removing REQUIREMENTS.md from `fileGenerators` in fresh mode, or making scaffold aware of which files have been populated by earlier init steps.
- The same pattern could affect STATE.json if the order of Steps 5 and 9 is not carefully maintained.
- The `upgrade` mode already handles this correctly (skips existing files), but `fresh` and `reinitialize` do not.

**Recommendation:** Introduce a `protectedFiles` parameter to `scaffoldProject` that lists files the scaffold should not overwrite even in fresh mode. The init skill passes `['REQUIREMENTS.md']` after Step 4D has written criteria. This is a 5-line change to init.cjs that prevents the entire class of same-file overwrite bugs.

### 13. DAG Show Command Does Not Display Groups or Dependencies

The `dag show` command (commands/dag.cjs line 43) currently displays sets grouped by wave with status colors. With the introduction of developer groups and cross-group dependencies, the display needs to show:

- Which developer group each set belongs to
- Cross-group dependency edges (not just wave ordering)
- Group completion status (all sets in group complete?)

**What gets skipped:** The display layer. Implementing DAG groups in the data model without updating `dag show` means the primary developer-facing tool for understanding project structure will not reflect the new grouping concept. Similarly, the `/rapid:status` skill (which calls `formatMarkIIStatus` in worktree.cjs) will not show group information.

**Recommendation:** Plan the `dag show` enhancements as part of the DAG central concept set, not as a separate set. Display is cheap to implement but expensive to retrofit if the data model is designed without considering display needs. Include `--group` filter flag and ASCII dependency graph in the initial implementation.

### 14. Stub File Extension and Language Generality

The current `generateStub` function (stub.cjs) generates CommonJS (`.cjs`) stubs exclusively. The scaffold overhaul targets "high-fidelity stubs for cross-group testing" which implies stubs for any language the target project uses. The `classifyProjectType` in scaffold.cjs detects `webapp`, `api`, `library`, `cli` types but not language. `detectCodebase` in context.cjs detects languages.

**What gets skipped:** Multi-language stub generation. Python projects need `.py` stubs, Go projects need `.go` stubs, TypeScript projects need `.ts` stubs. The current stub generator is CommonJS-only. Extending it to support multiple languages is a significant design decision: do you have one `generateStub` per language, or a template-based system?

**Recommendation:** Design stub generation as language-pluggable from the start. Define a `StubGenerator` interface: `{ canHandle(language): boolean, generateStub(contract, setName): string, fileExtension(): string }`. Register generators per language. This avoids a large refactor later when users request Python/Go/Rust stub support.

### 15. CLAUDE.md Principles Summary Length and Token Budget

Feature #5 stores a "summary in CLAUDE.md, full version in PRINCIPLES.md." CLAUDE.md is read by every agent at the start of every task. Each additional line in CLAUDE.md costs tokens across every agent invocation for the lifetime of the project.

**What gets skipped:** Token budget impact. If the principles summary is 50 lines, that is 50 lines multiplied by every agent invocation (6 research agents + synthesizer + roadmapper + N executors + reviewers). For a project with 10 sets and 3 waves each, that is roughly 100+ agent invocations, each paying the CLAUDE.md tax.

**Recommendation:** Enforce a strict line limit for the CLAUDE.md principles summary (10-15 lines maximum). The full PRINCIPLES.md can be as long as needed -- it is only loaded by `generateScopedClaudeMd` and `assembleExecutorPrompt`, which are selective context injection points. Document this constraint in the init skill's principles capture step.

### 16. Error Handling When DAG Dependencies Are Violated at Runtime

With DAG as the central concept, more operations will query DAG state. Currently, `start-set` does not check DAG dependencies, `execute-set` does not verify that dependency sets are merged, and `merge` uses DAG only for ordering (not for blocking). The DAG-central model implies stricter enforcement.

**What gets skipped:** Runtime enforcement of DAG constraints. If set B depends on set A and a developer tries to `execute-set B` before set A is merged, the executor will work against stubs. This may be intentional (stubs enable independent work), but there should be a warning. Without it, developers may not realize their execution is running against stubs vs. real implementations.

**Recommendation:** Add DAG dependency status checks to `start-set`, `execute-set`, and `merge` as informational warnings (not blockers, since stubs exist). Example: "Warning: set-api depends on set-auth (status: executing). Stubs will be used for set-auth exports." This is 10 lines of code per command but prevents confusion about why tests fail.

## Recommendations

Ordered by cost of adding later (most expensive to retrofit first):

1. **Developer group data model (#3):** Design the group schema on DAG nodes and state before writing any group-aware code. Decide: is a group a string label, an integer ID, or a structured object? Does it live in DAG.json, STATE.json, or both? Every downstream feature (scaffold stubs, dag show, merge ordering, status display) consumes this schema. Concrete first step: add optional `group: z.string().optional()` to `SetState` in state-schemas.cjs and `group` field to DAG node objects.

2. **Stub lifecycle on shared branch (#4):** Define the stub replacement protocol in writing before implementing scaffold stub generation. Key decisions: where do stubs live (main branch? dedicated `stubs` branch?), how are they replaced (merge deletes stub, real file takes precedence?), what happens when a worktree still references a deleted stub? Concrete first step: write a `STUB-LIFECYCLE.md` design doc in `.planning/` that the scaffold and merge sets reference.

3. **DAG schema migration (#1):** Commit to one DAG version for v6. The v2 schema already exists but is unused by `recalculateDAG`. Switch `recalculateDAG` to use `createDAGv2` and add a `tryMigrateDAG` function that converts v1 to v2 on load. Concrete first step: update `recalculateDAG` in add-set.cjs to call `createDAGv2` and update `getExecutionOrder` to handle both v1 (`.sets`) and v2 (`.nodes`) wave formats.

4. **PRINCIPLES.md propagation (#6):** Add principles loading alongside the existing style guide loading in `generateScopedClaudeMd`. Concrete first step: after the principles capture step in init, add `buildPrinciplesContext(cwd)` to execute.cjs following the `buildBrandingContext` pattern (graceful if missing, formatted as context section).

5. **STATE.json write validation (#2):** Route all skill-level state writes through a validating CLI command instead of raw Write tool. Concrete first step: add `state write-validated` subcommand that accepts JSON on stdin, validates against ProjectState schema, and writes atomically with locking.

6. **Foundational set #0 enforcement (#5):** Add DAG dependency check to `start-set`. Concrete first step: in worktree.cjs `setInit`, after checking set exists, load DAG and verify all dependency sets have status `merged` (or warn if not).

7. **Stub language generality (#14):** Make `generateStub` dispatch on language rather than hardcoding CJS. Concrete first step: extract current CJS generation into `generateCjsStub` and add a `generateStubForLanguage(contract, setName, language)` dispatcher.

8. **Flag naming consistency (#10):** Accept both `--desc` and `--description` as aliases. Concrete first step: in init.cjs line 32, add `case '--description':` as a fallthrough to `case '--desc':`.

9. **Init checkpoint/resume (#9):** Add `init status` command. Concrete first step: check for existence of STATE.json, research/*.md, ROADMAP.md, and return a JSON status object with `{ scaffold: boolean, research: boolean, synthesis: boolean, roadmap: boolean }`.

10. **REQUIREMENTS.md overwrite fix (#12):** Add `protectedFiles` parameter to `scaffoldProject`. Concrete first step: skip REQUIREMENTS.md generation in `scaffoldProject` when it already exists, regardless of mode.
