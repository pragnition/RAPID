# Domain & UX Research -- v4.2.0 (Discuss & Audit)

## Domain Conventions

RAPID operates in the **AI-assisted development orchestration** domain, specifically as a Claude Code plugin that manages parallel development through structured workflows. The closest comparable products are:

- **GSD (Get Shit Done)** -- The direct predecessor/inspiration. Uses milestone-based planning with audit-milestone workflow for gap detection. Presents findings as structured YAML with status-routed markdown output (Source: GSD audit-milestone.md workflow).
- **Cursor Composer / Windsurf Cascade** -- AI coding assistants with multi-file orchestration. Use inline question prompts with option cards for user decisions.
- **Nx / Turborepo** -- Monorepo orchestration tools. Use DAG-driven task execution with structured status tables for progress reporting.
- **Claude Code `/batch` skill** -- Claude's built-in parallel orchestration. Decomposes work into units, presents plans for approval, spawns parallel agents (Source: Claude Code skills documentation).

### Terminology conventions in this domain

| Term in RAPID | Industry Standard | Notes |
|---------------|-------------------|-------|
| Set | Phase / Epic / Work package | RAPID-specific, well-established in codebase |
| Gray area | Decision point / Design question | RAPID-specific term for architect-level choices |
| Milestone | Version / Release / Sprint | Standard; RAPID uses "milestone" and "version" interchangeably |
| Audit | Retrospective / Gap analysis / Verification | GSD uses "audit-milestone"; industry uses "retrospective" or "gap analysis" |
| Discuss | Discovery / Requirements gathering | Closest to "discovery session" in agile |

### Standard audit/review workflow patterns

Tools in this space follow a consistent pattern for post-completion audits (Source: GSD audit-milestone, Nx affected analysis, CI/CD pipeline reports):

1. **Automated scan** -- Machine-readable analysis of artifacts (no user input required during scan)
2. **Structured findings** -- Categorized by severity (critical/warning/info) or type (gap/debt/incomplete)
3. **Cross-reference validation** -- Multiple sources compared (requirements vs. implementation vs. tests)
4. **Actionable routing** -- Different next-step suggestions based on outcome (pass/gaps/debt)
5. **User confirmation gate** -- User decides what to act on vs. defer

## Interaction Models

### Primary interaction paradigm

RAPID uses a **CLI-driven, skill-orchestrated, question-prompt** interaction model. Users invoke slash commands (`/rapid:command`), and skills orchestrate multi-step workflows using `AskUserQuestion` for decision points.

### Input/output patterns users expect

1. **Banner + status summary first** -- Every skill displays a branded ANSI banner followed by current state summary. Users expect to see where they are in the workflow immediately (established pattern across init, discuss-set, plan-set, execute-set, new-version).

2. **AskUserQuestion for decisions** -- Structured prompts with pre-filled options. Users expect:
   - Clear question text with context (2-5 sentences)
   - Pre-filled options (not freeform) for bounded decisions
   - A "Claude decides" or skip option for low-priority items
   - `multiSelect: true` for checkbox-style selection

3. **Progress breadcrumb** -- Linear workflow indicator at completion and on errors:
   ```
   init [done] > start-set [done] > discuss-set [done] > plan-set > execute-set > review > merge
   ```

4. **Markdown tables for structured data** -- Pros/cons comparisons, status summaries, and findings use markdown tables. This is where the current discuss phase has alignment issues (see Recommendations).

5. **File artifact output** -- Every workflow step produces a named markdown or JSON file in `.planning/`. Users expect the file path to be displayed after writing.

### Feedback and progress indication conventions

- ANSI-colored banners with stage verbs (INITIALIZING, DISCUSSING SET, etc.)
- Breadcrumb trails showing workflow position
- Error messages prefixed with `[RAPID ERROR]` to stderr
- State transitions logged after completion

### Error communication patterns

- Structured recovery via AskUserQuestion (Retry / Skip / Cancel pattern)
- Error breadcrumbs showing failure point in workflow
- Descriptive error messages with clear next steps
- Never silent failures -- always surface to user with options

## Information Architecture

### How information should be organized and presented

**Audit Version findings** should follow the established pattern from GSD's audit-milestone while adapting to RAPID's terminology and file structure:

1. **Scan phase (automated):** Read all set definitions, verification reports, requirements, and ROADMAP.md. No user interaction during scan.
2. **Findings summary (structured table):** Present results as a markdown table with set-level granularity:
   ```
   | Set | Status | Requirements Met | Gaps Found | Severity |
   ```
3. **Detail drill-down:** For each gap, provide: what was expected, what was delivered, and evidence (file paths, missing exports, untested requirements).
4. **Action routing:** Based on findings, present different AskUserQuestion options:
   - **All passed:** Offer to close milestone or proceed to new-version
   - **Gaps found:** Offer to plan remediation sets or defer with written context
   - **Tech debt only:** Offer to accept debt or plan cleanup sets

### Navigation and discovery patterns

Users navigate RAPID through slash commands. The audit-version command should fit naturally into the lifecycle:

```
init > [start-set > discuss-set > plan-set > execute-set > review > merge]* > audit-version > new-version
```

The audit sits between the last merge and new-version, serving as a quality gate. Users should be able to run it at any time (even mid-milestone) for a progress snapshot.

### Content hierarchy and progressive disclosure

The discuss-set gray areas already use progressive disclosure well:
1. **Top level:** "Here are N areas that need your input" (multiSelect checkbox)
2. **Deep dive:** Individual questions only for selected areas
3. **Follow-up:** Only if gaps remain after all selected areas are covered

For audit-version, the same pattern applies:
1. **Top level:** Summary table (pass/fail per set)
2. **Detail level:** Gap descriptions per failing set (only shown for sets with issues)
3. **Action level:** Remediation options (only after user reviews findings)

## Accessibility Considerations

### Key accessibility patterns for the interaction model

RAPID operates in a terminal/CLI context within Claude Code. Key considerations:

- **Text-only output:** All ANSI formatting uses basic 16-color palette for maximum terminal compatibility (already implemented in display.cjs). No color-only information encoding -- text labels accompany all status indicators.
- **Structured options over freeform:** AskUserQuestion with pre-filled options is inherently more accessible than freeform text prompts -- users can select rather than type.
- **Consistent layout:** Tables should use fixed column widths where possible to aid screen reader navigation. The current pros/cons table misalignment issue reduces readability for all users.

### Screen reader and keyboard navigation

- Not directly applicable (Claude Code handles the UI chrome)
- AskUserQuestion options are navigable via keyboard in Claude Code's interface
- Markdown tables rendered by Claude Code are parseable by screen readers

### Color and contrast considerations

- ANSI colors are supplementary, not primary information carriers (already followed)
- Status indicators should use text labels ("PASS", "FAIL", "GAP") in addition to any color coding

### Inclusive design patterns

- Not applicable -- RAPID is a developer tool operating through Claude Code's accessibility layer. No additional accessibility infrastructure needed beyond what Claude Code provides.

## User Expectations

### Default behaviors users expect

Based on domain research, users of AI development orchestration tools expect:

1. **Audit should be non-destructive** -- Running audit-version should never modify state, create sets, or change files without explicit user confirmation. It is a read-only analysis followed by action prompts. (Source: GSD audit-milestone is read-only until user chooses an action; standard CI/CD audit patterns)

2. **Gray area consolidation should reduce prompt fatigue** -- When the discuss phase presents 8 or 12 gray areas across multiple AskUserQuestion prompts, users expect to handle selection efficiently. The current n-batch approach (2 prompts for 8 areas, 3 for 12) adds cognitive overhead vs. a single consolidated prompt. Users familiar with form-based UIs expect a single selection screen. (Source: todo.md user feedback)

3. **Table formatting should be consistent** -- Markdown tables in AskUserQuestion prompts should render with aligned columns. Users expect pros/cons tables to be scannable at a glance, not ragged. (Source: todo.md user feedback)

4. **Deferred items need written context** -- When audit-version identifies gaps that the user chooses to defer to the next version, users expect a written record of what was deferred and why. This prevents information loss across milestone boundaries. (Source: GSD audit-milestone captures deferred items; RAPID's own DEFERRED.md pattern in discuss-set)

5. **Audit results should be persistent** -- Users expect audit findings to be saved as a file (e.g., `.planning/v{version}-AUDIT.md`) that can be referenced later, not just shown ephemerally in the conversation. (Source: GSD writes `.planning/v{version}-MILESTONE-AUDIT.md`)

### Terminology users use vs. technical terms

- Users say "what did we miss" -- maps to "gap analysis" or "unsatisfied requirements"
- Users say "defer to next version" -- maps to "carry-forward with context"
- Users say "is everything done" -- maps to "milestone audit pass/fail"
- Users say "the table looks weird" -- maps to "markdown table column alignment issue"

### Mental models from competing products

- **From GSD:** Users familiar with GSD expect a 3-source cross-reference (verification reports, state, requirements) and status-routed output (passed/gaps/debt).
- **From agile retrospectives:** Users expect categorized findings (what went well, what was missed, action items).
- **From CI/CD pipelines:** Users expect pass/fail with drill-down detail and remediation suggestions.

### Onboarding and learning curve expectations

- Audit-version is a new command. Users need a clear help entry and it should appear in the workflow breadcrumb and `/rapid:status` output.
- The command should be discoverable from the natural workflow: after all sets are merged, `/rapid:status` should suggest running `/rapid:audit-version`.
- Gray area consolidation and table formatting are invisible improvements -- users should not need to learn anything new.

## Recommendations

1. **Adopt GSD's 3-source cross-reference for audit-version**: Compare ROADMAP.md requirements, set VERIFICATION-REPORTs, and STATE.json statuses to identify gaps. This is the industry-standard pattern for completeness audits. -- Priority: **critical** (core feature)

2. **Present audit findings as a summary table first, then detail on request**: Show a set-level pass/fail table, then use AskUserQuestion to let the user drill into specific gaps. This matches the progressive disclosure pattern already established in discuss-set. -- Priority: **critical** (core UX for audit feature)

3. **Route audit outcomes to specific next actions**: Three paths: (a) all passed -> offer `/rapid:new-version`, (b) gaps found -> offer to plan remediation sets with AskUserQuestion options, (c) defer -> write carry-forward context to a DEFERRED file for the next milestone. Mirrors GSD's action routing. -- Priority: **critical** (core feature)

4. **Consolidate gray area AskUserQuestion prompts from n batches to ceil(n/2) batches**: The user's request (todo.md) is to combine 8 gray areas into 2 questions with 4 checkboxes each instead of 2 separate 4-checkbox prompts. However, AskUserQuestion multiSelect currently batches at 4 options per prompt. The consolidation should increase options per prompt (e.g., 8 options in a single multiSelect prompt) rather than splitting across prompts. If Claude Code's AskUserQuestion supports >4 multiSelect options, use a single prompt for up to 8 areas and 2 prompts for 12. -- Priority: **high** (user-requested improvement)

5. **Fix pros/cons table alignment in discuss-set question prompts**: The markdown table in Format A (Step 6 of discuss-set) renders inconsistently because column content varies in length. Two approaches: (a) Use fixed-width padding in the table content (e.g., pad Pros/Cons cells to a minimum width), or (b) Switch from table format to a structured list format that does not depend on column alignment. Recommendation: Use padded table cells since users are already familiar with the table layout. -- Priority: **high** (user-reported issue)

6. **Write audit findings to persistent file**: Save audit results to `.planning/v{version}-AUDIT.md` (or `.planning/sets/AUDIT-{version}.md`) so findings are referenced across sessions. Include both machine-readable YAML frontmatter and human-readable markdown body, following GSD's pattern. -- Priority: **high** (data persistence expectation)

7. **Add `audit-version` to STAGE_VERBS and STAGE_BG in display.cjs**: Register the new command in the banner system with an appropriate verb ("AUDITING") and background color (bright red, matching review/merge as a quality gate stage). -- Priority: **medium** (branding consistency)

8. **Add `audit-version` to the progress breadcrumb**: Extend the workflow breadcrumb to include the audit step: `... > merge > audit-version > new-version`. This makes the command discoverable and establishes its position in the lifecycle. -- Priority: **medium** (discoverability)

9. **Support carry-forward context in deferred items**: When audit-version identifies gaps that the user defers, write structured context (what was expected, what was delivered, evidence) to a carry-forward file that `/rapid:new-version` reads as input. This prevents information loss across milestone boundaries and is consistent with RAPID's existing DEFERRED.md pattern. -- Priority: **medium** (information continuity)

10. **Fix context.cjs module path resolution**: The bug where `${RAPID_TOOLS}/../lib/context.cjs` fails because `RAPID_TOOLS` points to a `.cjs` file (not a directory) is a path resolution issue. The fix should use `path.dirname(RAPID_TOOLS)` or resolve relative to the known project structure. This is a UX issue because it causes init failures that users see as cryptic errors. -- Priority: **high** (bug fix affecting onboarding UX)
