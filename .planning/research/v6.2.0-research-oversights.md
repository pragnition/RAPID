# Oversights Research

## Cross-Cutting Concerns

### 1. Autoreload WebSocket/SSE Channel -- Transport and Lifecycle

The branding webserver overhaul replaces manual-refresh with autoreload. The current `branding-server.cjs` is a plain `node:http` static file server with no persistent connection mechanism. Implementing autoreload requires choosing a transport (WebSocket via `ws`, Server-Sent Events via native `EventSource`, or long-polling) and integrating a file-watcher (likely `fs.watch` or `chokidar`).

**Commonly forgotten aspects:**

- **Connection lifecycle management.** The server runs inside the Claude Code process (same PID written to `.server.pid`). If the agent session ends mid-branding, the HTTP server persists but the file watcher callback has nothing to notify. The autoreload channel must handle graceful degradation when no clients are connected, and must not leak watchers or timers.
- **Multiple-tab behavior.** Users may open the branding page in multiple browser tabs. Each tab establishes its own SSE/WebSocket connection. The watcher must broadcast to all connections, and connection cleanup on tab close must not tear down the watcher prematurely.
- **Debounce on file changes.** The agent writes artifacts (logo, theme, wireframe) via the Write tool, which may trigger multiple rapid `fs.watch` events for a single logical write. Without debouncing (50-100ms window), the browser will reload multiple times per artifact save.
- **CORS and CSP headers.** The current server does not set CORS or Content-Security-Policy headers. If autoreload uses WebSocket on a different port or SSE on the same port, the browser's security model may block connections depending on how the page is served. Self-contained inline `<script>` tags avoid CSP issues with external scripts, but WSS connections to `localhost` from a page served on `localhost` require matching protocols.
- **No external dependencies policy.** RAPID has only 4 production dependencies (zod, ajv, ajv-formats, proper-lockfile). Adding `ws` or `chokidar` as npm dependencies is a policy decision. SSE via `node:http` and `fs.watch` via `node:fs` avoid new dependencies but are less robust (especially `fs.watch` on Linux inotify limits and macOS FSEvents differences).

### 2. Artifact System Data Model

The milestone describes an "artifact system" where the model creates artifacts (logo, theme, wireframe, branding guidelines page) on a browsable page. This implies a structured registry of artifacts beyond flat file serving.

**Commonly forgotten aspects:**

- **Artifact metadata schema.** Each artifact needs at minimum: `id`, `type` (logo/theme/wireframe/guidelines), `filePath`, `createdAt`, `description`. Without a manifest file (e.g., `artifacts.json`), the hub page can only list files by name with no semantic grouping or ordering.
- **Artifact ordering and navigation.** The hub page currently lists files alphabetically. An artifact system implies a creation-order or category-grouped view. The `_generateHubPage()` function will need a complete rewrite to render from a manifest rather than `fs.readdirSync`.
- **Subdirectory support.** The current server only serves files directly in `.planning/branding/`. Logo generation may produce multiple variants (SVG, PNG at different sizes), wireframes may have multiple pages. The server's `_handleRequest` path resolution and the `_generateHubPage` directory listing both assume flat file structure -- they do not recurse into subdirectories.
- **File size limits for generated images.** Logo generation by the model produces SVG or text-based formats. If the system later supports raster image generation (via external tools), large PNG/JPEG files in `.planning/branding/` would be committed to git. Consider whether `.planning/branding/` should be partially gitignored (e.g., generated rasters) while keeping text-based artifacts tracked.

### 3. Branding-into-Init Integration Ordering

The plan inserts branding as Step 4B-post (after discovery areas, before granularity preference at 4C). The current init flow is already 10+ steps with 15+ AskUserQuestion calls. Branding adds 4-5 more interactive questions.

**Commonly forgotten aspects:**

- **AskUserQuestion budget exhaustion.** The init flow already consumes a large number of AskUserQuestion calls across Steps 4A, 4B (4 batches with potential follow-ups), 4C, 4D, and 4E (principles). Adding 4-5 branding questions pushes the total well past 20 interactive prompts. User fatigue is a real risk -- the branding interview at step 4B-post occurs after the user has already answered 12+ questions. Consider whether branding should be a brief opt-in during init (1 question: "Set up branding now? Yes/No") with the full interview deferred to `/rapid:branding`.
- **Skip path must be seamless.** Branding is documented as "FULLY OPTIONAL" in the existing SKILL.md. The init integration must preserve this -- skipping branding should not leave any broken state, half-created directories, or confusing progress indicators.
- **Brownfield projects already have style.** For brownfield projects, the codebase already has an implicit brand (existing README tone, existing CSS, existing documentation style). The branding interview questions ("What visual identity direction?") may feel redundant or confusing if the user is extending an existing project. The init integration should detect brownfield status and either skip the branding prompt or frame it as "review existing style" rather than "create from scratch."
- **Step numbering and breadcrumb coherence.** The init flow uses numbered steps (1-10+) and displays progress breadcrumbs on error. Inserting a new sub-step between 4B and 4C means every downstream step reference in error messages, breadcrumbs, and documentation must be updated. This is a mechanical but easily-missed task.

### 4. Update Reminder Timestamp Storage and Cross-Platform Persistence

The update reminder feature records `updated_on` during `/rapid:install` and prompts after 1 week. Storage location and format are decisions with cross-cutting implications.

**Commonly forgotten aspects:**

- **Storage location ambiguity.** Three candidates: the RAPID `.env` file, `config.json` at the plugin root, or a dedicated file (e.g., `.rapid-install-meta.json`). The `.env` file is already gitignored and written by `setup.sh`, but it is sourced via `export $(grep -v '^#' ...)` which does not handle values with spaces or special characters reliably. `config.json` is committed to the repo, so an install timestamp would create git noise. A separate metadata file is cleanest but adds yet another file to track.
- **Timestamp format and timezone.** ISO 8601 (`new Date().toISOString()`) is the correct choice, but the comparison logic must handle timezone differences if the user's system clock changes or if comparing against a server-reported latest version date.
- **Where to display the reminder.** The prompt "encourage the user to update" needs a trigger point. Options: at the start of every RAPID skill invocation (intrusive), only during `/rapid:status` (discoverable but passive), or as a one-time banner after the first stale invocation. The env preamble that loads RAPID_TOOLS runs at the start of every skill -- adding a version-check node call there would add latency to every single RAPID command.
- **Offline behavior.** If the reminder checks a remote source (npm registry, GitHub releases) for the latest version, it must handle network failures gracefully. The web-client.cjs already uses 2-second timeouts and never-throw patterns, which is a good precedent to follow. But a version check against npm or GitHub adds a network dependency to what is currently a fully offline tool.
- **Marketplace vs git clone update paths.** The install skill already handles two installation methods. Update reminders must provide different instructions for each: marketplace users update through Claude's plugin system, git clone users run `git pull`. The reminder message must detect which path applies and provide the correct command.
- **Stale `.env` after update.** After the user runs the update, the `updated_on` timestamp must be refreshed. If the user updates via `git pull` but does not re-run `/rapid:install`, the timestamp becomes stale and the reminder triggers again perpetually. The `setup.sh` script should write this timestamp, but currently it only writes `RAPID_TOOLS` to `.env`.

### 5. Branding Server Process Lifecycle in Claude Code

The branding server starts inside the Claude Code process and writes a PID file referencing `process.pid`. The overhaul introduces autoreload which adds persistent connections and file watchers.

**Commonly forgotten aspects:**

- **Claude Code agent session boundaries.** When a Claude Code conversation ends, spawned HTTP servers may or may not be cleaned up depending on how the process tree is managed. The current `stop()` only works if called within the same Node.js process that holds `_activeServer`. If the user closes Claude Code, the server process may become orphaned with a stale PID file. The existing `_isStale()` check handles this for subsequent runs, but the autoreload file watcher will continue consuming inotify watches until the orphaned process is killed.
- **Port conflict across projects.** The default port is 3141. If the user runs RAPID on multiple projects simultaneously (RAPID itself supports parallel development), each project's branding server will conflict on port 3141. The existing code returns `port_in_use` but the SKILL.md only offers manual port selection. Consider auto-incrementing or hashing the project path to a port range.
- **Server unref for non-blocking exit.** The current server does not call `server.unref()`, which means Node.js will not exit while the server is listening. If the branding server is started during init and the init skill completes, the process may hang if the server keeps the event loop alive. This matters for the new init integration where the server starts mid-flow.

### 6. Testing Strategy for the New Features

The existing test suite is thorough (21+ test files, co-located with source) and uses `node:test` with `assert/strict`. The new features introduce behaviors that are harder to unit test.

**Commonly forgotten aspects:**

- **Autoreload is inherently async and event-driven.** Testing that file changes trigger browser reloads requires either a real WebSocket/SSE client in tests or careful mocking of the event pipeline. The existing `branding-server.test.cjs` uses a simple `_fetch()` helper for HTTP GETs -- this is insufficient for testing persistent connections.
- **Init flow integration tests.** Inserting branding into the init flow changes the question sequence. If the init skill uses numbered steps referenced in tests (e.g., testing that Step 4C produces correct granularity output), inserting a new sub-step may shift expected outputs. No existing test covers the full init flow end-to-end (the tests are unit-level), but any future integration test must account for the new ordering.
- **Timestamp comparison edge cases.** The update reminder compares timestamps across potential timezone changes, system clock skew, and DST transitions. Tests should cover: exactly 7 days, just under 7 days, system clock set to the past, missing timestamp (first run), malformed timestamp string, and timestamp from a different timezone.
- **Artifact manifest validation.** If an `artifacts.json` manifest is introduced, it needs a Zod schema (consistent with the project's validation approach) and tests for malformed manifests, missing files referenced by manifest entries, and concurrent writes to the manifest.

### 7. Documentation and Onboarding for New Features

**Commonly forgotten aspects:**

- **SKILL.md version number.** The install skill currently references "v6.2.0" in its title and body text. The new milestone will produce v6.2.0. The version string in `skills/install/SKILL.md` line 1 and the post-install message (line 317: "RAPID v6.2.0 installation complete") must be updated. This is trivially forgettable.
- **`docs/` directory guides.** The project has 9 user documentation guides in `docs/`. If the branding overhaul changes the branding workflow significantly (from "generate a page" to "browsable artifact system with autoreload"), the relevant documentation must be updated. The user's mental model changes from "run branding, look at page" to "run branding, interact with live artifact gallery."
- **CLAUDE.md instructions.** The `CLAUDE.md` file is the authoritative source of project instructions for agents. If new environment variables are added (e.g., `RAPID_UPDATED_ON`), they should be documented in the context files or CLAUDE.md so agents are aware.

### 8. Configuration Management for New Features

**Commonly forgotten aspects:**

- **`.env` file grows incrementally.** The current `.env` contains only `RAPID_TOOLS`. The web dashboard adds `RAPID_WEB=true`. The update reminder may add `RAPID_UPDATED_ON`. Each feature appends lines to `.env` but there is no schema or validation for the `.env` file itself. Over time this file becomes a grab-bag of unrelated settings with no documentation of which are required vs optional.
- **`.env.example` drift.** `setup.sh` creates a `.env.example` with only `RAPID_TOOLS`. New environment variables added by features must also be added to `.env.example`, or new users will not know they exist.
- **`config.json` scope creep.** The runtime `config.json` currently has 2 keys (`lock_timeout_ms`, `agent_size_warn_kb`). The init flow writes to it via `init write-config`. If new features add configuration (e.g., default branding server port, update check interval), they should use `config.json` for runtime config and `.env` for environment-specific values. This boundary is not documented.

### 9. Accessibility of Branding HTML Pages

The branding hub page and artifact pages are rendered in a browser. The current `_generateHubPage()` produces dark-themed HTML with hardcoded colors.

**Not applicable in the traditional sense** (this is a developer tool, not a user-facing product), but worth noting:

- The branding page is the one place where RAPID produces user-facing HTML that a non-developer stakeholder might view (to approve branding). Basic accessibility (sufficient color contrast for the dark theme, semantic HTML elements, keyboard-navigable links) costs nothing and avoids embarrassment when showing the branding page to a design-conscious stakeholder.

### 10. Error Handling Across the Branding Overhaul

**Commonly forgotten aspects:**

- **Partial artifact creation recovery.** If the branding skill generates a theme artifact but fails mid-way through logo generation (agent context window exhaustion, network error), the artifact manifest would be in an inconsistent state. The next invocation needs to detect and recover from partial state rather than overwriting or duplicating.
- **Server start failure during init.** If branding is integrated into init and the server fails to start (port in use, permission error), the init flow must not abort. The SKILL.md already handles this with graceful error messages, but the init integration must inherit this same graceful pattern.
- **File watcher failure.** On Linux, `fs.watch` uses inotify which has a system-wide limit (`/proc/sys/fs/inotify/max_user_watches`, typically 8192-65536). Development environments with VS Code, webpack, and other watchers may exhaust this limit. The branding server should handle `ENOSPC` errors from `fs.watch` gracefully with a clear error message suggesting the user increase the limit.

### 11. Internationalization

**Not applicable.** RAPID is a developer tool with English-only interface. No i18n infrastructure is needed. The branding system itself is English-only in its interview questions and generated content.

## Recommendations

1. **Decide autoreload transport before implementation.** SSE (`EventSource`) over the existing HTTP server is the lowest-friction choice: no new npm dependencies, works with `node:http`, and degrades gracefully to manual refresh if the browser does not support SSE. WebSocket requires the `ws` package or a manual upgrade handler. This decision cascades into test strategy, client-side code, and the no-external-dependencies policy. Concrete first step: prototype a 20-line SSE endpoint on the existing server and verify it works with `fs.watch` callbacks.

2. **Design the artifact manifest schema before building the hub page.** Define `artifacts.json` with a Zod schema (consistent with `state-schemas.cjs` patterns). Include `id`, `type`, `fileName`, `createdAt`, and `description`. The hub page renders from this manifest, not from directory listing. Concrete first step: write the Zod schema and a `readArtifactManifest()` / `writeArtifactManifest()` pair with tests.

3. **Keep branding-in-init as a single opt-in question, not a full interview.** During init Step 4B-post, ask one question: "Set up project branding now? (Yes / Skip, run /rapid:branding later)". If "Yes", invoke the branding skill as a sub-flow. If "No", proceed to 4C. This avoids duplicating branding interview logic in the init skill and keeps the init flow from becoming exhausting. Concrete first step: add the opt-in question to the init SKILL.md after Step 4D and before Step 4E.

4. **Store `updated_on` in a dedicated `.rapid-meta.json` at the plugin root.** This avoids polluting `.env` (which is sourced as environment variables) and avoids git noise in `config.json` (which is committed). The file is simple: `{"updated_on": "2026-04-07T..."}`. Write it from `setup.sh` and from the install skill. Concrete first step: add `_writeMetaFile()` and `_readMetaFile()` functions to `version.cjs`.

5. **Display update reminders only during `/rapid:status`.** This is the natural "dashboard" command where users check project health. Adding a version check to the env preamble of every skill would add 50-200ms of latency to every invocation. The status command already aggregates project state; adding an "RAPID update available" line is low-friction. Concrete first step: add a `checkForUpdate()` call to the status display handler that reads `.rapid-meta.json` and compares against `Date.now()`.

6. **Call `server.unref()` after starting the branding server.** This allows the Node.js process to exit naturally when the skill completes, even if the server is still listening. The server will continue running (the OS keeps the socket open) but will not block process exit. This is critical for the init integration where the server starts mid-flow. Concrete first step: add `server.unref()` after `server.listen()` succeeds in `branding-server.cjs`.

7. **Add `fs.watch` debounce and `ENOSPC` error handling from day one.** Both are 5-10 line additions that prevent user-facing confusion. Debounce at 100ms prevents reload flicker. ENOSPC handling prints a clear message with the `sysctl` command to increase inotify watches. Concrete first step: wrap `fs.watch` in a helper function with built-in debounce and error handling.

8. **Update `.env.example` and version strings as part of the milestone's final wave.** Create a checklist task that updates: `skills/install/SKILL.md` version references, `.env.example` with any new variables, `package.json` version to 6.2.0, and `docs/` guides affected by the branding overhaul. Concrete first step: add this as an explicit task in the last wave of every set that adds new configuration.
