## Compatibility Matrix

### Branding Server Autoreload

The branding webserver overhaul requires:
1. **File watching** -- `fs.watch(dir, { recursive: true })` available on Node >=22 on all major platforms.
2. **Push notifications to browser** -- Server-Sent Events (SSE) via `node:http` with `text/event-stream`. No WebSocket library needed.
3. **Existing server infrastructure** -- `branding-server.cjs` already provides a working HTTP server on `node:http` with PID tracking, health probes, static file serving, hub page generation, and path traversal protection.

Constraint: The branding server currently serves files from `.planning/branding/`. The "artifact system" (logo, theme, wireframe, guidelines) would add more files to this directory. The `_generateHubPage()` function already dynamically lists directory contents, so new artifact types are automatically discoverable.

### Update Reminder (install timestamp)

Requirements:
1. **Timestamp storage** -- needs a persistent location. Options: `.env` file (already written by setup.sh), `config.json` (runtime config), or a new `.planning/config.json` field.
2. **Version comparison** -- `version.cjs` already exports `getVersion()` and `versionCheck()` with semver comparison via `prereqs.cjs::compareVersions()`.
3. **Time comparison** -- Node.js `Date` built-in. No dependency needed.

The `.env` file is the natural location since `setup.sh` already writes it and `/rapid:install` already modifies it. Adding an `RAPID_UPDATED_ON=<ISO timestamp>` line is trivial.

### Init Flow Branding Integration

The init skill (`skills/init/SKILL.md`) has this step structure:
- Step 4A: Logistics (name, team size, model)
- Step 4B: Deep project discovery (4 batches of questions)
- Step 4C: Granularity preference

The milestone brief specifies inserting branding as "step 4B, after areas, before granularity". This means between the current 4B (discovery) and 4C (granularity). The insertion point is well-defined.

Constraint: The branding skill (`skills/branding/SKILL.md`) is currently standalone with its own environment setup, codebase detection, interview flow, HTML generation, and server lifecycle. Integrating it into init means either:
- (a) Invoking `/rapid:branding` as a sub-skill from init (cleanest separation)
- (b) Inlining a condensed branding flow into the init SKILL.md (tighter integration but duplication risk)

No library-level code changes are needed for this -- it is purely a skill orchestration change.

## Upgrade Paths

### Zod v3 -> v4 (DEFERRED -- not for this milestone)

**Breaking changes that would affect RAPID:**
1. `.passthrough()` deprecated -- used on all 5 schema definitions in `state-schemas.cjs`. In v4, objects pass through unknown keys by default, so these calls would be removed.
2. `z.record()` now requires 2 arguments -- used in `review.cjs` as `z.record(z.string(), z.string())` (already 2 args, so safe).
3. `z.discriminatedUnion()` -- used in `returns.cjs`. Behavior should be compatible but needs testing.
4. Error customization API changes (`message` -> `error` param) -- RAPID does not customize error messages extensively, so low impact.
5. `.format()` and `.flatten()` deprecated -- grep shows no usage, so safe.
6. CommonJS support -- Zod v4 ships both CJS and ESM. `const { z } = require('zod')` should continue to work.

**Migration effort:** Moderate. A codemod exists (`npx @zod/codemod --transform v3-to-v4`). The main manual work would be removing `.passthrough()` calls and verifying that default passthrough behavior in v4 matches expectations. Estimated: 2-4 hours including test verification.

**Recommendation:** Defer to a future hygiene/cleanup milestone. Not worth the risk for a DX-focused milestone.

### execSync -> execFileSync (DEFERRED -- from bug-fixes-foundation)

- 346 occurrences of `execSync` across 22 files. Primarily in test files (mocking) and a few production files (`prereqs.cjs`, `core.cjs`, `worktree.cjs`, `execute.cjs`, `verify.cjs`, `merge.cjs`, `plan.cjs`, `review.cjs`).
- `execFileSync` avoids shell injection by not spawning a shell subprocess. Relevant for security hardening but not blocking for DX Refinements.

## Stack Risks

1. **`fs.watch` reliability on Linux (medium):** While recursive `fs.watch` works on Node 25/CachyOS, it relies on inotify which has a system-wide watch limit (`fs.inotify.max_user_watches`). For the branding directory (typically <20 files), this is a non-issue. But error handling should be robust -- if `fs.watch` fails, fall back gracefully (disable autoreload, serve normally).

2. **SSE connection lifecycle (low):** Browser SSE connections auto-reconnect, but if the server restarts (e.g., during development), stale connections may accumulate. The branding server's PID-based lifecycle management handles this since server restart cleans the PID file, but the SSE endpoint should handle connection cleanup on server shutdown.

3. **Branding server process model (low):** The server runs in-process (same Node.js process as the CLI invocation). If the CLI exits, the server stops. This is documented behavior and the PID file tracks it. For autoreload, `fs.watch` watchers are cleaned up when the process exits.

4. **Update reminder timestamp persistence (low):** The `.env` file is the simplest storage, but if the user runs `setup.sh` again it overwrites `.env` entirely (line 105-108 of setup.sh uses `cat >` not `cat >>`). The install flow should either: (a) append the timestamp rather than writing a fresh file, or (b) store the timestamp elsewhere (e.g., add to `.planning/config.json` or the root `config.json`). Storing in `config.json` avoids the `.env` overwrite issue.

## Recommendations

1. **Use `fs.watch` + SSE for branding autoreload (high):** Implement file watching with `fs.watch(brandingDir, { recursive: true })` and push change notifications via SSE (`text/event-stream`). Inject a small `<script>` tag into served HTML pages that opens an `EventSource` connection. Zero new dependencies required. Add error handling for inotify failures.

2. **Store `updated_on` in root `config.json` rather than `.env` (high):** The `.env` file is overwritten by `setup.sh`. The root `config.json` already stores runtime configuration (`lock_timeout_ms`, `agent_size_warn_kb`) and survives reinstalls. Add an `updated_on` ISO timestamp field there.

3. **Keep zero-dependency approach (high):** All three features (autoreload, update reminder, branding-in-init) are achievable with Node.js built-ins. Do not add chokidar, livereload, ws, or any other package.

4. **Do not upgrade Zod to v4 in this milestone (medium):** The `.passthrough()` deprecation and other breaking changes create unnecessary risk. Zod v3.25.76 is the latest v3 release and is fully functional. Defer to a dedicated migration set.

5. **Add `fs.watch` error handling for inotify limits (medium):** Wrap `fs.watch` in a try/catch and degrade gracefully to manual-refresh behavior if the watcher cannot be established.

6. **Consider artifact type registry for branding hub (low):** The current hub page lists all files in the branding directory. If the artifact system introduces structured types (logo, theme, wireframe, guidelines), a lightweight metadata approach (e.g., a `manifest.json` in the branding directory) could provide better organization than raw file listing. However, the current approach works and may be sufficient.
