# Architecture Research -- v7.0.0 Mission Control Autopilot

## Current Architecture (brownfield)

RAPID today has two distinct but loosely-coupled runtime surfaces, and this milestone grafts a third (agent runner) onto them:

### 1. Plugin / CLI side (Node.js, CommonJS)
- `src/bin/rapid-tools.cjs` is a thin router (~400 lines) dispatching to 23 handler modules under `src/commands/`.
- 41 library modules in `src/lib/` provide state, contracts, worktrees, merge, and review logic.
- State lives in `.planning/STATE.json` behind `withStateTransaction()` (lock -> read -> validate -> mutate -> write), with `proper-lockfile` (`.planning/.locks/`) as the concurrency primitive and Zod schemas as the integrity gate.
- Skills (`skills/*/SKILL.md`) are markdown orchestration specifications executed by Claude Code. They shell out to `node "${RAPID_TOOLS}" ...` and spawn agents via the in-harness `Agent` tool. All skills assume an interactive Claude Code TTY.

### 2. Mission Control webUI side (Python/FastAPI + React/Vite)
- Backend: FastAPI 0.135+, SQLModel on SQLite (WAL mode), Alembic migrations, uvicorn, Pydantic settings. Entry at `web/backend/app/main.py` (`create_app()` + `lifespan()`).
- Routers: `projects`, `kanban`, `notes`, `views` -- all REST under `/api/`, request-scoped `Session` via `Depends(get_db)`.
- A single long-lived background process: `FileWatcherService` (watchdog, inotify + polling fallback) that mirrors `.planning/STATE.json` and `REGISTRY.json` into SQLite metadata.
- A write-through `SyncEngine` keeps a `.rapid-web/` JSON mirror on disk -- the DB is the source of truth at runtime but the disk mirror is the handoff channel to the CLI.
- Frontend: React 19 + Vite 8, React Router 7, TanStack Query 5, Zustand, dnd-kit (kanban), CodeMirror (notes), Cytoscape (graph). Pure REST via `apiClient.ts` (`/api/*`); **there is no WebSocket or SSE layer today** -- the frontend refetches on focus/intervals via TanStack Query.

### Strengths relevant to this milestone
- **Skill-as-specification** is already machine-parseable. The Claude Agent SDK can invoke `/rapid:plan-set 1` as literal bash + agent directives -- no skill rewrite needed for non-interactive skills.
- **State lock machinery already exists.** `proper-lockfile` locks are PID-aware and stale-checked; a headless SDK-spawned agent writing state is architecturally indistinguishable from a TTY Claude Code session. Concurrency safety is already solved at the file level.
- **Kanban and projects are already first-class REST entities** with service layer, schemas, and SyncEngine disk mirror. Queueing agent todos is additive to the existing kanban schema.
- **FastAPI lifespan hook already manages background services** (file watcher). The same pattern extends naturally to an agent session manager.

### Areas needing extension
- **No long-lived server-to-client push channel.** The frontend is pure request/response. An autonomous agent emits events continuously (tool calls, assistant text, permission prompts) and the UI needs those in near-real-time.
- **No concept of a "chat" or "agent run" in the data model.** Today Projects own Kanban/Notes; chat transcripts and agent sessions are new entities.
- **No browser-side equivalent of `AskUserQuestion`, `Bash`, or file tools.** Interactive skills (`/rapid:init`, `/rapid:discuss-set`) depend on these. The SDK allows custom MCP tools to **replace** them, not wrap them -- the mapping has to be built.
- **Backend is Python; Claude Agent SDK for Python is the mandatory path.** The TypeScript SDK is better-documented but the existing FastAPI surface makes Python the correct choice. This is a constraint, not a weakness.
- **CLI and webUI can touch `.planning/STATE.json` at the same time.** File locks protect writes but nothing today coordinates *semantic* contention (e.g., user runs `/rapid:merge` in a terminal while an autopilot agent is mid-`/rapid:execute-set` in the browser).

## Recommended Architecture Pattern

- **Pattern:** **Extend the FastAPI backend with a long-lived AgentSessionManager service (in-process, asyncio-native) that owns Claude Agent SDK `ClaudeSDKClient` instances. Surface agent I/O to the browser via Server-Sent Events (SSE) with a polling fallback. Expose kanban operations to agents via an in-process SDK MCP server. Map browser-backed tools (AskUserQuestion, file ops) onto custom MCP tools that block on a request/response correlation table in SQLite.**
- **Rationale:**
  - The existing stack is FastAPI + SQLite + React Query. SSE fits natively in Starlette (`EventSourceResponse`), is firewall-friendlier than WebSockets, and matches the user's documented CLAUDE.md preference: the backend starts the long-running task, responds immediately with a run ID, and the frontend streams events (or polls `/runs/{id}/events`) without holding the HTTP socket open for the whole agent lifecycle.
  - In-process MCP servers (`create_sdk_mcp_server`) are the idiomatic way to expose Python functions to the agent -- they run in the same asyncio loop as the FastAPI app, so kanban writes go through the existing `kanban_service.py` without IPC.
  - The `ClaudeSDKClient` class is explicitly designed for bidirectional, streaming interactions and supports custom tools and hooks -- so a single class can serve `/rapid:plan-set` (autonomous, no input) and `/rapid:discuss-set` (input-requiring) without a fork in the code path.
- **Evidence:** The `claude-agent-sdk-demos` repo publishes a production-like reference architecture ("Long-Running Chat Sessions with WebSocket") using an `AgentSession` class wrapping `query()`/`ClaudeSDKClient` with a `MessageQueue`. We adopt the same shape on the Python side. FastAPI + watchdog is already proven in this codebase for long-lived background services (`file_watcher.py`).

**Rationale for structure decisions:**

- `app/agents/` is a package (not a flat module) because the runtime surface is large enough -- session manager, per-session state, MCP tool definitions, permission callback, transcript persistence -- that splitting is worth the import overhead.
- Tools live in `app/agents/tools/` as separate files mirroring the existing `app/routers/` convention (one concern per file).
- New DB models go in their own files (`app/models/agent_run.py`, `app/models/chat.py`) matching the existing `app/models/project.py` / `kanban.py` / `note.py` pattern.
- The frontend mirrors the backend surface: one page per tab, one hook per resource, one component per UI unit. This matches the existing KanbanBoard/KanbanColumn/KanbanCard decomposition.

## Module Organization

### Module: `app/agents/session_manager.py`
- **Responsibility:** Own the `dict[run_id, AgentSession]` registry for the process. Create sessions, tear them down on completion/error/timeout, enforce a per-project concurrent-run cap, and coordinate shutdown during app lifespan.
- **Public interface:** `start_run(project_id, skill, args, input_mode) -> run_id`, `get_session(run_id)`, `send_input(run_id, payload)`, `interrupt(run_id)`, `shutdown()`.
- **Dependencies:** `agent_session.py`, `models.agent_run`, `sqlmodel.Session`, `asyncio`.
- **Data ownership:** In-memory session registry. Persisted run metadata (status, started_at, completed_at, exit_reason) in `agent_run` table. Never touches `.planning/STATE.json` directly -- the spawned agent does that through bash/tools.

### Module: `app/agents/agent_session.py`
- **Responsibility:** Wrap a single `ClaudeSDKClient` instance. Own the asyncio task pumping messages from the SDK to the event bus. Expose a `MessageQueue`-style input channel so the browser can push follow-up messages and tool responses.
- **Public interface:** `AgentSession(run_id, project_path, skill_prompt, options)`, async `start()`, async `send(payload)`, async `interrupt()`, async `events() -> AsyncIterator[AgentEvent]`, `close()`.
- **Dependencies:** `claude_agent_sdk`, `event_bus.py`, `transcripts.py`.
- **Data ownership:** The SDK client, the asyncio task, the input queue. Writes transcript blocks to SQLite via `transcripts.py`.

### Module: `app/agents/event_bus.py`
- **Responsibility:** Per-run asyncio pub/sub. Multiple SSE subscribers may attach/detach during a run (browser tab reloads, multiple operators watching a run). Events are buffered (ring buffer, last N events) so a late subscriber can catch up without re-reading the DB.
- **Public interface:** `publish(run_id, event)`, async `subscribe(run_id) -> AsyncIterator[AgentEvent]`, `replay(run_id, since_seq) -> list[AgentEvent]`.
- **Dependencies:** `asyncio.Queue`, `collections.deque`.
- **Data ownership:** In-memory ring buffer. Durable storage is delegated to `transcripts.py`.

### Module: `app/agents/skill_runner.py`
- **Responsibility:** Convert a "run this skill" request into a concrete `(prompt_text, ClaudeAgentOptions)` pair. Two modes:
  1. **Slash command mode** (autonomous): prompt is `"/rapid:plan-set 1"` and options set `allowed_tools=["Bash", "Read", "Write", "Edit", "Grep", "Glob", "Agent", ...]` plus RAPID's own MCP tools.
  2. **Chat mode**: prompt is the user's free-form message, system prompt adds RAPID context, tool surface identical.
- **Public interface:** `build_run(project_path, skill, args, input_mode) -> (prompt, options)`.
- **Dependencies:** `claude_agent_sdk.ClaudeAgentOptions`, `permissions.py`, `tools/*`.
- **Data ownership:** None -- pure transform.

### Module: `app/agents/permissions.py`
- **Responsibility:** Implement the `can_use_tool` callback. For "safe" tools (Read, Grep, Glob, the RAPID MCP tools) auto-allow. For potentially-destructive tools in **interactive** runs (Bash edits outside the project worktree, Write outside `.planning/`), emit a `permission_request` event on the bus and await the user's decision (via `send_input`). For **autonomous** runs started from the kanban ("autopilot"), use a stricter policy that denies anything outside the worktree.
- **Public interface:** `build_can_use_tool(run_mode, project_root)` returning the callback.
- **Dependencies:** `claude_agent_sdk.PermissionResultAllow/Deny`, `event_bus.py`.
- **Data ownership:** None; coordinates through the event bus.

### Module: `app/agents/tools/ask_user.py`
- **Responsibility:** Define `mcp__rapid__webui_ask_user` as an `@tool`-decorated function. When the agent calls it, emit an `ask_user` event on the event bus, then await a matching `user_answer` on the session's input queue, and return it as the tool result.
- **Public interface:** A single async tool function registered into the agent's MCP server.
- **Dependencies:** `claude_agent_sdk.tool`, session context (wired via closure in `session_manager`).
- **Data ownership:** None -- purely request/response proxy.

### Module: `app/agents/tools/kanban_tools.py`
- **Responsibility:** Expose `mcp__rapid__kanban_board` (read full board), `kanban_add_card`, `kanban_move_card`, `kanban_update_card`, `kanban_get_next_todo` (pop next card from an "autopilot" column).
- **Public interface:** Five `@tool`-decorated functions registered into the same MCP server.
- **Dependencies:** `app.services.kanban_service`, `app.database.get_engine`, `sqlmodel.Session`.
- **Data ownership:** Writes go through the existing kanban_service (which already syncs to `.rapid-web/`).

### Module: `app/routers/agents.py`
- **Responsibility:** Expose HTTP surface: `POST /api/agents/runs` (start), `GET /api/agents/runs` (list), `GET /api/agents/runs/{id}` (status), `GET /api/agents/runs/{id}/events` (SSE stream), `GET /api/agents/runs/{id}/events?since=N` (polling fallback), `POST /api/agents/runs/{id}/input` (send follow-up), `POST /api/agents/runs/{id}/interrupt`, `DELETE /api/agents/runs/{id}` (cancel + cleanup).
- **Public interface:** Standard FastAPI router with Pydantic schemas.
- **Dependencies:** `app.services.agent_service`, `app.agents.session_manager`.
- **Data ownership:** None -- delegates to service layer.

### Module: `app/routers/chats.py`
- **Responsibility:** CRUD for chat entities (a thin wrapper over agent runs where the user controls the prompt). A chat is a named, persistent conversation; an agent run is the underlying execution. The mapping is 1:1 for new chats and 1:N for chats that the user resumes.
- **Public interface:** `GET/POST/PUT/DELETE /api/chats`, `POST /api/chats/{id}/messages` (which internally creates/continues an agent run).
- **Dependencies:** `app.services.chat_service`, `app.services.agent_service`.
- **Data ownership:** `Chat`, `ChatMessage` tables.

### Module: frontend `hooks/useAgentEvents.ts`
- **Responsibility:** Subscribe to `GET /api/agents/runs/{id}/events` via `EventSource`. On open error or reconnect, fall back to `GET .../events?since=lastSeq` polling at 2s. Expose events to components via a reducer-backed `useSyncExternalStore` so multiple tabs render consistently.
- **Public interface:** `useAgentEvents(runId)` returning `{events, status, reconnect}`.
- **Dependencies:** native `EventSource`, `@tanstack/react-query` for polling fallback.
- **Data ownership:** In-memory event log; can evict when the user navigates away.

## Data Flow

### Autonomous run lifecycle (e.g., user clicks "Run /rapid:plan-set 1" in the UI)

1. Browser `POST /api/agents/runs` with `{project_id, skill: "plan-set", args: ["1"], input_mode: "autonomous"}`.
2. `agents` router calls `agent_service.start_run(...)`:
   - Inserts an `agent_run` row (status=`pending`, run_id=uuid4).
   - Returns `{run_id, status: "pending"}` **immediately** -- the HTTP response does NOT wait for agent completion. (This matches the user's CLAUDE.md rule about not holding long-running sockets.)
3. `session_manager.start_run` schedules an asyncio task:
   - Calls `skill_runner.build_run` to produce `(prompt, options)`.
   - Constructs `ClaudeAgentOptions` with: `system_prompt` explaining the RAPID context, `cwd=project_path`, `allowed_tools` list, `mcp_servers={"rapid": rapid_mcp_server}`, `can_use_tool=permissions.build_can_use_tool("autonomous", ...)`, `permission_mode="default"`.
   - Instantiates `ClaudeSDKClient(options)`, enters it as an `async with` via `AgentSession`.
   - Sends the prompt `"/rapid:plan-set 1"`.
   - Pumps `client.receive_messages()` into the event bus, emitting `assistant_text`, `tool_use`, `tool_result`, `thinking`, `system`, `status_change` events.
4. Browser, upon receiving the `run_id`, opens `EventSource("/api/agents/runs/{id}/events")`.
5. Backend SSE endpoint subscribes to `event_bus.subscribe(run_id)`, yields each event as an SSE frame. On disconnect, browser polls `GET .../events?since={last_seq}`.
6. Meanwhile, the SDK client runs the actual skill -- it shells out to `node "${RAPID_TOOLS}" ...` via its Bash tool, spawns sub-agents via the Agent tool (these are Claude Code harness agents, NOT new SDK clients -- the harness handles them natively), writes files, creates git commits. The cwd is the project root, so `.planning/STATE.json` lock acquisition goes through the existing `proper-lockfile` machinery.
7. On completion, the pump loop exits, `agent_service` updates `agent_run.status = "complete"`, emits a final `run_complete` event, and the session is evicted from the manager.
8. Browser sees `run_complete`, invalidates TanStack queries for STATE / sets / kanban, and refetches to show the new state.

### Interactive run lifecycle (e.g., `/rapid:discuss-set 1`)

Steps 1-6 as above, but:
- The system prompt tells the agent that instead of `AskUserQuestion` (not available in SDK context) it MUST call `mcp__rapid__webui_ask_user`. The discuss-set skill gets a small patch: "If `RAPID_RUN_MODE=sdk`, call `mcp__rapid__webui_ask_user` instead of AskUserQuestion." This env var is set in the agent options.
- When the agent invokes `mcp__rapid__webui_ask_user(question, options)`, the tool function emits `ask_user` event and awaits the input queue.
- Browser renders `<AskUserModal>`, user picks an option.
- Browser `POST /api/agents/runs/{id}/input` with `{type: "user_answer", tool_use_id, value}`.
- `session.send(...)` puts the answer on the input queue; the `ask_user` tool returns; the agent continues.

### Kanban autopilot flow

1. User drags a kanban card into a column named "Autopilot" (detected by name or a column flag).
2. A separate `autopilot_worker` (started in `lifespan`) polls "Autopilot" columns every 10s per registered project, finds the top card, and calls `agent_service.start_run(project_id, skill="execute-card", args=[card_id], input_mode="autonomous")`.
3. The agent, with the kanban MCP tools in scope, calls `mcp__rapid__kanban_get_next_todo`, reads the description, plans and executes the work, and finally `mcp__rapid__kanban_move_card(card_id, "Done")`.
4. Concurrency: the autopilot worker holds a `proper-lockfile` lock on `.planning/.locks/autopilot.target` scoped per project, so two backend replicas (if that ever happens) do not double-pick.

### State management

- **Authoritative state live** (during a run): the `AgentSession` object in memory + the `ClaudeSDKClient`'s internal conversation history.
- **Authoritative state persisted** (across runs and restarts): SQLite tables `agent_run`, `agent_event`, `chat`, `chat_message`. Plus the existing `.planning/STATE.json` (owned by the skill execution, untouched by the session manager).
- **Event replay**: if the UI reconnects mid-run, `event_bus.replay(run_id, since_seq)` reads the tail from the in-memory ring buffer; if the event is older than the buffer window, `GET /api/agents/runs/{id}/events?since=N` falls through to `agent_event` table.
- **Consistency guarantee**: the event stream is strictly ordered per `run_id` (single-producer -- the pump task); no interleaving across runs needed since each run has its own bus partition.
- **Crash recovery**: on backend restart, in-flight runs are marked `interrupted` (status transition). There is no session resume -- the SDK does not persist its internal state. This is an intentional simplification; a user who wants to continue re-runs the skill.

## API Design

REST conventions for the new surface (following the existing `/api/projects`, `/api/kanban` patterns):

| Method | Path | Purpose |
|--------|------|---------|
| `POST` | `/api/agents/runs` | Start a new run. Returns 201 + `{run_id, status}`. |
| `GET`  | `/api/agents/runs?project_id=&status=` | List runs (paginated). |
| `GET`  | `/api/agents/runs/{run_id}` | Run metadata + latest status. |
| `GET`  | `/api/agents/runs/{run_id}/events` | SSE stream (`text/event-stream`). |
| `GET`  | `/api/agents/runs/{run_id}/events?since={seq}` | Polling fallback (JSON array). |
| `POST` | `/api/agents/runs/{run_id}/input` | Send user answer / permission response / chat follow-up. |
| `POST` | `/api/agents/runs/{run_id}/interrupt` | Stop the run. |
| `DELETE` | `/api/agents/runs/{run_id}` | Tombstone + cleanup. |
| `GET`  | `/api/chats?project_id=` | List chats. |
| `POST` | `/api/chats` | Create empty chat. |
| `GET`  | `/api/chats/{chat_id}/messages` | Message history. |
| `POST` | `/api/chats/{chat_id}/messages` | Send a message (internally starts/continues a run). |

- **Versioning:** none -- same convention as existing routes (`/api/...` without version prefix).
- **Auth:** none yet; the existing backend is localhost-only. If multi-tenant ever matters, it's a cross-cutting concern across every router.
- **Request/response format:** JSON, Pydantic-validated. SSE events are named (`event:` header) and JSON-payloaded.
- **Error format:** consistent with existing routes -- `{"detail": "..."}` with appropriate HTTP status codes.

### SSE event taxonomy

```
event: status         data: {"status": "running"|"complete"|"error"|"interrupted", "run_id": "..."}
event: assistant_text data: {"text": "...", "seq": 42}
event: thinking       data: {"text": "...", "seq": 43}
event: tool_use       data: {"tool": "Bash", "input": {...}, "id": "toolu_...", "seq": 44}
event: tool_result    data: {"tool_use_id": "...", "output": "...", "is_error": false, "seq": 45}
event: ask_user       data: {"tool_use_id": "...", "question": "...", "options": [...], "seq": 46}
event: permission_req data: {"tool_use_id": "...", "tool": "Write", "input": {...}, "seq": 47}
event: run_complete   data: {"exit_reason": "success"|"error", "summary": "...", "seq": 99}
```

Every event carries a monotonically-increasing `seq` scoped to the run, so polling can resume.

## Integration Points

| External System | Protocol | Purpose | Error Handling |
|----------------|----------|---------|----------------|
| Claude API (via `claude-agent-sdk`) | HTTPS | Agent inference, tool orchestration | SDK retries transient errors; we surface terminal errors as `status: error` event. ANTHROPIC_API_KEY from env or `.env`. |
| Claude Code CLI (subprocess spawned by SDK) | local stdio | SDK launches `claude` CLI under the hood for harness tool execution | If the CLI is not installed, startup fails with a clear error. Document prereq in setup. |
| `.planning/STATE.json` (file) | filesystem + `proper-lockfile` | State reads/writes by the spawned skill | Existing lock machinery handles contention; 300s stale threshold. |
| `.rapid-worktrees/*` (git) | git CLI (via spawned skill) | Set worktrees | Existing worktree service handles this; agent inherits cwd. |
| SQLite (`rapid-web.db`) | SQLAlchemy/SQLModel | New agent_run / chat tables | WAL mode handles concurrent reads during writes; `busy_timeout=5000` already set. |
| Frontend (browser) | HTTP + SSE | UI <-> backend | EventSource auto-reconnects; polling fallback covers proxy edge cases. |

## Scaling Considerations

- **Expected load:** single-user-per-project desktop tool. Concurrent runs per project: typically 1, max 3 (enforced cap). Concurrent runs across all projects: ~10 is a reasonable ceiling on a developer laptop given inference latency and memory. SQLite handles this trivially.
- **Bottlenecks:**
  - Anthropic API rate limits -- enforce per-account concurrency cap via a semaphore in `session_manager`.
  - SQLite write contention during event storms -- events from multiple concurrent runs interleave through WAL. Buffer writes in a per-run async queue that flushes every 200ms or on `run_complete`.
  - SSE client count per run -- cap to 5 subscribers per run (matches the existing branding server's `MAX_SSE_CLIENTS=10` convention).
- **Horizontal vs vertical:** not applicable at this scale. The backend is single-process by design (file watcher + session manager share in-process state). If multi-process ever matters, move the event bus to Redis pub/sub.
- **Caching:** none needed for agent state. For project/STATE.json reads during skill runs, the existing file watcher already updates a SQLite cache; the agent reads the files directly so this is transparent.

## Architecture Risks

1. **Risk: SDK's Claude Code CLI dependency.** The Python SDK spawns the `claude` CLI. If the user's CLI version drifts from what the SDK expects, or if the CLI isn't on `PATH` for the uvicorn process, skill runs fail opaquely. **Mitigation:** startup probe that runs `claude --version` and surfaces the result on `/api/health`; document `PATH` requirements; pin a minimum `claude-agent-sdk` version.

2. **Risk: Concurrent CLI and web access to `.planning/STATE.json`.** File locks prevent data corruption, but a user running `/rapid:merge` in a terminal while the browser has an active autopilot run can deadlock *semantically* (two actors both mid-transition). **Mitigation:** add a coarse project-scoped advisory lock in the SQLite `appconfig` table -- `agent_run_active=<run_id>` -- that both the session_manager and the CLI (via a new `rapid-tools lock check` subcommand) respect. If a user really wants to force it, they can `rapid-tools lock clear`.

3. **Risk: Browser tab close mid-run leaves agent orphaned.** The run continues server-side but no one is watching. **Mitigation:** this is actually desired behavior for autonomous runs -- the run logs to SQLite and the user can reattach later. For interactive runs (discuss/init), add a 2-minute "waiting for user input" timeout; on expiry, emit a `status: stalled` event and let the user resume or cancel.

4. **Risk: Permission-callback stream-keep-open gotcha.** Per the Python SDK docs, `can_use_tool` requires streaming mode AND a PreToolUse hook returning `{"continue_": true}` or the stream closes early. **Mitigation:** always register the no-op PreToolUse hook alongside `can_use_tool`; cover it in a unit test that spawns a fake tool call.

5. **Risk: In-process MCP server holds references to SQLModel sessions across async boundaries.** SQLAlchemy sessions are NOT asyncio-safe; reusing a session across awaits can corrupt cursors. **Mitigation:** every `@tool` function opens its own `Session(engine)` context manager, does its work, commits, closes. Never close over a session.

6. **Risk: MessageQueue back-pressure during long tool runs.** If the agent emits faster than SSE can fan out, the in-process queue grows unbounded. **Mitigation:** cap the ring buffer at 500 events per run; if overflowed, flush to SQLite and drop from memory (SSE subscribers that fall behind get a `caught_up_from_db` event and resume).

7. **Risk: Token leakage in event stream.** Assistant thinking blocks can contain ANTHROPIC_API_KEY or user secrets if the agent got them via Read. **Mitigation:** scrub known secret patterns in `transcripts.py` before emitting (regex for `sk-ant-*`, env-var-like patterns). Document that the event stream is not audit-grade.

8. **Risk: Autopilot kanban loop spawning runaway agents.** A card that fails repeatedly could be re-picked forever. **Mitigation:** per-card attempt counter stored in card `description` metadata (JSON blob); after N=3 failures, move to a "Needs Review" column and emit a notification.

9. **Risk: Python GIL + asyncio quirks under heavy tool-call load.** The event pump is asyncio, but SDK may block synchronously on subprocess I/O at moments. **Mitigation:** keep all session work on the uvicorn event loop; if CPU pressure appears, move transcript writes to a `run_in_executor` pool. Benchmark before optimizing.

10. **Risk: React state + SSE reconnection race.** On reconnect via `?since=N`, the client may replay events it already applied. **Mitigation:** client keeps a `Set<seq>` seen and ignores duplicates; server enforces monotonic `seq` per run.

## Recommendations

1. **Adopt SSE + polling fallback, not WebSocket** -- matches user's documented preference, fits FastAPI/Starlette natively, no auxiliary infra, trivial to degrade. **Priority: critical.**
2. **Use `ClaudeSDKClient` (not `query()`) for every run** -- it supports bidirectional input, custom tools, hooks, and `can_use_tool`. Unifies the autonomous and interactive code paths. **Priority: critical.**
3. **Make Kanban autopilot opt-in per column** -- detect by column title match (`"Autopilot"`) or by an explicit `is_autopilot` flag added to `KanbanColumn`. A flag is cleaner and survives renames. **Priority: high.**
4. **Patch existing skills minimally: wrap AskUserQuestion with "if sdk-mode, call mcp__rapid__webui_ask_user"** -- do NOT fork skills into CLI and web variants. Gate on `RAPID_RUN_MODE=sdk` env var (set in `ClaudeAgentOptions.env`). **Priority: critical.**
5. **Persist every SSE event to `agent_event` table** -- SQLite write volume is tiny (tens of events per run) and durable replay removes a whole class of "lost connection" bugs. **Priority: high.**
6. **Add a SQLite-backed project lock table (`project_run_lock`)** coordinated with CLI -- prevents CLI/web deadlocks without needing file-system primitives across language boundaries. **Priority: high.**
7. **Cap concurrent agent runs at 3 per project, 10 global** via an asyncio Semaphore in `session_manager`. **Priority: medium.**
8. **Design the "Chats" entity as a thin wrapper over one or more `agent_run` rows** -- a chat is a named transcript container, runs are the actual execution units. This keeps the run runtime stateless w.r.t. conversation identity. **Priority: medium.**
9. **Ship permission gating as two modes (`autonomous`, `interactive`) not one toggle** -- autopilot runs should reject Write outside worktree; interactive runs should prompt. Hardcode the policies in `permissions.py`. **Priority: high.**
10. **Unit-test the message pump with a fake `AsyncIterator` of SDK messages** before touching real network -- this is the highest-leverage tier of testing and `node:test`-style Python (pytest-asyncio) already ships in the dev dependency group. **Priority: critical.**
