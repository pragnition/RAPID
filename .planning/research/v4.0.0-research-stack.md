# Stack Research: Mission Control Dashboard (v4.0.0)

## Executive Summary

The Mission Control milestone introduces a full-stack web dashboard living at `~/.rapid/` as an optional companion to the RAPID CLI plugin. This is RAPID's first web service -- a Python backend with a JavaScript frontend -- layered on top of the existing Node.js/CommonJS plugin codebase. The stack choices below are grounded in confirmed version numbers and documentation, with confidence levels noted where inference was required.

---

## Core Stack Assessment

### 1. Python Runtime

| Property | Value |
|----------|-------|
| Detected on system | Python 3.14.3 |
| Minimum for FastAPI | Python >= 3.10 |
| Minimum for SQLModel | Python >= 3.10 |

**Key considerations:**
- Python 3.14 is the bleeding edge (released 2025). The user's system runs 3.14.3.
- FastAPI 0.135.x officially tests against 3.10-3.14. No known incompatibilities with 3.14.
- SQLModel 0.0.37 supports Python 3.10-3.14. Confirmed via PyPI metadata.
- **Recommendation:** Target Python >= 3.12 as minimum (balances modern features with broad compatibility), but test against 3.14 since that is the development environment.

### 2. FastAPI (Backend Framework)

| Property | Value |
|----------|-------|
| Latest stable | 0.135.1 (March 2026) |
| Recommended | 0.135.x |
| Python requirement | >= 3.10 |
| Source | Confirmed via PyPI and release notes |

**Key features relevant to Mission Control:**
- Native SQLite integration via SQLModel (official recommendation from FastAPI docs)
- Server-Sent Events (SSE) support added in recent releases -- useful for push-based status updates
- Pydantic v2 integration for request/response validation
- Auto-generated OpenAPI docs (Swagger UI at `/docs`, ReDoc at `/redoc`)
- Dependency injection system for database sessions
- `strict_content_type` checking now enabled by default (new in 2025+)

**Breaking changes to be aware of:**
- `fastapi-slim` is deprecated; use `fastapi[standard]` only
- Python 3.8/3.9 support dropped; internal syntax upgraded to 3.9+
- Starlette 1.0.0+ is now required (not compatible with older Starlette)
- `@app.on_event("startup")` is deprecated in favor of lifespan context managers

**Limitation:** FastAPI is async-first but SQLite with SQLModel uses synchronous sessions (via `check_same_thread=False`). This works fine for a local dashboard with low concurrency but would not scale to high-traffic scenarios. Not a concern for this use case.

### 4. Frontend Framework: Svelte 5 vs React 19

This is the key decision point. The spec says "Vite + React (or Svelte -- research which is better fit)."

#### React 19

| Property | Value |
|----------|-------|
| Latest stable | 19.2.4 (January 2026) |
| Bundle size (Hello World) | ~42 KB |
| Ecosystem maturity | Dominant -- largest library ecosystem |
| Source | Confirmed via react.dev/versions |

**Pros for this project:**
- Massive ecosystem of UI component libraries (Radix UI, Shadcn, Recharts, etc.)
- React DevTools are mature and well-known
- More AI agents and LLMs have deep React training data -- better agent-assisted code generation
- react-hotkeys-hook supports vim-style sequential hotkeys (`g>i>t`)
- @uiw/react-codemirror provides CodeMirror 6 markdown editor integration
- React Compiler in 19.x reduces need for manual memoization

**Cons:**
- Heavier runtime (~42 KB vs ~1.6 KB)
- JSX + hooks mental model has a steeper learning curve
- Virtual DOM overhead (negligible for this scale, but real)

#### Svelte 5

| Property | Value |
|----------|-------|
| Latest stable | 5.37.0 (March 2026) |
| Bundle size (Hello World) | ~1.6 KB |
| Ecosystem maturity | Growing -- ~20% developer adoption |
| Source | Confirmed via svelte.dev and GitHub releases |

**Pros for this project:**
- Dramatically smaller bundles -- compiles away the framework
- Svelte 5 runes (`$state`, `$derived`, `$effect`) provide fine-grained reactivity without a virtual DOM
- Simpler component syntax -- closer to vanilla HTML/CSS/JS
- Native TypeScript support without preprocessors (new in Svelte 5)
- svelte-codemirror-editor wraps CodeMirror 6 for markdown editing
- svelte-keyboard-shortcuts and @invokey/svelte for hotkey handling
- SvelteKit provides routing, SSR, and build tooling in one package
- Smaller runtime = faster load for a local dashboard

**Cons:**
- Smaller library ecosystem -- fewer off-the-shelf UI component kits
- Svelte 5 runes are a significant paradigm shift; some libraries may not yet support Svelte 5
- Less AI training data compared to React
- Smaller community for troubleshooting

#### Recommendation: Svelte 5

For a local developer dashboard that prioritizes:
- Fast startup and small footprint (it runs on every boot)
- Developer-tool aesthetic (Svelte's simplicity matches the "lean dev tool" vibe)
- Read-only UI with no complex form state management
- Single developer maintaining the code

Svelte 5 is the better fit. The smaller bundle, compiler-based approach, and simpler component model align with a lightweight always-on local service. The ecosystem gap is manageable because this is not a complex enterprise app -- it needs a handful of well-chosen libraries, not hundreds. If the team later finds Svelte's ecosystem insufficient, Vite supports both frameworks and a migration path exists (though it would be costly).

**If React is preferred instead:** React 19.2.x with Vite 8.x is a fully viable alternative. The recommendation is not strongly against React -- it is a matter of fit, not capability.

### 5. Vite (Frontend Build Tool)

| Property | Value |
|----------|-------|
| Latest stable | 8.0.1 (March 2026) |
| Previous stable | 7.x (mid-2025) |
| Recommended | 8.0.x |
| Source | Confirmed via vite.dev and npm |

**Key features for Mission Control:**
- Now uses Rolldown + Oxc internally (replacing esbuild + Rollup) -- faster builds
- `@vitejs/plugin-react` v6 uses Oxc for React Refresh (no Babel dependency)
- `@sveltejs/vite-plugin-svelte` for Svelte integration
- Built-in dev proxy for routing API calls to FastAPI backend during development
- `server.forwardConsole` forwards browser console to terminal (useful for agent-assisted development)
- Hot Module Replacement (HMR) works with both React and Svelte

**Breaking change:** Vite 8 default browser target changed from `modules` to `baseline-widely-available`. This is fine for a local dashboard where the user controls the browser.

### 6. SQLite + SQLModel (Database)

| Property | Value |
|----------|-------|
| SQLite | System-provided (typically 3.40+) |
| SQLModel latest | 0.0.37 (February 2026) |
| SQLAlchemy (dependency) | 2.0.x |
| Source | Confirmed via PyPI |

**Why SQLModel over raw SQLAlchemy:**
- Created by the same author as FastAPI (Sebastian Ramirez)
- Combines SQLAlchemy ORM with Pydantic validation in a single model class
- One model definition serves as both database schema and API request/response schema
- Official FastAPI documentation recommends SQLModel for SQLite projects

**SQLite-specific considerations:**
- File-based database stored at `~/.rapid/rapid.db` (or similar)
- `check_same_thread=False` required in `connect_args` for FastAPI multi-threaded access
- SQLite `ALTER TABLE` only supports adding/renaming columns -- Alembic must use `render_as_batch=True` for migrations
- Write concurrency is limited (WAL mode helps but single-writer lock remains) -- acceptable for a single-user local dashboard
- No server process needed -- the database is just a file

**Migration tool:** Alembic 1.x with `render_as_batch=True` for SQLite compatibility. This is well-documented for SQLModel + FastAPI.

### 7. Node.js (Existing Plugin Runtime)

| Property | Value |
|----------|-------|
| Detected on system | v25.8.0 |
| Required by RAPID plugin | Node.js 18+ |
| Source | Confirmed via package.json and system check |

**Interaction with Mission Control:**
- The RAPID CLI (Node.js/CommonJS) needs to register projects with the Mission Control dashboard
- This means the CLI will make HTTP calls to `localhost:8998` to register/update project state
- No shared runtime between Node.js CLI and Python backend -- they communicate via HTTP API
- The registration function should use Node.js built-in `http` or `https` module (no new dependencies)

## Compatibility Matrix

### Critical Constraints

| Constraint | Requirement | Impact |
|-----------|-------------|--------|
| FastAPI 0.135.x | Python >= 3.10 | Must target Python 3.10+ |
| SQLModel 0.0.37 | Python >= 3.10 | Aligned with FastAPI |
| Pydantic v2 | Required by FastAPI 0.135.x | SQLModel 0.0.37 uses Pydantic v2 |
| Starlette | >= 1.0.0 | Bundled with fastapi[standard] |
| Vite 8.x | Node.js >= 18 | System has Node 25.8.0 -- fine |
| Svelte 5 | Vite plugin compatibility | @sveltejs/vite-plugin-svelte supports Vite 8 |
| Tailwind CSS 4.x | PostCSS compatibility | v4 uses CSS-first config, no JS config needed |
| Alembic + SQLite | render_as_batch=True required | SQLite ALTER TABLE limitations |

### No Known Conflicts

- FastAPI + SQLModel + Pydantic v2 are all maintained by the same team -- version alignment is guaranteed.
- Vite 8 + Svelte 5 are tested together by the SvelteKit team.
- Tailwind CSS 4.x works with both Vite 8 and Svelte 5 via PostCSS or the Vite plugin.
- Node.js 25.x can run `http.request()` to communicate with the Python backend -- no dependency needed.

### Potential Friction Points

| Issue | Severity | Mitigation |
|-------|----------|------------|
| SQLite WAL mode + systemd service | Low | Enable WAL mode at engine creation; single-user so contention is minimal |
| Svelte 5 library ecosystem gaps | Medium | Verify each UI library supports Svelte 5 runes before adopting |
| Python virtual environment management | Medium | Use `uv` or `venv` at `~/.rapid/.venv/` for isolation |
| Two runtimes (Node.js + Python) | Medium | Clear separation: Node.js CLI calls Python HTTP API |

## Upgrade Paths

### From Initial Release

Since this is a greenfield component (no existing web service to migrate from), all dependencies start at their latest stable versions. The primary upgrade paths to monitor are:

| Dependency | Upgrade Concern | Timeline |
|-----------|----------------|----------|
| FastAPI | Starlette API changes, new lifespan patterns | Monitor quarterly |
| SQLModel | Still pre-1.0 (0.0.x versioning) -- API may shift | Monitor releases closely |
| Svelte | Svelte 5 runes are stable but ecosystem migration ongoing | Most libraries migrated by mid-2026 |
| Vite | Vite 8 is new; Vite 9 likely late 2026 | Expect smooth migration |
| Tailwind | v4 is stable; v5 not anticipated soon | Low risk |

### SQLModel Pre-1.0 Risk

SQLModel is at version 0.0.37 despite being widely used and recommended by FastAPI. The 0.0.x versioning means the API is not formally guaranteed to be stable. However:
- The author (Sebastian Ramirez) has been incrementally stabilizing the API
- FastAPI's own documentation recommends it as the primary ORM
- The risk is mitigated by the fact that SQLModel is a thin wrapper over SQLAlchemy 2.0 -- if SQLModel stalls, dropping down to raw SQLAlchemy is straightforward

## Stack Risks

| # | Risk | Impact | Mitigation |
|---|------|--------|------------|
| 1 | **SQLModel pre-1.0 instability** | Model API changes could break migrations | Pin exact version; wrap models in a thin abstraction layer; Alembic handles schema drift |
| 2 | **Two-runtime complexity** (Node.js + Python) | Increased install complexity; two dependency systems | Clear separation: Python lives in `~/.rapid/`; Node.js CLI just makes HTTP calls |
| 3 | **systemd user service reliability** | Service may not start if user has no lingering sessions enabled | Document `loginctl enable-linger $USER`; provide manual start fallback |
| 4 | **Svelte 5 ecosystem maturity** | Some libraries may not support Svelte 5 runes yet | Verify each dependency before adopting; framework-agnostic libraries (Cytoscape, CodeMirror) avoid this risk |
| 5 | **Python version availability** | Users may not have Python >= 3.10 installed | Check at install time; provide clear error message; consider bundling via `uv` |
| 6 | **SQLite file locking conflicts** | If CLI and web service both write to the same SQLite file | Use WAL mode; design the CLI to write via HTTP API, not directly to SQLite |
| 7 | **Port 8998 conflicts** | Another service may use port 8998 | Make port configurable via env var (RAPID_WEB_PORT); default to 8998 |
| 8 | **Langfuse dependency for non-AI features** | Langfuse is only needed if the dashboard makes external AI API calls | Make Langfuse optional; only import when AI features are active |

## Proposed Directory Structure

```
~/.rapid/
  rapid.db                    # SQLite database
  .venv/                      # Python virtual environment
  .env                        # Environment variables (LANGFUSE_*, etc.)
  .env.example                # Template for .env
  backend/
    main.py                   # FastAPI app entry point
    models/                   # SQLModel models
    routers/                  # API route modules
    services/                 # Business logic
    alembic/                  # Migration scripts
    alembic.ini
    requirements.txt          # or pyproject.toml
  frontend/
    vite.config.js
    package.json
    src/
      App.svelte
      routes/                 # Page components
      lib/                    # Shared utilities
      theme/                  # Everforest CSS variables
    dist/                     # Built frontend (served by FastAPI)
  rapid-dashboard.service     # systemd user unit template
```

**Note:** The frontend `dist/` is served by FastAPI as static files via `StaticFiles`. In development, Vite's dev server proxies API calls to FastAPI. In production (the installed service), FastAPI serves both the API and the built frontend.
