# Architecture Research -- v3.2.0 General Fixes

## Current Architecture

RAPID is a 5-layer Claude Code plugin architecture: Skill Layer (24 skills) -> Agent Layer (26 agents) -> CLI Layer (rapid-tools.cjs) -> Library Layer (21 modules in src/lib/) -> Storage Layer (.planning/, .rapid-worktrees/, git branches). State is managed through Zod-validated JSON (STATE.json) with file-based locking and atomic writes via `withStateTransaction()`.

### Strengths of Existing Architecture

1. **Clear layer separation.** Skills orchestrate, agents execute, CLI dispatches, libraries own logic, storage persists. Each layer has well-defined boundaries.
2. **Transaction pattern.** All state mutations go through `withStateTransaction(cwd, mutationFn)` which provides lock -> read -> validate -> mutate -> validate -> write -> unlock atomics. This prevents corruption from concurrent access.
3. **Worktree isolation.** Each set gets a physical git worktree, eliminating file-level conflicts between parallel agents.
4. **Contract-first planning.** Interface contracts define boundaries before implementation, validated at 3 gates.
5. **Structured return protocol.** Agents communicate via `<!-- RAPID:RETURN {...} -->` markers, enabling machine-parseable coordination.

### Weaknesses Relevant to This Milestone

1. **Dual-source set resolution.** `resolveSet()` in resolve.cjs uses `plan.listSets(cwd)` which reads the filesystem (`.planning/sets/` directories), while set identity lives in STATE.json milestones. When sets are archived (moved to `.planning/archive/`), the filesystem source goes stale but STATE.json retains set records. This is the root cause of the set name resolution bug.

2. **State name mismatch between layers.** The Zod schema (`state-schemas.cjs`) defines set statuses as past-tense: `pending`, `discussed`, `planned`, `executed`, `complete`, `merged`. But skills reference present-tense/gerund statuses: `discussing` (discuss-set/SKILL.md line 268), `planning` (plan-set/SKILL.md line 297), `executing` (execute-set/SKILL.md line 148), `reviewing` (review/SKILL.md line 80). The status-rename v3.1.0 migration changed the canonical names to past-tense but several skill files still attempt present-tense transitions.

3. **Phantom CLI commands.** The tool-docs.cjs TOOL_REGISTRY advertises `wave-plan` subcommands (`wave-plan-validate`, `wave-plan-list-jobs`, `wave-plan-resolve`, `wave-plan-create-dir`) but `rapid-tools.cjs` has no `wave-plan` command handler. The plan-set/SKILL.md (line 280) and review/SKILL.md (line 190) call these nonexistent commands.

4. **No review state in schema.** The review/SKILL.md (line 80) transitions sets to `reviewing`, but `reviewing` was explicitly removed from the SetStatus Zod enum in v3.1.0 (confirmed by test at state-schemas.test.cjs line 31-32). The transition will fail with a Zod validation error.

5. **Agent prompt/CLI desync.** The tool-docs.cjs ROLE_TOOL_MAP gives agents CLI command references, but some commands in TOOL_REGISTRY do not exist in rapid-tools.cjs (e.g., `plan check-gate`, all `wave-plan` commands). Agents see these as available tools and call them, producing errors.

6. **No post-merge review path.** The lifecycle is strictly linear: `pending -> discussed -> planned -> executed -> complete -> merged`. There is no branch from `merged` back to a review state, and no aggregate review across multiple merged sets.

## Recommended Architecture Pattern

- **Pattern:** Targeted fixes within existing modular monolith -- no architectural change needed
- **Rationale:** The 5-layer architecture is sound and well-established. All 8 milestone goals are bugs or UX issues within existing modules, not structural deficiencies requiring new patterns. The fixes involve: data source consolidation (resolve.cjs), schema alignment (state-transitions.cjs + skill files), CLI command pruning (tool-docs.cjs), UX improvements (discuss-set skill), and a new lifecycle path (review-after-merge).
- **Evidence:** The v3.1.0 status-rename migration proved the architecture supports cross-cutting changes. The existing transaction pattern, schema validation, and test infrastructure provide the safety net for surgical fixes.

## Module Organization

### Module: resolve.cjs
- **Responsibility:** Resolve user-provided set/wave references (numeric indices or string IDs) to canonical identifiers
- **Public interface:** `resolveSet(input, cwd)`, `resolveWave(input, state, cwd, setId)`
- **Dependencies:** `plan.cjs` (currently for `listSets()`; should switch to `state-machine.cjs` for `readState()`)
- **Data ownership:** None (read-only resolver)
- **Bug:** Uses filesystem-based `plan.listSets()` which reads `.planning/sets/` directories. When sets are archived, the directory listing no longer matches the milestone's set list in STATE.json. Numeric index "6" may resolve to the wrong set because the filesystem has gaps (sets 4,5 archived) but the index counts remaining directories.
- **Fix direction:** `resolveSet()` should read sets from STATE.json for the current milestone, not from the filesystem. The `plan.listSets()` function can remain for planning/decomposition contexts but must not be used for user-facing resolution.

### Module: state-schemas.cjs
- **Responsibility:** Zod schemas defining the shape of STATE.json
- **Public interface:** `ProjectState`, `SetStatus`, `WaveStatus`, `JobStatus`, and their state objects
- **Dependencies:** `zod`
- **Data ownership:** Schema definitions for all persistent state
- **Current status values:** `SetStatus = z.enum(['pending', 'discussed', 'planned', 'executed', 'complete', 'merged'])`
- **No `reviewing` state.** This was intentionally removed in v3.1.0.

### Module: state-transitions.cjs
- **Responsibility:** Valid state transition map and validation function
- **Public interface:** `SET_TRANSITIONS`, `WAVE_TRANSITIONS`, `JOB_TRANSITIONS`, `validateTransition()`
- **Dependencies:** None
- **Data ownership:** Transition rules
- **Current set transitions:** `pending -> [discussed, planned]`, `discussed -> [planned]`, `planned -> [executed]`, `executed -> [complete]`, `complete -> [merged]`
- **Note:** There is no `reviewing` transition. The review/SKILL.md's `complete -> reviewing` call will hit Zod validation failure before it even reaches transition validation.

### Module: display.cjs
- **Responsibility:** Branded ANSI terminal banner rendering
- **Public interface:** `renderBanner(stage, target)`, `STAGE_VERBS`, `STAGE_BG`
- **Dependencies:** None (zero-dependency, raw ANSI codes)
- **Data ownership:** Stage-to-color and stage-to-verb mappings
- **Current colors:** Planning stages = bright blue (`\x1b[104m`), execution = bright green (`\x1b[102m`), review/merge = bright red (`\x1b[101m`). The milestone wants dark purple instead of pink. Note: the current colors are bright blue/green/red, not pink. The "pink" perception may come from terminal color scheme variations. Dark purple in 256-color ANSI would be `\x1b[48;5;53m` or `\x1b[48;5;54m`.

### Module: tool-docs.cjs
- **Responsibility:** Static registry of CLI command documentation and role-to-command mappings for agent prompt generation
- **Public interface:** `TOOL_REGISTRY`, `ROLE_TOOL_MAP`, `getToolDocsForRole(role)`, `estimateTokens(text)`
- **Dependencies:** None
- **Data ownership:** Command documentation strings and role mappings
- **Bug:** TOOL_REGISTRY contains 4 phantom `wave-plan-*` entries that do not correspond to any handler in rapid-tools.cjs. ROLE_TOOL_MAP propagates `wave-plan-validate` to the `plan-verifier` role, causing that agent to call nonexistent commands.

### Module: plan.cjs
- **Responsibility:** Set creation, loading, listing, decomposition orchestration
- **Public interface:** `createSet()`, `loadSet()`, `listSets()`, `decomposeIntoSets()`, `surfaceAssumptions()`, etc.
- **Dependencies:** `dag.cjs`, `contract.cjs`
- **Data ownership:** `.planning/sets/` directory structure, DEFINITION.md, CONTRACT.json files
- **Relevance to bugs:** `listSets()` reads `.planning/sets/` filesystem entries -- this is the function resolve.cjs depends on. After archiving, this function returns an empty array or a subset, breaking numeric resolution.

## Data Flow

### Set Resolution Flow (Current -- Buggy)

```
User input "6" -> resolveSet("6", cwd)
  -> plan.listSets(cwd)
    -> fs.readdirSync('.planning/sets/')  // reads FILESYSTEM
    -> returns alphabetically sorted directory names
  -> index into sorted array: sets[5]  // 0-based
  -> WRONG: if sets 4,5 archived, filesystem has different entries than STATE.json
```

### Set Resolution Flow (Proposed -- Fixed)

```
User input "6" -> resolveSet("6", cwd, state?)
  -> milestone = state.milestones.find(m => m.id === state.currentMilestone)
  -> sets = milestone.sets.map(s => s.id).sort()
  -> index into sorted array: sets[5]  // 0-based
  -> CORRECT: uses canonical state, unaffected by filesystem archiving
```

### State Transition Flow

```
Skill calls: node "${RAPID_TOOLS}" state transition set <milestone> <set> <status>
  -> rapid-tools.cjs dispatches to transitionSet()
    -> withStateTransaction(): acquireLock -> readState -> findSet -> validateTransition -> mutate -> validate (Zod) -> write -> unlock
```

The failure chain for "discussing" transition:
1. discuss-set/SKILL.md calls `state transition set ... discussing`
2. `transitionSet()` calls `validateTransition('pending', 'discussing')`
3. `SET_TRANSITIONS['pending']` = `['discussed', 'planned']` -- `discussing` is not in the list
4. Throws: "Invalid transition: pending -> discussing. Valid: [discussed, planned]"

The failure chain for "reviewing" transition:
1. review/SKILL.md calls `state transition set ... reviewing`
2. `transitionSet()` calls `withStateTransaction()` -> `findSet()` succeeds
3. `validateTransition('complete', 'reviewing')`
4. `SET_TRANSITIONS['complete']` = `['merged']` -- `reviewing` is not valid
5. Throws: "Invalid transition: complete -> reviewing. Valid: [merged]"

### AskUserQuestion UX Flow (Current -- Suboptimal)

```
discuss-set Step 6:
  AskUserQuestion(question="Gray area title\n\nContext: ...\n\nQuestions:\n1. ...\n2. ...\n3. ...\n\nAnswer all questions...",
                  options=undefined)  // FREEFORM -- no pre-filled options
```

The user sees a text box with multiple questions but no structured options. They must type everything.

### AskUserQuestion UX Flow (Proposed -- Improved)

```
discuss-set Step 6:
  AskUserQuestion(question="Gray area title",
                  options=[
                    { value: "option-a", description: "Approach A: ..." },
                    { value: "option-b", description: "Approach B: ..." },
                    { value: "custom", description: "I'll answer with my own approach" }
                  ])
```

Each gray area gets pre-filled options reflecting the identified approaches, plus an escape hatch for custom answers.

Not applicable for this milestone. All 8 goals are bug fixes, UX improvements, and consistency enforcement within existing scale parameters. No new load characteristics are introduced.

## Architecture Risks

1. **Resolve.cjs STATE.json dependency loop** (Medium): Switching `resolveSet()` to read from STATE.json means it must call `readState()` which is async. Currently `resolveSet()` is synchronous. This change requires making `resolveSet()` async, which cascades to all callers: `resolveWave()`, all skill bash commands calling `resolve set`, and the CLI handler. Mitigation: Either (a) pass state as a parameter to avoid async read, or (b) add a sync `readStateSync()` variant, or (c) make `resolveSet` async and update all callers. Option (a) is simplest but requires callers to pre-load state. Option (c) is cleanest but most invasive.

2. **State transition name alignment** (High): Multiple skills reference old gerund-form status names. Simply updating the skill markdown files is the fix, but if any agent has cached the old instructions (unlikely with regeneration), or if there are other references in planning artifacts that agents read as context, stale names could resurface. Mitigation: Add a lint-style test (similar to status-rename.test.cjs) that scans all skill files for invalid status names.

3. **Review-after-merge lifecycle extension** (Medium): Adding a post-merge review path requires either (a) adding a new status to the enum (e.g., `reviewed` after `merged`) or (b) implementing review as an out-of-band process that does not modify set status. Option (b) is simpler and avoids schema changes: a separate review artifact and command that works on merged sets without transitioning their status. Mitigation: Use option (b) -- a new `/rapid:review-merged` skill or a `--post-merge` flag on `/rapid:review` that skips the state transition.

4. **Banner color terminal compatibility** (Low): Moving from basic 16-color ANSI to 256-color for dark purple (`\x1b[48;5;53m`) may render poorly in terminals that only support 16 colors. Mitigation: Use the closest 16-color equivalent (standard magenta `\x1b[45m`) with bright white text, or feature-detect terminal capabilities. Standard magenta is a reasonable dark purple approximation.

5. **Phantom command removal** (Low): Removing `wave-plan` commands from tool-docs.cjs affects the `plan-verifier` agent's available tools. If plan-set/SKILL.md no longer calls `wave-plan validate-contracts`, the plan-verifier agent may need its tool set updated or the validation inlined into the plan-set skill. Mitigation: Replace the CLI command with direct contract validation in the plan-set skill (read CONTRACT.json, validate against plan content).

## Recommendations

Ordered by implementation priority and dependency:

1. **Fix set resolution to use STATE.json** (Critical): `resolveSet()` must read sets from the current milestone in STATE.json rather than from the filesystem. This is the #1 user-facing bug. Approach: accept an optional `state` parameter; if not provided, read STATE.json synchronously (add `readStateSync()`) or require callers to pass it. The CLI `resolve set` handler already has access to cwd and can call `readState()`.

2. **Align state transition names in all skills** (Critical): Update discuss-set, plan-set, execute-set, and review skills to use the canonical past-tense status names from state-schemas.cjs (`discussed`, `planned`, `executed`, `complete`, `merged`). This is a multi-file find-and-replace with careful attention to context (some gerund uses are for display, not transitions).

3. **Remove or replace phantom wave-plan commands** (High): Remove `wave-plan-*` entries from tool-docs.cjs TOOL_REGISTRY and ROLE_TOOL_MAP. Update plan-set/SKILL.md and review/SKILL.md to either inline the functionality or use alternative approaches (direct file reads for job listing, inline validation for contracts).

4. **Fix discuss-set AskUserQuestion UX** (High): Rewrite Step 5 and Step 6 of discuss-set/SKILL.md to use AskUserQuestion with pre-filled options for each gray area. Each question should present 2-3 approach options plus "I'll answer with my own approach" as an escape hatch. The "Claude decides" option should still be available per-area.

5. **Implement review-after-merge** (Medium): Add a `--post-merge` flag to `/rapid:review` (or create `/rapid:review-merged`) that operates on merged sets without modifying set status. This runs the same adversarial pipeline (scoper -> unit-tester -> bug-hunter -> devils-advocate -> judge) on the merged main branch diff instead of a worktree branch diff.

6. **Add state-update enforcement to skills** (Medium): Each skill that completes a lifecycle phase (discuss-set, plan-set, execute-set, merge) must explicitly transition state AND commit STATE.json. Add a checklist/verification step at the end of each skill that confirms the state transition succeeded. The discuss-set skill already does this (Step 8), but execute-set and merge should be audited.

7. **Fix banner colors** (Low): Change STAGE_BG values in display.cjs from bright blue/green/red to dark purple. Use `\x1b[45m` (standard magenta background) for maximum terminal compatibility, or `\x1b[48;5;53m` (256-color dark purple) if 256-color support is acceptable. Apply consistently across all stages or use a unified brand color.

8. **Add "I'll answer" option to all AskUserQuestion prompts** (Low): Audit all skills that use AskUserQuestion and ensure every prompt includes pre-filled options. Never present a bare question where "Claude decides" is the only implicit option. Every AskUserQuestion should have structured options with an explicit freeform escape hatch.

## Detailed Fix Patterns

### Pattern 1: resolveSet() State-Based Resolution

Current:
```javascript
function resolveSet(input, cwd) {
  const sets = plan.listSets(cwd);  // reads filesystem
  // ...
}
```

Proposed:
```javascript
function resolveSet(input, cwd, state) {
  let sets;
  if (state) {
    const milestone = state.milestones.find(m => m.id === state.currentMilestone);
    sets = (milestone?.sets || []).map(s => s.id).sort();
  } else {
    // Fallback to filesystem for contexts without state (e.g., during init/planning)
    sets = plan.listSets(cwd);
  }
  // ... rest unchanged
}
```

The CLI `resolve set` handler would read state first and pass it:
```javascript
const readResult = await readState(cwd);
const state = readResult?.valid ? readResult.state : null;
const result = resolveSet(input, cwd, state);
```

### Pattern 2: State Transition Name Fix

In each skill file, change transition calls:
- `discussing` -> `discussed`
- `planning` -> `planned`
- `executing` -> `executed` (where applicable)
- `reviewing` -> remove entirely (review does not change set status in v3)

### Pattern 3: Review-After-Merge

New flag on `/rapid:review`:
```
/rapid:review <set-id> --post-merge
```

Behavior:
- Skip set status validation (set is already `merged`)
- Scope files from `git diff main~1..main` (last merge commit diff) instead of worktree branch diff
- Run review pipeline in the main working tree (no worktree needed)
- Write review artifacts to `.planning/waves/{setId}/POST-MERGE-REVIEW.md`
- Do not modify STATE.json set status

### Pattern 4: AskUserQuestion with Options

Replace freeform patterns:
```
AskUserQuestion("Question\n\n1. Sub-question\n2. Sub-question\n\nAnswer all...")
```

With structured patterns:
```
AskUserQuestion(
  question="Gray area: {title}",
  options=[
    "Option A: {description}",
    "Option B: {description}",
    "Option C: {description}",
    "I'll provide my own answer"
  ]
)
```
