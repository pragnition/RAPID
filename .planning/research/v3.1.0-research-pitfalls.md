# Pitfalls Research -- v3.1.0 Polish & Cleanup

## Critical Pitfalls

### Pitfall: Incomplete Status Rename Leaves Orphaned State
- **Category:** data-loss / correctness
- **Stack:** Zod 3.25.x `z.enum`, STATE.json, state-schemas.cjs, state-transitions.cjs
- **Description:** Renaming `discussing` -> `discussed`, `planning` -> `planned`, `executing` -> `executed` touches a Zod enum that is the single source of truth for state validation. If any one of the ~50 files referencing the old strings is missed, Zod will reject the state on parse, causing `ZodError` crashes throughout the system. Existing `STATE.json` files written under the old schema will fail validation on the next `readState()` call.
- **Trigger:** A user with a project in progress (e.g., a set in `discussing` status) upgrades to v3.1.0. The next `readState()` call parses `"discussing"` against the new `z.enum(['pending', 'discussed', 'planned', 'executed', 'complete', 'merged'])` and throws `ZodError`.
- **Impact:** Critical -- the entire state machine becomes inaccessible. No commands work. The user's project is effectively bricked until they manually edit STATE.json.
- **Prevention:**
  1. Add a migration function in `state-machine.cjs` (similar to existing `migrateStateVersion()` in rapid-tools.cjs) that runs on every `readState()` call. It should silently rewrite `discussing` -> `discussed`, `planning` -> `planned`, `executing` -> `executed` in parsed JSON before Zod validation.
  2. Alternatively, read raw JSON, apply string replacements, then parse with Zod. This is the safest approach since it happens before schema validation.
  3. The migration must be idempotent -- running it on already-migrated state should be a no-op.
- **Detection:** Any `ZodError` mentioning "Invalid enum value" with the old status strings after upgrade.
- **References:** Zod v3 `z.enum` docs at https://v3.zod.dev; `state-schemas.cjs` line 4.

### Pitfall: Two Parallel Naming Systems (SetStatus vs Registry Phase)
- **Category:** correctness
- **Stack:** state-schemas.cjs (lowercase statuses), worktree.cjs + rapid-tools.cjs (PascalCase phases)
- **Description:** The codebase has TWO independent naming systems for lifecycle stages:
  1. **SetStatus** (lowercase): `discussing`, `planning`, `executing` -- in `state-schemas.cjs`, `state-transitions.cjs`, STATE.json
  2. **Registry phases** (PascalCase): `Discussing`, `Planning`, `Executing` -- in `worktree.cjs` (line 369-371 phase map), `rapid-tools.cjs` (line 1815 `validPhases`), REGISTRY.json

  The rename milestone says to change `-ing` to past tense. If this is applied only to SetStatus but NOT to Registry phases (or vice versa), the two systems will become inconsistent. The phase map in worktree.cjs maps `Discussing` -> `Discuss`, `Planning` -> `Plan`, `Executing` -> `Execute` for display -- this will also need updating.
- **Trigger:** Renaming SetStatus values without correspondingly renaming the PascalCase registry phase strings and the display label map.
- **Impact:** Critical -- `deriveNextActions()` in worktree.cjs (line 804) switches on `'executing'` (lowercase SetStatus). The `execute update-phase` command validates against `['Discussing', 'Planning', 'Executing', ...]` (PascalCase). If only one system is renamed, status-dependent features silently stop matching.
- **Prevention:**
  1. Create an exhaustive inventory of ALL locations that reference each status string (both cases) before making any changes.
  2. Rename both systems in the same commit/set to prevent partial states.
  3. Update the phase map in `worktree.cjs` (lines 369-371) alongside the rename.
  4. Update the `validPhases` array in `rapid-tools.cjs` (line 1815).
- **Detection:** After rename, run `grep -rn 'discussing\|planning\|executing\|Discussing\|Planning\|Executing' src/ skills/` and verify zero matches for old strings.

### Pitfall: Git Commit Serialization Race in Parallel Wave Execution
- **Category:** data-loss / correctness
- **Stack:** git, child_process, proper-lockfile 4.1.2
- **Description:** When executing multiple waves concurrently within a single set, all waves share the same git worktree. Git does not support concurrent operations on the same repository -- running simultaneous `git add` + `git commit` from multiple processes leads to index corruption, merge conflicts in the index file, or silently lost changes. The `.git/index.lock` file that git uses internally will cause one process to fail with "fatal: Unable to create '.git/index.lock': File exists."
- **Trigger:** Two parallel wave agents both attempt to `git add` and `git commit` at the same time in the same worktree.
- **Impact:** Critical -- git index corruption can lose uncommitted work. Concurrent commits may silently overwrite each other's staged changes. The error "Unable to create index.lock" causes agent execution to fail mid-task.
- **Prevention:**
  1. Implement a git commit serialization lock using the existing `proper-lockfile` infrastructure. Create a `git-commit` lock that must be acquired before any `git add` + `git commit` sequence.
  2. The lock must cover the entire add-commit sequence (not just commit), since `git add` modifies the index which is shared.
  3. Consider using a queue pattern: each wave writes to a commit queue, and a single serializer process drains the queue.
  4. Alternative: use separate worktrees per wave (each wave gets its own branch/worktree), then merge wave branches. This avoids serialization entirely but adds merge complexity.
- **Detection:** Look for "index.lock" errors in agent output. Check commit history for missing expected files.
- **References:** git documentation on concurrent access; `lock.cjs` for existing locking pattern.

## High-Risk Pitfalls

### Pitfall: STATE.json Lock Contention Under Parallel Waves
- **Category:** performance / correctness
- **Stack:** proper-lockfile 4.1.2, state-machine.cjs
- **Description:** The current locking system uses a single `state` lock for all STATE.json mutations via `withStateTransaction()`. When multiple waves execute in parallel and each needs to update state (e.g., recording progress), they will contend on this single lock. The retry config has `retries: 10` with exponential backoff up to 2000ms, but under high parallelism (3+ concurrent waves), lock starvation or timeout becomes likely.
- **Trigger:** Three or more concurrent wave execution agents all try to update STATE.json within a short window.
- **Impact:** High -- lock acquisition failures cause agent crashes with unhandled rejection. The `onCompromised` callback in `lock.cjs` only logs to stderr and does not prevent data corruption if two processes both believe they hold the lock.
- **Prevention:**
  1. Reduce lock hold time: read state, compute mutation, acquire lock, re-read state (optimistic check), write, release. This minimizes the critical section.
  2. Consider batching state updates: waves queue state changes and a coordinator applies them periodically.
  3. Increase retry count or switch to infinite retries with a reasonable timeout for parallel scenarios.
  4. Add explicit error handling for lock acquisition failures in the execution orchestrator.
- **Detection:** Lock acquisition timeout errors in stderr. STATE.json with stale `lastUpdatedAt` despite recent activity.
- **References:** `lock.cjs` RETRY_CONFIG (lines 9-14); `state-machine.cjs` `withStateTransaction()` (line 129).

### Pitfall: Skills Reference Hardcoded Status Strings in Bash Commands
- **Category:** correctness
- **Stack:** SKILL.md files (discuss-set, plan-set, execute-set, review, merge)
- **Description:** Multiple skill files contain hardcoded status strings in bash `node "${RAPID_TOOLS}" state transition set ... discussing` commands. These are documentation/instruction files read by the LLM at runtime -- they are NOT validated by tests. If the enum values change but the skill files are not updated, the LLM will issue invalid state transition commands that fail at runtime.
- **Trigger:** Renaming status values in code without updating corresponding skill instructions.
- **Impact:** High -- skills are the primary user-facing interface. If `/rapid:discuss-set` tries to transition to `discussing` (old name) instead of `discussed` (new name), the command fails with a confusing error from `validateTransition()`.
- **Prevention:**
  1. Grep all SKILL.md files for old status strings: `grep -rn 'discussing\|planning\|executing' skills/`
  2. Known locations that MUST be updated:
     - `skills/discuss-set/SKILL.md` line 268: `discussing`
     - `skills/plan-set/SKILL.md` line 297: `planning`
     - `skills/execute-set/SKILL.md` line 148: `executing`
     - `skills/execute-set/SKILL.md` line 269: `complete` (this one stays the same)
     - `skills/review/SKILL.md` line 80: `reviewing` (NOTE: this references a status that does NOT exist in the current schema -- see separate pitfall)
     - `skills/merge/SKILL.md` line 364: `merged` (stays the same)
  3. Add a test that extracts valid statuses from `SetStatus.options` and greps all SKILL.md files for any `state transition set` calls that use non-valid status strings.
- **Detection:** Runtime errors: `Unknown status "discussing". Valid statuses: pending, discussed, planned, executed, complete, merged`.

### Pitfall: Review Skill References Non-Existent `reviewing` Status
- **Category:** correctness
- **Stack:** skills/review/SKILL.md, state-schemas.cjs
- **Description:** The review skill at line 80 issues `state transition set <milestone> <set-id> reviewing`, but `reviewing` is NOT a valid SetStatus. The valid statuses are `['pending', 'discussing', 'planning', 'executing', 'complete', 'merged']`. Additionally, `worktree.cjs` line 811-816 has `case 'reviewing':` and `case 'merging':` in `deriveNextActions()` that will never match because these are not valid statuses.
- **Trigger:** Running `/rapid:review` on a set.
- **Impact:** High -- the state transition call will throw "Unknown status" error, breaking the review workflow entirely.
- **Prevention:**
  1. The rename effort should NOT introduce `reviewed` as a new status. Instead, this existing bug should be fixed by either removing the transition call or adding `reviewing` to the schema if it is actually needed.
  2. If the status rename proceeds, decide whether the review/merge workflow needs its own statuses or should remain as `complete -> merged` without intermediate states.
  3. Clean up the dead `case 'reviewing':` and `case 'merging':` branches in `deriveNextActions()`.
- **Detection:** Running `/rapid:review` on any set will immediately fail.
- **References:** `skills/review/SKILL.md` line 75-80; `worktree.cjs` lines 811-823; `state-schemas.cjs` line 4.

### Pitfall: Partial Wave Failure Leaves Inconsistent State
- **Category:** correctness
- **Stack:** execute.cjs, state-machine.cjs, proper-lockfile
- **Description:** When waves execute in parallel and one wave fails mid-execution while others succeed, the system has no mechanism to record partial completion at the wave level. The current schema only tracks set-level status (`executing` -> `complete`). If wave 2 of 3 fails, there is no way to record "waves 1 and 3 complete, wave 2 failed" -- the set remains in `executing` and the user must re-execute all waves.
- **Trigger:** One wave encounters an error (test failure, git conflict, agent crash) while other parallel waves complete successfully.
- **Impact:** High -- wasted compute and time. Successfully executed waves must be re-run because there is no per-wave state tracking.
- **Prevention:**
  1. Add per-wave status tracking, even if lightweight (a JSON file per wave in `.planning/waves/{setId}/`).
  2. The reconciliation engine (`reconcileWave`, `reconcileWaveJobs`) already exists -- extend it to mark individual wave completion.
  3. On resume, only re-execute failed waves, not all waves.
  4. Consider an error boundary pattern: catch wave-level failures without aborting the entire set execution.
- **Detection:** After parallel execution with one failure, check if completed waves must be re-run.

## Medium-Risk Pitfalls

### Pitfall: Test Hardcoding of Status Strings Creates Brittle Tests
- **Category:** correctness
- **Stack:** node:test, state-schemas.test.cjs, state-transitions.test.cjs, state-machine.test.cjs, state-machine.lifecycle.test.cjs
- **Description:** Test files contain ~80+ hardcoded instances of `'discussing'`, `'planning'`, `'executing'` in string literals. These are direct value comparisons, not references to the schema. If the schema changes but any test assertion is missed, tests will fail with confusing messages.
- **Trigger:** Changing the enum values and missing a test assertion.
- **Impact:** Medium -- tests fail, but this is caught during development. The risk is that the volume of changes (~80 occurrences across 6 test files) makes it easy to miss one.
- **Prevention:**
  1. Use `SetStatus.options` or exported constants for test assertions instead of hardcoded strings.
  2. Before the rename, generate a complete inventory: `grep -c "'discussing'\|'planning'\|'executing'" src/lib/*.test.cjs`
  3. Consider a find-and-replace approach with verification: replace all occurrences, run tests, fix any that break.
- **Detection:** `npm test` will catch all mismatches immediately.

### Pitfall: Install Version Detection Logic May Be Fragile
- **Category:** compatibility
- **Stack:** setup.sh, skills/install/SKILL.md, package.json
- **Description:** Goal 4 requires making install version-aware. The current `setup.sh` only checks if `node_modules` exists (line 54) and skips `npm install` entirely. If the version detection compares RAPID_TOOLS path against package.json version, there are several edge cases:
  1. The path in `.env` may point to a different installation than the one being upgraded.
  2. `node_modules` may exist but be from an older version with different dependencies.
  3. The `proper-lockfile` or `zod` versions may be pinned differently between versions.
- **Trigger:** User upgrades RAPID from v3.0.0 to v3.1.0 while the old `RAPID_TOOLS` path still exists in their shell config.
- **Impact:** Medium -- the install may succeed but point to the wrong binary, or dependencies may be stale.
- **Prevention:**
  1. Store the version in `.env` alongside `RAPID_TOOLS`: `RAPID_VERSION=3.1.0`.
  2. On install, compare the stored version against `package.json` version.
  3. If versions differ, force `npm install` even if `node_modules` exists.
  4. Update `RAPID_TOOLS` path in shell config if it points to a different installation directory.
- **Detection:** `node "${RAPID_TOOLS}" --version` returning the wrong version after upgrade.

### Pitfall: Removing Deprecated Stubs Breaks Muscle Memory
- **Category:** compatibility
- **Stack:** skills/*.md (discuss, execute, plan, wave-plan, new-milestone, set-init)
- **Description:** Goal 3 removes deprecation stubs for old commands. Currently, running `/rapid:discuss` shows a helpful message: "Use /rapid:discuss-set instead." If the stub is deleted entirely, running `/rapid:discuss` will show Claude Code's generic "unknown skill" error with no guidance. Users who have muscle memory for v2 commands will get no help.
- **Trigger:** User types `/rapid:discuss` after upgrading to v3.1.0.
- **Impact:** Medium -- poor UX, user confusion, but no data loss. The user will eventually find the right command.
- **Prevention:**
  1. Consider whether "cleanup" means deleting the stubs or keeping minimal redirects.
  2. If deleting: update the help skill to prominently show the command mapping.
  3. If keeping minimal stubs: reduce them to a single line that says "Removed in v3.1. Use /rapid:discuss-set."
  4. Add the command mapping to a "migration notes" section that displays on first use after upgrade.
- **Detection:** User complaints about missing commands.

### Pitfall: `validateDiskArtifacts` Hardcodes Status Strings
- **Category:** correctness
- **Stack:** state-machine.cjs lines 243, 254
- **Description:** The `validateDiskArtifacts()` function contains hardcoded status arrays: `['planning', 'executing', 'complete', 'merged']` and `['executing', 'complete', 'merged']`. These will need updating to the new names. Since this function only returns warnings (not errors), a missed update would cause silent suppression of valid warnings.
- **Trigger:** A set is in `planned` status (new name) but `validateDiskArtifacts` checks for `'planning'` (old name) and never matches.
- **Impact:** Medium -- disk artifact warnings silently disappear, masking potential issues.
- **Prevention:** Replace hardcoded arrays with a computed set derived from `SetStatus.options` or a centralized constant.
- **Detection:** Run `validateDiskArtifacts` on a set in `planned` status and verify warnings are generated.

### Pitfall: `deriveNextActions` Missing Cases for New Status Names
- **Category:** correctness
- **Stack:** worktree.cjs lines 788-834
- **Description:** The `deriveNextActions()` function uses a switch statement on `set.status` with cases for `'pending'`, `'executing'`, `'reviewing'`, `'merging'`, `'complete'`. After the rename, `'executing'` must become `'executed'`. But this function also has cases for `'reviewing'` and `'merging'` which are NOT valid statuses -- these are dead code. The rename is an opportunity to clean this up.
- **Trigger:** A set transitions to `executed` but the switch still checks for `executing`.
- **Impact:** Medium -- the "next actions" feature stops suggesting actions for executing sets. No crash, but degraded UX.
- **Prevention:** Update the switch cases and remove dead `reviewing`/`merging` branches.
- **Detection:** `/rapid:status` shows no suggested next actions for active sets.

## Anti-Patterns for This Stack

### Anti-Pattern: Find-and-Replace Without Semantic Understanding
- **What people do:** Use global find-and-replace to change `discussing` -> `discussed` across the entire codebase.
- **Why it fails:** The word "discussing" appears in natural language contexts (agent descriptions, comments, help text) where it should NOT be changed to past tense. A global replace would turn "Start planning discussion for auth" into "Start planned discussion for auth" and "sets, each executing in its own git worktree" into "sets, each executed in its own git worktree."
- **Correct approach:** Only replace status string literals (in quotes: `'discussing'`, `"discussing"`) and status-specific identifiers. Leave natural language prose unchanged. Use regex: replace `'discussing'` and `"discussing"` but not bare word `discussing` in prose.
- **Example:** `grep -rn "'discussing'" src/` catches string literals. `grep -rn "discussing" src/` catches both literals and prose -- DO NOT replace all matches.

### Anti-Pattern: Adding Concurrency Without Error Boundaries
- **What people do:** Enable parallel wave execution by launching multiple subagents and hoping they all succeed.
- **Why it fails:** If one wave fails, the others may have already committed changes. Without error boundaries, the set is left in a partially-executed state with no clear recovery path. The current `reconcileWave` function detects issues after the fact but cannot prevent them.
- **Correct approach:** Implement a supervisor pattern:
  1. Launch all waves with individual error boundaries (try/catch per wave).
  2. Record per-wave success/failure.
  3. If any wave fails, allow running waves to complete (don't kill them).
  4. Report which waves succeeded and which failed.
  5. On re-execution, skip already-completed waves.

### Anti-Pattern: Version Check via String Comparison on Paths
- **What people do:** Compare `RAPID_TOOLS` file path strings to detect version changes.
- **Why it fails:** The path does not encode version information. A user could update the code in-place (git pull) without changing the path. Or they could have multiple installations at different paths.
- **Correct approach:** Read `package.json` version at the `RAPID_TOOLS` location and compare it against an expected version. Store the installed version in `.env` or a separate version file.

## Version-Specific Gotchas

- **Node.js 18+**: `fs.cpSync` is available (added in 16.7.0) but `fs.cp` callback API is not stable until 20. Stick to `fs.cpSync` or manual copy for directory operations.
- **Zod 3.25.76**: This is a very recent Zod v3 release. The `z.enum` API is stable, but be aware that Zod 4.x has a different API surface. Do not accidentally reference Zod 4 patterns.
- **proper-lockfile 4.1.2**: The `realpath: false` option (used in `lock.cjs` line 58) is important for symlinked paths. Git worktrees create symlinked `.git` files -- removing `realpath: false` would break locking in worktree directories.
- **Git worktrees**: Each worktree has its own working directory and index but shares the object store and refs with the main repo. Concurrent commits in different worktrees to the same branch will conflict. Parallel waves in the same worktree share the same index, making concurrent git operations unsafe.

## Security Pitfalls

1. **Lock file cleanup exposes PID information**: `lock.cjs` writes `{ pid, timestamp }` to lock target files. The `cleanStaleLocks()` function uses `process.kill(pid, 0)` to check if a process is alive. On multi-user systems, this could reveal process existence information to other users. -- **Prevention:** Restrict `.planning/.locks/` permissions to the current user.

2. **execSync in reconcileWave runs contract test files**: `execute.cjs` line 507 runs `node "${testFile}"` via `execSync`. If a contract test file is maliciously crafted, it executes arbitrary code. -- **Prevention:** This is by design (tests must run), but ensure contract test files are code-reviewed before execution. The 30-second timeout (line 507) limits blast radius.

## Performance Traps

1. **readState() is called on every CLI invocation**: Every `rapid-tools.cjs` command that needs project root calls `readState()`, which reads and parses STATE.json synchronously. With parallel wave execution adding more frequent state reads, this could become a bottleneck on large projects. -- **Mitigation:** Cache parsed state in memory within a single process lifetime. Multiple processes still need to read from disk.

2. **Lock contention scales quadratically with parallelism**: With N parallel waves, lock contention for STATE.json grows as O(N^2) because each wave's retry attempts interfere with other waves' retries. The exponential backoff with randomization helps, but with >5 concurrent waves, expect significant latency. -- **Mitigation:** Batch state updates or use a coordinator process that serializes state writes.

3. **reconcileWave reads every DEFINITION.md for every set in the wave**: The `reconcileWave()` function reads definition files and checks file existence for every artifact. With parallel waves, multiple reconciliation passes may run concurrently, each doing redundant I/O. -- **Mitigation:** Cache definition parsing results or run reconciliation as a post-execution step rather than inline.

## Summary Priority List

1. **STATE.json migration on status rename** -- Existing projects will brick if old status values are not auto-migrated before Zod validation. This MUST be implemented before or alongside the rename.
2. **Git commit serialization for parallel waves** -- Concurrent git operations in the same worktree will corrupt the index. A locking mechanism is mandatory for parallel wave execution.
3. **Two naming systems (SetStatus vs Registry Phase)** -- Both lowercase statuses AND PascalCase phases must be renamed together, or the system breaks silently.
4. **Skills reference hardcoded status strings** -- 6+ skill files contain old status names in bash commands that the LLM will execute verbatim. All must be updated.
5. **Review skill uses non-existent `reviewing` status** -- Pre-existing bug that will surface during the rename effort. Must be addressed.
6. **STATE.json lock contention under parallelism** -- The single-lock pattern will cause timeout failures with 3+ concurrent waves.
7. **Partial wave failure has no recovery path** -- No per-wave state tracking means failed parallel executions waste completed work on retry.
8. **Test files contain ~80 hardcoded status strings** -- High volume of changes needed in test files; easy to miss one.
9. **Install version detection edge cases** -- Path-based version detection is unreliable; need version stored in .env.
10. **Deprecated stub removal UX** -- Users lose migration guidance if stubs are deleted without alternative help.
