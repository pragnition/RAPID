# Domain & UX Research: Robustness & Fixes (v3.5.0)

## Domain Context

RAPID is a Claude Code plugin for parallel AI-assisted development. Its users are developers who interact primarily through slash commands in the Claude Code CLI (`/rapid:init`, `/rapid:start-set 1`, etc.). The tool orchestrates 26 specialized agents across a strict lifecycle: init -> start-set -> discuss-set -> plan-set -> execute-set -> review -> merge. Each stage transitions set state through a well-defined state machine.

This milestone focuses on robustness: fixing broken state transitions, incorrect CLI command references in agent prompts, AskUserQuestion constraint violations, worktree package management, and improving the discuss/planning stages to give more weight to UI/UX quality.

---

## Domain Conventions

### Comparable Products and the Specification-Driven Development Landscape

RAPID operates in the emerging domain of **specification-driven development (SDD) tools** alongside:

- **Kiro** (Amazon): Spec-driven IDE using requirements.md -> design.md -> tasks.md workflow. Each workflow step is one markdown document. Kiro guides users through 3 steps inside VS Code. (Source: [Martin Fowler SDD article](https://martinfowler.com/articles/exploring-gen-ai/sdd-3-tools.html))
- **GitHub Spec Kit**: Distributed as a CLI that creates workspace setups for coding assistants. Interaction via slash commands in the coding assistant. (Source: [Thoughtworks SDD analysis](https://www.thoughtworks.com/en-us/insights/blog/agile-engineering-practices/spec-driven-development-unpacking-2025-new-engineering-practices))
- **worktree-workflow** (forrestchang): Claude Code plugin specifically for git worktree parallel development, using a bash script + skill to manage worktrees. (Source: [GitHub repo](https://github.com/forrestchang/worktree-workflow))
- **SpillwaveSolutions/parallel-worktrees**: Runs parallel subagents synced with git worktrees. (Source: [GitHub repo](https://github.com/spillwavesolutions/parallel-worktrees))

**Industry standard terminology** in this space:
- "Set" / "Task" / "Wave" -- RAPID uses its own vocabulary. Kiro uses "requirements -> design -> tasks." Spec Kit uses "spec -> implementation plan." Users in the broader space expect **"task"** and **"plan"** as primary nouns. RAPID's "set" concept is unique and requires onboarding.
- "Slash commands" are the standard interaction model for Claude Code plugins (e.g., `/rapid:status`, `/rapid:merge 1`). This is consistent across the ecosystem.
- "Worktree" is understood by experienced git users but may require explanation for general developers.

### Standard Workflows

The dominant convention across SDD tools is a **linear pipeline with checkpoints**:
1. Requirements/specification capture
2. Design/planning decomposition
3. Execution/implementation
4. Review/testing
5. Integration/merge

RAPID follows this pattern faithfully. The key domain expectation is that **each stage is idempotent and resumable** -- a user should be able to re-run any stage without data loss.

## Interaction Models

### Primary Interaction Paradigm

RAPID operates as a **CLI plugin with structured prompting**. Users type slash commands, and the plugin orchestrates multi-agent pipelines behind the scenes. This is consistent with the broader Claude Code ecosystem.

**Key conventions from the domain** (Source: [Command Line Interface Guidelines](https://clig.dev/)):

1. **Suggest next actions.** When several commands form a workflow, suggesting to the user commands they can run next helps them learn and discover functionality. RAPID already does this well with its progress breadcrumbs (`init [done] > start-set [done] > discuss-set > ...`).

2. **Return zero exit code on success, non-zero on failure.** RAPID's CLI follows this via process exit codes. Agents report via RAPID:RETURN protocol (COMPLETE/CHECKPOINT/BLOCKED).

3. **Send output to stdout for machines, messaging to stderr for humans.** RAPID already follows this convention (`process.stdout.write(JSON.stringify(...))` for structured data, `process.stderr.write('[RAPID]...')` for human messages).

4. **Don't print scary-looking stack traces.** Errors should be human-readable with clear next steps. RAPID's error breadcrumbs serve this purpose.

### AskUserQuestion Constraints

The AskUserQuestion tool in Claude Code has a **hard limit of 4 options per question** (Source: [GitHub Issue #12420](https://github.com/anthropics/claude-code/issues/12420)). This is a platform constraint that RAPID must respect.

**Current violation found:** The discuss-set skill (Step 5, lines 166-170) lists 5 options for gray area selection despite the title saying "identify exactly 4 gray areas." The text contradicts itself and exceeds the AskUserQuestion limit. This must be fixed to present exactly 4 options (or use multi-select with 4 items).

**Additional AskUserQuestion conventions:**
- Each question supports 1-4 questions with 2-4 options each
- Questions have a header (max 12 characters) and full question text
- `multiSelect: true` allows selecting multiple options
- Users can always type a custom response ("Other" is implicitly available)
- 60-second timeout per question
- Cannot be used from sub-agents -- only the main skill can ask questions

### Input/Output Patterns Users Expect

1. **Numeric shorthand for references.** RAPID already supports this (`/rapid:start-set 1` instead of `/rapid:start-set auth-system`). This is a strong UX pattern consistent with how `git stash` and `docker` reference items.

2. **Progress indication.** Users expect to know what is happening during long-running agent operations. RAPID uses stage banners (colored ANSI output) and breadcrumbs. The domain convention (Source: [Evil Martians CLI UX patterns](https://evilmartians.com/chronicles/cli-ux-best-practices-3-patterns-for-improving-progress-displays)) recommends three patterns:
   - Spinner (for indeterminate operations)
   - X of Y counter (for countable steps)
   - Progress bar (for measurable operations)
   RAPID uses X of Y (`Wave 1: complete (skipping)`, `Wave 2: pending (will execute)`) which is appropriate for its step-based workflow.

3. **Error communication.** Users expect errors with three elements: what went wrong, why, and what to do next. RAPID's breadcrumb pattern achieves this well.

### Feedback and Progress Indication Conventions

For multi-agent orchestration tools, the standard expectation is:
- **Agent spawn notification:** "Spawning rapid-executor for wave 1..."
- **Activity indication during agent work:** Users in this domain understand that agent operations take time, but silence is unnerving.
- **Completion summary:** RAPID:RETURN protocol is a strong pattern for this.
- **Auto-advance on success:** RAPID already does this ("Only CHECKPOINT or BLOCKED stops"). This is the domain convention -- users do not want to confirm every successful step.

## Information Architecture

### How Information Should Be Organized

RAPID's information architecture follows a **hierarchical artifact model**:
```
.planning/
  STATE.json          <- single source of truth for status
  ROADMAP.md          <- top-level plan
  sets/{set-id}/
    DEFINITION.md     <- set scope
    CONTRACT.json     <- interface contract
    CONTEXT.md        <- discussion output
    wave-*-PLAN.md    <- per-wave plans
    WAVE-*-COMPLETE.md <- completion markers
```

This is consistent with the SDD domain convention of **durable markdown artifacts** as the source of truth. Kiro, Spec Kit, and RAPID all use markdown files to persist agent behavior across context window limits and session boundaries. (Source: [RedMonk: 10 Things Developers Want](https://redmonk.com/kholterhoff/2025/12/22/10-things-developers-want-from-their-agentic-ides-in-2025/))

### Navigation and Discovery

The `/rapid:status` command serves as the **dashboard entry point** -- showing all sets with statuses and suggested next actions. This matches the domain convention of a single "status" command as the orientation tool (analogous to `git status`, `docker ps`, `kubectl get pods`).

**Convention:** The status dashboard should be the first thing a confused user runs. It must always suggest the correct next action for each set based on its current state.

### Progressive Disclosure

RAPID implements progressive disclosure through its lifecycle stages. Users see only what they need at each step:
- During discuss: gray areas and options
- During plan: wave decomposition summary (not full plans)
- During execute: wave progress counters
- During review: scope summary and acceptance criteria
- During merge: conflict counts and resolution strategies

This is appropriate for the domain.

## Accessibility Considerations

### Terminal/CLI Accessibility

1. **Color as supplement, not sole indicator.** RAPID uses ANSI colors for banners and progress, with text labels always present alongside color. The display module uses basic 16-color ANSI palette for maximum terminal compatibility. However, there is no `NO_COLOR` environment variable support -- the code comment explicitly notes "Always outputs colors (no NO_COLOR checking)." The [clig.dev guidelines](https://clig.dev/) recommend respecting `NO_COLOR` for accessibility.

2. **Screen reader compatibility.** RAPID's text-based output is inherently screen-reader friendly. The structured table output (`| # | Set | Status | ...`) renders well in terminal screen readers. No issues identified.

3. **Keyboard navigation.** AskUserQuestion options are keyboard-navigable by default (Claude Code handles this). No RAPID-specific concerns.

### Cognitive Load

The primary cognitive load concern is RAPID's **unique vocabulary**: sets, waves, jobs, worktrees, contracts, DAG. For this milestone's goal of "improving discuss/planning to emphasize UI/UX considerations," the key insight is that **the planning agents should actively surface UI/UX tradeoffs as gray areas** rather than treating them as implementation details.

## User Expectations

### What Users Assume Based on Domain Conventions

1. **Idempotent commands.** Users expect they can re-run any command safely. If `/rapid:execute-set 1` fails midway, running it again should resume from where it left off. RAPID's artifact-based re-entry (WAVE-COMPLETE.md markers) addresses this, but the **execution-to-complete state transition failure** (milestone goal #5) violates this expectation -- if the set stays in "executed" instead of transitioning to "complete," the review skill rejects it.

2. **Graceful degradation.** If one sub-operation fails, the overall pipeline should continue where possible. RAPID already implements this for researcher failures (graceful degradation) but not consistently for state transitions.

3. **Consistent command syntax.** Agents that reference CLI commands MUST use the exact syntax that the CLI accepts. The milestone identifies that agents use wrong subcommands (e.g., `state set-status`, `state --help`, `state update-set`). Users (and agents acting as users) expect commands documented in one place to work everywhere. **The TOOL_REGISTRY in tool-docs.cjs is the canonical command reference** and should be the single source of truth agents consult.

4. **Solo mode as default.** The milestone goal #8 (project-wide solo mode config) reflects a real user expectation: many projects have one developer and do not need worktree isolation. Users expect a "simple mode" that avoids unnecessary complexity. The `config.json` currently has no `soloMode` field -- this needs to be added.

5. **Package availability in worktrees.** Users expect that `node_modules` is available in every worktree for running tests. The milestone identifies that worktrees may not have packages installed, causing test failures. The domain convention (Source: [worktree-workflow patterns](https://github.com/forrestchang/worktree-workflow)) is to either symlink `node_modules` or run `npm install` in each worktree automatically.

### Terminology Expectations

- Users familiar with git expect "branch" and "merge" to behave as git concepts.
- Users familiar with CI/CD expect "pipeline" to mean a sequential chain of stages.
- "Review" is expected to be a non-mutating observation step (RAPID correctly implements this).
- "Complete" is expected to mean "ready for the next stage" -- the execution-to-complete gap violates this.

### Mental Models from Competing Products

- From **Kiro/Spec Kit:** Users expect markdown files as the durable source of truth. RAPID matches this.
- From **git:** Users expect worktrees to be self-contained and functional. RAPID needs to ensure package availability.
- From **CI/CD tools (GitHub Actions, Jenkins):** Users expect failed stages to be clearly marked and retryable. RAPID's breadcrumb pattern handles this well.
- From **project management tools:** Users expect a single dashboard view of all work items. `/rapid:status` serves this role.

### Onboarding and Learning Curve

The learning curve for RAPID is steeper than Kiro (3 steps) or Spec Kit (CLI + slash commands). RAPID has 24 skills, 26 agents, and a 6-stage lifecycle. The breadcrumb navigation pattern helps orient users, but the vocabulary (sets, waves, contracts, DAG) requires explanation.

For this milestone, the key onboarding concern is **command syntax correctness in agent prompts**. When agents reference wrong CLI subcommands, users who read agent output cannot reproduce the commands -- this is a trust-breaking UX violation.

## Milestone-Specific UX Analysis

### Goal 1: Fix Agent Command Awareness

**UX Impact: CRITICAL**

Agents using wrong CLI subcommands (e.g., `state set-status`, `state --help`, `state update-set`) create a broken feedback loop. The tool-docs.cjs TOOL_REGISTRY defines the canonical commands, and the build-agents pipeline injects these into agent prompts. The fix must ensure:
- Every agent's `<tools>` section contains only valid commands from TOOL_REGISTRY
- The core-identity.md module references TOOL_REGISTRY commands, not free-form examples
- Agents that do not use CLI commands (research agents, scoper, etc.) should not have misleading command examples

**Domain convention:** CLI tools should have a single canonical help reference. In RAPID, `tool-docs.cjs` is that reference. Agent prompts are generated artifacts and should derive their command knowledge from this single source.

### Goal 2: Fix DEFINITION.md Generation

**UX Impact: HIGH**

The start-set skill generates scoped CLAUDE.md, which requires DEFINITION.md. If DEFINITION.md is missing, the scoped CLAUDE.md generation fails silently (warning in output). Users see "Warning: Scoped CLAUDE.md could not be generated (missing CONTRACT.json or DEFINITION.md)" but may not understand what DEFINITION.md is or how to create it.

**User expectation:** If a required artifact is missing, either generate it automatically or provide a specific command to create it. The error message should say exactly what to do, not just what is missing.

### Goal 3: Fix Discuss Skill 4-Option Limit

**UX Impact: HIGH**

The AskUserQuestion hard limit of 4 options means the discuss-set skill's Step 5 (which currently lists 5 gray area options) will fail or truncate. The fix should:
- Present exactly 4 gray areas (not 5)
- Use multiSelect to let users pick which to discuss
- The "Key Principles" section says "2-5 gray areas" but the AskUserQuestion constraint caps at 4 options

**Domain convention:** When platform constraints limit options, the tool should work within those constraints transparently. The user should never encounter a tool error from option overflow.

### Goal 4: Worktree Package Management

**UX Impact: HIGH**

Users expect `npm test` to work in a worktree immediately after creation. The domain convention for worktree tools is:
- **Option A (symlink):** Create a symlink from worktree `node_modules` to the main repo's `node_modules`. Fast, no disk overhead, but fragile if sets add different dependencies.
- **Option B (install):** Run `npm install` in the worktree after creation. Reliable but slow.
- **Option C (copy):** Copy `node_modules` from main repo. Fast for first setup but stale over time.

The **recommended pattern** from the worktree-workflow ecosystem is symlink as default with an `npm install` fallback if the worktree adds new dependencies. The merge agent should verify packages before running integration tests.

### Goal 5: Fix Execution-to-Complete State Transition

**UX Impact: CRITICAL**

The execute-set skill transitions `planned -> executed` at start and `executed -> complete` at end. If the executor fails to reach Step 6 (Complete Set), the set remains in "executed" state. The review skill then rejects it because it requires "complete" status.

**User expectation:** If all waves finished successfully, the set should be "complete." The transition should be resilient:
- Option A: The review skill should also accept "executed" status (defensive)
- Option B: The execute-set skill should ensure the transition happens even on partial failures
- Both options should be implemented for defense in depth.

**Domain convention (idempotent state transitions):** Running `/rapid:execute-set` on an already-executed set should detect all waves are complete and transition to "complete" without re-executing anything.

### Goal 6: Fix Merge Untracked File Handling

**UX Impact: MEDIUM**

Planning artifacts (`.planning/sets/*/` files) created during planning on main can block merges because git sees them as untracked files that would be overwritten. The domain convention is to commit all planning artifacts before merge.

**User expectation:** The merge pipeline should "just work" without requiring manual cleanup of planning files. The fix should ensure sets commit their planning docs in the worktree before merge starts.

### Goal 7: Fix Review File Discovery

**UX Impact: MEDIUM**

Post-merge review creates files (REVIEW-SCOPE.md, REVIEW-UNIT.md) on main, but downstream agents (unit-test, uat, bug-hunt) look for them in the worktree path. The fix should ensure review artifacts are discoverable regardless of whether the review is pre-merge (worktree) or post-merge (main).

**Domain convention:** Artifact paths should be deterministic and documented. Agents should resolve artifact paths based on the review mode (pre-merge vs post-merge), not assume a fixed location.

### Goal 8: Solo Mode Config

**UX Impact: MEDIUM**

Users working alone should not need to pass `--solo` on every `/rapid:start-set` command. The domain convention for default configuration is a project-level config file (already exists as `.planning/config.json`).

**User expectation:** Add `"soloMode": true` to config.json and have `start-set` read it as the default. The `--solo` flag should still work as an override. Similarly, `--no-solo` should override the config for cases where worktree isolation is wanted.

### Goal 9: Improve Discuss/Planning UI/UX Emphasis

**UX Impact: HIGH**

The discuss-set and plan-set skills currently focus on implementation/technical decisions. The milestone wants them to also surface UI/UX quality considerations.

**Domain convention (SDD tools):** The best SDD tools surface three categories of decisions:
1. **Technical architecture** (already well-covered by RAPID)
2. **User experience / interaction design** (gap identified)
3. **Quality and testing strategy** (partially covered)

**Recommendations for discuss-set:**
- One of the 4 gray areas should explicitly address UI/UX considerations if the set has user-facing components
- The gray area analysis should check whether the set modifies user-facing code (display, prompts, output formatting, error messages) and generate a UI/UX gray area if so

**Recommendations for plan-set:**
- The planner agent should include UI/UX acceptance criteria in wave plans for user-facing sets
- The plan-verifier should check for UI/UX coverage in sets that modify user-facing code

## Recommendations

Ordered by impact and relevance to the milestone:

1. **Fix AskUserQuestion option count in discuss-set Step 5**: Reduce gray areas from 5 to exactly 4 options to comply with the platform constraint. This is a hard blocker that causes runtime failures. -- Priority: **critical**

2. **Sweep all agents for correct CLI command syntax**: Audit every agent definition and role module against TOOL_REGISTRY in tool-docs.cjs. Replace any free-form command references with the canonical syntax. The `<tools>` section injected by build-agents is the authoritative reference. -- Priority: **critical**

3. **Make execution-to-complete transition resilient**: Both (a) accept "executed" in the review skill as a fallback, and (b) ensure execute-set performs the transition even during partial failures. Idempotent re-run of execute-set on an already-executed set with all waves complete should transition to "complete." -- Priority: **critical**

4. **Add UI/UX gray area bias to discuss-set**: When the set modifies user-facing code (display.cjs, skill SKILL.md files, agent prompts, error messages), the gray area analysis should explicitly include a UI/UX consideration. Add detection for user-facing files in the gray area generation logic. -- Priority: **high**

5. **Add UI/UX acceptance criteria generation to plan-set**: The planner role should check whether the set touches user-facing code and, if so, include UI/UX-specific acceptance criteria (e.g., "error messages include clear next-step guidance," "progress indication during long operations"). -- Priority: **high**

6. **Add `soloMode` to config.json schema**: Add a `planning.soloMode` boolean (default: false) to `.planning/config.json`. Have `start-set` read this value as the default, with `--solo` and `--no-solo` flags as overrides. -- Priority: **high**

7. **Ensure worktree package availability**: After worktree creation, symlink `node_modules` from the project root. Add a `--install` flag for cases where the set adds new dependencies. The merge agent should verify `node_modules` exists before running integration tests. -- Priority: **high**

8. **Commit planning artifacts in worktrees before merge**: The merge skill should check for uncommitted `.planning/` files in the worktree and commit them before attempting the git merge. This prevents untracked file conflicts. -- Priority: **medium**

9. **Fix review artifact path resolution for post-merge mode**: Review agents should use the SCOPE-META `postMerge` flag and `worktreePath` field to resolve artifact paths correctly. Post-merge artifacts live in `.planning/post-merge/{setId}/`; pre-merge artifacts live in `.planning/sets/{setId}/`. Downstream agents must check both locations. -- Priority: **medium**

10. **Generate DEFINITION.md during init or start-set if missing**: Rather than failing with a warning, auto-generate a minimal DEFINITION.md from ROADMAP.md set descriptions. This removes a confusing manual step. -- Priority: **medium**

11. **Add `NO_COLOR` support to display.cjs**: Respect the `NO_COLOR` environment variable per the [clig.dev guidelines](https://clig.dev/) for terminal accessibility. When `NO_COLOR` is set, strip ANSI escape codes from output. -- Priority: **low**
