# Domain & UX Research: v3.1.0 Polish & Cleanup

**Domain:** AI-powered developer tool (Claude Code plugin for parallel development)
**Milestone:** v3.1.0 -- Polish & Cleanup
**Researched:** 2026-03-13
**Overall confidence:** HIGH (patterns well-established across CLI tools, state machine frameworks, and developer tool ecosystems)

---

## Domain Conventions

### 1. State Machine Status Naming: Past Tense Is the Industry Standard for Completed States

The core question for this milestone is whether to rename `-ing` suffixes (`discussing`, `planning`, `executing`) to past tense (`discussed`, `planned`, `executed`). Domain research strongly supports the rename.

**Industry precedent: past tense / adjective for completed states**

| Tool/Framework | State Names | Pattern |
|----------------|-------------|---------|
| Kubernetes Pod Phases | `Pending`, `Running`, `Succeeded`, `Failed`, `Unknown` | `-ing` ONLY for active work; past tense/adjective for terminal |
| GitHub Actions | `queued`, `in_progress`, `completed` | `-ing` ONLY for active; past tense for done |
| Terraform workflow | `Initialized`, `Planned`, `Applied`, `Destroyed` | Past participle throughout |
| Git | `staged`, `committed`, `pushed`, `merged` | Past tense exclusively |
| HTTP | `200 OK`, `201 Created`, `301 Moved Permanently` | Past tense for result states |
| Docker containers | `created`, `running`, `paused`, `exited`, `dead` | `-ing` only for active; past tense for rest |

**Source:** [State Names (educery.dev)](https://educery.dev/patterns/state-names/) recommends: "For simple operational FSMs, name the machine states using the past tense of the transitive verb for each state transition operation."

**Source:** [Stately.ai -- What's in a Name?](https://stately.ai/blog/2024-01-23-state-machines-whats-in-a-name) states: "States represent different conditions or modes of being. Therefore, they are often named with nouns or adjectives that describe the status or quality of the system."

**RAPID's specific problem:** The `-ing` suffix implies "currently in progress" but RAPID sets these states AFTER the phase completes. `discussing` is set at the END of discuss-set (Step 8), meaning it represents "has been discussed" -- a past-tense concept forced into present-progressive grammar. The plan-set skill then reads `discussing` as a precondition, interpreting it as "discussion is complete." This semantic mismatch creates cognitive friction for both developers reading status output and for agents parsing state.

**The rename aligns RAPID with every major developer tool in the ecosystem.** No competing tool uses `-ing` for completed states.

### 2. Deprecation Stub Removal: Clean Break Is Standard After One Major Version

**Industry convention:** Deprecation stubs serve a transition period. After one major version cycle, removal is expected.

**Source:** [Command Line Interface Guidelines (clig.dev)](https://clig.dev/) -- "Warn before you make a non-additive change. Eventually, you'll find that you can't avoid breaking an interface. Before you do, forewarn your users in the program itself." The key word is "before" -- stubs are for warning, not permanent fixtures.

**Competing tools' patterns:**
- **npm CLI:** Deprecated commands show warnings for 1-2 minor versions, then are removed entirely. `npm ls` behavior changes were deprecated with warnings, then the old behavior was dropped.
- **Docker CLI:** `docker attach` behavior was deprecated with messages pointing to `docker exec`, then the old semantics were removed.
- **Kubernetes kubectl:** Deprecated commands (e.g., `kubectl run` for creating deployments) showed deprecation notices, then the generator flags were removed.
- **Linux networking:** `ifconfig`, `netstat`, `route` were deprecated in favor of `ip`, `ss`, etc. Stubs remained for years in some distros but were eventually removed from default installs.

**RAPID's situation:** The 6 deprecated commands (`/rapid:plan`, `/rapid:wave-plan`, `/rapid:set-init`, `/rapid:discuss`, `/rapid:execute`, `/rapid:new-milestone`) have stubs that show redirect messages. These were deprecated in v3.0.0. By v3.1.0, users have had one full version cycle to learn the v3.0 command names. The stubs occupy space in the command listing, add noise to the plugin surface, and violate the "7+4 command set" progressive disclosure principle established in the v3.0 UX research.

**Recommendation:** Remove the stubs. The help command already documents the old-to-new mapping. Users who type a removed command will get Claude Code's standard "unknown command" behavior, which is more honest than a perpetual redirect message.

### 3. Version-Aware Installation: Self-Update Is Expected but Hard

**Industry patterns for version-aware CLI tools:**

| Tool | Version Detection | Update Mechanism |
|------|------------------|------------------|
| pnpm | `pnpm --version` vs latest | `pnpm self-update` command |
| yarn | `yarn --version` vs latest | `yarn set version stable` |
| Homebrew | `brew --version` + `brew update` | `brew upgrade` |
| Claude Code | Internal version tracking | Auto-updates on restart |
| npm | `npm --version` vs `npm view npm version` | `npm install -g npm@latest` |

**Source:** [pnpm self-update](https://pnpm.io/cli/self-update) -- pnpm provides `self-update` that "updates pnpm to the latest version or the one specified."

**RAPID's situation:** The install skill checks for `RAPID_TOOLS` env var presence but not version. A user with an old v3.0.0-beta pointing `RAPID_TOOLS` at a stale path would pass the presence check but use outdated tooling. The `package.json` already contains `"version": "3.0.0"` -- this is the version source of truth.

**Key UX expectation:** Users expect install commands to be idempotent and version-aware. Running `/rapid:install` on an already-installed system should detect the version mismatch and offer to update, not silently succeed with stale tools.

## Interaction Models

### Status Display: What Users Expect to See

Users of RAPID interact with set statuses in two places:

1. **`/rapid:status` dashboard** -- The primary status surface. Displays a table with set names and statuses.
2. **Error breadcrumbs** -- Inline progress indicators shown when a command fails.

**Current display with `-ing` names:**
```
| # | Set | Status | Last Activity |
|---|-----|--------|---------------|
| 1 | auth-system | discussing | 2h ago: "commit msg" |
| 2 | api-layer | planning | 1d ago: "wave plans" |
```

A developer reading this reasonably thinks: "auth-system is currently being discussed somewhere." But it is not. The discussion is complete and the set is waiting for `/rapid:plan-set`.

**Proposed display with past-tense names:**
```
| # | Set | Status | Last Activity |
|---|-----|--------|---------------|
| 1 | auth-system | discussed | 2h ago: "commit msg" |
| 2 | api-layer | planned | 1d ago: "wave plans" |
```

Now the status communicates: "auth-system has been discussed -- ready for planning." The past tense makes the "ready for next step" semantics obvious.

**Status-to-action mapping becomes more natural:**

| Current (Confusing) | Proposed (Clear) | Next Action |
|---------------------|------------------|-------------|
| `pending` | `pending` | `/rapid:start-set` |
| `discussing` (sounds in-progress) | `discussed` (sounds done) | `/rapid:plan-set` |
| `planning` (sounds in-progress) | `planned` (sounds done) | `/rapid:execute-set` |
| `executing` (sounds in-progress) | `executed` (sounds done) | `/rapid:review` |
| `complete` | `complete` | `/rapid:merge` |
| `merged` | `merged` | (done) |

### Parallel Wave Execution: Progress Feedback Conventions

If waves within a set execute concurrently, the progress display needs to change.

**Current sequential display:**
```
Executing wave 1 of 3 for set 'auth-system'...
Wave 1: complete (skipping)
Wave 2: pending (will execute)
Wave 3: pending (will execute)
```

**Expected parallel display (based on CLI conventions):**

**Source:** [Evil Martians -- CLI UX best practices: 3 patterns for improving progress displays](https://evilmartians.com/chronicles/cli-ux-best-practices-3-patterns-for-improving-progress-displays) -- "Progress bars are best suited when used for several lengthy, similar processes running in parallel."

**Source:** [bash-concurrent](https://github.com/themattrix/bash-concurrent) -- Shows how to display multiple concurrent tasks with per-task status indicators.

For RAPID's text-based Claude Code terminal context, the recommended pattern is a table with per-wave status that updates as waves complete:

```
Executing set 'auth-system' (3 waves: 2 parallel, 1 sequential)

Wave 1: executing (agent spawned)
Wave 2: executing (agent spawned)   [parallel with wave 1]
Wave 3: waiting (depends on wave 1)

--- Wave 1 complete (3 commits) ---
--- Wave 2 complete (2 commits) ---

Wave 3: executing (agent spawned)

--- Wave 3 complete (4 commits) ---
All 3 waves complete. Running verification...
```

**Key conventions for parallel progress:**
- Show which waves are parallel vs sequential and why
- Update status as each wave completes (not just at the end)
- Show dependency relationships ("depends on wave 1")
- Report per-wave commit counts so the user knows work happened

### Error Handling for Partial Parallel Completion

When waves run in parallel and one fails while another succeeds, users expect:

1. **Immediate notification** of the failure, not silent swallowing
2. **Preservation** of successfully completed parallel work
3. **Clear re-entry path** -- "Wave 2 failed. Wave 1 and 3 completed. Re-run to retry wave 2 only."
4. **No cascading failures** -- A failed wave should not undo completed parallel waves

This matches how GitHub Actions, CI systems, and build tools handle parallel job failures: independent jobs run to completion; dependent jobs are skipped; failed jobs are retried individually.

## Information Architecture

### Command Surface After Cleanup

With 6 deprecated stubs removed, the command listing becomes:

**Before (v3.0.0):** 7 core + 4 auxiliary + 5 utilities + 6 deprecated stubs + 1 meta = 23 total skills
**After (v3.1.0):** 7 core + 4 auxiliary + 5 utilities + 1 meta = 17 total skills

This further aligns with the v3.0 UX principle of progressive disclosure. The deprecated stubs violated the "7+4 command set" target by padding the visible command surface.

### Help Command Update

The help command's "Deprecated Commands" section (lines 95-106 of `skills/help/SKILL.md`) should be removed entirely when stubs are deleted. The migration guide served its purpose during the v3.0 transition. Keeping a "Deprecated" section after removal creates confusion -- users read about commands that no longer exist at all.

### Version Display

With version-aware installation, the status and help outputs should show the RAPID version. Users need to verify they are running the expected version.

**Recommended addition to `/rapid:status`:**
```
## v3.1.0 -- Set Dashboard
```

**Recommended addition to `/rapid:help` footer:**
```
RAPID v3.1.0 | 7+4 commands | Rapid Agentic Parallelizable and Isolatable Development
```

The version should be read from `package.json` at runtime, not hardcoded.

## Accessibility Considerations

### Terminal Output Readability

RAPID operates in a terminal-text environment (Claude Code). Key accessibility patterns:

- **Status names should be unambiguous when read aloud.** Past-tense names (`discussed`, `planned`, `executed`) are more distinct spoken aloud than present-progressive (`discussing`, `planning`, `executing`), which can be confused with active processes.
- **Progress breadcrumbs** use `[done]` and `[FAILED]` markers that work with screen readers. No changes needed.
- **ANSI color codes** in banners (display.cjs) use 16-color palette for maximum terminal compatibility. The banner content is readable without color. No changes needed.

### Cognitive Load Reduction

- Removing 6 deprecated stubs reduces command-surface cognitive load by ~26% (from 23 to 17 visible skills)
- Past-tense status names eliminate the need to mentally translate "`discussing` means discussed" -- a small but repeated cognitive tax

## User Expectations

### What Users Will Assume About Status Names

Based on how every other developer tool names states:

1. **`discussing` = currently in progress.** Users will look for the active agent or wonder why discussion seems stuck. This is the wrong mental model -- the discussion is finished.
2. **`discussed` = complete, ready for next step.** Users immediately understand no further action is needed for this phase.
3. **Consistency with git terminology.** Git uses `staged`, `committed`, `merged` -- all past tense. RAPID uses `merged` already. `discussed`, `planned`, `executed` match.

### What Users Will Assume About Deprecated Commands

1. **If a deprecated command exists, it might still work.** Users see the stub in the command list and try it. The redirect message helps but adds friction.
2. **If a deprecated command does not exist, it was never there.** Users only learn the current commands.
3. **After one version cycle, stubs are cruft.** Any user who adopted v3.0 already learned the new names. New users never knew the old names.

### What Users Will Assume About Install

1. **Running install again should fix things.** If `RAPID_TOOLS` points to a stale version, `/rapid:install` should detect and fix this.
2. **The tool should tell me if I am outdated.** Users expect version-aware tools to proactively warn about version mismatches.
3. **The update should preserve my config.** Shell config lines should be updated in-place, not duplicated.

### What Users Will Assume About Parallel Execution

1. **Parallel is faster but not always applicable.** Users understand that not all waves can run in parallel. They expect the tool to make this determination, not ask them.
2. **Failures in one parallel lane don't cancel others.** This matches CI/CD expectations (GitHub Actions `fail-fast: false`).
3. **Re-entry after failure resumes only failed work.** Artifact-based re-entry (WAVE-COMPLETE.md markers) already handles this pattern. Users expect it to work the same with parallel waves.

## Recommendations

Ordered by priority for the v3.1.0 milestone:

1. **Rename `-ing` statuses to past tense:** `discussing` -> `discussed`, `planning` -> `planned`, `executing` -> `executed`. This is a cross-cutting rename (state-schemas.cjs, state-transitions.cjs, all skills, agents, CLI, tests) but the scope is well-defined and the semantic improvement is significant. -- **Priority: critical**
   - Rationale: Every major developer tool uses past tense for completed states. RAPID's current naming contradicts universal convention and creates cognitive friction at every status display.
   - Source: [Kubernetes Pod Lifecycle](https://kubernetes.io/docs/concepts/workloads/pods/pod-lifecycle/), [Terraform state machine](https://mattias.engineer/blog/2025/terraform-state-machine/), [educery.dev State Names](https://educery.dev/patterns/state-names/)

2. **Remove 6 deprecated command stubs:** Delete `skills/plan/`, `skills/wave-plan/`, `skills/set-init/`, `skills/discuss/`, `skills/execute/`, `skills/new-milestone/` directories. Remove the "Deprecated Commands" section from `skills/help/SKILL.md`. -- **Priority: high**
   - Rationale: One full version cycle (v3.0.0) has passed. Stubs add noise to the command surface and violate the 7+4 progressive disclosure target. [clig.dev](https://clig.dev/) establishes that deprecation warnings are transitional, not permanent.

3. **Make install version-aware:** Read version from `package.json`, compare against the version embedded at the `RAPID_TOOLS` path, and offer reinstall/update when mismatched. -- **Priority: high**
   - Rationale: Every modern CLI tool (pnpm, yarn, brew) detects version mismatches on install. RAPID currently only checks for presence, not correctness. A stale `RAPID_TOOLS` path silently uses outdated code.
   - Source: [pnpm self-update](https://pnpm.io/cli/self-update), [yarn self-update](https://classic.yarnpkg.com/lang/en/docs/cli/self-update/)

4. **Design parallel wave execution progress display:** When implementing parallel wave dispatch, show per-wave status with dependency awareness. Use the table format described in the Interaction Models section. -- **Priority: medium**
   - Rationale: Parallel execution without visible per-wave progress violates the "transparent agent activity" table stake from v3.0 UX research. Users need to see which waves are running, which are waiting, and which have completed.
   - Source: [Evil Martians CLI UX](https://evilmartians.com/chronicles/cli-ux-best-practices-3-patterns-for-improving-progress-displays)

5. **Handle partial parallel failure gracefully:** When a parallel wave fails, complete other running waves, mark the failed wave for retry, and give a clear re-entry message. -- **Priority: medium**
   - Rationale: Users of CI/CD systems expect independent parallel jobs to complete independently. Cascading failures from one lane to another is an anti-pattern.

6. **Update help command to remove deprecated section:** After stub removal, the help command should only show current commands. The old-to-new mapping is historical context, not actionable information. -- **Priority: medium**
   - Rationale: [clig.dev](https://clig.dev/) recommends static, predictable help output. Referencing removed commands in help creates confusion.

7. **Show RAPID version in status/help output:** Read version from `package.json` at runtime and display in the status dashboard header and help footer. -- **Priority: low**
   - Rationale: Users need to verify they are running the expected version, especially after the version-aware install feature is added. Hardcoded version strings in help output become stale.

8. **Update display.cjs STAGE_VERBS if needed:** If any banner stage names reference the old statuses, update the verb mappings. Current verbs (`DISCUSSING`, `PLANNING`, `EXECUTING`) are action verbs for banners (what is happening now), not status names -- these may actually be correct as-is since banners describe the current action, not the resulting state. -- **Priority: low**
   - Rationale: Banners describe the action being performed ("DISCUSSING SET") which is present progressive by nature. The rename only affects status values, not banner verbs. Verify during implementation.
