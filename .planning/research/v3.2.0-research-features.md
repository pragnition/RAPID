# Features Research -- v3.2.0 General Fixes

## Feature Inventory

### Feature 1: Set Name Resolution Bug (Archival Desync)

- **Description:** When previous planning documents are archived (via `/rapid:new-version`), `resolveSet()` in `resolve.cjs` resolves numeric indices against `plan.listSets()` which reads `.planning/sets/` on disk. After archiving sets 4,5 and adding 6,7,8,9, the on-disk set directories no longer match the sets in STATE.json, causing numeric index 6 to resolve to set 8.
- **Complexity:** moderate
- **Dependencies:** independent

#### Root Cause Analysis

`resolveSet()` (src/lib/resolve.cjs:19-53) calls `plan.listSets(cwd)` which reads `.planning/sets/` directory entries alphabetically (src/lib/plan.cjs:169-181). After archiving, the archived set directories are moved to `.planning/archive/{milestone}/sets/`, shrinking the on-disk list. STATE.json still tracks the full set history across milestones. The numeric index maps against the on-disk directory list, not the state-based set list.

Example scenario:
- Before archival: `.planning/sets/` has [set-1, set-2, set-3, set-4, set-5] -- index 4 = set-4
- After archiving 4,5 and adding 6,7,8,9: `.planning/sets/` has [set-1, set-2, set-3, set-6, set-7, set-8, set-9]
- Now index 6 = set-8 (should resolve to set-6)

#### Decomposition

1. **State-aware set listing** -- Modify `resolveSet()` to resolve numeric indices against STATE.json's current milestone sets rather than on-disk directories
2. **Fallback compatibility** -- Keep disk-based listing as fallback when STATE.json is unavailable (e.g., pre-init)
3. **Tests** -- Add test cases covering archival scenarios (sets removed from disk but present in state, new sets added after archival)

#### Implementation Strategies

**Option 1: State-First Resolution**
- Approach: Modify `resolveSet()` to accept an optional `state` parameter. When provided, resolve against `state.milestones[current].sets[].id` instead of `plan.listSets()`. Update all callers in skills that already have state loaded to pass it.
- Pros: Clean separation of concerns; state is the source of truth for set identity
- Cons: Requires updating all callers; `resolveSet()` currently has no state dependency
- Libraries/services: None (internal refactor)

**Option 2: Hybrid Resolution (Recommended)**
- Approach: Modify `resolveSet()` to first try STATE.json's current milestone sets. If STATE.json is missing or invalid, fall back to `plan.listSets()` disk listing. Internally, `resolveSet()` reads state via `readState()` or accepts a pre-loaded state object.
- Pros: Backward compatible; handles pre-init and post-archival scenarios; single function signature change
- Cons: Adds a readState call inside resolve (minor perf cost)
- Libraries/services: None (uses existing `state-machine.cjs:readState`)

**Recommended:** Option 2 -- The hybrid approach is self-healing and requires no caller changes. When state is available, resolution is always correct against the current milestone's sets. When state is missing (early init), disk fallback works as before.

#### Data Model Impact
- No new entities
- No schema changes
- `resolveSet()` function signature gains optional `state` parameter

#### API Surface
- Modified: `resolve set <input>` CLI command (internally uses state-first resolution)
- Modified: `resolveSet(input, cwd, [state])` library function

### Feature 2: Batched Questions UX (Discuss Phase)

- **Description:** During the discuss phase, agents sometimes ask multiple freeform questions without pre-filled options in a single AskUserQuestion call. The user experience should use AskUserQuestion's native multi-header batching where each header = one question with pre-filled options.
- **Complexity:** moderate
- **Dependencies:** independent

#### Root Cause Analysis

The discuss-set SKILL.md Step 6 instructs agents to batch 2-3 questions per gray area into a single freeform AskUserQuestion. The issue is twofold:

1. The skill's Step 6 template says "Answer all questions above, or type 'Claude decides'" -- this is a freeform prompt without structured options.
2. AskUserQuestion supports multi-header mode where each header gets its own set of options, but the skill template does not use this format.

Additionally, the skill Step 5 properly uses structured options (with "Let Claude decide all" and per-area selections), but Step 6's per-area deep-dive lacks structured options.

#### Decomposition

1. **Restructure Step 6 template** -- Convert the freeform multi-question AskUserQuestion into batched individual AskUserQuestion calls, each with pre-filled options
2. **Add "I'll answer" option** -- Every AskUserQuestion must include an "I'll answer" freeform option alongside pre-filled choices
3. **Update Step 5 template** -- Ensure the gray area selection also follows the multi-header pattern with options
4. **Document the AskUserQuestion pattern** -- Add a conventions note about always providing pre-filled options

#### Implementation Strategies

**Option 1: One AskUserQuestion Per Sub-Question (Recommended)**
- Approach: Replace the batched 2-3 questions per gray area with individual AskUserQuestion calls. Each call has: a clear question header, 2-4 pre-filled option selections, and an "I'll answer" freeform option. Process sequentially per gray area.
- Pros: Clean UX; user navigates one question at a time with clear options; matches the "never show Claude decides without options" requirement
- Cons: More round-trips with the user (4 areas x 2-3 questions = 8-12 questions total)
- Libraries/services: Native AskUserQuestion tool

**Option 2: Multi-Header AskUserQuestion**
- Approach: Use AskUserQuestion's native multi-header batching where the `question` param contains markdown with `###` headers for each sub-question, and options are provided as a combined list.
- Pros: Fewer round-trips; all questions visible at once
- Cons: AskUserQuestion multi-header mode may not be well-supported; option disambiguation becomes harder with many options
- Libraries/services: Native AskUserQuestion tool

**Recommended:** Option 1 -- Sequential individual AskUserQuestion calls per question. This is clearer for the user and aligns with the "never overwhelm" design principle. The discuss phase already takes 4 areas; adding 2-3 questions per area is manageable since each is quick with pre-filled options.

#### Data Model Impact
- No data model changes
- CONTEXT.md output format unchanged

#### API Surface
- Modified: `skills/discuss-set/SKILL.md` (Step 5 and Step 6 templates)
- No CLI changes

### Feature 3: Subagent Banner Colours

- **Description:** The RAPID stage banner background colour is pink/bright blue. It should be changed to dark purple for better readability with white text.
- **Complexity:** trivial
- **Dependencies:** independent

#### Root Cause Analysis

The `display.cjs` STAGE_BG map uses ANSI bright blue (`\x1b[104m`) for planning stages, bright green (`\x1b[102m`) for execution stages, and bright red (`\x1b[101m`) for review stages. The user reports this looks bad, particularly the bright blue rendering as pinkish in some terminals, and wants dark purple instead.

ANSI 256-color or RGB escape codes can produce a proper dark purple. The 16-color palette does not have a purple option (closest is bright magenta `\x1b[105m`). For true dark purple, use 256-color mode: `\x1b[48;5;53m` (dark purple) or `\x1b[48;5;54m` (slightly lighter).

#### Decomposition

1. **Change STAGE_BG colours** -- Update the ANSI escape codes in `display.cjs` to use dark purple (256-color)
2. **Update tests** -- Update `display.test.cjs` assertions that check for specific ANSI codes
3. **Verify terminal compatibility** -- 256-color mode is supported by all modern terminals (iTerm2, Terminal.app, Windows Terminal, Alacritty, Kitty, most Linux terminals)

#### Implementation Strategies

**Option 1: 256-Color Dark Purple (Recommended)**
- Approach: Replace the 16-color bright backgrounds with 256-color dark purple: `\x1b[48;5;53m` for planning stages, `\x1b[48;5;22m` (dark green) for execution, `\x1b[48;5;88m` (dark red) for review. This gives readable contrast with white text.
- Pros: True dark purple; good contrast with white text; widely supported
- Cons: Not supported in extremely old terminals (pre-2010)
- Libraries/services: None (raw ANSI codes)

**Option 2: 16-Color Bright Magenta**
- Approach: Use `\x1b[105m` (bright magenta background) as the closest 16-color purple.
- Pros: Maximum terminal compatibility
- Cons: Bright magenta is not dark purple; may still look bad with white text

**Recommended:** Option 1 -- 256-color mode is universally supported in modern terminals. Dark purple (`\x1b[48;5;53m`) provides the requested look.

#### Data Model Impact
- None

#### API Surface
- Modified: `display banner <stage>` output changes colour
- Modified: `src/lib/display.cjs` STAGE_BG map
- Modified: `src/lib/display.test.cjs` test assertions

### Feature 4: Never Show "Claude Decides" Without Options

- **Description:** AskUserQuestion should never show a plain "Claude decides" option with only "I'll answer" and "Type something" as alternatives. Every question must have pre-filled answer options plus an "I'll answer" escape hatch.
- **Complexity:** moderate
- **Dependencies:** depends on Feature 2 (discuss-set UX improvements)

#### Root Cause Analysis

The discuss-set SKILL.md Step 6 has a pattern where the agent presents "Claude decides" alongside "I'll answer" and "Type something". The user sees:

```
1. Claude decides
2. I'll answer
3. Type something.
4. Chat about this
```

This happens because the SKILL.md template does not mandate specific answer options for each question. The agent falls back to these generic meta-options when it lacks concrete pre-filled choices.

#### Decomposition

1. **Mandate concrete options in SKILL.md** -- Every AskUserQuestion in discuss-set (and all other skills) must list 2-4 concrete answer options
2. **Add "I'll answer" as escape hatch** -- The last option in every AskUserQuestion should be "I'll answer" with freeform description
3. **Audit all skills** -- Check all 24 skills for AskUserQuestion calls that lack pre-filled options
4. **Add convention rule** -- Document in core-conventions.md that all AskUserQuestion calls require pre-filled options

#### Implementation Strategies

**Option 1: Template Enforcement in SKILL.md (Recommended)**
- Approach: Rewrite every AskUserQuestion call in discuss-set SKILL.md to include concrete option examples. Add "I'll answer -- I'll provide my own input" as the final option in every question. Add a convention rule in `src/modules/core/core-conventions.md` that all AskUserQuestion calls must have pre-filled options.
- Pros: Fixes the root cause at the template level; convention rule prevents regression
- Cons: Requires auditing all 24 skills
- Libraries/services: None

**Recommended:** Option 1 -- Template-level enforcement is the only reliable fix since agents follow SKILL.md instructions.

#### Data Model Impact
- None

#### API Surface
- Modified: `skills/discuss-set/SKILL.md` (all AskUserQuestion calls get concrete options)
- Modified: `src/modules/core/core-conventions.md` (new convention rule)

### Feature 6: Review After Merge Support

- **Description:** Users may want to run the review pipeline after merging (on the main branch) rather than after each individual set completes. Currently, review only works on set branches before merge.
- **Complexity:** complex
- **Dependencies:** independent

#### Root Cause Analysis

The review pipeline (`/rapid:review`) is currently designed to run on a set's worktree branch, scoping changed files via `git diff main..rapid/{setName}`. After merge, the set's branch is already integrated into main, so `git diff main..rapid/{setName}` shows nothing (they are the same).

The review skill (Step 0c) validates that the set status is `complete` or `reviewing`. After merge, the status is `merged`, which the skill rejects.

#### Decomposition

1. **Post-merge review scope** -- Create a new scope mode that diffs against the merge commit's parent(s) to identify the set's changes on main
2. **Review skill status gate** -- Allow `merged` status as a valid review entry point with a different scope strategy
3. **Review-after-merge milestone option** -- In the merge skill, add an option to queue review for after merge instead of requiring it before
4. **Milestone-level review** -- Support reviewing all sets in a milestone at once (aggregate scope)

#### Implementation Strategies

**Option 1: Post-Merge Review via Merge Commit Ancestry (Recommended)**
- Approach: Add a `--post-merge` flag to the review skill. When set, the scope command uses `git log --merges --grep="rapid/{setName}"` to find the merge commit, then diffs against its first parent (`git diff <merge-commit>^...<merge-commit>`) to recover the set's changes. The review skill accepts `merged` status when `--post-merge` is active. The merge skill offers a "Review after merge" option alongside "Merge now".
- Pros: Works on the integrated codebase (catches integration bugs the pre-merge review would miss); reuses existing review infrastructure
- Cons: Merge commit must be identifiable (convention: use `--no-ff` which is already enforced); review findings cannot be fixed in an isolated branch
- Libraries/services: Git (merge commit ancestry)

**Option 2: Skip-Review-Before-Merge Flag**
- Approach: Add a `--skip-review` flag to the merge skill that allows merging without prior review. After merge completes, prompt the user to run `/rapid:review --post-merge`. Simpler scope: just use the merge commit diff.
- Pros: Simpler implementation; user explicitly opts in
- Cons: Less automated; user might forget to review

**Recommended:** Option 1 -- Full post-merge review support with automatic scope detection. The `--post-merge` flag is clean and reuses existing pipeline stages.

#### Data Model Impact
- No schema changes needed
- Review artifacts would be written to `.planning/waves/{setId}/` as before (or a new `.planning/review/{setId}/` path)

#### API Surface
- Modified: `review scope <set-id>` gains `--post-merge` flag
- Modified: `skills/review/SKILL.md` accepts `merged` status with `--post-merge`
- Modified: `skills/merge/SKILL.md` offers review-after-merge option
- Modified: `src/lib/review.cjs` scopeSetForReview gains post-merge mode

### Feature 7: State Update Enforcement

- **Description:** Agents MUST update state (STATE.json) after merging, executing, and discussing. Currently, agents sometimes skip state transitions, leaving STATE.json stale.
- **Complexity:** moderate
- **Dependencies:** depends on Feature 8 (state transition fixes)

#### Root Cause Analysis

The state transition calls in skills are often: (a) at the end of multi-step processes where agent context may have been exhausted, (b) silently suppressed with `2>/dev/null || true` (discuss-set SKILL.md line 268), or (c) not wired at all in certain paths.

Key gaps identified:
- discuss-set: transitions to "discussing" but the valid transition map expects "discussed" (see Feature 8)
- execute-set: Step 3 transitions to "executing" but the schema expects "executed"
- The transition errors cause agents to proceed without the state being updated
- No post-transition verification step exists (agent does not re-read state to confirm)

#### Decomposition

1. **Add state verification after every transition** -- Each skill's transition step should verify the transition succeeded by re-reading state
2. **Remove silent error suppression** -- Change `2>/dev/null || true` to proper error handling
3. **Add transition as MANDATORY step** -- Mark state transitions as non-optional in skill templates
4. **Add post-execution state commit** -- Ensure STATE.json is always committed alongside artifacts
5. **Add state-check CLI command** -- A `state check <setId> <expected-status>` command that verifies current status matches expected

#### Implementation Strategies

**Option 1: Verification-After-Transition Pattern (Recommended)**
- Approach: After every `state transition set` call in skills, add a verification step that re-reads state and asserts the expected status. Remove all `|| true` suppression. Add a `state check` CLI subcommand. Update all skills to use this pattern:
  ```bash
  node "${RAPID_TOOLS}" state transition set "${MILESTONE}" "${SET_ID}" <target>
  node "${RAPID_TOOLS}" state check "${MILESTONE}" "${SET_ID}" <target>
  ```
- Pros: Guarantees state consistency; fails fast on transition errors
- Cons: One extra CLI call per transition
- Libraries/services: None (new CLI subcommand)

**Recommended:** Option 1.

#### Data Model Impact
- No schema changes

#### API Surface
- New: `state check <milestoneId> <setId> <expectedStatus>` CLI subcommand
- Modified: All skill SKILL.md files (add verification after transitions)

### Feature 8: Invalid State Transitions Fix

- **Description:** Two specific invalid transition errors need fixing: (a) "discussing -> complete" error during merge, and (b) unknown "wave-plan" command. The root cause is a status naming mismatch between skills and the code.
- **Complexity:** complex
- **Dependencies:** independent (but Features 2, 4, 7 depend on this being fixed first)

#### Root Cause Analysis

**Critical mismatch discovered:** The codebase has a status naming inconsistency between the Zod schemas/transitions and the skill templates:

| Code (state-schemas.cjs, state-transitions.cjs) | Skills (SKILL.md files) |
|---|---|
| `discussed` | `discussing` |
| `planned` | `planning` |
| `executed` | `executing` |

The `SetStatus` enum in `state-schemas.cjs` line 4 is:
```javascript
z.enum(['pending', 'discussed', 'planned', 'executed', 'complete', 'merged'])
```

The `SET_TRANSITIONS` in `state-transitions.cjs` are:
```javascript
pending -> ['discussed', 'planned']
discussed -> ['planned']
planned -> ['executed']
executed -> ['complete']
complete -> ['merged']
```

But the skills use progressive tense:
- discuss-set SKILL.md line 268: `transition set ... discussing`
- plan-set SKILL.md line 297: `transition set ... planning`
- execute-set SKILL.md line 148: `transition set ... executing`

The `status-rename.test.cjs` was created to detect old progressive-tense status literals in `.cjs` source files, confirming a rename from progressive to past tense was done in the code. However, the skills (Markdown files) were never updated.

**The "discussing -> complete" error during merge:** A set gets stuck in "discussing" status (from the Zod schema's perspective, "discussed") because the skill writes the wrong literal. When merge later tries to check status, it finds an invalid state.

Wait -- actually, re-reading the schemas, `SetStatus` uses `['pending', 'discussed', 'planned', 'executed', 'complete', 'merged']`. The discuss-set skill tries `state transition set ... discussing`. The transition handler reads the current status ("pending"), looks up valid transitions: `['discussed', 'planned']`, and the target "discussing" does not match "discussed". So the transition FAILS, but the `2>/dev/null || true` suppresses the error. The set stays at "pending", but CONTEXT.md is written. Later, other skills find the set in unexpected states.

**The "wave-plan" error:** Covered in Feature 5 above. The plan-set SKILL.md calls `wave-plan validate-contracts` which is a removed v2 command.

Additionally, the review SKILL.md references a "reviewing" status (line 80: `transition set ... reviewing`) which does not exist in the SetStatus enum at all. The valid status after "complete" is only "merged".

#### Decomposition

1. **Fix skill status literals** -- Update all 24 skills to use past-tense status names matching the code: `discussed`, `planned`, `executed`
2. **Fix review skill's "reviewing" status** -- The review pipeline needs a design decision: either add "reviewing" to the SetStatus enum, or have review operate without a state transition (just check that status is "complete")
3. **Fix status references in agent modules** -- Update `src/modules/roles/*.md` and `agents/*.md` for correct status names
4. **Fix status references in display/UI** -- Update `skills/status/SKILL.md` and `skills/new-version/SKILL.md` references
5. **Fix worktree.cjs "reviewing" case** -- Line 683 and 811 reference `reviewing` status that does not exist

#### Implementation Strategies

**Option 1: Update Skills to Match Code (Recommended)**
- Approach: The code (schemas, transitions) is the source of truth since a deliberate rename was already done and `status-rename.test.cjs` enforces it. Update all skills and agent modules to use the past-tense status names. For the "reviewing" status, decide: (a) add it to the enum and transitions, or (b) remove it from the review skill.
- Pros: Minimal code changes (skills are Markdown, not executable code); aligns with the deliberate rename
- Cons: Large number of Markdown files to update (24 skills + 26 agents)
- Libraries/services: None

**Sub-decision: "reviewing" status**
- **Option A: Add "reviewing" to SetStatus and transitions.** Add `reviewing` between `complete` and `merged`, with transition `complete -> ['reviewing', 'merged']` and `reviewing -> ['merged']`. This is clean and explicit.
- **Option B: Remove "reviewing" from review skill.** Review operates on sets with status "complete" without transitioning. Simpler but loses the ability to track review-in-progress.

**Recommended:** Option 1 with Sub-decision Option A. Adding "reviewing" gives the review pipeline its own trackable state and matches the skill's existing design intent.

#### Data Model Impact
- Modified: `state-schemas.cjs` SetStatus enum -- add `reviewing` between `complete` and `merged`
- Modified: `state-transitions.cjs` SET_TRANSITIONS -- add `reviewing` key
- Modified: `state-transitions.test.cjs` -- new transition tests

#### API Surface
- Modified: All 24 skill SKILL.md files (status literal fixes)
- Modified: 26 agent .md files (rebuilt from modules)
- Modified: `src/lib/state-schemas.cjs` (new reviewing status)
- Modified: `src/lib/state-transitions.cjs` (new transition path)
- Modified: `src/lib/worktree.cjs` (reviewing case already exists, now valid)
