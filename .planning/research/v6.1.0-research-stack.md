## Zod 3 to Zod 4 Migration Analysis

**This is the most significant stack finding for this milestone.** Zod 4 is a major release with substantial breaking changes. RAPID uses Zod in 5 library files for state validation. Here is a detailed impact assessment.

### Affected Files

| File | Zod APIs Used | Breaking Changes Hit |
|------|--------------|---------------------|
| `src/lib/state-schemas.cjs` | z.enum, z.object, z.string, z.number, z.array, .default(), .optional(), **.passthrough()** | `.passthrough()` deprecated |
| `src/lib/hooks.cjs` | z.object, z.string, z.number, z.boolean, z.array, .safeParse() | None identified |
| `src/lib/returns.cjs` | z.object, z.string, z.number, z.boolean, z.array, z.enum, z.union, z.literal, .safeParse() | None identified |
| `src/lib/merge.cjs` | z.object, z.string, z.number, z.boolean, z.array, z.enum, z.record(), .optional(), .default(), .parse() | z.record() -- OK (2-arg form used) |
| `src/lib/review.cjs` | z.object, z.string, z.number, z.boolean, z.array, z.enum, z.record(), .optional(), .default(), .parse() | z.record() -- OK (2-arg form used) |

### Critical Breaking Change: `.passthrough()`

`state-schemas.cjs` uses `.passthrough()` on **all 5 core schemas** (JobState, WaveState, SetState, MilestoneState, ProjectState). In Zod 4, `.passthrough()` is deprecated -- the replacement is `z.looseObject()` or the default behavior change.

**In Zod 4:** `z.object()` strips unknown keys by default (same as Zod 3). To preserve unknown keys, you must use `z.looseObject()` instead of `z.object({...}).passthrough()`.

**Migration path:**
```javascript
// Zod 3 (current)
const SetState = z.object({ id: z.string(), ... }).passthrough();

// Zod 4
const SetState = z.looseObject({ id: z.string(), ... });
```

This is a mechanical change across 5 schema definitions.

### Other Zod 4 Breaking Changes -- NOT Affecting RAPID

| Breaking Change | Impact on RAPID |
|----------------|----------------|
| `{ message: }` -> `{ error: }` in validators | Not used -- RAPID does not pass custom error messages to Zod validators |
| `z.nativeEnum()` deprecated | Not used |
| `z.record()` single-arg removed | Not used -- all usages are 2-arg form: `z.record(z.string(), z.string())` |
| `.deepPartial()` removed | Not used |
| `.merge()` deprecated | Not used |
| `.format()` / `.flatten()` deprecated | Not used |
| `z.function()` restructured | Not used |
| `z.promise()` deprecated | Not used |
| `z.ostring()` / `z.onumber()` removed | Not used |
| `.refine()` type narrowing removed | Not used |
| `ZodError.addIssue()` deprecated | Not used |
| `.strict()` deprecated | Not used |
| `z.any()` / `z.unknown()` no longer optional in objects | Not used as object properties |

### CommonJS Compatibility Concern

Zod 4 has a **4x bundle size increase for CommonJS** compared to v3 (68KB -> 254-288KB unbundled). For RAPID this is acceptable since:
- RAPID is a CLI plugin, not a browser bundle
- Cold-start time is not a constraint (CLI runs are infrequent, not Lambda-scale)
- The `require('zod')` import path continues to work in Zod 4

There were early CJS loading bugs in Zod 4 betas, but the stable 4.x release has resolved these.

### Upgrade Recommendation

**Do NOT upgrade Zod in this milestone.** Rationale:
1. The v6.1.0 milestone is focused on UX/onboarding, not internal refactoring
2. Zod 3.25.76 has zero CVEs and continues to work correctly
3. The migration is mechanical (5 `.passthrough()` changes) but requires test validation across all schema consumers
4. Recommend deferring to a future "hygiene" or "dependency upgrade" set

**Priority: low** -- No security concern, no functional gap. Track for next milestone.

### proper-lockfile (Priority: low)

| Field | Value |
|-------|-------|
| Current | 4.1.2 (last published 2021) |
| Target | No newer version available |
| Alternatives | async-lock, lockfile (both less feature-complete) |
| Migration effort | Significant -- core concurrency primitive |

The package is dormant but functional. Its `mkdir` atomicity strategy is well-suited for RAPID's file-based locking. Alternatives like `async-lock` are in-process only (no cross-process locking), which does not meet RAPID's multi-agent concurrency requirements. No action needed unless a CVE is discovered.

### execSync -> execFileSync Migration (Priority: medium, deferred)

| Field | Value |
|-------|-------|
| Current state | 346 execSync occurrences across 22 files; 28 execFileSync in 5 files |
| Security concern | execSync invokes a shell, enabling command injection if inputs are not sanitized |
| Migration effort | Significant -- audit each call site for shell features (pipes, globbing, env expansion) |

This is explicitly listed as a deferred decision from the bug-fixes-foundation set. The audit-to-set handoff mechanism being built in this milestone could generate a remediation set for this work. Not in scope for v6.1.0 implementation.

## Milestone-Specific Stack Notes

### Audit-to-Set Handoff Feature

The current stack fully supports this feature. Key observations:
- `DEFERRED.md` files already exist as artifacts from the audit pipeline (7 files found in v6.0.0 archive)
- `addMilestone()` in state-machine.cjs already accepts `carryForwardSets` parameter
- The `add-set` skill reads state but has no mechanism to ingest deferred context
- Implementation requires: a machine-readable deferral index file (JSON) and a read path in the add-set discovery flow
- No new dependencies needed

### Clear-Before-Next-Command Guidance

Pure UX/display concern. The display.cjs module handles banner output and could be extended with a footer/guidance line. No stack implications.

### README Improvements

Stack implication: version badges, install instructions, and prerequisite documentation all need updating for Node.js >=22. The setup.sh script's prerequisite check (Step 1) needs the version threshold raised.

### General UX Audit

No stack-level blockers. The display system (display.cjs, 16-color ANSI with NO_COLOR support) is adequate. The skill/command output format is text-based and does not require additional dependencies.
