# Research Summary

## Project Profile
- **Type:** brownfield CLI plugin (Claude Code skill-based development orchestration)
- **Stack:** Node.js (CommonJS .cjs) + Zod 3 + ajv 8 + proper-lockfile + Git worktrees
- **Scale:** small (4 features: 1 new command, 2 UX refinements, 1 bug fix)
- **Key Constraint:** All changes are specification-level (SKILL.md) or thin library additions; the existing skill/agent/CLI/library architecture fully supports every item without structural changes

## Unified Findings

### Stack and Infrastructure
- Node.js 25.6.1 (Current) detected; project claims Node 18+ minimum but Node 18 is EOL. Practical minimum is Node 22+ (sources: STACK, OVERSIGHTS)
- All dependencies (zod 3.25.76, ajv 8.18.0, ajv-formats 3.0.1, proper-lockfile 4.1.2) are current or stable with zero CVEs (source: STACK)
- Zod 4 is available as `require('zod/v4')` in the same package but should NOT be upgraded in this milestone -- state-schemas.cjs is load-bearing infrastructure (source: STACK)
- proper-lockfile is unmaintained (5+ years) but functionally complete; the project's `lock.cjs` wrapper provides an abstraction layer for future replacement (source: STACK)
- No CI/CD pipeline, no linter/formatter configured; tests run locally via `node --test` (source: STACK)
- CommonJS throughout; no ESM migration pressure; all dependencies export CJS-compatible modules (source: STACK)

### Feature Landscape

**Feature groups with dependencies:**

All four items are fully independent with no cross-dependencies. They touch different files and can be implemented in any order or in parallel (source: FEATURES).

1. **Bug Fix: context.cjs path resolution** -- Trivial fix affecting 4 lines across 2 SKILL.md files. Fixes `require('${RAPID_TOOLS}/../lib/...')` pattern where Node treats the .cjs filename as a directory component (sources: FEATURES, STACK, PITFALLS, OVERSIGHTS)
2. **Discuss Table Formatting** -- SKILL.md Step 6 Format A template rewrite. Replace markdown tables with structured list format for reliable rendering (source: FEATURES)
3. **Discuss Gray Areas Consolidation** -- SKILL.md Step 5 rewrite. Present all gray areas in a single multiSelect AskUserQuestion instead of multiple batched prompts (sources: FEATURES, OVERSIGHTS)
4. **Audit Version Command** -- New skill (`/rapid:audit-version`) with optional thin library module. Reads milestone artifacts, cross-references requirements vs. delivery, produces gap report, offers remediation (sources: FEATURES, ARCHITECTURE, UX)

**Recommended implementation order:**
1. Path resolution bug fix (unblocks init; trivial effort)
2. Discuss table formatting + gray areas consolidation (both modify same SKILL.md; moderate effort)
3. Audit version command (most complex; additive feature, no blockers)

**Third-party service dependencies:** None. All features are internal tooling changes (source: FEATURES).

### Architecture Direction

- **Recommended pattern:** Extend existing modular skill architecture; no new architectural pattern needed (source: ARCHITECTURE)
- **Module boundaries:** New audit-version gets: `skills/audit-version/SKILL.md`, `src/lib/audit.cjs` (thin), `src/commands/audit.cjs`, `src/modules/roles/role-auditor.md`, display.cjs banner entry. Discuss changes are SKILL.md-only. Path fix is SKILL.md-only (source: ARCHITECTURE)
- **Data flow:** Audit-version is read-only against state; produces `.planning/v{version}-AUDIT.md` artifact. Does not mutate STATE.json. Remediation uses existing `/rapid:add-set` (sources: ARCHITECTURE, FEATURES)
- **State management:** No new state transitions needed. Milestone completion derived from `sets.every(s => s.status === 'merged')` rather than adding a milestone-level status field. This avoids schema migration complexity (sources: ARCHITECTURE, PITFALLS, OVERSIGHTS)
- **Discuss changes are specification-only:** Gray area consolidation and table formatting modify SKILL.md instructions, not executable code. The `discussed` state transition is unchanged (source: ARCHITECTURE)

### User Experience Direction

**Domain conventions and standard terminology:**
- RAPID operates in the AI-assisted development orchestration domain. Audit-version maps to industry concepts of "retrospective" and "gap analysis" (source: UX)
- Users say "what did we miss" (gap analysis), "defer to next version" (carry-forward), "is everything done" (milestone audit), "the table looks weird" (alignment issue) (source: UX)
- Industry-standard audit pattern: automated scan -> structured findings -> cross-reference validation -> actionable routing -> user confirmation gate (source: UX)

**Interaction models and user expectations:**
- Audit should be non-destructive: read-only analysis followed by action prompts with explicit user confirmation (sources: UX, ARCHITECTURE)
- Audit results must be persistent: saved to `.planning/v{version}-AUDIT.md` for cross-session reference (source: UX)
- Three outcome paths: (a) all passed -> offer `/rapid:new-version`, (b) gaps found -> offer remediation sets, (c) defer -> write carry-forward context (sources: UX, FEATURES)
- Gray area consolidation should reduce prompt fatigue: single multiSelect for up to 8 areas (sources: UX, FEATURES)
- Table formatting should be consistent and scannable at a glance (sources: UX, OVERSIGHTS)

**Information architecture patterns:**
- Progressive disclosure: summary table first, detail drill-down on request, action routing last (source: UX)
- Banner + status summary at start of every skill invocation (source: UX)
- File artifact output with displayed path after writing (source: UX)

**Key accessibility considerations:**
- Text-only output with ANSI 16-color palette; no color-only information encoding (source: UX)
- Structured options over freeform input for decisions (source: UX)
- Status indicators should use text labels ("PASS", "FAIL", "GAP") alongside any color coding (source: UX)

## Contradictions

### Contradiction: Audit implementation approach (skill-only vs. hybrid)
- **FEATURES says:** Option 1 recommended -- pure skill-only approach with no new library module; follows `/rapid:quick` and `/rapid:new-version` precedents
- **ARCHITECTURE says:** Create `src/lib/audit.cjs` library module with `auditMilestone()`, `classifyGaps()`, `formatAuditReport()` functions, plus `src/commands/audit.cjs` CLI handler
- **Resolution:** Both are valid but differ in complexity. The FEATURES recommendation (skill-only) is simpler and appropriate for a first iteration. The ARCHITECTURE recommendation (hybrid) provides deterministic artifact harvesting and testability. **Recommend starting with skill-only and extracting to library only if deterministic harvesting proves necessary.** This is a design decision for the implementer.

### Contradiction: Gray area consolidation target (single prompt vs. capped batches)
- **FEATURES says:** Use a single multiSelect prompt with ALL gray areas regardless of count (even 12)
- **ARCHITECTURE says:** Cap at 8 per prompt; for n=3 (12 areas) use 2 prompts of 6
- **PITFALLS says:** 12 checkboxes in one prompt may feel overwhelming; multi-batch approach creates fragmented experience
- **Resolution:** The cap at 8 per prompt is a reasonable middle ground. **Recommend single prompt for up to 8 areas, 2 prompts for 9-12 areas.** This balances prompt fatigue reduction against checkbox overload.

### Contradiction: Table formatting fix approach (padded tables vs. structured lists)
- **FEATURES says:** Option 2 recommended -- replace tables with structured list format (eliminates alignment problem entirely)
- **UX says:** Use padded table cells since users are already familiar with the table layout
- **OVERSIGHTS says:** Need to clarify rendering context first; fix depends on whether issue is in AskUserQuestion HTML vs. terminal
- **Resolution:** The OVERSIGHTS point is valid -- diagnosis should come first. However, **structured lists are the safer default** because they work in all rendering contexts. If the rendering context turns out to be HTML (where tables align fine), padding is sufficient. **Recommend structured lists as the primary approach** unless investigation reveals the issue is terminal-only.

### Contradiction: Display banner color for audit-version
- **OVERSIGHTS says:** Bright yellow (`\x1b[103m`) to distinguish from planning/execution/review
- **UX says:** Bright red, matching review/merge as a quality gate stage
- **Resolution:** Minor cosmetic choice. **Needs implementer decision.** Yellow emphasizes distinctness; red emphasizes quality-gate severity.

## Recommended Approach

The v4.2.0 milestone is a focused, low-risk release with four independent items that touch largely non-overlapping parts of the codebase. The path resolution bug fix is the clear first priority -- it is trivial to implement, unblocks core init functionality, and all six research streams independently flagged it. The fix should use `path.dirname()` inside Node inline scripts (reading from `process.env.RAPID_TOOLS` at runtime rather than shell interpolation) and be applied to all 4 affected locations in `skills/init/SKILL.md` and `skills/register-web/SKILL.md`. Alongside the fix, establish a convention (documented in CONVENTIONS.md or the env preamble) to prevent the pattern from recurring.

The two discuss-set improvements (table formatting and gray area consolidation) should be implemented together since both modify `skills/discuss-set/SKILL.md`. The critical trap here is the existing test suite: `SKILL.test.cjs` has 5 assertions hardcoded to "exactly 4 gray areas" that will break immediately. Tests must be updated in the same commit as the SKILL.md changes. For gray area consolidation, present all areas in a single multiSelect prompt for counts up to 8, and split into 2 prompts for 9-12 areas. For table formatting, replace the markdown table in Format A with a structured list format that renders reliably regardless of content length or rendering context.

The audit-version command is the most complex item but carries low architectural risk because RAPID's existing patterns (skills, agents, CLI commands, display banners) provide all the scaffolding. The recommended approach is skill-only for the first iteration: the audit skill reads STATE.json, ROADMAP.md, REQUIREMENTS.md, and per-set artifacts; cross-references requirements against delivered work; produces a `.planning/v{version}-AUDIT.md` report; and presents findings via AskUserQuestion with three outcome paths (passed, gaps, defer). The audit must be non-destructive (read-only against state) and must cross-reference existing DEFERRED.md files to distinguish intentional deferrals from actual gaps. Milestone completion should be derived from set statuses rather than adding a new schema field -- this avoids migration complexity and keeps the audit observational rather than lifecycle-altering.

All four items can be parallelized across sets since they have no cross-dependencies. The only coordination point is ensuring the discuss-set changes and their test updates ship together.

### Suggested Set Boundaries

- **Set A: Path Resolution Bug Fix** -- Fix `require()` path resolution in `skills/init/SKILL.md` and `skills/register-web/SKILL.md` (4 affected lines). Establish path resolution convention. Optionally add `resolve lib-path` CLI subcommand. Rationale: trivial, self-contained, highest priority bug fix that unblocks init.

- **Set B: Discuss-Set UX Improvements** -- Gray area consolidation (Step 5) + table formatting (Step 6 Format A) in `skills/discuss-set/SKILL.md`, plus coordinated `SKILL.test.cjs` updates. Rationale: both changes modify the same file and benefit from unified testing; the test suite coordination requirement makes them natural to group.

- **Set C: Audit Version Command** -- New `skills/audit-version/SKILL.md`, display.cjs banner entry, optional `src/lib/audit.cjs` + `src/commands/audit.cjs` if deterministic harvesting is needed, agent role module. Rationale: largest item, self-contained new feature, no dependencies on Sets A or B.
