# Research Summary -- v4.3.0 Reliability & State

## Project Profile
- **Type:** brownfield plugin/CLI tool (Claude Code plugin for parallel AI-assisted development)
- **Stack:** Node.js (v25.8.0 system, 18+ minimum) + CommonJS + Zod 3.25 + proper-lockfile 4.1.2
- **Scale:** medium (5 features/fixes, 21+ test files, 24 skills, 26 agents, 7-step lifecycle)
- **Key Constraint:** All bugs are code-level issues (wrong paths, missing implementations, opaque errors) -- no dependency or infrastructure changes required

## Unified Findings

### Stack and Infrastructure
- Node.js v25.8.0 is well above the v18+ minimum; `node:test` runner is stable and fully supports all test patterns used (sources: STACK, OVERSIGHTS)
- All dependencies are current and secure -- `npm audit` reports 0 vulnerabilities (source: STACK)
- Zod 3.25.76 is in maintenance mode; v4 is ESM-first and would be a non-trivial migration for this 100% CJS project -- defer to future milestone (sources: STACK, PITFALLS)
- proper-lockfile 4.1.2 is the latest version; stable but in maintenance mode since 2022 -- no immediate risk (sources: STACK, PITFALLS)
- No build step, no linter, no CI pipeline configured -- tests run locally via `node --test` (sources: STACK, OVERSIGHTS)
- No code coverage reporting configured (source: STACK)
- `npm test` script is missing from package.json (source: OVERSIGHTS)

### Feature Landscape

**Bug Fixes (independent, parallelizable):**

1. **DAG.json Not Found During Merge Order** -- Three distinct root causes: (a) wrong hardcoded path in `merge.cjs:274`, (b) `getMergeOrder()` does not resolve worktree paths to project root, (c) `/new-version` skill never calls `dag generate` after writing STATE.json (sources: FEATURES, ARCHITECTURE, PITFALLS)

2. **STATE.json Missing/Invalid During State Transitions** -- Opaque error in `withStateTransaction()` conflates file-missing, JSON-parse-failure, and schema-validation-failure into one message; potential race conditions with concurrent agents; no recovery path for orphaned `.tmp` files (sources: FEATURES, ARCHITECTURE, PITFALLS, OVERSIGHTS)

3. **/plan-set --gaps Treats Merged Sets as Complete** -- Flag referenced in execute-set output but never implemented in plan-set SKILL.md; status validation blocks all post-discussed states; `SET_TRANSITIONS` defines `merged: []` as terminal with no re-entry (sources: FEATURES, ARCHITECTURE, PITFALLS)

**UX Improvements (share SKILL.md surface area):**

4. **/new-version Spec-Aware** -- Accept spec file argument, pre-populate goals categories, reduce 5-prompt sequence to 1-prompt confirmation (source: FEATURES)

5. **/new-version Deferred Auto-Discovery** -- Include all DEFERRED.md contents in researcher briefs, not just user-selected items (source: FEATURES)

**Dependency graph:** Features 1-3 are fully independent. Features 4 and 5 share the same SKILL.md file but modify different sections. Feature 5 has a soft dependency on Feature 4 (shared modification surface).

### Architecture Direction
- **Pattern:** Targeted surgical fixes within existing modular monolith -- architecture is sound, bugs are localized (source: ARCHITECTURE)
- **Key architectural issue:** Two incompatible project root resolution functions -- `findProjectRoot()` in `core.cjs` (directory walk) vs `resolveProjectRoot()` in `plan.cjs` (git worktree-aware via `git rev-parse --git-common-dir`). The CLI backbone uses the non-worktree-aware version (sources: ARCHITECTURE, PITFALLS)
- **Recommended fix:** Extract `resolveProjectRoot()` to a shared module (e.g., `core.cjs`) and replace all uses of `findProjectRoot()` with the worktree-aware version (sources: FEATURES, ARCHITECTURE, PITFALLS)
- **State machine design:** The transaction pattern (lock -> read -> validate -> mutate -> validate -> write -> unlock) is well-designed; the bugs are in error reporting and path resolution, not in the transaction model itself (source: ARCHITECTURE)
- **Gap-closure workflow:** Should be a lightweight overlay on the existing lifecycle, not a state machine modification. Sets remain in terminal state; gap-closure waves are additional planning artifacts (sources: FEATURES, ARCHITECTURE)

### User Experience Direction

**Domain conventions:** RAPID operates in the developer tooling / build orchestration space alongside Nx, Turborepo, and CI orchestrators. Its terminology (set, wave, DAG, worktree) is well-chosen and domain-appropriate (source: UX)

**Error communication gap:** All three bugs share a pattern -- errors describe the symptom but not the cause or recovery path. Domain convention (Rust/Cargo-style) calls for three-part errors: (1) what happened, (2) why it happened, (3) what to do (sources: UX, OVERSIGHTS)

**Key UX patterns:**
- Users expect idempotent retries or clear explanations of why retry is not possible (source: UX)
- Users expect transient failure recovery with automatic retry and backoff (source: UX)
- Users expect DAG availability after init/new-version completes -- the "DAG.json not found" error violates the "install works after init" mental model (source: UX)
- The gap-closure workflow violates user expectations: execute-set suggests `/plan-set --gaps` but the command does not work on merged sets (sources: UX, PITFALLS)

**Missing UX pattern:** A `rapid-tools doctor` diagnostic command (analogous to `brew doctor`, `flutter doctor`, `npm doctor`) that checks STATE.json validity, DAG.json existence, lock staleness, and worktree consistency (source: UX)

**Accessibility:** RAPID's text-based CLI output with `[RAPID]`/`[RAPID ERROR]` prefixes and breadcrumb progress indicators is accessible without color dependency. No accessibility changes needed (source: UX)

## Contradictions

### Contradiction: Gap-Closure State Transitions

- **FEATURES says:** Option 1 (Skill-only fix) is recommended -- skip state transition entirely in `--gaps` mode, keep set in terminal state, do not modify SET_TRANSITIONS
- **ARCHITECTURE says:** Option A recommends adding `merged -> planned` and `complete -> planned` transitions to SET_TRANSITIONS, and transitioning the set back to `planned` status
- **PITFALLS says:** Add reverse transitions `merged -> planned` and `complete -> planned`, gated behind a `--gaps` flag
- **Resolution:** This is a genuine design decision. FEATURES' approach (no state machine changes) is simpler and avoids complicating the linear lifecycle. ARCHITECTURE's approach (add transitions) makes the state machine accurate. Both are valid. **Recommend FEATURES' approach** (skill-only, no state transitions) for simplicity, since gap-closure waves are additive artifacts that do not change the set's fundamental completion status. The roadmapper should make the final call.

### Contradiction: readState Lock Protection

- **ARCHITECTURE says:** `readState()` lacks lock protection and could read stale data during concurrent writes; suggests adding a shared/read lock for standalone calls (priority: low)
- **STACK says:** `readState()` not having a lock is safe because `renameSync` is atomic on POSIX; the real issue is path resolution, not locking
- **Resolution:** STACK is correct for the specific bug #2 scenario -- atomic rename means reads never see partial writes on local POSIX filesystems. ARCHITECTURE's concern about stale reads is valid but low-impact since `withStateTransaction` already acquires a lock before reading. No lock changes to `readState()` are needed for this milestone.

## Recommended Approach

This milestone is a reliability-focused release with three well-scoped bug fixes and two UX improvements. The architecture is sound -- all bugs are localized to specific modules and skill definitions, not systemic failures. The correct approach is surgical fixes with improved error reporting, not architectural refactoring.

The first priority should be consolidating the project root resolution into a single worktree-aware function. This is the foundational fix that unblocks both the DAG.json-not-found bug and prevents the same class of path resolution error from occurring in future modules. The current situation -- two functions (`findProjectRoot` and `resolveProjectRoot`) with incompatible behavior -- is the root cause of multiple reported failures. Moving `resolveProjectRoot` to `core.cjs` as a shared utility is a small change with outsized impact.

The second priority is the STATE.json error diagnostics improvement. The intermittent "missing or invalid" error is the most user-facing pain point, and the fix (splitting the error into specific types with recovery guidance) is straightforward. This should be paired with orphaned `.tmp` file detection and the `onCompromised` abort flag to address the full spectrum of STATE.json reliability concerns. Concurrency tests should be written alongside these fixes to verify correctness under parallel agent load.

The third priority is the plan-set `--gaps` implementation. This is a skill-only change (no code modifications needed) that completes the gap-closure workflow referenced by execute-set. The recommended approach is to bypass status validation when `--gaps` is present and skip state transitions entirely -- gap-closure waves are additive artifacts that do not change the set's lifecycle position.

The two `/new-version` UX improvements (spec-aware and deferred auto-discovery) are lower priority but high value. They share the same SKILL.md file and can be implemented together. The spec-aware feature reduces the 5-prompt goal-gathering sequence to a 1-prompt confirmation, which significantly improves the milestone creation experience for users with existing spec documents.

### Suggested Set Boundaries

Rationale: all changes relate to path resolution and DAG lifecycle; they share the same root cause and touch related modules (`core.cjs`, `merge.cjs`, `dag.cjs`, `new-version/SKILL.md`).
