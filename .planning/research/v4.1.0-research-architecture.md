# Architecture Research: v4.1.0 Polish & Fixes

## Current Architecture

RAPID is a Claude Code plugin structured as a 5-layer system: Skills (user-facing commands) > Agents (specialized AI workers) > CLI (rapid-tools.cjs command router) > Library (21 core modules in src/lib/) > Storage (.planning/ state, .rapid-worktrees/ isolation). The system uses CommonJS throughout, Zod for state validation, proper-lockfile for concurrency, and git worktrees for physical isolation between parallel sets.

### Strengths Relevant to This Milestone

1. **Skill-level modularity:** Each skill is a self-contained SKILL.md file with clear step sequences. The branding, discuss-set, init, unit-test, and new-version skills can each be modified independently without cross-skill coupling. This directly supports the milestone's scattered enhancement targets.

2. **DAG infrastructure already exists:** `src/lib/dag.cjs` provides complete toposort, wave assignment, v1/v2 DAG creation, validation, and loading. The `add-set.cjs` module already accepts `--deps` and calls `recalculateDAG()` after set insertion. The web dashboard already has a `KnowledgeGraphPage` using cytoscape-dagre for DAG visualization. The plumbing exists; it needs wiring back into the workflow.

3. **State schema is extensible:** The Zod schemas in `state-schemas.cjs` use `.passthrough()` on all object types, meaning additional fields (like `deps` on SetState) can be persisted without schema migration. This is important for the DAG dependency resolution enhancement.

4. **Test framework detection exists in context.cjs:** The `detectCodebase()` function already detects testing config files (jest, vitest, mocha, pytest) and identifies test vs. source files. This provides a foundation for making the unit-test skill framework-agnostic.

### Areas Requiring Extension

1. **Discuss-set skill (SKILL.md):** The current flow hardcodes exactly 4 gray areas and question format lacks research context (pros/cons, recommendations). The CONTEXT.md output lacks `<canonical_refs>` and `<deferred>` sections are minimal. The deferred decisions have no persistent storage beyond the inline CONTEXT.md section.

2. **Init skill acceptance criteria:** Step 4D writes REQUIREMENTS.md, then Step 5 scaffold unconditionally overwrites it. The `scaffoldProject()` function in `init.cjs` has no guard against overwriting REQUIREMENTS.md in `fresh` mode.

3. **RAPID_ROOT:** Computed in every skill preamble (`RAPID_ROOT="${CLAUDE_SKILL_DIR}/../.."`) but never used for anything besides deriving `.env` path. It was a v1.0 concept that should have been removed when `RAPID_TOOLS` became the canonical path. Found in 26 skill files (124 total occurrences, most as part of the env preamble).

4. **Review log-issue:** The CLI handler at `src/commands/review.cjs` lines 66-91 reads issue data from stdin JSON, but skill docs in `skills/unit-test/SKILL.md` show CLI flag syntax (`--set-id`, `--type`, `--severity`, etc.) that does not work.

5. **Unit-test 5-group cap:** The skill hardcodes "up to 5 concern groups maximum" in Step 3 with no fallback strategy when concern groups exceed 5.

## Recommended Architecture Pattern

- **Pattern:** Incremental refinement of existing modular skill architecture
- **Rationale:** This is a polish/bugfix milestone. The existing architecture is sound -- the 5-layer system with skill-level modularity cleanly supports targeted changes. Introducing new architectural patterns would be counter-productive. Each change in this milestone touches a different skill or library module, with minimal cross-cutting concerns. The correct approach is to refine each module in place while respecting existing conventions.
- **Evidence:** The existing modular structure has been stable through v1.0 to v4.0.0, demonstrating that skill-level changes can be made independently. The `.passthrough()` schema design, the DAG library's existing API, and the context detection infrastructure all support the planned enhancements without structural changes.

## Module Organization

### Module: discuss-set Skill

- **Responsibility:** Capture developer implementation vision for a set through structured gray area discussion
- **Public interface:** Skill command `/rapid:discuss-set <set-id>` with `--skip` flag
- **Dependencies:** rapid-tools CLI (resolve, state, display), AskUserQuestion, Write tool
- **Data ownership:** `.planning/sets/{set-id}/CONTEXT.md`, deferred decisions
- **Changes needed:**
  - Variable gray area count (multiples of 4: 4, 8, 12) instead of fixed 4
  - Richer question format: 3-5 sentence context, pros/cons table, "(Recommended)" tagging per get-shit-done reference
  - Add `<canonical_refs>` section to CONTEXT.md output
  - Deferred decisions written to `.planning/sets/{set-id}/DEFERRED.md` or a shared `.planning/deferred/` directory
  - Architect-level and UI/UX questions instead of implementation details

### Module: init Skill

- **Responsibility:** Orchestrate project initialization: prereqs, scaffold, research, roadmap
- **Public interface:** Skill command `/rapid:init`
- **Dependencies:** rapid-tools CLI (prereqs, init, context, display), Agent spawning, AskUserQuestion
- **Data ownership:** `.planning/` scaffold, REQUIREMENTS.md, research files, STATE.json
- **Changes needed:**
  - Reorder Step 4D (write REQUIREMENTS.md) and Step 5 (scaffold) so scaffold runs first, OR add a guard in `scaffoldProject()` to skip REQUIREMENTS.md if it has non-template content
  - Encode acceptance criteria with domain-specific IDs (UIUX-001, PERF-001, FUNC-001, etc.)
  - Store encoded criteria in REQUIREMENTS.md with structured format that the verifier agent can programmatically check

### Module: branding Skill

- **Responsibility:** Capture project tone, voice, and visual style preferences
- **Public interface:** Skill command `/rapid:branding`
- **Dependencies:** rapid-tools CLI (display), AskUserQuestion, Write tool, Glob, Grep
- **Data ownership:** `.planning/branding/BRANDING.md`, `.planning/branding/index.html`
- **Changes needed:**
  - Refocus from documentation style to UI/UX visual branding
  - Add codebase detection: if no web frontend detected (no package.json with React/Vue/Angular, no HTML templates), skip visual branding questions and focus on CLI/API output style
  - May integrate with `context.cjs` detectCodebase() to determine project type

### Module: unit-test Skill

- **Responsibility:** Run unit test pipeline on a scoped set
- **Public interface:** Skill command `/rapid:unit-test <set-id>` with `--post-merge` flag
- **Dependencies:** REVIEW-SCOPE.md input, rapid-tools CLI (review, resolve, state), Agent spawning
- **Data ownership:** REVIEW-UNIT.md, test files in worktree
- **Changes needed:**
  - Remove the hardcoded "up to 5 concern groups maximum" cap in Step 3
  - Generalize dispatch: if N concern groups, spawn N agents (with token-budget awareness note)
  - Make test runner framework-agnostic: detect from project config (package.json scripts.test, pytest.ini, Cargo.toml, go.mod) instead of hardcoding `node --test`
  - Agent prompt should specify detected test framework and conventions

### Module: review.cjs (CLI handler)

- **Responsibility:** CLI command routing for review operations (scope, log-issue, list-issues, etc.)
- **Public interface:** `review log-issue <set-id>` via stdin JSON
- **Dependencies:** src/lib/review.cjs, src/lib/args.cjs, src/lib/stdin.cjs
- **Data ownership:** REVIEW-ISSUES.json per set
- **Changes needed:**
  - Either update handler to parse CLI flags (`--type`, `--severity`, etc.) and construct JSON internally
  - Or update skill docs to show the stdin JSON format
  - Decision: parsing CLI flags is more user-friendly and aligns with how other commands work

### Module: init.cjs (scaffold library)

- **Responsibility:** Generate .planning/ scaffold files
- **Public interface:** `scaffoldProject(cwd, opts, mode)`, `detectExisting(cwd)`
- **Dependencies:** state-machine.cjs (createInitialState), version.cjs
- **Data ownership:** All .planning/ template files
- **Changes needed:**
  - Add guard in `scaffoldProject()` for REQUIREMENTS.md: before writing, check if existing file contains non-template content (line count > template line count or content differs from generator output). If so, skip it and include in `skipped` array.
  - This respects the "additive-only" principle already used in `upgrade` mode

### Module: new-version Skill

- **Responsibility:** Create new milestone with research pipeline and roadmap generation
- **Public interface:** Skill command `/rapid:new-version`
- **Dependencies:** rapid-tools CLI (state, display), 6 research agents, synthesizer, roadmapper
- **Data ownership:** New milestone in STATE.json, research files, ROADMAP.md
- **Changes needed:**
  - Step 2 milestone goals gathering needs more structured requirements capture
  - The goals freeform question should be decomposed into multiple focused questions (features, improvements, bug fixes, removed scope)
  - Research agent prompts should include the full structured goals, not just a single freeform string
  - Roadmapper should receive the structured goals to ensure no requirements are missed

## Data Flow

### Discuss-Set Enhanced Flow

```
1. User invokes /rapid:discuss-set <set>
2. Resolve set reference -> load CONTRACT.json, DEFINITION.md, SET-OVERVIEW.md
3. Codebase scout: Glob/Grep source files in file ownership
4. Identify gray areas (multiples of 4, adaptive to set complexity)
   - Each gray area includes: 3-5 sentence context, pros/cons, recommendation
5. Present selection (1 AskUserQuestion per 4 areas)
6. For each selected area: deep-dive with enriched questions
   - Each question shows: tradeoff context, options with pros/cons, recommended tag
7. Follow-up questions if understanding gaps remain
8. Write CONTEXT.md with enriched sections:
   - <domain>, <decisions>, <canonical_refs>, <code_context>, <specifics>, <deferred>
9. Write deferred decisions to DEFERRED.md
10. State transition: pending -> discussed
11. Commit CONTEXT.md + DEFERRED.md + STATE.json
```

### Init Enhanced Flow (REQUIREMENTS.md fix)

```
1. Steps 1-4B: unchanged
2. Step 4C: Granularity preference (unchanged)
3. Step 4D: Generate encoded acceptance criteria
   - Each criterion gets an encoded ID: {DOMAIN}-{NNN}
   - Domains: FUNC (functional), UIUX (UI/UX), PERF (performance),
     SEC (security), A11Y (accessibility), I18N (internationalization),
     INTG (integration), COMP (compliance)
   - Write to REQUIREMENTS.md
4. Step 5: Scaffold
   - scaffoldProject() skips REQUIREMENTS.md if it already has non-template content
   - All other files generated normally
5. Steps 6+: Research, synthesis, roadmap (unchanged)
```

### DAG Dependency Resolution Flow

```
1. During /rapid:init or /rapid:new-version, roadmapper defines set dependencies
2. add-set CLI already accepts --deps flag, validates deps exist, calls recalculateDAG()
3. DAG.json written to .planning/sets/DAG.json
4. Web dashboard reads DAG.json via backend API -> renders in KnowledgeGraphPage
5. /rapid:merge uses merge order (already calls dag.getExecutionOrder())
6. /rapid:status could display sets in DAG wave order (not alphabetically)
```

The DAG infrastructure exists end-to-end. The gap is that roadmapper and start-set flows need to consistently produce dependency edges and that the command center display uses DAG ordering.

### Unit-Test Framework Detection Flow

```
1. Unit-test skill loads REVIEW-SCOPE.md
2. NEW: Detect test framework from project root:
   a. Check package.json scripts.test -> extract runner (jest, vitest, mocha, node --test)
   b. Check for pytest.ini, pyproject.toml [tool.pytest] -> pytest
   c. Check for Cargo.toml -> cargo test
   d. Check for go.mod -> go test
   e. Fallback: node --test (current default)
3. Pass detected framework to rapid-unit-tester agent prompt
4. Agent uses framework-specific test file naming, assertion library, and runner command
5. Results collected in same format regardless of framework
```

### State Management

- **Where state lives:** `.planning/STATE.json` (machine state), `.planning/sets/*/` (per-set artifacts), `.planning/branding/` (branding), `.planning/config.json` (runtime config)
- **How state is read/written:** All mutations through `withStateTransaction()` (lock + read + validate + mutate + validate + write + unlock). Skills write artifact files directly via the Write tool.
- **Consistency guarantees:** Zod validation on every state write. File locks prevent concurrent mutation. `.passthrough()` on schemas allows forward-compatible field additions.
- **No schema migration needed:** The SetState schema uses `.passthrough()`, so adding `deps` or `encodedCriteria` fields to sets or milestones does not require version bumping.

## Scaling Considerations

Not applicable to this milestone. All changes are to skill definitions (Markdown files) and library modules. No new runtime services, no new data stores, no performance-sensitive code paths.

The only consideration is the unit-test skill's removal of the 5-group cap -- spawning N agents for N concern groups has token cost implications. The skill should include a soft warning when N > 8, suggesting the user consider merging related concerns, but should not enforce a hard cap.

## Architecture Risks

1. **RAPID_ROOT sweep may miss edge cases in archived files:** 103 files reference RAPID_ROOT (many in .planning/archive/). The sweep should focus on active skill files (26 in skills/) and active source code, not rewrite archived planning artifacts.
   - Impact: Low -- archived files are not executed
   - Mitigation: Grep for RAPID_ROOT in skills/ and src/ only; leave archives untouched

2. **Discuss-set variable gray area count may produce inconsistent CONTEXT.md quality:** Going from fixed 4 to variable (4, 8, 12) gray areas means some sets get deeper exploration than others.
   - Impact: Medium -- downstream plan-set quality depends on CONTEXT.md depth
   - Mitigation: Set a minimum of 4 and maximum of 12. Use set complexity (file count in CONTRACT.json, number of exports/imports) as a heuristic for how many gray areas to surface.

3. **Review log-issue backward compatibility:** Changing from stdin-only to CLI flags could break existing agent prompts that use the stdin JSON format.
   - Impact: Medium -- agents reference log-issue in their role prompts
   - Mitigation: Support BOTH interfaces. If stdin has data, use it (current behavior). If no stdin data but CLI flags present, construct JSON from flags. This preserves backward compatibility.

4. **Encoded criteria IDs (UIUX-001) add complexity to init flow:** Users may find domain encoding unfamiliar or overly formal.
   - Impact: Low -- encoding is generated by RAPID, not typed by users
   - Mitigation: Auto-generate IDs from the acceptance criteria content. Users see readable criteria with IDs as prefixes. The verifier checks IDs programmatically.

5. **Unit-test framework detection may not cover all project types:** The context.cjs detection is limited to known manifests.
   - Impact: Low -- fallback to `node --test` maintains current behavior
   - Mitigation: Allow override via `.planning/config.json` testRunner field. Detection is best-effort; the user can always specify explicitly.
