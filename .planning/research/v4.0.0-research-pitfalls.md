# Pitfalls Research -- v4.0.0 Mission Control Dashboard

## Critical Pitfalls

### Pitfall: SQLite Write Contention Under Concurrent Access
- **Category:** data-loss / correctness
- **Stack:** SQLite, FastAPI, aiosqlite/SQLAlchemy
- **Description:** SQLite uses a database-level file lock for writes. When the FastAPI backend handles concurrent requests (multiple browser tabs, SSE connections, CLI registration commands), simultaneous write operations collide, producing `SQLITE_BUSY` or `OperationalError: database is locked` errors. The default SQLite timeout is 5 seconds, after which writes fail silently or raise exceptions that may not be caught by generic error handlers.
- **Trigger:** Two or more simultaneous write operations -- e.g., a file watcher updating project state while a user action triggers a kanban update, or multiple projects registering at the same time. Also triggered when the CLI `rapid-tools.cjs` writes to the same database that the FastAPI service reads/writes (cross-process contention).
- **Impact:** Severity CRITICAL. Lost writes, corrupted kanban state, or stale data displayed to users. In the worst case, repeated lock timeouts cause the file watcher to fall behind and display permanently stale information.
- **Prevention:**
  1. Enable WAL (Write-Ahead Logging) mode at database creation: `PRAGMA journal_mode=WAL;`. WAL allows concurrent readers while a single writer operates.
  2. Set `PRAGMA busy_timeout=5000;` (or higher) so writes retry instead of failing instantly. Production crash rates can drop 40% from this single pragma alone.
  3. Funnel all writes through a single async writer task (a dedicated write queue) rather than allowing arbitrary connections to write. This serializes writes at the application level and avoids SQLite-level lock contention entirely.
  4. Keep transactions short -- do data processing outside the transaction, then write in a quick batch.
  5. Set `PRAGMA synchronous=NORMAL;` with WAL mode to reduce fsync overhead without sacrificing durability.
- **Detection:** Monitor for `OperationalError: database is locked` in application logs. Add a counter metric for write retries. If the WAL file grows unboundedly, checkpoint frequency is too low.
- **References:**
  - [SQLite WAL Mode Documentation](https://sqlite.org/wal.html)
  - [SQLite Connection Pool Pitfalls](https://www.jtti.cc/supports/3154.html)
  - [beets: the SQLite Lock Timeout Nightmare](https://beets.io/blog/sqlite-nightmare.html)

### Pitfall: Blocking the FastAPI Event Loop with Synchronous Code
- **Category:** performance / correctness
- **Stack:** FastAPI, uvicorn, Python asyncio
- **Description:** FastAPI routes defined with `async def` run directly on the event loop. If any synchronous/blocking operation occurs inside an `async def` route (file I/O, CPU computation, synchronous database calls, `time.sleep()`), the entire event loop freezes. No other request can be processed until the blocking call completes. This is particularly dangerous because the dashboard will use file watching (synchronous disk reads), SQLite operations, and potentially CPU-intensive graph computation.
- **Trigger:** Using `open()` / `os.path.exists()` / `json.load()` inside `async def` routes. Using synchronous SQLite drivers (stdlib `sqlite3`) inside async endpoints. Performing graph layout calculations inline in request handlers.
- **Impact:** Severity CRITICAL. The entire dashboard becomes unresponsive during blocking operations. SSE streams stall, WebSocket connections drop, and all concurrent users experience hangs. Under load, this manifests as request queuing and eventual timeouts.
- **Prevention:**
  1. Use `async def` only for endpoints that exclusively call `await`-compatible I/O (aiosqlite, aiofiles, httpx).
  2. Use plain `def` for endpoints that must call synchronous code -- FastAPI will automatically run them in a thread pool.
  3. For CPU-bound work (graph layout, codebase analysis), offload to `asyncio.to_thread()` or a process pool via `concurrent.futures.ProcessPoolExecutor`.
  4. Never use `time.sleep()` in async code; use `await asyncio.sleep()` instead.
  5. Use `aiosqlite` (not stdlib `sqlite3`) for all database access in async endpoints.
- **Detection:** Add middleware that logs request duration. Any request taking >100ms in an async route is suspicious. Use `asyncio` debug mode (`PYTHONASYNCIODEBUG=1`) which warns when the event loop is blocked for >100ms.
- **References:**
  - [FastAPI Async Documentation](https://fastapi.tiangolo.com/async/)
  - [FastAPI Mistakes That Kill Performance](https://dev.to/igorbenav/fastapi-mistakes-that-kill-your-performance-2b8k)
  - [FastAPI Issue #2488: async def blocks def functions](https://github.com/fastapi/fastapi/issues/2488)

### Pitfall: inotify Watch Limit Exhaustion
- **Category:** correctness / performance
- **Stack:** Python watchdog library, Linux inotify
- **Description:** The dashboard must watch STATE.json and planning artifacts across multiple registered RAPID projects. Each project has a `.planning/` directory tree with STATE.json, sets/, contracts/, worktrees/REGISTRY.json, and other artifacts. On Linux, the `watchdog` library uses `inotify` under the hood. Each watched directory consumes an inotify watch. The default Linux limit is often 8192 watches. With recursive watching across even 10-20 projects, the limit can be reached, causing `OSError: [Errno 28] inotify watch limit reached`.
- **Trigger:** Registering many RAPID projects, especially those with large `.planning/` directory trees. Recursive watching (which watchdog enables by default) on project directories that contain `node_modules/`, `.git/`, or other large subtrees multiplies the problem.
- **Impact:** Severity CRITICAL. Once the inotify limit is hit, no new watches can be created. The file watcher crashes or silently stops detecting changes. The dashboard displays permanently stale data with no error visible to the user.
- **Prevention:**
  1. Watch only specific files (STATE.json, REGISTRY.json), not entire directory trees. Use targeted file watches rather than recursive directory watches.
  2. If directory watching is needed, set `recursive=False` and manually add watches only for known subdirectories (`.planning/sets/`, `.planning/worktrees/`).
  3. Implement a fallback to `PollingObserver` when inotify watches are exhausted, with a configurable polling interval (e.g., 5 seconds).
  4. Document that users may need to increase the inotify limit: `echo fs.inotify.max_user_watches=524288 | sudo tee /etc/sysctl.d/40-max-user-watches.conf && sudo sysctl --system`.
  5. On macOS, FSEvents does not have this limitation, but be aware of different event delivery semantics.
- **Detection:** Catch `OSError` with errno 28 specifically and log a clear message with the fix instruction. Add a health check endpoint that reports the number of active watches.
- **References:**
  - [Watchdog inotify watch limit issue #673](https://github.com/gorakhargosh/watchdog/issues/673)
  - [Watchdog inotify instance limit issue #846](https://github.com/gorakhargosh/watchdog/issues/846)
  - [How to fix inotify watch limit reached](https://gist.github.com/fastfingertips/20deb7c4235bd51800fcc4edad1fbf36)

## High-Risk Pitfalls

### Pitfall: systemd User Service Fails After Logout
- **Category:** correctness / compatibility
- **Stack:** systemd, loginctl, XDG_RUNTIME_DIR
- **Description:** systemd user services (managed via `systemctl --user`) are tied to the user's login session. By default, when the user logs out of all sessions (including closing SSH, graphical logout), `user@.service` is terminated, taking all user services (including the RAPID dashboard) with it. The `XDG_RUNTIME_DIR` (/run/user/UID) is also cleaned up, which breaks the service's socket and runtime files.
- **Trigger:** User logs out of their desktop session, closes all SSH sessions, or the system restarts without the user logging in. On WSL2, `/run/user/1000` may be unexpectedly deleted.
- **Impact:** Severity HIGH. The dashboard stops running when the user is logged out, defeating the purpose of a background service. Users must manually restart the service after every login.
- **Prevention:**
  1. Enable lingering for the user: `loginctl enable-linger $USER`. This keeps `user@.service` running even when the user has no active sessions.
  2. The install command must check for and enable lingering, or at minimum instruct the user to do so.
  3. Store PID files and sockets in a location that survives logout (e.g., `~/.rapid/run/` instead of `$XDG_RUNTIME_DIR`).
  4. Add a `WantedBy=default.target` in the unit file to ensure the service starts on boot (with lingering enabled).
  5. Test on WSL2 explicitly -- `/run/user/1000` deletion is a known 2025 issue there.
- **Detection:** After enabling the service, verify it survives logout: log out, log back in, check `systemctl --user status rapid-dashboard`. Add a health check that the install process runs.
- **References:**
  - [systemd/User -- ArchWiki](https://wiki.archlinux.org/title/Systemd/User)
  - [systemd issue #10574: XDG_RUNTIME_DIR not set](https://github.com/systemd/systemd/issues/10574)
  - [WSL2 runtime dir deletion](https://hy2k.dev/en/blog/2025/01-26-wsl2-runtime-dir/)

### Pitfall: Cross-Platform Service Management (systemd vs launchd vs None)
- **Category:** compatibility
- **Stack:** systemd (Linux), launchd (macOS), Windows
- **Description:** The spec calls for a background service that starts on boot. systemd is Linux-only. macOS uses launchd with plist files (fundamentally different config format). Windows has no direct equivalent (closest is Task Scheduler or NSSM for running services). Each platform has different directory conventions, permission models, and lifecycle behaviors.
- **Trigger:** Any user on macOS or Windows trying to use the `--service` install feature.
- **Impact:** Severity HIGH. If only systemd is implemented, macOS and Windows users cannot use the auto-start feature. If cross-platform is attempted naively, subtle bugs arise: launchd user agents only run while the user is logged in (similar to systemd without lingering), Windows Task Scheduler has different trigger semantics, etc.
- **Prevention:**
  1. Abstract service management behind a platform detection layer. Detect the OS and generate the appropriate config (systemd unit, launchd plist, or manual instructions).
  2. For macOS: generate a `~/Library/LaunchAgents/com.rapid.dashboard.plist` file. Use `launchctl load` to register.
  3. For Windows: generate a startup shortcut or provide a PowerShell script. Document that this is less integrated than Linux/macOS.
  4. Make the service entirely optional (the spec already says this). The dashboard should work as a manually-started process with equal functionality.
  5. Provide a `rapid-dashboard start` command that works everywhere as a foreground/background process, independent of init system integration.
- **Detection:** Test on all three platforms during development. CI/CD should include at least Linux and macOS runners.
- **References:**
  - [Creating Launch Daemons and Agents -- Apple Developer](https://developer.apple.com/library/archive/documentation/MacOSX/Conceptual/BPSystemStartup/Chapters/CreatingLaunchdJobs.html)
  - [launchd Tutorial](https://www.launchd.info/)

### Pitfall: Watchdog Memory Leak with InotifyObserver
- **Category:** performance / correctness
- **Stack:** Python watchdog, Linux inotify
- **Description:** The watchdog library's `InotifyObserver` has a documented memory leak: `InotifyEvent` objects are never garbage collected until the `Observer` object stops/joins. For a long-running dashboard service that watches file changes 24/7, memory usage grows unboundedly over time, eventually consuming all available RAM.
- **Trigger:** Running the dashboard for extended periods (days/weeks) with active file watching on projects that have frequent state changes (active development with many set transitions, worktree operations).
- **Impact:** Severity HIGH. The service gradually consumes more memory until the OS kills it (OOM killer) or the system becomes unresponsive. Since this is a background service, users may not notice until the dashboard suddenly stops responding.
- **Prevention:**
  1. Use `PollingObserver` instead of the default `Observer` (which uses inotify on Linux). Polling is less efficient but does not leak memory.
  2. Alternatively, periodically stop and restart the Observer on a schedule (e.g., every 6 hours) to force garbage collection of accumulated events.
  3. Watch only specific files rather than directories to minimize event volume.
  4. Implement a memory usage monitor that logs warnings when the process exceeds a threshold (e.g., 200MB).
  5. Consider using a custom minimal inotify wrapper via `ctypes` or the `inotify_simple` package instead of the full watchdog library.
- **Detection:** Monitor process RSS memory over time. A steadily increasing trend without plateau indicates a leak. Add a `/health` endpoint that reports current memory usage.
- **References:**
  - [Watchdog issue #587: InotifyEvents not garbage collected](https://github.com/gorakhargosh/watchdog/issues/587)
  - [watchdog-v6-without-memory-leak-adi on PyPI](https://pypi.org/project/watchdog-v6-without-memory-leak-adi/)

### Pitfall: Dual-Runtime Dependency Chain Complexity
- **Category:** correctness / compatibility
- **Stack:** Node.js (RAPID core), Python (FastAPI dashboard)
- **Description:** RAPID is a Node.js Claude Code plugin. The dashboard introduces Python/FastAPI as a second runtime. This creates a dual-dependency chain: `npm` for the Node.js side and `pip`/`uv`/`poetry` for the Python side. Users must have both Node.js and Python installed. Virtual environment management, PATH conflicts, and version mismatches create a combinatorial explosion of potential installation failures.
- **Trigger:** Installing RAPID on a system where Python is not available, or where the system Python is an incompatible version (e.g., Python 3.8 which lacks some async features used by modern FastAPI). Also triggered when `pip install` pollutes the global Python environment, or when the virtual environment is created in a location that conflicts with git worktree operations.
- **Impact:** Severity HIGH. Installation fails or the dashboard silently fails to start. Users who installed RAPID for its Claude Code plugin functionality now face unexpected Python dependency requirements. Debug difficulty is high because errors come from a different ecosystem than what RAPID users are familiar with.
- **Prevention:**
  1. Bundle the Python environment setup in the install script. Use `python -m venv ~/.rapid/venv` to create an isolated virtual environment.
  2. Pin Python version requirements clearly (e.g., Python 3.10+) and fail fast with a clear error if the version is too old.
  3. Use `uv` as the Python package manager if available (10-100x faster than pip, resolves dependencies deterministically).
  4. Keep the Python dependency list minimal. FastAPI + uvicorn + aiosqlite + watchdog is likely the entire list.
  5. The systemd unit / launchd plist must activate the virtual environment before starting uvicorn (set `Environment=PATH=~/.rapid/venv/bin:...` or use the venv's full python path).
  6. Provide a `rapid-dashboard doctor` command that checks all prerequisites (Python version, venv exists, dependencies installed, port available).
- **Detection:** The install script should verify Python availability and version before proceeding. Add a pre-flight check that imports all required packages.
- **References:** Project-specific concern based on the existing Node.js codebase at `/home/kek/Projects/RAPID/package.json`

### Pitfall: SPA Routing vs FastAPI API Route Conflicts
- **Category:** correctness
- **Stack:** Vite (frontend SPA), FastAPI (backend)
- **Description:** A Vite-built SPA uses client-side routing (e.g., `/projects`, `/kanban`, `/graph`). When the user directly navigates to these URLs (browser refresh, bookmark), the request goes to FastAPI, which does not have routes for `/projects` or `/kanban`. FastAPI returns 404. The fix (a catch-all route serving `index.html`) can conflict with API routes and static file serving if the mount order is wrong.
- **Trigger:** User refreshes the browser on any non-root URL. User bookmarks a dashboard page and opens it later. User shares a dashboard URL with another person.
- **Impact:** Severity HIGH. Users see 404 errors instead of the expected page. If the catch-all is misconfigured, it can intercept API calls (returning HTML instead of JSON) or prevent static assets from loading.
- **Prevention:**
  1. Prefix all API routes with `/api/` (e.g., `/api/projects`, `/api/state`). This creates a clean namespace separation.
  2. Mount static files (Vite build output) before the catch-all route but after API routes.
  3. Implement a custom `StaticFiles` subclass that falls back to `index.html` for any path that does not match a real file, rather than using a generic catch-all route.
  4. During development, use Vite's dev server proxy to forward `/api/*` requests to FastAPI, avoiding CORS issues entirely.
  5. In production, ensure the Vite `base` config is set correctly (usually `/`) so asset paths resolve properly.
- **Detection:** Test direct URL navigation for every frontend route after deployment. Verify that `/api/*` routes return JSON, not HTML.
- **References:**
  - [Serving a React Frontend with FastAPI](https://davidmuraya.com/blog/serving-a-react-frontend-application-with-fastapi/)
  - [FastAPI Discussion #5134: Vite + FastAPI via Docker](https://github.com/fastapi/fastapi/discussions/5134)

## Medium-Risk Pitfalls

### Pitfall: Knowledge Graph Rendering Performance with Large Codebases
- **Category:** performance
- **Stack:** Frontend (Vite/JS), graph visualization libraries (D3.js, vis.js, Cytoscape.js)
- **Description:** Large codebases can produce dependency graphs with hundreds or thousands of nodes and edges. Naive DOM-based rendering (SVG with D3) bogs down at ~500 nodes. Even Canvas/WebGL renderers struggle with force-directed layouts on 1000+ nodes due to O(n^2) force calculations per frame.
- **Trigger:** Registering a large project with many modules/files. Rendering the full codebase dependency graph rather than a filtered subset. Applying force-directed layout without spatial partitioning.
- **Impact:** Severity MEDIUM. The graph view becomes unusable -- browser tab freezes, high CPU usage, potential browser crash. Other dashboard views are unaffected.
- **Prevention:**
  1. Aggregate the graph server-side. Send a pre-computed layout or a summarized graph (module-level, not file-level) to the frontend.
  2. Implement level-of-detail: show module clusters at high zoom, expand to individual files on zoom-in.
  3. Use WebGL-based renderers (Sigma.js, Cytoscape.js with WebGL extension) for graphs exceeding 200 nodes.
  4. Set a hard node count limit. If the graph exceeds the limit, switch to a hierarchical tree layout (O(n log n)) instead of force-directed (O(n^2)).
  5. Paginate or virtualize: only render nodes visible in the current viewport.
- **Detection:** Measure render time for the graph view. If initial render exceeds 500ms or frame rate drops below 30fps, the graph is too large for the current renderer.
- **References:**
  - [Graph visualization efficiency -- PMC](https://pmc.ncbi.nlm.nih.gov/articles/PMC12061801/)
  - [Large dataset rendering in React](https://www.syncfusion.com/blogs/post/render-large-datasets-in-react)

### Pitfall: FastAPI Lifespan vs Deprecated Startup/Shutdown Events
- **Category:** correctness
- **Stack:** FastAPI 0.95+, Starlette
- **Description:** FastAPI deprecated `@app.on_event('startup')` and `@app.on_event('shutdown')` in favor of the `lifespan` context manager. The two mechanisms are mutually exclusive -- if a `lifespan` handler is defined, any `@app.on_event()` decorators silently fail. This is a trap for developers who copy examples from older tutorials or use third-party libraries that register their own startup events.
- **Trigger:** Defining a `lifespan` context manager for database/watcher initialization while also using `@app.on_event('startup')` for something else (e.g., a third-party middleware's initialization hook).
- **Impact:** Severity MEDIUM. Resources that should be initialized at startup are not, leading to NoneType errors when the first request arrives. The failure is silent -- no error is logged, the startup handler simply never runs.
- **Prevention:**
  1. Use only the `lifespan` context manager. Do not mix with `@app.on_event()`.
  2. Initialize all resources (database connection, file watcher, SSE manager) inside the `lifespan` async context manager.
  3. Store initialized resources on `app.state` and access them via dependency injection.
  4. Verify in tests using `with TestClient(app) as client:` to trigger lifespan events.
- **Detection:** Add assertions in the first request middleware that checks `app.state` for expected resources. If they are `None`, the lifespan handler did not run.
- **References:**
  - [FastAPI Lifespan Events](https://fastapi.tiangolo.com/advanced/testing-events/)
  - [FastAPI Discussion #9604: Lifespan causes deprecated events to silently fail](https://github.com/fastapi/fastapi/discussions/9604)

### Pitfall: CORS Misconfiguration Between Vite Dev Server and FastAPI
- **Category:** security / correctness
- **Stack:** FastAPI CORSMiddleware, Vite dev server
- **Description:** During development, the Vite dev server runs on port 5173 and the FastAPI backend on port 8998. Cross-origin requests from Vite to FastAPI are blocked by the browser's same-origin policy unless CORS is configured correctly. A common mistake is using `allow_origins=["*"]` during development and forgetting to restrict it for production, or misconfiguring the allowed methods/headers causing preflight failures.
- **Trigger:** Any fetch/XHR request from the Vite frontend (localhost:5173) to the FastAPI backend (localhost:8998) during development.
- **Impact:** Severity MEDIUM. Development workflow is blocked -- the frontend cannot communicate with the backend. If `allow_origins=["*"]` leaks to production, any website can make authenticated requests to the dashboard API.
- **Prevention:**
  1. Use Vite's built-in proxy during development: configure `server.proxy` in `vite.config.ts` to forward `/api/*` to `http://localhost:8998`. This avoids CORS entirely during development.
  2. In production, the FastAPI server serves both the API and the static frontend, so CORS is not needed (same origin).
  3. If CORS must be configured, restrict `allow_origins` to specific origins, not `["*"]`. Since this is a local service, `["http://localhost:8998"]` is sufficient.
  4. Ensure `CORSMiddleware` is added early in the middleware stack (before authentication or other middleware).
- **Detection:** Open browser DevTools Network tab. Failed CORS preflight requests appear as blocked/failed with a clear error message.
- **References:**
  - [FastAPI CORS Documentation](https://fastapi.tiangolo.com/tutorial/cors/)
  - [CORS Configuration in FastAPI](https://www.stackhawk.com/blog/configuring-cors-in-fastapi/)

### Pitfall: Stale Data in Multi-View Dashboard
- **Category:** correctness
- **Stack:** Frontend state management (React/Vue/Svelte), SSE/WebSocket
- **Description:** The dashboard has multiple views (kanban, knowledge graph, project list, notes, worktree tracker) that all consume the same underlying project data. If one view updates (e.g., a file watcher detects a state change) but other views are not notified, the user sees inconsistent data across tabs/views. React's `useRef` vs `useState` distinction makes this worse: state captured in closures may point to stale snapshots.
- **Trigger:** Switching between dashboard views while a project's state is changing. Having multiple browser tabs open to different views. Receiving an SSE update while viewing a different tab (the update handler may not re-render the currently visible view).
- **Impact:** Severity MEDIUM. Users see outdated information. In a kanban board, a task appears in the wrong column. In the project list, a set appears as "executing" when it is already "complete." Users lose trust in the dashboard accuracy.
- **Prevention:**
  1. Use a centralized state store (Zustand, Pinia, or Svelte stores) that all views subscribe to. Updates propagate to all active views automatically.
  2. For SSE/WebSocket updates, update the central store, not individual component state.
  3. Implement a "last updated" timestamp visible in the UI so users know when data was last refreshed.
  4. Add a manual "refresh" button as a safety valve.
  5. When a view becomes visible (tab focus, route navigation), trigger a data re-fetch to ensure freshness.
- **Detection:** Compare data displayed in two different views of the same project. If they disagree, the state management has a staleness bug.
- **References:**
  - [Real-Time Web Apps: WebSockets and SSE Guide](https://www.debutinfotech.com/blog/real-time-web-apps)

### Pitfall: Vim Keybinding Conflicts with Browser and Form Inputs
- **Category:** correctness / compatibility
- **Stack:** Frontend, browser keyboard event handling
- **Description:** If the dashboard implements vim-style keyboard shortcuts (hjkl navigation, `:` command mode, `/` search), these conflict with browser defaults (Ctrl+F for find, `/` for quick find in Firefox), browser extensions (Vimium, which many target users will have), accessibility features (screen reader shortcuts), and form inputs (typing "j" in a search box triggers navigation instead of inserting the character).
- **Trigger:** User types in any text input, textarea, or the markdown note editor. User has Vimium/Tridactyl installed. User uses a screen reader. User presses Escape (which may both exit vim mode and trigger browser behavior like closing fullscreen).
- **Impact:** Severity MEDIUM. Text editing becomes impossible in the markdown editor. Search inputs are unusable. Accessibility is broken. Users with browser vim extensions experience double-handling of keystrokes.
- **Prevention:**
  1. Disable all custom keyboard shortcuts when focus is inside an input, textarea, or contenteditable element. Check `document.activeElement.tagName` before handling keyboard events.
  2. Make vim keybindings opt-in, not default. Most dashboard users will not expect vim navigation in a web UI.
  3. Use a library like `hotkeys-js` or `mousetrap` that handles focus detection and modifier key management.
  4. Never override Ctrl/Cmd+key combinations that browsers reserve (Ctrl+C, Ctrl+V, Ctrl+F, Ctrl+T, Ctrl+W).
  5. Provide a clear visual indicator of the current mode (normal/insert) if implementing a modal keyboard system.
- **Detection:** Test the dashboard with Vimium enabled. Test typing in all text inputs with vim mode active. Run a keyboard accessibility audit.
- **References:**
  - [Vimium -- GitHub](https://github.com/philc/vimium)
  - [Vim key bindings for web browsers -- Vim Tips Wiki](https://vim.fandom.com/wiki/Vim_key_bindings_for_web_browsers)

### Pitfall: XDG Base Directory Non-Compliance
- **Category:** compatibility / correctness
- **Stack:** Python, filesystem, Linux/macOS
- **Description:** The spec places the dashboard at `~/.rapid/`. While `~/.rapid/` is a reasonable conventional location, it does not follow the XDG Base Directory Specification, which expects config in `~/.config/rapid/`, data in `~/.local/share/rapid/`, cache in `~/.cache/rapid/`, and runtime files in `$XDG_RUNTIME_DIR/rapid/`. On some Linux distributions, tools and cleanup scripts may not account for dotfiles in `$HOME` that are not XDG-compliant.
- **Trigger:** Users who have strict home directory policies. Backup tools that respect XDG directories but skip arbitrary dotfiles. NixOS or other declarative systems that manage `$HOME` contents.
- **Impact:** Severity LOW-MEDIUM. Not a functional failure, but causes friction with users who expect XDG compliance. The SQLite database, virtual environment, and configuration all end up in a single directory, making selective backup/cleanup harder.
- **Prevention:**
  1. Use `~/.rapid/` as the single conventional location (as the spec dictates) but document this choice.
  2. If future versions want XDG compliance, split: config (`~/.config/rapid/`), database/data (`~/.local/share/rapid/`), logs/cache (`~/.cache/rapid/`), runtime (`$XDG_RUNTIME_DIR/rapid/`).
  3. For now, keep it simple. `~/.rapid/` is easier to reason about and the target audience (Claude Code users / developers) will not be confused by it.
  4. Ensure all paths use `os.path.expanduser("~")` in Python, never rely on `$HOME` environment variable directly (which may not be set in systemd service contexts).
- **Detection:** Verify that the service works when `$HOME` is set but `$XDG_*` variables are not (common on macOS).
- **References:**
  - [XDG Base Directory -- ArchWiki](https://wiki.archlinux.org/title/XDG_Base_Directory)
  - [XDG Base Directory Specification](https://specifications.freedesktop.org/basedir/latest/)

## Anti-Patterns for This Stack

### Anti-Pattern: Multiple SQLite Connections from Multiple Writers
- **What people do:** Create a new SQLite connection per request handler (like traditional ORMs with PostgreSQL/MySQL), allowing concurrent writes from multiple connections.
- **Why it fails:** SQLite uses a database-level write lock. Multiple writers contend on this lock, causing `SQLITE_BUSY` errors and worse throughput than a single-writer setup. Connection pools exacerbate this because the pool spawns multiple writers that all contend.
- **Correct approach:** Use a single writer pattern. All write operations go through one dedicated connection (or a write queue). Multiple reader connections are fine with WAL mode enabled.
- **Example:** Create a `WriteQueue` class that accepts write operations as async callables and executes them serially on a single connection.

### Anti-Pattern: Recursive Directory Watching for State Changes
- **What people do:** Use `watchdog.Observer` with `recursive=True` on the project root or `.planning/` directory to catch any state change.
- **Why it fails:** Recursive watching creates an inotify watch for every subdirectory, consuming watches quadratically as projects grow. It also generates massive amounts of irrelevant events (editor temp files, git index changes, node_modules updates).
- **Correct approach:** Watch specific files: `.planning/STATE.json`, `.planning/worktrees/REGISTRY.json`. Poll for structural changes (new sets, new contracts) on a timer rather than watching entire directory trees.
- **Example:** `observer.schedule(handler, path="/project/.planning/STATE.json", recursive=False)`

### Anti-Pattern: Serving Vite Build Output with Generic Static File Mount
- **What people do:** Mount the entire Vite `dist/` directory as a `StaticFiles` app at `/` and assume client-side routing works.
- **Why it fails:** Direct navigation to `/projects/my-project` returns 404 because there is no `projects/my-project/index.html` file in the dist directory. The SPA catch-all route must serve `index.html` for all non-file, non-API paths.
- **Correct approach:** Subclass `StaticFiles` to override `get_response()`, falling back to `index.html` when the requested path does not match a real file. Or use a catch-all route mounted after API routes and static file mounts.
- **Example:**
```python
@app.get("/{path:path}")
async def spa_catch_all(path: str):
    file_path = DIST_DIR / path
    if file_path.is_file():
        return FileResponse(file_path)
    return FileResponse(DIST_DIR / "index.html")
```

### Anti-Pattern: Using SSE for Bidirectional Communication
- **What people do:** Use Server-Sent Events for real-time updates (correct) but then also try to use them for client-to-server communication (incorrect), or add WebSockets for features that only need server-to-client push.
- **Why it fails:** SSE is unidirectional (server to client only). Adding WebSockets for a dashboard that only needs push updates adds unnecessary complexity (connection management, heartbeats, reconnection logic). For a read-only dashboard, SSE is simpler and sufficient.
- **Correct approach:** Use SSE for pushing state updates from the backend to the frontend. Use regular HTTP POST/PUT for the rare client-to-server interactions (registering a project, updating notes). SSE handles reconnection automatically via the `EventSource` API.
- **Example:** FastAPI's `StreamingResponse` with `media_type="text/event-stream"` for SSE endpoints.

### Anti-Pattern: Polling STATE.json from the Frontend
- **What people do:** Set up a `setInterval` in the frontend that fetches `/api/state` every N seconds to detect changes.
- **Why it fails:** Polling at short intervals (1-2s) creates unnecessary load on both the server and the SQLite database. Polling at long intervals (10-30s) makes the dashboard feel stale. Neither interval is optimal, and polling does not scale with the number of projects.
- **Correct approach:** The backend watches STATE.json via file system events and pushes updates to connected clients via SSE. The frontend subscribes to the SSE stream and updates the UI reactively. Zero polling, instant updates.

## Version-Specific Gotchas

- **Python 3.10-3.11:** `asyncio.TaskGroup` is only available in 3.11+. If targeting 3.10, use `asyncio.gather()` instead.
- **Python 3.12:** Changed the default event loop policy. `asyncio.get_event_loop()` now raises a `DeprecationWarning` if no event loop is running. Use `asyncio.get_running_loop()` inside async contexts.
- **FastAPI 0.100+:** Pydantic v2 is required. If any dependency pins Pydantic v1, there will be an import conflict. Ensure all dependencies are Pydantic v2 compatible.
- **SQLite 3.34 (ships with Python 3.9):** WAL mode works but `RETURNING` clause is not supported until SQLite 3.35. If using `INSERT ... RETURNING`, ensure the bundled SQLite version supports it.
- **Vite 6.x (2025+):** Changed the default module resolution strategy. Ensure `vite.config.ts` explicitly sets `resolve.conditions` if using server-side rendering or hybrid rendering.
- **uvicorn 0.30+:** The `--reload-dir` flag behavior changed. Without explicit `--reload-dir`, uvicorn watches the current working directory recursively, which can be slow if run from a large project root.

## Security Pitfalls

1. **Unprotected local service:** The dashboard runs on port 8998 with no authentication. Any local process or user on the same machine can access it. If the machine is shared or the port is accidentally exposed (e.g., via SSH tunnel, Docker port forwarding), project data is readable by unauthorized parties. -- **Prevention:** Bind to `127.0.0.1` only (never `0.0.0.0`). Add optional token-based auth for multi-user machines.
2. **Path traversal via project registration:** If the `/api/register` endpoint accepts arbitrary paths and the backend reads files from those paths, a malicious registration could read arbitrary files (e.g., `../../etc/passwd`). -- **Prevention:** Validate and canonicalize all paths. Ensure the path contains a `.planning/` directory before accepting it.
3. **SQL injection via project names:** If project names or set names are interpolated into SQL queries instead of parameterized, injection is possible. -- **Prevention:** Always use parameterized queries (`?` placeholders). Never f-string or %-format SQL.
4. **Markdown XSS in note editor:** If the markdown note editor renders user-supplied markdown to HTML without sanitization, `<script>` tags or `javascript:` URLs in markdown links can execute arbitrary code. -- **Prevention:** Use a sanitizing markdown renderer (e.g., `DOMPurify` on the frontend, `bleach` on the backend). Disable raw HTML in markdown parsing.
5. **Sensitive data in SQLite database:** The SQLite database at `~/.rapid/dashboard.db` contains project paths, state information, and user notes. If the file permissions are too open (world-readable), other users on the system can read this data. -- **Prevention:** Set file permissions to `0600` on the database file. Verify permissions on startup.

## Performance Traps

1. **Full state re-serialization on every change:** If the file watcher triggers a full re-read and re-serialize of STATE.json for every file change event, rapid successive changes (common during batch operations like `execute-set`) cause a storm of reads. -- **Mitigation:** Debounce file change events (100-500ms window). Only process the latest event after the debounce period.
2. **Unbounded SSE connections:** Each open browser tab holds an SSE connection. If the user opens many tabs or a monitoring tool opens many connections, the backend accumulates connections that consume file descriptors and memory. -- **Mitigation:** Limit SSE connections per client (by IP or session token). Set a maximum connection limit (e.g., 50 total). Close idle connections after a timeout.
3. **Frontend re-rendering entire graph on any state change:** If the knowledge graph re-runs layout computation whenever any project state changes (not just graph-related changes), CPU usage spikes unnecessarily. -- **Mitigation:** Separate graph data from project state in the store. Only re-render the graph when graph-specific data changes.
4. **SQLite without VACUUM:** Over time, deleted rows leave fragmentation in the SQLite database file. The file grows but never shrinks. -- **Mitigation:** Run `PRAGMA auto_vacuum=INCREMENTAL;` at database creation, and periodically run `PRAGMA incremental_vacuum;` in a background task.
5. **Python virtual environment in ~/.rapid/:** The venv can be 50-200MB depending on dependencies. If `~/.rapid/` is on a small partition or a network-mounted home directory, this causes space issues. -- **Mitigation:** Document the space requirement. Allow configuring the venv location via environment variable.

## Summary Priority List

1. **SQLite write contention** -- concurrent writes from file watcher, API, and CLI will cause data loss without WAL mode + busy_timeout + single-writer pattern
2. **Blocking the event loop** -- synchronous file/DB operations in async routes will freeze the entire dashboard for all users
3. **inotify watch limit exhaustion** -- watching multiple projects recursively will hit kernel limits and silently stop detecting changes
4. **systemd user service dies on logout** -- without lingering enabled, the background service stops when the user logs out
5. **Watchdog memory leak** -- InotifyObserver leaks memory over days/weeks of continuous operation, eventually OOM-killing the service
6. **Dual-runtime installation complexity** -- requiring both Node.js and Python creates a fragile installation process
7. **SPA routing 404s** -- browser refresh on any non-root URL returns 404 without proper catch-all configuration
8. **Cross-platform service management** -- systemd-only implementation locks out macOS and Windows users
9. **Knowledge graph rendering performance** -- large codebases produce graphs that freeze the browser
10. **FastAPI lifespan silent failures** -- mixing deprecated startup events with lifespan handlers causes resources to not initialize
11. **CORS misconfiguration** -- blocks all frontend-backend communication during development
12. **Stale data across dashboard views** -- users see inconsistent state in different tabs/views
13. **Vim keybinding conflicts** -- custom keyboard shortcuts break text editing, accessibility, and browser extension compatibility
14. **Unprotected local service** -- no authentication on a network-accessible port exposes project data
15. **Markdown XSS** -- unsanitized user-authored markdown in the note editor enables script injection
