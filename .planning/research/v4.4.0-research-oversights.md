# Oversights Research: v4.4.0 Polish & Documentation

## Cross-Cutting Concerns

### 1. Review Pipeline State Persistence -- Schema and Storage Design

The milestone's primary feature is "store the state of the review pipeline so that subsequent commands don't re-ask the user to run earlier stages." Currently, the review pipeline is stateless -- `/rapid:review` produces `REVIEW-SCOPE.md`, then `/rapid:unit-test`, `/rapid:bug-hunt`, and `/rapid:uat` each independently check for `REVIEW-SCOPE.md` and produce their respective artifacts (`REVIEW-UNIT.md`, `REVIEW-BUGS.md`, `REVIEW-UAT.md`). There is no persistent record of which stages have been completed.

**What gets overlooked:**

1. **Where to store review state.** The obvious candidates are:
   - A new `REVIEW-STATE.json` file per set (in `.planning/sets/{setId}/` or `.planning/post-merge/{setId}/`)
   - A `reviewState` field on the `SetState` Zod schema in `state-schemas.cjs`
   - A standalone review state file next to REVIEW-SCOPE.md

   Adding a field to `SetState` in `state-schemas.cjs` triggers a schema version bump, which requires a migration in `migrate.cjs`. A standalone file avoids this but introduces a second state source that can desync from STATE.json. The decision ripples into every review skill.

2. **Post-merge vs standard path divergence.** Review artifacts live in two locations: `.planning/sets/{setId}/` for standard reviews and `.planning/post-merge/{setId}/` for post-merge reviews. The review state must handle both paths. The auto-detect logic in `unit-test`, `bug-hunt`, and `uat` SKILL.md files already has dual-path checking -- adding state persistence doubles this complexity.

3. **Partial completion semantics.** If bug-hunt ran but found 0 issues vs bug-hunt ran and found 5 issues that were all fixed -- both are "complete" but with different confidence levels. The state schema needs to capture at minimum: `{stage: string, status: 'pending'|'complete'|'skipped', completedAt: string}`, but might also need `issuesFound`, `issuesFixed`, etc. for downstream skills to make informed decisions.

4. **Compaction hook integration.** `compaction.cjs` already has a `review-stage-complete` event registered as a no-op placeholder (line 350). This was designed for exactly this use case but never wired up. The review state feature should fire this trigger when a review stage completes.

**Why it matters for this milestone:** This is the core feature. Getting the state schema wrong means either a painful migration later or a schema that cannot express what downstream skills need to know. The compaction hook is already waiting -- not connecting it is a missed integration point.

**Recommended action:** Use a standalone `REVIEW-STATE.json` per set (not a SetState schema change) to avoid triggering a state migration. Structure it as:
```json
{
  "setId": "...",
  "scope": { "completedAt": "...", "postMerge": false },
  "unitTest": { "status": "complete", "completedAt": "...", "summary": "..." },
  "bugHunt": { "status": "pending" },
  "uat": { "status": "skipped" }
}
```
Each review skill reads and writes this file. Wire the `review-stage-complete` compaction trigger.

### 2. Review State -- Idempotency and Re-entry

Once review state is persisted, each review skill must decide what to do when invoked for a stage that is already "complete." Three options exist:

1. **Skip silently** -- show "unit-test already completed, skipping" and exit
2. **Confirm re-run** -- ask the user "Unit tests already passed. Re-run?" via AskUserQuestion
3. **Always re-run** -- ignore prior state, re-execute, update the state

The milestone brief says "don't re-ask the user to run earlier stages." This implies option 1 (skip) for prerequisite stages, but what about the current stage? If the user explicitly runs `/rapid:bug-hunt` and bug-hunt is already complete, they probably want to re-run. If they run `/rapid:uat` and unit-test/bug-hunt are complete, those should be skipped.

**What gets overlooked:** The distinction between "I am running this stage" vs "this stage is a prerequisite for what I'm running" is not currently modeled. Each skill is standalone -- there is no orchestrator that chains them. The state file enables detection but each skill needs consistent skip/re-run logic.

**Recommended action:** Each skill should:
- Check REVIEW-STATE.json for its own stage: if complete, ask "Re-run? Results from {date} exist."
- Check prerequisite stages: if REVIEW-SCOPE.md missing, error. If upstream stage not complete, warn but allow proceeding (user may have intentionally skipped bug-hunt to go straight to UAT).

### 3. Display Color Change -- Test Suite Impact

The milestone asks to "change bright agent colours to dark colours that play well with white text on dark terminals." Currently `display.cjs` uses bright background ANSI codes (10Xm range): `\x1b[104m` (bright blue), `\x1b[102m` (bright green), `\x1b[101m` (bright red). The milestone wants dark variants (40-47 range): `\x1b[44m` (dark blue), `\x1b[42m` (dark green), `\x1b[41m` (dark red).

**What gets overlooked:**

1. **`display.test.cjs` has hardcoded ANSI assertions.** Lines 76, 88, 100 explicitly assert `\x1b[104m`, `\x1b[102m`, `\x1b[101m` for planning, execution, and review stages respectively. Lines 278, 290, 302 test banner rendering with the same codes. Changing the colors without updating these 6+ assertions will break tests immediately.

2. **Agent `color` field is separate.** The `ROLE_COLORS` in `build-agents.cjs` use named colors (`blue`, `green`, `red`, `cyan`, `yellow`, `purple`) that are interpreted by the Claude Code plugin runtime, not by RAPID's display.cjs. These are two completely independent color systems. The milestone brief says "agent colours" which could mean either or both. Clarification is needed: does "agent colours" mean the banner background colors in display.cjs, or the `color:` field in agent frontmatter, or both?

3. **NO_COLOR standard.** The current display.cjs comment explicitly says "Always outputs colors (no NO_COLOR checking)" (line 7). If the goal is better terminal compatibility, the standard `NO_COLOR` environment variable (https://no-color.org/) should be respected. This is a separate concern from changing bright-to-dark but is commonly forgotten during color refactoring.

4. **`audit-version` color was just added.** The v4.2.0 milestone added `audit-version: '\x1b[101m'` (bright red). If the color scheme changes, this needs updating too. The `STAGE_BG` object has 17 entries total -- all must be changed consistently.

**Why it matters:** Color changes seem trivial but the test assertions make them a guaranteed test failure if not coordinated. The agent color vs banner color ambiguity could lead to changing the wrong thing.

**Recommended action:** Update display.cjs and display.test.cjs in the same task. Clarify whether "agent colours" refers to banner backgrounds, agent frontmatter colors, or both. Consider adding NO_COLOR support as a low-cost addition.

### 4. Documentation Scope -- What Exists vs What Is Needed

The milestone asks for "documentation for open-sourcing" covering the canonical workflow. Current documentation state:

**Already exists (docs/):**
- `setup.md` -- covers `/install`, `/init`, `/context`, `/register-web`
- `planning.md` -- covers `/start-set`, `/discuss-set`, `/plan-set`
- `execution.md` -- covers `/execute-set`
- `review.md` -- covers `/review`, unit-test, bug-hunt, uat
- `merge-and-cleanup.md` -- covers `/merge`, `/cleanup`
- `agents.md` -- agent architecture
- `configuration.md` -- config.json options
- `troubleshooting.md` -- common issues
- `state-machines.md` -- state transition diagrams
- `CHANGELOG.md` -- version history

**Missing or incomplete:**
- No top-level `README.md` at the project root (critical for open-source)
- No documentation for `/rapid:branding`
- No documentation for `/rapid:scaffold`
- No documentation for `/rapid:audit-version`
- No documentation for `/rapid:new-version`
- No documentation for `/rapid:quick`
- No documentation for `/rapid:bug-fix`
- No documentation for `/rapid:add-set`
- `help/SKILL.md` references v4.3.0 but documentation links point to existing docs, not missing commands
- No "Getting Started" or "Quick Start" guide for new users

**What gets overlooked:**

1. **README.md is the open-source front door.** Without it, GitHub shows the raw file list. A README needs: project description, installation, quick start, link to full docs, license badge, contributing guidelines.

2. **The help command (`/rapid:help`) doubles as inline documentation.** It must stay in sync with the docs/ folder. Currently help/SKILL.md lists all commands but the docs/ folder does not cover all of them. Updating docs without updating help creates drift.

3. **Screenshot/terminal-recording for the workflow.** Open-source documentation for CLI tools benefits enormously from a single terminal recording or screenshot showing the workflow in action. Text-only docs are harder to grok for newcomers.

**Why it matters:** Documentation is the highest-visibility artifact for open-sourcing. Missing a README is immediately visible. Missing individual command docs is discoverable. The help command sync issue creates long-term maintenance burden.

**Recommended action:** Prioritize README.md creation first. Then fill missing command docs. Update help/SKILL.md version number. Consider whether docs should live in docs/ (current) or a separate documentation site.

### 5. Documentation -- Canonical Workflow Order Accuracy

The milestone brief specifies the canonical workflow as: `/install > /init > /branding > /scaffold > /start-set > /discuss-set > /plan-set > /execute-set > /review > /merge > /audit-version > /new-version`. But the existing help/SKILL.md shows a different grouping:

- 7 Core Lifecycle Commands (init through merge)
- 3 Review Pipeline Commands (unit-test, bug-hunt, uat)
- Utility Commands (quick, add-set, scaffold, branding, etc.)

**What gets overlooked:**

1. **`/branding` and `/scaffold` placement.** These are listed in the canonical flow between `/init` and `/start-set`, but in practice `/branding` generates agent visual identity (optional) and `/scaffold` sets up project structure (also optional). The documentation must clarify that these are optional steps, not mandatory gates.

2. **`/audit-version` and `/new-version` are post-milestone commands.** They operate after all sets are merged, outside the per-set lifecycle. The documentation should separate the per-set loop from the milestone-level lifecycle.

3. **Review sub-skills ordering.** The canonical flow shows `/review` as a single step, but it actually requires running `/review` (scoping) then optionally `/unit-test`, `/bug-hunt`, and `/uat` as separate commands. The documentation must make this pipeline visible without overwhelming new users.

**Recommended action:** Document two levels -- a high-level "milestone lifecycle" (init > branding? > scaffold? > [start-set > discuss-set > plan-set > execute-set > review > merge] x N > audit-version > new-version) and a detailed per-command reference. Mark optional steps clearly.

### 6. Deferred Decisions -- Continuous Audit Hook

The milestone explicitly lists two deferred decisions:
1. Continuous audit hook -- post-merge hook that auto-audits after each set completes
2. New-version integration -- researchers should auto-discover DEFERRED.md

**What gets overlooked:** The deferred decisions section says these are deferred to future milestones, but the documentation work in this milestone could accidentally document these features as if they exist. When writing docs for `/audit-version`, the documentation should describe what it does today (manual invocation) without implying automatic post-merge triggering exists.

Similarly, when documenting `/new-version`, the docs should note that DEFERRED.md items must be manually reviewed, not that they are auto-discovered.

**Recommended action:** In documentation for audit-version and new-version, add a "Planned enhancements" or "Roadmap" subsection that mentions these deferred capabilities without documenting them as current features.

### 7. Review State -- Interaction with Solo Sets and Post-Merge Mode

Solo sets (created via `/rapid:quick` or with solo flag) auto-merge during execution and bypass the normal complete-then-review-then-merge flow. The review skill already handles this with auto-detection of solo+merged status. Adding review pipeline state introduces a new concern:

**What gets overlooked:** Solo sets that auto-merge may never have had `/rapid:review` run on them at all. If a user later runs `/rapid:unit-test solo-set-name`, the skill expects REVIEW-SCOPE.md to exist. Currently it errors with "REVIEW-SCOPE.md not found." With review state, should it also auto-scope (run the equivalent of `/rapid:review`) before proceeding?

The review-auto-detect.test.cjs tests verify the dual-path checking (standard and post-merge paths) but do not test what happens when neither path has REVIEW-SCOPE.md. The new state file adds a third thing to check.

**Recommended action:** Define the behavior: if REVIEW-STATE.json does not exist and REVIEW-SCOPE.md does not exist, either auto-scope or provide a clear error saying "Run /rapid:review first." Do not introduce silent auto-scoping without user confirmation.

### 8. Banner Stage Registration for Missing Sub-Skills

The `STAGE_VERBS` and `STAGE_BG` maps in display.cjs do not include entries for `unit-test`, `bug-hunt`, or `uat`. These skills call `node "${RAPID_TOOLS}" display banner unit-test` (etc.) in their SKILL.md files. This means they currently get the fallback `[RAPID] Unknown stage: unit-test` instead of a branded banner.

**What gets overlooked:** This is an existing bug, not a new concern introduced by this milestone. But since the milestone touches display.cjs for color changes, it is the ideal time to also add the missing stage registrations.

**Recommended action:** Add these entries to STAGE_VERBS and STAGE_BG:
- `'unit-test': 'UNIT TESTING'` (review stage color)
- `'bug-hunt': 'BUG HUNTING'` (review stage color)
- `'uat': 'ACCEPTANCE TESTING'` (review stage color)
- `'bug-fix': 'BUG FIXING'` (execution stage color)

### 9. Documentation Testing -- Verifying Code Examples Work

The documentation will contain bash commands and workflow examples. These are instructions for humans, not executable code. However, if a documented command references a CLI subcommand that does not exist or has a different name, the documentation is worse than useless.

**What gets overlooked:** There is no mechanism to verify that documented commands match the actual CLI interface. For example, `rapid-tools.cjs` has specific subcommands and argument formats. If documentation says `node rapid-tools.cjs review pipeline-status` but the actual subcommand is `review list-issues`, users will hit errors.

**Recommended action:** After writing documentation, run a quick grep to verify that every `node "${RAPID_TOOLS}"` command in the docs corresponds to an actual command handler in `rapid-tools.cjs`. This can be a manual verification step or a lightweight test.

### 10. CHANGELOG and Version Bump Coordination

This milestone is v4.4.0. The existing CHANGELOG.md and help/SKILL.md reference v4.3.0. When shipping this milestone:

1. CHANGELOG.md needs a v4.4.0 entry
2. help/SKILL.md header says "RAPID v4.3.0 Workflow" -- needs updating
3. `version.cjs` returns the current version -- needs bumping
4. `package.json` version field needs updating

**What gets overlooked:** Version bumps are typically done last but the documentation explicitly references version numbers. If docs are written with v4.3.0 references and the version bump happens in a different set, there will be a race to update. The version bump should happen early or documentation should use a placeholder.

**Recommended action:** Include version bump in the documentation set, not as a separate task. Update all version references atomically.

## Recommendations

Ordered by "cost of getting it wrong" -- items where a wrong early decision causes the most rework:

1. **Review state storage design (concern 1):** Choose standalone REVIEW-STATE.json over SetState schema extension to avoid a state migration. Design the schema before any skill modifications. Wire the existing `review-stage-complete` compaction trigger. -- *First step: Create the REVIEW-STATE.json Zod schema in review.cjs and add read/write helper functions.*

2. **Review state idempotency rules (concern 2):** Define skip/re-run behavior for each stage before modifying skill files. The rule should be: "prerequisite stages skip silently; current stage asks for confirmation if already complete." -- *First step: Add a `checkReviewState(setId, stage)` function that returns `{shouldRun: bool, reason: string}` for each skill to call.*

3. **Display color + test update (concern 3):** Change all 17 STAGE_BG entries and all 6+ display.test.cjs assertions in one atomic task. Also add the 4 missing stage registrations (concern 8) since you are already editing the same files. -- *First step: Map current bright codes to dark equivalents: `\x1b[104m` -> `\x1b[44m`, `\x1b[102m` -> `\x1b[42m`, `\x1b[101m` -> `\x1b[41m`.*

4. **README.md creation (concern 4):** Write the project README before detailed command documentation. Include: project description, installation (`/rapid:install`), quick start (the canonical workflow), link to docs/, license. -- *First step: Draft README.md with sections: What is RAPID, Installation, Quick Start, Full Documentation, Contributing, License.*

5. **Canonical workflow documentation (concern 5):** Document two tiers -- milestone lifecycle and per-command reference. Mark `/branding` and `/scaffold` as optional. Separate the per-set loop from post-milestone commands (`/audit-version`, `/new-version`). -- *First step: Create a workflow diagram in the README showing the two-tier structure.*

6. **Version bump (concern 10):** Include version bump in the first set that touches help/SKILL.md or docs/. -- *First step: Add version bump to the documentation set's task list.*

7. **Solo set review state edge case (concern 7):** After review state is implemented, add explicit handling for "no REVIEW-STATE.json exists yet" in each review sub-skill. Error clearly rather than auto-scoping. -- *First step: Add a guard clause at the top of each review sub-skill's Step 1.*
