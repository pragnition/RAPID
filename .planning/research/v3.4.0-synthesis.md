# Research Summary -- v3.4.0 Agent Intelligence

## Project Profile
- **Type:** brownfield CLI plugin (Claude Code plugin for multi-agent development orchestration)
- **Stack:** Node.js (CommonJS) + Zod + Ajv + proper-lockfile, no build step, no bundler
- **Scale:** medium (7 features, ~37 sub-features, 21 existing library modules, 40 test files)
- **Key Constraint:** All new features must extend the existing 5-layer architecture (Skill > Agent > CLI > Library > Storage) without restructuring it, and context window budget pressure means enriched context must be carefully token-budgeted (compaction.cjs DEFAULT_BUDGET_TOKENS = 120000)

## Unified Findings

### Stack and Infrastructure

- **Node 18 is EOL (since April 2025):** The project declares Node 18+ minimum in `setup.sh` but Node 18 has been end-of-life for nearly a year. No `engines` field in package.json, no `.nvmrc`/`.node-version`. Bump minimum to Node 20, add `"engines": { "node": ">=20" }` to package.json. (sources: STACK, OVERSIGHTS)
- **Zod 3.25.76 is safe but watch for instability:** Zod 3.25.x bundles both Zod 3 and Zod 4 internally, doubling package size. CJS loading issues are reported for Zod 4. The `^3.25.76` semver range is safe (will not auto-upgrade to v4), but a fresh `npm install` could pull a different 3.25.x patch. Recommendation: stay on 3.25.76, consider pinning exact version. Defer full Zod 4 migration to a future milestone. (sources: STACK, PITFALLS)
- **proper-lockfile is unmaintained but stable:** Last published ~5 years ago, but functionally correct for local filesystem locking. Abstracted behind `lock.cjs` which provides a clean replacement boundary. No action needed. (sources: STACK)
- **Ajv 8.18.0 is current:** No issues. The `require('ajv').default` CJS import is fragile but correct for v8. Consider centralizing Ajv instantiation into a single module to avoid duplication when UI contracts add more validation. (sources: STACK, PITFALLS)
- **No hooks.json exists:** The existing `rapid-task-completed.sh` hook script is not wired into the plugin hook system via `hooks/hooks.json`. This is a gap that must be resolved before building the hooks feature. (sources: STACK, OVERSIGHTS)
- **No npm test script:** `package.json` has zero scripts. Tests run ad-hoc via `node --test`. This gap has been identified in two consecutive milestones and remains unaddressed. (sources: OVERSIGHTS)
- **Configuration is fragmented across 5 locations:** Version stored in 3 places (`.claude-plugin/plugin.json`, `.planning/config.json`, `package.json`) with `.planning/config.json` already stale at version 3.0.0. No unified config reader exists. (sources: OVERSIGHTS)
- **CommonJS module system is correct and should not change:** 100% CJS with `.cjs` extensions, deliberate choice that simplifies plugin ecosystem. All 4 dependencies support CJS. (sources: STACK)
- **No build step needed:** Source files run directly. Agent `.md` files assembled by `build-agents` command. This is the right approach for a CLI plugin. (sources: STACK)
- **Zero CVEs, minimal dependency tree:** 4 direct dependencies, 7 transitive. `npm audit` clean. (sources: STACK)

### Feature Landscape

Seven features are identified, all structurally independent (no hard blocking dependencies), with beneficial soft synergies:

**Trivial complexity:**
1. **Quick task logging** -- Add `.planning/quick/LOG.jsonl` append-only log, CLI query commands, SKILL.md update. Establishes the JSONL logging pattern reused by memory. (source: FEATURES)
2. **add-set enhancement** -- Move logic into CLI command `state add-set`, add DAG recalculation, OWNERSHIP.json regeneration. Existing skill needs audit for edge cases. (source: FEATURES)

**Moderate complexity:**
3. **Hooks (state verification)** -- Node.js hook script reusing existing state-machine.cjs, RAPID:RETURN parsing, remediation prompts. Must avoid deadlock on STATE.json lock. (source: FEATURES)

**Complex:**
4. **Memory system** -- JSONL decision log, mid-execution change flow, context injection, cross-milestone preferences. Foundational for code quality and documentation. (source: FEATURES)
5. **Code quality (richer context)** -- Quality profile system, enhanced context injection, pattern library, quality gates. Depends on memory for decision injection. (source: FEATURES)
6. **UI specs/contracts** -- Separate UI-CONTRACT.json per set, Ajv validation, cross-set consistency checks, context injection into executors. (source: FEATURES)
7. **Documentation generation** -- Agent-driven with template scaffolding, changelog extraction, diff-aware updates. Best done last to document other features. (source: FEATURES)

**Dependency graph:**
- memory -> code-quality (soft dependency: quality agents benefit from decision history)
- memory -> quick-logging (soft synergy: shared JSONL infrastructure)
- hooks -> memory (soft synergy: hooks could log to memory)
- documentation -> code-quality (soft synergy: better context improves doc accuracy)
- All other pairs are independent

**Third-party services:** None required. All features implementable with existing deps. jq used in existing bash hooks but new hooks should prefer Node.js to avoid the dependency.

(sources: FEATURES, ARCHITECTURE, OVERSIGHTS)

### Architecture Direction

- **Recommended pattern: Event-driven module extension with persistence layer.** The 7 features all extend the existing layered architecture rather than replacing it. The common thread is: capture more information at lifecycle transition points (events), persist durably (storage), and make available to downstream agents (context enrichment). (source: ARCHITECTURE)
- **Module organization follows existing conventions:** New modules (`hooks.cjs`, `memory.cjs`, `documentation.cjs`, `quick-log.cjs`, `ui-spec.cjs`) slot into `src/lib/` with corresponding command handlers in `src/commands/`. No new architectural concepts needed. (source: ARCHITECTURE)
- **Memory as append-only JSONL:** `.planning/memory/` directory with DECISIONS.jsonl and CORRECTIONS.jsonl. Append-only for durability (no data loss on crash), no database dependency. Integrates with existing compaction pipeline for token budgeting. (sources: ARCHITECTURE, FEATURES, OVERSIGHTS)
- **Hook system via dependency injection:** Avoid circular requires between hooks.cjs and state-machine.cjs. The CLI layer wires `hooks.executeHooks` as a callback passed to `state-machine.withStateTransaction`. Keeps modules independently testable. (source: ARCHITECTURE)
- **CONTRACT.json UI extension:** Add optional `ui` section to existing contract schema. Backward compatible -- sets without UI simply omit it. Alternatively, separate `UI-CONTRACT.json` per set for clean separation. (sources: ARCHITECTURE, FEATURES)
- **Lazy initialization for new .planning/ subdirectories:** All new directories (`.planning/memory/`, `.planning/hooks/`, etc.) created on first use, not during init. Ensures backward compatibility with existing v3.3.0 projects. (source: ARCHITECTURE)
- **Token budgets for enriched context:** Memory digests, UI specs, and enriched plans all compete for context window space. Implement budget allocation: e.g., 100k for wave context, 10k for memory, 5k for UI specs, 5k for contract summaries. Make configurable. (source: ARCHITECTURE)

### Cross-Cutting Requirements

**Must address early (high retrofit cost):**
- **Unified memory persistence architecture:** Three features (user request memory, quick logging, richer context) need persistent storage. Without a shared `.planning/memory/` pattern established early, each will build ad-hoc storage that is expensive to unify later. (sources: OVERSIGHTS, ARCHITECTURE, FEATURES)
- **Context consumption path update:** `execute.cjs:prepareSetContext()` currently loads only 3 things (scopedMd, definition, contractStr). Richer context is useless without an updated injection path. Must update before adding new context types. (sources: OVERSIGHTS, ARCHITECTURE)
- **Hook design decision (SKILL.md checks vs hook system):** Must decide between Claude Code `TaskCompleted` hooks (shell scripts, fire-and-forget), RAPID SKILL.md-level post-checks (inline state validation), or hybrid approach before any hook is implemented. (sources: OVERSIGHTS, UX)
- **UI contract schema design:** CONTRACT.json schema extension must happen before any UI spec is written. Schema changes after contracts exist require migration. (sources: OVERSIGHTS, ARCHITECTURE)
- **Configuration unification:** Adding a `loadConfig()` function and merging config sources is cheap now, expensive after 4+ features add their own config reads. (source: OVERSIGHTS)

**Can address later (low retrofit cost):**
- **npm test script:** 1-line addition to package.json; can happen any time (source: OVERSIGHTS)
- **Documentation staleness tracking:** DOC-MANIFEST.json can be added at any time; cost increases linearly with docs added without tracking (source: OVERSIGHTS)
- **Code quality measurement baselines:** Baselines can be established at any point; earlier is better for comparison but not structurally expensive to retrofit (source: OVERSIGHTS)
- **/quick branch guard:** 5-line bash check in SKILL.md; simple but easily forgotten (source: OVERSIGHTS)
- **NO_COLOR environment variable support in display.cjs:** Industry standard accessibility feature, small change (source: UX)
- **SKILL.md integration testing:** Integration tests can be added incrementally; the gap is uncomfortable but not structurally expensive to retrofit (source: OVERSIGHTS)

### User Experience Direction

- **Interaction model is well-aligned with domain conventions:** CLI-first slash commands, subagent orchestration, and interactive prompts match the emerging standard in the Claude Code ecosystem. No paradigm changes needed. (source: UX)
- **Memory should be automatic, not manual:** Users coming from GitHub Copilot expect persistent memory that "just works" across sessions. Memory should be auto-populated during discuss-set and new-version, and auto-surfaced at session start. (sources: UX, OVERSIGHTS)
- **Hooks should be transparent:** Users expect hooks to fire automatically and report only on failure. Follow CI/CD conventions: configuration-driven (JSON), not code-driven. Use Claude Code's native `SubagentStop`/`Stop` hook events where possible. (source: UX)
- **Activity logging is a user expectation gap:** Users expect a consolidated "what happened" view. Neither quick tasks nor full set executions have a unified activity log. Quick task logging (todo item 7) addresses part of this; a broader ACTIVITY.jsonl is a future consideration. (source: UX)
- **Error UX is a competitive advantage:** The "What happened / What's done / What to do next" error pattern and progress breadcrumbs are strong and somewhat unique. No changes needed. (source: UX)
- **Progressive disclosure hierarchy is correct:** New features should slot into existing levels -- memory at Level 1 (auto-surfaced), hooks/UI contracts at Level 2 (on request), documentation at Level 2 (on invocation). (source: UX)
- **Competitive landscape context:** GitHub Copilot Memory, Deep Agent CLI, and Agent Orchestrator (ComposioHQ) are converging on persistent memory and agent coordination. RAPID's discuss-set phase and adversarial review pipeline remain competitive differentiators. (source: UX)
- **Domain terminology is well-aligned:** "Hook", "contract", "skill" match industry usage. "Set" and "wave" are RAPID-unique but understood. "Quick task" could also be called "hotfix" or "ad-hoc task" per industry norms. (source: UX)

## Contradictions

### Contradiction: Hook Implementation Strategy
- **ARCHITECTURE says:** Build a first-class event system within RAPID's lifecycle. Create `hooks.cjs` with `registerHook()`, `executeHooks()`, and `validateAgentCompletion()`. Wire into `state-machine.cjs` via dependency injection callback pattern.
- **OVERSIGHTS says:** The simplest solution may not be hooks at all -- it may be adding a state-check step to SKILL.md orchestrators. SKILL.md-level checks are more reliable than a separate hook system because SKILL.md already has access to state and transition logic.
- **UX says:** Use Claude Code's native hook system (`SubagentStop`, `Stop` events) via `hooks/hooks.json`. This is the industry-standard approach and avoids reinventing hook infrastructure.
- **Resolution:** These are not fully contradictory but represent a design spectrum. Recommended hybrid: (a) use Claude Code native hooks for optional telemetry/logging, (b) add SKILL.md-level post-checks for critical state verification (the most reliable path), (c) build `hooks.cjs` as the programmatic interface that both paths can use for validation logic. This needs a design decision before implementation.

### Contradiction: UI Contract Storage Model
- **FEATURES says:** Separate `UI-CONTRACT.json` per set gives clean separation from API contracts while remaining machine-validatable.
- **ARCHITECTURE says:** Extend CONTRACT.json with an optional `ui` section. UI specs extend the contract rather than creating a parallel specification system -- agents already know how to read and validate contracts.
- **OVERSIGHTS says:** UI consistency is cross-cutting, not per-set. A project-wide `.planning/UI-SPECS.md` is more appropriate because individual sets reference the global UI spec.
- **Resolution:** These represent three different scoping models (per-set separate file, per-set embedded in contract, project-wide). The right answer likely depends on project type. For RAPID's needs, the recommended approach is: project-wide `.planning/UI-SPEC.md` for design system rules (colors, typography, layout patterns), plus per-set `ui` section in CONTRACT.json for set-specific component declarations. This gives both cross-cutting consistency and set-level specificity. Needs user/roadmapper decision.

### Contradiction: Memory Storage File vs Log
- **FEATURES says:** JSONL decision log (`.planning/decisions.jsonl`) combined with human-readable DECISIONS.md regenerated from JSONL on demand.
- **ARCHITECTURE says:** `.planning/memory/` directory with DECISIONS.jsonl, CORRECTIONS.jsonl, and CONTEXT-NOTES.md. Separate files for decisions, corrections, and human-readable notes.
- **OVERSIGHTS says:** Unified `.planning/memory/` directory with standard JSONL format for all structured events. All three memory features should write through a single `memory.cjs` module.
- **Resolution:** All three converge on JSONL in a `.planning/memory/` directory. The difference is granularity (one file vs multiple). OVERSIGHTS' recommendation of a unified module with multiple JSONL files per event type is the most practical: `DECISIONS.jsonl`, `CORRECTIONS.jsonl`, and `quick-tasks.jsonl` all under `.planning/memory/`, all accessed through `memory.cjs`. No real contradiction -- just different levels of detail.

### Contradiction: Quick Task Log File Name and Location
- **FEATURES says:** `.planning/quick/LOG.jsonl`
- **ARCHITECTURE says:** `.planning/quick/INDEX.json`
- **OVERSIGHTS says:** `.planning/memory/quick-tasks.jsonl`
- **UX says:** `.planning/quick/ACTIVITY.jsonl`
- **STACK says:** `.planning/quick/HISTORY.jsonl`
- **Resolution:** Five different proposals for the same file. The key decisions are: (a) JSONL vs JSON -- JSONL is correct for append-only logging, (b) location in `.planning/quick/` vs `.planning/memory/` -- storing under memory is consistent with the unified memory architecture. Recommended: `.planning/memory/quick-tasks.jsonl` (OVERSIGHTS' recommendation) because it integrates with the unified memory module. The name matters less than the consistency. Needs roadmapper decision.

## Recommended Approach

The v3.4.0 "Agent Intelligence" milestone is fundamentally about making RAPID's agents smarter by giving them better memory, richer context, and stronger guardrails. The 7 features share a common architectural need: capture more information at lifecycle transition points, persist it durably, and make it available to downstream agents. This is an extension of the existing architecture, not a redesign.

The most important architectural decision is establishing the memory persistence layer first. Three of the seven features (memory, quick logging, code quality) either directly produce or consume persistent memory, and two more (hooks, documentation) benefit from it. Building memory.cjs with JSONL append/query/digest functions in `.planning/memory/` gives all subsequent features a shared foundation. Without this, each feature will create ad-hoc storage that must be unified later at high cost.

The second critical decision is the hook implementation strategy. Three research streams (ARCHITECTURE, OVERSIGHTS, UX) propose different approaches ranging from a full custom event system to simple SKILL.md post-checks to Claude Code native hooks. The recommended hybrid is: SKILL.md-level state verification for critical paths (most reliable because it runs in the agent's own context), Claude Code native hooks for optional telemetry/logging, and a `hooks.cjs` library module that provides the validation logic both paths can call. This avoids overengineering while ensuring state updates are never forgotten.

The third priority is updating the context consumption path. The existing `prepareSetContext()` function is the bottleneck -- it loads only 3 artifacts. Until this is extended to accept memory digests, UI specs, and enriched context, no amount of new context generation matters. This update must include token budget allocation across context sources to avoid exceeding the 120k compaction budget.

Implementation should proceed in three waves. Wave 1 establishes infrastructure: memory module, context path update, hook design, quick task logging, add-set CLI refactor, and Node.js version bump. Wave 2 builds the substantive features: code quality profile, UI contracts, memory injection into agents, and hook state-verification. Wave 3 delivers documentation generation (which benefits from documenting the other features) and polish items (NO_COLOR, npm test script, config unification). The trivial features (quick logging, add-set) should be early in Wave 1 as warm-ups that establish patterns for the complex features.

A key risk throughout is context window pressure. Every new information source (memory, UI specs, quality profiles, richer plans) competes for the same ~120k token budget. The compaction system must be extended with per-source budgets, and memory digests must be pre-compacted summaries rather than raw logs. The existing compaction infrastructure from v3.3.0 provides the foundation, but it needs explicit allocation per context type.

### Suggested Set Boundaries

- **Set A: Memory Infrastructure** -- Create `memory.cjs` module with JSONL append/query/digest, `.planning/memory/` directory structure, decision recording in discuss-set and new-version skills. Rationale: foundational for 3+ other features; must be first.

- **Set B: Quick Logging + add-set** -- Quick task JSONL logging, monotonic ID fix, query CLI, add-set CLI refactor with `withStateTransaction`, DAG recalculation. Rationale: two trivial features grouped together; both produce small CLI commands; add-set fix eliminates a known data-loss race condition.

- **Set C: Hooks + State Verification** -- Hook design (SKILL.md checks + native hooks hybrid), `hooks.cjs` validation module, state-check post-execution step in SKILL.md files, hook registration in plugin manifest. Rationale: self-contained reliability feature; depends on memory for logging but can proceed in parallel with minimal coordination.

- **Set D: Code Quality + Context Enrichment** -- Quality profile system, `prepareSetContext()` extension, memory injection into agent prompts, token budget allocation, enhanced scoped CLAUDE.md. Rationale: the "make agents smarter" core; depends on memory infrastructure (Set A); touches execute.cjs, compaction.cjs, worktree.cjs.

- **Set E: UI Specs/Contracts** -- UI contract schema design, `ui-spec.cjs` module, Ajv validation, cross-set consistency checks, planner/executor integration. Rationale: complex but self-contained; independent of memory; can parallel with Set D.

- **Set F: Documentation Generation** -- Documentation agent, template system, changelog extraction, diff-aware updates, `/rapid:docs` skill. Rationale: most standalone; benefits from all other features being in place; should be last.

- **Set G: Infrastructure Polish** (optional, can merge into other sets) -- Node.js 20 minimum bump, npm test script, NO_COLOR support, config unification, .nvmrc, Zod version pinning. Rationale: small independent improvements that can be sprinkled across other sets or grouped as a cleanup pass.
