# Research Summary -- v3.6.0 Workflow & UX Polish

## Project Profile
- **Type:** brownfield plugin (Claude Code CLI plugin for parallel development)
- **Stack:** Node.js (CommonJS) + Zod + proper-lockfile + Git worktrees
- **Scale:** medium (6 features, 24 sub-features across workflow fixes, UX improvements, and one new capability)
- **Key Constraint:** All changes are application-logic and agent-prompt modifications; no dependency upgrades or infrastructure changes are needed for this milestone

## Unified Findings

### Stack and Infrastructure

- **Node.js 18 minimum is outdated.** Node.js 18 reached EOL on 2025-04-30. RAPID's `prereqs.cjs` (line 118) sets `minVersion: '18'`. Should bump to 20 or 22. No milestone features require version-specific APIs; this is a hygiene item. (sources: STACK, OVERSIGHTS)
- **CommonJS is stable and sufficient.** All 21+ library modules, 50+ test files, and the CLI use `require()`. No ESM migration needed. `node:test` works correctly with CJS across all supported versions. (source: STACK)
- **Zod 3.25.76 should NOT be upgraded.** Zod v4 has 6+ breaking changes affecting `state-schemas.cjs`, `.passthrough()` removal, `.enum` API changes, and CommonJS bundle issues (4x size increase, module resolution problems). Current version is functional. (source: STACK)
- **proper-lockfile 4.1.2 is unmaintained but adequate.** Last updated ~5 years ago. No CVEs, no alternatives for inter-process file locking. Its API surface is small and stable. (sources: STACK, PITFALLS)
- **No build step, no linter, no CI/CD.** Zero-build approach is a strength for a Claude Code plugin. Code style enforced through agent prompts. No changes needed. (source: STACK)
- **No test coverage metrics.** 53 test files exist but no coverage reporting. Consider `--experimental-test-coverage` flag (Node 20+). (source: STACK)
- **No version conflicts exist for the current stack.** Git 2.30+, Zod 3.x in CJS, proper-lockfile + Node 25, `node:test` runner -- all compatible. (source: STACK)

### Feature Landscape

**Six features with clear dependency ordering:**

```
Feature 1: Bug Fixes (DAG.json ENOENT, path fix, execute state)
  +-- Feature 2: DAG.json Management (depends on Feature 1's tryLoadDAG)
  +-- Feature 4: Solo Mode Improvements (depends on state transition fixes)

Feature 3: Init Flow Redesign (independent)
Feature 5: Branding System (independent)
Feature 6: Review Cycle Confirmation (independent)
```

- **Feature 1 (Bug Fixes):** 4 sub-features. Moderate complexity. Foundation work -- fixes DAG.json ENOENT crash, path inconsistency (`.planning/DAG.json` vs `.planning/sets/DAG.json`), and execute-state completion bug. (source: FEATURES)
- **Feature 2 (DAG.json Management):** 4 sub-features. Moderate complexity. Adds DAG.json creation to init flow, lazy regeneration via `ensureDAG()`. Depends on Feature 1. (source: FEATURES)
- **Feature 3 (Init Flow Redesign):** 4 sub-features. Complex. Major SKILL.md rewrite of Step 4B with structured AskUserQuestion, granularity preference, summary + confirmation. (sources: FEATURES, UX)
- **Feature 4 (Solo Mode Improvements):** 4 sub-features. Moderate complexity. Execute-set auto-merge transition for solo sets, merge agent messaging. Depends on Feature 1. (sources: FEATURES, UX)
- **Feature 5 (Branding System):** 5 sub-features. Complex. New `/rapid:branding` skill, new agent role, BRANDING.md artifact, agent awareness integration. (sources: FEATURES, UX)
- **Feature 6 (Review Cycle Confirmation):** 3 sub-features. Trivial complexity. AskUserQuestion between bug-hunt cycles. Smallest effort, quickest win. (sources: FEATURES, UX)

**Third-party service dependencies:** None. All features are internal RAPID workflow modifications. (source: FEATURES)

**Recommended implementation order:** Feature 1 (bug fixes) -> Feature 6 (review confirmation) -> Feature 2 (DAG management) -> Feature 4 (solo mode) -> Feature 3 (init redesign) -> Feature 5 (branding). (source: FEATURES)

### Architecture Direction

- **Pattern: Incremental extension with defensive hardening.** The existing 5-layer architecture (Skill > Agent > CLI > Library > Storage) is sound. No new layers or structural changes needed. Each change operates within its layer boundaries. (source: ARCHITECTURE)
- **Module boundaries are well-defined.** DAG management (dag.cjs + plan.cjs + add-set.cjs), init flow (init.cjs + init SKILL.md), execute & state (execute.cjs + execute-set SKILL.md), solo mode (worktree.cjs), merge pipeline (merge.cjs + merge SKILL.md), review cycles (bug-hunt/unit-test SKILL.md), branding system (new skill + new role). (source: ARCHITECTURE)
- **DAG.json lifecycle gap is the primary architectural issue.** DAG.json is currently created in 2-3 places but consumed by 6. The init flow never creates it. The proposed fix adds creation at init time (post-roadmap acceptance) and defensive handling everywhere else. (sources: ARCHITECTURE, FEATURES, OVERSIGHTS)
- **State machine is correct.** SET_TRANSITIONS map (`pending -> discussed -> planned -> executed -> complete -> merged`) supports all needed transitions including `complete -> merged` for solo auto-merge. Self-transitions exist for idempotent re-entry. (source: ARCHITECTURE)
- **Solo mode foundation exists.** `worktree.cjs` has `setInitSolo()`, `isSoloMode()`, `getSetDiffBase()`. Registry marks solo entries with `solo: true` and `startCommit`. Merge command detects and fast-paths solo sets. Gaps are in post-execution state management and messaging. (sources: ARCHITECTURE, OVERSIGHTS)
- **Branding system needs a new skill and agent role** but integrates into the existing `enrichedPrepareSetContext()` optional context injection pattern (alongside `qualityContext` and `uiContext`). (sources: ARCHITECTURE, OVERSIGHTS)
- **Display system is extensible.** Adding new stages (like `branding`) requires only two map entries (`STAGE_VERBS` and `STAGE_BG`). (source: ARCHITECTURE)

### Cross-Cutting Requirements

**Must address early (high retrofit cost):**
- DAG.json lifecycle management -- every merge and execution operation fails without it; creating DAG.json at init and fixing the path inconsistency must happen first (sources: OVERSIGHTS, PITFALLS, ARCHITECTURE)
- Execute-set state transition reliability -- root cause of review pipeline blocking; restructure Step 6 ordering and add error recovery (sources: OVERSIGHTS, PITFALLS, ARCHITECTURE)
- Solo mode cross-skill state consistency -- auto-merging solo sets to `merged` affects review, merge, cleanup, and status display; all consuming skills need awareness (sources: OVERSIGHTS, ARCHITECTURE)

**Can address later (low retrofit cost):**
- AskUserQuestion convention (2-5 options max) -- document but do not enforce yet (source: OVERSIGHTS)
- Configuration propagation for new features -- most new features do not need config entries; avoid unnecessary config complexity (source: OVERSIGHTS)
- Skill validation tests -- valuable but not blocking; write alongside fixes (source: OVERSIGHTS)
- Documentation and agent discovery for `/branding` -- standard skill creation process, follow existing patterns (source: OVERSIGHTS)
- `NO_COLOR` environment variable support -- minor CLI accessibility concern, not related to milestone goals (source: UX)
- Node.js minimum version bump to 20/22 -- hygiene item, trivial change, no blocking risk (source: STACK)

### User Experience Direction

**Domain conventions and standard terminology:**
- RAPID users think in terms of "tasks" (sets), "branches" (worktrees), and "what's next?" (status). The CLI uses numeric shorthand for set references. The interaction model is a **conversational CLI** alternating between commands, structured questions, and agent output. (source: UX)

**Interaction models and user expectations:**
- Solo sets should "just work" on main without merge ceremony. (sources: UX, FEATURES)
- Review cycles should not run away without user confirmation. (sources: UX, FEATURES)
- Missing files should degrade gracefully, not crash. (sources: UX, PITFALLS)
- Init should be interactive wizard-style, not essay-writing. Structured AskUserQuestion with pre-filled options matches the mental model of modern CLI setup wizards (Nx, Vite, Yeoman). (sources: UX, FEATURES)
- Post-execution state should accurately reflect reality -- users should not diagnose why review refuses to run. (sources: UX, OVERSIGHTS)

**Information architecture patterns:**
- Stage banners, workflow breadcrumbs, per-wave progress, next-step hints form the feedback system. (source: UX)
- Flat command namespace with semantic grouping (`/rapid:init`, `/rapid:start-set N`, etc.). The proposed `/rapid:branding` fits naturally as an optional post-init step. (source: UX)
- Progressive disclosure via staged question batches (4 questions at a time) aligns with Miller's Law (7+/-2 items). (sources: UX, OVERSIGHTS)

**Key accessibility considerations:**
- `NO_COLOR` environment variable not supported (display.cjs always outputs ANSI). Minor concern for this milestone. (source: UX)
- Structured choices over freeform improve cognitive accessibility. (source: UX)
- "Other" option and `--skip` flag provide escape hatches for power users. (source: UX)

## Contradictions

### Contradiction: Unit-Test Cycle Confirmation Scope

- **FEATURES says:** "Add a confirmation prompt after Step 5 (execute tests) if there are failures: 'Run tests again after fixes?'" -- implying unit-test needs a re-run loop.
- **ARCHITECTURE says:** "The unit-test skill is single-cycle by design and does not need this change."
- **OVERSIGHTS says:** "The milestone may mean that unit-test should also confirm before writing/running tests (currently it already has plan approval in Step 4)" -- suggesting the existing plan approval may satisfy the requirement.
- **Resolution:** ARCHITECTURE is correct that unit-test is single-cycle. The existing Step 4 plan approval already provides the confirmation gate. The FEATURES suggestion of a "re-run on failure" prompt is an enhancement beyond the milestone scope. The primary cycle confirmation target is bug-hunt only. Unit-test and UAT may benefit from a simple "retry on failure" prompt but this is low priority.

### Contradiction: Branding Artifact Location

- **FEATURES says:** Store branding at `.planning/BRANDING.md` at the planning root.
- **ARCHITECTURE says:** Store at `.planning/branding/BRANDING.md` (in a subdirectory).
- **Resolution:** `.planning/BRANDING.md` (planning root, no subdirectory) is more consistent with existing artifacts (`PROJECT.md`, `ROADMAP.md`, `REQUIREMENTS.md` are all at `.planning/` root). The subdirectory approach adds unnecessary nesting for a single file. Use `.planning/BRANDING.md`.

### Contradiction: DAG.json Locking Strategy

- **PITFALLS says:** DAG.json writes need locking via `withDAGTransaction()` to prevent concurrent write data loss.
- **ARCHITECTURE says:** DAG.json can be regenerated from STATE.json + CONTRACT.json using `recalculateDAG()`, implying regeneration replaces the need for incremental locked writes.
- **Resolution:** Both are valid but ARCHITECTURE's approach is better for this milestone. Treating DAG.json as a derived artifact (always regenerated from source-of-truth STATE.json + CONTRACT.json) eliminates the concurrent write problem entirely. `recalculateDAG()` already implements this. No new locking mechanism needed -- just ensure all agents call `recalculateDAG()` instead of doing incremental DAG.json writes.

### Contradiction: Granularity Preference Storage

- **OVERSIGHTS says:** "For this milestone, avoid adding new config entries unless absolutely necessary. `targetSetCount` should be a runtime input to the roadmapper (passed in the agent prompt), not persisted config."
- **FEATURES says:** "`config.json` gains a `granularity` field (e.g., `{ "granularity": { "suggestedSets": 8 } }`)."
- **Resolution:** OVERSIGHTS' recommendation is better. Passing `targetSetCount` as a runtime input to the roadmapper prompt is simpler and avoids config schema complexity. If persistence becomes needed later (e.g., for add-set warnings), it can be added then. For this milestone, keep it as a prompt-time parameter.

## Recommended Approach

The v3.6.0 milestone is a polish release addressing concrete UX friction points and bugs observed during real-world usage. All six goals are achievable with the current stack -- no dependency upgrades, infrastructure changes, or architectural modifications are needed. The work is entirely in application logic (library modules), agent prompts (SKILL.md files and role modules), and state management.

The critical path begins with three tightly related bug fixes: DAG.json path consolidation and ENOENT handling, execute-set state transition reliability, and DAG.json creation at init time. These three items form a coherent unit of work because they all address the same root cause: the DAG.json lifecycle has gaps that cause downstream failures. A centralized `tryLoadDAG()` function with canonical path `.planning/sets/DAG.json` eliminates the path inconsistency, while adding `recalculateDAG()` to the init flow ensures DAG.json exists from project creation onward. The execute-set state transition fix should restructure Step 6 to run the state transition before git commit operations, use explicit CWD, and add retry logic for lock contention.

Solo mode improvements and review cycle confirmation form the next tier. Solo auto-merge is a small but important behavioral change: after execution, check the registry for `solo: true` and auto-transition to `merged`. This has a cross-skill impact that must be handled carefully -- the review skill needs to auto-detect solo+merged sets and switch to post-merge mode rather than rejecting them. The review cycle confirmation is the quickest win in the milestone: a single AskUserQuestion insertion between bug-hunt cycles with an early-exit path that preserves findings.

The init flow redesign and branding system are the two complex features that can be developed independently and in parallel. The init redesign replaces prose-based discovery with structured AskUserQuestion batches, adds a granularity preference, and adds summary confirmation. This is a significant SKILL.md rewrite but does not touch library code. The branding system is entirely new: a skill, an optional agent role, an artifact format, and integration into the existing context injection pipeline. Both features should be developed after the critical path items are stable.

A key technical decision that emerged from synthesizing the research: DAG.json should be treated as a **derived artifact** that is always regenerated from STATE.json + CONTRACT.json via `recalculateDAG()`, rather than incrementally updated with locking. This eliminates the concurrent write problem identified in PITFALLS and simplifies the lifecycle management identified in FEATURES and OVERSIGHTS.

### Suggested Set Boundaries

- **Set A: DAG & State Fixes** -- DAG.json path consolidation, centralized `tryLoadDAG()`, ENOENT defensive handling across all consumers, execute-set state-to-complete transition fix, DAG.json creation at init time. Rationale: these are interdependent bug fixes that share the same code areas (`plan.cjs`, `merge.cjs`, `execute.cjs`, `execute-set/SKILL.md`). Must be done first.

- **Set B: Solo Mode + Review Confirmation** -- Solo auto-merge in execute-set, review skill solo+merged detection, merge skill solo messaging, bug-hunt cycle confirmation with early-exit path. Rationale: solo mode and review confirmation both modify SKILL.md files in the execution and review pipeline and touch the state transition logic. They can be grouped because they affect adjacent skill areas without file conflicts.

- **Set C: Init Flow Redesign** -- Structured AskUserQuestion discovery, granularity preference, summary confirmation, roadmapper prompt updates for targetSetCount. Rationale: contained entirely within `skills/init/SKILL.md` and `src/modules/roles/role-roadmapper.md`. No library code changes needed. Independent of Sets A and B.

- **Set D: Branding System** -- New `/rapid:branding` skill, `role-branding.md` agent, BRANDING.md artifact format, `enrichedPrepareSetContext()` branding context injection, display.cjs stage entries. Rationale: entirely new feature with no overlap with existing code being modified by other sets. Can be developed fully in parallel.
