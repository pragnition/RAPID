# Research Synthesis -- v7.0.0 Mission Control Autopilot

**Synthesized:** 2026-04-15
**Research inputs:** 6/6 (STACK, FEATURES, ARCHITECTURE, PITFALLS, OVERSIGHTS, UX)
**Target:** Unified input for `rapid-roadmapper` to produce the v7.0.0 project roadmap.

---

## Project Profile

- **Type:** brownfield / plugin with parallel web control plane (Claude Code plugin + FastAPI/React Mission Control webUI)
- **Stack:** Python 3.12+ FastAPI / SQLModel / SQLite (web backend); React 19 + Vite 8 + TanStack Query + Zustand (frontend); Node 22+ CommonJS (CLI plugin); new dep `claude-agent-sdk` (Python, >=0.1.59,<1.0)
- **Scale:** medium. 5 distinct feature subsystems, ~8-10 waves, ~30 new backend files, ~15 new React components, 1 new DB migration, 4 new tables, ~10 new HTTP endpoints, 1 new SSE/WS streaming endpoint, 30 SKILL.md frontmatter migrations
- **Key Constraint:** The Claude Agent SDK ships only for Node and Python, and RAPID's web backend is already Python/FastAPI. Adopting the Python SDK in-process inside the existing FastAPI app is the load-bearing architectural decision that shapes data model, concurrency, deployment, security, and every downstream feature. The alternative (Node sidecar) adds a second runtime for no benefit.

## Unified Findings

### Stack and Infrastructure

- **Adopt `claude-agent-sdk` (Python) at `>=0.1.59,<0.2`** (sources: STACK, FEATURES, ARCHITECTURE, OVERSIGHTS). The package bundles the Claude Code CLI binary -- do NOT also globally install `@anthropic-ai/claude-code`. Pin with a bounded range, not exact-pin (pre-1.0 weekly churn; SDK 0.2.101+ requires minimum for MCP-leak fix).
- **No new runtime added.** Python 3.12 satisfies SDK's >=3.10 floor; FastAPI's async model is asyncio-native like the SDK (sources: STACK, FEATURES, ARCHITECTURE).
- **Frontend stack is current.** React 19.2, Vite 8.0, Tailwind 4.2, TanStack Query 5.91, Zustand 5.0, CodeMirror 6 are all already installed -- no new frontend deps required for the Agents/Chats tabs (sources: STACK, FEATURES, UX).
- **Transport: SSE (primary) + short-interval polling (fallback) -- NOT WebSocket.** Matches user's global CLAUDE.md preference for poll-based long-running tasks, fits Starlette natively, firewall-friendlier, simpler reconnect (sources: ARCHITECTURE, PITFALLS, UX, OVERSIGHTS). UX specifies 2-5s polling for run list/status + optional SSE for chat token streaming only.
- **Brandable constraint: cannot call the tabs "Claude Code".** Anthropic restricts partner naming (source: STACK). Use "Agents" / "Chats" / "Mission Control Agents".
- **`setting_sources=["project"]` is load-bearing.** Without it, the SDK silently skips `.claude/skills/*`, CLAUDE.md, and slash commands -- every `rapid:*` skill fails invisibly (sources: STACK, FEATURES, ARCHITECTURE). Centralise option construction in one helper and unit-test this assertion.
- **Concurrency ceiling needed.** SDK pre-0.2.101 warned on 11+ concurrent `query()` calls. Enforce `asyncio.Semaphore` (recommended 3 per project, 10 global) sized to machine resources (sources: STACK, ARCHITECTURE, PITFALLS, OVERSIGHTS).
- **CLI drift risk.** SDK spawns bundled `claude` CLI; if the user's PATH-level CLI mismatches, tool behavior drifts silently (sources: STACK, PITFALLS, OVERSIGHTS). Probe `claude --version` at startup; surface in `/api/health` and in the Agents tab's session metadata.
- **`zod 3.25.76` (core RAPID) satisfies SDK's `^3.24.1 || ^4.0.0` peer.** No upgrade required (source: STACK).

### Architecture Direction

- **Pattern:** Extend the FastAPI backend with a long-lived `AgentSessionManager` service (in-process, asyncio-native) that owns `ClaudeSDKClient` instances. Surface agent I/O via SSE with polling fallback. Expose kanban + AskUser tools as in-process SDK MCP servers (`create_sdk_mcp_server` + `@tool`). Map `AskUserQuestion` onto a custom MCP tool (`mcp__rapid__webui_ask_user`) rather than forking skills (source: ARCHITECTURE).
- **Module organisation** (source: ARCHITECTURE):
  - `app/agents/` package: `session_manager.py` (registry + lifecycle), `agent_session.py` (wraps `ClaudeSDKClient` + asyncio pump task), `event_bus.py` (per-run asyncio pub/sub with ring buffer), `skill_runner.py` (builds `(prompt, options)` from skill+args), `permissions.py` (`can_use_tool` callback), `tools/ask_user.py`, `tools/kanban_tools.py`, `transcripts.py` (SQLite persistence)
  - `app/routers/agents.py`, `app/routers/chats.py`
  - `app/models/agent_run.py`, `app/models/chat.py`
  - Frontend mirror: `pages/AgentsPage.tsx`, `pages/ChatsPage.tsx`, `pages/AgentRunPage.tsx`, `hooks/useAgentEvents.ts`, `components/agents/*`, `components/chat/*`, `lib/sseClient.ts`
- **Use V1 `query()` / `ClaudeSDKClient` (unified).** V2 preview is unstable and lacks `forkSession`; revisit post-milestone (sources: STACK, ARCHITECTURE). `ClaudeSDKClient` works for both autonomous (no input) and interactive (input-requiring) modes; no code-path fork needed.
- **Data flow:** browser `POST /api/agents/runs` returns `{run_id}` in <200ms; backend schedules `asyncio.Task`; browser attaches `EventSource` to `/api/agents/runs/{id}/events`; pump task emits `assistant_text | thinking | tool_use | tool_result | ask_user | permission_req | status | run_complete` events with monotonic `seq` (source: ARCHITECTURE).
- **Authoritative state:** `.planning/STATE.json` remains canonical for RAPID state (untouched by session manager; accessed only through spawned skills). New SQLite tables hold `agent_run`, `agent_event`, `chat`, `chat_message`. In-memory state is ephemeral; crash recovery marks in-flight runs `interrupted` -- no SDK session resume across backend restarts (source: ARCHITECTURE).
- **Kanban board is canonical as SQLite; `board.json` is a read-only projection.** All agent mutations go through `kanban_service.move_card` (optimistic concurrency + re-read before write + per-card rev numbers) -- never touch `board.json` directly (sources: PITFALLS, OVERSIGHTS).

### Cross-Cutting Requirements

**Must address early (high retrofit cost):**

- **Run mutex + per-set concurrency lock as SQLite unique constraint.** Without it, browser UX invites multi-run races (source: OVERSIGHTS risk #8). Database schema choice; migration later is expensive.
- **Logging substrate with `run_id` correlation threaded through SDK env + SQLite tables + SSE events + `.planning/` writes.** Schema locks in UI capabilities; backfilling after 1000 runs = migration + backfill (source: OVERSIGHTS).
- **Tool shim compatibility audit for all 30 skills.** Enumerate each skill's `allowed-tools` frontmatter; determine SDK behavior (works / needs shim / needs skill-level change); produce `docs/SKILL_SDK_MATRIX.md` (source: OVERSIGHTS).
- **Authentication + credential model.** Default: localhost-only bind, no auth. Network mode: opt-in bearer token via `rapid-web register-web`. Anthropic credential via reused Claude Code OAuth if SDK supports it, else `~/.rapid/config.toml`. Changing auth after launch is breaking (sources: PITFALLS, OVERSIGHTS).
- **Worktree isolation policy for autonomous model.** Pre-flight `git status --porcelain`; per-set DB mutex; registry re-read per run; `.rapid-worktrees/<set>/.RUNNING` marker file for CLI coordination (source: OVERSIGHTS).
- **Kanban schema v2 migration (`created_by`, `locked_by_run_id`, `completed_by_run_id`, `agent_status`, optional `metadata` JSON, optional `agent_run_id` FK).** Migration + backfill for existing cards (sources: FEATURES, OVERSIGHTS).
- **Chat history schema (sessions, messages, optional attachments-nullable column).** Stub now even if unused (source: OVERSIGHTS).
- **Additive-only CLI parity constraint.** Every new shim session-scoped; CLI sessions keep default behavior. Enforce via PR template + CI smoke test (source: OVERSIGHTS).
- **Error taxonomy (`SdkError`, `RunError`, `StateError`, `ToolError`, `UserError`) + BLOCKED resume skeleton.** Define once; apply consistently (source: OVERSIGHTS).
- **Token budget + cost cap (per-project daily default $10; per-run `max_turns` per skill).** Config before first ship prevents user complaints (sources: PITFALLS, OVERSIGHTS).
- **Test infrastructure (SDK mocks + real-SDK integration gated on `CI_LIVE_CLAUDE=1` + Playwright smoke).** One-time cost; delaying compounds (sources: STACK, OVERSIGHTS).
- **SDK option construction centralised in `agent_sdk.py` helper with `setting_sources=["project"]` unit-tested.** Single source of truth for every skill-route's SDK invocation (source: STACK).

**Can address later (low retrofit cost):**

- Data retention policies (add `rapid-web prune` when db grows)
- Backup + export (`rapid-web backup`, `rapid-web export --run-id`)
- Artifact side panel in chat (diff/plan viewer) -- defer to v7.1
- "Open as chat" button on run detail (mid-run steering) -- defer to v7.1
- Fork run action -- defer to v7.1
- Internationalization (only `Intl.DateTimeFormat` as cheap hedge)
- Pool SDK clients per-project for startup-cost optimisation (post-launch profiling)

### User Experience Direction

- **Domain conventions:** "run" (preferred term; matches Replit, Cursor, CI/CD mental model) over "task"/"job"/"session". "Autopilot" over "auto mode". "Approval" (user-facing) / "interrupt" (backend). "Dispatch" / "run a skill" over "trigger"/"invoke". Avoid "Claude Code" in UI copy (Anthropic branding restriction) (sources: UX, STACK).
- **Three surfaces, one dashboard** (source: UX): the single most load-bearing UX decision. Users must know from chrome alone which mode they are in:
  - **Chats tab** -- persistent composer at bottom, streaming cursor, inline tool-call cards, inline structured-question forms (not freeform textareas for bounded decisions). For `/rapid:discuss-set`, `/rapid:init`, `/rapid:bug-fix`, `/rapid:quick`.
  - **Agents tab (run detail)** -- NO composer, prominent status pill (`RUNNING / 00:04:32`), live activity feed, pause/stop controls. For autonomous skills like `/rapid:plan-set`, `/rapid:execute-set`.
  - **Kanban** -- work queue / agent handoff with "Agent claimed" badges and "Run with RAPID" per-card action.
- **Navigation: extend left sidebar, NOT introduce top tabs.** Add `Agents` (`ga`) and `Chats` (`gh`) alongside Kanban. Group driver surfaces (Agents / Chats / Kanban) separated from observer surfaces (State / Worktrees / Graph / Codebase). Deep-linkable routes `/agents/:runId`, `/chats/:threadId` (source: UX).
- **Run = durable server-side resource.** Tab close does not kill run; reopen re-attaches. Matches Replit Agent 3, Devin, Manus; matches CLAUDE.md's polling preference; matches existing `/rapid:pause` / `/rapid:resume` CLI semantics (source: UX).
- **Skill launcher modal modeled on GitHub Actions `workflow_dispatch`.** Typed inputs (string / choice / bool / multi-line), description, required flag, default value per input. Dedicated "Skip discuss" toggle for skills that support it. Capped at ~10 inputs per skill (GH Actions limit; RAPID skills rarely exceed 4) (sources: FEATURES, UX).
- **Per-skill argument schema via YAML frontmatter in each `SKILL.md`.** Co-located with skill definition; migration to 30 files is mechanical (source: FEATURES).
- **Inline tool-call cards.** Spinner while running, checkmark + duration + expandable args/result when complete. Industry standard in 2026 chat UIs (source: UX).
- **Approval modal triad: Approve / Reject / Edit** (modify tool args before approving). Focus-trapped, ESC = reject, default = reject for destructive. Wire via `can_use_tool` (source: UX).
- **Status pills = color + text label** (WCAG 1.4.1). `RUNNING`, `WAITING`, `FAILED`, `COMPLETED` (source: UX).
- **ARIA live regions for streaming content; `aria-busy` during token streams; focus-trapped approval modals; keyboard-accessible pause/stop/approve (`Shift+P`/`Shift+S`); respect `prefers-reduced-motion`; auto-scroll opt-out** (source: UX).
- **Empty-state onboarding on Agents/Chats tabs** explains the chat-vs-run distinction with 3 example actions (source: UX).
- **Cost/token/duration telemetry on every run** (table-stakes 2026 expectation) (source: UX).
- **Kanban autopilot: "drop it, let the agent handle it, check back later."** Linear AI is the reference. Card metadata determines subflow: `bug` label -> `/rapid:bug-fix`; `feature` -> `/rapid:add-set`; default -> `/rapid:quick` (sources: FEATURES, UX).
- **Column semantic mapping:** `Todo` / `In Progress` / `Done` / `Blocked` convention; explicit `is_autopilot` flag on `KanbanColumn` is cleaner than name-match (sources: FEATURES, ARCHITECTURE).
- **Autopilot safety: per-card max-retry counter (N=3); card-level timeout; "Blocked" / "Needs Review" column halts retries.** Prevents runaway loops (sources: FEATURES, ARCHITECTURE, PITFALLS).

## Contradictions

Four contradictions detected. All resolvable; none require new user decisions.

### Contradiction 1: Transport -- WebSocket vs SSE

- **STACK says:** "WebSocket on backend: FastAPI's native `WebSocket` + Starlette is sufficient. No Socket.IO dep required ... the demo `simple-chatapp` pattern from the Anthropic SDK demos is directly adaptable." (line 76)
- **FEATURES says:** Lists `WS /api/projects/{pid}/agents/runs/{run_id}/stream` as the streaming endpoint (line 114) and describes a WebSocket-based `canUseTool` round-trip (Option 1, line 140).
- **ARCHITECTURE says:** Explicitly argues **SSE with polling fallback** over WebSocket: "SSE fits natively in Starlette (`EventSourceResponse`), is firewall-friendlier than WebSockets, and matches the user's documented CLAUDE.md preference." (line 37); later "Adopt SSE + polling fallback, not WebSocket -- matches user's documented preference." (Recommendation #1)
- **PITFALLS says:** "SSE, not WebSocket, for event streaming. The existing `src/lib/branding-server.cjs` already uses SSE ... auto-reconnects in the browser via `EventSource`." (line 64). "If you must use WebSocket (bidirectional -- which you will for tool-bridge responses)" with explicit backpressure warnings.
- **UX says:** "Short-interval polling (2-5s) on the runs list and current run detail, with optional SSE for the chat message stream only." (line 229)
- **Resolution: SSE + short-interval polling, NOT WebSocket.** Four of five reports agree (ARCHITECTURE / PITFALLS / UX / global CLAUDE.md); STACK and FEATURES mention WebSocket only as a surface-level default and do not consider the CLAUDE.md polling preference or reverse-proxy simplicity. Server-to-client uses SSE; client-to-server uses plain `POST` (for answers, input, interrupts). Reconnect via `EventSource` auto-retry + `?since=N` replay from ring buffer + SQLite backfill.

### Contradiction 2: Bridge Mechanism for `AskUserQuestion`

- **FEATURES (Option 3, line 162):** Hybrid -- `canUseTool` for built-in `AskUserQuestion` (zero skill changes) + custom MCP tools only for net-new browser capabilities. Recommends this because existing 9+ skills already use `AskUserQuestion` and must stay CLI-compatible.
- **ARCHITECTURE (line 197):** "Patch existing skills minimally: wrap `AskUserQuestion` with 'if sdk-mode, call `mcp__rapid__webui_ask_user`' -- do NOT fork skills into CLI and web variants. Gate on `RAPID_RUN_MODE=sdk` env var."
- **PITFALLS (Pitfall "AskUserQuestion 4-Question Limit"):** Identifies that the SDK's native `AskUserQuestion` has a 4-question-max + no-free-text schema limit. Recommends adding `rapid__ask_free_text` as a custom MCP tool for free-form input, validating server-side to split >4 questions.
- **Resolution: Hybrid, as FEATURES Option 3 describes, refined by PITFALLS.**
  - `can_use_tool` intercepts built-in `AskUserQuestion` for the multi-choice cases it already handles. Zero skill changes for the CLI path.
  - Add a custom MCP tool (`mcp__rapid__webui_ask_user` and/or `mcp__rapid__ask_free_text`) ONLY for cases the built-in can't handle: free-form text, N>4 questions, richer input. Gate skill prose to call the custom tool when `RAPID_RUN_MODE=sdk` env var is set (via `ClaudeAgentOptions.env`). ARCHITECTURE's env-gate is a narrower version of Option 3 -- they are compatible, not contradictory.
  - Server-side validation in `can_use_tool` splits >4 questions into multiple prompts before forwarding to browser (PITFALLS).

### Contradiction 3: Permission Mode Default

- **STACK (Risk #6):** Warns `permission_mode` default prompts for every non-allowlisted tool -- without a UI approval route, session hangs. Suggests "pre-populate `allowed_tools` per skill category" as an alternative.
- **PITFALLS (Critical pitfall "Permission Escalation"):** Explicitly insists: "Default to `permission_mode=\"default\"` ... Autopilot means 'user approved this task up front via the kanban card'; it does NOT mean 'auto-approve every tool call.' Per-skill permission policy: `plan-set`/`execute-set` can use `acceptEdits` but with `disallowed_tools`; `merge`/`cleanup` stay in `default`."
- **ARCHITECTURE:** Ships `can_use_tool` as the primary gate with two modes (`autonomous`, `interactive`), both with tightened allowlists.
- **Resolution: Per-skill permission policy, NEVER `bypassPermissions`.** PITFALLS is correct and STACK's concern is met by the `can_use_tool` bridge going live from day one (STACK's own Recommendation #3). Default stays `"default"` for interactive runs and destructive skills (merge, cleanup). `acceptEdits` is acceptable ONLY for file-editing skills (`plan-set`, `execute-set`) and ONLY when paired with `disallowed_tools` for the destructive command set. `bypassPermissions` is never used; it propagates to subagents and nullifies the safety model.

### Contradiction 4: cwd for SDK Session

- **ARCHITECTURE (line 185):** "Constructs `ClaudeAgentOptions` with ... `cwd=project_path`" and "The cwd is the project root, so `.planning/STATE.json` lock acquisition goes through the existing `proper-lockfile` machinery." (lines 185, 191). Also notes passing `cwd` to the worktree path for set runs.
- **PITFALLS (Pitfall "Sessions Keyed on cwd"):** "If an agent run's cwd is `/home/kek/Projects/MyProject/.rapid-worktrees/set-01`, its session is keyed on that specific path. When `/rapid:cleanup` removes the worktree, the cwd path is gone ... Run agent sessions with `cwd` set to a stable path (e.g., the project root) ... rather than the worktree path. Pass the worktree path as `additionalDirectories` instead."
- **OVERSIGHTS (Concern #2):** "The SDK session `cwd` must be set to this path [worktree] for worktree isolation to work" -- advocates worktree cwd for isolation.
- **Resolution: cwd = project root; worktree path via `additionalDirectories` + env var.** PITFALLS wins because the retrofit cost of losing transcripts/resume on cleanup is high. Worktree isolation is still achieved because: (a) the spawned skill invokes `cd` or bash-level cwd for git operations, (b) `additionalDirectories` whitelists worktree-scoped Read/Write access, (c) `.planning/` at project root is reachable from project-root cwd anyway (which was OVERSIGHTS's original concern). ARCHITECTURE should set `cwd=project_root` and pass `additional_directories=[worktree_path]`. The spawned skill (not the SDK session) can `cd` into the worktree for its bash operations.

## Recommended Approach

**Build a Python-native agent control plane in-process in FastAPI, grafted onto the existing Mission Control backend.** The Claude Agent SDK runs as Python, owning `ClaudeSDKClient` instances inside a new `AgentSessionManager` in the lifespan hook. Each run is a durable server-side resource with a SQLite row, an in-memory session, and an event bus streaming to the browser via SSE. The browser is a thin control panel -- closing it does not kill the run; reopening re-attaches via polling. This is the load-bearing architectural shape that CLAUDE.md mandates, that PITFALLS and ARCHITECTURE converge on, and that the existing FastAPI+React+SQLite stack already supports without new runtimes.

**Treat "autopilot" as "the user consented to this specific task," not "auto-approve every tool call."** Permission-mode defaults stay conservative. Destructive commands (`git push --force`, `rm -rf`, branch deletion) are unconditionally firewalled via `disallowed_tools` regex + PreToolUse hook. Credentials never enter the agent's environment; the proxy pattern injects `ANTHROPIC_API_KEY` outbound. Per-skill permission policy ships with the runner -- `plan-set`/`execute-set` use `acceptEdits` with scoped disallow-lists; `merge`/`cleanup` stay in `default`. This is the single most important design stance: the CLI UX gets its safety from a human-in-TTY; the webUI replicates that safety through automated gates plus a `can_use_tool` SSE round-trip for ambiguous cases.

**Ship in 4 waves across 5 sets.** Wave 1 delivers the agent runner end-to-end for the 17 autonomous skills (plan-set, execute-set, merge, review, start-set, cleanup, status, pause, resume, backlog, unit-test, bug-hunt, uat, audit-version, documentation, migrate, new-version-planning). Wave 2 lands the web-tool bridge (`can_use_tool` + `mcp__rapid__webui_ask_user` / `ask_free_text`) and the skill launcher UI (YAML frontmatter + catalog endpoint + launcher modal) in parallel -- they are independent (Feature 2 is runtime, Feature 3 is pre-dispatch). Wave 3 lands the Kanban MCP tools + `/rapid:autopilot` skill + Kanban schema v2 in parallel with the WebUI navigation overhaul (new `/agents` and `/chats` routes replacing the provisional Wave 1 `/runs` page). Wave 4 is integration polish, observability, docs, and the browser-native end-to-end walkthrough. The meta-goal "Browser-Native Workflow" is an emergent property of Waves 1-3 landing, not an independent feature.

**Enforce additive-only CLI compatibility as a design constraint.** Every shim is session-scoped via `mcp_servers` registration on the SDK side; CLI sessions do not register the shim and keep built-in behavior. Each new custom MCP tool is gated on `RAPID_RUN_MODE=sdk` so skill prose can branch without forking the skill file. The `Skill` tool chaining gap (built-in in Claude Code harness, absent in SDK) needs an in-process shim implementation or skills that chain must call CLI commands directly. CI smoke-tests `/rapid:execute-set` via both CLI and SDK every build to catch regressions.

**Invest in the five load-bearing foundations in Wave 1 even if they add two weeks:** run mutex (SQLite unique constraint + semaphore), logging substrate with `run_id` correlation, authentication model (localhost-only + optional bearer token), tool shim compatibility audit (all 30 skills), and kanban schema v2 migration. Every one of these has a very-high retrofit cost; skipping them for velocity in Wave 1 creates breaking migrations in Wave 3. The core pattern -- task queue + polling + SSE for streams + SQLite canonical + in-process MCP tools -- is boring, and that is its virtue: every piece is well-understood by FastAPI, the user's CLAUDE.md matches, and the 30 existing skills run verbatim on top.
