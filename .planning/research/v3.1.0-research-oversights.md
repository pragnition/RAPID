# Oversights Research -- v3.1.0 Polish & Cleanup

## Cross-Cutting Concerns

### 1. Status Rename: Backward Compatibility and Migration

The rename from `-ing` suffixes to past tense (`discussing` -> `discussed`, `planning` -> `planned`, `executing` -> `executed`) is a cross-cutting change that touches the Zod schema, transition map, state machine, CLI, display utilities, tests, and agent role prompts. The biggest oversight risk is **existing STATE.json files** in user projects.

**What gets forgotten:**

- **Runtime migration for existing STATE.json files.** Any project currently mid-workflow has sets with `discussing`, `planning`, or `executing` status values persisted in `.planning/STATE.json`. After the rename, `ProjectState.parse()` will reject these as invalid Zod enum values. There is no migration path coded today -- the `readState()` function does `ProjectState.safeParse()` and will return `{ valid: false }` for any project with old status names.
- **Worktree REGISTRY.json phase values.** The `update-phase` command in rapid-tools.cjs (line 1815) has its own hardcoded list of valid phases: `['Discussing', 'Planning', 'Executing', 'Verifying', 'Done', 'Error', 'Paused']`. These are capitalized variants of the same names and must be renamed in sync.
- **STATUS_SORT_ORDER in worktree.cjs** (line 681-688) uses `executing`, `reviewing`, `merging`, `planning` as keys. This must be updated.
- **deriveNextActions in worktree.cjs** (line 782-838) has `case 'executing':`, `case 'reviewing':`, `case 'merging':` switch branches. Note: `reviewing` and `merging` are already removed from the Zod schema (tests at state-schemas.test.cjs:31-36 confirm they are rejected), but the display code still references them -- this is a pre-existing inconsistency that the rename should also clean up.
- **STAGE_VERBS and STAGE_BG in display.cjs** use command names (discuss, execute, etc.) not status names, so these are unaffected. But the distinction is easy to confuse.
- **Hardcoded status string arrays in state-machine.cjs** (lines 243, 254): `['planning', 'executing', 'complete', 'merged']` and `['executing', 'complete', 'merged']` in `validateDiskArtifacts()`.
- **MergeStateSchema in merge.cjs** (line 44-53) has its own independent status enum including `'detecting'`, `'resolving'`, `'merging'`, `'testing'`. The milestone brief only mentions set status renames (`discussing`, `planning`, `executing`), but `merging` in MergeStateSchema refers to the merge pipeline state, not the set lifecycle state -- this distinction must be preserved, not accidentally renamed.
- **Test assertions that validate the exact set of status strings.** The state-schemas.test.cjs (line 25) asserts `['pending', 'discussing', 'planning', 'executing', 'complete', 'merged']`. The state-transitions.test.cjs (line 12) asserts sorted key list. Every test file referencing these strings needs updating.
- **Agent prompt modules.** While grep shows agent role modules mostly reference `.planning/research/` paths rather than status strings, the `executing` status appears in worktree.test.cjs test data extensively (lines 1014, 1015, 1017, 1041, 1042, 1081, 1082, 1088, 1089, 1097, 1127, 1128, 1182).

**Concrete risk:** A user updates RAPID mid-project. Their STATE.json contains `"status": "executing"`. The new code rejects it via Zod. The project is now unreadable until they manually edit STATE.json.

### 2. Parallel Wave Execution: Git Commit Serialization

The milestone calls for enabling concurrent wave execution with "git commit serialization." This is easy to underestimate.

**What gets forgotten:**

- **Lock contention under parallel commits.** The current `acquireLock()` in lock.cjs uses `mkdir` for atomic lock acquisition. When multiple jobs in parallel waves try to commit STATE.json simultaneously, the lock system works -- but the lock's timeout/retry behavior determines whether agents deadlock or succeed. Current lock implementation should be audited for its retry interval and max wait time under parallel load.
- **Commit ordering determinism.** When two waves execute concurrently and both modify STATE.json (e.g., transitioning their respective set statuses), the commit that lands first wins. The second agent will re-read state after lock acquisition, but its `readState()` call happens inside `withStateTransaction()` AFTER lock acquisition -- this is correct. However, if both agents try to commit git changes (not just STATE.json), the commits may interleave in surprising ways on the shared branch.
- **Worktree branch isolation for parallel waves.** Each set already has its own worktree branch. But "parallel waves" within a single set means executing multiple waves of the same set simultaneously. The current worktree model is one-worktree-per-set, not one-worktree-per-wave. Parallel wave execution within a set would require either (a) sequential commits on the same branch, or (b) sub-branches per wave. Neither is currently supported.
- **Wave dependency graph enforcement.** The DAG in dag.cjs handles set-level dependencies. For parallel wave execution, waves within a set are sequentially numbered (Wave 1, Wave 2, etc.) and currently assumed sequential. "Independent waves" implies some mechanism to determine which waves can overlap -- this requires a new intra-set dependency model.
- **Error handling for partial wave completion.** If Wave 1 has jobs A and B running in parallel, and job A succeeds but job B fails, what happens? The current reconciliation model (`reconcileWave`, `reconcileWaveJobs`) assumes all jobs complete before reconciliation runs. Partial completion tracking per-job is needed, and the execute.cjs `formatProgressBanner()` function already supports per-job status -- but the orchestration logic to pause-and-retry individual jobs while leaving completed ones alone does not exist.
- **STATE.json wave/job transitions are broken.** The CLI at rapid-tools.cjs:355 calls `sm.transitionWave()` and at line 365 calls `sm.transitionJob()`, but these functions do NOT exist in state-machine.cjs. The tests at state-machine.test.cjs:273-278 explicitly assert they are `undefined`. This means wave/job state transitions will crash at runtime. This is a pre-existing bug that must be fixed before parallel wave execution can work.
- **Wave and job state not in the Zod schema.** The `ProjectState` schema in state-schemas.cjs defines `SetState` with `id` and `status`, but there is no `WaveState` or `JobState` Zod schema. The rapid-tools.cjs test at line 1169 asserts `job.status === 'executing'`, implying jobs have status fields, but these are not validated by any schema. Parallel wave execution needs schema-validated wave and job states.
- **Reconciliation timing.** The `reconcileWave()` function in execute.cjs checks contract test compliance, artifact existence, and commit counts. Under parallel execution, reconciliation cannot run until ALL parallel jobs complete. But how does the orchestrator know when all parallel jobs are done? Currently there is no aggregation mechanism -- each agent reports its own COMPLETE/CHECKPOINT/BLOCKED return independently. An aggregator that collects returns from parallel agents and triggers reconciliation only when all have reported is needed.
- **HANDOFF.md conflicts under parallelism.** The `generateHandoff()` function writes to `.planning/sets/{setName}/HANDOFF.md`. If two parallel jobs within the same set both checkpoint, they would overwrite each other's HANDOFF.md. The job-level `generateJobHandoff()` function mitigates this by writing to separate per-job files, but the set-level pause mechanism does not account for partial completion of parallel work.

### 3. Old Command Cleanup: Identifying the Full Deprecation Surface

The milestone says "Remove deprecation stubs for old commands." The codebase has evolved through v1.0 -> v1.1 -> v2.0 -> v2.1 -> v3.0 with several command renames.

**What gets forgotten:**

- **TOOL_REGISTRY in tool-docs.cjs still lists wave-plan commands** (lines 66-69): `wave-plan-resolve`, `wave-plan-create-dir`, `wave-plan-validate`, `wave-plan-list-jobs`. If these were collapsed into plan-set in v3.0, the tool-docs references should be removed.
- **ROLE_TOOL_MAP references.** When commands are removed from TOOL_REGISTRY, any ROLE_TOOL_MAP entries referencing them will cause `getToolDocsForRole()` to throw an error (line 154: `throw new Error('Unknown tool key...')`). This is a runtime crash if a role references a removed command.
- **Agent modules referencing removed commands.** The built agent .md files in `.claude/agents/` (if generated) embed command references. A `build-agents` run is required after any command removal.
- **Existing HANDOFF.md or LAST_RETURN.json files** may reference old command names in their resume instructions. These are user data artifacts that cannot be retroactively updated.

### 4. Install Command Version Awareness: Version Detection Strategy

The milestone wants install to "detect outdated RAPID_TOOLS and reinstall/update."

**What gets forgotten:**

- **Version exists in package.json but is not exposed to CLI.** The package.json has `"version": "3.0.0"`, but rapid-tools.cjs has no `--version` flag to query it. The install skill needs to be able to run `node rapid-tools.cjs --version` and compare the result against a cached version.
- **The .env file stores a path, not a version.** The current `.env` just has `RAPID_TOOLS=/path/to/rapid-tools.cjs`. Version detection requires either (a) embedding a version in rapid-tools.cjs that can be queried, (b) using git commit hash of the RAPID repo, or (c) comparing file checksums.
- **Shell config persistence across updates.** The setup.sh writes to `.env` and optionally runs `claude plugin add`. If the path changes during an update (e.g., user moves the RAPID directory), all existing shell configs and .env files point to the old location.
- **The `detect_install_method()` function** in setup.sh (line 14) only distinguishes "marketplace" vs "clone". Version-aware updates need different strategies for each (marketplace = wait for update, clone = git pull + npm install).
- **node_modules staleness.** setup.sh skips `npm install` if `node_modules` exists (line 54). A version update may add new dependencies. The install command should check `package-lock.json` timestamp against `node_modules` mtime or run `npm ci` on updates.
- **build-agents regeneration on update.** When a user updates RAPID, the agent .md files in `.claude/agents/` may be stale (built from old module sources). The install/update flow should automatically run `build-agents` after detecting a version change. Currently setup.sh does this (line 80), but a version-aware reinstall path might skip it.
- **Multiple RAPID installations.** A user might have RAPID installed in two locations (e.g., a development clone and a production clone). The .env file in each project's RAPID directory would point to different paths. The install command needs to handle the case where `RAPID_TOOLS` in the shell environment points to installation A but the user is running `/rapid:install` from installation B.
- **package.json version field already exists.** The package.json has `"version": "3.0.0"` which should be bumped to `"3.1.0"` for this milestone. The install command can compare `require('../package.json').version` against a cached version stored in the project's .env (e.g., `RAPID_VERSION=3.0.0`). This is the standard Node.js convention and avoids inventing a custom version mechanism.

### 5. Error Handling Strategy for Status Rename

- **Error messages embed status names.** The `validateTransition()` function in state-transitions.cjs builds error messages like `"Valid transitions from 'discussing': [planning]"`. After the rename, these become `"Valid transitions from 'discussed': [planned]"`. While technically correct, the past-tense naming makes error messages read oddly: "discussed -> planned" sounds like describing what already happened, not what is allowed. This is a UX concern worth acknowledging.
- **Cascading error on corrupted state.** If a user's STATE.json has old status names, `readState()` returns `{ valid: false }`. Every downstream function that calls `readState()` will see this as corruption. The `detectCorruption()` function at state-machine.cjs:275 will flag it as corrupt. The `recoverFromGit()` function will try to restore from the last git commit -- which may also have the old status name. There is no path forward except manual editing.

### 6. Testing Strategy for Cross-Cutting Changes

**What gets forgotten:**

- **Test files affected by the status rename are numerous.** At minimum: state-schemas.test.cjs, state-transitions.test.cjs, state-machine.test.cjs, state-machine.lifecycle.test.cjs, worktree.test.cjs, rapid-tools.test.cjs. A grep found 80+ lines across test files referencing `discussing`, `planning`, or `executing` as status values. Missing even one will cause test failures.
- **Integration test coverage for parallel wave execution.** No integration test infrastructure exists for testing concurrent agent execution. The current test suite uses node:test with synchronous assertions. Parallel execution testing needs either (a) spawning multiple processes, (b) mocking concurrency with interleaved async calls, or (c) a dedicated concurrency test harness.
- **Snapshot-style assertions.** Tests like state-transitions.test.cjs:12 assert the exact sorted key list `['complete', 'discussing', 'executing', 'merged', 'pending', 'planning']`. These are effectively snapshot tests that must be updated atomically with the schema change.

### 8. CI/CD Pipeline

Not applicable. RAPID is a Claude Code plugin that does not have its own CI/CD pipeline. Tests are run manually via `node --test`. However:

- **Pre-commit verification.** There is no pre-commit hook or CI gate. A developer could commit a partial rename (e.g., update the schema but forget the tests), and the project would be in a broken state until the next test run.
- **build-agents pipeline.** After any status rename or command removal, `node rapid-tools.cjs build-agents` must be run to regenerate `.claude/agents/` files. This is easy to forget because the build step is not automatic.

### 9. Developer Experience

- **Atomic rename strategy.** The status rename touches ~10 source files and ~6 test files simultaneously. A partial rename (e.g., schema updated but transitions not) will cause immediate runtime crashes. All changes must land in a single commit.
- **Search-and-replace safety.** Naive find-replace of `discussing` -> `discussed` could accidentally modify non-status occurrences: the word `discussing` appears in agent prompt text (e.g., "You are reviewing the set before implementation begins" uses `discussing` as a phase name in prompts, not as a status value). The word `planning` appears 160+ times across the codebase in comments, descriptions, directory names (`.planning/`), and documentation -- not all of which should be renamed.
- **The `VALID_PHASES` array in execute.cjs** (line 26) is `['discuss', 'plan', 'execute']`. These are phase/command names, NOT status values. They must NOT be renamed. The distinction between phase names (verb: discuss) and status values (state: discussed/discussing) is critical and easy to confuse.

### 12. Data Concerns: STATE.json Migration

- **No schema version field in STATE.json.** The `ProjectState` schema has `version: z.literal(1)` but this is a schema version, not a data format version. There is no mechanism to detect "this STATE.json uses v3.0 status names vs v3.1.0 status names."
- **Recommendation:** Add a migration layer to `readState()` that detects old status names and auto-migrates them. This is cheap to implement (a simple string map applied before Zod parsing) and prevents the hard-crash scenario described in Section 1.
- **Git recovery limitation.** The `recoverFromGit()` function does `git checkout HEAD -- .planning/STATE.json`. If HEAD already has the new code but the STATE.json in git also has old names, recovery will not help. The migration layer must be in the read path, not the write path.

## Recommendations

1. **STATE.json migration layer (MUST DO FIRST):** Before renaming any status values, implement a migration function in `readState()` that maps `discussing -> discussed`, `planning -> planned`, `executing -> executed` before Zod parsing. This is ~10 lines of code and prevents breaking every active user's project. Without this, the rename is a breaking change with no recovery path.

2. **Fix transitionWave/transitionJob before parallel wave work:** The CLI calls `sm.transitionWave()` and `sm.transitionJob()` which do not exist. The tests explicitly assert they are undefined. This is a blocking prerequisite for the parallel wave execution feature -- either implement these functions or remove the CLI entry points.

3. **Single-commit atomic rename:** The status rename must touch state-schemas.cjs, state-transitions.cjs, state-machine.cjs (validateDiskArtifacts), worktree.cjs (STATUS_SORT_ORDER, deriveNextActions), rapid-tools.cjs (update-phase valid phases), and all corresponding test files in a single commit. Use a script or careful grep to ensure no occurrences are missed. Be especially careful to distinguish status values (`discussing`) from phase names (`discuss`), command names (`/discuss-set`), directory names (`.planning/`), and natural language in prompts.

4. **Identify the full old-command surface before removal:** Audit TOOL_REGISTRY, ROLE_TOOL_MAP, and rapid-tools.cjs USAGE string together. Any command removed from the CLI must also be removed from TOOL_REGISTRY, and any ROLE_TOOL_MAP entry referencing it must be updated. Run `build-agents` after removal and verify no agent prompts reference deleted commands.

5. **Expose version via CLI flag:** The package.json already has `"version": "3.0.0"`. Add a `--version` flag to rapid-tools.cjs that reads and outputs `require('./package.json').version`. Store the installed version in the project's `.env` as `RAPID_VERSION=3.0.0` so the install command can compare current vs installed versions. Bump package.json to `3.1.0` for this milestone.

6. **Design the parallel wave dependency model:** Before implementing parallel wave execution, decide how waves within a set declare independence. Options: (a) all waves are independent by default unless edges exist, (b) waves are sequential by default with explicit `can_parallel: true` markers, (c) use the existing DAG edge system extended to wave-type nodes. The DAG already supports wave-type nodes via `createDAGv2()` -- leverage this.

7. **Run the full test suite after each sub-change:** With 80+ test assertions touching status strings, incremental verification after each file change prevents cascading failures.
