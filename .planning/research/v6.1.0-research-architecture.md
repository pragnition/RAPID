# Architecture Research

## Current Architecture

RAPID is a Claude Code plugin organized into five layers: Skill Layer (24 skills), Agent Layer (26 agents), CLI Layer (rapid-tools.cjs router), Library Layer (21 modules in src/lib/), and Storage Layer (.planning/ state files + .rapid-worktrees/ git worktrees). The architecture follows a modular monolith pattern with clear separation between layers.

### Strengths Relevant to This Milestone

- **File-based persistence already exists.** DEFERRED.md, CONTRACT.json, and STATE.json provide persistent artifacts that survive `/clear`. The audit-to-set handoff problem is not a storage problem -- the persistence layer is robust. The problem is a _routing_ problem: audit output tells users to run `/rapid:add-set` but provides no way to carry context forward.
- **Progress breadcrumbs are partially implemented.** Skills like start-set, discuss-set, plan-set, and execute-set already render breadcrumbs (`init [done] > start-set [done] > discuss-set > ...`). However, these breadcrumbs do not include a `/clear` reminder, and not all skills render them consistently.
- **Display module is minimal and extensible.** The display.cjs module currently only handles banner rendering (ANSI-colored stage headers). It has no concept of a "footer" or "next-step" renderer, which would need to be added for systematic `/clear` guidance.
- **Memory system (DECISIONS.jsonl / CORRECTIONS.jsonl) provides structured cross-session persistence.** This system could be leveraged for handoff context, though its category taxonomy (architecture, integration, ux, etc.) does not natively include an "audit-remediation" category.

### Gaps Requiring Architectural Extension

1. **No file-based handoff between audit and add-set.** The audit skill generates an AUDIT.md report and prints remediation commands to the terminal. After `/clear`, the terminal output is lost. The add-set skill has no mechanism to ingest pre-populated scope from an audit output.
2. **No consistent "end-of-command" footer pattern.** Each skill implements its own next-step suggestion with varying formats. There is no shared footer that could uniformly inject `/clear` guidance.
3. **README lacks progressive disclosure.** The current README dives into architecture and command reference immediately. There is no "what problem does this solve" framing that would help beginners understand context rot and why `/clear` matters.

## Recommended Architecture Pattern

- **Pattern:** Targeted extension of the existing modular monolith -- new file-based handoff artifact plus standardized UX patterns in skill prompts
- **Rationale:** The milestone's scope is UX-focused, not structural. The existing five-layer architecture is sound and does not require reorganization. All four features (handoff, README, clear-guidance, UX audit) can be achieved through: (a) a new persistent artifact type, (b) skill prompt updates, and (c) documentation improvements. No new modules or architectural layers are needed.
- **Evidence:** RAPID's existing DEFERRED.md pattern demonstrates that file-based handoff between commands works well. The pattern of "write a structured file, read it in the next command" is proven across the discuss-set -> plan-set -> execute-set pipeline. Extending this pattern to audit -> add-set is architecturally consistent.

## Project Structure

The milestone does not require structural changes to the directory layout. The affected areas are:

```
.planning/
  v{VERSION}-AUDIT.md          # Existing -- audit report output
  v{VERSION}-REMEDIATION.json  # NEW -- structured remediation handoff artifact
  sets/
    {set-id}/
      DEFINITION.md            # Existing -- modified by add-set to accept pre-populated scope

skills/
  audit-version/SKILL.md       # MODIFIED -- writes REMEDIATION.json alongside AUDIT.md
  add-set/SKILL.md             # MODIFIED -- reads REMEDIATION.json for pre-populated scope
  */SKILL.md                   # MODIFIED -- standardized end-of-command footer with /clear guidance

docs/
  setup.md                     # MODIFIED -- beginner-friendly language
  
README.md                      # MODIFIED -- restructured for progressive disclosure
```

### Rationale for Structure Decisions

No new directories or modules are introduced because:
1. The remediation handoff is a single JSON file in the existing .planning/ directory
2. UX improvements are skill prompt changes, not code changes
3. The README rewrite replaces the existing file in-place
4. The `/clear` guidance is a pattern change across existing skills, not a new system

## Module Organization

### Module: Audit-Version Skill (skills/audit-version/SKILL.md)

- **Responsibility:** Produce gap analysis report and structured remediation artifact
- **Public interface:** Writes `.planning/v{VERSION}-AUDIT.md` (existing) and `.planning/v{VERSION}-REMEDIATION.json` (new)
- **Dependencies:** rapid-tools.cjs (state, display), rapid-auditor agent
- **Data ownership:** Owns AUDIT.md and REMEDIATION.json for the audited version

### Module: Add-Set Skill (skills/add-set/SKILL.md)

- **Responsibility:** Interactive set creation with optional pre-populated scope from audit remediation
- **Public interface:** Creates `.planning/sets/{id}/` directory with DEFINITION.md and CONTRACT.json
- **Dependencies:** rapid-tools.cjs (state add-set, display), REMEDIATION.json (optional read)
- **Data ownership:** Owns the newly created set directory

### Module: Display Library (src/lib/display.cjs)

- **Responsibility:** Branded banner rendering with ANSI color codes
- **Public interface:** `renderBanner(stage, target)`, `STAGE_VERBS`, `STAGE_BG`
- **Dependencies:** None (zero dependencies by design)
- **Data ownership:** None (pure rendering utility)
- **Extension for this milestone:** Potentially add a `renderFooter(stage, setIndex)` function that produces the standardized next-step + clear guidance. However, since skills are prompt-driven (not code-driven), the more practical approach is standardizing the footer pattern in skill prompts directly.

### Module: README.md (project root)

- **Responsibility:** First-contact documentation for new users
- **Public interface:** GitHub rendering, npm package description
- **Dependencies:** Branding assets (SVGs in branding/)
- **Data ownership:** N/A

## Data Flow

### Audit-to-Set Handoff (New Flow)

```
1. /rapid:audit-version
   a. Spawns rapid-auditor agent -> produces GAP_DATA
   b. Writes .planning/v{VERSION}-AUDIT.md (existing behavior)
   c. For each "Create remediation set" decision:
      - Collects set name + scope + remediation description
   d. Writes .planning/v{VERSION}-REMEDIATION.json (NEW)
   e. Prints remediation commands (existing behavior, now also says "context persisted in REMEDIATION.json")

2. User runs /clear (context is lost from conversation, but REMEDIATION.json persists on disk)

3. /rapid:add-set [set-name]
   a. Checks for .planning/v*-REMEDIATION.json (newest by version)
   b. If found and set-name matches a remediation entry:
      - Pre-populates SET_SCOPE from remediation description
      - Pre-populates SET_FILES_AND_DEPS from remediation context
      - Skips interactive discovery (Step 2) or uses it for confirmation only
   c. If not found or no match: falls through to standard interactive flow (backward compatible)
```

### REMEDIATION.json Schema

```json
{
  "version": "v6.0.0",
  "milestone": "Scale & Quality",
  "auditedAt": "2026-04-06",
  "remediations": [
    {
      "setName": "fix-stub-cleanup",
      "requirement": "Merge-time auto-deletion of stub sidecar files",
      "scope": "Wire cleanupStubSidecars() into merge pipeline post-resolution",
      "severity": "medium",
      "suggestedFiles": ["src/lib/merge.cjs", "src/lib/stub.cjs"],
      "suggestedDeps": []
    }
  ]
}
```

This schema is intentionally minimal. The `setName` field is keyed by the user's response during audit remediation (Step 4e). The `scope` and `suggestedFiles` provide the handoff context that would otherwise be lost on `/clear`.

### Request Lifecycle: Skill Invocation

No changes to the core request lifecycle. The existing pattern remains:

```
1. User types /rapid:{skill-name} [args]
2. Claude Code loads skills/{skill-name}/SKILL.md
3. Skill orchestrates CLI calls via node "${RAPID_TOOLS}" ...
4. CLI dispatches to src/commands/{handler}.cjs
5. Handler calls src/lib/{module}.cjs for business logic
6. Results written to .planning/ artifacts
7. Skill displays next-step suggestion + progress breadcrumb
   (NEW: + /clear reminder in standardized footer)
```

### State Management

No changes to state management. The REMEDIATION.json artifact is NOT part of STATE.json -- it is a standalone file in .planning/ that is read opportunistically by add-set. This avoids:
- Schema migration in state-schemas.cjs
- Zod validation changes
- withStateTransaction complexity

The file is written by audit-version (Write tool) and read by add-set (Read tool). There is no concurrent access concern because these commands run sequentially by design.

## API Design (CLI)

### Existing CLI Commands -- No Changes

The rapid-tools.cjs CLI does not need new subcommands for this milestone. The handoff mechanism operates entirely through file I/O in skill prompts.

### Potential CLI Enhancement (Optional)

If a CLI-based approach is preferred over skill-prompt-based file I/O:

```bash
# Write remediation entries
node "${RAPID_TOOLS}" audit write-remediation --version v6.0.0 --set-name "fix-stub" --scope "..." --severity medium

# Read remediation entries
node "${RAPID_TOOLS}" audit read-remediation --version v6.0.0 [--set-name "fix-stub"]
```

However, this is likely unnecessary. The skill prompt can directly use the Write/Read tools to manage the JSON file. Adding CLI commands would increase code surface area without meaningful benefit, since the only consumers are two skill prompts.

**Recommendation:** Use direct file I/O in skill prompts. Do not add CLI commands.

## Scaling Considerations

This milestone is UX-focused and does not introduce scaling concerns. The REMEDIATION.json file is small (typically < 1KB), written once per audit, and read once per add-set invocation.

The only scaling consideration is the number of skills that need `/clear` guidance added. There are approximately 20+ skill SKILL.md files. The `/clear` guidance should be added as a standardized pattern (copy-paste or template) rather than through code generation, to avoid over-engineering.

### Approach to Breadth of Skill Updates

Two strategies for adding `/clear` guidance across all skills:

**Strategy A: Targeted update (recommended).** Add `/clear` guidance only to skills that produce a "Next step" suggestion (approximately 10 skills: start-set, discuss-set, plan-set, execute-set, review, merge, audit-version, add-set, new-version, init). These are the skills where the user will actually run another RAPID command next.

**Strategy B: Universal update.** Add `/clear` guidance to all 24 skills. This is more thorough but includes skills like `/rapid:status` (read-only, no next step) and `/rapid:quick` (self-contained, no next step) where the guidance is unnecessary.

## Architecture Risks

1. **REMEDIATION.json staleness:** If the user runs audit, creates REMEDIATION.json, but never runs add-set, the file lingers. Multiple audits could create multiple REMEDIATION.json files. **Impact:** Low -- the file is small and harmless. **Mitigation:** add-set reads the newest version; cleanup can optionally delete consumed entries after set creation.

2. **Skill prompt drift in /clear guidance:** Adding a footer pattern to 10+ skill prompts creates a maintenance burden. If the footer wording changes, all skills must be updated. **Impact:** Medium -- inconsistency degrades UX. **Mitigation:** Keep the footer text short and formulaic so updates are trivial. Consider a shared snippet in CLAUDE.md or a display CLI subcommand that renders the footer programmatically.

3. **README length vs. comprehension trade-off:** A beginner-friendly README needs more text (explaining context rot, showing the problem, progressive disclosure) but RAPID's current README is already 130+ lines. **Impact:** Low -- README length is acceptable for a tool of this complexity. **Mitigation:** Use collapsible sections (`<details>`) for advanced content; keep the core quickstart path short.

4. **Backward compatibility of add-set with REMEDIATION.json:** If REMEDIATION.json does not exist (projects that never ran audit, or older versions), add-set must fall through cleanly to the interactive flow. **Impact:** None if implemented correctly -- the check is a simple `fs.existsSync`. **Mitigation:** Guard clause at the top of the add-set flow; existing interactive path is the default.

## Recommendations

### Ordered by priority:

1. **Audit-to-set handoff via REMEDIATION.json:** Create a new persistent artifact written by audit-version and consumed by add-set. This is the core architectural decision for the milestone. **Rationale:** File-based handoff is the proven pattern in RAPID (DEFERRED.md, CONTRACT.json, CONTEXT.md all follow this pattern). It survives `/clear` by design. **Priority: critical**

2. **Standardized end-of-command footer with /clear guidance:** Define a consistent footer pattern for skill prompts that includes the next-step suggestion AND a `/clear` reminder. Pattern:
   ```
   > **Next step:** `/rapid:{next-command} {args}`
   > Tip: Run `/clear` before your next RAPID command to free context.
   
   init [done] > start-set [done] > discuss-set > plan-set > execute-set > review > merge
   ```
   **Rationale:** Users lose significant context window by not clearing between commands. A consistent reminder at every command boundary addresses this. **Priority: high**

3. **README restructure for progressive disclosure:** Rewrite the README to lead with the problem statement (context rot, parallel development coordination), then show the solution, then the quickstart, then the architecture. Move the current architecture diagrams and detailed descriptions into collapsible sections or separate documentation pages. **Rationale:** The current README assumes the reader already understands why they need RAPID. Beginners need the "why" before the "how". **Priority: high**

4. **General UX audit of command output consistency:** Review all skill prompts for inconsistencies in: breadcrumb format, next-step suggestion format, error message format, and banner rendering. Standardize where inconsistencies are found. This is discovery-driven work -- scope depends on what the audit finds. **Rationale:** Consistent output patterns reduce cognitive load and make RAPID feel more polished. **Priority: medium**

5. **Deferred decisions pass-through:** The four deferred items from v6.0.0 (LLM-based build-agents validation, execSync migration, comprehensive execSync audit, auto-regroup on DAG changes) should be captured in the milestone's planning artifacts without implementation. They are carry-forward context for future milestones. **Rationale:** These items are explicitly deferred and should not consume implementation effort in this milestone. **Priority: low**
