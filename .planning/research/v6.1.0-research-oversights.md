# Oversights Research: v6.1.0 UX & Onboarding

## Cross-Cutting Concerns

### 1. Audit-to-Set Handoff Persistence

The audit-version skill (Step 4e) prints remediation commands like `/rapid:add-set fix-auth-validation (scope: ...)` to the screen. After the user runs `/clear` (which they are expected to do after every command), all that context is gone. The add-set skill (Step 2) then asks "What should this new set accomplish?" from scratch -- the user must manually re-type or remember the audit's remediation recommendation.

**What gets forgotten:** The handoff data that needs to survive `/clear` includes: (a) the requirement description from the audit gap, (b) the recommended remediation scope, (c) the severity level, and (d) which original set was responsible. None of this is persisted to a file that add-set reads.

**Why it matters for this milestone:** This is the core feature request in the milestone brief. The v6.0.0-AUDIT.md and v{version}-DEFERRED.md files already exist on disk after audit-version runs. The DEFERRED.md has a `Carry-Forward Context` column that holds remediation recommendations. However, neither add-set nor new-version currently reads audit remediation recommendations proactively. The deferred pipeline flows into new-version (Step 2C-v reads DEFERRED.md files), but the remediation-to-add-set pipeline has no file-based bridge at all.

**Concrete handoff mechanism options:**
1. Write a `.planning/REMEDIATION-QUEUE.md` file during audit-version Step 4e with structured entries that add-set can auto-read in Step 2
2. Extend the existing DEFERRED.md format with a `type: remediation` field and have add-set check for pending remediation items on startup
3. Write individual `.planning/sets/{set-name}/HANDOFF.md` pre-creation files that add-set discovers and pre-populates from

The key constraint: add-set must be able to discover pending handoffs without the user providing any context. It should detect them automatically and offer to use them.

### 2. Clear-Before-Next-Command Guidance (Missing from ALL Skills)

A grep of all 24 skill SKILL.md files reveals that zero of them include `/clear` guidance in their completion output. Every skill has a "Next step" suggestion (e.g., "Next step: /rapid:plan-set 3") and a progress breadcrumb, but none say "Run /clear before your next command."

**What gets forgotten:** Without this guidance, users accumulate context across commands. For small projects this is harmless, but for RAPID's typical use case (multi-set milestones with 6+ research agents, executors, and reviewers), a single conversation can easily exceed 200K tokens. Context compaction kicks in unpredictably, causing agent instructions to get truncated. The whole architecture is designed for `/clear`-resilience, but users do not know they should be doing it.

**Why it matters for this milestone:** The milestone brief explicitly calls this out: "Tell users to /clear before the next command (at the end of each command)." This is a cross-cutting change that affects every skill's completion output, not just one skill.

**Retrofit cost:** Low if implemented as a standard footer pattern. The change touches all 24 SKILL.md files but each change is mechanical: add a line after the progress breadcrumb. The risk is inconsistency -- some skills might get the footer and others might not, creating an uneven experience.

**Recommended approach:** Define the footer once in a shared pattern (could be documented in CONVENTIONS.md or as a display.cjs utility), then apply it uniformly across all skills. The text should be concise and non-alarming -- something like:

```
Tip: Run /clear before your next RAPID command to keep context fresh.
```

### 3. README Install Instructions Inconsistency

The README.md uses:
```
/plugin install rapid@pragnition/pragnition-public-plugins
```

The DOCS.md uses:
```
claude plugin add pragnition/RAPID
```

These are two completely different install mechanisms pointing at two different repositories. The README references `pragnition-public-plugins` (a marketplace aggregator repo), while DOCS.md references `pragnition/RAPID` (the direct repo). A beginner encountering both will be confused about which is correct.

Additionally, the README install section lacks any explanation of what happens after installation. It jumps from the install command to a "60-Second Quickstart" with 10 commands. A beginner does not know:
- What `/rapid:install` does or why it is needed after `plugin install`
- What RAPID_TOOLS is or why it matters
- That they need Node.js 20+ and git 2.30+ installed first
- What Claude Code even is (the README assumes this knowledge)

**Why it matters:** The milestone brief explicitly states "Install instructions are confusing for beginners." The current README optimizes for brevity over clarity. It fails the "cold open" test: a developer who finds RAPID on GitHub cannot get it running without prior knowledge of the Claude Code plugin ecosystem.

### 4. README Does Not Explain the Problem RAPID Solves

The README opens with a quote about collaboration, then immediately dives into features (isolation, ownership, contracts, merge strategy). It never clearly states the core problem: "AI coding agents lose context when conversations get long, and Claude Code's /clear is the only reliable way to manage this -- but /clear means every command must be self-contained, and coordinating parallel work across /clear boundaries requires careful state management."

**What gets forgotten:** Without a clear problem statement, beginners cannot evaluate whether RAPID is relevant to them. They see "parallel development" and think "git branches," missing that RAPID's value proposition is specifically about orchestrating AI agents across context boundaries.

**Why it matters:** The milestone brief says "it is not clear that we are trying to solve the problem of context rot by allowing users to /clear after every command." This is a documentation oversight, not a code change, but it is the single most important thing for onboarding new users.

### 5. Breadcrumb and Next-Step Inconsistency Across Skills

After auditing all skill completion outputs, the patterns are inconsistent:

| Skill | Has Next Step | Has Breadcrumb | Has /clear Tip |
|-------|:---:|:---:|:---:|
| init | Yes | Yes | No |
| start-set | Yes | Yes | No |
| discuss-set | Yes | Yes | No |
| plan-set | Yes | Yes | No |
| execute-set | Yes | Yes | No |
| review | Yes | No | No |
| merge | Yes | No | No |
| add-set | Yes | Yes | No |
| new-version | Yes | Yes | No |
| audit-version | Yes | No | No |
| status | Yes | No | No |
| unit-test | Yes | No | No |
| bug-hunt | Yes | No | No |
| uat | Yes | No | No |
| quick | No | No | No |
| branding | No | No | No |
| scaffold | Yes | No | No |
| bug-fix | No | No | No |

The review pipeline commands (review, unit-test, bug-hunt, uat) and lifecycle commands (merge, audit-version, new-version) do not show progress breadcrumbs. The core lifecycle commands (init through execute-set) and add-set do.

**What gets forgotten:** A user completing a review or merge has no visual indicator of where they are in the overall workflow. They know what to do next (the "Next step" line tells them), but they lack the at-a-glance progress overview.

**Retrofit cost:** Low-moderate. Adding breadcrumbs to the ~10 skills that lack them is straightforward, but the breadcrumb content varies by context (standard path vs gap-closure vs post-merge review). Getting the breadcrumb correct for each skill path requires understanding that skill's state machine.

### 6. Help Command Shows Stale Information

The `/rapid:help` skill (SKILL.md) has `disable-model-invocation: true` and outputs static content. It states:
- "Add sets mid-milestone (Phase 44 -- not yet available)" -- but add-set IS available and has been since v4.x
- References "v6.0.0" throughout -- will need updating for v6.1.0
- Does not mention cleanup, bug-fix, scaffold, branding, context, documentation, register-web, migrate, pause, or resume in the core tables (some are in "Kept Utilities" but several are missing entirely)

**What gets forgotten:** Because help is static (`disable-model-invocation: true`), it cannot dynamically reflect the current version or available commands. Every version bump requires a manual update to the help skill. If this milestone adds new commands or changes existing ones, the help text will be stale immediately.

**Why it matters:** Help is often the first command a new user runs. Stale content creates immediate distrust.

### 7. Error Message UX -- No Recovery Guidance Pattern

The error handling convention is `process.stderr.write('[RAPID ERROR] message\n')` but there is no convention for including recovery guidance. Skills include "Error breadcrumb" patterns, but the underlying library modules (core.cjs, state-machine.cjs, worktree.cjs) emit raw error messages without actionable next steps.

**Example:** If `state get --all` fails because STATE.json is corrupted, the user sees `[RAPID ERROR] Failed to parse STATE.json: Unexpected token...`. They do not see: "Try running /rapid:init to reinitialize, or check .planning/STATE.json manually."

**What gets forgotten:** Skills wrap CLI errors with breadcrumbs and recovery guidance, but if the skill itself fails to handle an error case, the raw library error bubbles up with no context. This is particularly problematic for beginners who do not know the internal architecture.

**Retrofit cost:** Moderate. Adding recovery hints to library-level errors means touching core.cjs, state-machine.cjs, and worktree.cjs. However, the scope can be limited to the most common failure modes (STATE.json missing/corrupt, worktree creation failures, git branch conflicts).

### 8. Status Command Does Not Show Deferred/Remediation Items

The `/rapid:status` skill displays sets and their statuses, but it does not surface pending remediation items from audit reports or deferred items from previous milestones. A user running `/rapid:status` after an audit gets no indication that there are outstanding gaps to address.

**What gets forgotten:** The handoff gap is not just about audit-to-add-set. It is also about status visibility. If the user clears context and runs `/rapid:status`, they should see that there are pending remediation items, not just set statuses. This connects directly to the audit-to-set handoff concern (Section 1) but from the status/dashboard perspective.

**Retrofit cost:** Low. The status skill already loads STATE.json. Adding a check for `.planning/v*-AUDIT.md` and `.planning/v*-DEFERRED.md` files and displaying a summary line ("2 remediation items pending from v6.0.0 audit") is a small addition.

### 9. Onboarding Flow -- No "First Run" Experience

After a user installs RAPID and runs `/rapid:install`, the install skill offers next actions (help, init, status, register-web, done). But there is no guided onboarding path for someone who has never used RAPID before. The transition from install to init is:
1. User picks "Run /rapid:init" from install's completion options
2. BUT the install skill says "Run /rapid:install inside Claude Code" -- meaning the user is already IN Claude Code
3. The init flow immediately asks complex questions (project description, team size, spec file)

There is no intermediate step that explains:
- What RAPID will do to their project (create .planning/, modify git branches)
- That RAPID is designed around `/clear` -- they should clear between every command
- What the lifecycle looks like at a high level (init -> start-set -> ... -> merge)
- That they can run `/rapid:help` to see the command reference anytime

**What gets forgotten:** The install -> init transition assumes the user already understands RAPID's mental model. A 30-second "here's what you're about to do" orientation would dramatically reduce confusion.

**Retrofit cost:** Low. This is a documentation/prompt addition to either the install completion step or the init preamble, not a code change.

### 10. Testing Strategy for UX Changes

UX changes to skills are primarily Markdown modifications (SKILL.md files). The existing test infrastructure includes:
- `skills/review/review-auto-detect.test.cjs` -- one skill test
- No other skill-level tests in the skills/ directory
- `discuss-set` has structural tests referenced in the v4.2.0 oversights report (asserting option counts)

**What gets forgotten:** When modifying 24 SKILL.md files to add `/clear` guidance, there is no automated way to verify the changes are consistent. If one skill gets the wrong text or misses the footer, there is no test to catch it. Similarly, the help command's static content has no test verifying it matches the actual command inventory.

**Why it matters:** The breadcrumb and `/clear` guidance changes are mechanical but numerous. Without validation, regressions will go unnoticed until a user encounters them.

**Recommended approach:** Consider adding a lightweight structural test that greps all SKILL.md files for required patterns (e.g., every SKILL.md must contain "Run /clear" in its completion section, every help output must list all commands in the skills/ directory).

### 11. Version String Propagation

The project currently has `6.0.0` hardcoded in: README.md (badge), DOCS.md (header), plugin.json (version field), package.json (version field), skills/help/SKILL.md (title), skills/install/SKILL.md (title and references), skills/status/SKILL.md (references), technical_documentation.md, and the CHANGELOG header.

If this milestone bumps the version to 6.1.0, all of these files need coordinated updates. The v6.0.0 milestone included a dedicated `docs-version-bump` set for this, but there is no automation or script that updates all version references.

**What gets forgotten:** Version bumps are manual and error-prone. The v4.2.0 oversights report did not flag this because v4.2.0 had a dedicated version bump set. This milestone may or may not need one, but the concern should be tracked.

**Retrofit cost:** Very low. A `sed` command or a small script in rapid-tools.cjs could handle this, but it is not critical for this milestone if version does not bump.

---

### 12. Accessibility of RAPID's Output -- Terminal Readability

RAPID's display.cjs uses ANSI escape codes for banners and respects `NO_COLOR`. However, the progress breadcrumbs, completion summaries, and next-step guidance are rendered as raw Markdown in the terminal. Markdown tables (e.g., in status output) do not render as tables in a terminal -- they display as pipe-separated text.

**What gets forgotten:** The status command outputs Markdown tables that look great in rendered Markdown viewers but are hard to read as raw text in Claude Code's response. This is a minor UX concern but affects readability for the dashboard, which is one of the most-used commands.

**Why it matters:** For a UX-focused milestone, the output formatting is part of the user experience. However, Claude Code renders Markdown in its UI, so this may be a non-issue in practice. The concern is specifically about users who read output in non-rendering terminals.

**Retrofit cost:** Low if addressed. The display.cjs module already exists and could be extended with a `renderTable()` function that produces fixed-width aligned output. However, since Claude Code does render Markdown, this may be "Not applicable."

---

### 13. Documentation Gap -- No "Concepts" Guide

RAPID has extensive technical documentation (technical_documentation.md, 9 docs in docs/), but no conceptual guide that explains the "why" behind key decisions:
- Why `/clear` after every command?
- Why worktrees instead of branches?
- Why contracts instead of just file ownership?
- Why 6 parallel research agents instead of 1?

The README touches on these ("each set gets its own git worktree") but does not explain the reasoning. A new user reading the README cannot answer "why should I use this instead of just branching manually?"

**What gets forgotten:** This is the "curse of knowledge" blindspot -- the team that built RAPID understands the reasoning, but new users see the complexity without the context that motivates it.

**Retrofit cost:** Low (it is a documentation file), but getting the content right requires thoughtful writing. This could be a section in the README or a separate `docs/concepts.md`.

## Recommendations

1. **Audit-to-set handoff mechanism** (Concern #1): Design the persistence format first. A `.planning/REMEDIATION-QUEUE.md` or similar file that audit-version writes and add-set reads is the simplest approach. This is the highest-risk item if deferred -- without it, the audit -> add-set flow remains broken across /clear boundaries. The format should be simple enough that add-set can auto-detect and pre-populate from it, and status can display pending items (Concern #8).

2. **/clear guidance standardization** (Concern #2): Before touching 24 SKILL.md files, define the exact text and placement as a convention (documented in CONVENTIONS.md or the role modules). Consider adding a lightweight structural test (Concern #10) that validates all skills include the footer. This prevents future skills from omitting it.

3. **README rewrite** (Concerns #3, #4, #9, #13): The README should be rewritten with a clear problem statement, consistent install instructions, a brief "what RAPID does to your project" section, and the /clear mental model explained upfront. The install section should match DOCS.md exactly. This is the single highest-impact change for onboarding.

4. **Status + remediation visibility** (Concern #8): When implementing the handoff mechanism (#1), also update the status skill to display pending remediation items. This ensures the remediation queue is visible from the most-used dashboard command.

5. **Help command update** (Concern #6): Update the static help text to reflect the current command inventory. Remove the "not yet available" note for add-set. If the version bumps, update the version reference.

6. **Breadcrumb standardization** (Concern #5): Extend breadcrumbs to the review pipeline and merge skills. This is lower priority than the core UX features but improves workflow orientation.

7. **Error recovery guidance** (Concern #7): Defer to a future milestone unless specific error messages are identified as blocking onboarding. The skills already handle the most common errors with recovery steps.

8. **Version propagation** (Concern #11): If this milestone bumps the version, include version string updates in the same set that changes documentation. If not, defer.
