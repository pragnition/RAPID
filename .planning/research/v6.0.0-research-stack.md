# Stack Research -- v6.0.0 Scale & Quality

## Core Stack Assessment

### Node.js Runtime

| Property | Value |
|----------|-------|
| Detected minimum | 18+ (per CODEBASE.md) |
| Installed on dev machine | v25.8.0 |
| Current LTS | 24.14.1 (active support until Oct 2026) |
| Module system | CommonJS (.cjs throughout) |

**Node.js 18 reached end-of-life on April 30, 2025.** It receives no security patches. The project's documented minimum of "Node.js 18+" allows users to run on an unsupported runtime. Node 20 enters maintenance-only on April 30, 2026 (one month from now). Node 22 is in maintenance LTS (support ends April 2027). Node 24 is the current active LTS.

**Relevant to this milestone:** The `node:test` runner used for all RAPID tests has matured significantly across versions:
- Node 22: `--test-force-exit`, 8-10x performance improvement, filtered tests removed from output.
- Node 24: Test runner is no longer experimental (`--experimental` flag removed), built-in coverage, automatic subtest lifecycle management.

The project currently runs 1832 tests in ~17 seconds. No issues observed with the test runner on Node 25, but the minimum version should be raised to at least Node 20 to avoid shipping with an EOL runtime floor.

**Recommendation:** Bump minimum to Node 20 (maintenance LTS, supported through April 2026) or Node 22 (maintenance LTS, supported through April 2027). Since the codebase uses no Node 20+ specific APIs, this is documentation-only (prereqs.cjs version check).

### CommonJS Module System

The entire codebase uses `.cjs` extension with `require()` / `module.exports`. This is a deliberate choice for Claude Code plugin compatibility and maximum Node.js version compatibility.

**Relevant to this milestone:** The DAG-driven set grouping and scaffold overhaul features will add new modules (likely `src/lib/dag-groups.cjs` or extensions to `dag.cjs`). The CommonJS choice remains appropriate -- no ESM migration is warranted. However, one dependency (Zod) has CommonJS compatibility concerns at v4 (see below).

### Zod (^3.25.76 -> 4.3.6 available)

**Current usage (5 files):** `state-schemas.cjs`, `returns.cjs`, `hooks.cjs`, `review.cjs`, `merge.cjs`. All use `const { z } = require('zod')`.

**Critical patterns that would break with Zod 4:**

1. **`.passthrough()` -- used 5 times in `state-schemas.cjs`**. This method is deprecated in Zod 4. Replacement is `z.looseObject()` instead of `z.object({...}).passthrough()`. This is the most impactful breaking change for RAPID.

2. **`.optional()` + `.default()` ordering** -- Zod 4 changes behavior: `z.string().default("x").optional()` now applies the default inside optional. RAPID uses `StatusEnum.default('pending')` inside object schemas, not chained with `.optional()`, so this specific pattern is likely safe. Confirmed by inspection: no `.default().optional()` chains exist.

3. **`z.string().optional()`** -- Used in `returns.cjs`, `review.cjs`, `merge.cjs`. Still works in Zod 4 but the string format validators (`.email()`, `.uuid()`, etc.) moved to top-level. RAPID does not use string format validators, so no impact.

4. **`z.enum()`** -- Used for `SetStatus`, `WaveStatus`, `JobStatus`. Zod 4 changes: `z.nativeEnum()` deprecated (not used by RAPID), `.Enum` and `.Values` accessors removed. RAPID does not reference these accessors, so `z.enum()` usage is safe.

5. **CommonJS compatibility** -- Zod 4 has reported 4x bundle size increase for CJS consumers (issue #4637). For a CLI tool like RAPID this is inconvenient but not blocking (no Lambda cold-start concerns). More critically, Zod 3.25.x ships both v3 and v4 at subpaths, and there were CJS module resolution breakages reported in 3.25.0-beta (issue #4434). The currently pinned 3.25.76 appears stable.

**Assessment:** Staying on Zod 3.25.76 is safe for this milestone. The `^3.25.76` range will resolve to 3.25.x, which still exports Zod 3 APIs at the package root. A Zod 4 migration is not required and would add risk without benefit. If the `^` range inadvertently pulls in `4.x`, the `.passthrough()` calls will break. The `^3` range protects against this (npm semver treats 3.x -> 4.x as a major bump).

### Ajv (^8.17.1 -> 8.18.0)

**Installed version is already 8.18.0** (resolved by `^8.17.1`). This version includes:
- Tree-shaking support via `"sideEffects": false`
- Fix for Infinity/NaN serialization
- **CVE-2025-69873 security patch** (ReDoS mitigation)

The codebase uses `require('ajv').default` (CJS import for ESM-first package), which works correctly. No action needed -- already on latest.

### ajv-formats (^3.0.1 -> 3.0.1)

At latest. No updates available. No action needed.

### proper-lockfile (^4.1.2 -> 4.1.2)

**Last commit: January 25, 2021 (over 5 years ago).** 15 open issues on GitHub. No releases since 4.1.2. The package is functionally abandoned despite not being formally deprecated.

**Current usage:** File-based distributed locking in `src/lib/lock.cjs` for concurrent state access. This is a critical path -- if locking breaks, state corruption follows.

**Risk assessment:** The package works correctly today and uses the `mkdir` strategy which is OS-level atomic. Node.js 22+ has not broken `fs.mkdir` semantics, so there is no immediate runtime risk. However:
- No security patches will be issued if vulnerabilities are found
- No compatibility fixes for future Node.js versions
- If `fs.mkdir` behavior changes in a future Node.js release, this package will not adapt

**For this milestone:** No action required. The locking system is stable and the DAG/scaffold features do not change concurrency patterns. Flag for future consideration.

## Stack Risks

### 1. Zod semver range pulls in v4 unexpectedly
**Impact:** `.passthrough()` calls in `state-schemas.cjs` would throw, breaking all state operations.
**Likelihood:** Low -- `^3.25.76` is constrained to `>=3.25.76 <4.0.0` by npm semver rules.
**Mitigation:** `package-lock.json` pins the resolved version. Risk only materializes if lockfile is deleted and reinstalled after a hypothetical Zod 4.x published under `zod@3.x` (extremely unlikely).

### 2. proper-lockfile unmaintained
**Impact:** No patches for future Node.js incompatibilities or security issues.
**Likelihood:** Low in short term (mkdir atomicity is an OS guarantee, not a Node API).
**Mitigation:** The `lock.cjs` wrapper already abstracts proper-lockfile. A future replacement can be swapped without affecting callers.

### 3. Node.js 18 EOL minimum version
**Impact:** Users on Node 18 run without security patches. New users may encounter subtle incompatibilities.
**Likelihood:** Medium -- Node 18 is still commonly installed.
**Mitigation:** Update prereqs.cjs minimum version check and documentation.

### 4. No CI pipeline for 1832-test suite
**Impact:** Regressions introduced by DAG grouping, scaffold overhaul, or init changes may not be caught before merge.
**Likelihood:** Medium -- this milestone touches core systems (DAG, scaffold, init, state).
**Mitigation:** Run full test suite locally before merge. Consider adding a basic CI workflow as part of the milestone.

### 5. CLI backbone complexity growth
**Impact:** `rapid-tools.cjs` currently imports 20 command handlers. The DAG grouping, scaffold overhaul, and init changes will add or modify commands. No command routing framework exists -- it is a manual switch/case tree.
**Likelihood:** Low immediate risk, growing maintenance burden.
**Mitigation:** The existing command handler pattern (one file per command group in `src/commands/`) keeps individual files manageable. The main router just dispatches.

## Milestone-Specific Stack Implications

### DAG.json as Central Concept / DAG-Driven Set Grouping

The DAG module (`src/lib/dag.cjs`, 500 lines) currently supports:
- v1 DAG: `{nodes, edges, waves, metadata}` -- nodes have `{id, wave, status}`
- v2 DAG: adds `type` field (`set|wave|job`), same-type edge constraint, version field

**For group assignment:** The DAG node schema needs extension with a `group` property (e.g., `{id, type, wave, status, group: 1}`). The `createDAGv2()` function would need to accept group annotations. The `assignWaves()` function remains unchanged -- waves and groups are orthogonal concepts (waves = dependency ordering, groups = developer assignment).

**The `.passthrough()` in state-schemas.cjs is actually helpful here** -- it means the Zod schema already allows unknown properties on set/wave/job state objects. A `group` property added to DAG nodes would survive state validation without schema changes. This is a feature, not a bug.

### /scaffold Command Overhaul

Current scaffold system (`src/lib/scaffold.cjs`, 694 lines) generates language-specific template files (webapp, api, library, cli). It has no concept of "stubs for cross-group testing."

**Stack implication:** The stub system already exists in `src/lib/stub.cjs` (157 lines) -- it generates CommonJS stub modules from `CONTRACT.json` exports. The scaffold overhaul can build on this existing infrastructure rather than creating a new stub mechanism. The stub generator currently:
- Reads CONTRACT.json exports (functions and types)
- Generates `throw new Error('Stub: ...')` implementations
- Writes to `.rapid-stubs/{setName}-stub.cjs`

This architecture can be extended for cross-group stubs with minimal new dependencies.

### /init Spec File Integration and Meta Principles

No stack implications beyond file I/O. The init module (`src/lib/init.cjs`) already handles file generation and the scaffoldProject function can be extended to accept spec file paths. The `--desc` / `--description` flag mismatch (confirmed: CLI accepts `--desc` at line 32, SKILL.md instructs `--description` at line 481) should be fixed as part of this work.

### Bug Fixes (REQUIREMENTS.md overwrite, state overwrite)

Both bugs are logic errors in the init skill flow, not stack issues. The REQUIREMENTS.md overwrite happens because `scaffoldProject()` unconditionally generates REQUIREMENTS.md (line 240 of init.cjs). The fix is to either skip REQUIREMENTS.md in the `fileGenerators` map when mode allows it, or have the skill save and restore the file. No dependency or stack changes needed.
