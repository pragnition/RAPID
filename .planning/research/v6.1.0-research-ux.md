# Domain & UX Research -- v6.1.0 UX & Onboarding

## Domain Conventions

RAPID operates in the **AI agent harness / meta-prompting framework** domain. Competing products include GSD (Get Shit Done), OpenSpec, Claude Harness, and various Claude Code workflow plugins. Key domain conventions observed across these products:

### Standard Terminology

| RAPID Term | Domain Standard | Notes |
|------------|----------------|-------|
| Set | Phase (GSD), Spec/Change (OpenSpec) | "Set" is unique to RAPID; most competitors use "phase" |
| Wave | Wave (GSD), Task group | Consistent with GSD convention |
| `/clear` between commands | "Fresh context per phase" (GSD), "Context hygiene" (OpenSpec) | The principle is universal; terminology varies |
| Worktree | N/A | RAPID-unique; no competitor uses git worktrees for isolation |
| CONTRACT.json | Spec (OpenSpec), PLAN.md (GSD) | RAPID's approach is more machine-verifiable |

Source: [GSD README](https://github.com/gsd-build/get-shit-done), [OpenSpec README](https://github.com/Fission-AI/OpenSpec), [GSD Deep Dive (codecentric)](https://www.codecentric.de/en/knowledge-hub/blog/the-anatomy-of-claude-code-workflows-turning-slash-commands-into-an-ai-development-system)

### Common Workflows

All competing products follow a **Plan -> Execute -> Review** lifecycle with these shared conventions:
- Each phase produces file artifacts (not conversation memory) as the primary state persistence mechanism
- State survives `/clear` because it lives on disk, not in conversation context
- Next-step guidance is provided at the end of each command
- Fresh context windows are used per phase to prevent context rot

GSD explicitly recommends: "Use `/clear` between phases, treat it like a commit checkpoint -- you're signaling a phase transition." ([MindStudio GSD article](https://www.mindstudio.ai/blog/gsd-framework-claude-code-clean-context-phases))

OpenSpec documents: "Clear your context before starting implementation and maintain good context hygiene throughout your session." ([OpenSpec README](https://github.com/Fission-AI/OpenSpec))

### Industry Standard: Context Rotation

Anthropic's official best practices explicitly recommend:
- "Run `/clear` between unrelated tasks to reset context" ([Claude Code Best Practices](https://code.claude.com/docs/en/best-practices))
- "If you've corrected Claude more than twice on the same issue in one session, the context is cluttered with failed approaches. Run `/clear` and start fresh."
- The `/clear` between phases is the **industry standard** for Claude Code agent harnesses

**Critical gap identified:** RAPID's architecture is perfectly designed for context rotation (all state on disk, artifact-based recovery), but this design philosophy is invisible to users. Neither the README, the command outputs, nor the skill instructions explicitly tell users to `/clear` between commands.

## Interaction Models

### Primary Interaction Paradigm

RAPID is a **slash-command-driven CLI-in-CLI** tool. Users interact through Claude Code's slash command interface (`/rapid:*`), which orchestrates subagent spawning, file creation, and state management. This is consistent with the domain -- all competitors use the same paradigm.

### Input/Output Patterns Users Expect

Based on analysis of GSD, OpenSpec, Claude Harness, and Anthropic's own guidelines:

1. **Pre-command context:** Users expect to start each command from a fresh context with no accumulated conversation baggage. This is the fundamental value proposition of agent harnesses.

2. **Post-command guidance:** Users expect three things at the end of every command:
   - What was accomplished (summary)
   - What to do next (next action with exact command)
   - How to get there (context management guidance)

3. **Progress visibility:** Users expect to see where they are in the overall workflow. RAPID already provides this via progress breadcrumbs (e.g., `init [done] > start-set [done] > discuss-set > plan-set > ...`). This is a strong UX pattern that should be preserved and made consistent.

4. **Error recovery breadcrumbs:** RAPID already provides error breadcrumbs showing the failure point. This is a best practice that competitors have not implemented as thoroughly.

### Feedback and Progress Patterns

| Pattern | RAPID Status | Domain Standard |
|---------|-------------|-----------------|
| Progress breadcrumbs | Present in most skills | Not common in competitors (RAPID advantage) |
| Next-step suggestion | Present in most skills | Universal (GSD, OpenSpec both do this) |
| `/clear` reminder | **Missing** | GSD and OpenSpec both recommend this |
| Colored banners | Present (display.cjs) | GSD uses similar stage headers |
| Error breadcrumbs | Present with detail | Not common in competitors (RAPID advantage) |
| Numeric shorthand | Present (set references) | Not common (RAPID convenience feature) |

### Error Communication Patterns

RAPID uses `[RAPID ERROR]` prefix for stderr errors and `[RAPID]` for stdout messages. This is consistent with CLI conventions. The error breadcrumb pattern (showing what completed and what failed) is a domain-leading practice.

## Information Architecture

### How Information Is Organized

RAPID's information architecture follows a **progressive disclosure** model:
1. `/rapid:help` -- Static command reference (overview)
2. `DOCS.md` -- Detailed command reference with usage examples
3. `docs/*.md` -- Topic-specific deep-dives
4. `technical_documentation.md` -- Architectural narrative

This is well-structured. However, the **entry point** (README.md) has significant gaps:

### README Gaps vs. Domain Standards

Comparing RAPID's README to GSD and OpenSpec:

| Element | GSD | OpenSpec | RAPID | Assessment |
|---------|-----|---------|-------|------------|
| One-line value proposition | "A powerful meta-prompting...system" | "Spec-driven development for AI coding assistants" | "Coordinates parallel AI-assisted development" | RAPID's is good but buried |
| Problem statement | Implicit (context management) | "Agree on what to build before any code is written" | **Missing** | Users don't understand WHY to use RAPID |
| Install command | `npx get-shit-done-cc@latest` | `npm install -g @fission-ai/openspec` | Plugin marketplace + setup | Confusing; two paths not clearly differentiated |
| "Hello world" quickstart | Plan -> Execute -> Review (3 steps) | `/opsx:propose "idea"` (1 step) | 12-step quickstart | Too many steps for a quickstart |
| Context rotation guidance | Explicit in docs and workflow | "Clear your context before starting" | **Missing from README** | Major gap |
| Why this over alternatives | Implicit (sophisticated workflows) | Multi-tool support | "Not built for collaboration amongst teams" | Could be stronger |

### Navigation and Discovery

The current README directly links to DOCS.md, technical_documentation.md, and docs/. This is appropriate. However, there is no clear path from "I just installed RAPID" to "I understand what it does and why."

### Help and Documentation Conventions

The `/rapid:help` skill is well-designed as a static, no-analysis reference. This is good UX -- help should be instant. The help output includes a workflow diagram, command tables, and a typical workflow section. This is above average for the domain.

## Accessibility Considerations

### Key Patterns for the CLI Interaction Model

1. **NO_COLOR support:** RAPID already respects the `NO_COLOR` environment variable in `display.cjs` per the [no-color.org](https://no-color.org) standard. This is correct and follows best practices.

2. **Screen reader compatibility:** Terminal output is plain text by nature, which is inherently accessible. The ANSI color codes include a NO_COLOR fallback, which is correct.

3. **Color contrast:** RAPID uses dark background variants (`\x1b[4Xm`) with bright white text (`\x1b[97m`). This provides good contrast ratios. The three-color scheme (blue = planning, green = execution, red = review) provides semantic meaning through color, but the stage verbs (INITIALIZING, EXECUTING, etc.) also convey meaning textually. This is good accessible design.

4. **Keyboard navigation:** Not applicable -- RAPID is a terminal/CLI tool where all interaction is keyboard-driven by default.

**Assessment:** Accessibility is well-handled for a CLI tool. No significant gaps identified. Mark as "adequately addressed."

## User Expectations

### What Users Will Assume (Based on Domain Research)

1. **"/clear between commands" is implicit:** Users familiar with GSD or Claude Code best practices will assume they should `/clear` between RAPID commands. Users new to agent harnesses will NOT know this, leading to context rot and degraded performance. This is the single biggest onboarding gap.

2. **Install means "one command":** Users from the npm/plugin ecosystem expect installation to be a single command. RAPID's two-path installation (marketplace + `/rapid:install`) adds friction. GSD's `npx get-shit-done-cc@latest` is the benchmark.

3. **README tells me what problem this solves:** Users expect the README to answer "Why should I use this?" before "How do I use this?" RAPID's README starts with quotes about team collaboration, which is the right direction, but doesn't clearly state the problem: **context rot makes long Claude Code sessions unreliable; RAPID solves this by letting you `/clear` after every command because all state lives on disk.**

4. **Audit findings persist for action:** When `/rapid:audit-version` identifies gaps and suggests remediation sets, users expect that context to survive a `/clear`. Currently, the audit output recommends running `/rapid:add-set` with specific scope descriptions, but after `/clear`, the user has no record of what the audit found. The audit DOES write `.planning/v{version}-AUDIT.md` and `.planning/v{version}-DEFERRED.md`, but `/rapid:add-set` does not read these files. This is the handoff gap identified in the milestone brief.

5. **"What do I do next?" is always answered:** Every command should end with an unambiguous next action. RAPID already does this well with next-step suggestions and breadcrumbs, but the `/clear` guidance is the missing piece that connects the next-step suggestion to actual user behavior.

6. **Mental model from GSD:** Users coming from GSD expect a Plan -> Execute -> Review cycle. RAPID's 7-step lifecycle (init -> start-set -> discuss-set -> plan-set -> execute-set -> review -> merge) is more complex. The quickstart should acknowledge this complexity and provide a "minimal path" option.

### Terminology: User Language vs. Technical Language

| User Says | RAPID Calls It | Potential Confusion |
|-----------|---------------|---------------------|
| "phase" | "set" | Users from GSD will call sets "phases" |
| "task" | "wave" / "job" | Wave/job hierarchy is internal; users think in terms of tasks |
| "save my progress" | Artifact-based state | Users don't realize state persists through `/clear` |
| "what's next?" | `/rapid:status` | Users may not know this command exists |
| "start over" / "reset" | `/clear` + re-run | Users may think `/clear` loses progress |

## Recommendations

Ordered by priority, addressing the specific goals from the milestone brief:

1. **Add `/clear` guidance to every command's completion output:** Every skill should end its output with a line like: `> Run /clear before your next RAPID command to keep context fresh.` This is the most impactful single UX change. Competing products (GSD, OpenSpec) and Anthropic's own best practices all recommend this. Currently, zero RAPID skills include this guidance. -- **Priority: critical**

2. **Implement audit-to-set handoff mechanism:** The `/rapid:audit-version` skill already writes `.planning/v{version}-AUDIT.md` and `.planning/v{version}-DEFERRED.md`. The handoff should work by having `/rapid:add-set` auto-detect and read these files when invoked, presenting remediation items as pre-populated scope suggestions. This requires: (a) `audit-version` writing a machine-readable handoff artifact (a structured section in the audit report or a separate `AUDIT-HANDOFF.json`), and (b) `add-set` checking for this artifact and offering to pre-fill scope from it. No new commands needed -- just artifact-based handoff through the existing file system. -- **Priority: critical**

3. **Rewrite README.md for beginners:** The README should follow this structure: (a) One-line problem statement ("Context rot makes long AI coding sessions unreliable"), (b) One-line solution ("RAPID lets you `/clear` after every command -- all state lives on disk"), (c) Single install command with link to detailed setup, (d) 3-step minimal quickstart (init -> start-set -> execute-set), (e) "What just happened?" explanation, (f) Full lifecycle for when users want to dive deeper. GSD's README structure is the benchmark. -- **Priority: critical**

4. **Standardize post-command output format across all skills:** Define a consistent "command footer" that every skill renders at completion: (a) Summary of what was accomplished, (b) Next step with exact command, (c) Progress breadcrumb, (d) `/clear` reminder. Some skills already have most of these elements; others are missing pieces. A consistent footer template would ensure uniformity. -- **Priority: high**

5. **Add "Why RAPID?" section to README:** Users need to understand the value proposition before learning commands. Key points: context rot is the enemy, `/clear` between commands is the solution, RAPID's artifact-based architecture makes this possible, parallel development via worktrees is the bonus. -- **Priority: high**

6. **Improve install instructions clarity:** The README shows two install paths (plugin marketplace and git clone) without clearly explaining which to use when. Simplify to: "Install via plugin marketplace" as the primary path, with git clone as a secondary option for contributors. The `npx` or single-command install pattern (like GSD's `npx get-shit-done-cc@latest`) should be the aspiration. -- **Priority: high**

7. **General UX audit findings:** Several additional UX improvements identified during research:
   - The `/rapid:help` skill references "Phase 44 -- not yet available" for add-set, but add-set IS available. This is stale content. -- **Priority: medium**
   - The DOCS.md installation section shows `claude plugin add pragnition/RAPID` but README shows different install commands. These should be consistent. -- **Priority: medium**
   - Skills that end with AskUserQuestion (like `/rapid:status`) should include a "What does this do?" option for each suggested action, not just the action name. -- **Priority: low**
   - The progress breadcrumb format varies slightly between skills (some use `[done]`, some use `[FAILED: ...]`). Standardize the format. -- **Priority: low**
   - Consider adding a `/rapid:next` command (like GSD's `/gsd-next`) that auto-detects the next logical action from STATE.json. This would reduce cognitive load for new users who don't know which command to run next. -- **Priority: medium**

8. **Document the "context rotation" design philosophy:** RAPID's architecture is explicitly designed for context rotation (STATE-04 requirement: "Every command bootstraps exclusively from STATE.json + disk artifacts, self-contained after /clear"). This design philosophy should be surfaced prominently in documentation, README, and the install experience. Users should understand that `/clear` is not losing progress -- it is the intended workflow. -- **Priority: high**
