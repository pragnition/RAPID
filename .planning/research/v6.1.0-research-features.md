# Features Research

## Feature Inventory

### Feature: Audit-to-Set Handoff Mechanism

- **Description:** After `/rapid:audit-version` completes and suggests creating remediation sets, the context is lost when the user runs `/clear`. The add-set agent currently has no way to know what the audit discovered or what scope the remediation set should have. A persistent handoff mechanism is needed so that `/rapid:add-set` can consume audit findings without requiring them to be in the conversation context.
- **Complexity:** moderate
- **Dependencies:** independent

#### Decomposition

1. **Audit-side remediation artifact writer** -- When the audit identifies gaps that the user chooses to remediate (Step 4e of audit-version/SKILL.md), write a structured JSON file per remediation item to `.planning/pending-sets/` with full context (requirement, severity, remediation scope, suggested set name). Currently the audit only prints text commands to stdout; no persistent artifact captures the remediation intent.

2. **add-set pending-set auto-discovery** -- Modify the add-set skill to check `.planning/pending-sets/` at startup. If pending set proposals exist, present them to the user and pre-populate the discovery flow (Step 2 questions) with the audit's context, skipping interactive questions for that set.

3. **Pending-set cleanup after consumption** -- After add-set successfully creates a set from a pending proposal, remove (or archive) the consumed pending-set file so it is not re-offered.

4. **Status integration** -- When `/rapid:status` runs, show a notification if pending sets exist: "N remediation sets pending from audit. Run /rapid:add-set to create them."

#### Implementation Strategies

**Option 1: File-based pending-sets directory**
- Approach: Create `.planning/pending-sets/{set-name}.json` with structured remediation context. The audit writes these files; add-set reads and consumes them. Each file contains: `{ setName, requirement, severity, scope, remediation, sourceAudit, createdAt }`.
- Pros: Zero new dependencies; follows existing artifact-based patterns (DEFERRED.md, CONTRACT.json); survives `/clear`; human-readable; git-trackable.
- Cons: Requires coordination between two skills (audit-version writes, add-set reads) -- but this is standard RAPID file-ownership patterns.
- Libraries/services: None (plain Node.js fs operations).

**Option 2: Extend DEFERRED.md with machine-readable markers**
- Approach: Augment the existing `v{VERSION}-DEFERRED.md` with embedded JSON blocks for remediation items (similar to RAPID:RETURN markers). Add-set parses DEFERRED.md at startup.
- Pros: Reuses existing artifact; no new directory.
- Cons: DEFERRED.md is a human-readable Markdown file -- embedding machine-readable JSON in it conflates two concerns. Also, DEFERRED.md captures deferrals (items NOT to act on now), while remediation sets are items TO act on now. Semantically wrong container.

**Recommended:** Option 1 (file-based pending-sets directory). It cleanly separates "deferred" items (user chose to skip) from "remediation" items (user chose to create sets for). It follows existing artifact patterns and is naturally consumed by downstream skills.

#### Data Model Impact

- New entities: `.planning/pending-sets/{set-name}.json` -- one file per remediation proposal
- Existing entities modified: None (audit-version already writes artifacts; add-set already reads state)
- Relationships: audit-version produces pending-set proposals; add-set consumes and deletes them; status reads them for display

#### API Surface

- New CLI subcommand: `node rapid-tools.cjs pending-sets list` -- returns JSON array of pending set proposals (for status display)
- New CLI subcommand: `node rapid-tools.cjs pending-sets consume {set-name}` -- marks a pending set as consumed (deletes the file)
- Modified skill: `skills/audit-version/SKILL.md` Step 4e -- write pending-set files instead of (or in addition to) printing text commands
- Modified skill: `skills/add-set/SKILL.md` Step 2 -- check for pending-sets before interactive discovery
- Modified skill: `skills/status/SKILL.md` -- display pending-set count notification

### Feature: Beginner-Friendly README.md

- **Description:** The current README.md has install instructions that are confusing for beginners. It does not clearly explain RAPID's core value proposition (solving context rot via `/clear` between commands) or provide a clear beginner onboarding path. The install section has two alternative approaches presented ambiguously; the quickstart lists commands without explaining when/why to use `/clear`.
- **Complexity:** moderate
- **Dependencies:** depends on "Clear-Before-Next-Command Guidance" (the README should reference the `/clear` pattern)

#### Decomposition

1. **Rewrite install section** -- Simplify to a single primary install path (plugin marketplace) with the git clone as a secondary "advanced" option. Add prerequisite callouts (Node.js 20+, Claude Code). Remove the nested tip blocks that currently confuse the hierarchy.

2. **Add "Why RAPID?" section** -- Before the quickstart, add a concise explanation of the problem RAPID solves: context rot in long Claude Code sessions, and how RAPID's `/clear`-between-commands architecture prevents it by persisting all state to disk artifacts.

3. **Annotate quickstart with /clear guidance** -- After each command in the quickstart, show that users should `/clear` before the next command. Explain why (context rot, memory limits).

4. **Add "First Project" walkthrough** -- A brief narrative walkthrough (not just a command list) that walks through init -> start-set -> discuss-set -> plan-set -> execute-set -> review -> merge for a single set, explaining what each step produces and why.

5. **Improve command reference table** -- Add a "When to use" column or group commands by lifecycle phase so beginners can understand the flow.

#### Implementation Strategies

**Option 1: In-place README rewrite**
- Approach: Restructure the existing README.md with improved sections: Hero -> Why RAPID -> Install -> First Project Walkthrough -> Quickstart Reference -> Architecture -> Command Reference -> Documentation links -> Credits.
- Pros: Single file to maintain; standard open-source convention.
- Cons: README could get long if walkthrough is too detailed.
- Libraries/services: None.

**Option 2: README + separate GETTING-STARTED.md**
- Approach: Keep README concise (hero, install, quickstart) and add a `docs/getting-started.md` for the full walkthrough.
- Pros: Clean separation of concerns.
- Cons: Beginners might miss the getting-started doc; extra navigation step.

**Recommended:** Option 1 (in-place README rewrite). Beginners look at README first and may not navigate to docs. Keep it under 200 lines by being concise in the walkthrough section and linking to docs for deep dives.

#### Data Model Impact

- New entities: None
- Existing entities modified: `README.md` (full rewrite of content, same file)
- Relationships: References docs/ directory for detailed documentation

#### API Surface

- No API changes. This is a documentation-only feature.

### Feature: Clear-Before-Next-Command Guidance

- **Description:** Users need to be told to run `/clear` before the next RAPID command. Currently, no skill outputs this guidance consistently. The `/clear` pattern is fundamental to RAPID's design (preventing context rot from accumulated conversation history) but is nowhere documented in the user-facing output.
- **Complexity:** moderate
- **Dependencies:** independent

#### Decomposition

1. **Define standard footer block** -- Create a consistent "next step + clear" footer format that all skills use. Something like:
   ```
   Next step: /rapid:plan-set 1
   Tip: Run /clear before your next RAPID command to prevent context rot.
   ```

2. **Update all skill completion outputs** -- Modify every skill's completion banner/breadcrumb to include the `/clear` reminder. Skills that currently show "Next step: /rapid:X" (start-set, discuss-set, plan-set, execute-set, add-set, review, new-version, audit-version) all need this footer.

3. **Add /clear guidance to error outputs** -- When a skill fails, the error breadcrumb should also remind users to `/clear` before retrying, since the failed context is even more wasteful to carry.

4. **Optional: CLI-level footer utility** -- Add a `display footer` subcommand to rapid-tools.cjs that renders the standard footer, so skills can call `node rapid-tools.cjs display footer "{next-command}"` instead of each skill reimplementing the pattern.

#### Implementation Strategies

**Option 1: Skill-level text injection**
- Approach: Add the `/clear` reminder text directly into each SKILL.md file at every completion/error point. Each skill handles its own footer.
- Pros: No code changes; pure prompt-level change.
- Cons: Duplicated across 15+ skills; inconsistent if one skill is missed; harder to change the message later.
- Libraries/services: None.

**Option 2: CLI display footer utility + skill updates**
- Approach: Add `renderFooter(nextCommand)` to `display.cjs` and expose via `node rapid-tools.cjs display footer "{next-command}"`. Each skill calls this function at completion. The footer includes the `/clear` reminder, next-command suggestion, and progress breadcrumb in a unified format.
- Pros: Single source of truth for the footer format; easy to update message globally; consistent rendering across all skills.
- Cons: Requires a small code change to display.cjs plus updates to all skills to call the new function.
- Libraries/services: None.

**Recommended:** Option 2 (CLI display footer utility). The `/clear` message will inevitably be refined over time. Having a single source of truth prevents drift across 15+ skill files.

#### Data Model Impact

- New entities: None
- Existing entities modified: `src/lib/display.cjs` (add `renderFooter` function)
- Relationships: All skills call `display footer` at completion

#### API Surface

- New CLI subcommand: `node rapid-tools.cjs display footer "{next-command}"` -- renders standard footer with /clear guidance
- Modified skills: All 15+ skills that currently show completion banners

### Feature: General UX Audit

- **Description:** A systematic review of the full RAPID workflow to identify and address UX friction points beyond the specific issues already listed (install confusion, /clear guidance, audit handoff). This includes error messages, progress feedback, command discoverability, and workflow navigation.
- **Complexity:** complex
- **Dependencies:** depends on "Clear-Before-Next-Command Guidance" (the footer pattern should be established first, then the UX audit can validate it and find additional issues)

#### Decomposition

1. **Error message audit** -- Review all error outputs across skills and library modules. Ensure every error includes: what went wrong, why, and what to do next (actionable recovery command). Current errors sometimes lack recovery guidance.

2. **Progress feedback audit** -- Verify that long-running operations (research pipeline, execution, merge) provide intermediate progress indicators. The current system relies on agent output streaming, but when agents are silent for long periods, users do not know if progress is being made.

3. **Command discoverability improvements** -- The status dashboard (status/SKILL.md) is the main entry point for "what do I do next?" Evaluate whether it surfaces enough context. Consider adding a "Getting started?" prompt for first-time users.

4. **Breadcrumb consistency audit** -- Several skills have breadcrumbs (start-set, discuss-set, plan-set, execute-set, add-set) but formats vary. Standardize the breadcrumb format and ensure all skills use it.

5. **First-run experience** -- When a user runs any RAPID command for the first time without having run `/rapid:init`, the error messages vary. Standardize to always suggest `/rapid:init` with a brief explanation.

6. **Numeric shorthand documentation** -- The status skill mentions numeric shorthand (e.g., `/rapid:start-set 1` instead of `/rapid:start-set set-name`) but this is not documented elsewhere. Add consistent documentation of this pattern.

#### Implementation Strategies

**Option 1: Targeted fixes from predefined checklist**
- Approach: Create a checklist of known UX issues (error messages without recovery paths, missing breadcrumbs, inconsistent formats) and fix them systematically. Prioritize by user impact.
- Pros: Focused, measurable, predictable scope.
- Cons: May miss issues not on the checklist.
- Libraries/services: None.

**Option 2: Full workflow walkthrough + fix**
- Approach: Walk through the entire RAPID lifecycle (init -> ... -> merge -> audit -> new-version) from a beginner's perspective, documenting every friction point, then prioritize and fix.
- Pros: Comprehensive coverage.
- Cons: Very broad scope; risk of scope creep.

**Recommended:** Option 1 (targeted fixes from predefined checklist). The UX audit should be scoped to specific, identifiable improvements rather than an open-ended exploration. The checklist should be built during the discuss-set phase based on known friction points.

#### Data Model Impact

- New entities: None
- Existing entities modified: Multiple SKILL.md files (error messages, breadcrumbs, footers), `src/lib/display.cjs` (breadcrumb standardization), possibly `src/lib/core.cjs` (standardized error formatting)
- Relationships: Cross-cutting -- affects most skills and some library modules

#### API Surface

- Potentially new CLI subcommand: `node rapid-tools.cjs display breadcrumb "{current-stage}"` -- renders standardized breadcrumb
- Modified: Error output patterns across all skills
- Modified: Status skill for first-run detection

### Feature: Deferred Decision -- LLM-Based build-agents Validation

- **Description:** After `build-agents` generates agent definition files, spawn an executor agent to validate that all generated agents comply with namespace isolation rules (no references to non-RAPID skills/agents). Currently, namespace compliance is manually enforced.
- **Complexity:** moderate
- **Dependencies:** independent (from agent-namespace-enforcement set)

#### Decomposition

1. **Post-build validation hook** -- After `build-agents` completes, invoke a validation pass that checks each generated `.md` file in `agents/` for namespace violations.

2. **Validation rules engine** -- Define the set of patterns that constitute namespace violations: references to `gsd:*`, `openspec:*`, or other non-RAPID prefixed skills/agents in agent body text (outside of the identity/deny-list sections).

3. **Reporting** -- Output a pass/fail report with specific violations listed per agent file, including line numbers and the violating text.

#### Implementation Strategies

**Option 1: Static regex scan (no LLM)**
- Approach: After build-agents, run a regex-based scanner over all `agents/*.md` files looking for known non-RAPID skill patterns (e.g., `/gsd:`, `gsd-`, `/openspec:`). Report violations with file and line number.
- Pros: Fast, deterministic, no token cost, no LLM dependency.
- Cons: Cannot detect subtle semantic violations (e.g., agent referring to a concept by generic name that happens to be a GSD feature).
- Libraries/services: None (pure Node.js regex).

**Option 2: LLM-based validation via executor agent**
- Approach: Spawn a lightweight validation agent that reads each generated agent file and checks for namespace compliance using LLM understanding.
- Pros: Can detect semantic violations, not just lexical patterns.
- Cons: Expensive (token cost per build), slow, non-deterministic.
- Libraries/services: Claude Code Agent tool.

**Recommended:** Option 1 (static regex scan) as the primary validator, with Option 2 as an optional `--deep` flag for thorough validation when desired. The regex scan catches 95%+ of violations and is free/instant.

#### Data Model Impact

- New entities: None (validation report printed to stdout)
- Existing entities modified: `src/bin/rapid-tools.cjs` (add post-build validation step to `build-agents` command handler)
- Relationships: Runs after build-agents; blocks commit if violations found

#### API Surface

- New CLI flag: `node rapid-tools.cjs build-agents --validate` (run namespace validation after build)
- New CLI subcommand: `node rapid-tools.cjs validate-agents` (standalone validation)

### Feature: Deferred Decision -- execSync to execFileSync Migration

- **Description:** The `prereqs.cjs` module and other files still use `execSync` for shell command execution. `execFileSync` is safer (no shell injection) and was adopted in `worktree.cjs` and parts of `merge.cjs` during v6.0.0. A comprehensive codebase audit and migration is needed.
- **Complexity:** moderate
- **Dependencies:** independent (from bug-fixes-foundation set)

#### Decomposition

1. **Codebase audit** -- Grep all `.cjs` files for `execSync` usage. Categorize each call: (a) can be directly replaced with `execFileSync`, (b) requires shell features (pipes, redirects, `&&` chains) and needs refactoring, (c) is in test code and can remain.

2. **Production code migration** -- Replace all production-code `execSync` calls with `execFileSync` equivalents. For calls requiring shell features, refactor to use sequential `execFileSync` calls or Node.js built-in alternatives.

3. **Test code assessment** -- Decide whether test code should also be migrated. Test code using `execSync` with hardcoded strings is lower risk since the inputs are not user-controlled. Pragmatic approach: migrate where straightforward, leave where refactoring would obscure test intent.

#### Implementation Strategies

**Option 1: Targeted production-only migration**
- Approach: Migrate only production code in `src/lib/*.cjs` and `src/bin/rapid-tools.cjs`. Leave test files as-is since their inputs are controlled.
- Pros: Focused scope; tests remain readable; addresses the security concern where it matters.
- Cons: Inconsistency between production and test code patterns.
- Libraries/services: None.

**Option 2: Full codebase migration**
- Approach: Migrate all files including tests. For test files using `&&` chains, refactor to sequential calls.
- Pros: Full consistency; no `execSync` anywhere.
- Cons: Test code refactoring is noisy and obscures test intent.

**Recommended:** Option 1 (production-only migration). The security benefit of `execFileSync` applies to production code where inputs might be partially user-controlled. Test code with hardcoded commands has no injection risk.

#### Data Model Impact

- New entities: None
- Existing entities modified: `src/lib/merge.cjs` (remaining execSync calls), test files only if Option 2
- Relationships: None

#### API Surface

- No API changes. Internal implementation detail.

### Feature: Deferred Decision -- Auto-Regroup on DAG Structural Changes

- **Description:** When `/rapid:add-set` adds a new set with dependencies, the DAG structure changes but developer group assignments are not automatically recalculated. Currently, users must manually run `dag regroup` after adding a set. This should be automatic.
- **Complexity:** trivial
- **Dependencies:** independent (from dag-central-grouping set)

#### Decomposition

1. **Wire regroup into add-set state mutation** -- The `state add-set` CLI command already recalculates DAG.json. Add a call to `partitionIntoGroups()` (from dag.cjs) after the DAG recalculation in the same transaction.

2. **Test coverage** -- Add a test that verifies adding a set triggers group reassignment.

#### Implementation Strategies

**Option 1: Inline in state add-set handler**
- Approach: After `recalculateDAG()` in the `state add-set` handler (in rapid-tools.cjs), call `partitionIntoGroups()` to update group assignments. This is a 3-5 line change.
- Pros: Minimal code; atomic with the state transaction; users never need to think about regrouping.
- Cons: Tight coupling between add-set and DAG grouping.
- Libraries/services: Existing `dag.cjs` module.

**Recommended:** Option 1 (inline in state add-set handler). This is a trivial wiring change with clear benefits. The `partitionIntoGroups` function already exists and is idempotent.

#### Data Model Impact

- New entities: None
- Existing entities modified: DAG.json groups are updated automatically on add-set
- Relationships: `state add-set` -> `recalculateDAG()` -> `partitionIntoGroups()`

#### API Surface

- No new API. Existing `state add-set` gains implicit regroup behavior.

## Implementation Order

1. **Clear-Before-Next-Command Guidance**: Foundation for all other UX work. Creates the `renderFooter` utility in display.cjs that other skills will use. Must be done first so the README and UX audit can reference the established pattern.

2. **Audit-to-Set Handoff Mechanism**: Core feature request. Creates the `.planning/pending-sets/` directory pattern and modifies both audit-version and add-set skills. Independent of UX work, so can be done in parallel with item 1 if desired.

3. **Auto-Regroup on DAG Structural Changes**: Trivial 3-5 line change. Can be done in any order; putting it early reduces scope of the UX audit (one fewer known issue to track).

4. **Beginner-Friendly README.md**: Depends on the `/clear` guidance pattern being established (item 1). Best done after the footer utility exists so the README can accurately describe the workflow.

5. **General UX Audit**: Depends on items 1 and 3 being done (so the audit validates the new patterns rather than flagging known gaps). This is the broadest scope item and benefits from having other improvements in place first.

6. **execSync to execFileSync Migration**: Pure hygiene/security hardening. Independent of everything else. Can be done at any point.

7. **LLM-Based build-agents Validation**: Lowest priority deferred decision. Pure tooling improvement. Can be done at any point.
