### Feature 1: Solo Mode

- **Description:** Skip worktree/merge ceremonies for solo developers. All sets work directly on main branch, eliminating branch management overhead while preserving the full lifecycle pipeline (discuss, plan, execute, review).
- **Complexity:** complex
- **Dependencies:** independent (but touches many existing modules)

#### Decomposition

1. **Config schema extension** -- Add `soloMode: boolean` to STATE.json or a new `config.json` field. Extend Zod schemas in `state-schemas.cjs` to include a `soloMode` flag. Add init-time prompting.
2. **Init-time solo detection** -- Add an AskUserQuestion prompt during `/init` ("Will you be working solo or with a team?"). Persist the answer. Modify `skills/init/SKILL.md` Step 3-4.
3. **Per-set `--solo` override** -- Accept `--solo` flag on `/start-set`. When solo, bypass `createWorktree()` in `worktree.cjs` and instead register the set as operating on `main`. Modify `skills/start-set/SKILL.md` and `set-init create` handler.
4. **Worktree bypass in execute** -- In `execute.cjs` `prepareSetContext()`, when solo, generate context against cwd (main) rather than worktree path. Modify `generateScopedClaudeMd()` to work without a worktree.
5. **Merge no-op in solo mode** -- In `merge.cjs` and `skills/merge/SKILL.md`, detect solo mode and short-circuit with a friendly message ("Solo mode: commits already on main, no merge needed.").
6. **Status dashboard indicator** -- In `display.cjs` and `/status` skill, show "[SOLO]" badge when solo mode is active.
7. **Solo-to-team transition guard** -- Warn if user attempts to switch from solo to team mode mid-project. Provide guidance but do not block.

#### Implementation Strategies

**Option 1: Config-level flag with behavioral branching**
- Approach: Add `soloMode` to project config (either STATE.json or `.planning/config.json`). Each module checks the flag and branches behavior.
- Pros: Simple to implement, easy to understand, minimal refactoring.
- Cons: Scattered `if (soloMode)` checks across multiple modules.
- Libraries/services: None additional -- uses existing Zod schemas.

**Option 2: Strategy pattern with worktree abstraction**
- Approach: Create a `WorkMode` abstraction that has two implementations: `TeamWorkMode` (uses worktrees) and `SoloWorkMode` (operates on main). Each lifecycle command delegates to the active work mode.
- Pros: Clean separation, avoids scattered conditionals, easier to test.
- Cons: More upfront refactoring, risk of over-engineering for a boolean toggle.
- Libraries/services: None additional.

**Recommended:** Option 1. The behavioral differences are small enough (skip worktree creation, skip merge) that conditional checks are clearer than an abstraction layer. Solo mode is fundamentally a "skip these steps" feature, not a different execution model.

#### Data Model Impact

- New fields: `soloMode: boolean` in project-level config (add to `config.json` schema or STATE.json root)
- Modified entities: `REGISTRY.json` entries need to support `worktreePath: null` for solo sets
- Relationships: None changed

#### API Surface

- Modified commands: `init scaffold` (add solo mode prompt), `set-init create` (add `--solo` flag), `merge execute` (add no-op path)
- New: `state get-config` or extend `state get --all` to include solo mode flag
- Modified skills: `/init`, `/start-set`, `/execute-set`, `/merge`, `/status`

### Feature 2: Project Scaffold Command (`/scaffold`)

- **Description:** Generate a shared project foundation ("set-0") that all parallel sets build upon -- layout shell, routing skeleton, test harness, shared types. Not a replacement for framework CLIs, but the RAPID-specific glue layer.
- **Complexity:** very-complex
- **Dependencies:** independent (runs after `/init`, before `/start-set`)

#### Decomposition

1. **Scaffold skill definition** -- Create `skills/scaffold/SKILL.md` that orchestrates project type detection and scaffold generation. Define allowed tools and step sequence.
2. **Project type detection** -- Extend `context.cjs` `detectCodebase()` to classify projects as webapp, API, library, or CLI based on manifest files, framework indicators, and directory structure.
3. **Scaffold templates per project type** -- Define template generators for each project type:
   - **Webapp:** Layout component, routing skeleton, shared styles, test setup
   - **API:** Route structure, middleware chain, error handler, DB connection boilerplate
   - **Library:** Module structure, public API surface, test config
   - **CLI:** Command router, argument parser, help system
4. **Scaffold agent** -- Create a `rapid-scaffolder` agent role (`src/modules/roles/role-scaffolder.md`) that receives the project type and generates appropriate foundation code.
5. **Roadmap awareness** -- Modify the roadmapper agent to recognize when a scaffold exists and plan sets that build on top of it rather than duplicating foundation work. Add a `hasScaffold: true` flag to STATE.json.
6. **Set-0 registration** -- Register the scaffold as an implicit "set-0" in STATE.json that all real sets depend on. This integrates with the DAG in `dag.cjs`.

#### Implementation Strategies

**Option 1: Agent-driven scaffold generation**
- Approach: The `/scaffold` skill spawns a `rapid-scaffolder` agent with the project type and codebase analysis. The agent reads the existing codebase, determines what foundation is needed, and generates files. Works like the existing research agent pipeline.
- Pros: Flexible, context-aware, adapts to any project type.
- Cons: Non-deterministic output, may generate inconsistent scaffolds across runs.
- Libraries/services: None additional -- uses existing Agent tool.

**Option 2: Template-based scaffold with agent refinement**
- Approach: Define starter templates for each project type (stored as template strings or files in the plugin). The scaffold skill applies the template, then spawns an agent to customize it based on the specific project's needs.
- Pros: Deterministic base, consistent structure, agent adds customization.
- Cons: Templates need maintenance, may not cover all framework variants.
- Libraries/services: None additional.

**Recommended:** Option 1. Given RAPID is framework-agnostic and supports many project types, hardcoded templates would be brittle. The agent-driven approach leverages Claude's ability to understand the specific project and generate appropriate scaffolding. The `/init` already established a research-then-synthesize pattern that works well here.

#### Data Model Impact

- New entities: `hasScaffold: boolean` flag in STATE.json or config
- Modified entities: DAG dependencies -- sets should implicitly depend on scaffold completion
- Relationships: Scaffold is a prerequisite for all sets (set-0 in DAG)

#### API Surface

- New skill: `/scaffold` (`skills/scaffold/SKILL.md`)
- New agent: `rapid-scaffolder` (`agents/rapid-scaffolder.md`)
- New role module: `src/modules/roles/role-scaffolder.md`
- Modified: `/init` skill (suggest scaffold as next step), roadmapper agent (scaffold awareness)

### Feature 3: Review Pipeline Decomposition

- **Description:** Split the monolithic `/review` into four independent, composable skills: `/review` (scope), `/unit-test`, `/bug-hunt`, `/uat`. Each independently invocable, no stage selection prompt.
- **Complexity:** complex
- **Dependencies:** independent

#### Decomposition

1. **Extract `/review` into scope-only skill** -- Strip the current `/review` SKILL.md down to Steps 0-2.5 (resolve, validate, scope, concern-based scoping). Output: `REVIEW-SCOPE.md` with scoped files, concern groups, chunks, and wave attribution.
2. **Create `/unit-test` skill** -- New `skills/unit-test/SKILL.md` containing Steps 4a.1-4a.6 from current review. Reads `REVIEW-SCOPE.md` as input. Writes `REVIEW-UNIT.md`.
3. **Create `/bug-hunt` skill** -- New `skills/bug-hunt/SKILL.md` containing Steps 4b.1-4b.9 from current review. Reads `REVIEW-SCOPE.md` as input. Includes the full hunter/advocate/judge/bugfix cycle. Writes `REVIEW-BUGS.md`. Add judge confidence indicators.
4. **Create `/uat` skill** -- New `skills/uat/SKILL.md` containing Steps 4c.1-4c.7. Reads `REVIEW-SCOPE.md`. Writes `REVIEW-UAT.md`.
5. **REVIEW-SCOPE.md as shared contract** -- Define the schema for REVIEW-SCOPE.md (or a JSON equivalent like `REVIEW-SCOPE.json`) that all downstream skills consume. Include: scoped files, concern groups, chunks, wave attribution, working directory.
6. **Dependency detection** -- Each decomposed skill checks for REVIEW-SCOPE.md existence and prompts user to run `/review` first if missing.
7. **Summary generation consolidation** -- Modify `review summary` CLI command to work across the independently generated artifact files.

#### Implementation Strategies

**Option 1: Clean skill split with shared JSON scope artifact**
- Approach: `/review` outputs `REVIEW-SCOPE.json` (machine-readable). Each downstream skill reads it. Skills are fully independent -- no shared state beyond the scope file on disk.
- Pros: Clean boundaries, each skill is self-contained, easy to re-run individual stages.
- Cons: Scope file must contain all context (could be large), slight duplication in environment setup.
- Libraries/services: None additional. Uses existing review.cjs library functions.

**Option 2: CLI-based scope passing**
- Approach: `/review` generates scope, then each downstream skill is invoked with scope data piped via stdin or CLI args.
- Pros: No intermediate files.
- Cons: Tight coupling, harder to re-run, cannot inspect intermediate state.

**Recommended:** Option 1. Disk-based artifacts are the RAPID convention (CONTRACT.json, DEFINITION.md, CONTEXT.md). A `REVIEW-SCOPE.json` follows the same pattern and enables independent re-runs.

#### Data Model Impact

- New entities: `REVIEW-SCOPE.json` (schema for scoped files, concern groups, chunks, wave attribution)
- Existing entities modified: Review artifact paths unchanged (REVIEW-UNIT.md, REVIEW-BUGS.md, REVIEW-UAT.md)
- Relationships: `/unit-test`, `/bug-hunt`, `/uat` all depend on `/review` having run first

#### API Surface

- New skills: `/unit-test`, `/bug-hunt`, `/uat`
- Modified skill: `/review` (reduced to scope-only)
- Modified CLI commands: `review scope` (output `REVIEW-SCOPE.json` instead of just returning JSON to stdout)
- Existing CLI commands: `review log-issue`, `review list-issues`, `review update-issue`, `review summary` -- unchanged

### Feature 4: Context-Aware Internal Compaction

- **Description:** Keep working context below ~120k tokens during RAPID orchestration by compacting completed work intelligently, preserving RAPID-specific context structure.
- **Complexity:** very-complex
- **Dependencies:** independent

#### Decomposition

1. **Token estimation utility** -- Create a `tokenEstimate(text)` function that approximates token count (chars/4 heuristic for Claude, or use tiktoken-like library). Add to `core.cjs` or a new `tokens.cjs` module.
2. **Compaction trigger points** -- Identify where compaction should fire: between waves in execute-set, between review cycles, after agent returns are processed. Add hooks in the execute-set and review skill SKILL.md files.
3. **Safe-to-compact classifier** -- Build rules for what can be compacted vs what must be preserved:
   - Completed wave results: keep summary, discard full output
   - Resolved discussions: keep decisions, discard deliberation
   - Old agent returns: keep status + artifacts, discard full return body
   - Active task context, contracts, blockers: MUST preserve
4. **Compaction format** -- Define a structured summary format that agents can still reason about. Something like a compressed HANDOFF.md.
5. **Disk-as-truth recovery** -- Ensure all compactable data is first written to disk artifacts. If needed later, agents can re-read from disk.
6. **Context budget tracking** -- Add a running token budget tracker to orchestration skills that estimates current context usage and triggers compaction when approaching threshold.

#### Implementation Strategies

**Option 1: Proactive context minimization (preferred)**
- Approach: Rather than compacting after accumulation, reduce what is injected into context in the first place. Use leaner agent returns, shorter prompts, and always read from disk artifacts instead of carrying context forward in conversation.
- Pros: Simpler, no interaction with Claude Code's built-in compaction, lower risk of silent quality degradation.
- Cons: Requires auditing all skills and agent prompts, may miss some context that is convenient to carry.
- Libraries/services: None additional.

**Option 2: Active compaction layer**
- Approach: Build a compaction module that processes conversation context at defined trigger points, replacing detailed content with summaries while writing full content to disk.
- Pros: Handles the problem directly, can be tuned.
- Cons: Fights Claude Code's built-in context management, complex interaction effects, risk of discarding something needed later, hard to debug when compaction causes quality loss.
- Libraries/services: Possibly tiktoken (for accurate token counting), but chars/4 heuristic is sufficient.

**Option 3: Hybrid -- lean generation + disk-first architecture**
- Approach: Combine proactive minimization (Option 1) with a lightweight monitor that warns when context is growing large. No active compaction, just awareness and guidance.
- Pros: Low risk, observable, pushes agents toward disk artifacts naturally.
- Cons: Does not actively solve the problem, relies on agent compliance.

**Recommended:** Option 3 (Hybrid). Active compaction (Option 2) is high-risk and hard to debug. The better approach is to make the system inherently leaner: shorter prompts, disk-first artifact access, and a warning system that flags when context is growing. This avoids fighting Claude Code's built-in compaction while providing RAPID-aware context awareness.

#### Data Model Impact

- New entities: None for the hybrid approach
- Modified entities: Potentially add `contextBudgetTokens` to config.json
- Relationships: None

#### API Surface

- New utility: `tokenEstimate()` function in `core.cjs` or `tokens.cjs`
- Modified skills: Execute-set, review skills -- add context budget awareness annotations
- No new CLI commands needed for the hybrid approach

### Feature 5: Version Migration Command (`/migrate`)

- **Description:** Detect the current RAPID version from on-disk state, run versioned migration steps in order, handle schema migrations, artifact relocation, agent rebuilds, and config updates. Provide dry-run mode and backup before migration.
- **Complexity:** complex
- **Dependencies:** independent

#### Decomposition

1. **Version detection** -- Extend `version.cjs` with `detectInstalledVersion(cwd)` that infers version from STATE.json shape, artifact directory structure, or an explicit `rapidVersion` field. Use a version fingerprinting approach: check for presence of known version-specific fields/paths.
2. **Migration registry** -- Create `src/lib/migrate.cjs` with a registry of versioned migrations. Each migration is a function `(cwd) => void` with a `from` and `to` version. Registry: `[{ from: '2.0', to: '3.0', migrate: fn }, { from: '3.0', to: '3.1', migrate: fn }, ...]`.
3. **Schema migrations** -- Write migration functions for known schema changes (e.g., rename fields, add new required fields with defaults, remove deprecated fields). Use `withStateTransaction()` for atomic writes.
4. **Artifact relocation** -- Write migration functions for file/directory moves (e.g., `.planning/waves/` to `.planning/sets/`). Handle partial-move recovery.
5. **Agent rebuild integration** -- After schema migrations, trigger `build-agents` to regenerate agent prompts from updated modules.
6. **Config merge** -- Merge new default config values into existing `config.json` without overwriting user customizations (deep merge with user-wins semantics).
7. **Backup and dry-run** -- Before applying migrations, copy `.planning/` to `.planning/.pre-migrate-backup/`. Implement `--dry-run` flag that reports changes without writing.
8. **Migrate skill** -- Create `skills/migrate/SKILL.md` that orchestrates the migration pipeline with user confirmation at each major step.
9. **Version stamping** -- After successful migration, write the new version into STATE.json as `rapidVersion` field.
10. **Migration report** -- Generate a `MIGRATION-REPORT.md` summarizing all changes made.

#### Implementation Strategies

**Option 1: Sequential migration functions with version chain**
- Approach: Each version bump gets a migration function. `migrate()` detects current version, finds the chain of migrations needed (v2.0 -> v3.0 -> v3.1 -> ...), and runs them in order. Each migration is atomic (transaction-protected).
- Pros: Simple, predictable, easy to add new migrations, each migration is testable in isolation.
- Cons: Long migration chains for old projects, each intermediate state must be valid.
- Libraries/services: None additional -- uses existing Zod, fs, state-machine.cjs.

**Option 2: Declarative migration manifests**
- Approach: Define migrations as JSON manifests describing field renames, path moves, and transforms. A generic migration engine interprets the manifests.
- Pros: Declarative, easy to audit, no code for simple migrations.
- Cons: Complex transforms (like restructuring nested objects) don't fit a declarative model, needs a custom DSL.

**Recommended:** Option 1. The migration needs are varied (schema changes, file moves, agent rebuilds) and don't fit a purely declarative model. Sequential functions are simple, testable, and match the existing code style (imperative CommonJS with Zod validation).

#### Data Model Impact

- New fields: `rapidVersion: string` in STATE.json (added via ProjectState schema extension)
- Modified entities: STATE.json schema evolves per migration
- New files: `.planning/.pre-migrate-backup/` (backup), `MIGRATION-REPORT.md`
- Relationships: None

#### API Surface

- New skill: `/migrate` (`skills/migrate/SKILL.md`)
- New library module: `src/lib/migrate.cjs` (migration registry and runner)
- New CLI commands: `migrate detect-version`, `migrate run [--dry-run]`, `migrate status`
- Modified: `state-schemas.cjs` (add `rapidVersion` to ProjectState)

### Feature 15: Monolithic CLI Split (Design)

- **Description:** Split the ~2,700-line `src/bin/rapid-tools.cjs` into `src/commands/{command}.cjs` files with a thin router remaining.
- **Complexity:** complex
- **Dependencies:** Feature 11 (Error format standardization should ideally happen first or alongside)

#### Decomposition

1. **Design command module structure** -- Create `src/commands/` directory. Each top-level command group gets its own file: `state.cjs`, `execute.cjs`, `merge.cjs`, `worktree.cjs`, `review.cjs`, `plan.cjs`, `init.cjs`, `resolve.cjs`, `display.cjs`, `lock.cjs`, `set-init.cjs`, `misc.cjs` (for prereqs, parse-return, verify-artifacts, build-agents).
2. **Extract handler functions** -- Move each handler function from rapid-tools.cjs to its corresponding command file. Preserve the exact function signatures.
3. **Create thin router** -- Reduce rapid-tools.cjs to ~100 lines: argument parsing, command dispatch table, and `require()` calls to command modules.
4. **Update test imports** -- Modify `rapid-tools.test.cjs` to test command modules directly where possible.
5. **Verify CLI interface** -- Ensure all commands still work identically (same args, same output format).

#### Implementation Strategies

**Option 1: Extract-and-import (recommended)**
- Approach: Create `src/commands/{group}.cjs` files. Each exports an `async function handle(cwd, args)`. rapid-tools.cjs becomes a router that dispatches to the right handler.
- Pros: Minimal risk, incremental, each file is independently testable.
- Cons: Some shared utilities may need to be factored out (already in core.cjs).
- Libraries/services: None additional.

**Option 2: Commander.js or yargs**
- Approach: Replace manual routing with a CLI framework.
- Pros: Better help generation, built-in argument parsing.
- Cons: Adds a dependency, significant refactoring, overkill for an internal CLI.

**Recommended:** Option 1. The CLI is internal (called by skills, not users directly). A thin router is simpler and dependency-free.

#### Data Model Impact

- None

#### API Surface

- No external changes -- same CLI interface
- Internal: New `src/commands/` directory with ~12 module files

## Implementation Order

Recommended build order based on dependencies, risk, and value delivered:

1. **Critical fixes first (8, 9, 10):** Resume duplication, state mutation bypass, MERGE-STATE inconsistency. These address data corruption and race condition risks. Low effort, high impact.
2. **Trivial bug fixes (6, 7):** DEFINITION.md path fix and discuss-set UX fix. Quick wins that improve daily developer experience.
3. **Trivial structural fixes (12, 14, 18, 19):** Path resolution, agent build detection, deprecated cleanup, registry naming. Mechanical refactoring that reduces technical debt.
4. **Error standardization (11):** Prerequisite for the CLI split, establishes consistent patterns.
5. **CLI split (15) + Arg parsing (16):** The biggest structural refactoring. Reduces merge conflicts, enables parallel development on commands, improves testability. Do together.
6. **Stdin validation (17):** Apply after CLI split for cleaner per-command schemas.
7. **Review path mismatch (13):** Fix artifact paths before decomposing the review pipeline.
8. **Review decomposition (3):** High-value feature -- gives users control over review stages. Depends on path fix.
9. **Solo mode (1):** High-value feature for solo developers. Touch many modules but changes are behavioral (skip steps), not structural.
10. **Version migration (5):** Important for upgrade path. Build after other schema changes settle.
11. **Context compaction (4):** Hybrid approach (lean generation + monitoring). Lower priority since it is a quality-of-life improvement, not a functional gap.
12. **Project scaffold (2):** Most complex feature. Build last when the platform is stable.
