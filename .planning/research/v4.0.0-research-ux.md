# Domain & UX Research: Mission Control Dashboard (v4.0.0)

## Domain Conventions

### How Similar Products Work

The Mission Control dashboard sits at the intersection of **developer project management tools** (Linear, GitHub Projects), **developer dashboards** (Grafana, GitLens), **productivity launchers** (Raycast), and **knowledge management tools** (Notion, Obsidian). Each category contributes conventions that target users will expect.

**Developer Project Management (Linear, GitHub Projects):**
- Linear's UI redesign reduced visual noise while maintaining information density, using an "inverted L-shape" chrome (sidebar + top bar) controlling the main view. Linear explicitly cut back on color saturation, swapping monochromatic blue for monochromatic black/white with fewer bold accent colors. (Source: [Linear UI Redesign](https://linear.app/now/how-we-redesigned-the-linear-ui))
- GitHub Projects provides three layout modes: high-density table, kanban board, and timeline-style roadmap, letting users switch between views of the same data. (Source: [GitHub Projects Docs](https://docs.github.com/en/issues/planning-and-tracking-with-projects/learning-about-projects/about-projects))
- Both tools use keyboard-first interaction as a primary design goal, with Linear providing full J/K navigation and `G` prefix for "Go to" destinations.

**Developer Dashboards (Grafana):**
- Grafana uses a 24-column grid system as the dashboard layout foundation, with support for tabs, rows, and conditional visibility. (Source: [Grafana Dashboard Best Practices](https://grafana.com/docs/grafana/latest/visualizations/dashboards/build-dashboards/best-practices/))
- Navigation uses a megamenu (sidebar) with three states: opened, closed, and docked. When docked, it narrows the viewport for page content. Breadcrumbs are auto-generated from page hierarchy. (Source: [Grafana Navigation Patterns](https://grafana.com/developers/saga/patterns/navigation/))
- Dashboard content outline provides a tree-view navigation pane for jumping between sections.

**Productivity Launchers (Raycast):**
- Raycast's entire UX centers on the command palette pattern -- a single keyboard shortcut opens a search-first interface that provides access to all functionality. Extensions add commands to the same palette. (Source: [Raycast](https://www.raycast.com/))
- Raycast uses list views, detail views, and forms as its primary UI components, treating everything as a searchable, keyboard-navigable list.

**Knowledge Management (Notion, Obsidian):**
- Notion uses slash commands (`/`) to insert content blocks, inline markdown shortcuts for formatting, and a hierarchical page/database structure. (Source: [Notion Help Center](https://www.notion.com/help/keyboard-shortcuts))
- Obsidian popularized the local-first knowledge graph visualization, bidirectional linking, and split-pane markdown editing for developer-oriented note-taking.

### Standard Terminology

Users in this domain expect:
- **Project** (not "workspace" or "space") -- a codebase being managed
- **Set** -- RAPID-specific term; should be explained with tooltip on first encounter
- **Wave** -- RAPID-specific term; same treatment as Set
- **Milestone** -- universally understood project management term
- **Board/Kanban** -- universally understood view type
- **Graph** -- understood as dependency/relationship visualization
- **Status** -- pending, in-progress, complete, etc. (standard status vocabulary)

### Competitive Landscape Patterns

All tools in this space share these conventions:
1. Dark theme as default (or prominent option) for developer tools
2. Command palette (Cmd/Ctrl+K) as universal entry point
3. Keyboard shortcuts prominently documented
4. Information density prioritized over whitespace
5. Monospace font usage for code-related content
6. Status indicated by color-coded badges/dots, not just text
7. Sidebar navigation with collapsible sections

## Interaction Models

### Primary Interaction Paradigm

Mission Control is a **read-only dashboard web application** accessed via browser at `localhost:8998`. The primary interaction model is:

1. **Navigation** -- Moving between views (projects, kanban, graph, notes, codebase map)
2. **Inspection** -- Drilling into project details, set status, dependency graphs
3. **Note-taking** -- The only write operation: markdown notes and todo lists
4. **Search** -- Finding projects, sets, notes, and codebase elements

### Keyboard-First Design (Vim-Style)

Based on research into Linear, Vimium, and vim-style web apps, the recommended keyboard model is:

**Mode System:**
- **Normal mode** (default): Single-key navigation. J/K moves between items, H/L collapses/expands or moves between panels, Enter opens/selects.
- **Insert mode**: Activated when focusing text inputs (search bar, markdown editor). Esc returns to normal mode.
- **Command mode**: Activated with `:` or Cmd/Ctrl+K for the command palette.

**Navigation Shortcuts (following Linear's established patterns):**
| Key | Action | Source Convention |
|-----|--------|-------------------|
| `j` / `k` | Move down/up in lists | Vim, Linear |
| `h` / `l` | Navigate left/right between panels/columns | Vim |
| `Enter` | Open/select focused item | Universal |
| `Esc` | Go back / exit mode / close panel | Universal |
| `/` | Open search | Linear, Vim |
| `:` | Open command palette | Vim |
| `Ctrl+K` / `Cmd+K` | Open command palette (alternative) | Linear, Raycast, VS Code |
| `g` then `p` | Go to Projects | Linear (`G, P`) |
| `g` then `k` | Go to Kanban | Custom (Linear-style) |
| `g` then `g` | Go to Graph | Custom (Linear-style) |
| `g` then `n` | Go to Notes | Custom (Linear-style) |
| `g` then `m` | Go to Codebase Map | Custom (Linear-style) |
| `g` then `w` | Go to Worktrees | Custom (Linear-style) |
| `?` | Show keyboard shortcut overlay | Vim, GitHub |
| `Tab` | Move focus between major sections | W3C keyboard convention |

(Source: [Linear Shortcuts](https://keycombiner.com/collections/linear/), [W3C Keyboard Interface](https://www.w3.org/WAI/ARIA/apg/practices/keyboard-interface/))

**Mode Indicator:**
Vimium and vim-based web apps display a mode indicator in a fixed corner position (typically bottom-right or bottom-left). For Mission Control:
- Display a small, unobtrusive mode badge: "NORMAL", "INSERT", "COMMAND"
- Use the Everforest green accent (`#a7c080`) for the active mode indicator
- Fade it to low opacity when not recently changed; briefly highlight on mode switch

(Source: [Vimium](https://github.com/philc/vimium), [Vim Mode Feedback](https://vim.fandom.com/wiki/More_visible_mode_feedback))

### Mouse-Friendly Alternative

Every keyboard action must have a mouse equivalent:
- Click to select/open items
- Hover states on all interactive elements
- Context menus (right-click) for secondary actions
- Drag-and-drop for kanban cards (with keyboard alternative)
- Scrollable panels with visible scrollbars in dark theme

### Input/Output Patterns

- **Input**: Navigation, search queries, markdown text, kanban card movement
- **Output**: Rendered dashboards, graphs, formatted markdown, status visualizations
- **Feedback**: Subtle animations on state changes (opacity transitions, not flashy effects), toast notifications for async operations (project registration), loading skeletons for data fetch

### Error Communication

- Use inline error messages near the source of the problem (e.g., "Cannot connect to project at /path/to/project")
- Toast notifications for transient errors (network issues, service unavailable)
- Empty states with clear messaging: "No projects registered. Run `/rapid:init` in a project to register it."
- Never show raw stack traces in the UI; log them to console

## Information Architecture

### Recommended View Structure

Based on research into Grafana's sidebar + content model and Linear's "inverted L-shape":

```
+------------------+------------------------------------------------+
|                  |                                                  |
|    Sidebar       |              Main Content Area                   |
|    (fixed)       |                                                  |
|                  |  +------------------------------------------+   |
|  [Logo/Brand]    |  |  View Header (breadcrumb + actions)      |   |
|                  |  +------------------------------------------+   |
|  [Search]        |  |                                          |   |
|                  |  |  View Content                            |   |
|  Projects        |  |  (kanban, graph, editor, etc.)           |   |
|   > Project A    |  |                                          |   |
|   > Project B    |  |                                          |   |
|                  |  |                                          |   |
|  Views           |  |                                          |   |
|   Dashboard      |  |                                          |   |
|   Kanban         |  |                                          |   |
|   Graph          |  |                                          |   |
|   Notes          |  |                                          |   |
|   Codebase       |  |                                          |   |
|   Worktrees      |  |                                          |   |
|                  |  +------------------------------------------+   |
| [Mode Indicator] |  [Status Bar]                                   |
+------------------+------------------------------------------------+
```

**Sidebar** (collapsible, Grafana-style three states: open, closed/icon-only, docked):
- Project selector with status dots (green = active, grey = idle, red = error)
- View navigation with icons + labels
- Collapsible to icon-only mode on smaller screens or via keyboard shortcut (`Ctrl+B` following Linear)
- Fixed position, does not scroll with content

**Main Content Area:**
- View header with breadcrumb trail (auto-generated from navigation path)
- View-specific content (kanban columns, graph canvas, editor panes)
- Status bar at bottom (connection status, last sync time, mode indicator)

### Navigation and Discovery

- **Breadcrumbs**: Show navigation path (e.g., "Projects > my-app > Sets > auth-system")
- **Command palette**: Global search across projects, sets, notes, and navigation destinations
- **Recent items**: Command palette shows recently visited items on open (Source: [Command Palette Design](https://destiner.io/blog/post/designing-a-command-palette/))

### Content Hierarchy and Progressive Disclosure

Follow a three-level information hierarchy:
1. **Overview level**: Project list with status summaries, aggregate progress
2. **Project level**: Dashboard with set status, kanban, graph access
3. **Detail level**: Individual set details, note content, codebase file inspection

Each level reveals more detail without requiring the user to leave context -- use expandable sections, slide-out panels, and inline detail views rather than full page navigations where possible.

### Help and Documentation

- `?` key opens a keyboard shortcut cheatsheet overlay (following GitHub convention)
- Tooltips on all icons showing both the action name and keyboard shortcut
- First-time user onboarding: a dismissable banner explaining vim-style navigation
- Link to full RAPID documentation in sidebar footer

## Accessibility Considerations

### Keyboard Navigation (Critical for Vim-Mode App)

- All interactive elements must be reachable via Tab key in logical order (Source: [W3C Keyboard Interface](https://www.w3.org/WAI/ARIA/apg/practices/keyboard-interface/))
- Visible focus indicators: 2-3px outline using Everforest aqua (`#83c092`) on dark backgrounds for high contrast
- Focus must never become trapped -- Esc should always provide an escape route
- Vim keybindings must not conflict with screen reader shortcuts; provide a setting to disable vim mode entirely for assistive technology users
- Kanban drag-and-drop must have keyboard alternative: select card with Enter, arrow keys to move, Enter to drop (Source: [Kanban Accessibility](https://uxpatterns.dev/patterns/data-display/kanban-board))

### Screen Reader Support

- Semantic HTML throughout: `<nav>`, `<main>`, `<aside>`, `<article>`, `<section>` with proper headings hierarchy
- ARIA live regions for dynamic content updates (status changes, toast notifications)
- `aria-label` on icon-only buttons
- Kanban cards: `aria-grabbed`, `aria-dropeffect`, and live region announcements for card moves (Source: [Drag-and-Drop Accessibility](https://appinstitute.com/drag-and-drop-design-accessibility-best-practices/))

### Color and Contrast

- WCAG AA minimum: 4.5:1 contrast ratio for normal text, 3:1 for large text
- Everforest palette analysis:
  - `#d3c6aa` (fg) on `#2d353b` (bg0): contrast ratio ~8.2:1 -- passes AAA
  - `#a7c080` (green) on `#2d353b` (bg0): contrast ratio ~5.7:1 -- passes AA
  - `#83c092` (aqua) on `#2d353b` (bg0): contrast ratio ~5.8:1 -- passes AA
  - `#7a8478` (grey0) on `#2d353b` (bg0): contrast ratio ~3.1:1 -- borderline, use only for decorative/non-essential text
- Never rely solely on color to convey status; pair with icons or text labels (e.g., green dot + "Active" label)

### Dark Theme Accessibility

- Avoid pure black (#000000) backgrounds; Everforest's `#2d353b` is appropriately dark grey (Source: [Dark Mode Best Practices](https://atmos.style/blog/dark-mode-ui-best-practices))
- Avoid pure white (#ffffff) text; Everforest's `#d3c6aa` (warm cream) provides a softer reading experience
- Offer a light theme toggle for users with astigmatism, as dark UIs can cause halation (text blur from iris dilation)
- Desaturate accent colors by ~20 points relative to light theme equivalents (Everforest already does this by design)

### Inclusive Design

- Support prefers-reduced-motion: disable animations for users who request it
- Support prefers-color-scheme: auto-detect system theme preference
- Ensure all views function at 200% zoom without horizontal scrolling
- Minimum touch target size of 44x44px for mouse-click targets

## User Expectations

### What Users Will Assume

Based on domain research, users of a developer dashboard tool will expect:

1. **Instant startup**: The dashboard should load in under 2 seconds. Developers abandon slow tools. Use skeleton loading states rather than spinners.

2. **Command palette**: Pressing Cmd/Ctrl+K should open a search palette. This is the de facto standard established by VS Code, Linear, Raycast, Notion, Slack, and GitHub. Absence of this feature will feel like a gap. (Source: [Command Palette Design](https://destiner.io/blog/post/designing-a-command-palette/))

3. **Dark theme by default**: Developer tools overwhelmingly default to dark themes. The Everforest dark theme should be the default, with light as an opt-in toggle.

4. **Keyboard-driven operation**: Developer audiences expect most actions to be performable without touching the mouse. However, they also expect the mouse path to work seamlessly -- keyboard-first does not mean keyboard-only.

5. **Information density**: Developers prefer dense, data-rich interfaces over spacious, marketing-style layouts. Show more information at a glance; use progressive disclosure for details rather than hiding content behind clicks.

6. **Real-time updates**: If a project's state changes (e.g., a set moves from "executing" to "complete"), the dashboard should reflect this promptly. Polling at 5-10 second intervals is acceptable; websockets are preferred but not essential for a read-only tool.

7. **Monospace font for code content**: File paths, branch names, set IDs, and technical identifiers should use a monospace font. UI labels and prose should use a proportional font.

8. **No account/login wall**: A local development tool should not require authentication. It should work immediately when accessed at `localhost:8998`.

### Terminology Expectations

- Developers will understand "worktree", "branch", "merge", "DAG" without explanation
- RAPID-specific terms ("set", "wave", "job") need contextual explanation (tooltips, first-use callouts)
- Use "Status" not "State" in user-facing UI (despite internal naming conventions)
- Use "Progress" not "Completion" for percentage indicators
- Use "Notes" not "Documents" for the markdown editor section

### Mental Models from Competing Products

- **From Linear**: Users expect `G` prefix for "Go to" navigation, `C` for create, `S` for status changes
- **From VS Code**: Users expect Cmd/Ctrl+P for file search, Cmd/Ctrl+Shift+P for commands
- **From GitHub**: Users expect `/` to focus search, `?` for shortcut help
- **From Notion**: Users expect `/` for slash commands in editor contexts
- **From Grafana**: Users expect panel-based dashboard layouts with sidebar navigation

### Onboarding and Learning Curve

- **Zero-friction start**: Dashboard should show useful information immediately after first project registration
- **Empty states**: Every view needs a meaningful empty state with instructions (e.g., "No notes yet. Press `n` to create your first note.")
- **Shortcut discovery**: Use "K-bar" style tooltip hints -- when hovering over a button, show the keyboard shortcut in the tooltip (e.g., "Search (/)"). Do not show all shortcuts at once unless explicitly requested with `?`.
- **No tutorial modals**: Developer audiences dislike forced tutorials. Use contextual hints and an optional help overlay instead.

## Knowledge Graph Visualization

### Library Comparison for RAPID's Use Case

RAPID's knowledge graph will display project sets, their dependencies (DAG), and relationships between code modules. Expected scale: 5-50 nodes (sets) with 0-100 edges (dependencies). This is a small graph by visualization standards.

| Library | Max Nodes | Rendering | Learning Curve | Best For | License |
|---------|-----------|-----------|----------------|----------|---------|
| **Cytoscape.js** | 100,000+ | Canvas | Medium | Network graphs, dependency DAGs | MIT |
| **D3.js** | ~5,000 | SVG/Canvas | High | Custom, highly branded visuals | BSD |
| **vis.js** | ~3,000 | Canvas | Low | Quick setup, simple graphs | MIT/Apache |

**Recommendation: Cytoscape.js** for the following reasons:
1. Built-in graph algorithms (shortest path, BFS/DFS, centrality) -- useful for DAG analysis
2. Built-in layout algorithms including `dagre` for directed acyclic graphs -- perfect for RAPID's set dependency visualization
3. Excellent interactivity: pan, zoom, click, hover events out of the box
4. Headless mode for server-side rendering if needed
5. Mature ecosystem with extension plugins

(Source: [Top 10 JS Graph Libraries](https://www.getfocal.co/post/top-10-javascript-libraries-for-knowledge-graph-visualization), [Cytoscape.js](https://js.cytoscape.org/))

**For the codebase map visualization**, consider D3.js circle packing layout (following GitHub's approach) or a treemap. GitHub's repo visualization uses React + D3 with circle packing where color = file type and size = file size. This approach provides a "fingerprint" view that enables glance comprehension. (Source: [GitHub Next Repo Visualization](https://githubnext.com/projects/repo-visualization/))

### Interaction Patterns for Graph Views

- **Hover**: Highlight node and its direct connections; dim unrelated nodes (GitHub's "show connections on hover" pattern)
- **Click**: Select node to show details in side panel
- **Zoom**: Scroll wheel to zoom; pinch on trackpad; fit-all button (`0` key)
- **Pan**: Click-and-drag background; arrow keys when graph is focused
- **Layout toggle**: Switch between DAG layout (hierarchical), force-directed, and circular
- **Filter**: Show/hide nodes by status (pending, executing, complete, merged)

## Kanban Board UX

### Card Design

Based on research into Linear and GitHub Projects kanban patterns:

```
+---------------------------------------+
| [Status Dot] Set Name          [Icon] |
| Wave 2 of 3  |  3/5 tasks complete    |
| [=========>          ] 60%            |
+---------------------------------------+
```

- **Primary info**: Set name (bold), status dot (color-coded)
- **Secondary info**: Wave progress, task count
- **Tertiary info**: Progress bar
- **Card width**: Consistent within column; no truncation of set names
- **Card actions**: Click to open details; keyboard shortcut `Enter` on focused card

### Column Design

Columns represent set statuses from the RAPID lifecycle:
- **Pending** | **Discussing** | **Planning** | **Executing** | **Complete** | **Merged**

Each column shows:
- Column header with count badge
- WIP limit indicator (optional, visual only since read-only)
- Scrollable card list

### Keyboard Accessibility for Kanban

- Tab to enter the board, J/K to move between cards in a column
- H/L to move focus between columns
- Since Mission Control is read-only (no editing features yet), drag-and-drop is not needed for the initial version
- If card reordering is added later: Enter to grab, arrow keys to move, Enter to drop; announce moves via ARIA live regions

(Source: [Kanban Board UX Pattern](https://uxpatterns.dev/patterns/data-display/kanban-board), [dnd-kit Accessibility](https://plaintext-engineering.com/blog/drag-n-drop-kanban-board-react/))

## Markdown Editor Patterns

### Recommended Approach: Split-Pane with Toggle

For the note editor, use a split-pane layout with three modes:
1. **Source mode**: Raw markdown on left, rendered preview on right (default)
2. **Preview mode**: Full-width rendered view (toggle with `Ctrl+Shift+P`)
3. **Edit mode**: Full-width source editor (toggle with `Ctrl+Shift+E`)

Key features:
- **Synchronized scrolling**: Bind scroll positions between editor and preview panes (Source: [StackEdit](https://stackedit.io/))
- **Vim keybindings in editor**: Since the app is vim-themed, the markdown editor should support vim motions (via CodeMirror vim extension or Monaco vim mode)
- **Slash commands**: Support Notion-style `/` commands for inserting headers, lists, code blocks, and links
- **Toolbar**: Minimal floating toolbar above editor for common formatting (bold, italic, heading, list, code, link) with keyboard shortcut hints
- **Auto-save**: Save on every pause in typing (debounced 1-2 seconds), with "Saved" indicator

### Markdown Formatting Shortcuts

Follow Notion's established patterns:
| Shortcut | Action |
|----------|--------|
| `Ctrl+B` | Bold |
| `Ctrl+I` | Italic |
| `Ctrl+E` | Inline code |
| `Ctrl+K` | Insert link (context-dependent: overrides command palette when editor is focused) |
| `Ctrl+Shift+1/2/3` | Heading 1/2/3 |
| `Ctrl+Shift+7` | Ordered list |
| `Ctrl+Shift+8` | Unordered list |
| `Ctrl+Shift+9` | Checklist |

(Source: [Notion Keyboard Shortcuts](https://www.notion.com/help/keyboard-shortcuts))

### Elevation and Depth

In dark themes, communicate elevation through lighter surfaces rather than shadows (Source: [Dark Mode Best Practices](https://atmos.style/blog/dark-mode-ui-best-practices)):
- Higher elevation = lighter background (bg_dim < bg0 < bg1 < bg2)
- Sidebar: `bg_dim` (#232A2E)
- Main content: `bg0` (#2D353B)
- Cards/panels: `bg1` (#343F44)
- Hover state: `bg2` (#3D484D)
- Use thin borders (`bg3` / `#475258`) sparingly to separate sections; prefer spacing and elevation differences

- **Headings**: Inter Display or system sans-serif, `#D3C6AA`

- **Code/technical**: JetBrains Mono, Fira Code, or system monospace, `#D3C6AA`
- **Accent text**: `#A7C080` (green) for active/important, `#83C092` (aqua) for links

## Tooltip and Shortcut Discovery

### Progressive Disclosure Strategy

Based on research into progressive disclosure patterns (Source: [IxDF Progressive Disclosure](https://ixdf.org/literature/topics/progressive-disclosure)):

**Layer 1 -- Passive Discovery (always present):**
- Tooltips on all icon buttons showing action name + shortcut (e.g., "Search (/)")
- Mode indicator in status bar showing current vim mode
- Cursor changes on hoverable/clickable elements

**Layer 2 -- On-Demand Discovery (user-initiated):**
- `?` key opens a full keyboard shortcut overlay/cheatsheet (following GitHub's convention)
- Command palette (`Ctrl+K` / `:`) shows shortcut next to each command
- Help icon in sidebar footer links to documentation

**Layer 3 -- First-Time Discovery (dismissable):**
- On first visit, show a subtle bottom banner: "This dashboard supports Vim-style navigation. Press ? for shortcuts."
- Store dismissal in localStorage; never show again after dismissed
- Empty states include contextual shortcut hints ("Press `n` to create a note")

**Layer 4 -- Contextual Discovery (situation-dependent):**
- When a user performs an action with the mouse that has a keyboard shortcut, briefly flash the shortcut in a toast: "Tip: Press `g` then `k` to jump to Kanban"
- Limit to one tip per session to avoid annoyance; track shown tips in localStorage
- Disable tips entirely via a "Don't show tips" setting

### Implementation Notes

- Shortcut hints in tooltips should use styled keyboard key indicators (`<kbd>` elements) with Everforest `bg2` background
- Command palette should support fuzzy search (Source: [Command Palette Design](https://destiner.io/blog/post/designing-a-command-palette/))
- Command palette should show recent items on empty query, not an alphabetical list
- Shortcut overlay should be organized by category (Navigation, Views, Editor, General)

## Dashboard Layout Patterns

### Resizable Panels

For views that benefit from flexible layout (e.g., graph + details, editor + preview):
- Use a split-pane layout with draggable divider
- Persist panel sizes in localStorage
- Provide keyboard shortcuts to toggle panel visibility
- Double-click divider to reset to default proportions
- The allotment library (based on VS Code's split view) provides an industry-standard look and feel

(Source: [react-resplit](https://github.com/kenanyusuf/react-resplit), [allotment](https://github.com/johnwalley/allotment))

### Responsive Behavior

Since this runs on localhost and is primarily a desktop tool:
- **Primary target**: 1280px+ desktop screens
- **Minimum supported**: 1024px (sidebar collapses to icon-only)
- **No mobile layout needed** (localhost tool, not accessed from phones)
- Sidebar should auto-collapse below 1280px, accessible via hamburger icon or `Ctrl+B`
- Graph and kanban views should scale their content to the available viewport

### Codebase Map Layout

Following GitHub's repo visualization approach:
- Use D3.js circle packing for file/directory visualization
- Color = file type (language), Size = file size
- Nesting = directory hierarchy
- Hover to show file path and metadata
- Click to show file details in side panel
- Provide a "bird's eye view" that serves as a project fingerprint

(Source: [GitHub Next Repo Visualization](https://githubnext.com/projects/repo-visualization/))

---

## Recommendations

Ordered by priority for the Mission Control roadmap:

1. **Implement command palette (Ctrl+K and : triggers)**: This is the single most expected interaction pattern in modern developer tools. Show recent items, fuzzy search, and keyboard shortcut hints alongside commands. -- **Priority: critical**

2. **Build with Everforest dark theme as default**: Use the complete Everforest palette with surface elevation model (lighter = higher). Provide light theme toggle but make dark the default experience. -- **Priority: critical**

3. **Implement vim-style J/K/H/L navigation with mode indicator**: Follow Linear's proven model of vim-style single-key navigation in normal mode with a visible mode indicator. Include `G` prefix navigation for view switching. -- **Priority: critical**

4. **Use collapsible sidebar navigation (three states)**: Follow Grafana's pattern of open/icon-only/docked sidebar states. Sidebar should contain project selector and view navigation. -- **Priority: critical**

5. **Add keyboard shortcut overlay on `?` key**: GitHub's convention of `?` for shortcut help is widely expected. Organize shortcuts by category with `<kbd>` styling. -- **Priority: high**

6. **Choose Cytoscape.js for dependency graph visualization**: Best fit for DAG visualization with built-in dagre layout, good interactivity, and appropriate scale for RAPID's use case (5-50 nodes). -- **Priority: high**

7. **Use D3.js circle packing for codebase map**: Follow GitHub's established pattern of color-by-type, size-by-file-size circle packing for an intuitive codebase fingerprint view. -- **Priority: high**

8. **Implement split-pane markdown editor with vim keybindings**: Use CodeMirror 6 with vim extension for the editor component. Provide source/preview split with synchronized scrolling. -- **Priority: high**

9. **Design kanban board as read-only status view**: Since Mission Control is read-only, focus on clear status visualization with column-per-status design rather than drag-and-drop interaction. -- **Priority: high**

10. **Add progressive shortcut discovery system**: Four-layer approach from passive tooltips to contextual tips. Never force tutorials; respect developer autonomy. -- **Priority: medium**

11. **Implement multi-segment progress bars for project status**: Glanceable progress visualization using Everforest semantic colors. Show set status distribution at a glance. -- **Priority: medium**

12. **Support prefers-reduced-motion and prefers-color-scheme**: Respect system accessibility preferences for animations and theme auto-detection. -- **Priority: medium**

13. **Provide vim-mode disable option**: Allow users to disable vim keybindings entirely for accessibility or preference reasons, falling back to standard Tab/Arrow navigation. -- **Priority: medium**

14. **Persist UI state in localStorage**: Panel sizes, sidebar state, selected project, last viewed section, dismissed tips -- all should persist across browser sessions. -- **Priority: low**
