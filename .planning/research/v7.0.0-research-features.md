# v7.0.0 Mission Control Autopilot -- Features Research

**Milestone:** v7.0.0 Mission Control Autopilot
**Research domain:** Feature implementation approach (decomposition, strategies, third-party libraries, dependency ordering)
**Companion reports expected:** architecture, stack, pitfalls, oversights (to be synthesized separately)

This report is a brownfield feature-research document. It takes the four features and two UX improvements from the milestone brief, decomposes each into concrete subsystems, evaluates 2-3 implementation strategies per subsystem against the existing RAPID codebase (React 19 + Vite SPA, FastAPI backend, Node.js CLI plugin), and recommends a build order.

---

## Executive Orientation

### What the milestone is actually asking for

The milestone reads as "ship an autopilot runtime for RAPID in the browser", but it decomposes into **five distinct subsystems** that can be built largely in parallel:

1. **SDK-based agent runner** -- a long-lived process on the backend that spawns Claude Agent SDK queries, streams their messages, and exposes REST/WebSocket endpoints for listing / starting / stopping / inspecting runs.
2. **Web-tool bridge** -- a mapping from SDK tool invocations (especially `AskUserQuestion`) to browser UI controls, via WebSocket round-trip. Uses the SDK's `canUseTool` callback and in-process MCP servers.
3. **Skill invocation UI** -- a frontend form-generation layer that reads each skill's expected arguments and renders an input surface, with the option to skip discuss-phases when invoked via webUI.
4. **Kanban-agent MCP tools** -- custom SDK MCP tools (`add_card`, `view_board`, `move_card`, `archive_card`) that let agents read the Kanban board as a task queue and pull work from it.
5. **WebUI navigation overhaul** -- add "Agents" and "Chats" tabs to the existing sidebar without disturbing the v4.0.0 Mission Control layout (Dashboard, Projects, State, Worktrees, Code Graph, Codebase, Kanban, Notes, Settings).

### Key architecture decision that shapes everything

The **backend must run a persistent process** that holds open Claude Agent SDK sessions. The existing `web/backend/` is Python (FastAPI + SQLModel). The Claude Agent SDK has **two runtimes**: `@anthropic-ai/claude-agent-sdk` (TypeScript/Node) and `claude_agent_sdk` (Python, `claude-agent-sdk` on PyPI). Both expose `query()`, `ClaudeSDKClient`, custom MCP tools via `create_sdk_mcp_server`, and `can_use_tool` callbacks. **Recommended:** use the Python SDK so the agent runner lives inside the existing FastAPI app and shares its database, file watcher, and DB session infrastructure. This avoids a second backend process, a second set of auth/CORS rules, and a second deployment surface.

### Which skills run trivially via SDK vs. need adaptation

Auditing the 30 skills in `skills/`:

| Category | Skills | SDK compatibility |
|----------|--------|-------------------|
| **Fully autonomous (SDK-trivial)** | `plan-set`, `execute-set`, `review`, `merge`, `start-set`, `audit-version`, `documentation`, `migrate`, `new-version` (planning half), `status`, `pause`, `resume`, `cleanup`, `backlog`, `unit-test`, `bug-hunt`, `uat` | Run verbatim via `query()` with `Skill` tool allowed. No UI interaction needed. |
| **AskUserQuestion-gated (need bridge)** | `discuss-set`, `init`, `new-version` (discovery half), `bug-fix` (during investigation), `scaffold`, `branding`, `quick` (when scoping is ambiguous), `assumptions`, `add-set` (mid-milestone discovery) | Need `canUseTool` interception and WebSocket round-trip to browser. |
| **Human-in-loop by design** | `uat` (user acceptance is the whole point), `review` cycle confirmation gates | Can be SDK-backed but UI must render approval prompts. |
| **Environment / plugin-meta** | `install`, `register-web`, `help`, `context` | Out of scope -- these are setup tasks, not autopilot workflows. |

The split suggests a natural MVP cut: **ship the SDK runner + WebUI-skill invocation for the 17 autonomous skills in Wave 1**, then add the `AskUserQuestion` bridge for the interactive skills in Wave 2.

## Feature Inventory

### Feature 1: Claude Agent SDK Integration (Agent Runner)

- **Description:** Run RAPID skills programmatically from the FastAPI backend, so users can click "Run /rapid:plan-set 1" in the browser and see streaming output in real time.
- **Complexity:** complex
- **Dependencies:** none (foundational -- every other feature depends on this)

#### Decomposition

1. **Agent session lifecycle manager** (`AgentSessionService`) -- creates, starts, stops, lists, and persists agent runs. One session = one `query()` call (or `ClaudeSDKClient` instance for multi-turn). Sessions live in memory (dict keyed by UUID) with a DB-backed audit trail.
2. **Message streaming to frontend** -- each session produces a stream of SDK messages (`system/init`, `assistant`, `tool_use`, `tool_result`, `result`, `partial`). These are broadcast to connected WebSocket clients filtered by `sessionId`.
3. **Settings wiring** -- the SDK must load RAPID's skills/agents. Pass `setting_sources=["project"]` + `cwd=<project_root>` + `allowed_tools=["Skill", "Bash", "Read", "Write", "Edit", "Glob", "Grep", "Task"]` so the SDK reads `.claude/skills/*` (which symlinks or is equivalent to the plugin's `skills/` directory) and respects the project's RAPID_TOOLS env var.
4. **Model selection & options** -- expose per-run overrides (`model`, `maxTurns`, `permissionMode: "acceptEdits" | "bypassPermissions" | "plan"`, `systemPrompt`).
5. **Session persistence & resume** -- store SDK `session_id` in SQLite, allow resuming via `options.resume=session_id`. Survives backend restarts.
6. **Log & transcript storage** -- write full message stream to `.planning/agent-runs/<session-id>.jsonl` for post-hoc inspection and `rapid:resume`-style continuation.
7. **Concurrency limiter** -- cap simultaneous running sessions (default 3) to avoid API rate-limit collisions. Queue excess requests.

#### Implementation Strategies

**Option 1: Python Claude Agent SDK + FastAPI async**

- Approach: Install `claude-agent-sdk` (PyPI) into `web/backend/pyproject.toml`. Spawn each run in an `asyncio.Task` inside the FastAPI app. Use `ClaudeSDKClient` for multi-turn sessions and `query()` for one-shots. WebSocket endpoint (`/api/projects/{id}/agents/{session_id}/stream`) yields messages as JSON.
- Pros:
  - Lives in the existing backend process -- shared DB, shared config, no new deploy surface.
  - Python SDK has full parity with Node SDK for `can_use_tool`, `create_sdk_mcp_server`, `@tool` decorator, settings sources, session resume.
  - FastAPI's native WebSocket support is ergonomic.
- Cons:
  - Python SDK is slightly less battle-tested for very long streams than the Node SDK.
  - Requires adding `claude-agent-sdk` + `anthropic` deps to the backend.
- Libraries: `claude-agent-sdk>=0.2`, `anthropic>=0.39`, `websockets` (already transitive via `fastapi[standard]`).

**Option 2: Node.js sidecar process**

- Approach: Build a small Node.js service (`web/agent-runner/`) using `@anthropic-ai/claude-agent-sdk`. FastAPI proxies requests to it via HTTP. Node service streams over WebSocket directly to the browser.
- Pros:
  - Node SDK is the reference implementation -- best tool ecosystem.
  - Keeps the agent runner cleanly isolated from the web backend.
- Cons:
  - Second process to manage, deploy, and supervise.
  - Two languages maintaining RAPID state (CLI already uses `rapid-tools.cjs`).
  - Harder to share the SQLite DB (would need an HTTP boundary for reads).
- Libraries: `@anthropic-ai/claude-agent-sdk@^0.2`, `ws@^8`, `zod@3.25.76`.

**Option 3: Shell out to Claude Code CLI headlessly**

- Approach: Backend spawns `claude-code --print --output-format stream-json --allowed-tools ...` as a subprocess, parses the NDJSON stream. No SDK dependency.
- Pros: Zero new runtime deps; uses the CLI Claude Code already ships.
- Cons:
  - No `canUseTool` callback from outside the CLI process -- AskUserQuestion becomes impossible to intercept, which kills Feature 2.
  - Subprocess lifecycle management is fragile across long-running sessions.
  - Lose the custom SDK MCP server ergonomics (would need a separate MCP server process).
- Libraries: `asyncio.create_subprocess_exec`.

**Recommended: Option 1 (Python SDK in FastAPI).** The existing backend is FastAPI, it already manages a SQLite DB, file watcher, and per-project workflows. Adding the Python SDK keeps the deployment model (one process, `uvicorn`) unchanged. Option 3 is a non-starter because it can't intercept `AskUserQuestion` -- and the whole milestone depends on that.

#### Data Model Impact

- **New tables:**
  - `agent_runs` -- `id (UUID)`, `project_id`, `skill_name`, `skill_args (JSON)`, `status (pending|running|awaiting_input|complete|failed|cancelled)`, `sdk_session_id`, `model`, `permission_mode`, `started_at`, `completed_at`, `error_message`, `transcript_path`.
  - `agent_run_messages` -- append-only; `id`, `run_id`, `seq`, `type (assistant|user|tool_use|tool_result|system|result|partial)`, `content (JSON)`, `created_at`. Optional -- transcripts on disk may be sufficient.
- **Existing entities modified:** None. `Project` already has `root_path`; the runner uses that as `cwd`.
- **Relationships:** `agent_runs.project_id -> projects.id` (FK cascade delete). `agent_run_messages.run_id -> agent_runs.id`.

#### API Surface

- `POST /api/projects/{pid}/agents/runs` -- body `{ skill, args, options }`, returns `{ run_id, session_id }`.
- `GET /api/projects/{pid}/agents/runs` -- list runs (filter by status).
- `GET /api/projects/{pid}/agents/runs/{run_id}` -- metadata + latest status.
- `GET /api/projects/{pid}/agents/runs/{run_id}/transcript` -- full message history (JSON).
- `POST /api/projects/{pid}/agents/runs/{run_id}/cancel` -- abort via `AbortController`.
- `POST /api/projects/{pid}/agents/runs/{run_id}/resume` -- resume a completed/cancelled session.
- `WS /api/projects/{pid}/agents/runs/{run_id}/stream` -- push-based message stream.

### Feature 2: Custom Chat Interface (Web-Tool Bridge)

- **Description:** When an SDK-run skill invokes `AskUserQuestion` (or any interactive tool), render the question in the browser and ship the answer back so the skill continues.
- **Complexity:** complex
- **Dependencies:** Feature 1 (agent runner must exist to intercept tools)

#### Decomposition

1. **`canUseTool` bridge handler** -- Python callback that, on `AskUserQuestion` (or any whitelisted tool), emits a `ui_question` event over the run's WebSocket and `await`s a response from the frontend.
2. **Pending-question queue** -- per-run `asyncio.Future` map keyed by `question_id`. Callback blocks until frontend posts an answer.
3. **Browser `<AskUserQuestion>` renderer** -- React component matching the SDK input schema (`questions[]`, `options[]`, `multiSelect`, `header`). Reuses existing UI primitives in `web/frontend/src/components/ui/`.
4. **Chat thread persistence** -- each run has an associated chat thread (list of `{role: user|assistant|tool|system, content, toolName?, questionId?, answeredAt?}`). Rendered in the "Chats" tab for real-time and historical viewing.
5. **Additional browser-backed tools (Phase 2)** -- wrap existing Claude Code tools that need human input or browser-specific rendering:
   - `Bash` (with approval gate) -- show the command before approving.
   - `Edit` / `Write` (with diff preview) -- render a side-by-side diff in the browser before approving.
   - `WebFetch` / `WebSearch` -- forward to a browser-side `fetch()` to leverage the user's logged-in session cookies (optional future work).
6. **Timeout & cancellation** -- if no answer arrives within N minutes (default 10), return `behavior: "deny"` with a timeout message. User can always cancel the run.

#### Implementation Strategies

**Option 1: `canUseTool` callback with WebSocket round-trip**

- Approach: Pass an async `can_use_tool` to `ClaudeAgentOptions`. On `AskUserQuestion`, it broadcasts the question over the run's WebSocket and awaits an `asyncio.Future` keyed by a generated `question_id`. Frontend posts the answer via `POST /api/projects/{pid}/agents/runs/{run_id}/answer` which resolves the Future. Return `{behavior: "allow", updatedInput: {...questions, answers}}`.
- Pros:
  - Matches the SDK's documented pattern exactly (`permissions.md` explicitly covers `AskUserQuestion`).
  - Works for any tool, not just AskUserQuestion -- can extend to diff-approval for `Edit`.
  - No need to build custom MCP tool wrappers for built-in tools.
- Cons:
  - Tight coupling between the run's event loop and the frontend -- if the user closes the tab, the skill blocks until timeout.
  - WebSocket reconnection logic must re-subscribe to pending questions after a tab reload.
- Libraries: `fastapi.WebSocket`, `asyncio.Future`, `uuid.uuid4`.

**Option 2: Custom SDK MCP tool `ask_via_browser` + hide built-in AskUserQuestion**

- Approach: Disable the built-in `AskUserQuestion` in `allowed_tools`. Register a custom SDK MCP tool `mcp__webui__ask_user` via `create_sdk_mcp_server`. Modify the discuss-set/init skill prompts to prefer this tool. The MCP tool implementation does the WebSocket round-trip internally.
- Pros:
  - Cleaner separation: agents explicitly call a browser tool, no `canUseTool` indirection.
  - Can add richer input types (free-text, file upload, multi-step forms) that `AskUserQuestion` doesn't support.
- Cons:
  - Requires modifying all skills that currently call `AskUserQuestion` -- touches 9+ skill files.
  - Loses the CLI fallback: if the user runs `/rapid:discuss-set` from the CLI, the `ask_user` MCP tool isn't available, so skills must handle both.
- Libraries: `claude-agent-sdk.create_sdk_mcp_server`, `@tool` decorator.

**Option 3: Hybrid -- `canUseTool` for `AskUserQuestion` + custom tools for new capabilities**

- Approach: Use Option 1 for AskUserQuestion (zero skill changes). Add new custom MCP tools only for net-new browser-specific capabilities (e.g. `mcp__webui__show_kanban`, `mcp__webui__render_diff`, `mcp__webui__pick_file`).
- Pros: Best of both. Zero skill churn for existing flows; richer tooling for new flows.
- Cons: Slightly more infrastructure surface -- both `canUseTool` callback AND a custom MCP server.

**Recommended: Option 3 (Hybrid).** Rationale: existing skills in `skills/discuss-set/SKILL.md`, `skills/init/SKILL.md`, `skills/bug-fix/SKILL.md`, etc. already use `AskUserQuestion` and work perfectly from the CLI -- forcing them to switch to a custom MCP tool would break CLI compatibility. Intercepting via `canUseTool` is a zero-touch bridge. Reserve custom MCP tools for new capabilities that don't exist as built-ins.

#### Data Model Impact

- **New tables:**
  - `agent_chat_messages` -- `id`, `run_id`, `seq`, `role (user|assistant|tool|system|question|answer)`, `content (JSON)`, `tool_name`, `question_id`, `answered_at`, `created_at`. Drives the "Chats" tab. Alternatively fold into `agent_run_messages` from Feature 1.
- **Existing entities modified:** None.

#### API Surface

- `POST /api/projects/{pid}/agents/runs/{run_id}/answer` -- body `{ question_id, answers: {[question]: string|string[]} }`. Resolves the pending Future.
- `POST /api/projects/{pid}/agents/runs/{run_id}/deny` -- body `{ question_id, message }`. For user-initiated cancellation of a specific prompt.
- `GET /api/projects/{pid}/agents/runs/{run_id}/pending-questions` -- list currently-awaited questions (for reconnect-after-refresh).
- WebSocket events (server->client):
  - `{type: "ui_question", question_id, questions: [...]}`
  - `{type: "tool_use", tool_name, input}`
  - `{type: "tool_result", tool_name, output}`
  - `{type: "assistant_text", text}`
  - `{type: "run_complete", result}`

### Feature 3: Skill Arguments from WebUI (Invocation UI)

- **Description:** Render a form for each skill's arguments (skill name, positional args, flags like `--skip`, `--gaps`, `--wave-size`) so users can launch skills without hand-typing CLI syntax. Let users opt to skip discuss-phases when launching from WebUI.
- **Complexity:** moderate
- **Dependencies:** Feature 1 (runner), Feature 2 (bridge for interactive skills)

#### Decomposition

1. **Skill argument registry** -- a machine-readable manifest of each skill's accepted args and flags. Lives in `.claude/skills/<name>/SKILL.md` frontmatter or a sibling `ARGS.json`. Parsed by a new `src/lib/skill-manifest.cjs` module.
2. **Form generator** -- React component that reads the manifest and renders inputs: set-picker (dropdown from current milestone's sets), text input, toggle (flag), numeric input (e.g. `--wave-size`).
3. **`--skip-discuss` wrapper flag** -- top-level WebUI concept; when set, the invocation path for an unmerged set automatically runs `/rapid:discuss-set --skip` before `/rapid:plan-set`, chained via the agent runner.
4. **Skill gallery page** -- browsable catalog in the "Agents" tab showing all 30 skills with descriptions, required status preconditions, and "Run" buttons.
5. **Precondition guard** -- before launching, hit `GET /api/projects/{pid}/state` to verify the skill's preconditions (e.g. plan-set requires set status `discussed`) and show a warning or auto-fix option.

#### Implementation Strategies

**Option 1: Manifest via YAML frontmatter in SKILL.md**

- Approach: Extend each `SKILL.md` frontmatter with an `args` schema (Zod-compatible):

  ```yaml
  ---
  description: ...
  allowed-tools: ...
  args:
    - name: set
      type: set-ref
      required: true
      description: Set identifier (numeric or string)
    - name: skip
      type: flag
      default: false
  ---
  ```

  Backend parses frontmatter (new `web/backend/app/services/skill_catalog_service.py`). Frontend fetches via `GET /api/skills`.
- Pros: Co-located with skill definition, version-controlled with the skill. One source of truth.
- Cons: Must update 30 SKILL.md files. Any new skill requires remembering to add args.

**Option 2: External ARGS.json per skill**

- Approach: Sibling file `skills/<name>/ARGS.json` with JSON Schema. Parsed via Ajv (already a RAPID dep).
- Pros: JSON Schema ergonomics; 3rd-party form generators (`@rjsf/core`) can render it.
- Cons: Another file to maintain; drift risk between SKILL.md docs and ARGS.json.

**Option 3: Centralized registry in `src/lib/skill-manifest.cjs`**

- Approach: One Node module exporting the full registry. Backend reads it via `node -e 'console.log(JSON.stringify(require("...")))'`.
- Pros: Single file to review; easy to refactor.
- Cons: Duplicates metadata already in SKILL.md; tight coupling between backend and RAPID CLI internals.

**Recommended: Option 1 (YAML frontmatter).** Rationale: skill authors already read and write SKILL.md. Co-locating the argument schema there keeps docs and schema in lockstep. The migration effort (adding `args:` to 30 files) is mechanical and can be partially automated by generating schemas from each skill's existing "Accept arguments" prose. For rendering, use a simple custom form generator (not `@rjsf`) since the type surface is small (set-ref, flag, text, number).

#### Data Model Impact

- **New entities:** None (skill catalog is read-only metadata sourced from filesystem).
- **Optionally persisted:** recent skill launches (for "Run again" UX) -- can live in `agent_runs` already.

#### API Surface

- `GET /api/skills` -- returns catalog: `[{name, description, args: [{name, type, required, default}], preconditions: {...}}]`.
- `GET /api/skills/{name}` -- single-skill manifest + full SKILL.md markdown body (for tooltip/inline docs).
- `GET /api/projects/{pid}/skills/{name}/preconditions` -- project-scoped precondition check.

### Feature 4: Kanban Agent Integration (Kanban MCP Tools)

- **Description:** Expose the existing Kanban board to agents as an MCP-style tool surface. Users queue todos as cards; agents pull cards, plan them, execute them, and move them through columns autonomously.
- **Complexity:** moderate
- **Dependencies:** Feature 1 (runner). Independent of Features 2 and 3 for basic read/move ops; benefits from Feature 3 for launching the autopilot loop.

#### Decomposition

1. **Kanban SDK MCP server** -- Python `create_sdk_mcp_server` with tools:
   - `mcp__kanban__list_cards(column?, status?)` -- returns cards as JSON.
   - `mcp__kanban__get_card(card_id)` -- returns full card detail.
   - `mcp__kanban__add_card(column, title, description?)` -- creates a new card.
   - `mcp__kanban__move_card(card_id, column, position?)` -- moves between columns.
   - `mcp__kanban__update_card(card_id, title?, description?)` -- updates fields.
   - `mcp__kanban__comment_card(card_id, comment)` -- appends a comment (if comments are modeled).
2. **Column semantic mapping** -- convention-based: `Todo`, `In Progress`, `Done`, `Blocked`. The agent moves cards to `In Progress` when it starts and `Done` when it finishes. Configurable per-project mapping lives in `.planning/config.json`.
3. **Autopilot loop skill** -- a new `/rapid:autopilot` skill that:
   1. Lists cards in `Todo` column.
   2. For each card (in order), launches `/rapid:start-set`, `/rapid:plan-set`, `/rapid:execute-set`, `/rapid:review`, `/rapid:merge` based on card metadata.
   3. Moves the card to `Done` on success or `Blocked` with a comment on failure.
4. **Progress callbacks to Kanban UI** -- the autopilot writes `status: running` / `progress: 45%` back to the card via the MCP tool, which the existing Kanban frontend polls (existing functionality).
5. **Label/priority handling** -- if the Kanban data model has labels (e.g. `bug`, `feature`, `chore`), the autopilot skill routes cards to the appropriate subflow (`bug-fix` vs `add-set` vs `quick`).

#### Implementation Strategies

**Option 1: Custom SDK MCP server calling existing FastAPI kanban endpoints**

- Approach: The MCP tools internally call `get_board`, `create_card`, `move_card` from `app.services.kanban_service`. No new HTTP hop -- direct function calls inside the same process.
- Pros:
  - Zero drift: the UI and agents hit the same service layer.
  - No authentication overhead between MCP tools and the backend.
  - Easy to write.
- Cons: Couples the Kanban MCP server to the agent runner process (they must run in the same Python process, which is already the plan).
- Libraries: `claude-agent-sdk.create_sdk_mcp_server`, `@tool` decorator.

**Option 2: External MCP server (stdio or HTTP)**

- Approach: Standalone `.mcp/kanban-server.py` that speaks MCP stdio. Registered via `ClaudeAgentOptions.mcp_servers["kanban"] = {"type": "stdio", "command": "python -m kanban_mcp"}`.
- Pros: Reusable outside the RAPID web backend (e.g. from Claude Code CLI directly). Standard MCP protocol.
- Cons: Second process. Must re-implement DB access / re-use SQLModel models in a second context. No real benefit for RAPID since the agents only run inside the backend.

**Option 3: HTTP-based MCP via `@tool` wrappers**

- Approach: MCP tools simply `httpx.post()` to the FastAPI kanban endpoints.
- Pros: Bypasses in-process coupling; clean layering.
- Cons: Unnecessary network hop; auth headers required; circular (backend calling itself).

**Recommended: Option 1 (In-process MCP with direct service calls).** Rationale: the agent runner lives in the same FastAPI process (per Feature 1 recommendation). Direct service calls are the fastest path, zero drift with the UI, and need no auth handshake. Option 2 is only worth it if we later want to expose Kanban tools to the Claude Code CLI itself -- which is not an MVP requirement.

#### Data Model Impact

- **Existing entities:** `KanbanColumn`, `KanbanCard` (from `web/backend/app/models/kanban.py`) -- used as-is.
- **New columns on `KanbanCard`:**
  - `agent_run_id` (FK to `agent_runs.id`, nullable) -- which run is currently processing this card.
  - `agent_status` (enum: `queued | in_progress | blocked | done`, nullable) -- for autopilot lifecycle.
  - `metadata` (JSON, nullable) -- card-level overrides (skill to invoke, flags, etc.).
- **New entity (optional):** `KanbanCardComment` for autopilot progress notes. The Kanban data model may already support this -- check `web/backend/app/models/kanban.py` before adding.
- **Migration:** Alembic migration to add the three new columns.

#### API Surface

No new user-facing HTTP endpoints (agents hit the MCP tool surface, not HTTP). For UI polling after agent mutations, the existing Kanban endpoints are sufficient.

Internally (MCP tools, inside the SDK server):
- `mcp__kanban__list_cards`, `mcp__kanban__get_card`, `mcp__kanban__add_card`, `mcp__kanban__move_card`, `mcp__kanban__update_card`, `mcp__kanban__comment_card`.

### Feature 5: WebUI Navigation Overhaul (Agents + Chats tabs)

- **Description:** Preserve the existing sidebar layout but add two new top-level tabs: "Agents" (skill catalog + run history + launch UI) and "Chats" (live and historical chat threads per run).
- **Complexity:** moderate
- **Dependencies:** Feature 1 (runs), Feature 2 (chat messages), Feature 3 (skill invocation)

#### Decomposition

1. **Route registration** -- add `/agents` and `/chats` to `web/frontend/src/router.tsx`. Point at new lazy-loaded pages.
2. **Sidebar entries** -- update `AppLayout` to include the two new items with appropriate icons (spark/robot for Agents, speech-bubble for Chats).
3. **Agents page** -- top-level tabs: "Catalog" (browse skills), "Runs" (history), "New Run" (quick launch). Default to "Runs" view.
4. **Chats page** -- list of chat threads, filterable by project/run/status. Each thread links to a detail view that renders the message stream, with pending questions shown inline as interactive forms.
5. **Dashboard widget** -- optional: add a "Recent agent runs" card to the existing dashboard so users don't have to navigate away to see active work.
6. **Real-time indicators** -- sidebar badges showing count of active runs / pending questions. Updated via a global WebSocket subscription.
7. **Keyboard shortcuts** -- integrate with existing `KeyboardProvider` (see `web/frontend/src/context/KeyboardContext.tsx`). Map `g a` (go agents), `g c` (go chats), `n r` (new run).

#### Implementation Strategies

**Option 1: Two new routes, shared design language with existing pages**

- Approach: Use existing `<AppLayout>`, CodeMirror for transcript rendering (already a dep), `@tanstack/react-query` for data fetching, `zustand` for run-list state (already a dep). WebSocket handled by a custom hook `useAgentRunStream(runId)`.
- Pros: Zero new dependencies; consistent UX with rest of Mission Control.
- Cons: Must build a chat-style message renderer from scratch (not trivial for streaming Markdown + tool-use rendering).
- Libraries: existing React 19, react-router 7, zustand 5, tanstack-query 5, codemirror 6 (markdown lang).

**Option 2: Embedded chat UI library (e.g. `@assistant-ui/react`)**

- Approach: Use a drop-in chat library for the Chats page.
- Pros: Faster time-to-MVP for the chat UI.
- Cons: Adds a heavyweight dep; likely fights the Everforest theme; unclear how it handles custom tool-use rendering and question-form interspersing.
- Libraries: `@assistant-ui/react` or similar.

**Option 3: Dashboard-only integration (defer full Agents/Chats pages)**

- Approach: Don't add new routes; stuff the agent runs list and chat into the existing Dashboard as expandable panels.
- Pros: Minimal routing surface; preserves existing nav strictly.
- Cons: Dashboard becomes cluttered; doesn't match the "new tabs" requirement in the milestone brief.

**Recommended: Option 1.** The milestone explicitly says add tabs while preserving the layout, and the existing React stack is fully capable. Custom message rendering is a ~300-line component; it pays off in long-term maintainability and design consistency.

#### Data Model Impact

- **None.** Pages consume Feature 1 and Feature 2 APIs.

#### API Surface

- Already covered by Features 1-3 APIs. No new endpoints required.

### Feature 6 (UX): Browser-Native Workflow

- **Description:** Full RAPID experience from the browser -- no CLI required for core flows. This is an **outcome**, not an independent feature. It is satisfied by Features 1-5 landing successfully.
- **Complexity:** n/a (meta-goal)
- **Dependencies:** All of Features 1-5

#### What "full browser-native" means concretely

Users should be able to:
- Initialize a new RAPID project from the browser (`/rapid:init` via UI).
- Start, discuss, plan, execute, review, and merge sets without opening a terminal.
- Queue work via Kanban cards and have agents execute them autopilot-style.
- See all agent output, ask/answer questions, and inspect transcripts from the browser.

#### Gap analysis vs. CLI-only flows

The `install`, `register-web`, and `context` skills are bootstrap/setup flows that **still require CLI access** (they configure env vars, create `.env` files, and set up the plugin). Accept this scope boundary: "Browser-native after install." Document it in the init UX.

## Recommended Implementation Order (Wave Plan)

### Wave 1 (foundation, 1 set)

- **Feature 1: SDK Agent Runner**
  - End state: backend can spawn, stream, persist, list, resume, and cancel SDK runs for the 17 non-interactive skills. WebSocket streaming of messages works. Minimal web UI: a "Runs" page at `/runs` (not yet the Agents tab). No AskUserQuestion bridge yet.
  - Why first: everything depends on this. Ship it with the 17 auto-skills working end-to-end so the value is visible immediately.

### Wave 2 (parallel, 2 sets)

- **Feature 2: Web-Tool Bridge**
  - End state: `canUseTool` intercepts `AskUserQuestion` and round-trips through WebSocket. Discussable skills (discuss-set, init, bug-fix, etc.) work from browser. Chat thread persisted.
- **Feature 3: Skill Invocation UI**
  - End state: SKILL.md frontmatter extended with `args:` for all 30 skills. `/api/skills` catalog endpoint. Form-generating React component. "New Run" flow from skill catalog.
  - Note: these two are **independent** because Feature 3 operates on the launch side (before the SDK query starts) while Feature 2 operates on the runtime side (after the query is running). They can land in parallel in two worktrees.

### Wave 3 (parallel, 2 sets)

- **Feature 4: Kanban Agent Tools**
  - End state: Kanban MCP server registered with every SDK run. A new `/rapid:autopilot` skill pulls cards from "Todo" and drives them through the set lifecycle. Cards update their status column as work progresses.
- **Feature 5: WebUI Navigation Overhaul**
  - End state: `/agents` and `/chats` routes with full catalog, run history, chat threads, pending-question UI, and dashboard widget. Replaces the provisional `/runs` page from Wave 1.
  - Note: these two are independent. Feature 4 is backend + new skill; Feature 5 is frontend-only.

### Wave 4 (polish, 1 set)

- Browser-native workflow validation -- end-to-end walkthrough of a fresh project going init -> plan -> execute -> merge entirely in the browser.
- Observability polish: run log viewer, error retry UX, timeout tuning.
- Docs + demo video.

## Open Questions / Ambiguities

These should be resolved in `/rapid:discuss-set` for each set, not pre-decided here:

1. **Does the Python SDK need to live inside the FastAPI app or in a sidecar process?** Strongly recommend in-process (per Feature 1 Option 1), but the synthesis agent may prefer isolation for fault tolerance.
2. **Should the Kanban board be the canonical work queue, or just one of many queue sources?** If canonical, the autopilot skill ships as part of core RAPID; if optional, it lives behind a feature flag.
3. **How do we handle auth between browser and backend?** The existing backend is single-user localhost; adding multi-user auth is a separable concern. Recommend: MVP assumes single-user localhost; document this clearly.
4. **Concurrent run limit default?** 3 feels right for Claude Max/API rate limits, but should be a project-level setting in `.planning/config.json`.
5. **Chat thread retention?** Transcripts can grow large. Propose: full transcript on disk at `.planning/agent-runs/<id>.jsonl`, DB stores only metadata + last N messages (100?) for fast list queries.
6. **Should `/rapid:autopilot` be a new skill or an extension of `/rapid:quick`?** `/rapid:quick` already handles ad-hoc tasks; Kanban-driven autopilot is its spiritual cousin. Propose: new skill, but share the planner/executor pipeline.

## Risks & Mitigations (abbreviated -- full treatment in pitfalls research)

| Risk | Impact | Mitigation |
|------|--------|-----------|
| SDK `canUseTool` semantics change between minor versions | Feature 2 breaks silently | Pin SDK version; add integration test that exercises AskUserQuestion round-trip |
| Long-running runs leak memory / sockets | Backend degrades over time | Session TTL; aggressive cleanup on `result` message; WebSocket heartbeat |
| User closes browser during pending question | Skill hangs until timeout | Persist pending questions in DB; resume on reconnect; short default timeout (10min) with clear UX |
| Agents modify files outside worktree | Breaks RAPID isolation model | Enforce `cwd` per run; pass `permission_mode: "acceptEdits"` with `allowedTools` scoped to worktree path; reuse existing worktree-scoped CLAUDE.md |
| Autopilot infinite-loops on a broken card | Burns API tokens | Per-card max-retry counter; card-level timeout; "Blocked" column halts retries |
| Multiple runs trying to merge concurrently | State corruption | Reuse existing `proper-lockfile` via `rapid-tools.cjs` -- SDK calls through CLI for state mutations, not direct DB writes |

## Summary for Synthesizer

**Core recommendation:** Build this in 5 sets across 4 waves. Start with the Python SDK integrated directly into the existing FastAPI backend (Wave 1). Add the canUseTool bridge and skill invocation UI in parallel (Wave 2). Add Kanban MCP tools and the navigation overhaul in parallel (Wave 3). Polish and ship (Wave 4).

**Critical design decision that cascades:** the Python Claude Agent SDK in-process inside FastAPI, not a Node sidecar. This shapes data model, concurrency, deployment, and every subsequent feature.

**17 skills ship autonomously via SDK on day one; 9 need the bridge; 4 stay CLI-only by scope.**

**The Kanban board is the natural home for an autopilot task queue** -- cards already exist, the model supports columns and positions, and users already interact with it. The autopilot skill is a thin orchestration layer on top of existing skills.

**No blocking third-party dependency decisions** -- the only new runtime deps are `claude-agent-sdk` and `anthropic` (both well-maintained, MIT-licensed, pinnable).
