# Domain & UX Research -- v7.0.0 (Mission Control Autopilot)

**Domain:** Browser-native autonomous agent control plane for a CLI-first developer tool
**Researched:** 2026-04-15
**Overall confidence:** HIGH on chat/approval/dashboard patterns (well-established in Cursor/Zed/Replit/Devin/ChatGPT/Claude.ai); MEDIUM on the RAPID-specific grafting problem (few direct precedents for "CLI plugin with a parallel webUI driver").

This milestone extends the existing Mission Control webUI (React + Vite frontend, Python FastAPI backend) so that the **browser becomes a first-class driver for RAPID workflows** -- not a read-only observer. The skill-catalog of `/rapid:init`, `/rapid:discuss-set`, `/rapid:plan-set`, `/rapid:execute-set`, `/rapid:merge` must be invokable from the browser. Some skills have no user input (plan, execute) and should run autonomously. Others (init, discuss-set) require structured prompts -- they must be translated from `AskUserQuestion`-style CLI prompts to browser form controls via the Claude Agent SDK's custom-tool / UI-state primitives.

## Domain Conventions

### Comparable Products (How Similar Products Work)

RAPID's v7 problem is "put a chat-driven autonomous agent frontend on top of an existing CLI tool." The precedents cluster into four groups:

| Group | Examples | Interaction Model | What RAPID Should Borrow |
|-------|----------|------------------|--------------------------|
| **IDE chat panels** | Cursor Composer, Zed Agent Panel, JetBrains Junie | Chat pane beside editor; streaming text + inline tool cards | Per-message tool-call cards (spinner then checkmark); "approve/reject" inline on destructive tool calls; persistent thread history |
| **Web agent playgrounds** | Claude.ai (with artifacts), Anthropic Console Workbench, v0.dev | Chat left / artifact right; SSE streaming; "typing" cursor | Side-panel artifacts pattern (diff, log, plan file on right); token-level streaming; suggested-action chips at start |
| **Autonomous agent dashboards** | Replit Agent 3/4, Devin 2.0, Manus, OpenAI Operator | List/grid of agent "runs"; live timeline per run; "step in to steer" | Agent cards with status/elapsed/last-step; phone-style live-monitoring of long runs; "cut losses early" banners |
| **Work-item autopilots** | Linear AI agent, Factory.ai, Port.io + coding agents, "Claude Project Tracker" | Tickets as agent input units; agent picks up, works, reports back | Kanban card -> autonomous set (discuss -> plan -> execute) handoff; "agent claimed this ticket" badges |

RAPID already has a kanban board, worktrees, and the full skill catalog. The design job is to **compose these existing pieces into an autopilot flow**, not to invent new metaphors. Every pattern below maps back to a precedent.

### Sources for Each Cluster

- **IDE chat panels:** Cursor 2.0 Composer, [Zed Agent Panel (May 2025)](https://zed.dev/blog/zed-ai), Zed Text Threads docs. Zed's Agent Panel is the closest direct reference -- full text editor that exposes the entire LLM request with code/history/files visible as text; supports tool calls.
- **Web agent playgrounds:** [Claude.ai artifacts side panel](https://albato.com/blog/publications/how-to-use-claude-artifacts-guide), [Anthropic Console Workbench](https://learnprompting.org/blog/upgraded-anthropic-console), [v0.dev chat UX](https://v0.app/chat/chatbot-interface-7z2FadaDqOw).
- **Autonomous agent dashboards:** [Replit Agent 3 (200-min autonomy + live monitoring)](https://blog.replit.com/introducing-agent-3-our-most-autonomous-agent-yet), [Replit Agent 4 parallel tasks](https://blog.replit.com/introducing-agent-4-built-for-creativity), [Devin 2.0 design](https://medium.com/@takafumi.endo/agent-native-development-a-deep-dive-into-devin-2-0s-technical-design-3451587d23c0), [Manus autonomy](https://www.asad.pw/ai-agents-experiences-with-manus-operator-and-more/).
- **Work-item autopilots:** [Linear AI agent-aware workflows](https://linear.app/), [Factory.ai for Linear/Jira](https://factory.ai/product/ai-project-manager), [Port.io auto-resolve tickets with coding agents](https://docs.port.io/guides/all/automatically-resolve-tickets-with-coding-agents/), [Claude Project Tracker `.project/` folder pattern](https://dev.to/rpostulart/i-built-jira-for-ai-agents-heres-why-your-ai-coding-assistant-needs-its-own-project-management-1m5f).

### Terminology Conventions

| Term in RAPID v7 | Industry Standard | Notes |
|------------------|-------------------|-------|
| Agent run | Run / Session / Thread / Job | Devin: "session"; Replit: "task"; Cursor: "composer run"; Linear: "agent run". **Recommendation:** use **"run"** (shortest, unambiguous, matches Replit & Cursor). |
| Autopilot | Autopilot / Autonomous mode / Headless | Pause.do, Replit Agent 3, Devin all use "autopilot" or "autonomous". Anti-term: "auto mode" (ambiguous). |
| Skill launch | Trigger / Dispatch / Invoke | GitHub Actions: `workflow_dispatch`; Zapier: "trigger"; Linear agents: "assign issue". **Recommendation:** "run a skill" (user-facing) / "dispatch" (technical). |
| Approval gate | Interrupt / Approval / Confirm step | [OpenAI Agents SDK](https://openai.github.io/openai-agents-js/guides/human-in-the-loop/), [AG-UI protocol](https://www.codecademy.com/article/ag-ui-agent-user-interaction-protocol), Claude Agent SDK: all use "interrupt" or "approval". **Recommendation:** "approval" (user-facing) / "interrupt" (backend). |
| AskUserQuestion (CLI) | Form / Prompt / Question | Browser equivalent = form with pre-filled radio/checkbox options, never freeform textarea (preserves the CLI's "bounded decision" pattern). |

### User Journey (End-to-End Browser Workflow)

Based on composite patterns from Replit Agent 3, Linear AI, and Devin 2.0, the expected v7 browser journey is:

1. **Capture** -- User drops an idea onto the kanban board (a card). No agent involved yet.
2. **Assign** -- User drags the card to an "Agent Queue" column, or clicks "Run with RAPID". The autopilot picks it up.
3. **Chat (discuss)** -- Chat tab opens. Agent asks the 2-5 structured questions from `/rapid:discuss-set` as browser form controls. User answers inline; no CLI round-trip.
4. **Plan & execute (autonomous)** -- Agent runs `/rapid:plan-set` and `/rapid:execute-set` silently in the background. Dashboard shows a "run" card with progress: step name, elapsed time, current agent.
5. **Approval interrupts** -- If a destructive action comes up (merge conflict resolution, force-push, delete worktree), the run pauses and surfaces an approval modal. User approves / rejects / edits the action.
6. **Review & merge** -- `/rapid:review` and `/rapid:merge` run with the same chat + approval surface. User is the final gate; agent does the grunt work.
7. **Done** -- Card moves to "Merged" column. Run archived in Runs tab. Chat thread archived in Chats tab.

## Interaction Models

### Primary Interaction Paradigm: Three Surfaces, One Dashboard

The webUI already has a sidebar with Dashboard / Projects / State / Worktrees / Graph / Codebase / Kanban / Notes / Settings. v7 adds **two more siblings** (not sub-tabs, not a separate nav): **Agents** and **Chats**.

Three distinct *modes* exist for user-agent interaction. Users should be able to tell from the UI chrome alone which mode they're in:

| Surface | Mode | Visual Signal | When Used |
|---------|------|---------------|-----------|
| **Chats tab** | Interactive conversation | Input composer visible at bottom; "typing" indicator when agent responds; message history flows top-to-bottom | User is actively talking with an agent (e.g., `/rapid:discuss-set`, `/rapid:init`, `/rapid:quick`) |
| **Agents tab (run detail)** | Autonomous monitoring | No input composer; timeline/activity feed; big status pill ("RUNNING / 4m 32s"); "Pause" or "Interrupt" button | Skill is running headless (e.g., `/rapid:plan-set`, `/rapid:execute-set`) |
| **Kanban** | Work queue + handoff | Card with "Agent claimed" badge; status chip on card; "Pickup" button per card | User is queuing / reviewing / reassigning work items |

The mode-distinction is load-bearing. [Smashing Mag's agentic UX patterns](https://www.smashingmagazine.com/2026/02/designing-agentic-ai-practical-ux-patterns/) and the [AG-UI protocol](https://www.codecademy.com/article/ag-ui-agent-user-interaction-protocol) both emphasize that "is this a chat or an autonomous run?" is the single most confusing question users face. Making the surface itself the answer avoids ambiguity.

### Input/Output Patterns Users Expect

#### Chat Tab

Established expectations (from Claude.ai, Cursor Composer, Zed Agent Panel, v0.dev, ChatGPT):

1. **Scrolling message list + sticky composer at bottom.** Textarea auto-grows. Send button right-aligned inside or beside the textarea. Enter sends, Shift+Enter newline. ([v0 chatbot pattern](https://v0.app/chat/chatbot-interface-7z2FadaDqOw))
2. **Token-level streaming.** Text appears character-by-character with a blinking cursor at the tail; code blocks render with syntax highlighting after the closing fence streams. No "thinking..." spinner as long as tokens are flowing.
3. **Inline tool-call cards.** When the agent calls a tool, a small card renders inline in the message with: tool name, spinner while running, checkmark + duration when complete, expandable to show args + result. This is the [assistant-ui](https://github.com/assistant-ui/assistant-ui) / [prompt-kit](https://www.prompt-kit.com/chat-ui) convention and matches Cursor Composer / Claude.ai.
4. **Structured question prompts (not freeform).** When the agent needs input (e.g., discuss-set gray area), it renders a form inline in the message: radio/checkbox options with pre-filled choices, an optional "Let Claude decide" button, and a Submit action. This preserves the CLI's `AskUserQuestion` contract in the browser. Map via a Claude Agent SDK custom tool like `ask_user_question(question, options, multiselect)` that renders a `ComponentInstance` in the chat. ([Claude Agent SDK demos UI state pattern](https://github.com/anthropics/claude-agent-sdk-demos/blob/main/email-agent/UI_STATE_COMPONENTS_SPEC.md))
5. **Suggested-action chips at start.** Empty-state chat shows 3-5 "Try: /rapid:init", "Try: discuss set 'auth'" chips. ([v0.dev suggested actions](https://v0.app/chat/chatbot-interface-7z2FadaDqOw))
6. **Artifact side panel (optional).** For diffs, plan files, generated CONTRACT.json, open a collapsible right pane instead of flooding the chat. Matches [Claude.ai artifacts](https://albato.com/blog/publications/how-to-use-claude-artifacts-guide). MEDIUM priority -- can defer if complexity balloons.

#### Agents Tab (Run Detail)

Established expectations (from Replit Agent 3, Devin 2.0, Manus, AgentCenter):

1. **Prominent status pill.** Top of page: `RUNNING / 00:04:32` or `WAITING FOR APPROVAL` or `FAILED / 00:12:08 / last step: execute-wave-2`. Pill color-codes: green=running, amber=waiting, red=failed, gray=done.
2. **Live activity feed.** Chronological stream of agent actions: "Spawned rapid-planner", "Wrote .planning/PLAN.md", "Committed abc1234", "Called rapid-tools state set ..." -- similar to Replit's "Live Activity Feed" and [Replit Agent 3's phone monitoring](https://blog.replit.com/introducing-agent-3-our-most-autonomous-agent-yet).
3. **Current-step breadcrumb.** "Set `auth` > Wave 2 > Task 3 of 5 > Agent rapid-executor". One line, wrapping sparingly. Mirrors RAPID's existing CLI breadcrumb format.
4. **Cost / token / duration telemetry.** Small row: "Tokens: 48.3k in / 12.1k out  |  Duration: 4m 32s  |  Cache hit: 72%". This is the "agent-specific UI" that [is table-stakes for 2026 chat UIs](https://dev.to/alexander_lukashov/i-evaluated-every-ai-chat-ui-library-in-2026-heres-what-i-found-and-what-i-built-4p10).
5. **Pause / Stop / Fork controls.** Top-right. "Pause" = send SIGSTOP-equivalent, keep run resumable. "Stop" = terminate and archive. "Fork" = spawn a new run from the current state (nice-to-have, v7.1).
6. **No input composer.** A run is headless by definition. To talk to it, user clicks "Open as chat" (if the underlying skill supports mid-run interaction) which opens a linked chat in the Chats tab.

#### Agents Tab (Run List / Dashboard View)

Expectations (from AgentCenter, Replit, Autensa, Devin session list):

1. **Card grid or table.** Each run = one card/row with: name, status pill, elapsed, last step, project/set, actions (View / Pause / Cancel).
2. **Filter chips.** "Running", "Waiting approval", "Failed", "Completed", "Archived". Persistent as URL params.
3. **"Start a run" affordance.** Top-right button opens a skill launcher modal (see below).

### Feedback and Progress Indication

| Signal | Chat Tab | Agents Tab | Kanban |
|--------|---------|-----------|--------|
| Agent is thinking | Blinking cursor + text streaming | "Step: researching codebase..." in activity feed | Card badge: "Agent working..." |
| Tool call in progress | Inline spinner card ("Reading src/lib/state.cjs...") | Activity feed line + elapsed counter on current step | -- |
| Awaiting user input | Inline form renders; composer disables | Big amber pill "AWAITING APPROVAL"; modal pops | Card badge: "Needs review" |
| Completed | Final assistant message; "Done" indicator | Status pill -> green "COMPLETED / Xm Ys" | Card auto-moves to "Done" column |
| Failed | Red error banner in chat + "Retry" button | Status pill -> red "FAILED"; log excerpt below pill | Card moves back to "Todo" + red flag |

### Error Communication Patterns

Match the CLI's established "errors-as-navigation" pattern (see `.planning/research/DOMAIN-UX.md` v3 research) in browser form:

- **Error card in chat:** Red left border, error title, 1-2 sentence explanation, one clear "Next" action as a button ("Retry", "Open logs", "Continue manually in CLI"). Never just a stacktrace.
- **Toast for transient issues:** Network blips, SSE disconnects, polling failures. Use existing toast/notify system.
- **Full-page for fatal:** If the backend is down entirely, the whole Agents tab shows a "Mission Control backend unreachable" state with a "Retry" button. Do not let the UI render a half-broken shell.

## Information Architecture

### Navigation: Sidebar Extension, Not Top Tabs

**Decision: extend the existing left sidebar with two new items (Agents, Chats). Do not introduce top tabs.** Confidence: HIGH.

Rationale:

1. **Existing pattern is already a left sidebar** (`web/frontend/src/components/layout/Sidebar.tsx`, `NAV_ITEMS` in `types/layout.ts`). The brief explicitly says "preserve the current dashboard layout."
2. **Sidebars scale to 9-12 items gracefully** (current RAPID has 9). Adding two brings it to 11. The UX literature supports up to ~12 primary items in a sidebar before breaking into subgroups ([Sidebar Design for Web Apps 2026](https://www.alfdesigngroup.com/post/improve-your-sidebar-design-for-web-apps)).
3. **Top tabs are for equal-weight peers within a single page**, not for top-level navigation in an admin dashboard ([Tabs UX best practices](https://www.eleken.co/blog-posts/tabs-ux), [LogRocket](https://blog.logrocket.com/ux-design/tabs-ux-best-practices/)). Agents and Chats are not sub-views of one page -- they're distinct surfaces.
4. **Keyboard shortcuts compose.** RAPID already uses `g<letter>` nav shortcuts (`gd`, `gp`, `gs`, `gw`, `gk`, `gc`, `gb`). Add `ga` (agents), `gh` (chats -- "h" for "history" since `gc` is codebase). No collision with existing bindings in `AppLayout.tsx`.
5. **Runs/chats need deep-linkable detail pages.** Top tabs typically swap in-page content; sidebar nav + router routes (`/agents/:runId`, `/chats/:threadId`) gives shareable URLs and browser-back semantics for free.

Proposed updated `NAV_ITEMS` order (grouped by lifecycle):

```
Dashboard  (gd)
Projects   (gp)
---
Agents     (ga)   <- NEW
Chats      (gh)   <- NEW
Kanban     (gb)
---
State      (gs)
Worktrees  (gw)
Graph      (gk)
Codebase   (gc)
Notes
Settings
```

Rationale for placement: Agents/Chats/Kanban are the "driver" surfaces (user actively works); State/Worktrees/Graph/Codebase are the "observer" surfaces (user inspects). Group them with a separator.

### Sub-Page Structure for Agents Tab

Two-level IA to keep the page manageable:

- **`/agents`** -- dashboard/list. Card grid or table. Filter chips. "Start a run" button.
- **`/agents/:runId`** -- run detail. Status pill, activity feed, controls, linked chat.

For Chats:

- **`/chats`** -- thread list. Most-recent-first. Filter: "Active", "Archived", "Has pending approval".
- **`/chats/:threadId`** -- full chat view. Same layout as detail pages in Claude.ai / ChatGPT.

### Skill Launcher (Triggering Parameterized Skills from Web)

Pattern: GitHub Actions' `workflow_dispatch` modal is the canonical reference ([GitHub Actions input types](https://github.blog/changelog/2021-11-10-github-actions-input-types-for-manual-workflows/)). Zapier and n8n use similar form-based triggers with typed inputs. All three converge on:

- **Per-skill argument schema** (declared once alongside the SKILL.md).
- **Typed inputs:** string / choice (enum) / boolean / multi-line-string. GitHub Actions caps at 10 inputs; RAPID skills rarely exceed 4, so 10 is plenty.
- **Description + required flag + default value** per input.
- **"Skip discuss" toggle** if the skill supports it (matches the brief's requirement).

Recommended launcher UI:

1. User clicks "Run a skill" button (from Agents tab dashboard, or from kanban card's "Run with RAPID" action).
2. Modal opens with two-pane: left = skill picker (searchable list), right = argument form for selected skill.
3. Form renders dynamically from the skill's argument schema. Unknown skills fall back to a single "Extra arguments" textarea.
4. "Skip discuss phase" checkbox is offered when the chosen skill is `/rapid:init` or `/rapid:start-set` (per the brief).
5. Submit -> create run, navigate to `/agents/:runId`.

Store skill arg schemas in the same place as the SKILL.md so they stay co-located. A minimal schema is enough -- do not over-engineer into a dynamic form DSL.

### Progressive Disclosure

Match the CLI's existing cognitive-load budget: **surface the next thing to do, hide everything else**.

- Empty Agents tab: big "Start a run" CTA + 3 example skill chips. No agent-monitoring cruft until there's something to monitor.
- Empty Chats tab: "Start a chat" with skill-picker chips.
- Kanban unchanged in empty state; "Run with RAPID" button appears on-hover per card.
- Run detail page: activity feed collapsed to last 5 events by default with "Show full log" expander.

## Accessibility Considerations

Baseline: the webUI already has [`TooltipOverlay`](file:///home/kek/Projects/RAPID/web/frontend/src/components/ui/TooltipOverlay.tsx), keyboard shortcuts, mobile drawer. v7 additions must preserve these patterns.

### Critical Patterns

1. **Status pills must not be color-only.** Use both a color and a text label (`RUNNING`, `WAITING`, `FAILED`). Do not ship "green dot = success" without the word. WCAG 1.4.1 (Use of Color). Matches the CLI's "text label + color" convention from `display.cjs`.
2. **Tool-call cards need ARIA live regions.** When a new tool call streams in, announce via `aria-live="polite"`. Do not spam -- debounce to one announcement per 2s. Screen-reader users otherwise can't follow a streaming agent.
3. **Chat streaming + `aria-busy`.** While tokens stream, `aria-busy="true"` on the message region; flip to `false` on completion. Otherwise screen readers re-read the entire partial message on each chunk. Reference: [assistant-ui's streaming accessibility](https://github.com/assistant-ui/assistant-ui).
4. **Approval modals trap focus.** Destructive-action dialogs must be focus-trapped and have a clear default (usually "Reject" for destructive). ESC = cancel/reject, not approve. AG-UI convention.
5. **Keyboard-accessible pause/stop/approve.** Every action reachable via the keyboard. Pause/stop on run detail: `Shift+P` / `Shift+S`. Approve/reject on modal: `Enter` = default (reject for destructive), `Shift+Enter` = approve.
6. **Auto-scroll can be disabled.** Long activity feeds and chats must let users scroll up without being yanked back by streaming updates. Follow [Chrome DevTools Console pattern](https://developer.chrome.com/docs/devtools): if user scrolls away from bottom, stop auto-scrolling until they click the "new messages" pill that appears.
7. **Reduced motion.** The spinner-to-checkmark animation on tool-call cards should respect `prefers-reduced-motion`. Use a simple opacity swap instead of a spin.

### Not Applicable / Deferred

- Speech input: no RAPID user has asked for it; defer indefinitely.
- Internationalization: RAPID is English-only in the CLI; keep parity.

## User Expectations

Based on domain research, users of browser-native autonomous agent control panels will assume:

### Default Behaviors Users Expect

1. **The browser is the control panel; the agent lives on the backend.** Closing the tab does not kill the run. The run keeps going; reopening the tab re-attaches to its stream. Matches Replit Agent 3 (phone monitoring), Devin (cloud VM), Manus. Anti-pattern: treating the browser tab as the session owner. This aligns with the user's CLAUDE.md preference for polling over long-lived sockets -- the run is a durable server-side resource.
2. **Status updates stream in real-time (or feel like they do).** Users hate "refresh to see progress." Given the CLAUDE.md preference, the recommended pattern is **short-interval polling (2-5s) on the runs list and current run detail**, with **optional SSE** for the chat message stream only (because token-level streaming is the one place polling is visibly inferior). Hybrid is normal -- AG-UI [calls out that its protocol works over HTTP, SSE, or WebSockets](https://www.codecademy.com/article/ag-ui-agent-user-interaction-protocol) precisely because different update rates need different transports.
3. **Destructive actions require explicit approval.** Merge conflict resolution, force-push, delete worktree, overwrite existing file -- all must interrupt and wait. [Industry consensus](https://brainroad.com/when-your-ai-agent-needs-permission-building-approval-workflows/): "everything reversible runs autonomously; destructive/irreversible requires approval." Approve / Reject / **Edit** (modify tool args before approving) is the right triad -- binary approve/reject forces black-and-white decisions when users want nuance.
4. **A run is a durable, resumable thing.** Users expect to pause, come back tomorrow, and resume. Not restart from scratch. Matches how [OpenAI Agents SDK RunState](https://openai.github.io/openai-agents-js/guides/human-in-the-loop/) and RAPID's existing `/rapid:pause` / `/rapid:resume` commands already work on the CLI side.
5. **Chats and runs have history.** Users expect an archive; `gpt_chat_history`-style lists sorted by most recent. Runs must be findable by set name, project, date range, status.
6. **The kanban flow is "drop it, let the agent handle it, check back later."** Linear's AI agent is the reference: drop a ticket on the "Agent" lane; the agent picks it up, runs the pipeline, reports progress on the card, and moves the card when done. User never has to manually trigger each skill step. RAPID's autopilot = discuss (with chat handoff if needed) -> plan -> execute -> review, all triggered off the card.
7. **"Running" vs "chatting" is obvious at a glance.** The surface distinction above (no composer on Agents tab; composer prominent on Chats tab) serves this. Matches how Cursor Composer separates "agent is running" view from "talk to the agent" view.

### Terminology Users Prefer

Most users in this domain say "run" (short, matches CI/CD mental model), "agent" (a specific named worker), "skill" or "action" (what gets run). Avoid:

- "Task" -- overloaded with kanban cards and Claude Code's own `/tasks`.
- "Job" -- RAPID already uses "job" internally for wave sub-units; don't promote it to the UI.
- "Session" -- implies connection-owned, not durable; conflicts with the "browser is control panel" expectation.

### Mental Models from Competing Products

- **GitHub Actions:** "workflow_dispatch" modal + a "runs" list per workflow. Users will expect RAPID's skill launcher to look similar. ([GitHub input types](https://github.blog/changelog/2021-11-10-github-actions-input-types-for-manual-workflows/))
- **Linear agents:** "this ticket has an agent assigned" badge. Users will expect RAPID kanban cards to show the same.
- **Replit Agent 3:** phone-friendly live-monitoring view. Users will expect the webUI to feel usable on a tablet if not a phone.
- **Devin 2.0:** "step in to steer" UX -- a running agent can be paused and given new instructions mid-run. Nice-to-have for v7.1; v7.0 can ship with just pause/stop/resume.

### Onboarding Expectations

- First-time Agents/Chats tab visit: show an empty-state hero with "What can I do here?" + 3 example buttons ("Plan a set", "Execute a set", "Discuss a set").
- Skill launcher modal should show a one-line description of each skill picked from the SKILL.md. Users should not have to know the 30 slash-commands by heart.
- Chats tab for users new to the CLI: explain in the empty state that "chat" = interactive mode, "agents" = autonomous mode. This is the single most-likely-to-confuse distinction.

## Recommendations

Ordered by priority. Each item is an action for the roadmap, not an implementation spec -- the feature-research and architecture-research outputs translate these into tasks.

1. **[CRITICAL]** Add **Agents** and **Chats** items to the existing left sidebar (not top tabs), grouped with Kanban as "driver" surfaces separated from the observer surfaces. Use `ga` / `gh` keyboard shortcuts. -- Rationale: preserves the existing dashboard aesthetic per the brief; sidebars scale; deep-linkable detail routes matter for shareable run/chat URLs.

2. **[CRITICAL]** Make "**is this a chat or a run?**" answerable from UI chrome alone. Chat tab = persistent composer at bottom; Agents tab = no composer, prominent status pill + activity feed. -- Rationale: the single most confusing question in agentic UX; making the surface itself the answer eliminates the confusion entirely.

3. **[CRITICAL]** Run = durable server-side resource; the tab is just a view. Closing the tab does not kill the run. Reopening re-attaches. -- Rationale: matches Replit / Devin / Manus expectations; matches RAPID's existing `/pause` / `/resume` CLI semantics; aligns with user's CLAUDE.md preference for polling over long-lived HTTP sockets (run state lives in backend, polled by browser).

4. **[CRITICAL]** Use the **Claude Agent SDK's custom-tool + UI-state primitives** to map CLI `AskUserQuestion` prompts to browser form components inline in chat. Do not render freeform textareas for bounded decisions. -- Rationale: preserves the CLI's "bounded decision" UX contract; Claude Agent SDK already has `ComponentTemplate` / `ComponentInstance` / `triggerAction` designed exactly for this; skill compatibility with both CLI and webUI hinges on this abstraction.

5. **[HIGH]** Inline **tool-call cards** in chat: spinner while running, checkmark + duration + expandable args/result when complete. -- Rationale: table-stakes in 2026 chat UIs (Cursor, Claude.ai, Zed, assistant-ui, prompt-kit all do this); establishes transparency and trust.

6. **[HIGH]** **Approval modal** pattern for destructive tool calls: Approve / Reject / Edit (modify args before approve). Focus-trapped, ESC = reject. Wire via Claude Agent SDK interrupt/approval flow. -- Rationale: "destructive = require approval, reversible = autonomous" is industry consensus; Edit (not just approve/reject) prevents forcing binary decisions.

7. **[HIGH]** **Short-interval polling (2-5s)** for runs list and run-detail status. **Optional SSE** only for chat token streaming. -- Rationale: user's CLAUDE.md explicitly prefers polling for long-running processes; hybrid is conventional (AG-UI uses both); token streaming is the single exception where polling is visibly worse.

8. **[HIGH]** **Kanban-to-autopilot handoff.** Kanban card gets a "Run with RAPID" action; running becomes visible on the card (badge + status chip); card auto-moves on status change. -- Rationale: matches the brief's "queue todos on the kanban, agents read/plan/execute autonomously"; Linear AI + Port.io + Factory.ai all converge on this pattern.

9. **[HIGH]** **Skill launcher modal** modeled on GitHub Actions `workflow_dispatch`. Typed inputs (string/choice/bool/multi-line), descriptions, defaults, required flag. Dedicated "Skip discuss" toggle for skills that support it. -- Rationale: brief explicitly calls for skill arguments from webUI and discuss-skip; GitHub Actions pattern is familiar to every developer.

10. **[HIGH]** **Run dashboard:** card grid with status pill, elapsed, last-step, filter chips ("Running / Waiting / Failed / Completed / Archived"). Links to run-detail. -- Rationale: Replit/AgentCenter/Autensa convention; filter chips avoid pagination noise.

11. **[MEDIUM]** **Token / cost / duration telemetry** on every run. -- Rationale: 2026 expectation (cost transparency is table-stakes per dev.to evaluation of AI chat libraries); RAPID already has the data via Claude Agent SDK usage responses.

12. **[MEDIUM]** **Live activity feed** on run detail: one line per significant agent action (tool call, commit, state transition). Auto-scroll with opt-out. -- Rationale: Replit Live Activity Feed and Devin timeline are the references; grows trust in autonomous runs.

13. **[MEDIUM]** **Status pills + color + text label** (no color-only status signalling). ARIA live regions for streaming content. `aria-busy` during token streams. Focus-trapped approval modals. -- Rationale: WCAG 1.4.1 and standard accessibility practice; avoids regressing existing webUI accessibility posture.

14. **[MEDIUM]** **Empty-state onboarding** on Agents / Chats tabs explaining the chat-vs-run distinction and offering 3 example actions. -- Rationale: most-likely-to-confuse UX distinction; one-time investment pays off forever.

15. **[LOW]** **Artifact side panel** in chat for diffs / plan files / CONTRACT.json. -- Rationale: Claude.ai pattern; keeps chat flow readable. Can defer to v7.1 if complexity balloons.

16. **[LOW]** **"Open as chat" button** on run detail for skills that support mid-run interaction. -- Rationale: Devin's "step in to steer" pattern; edge case in v7.0 since most RAPID skills are either fully-interactive (chat) or fully-autonomous (run), not mixed.

17. **[LOW]** **Fork run** action. -- Rationale: nice-to-have from Replit Agent 4; not in brief; defer to v7.1.
