# Features Research -- Mission Control (v4.0)

## Feature Inventory

### Feature: Project Registry

- **Description:** A central registry that tracks all active RAPID projects the user is working on. Projects self-register when `/rapid:init` detects the `RAPID_WEB=true` env var, or manually via a `/register_web` command for older projects. The registry provides the data backbone for every other dashboard view.
- **Complexity:** moderate
- **Dependencies:** Service Management (the web service must be running to accept registrations)

#### Decomposition

1. **SQLite project table** -- Schema for storing project name, root path, git remote, current milestone, registration timestamp, last-seen timestamp.
2. **Registration HTTP endpoint** -- `POST /api/projects/register` accepts project metadata from `rapid-tools.cjs` and upserts into the database.
3. **Heartbeat / refresh endpoint** -- `POST /api/projects/{id}/heartbeat` updates last-seen timestamp; `GET /api/projects/{id}/state` reads the project's `.planning/STATE.json` on demand.
4. **rapid-tools.cjs registration helper** -- A new `web register` subcommand in the CLI that POSTs project metadata to `http://localhost:8998/api/projects/register`.
5. **Init hook integration** -- Modify the init flow to call the registration helper when `RAPID_WEB=true` is detected in the environment.
6. **`/register_web` skill** -- A new Claude Code skill that wraps the CLI registration helper for older projects.
7. **Project list frontend** -- Sidebar or landing page listing all registered projects with status indicators (active/stale based on last-seen).
8. **Deregistration endpoint** -- `DELETE /api/projects/{id}` for cleanup.

#### Implementation Strategies

**Option 1: SQLite via SQLModel (recommended)**
- Approach: Use SQLModel (Pydantic + SQLAlchemy hybrid) with FastAPI dependency injection. Each project is a row with columns: `id` (UUID), `name`, `root_path` (unique), `git_remote`, `current_milestone`, `rapid_version`, `registered_at`, `last_seen_at`.
- Pros: First-class FastAPI integration, type-safe models, automatic OpenAPI docs, migrations via Alembic.
- Cons: Adds Python dependency management (pip/uv).
- Libraries: SQLModel, aiosqlite (async SQLite driver), Alembic (migrations).

**Option 2: Raw sqlite3 with Pydantic models**
- Approach: Use Python's built-in `sqlite3` module directly with Pydantic models for validation.
- Pros: Zero additional dependencies beyond FastAPI/Pydantic, simpler.
- Cons: Manual SQL, no migration framework, more boilerplate.

**Recommended:** Option 1 (SQLModel). The ORM layer pays for itself in reduced boilerplate, and SQLModel is designed specifically for FastAPI.

#### Data Model Impact

- **New entities:** `Project` table (id, name, root_path, git_remote, current_milestone, rapid_version, registered_at, last_seen_at, metadata JSON blob).
- **Existing entities modified:** None -- the web service has its own database at `~/.rapid/rapid.db`, completely separate from per-project `.planning/STATE.json`.
- **Relationships:** Project is the root entity; all other features (kanban, notes, knowledge graph) reference a project via foreign key.

#### API Surface

- `POST /api/projects/register` -- Register or update a project
- `GET /api/projects` -- List all registered projects
- `GET /api/projects/{id}` -- Get single project details
- `GET /api/projects/{id}/state` -- Read live STATE.json from project root
- `POST /api/projects/{id}/heartbeat` -- Update last-seen timestamp
- `DELETE /api/projects/{id}` -- Deregister a project

### Feature: Knowledge Graph

- **Description:** Interactive visual graph showing each project's structure -- sets, waves, dependencies between sets (from DAG.json), and component relationships. Provides a birds-eye view of project decomposition.
- **Complexity:** complex
- **Dependencies:** Project Registry (needs project root path to read DAG.json and STATE.json)

#### Decomposition

1. **DAG data endpoint** -- `GET /api/projects/{id}/dag` reads `.planning/sets/DAG.json` from the registered project root and returns structured node/edge data.
2. **State overlay endpoint** -- `GET /api/projects/{id}/graph` merges DAG topology with current set statuses from STATE.json, producing enriched nodes with status colors.
3. **React Flow graph component** -- Interactive node-based graph using React Flow with dagre auto-layout. Custom node components render set/wave/job types with status badges.
4. **Custom node types** -- Three node types: `SetNode` (shows set name, status, wave assignment), `WaveNode` (groups sets by execution wave), `JobNode` (individual tasks within waves).
5. **Layout engine integration** -- Use dagre for automatic hierarchical layout (top-to-bottom or left-to-right), matching the DAG's wave structure.
6. **Status-colored edges** -- Edge styling reflects dependency state: grey (pending), blue (in progress), green (complete), gold (merged).
7. **Node detail panel** -- Click a node to see details: contract summary, file ownership, worktree branch, wave assignment.
8. **Graph controls** -- Zoom, pan, minimap, fit-view, layout direction toggle (TB/LR).

#### Implementation Strategies

**Option 1: React Flow + dagre (recommended)**
- Approach: Use `@xyflow/react` (React Flow v12) for the interactive canvas with `dagre` for automatic node positioning. Map RAPID's DAG.json nodes/edges directly to React Flow's node/edge format. Custom node components styled with Everforest palette.
- Pros: Production-ready, well-documented, built-in zoom/pan/minimap/controls, excellent custom node support, dagre integration is a solved pattern.
- Cons: React Flow v12 has a pro license for some features (but core is MIT), dagre is unmaintained (but stable).
- Libraries: `@xyflow/react` (MIT core), `dagre` (auto-layout), `@dagrejs/graphlib`.

**Option 2: D3-force with custom SVG**
- Approach: Use D3's force-directed layout with custom SVG rendering.
- Pros: Full control over rendering, no framework lock-in.
- Cons: Significantly more implementation effort, must build zoom/pan/selection from scratch, force layout is less appropriate for DAGs than hierarchical layout.

**Recommended:** Option 1. React Flow + dagre is the standard approach for DAG visualization in React. The RAPID DAG format maps almost 1:1 to React Flow's data model.

#### Data Model Impact

- **New entities:** None -- reads existing DAG.json and STATE.json.
- **Existing entities modified:** None.
- **Relationships:** Graph data is derived from the project's filesystem, not stored in the dashboard database.

#### API Surface

- `GET /api/projects/{id}/dag` -- Raw DAG.json data
- `GET /api/projects/{id}/graph` -- Enriched graph with status overlay

### Feature: Kanban Board

- **Description:** Drag-and-drop task tracking board for organizing work items across columns (e.g., Backlog, In Progress, Review, Done). Persisted in the dashboard's SQLite database, associated with a registered project.
- **Complexity:** complex
- **Dependencies:** Project Registry (board is scoped to a project)

#### Decomposition

1. **Board data model** -- SQLite tables: `Board` (id, project_id, name, created_at), `Column` (id, board_id, title, position), `Card` (id, column_id, title, description, labels, position, created_at, updated_at).
2. **Board CRUD endpoints** -- Standard REST for boards, columns, and cards with position-aware ordering.
3. **Drag-and-drop frontend** -- React component using dnd-kit for card reordering within and across columns. Optimistic UI updates with backend sync.
4. **Card detail modal** -- Expanded view for editing card title, description (markdown), labels, and linked set references.
5. **Auto-populate from state** -- Optional one-time import: generate cards from STATE.json sets/waves, mapping set statuses to column positions.
6. **Position management** -- Fractional indexing (e.g., LexoRank-style) for efficient reordering without renumbering all positions.
7. **Label system** -- Color-coded labels (using Everforest accent colors) for categorizing cards by type (feature, bug, docs, etc.).
8. **Keyboard navigation** -- Vim-style: `j/k` to move between cards, `h/l` to move between columns, `Enter` to open card detail, `n` to create new card.

#### Implementation Strategies

**Option 1: dnd-kit + custom board (recommended)**
- Approach: Use `@dnd-kit/core` and `@dnd-kit/sortable` for drag-and-drop mechanics. Build the board layout with CSS Grid. Cards are sortable items within droppable column containers.
- Pros: Highly flexible, accessible (keyboard + screen reader), lightweight (~10kb), actively maintained, no opinionated styling.
- Cons: More implementation work than a pre-built kanban component; must build column/card components from scratch.
- Libraries: `@dnd-kit/core`, `@dnd-kit/sortable`, `@dnd-kit/utilities`.

**Option 2: @hello-pangea/dnd (react-beautiful-dnd fork)**
- Approach: Use the maintained fork of react-beautiful-dnd, which provides higher-level abstractions for list-based drag-and-drop.
- Pros: Less boilerplate for simple kanban, built-in animations.
- Cons: Less flexible than dnd-kit for custom behaviors, slightly larger bundle.

**Recommended:** Option 1 (dnd-kit). Better accessibility support, more flexible for custom keyboard navigation (which is a design constraint), and actively maintained.

#### Data Model Impact

- **New entities:** `Board` (id, project_id FK, name, created_at), `Column` (id, board_id FK, title, position), `Card` (id, column_id FK, title, description, labels JSON, position, created_at, updated_at).
- **Existing entities modified:** None.
- **Relationships:** Board -> Project (many-to-one), Board -> Columns (one-to-many), Column -> Cards (one-to-many).

#### API Surface

- `GET /api/projects/{id}/boards` -- List boards for a project
- `POST /api/projects/{id}/boards` -- Create a board
- `GET /api/boards/{id}` -- Get board with columns and cards
- `PUT /api/boards/{id}` -- Update board metadata
- `POST /api/boards/{id}/columns` -- Add a column
- `PUT /api/columns/{id}` -- Update column (title, position)
- `DELETE /api/columns/{id}` -- Remove a column
- `POST /api/columns/{id}/cards` -- Add a card
- `PUT /api/cards/{id}` -- Update card (title, description, labels, position, column_id for moves)
- `DELETE /api/cards/{id}` -- Remove a card
- `POST /api/boards/{id}/import-state` -- Auto-populate board from STATE.json

### Feature: Markdown Note Editor

- **Description:** Rich markdown editing interface for writing todo lists, notes, and documentation within the dashboard. Notes are persisted per-project in the dashboard database.
- **Complexity:** moderate
- **Dependencies:** Project Registry (notes are scoped to a project)

#### Decomposition

1. **Note data model** -- SQLite table: `Note` (id, project_id FK, title, content TEXT, pinned BOOLEAN, tags JSON, created_at, updated_at).
2. **Note CRUD endpoints** -- REST API for creating, reading, updating, deleting, and listing notes with search/filter by tags.
3. **Markdown editor component** -- WYSIWYG editor using MDXEditor or @uiw/react-md-editor with live preview, toolbar, and keyboard shortcuts.
4. **Note list sidebar** -- Searchable, filterable list of notes with pin-to-top support and tag filtering.
5. **Auto-save** -- Debounced auto-save (500ms after last keystroke) with save indicator.
6. **Todo list support** -- Checkbox rendering in markdown (GFM task lists) with click-to-toggle that updates the markdown source.
7. **Vim keybindings** -- Editor supports vim mode for text editing (most markdown editors support this via CodeMirror or similar).
8. **Export** -- Download note as `.md` file.

#### Implementation Strategies

**Option 1: @uiw/react-md-editor (recommended)**
- Approach: Use the lightweight split-pane markdown editor with built-in preview, syntax highlighting, and toolbar. ~4.6kb gzipped.
- Pros: Small bundle, simple API, built-in preview mode, active maintenance, supports custom toolbar commands, good GFM support.
- Cons: Split-pane rather than true WYSIWYG, no vim mode built-in.
- Libraries: `@uiw/react-md-editor`, `rehype-sanitize` (XSS protection).

**Option 2: MDXEditor**
- Approach: True WYSIWYG markdown editor built on Lexical framework.
- Pros: Inline editing (no preview pane needed), rich plugin system, table support.
- Cons: Heavier bundle (~50kb+), more complex configuration, may be overkill for notes.

**Option 3: Milkdown**
- Approach: WYSIWYG editor built on ProseMirror with plugin system.
- Pros: Beautiful out-of-box, plugin architecture, vim mode via prosemirror-vim.
- Cons: Smaller community, more complex setup.

**Recommended:** Option 1 (@uiw/react-md-editor) for v4.0. It is lightweight, simple to integrate, and covers the core use case. Upgrade to MDXEditor or Milkdown in a future version if richer editing is needed.

#### Data Model Impact

- **New entities:** `Note` (id, project_id FK, title, content, pinned, tags, created_at, updated_at).
- **Existing entities modified:** None.
- **Relationships:** Note -> Project (many-to-one).

#### API Surface

- `GET /api/projects/{id}/notes` -- List notes (with optional tag/search filter)
- `POST /api/projects/{id}/notes` -- Create a note
- `GET /api/notes/{id}` -- Get a single note
- `PUT /api/notes/{id}` -- Update a note (title, content, pinned, tags)
- `DELETE /api/notes/{id}` -- Delete a note
- `GET /api/notes/{id}/export` -- Download as .md file

### Feature: Codebase Mapping

- **Description:** Visual codebase analysis for compatible codebases. A Python backend uses AST parsing (tree-sitter for multi-language support) to extract function/class/module relationships and render them as interactive graphs.
- **Complexity:** very-complex
- **Dependencies:** Project Registry (needs project root path), Knowledge Graph (reuses React Flow infrastructure)

#### Decomposition

1. **AST analysis engine** -- Python module using tree-sitter (via `tree-sitter-language-pack`) to parse source files in JavaScript, TypeScript, Python, Go, Rust, etc. Extracts: functions, classes, imports, exports, call graphs.
2. **File scanner** -- Walk the project directory (respecting `.gitignore` and SKIP_DIRS from RAPID's existing `context.cjs`), collect source files by language.
3. **Dependency graph builder** -- Build import/export dependency graph between modules. Identify strongly connected components.
4. **Complexity metrics** -- Calculate cyclomatic complexity, function length, and nesting depth per function.
5. **Analysis cache** -- Cache parsed results in SQLite (per-project, keyed by file path + mtime hash) to avoid re-parsing unchanged files.
6. **Analysis trigger endpoint** -- `POST /api/projects/{id}/analyze` triggers background analysis. Returns a task ID for polling.
7. **Analysis status polling** -- `GET /api/projects/{id}/analysis/status` returns progress (files scanned / total files, current phase).
8. **Codebase graph endpoint** -- `GET /api/projects/{id}/codebase/graph` returns module dependency graph for React Flow rendering.
9. **File detail endpoint** -- `GET /api/projects/{id}/codebase/files/{path}` returns functions, classes, complexity metrics for a single file.
10. **React Flow codebase view** -- Reuse graph infrastructure from Knowledge Graph with different custom nodes (file nodes, module nodes, function nodes). Color by complexity or language.
11. **Directory treemap** -- Optional treemap visualization showing file sizes/complexity as nested rectangles.

#### Implementation Strategies

**Option 1: tree-sitter via tree-sitter-language-pack (recommended)**
- Approach: Use the `tree-sitter-language-pack` Python package that bundles 170+ language parsers. Parse each source file into a CST, then extract semantic information (function definitions, class definitions, import statements) using tree-sitter queries. Build a dependency graph from import/export relationships.
- Pros: Multi-language support from a single dependency, incremental parsing, production-proven, fast native parsing.
- Cons: tree-sitter grammars vary in quality across languages, query patterns differ per language, native binary dependency.
- Libraries: `tree-sitter-language-pack` (bundled parsers), `tree-sitter` (Python bindings).

**Option 2: Python ast module + language-specific parsers**
- Approach: Use Python's built-in `ast` module for Python files, and separate tools (e.g., TypeScript compiler API via subprocess) for other languages.
- Pros: No native dependencies for Python analysis, Python `ast` module is well-documented.
- Cons: Each language needs its own parser integration, much more code, slower for non-Python languages.

**Recommended:** Option 1 (tree-sitter-language-pack). It provides consistent multi-language support through a unified API, which is essential for a tool that analyzes arbitrary codebases.

#### Data Model Impact

- **New entities:** `AnalysisResult` (id, project_id FK, status, started_at, completed_at, file_count, error_log), `FileAnalysis` (id, analysis_id FK, file_path, language, mtime_hash, functions JSON, classes JSON, imports JSON, exports JSON, complexity_metrics JSON).
- **Existing entities modified:** None.
- **Relationships:** AnalysisResult -> Project (many-to-one), FileAnalysis -> AnalysisResult (many-to-one).

#### API Surface

- `POST /api/projects/{id}/analyze` -- Trigger background analysis (returns task_id)
- `GET /api/projects/{id}/analysis/status` -- Poll analysis progress
- `GET /api/projects/{id}/codebase/graph` -- Module dependency graph
- `GET /api/projects/{id}/codebase/files` -- List analyzed files with summary stats
- `GET /api/projects/{id}/codebase/files/{path}` -- Detailed file analysis (functions, classes, complexity)
- `GET /api/projects/{id}/codebase/stats` -- Aggregate codebase statistics (LOC, language breakdown, avg complexity)

### Feature: Worktree Tracking

- **Description:** Monitor active git worktrees, their status, branch associations, and mapping to RAPID sets. Shows which worktrees are active, orphaned, or cleaned up.
- **Complexity:** moderate
- **Dependencies:** Project Registry (needs project root to read REGISTRY.json and run git commands)

#### Decomposition

1. **Worktree data endpoint** -- `GET /api/projects/{id}/worktrees` reads `.planning/worktrees/REGISTRY.json` from the project root and returns enriched worktree data.
2. **Live git status enrichment** -- For each active worktree, run `git status --porcelain` and `git log --oneline -5` to show uncommitted changes and recent commits.
3. **Worktree list component** -- Table/card view showing: set name, branch, status (active/orphaned), dirty/clean indicator, last commit message, creation date.
4. **Reconciliation trigger** -- Button to trigger `rapid-tools worktree reconcile` via API, then refresh the view.
5. **Worktree detail panel** -- Click a worktree to see: full branch name, absolute path, diff summary, recent commits, associated set status.
6. **Status polling** -- Frontend polls `GET /api/projects/{id}/worktrees` every 30 seconds when the worktree view is active.

#### Implementation Strategies

**Option 1: Read REGISTRY.json + subprocess git (recommended)**
- Approach: FastAPI endpoint reads the project's `REGISTRY.json` file and enriches each entry by running git commands (status, log) in the worktree directory via `subprocess.run`.
- Pros: Uses existing RAPID data structures, no duplication, always reflects ground truth.
- Cons: Subprocess calls add latency (~100-200ms per worktree); must handle projects with many worktrees.
- Libraries: Python `subprocess`, `json` (stdlib).

**Option 2: Mirror registry into dashboard SQLite**
- Approach: Periodically sync REGISTRY.json into the dashboard database, query from there.
- Pros: Faster reads, can add dashboard-specific metadata.
- Cons: Data staleness, sync logic complexity, duplication of truth.

**Recommended:** Option 1. Read from source of truth on each request. Cache responses with short TTL (5s) to avoid hammering git for rapid polling.

#### Data Model Impact

- **New entities:** None -- reads from existing `.planning/worktrees/REGISTRY.json`.
- **Existing entities modified:** None.
- **Relationships:** Worktree data is derived from the project's filesystem.

#### API Surface

- `GET /api/projects/{id}/worktrees` -- List worktrees with status
- `GET /api/projects/{id}/worktrees/{set_name}` -- Detailed worktree info (git status, recent commits)
- `POST /api/projects/{id}/worktrees/reconcile` -- Trigger reconciliation

### Feature: Project State View

- **Description:** Display the full project state hierarchy: milestones, sets, set statuses, wave assignments, job progress. This is the read-only equivalent of running `rapid-tools worktree status-v2`.
- **Complexity:** moderate
- **Dependencies:** Project Registry (needs project root to read STATE.json)

#### Decomposition

1. **State data endpoint** -- `GET /api/projects/{id}/state` reads `.planning/STATE.json` and returns the full validated ProjectState object.
2. **Milestone progress component** -- Visual progress bar showing sets completed/total, with color-coded status indicators matching set lifecycle (pending -> discussed -> planned -> executed -> complete -> merged).
3. **Set list with status badges** -- Expandable table/accordion: each set shows status badge, wave assignment, job count, and expand to show waves/jobs.
4. **Wave grouping view** -- Alternative view grouping sets by execution wave (from DAG.json), showing which wave is currently active.
5. **Status timeline** -- Horizontal timeline showing set transitions over time (using `lastUpdatedAt` timestamps).
6. **Version display** -- Show project name, current milestone, RAPID version, last updated timestamp.
7. **Auto-refresh** -- Poll state every 10 seconds when the project view is active.

#### Implementation Strategies

**Option 1: Direct STATE.json read with Pydantic validation (recommended)**
- Approach: FastAPI reads STATE.json from the project root, validates against a Pydantic model mirroring RAPID's Zod schema (ProjectState -> MilestoneState -> SetState -> WaveState -> JobState).
- Pros: Always shows current state, no sync lag, validates data integrity.
- Cons: File I/O on each request (mitigated by caching).
- Libraries: Pydantic (already a FastAPI dependency).

**Recommended:** Option 1. This is a read-only dashboard -- reading the source of truth directly is the simplest and most reliable approach. Add 5-second response caching to avoid excessive file I/O.

#### Data Model Impact

- **New entities:** None -- reads existing STATE.json.
- **Existing entities modified:** None.
- **Relationships:** State data is derived from the project's filesystem.

#### API Surface

- `GET /api/projects/{id}/state` -- Full project state (milestones, sets, waves, jobs)
- `GET /api/projects/{id}/state/summary` -- Lightweight summary (milestone progress, set counts by status)
- `GET /api/projects/{id}/state/timeline` -- Status transition history (derived from git log of STATE.json changes)

### Feature: Service Management

- **Description:** The web service lifecycle: installation as a systemd user service, `RAPID_WEB=true` env var detection, and start/stop/status management. Includes setup guidance for non-elevated permissions.
- **Complexity:** moderate
- **Dependencies:** independent (foundational infrastructure)

#### Decomposition

1. **FastAPI application scaffold** -- Main `app.py` with CORS middleware, lifespan handler for database init, and router registration. Runs on `0.0.0.0:8998` (or `127.0.0.1:8998` for security).
2. **SQLite database initialization** -- Create `~/.rapid/rapid.db` on first startup with all required tables.
3. **Systemd user unit file** -- `~/.config/systemd/user/rapid-web.service` that runs the FastAPI server via uvicorn.
4. **Install command** -- Extension to `/rapid:install` skill that (a) asks user if they want web dashboard, (b) if yes: creates `~/.rapid/` directory, generates systemd unit file, writes `RAPID_WEB=true` to shell config, runs `systemctl --user enable --now rapid-web`.
5. **Linger enablement** -- Guide user to run `loginctl enable-linger $USER` so the service persists beyond login sessions.
6. **Health check endpoint** -- `GET /api/health` returns server status, uptime, database path, version.
7. **Static file serving** -- Serve the Vite-built frontend from a `static/` directory within the FastAPI app.
8. **Development mode** -- `uvicorn app:app --reload` with Vite dev server proxy for development.
9. **Python dependency management** -- Use `uv` (fast pip replacement) with a `pyproject.toml` and lockfile. Virtual environment at `~/.rapid/.venv/`.
10. **RAPID_WEB env var** -- Add `RAPID_WEB=true` to shell config alongside existing `RAPID_TOOLS` during install.

#### Implementation Strategies

**Option 1: Systemd user unit + uv-managed venv (recommended)**
- Approach: Ship a template systemd unit file. During install, create `~/.rapid/`, set up a Python venv with `uv`, install dependencies, write the unit file to `~/.config/systemd/user/rapid-web.service`, enable and start. The unit file uses `ExecStart` pointing to the venv's uvicorn.
- Pros: Standard Linux service management, survives reboots, user-level (no root needed), `uv` is fast and reliable.
- Cons: Systemd-specific (not portable to macOS launchd), requires Python 3.10+.
- Libraries: `uvicorn` (ASGI server), `uv` (package manager), systemd user services.

**Option 2: Process manager (pm2 or supervisor)**
- Approach: Use pm2 or supervisor to manage the FastAPI process.
- Pros: Cross-platform.
- Cons: Additional dependency, less integrated with OS, user must learn another tool.

**Recommended:** Option 1. Systemd user services are the standard for user-level daemons on Linux. For macOS, a launchd plist can be added later as a separate concern. The spec explicitly targets systemd.

#### Data Model Impact

- **New entities:** The SQLite database itself at `~/.rapid/rapid.db`. All tables for other features live here.
- **Existing entities modified:** None in the RAPID core.
- **Relationships:** The database is the persistence layer for all dashboard-owned data.

#### API Surface

- `GET /api/health` -- Health check (status, uptime, version, db path)
- `GET /` -- Serve frontend SPA

## Feature Dependency Graph

```
Service Management (foundational)
  └── Project Registry (requires running web service)
        ├── Knowledge Graph (requires project root path for DAG.json)
        ├── Kanban Board (boards scoped to projects)
        ├── Markdown Note Editor (notes scoped to projects)
        ├── Codebase Mapping (requires project root, reuses graph infra)
        │     └── (also depends on Knowledge Graph for React Flow reuse)
        ├── Worktree Tracking (requires project root for REGISTRY.json)
        └── Project State View (requires project root for STATE.json)
```

All features except Service Management depend on Project Registry. Codebase Mapping additionally benefits from Knowledge Graph's React Flow infrastructure being built first.

## Implementation Order

1. **Service Management** -- Foundational infrastructure. Without the web server, nothing else can run. Includes FastAPI scaffold, SQLite init, systemd unit, install flow.
   - *Rationale:* Everything else depends on this.

2. **Project Registry** -- Core data backbone. Every other feature needs to know which projects exist and where their root directories are.
   - *Rationale:* Direct dependency for all remaining features.

3. **Project State View** -- Simplest read-only view. Reads STATE.json and renders milestone/set/wave hierarchy. Proves the end-to-end pipeline (register project -> read its data -> render in frontend).
   - *Rationale:* Low complexity, validates the full stack, high value for users monitoring set progress.

4. **Worktree Tracking** -- Similar pattern to State View (read from project filesystem, render). Adds git subprocess enrichment.
   - *Rationale:* Builds on patterns from State View, moderate complexity, high utility for active development.

5. **Knowledge Graph** -- Introduces React Flow infrastructure that Codebase Mapping will reuse. Maps existing DAG.json to interactive graph.
   - *Rationale:* Must precede Codebase Mapping; provides visual "wow factor" for the dashboard.

6. **Kanban Board** -- Independent feature with its own data model. Can be built in parallel with Knowledge Graph once the project scaffold is solid.
   - *Rationale:* Complex but self-contained; drag-and-drop requires careful implementation.

7. **Markdown Note Editor** -- Straightforward CRUD with a markdown editor component. Lower priority since users already have editors.
   - *Rationale:* Moderate complexity, nice-to-have for knowledge management.

8. **Codebase Mapping** -- Most complex feature. Requires tree-sitter parsing, background task infrastructure, and reuses React Flow from Knowledge Graph.
   - *Rationale:* Highest complexity, benefits from all preceding infrastructure being stable.

## Cross-Cutting Concerns

### Frontend Architecture
- **State management:** TanStack Query for server state (polling, caching, background refetch). React context for UI state (selected project, theme, sidebar collapse).
- **Routing:** React Router v7 with project-scoped routes: `/projects/:id/state`, `/projects/:id/graph`, `/projects/:id/board`, etc.
- **Theme:** Everforest dark as default. CSS custom properties for theme switching. Tailwind CSS with custom color palette mapping Everforest hex values to Tailwind tokens.
- **Keyboard navigation:** Global vim-style hotkeys via a custom `useVimNavigation` hook. `g` prefix for goto commands (e.g., `gs` = go to state, `gg` = go to graph). Helper tooltips shown on `?` keypress.
- **Polling pattern:** All data-fetching endpoints use TanStack Query's `refetchInterval` for polling. Intervals vary by view (10s for state, 30s for worktrees, manual for analysis).

### Backend Architecture
- **Project structure:** `~/.rapid/server/` containing `app.py`, `models.py`, `routers/`, `services/`, `analysis/`.
- **Background tasks:** FastAPI `BackgroundTasks` for short operations. For codebase analysis (which can run 30+ seconds), use a task queue pattern: write task to SQLite, worker thread picks it up, frontend polls for status.
- **Error handling:** Pydantic validation errors return 422. File-not-found (missing STATE.json, etc.) returns 404 with descriptive message. Project root inaccessible returns 503.
- **Security:** Bind to `127.0.0.1:8998` (localhost only). No authentication needed for v4.0 since it is a local-only service.
