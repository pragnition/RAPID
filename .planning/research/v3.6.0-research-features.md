# Features Research -- v3.6.0 Workflow & UX Polish

## Feature Inventory

### Feature 1: Bug Fixes (Error Logs, DAG.json ENOENT, Execute State Completion)

- **Description:** Three distinct bugs that need fixing: (1) error logs from merge pipeline when DAG.json is missing at `.planning/sets/DAG.json`, (2) DAG.json ENOENT crash during merge order computation, (3) execute-set agent not transitioning set state to `complete` after execution, causing the review agent to refuse review.
- **Complexity:** moderate
- **Dependencies:** independent (but should be fixed before other features)

#### Decomposition

1. **DAG.json ENOENT guard in merge pipeline** -- The `getMergeOrder()` function in `src/lib/merge.cjs` (line 1561) does a raw `fs.readFileSync` on `.planning/sets/DAG.json` without an existence check. The `detectCascadeImpact()` function (line 1888) looks at `.planning/DAG.json` (wrong path). Both need graceful handling when DAG.json does not exist.
2. **DAG.json path inconsistency fix** -- `detectCascadeImpact()` in merge.cjs reads from `.planning/DAG.json` while every other consumer reads from `.planning/sets/DAG.json`. This is a bug introduced by a path mismatch.
3. **Execute-set state completion fix** -- The execute-set SKILL.md (Step 6) does call `state transition set ... complete`, but the issue is that executors may not always emit a valid RAPID:RETURN marker, causing the orchestrator to exit early before reaching Step 6. The fix needs to ensure the transition happens even in edge cases (e.g., when the executor returns without the marker but did make commits).
4. **Error log sanitization** -- Review all DAG.json consumers in `src/commands/execute.cjs`, `src/commands/worktree.cjs`, `src/commands/merge.cjs` to ensure graceful fallback instead of throwing ENOENT.

#### Implementation Strategies

**Option 1: Guard-and-fallback pattern**
- Approach: Add `fs.existsSync()` checks before every `fs.readFileSync` of DAG.json. Where DAG is missing, provide meaningful fallback behavior (e.g., return empty merge order, skip cascade detection, list sets alphabetically instead of by wave).
- Pros: Minimal code changes, backwards compatible, each consumer decides its own fallback.
- Cons: Repetitive guard logic in multiple files.
- Libraries/services: None needed (native fs).

**Option 2: Centralized DAG loader with tryLoadDAG()**
- Approach: Create a `tryLoadDAG(cwd)` function in `src/lib/plan.cjs` that returns `{ dag: object | null, path: string }` with null when the file doesn't exist. All consumers import this one function.
- Pros: Single source of truth for DAG loading, consistent error handling, easy to add future DAG version migration logic.
- Cons: Requires updating all DAG consumers to use the new function.
- Libraries/services: None.

**Recommended:** Option 2. A centralized loader eliminates the path inconsistency bug entirely (one canonical path) and prevents future ENOENT regressions.

#### Data Model Impact
- No new entities.
- Existing: `src/lib/plan.cjs` gains a `tryLoadDAG(cwd)` function.
- Fix `detectCascadeImpact` path from `.planning/DAG.json` to `.planning/sets/DAG.json`.

#### API Surface
- New function: `tryLoadDAG(cwd)` in plan.cjs (internal, not CLI-facing).
- Modified: `getMergeOrder()`, `detectCascadeImpact()` in merge.cjs; `wave-status` and `reconcile` handlers in execute.cjs; `status` handler in worktree.cjs.
- Execute-set SKILL.md Step 4c "no RAPID:RETURN found" logic needs hardening.

### Feature 2: DAG.json Management (Created/Updated by All Relevant Agents)

- **Description:** DAG.json should be created and kept up-to-date by all agents that modify the set structure: init (roadmapper), add-set, plan-set, and any agent that changes set dependencies. Currently only `add-set` (via `recalculateDAG()`) and `plan decompose` (via `decomposeIntoSets()`) write DAG.json. The init/roadmapper flow does not create DAG.json at all.
- **Complexity:** moderate
- **Dependencies:** depends on Feature 1 (bug fixes for DAG.json loading)

#### Decomposition

1. **Init/roadmapper DAG creation** -- After the roadmapper returns contracts and state, the init SKILL.md should call `plan write-dag` to create DAG.json from the set dependency information in the contracts. This is the primary gap.
2. **Plan-set DAG refresh** -- When `/rapid:plan-set` modifies wave structure within a set, the DAG node metadata could be updated to reflect planning status. This is optional but good hygiene.
3. **DAG auto-regeneration on state changes** -- Any CLI command that modifies sets (state transition, add-set) should trigger DAG recalculation. `add-set` already does this. `state transition` does not.
4. **DAG.json creation in start-set** -- If DAG.json doesn't exist when `/rapid:start-set` runs, regenerate it from current state. This is a fallback safety net.

#### Implementation Strategies

**Option 1: Inject DAG write into init SKILL.md + lazy regeneration elsewhere**
- Approach: Add a DAG.json write step to init SKILL.md Step 9 (after roadmap acceptance). For all other agents, add a "lazy regeneration" check: if the DAG.json is missing and STATE.json has sets, auto-regenerate before proceeding. Implement this as a `ensureDAG(cwd)` function.
- Pros: Minimal SKILL.md changes (only init needs modification). Other agents get automatic DAG recovery.
- Cons: Lazy regeneration may mask deeper issues (agent should have created it).
- Libraries/services: None.

**Option 2: Explicit DAG write at every lifecycle boundary**
- Approach: Add `plan write-dag` or `recalculateDAG()` calls to init, start-set, plan-set, and execute-set skills at the points where set structure is finalized.
- Pros: Deterministic -- DAG.json is always current after each lifecycle step.
- Cons: More SKILL.md modifications, redundant regeneration.
- Libraries/services: None.

**Recommended:** Option 1. The init SKILL.md is the only place that truly needs an explicit DAG write (since it creates sets). For everything else, `ensureDAG()` with lazy regeneration handles edge cases without cluttering every skill.

#### Data Model Impact
- No new entities.
- New function: `ensureDAG(cwd)` in plan.cjs -- reads STATE.json + CONTRACT.json files, calls `recalculateDAG()` if DAG.json is missing.

#### API Surface
- New CLI subcommand: `plan ensure-dag` (calls `ensureDAG()`).
- Modified: init SKILL.md Step 9 (add `plan write-dag` call after contract/state writes).
- Modified: merge SKILL.md Step 1b (call `plan ensure-dag` before `merge order`).

### Feature 3: Init Flow Redesign (Structured AskUserQuestion)

- **Description:** Replace the current prose-based Q&A discovery conversation (Step 4B of init) with a structured `AskUserQuestion` flow that asks 4 sub-questions at a time until the agent is satisfied. After collecting enough information, present a summary and ask for confirmation. Also add a granularity preference question (number of sets).
- **Complexity:** complex
- **Dependencies:** independent

#### Decomposition

1. **Replace Step 4B discovery batches with AskUserQuestion calls** -- Convert the 4 batch prompts (Vision/Users, Features/Technical, Scale/Integrations, Context/Success) from freeform text prompts to structured `AskUserQuestion` calls with 4 questions each. Each call uses freeform mode (the user types answers, not selects options).
2. **Adaptive follow-up loop** -- After each batch of 4 questions, analyze responses for completeness. If any area is insufficiently covered, ask targeted follow-up questions (also via AskUserQuestion). Continue until satisfied.
3. **Summary + confirmation step** -- After discovery is complete, compile the project brief and present it to the user. Use AskUserQuestion with options: "Confirm", "Add more details", "Start over".
4. **Granularity preference question** -- Add a new question in Step 4A (logistics) or as the final discovery question: "How finely grained should the project be decomposed?" with options suggesting number of sets (e.g., "3-5 sets (coarse)", "6-10 sets (medium)", "11-20 sets (fine-grained)"). Pass this preference to the roadmapper agent.

#### Implementation Strategies

**Option 1: AskUserQuestion with freeform multi-question batches**
- Approach: Use AskUserQuestion in freeform mode, packing 4 sub-questions into a single prompt. The user answers all 4 in one response. This preserves the batch structure but uses the formal `AskUserQuestion` tool instead of prose conversation.
- Pros: Minimal structural change from current approach. AskUserQuestion provides better UX than raw conversation. 4 questions per call is manageable.
- Cons: AskUserQuestion freeform mode doesn't enforce answering all 4 questions -- user might skip some.
- Libraries/services: AskUserQuestion tool (already available in init SKILL.md).

**Option 2: Individual AskUserQuestion calls per question**
- Approach: Ask each question individually using AskUserQuestion with structured options where possible, freeform where needed. 10+ individual calls.
- Pros: Maximum structure, each question guaranteed to be answered.
- Cons: Very slow UX -- 10+ separate prompts is tedious. Loses the conversational discovery quality.
- Libraries/services: AskUserQuestion tool.

**Recommended:** Option 1. Batching 4 questions per AskUserQuestion call matches the user's stated preference ("4 sub-questions at a time") and keeps the interaction efficient. The adaptive follow-up ensures completeness.

#### Data Model Impact
- Modified: `.planning/config.json` gains a `granularity` field (e.g., `{ "granularity": { "suggestedSets": 8 } }`).
- Modified: The project brief structure gains a `granularity` section.

#### API Surface
- Modified: init SKILL.md Steps 4A and 4B (rewrite discovery flow).
- Modified: Roadmapper agent prompt (receives granularity preference).
- No CLI changes needed.

### Feature 4: Solo Mode Improvements

- **Description:** Solo sets should work on the main branch without worktree creation. The merge agent should detect solo sets and inform the user that merging is not needed. After solo execution completes, the set state should automatically be marked as `merged`.
- **Complexity:** moderate
- **Dependencies:** depends on Feature 1 (bug fixes)

#### Decomposition

1. **Verify solo set works on main branch** -- The current `start-set` SKILL.md and `set-init create --solo` command already handle this: they register the set with `solo: true` and `startCommit` in the registry without creating a worktree or branch. Verify this works correctly end-to-end.
2. **Auto-mark solo set as merged after execution** -- In the execute-set SKILL.md Step 6, after transitioning to `complete`, check if the set is solo. If so, also transition to `merged` immediately. This requires a new state transition path: `complete -> merged` (already exists in `SET_TRANSITIONS`).
3. **Merge agent solo detection and messaging** -- The merge SKILL.md Step 1d already detects solo sets and annotates them. The `mergeSet()` function in merge.cjs already returns `{ merged: true, solo: true }` for solo sets. What's needed: (a) clearer messaging in the merge skill output ("Solo set -- already on main, no merge needed"), (b) skip the merge-tree fast-path check for solo sets, (c) auto-transition state if not already merged.
4. **State consistency check** -- Ensure that after execute-set marks a solo set as `merged`, the merge pipeline correctly identifies it as already merged and skips it with a clear message.

#### Implementation Strategies

**Option 1: Execute-set auto-transitions solo to merged**
- Approach: In execute-set SKILL.md Step 6, after the `complete` transition, check the registry for `solo: true`. If solo, immediately run `state transition set ... merged`. The merge pipeline then sees state=merged and skips. Add a clear message: "Solo set -- automatically marked as merged (no worktree merge needed)."
- Pros: Clean and simple. Solo sets complete the full lifecycle in execute-set. User never needs to run /merge for solo projects.
- Cons: Changes the lifecycle semantics slightly (execute-set now does two transitions for solo sets).
- Libraries/services: None.

**Option 2: Merge agent handles solo transition**
- Approach: Keep execute-set as-is (transitions to `complete`). When /merge runs, it detects solo sets and auto-transitions them to `merged` without any git operations.
- Pros: Lifecycle transitions stay in their canonical skill (merge handles merging).
- Cons: User still needs to run /merge even though nothing happens. Confusing UX.

**Recommended:** Option 1. Auto-transitioning to `merged` in execute-set for solo sets provides the best UX -- the user doesn't need to run a no-op merge command. The merge pipeline should also gracefully handle solo sets that are already merged (which it already does via Step 3a idempotent re-entry).

#### Data Model Impact
- No new entities.
- State transition: `complete -> merged` (already allowed in `SET_TRANSITIONS`).

#### API Surface
- Modified: execute-set SKILL.md Step 6 (add solo check and auto-merge transition).
- Modified: merge SKILL.md Step 1d and Step 3a-solo (improve messaging).
- No new CLI commands needed.

### Feature 5: Branding System (/branding command)

- **Description:** New optional `/branding` command that runs after `/init`. Triggers an in-depth discussion with the user to formalize UI branding guidelines. Produces a `BRANDING.md` file that other agents can reference for consistent UI implementation.
- **Complexity:** complex
- **Dependencies:** independent

#### Decomposition

1. **New SKILL.md for `/rapid:branding`** -- Create `skills/branding/SKILL.md` defining the branding discussion flow. Uses AskUserQuestion to gather: color palette, typography, spacing/layout preferences, component style (rounded vs sharp, shadows, borders), iconography, tone of voice, accessibility requirements.
2. **New agent role: `role-branding.md`** -- Create `src/modules/roles/role-branding.md` for a branding research agent that can analyze existing brand assets, reference websites, and produce structured guidelines.
3. **BRANDING.md output format** -- Define a structured format for `.planning/BRANDING.md` covering: color tokens (primary, secondary, accent, background, text), typography (font families, sizes, weights), spacing scale, border radius conventions, component patterns, dark/light mode considerations.
4. **Agent awareness integration** -- Update the executor agent role (`role-executor.md`) and related agents to check for and reference `BRANDING.md` when implementing UI components. Add branding context loading to `execute.cjs:enrichedPrepareSetContext()`.
5. **Display module integration** -- Optionally update `src/lib/display.cjs` to reference branding colors for RAPID's own banner output (stretch goal).

#### Implementation Strategies

**Option 1: Conversational branding interview + structured output**
- Approach: The /branding skill conducts a multi-round AskUserQuestion interview covering 6-8 branding areas (colors, typography, layout, components, tone, accessibility). After each area, it presents the captured data. At the end, it compiles everything into BRANDING.md and asks for confirmation. Agents read BRANDING.md at execution time.
- Pros: Deep, thorough branding capture. User-controlled. Produces a reference document agents can follow.
- Cons: Can be lengthy for simple projects.
- Libraries/services: AskUserQuestion tool.

**Option 2: Template-based branding with defaults**
- Approach: Offer preset branding templates (minimal, corporate, playful, dark-mode-first) and let the user customize specific aspects. Faster but less personalized.
- Pros: Quick setup. Good defaults for users who don't have strong preferences.
- Cons: Less thorough. May not capture unique brand requirements.
- Libraries/services: AskUserQuestion tool.

**Recommended:** Option 1, with an optional "quick mode" that offers preset templates (from Option 2) as an accelerator. Users who want depth get the full interview; users who want speed pick a template and customize.

#### Data Model Impact
- New file: `.planning/BRANDING.md` (structured branding guidelines).
- Modified: `.planning/config.json` gains `branding: { enabled: true }` flag.
- No database or state machine changes.

#### API Surface
- New skill: `/rapid:branding` (`skills/branding/SKILL.md`).
- New agent: `rapid-branding` (built from `role-branding.md`).
- New CLI subcommand: `display branding-status` -- check if BRANDING.md exists.
- Modified: `execute.cjs:enrichedPrepareSetContext()` -- add branding context.
- Modified: `src/modules/roles/role-executor.md` -- reference BRANDING.md.
- Modified: init SKILL.md Step 11 -- mention `/rapid:branding` as optional next step.

### Feature 6: Review Cycle Confirmation

- **Description:** Bug hunt and unit-test currently run up to 3 cycles automatically. The agents should now prompt the user for confirmation before starting a new cycle (cycle 2+), instead of automatically looping.
- **Complexity:** trivial
- **Dependencies:** independent

#### Decomposition

1. **Bug hunt cycle confirmation** -- In `skills/bug-hunt/SKILL.md` Step 3.9 (after bugfix agent completes), add an AskUserQuestion before incrementing the cycle: "Cycle {N} complete. {accepted} bugs found and fixed. Run another cycle?" with options: "Continue to cycle {N+1}", "Stop here".
2. **Unit-test cycle confirmation** -- The unit-test skill (`skills/unit-test/SKILL.md`) does not have an explicit cycle loop like bug-hunt. However, if there are failed tests, the user might want to re-run. Add a confirmation prompt after Step 5 (execute tests) if there are failures: "Run tests again after fixes?" with options: "Yes", "No, proceed to results".
3. **UAT cycle confirmation** -- Check `skills/uat/SKILL.md` for similar automatic cycling behavior and add confirmation if present.

#### Implementation Strategies

**Option 1: AskUserQuestion between cycles**
- Approach: Add a single AskUserQuestion call between each cycle in the bug-hunt skill. After the bugfix agent returns, show cycle results and ask whether to continue. For unit-test, add a re-run prompt after failures.
- Pros: Simple, explicit user control. Prevents wasted compute on diminishing-returns cycles.
- Cons: Adds one interaction per cycle.
- Libraries/services: AskUserQuestion tool (already available in both skills).

**Recommended:** Option 1. This is the natural approach. The AskUserQuestion call should include cycle statistics (bugs found, bugs fixed, files modified) to help the user decide.

#### Data Model Impact
- No new entities.
- No state changes.

#### API Surface
- Modified: `skills/bug-hunt/SKILL.md` Step 3 loop (add confirmation between cycles).
- Modified: `skills/unit-test/SKILL.md` (add optional re-run confirmation after failures).
- Modified: `skills/uat/SKILL.md` (add confirmation if cycling exists).
- No CLI changes needed.

## Key Technical Details

### DAG.json Canonical Path
The canonical path is `.planning/sets/DAG.json`. The bug in `detectCascadeImpact()` at `src/lib/merge.cjs:1888` uses the wrong path `.planning/DAG.json`. All 8 consumers of DAG.json should go through a centralized `tryLoadDAG()` function.

### Current DAG.json Consumers (files that read DAG.json)
1. `src/commands/execute.cjs` -- `wave-status` and `reconcile` subcommands
2. `src/commands/worktree.cjs` -- `status` subcommand (graceful fallback already)
3. `src/commands/merge.cjs` -- `resolve` subcommand (for ownership/heuristic context)
4. `src/lib/merge.cjs` -- `getMergeOrder()` and `detectCascadeImpact()`
5. `src/lib/add-set.cjs` -- `recalculateDAG()` writes DAG.json
6. `src/lib/plan.cjs` -- `writeDAG()` and `decomposeIntoSets()`

### State Transition Map for Solo Sets
```
pending -> discussed -> planned -> executed -> complete -> merged
```
For solo sets, execute-set should perform: `planned -> executed` (start) + `executed -> complete` + `complete -> merged` (end). The `SET_TRANSITIONS` map already permits `complete -> merged`.

### Init SKILL.md DAG.json Gap
The init SKILL.md (Step 9) writes ROADMAP.md, STATE.json, CONTRACT.json, and DEFINITION.md -- but NOT DAG.json. The roadmapper returns contract data with dependency information (`imports.fromSets`) that contains the edges needed to build a DAG. The fix is to call `recalculateDAG()` (from add-set.cjs) or the equivalent CLI command after writing STATE.json and contracts.

### Branding File Location
`.planning/BRANDING.md` at the planning root, similar to `PROJECT.md` and `ROADMAP.md`. Referenced by agents through the `enrichedPrepareSetContext()` function, which already loads optional context files like quality guidelines and UI contracts.

### Review Cycle Confirmation Insertion Points
- **Bug hunt:** Between Step 3.9 (bugfix agent return) and the cycle increment at the end of Step 3. Insert AskUserQuestion after `modifiedFiles` is updated.
- **Unit test:** After Step 5 (execute tests), if there are failures, add a re-run option. The unit-test skill does not currently have an explicit cycle loop, so this is more of a "retry on failure" prompt.
- **UAT:** Check if Step 3 in `skills/uat/SKILL.md` has automatic cycling; if so, add confirmation there too.
