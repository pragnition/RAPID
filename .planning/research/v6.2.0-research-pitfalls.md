# Pitfalls Research

## Critical Pitfalls

### Pitfall: fs.watch Unreliability for Autoreload Server

- **Category:** correctness
- **Stack:** Node.js `fs.watch` (built-in)
- **Description:** The milestone requires replacing the manual-refresh branding server with autoreload. The natural approach is `fs.watch()` on `.planning/branding/` to detect file changes and push reload events to the browser. However, `fs.watch()` is notoriously unreliable across platforms: it fires duplicate events for single writes, does not consistently report filenames on macOS, has no recursive watching support on Linux until Node v19.1+, and can silently drop events under inode-reuse conditions.
- **Trigger:** Agent writes a file to `.planning/branding/` (e.g., new artifact), `fs.watch` listener fires 0, 1, or 2+ events depending on OS and editor behavior. On macOS, rename events are common even for simple overwrites.
- **Impact:** High -- the autoreload feature is the centerpiece of the webserver overhaul. If the watcher is unreliable, artifacts appear stale in the browser, destroying the interactive branding workflow.
- **Prevention:**
  1. Debounce all `fs.watch` events with a 200-500ms window. Treat any burst of events within the window as a single "files changed" signal.
  2. Do NOT rely on the `filename` argument from the callback; instead, re-scan the branding directory on any event.
  3. Use `{ recursive: true }` only when `process.platform !== 'linux'` or Node >= 19.1. On older Linux, watch individual files or fall back to polling.
  4. Add a manual refresh fallback (e.g., a "Refresh" button in the hub page) so users are never stuck.
- **Detection:** If the browser does not auto-update when a new artifact is written, the watcher is misfiring. Add debug logging (gated behind `RAPID_DEBUG`) that logs every `fs.watch` event to stderr.
- **References:** [Node.js fs.watch documentation](https://nodejs.org/docs/latest/api/fs.html), [chokidar rationale](https://github.com/paulmillr/chokidar)

### Pitfall: SSE/WebSocket Browser Connection Limit (6 per domain on HTTP/1.1)

- **Category:** correctness
- **Stack:** Server-Sent Events (SSE) or WebSocket over `node:http`
- **Description:** The autoreload mechanism will likely use SSE or WebSocket to push change notifications to the browser. On HTTP/1.1, browsers enforce a hard limit of 6 concurrent connections per domain. If the user opens the branding hub in multiple tabs (or other RAPID services also use localhost:3141), SSE connections will saturate the limit, causing the page to hang or fail to load entirely.
- **Trigger:** User opens branding hub in 2+ tabs, or browser dev tools open alongside the page (Chrome DevTools can consume connections).
- **Impact:** High -- the branding page appears to "freeze" and never loads new content. No error is displayed. This is especially confusing because it works fine with 1 tab.
- **Prevention:**
  1. Use a single SSE event stream endpoint and use `event:` types to multiplex different notification types over one connection.
  2. Set `Connection: close` headers on non-SSE responses to free connections quickly.
  3. Consider using a short-polling fallback (e.g., `setInterval` fetching `/_changes?since=<timestamp>` every 2 seconds) instead of SSE, which avoids the connection limit entirely and is simpler to implement for a dev-only tool.
  4. If using SSE, add `retry: 3000` field so the browser reconnects gracefully after disconnection.
- **Detection:** Open the branding hub in 7+ tabs -- the 7th tab will hang. Also check `chrome://net-internals/#sockets` to see connection pool saturation.
- **References:** [MDN SSE documentation](https://developer.mozilla.org/en-US/docs/Web/API/Server-sent_events/Using_server-sent_events), [SSE browser limits](https://www.javascriptroom.com/blog/server-sent-events-and-browser-limits/)

### Pitfall: Path Traversal Bypass in Branding Server Static File Serving

- **Category:** security
- **Stack:** `node:http`, `node:path`
- **Description:** The existing branding server at `src/lib/branding-server.cjs` has a path traversal check on line 278: `if (!resolved.startsWith(brandingDir + path.sep) && resolved !== brandingDir)`. This check has a subtle bypass: if `brandingDir` does not have a trailing separator and the attacker crafts a path that resolves to a sibling directory starting with the same prefix (e.g., `brandingDir` is `/project/.planning/branding` and attacker requests `../branding-secrets/file`), `path.resolve` could resolve to `/project/.planning/branding-secrets/file` which DOES start with `/project/.planning/branding`. The existing code appends `path.sep` to mitigate this, but adding the artifact system with subdirectories or symlinks could reintroduce the vulnerability.
- **Trigger:** An HTTP request with a crafted path segment that resolves to a path sharing the prefix of the branding directory.
- **Impact:** Medium-High -- could expose files outside the branding directory, including STATE.json, config.json, or .env files containing RAPID_TOOLS paths. The server binds to 127.0.0.1 which limits exposure to local-only, but any local process or browser JavaScript could exploit it.
- **Prevention:**
  1. After `path.resolve()`, verify the resolved path starts with `brandingDir + path.sep` OR equals `brandingDir` exactly (the existing check does this correctly).
  2. When adding subdirectory support for artifacts, ensure symlinks are not followed by using `fs.realpathSync()` on the resolved path and re-checking the prefix against the realpath of the branding dir.
  3. Reject paths containing `..` segments before resolution as a defense-in-depth measure.
  4. Reject null bytes in the URL path (`%00`).
- **Detection:** Write a test that requests `/../../../etc/passwd`, `/../.env`, and variations with URL-encoded sequences. Verify 403 responses.
- **References:** [CVE-2025-27210](https://zeropath.com/blog/cve-2025-27210-nodejs-path-traversal-windows), [Node.js path traversal secure coding](https://www.nodejs-security.com/blog/secure-coding-practices-nodejs-path-traversal-vulnerabilities)

## High-Risk Pitfalls

### Pitfall: Stale PID File Leaves Orphaned Server or Blocks Restart

- **Category:** correctness
- **Stack:** `node:http`, `branding-server.cjs` PID file management
- **Description:** The branding server writes a PID file (`.planning/branding/.server.pid`) and checks it on start. If the Node.js process crashes without cleanup (SIGKILL, OOM, power loss), the PID file remains. The existing `_isStale()` check uses `process.kill(pid, 0)` and an HTTP health probe. However, PID reuse is real -- if the OS reassigns the same PID to another process (common on Linux with PID recycling), `_isStale()` returns false, and the server refuses to start, claiming "already_running" when the old server is actually dead.
- **Trigger:** Agent process is killed mid-execution (SIGKILL during branding), then another long-running process reuses the PID. Next branding invocation is blocked.
- **Impact:** Medium-High -- the user cannot start the branding server and gets a confusing "already_running" error pointing to a PID that belongs to an unrelated process.
- **Prevention:**
  1. Store the process start time in the PID file (already done: `startedAt`). On stale check, compare `startedAt` against the actual process start time (readable from `/proc/<pid>/stat` on Linux or `ps -p <pid> -o lstart` cross-platform).
  2. Add a `--force` flag to `start()` that removes the PID file unconditionally.
  3. The health probe is a good defense -- ensure it always runs even when `_isProcessAlive()` returns true.
- **Detection:** Kill the server with `kill -9`, then immediately start a long-running process. Try starting the branding server again.
- **References:** [proper-lockfile stale lock discussion](https://github.com/moxystudio/node-proper-lockfile)

### Pitfall: Zod 3.25.x Bundling Instability with `^` Semver Range

- **Category:** compatibility
- **Stack:** Zod 3.25.76 (`^3.25.76` in package.json)
- **Description:** Zod 3.25.x internally bundles both Zod 3 and Zod 4, doubling package size. Multiple GitHub issues report that `npm install` on a fresh machine can pull a newer 3.25.x patch that breaks CJS loading or introduces TypeScript incompatibilities. Issue #4923 specifically reports that "Latest Zod3 now includes Zod4, which causes breaking changes on minor zod upgrade." The RAPID codebase uses `^3.25.76` which allows automatic patch upgrades within 3.25.x.
- **Trigger:** Running `npm install` in a new worktree, CI environment, or after deleting `node_modules` pulls a newer 3.25.x patch that has CJS loading issues.
- **Impact:** Medium-High -- all state validation breaks, blocking every RAPID command that reads or writes STATE.json. Because Zod validation is on the critical path of every state mutation, this is a silent time bomb.
- **Prevention:**
  1. Pin the exact version: change `"zod": "^3.25.76"` to `"zod": "3.25.76"` in package.json.
  2. Commit `package-lock.json` to ensure reproducible installs.
  3. Add a smoke test in CI that does `rm -rf node_modules && npm install && node -e "require('zod')"` to catch CJS loading regressions.
- **Detection:** If `require('zod')` throws or any Zod schema parse call fails with unexpected errors after a fresh install, Zod has been silently upgraded to a broken patch.
- **References:** [Zod #4434 CJS loading broken](https://github.com/colinhacks/zod/issues/4434), [Zod #4923 breaking changes](https://github.com/colinhacks/zod/issues/4923), [Zod #4372 CJS v4 issue](https://github.com/colinhacks/zod/issues/4372)

### Pitfall: Update Reminder Timestamp Storage Location Conflict

- **Category:** correctness
- **Stack:** RAPID install/config system (.env, config.json, setup.sh)
- **Description:** The "update reminder" feature needs to record an `updated_on` timestamp during `/rapid:install`. The todo.md says "env var or temp file somewhere." The challenge is choosing WHERE to store it. The `.env` file is in the plugin root (shared across all projects). The `.planning/config.json` is per-project. If stored in `.env`, every project shares one timestamp. If stored in per-project config, the reminder only fires in initialized projects. If stored in a temp file (e.g., `/tmp`), it gets wiped on reboot.
- **Trigger:** User runs `/rapid:install`, timestamp is recorded. One week later, the reminder should fire. But if the storage location is wrong, the reminder either never fires, fires on every project, or resets on reboot.
- **Impact:** Medium -- the feature silently fails (no reminder ever shown) or becomes annoying (shown on every command in every project). Neither outcome is correct.
- **Prevention:**
  1. Store `updated_on` in the plugin-root `.env` file (next to `RAPID_TOOLS=...`). This is the natural location because install is a plugin-level operation, not a project-level one.
  2. Read it from `.env` using the existing `_parseEnvFile()` pattern in `web-client.cjs`.
  3. Check the timestamp in a lightweight function called early in the CLI command router (not in every library module). Display the reminder to stderr so it does not interfere with JSON output.
  4. Use ISO 8601 date format (`YYYY-MM-DDTHH:mm:ssZ`) for the timestamp to avoid timezone parsing issues.
- **Detection:** Set `updated_on` to a date 8 days in the past, run any `/rapid:*` command, and verify the update reminder appears.

### Pitfall: Branding Init Flow Integration Creates Circular Dependency

- **Category:** correctness
- **Stack:** RAPID skill orchestration (`skills/init/SKILL.md`, `skills/branding/SKILL.md`)
- **Description:** The milestone requires inserting branding into the init flow at step 4B (after areas, before granularity). However, the branding skill currently expects a project that has already been scaffolded (it reads PROJECT.md, ROADMAP.md for context in Step 2, and requires `.planning/branding/` to exist). During init step 4B, scaffolding has not happened yet (that is Step 5 or later in the init flow). If the branding interview is naively inserted at 4B, it will fail to find project artifacts and may write to a directory that does not yet exist.
- **Trigger:** User runs `/rapid:init` for a fresh project. Init reaches step 4B, invokes branding. Branding tries to read `.planning/branding/BRANDING.md` or create `.planning/branding/`, but `.planning/` does not exist yet.
- **Impact:** Medium-High -- the init flow crashes or branding silently skips its codebase-aware detection, producing a generic/inferior branding output.
- **Prevention:**
  1. The branding integration in init must be a SUBSET of the full branding skill -- specifically just the interview rounds (Steps 4-5 of the branding SKILL.md). Skip Steps 1-3 (env setup, codebase detection, existing check) and Step 6 (server start).
  2. Create `.planning/branding/` directory early in the init flow if the user opts into branding.
  3. Pass the project type detected during init discovery (from Area 5/6 of step 4B) directly to the branding interview instead of re-detecting it.
  4. Do NOT start the branding server during init -- save that for the standalone `/rapid:branding` invocation.
- **Detection:** Run `/rapid:init` in a completely empty directory. Verify the flow completes without errors and produces a valid BRANDING.md.

## Medium-Risk Pitfalls

### Pitfall: HTTP Server Memory Leak from Unclosed SSE/WebSocket Connections

- **Category:** performance
- **Stack:** `node:http` server, SSE connections
- **Description:** When adding autoreload via SSE, each browser connection holds an open HTTP response object. If the browser disconnects without proper close (tab close, network drop), the server-side response object and its associated `fs.watch` listener may not be cleaned up, leading to gradual memory growth and file descriptor exhaustion.
- **Trigger:** User opens and closes branding tabs repeatedly, or leaves tabs open overnight. Each uncleaned connection leaks the response object and any associated event listeners.
- **Impact:** Medium -- the server slowly consumes memory. For a dev-only tool this is tolerable short-term, but if left running for days it can impact the system.
- **Prevention:**
  1. Listen for the `close` event on the response object to detect client disconnection: `res.on('close', () => { /* cleanup watcher, remove from connections set */ })`.
  2. Maintain a `Set` of active SSE connections. On `fs.watch` event, iterate the set and write to each. On `close`, remove from set.
  3. Set `server.keepAliveTimeout` and `server.headersTimeout` to reasonable values (30s, 60s respectively).
  4. Add a connection count to the `/_health` endpoint for monitoring.
- **Detection:** Open 50 tabs to the branding hub, close them all, check `process.memoryUsage()` via the health endpoint.
- **References:** [Node.js http memory leak issue #48604](https://github.com/nodejs/node/issues/48604), [CVE-2025-23085 HTTP/2 memory leak](https://vulert.com/vuln-db/CVE-2025-23085)

### Pitfall: Artifact System File Naming Collisions

- **Category:** data-loss
- **Stack:** Branding artifact system (new feature)
- **Description:** The new artifact system allows the model to create artifacts (logo, theme, wireframe, branding guidelines) in `.planning/branding/`. If two artifacts are generated with the same filename (e.g., the model generates `logo.svg` twice with different content), the second write silently overwrites the first. There is no versioning or conflict detection.
- **Trigger:** Model generates a logo, user asks for revisions, model writes a new logo with the same filename.
- **Impact:** Medium -- previous artifact version is lost with no undo. For creative assets like logos this is particularly painful since the user might prefer the earlier version.
- **Prevention:**
  1. Include a timestamp or sequence number in artifact filenames: `logo-001.svg`, `logo-002.svg`.
  2. Before writing an artifact, check if the file exists. If it does, append a numeric suffix.
  3. Keep a manifest file (e.g., `artifacts.json`) that tracks all generated artifacts with metadata (type, timestamp, description).
- **Detection:** Generate the same artifact type twice and verify both versions are preserved.

### Pitfall: Branding Server Port Conflict with Web Backend

- **Category:** correctness
- **Stack:** `branding-server.cjs` (port 3141), `web-client.cjs` (port 8998)
- **Description:** The branding server defaults to port 3141 and the web backend to 8998, so they do not conflict directly. However, the EADDRINUSE handling in `start()` returns `{ error: 'port_in_use', port }` but the SKILL.md only handles this with a user prompt for alternative ports. If the branding server is integrated into an automated flow (init), there is no user present to answer the prompt, and the flow will stall or fail.
- **Trigger:** Another process (dev server, another RAPID instance) occupies port 3141 during an automated init flow that includes branding.
- **Impact:** Medium -- the init flow hangs waiting for user input that may never come in an automated context.
- **Prevention:**
  1. Implement auto-port-increment: if 3141 is in use, try 3142, 3143, etc., up to 3150 before giving up.
  2. Store the actual port used in the PID file (already done) and in the server status output.
  3. For the init-flow integration specifically, make server start optional/skippable with no user prompt.
- **Detection:** Start a dummy server on port 3141, then run the branding server start. Verify it finds an alternative port automatically.

### Pitfall: execSync Usage in worktree.cjs Deferred Migration Risk

- **Category:** security
- **Stack:** `child_process.execSync` in `src/lib/worktree.cjs`
- **Description:** The deferred decision list includes "execSync to execFileSync migration." The existing code at `worktree.cjs:113` uses `execSync(pm.command, ...)` for package manager commands and at line 417 uses `execSync` with a dynamically constructed command string. While `execFileSync` is used for `gitExec`, the remaining `execSync` calls with string interpolation are vulnerable to command injection if any input contains shell metacharacters.
- **Trigger:** A set name, worktree path, or package manager command containing shell metacharacters (`;`, `|`, `$()`, backticks) is passed to `execSync`.
- **Impact:** Medium -- command injection. The attack surface is limited because set names are typically controlled by the user/agent, but malformed project paths or package manager outputs could trigger it.
- **Prevention:**
  1. While the full migration is deferred, at minimum validate that set names and worktree paths do not contain shell metacharacters before passing to `execSync`.
  2. For the new features in this milestone, ensure ALL new shell invocations use `execFileSync` (array form) rather than `execSync` (string form).
  3. Add a linting rule or test that greps for new `execSync` calls in changed files.
- **Detection:** The existing test at `worktree.test.cjs:1835` checks that `gitExec` does not use `execSync` with template strings. Extend this test to cover all production modules.

## Anti-Patterns for This Stack

### Anti-Pattern: Adding npm Dependencies for Dev-Only Features

- **What people do:** Add `chokidar`, `ws`, `livereload`, or `socket.io` as dependencies to implement file watching and autoreload.
- **Why it fails:** RAPID has exactly 4 production dependencies by design. Adding dependencies for a dev-only branding preview tool inflates the install footprint for every user, introduces supply chain risk, and violates the project's minimalist dependency philosophy.
- **Correct approach:** Use `fs.watch` with debouncing for file watching and either SSE or short-polling for browser reload. Both are zero-dependency with `node:http` and `node:fs`. Accept the `fs.watch` quirks and compensate with debouncing and fallbacks.
- **Example:** Instead of `chokidar.watch(dir).on('change', handler)`, use `fs.watch(dir, { recursive: true }, debounce(handler, 300))`.

### Anti-Pattern: Storing Installation Metadata in STATE.json

- **What people do:** Add `updated_on`, `installedAt`, or version check fields to the project STATE.json schema.
- **Why it fails:** STATE.json is per-project and validated by Zod schemas on every read/write. Adding install-time metadata to it (a) requires a schema migration for all existing projects, (b) conflates project state with plugin state, and (c) means the reminder only works in initialized projects.
- **Correct approach:** Store plugin-level metadata in the plugin-root `.env` file (alongside `RAPID_TOOLS=...`). The `.env` file is already loaded by every skill preamble and by `web-client.cjs`. Adding `UPDATED_ON=2026-04-07T00:00:00Z` requires no schema changes and no migration.

### Anti-Pattern: Starting the Branding Server During Init

- **What people do:** Because branding is being integrated into init, start the branding server as part of the init flow so the user can see the preview immediately.
- **Why it fails:** Init is a long, multi-agent pipeline. Starting a server mid-flow (a) creates a background process that outlives the init conversation, (b) adds complexity to error handling (what if the server fails?), and (c) requires the init skill to handle server lifecycle management that belongs to the branding skill.
- **Correct approach:** During init, only run the branding interview and write BRANDING.md + index.html. Display a message: "Run `/rapid:branding` to preview your branding guidelines in the browser." The server lifecycle remains the responsibility of the standalone branding skill.

### Anti-Pattern: Using Module-Level `_activeServer` Singleton for Multi-Instance Safety

- **What people do:** Store the HTTP server reference in a module-level variable (as `branding-server.cjs` currently does with `_activeServer`).
- **Why it fails:** If multiple RAPID CLI invocations happen in the same Node.js process (e.g., tests, or a future watch mode), the singleton is shared and can be overwritten, causing the first server to become untrackable. The `stop()` function only closes `_activeServer`, so if it has been reassigned, the previous server leaks.
- **Correct approach:** Return the server object from `start()` (already done) and let the caller manage its lifecycle. The PID file is the durable record of server state. When adding autoreload watchers, attach them to the server's close event for cleanup rather than relying on module-level state.

## Version-Specific Gotchas

- **Node.js 22.x:** `fs.watch({ recursive: true })` is supported on macOS and Windows but still experimental on Linux. If targeting Linux users (likely, given this is a dev tool), do not rely on recursive watching.
- **Node.js 22.x:** `AbortSignal.timeout()` is stable and used in `web-client.cjs`. Safe to use for new HTTP client code.
- **Zod 3.25.76:** `.passthrough()` is used on all state schemas. This means unknown fields are preserved through validation -- good for forward compatibility but means typos in field names (e.g., `updatedOn` vs `updated_on`) will silently pass validation.
- **proper-lockfile 4.1.2:** The `stale` option defaults to 0 (never stale). RAPID configures `lock_timeout_ms: 300000` in config.json. Ensure the branding server and any new features respect this timeout, especially if they acquire locks for writing artifacts.

## Security Pitfalls

1. **Path traversal in artifact serving:** When the artifact system adds subdirectory support, every new path resolution point is a potential traversal vector. Use `path.resolve()` + prefix check + `fs.realpathSync()` for symlink defense.
2. **PID file race condition:** Between reading the PID file and checking process liveness, another process could start with the same PID. Use file locking (proper-lockfile) around PID file read-check-write sequences.
3. **Unescaped HTML in hub page:** The `_generateHubPage()` function inserts filenames directly into HTML (`<a href="/${f}">${f}</a>`). If an artifact filename contains `<script>` or HTML entities, this is an XSS vector (local-only, but still a correctness issue). HTML-escape filenames before insertion.
4. **Environment variable injection via .env:** The `.env` file is parsed with a simple split-on-`=` parser. If the `updated_on` value contains newlines or quotes, it could inject additional environment variables. Use strict parsing that rejects values with newlines.

## Performance Traps

1. **fs.watch + full directory re-scan on every event:** If the branding directory grows large (many artifacts), re-scanning on every `fs.watch` event becomes expensive. Use a timestamp-based change detection (compare directory mtime) rather than re-listing all files.
2. **Synchronous file I/O in HTTP request handler:** The current `_handleRequest` uses `fs.statSync()` and `fs.createReadStream()`. The stat is synchronous and blocks the event loop. For a single-user dev tool this is acceptable, but if autoreload adds concurrent connections, consider `fs.stat()` (async) or caching stat results.
3. **Hub page regeneration on every request:** `_generateHubPage()` re-reads the directory and builds HTML on every `/` request. If autoreload causes frequent polling, cache the generated HTML and invalidate on `fs.watch` events.

## Summary Priority List

1. **fs.watch unreliability for autoreload** -- the core feature of the webserver overhaul depends on a notoriously flaky API; debouncing and fallbacks are essential
2. **SSE browser connection limit** -- invisible failure mode that makes the autoreload appear broken with multiple tabs; short-polling is simpler and avoids the issue entirely
3. **Branding init flow integration ordering** -- inserting branding at step 4B before scaffolding creates a dependency on non-existent directories; requires careful init flow surgery
4. **Path traversal in artifact serving** -- expanding the server to serve subdirectories/artifacts multiplies the attack surface; existing defense is correct but fragile
5. **Zod 3.25.x CJS instability** -- `^` semver range allows automatic upgrades to broken patches; pin exact version
6. **Stale PID file with PID reuse** -- blocks server restart with a confusing "already_running" error; add process start-time comparison
7. **Update reminder storage location** -- wrong choice silently breaks the feature or makes it annoying; store in plugin-root .env
8. **Artifact filename collisions** -- silent data loss when regenerating artifacts; use sequence numbers
9. **execSync command injection surface** -- deferred migration means new features must be extra careful to use execFileSync
10. **XSS via unescaped filenames in hub page** -- local-only but still a correctness bug; HTML-escape all dynamic content
