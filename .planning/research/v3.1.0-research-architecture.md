# Architecture Research: v3.1.0 Polish & Cleanup

## Current Architecture

### Overview

RAPID is a Claude Code plugin built as a monolithic CommonJS Node.js application. The architecture follows a layered library pattern where:

- **CLI entry point** (`src/bin/rapid-tools.cjs`) is a single dispatcher routing to library functions
- **Library modules** (`src/lib/*.cjs`) contain all business logic, organized by domain
- **Skills** (`skills/*/SKILL.md`) are Markdown-based instruction files consumed by Claude Code agents
- **Agents** (`agents/*.md`) are generated from source modules (`src/modules/`) via `build-agents`
- **State** lives in `.planning/STATE.json`, validated by Zod schemas, and mutated through lock-protected transactions

### Module Dependency Graph (relevant to this milestone)

```
state-schemas.cjs    <-- SetStatus enum: ['pending','discussing','planning','executing','complete','merged']
       |
state-transitions.cjs <-- SET_TRANSITIONS map: pending->discussing/planning, discussing->planning, etc.
       |
state-machine.cjs    <-- readState, writeState, withStateTransaction, transitionSet, validateDiskArtifacts
       |
       +-- lock.cjs (proper-lockfile based, mkdir atomic locking)
       |
       +-- Used by: rapid-tools.cjs, execute.cjs, worktree.cjs, plan.cjs, review.cjs, merge.cjs
```

### State Machine Design

The current state machine uses a simple enum + transition map pattern:

1. **Schema** (`state-schemas.cjs`): Zod enum defines valid statuses
2. **Transitions** (`state-transitions.cjs`): Plain object maps `current -> [allowed_next]`
3. **Enforcement** (`state-machine.cjs`): `validateTransition()` checks the map before mutation
4. **Lock protection**: All state writes go through `withStateTransaction()` which acquires a lock, reads, mutates, validates, writes atomically (tmp+rename), and releases

This pattern is clean and works well. The status values permeate through:
- Zod enum validation (schema layer)
- Transition map keys/values (transition layer)
- String comparisons in skills (instruction layer)
- Display formatting in worktree.cjs (UI layer)
- Disk artifact validation paths in state-machine.cjs (validation layer)

### Skill Architecture

Skills are Markdown files with YAML frontmatter that Claude Code interprets as instructions. They reference status values as string literals in inline bash scripts and prose instructions. Key pattern:

```bash
node "${RAPID_TOOLS}" state transition set "${MILESTONE_ID}" "${SET_ID}" discussing
```

Skills check status with conditional logic in natural language ("If `discussing`: this is expected. Continue.").

### Deprecated Command Stubs

Six skills serve as deprecation redirects:
- `skills/set-init/` -> `/rapid:start-set`
- `skills/discuss/` -> `/rapid:discuss-set`
- `skills/execute/` -> `/rapid:execute-set`
- `skills/plan/` -> `/rapid:plan-set`
- `skills/wave-plan/` -> `/rapid:plan-set`
- `skills/new-milestone/` -> `/rapid:new-version`

Each is a minimal SKILL.md with `disable-model-invocation: true` and a one-line redirect message. They add noise to the skill registry but have zero runtime cost since `disable-model-invocation` prevents agent spawning.

### Wave Execution Architecture

Current execution is strictly sequential in `skills/execute-set/SKILL.md`:
- Step 4 iterates waves in order (1, 2, 3...)
- Each wave spawns one `rapid-executor` agent
- Completion is tracked via `WAVE-N-COMPLETE.md` marker files
- Re-entry detection skips completed waves

The DAG module (`dag.cjs`) already supports wave assignment via BFS level grouping (`assignWaves()`), and `getExecutionOrder()` returns waves as parallel groups. However, this is currently used only at the **set level** (sets in the same wave can run in parallel via worktrees), not at the **wave level within a set**.

### Install Architecture

Install is split between:
- `setup.sh`: Non-interactive bootstrap (prereqs, npm install, validate, .env, plugin registration, build-agents)
- `skills/install/SKILL.md`: Interactive wrapper (shell detection, config file selection, verification)
- `commands/install.md`: Simpler non-interactive variant

Current install checks for `RAPID_TOOLS` presence only. There is no version comparison between the installed binary and the current plugin version (`package.json` version `3.0.0`).

## Recommended Architecture Patterns Per Goal

### Goal 1: Set Status Naming Fix (discussing -> discussed, planning -> planned, executing -> executed)

**Pattern: Coordinated Cross-Cutting Rename with Schema-First Migration**

**Rationale:** This is a data model change that cascades through every layer. The safest approach is schema-first: change the canonical definitions first (Zod enum, transition map), then update all consumers. Since RAPID is a single codebase with no external API contracts or persisted state files from other systems, this can be done as a direct rename without migration logic.

**Architecture Decision: No State Migration Needed**

`STATE.json` files in existing projects will have old status values. However, the Zod schema uses `.parse()` which will reject unknown enum values. Two options:

1. **Hard break (recommended):** Change the enum. Old STATE.json files fail validation. Users re-run `/rapid:init` for new milestones. Since sets within a milestone are transient (they complete and merge), any in-progress milestone can be manually updated by editing STATE.json. This is acceptable for a minor version bump.

2. **Soft migration:** Add a migration function in `readState()` that maps old values to new ones transparently. This adds complexity for a one-time transition.

Recommendation: Option 1 (hard break) with a one-line migration note in release notes. The project is pre-1.0-adoption, and the rename is the entire point of this change.

**Impacted Layers (ordered by change sequence):**

```
1. state-schemas.cjs         -- Change Zod enum values
2. state-transitions.cjs     -- Change transition map keys/values
3. state-machine.cjs         -- Update validateDiskArtifacts status checks
4. state-schemas.test.cjs    -- Update valid statuses array
5. state-transitions.test.cjs -- Update transition test cases
6. state-machine.lifecycle.test.cjs -- Update lifecycle test chain
7. state-machine.test.cjs    -- Any inline status references
8. worktree.cjs              -- formatWaveSummary phase strings, suggestActions status cases
9. display.cjs               -- STAGE_VERBS and STAGE_BG (these are stage names, not statuses, but verify)
10. execute.cjs              -- validateDiskArtifacts status array
11. plan.cjs                 -- Any status references
12. rapid-tools.cjs          -- 'pending' reference in list-available
13. skills/discuss-set/       -- 'discussing' in transition, status checks
14. skills/plan-set/          -- 'discussing' expected, 'planning' transition
15. skills/execute-set/       -- 'planning'/'executing' checks and transitions
16. skills/start-set/         -- Status checks
17. skills/status/            -- Status display
18. skills/review/            -- Status checks
19. skills/merge/             -- Status checks
20. agents/ (26 files)        -- Generated from src/modules/, so update modules instead
21. src/modules/ (roles)      -- Status string references
22. All test files             -- Status string assertions
```

**Key Risk: Ambiguous string matches.** The word "planning" appears in non-status contexts (e.g., ".planning/" directory, "planning phase" in prose, "plan-set" stage names). The rename to "planned" eliminates this ambiguity -- "planned" is clearly a past-tense status, not a directory name or activity. Similarly, "discussed" and "executed" are unambiguous. This actually makes the codebase cleaner.

**Testing Strategy:** After the rename, run `grep -rn "discussing\|planning\|executing" src/ skills/ agents/` and manually verify each remaining match is a non-status usage (directory paths, prose descriptions, stage names).

### Goal 2: Parallel Wave Execution

**Pattern: Dependency-Aware Parallel Dispatch with Serialized Commit Gate**

**Rationale:** The infrastructure for parallel execution already exists at the set level (worktrees + DAG). Within a set, waves execute in a single worktree, which creates a serialization constraint on git operations. The architecture should enable concurrent agent execution with coordinated commit sequencing.

**Architecture Overview:**

```
execute-set SKILL.md
  |
  +-- Step 4: Wave Execution (currently sequential)
  |     |
  |     +-- CURRENT: for each wave N: spawn executor, wait, next
  |     |
  |     +-- PROPOSED: analyze wave dependency graph, identify independent groups,
  |           spawn independent waves concurrently via parallel Agent tool calls,
  |           gate subsequent waves on all predecessors completing
  |
  +-- Commit Serialization
        |
        +-- Option A: File-system lock per worktree (lock.cjs already supports this)
        +-- Option B: Each executor commits freely (git handles concurrent commits)
        +-- Option C: Wave-level branches + sequential fast-forward merge
```

**Recommended Approach: Simplified Parallel Dispatch (Option B with guardrails)**

The key insight from `todo.md` is: "the payoff depends on how often plans produce independent waves -- if most waves are sequential by nature (wave 2 builds on wave 1's output), the gain is minimal."

Given this, the architecture should be:

1. **Wave dependency analysis:** At the start of execution, analyze wave plans to determine which waves are independent. Independence criteria:
   - No file overlap between waves (check "Files to Create/Modify" sections)
   - No explicit dependency declared (wave N references output of wave M)
   - Waves at the same DAG level are candidates for parallelism

2. **Parallel dispatch in SKILL.md:** Use the Claude Code Agent tool's built-in parallelism. Multiple Agent tool calls in the same step execute concurrently. This is the simplest approach -- no new infrastructure needed.

3. **Git commit serialization:** Since each wave modifies disjoint files, concurrent `git add` + `git commit` operations in the same worktree should not conflict. The worktree's filesystem provides natural serialization for file writes. However, git's index is a shared resource. Two mitigations:
   - **Preferred:** Use `git add <specific files>` (already enforced by convention) + sequential commit via the existing `state` lock
   - **Fallback:** Each executor commits immediately; if two race, one will get a "nothing to commit" or merge conflict -- detectable and recoverable

4. **Error handling for partial completion:** If wave A succeeds but wave B fails in a parallel group:
   - Wave A's `WAVE-A-COMPLETE.md` marker is written
   - Wave B has no marker
   - Re-entry detection (Step 2 of execute-set) naturally handles this: it skips complete waves and re-executes incomplete ones
   - The existing re-entry mechanism requires zero changes

**Module Changes Required:**

```
skills/execute-set/SKILL.md  -- Primary change: parallel dispatch logic in Step 4
execute.cjs                  -- Optional: add wave-independence detection helper
dag.cjs                      -- No changes needed (already supports wave grouping)
lock.cjs                     -- No changes needed (can be used for commit serialization)
```

**Scope Consideration:** The skill-level change is the most impactful. The execute-set SKILL.md Step 4 needs to:
1. Read all wave PLAN.md files
2. Determine which waves can run in parallel (file ownership disjointness check)
3. For parallel-eligible waves: spawn multiple Agent calls in one step
4. For sequential-dependent waves: preserve current behavior
5. After each parallel group completes: verify all markers, then proceed to next group

**Trade-off: Skill Complexity vs. Library Code**

Two implementation strategies:
- **Strategy A (recommended):** Keep the parallelism logic entirely in SKILL.md. The skill already orchestrates sequential waves; adding parallel dispatch is a natural extension. No new library code needed.
- **Strategy B:** Add a `determineWaveParallelism(cwd, setId)` function to `execute.cjs` that reads wave plans and returns parallel groups. The skill calls this function and dispatches accordingly. This is cleaner but adds library surface area.

Recommendation: Strategy A for v3.1.0 (minimal changes, prove the concept), consider Strategy B for future versions if parallel wave execution proves valuable.

### Goal 4: Install Command Flexibility (Version-Aware Install)

**Pattern: Version Comparison with Conditional Reinstall**

**Rationale:** The install skill currently checks only for `RAPID_TOOLS` presence (is the env var set?). It needs to also check version currency (does the installed version match the current plugin version?).

**Architecture Design:**

```
package.json               -- Already has version: "3.0.0" (will be "3.1.0" for this release)
prereqs.cjs                -- Already has compareVersions() utility
setup.sh                   -- Needs version stamping + version check
skills/install/SKILL.md    -- Needs version comparison step
commands/install.md         -- Needs version comparison step
```

**Version Detection Flow:**

```
1. Read current plugin version from package.json (source of truth)
2. Check if RAPID_TOOLS env var is set
3. If set, check if RAPID_TOOLS path exists and is executable
4. If executable, extract installed version:
   - Option A: Run `node "${RAPID_TOOLS}" --version` (add --version flag to CLI)
   - Option B: Read package.json from the RAPID_TOOLS parent directory
   - Option C: Read a VERSION file stamped during setup
5. Compare installed version vs current version
6. If mismatch: reinstall (re-run setup.sh, re-write .env, re-register plugin)
7. If match: skip reinstall, verify prereqs only
```

**Recommended: Option A (--version flag)**

Adding `--version` to the CLI is the most robust approach:
- No path manipulation needed
- Works regardless of where RAPID is installed
- Follows standard CLI conventions

Implementation:
1. Add `--version` handler to `rapid-tools.cjs` that reads and outputs `package.json` version
2. In `setup.sh`, stamp the version into the `.env` file as `RAPID_VERSION=X.Y.Z`
3. In `skills/install/SKILL.md`, add a Step 0.5 that compares versions before deciding whether to run full setup

**Version Comparison Logic in Install Skill:**

```
Step 0.5: Check Existing Installation
  1. Check RAPID_TOOLS env var
  2. If not set: fresh install, run full setup
  3. If set but path doesn't exist: stale config, run full setup
  4. If set and path exists:
     a. Run: node "${RAPID_TOOLS}" --version
     b. Read current version from ${RAPID_ROOT}/package.json
     c. If versions match: skip to verification (Step 4)
     d. If installed < current: "Outdated installation detected (installed: X, current: Y). Updating..."
        Re-run setup.sh to update paths and rebuild agents
     e. If installed > current: "Installed version is newer than plugin. This may indicate a mixed installation."
        Warn and proceed with verification
```

**Edge Cases:**
- `RAPID_TOOLS` points to a deleted file: `node "${RAPID_TOOLS}"` will fail; treat as fresh install
- `RAPID_TOOLS` points to a different RAPID installation: version comparison catches this; reinstall updates to current
- `--version` flag not recognized by old CLI: the command will fail with usage error; treat as outdated

## Module Organization

### Module: state-schemas

- **Responsibility:** Define the canonical set status values as a Zod enum; define the shape of SetState, MilestoneState, and ProjectState
- **Public interface:** `SetStatus`, `SetState`, `MilestoneState`, `ProjectState` (Zod schemas)
- **Dependencies:** `zod`
- **Data ownership:** The status enum is the single source of truth for valid status values

### Module: state-transitions

- **Responsibility:** Define which status transitions are valid; provide validation function
- **Public interface:** `SET_TRANSITIONS` (map), `validateTransition(current, next)`
- **Dependencies:** None (pure logic)
- **Data ownership:** Owns the transition graph

### Module: state-machine

- **Responsibility:** CRUD operations on STATE.json with lock protection and schema validation
- **Public interface:** `readState`, `writeState`, `withStateTransaction`, `transitionSet`, `findSet`, `findMilestone`, `validateDiskArtifacts`, `commitState`, etc.
- **Dependencies:** `state-schemas`, `state-transitions`, `lock`
- **Data ownership:** Owns `.planning/STATE.json` read/write lifecycle

### Module: execute

- **Responsibility:** Execution context preparation, prompt assembly, post-execution verification, wave reconciliation, handoff generation
- **Public interface:** `prepareSetContext`, `assembleExecutorPrompt`, `verifySetExecution`, `reconcileWave`, `generateHandoff`, etc.
- **Dependencies:** `worktree`, `plan`, `verify`, `contract`
- **Data ownership:** Owns wave reconciliation results and handoff files

### Module: display

- **Responsibility:** ANSI banner rendering for CLI stages
- **Public interface:** `renderBanner`, `STAGE_VERBS`, `STAGE_BG`
- **Dependencies:** None
- **Data ownership:** Stage-to-verb and stage-to-color mappings

### Module: prereqs

- **Responsibility:** Validate system prerequisites (git, Node.js, jq) with version comparison
- **Public interface:** `validatePrereqs`, `compareVersions`, `checkGitRepo`, `formatPrereqSummary`
- **Dependencies:** `child_process`
- **Data ownership:** None (read-only checks)

### Module: lock

- **Responsibility:** Atomic lock acquisition using proper-lockfile (mkdir-based locking)
- **Public interface:** `acquireLock`, `isLocked`, `cleanStaleLocks`
- **Dependencies:** `proper-lockfile`
- **Data ownership:** Owns `.planning/.locks/` directory

## Data Flow

### Status Rename Data Flow

```
1. User runs skill (e.g., /rapid:discuss-set)
2. Skill runs: node "${RAPID_TOOLS}" state transition set <ms> <set> "discussed"
3. rapid-tools.cjs dispatches to transitionSet()
4. transitionSet() calls withStateTransaction():
   a. acquireLock('state')
   b. readState() -> Zod parse with SetStatus enum
   c. validateTransition(current, "discussed") -> checks SET_TRANSITIONS map
   d. set.status = "discussed"
   e. ProjectState.parse(state) -> validates new status against Zod enum
   f. atomic write (tmp + rename)
   g. release lock
5. Skill continues with downstream logic
```

The rename propagates through this chain: Zod enum -> transition map -> validation -> all string comparisons in skills and library code.

### Parallel Wave Execution Data Flow

```
1. execute-set SKILL.md Step 4 (proposed):
   a. Read all wave-N-PLAN.md files
   b. Analyze file ownership across waves to identify independent groups
   c. Group waves into parallel batches:
      - Batch 1: waves with no dependencies (typically wave 1 only)
      - Batch 2: waves depending only on Batch 1 outputs
      - etc.
   d. For each batch:
      - Spawn all waves in batch as parallel Agent tool calls
      - Wait for all to complete (Agent tool handles this)
      - Check RAPID:RETURN from each
      - Write WAVE-N-COMPLETE.md markers for successes
      - If any BLOCKED/CHECKPOINT: handle per existing logic
   e. Proceed to next batch only when all current batch waves complete
```

### Install Version Check Data Flow

```
1. User runs /rapid:install
2. Skill checks RAPID_TOOLS env var
3. If set: run node "${RAPID_TOOLS}" --version
4. Compare with current package.json version
5. If mismatch: re-run setup.sh (which re-writes .env, re-registers plugin, rebuilds agents)
6. If match: skip to verification
```

### State Management

- **Where state lives:** `.planning/STATE.json` (file system, git-tracked)
- **How state is read:** `readState()` -> JSON parse -> Zod schema validation
- **How state is written:** `writeState()` / `withStateTransaction()` -> Zod validation -> lock acquire -> atomic tmp+rename -> lock release
- **Consistency guarantees:** Lock-protected single-writer; atomic rename prevents partial writes; Zod validation prevents schema violations

## Scaling Considerations

Not applicable for this milestone. RAPID operates as a single-user development tool per project instance. The parallel wave execution feature adds concurrency within a single worktree but does not change the scaling profile.

**Concurrency concern for parallel waves:** Multiple executor agents writing to the same git worktree. Git's index (`/.git/index`) is a shared resource. Concurrent `git add` operations can corrupt the index. Mitigation: use the existing lock system to serialize git operations, or ensure file-disjoint waves so that `git add <specific-files>` never targets the same index entries.

## Architecture Risks

1. **Status rename string matching ambiguity (Medium):** The word "planning" appears in directory paths (`.planning/`), prose ("planning phase"), and stage names (`plan-set`). A naive find-and-replace could break non-status usages. **Mitigation:** Rename to "planned" (past tense) which has no collisions with existing non-status strings. Use targeted replacement per file rather than global sed.

2. **Git index corruption during parallel wave execution (High):** Two concurrent `git add` + `git commit` sequences in the same worktree can corrupt the git index file. **Mitigation:** Serialize git operations using the existing lock system. Each executor completes its file writes freely but acquires a lock before `git add` + `git commit`. Alternative: only parallelize waves with strictly disjoint file sets and rely on `git add <specific-files>` not touching overlapping index entries.

3. **Old STATE.json files with deprecated status values (Low):** Existing projects with in-progress milestones will have `discussing`/`planning`/`executing` in their STATE.json. After upgrade, Zod validation will reject these. **Mitigation:** Document in release notes. Optionally add a one-time migration utility or a `readState()` shim that maps old values.

4. **Install version detection for old CLI without --version flag (Low):** If `RAPID_TOOLS` points to a pre-3.1.0 CLI that lacks the `--version` flag, the version check command will fail. **Mitigation:** Treat command failure as "version unknown, assume outdated, proceed with reinstall." This is the correct behavior.

5. **Skill instruction complexity for parallel dispatch (Medium):** Adding parallel dispatch logic to execute-set SKILL.md increases the skill's complexity. Skills are interpreted by LLMs, not executed as code -- more complex instructions increase the chance of misinterpretation. **Mitigation:** Keep the parallel logic simple (analyze file sets, group waves, spawn in parallel). Add explicit anti-patterns to prevent common mistakes.

## Recommendations

Ordered by implementation priority:

1. **Status rename: Schema-first, test-second, consumer-last (Critical):**
   Change `state-schemas.cjs` and `state-transitions.cjs` first. Run tests to get a clear list of all failures. Fix test files next. Then update library consumers (state-machine, execute, worktree, display). Finally update skills and modules. Regenerate agents via build-agents as the last step. This ordering ensures the canonical source of truth changes first and broken tests guide the consumer updates.

2. **Old command cleanup: Direct deletion (High):**
   Delete the six deprecated skill directories. Update DOCS.md and display.cjs. No migration or compatibility concern. This is the simplest change in the milestone.

3. **Install version awareness: Add --version flag + comparison logic (High):**
   Add `--version` to rapid-tools.cjs CLI. Update `setup.sh` to stamp version in .env. Update install skill to compare versions before deciding on full vs. skip install. Bump `package.json` to `3.1.0`.

4. **Parallel wave execution: Skill-level parallel dispatch (Medium):**
   Modify execute-set SKILL.md to analyze wave independence and spawn parallel Agent calls for independent waves. Add a `detectWaveIndependence()` helper to execute.cjs that parses wave PLAN.md files and checks file ownership disjointness. This is the most complex change and should be implemented last after the simpler changes are verified.

### Set Decomposition Recommendation

The four goals decompose naturally into sets based on independence:

- **Set A (Status Rename):** Cross-cutting but self-contained. Changes state-schemas, state-transitions, state-machine, all tests, all skills, all modules. No dependency on other sets.
- **Set B (Old Command Cleanup):** Fully independent. Deletes files and updates docs. No dependency on other sets.
- **Set C (Install Flexibility):** Independent. Changes rapid-tools.cjs, setup.sh, install skill. No dependency on other sets.
- **Set D (Parallel Wave Execution):** Depends on Set A if execute-set SKILL.md is being modified for status rename simultaneously. Otherwise independent. Changes execute-set skill and optionally execute.cjs.

Sets B and C can run in parallel with Set A. Set D should run after Set A if both touch execute-set SKILL.md, or can run in parallel if the status rename in execute-set is handled as part of Set A's scope.
