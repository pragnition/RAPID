# Oversights Research: v4.2.0 Discuss & Audit

## Cross-Cutting Concerns

### 1. Audit-Version State Transition Gap

The audit-version command needs to determine "is this milestone done?" by checking whether all sets reached `merged` status. However, the current state machine has no milestone-level status field -- `MilestoneState` only has `id`, `name`, and `sets[]`. There is no `milestone.status` to transition. The audit command will need to derive milestone completion from aggregated set statuses, but there is no existing helper for this. If the audit finds gaps and plans new sets, those sets need to be added to an already-"complete" milestone, which has no precedent in the current workflow.

**Why it matters for this milestone:** The audit-version command is the primary feature. Without a clear answer to "how does a milestone get marked as audited" vs "complete" vs "still has gap-fill sets pending," the command cannot provide reliable status tracking. The `add-set` command exists but was designed for mid-milestone additions, not post-completion gap-filling.

**Recommended action:** Decide early whether to add a milestone-level status enum (e.g., `active | audited | closed`) or keep the current implicit derivation. This affects state-schemas.cjs, state-transitions.cjs, and the migrate.cjs module if the schema version bumps.

---

### 2. Audit-Version Evidence Sources

GSD's audit-milestone cross-references three independent sources: REQUIREMENTS.md traceability, VERIFICATION-REPORT.md per phase, and SUMMARY.md frontmatter. RAPID has analogous artifacts but they are inconsistently populated:

- **REQUIREMENTS.md** exists and has checkbox traceability (`[x]`) for v3.0, but later versions (v3.1.0+) do not appear to have requirement-level entries.
- **VERIFICATION-REPORT.md** exists per set (5 currently in `.planning/sets/`) but not all completed sets have one -- it depends on whether the review phase generated one.
- **SET-OVERVIEW.md** exists per set but has no structured "requirements satisfied" field.

**Why it matters:** If the audit command relies on artifacts that may not exist, it will produce false "gap found" results for sets that are actually complete but lack documentation. The command must distinguish "missing artifact = gap" from "missing artifact = artifact never generated."

**Recommended action:** The audit command should define a minimal evidence threshold (e.g., set status `merged` is sufficient baseline evidence; VERIFICATION-REPORT.md is supplementary). Do not require artifacts that previous milestones never produced.

### 3. Discuss Gray Areas Consolidation -- AskUserQuestion Constraint Awareness

The milestone brief says to consolidate >4 gray areas into fewer AskUserQuestion prompts (e.g., 8 gray areas in 2 questions with 4 checkboxes each). The current implementation already batches gray areas into groups of 4, using multiple AskUserQuestion calls (Step 5 in SKILL.md). The todo.md says "we can actually just have 1 AskUserQuestion prompt with multiple questions in it."

**The overlooked concern:** AskUserQuestion's `multiSelect: true` mode renders checkboxes. But if you put 8 checkboxes in a single prompt, the user sees a wall of options without visual grouping. There is no built-in "group header" or "section divider" within a single AskUserQuestion call. The descriptions field on each option is the only way to provide context, and it is limited.

Additionally, the existing SKILL.test.cjs has structural tests that explicitly assert "Step 5 lists exactly 4 options" (Test 1), "Step 5 does NOT contain a 5th option" (Test 2), and "Key Principles says 'Exactly 4 gray areas'" (Test 3). These tests will break when the consolidation changes are made. The test file needs coordinated updates.

**Recommended action:** Update the structural tests in `skills/discuss-set/SKILL.test.cjs` as part of the same task that modifies the SKILL.md. Consider whether the consolidated prompt should use numbered prefixes in option labels (e.g., "1. Architecture: {title}" ... "5. Integration: {title}") to provide visual grouping within a flat list.

### 4. Discuss Table Formatting -- Rendering Environment Mismatch

The todo says pros/cons tables are "sometimes unaligned." This happens because Markdown tables render differently in:
1. **Claude Code's AskUserQuestion modal** -- rendered as styled HTML
2. **Terminal output** -- raw Markdown text (no rendering)
3. **Written CONTEXT.md** -- Markdown file (rendered only when viewed in an editor/GitHub)

The Format A tables in Step 6 of SKILL.md use standard Markdown pipes (`| Option | Pros | Cons |`). Alignment issues likely occur in the AskUserQuestion context string, where the table is embedded as text within the prompt parameter. If the AI model generates uneven column widths, the raw Markdown looks misaligned in fixed-width displays.

**Why it matters:** The "fix" depends entirely on which rendering context has the alignment problem. If it is the AskUserQuestion modal (HTML-rendered), Markdown tables should already align fine and the issue may be in how the model generates inconsistent column content lengths. If it is terminal output, Markdown tables will never look good without a dedicated formatter.

**Recommended action:** Clarify the rendering context before implementing. If the issue is AskUserQuestion's HTML rendering, the fix may be simply constraining pros/cons text length in the SKILL.md instructions. If it is terminal output, consider using a fixed-width table formatter in display.cjs (but the display module currently only handles banners, not tables).

### 5. Context.cjs Path Resolution -- Broader Impact Assessment

The bug report says `${RAPID_TOOLS}/../lib/context.cjs` fails because RAPID_TOOLS points to a file (`src/bin/rapid-tools.cjs`), not a directory. Shell path resolution treats `file/../` differently than `dir/../`. The actual resolution: `/home/kek/Projects/RAPID/src/bin/rapid-tools.cjs/../lib/context.cjs` becomes `/home/kek/Projects/RAPID/src/bin/lib/context.cjs` (wrong).

**Overlooked scope:** This pattern appears in exactly one place: `skills/init/SKILL.md` line 564. But the pattern `${RAPID_TOOLS}/../...` could easily be copied into future skills or plan files. This is also tracked as GitHub Issue #3, which notes a broader concern: RAPID_ROOT is computed but not used for binary discovery.

**Why it matters:** The fix for this one instance is trivial (`$(dirname "${RAPID_TOOLS}")/../lib/context.cjs` or use RAPID_ROOT). But the real oversight is that there is no convention or documented pattern for referencing sibling modules from skill/plan Bash code. Without establishing a convention, the same bug will recur.

**Recommended action:** Fix the immediate bug, but also:
1. Add a documented convention in CONVENTIONS.md or CLAUDE.md for how skills should reference library modules
2. Consider exporting a `RAPID_LIB` or `RAPID_ROOT` env var from the env preamble so skills never need to compute relative paths
3. Add a grep-based lint or test that flags `${RAPID_TOOLS}/../` patterns as errors

### 6. Audit-Version Deferred Items Integration

GSD's audit-milestone aggregates deferred items from all phases and presents them as a "tech debt" category that does not block milestone closure. RAPID already has a per-set DEFERRED.md file (written during discuss-set). However:

- There is no mechanism to aggregate DEFERRED.md files across all sets in a milestone.
- There is no convention for how deferred items flow into the next version's planning. The DEFERRED.md doc says "These items should be reviewed during `/rapid:new-version` planning" but `/rapid:new-version` SKILL.md does not reference DEFERRED.md files.
- The audit command should surface deferred items as context for "is this gap actually deferred intentionally or was it missed?"

**Why it matters:** Without deferred item awareness, the audit command will flag intentionally deferred work as gaps, creating noise. The user will have to manually distinguish "we decided to skip this" from "we forgot this."

**Recommended action:** The audit-version command should read all `.planning/sets/*/DEFERRED.md` files for the audited milestone and cross-reference them against detected gaps. Items that match a deferred decision should be labeled as "deferred (intentional)" rather than "gap."

### 7. Audit-Version -- New Skill Registration

Adding a new skill (`audit-version`) requires:
1. Creating `skills/audit-version/SKILL.md`
2. Potentially adding a new agent role if the audit logic is complex enough to warrant a subagent
3. No plugin.json update needed (skills are auto-discovered)
4. Potentially adding display banner support (`STAGE_VERBS` and `STAGE_BG` in display.cjs)
5. Updating `STAGE_VERBS` with an 'audit-version' entry

The display.cjs module has hardcoded stage-to-verb and stage-to-color mappings. If audit-version is not added to both `STAGE_VERBS` and `STAGE_BG`, the banner command will fall through to the unknown-stage fallback (which works but looks generic).

**Recommended action:** Add `'audit-version': 'AUDITING'` to STAGE_VERBS and assign it a distinct background color (suggestion: bright yellow `\x1b[103m` to distinguish from planning/execution/review).

### 8. Test Strategy for Skill-Level Changes

The discuss-set changes (table formatting, gray area consolidation) modify SKILL.md, which is a Markdown prompt file -- not executable code. The existing test approach (`SKILL.test.cjs`) uses string matching and regex against the Markdown content. This works for structural assertions ("does Step 5 contain 4 options?") but cannot validate:

- Whether the AI model actually follows the updated instructions
- Whether the consolidated gray area prompt produces a usable UX
- Whether table formatting instructions produce aligned output in practice

**Why it matters:** SKILL.md changes are instructions to an LLM, not deterministic code. String-match tests can verify the instructions are present but cannot verify they are effective. If the table formatting fix adds a new instruction like "keep pros/cons under 50 characters," the test can verify the instruction exists but not that models follow it.

**Recommended action:** For the SKILL.test.cjs updates, focus on structural invariants that must hold (e.g., "gray areas are presented in groups of 4," "Format A includes a table," "all three formats are documented"). Do not attempt to test LLM behavioral compliance -- that requires manual QA via actual `/rapid:discuss-set` runs.

### 9. Backward Compatibility of Discuss-Set Changes

The gray area consolidation changes the number of AskUserQuestion calls from potentially 2-3 (for 8-12 gray areas) down to fewer calls with more options each. The CONTEXT.md output format remains unchanged. However:

- Existing CONTEXT.md files from previous milestones were generated with the old flow. The audit-version command (if it reads historical CONTEXT.md) should not assume a specific discussion format.
- The `plan-set` skill parses CONTEXT.md using XML tags (`<domain>`, `<decisions>`, etc.). As long as the tags are preserved, downstream consumption is unaffected.
- The `discussed` self-transition in SET_TRANSITIONS allows re-discussion, so a set can be re-discussed after the SKILL.md update without state issues.

**Why it matters:** Low risk, but worth noting: if the plan-set skill has any logic that counts decisions or gray areas based on CONTEXT.md content patterns, the consolidation could produce different patterns. The XML tags are the safe boundary.

**Recommended action:** Verify that plan-set only reads CONTEXT.md via XML tag parsing, not by counting section headers or bullet points.

---

### 10. Documentation of Audit-Version Workflow Position

RAPID has a clear lifecycle sequence: `init > start-set > discuss-set > plan-set > execute-set > review > merge`. The audit-version command sits outside this sequence -- it runs after all sets in a milestone are merged. This needs to be documented:

- The progress breadcrumb shown by skills (`init [done] > start-set [done] > ...`) does not include audit-version.
- The ARCHITECTURE.md "Set Lifecycle" and "State Flow" sections do not mention auditing.
- The ROADMAP.md does not describe when audit-version should be invoked.

**Not applicable:** CI/CD pipeline concerns, internationalization, and accessibility -- RAPID is a CLI tool for developers, not an end-user application. These sections are not relevant.

## Recommendations

Ordered by "cost of getting it wrong" -- the items where a wrong early decision causes the most rework:

1. **Milestone status model (concern 1):** Decide whether audit-version creates a milestone-level status or derives completion from set statuses before writing any audit code. If you add a status field, the migration module needs to handle existing milestones without the field. -- *First step: Design the milestone status enum and document it in the audit-version DEFINITION.md.*

2. **Deferred items cross-referencing (concern 6):** Build deferred-item awareness into the audit-version gap detection from the start, not as a follow-up. Reading `.planning/sets/*/DEFERRED.md` and tagging known deferrals is trivial at design time but painful to retrofit after gap detection logic is written. -- *First step: Add a "load all DEFERRED.md" utility function that the audit command calls before comparing gaps.*

3. **Path resolution convention (concern 5):** Fix the context.cjs bug and simultaneously establish the convention to prevent recurrence. Adding `RAPID_ROOT` or `RAPID_LIB` to the env preamble is a 3-line change with permanent benefit. -- *First step: Add `RAPID_LIB=$(dirname "${RAPID_TOOLS}")/../lib` to the env preamble template and update the one broken reference in init SKILL.md.*

4. **Gray area test coordination (concern 3):** When modifying SKILL.md for the consolidation, update SKILL.test.cjs in the same task/commit. The tests explicitly assert "exactly 4 options" which will fail immediately. -- *First step: List all assertions in SKILL.test.cjs that reference the number 4 and plan their updates alongside the SKILL.md changes.*

5. **Table formatting diagnosis (concern 4):** Before implementing any table fix, reproduce the misalignment by running `/rapid:discuss-set` on a test set and observing where the tables render badly. The fix differs completely depending on whether the issue is in AskUserQuestion's HTML modal vs. terminal output. -- *First step: Run a discuss-set session, screenshot the misaligned table, identify the rendering context.*

6. **Display banner for audit-version (concern 7):** Add the stage verb and color mapping when creating the skill. -- *First step: Add `'audit-version': 'AUDITING'` and `'audit-version': '\x1b[103m'` to display.cjs.*

7. **Audit evidence strategy (concern 2):** Define what counts as "sufficient evidence of set completion" before implementing gap detection. Rely on set status (`merged`) as the primary signal; treat VERIFICATION-REPORT.md as supplementary. -- *First step: Document the evidence hierarchy in the audit-version SKILL.md preamble.*

8. **Plan-set backward compatibility (concern 9):** Verify plan-set parses CONTEXT.md only by XML tags. -- *First step: Grep plan-set SKILL.md and plan.cjs for CONTEXT.md parsing logic.*
