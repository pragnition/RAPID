# Oversights Research -- v3.5.0 Robustness & Fixes

## Cross-Cutting Concerns

### 1. Agent CLI Command Awareness (Systematic Drift)

The milestone explicitly calls out agents using wrong CLI subcommands (`state set-status`, `state --help`, `state update-set`). This is not an isolated bug -- it is a systemic oversight in how agents learn valid command syntax.

**Why this matters for THIS project:**
- RAPID's agents are generated from `src/modules/roles/*.md` and `src/modules/core/core-identity.md`. The core identity module tells agents to use `node "${RAPID_TOOLS}" state ...` but does not enumerate valid subcommands.
- The `state` command handler (`src/commands/state.cjs`) supports exactly: `get`, `transition`, `add-milestone`, `add-set`, `remove-set` (and others defined there). Agents have no structured reference to this list.
- Skills (e.g., `skills/execute-set/SKILL.md`, `skills/discuss-set/SKILL.md`) embed correct CLI calls as copy-paste templates, but agents operating outside skill contexts (e.g., the executor subagent, the verifier subagent) rely on their own knowledge, which drifts.

**What gets overlooked:**
- Fixing the known wrong commands without adding a mechanism to prevent future drift. A one-time sweep will become stale as commands evolve.
- The `tool-docs.cjs` module generates YAML tool documentation for agents but does not include a CLI command reference. Adding a canonical command reference to the agent build pipeline would be a lasting fix.
- The core-identity module instructs agents to "never edit .planning/ directly" and to use the CLI, but does not provide a command catalog. Agents infer commands from context, and inference fails for subcommands.

**Overlooked fix dimension:** The `build-agents` pipeline assembles agents from modules. A new module or section in `core-identity.md` listing all valid `state` subcommands (and other top-level commands) would propagate automatically to all 26 agents on rebuild. Without this, every agent must be individually audited every time a command changes.

### 2. DEFINITION.md Generation Timing vs. Consumption Timing

The DEFINITION.md file is created by `plan.cjs:createSet()` during the init/roadmap phase (when `plan create-set` runs). It is consumed by `plan.cjs:loadSet()` which is called by `worktree.cjs:generateScopedClaudeMd()` during `start-set`. The failure path reported in `todo-new.md` is:

```
Warning: Scoped CLAUDE.md could not be generated (missing DEFINITION.md)
```

**What gets overlooked:**
- The `add-set` skill (`skills/add-set/SKILL.md`) creates the set directory with `mkdir -p` and writes CONTRACT.json, SET-OVERVIEW.md, and DEFINITION.md manually. But the DEFINITION.md write is done by the skill's inline instructions, not by calling `plan create-set`. If the skill agent skips or fails the DEFINITION.md write (which is a markdown document written by the agent, not a CLI command), the file is silently missing.
- The `loadSet()` function in `plan.cjs` (line 202) does a raw `fs.readFileSync` on DEFINITION.md with no existence check -- it throws a hard error. There is an `fs.existsSync(setDir)` check for the directory but no graceful handling for a missing DEFINITION.md inside an existing directory.
- The `setInit()` function in `worktree.cjs` (line 371-378) catches the error gracefully and sets `claudeMdError`, but the worktree is still created. The result is a worktree without a scoped CLAUDE.md, which means the executor agent receives no file ownership boundaries -- a silent correctness issue, not just a cosmetic one.
- The init path via `plan.cjs:createSet()` always generates DEFINITION.md because it calls `generateDefinition()`. The only paths that produce sets WITHOUT DEFINITION.md are: (a) `add-set` if the agent fails to write it, and (b) manual set creation.

**What the fix should guard against:** The `loadSet()` function should either gracefully degrade when DEFINITION.md is missing (return empty string for definition) or the `start-set` flow should validate DEFINITION.md existence before attempting scoped CLAUDE.md generation.

### 3. State Transition Fragility: executed-to-complete Gap

The state transition map in `state-transitions.cjs` defines:
```
planned -> [executed]
executed -> [complete]
complete -> [merged]
```

The `execute-set` skill performs TWO transitions:
1. Step 3: `planned -> executed` (before wave execution starts)
2. Step 6: `executed -> complete` (after all waves finish)

**What gets overlooked:**
- If the executor agents complete all waves but the orchestrator skill fails before Step 6 (e.g., context limit hit, verifier agent hangs, user interrupts), the set is stuck in `executed` status. This is the exact scenario reported in `todo-new.md`.
- The review skill (`skills/review/SKILL.md` line 122-126) ONLY accepts `complete` status. It explicitly rejects `executed`:
  ```
  The set status MUST be `complete`. If the set is in any other status (e.g., `pending`, `planned`, `executed`): Exit.
  ```
- The fix is not just about making review accept `executed` -- it is about recognizing that `executed` and `complete` represent an artificial distinction. The `executed` state means "waves ran" and `complete` means "verification passed." But verification is non-blocking (gaps do not prevent the `complete` transition), so `complete` is always reachable from `executed` -- making the two-step transition a ceremony that creates a failure window.
- Re-running `execute-set` on a set in `executed` status does work (the skill accepts it for re-entry at line 79), but the user must know to re-run execute-set rather than manually transitioning. The error message from the review skill does not suggest this.

**Recommended approach:** Either (a) make the review skill accept both `executed` and `complete`, or (b) have the execute-set skill make the `executed -> complete` transition earlier/more reliably. Option (a) is cheaper. But both should be done -- the review skill should be lenient, AND the execute-set should commit the state transition atomically with the last wave commit.

### 4. Untracked Planning Artifacts Blocking Merges

The merge failure reported in `todo-new.md` is specific:
```
Untracked files on main would be overwritten by the merge:
- .planning/sets/code-quality/WAVE-1-COMPLETE.md
- .planning/sets/code-quality/wave-1-PLAN-DIGEST.md
```

**What gets overlooked:**
- These files are created by the executor agent in the worktree (on the `rapid/code-quality` branch). But the execute-set orchestrator also creates some files on main (WAVE-*-COMPLETE.md markers) during Step 6. The execute-set skill runs `git add` and `git commit` at the end, but the orchestrator itself runs on main, while the executor runs in the worktree. If the orchestrator creates marker files but does not commit them (e.g., the commit fails due to nothing staged, or the files were already committed in the worktree), they remain as untracked files on main.
- The `mergeSet()` function in `merge.cjs` does a `git merge --no-ff` which will fail if untracked files on the target branch (main) would be overwritten by the merge source branch. Git does not automatically resolve this.
- The deeper issue: the execute-set skill commits WAVE-*-COMPLETE.md marker files on whatever branch the orchestrator is running on. For worktree sets, the orchestrator runs on main (the project root), but marker files should be committed in the worktree. For solo sets, the orchestrator runs on main and everything is fine.
- The `merge.cjs:mergeSet()` function has no pre-merge cleanup step. It does not check for untracked files that would conflict. The fix needs to happen at two levels: (1) ensure marker files are committed in the worktree, and (2) add a pre-merge cleanup that handles stale untracked files.

**Where the execute-set commit goes wrong:** Step 6 of execute-set runs `git add .planning/sets/${SET_ID}/WAVE-*-COMPLETE.md` -- but this is from the orchestrator context, which may be the project root (main branch) or the worktree. The skill does not explicitly `cd` to the worktree path before this commit. If the WAVE-COMPLETE.md files were written to the project root's `.planning/` (which is shared across worktrees since `.planning/` lives at the project root, not inside `.rapid-worktrees/`), they are untracked on main.

### 5. Review File Discovery: Post-Merge Path Mismatch

The review skill writes REVIEW-SCOPE.md to:
- Standard mode: `.planning/sets/{setId}/REVIEW-SCOPE.md`
- Post-merge mode: `.planning/post-merge/{setId}/REVIEW-SCOPE.md`

Downstream skills (unit-test, bug-hunt, uat) read from:
- Standard mode: `.planning/sets/{setId}/REVIEW-SCOPE.md`
- Post-merge mode: `.planning/post-merge/{setId}/REVIEW-SCOPE.md`

**What gets overlooked:**
- The path logic is correct in the skill definitions. The actual bug is likely that: (a) the review skill writes to `.planning/post-merge/{setId}/` but the downstream skill does not detect `--post-merge` mode correctly, or (b) the `post-merge` directory does not exist and the write fails silently.
- The `review.cjs:logIssuePostMerge()` function (line 641) creates the directory with `fs.mkdirSync(setDir, { recursive: true })` before writing. But the REVIEW-SCOPE.md is written by the review SKILL (via the `Write` tool from the agent), not by `review.cjs`. The skill must create the directory manually or the Write tool must handle directory creation.
- The review skill does not explicitly `mkdir -p .planning/post-merge/{setId}/` before writing REVIEW-SCOPE.md. If the directory does not exist, the Write tool will fail.
- More fundamentally: when the user runs `/rapid:review <set-id> --post-merge` followed by `/rapid:unit-test <set-id>`, the unit-test skill needs to know it is in post-merge mode. If the user forgets `--post-merge` on the downstream skill, it looks in `.planning/sets/{setId}/` and finds nothing.

**What the fix should address:** (1) Ensure the `post-merge` directory is created before writing, (2) consider having downstream skills auto-detect post-merge mode by checking if `.planning/post-merge/{setId}/REVIEW-SCOPE.md` exists when `.planning/sets/{setId}/REVIEW-SCOPE.md` does not.

### 6. Worktree Package Management

The milestone calls out that worktrees need independent test execution. Currently:
- `worktree.cjs:createWorktree()` creates a git worktree with `git worktree add`
- Git worktrees share the `.git` directory but have independent working trees
- `node_modules/` is typically gitignored, so worktrees do NOT get a copy of `node_modules/`

**What gets overlooked:**
- There is zero `npm install` logic anywhere in the worktree creation path. The `createWorktree()` function, the `setInit()` function, and the `start-set` skill all skip package installation.
- The project itself (RAPID) only has 3 production dependencies (zod, ajv, proper-lockfile), but projects USING RAPID may have hundreds. When an executor agent runs `node --test` in a worktree, it will fail if node_modules is missing.
- For the RAPID project itself, worktrees may work accidentally because `node_modules/` is often symlinked or resolved from parent directories in certain Node.js resolution configurations. But this is fragile and project-specific.
- The merge agent should verify packages are installed, but so should the executor. By the time the merge agent catches a missing dependency, the damage is done -- the execution was incomplete.
- For non-Node ecosystems (Python/uv, Ruby/bundler, etc.), the problem is worse because virtual environments are not shared across worktrees at all.

**What the fix should address:** (1) Add a post-worktree-creation hook that runs package installation in the worktree, (2) make this ecosystem-aware (detect package.json, requirements.txt, pyproject.toml, etc.), (3) add a verification step in execute-set that confirms the test runner works before executing waves.

### 7. AskUserQuestion Option Limit (4 max)

The milestone notes that AskUserQuestion has a maximum of 4 options. The discuss-set skill Step 5 currently presents 5 options (lines 166-171 of `skills/discuss-set/SKILL.md`):

```
1. "{Gray area 1 title}" -- "..."
2. "{Gray area 2 title}" -- "..."
3. "{Gray area 3 title}" -- "..."
4. "{Gray area 4 title}" -- "..."
5. "{Gray area 5 title}" -- "..."
```

**What gets overlooked:**
- The text says "4 gray areas" but the template shows 5 options. The fix is simple: remove option 5.
- But the constraint applies EVERYWHERE AskUserQuestion is used. The previous v3.2.0 UX improvements wave already audited all 24 skills for AskUserQuestion usage. A new sweep is needed to verify none of the fixes introduced >4 options.
- The Step 5 text says "exactly 4 gray areas" but the Key Principles section says "2-5 gray areas, more is better than less." These contradict. The fix needs to reconcile both: the limit should be 4 (matching the AskUserQuestion constraint), and the Key Principles should say "2-4."
- The Anti-Patterns section says "Do NOT ask about less than 2 gray areas" -- this is fine, the range should be 2-4.
- Step 6 asks questions WITHIN each gray area, each with options like "Option A", "Option B", "Option C", "Claude decides". That is 4 options max, which is fine. But if the agent generates 5 sub-questions within an area, each presented as a separate AskUserQuestion, that is acceptable (each call has <= 4 options).

### 8. Solo Mode Project-Wide Config

The milestone calls for defaulting all sets to solo mode via `config.json`. Currently:
- Solo mode is per-set via `--solo` flag on `/start-set`
- The `.planning/config.json` has `project`, `model`, and `planning` keys but no `solo` flag
- The `config.json` at project root has `lock_timeout_ms` and `agent_size_warn_kb` but no `solo` flag

**What gets overlooked:**
- There are TWO config files: `.planning/config.json` (planning config) and `config.json` (runtime config). The solo mode setting should go in `.planning/config.json` since it affects the planning/execution workflow, not runtime behavior.
- Every skill that checks for `--solo` must also check the project-wide config. This includes: `start-set`, `execute-set`, `merge`, `review`. If they only check the CLI flag, the config file default is ignored.
- The `set-init` command handler (`src/commands/set-init.cjs` line 13) checks `args.includes('--solo')`. It needs to also check `.planning/config.json` for a project-wide solo default.
- The `/init` skill (`skills/init/SKILL.md`) should ask "Will you be working solo or with a team?" and persist the answer. This is already specified in `user-todo.md` but not yet implemented.
- The `/status` skill should display solo mode status. Currently it reads from the registry, which already has `solo: true` per entry, but it does not distinguish project-wide solo vs per-set solo.

### 9. UI/UX Emphasis in Discuss/Planning

The milestone calls for improving discuss/planning to emphasize UI/UX considerations. Currently, the discuss-set skill (Step 5) mentions "UI/UX decisions need to be made" in its gray area criteria, but:

**What gets overlooked:**
- The gray area identification is entirely agent-discretionary. The skill says "identify exactly 4 gray areas" but does not require any to be UI/UX related. For non-frontend projects, this is correct. For frontend projects, the agent may still skip UI/UX in favor of backend concerns.
- The plan-set skill does not mention UI/UX at all. Wave plans are decomposed by task complexity, not by concern type. A "UI considerations" section in wave plans would help executors remember to implement accessible, responsive, styled components rather than just functional code.
- The SET-OVERVIEW.md template (generated during init) does not have a UI/UX section. Adding one would ensure that UI/UX requirements are captured at the set level before planning begins.
- The CONTRACT.json schema does not have a `ui` or `ux` section. UI contracts (screen mockups, component hierarchy, interaction patterns) are informal or missing. The `ui-contract.cjs` module exists but is separate from the main contract flow.

### 10. Error Recovery After Partial Execution Failure

**What gets overlooked across the milestone goals:**
- Several goals involve fixing failure modes (state transition failures, merge failures, file discovery failures). Each fix addresses a specific symptom, but there is no systematic error recovery strategy.
- The execute-set skill has re-entry logic (checking WAVE-*-COMPLETE.md markers) which is good, but the other skills do not. The review skill cannot resume if it fails mid-scoping. The merge skill has idempotent re-entry via MERGE-STATE.json.
- When a state transition fails (e.g., `executed -> complete`), there is no retry mechanism. The user must manually re-run the skill. An auto-retry with backoff would handle transient lock failures.
- The `withStateTransaction()` function handles locking and atomic writes, but it does not log failed mutations for debugging. When a transition fails, the error message says what went wrong but not what state the system is in. Adding a "current state dump" to transition error messages would save users from having to run `state get --all` after every failure.

### 11. Git Commit Ordering in Worktree vs. Main

**What gets overlooked:**
- The execute-set skill transitions state (`STATE.json`) at Step 3 (before execution) and Step 6 (after execution). STATE.json lives in `.planning/` which is at the project root, NOT inside the worktree. Git commits of STATE.json from the project root go to main. But the executor agents' code commits go to the worktree branch.
- This means the state transition commit and the code commit are on different branches. If someone looks at the worktree branch history, they will not see the state transition. If they look at main, they will not see the code changes.
- The execute-set Step 6 commits `WAVE-*-COMPLETE.md` and `GAPS.md` from `.planning/sets/{SET_ID}/` on main (or wherever the orchestrator is). These are planning artifacts that describe the worktree branch's work but are committed to main.
- When the merge happens later, the worktree branch does not contain these planning artifacts (they were committed to main). So the merge brings code changes from the worktree but NOT the planning artifact commits. The planning artifacts were already on main before the merge.
- This ordering is actually correct (state should be on main so other sets can read it), but it is confusing and easy to break. The fix for "untracked files blocking merge" must account for this two-branch commit model.

### 12. Testing Strategy for This Milestone

**What gets overlooked:**
- The fixes in this milestone span skills (markdown prompt files), library code (`.cjs` modules), and agent definitions (generated `.md` files). The skills have no automated testing -- they are prompt documents executed by Claude Code.
- The library code has excellent test coverage (co-located `.test.cjs` files). Fixes to `state-transitions.cjs`, `worktree.cjs`, `review.cjs`, `merge.cjs`, and `plan.cjs` should all have corresponding test updates.
- Agent definitions are generated by `build-agents`. If the fix involves adding CLI command references to `core-identity.md`, the change must be verified by rebuilding agents and checking the output.
- There is no integration test that runs the full `start-set -> discuss-set -> plan-set -> execute-set -> review -> merge` pipeline. Each skill is tested in isolation by running it manually. A regression in one step may not surface until a user hits it.

| Git commit ordering model (#11) | HIGH | Wave 1 | Foundational design issue that every other fix must account for. Not a bug to fix -- a constraint to document and respect. |
| State transition fragility (#3) | MEDIUM | Wave 1 | Simple fix (accept `executed` in review), but understanding the two-step transition ceremony is needed for other fixes. |

## Recommendations

Ordered by urgency (things to include in the roadmap early):

1. **Agent CLI command catalog (#1):** Add a `## Valid CLI Commands` section to `core-identity.md` listing all valid commands and subcommands with their argument patterns. This propagates to all agents on rebuild. Without this, the agent sweep fix will rot within one milestone. **First step:** Extract the command routing table from `rapid-tools.cjs` into a reference section for `core-identity.md`.

2. **Fix the two-branch commit model for planning artifacts (#4, #11):** Decide once: do WAVE-COMPLETE.md markers, STATE.json transitions, and PLAN-DIGEST.md files belong on main or on the worktree branch? Currently they are split. The simplest fix: commit all `.planning/sets/{SET_ID}/` artifacts in the worktree branch (not main), and only commit STATE.json transitions on main. This means the merge will bring planning artifacts from the worktree branch to main cleanly. **First step:** Change execute-set Step 6 to commit marker files in the worktree, not on main.

3. **Make review accept `executed` status (#3):** Change the review skill's status validation to accept both `executed` and `complete`. Also keep the `executed -> complete` transition in execute-set, but make it more resilient (catch and warn rather than fail). **First step:** Edit `skills/review/SKILL.md` line 122 to add `executed` to the accepted statuses.

4. **Graceful DEFINITION.md handling (#2):** Add an existence check in `plan.cjs:loadSet()` so that a missing DEFINITION.md returns an empty string rather than throwing. Also add DEFINITION.md validation in the `add-set` skill flow. **First step:** Add `fs.existsSync` check before `fs.readFileSync` in `loadSet()`.

5. **Post-merge directory auto-creation and mode detection (#5):** Ensure the review skill creates `.planning/post-merge/{setId}/` before writing REVIEW-SCOPE.md. Have downstream skills auto-detect post-merge mode when the standard path file is missing but the post-merge path file exists. **First step:** Add `mkdir -p` equivalent in the review skill before the Write tool call.

6. **AskUserQuestion constraint enforcement (#7):** Fix discuss-set to present exactly 4 gray areas (not 5). Update Key Principles to say "2-4 gray areas." Audit all other skills for >4 options in any AskUserQuestion call. **First step:** Remove option 5 from discuss-set Step 5 template.

7. **Worktree package management (#6):** Add a `postCreateHook` in `worktree.cjs:setInit()` that detects the project's package manager and runs installation. Start with Node.js (`npm install`) since RAPID is a Node.js tool. **First step:** After `createWorktree()`, check for `package.json` in the worktree and run `npm install --production` if found.

8. **Solo mode project config (#8):** Add `solo: true/false` to `.planning/config.json`. Have `set-init` check this config when `--solo` is not explicitly passed. Update `/init` to ask the solo question. **First step:** Add `defaultSolo` field to `.planning/config.json` schema.

9. **UI/UX emphasis in discuss/plan (#9):** Add a "UI/UX Considerations" section to the discuss-set gray area template. For frontend-detected projects, make at least one gray area UI/UX related. Add a UI/UX checklist section to wave plan templates. **First step:** Add conditional UI/UX gray area guidance to discuss-set Step 5 when the project has frontend files.
