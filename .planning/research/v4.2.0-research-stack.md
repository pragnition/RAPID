### Zod 3 vs Zod 4

The project uses `zod@3.25.76` via `require('zod')`. Zod 4 is available as a subpath import (`require('zod/v4')`) within the same package. Key differences relevant to this project:

- **Breaking:** Error customization API changed (`message` -> `error`, `invalid_type_error`/`required_error` replaced with error function).
- **Breaking:** `.strict()` and `.passthrough()` replaced by `z.strictObject()` and `z.looseObject()`.
- **Breaking:** String format methods moved to top-level (`z.string().email()` -> `z.email()`).
- **Non-breaking for RAPID:** The project's Zod usage in `state-schemas.cjs` primarily uses `z.object()`, `z.enum()`, `z.array()`, `z.string()`, `z.optional()` -- basic constructs that are largely compatible.
- **CommonJS support:** Zod 4 is consumable via `require('zod/v4')` in CJS, confirmed by Zod docs.
- **Recommendation for v4.2.0:** Do NOT upgrade to Zod 4 in this milestone. The state schema module is load-bearing infrastructure. A Zod upgrade warrants its own focused set with migration testing.

### proper-lockfile Longevity

- Last published 5+ years ago. The maintainer (moxystudio) has not archived the repository but activity is minimal.
- The package is functionally complete for its scope (file locking with stale detection, retries, mkdir-based locking).

- **Risk level:** Low for now. If the package becomes formally deprecated or a Node.js update breaks compatibility, alternatives include `lockfile` (npm) or implementing a simple mkdir-based lock (the project's `lock.cjs` already wraps proper-lockfile).

## Stack Risks

1. **Path resolution bug (`${RAPID_TOOLS}/../lib/context.cjs`):** **High impact** -- Breaks `/rapid:init` test framework detection. The bug is confirmed: when `RAPID_TOOLS` points to a file path like `.../src/bin/rapid-tools.cjs`, Node's `require()` treats `rapid-tools.cjs/..` as navigating up from a file-as-directory, resolving to `.../src/bin/lib/context.cjs` instead of the intended `.../src/lib/context.cjs`. The same pattern appears in 4 other locations (`web-client.cjs` references in `init/SKILL.md` and `register-web/SKILL.md`). **Mitigation:** Use `path.dirname(RAPID_TOOLS)` or `$(dirname "${RAPID_TOOLS}")` in shell before constructing the require path.

2. **proper-lockfile maintenance risk:** **Low impact** -- Package is unmaintained but functionally stable. No CVEs. The project's `lock.cjs` wrapper provides an abstraction layer that would make replacement straightforward if needed. **Mitigation:** Monitor; no action needed now.

3. **Zod 3 eventual deprecation:** **Low impact** -- Zod 3 continues to be exported from the `zod` package and the maintainer has committed to a transitionary period. No urgency. **Mitigation:** Plan Zod 4 migration as a standalone set in a future milestone.

4. **Node 18 minimum claim is stale:** **Low impact** -- Documentation says Node 18+ but Node 18 is EOL. Does not affect functionality. **Mitigation:** Update documentation reference to Node 22+.

## Recommendations

1. **Fix RAPID_TOOLS path resolution (critical):** Replace all `require('${RAPID_TOOLS}/../lib/...')` patterns with `require(path.join(path.dirname('${RAPID_TOOLS}'), '..', 'lib', '...'))` or use shell `dirname` before interpolation. Affects `skills/init/SKILL.md` (2 occurrences) and `skills/register-web/SKILL.md` (2 occurrences). This is a v4.2.0 deliverable. -- **Priority: critical**

2. **Add `audit-version` command to CLI and skill layer (high):** No stack constraints identified. The audit-version feature is pure application logic -- reading state, comparing planned vs. completed work, generating reports. The existing state-machine, plan, and resolve libraries provide all needed infrastructure. -- **Priority: high**

3. **Improve discuss table formatting (medium):** The discuss skill (`skills/discuss-set/SKILL.md`) uses Markdown tables in AskUserQuestion prompts (Format A, lines 236-252). Table alignment in AskUserQuestion depends on the Claude Code UI rendering, not on the RAPID codebase. The improvement likely needs to pad cell content or use a different visual format. No dependency changes needed. -- **Priority: medium**

4. **Consolidate gray area prompts (medium):** The discuss skill already batches gray areas in groups of 4 (lines 183-216). The requested change is to further consolidate when >4 gray areas exist (e.g., 8 gray areas in 2 prompts with 4 checkboxes each instead of separate prompts). The current SKILL.md already implements this batching pattern -- the task may be to verify it works correctly or adjust the batching logic. No stack implications. -- **Priority: medium**

5. **Update Node.js minimum version in docs (low):** Change "Node.js 18+" to "Node.js 22+" in CODEBASE.md and any setup documentation. -- **Priority: low**

6. **Plan Zod 4 migration for future milestone (low):** Not for v4.2.0. When ready, use the `zod/v4` subpath import for incremental migration. Test state-schemas.cjs thoroughly. -- **Priority: low**
