# Domain & UX Research: OSS Presentation Milestone

**Domain:** Open-source GitHub presentation for a Claude Code plugin (developer tool)
**Researched:** 2026-03-30
**Overall confidence:** HIGH (GitHub OSS conventions are well-established; Everforest palette is fully documented; competing plugin READMEs provide clear patterns)

---

## Domain Conventions

### How Similar Products Present on GitHub

Developer tools in the Claude Code plugin ecosystem and the broader CLI/developer-tool space follow a remarkably consistent GitHub presentation pattern. Research across Anthropic's own claude-code repo, the everything-claude-code project, the OpenSpec plugin, and top-starred developer tools reveals these conventions:

1. **Header stack pattern.** The dominant layout is: centered banner image, then a row of shields.io badges, then a one-line attribution or tagline. This is the single most recognizable pattern across high-quality OSS repos. (Source: anthropics/claude-code, Fission-AI/OpenSpec, affaan-m/everything-claude-code, badges/shields README)

2. **Progressive disclosure.** READMEs universally follow a top-down funnel: visual hook (banner + badges) -> problem statement -> install -> quickstart -> deep architecture -> reference docs -> contributing. The user reads as deep as they need and stops. (Source: awesome-readme curated list, Tilburg Science Hub README best practices)

3. **Collapsible architecture sections.** Complex projects wrap architecture diagrams and detailed trees inside `<details><summary>` blocks. This keeps the README scannable while preserving depth for those who want it. (Source: GitHub Flavored Markdown docs, copyprogramming.com collapsible guide)

4. **Static SVGs over Mermaid.** High-polish OSS projects use hand-crafted SVGs for diagrams rather than Mermaid or ASCII art. SVGs render consistently across light/dark mode when they include their own background, are version-controllable, and allow palette-matched branding. Mermaid rendering varies by GitHub's server-side renderer and cannot use custom fonts or fine color control. (Source: dev.to architecture diagrams discussion, GitHub rendering behavior)

5. **Minimal badge count.** The sweet spot is 3-5 badges. More than 8 starts looking cluttered. The most common badges for developer tools are: version/release, license, platform/runtime, and a community link (Discord, discussions). (Source: shields.io best practices, daily.dev badge guide)

6. **Arrow-prefixed doc links.** The OpenSpec README and similar minimalist projects use Unicode arrow characters (`->`) as visual tokenization for link lists, replacing verbose bullet-point descriptions. This creates scannable navigation that feels intentional and compact. (Source: Fission-AI/OpenSpec README)

### Standard Terminology in Claude Code Plugin Ecosystem

| User-Facing Term | Internal/Technical Term | Convention Source |
|---|---|---|
| "plugin" | plugin (not "extension" or "package") | Anthropic plugin system |
| "skill" / "command" | slash command (`/rapid:init`) | Claude Code CLI convention |
| "agent" | subagent, agent definition | Anthropic agent SDK |
| "install" | `claude plugin add <org>/<repo>` | Official install path |
| "set" | workstream, feature branch | RAPID-specific terminology |

Users in this ecosystem expect the `/namespace:command` syntax. The install path is `claude plugin add`, not npm/pip/brew. The README install section must use this exact phrasing -- users will ctrl+F for "install" or "getting started." (Source: anthropics/claude-code plugins README, claude-plugins-official marketplace.json)

## Interaction Models

### Primary Interaction: README as Landing Page

For an OSS presentation milestone, the primary "interaction" is not a CLI session -- it is a developer landing on the GitHub repository page. The README IS the product interface. Every UX decision in this milestone optimizes for that first-impression read.

**Interaction flow users expect:**
1. See banner -> immediately understand "what is this" (visual identity)
2. See badges -> gauge maturity (version, license, platform)
3. Read one-liner -> confirm relevance ("is this for me?")
4. Scan install -> decide effort ("how hard is this to try?")
5. Skim quickstart -> evaluate complexity ("do I understand the workflow?")
6. (Optional) Explore architecture -> assess sophistication
7. (Optional) Read contributing -> decide whether to participate

**Time budget:** Research on GitHub browsing behavior suggests developers spend 10-30 seconds deciding whether to star/bookmark a repo. The banner, badges, and first paragraph must work within that window. (Source: Hatica README best practices, freeCodeCamp README structure guide)

### Issue and PR Templates as Interaction Design

GitHub issue forms (YAML) are a form of interaction design. They replace free-text issues with structured forms that guide reporters through the information a maintainer needs.

**Standard field patterns for bug reports:**
- Description (textarea, required)
- Steps to reproduce (textarea, required)
- Expected vs actual behavior (textarea, required)
- Version/environment (input or dropdown, required)
- Screenshots (textarea, optional)

**Standard field patterns for feature requests:**
- Problem description / use case (textarea, required)
- Proposed solution (textarea, optional)
- Alternatives considered (textarea, optional)

**config.yml conventions:** Disable blank issues (force template selection), add external links to documentation or discussions. (Source: GitHub Docs issue forms syntax, Everhour GitHub templates guide)

### PR Template Conventions

Short and checklist-oriented. The standard pattern is:
- "What changed" (free text, 1-2 sentences)
- "Why" (free text, link to issue)
- "Testing done" (checklist: manual test, unit test, etc.)
- "Checklist" (checkboxes: tests pass, docs updated, no breaking changes)

Long PR templates get ignored. The ideal template is 10-20 lines. (Source: Graphite PR checklist guide, Microsoft Engineering Playbook, pullchecklist.com)

## Information Architecture

### README Section Hierarchy

Based on research across 20+ high-profile developer tool READMEs, the information architecture should follow this priority:

| Priority | Section | Purpose | Convention |
|---|---|---|---|
| 1 | Banner + Badges | Visual identity, maturity signals | Centered `<p align="center">` |
| 2 | Title + One-Liner | "What is this?" in under 10 words | H1 heading + bold paragraph |
| 3 | Problem Statement | "Why does this exist?" | 2-3 paragraphs, narrative voice |
| 4 | Install | "How do I try this?" | Code block, minimal steps |
| 5 | Quickstart | "What does usage look like?" | Terminal transcript or code blocks |
| 6 | How It Works | "What happens under the hood?" | Prose + diagram |
| 7 | Architecture | Deep technical detail | Collapsible (`<details>`) |
| 8 | Command Reference | Complete reference | Tables |
| 9 | Documentation Links | Navigation to deep docs | Arrow-prefixed link list |
| 10 | Contributing | "How do I participate?" | One-liner linking to CONTRIBUTING.md |
| 11 | License | Legal | One line |

**Key insight:** The spec's proposed section ordering matches this convention exactly. The spec already aligns with industry best practice. (Source: freeCodeCamp README structure, Tilburg Science Hub, Medium README Rules)

### CONTRIBUTING.md Architecture

For developer tools, the standard CONTRIBUTING.md sections are:

1. **Dev install** -- how to clone and set up locally (not plugin install, but contributor setup)
2. **Bug reports** -- link to issue template, describe what info is needed
3. **Feature proposals** -- link to issue template or discussions
4. **Code style** -- link to or summarize style guide
5. **Pull request process** -- what to do before submitting

The spec calls for "brief, no-nonsense" which aligns with the convention. Lengthy CONTRIBUTING files get skimmed; the best ones are under 100 lines. (Source: Mozilla Science CONTRIBUTING guide, GitHub Docs contributor guidelines, executablebooks/.github CONTRIBUTING.md)

### Collapsible Sections

GitHub supports `<details><summary>` natively. The convention for architecture diagrams is:

```html
<details>
<summary>Architecture</summary>

[diagram or detailed content]

</details>
```

An empty line is required between the `<summary>` closing tag and the Markdown content inside. This is a common gotcha. (Source: GitHub Flavored Markdown spec, emmanuelgautier.com collapsible sections)

## Accessibility Considerations

### SVG Accessibility

SVGs embedded in README files via `<img>` tags should include:
- `alt` text that describes the diagram content (not just "diagram")
- Sufficient color contrast against the SVG's own background (not relying on GitHub's page background)
- Solid backgrounds rather than transparency, because GitHub renders READMEs in both light and dark mode

The Everforest dark palette provides good contrast ratios:
- Foreground `#d3c6aa` on Background `#2d353b` = contrast ratio ~7.4:1 (exceeds WCAG AAA)
- Grey `#859289` on Background `#2d353b` = contrast ratio ~4.1:1 (meets WCAG AA)
- Red `#e67e80` on Background `#2d353b` = contrast ratio ~5.1:1 (meets WCAG AA)

These ratios are sufficient for text in diagrams and banners.

### Dark/Light Mode Handling

GitHub renders READMEs in both light and dark mode. SVGs with solid dark backgrounds will look correct in dark mode and stand out (in a good way) in light mode -- this is the standard approach for dark-themed developer tools. The `<picture>` tag with `<source media="(prefers-color-scheme: dark)">` can provide mode-specific images, but for a consistently dark-branded project like RAPID, a single dark SVG is the conventional choice. Most Everforest-themed projects use a single dark variant. (Source: GitHub Docs customizing social preview, OpenSpec approach)

### Screen Reader and Keyboard Navigation

README content on GitHub is navigable via screen readers through heading hierarchy. The proposed structure (H1 for title, H2 for sections, H3 for subsections) follows proper semantic hierarchy. `<details>` elements are natively accessible -- screen readers announce them as disclosure widgets. No special ARIA attributes are needed. (Source: GitHub Flavored Markdown accessibility, W3C details element spec)

## User Expectations

### What Developers Assume When They Land on This Repo

Based on domain research and competitive analysis of the Claude Code plugin ecosystem:

1. **Install must be one command.** Developers expect `claude plugin add <org>/<repo>` as the ONLY install step. Any additional steps (running `/rapid:install`) should be clearly separated as "configuration" not "installation." The spec's `> [!TIP]` callout for install is the right approach -- it visually elevates the install command above surrounding content. (Source: anthropics/claude-code install pattern)

2. **Badges signal maturity.** A repo without badges looks like a side project. Three badges (version, license, platform) is the minimum for "this is maintained software." The spec's three badges (version, MIT license, Claude Code Plugin) hit this threshold exactly. (Source: daily.dev badge best practices, shields.io conventions)

3. **The banner IS the brand.** Developers will screenshot and share repos with distinctive banners. The pixel-art wave banner currently in `branding/banner.svg` is creative but does not communicate "professional developer tool." The proposed G1 Everforest serif typography banner communicates: minimalist, intentional, confident. This matches the aesthetic of tools developers trust (terminal-native, no-nonsense). (Source: OpenSpec's banner approach, Vercel's Geist branding)

4. **`pragnition/RAPID` not `fishjojo1/RAPID`.** The reference migration from fishjojo1 to pragnition signals organizational ownership, which increases perceived longevity. Developers are more likely to adopt tools owned by orgs than individual accounts. This is a well-known GitHub trust signal. (Source: GitHub repository transfer docs, enterprise migration guides)

5. **Issue templates signal "we accept contributions."** A repo with YAML issue forms looks maintained and welcoming. A repo with no templates or blank-issue-only looks like it does not want outside input. For an OSS presentation milestone, the templates are as much a signal as a functional feature. (Source: GitHub Docs issue template configuration)

6. **A short CONTRIBUTING.md signals "we respect your time."** Lengthy contributor guides are associated with enterprise projects and bureaucratic processes. For a developer tool, brief = respectful. The spec's "brief, no-nonsense" tone is exactly right.

### Terminology Expectations

| What Users Say | What They Mean | RAPID Equivalent |
|---|---|---|
| "How do I install this?" | `claude plugin add` | `/rapid:install` for config |
| "What commands does it have?" | Slash command reference | Command Reference tables |
| "How does it work?" | Workflow overview | Lifecycle flow diagram |
| "Can I contribute?" | Where is CONTRIBUTING.md? | Root-level CONTRIBUTING.md |
| "Is this maintained?" | Recent commits, badges, version | Badges + commit history |
| "What license?" | MIT/Apache/ISC badge | License badge + LICENSE file |

### Mental Models from Competing Products

Developers coming from other Claude Code plugins expect:
- A flat `/namespace:command` structure (not nested subcommands)
- A single entry point command (like `/rapid:init` to start)
- An install path via `claude plugin add` (the canonical method)
- Documentation in DOCS.md or docs/ (not a separate docs site)

RAPID already matches all of these. The OSS presentation milestone reinforces these conventions visually. (Source: anthropics/claude-code plugins directory, awesome-claude-code curated lists)

## Recommendations

Ordered by priority. Each recommendation is specific to this OSS Presentation milestone.

1. **Use solid dark backgrounds on all SVGs**: Never use transparent SVG backgrounds. GitHub renders in both light and dark mode. Everforest `#2D353B` as the background on every SVG ensures consistent rendering in both modes. -- Priority: **critical**

2. **Keep badge count to exactly 3**: Version, License (MIT), Claude Code Plugin. Three badges hit the maturity signal threshold without clutter. Adding more (stars, downloads, CI status) can come after the repo gains traction. -- Priority: **critical**

3. **Use `flat-square` badge style with Everforest colors**: The spec specifies `flat-square` style with custom Everforest hex colors. This is the right choice -- it creates visual cohesion between badges and banner. Avoid default shields.io green/blue colors. -- Priority: **high**

4. **Social preview must be PNG with solid background**: GitHub's og:image spec requires PNG, JPG, or GIF under 1MB at 1280x640. Do not use transparency -- social media platforms render transparent backgrounds differently (Twitter: white, Discord: black, Slack: white). A solid `#2D353B` background ensures consistency. -- Priority: **high**

5. **Lifecycle flow diagram: horizontal, left-to-right, 7 nodes**: The standard flow is init -> start-set -> discuss-set -> plan-set -> execute-set -> review -> merge. Each node should show the command name and its primary artifact. Horizontal flow matches the spec's ~1280x200 dimensions and the left-to-right reading convention for process flows. -- Priority: **high**

6. **Agent dispatch diagram: wrap in collapsible `<details>`**: This diagram is ~1280x600 and will dominate the README if shown inline. Collapsible sections are the convention for "deep detail that not everyone needs." The summary text should be descriptive: "Agent Dispatch Architecture" or similar. -- Priority: **high**

7. **Reference migration: prioritize user-facing files first**: The `fishjojo1/RAPID` -> `pragnition/RAPID` migration affects 52 files. Prioritize: plugin.json, README.md, CONTRIBUTING.md, PROJECT.md, CODEBASE.md. Archive files (`.planning/archive/`, `.planning/milestones/`) can be updated in bulk but are lower priority since they are not user-facing. -- Priority: **high**

8. **Issue form fields: require version and Claude Code version**: Bug reports in the Claude Code plugin ecosystem almost always need both the plugin version AND the Claude Code version. These should be separate required `input` fields in bug_report.yml, not a single "environment" textarea. -- Priority: **medium**

9. **PR template: keep under 15 lines**: The spec calls for "short checklist" -- enforce this. Three sections max: What (1-2 sentences), Why (link to issue), Testing (3-4 checkboxes). Developers skip long templates. -- Priority: **medium**

10. **CONTRIBUTING.md: lead with `git clone` not `claude plugin add`**: Contributors need the dev install path (clone, npm install, run tests), not the user install path. Make this distinction explicit in the first section. -- Priority: **medium**

11. **Use Georgia serif for banner typography**: The spec calls for Georgia, which is a web-safe serif font available on all platforms. This is the correct choice for SVG text elements -- it will render reliably without font embedding. Avoid Google Fonts or custom typefaces in SVGs that need to render in GitHub's image proxy. -- Priority: **medium**

12. **Arrow-prefixed doc links: use Unicode `\u2192` not ASCII `-->`**: The spec shows `->` but the OpenSpec convention uses the Unicode right arrow character. Unicode arrows are more visually polished and render consistently on GitHub. -- Priority: **low**
