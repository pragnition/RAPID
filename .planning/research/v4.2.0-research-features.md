## Feature Inventory

### Feature: Audit Version Command (`/rapid:audit-version`)

- **Description:** After all sets in a milestone complete, audits the version for missed features, incomplete implementations, or divergence from original requirements. Offers to plan remediation sets or defer items to the next version with written context.
- **Complexity:** complex
- **Dependencies:** independent (reads existing state and artifacts)

#### Decomposition

1. **Audit skill scaffold** -- Create `skills/audit-version/SKILL.md` with environment preamble, banner, and step-by-step flow
2. **Requirement harvesting** -- Read REQUIREMENTS.md, ROADMAP.md, and per-set DEFINITION.md/CONTRACT.json to build a requirements inventory
3. **Artifact verification** -- For each set, read VERIFICATION-REPORT.md (if exists), WAVE-COMPLETE.md files, and SET-OVERVIEW.md to determine what was actually delivered
4. **Gap analysis engine** -- Cross-reference requirements against delivered artifacts; classify each requirement as satisfied/partial/unsatisfied/orphaned
5. **Display banner integration** -- Add `audit-version` stage verb and background color to `src/lib/display.cjs` STAGE_VERBS and STAGE_BG maps
6. **Interactive remediation offering** -- Present gaps to user via AskUserQuestion; offer to plan new sets (via `/rapid:add-set`) or defer items to next version
7. **Deferral context writing** -- Write deferred items to `.planning/DEFERRED-{version}.md` with rich context for carry-forward into `/rapid:new-version`

#### Implementation Strategies

**Option 1: Skill-only approach (prompt-driven audit)**
- Approach: Implement entirely as a SKILL.md file. The skill reads all planning artifacts, uses the model's reasoning to cross-reference requirements vs. delivered work, and produces the audit report. No new library code.
- Pros: Consistent with existing skill patterns (quick, new-version); no new library modules to maintain; leverages model's ability to do fuzzy requirement matching
- Cons: Audit quality depends on model reasoning; no programmatic verification; harder to unit test
- Libraries/services: None new

**Option 2: Hybrid skill + library approach**
- Approach: Add an `audit.cjs` library module that programmatically harvests artifacts and builds a structured requirements-vs-delivery matrix. The SKILL.md orchestrates the flow and calls CLI commands backed by the library. The model handles gap interpretation and user interaction.
- Pros: Programmatic artifact harvesting is deterministic and testable; structured data enables consistent reporting; follows the pattern of execute.cjs and review.cjs
- Cons: More implementation effort; requires CLI command routing in rapid-tools.cjs; additional test surface
- Libraries/services: None new (uses existing fs, path, Zod)

**Recommended:** Option 1 (skill-only). The audit is fundamentally a reasoning task -- matching requirements descriptions against implementation descriptions. A pure skill approach keeps it simple, follows the `/rapid:quick` and `/rapid:new-version` precedents, and avoids the overhead of a new library module for a feature that runs once per milestone. If deterministic harvesting proves necessary later, it can be extracted into a library.

#### Data Model Impact
- New entities/tables needed: None. The audit reads existing STATE.json, REQUIREMENTS.md, ROADMAP.md, and per-set artifacts.
- Existing entities modified: None. The audit is read-only against the state. (Remediation uses existing `/rapid:add-set` to create new sets.)
- New artifact: `.planning/AUDIT-{version}.md` -- the audit report. `.planning/DEFERRED-{version}.md` -- deferred items with carry-forward context.

#### API Surface
- New CLI command: `node "${RAPID_TOOLS}" display banner audit-version` (banner only; the audit logic lives in the skill)
- New skill: `skills/audit-version/SKILL.md`
- Existing endpoints modified: None

### Feature: Discuss Table Formatting

- **Description:** Improve alignment and formatting of pros/cons tables in the discuss-set skill's Step 6 (deep-dive questions). Currently tables rendered via AskUserQuestion can be misaligned.
- **Complexity:** moderate
- **Dependencies:** independent

#### Decomposition

1. **Diagnose alignment issue** -- The AskUserQuestion prompt body is rendered by Claude Code's UI. Markdown tables in the prompt string may not render with proper column alignment if cell content varies in length. Identify whether the issue is in the prompt template (SKILL.md) or in Claude Code's rendering.
2. **Rewrite table format in SKILL.md** -- Update Format A (Option Descriptions) in Step 6 of `skills/discuss-set/SKILL.md` to use a format that renders more consistently
3. **Test with varied content lengths** -- Verify the new format handles short and long option names/descriptions gracefully

#### Implementation Strategies

**Option 1: Padded markdown tables with fixed-width hints**
- Approach: Pad markdown table cells to consistent widths using spaces, and keep cell content concise (under 40 chars per column). Add explicit guidance in SKILL.md about maximum cell content length.
- Pros: Simple change; stays within standard markdown
- Cons: Padding may not survive Claude Code's rendering; still depends on monospace rendering

**Option 2: Replace tables with structured list format**
- Approach: Replace the markdown table in Format A with a structured list format that does not depend on column alignment. For example:
  ```
  **Option A: {name}**
  - Pros: {pros}
  - Cons: {cons}

  **Option B: {name}**
  - Pros: {pros}
  - Cons: {cons}
  ```
- Pros: Lists render reliably regardless of content length; no alignment issues; easier to scan vertically
- Cons: Takes more vertical space; loses the at-a-glance comparison that tables provide

**Option 3: Compact comparison blocks**
- Approach: Use a hybrid format with option name as header and a compact two-column layout:
  ```
  | | A: {name} | B: {name} | C: {name} |
  |---|---|---|---|
  | Pros | {pros} | {pros} | {pros} |
  | Cons | {cons} | {cons} | {cons} |
  ```
  This transposes the table so options are columns (fewer, fixed count) and rows are the fixed categories (Pros/Cons).
- Pros: Fixed number of rows means consistent height; columns correspond to options which is natural for comparison
- Cons: Wide tables if many options; may still have alignment issues with long text

**Recommended:** Option 2 (structured list format). The core problem is that markdown tables depend on monospace rendering and consistent column widths, neither of which is guaranteed in Claude Code's AskUserQuestion UI. A structured list eliminates the alignment problem entirely. The skill already uses lists elsewhere (Format C: Question Context Block), so this is consistent.

#### Data Model Impact
- New entities/tables needed: None
- Existing entities modified: None
- Relationships: N/A

#### API Surface
- New endpoints/commands needed: None
- Existing endpoints modified: `skills/discuss-set/SKILL.md` (Step 6, Format A template)

### Feature: Discuss Gray Areas Consolidation

- **Description:** When more than 4 gray areas exist (n=2 or n=3), consolidate them into fewer AskUserQuestion prompts instead of presenting multiple sequential prompts. For example, 8 gray areas become 2 questions with 4 checkboxes each within a single prompt, rather than 2 separate AskUserQuestion calls.
- **Complexity:** moderate
- **Dependencies:** independent

#### Decomposition

1. **Understand AskUserQuestion capabilities** -- Determine whether AskUserQuestion supports multiple independent question groups within a single call, or whether the consolidation must use a different approach
2. **Redesign gray area presentation logic** -- Rewrite Step 5 of `skills/discuss-set/SKILL.md` to present all gray areas in a single AskUserQuestion call with grouped checkboxes
3. **Update response parsing logic** -- Ensure the skill correctly maps responses back to individual gray areas when they come from a consolidated prompt
4. **Update anti-patterns section** -- Remove/update the anti-pattern about "Do NOT batch multiple questions" since we are now intentionally batching gray area selection (but NOT deep-dive questions)

#### Implementation Strategies

**Option 1: Single multiSelect AskUserQuestion with all gray areas**
- Approach: Present all 8 or 12 gray areas in a single AskUserQuestion with `multiSelect: true`. Group them visually using numbered prefixes or section headers in the option descriptions.
- Pros: Single user interaction instead of 2-3; simpler flow; user sees all options at once for better decision-making
- Cons: 12 checkboxes in one prompt may feel overwhelming; no visual grouping beyond option text

**Option 2: Restructured prompt with category headers**
- Approach: Use a single AskUserQuestion but organize options by category (architecture, API design, state management, etc.) using the option description field to indicate grouping. For 8 gray areas: present as 8 options in one multiSelect prompt with category prefixes like "[Architecture] {title}" and "[API] {title}".
- Pros: Single interaction; categorical grouping aids scanning; user sees full picture
- Cons: Category prefixes add visual noise; still a long list for n=3

**Recommended:** Option 1 (single multiSelect with all gray areas). The user's request is specifically to reduce the number of AskUserQuestion prompts. A single multiSelect prompt with 8 or even 12 options is standard UI (comparable to a settings page). The current batching into groups of 4 was an arbitrary constraint that adds friction without clear benefit. The skill should present ALL gray areas in one prompt regardless of count, with clear numbering for reference.

- Existing entities modified: None

### Bug Fix: context.cjs Module Path Resolution

- **Description:** During `/rapid:init`, the path `${RAPID_TOOLS}/../lib/context.cjs` fails because Node.js treats `rapid-tools.cjs` (a file) as a directory when resolving `..`. The path `rapid-tools.cjs/../lib/context.cjs` resolves to `src/bin/lib/context.cjs` instead of the intended `src/lib/context.cjs`.
- **Complexity:** trivial
- **Dependencies:** independent

#### Decomposition

1. **Fix path resolution in init SKILL.md** -- Replace `require('${RAPID_TOOLS}/../lib/context.cjs')` with `require(require('path').join(require('path').dirname('${RAPID_TOOLS}'), '..', 'lib', 'context.cjs'))` or use a shell-level `dirname` approach
2. **Fix identical pattern in register-web SKILL.md** -- Same bug exists for `web-client.cjs` requires
3. **Add a CLI command for module resolution** -- Optionally, add a `node "${RAPID_TOOLS}" resolve module context` command that returns the correct absolute path, avoiding the pattern entirely

#### Root Cause Analysis

`RAPID_TOOLS` is set to `/home/kek/Projects/RAPID/src/bin/rapid-tools.cjs` (a file path). When used in `require('${RAPID_TOOLS}/../lib/context.cjs')`, Node's `require()` resolves `..` relative to the full string path. Since `rapid-tools.cjs` is a file (not a directory), `rapid-tools.cjs/..` does NOT navigate to `src/bin/` -- instead it treats `rapid-tools.cjs` as a directory component, yielding `src/bin/rapid-tools.cjs/../lib/context.cjs` which resolves to `src/bin/lib/context.cjs` (non-existent).

Confirmed affected locations:
- `skills/init/SKILL.md` line 564: `require('${RAPID_TOOLS}/../lib/context.cjs')` for detectTestFrameworks
- `skills/init/SKILL.md` line 1000: `require('${RAPID_TOOLS}/../lib/web-client.cjs')` for web registration
- `skills/register-web/SKILL.md` line 22 and 44: `require('${RAPID_TOOLS}/../lib/web-client.cjs')`

#### Implementation Strategies

**Option 1: Shell dirname approach**
- Approach: Use shell `dirname` to strip the filename before constructing the path:
  ```bash
  node -e "
    const { detectTestFrameworks } = require(require('path').join(require('path').dirname('${RAPID_TOOLS}'), '..', 'lib', 'context.cjs'));
    ...
  "
  ```
- Pros: Correct path resolution; self-contained in each usage site
- Cons: Verbose; repeated pattern across multiple skills

**Option 2: RAPID_LIB environment variable**
- Approach: In `setup.sh` and `.env`, derive and export `RAPID_LIB` pointing to `src/lib/`. Skills use `require('${RAPID_LIB}/context.cjs')` directly.
- Pros: Clean usage pattern; one fix propagates everywhere; no path arithmetic in skills
- Cons: Requires setup.sh change and re-install; adds another env var

**Option 3: CLI command to expose lib path**
- Approach: Add `node "${RAPID_TOOLS}" resolve lib-path` that prints the absolute path to `src/lib/`. Skills capture this and use it for requires.
- Pros: Uses existing CLI pattern; no new env vars; self-correcting if installation moves
- Cons: Requires an extra CLI call before the node -e invocation; two-step pattern

**Recommended:** Option 1 (inline path.dirname) for the immediate fix, as it requires no infrastructure changes and can be applied to each affected SKILL.md independently. Option 2 could be considered as a follow-up improvement but is out of scope for a bug fix.

#### API Surface
- New endpoints/commands needed: None
- Existing endpoints modified: `skills/init/SKILL.md`, `skills/register-web/SKILL.md` (path patterns in inline node -e scripts)

## Audit Version -- Detailed Design Notes

The GSD audit-milestone workflow provides strong inspiration but must be adapted to RAPID's architecture:

### GSD Concepts to Adopt
- **Requirements traceability matrix**: Cross-reference REQUIREMENTS.md entries against per-set VERIFICATION-REPORT.md and SET-OVERVIEW.md deliverables
- **Three-source cross-reference**: Check requirements in (1) REQUIREMENTS.md, (2) per-set verification artifacts, (3) ROADMAP.md set descriptions
- **Structured output artifact**: Write `.planning/AUDIT-{version}.md` with YAML frontmatter, scores, gaps, and tech debt sections
- **Remediation routing**: Present three outcome paths (passed, gaps found, tech debt only) with clear next-step commands

### GSD Concepts to Adapt
- **Phase VERIFICATION.md**: RAPID uses per-set VERIFICATION-REPORT.md (when review was run) and WAVE-COMPLETE.md files. Not all sets will have verification reports.
- **Subagent for cross-phase wiring**: RAPID can use a single-pass model analysis since sets have explicit CONTRACT.json defining interfaces
- **Nyquist compliance**: Not applicable to RAPID; skip entirely
- **REQ-ID traceability**: RAPID's REQUIREMENTS.md uses checkbox-style requirements (e.g., `[x] STATE-01`). The audit should parse these markers.

### Proposed Audit Flow
1. Read STATE.json to enumerate all sets and their statuses
2. Read REQUIREMENTS.md to build requirements inventory (parse `[x]`/`[ ]` markers and REQ-IDs)
3. Read ROADMAP.md for set descriptions and scope
4. For each set: read DEFINITION.md, SET-OVERVIEW.md, VERIFICATION-REPORT.md (if exists)
5. Cross-reference: which requirements map to which sets? Which were marked complete? Which have no evidence of implementation?
6. Generate gap report with categories: satisfied, partial, unsatisfied, orphaned
7. Present to user via AskUserQuestion with remediation options:
   - "Plan remediation sets" -- use `/rapid:add-set` to create new sets for gaps
   - "Defer to next version" -- write DEFERRED-{version}.md with context
   - "Accept as-is" -- mark audit complete without remediation
8. Write `.planning/AUDIT-{version}.md` with full report

### Discuss Consolidation -- Detailed Design Notes

The current SKILL.md Step 5 uses a batching model that presents gray areas in groups of 4 via separate AskUserQuestion calls. The consolidation changes this to:

**Before (n=2, 8 gray areas):**
- AskUserQuestion 1: "Gray Areas (1 of 2)" with options 1-4
- AskUserQuestion 2: "Gray Areas (2 of 2)" with options 5-8

**After (n=2, 8 gray areas):**
- AskUserQuestion 1: Single prompt with all 8 options, multiSelect: true

The SKILL.md anti-pattern "Do NOT batch multiple questions into a single freeform AskUserQuestion" must be updated. This anti-pattern was about deep-dive questions (Step 6), not gray area selection (Step 5). The updated anti-pattern should clarify: "Do NOT batch deep-dive questions into a single AskUserQuestion -- each deep-dive question gets its own AskUserQuestion. Gray area selection (Step 5) uses a single multiSelect prompt."
