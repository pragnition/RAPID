### Zod (3.25.76 -> 4.3.6 available)

**Current usage pattern:**
The project uses Zod in a focused, minimal way -- only in `state-schemas.cjs`:
- `z.enum()` for `SetStatus` with string array input
- `z.object()` with `.default()` for `SetState`, `MilestoneState`, `ProjectState`
- `z.literal(1)` for version pinning
- `.parse()` and `.safeParse()` throughout `state-machine.cjs` and tests

**Zod 4 breaking changes relevant to this project (confirmed via Context7 docs):**

1. **`z.enum()` overloaded**: Now supports native TS enums directly. However, `z.enum(['pending', 'discussing', ...])` with a string array still works in Zod 4 -- this is NOT a breaking change for the project's usage.

2. **`.default()` semantics changed**: In Zod 4, `.default()` short-circuits parsing and requires the default value to match the output type (not input type). The project uses `.default('pending')` on SetStatus and `.default([])` on arrays -- both match output types, so no breakage expected.

3. **`.Enum` and `.Values` removed**: The project does NOT use these properties -- no impact.

4. **Error issue types renamed**: `ZodError` name remains, but issue type internals changed. The project's test `state-schemas.test.cjs` checks `{ name: 'ZodError' }` which still works. However, tests checking `result.error.issues[].path` may need minor adjustments if issue shapes changed.

5. **`.strict()` and `.passthrough()` deprecated**: Not used in the project -- no impact.

6. **`._def` moved to `._zod.def`**: Not used in the project -- no impact.

**Assessment:** Zod upgrade to v4 is NOT required for this milestone. The current `z.enum()` with string arrays, `.parse()`, `.safeParse()`, `.default()`, `z.object()`, and `z.literal()` all work identically in Zod 4. An upgrade is feasible but should be deferred to a dedicated cleanup milestone to avoid scope creep. A community codemod `zod-v3-to-v4` exists for future use.

## Stack Risks

### Risk 1: String-Based Status Rename Scope (Goal 1)
- **Risk:** The status values (`discussing`, `planning`, `executing`) appear as raw strings in 662 occurrences across 51 files. A partial rename risks runtime failures from missed occurrences.
- **Impact:** High -- state machine validation will reject any unrenamed status string, causing hard failures.
- **Mitigation:** The Zod schema (`SetStatus`) acts as a runtime validator. After renaming the schema, any code path using an old status string will fail at `parse()`/`safeParse()` time, making missed renames detectable. Run the full test suite after rename to catch all failures. Use systematic find-and-replace rather than ad-hoc edits.

### Risk 2: Parallel Git Commit Serialization (Goal 2)
- **Risk:** Git is not concurrency-safe. Multiple concurrent waves writing to the same worktree and running `git commit` simultaneously will corrupt the repository.
- **Impact:** Critical -- data loss, corrupted git history.
- **Mitigation:** The existing `acquireLock` mechanism in `lock.cjs` is designed for exactly this. Parallel wave execution must serialize all git operations through the lock. The lock's 5-minute stale threshold and exponential backoff are adequate. Each wave should run in its own worktree (already the architecture), and git operations within a single worktree must be serialized.

### Risk 3: State Machine Transition Integrity During Parallel Execution (Goal 2)
- **Risk:** Multiple parallel wave agents transitioning the same set's state concurrently could cause race conditions in `STATE.json`.
- **Impact:** Medium -- corrupted state, lost transitions.
- **Mitigation:** `withStateTransaction()` in `state-machine.cjs` already uses `acquireLock('state')` to serialize reads and writes. The atomic write pattern (write to `.tmp` then `rename`) prevents partial writes. This is sufficient for parallel wave execution as long as all state mutations go through this function.

### Risk 4: proper-lockfile Dormancy (Long-term)
- **Risk:** The lockfile library has not been updated since January 2021. Future Node.js versions or OS changes could introduce compatibility issues.
- **Impact:** Low (for this milestone) -- the library works correctly on Node 25 today.
- **Mitigation:** Monitor for issues. If problems arise, the lock module (`lock.cjs`) is well-encapsulated (4 exported functions, 133 lines) and could be replaced with a custom mkdir-based lock or an alternative library with minimal effort.

### Risk 5: Test Coverage for Renamed Status Values (Goal 1)
- **Risk:** Tests explicitly assert on status string values (e.g., `assert.equal(SetStatus.parse('discussing'), 'discussing')`). Every such assertion must be updated.
- **Impact:** Medium -- tests will fail but won't cause production issues.
- **Mitigation:** The test files contain approximately 200+ occurrences of the old status strings. Systematic find-and-replace across test files in the same pass as source files will handle this.

## Recommendations

1. **Use systematic string replacement for status rename (Goal 1):** Replace `discussing` -> `discussed`, `planning` -> `planned`, `executing` -> `executed` across all 51 affected files. Start with the source-of-truth files (`state-schemas.cjs`, `state-transitions.cjs`), then propagate to consumers and tests. Run full test suite to verify completeness.
   - **Priority:** Critical (core milestone goal)

2. **Serialize git operations in parallel wave execution (Goal 2):** Reuse the existing `acquireLock` mechanism for git commit serialization. Each concurrent wave agent should acquire the lock before any git operation and release after. The existing retry/stale configuration is adequate.
   - **Priority:** Critical (core milestone goal)

3. **Do NOT upgrade Zod for this milestone:** Zod 3.25.76 works correctly and the status rename changes only enum values, not Zod API usage. Deferring the Zod 4 upgrade avoids introducing unrelated risk.
   - **Priority:** Low (defer to future milestone)

4. **Keep proper-lockfile for this milestone:** The library works correctly for the parallel execution use case. Its encapsulation in `lock.cjs` makes future replacement straightforward if needed.
   - **Priority:** Low (monitor only)

5. **Verify setup.sh install detection after changes (Goal 4):** The install version-awareness feature should compare the installed `RAPID_TOOLS` path's package version against the current source's `package.json` version. The version field in `package.json` (currently `3.0.0`) provides the comparison value. Update to `3.1.0` as part of this milestone release.
   - **Priority:** High (milestone goal)

6. **No new dependencies required:** All four milestone goals can be achieved with the existing dependency set. Parallel wave execution uses `child_process` (built-in) and the existing lock module. The status rename is a pure refactor. Old command cleanup is deletion. Install versioning reads `package.json`.
   - **Priority:** Informational
