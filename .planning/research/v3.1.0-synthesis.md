# Research Summary -- v3.1.0 Polish & Cleanup

## Project Profile
- **Type:** brownfield plugin (Claude Code CLI plugin for parallel development)
- **Stack:** Node.js (v25.8.0, min v18) + CommonJS (.cjs) + Zod 3.25.x + proper-lockfile 4.1.2
- **Scale:** medium (4 features, ~73 affected files, ~800 string occurrences to change)
- **Key Constraint:** The status rename is a cross-cutting change touching ~80 files and ~800 occurrences; partial completion causes hard runtime crashes via Zod validation failures

## Unified Findings

### Stack and Infrastructure

- Node.js v25.8.0 (non-LTS "Current" release) is fully compatible with all dependencies and milestone goals; no version constraints block any work (source: STACK)
- CommonJS module system is used exclusively across 48+ source files; no ESM migration needed; concurrent dispatch should use `child_process` or `Promise.all` patterns (source: STACK)
- Zod 3.25.76 is one major version behind (v4.3.6 available) but the project's usage pattern (`z.enum`, `z.object`, `.parse`, `.safeParse`, `.default`) is fully compatible with Zod 4; upgrade is feasible but NOT required for this milestone and should be deferred (source: STACK)
- proper-lockfile 4.1.2 is at latest version but effectively dormant (last release Jan 2021, 15 open issues); the library works correctly for current use cases and is well-encapsulated in `lock.cjs` (133 lines, 4 exports) for easy future replacement (source: STACK, OVERSIGHTS)
- ajv 8.18.0 + ajv-formats 3.0.1 are at latest versions; no changes needed (source: STACK)
- No build step exists (raw .cjs files); `build-agents` assembles agent .md files from `src/modules/` templates and MUST be re-run after any status rename or command removal (source: STACK, OVERSIGHTS)
- No CI pipeline, linter, or formatter configured; tests run manually via `node --test` (source: STACK)
- No new dependencies are required for any of the four milestone goals (source: STACK, FEATURES)

### Feature Landscape

**Feature 1: Set Status Naming Fix** (moderate complexity)
- Rename `discussing` -> `discussed`, `planning` -> `planned`, `executing` -> `executed` across the entire codebase (source: FEATURES)
- Touches ~73 files with ~800 occurrences: 17 source libs, 15 test files, 1 CLI, 19 skills, 13 agent modules, 1 hook (source: FEATURES, OVERSIGHTS)
- Two naming systems must be renamed in sync: SetStatus (lowercase) and Registry phases (PascalCase in worktree.cjs and rapid-tools.cjs) (source: PITFALLS, OVERSIGHTS)
- Existing STATE.json files need migration; without it, projects are bricked on upgrade (source: PITFALLS, OVERSIGHTS, ARCHITECTURE)
- Recommended approach: Big-bang find-and-replace with schema-first ordering (source: FEATURES, ARCHITECTURE)

**Feature 2: Parallel Wave Execution** (complex)
- Enable concurrent execution of independent waves within a set (source: FEATURES)
- Current architecture is strictly sequential: one executor per wave, waves processed in order (source: ARCHITECTURE)
- DAG module already supports wave assignment via BFS level grouping but only at the set level, not intra-set (source: ARCHITECTURE, FEATURES)
- Four implementation strategies analyzed; Option 4 (wave-level plan splitting) recommended for clean isolation, with Option 3 (set-level parallelism only) as pragmatic fallback (source: FEATURES)
- Pre-existing blocking bug: `transitionWave()` and `transitionJob()` called by CLI but do not exist in state-machine.cjs (tests explicitly assert they are `undefined`) (source: OVERSIGHTS)
- No WaveState or JobState Zod schemas exist; wave/job state is unvalidated (source: OVERSIGHTS)

**Feature 3: Old Command Cleanup** (trivial complexity)
- Remove 6 deprecated skill directories: `set-init`, `discuss`, `execute`, `plan`, `wave-plan`, `new-milestone` (source: FEATURES, ARCHITECTURE)
- Update help skill to remove "Deprecated Commands" section (source: FEATURES, ARCHITECTURE, UX)
- Remove legacy stage entries from `display.cjs` (source: FEATURES, ARCHITECTURE)
- TOOL_REGISTRY in tool-docs.cjs still lists `wave-plan` commands that must also be removed, along with corresponding ROLE_TOOL_MAP entries (source: OVERSIGHTS)
- Recommended approach: full removal (no redirects); users had one full version cycle to migrate (source: FEATURES, UX)

**Feature 4: Install Command Flexibility** (moderate complexity)
- Make install version-aware by adding `--version` CLI flag and comparing against cached version (source: FEATURES, ARCHITECTURE)
- Current install only checks RAPID_TOOLS env var presence, not version currency (source: FEATURES, ARCHITECTURE)
- `prereqs.cjs` already has `compareVersions()` utility that can be reused (source: ARCHITECTURE)
- Store `RAPID_VERSION` in `.env` alongside `RAPID_TOOLS` path (source: FEATURES, OVERSIGHTS)
- Must handle edge cases: stale paths, missing binaries, old CLI without `--version` flag (source: PITFALLS, OVERSIGHTS)
- setup.sh skips `npm install` if `node_modules` exists; version update may need forced reinstall (source: OVERSIGHTS)
- build-agents regeneration must be triggered after version-aware update (source: OVERSIGHTS)

**Recommended Implementation Order** (source: FEATURES):
1. Set Status Naming Fix -- do first to avoid merge conflicts with other features
2. Old Command Cleanup -- quick win, reduces noise
3. Install Command Flexibility -- moderate, self-contained
4. Parallel Wave Execution -- most complex, do last

### Architecture Direction

- **Pattern: Layered library with skill orchestration.** CLI entry point dispatches to library modules; skills are Markdown instruction files consumed by Claude Code agents; state lives in Zod-validated JSON with lock-protected transactions (source: ARCHITECTURE)
- **State machine design is clean and sufficient:** Zod enum -> transition map -> validation -> lock-protected atomic writes. No architectural changes needed for the rename. (source: ARCHITECTURE)
- **Status rename follows schema-first migration pattern:** Change canonical definitions first (Zod enum, transition map), then update consumers layer by layer. Tests guide the consumer updates by failing on each missed occurrence. (source: ARCHITECTURE)
- **Parallel wave execution should use simplified parallel dispatch:** Keep parallelism logic in SKILL.md using Claude Code Agent tool's built-in parallel invocations. Serialize git operations via existing lock system. No new infrastructure needed for v3.1.0. (source: ARCHITECTURE)
- **Old command cleanup is direct deletion with doc updates:** Zero runtime risk; `disable-model-invocation: true` stubs have no behavioral dependencies (source: ARCHITECTURE)
- **Install version detection should use --version CLI flag:** Most robust approach; follows standard CLI conventions; combined with version stamping in `.env` (source: ARCHITECTURE, FEATURES)
- **Module boundaries are well-defined and require no restructuring** for this milestone (source: ARCHITECTURE)

### Cross-Cutting Requirements

**Must address early (high retrofit cost):**
- STATE.json migration layer for status rename -- must be in same commit as the rename; without it every active project breaks on upgrade (source: OVERSIGHTS, PITFALLS)
- Atomic rename across all files in single commit -- partial rename causes immediate runtime crashes via Zod validation (source: OVERSIGHTS)
- Fix transitionWave/transitionJob bug before parallel wave work -- CLI calls non-existent functions; parallel waves cannot work without this (source: OVERSIGHTS)
- Rename both SetStatus (lowercase) and Registry phases (PascalCase) together -- divergence causes silent failures in `deriveNextActions()` and `update-phase` (source: PITFALLS, OVERSIGHTS)
- TOOL_REGISTRY cleanup for removed commands -- same commit as command removal; `getToolDocsForRole()` throws on missing keys (source: OVERSIGHTS)

**Can address later (low retrofit cost):**
- build-agents re-run after changes -- forgetting leaves stale agent prompts but is trivially fixable (source: OVERSIGHTS)
- Version display in status/help output -- cosmetic enhancement (source: UX)
- Error message UX for past-tense statuses -- reads oddly but is not a bug (source: OVERSIGHTS)
- node_modules staleness check during version-aware install -- only matters for users updating RAPID (source: OVERSIGHTS)
- STAGE_VERBS/STAGE_BG banner text -- these use command names (discuss, plan, execute) not status names, so likely unaffected; verify during implementation (source: OVERSIGHTS, UX)

### User Experience Direction

- **Past tense is the universal industry standard for completed states:** Kubernetes, GitHub Actions, Terraform, Git, Docker all use past tense for completed/result states; `-ing` only for active in-progress work. RAPID's current naming contradicts every major developer tool. (source: UX)
- **Status-to-action mapping becomes natural with past tense:** `discussed` clearly communicates "done, ready for planning" while `discussing` falsely implies ongoing work. The rename eliminates a repeated cognitive tax at every status display. (source: UX)
- **Deprecation stub removal aligns with CLI convention:** One major version cycle is the standard transition period (npm, Docker, kubectl all follow this pattern). Stubs add noise to the command surface and violate the 7+4 progressive disclosure target from v3.0. (source: UX)
- **Command surface shrinks from 23 to 17 skills** (~26% reduction in cognitive load) after removing 6 deprecated stubs (source: UX)
- **Version-aware install is expected behavior:** Every modern CLI tool (pnpm, yarn, brew) detects version mismatches; users expect `/rapid:install` to be idempotent and version-aware (source: UX)
- **Parallel wave progress display should show per-wave status with dependency awareness:** Table format showing which waves are parallel vs sequential, with status updates as each wave completes (source: UX)
- **Partial parallel failure should preserve completed work and offer targeted retry:** Matches CI/CD expectations (GitHub Actions `fail-fast: false` pattern) (source: UX)
- **RAPID version should be visible in status/help output** after version-aware install feature is added (source: UX)

## Contradictions

### Contradiction: STATE.json Migration Strategy
- **ARCHITECTURE says:** "Hard break (recommended). Old STATE.json files fail validation. Users re-run /rapid:init for new milestones. This is acceptable for a minor version bump."
- **PITFALLS says:** "STATE.json migration on status rename -- Existing projects will brick if old status values are not auto-migrated before Zod validation. This MUST be implemented before or alongside the rename."
- **OVERSIGHTS says:** "STATE.json migration layer (MUST DO FIRST): Before renaming any status values, implement a migration function in readState() that maps old values before Zod parsing. Without this, the rename is a breaking change with no recovery path."
- **Resolution:** PITFALLS and OVERSIGHTS are correct. A migration layer is essential. The ARCHITECTURE recommendation of "hard break" underestimates user impact -- an active project whose STATE.json suddenly fails Zod validation becomes unreadable with no self-service recovery. The migration is ~10 lines of code (string map before Zod parse) and prevents bricking active projects. Implement the migration.

### Contradiction: Parallel Wave Execution Implementation Strategy
- **FEATURES says:** "Recommended: Option 4 (Wave-Level Plan Splitting with separate worktrees per sub-plan and branch merge). However, the pragmatic first step might be Option 3 (set-level parallelism only)."
- **ARCHITECTURE says:** "Recommended: Simplified Parallel Dispatch (keep parallelism logic in SKILL.md, serialize git operations via existing lock, no new infrastructure). Strategy A for v3.1.0."
- **PITFALLS says:** "Concurrent git operations in the same worktree will corrupt the index. A locking mechanism is mandatory."
- **OVERSIGHTS says:** "Pre-existing blocker: transitionWave/transitionJob do not exist. Wave/job state schemas do not exist. These must be fixed before parallel waves can work."
- **Resolution:** The ARCHITECTURE recommendation (simplified parallel dispatch in SKILL.md with git lock serialization) is the most pragmatic for v3.1.0 scope. FEATURES Option 4 (separate worktrees per wave) is overengineered for a first implementation. However, OVERSIGHTS correctly identifies blocking prerequisites (missing functions, missing schemas) that must be addressed first. The recommended approach is: (1) fix blocking prerequisites, (2) implement simplified parallel dispatch per ARCHITECTURE, (3) defer wave-level plan splitting to a future version if needed.

### Contradiction: Display Module Changes
- **FEATURES says:** "Remove legacy stage names from STAGE_VERBS and STAGE_BG in display.cjs (entries for set-init, discuss, wave-plan, execute)."
- **OVERSIGHTS says:** "STAGE_VERBS and STAGE_BG in display.cjs use command names (discuss, execute, etc.) not status names, so these are unaffected."
- **UX says:** "Banners describe the action being performed ('DISCUSSING SET') which is present progressive by nature. The rename only affects status values, not banner verbs. Verify during implementation."
- **Resolution:** Both FEATURES and OVERSIGHTS are partially correct about different aspects. The LEGACY entries for removed commands (`set-init`, `discuss`, `wave-plan`, `execute` -- the old v2 command names) should be removed as part of command cleanup. The CURRENT entries (`discuss-set`, `plan-set`, `execute-set`) should keep their present-progressive banner verbs since banners describe active actions, not status results. The distinction is: remove entries for deleted commands, keep entries for active commands unchanged.

## Recommended Approach

The v3.1.0 milestone has a clear critical path: the status rename is the most impactful and highest-risk change, touching ~73 files with ~800 occurrences across every layer of the codebase. It should be implemented first because (a) it affects nearly every file, creating merge conflict risk with parallel work, and (b) it is mechanically straightforward despite its breadth -- systematic find-and-replace guided by the test suite. The key prerequisite is implementing a STATE.json migration layer (~10 lines) that maps old status values to new ones before Zod parsing, preventing active projects from breaking on upgrade. The rename must change both the lowercase SetStatus values AND the PascalCase Registry phase values in a single atomic commit. The recommended ordering within the rename is: schema (state-schemas.cjs) -> transitions (state-transitions.cjs) -> state machine -> library modules -> tests -> CLI -> skills -> agent modules -> build-agents regeneration. Special care must be taken to NOT rename: the `.planning/` directory path, phase/command names (discuss, plan, execute), STAGE_VERBS banner text, merge pipeline states (merging in MergeStateSchema), or natural language prose.

Old command cleanup is the simplest change -- deleting 6 deprecated skill directories, removing corresponding TOOL_REGISTRY and ROLE_TOOL_MAP entries, updating the help skill, and removing legacy display.cjs entries. This yields an immediate UX improvement (23 -> 17 visible skills, 26% cognitive load reduction) with zero runtime risk. Install version awareness is a moderate, self-contained change: add --version CLI flag, store RAPID_VERSION in .env, compare versions in the install skill. The `prereqs.cjs` module already provides `compareVersions()` for reuse.

Parallel wave execution is the most complex feature and should be implemented last. Two pre-existing bugs must be fixed first: `transitionWave()`/`transitionJob()` not existing despite CLI calls, and the review skill's reference to a non-existent `reviewing` status. The recommended implementation for v3.1.0 is the simplified approach: keep parallelism logic in execute-set SKILL.md, use Agent tool's built-in parallel invocations for independent waves, serialize git operations through the existing lock system. Wave-level plan splitting with separate worktrees (FEATURES Option 4) should be deferred to a future version if the simplified approach proves insufficient. The payoff depends on how often plans produce genuinely independent waves -- if most waves are inherently sequential (wave 2 builds on wave 1), the practical benefit of parallel wave execution may be limited.

All four features can be accomplished with zero new dependencies. The existing stack (Node.js 25, Zod 3, proper-lockfile 4, ajv 8) is fully sufficient. The Zod 4 upgrade should be explicitly deferred to avoid scope creep.

### Suggested Set Boundaries

- **Set A (Status Rename + Migration):** state-schemas.cjs, state-transitions.cjs, state-machine.cjs, worktree.cjs (STATUS_SORT_ORDER, deriveNextActions, phase map), rapid-tools.cjs (validPhases), display.cjs (only legacy entries if overlapping with cleanup), all test files, all skill files (status strings in bash commands), all agent module files, build-agents regeneration. Includes STATE.json migration layer and fix for pre-existing `reviewing`/`merging` dead code. -- Rationale: cross-cutting change that must be atomic; doing it as one set avoids partial-rename states and merge conflicts with all other sets.

- **Set B (Old Command Cleanup):** Delete 6 deprecated skill directories, remove TOOL_REGISTRY entries for wave-plan commands, update ROLE_TOOL_MAP, remove "Deprecated Commands" section from help skill, remove legacy STAGE_VERBS/STAGE_BG entries from display.cjs. -- Rationale: purely additive (deletion) with no dependencies on other sets; can run in parallel with Set C.

- **Set C (Install Version Awareness):** Add --version handler to rapid-tools.cjs, add RAPID_VERSION to .env, update setup.sh with version stamping and forced npm install on version change, update install skill with version comparison step, bump package.json to 3.1.0. -- Rationale: self-contained feature touching install/setup infrastructure; no overlap with other sets.

- **Set D (Parallel Wave Execution):** Fix transitionWave/transitionJob (implement or remove), add wave independence detection to execute.cjs or execute-set SKILL.md, modify execute-set SKILL.md Step 4 for parallel dispatch, add git commit serialization, add per-wave progress display, handle partial failure and re-entry. -- Rationale: most complex feature; depends on Set A completing first (execute-set SKILL.md is modified in both); should be the last set executed.

**Dependency graph:** Set A should complete first (avoids merge conflicts). Sets B and C can run in parallel with each other (and potentially in parallel with Set A if file ownership is carefully managed). Set D depends on Set A (shared file: execute-set SKILL.md) and should run last.
