## Upgrade Paths

### Zod: v3.25.76 -> v4.x
- **Recommendation:** Do NOT upgrade for v3.2.0. Stay on 3.25.76.
- **Rationale:** v4 has breaking changes in error customization APIs, `.default()` behavior with optional fields, and string format methods. The project uses `z.enum()`, `z.object()`, `z.array()`, `z.string()`, `z.literal()`, and `.default()` extensively in `state-schemas.cjs`. The `.default()` behavior change (defaults applied even within optional fields) could cause subtle state validation bugs.
- **Breaking changes that would affect this project:**
  1. `.default()` inside optional fields now always applies -- impacts `WaveState`, `SetState`, `MilestoneState`, `ProjectState` schemas
  2. Error customization API changes -- impacts any `.safeParse()` error handling in `state-machine.cjs`
  3. CJS `require('zod/v4')` has known resolution issues
- **Migration effort:** Moderate. Would require auditing all 4 files that use Zod, updating `.default()` semantics, and verifying all 21+ test files still pass.
- **Priority:** Low. v3.25.76 is the latest on its line and receives security fixes.

### proper-lockfile: 4.1.2 (no upgrade available)
- **Status:** No newer version exists. The package is functionally complete but minimally maintained.
- **Risk:** If a Node.js future version changes `fs.mkdir` or `fs.realpath` behavior, locking could break silently.
- **Alternative:** Consider `lockfile` (npm) or implementing a simple `fs.mkdirSync`-based lock (the project already has `lock.cjs` as a wrapper). However, no action needed for v3.2.0.

### ajv / ajv-formats: No upgrade needed
- Both at latest versions. Fully current.

## Stack Risks

Ordered by relevance to the v3.2.0 milestone:

1. **Ghost `wave-plan` commands in tool-docs.cjs:** The `TOOL_REGISTRY` in `tool-docs.cjs` (lines 66-69) defines 4 `wave-plan-*` commands (`resolve-wave`, `create-wave-dir`, `validate-contracts`, `list-jobs`), but `rapid-tools.cjs` has NO handler for the `wave-plan` command group (confirmed: grep returns 0 matches in rapid-tools.cjs). The `plan-verifier` role references `wave-plan-validate` (line 126). This directly causes the "agents call commands that don't exist" bug (milestone goal 5) and the "unknown wave-plan command" error (milestone goal 8). **Impact: High** -- agents given these tool docs will attempt nonexistent commands and fail.

2. **Set name resolution depends on filesystem sort order:** `plan.listSets()` reads `.planning/sets/` directory entries and sorts alphabetically. When sets are archived (moved out of `.planning/sets/`), the remaining sets shift indices. If a user references "set 6" by numeric index, it resolves to a different set after archiving. This is the root cause of milestone goal 1. **Impact: High** -- the resolution algorithm has no concept of stable IDs vs. positional indices.

3. **State transition map missing intermediate states:** `SET_TRANSITIONS` allows `pending -> discussed -> planned -> executed -> complete -> merged`. There is no `discussing` state in the enum (`SetStatus` is `['pending', 'discussed', 'planned', 'executed', 'complete', 'merged']`). If an agent attempts `discussing -> complete` (milestone goal 8), it will fail because `discussing` is not a valid status in the schema. The question is whether agents are generating a `discussing` status that doesn't exist in the Zod enum, or whether the transition map needs a `discussing` intermediate state. **Impact: Medium** -- this is a schema/transition map design question.

4. **proper-lockfile maintenance risk:** The package has not been updated in ~5 years with 15 open issues. While it works today, it represents a long-term risk if Node.js changes filesystem semantics. **Impact: Low for v3.2.0** -- no immediate issues.

5. **No Zod v4 migration pressure:** The project is pinned to 3.25.76 with `^3.25.76` which correctly resolves to the latest 3.x. There is no urgency to migrate, but the ecosystem (libraries, tools) will gradually move to v4. **Impact: Low for v3.2.0** -- no action needed.

6. **ANSI color codes use basic 16-color palette only:** The `display.cjs` module uses `\x1b[104m` (bright blue background) for planning stages. The milestone requests dark purple, which requires 256-color mode (`\x1b[48;5;54m` for dark purple background). The current architecture uses only the basic ANSI palette, meaning a change to 256-color codes may not render correctly on all terminals. **Impact: Low** -- most modern terminals support 256-color mode, but the change breaks the "maximum terminal compatibility" design goal stated in the display.cjs header comment.

## Recommendations

Ordered by relevance to the v3.2.0 milestone goals:

1. **Remove or implement `wave-plan` commands:** Either remove the 4 `wave-plan-*` entries from `TOOL_REGISTRY` and the `plan-verifier` reference in `ROLE_TOOL_MAP`, or implement the actual command handlers in `rapid-tools.cjs`. Given v3.0 eliminated wave-level planning as a separate concept, removal is the correct path. -- **Priority: Critical** (directly blocks goals 5, 8)

2. **Add `discussing` status to state machine or fix agent prompts:** Either add `discussing` as a valid `SetStatus` enum value with appropriate transitions (`discussing -> discussed`, `discussing -> planned`), or ensure agent prompts never emit a `discussing` status. Check whether agents are attempting to transition to/from states that don't exist in the Zod schema. -- **Priority: Critical** (directly blocks goal 8)

3. **Stabilize set name resolution against archival:** The `resolveSet()` function in `resolve.cjs` uses positional indexing into alphabetically-sorted directory listings. Consider one of: (a) storing a stable numeric-to-name mapping in STATE.json, (b) using the set's `id` field from the milestone's `sets[]` array in STATE.json instead of filesystem enumeration, or (c) emitting a warning when numeric indices are used after archival changes the set count. -- **Priority: High** (directly blocks goal 1)

4. **Use 256-color ANSI for dark purple banner:** Change `STAGE_BG` entries from `\x1b[104m` (bright blue) to `\x1b[48;5;54m` (dark purple) or `\x1b[48;5;55m` (slightly lighter purple) for the desired visual. Test across common terminals (iTerm2, Terminal.app, Windows Terminal, Alacritty). The `display.cjs` header comment about "basic 16-color palette" should be updated to reflect the new approach. -- **Priority: Medium** (goal 3)

5. **Audit all role modules for command references:** Cross-reference every `rapid-tools` command mentioned in `src/modules/roles/*.md` agent prompts against the actual USAGE string in `rapid-tools.cjs` and the `TOOL_REGISTRY` in `tool-docs.cjs`. Any command referenced in a prompt that doesn't exist in rapid-tools.cjs is a guaranteed runtime failure. -- **Priority: High** (goal 5)

6. **No dependency upgrades needed for v3.2.0:** All dependencies are at their latest compatible versions. Zod 3.25.76, ajv 8.18.0, ajv-formats 3.0.1, and proper-lockfile 4.1.2 are all current on their respective lines with zero known vulnerabilities. -- **Priority: None** (informational)

7. **State transition enforcement is a code-level fix, not a stack concern:** The v3.2.0 goals around state update enforcement (goal 7) and state transition fixes (goal 8) are implementable within the existing `state-transitions.cjs` and agent prompt system. No new dependencies or tooling changes are required. -- **Priority: Informational**
