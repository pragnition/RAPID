# Oversights Research -- v7.0.0 Mission Control Autopilot

## Preamble

This milestone moves RAPID from a CLI-driven, human-in-the-loop tool into a partially-autonomous, browser-orchestrated system. The orchestration locus shifts from the Claude Code CLI harness (which owns the terminal, a single cwd, an implicit OAuth session, and the user's attention) to a FastAPI backend that spawns `ClaudeSDKClient` sessions on behalf of a browser user who may have several tabs open, several agents running, and zero terminal context.

Every assumption baked into the current skills -- "a human is reading this output", "`cwd` is the worktree", "`RAPID_TOOLS` is in the shell env", "commits are visible in the active terminal", "only one agent touches the repo at a time", "`AskUserQuestion` pauses the world until answered" -- needs to be re-examined. This report catalogs the blind spots that are easy to skip during feature planning but expensive to retrofit after the SDK integration has shipped.

**Scope boundary:** This report focuses on things the team will *forget* (logging, permissions, concurrency models, config precedence, state lifecycle, observability, developer experience). Things that will actively *break* (crashes, deadlocks, security holes) belong in `v7.0.0-research-pitfalls.md`. Where a concern straddles both, it is noted here only if the primary risk is *omission*, not *failure*.

---

## Cross-Cutting Concerns

### 1. Tool Shim Compatibility (Skill Behavior Under SDK)

**The blind spot:** Every one of the 30 existing skills was written against the Claude Code harness, where tools like `AskUserQuestion`, `Bash`, `Read`, `Edit`, `Glob`, `Grep`, and `Skill` have well-defined semantics. When the SDK is driving, the shape of these tools changes. If shim behavior is not audited skill-by-skill, subtle breakages will only surface at runtime.

- **`AskUserQuestion` semantics are load-bearing in non-obvious places.** `/rapid:discuss-set` uses it at every decision point. `/rapid:init` uses it for prereq retries, spec confirmation, branding opt-in, and roadmap approval. `/rapid:execute-set` uses it to ask which set to run. If the webUI shim returns a value asynchronously (or via a polling mechanism), any skill step that does `AskUserQuestion` inside a loop (retry prereqs, re-discuss) needs to handle promise-like semantics. The skills currently assume synchronous blocking.
- **`Bash` shim fidelity.** Every skill in the repo has the `RAPID_TOOLS` env-loading preamble (see `skills/execute-set/SKILL.md` step 0, `skills/discuss-set/SKILL.md` step 1, `skills/init/SKILL.md` "Environment Setup"). If the SDK's Bash tool runs commands in a different shell (e.g., `/bin/sh` vs `/bin/fish` vs `bash`), the `export $(grep -v '^#' .env | xargs)` pattern may fail silently -- `xargs` in POSIX sh handles quoting differently than in bash. This failure mode is invisible: commands run, `RAPID_TOOLS` is empty, the skill exits with "RAPID_TOOLS is not set".
- **`Skill` tool chaining.** `skills/init/SKILL.md` declares `Skill` in its `allowed-tools` frontmatter. If the SDK does not expose a `Skill` tool (it is a Claude Code harness feature, not a generic tool), init will not be runnable via SDK at all. This is different from "runs but with no user input" -- it simply cannot chain into sub-skills.
- **`Glob` and `Grep` scope.** These tools are sandboxed to the cwd in the Claude Code harness. The SDK's working directory is set via `ClaudeAgentOptions.cwd`. If the backend sets this to the project root but the skill expects the worktree, `Glob('.planning/sets/*/wave-*-PLAN.md')` returns the right thing only because `.planning/` lives at the project root. But `Glob('src/lib/*.cjs')` called from within a set context would return main-branch files, not worktree files. Several skills assume worktree-relative reads.
- **`Read`/`Edit`/`Write` path resolution.** Absolute paths work everywhere; relative paths depend on `cwd`. The RAPID conventions already require absolute paths in Bash agents, but `Read` calls in skill prose ("Read CONTEXT.md") are frequently relative. Any relative-path read in a skill step will resolve against whatever `cwd` the SDK is given.
- **Spawning sub-agents via Agent tool.** `/rapid:execute-set`, `/rapid:plan-set`, `/rapid:merge`, `/rapid:review` all spawn RAPID subagents (`rapid-executor`, `rapid-set-planner`, `rapid-merger`, etc.) via the `Agent` tool. If the SDK maps `Agent` to its `subagent` / session-fork feature, the subagent inherits the parent session's tool shims (including the webUI-shimmed `AskUserQuestion`). This means a `rapid-executor` that normally would never hit `AskUserQuestion` could, if something goes wrong, block on a UI prompt the user did not know was coming. The fix is to disallow `AskUserQuestion` from spawned executor/planner agents by design -- but this must be explicit in the SDK config.

**Why it matters early:** Each skill needs a compatibility audit *before* the SDK is wired in, not after. Fixing this after 10 skills have been "integrated" means re-testing all 10.

### 2. Worktree Isolation Model and Concurrency

**The blind spot:** The worktree model was designed around "a developer runs `/rapid:execute-set auth`, goes and gets coffee, and the agent does its thing." That mental model extends tenuously to "three browser tabs each kick off an autonomous execute on three different sets simultaneously."

- **Worktree-per-set holds, but does the SDK know?** `REGISTRY.json` records `{ setName: { path: ".rapid-worktrees/auth-system", branch: "rapid/auth-system" } }`. The SDK session `cwd` must be set to this path for worktree isolation to work. The backend spawning the SDK session must look this up *before* starting the session -- and must handle the case where the registry entry is missing or `orphaned` (the registry already has orphaned entries from past milestones, as shown in the current `REGISTRY.json`).
- **Solo mode breaks worktree isolation.** The registry supports a `solo: true` entry where the worktree path *is* the project root. If two autonomous runs target the same solo set (or one solo set and the project root), they will stomp each other's git index. The current CLI flow prevents this by the human only running one skill at a time; the webUI breaks that invariant.
- **Locking is per-project, not per-set.** `proper-lockfile` guards `.planning/STATE.json` at the project root (`.planning/.locks/`). Two agents running on different sets *will* contend for this lock during state transitions. The lock timeout is 300 seconds, which is generous for a human-scale flow but could pile up under the autonomous scenario (agent 1 is mid-transaction while agent 2-10 queue). The retry logic in `execute-set` (lines 489-498) retries 3 times with 2s sleep -- fine for humans, but with 5+ concurrent agents, cascading timeouts become plausible.
- **Multiple agents on the same set.** Nothing in the current architecture prevents two simultaneous `/rapid:execute-set auth` invocations. The second one will observe `status: executed` and re-enter from WAVE-COMPLETE markers, potentially racing with the first. The backend MUST enforce a per-set run mutex (in the database, not in the filesystem) and surface "already running, queue or cancel?" to the user.
- **Worktree git state during an active run.** If a user's browser autonomously runs `/rapid:execute-set` and then, mid-run, the user opens a terminal and does `git status` in the worktree, they see uncommitted partial changes. Worse, if they `git reset --hard` or switch branches, the agent's work is destroyed. The "chats" tab / "agents" tab needs a clear "this worktree has a running agent -- do not touch" indicator that also writes to a file the worktree itself can advertise (e.g., `.rapid-worktrees/<set>/.RUNNING`).
- **Registry reconciliation.** The `REGISTRY.json` shows `orphaned` entries from historical milestones. A long-running backend may hold stale registry state in memory. The backend must re-read the registry at run start, not cache it.
- **Branch state at run start.** Autonomous runs assume a clean git state. If the worktree has uncommitted changes from a previous aborted run (hung subprocess, killed via kill -9), the next run's commit will include garbage. The orchestrator needs a pre-flight `git status --porcelain` check with a user-visible "pre-run cleanup needed" dialog in the UI.

**Why it matters early:** Concurrency assumptions bleed into the database schema (runs table, locks), the UI state model (what is "running" means), and the SDK session lifecycle. Retrofitting a run-mutex after launch requires data migration.

### 3. Logging, Telemetry, and Autonomous Run Observability

**The blind spot:** The user needs to see what an autonomous agent is doing. The current system has three disjoint logging substrates (Claude Code CLI stdout, Node CLI `[RAPID ERROR]` on stderr, Python web backend JSON logs via `python-json-logger`). None of them is set up to stream a live narrative of an agent run into a browser UI.

- **Where do autonomous run logs live?** Three candidates, each with tradeoffs:
  - **SQLite (rapid.db) runs + messages tables.** Queryable, survives restarts, integrates with the existing `app.database` infrastructure. But SQLite + streaming many writes (stdout, tool calls, tool results) under WAL mode needs care -- the `busy_timeout=5000` pragma already set in `database.py` should be verified against the expected write rate.
  - **Append-only log files per run (`~/.rapid/logs/runs/{run_id}.log`).** Simple, cheap, streamable via SSE tail. But querying ("show me all failed runs for set 'auth'") requires grepping or a separate index.
  - **Hybrid: metadata in SQLite, raw stream in file.** Best for scale, worst for consistency.
  A decision here ripples into the UI (what the "agents" tab shows), retention policy (how long are logs kept?), and privacy (logs likely contain code snippets).
- **Structured event stream.** The SDK yields messages in a loop (`async for msg in client.receive_response()`). These include `ToolUseBlock`, `ToolResultBlock`, text deltas, and lifecycle markers. Persisting them as opaque JSON preserves fidelity but makes the UI work harder. The schema choice (one row per message? one row per tool-use + its result?) determines whether "show me what Bash commands the agent ran" is a `WHERE tool_name = 'Bash'` query or a JSON-parsing job.
- **Live streaming to the browser.** SSE is the obvious choice (FastAPI supports it natively, and the branding server already uses SSE at port 3141 -- reuse the pattern). WebSocket is overkill for one-way stream. The backend needs to multicast SDK messages to all connected viewers of a run, and must buffer recent messages for clients connecting mid-stream.
- **Correlation IDs.** A single `/rapid:execute-set` run spans: user action in UI -> backend run record -> SDK session -> potentially multiple sub-sessions (executor agents per wave) -> many Bash calls into `rapid-tools.cjs` -> state mutations in STATE.json. Without a `run_id` threaded through every log line and state mutation, debugging "why did wave 3 fail" means correlating timestamps. Add `run_id` as a first-class column in every runs-related table and inject it into SDK session env.
- **Cost and token accounting.** SDK sessions consume tokens. Long autonomous runs could rack up a bill. The UI should expose a per-run and per-project token count. The SDK returns token usage in its message stream -- capture it. The "agents" tab needs a cost column or the user will be blindsided.
- **Dangling runs on backend restart.** If uvicorn is restarted while a run is in progress, the SDK subprocess may continue or may die depending on how the backend shut down. Any run record in `running` status after startup needs to be reconciled: poll the PID, check the worktree, mark as `unknown` if unrecoverable. This is a lifecycle path that is easy to forget.
- **CLI parity: `node "${RAPID_TOOLS}"` output.** Every RAPID skill shells out to Node. That subprocess writes to stdout/stderr that the SDK captures as Bash tool results. If the user expects to see "plan-set created 4 waves" in the agents tab, that output must be extracted from the SDK message stream and surfaced -- otherwise it is buried in a raw `ToolResultBlock`.

**Why it matters early:** Logging schema is a database migration. Adding `run_id` or `tokens_used` columns after 1000 runs exist means an Alembic migration plus backfill. Designing the schema before the first SDK run ships is nearly free.

### 4. Persistent Chat History and Conversation State

**The blind spot:** The "chats" tab implies users have ongoing conversations with agents that span more than a single skill run. The current RAPID model has zero concept of a conversation -- each skill is a one-shot.

- **Where are conversations stored?** Candidates: new `chat_session` and `chat_message` tables in rapid.db; blob storage per session; SDK session-fork state. Chats that reference code snippets, kanban cards, or skill runs need foreign keys to keep them connected when the underlying entity is deleted (card deleted -> chat reference dangling).
- **SDK session resumption.** The SDK supports session forking. If a user closes a browser tab and reopens it, can they resume the conversation where they left off? Token-wise, the entire history needs to be re-fed unless the SDK supports resumption via a session ID. This is a vendor API question that needs answering up front.
- **Multi-user / multi-tab.** If two browser tabs are viewing the same chat session, edits in one need to appear in the other. SSE or WebSocket pub-sub solves this, but the initial load + reconciliation needs design.
- **History retention.** Chats accumulate. Without a retention policy, the db grows unbounded. Offer: never delete / delete after N days / user-triggered archive.
- **What counts as a "chat"?** A `/rapid:execute-set` autonomous run has a conversation (user prompt -> agent messages). Does that show in the chats tab, or only in the agents tab? The cleanest model is: every SDK session is a chat; "agents" is a filtered view of chats tied to a skill; "chats" is the unfiltered view including free-form questions.
- **Privacy.** Chats contain code. If the project is shared (RAPID is open-source and the dashboard is registerable), who can see whose chats? Even single-user deployments should set permission bits correctly on the sqlite file (`~/.rapid/rapid.db` -- chmod 600).
- **Attachment support.** Users will eventually want to paste images, attach files. Budget for it in the schema (e.g., a `chat_attachments` table with paths) or explicitly defer.

**Why it matters early:** Chat schema choices are sticky. Bolting on attachment support after launch is orders of magnitude harder than adding an unused `attachments_json` column up front.

### 5. Authentication and Credential Flow

**The blind spot:** The FastAPI backend currently binds to `127.0.0.1:9889` and has no authentication -- it is an unauthenticated local HTTP server. The SDK integration introduces credentials (`ANTHROPIC_API_KEY` at minimum) that the backend now holds and uses on behalf of browser users.

- **Browser users are not authenticated.** Anyone with local network access to port 9889 can trigger an autonomous agent run. If the user is on a coffee-shop wifi and binds to `0.0.0.0` (documentation may encourage this for mobile access), any local-network peer can spend their Anthropic credits. The backend needs a localhost-only default + optional bearer-token or session-cookie auth for network-exposed deployments. The `rapid-tools.cjs register-web` command already exists; extend it to generate a per-project API token.
- **Where is `ANTHROPIC_API_KEY` stored?** Options: (a) backend process env (simplest, but requires the user to export before starting rapid-web), (b) `~/.rapid/config.toml` or secrets file (persists, but another secret on disk), (c) Claude Code's existing OAuth credentials (if the SDK can reuse them, this is ideal since most users already have Claude Code authenticated).
- **SDK authentication model.** The SDK docs say `ANTHROPIC_API_KEY` is required. But Claude Code users are typically authenticated via OAuth, not API key -- the question is whether the bundled CLI respects the user's existing `claude` auth or requires a separate API key. This is a critical question for UX: making users set `ANTHROPIC_API_KEY` when they already have `claude` authenticated is a poor experience, and could lead to users putting API keys in dotfiles that get committed to git.
- **Multi-project credentials.** RAPID can have multiple registered projects. If each project uses a different Anthropic account (unlikely but possible), credentials need per-project scoping. Simpler: one global credential, but document it clearly.
- **Session secret for chat tabs.** If chats are persisted, the db file needs reasonable protection. File permissions (0600) for the sqlite file. Default location `~/.rapid/rapid.db` is fine; warn if users override `RAPID_WEB_DB_PATH` to a world-readable location.
- **CSRF for state-changing endpoints.** FastAPI's CORS config currently allows `localhost:5173` and `127.0.0.1:5173`. Once SDK endpoints ship, a malicious webpage the user visits could POST to `localhost:9889/api/...` from their browser and trigger a run. Standard CSRF mitigations (same-origin checks, token in header) are needed. This is a pitfall, not an oversight -- cross-listed so it is not missed.
- **Credential leakage in logs.** If the backend logs SDK input or MCP tool arguments verbatim, and the user pastes an API key into a chat, it ends up in `rapid-web.log`. Redaction rules in the structured logger (a pattern for `sk-ant-*`) avoid this.

**Why it matters early:** Adding auth after launch is a breaking change for every API endpoint and requires a migration path for existing users. Shipping localhost-only by default is a 2-line config change now.

### 6. Additive vs Breaking Changes to Existing CLI Skills

**The blind spot:** The milestone brief wants SDK integration, but does not commit to whether existing CLI flows must continue to work. If someone's muscle memory is `claude /rapid:execute-set auth`, does that keep working in v7.0.0? If yes, every new SDK-facing shim must have a CLI-compatible fallback. If no, that is a breaking change worth flagging in release notes.

- **Default path: fully additive.** The SDK becomes a new entry point that invokes the same skills the CLI does. Skills are unchanged. Tool shims (webUI-backed `AskUserQuestion`) only kick in when the SDK session has them registered. CLI sessions still use the built-in AskUserQuestion. This is the most user-friendly option.
- **Achievability depends on shim mechanism.** If the SDK shim replaces `AskUserQuestion` globally for a session (the SDK docs suggest per-session `mcp_servers` and tool registration, which *is* session-scoped), then additive-only is achievable. CLI sessions do not register the webUI shim, so they get the default behavior.
- **Skill frontmatter may need expansion.** `allowed-tools` in skill frontmatter lists the tools a skill uses. If the SDK runs a skill and does not have all those tools, the skill errors. To support webUI-only tools (e.g., `mcp__webui__ShowKanban`), either (a) add them to `allowed-tools` for all skills that might use them, or (b) make the tool list extensible at runtime. (a) is simpler; (b) is cleaner.
- **`Skill` tool / sub-skill chaining.** If the SDK does not expose a `Skill` tool, skills that chain to other skills (like `init` chaining to `scaffold`) will not run via SDK. Either implement a `Skill` equivalent as an in-process MCP tool, or refactor chainers to call CLI commands directly.
- **"Skip discuss" flag.** The brief says "optionally skip discuss phases when invoking from the browser". This aligns with the existing `/rapid:discuss-set --skip` flag. Verify that all human-input skills have a `--skip` or `--auto` path. `init`'s interactive branding step has an opt-in toggle; verify it works non-interactively.
- **Deprecation path.** If any skill gets a breaking change (new required tool, different output format to support UI streaming), bump the plugin version and document it in `docs/`. A deprecation warning in the CLI output gives users a heads-up.

**Why it matters early:** "Additive only" is a design constraint that prevents entire classes of accidental-breakage bugs. Promising it up front tells the team to reject any design that makes CLI use harder.

### 7. Configuration Management

**The blind spot:** RAPID already has multiple config surfaces (`.env`, `config.json`, `package.json`, `.planning/config.json`, `pyproject.toml`, `~/.rapid/rapid.db` AppConfig table, `~/.rapid/projects.json`). Adding "autonomous run config" (token budgets, timeouts, retry policies, model selection) on top risks config sprawl.

- **Model selection.** The SDK lets you pick the model (Opus / Sonnet / Haiku) per session. Autonomous runs benefit from cheaper models (Haiku for kanban queries, Sonnet for execute). Expose as per-skill config with a sensible default (`model_default` in AppConfig). Do not hardcode.
- **Token budgets / cost caps.** Without a cap, a looping agent could spend arbitrary money. Budget a per-run `max_tokens` and per-project-per-day cost cap. The UI surfaces "budget remaining" prominently.
- **Timeout policy.** `execute-set` can legitimately run for an hour. But a hung run that makes no progress should be killable. Per-skill timeout defaults, overrideable per run.
- **Precedence order.** Classic precedence: CLI flag > session override > project config > user config > system default. Document this or users will be baffled when their `.env` setting is ignored.
- **`.env` vs `~/.rapid/`.** RAPID's existing convention is project-local `.env` (RAPID_TOOLS, RAPID_WEB). The web backend uses `~/.rapid/` (db, projects.json, logs). Mixing these confuses users -- pick a lane per config type. Recommend: project-scoped config in `.env` + `.planning/config.json`; user-scoped config (Anthropic API key, default model) in `~/.rapid/config.toml`.
- **Secrets in `.env.example`.** RAPID's user-level CLAUDE.md mandates `.env.example`. The new config surface needs an `.env.example` entry for every new var (`ANTHROPIC_API_KEY`, `RAPID_WEB_SDK_ENABLED`, `RAPID_WEB_DEFAULT_MODEL`).
- **Hot reload.** Changing `rapid-web` config should not require a backend restart if the user is mid-run. Most settings (model, timeout) can be read per-run. A few (port, db path) genuinely require restart -- document which.

**Why it matters early:** Token budgets prevent runaway spending. Model selection prevents "autonomy = Opus-only = expensive".

### 8. Testing Strategy

**The blind spot:** The current test surface is Node `node:test` for the CLI and pytest for the web backend. SDK integration introduces a third test mode: "does a skill actually run end-to-end via SDK against a real Anthropic endpoint?" That is expensive (money + slow) but the only true integration test.

- **Unit tests for tool shims.** Every custom MCP tool (kanban add/view/move, AskUserQuestion replacement, file-backed prompt viewers) needs a unit test with the `@tool` decorator behavior mocked. Test the input schema validation, success path, error path, and permission denial.
- **SDK session mocking.** For the backend run orchestrator, mock the SDK's message stream. Write a fixture that yields a canned sequence of `ToolUseBlock` / `ToolResultBlock` / text deltas. The orchestrator code should be testable without any Anthropic calls.
- **Integration tests with real SDK, fake anthropic.** The SDK's bundled CLI can be pointed at a mock Anthropic endpoint. Expensive to set up, but catches SDK-level bugs that unit tests miss. Can also use a local Claude alternative (if cost is prohibitive) but fidelity suffers.
- **Skill smoke tests.** At least one end-to-end test per skill via SDK on a throwaway fixture project, running nightly against real Anthropic. Budget allowance of $5-10/day is likely sufficient for a handful of smoke tests.
- **Concurrency tests.** Fire 3 simultaneous runs against 3 different sets; assert no state corruption, both end `complete`. Fire 2 simultaneous runs against the same set; assert the second is rejected or queued. Fire a run, SIGKILL the backend mid-run, restart, assert reconciliation marks the run as failed.
- **Front-end tests.** The React frontend already has some infrastructure but no tests visible in the repo pass. Add Playwright-driven UI tests for: triggering a run, watching logs stream, cancelling a run, moving a kanban card.
- **Test database isolation.** The backend tests use `tests/` in the FastAPI project. Ensure SDK-testing uses a fresh sqlite db per test (current pattern with `get_engine(db_path=...)` supports this).

**Why it matters early:** Without integration tests, every SDK version bump is a gamble. Integration-test infrastructure is a one-time cost; skipping it means it never gets built.

### 9. CI/CD Pipeline

**The blind spot:** Adding autonomous runs introduces new failure surfaces (SDK version drift, Anthropic API changes, quota) that the CI needs to guard against. The current CI (if any) focuses on Node lib + Python backend tests.

- **SDK version pinning.** Pin the SDK to an exact version in `pyproject.toml`. Renovate or Dependabot bumps to PRs for visibility. Breaking SDK changes should surface in CI via integration tests, not in production.
- **Anthropic API version pinning.** If the SDK lets you pin the API version, do so. Avoids silent behavior changes from server-side model updates.
- **Tests require API key.** CI needs `ANTHROPIC_API_KEY` as a secret for integration tests. Non-fork PRs only (forks should not have access to the secret). Default to skipping integration tests on fork PRs, gating merge on them for internal PRs.
- **Build + deploy of the frontend.** Frontend builds with Vite; backend serves from `frontend/dist`. CI should build frontend and verify the output loads in a smoke test (curl the health endpoint + check the SPA index responds).
- **Migration tests.** Alembic migrations need a test that runs them against an old-version database to verify they apply cleanly. Every PR touching `models/` should include a migration and a test.
- **Release artifacts.** If RAPID ships a Python wheel for `rapid-web` + Node plugin package, the release pipeline must build both and version them in lockstep. Mismatched versions (v7.0.0 plugin + v6.x web backend) cause subtle bugs.
- **Hooks.** The project already has a `rapid-task-completed.sh` hook. Adding new hooks for SDK run lifecycle (`rapid-run-started`, `rapid-run-ended`) needs careful review so they do not spam or slow down runs.

**Why it matters early:** A broken CI at v7.0.0 launch means every follow-up bugfix is a gamble on untested code. Setting up integration tests in advance avoids the mid-release scramble.

### 10. Documentation Needs

**The blind spot:** RAPID has 9 docs in `docs/`. Adding "autonomous mode" is a major cognitive shift for existing users. Without clear docs, users will either ignore the new feature or use it in ways that surprise them.

- **"What is autonomous mode?" primer.** A short doc explaining: you queue a kanban card, RAPID picks it up, plans, executes, and merges. Reads logs. Stops it if it goes sideways. This is step 0 for new users.
- **"Migration from CLI-only" guide.** For existing users: nothing breaks. The SDK is additive. Here is how to opt in.
- **"When to use autonomy vs CLI" decision guide.** Autonomy is great for boilerplate sets (well-scoped, clear success criteria). CLI is better for exploratory or ambiguous work. Give guidance.
- **API reference.** The new REST endpoints for the SDK integration need OpenAPI docs (FastAPI generates these, but they need meaningful descriptions). Surface `/docs` and `/redoc` prominently.
- **Troubleshooting.** "My run hung." "My run failed with no logs." "My kanban card disappeared." These need runbook entries.
- **Privacy / security disclosure.** If chats and logs persist locally, state that. If analytics are sent anywhere, disclose. Users will ask.
- **Changelog / release notes.** v7.0.0 is a major version; the CHANGELOG should highlight: new WebUI tabs, autonomous mode, SDK requirement, any breaking changes.
- **Architecture diagrams.** Update `docs/ARCHITECTURE.md` (if exists) and `.planning/context/ARCHITECTURE.md` to reflect the new SDK-driven run lifecycle.

**Why it matters early:** Writing docs at the end of a milestone leads to skipped docs. Stub the doc files during planning; fill them as features land.

### 11. Accessibility

**The blind spot:** The webUI overhaul is adding new tabs with complex interactions (real-time log streaming, kanban DnD, chat UI). Each has known accessibility traps.

- **Live regions for streaming logs.** Use `aria-live="polite"` (not `"assertive"`) on the log output container. Screen readers should hear new log lines without being overwhelmed.
- **Kanban DnD alternative.** `@dnd-kit` (already in deps) supports keyboard-driven drag and drop if enabled. The accessibility config needs explicit setup; default keyboard support is minimal.
- **Color contrast.** The current UI uses Tailwind; check that the agents-tab status colors (running/complete/failed/cancelled) meet WCAG AA contrast against the theme background in both light and dark modes.
- **Focus management in chat.** When a new message arrives, focus should not steal from the input. When the user sends a message, focus returns to the input. Standard chat UI, but easy to get wrong.
- **Screen reader announcements for run state.** "Run started", "Run completed", "Run failed" should announce via live region. Silent UI updates leave screen reader users guessing.
- **Not applicable:** WCAG AAA level is overkill for an internal developer tool. AA is the target.

**Why it matters early:** Retrofitting accessibility onto a DnD kanban board after it ships is hours per component. Getting `dnd-kit` accessibility config right once is cheap.

### 12. Internationalization

**Not applicable (with rationale).** RAPID is a developer tool with English-only source, English-only skill prose, English-only documentation, and no current user base outside English-speaking devs. Adding i18n now would be substantial effort for zero current benefit. If the project ever adds enterprise customers in non-English markets, revisit -- but the cost to add i18n at v10 is similar to adding it now, and the current evidence does not justify it.

One narrow exception: date/time formatting in the UI. Use the user's locale (`Intl.DateTimeFormat`) for displayed timestamps rather than hardcoding a format. This is cheap and handles the timezone issue for free.

### 13. Data Concerns (Backup, Migration, Privacy)

**The blind spot:** The db is `~/.rapid/rapid.db`. The project data is `.planning/STATE.json`. Logs are `~/.rapid/logs/`. None of these have a defined backup strategy. The db grows with runs and chats; there is no retention policy.

- **Backup.** Sqlite under WAL mode is safe-ish to copy while running (checkpoint + cp). Document a `rapid-web backup` command that writes a point-in-time snapshot. Do not auto-backup without user consent.
- **Schema migrations.** Alembic is set up. Every new table/column needs a migration. Enforce via CI: a test that fails if models change without a migration.
- **Data retention.** Propose: runs kept indefinitely (metadata only), run logs purged after 90 days (user-configurable), chats kept indefinitely (user-deletable). Default policy documented; user can override.
- **Privacy scan.** Chat content + run logs may contain API keys, credentials pasted inadvertently, proprietary code. Document that the db is local-only unless the user registers with a remote dashboard, and that remote registration opts them in to sending data. The existing `register-web` command's privacy implications should be audited.
- **Export / import.** Users will want to move their chats/runs between machines or export for debugging. A `rapid-web export --run-id <id>` that dumps run + logs + messages to a zip helps both. Not urgent for v7.0.0, but stub the command so users can ask for it.
- **Data validation at boundaries.** Every SDK-facing endpoint validates input with Pydantic (FastAPI does this natively). Every input to MCP tools validated via Zod-equivalent in the SDK. Outputs from the agent (e.g., a kanban card title) should not trust the model -- sanitize for length, reject HTML/control chars for display.
- **GDPR / compliance.** Not currently applicable for a single-user local tool. If a hosted version ships in the future, this becomes significant. Document that v7.0.0 is local-only and any network egress (the dashboard registration) is clearly disclosed.

**Why it matters early:** Data retention + privacy positions need to be decided before users accumulate data under one assumption and have it flipped later.

### 14. Developer Experience

**The blind spot:** Adding a Python backend + React frontend + Node CLI + SDK integration means onboarding developers to four stacks. Without deliberate DX work, new contributors will stall at "how do I run this locally".

- **Unified dev command.** The `web/Makefile` exists. Extend it: `make dev` starts the backend + frontend + optional SDK mock in tmux panes. Zero config, one command.
- **Hot reload.** Vite handles frontend HMR. The Python backend with `uvicorn --reload` handles that side. The Node CLI does not need hot reload (it is one-shot). But when a dev changes a skill's SKILL.md, the SDK-driven run should pick it up without a backend restart -- skills are read from disk per run, so this is already free, but confirm.
- **Debugging SDK runs.** When a run misbehaves, dev needs: session id, full message stream, backend logs, worktree git state. A `rapid-web debug-run <id>` command that dumps all of these to a directory accelerates bug triage 10x.
- **Sample kanban workflow.** A demo script that seeds a fresh project with 5 kanban cards across 3 columns, so new users can see the system in action without setting up a real project.
- **Quality gates before commit.** `ruff check`, `pytest`, `node --test`, `tsc -b --noEmit` -- all should be in a single pre-commit hook. RAPID already has a `rapid-task-completed.sh`; audit whether it covers the new stack.
- **Error messages must be actionable.** "Port 9889 in use" -> include the lsof command (already done in `main.py`). "ANTHROPIC_API_KEY not set" -> include how to set it and where. "Run failed: stderr..." -> link to the log viewer URL.
- **Clear error surfacing in UI.** When a backend error occurs mid-run, the UI should show the error, not a generic spinner. Standard React error boundaries around the agents/chats tabs.

**Why it matters early:** Developer experience dictates velocity. Saving 10 minutes per dev per day across a team for a year is many weeks of engineering.

### 15. Kanban-Agent Integration Specifics

**The blind spot:** The brief lists "custom tools that let agents add, view, and move kanban cards". But the kanban is currently a human artifact -- users put things there for themselves. Blending agent and human writes has subtle UX pitfalls.

- **Ownership of cards.** Who created a card -- human or agent? Add a `created_by` field to `KanbanCard` (enum: `human`, `agent`) and surface it with a small icon in the UI. Users deserve to know which cards the agent added.
- **Agent-card safety.** When an agent "picks up" a card to execute, it should move it to an "in progress" column *and* lock writes to it from other agents. A second agent trying to pick up the same card should find it already locked.
- **Completion convention.** When the agent finishes, does the card move to "done" or is it deleted? UX-wise, moving to done is better (user can see what got done). Schema-wise, keep the card with a completion record attached (set_id, run_id).
- **Kanban "todo" as skill input.** The brief says users queue todos and agents read, plan, execute. Does every kanban card become a set? A quick task? A merge request? Define the card-to-skill mapping up front: a card with title "Add auth system" -> `/rapid:add-set auth-system`; a card with title "Fix typo in README" -> `/rapid:quick`.
- **Card -> run -> commit traceability.** Every commit made by an autonomous agent should reference the card that triggered it in the commit message (`feat(auth): add login form\n\nCard: <card_id>`). That way the user can trace any git history back to a kanban card.
- **View vs move permissions.** Agents view everything; move permission should be limited (the agent should not reorganize the user's columns without asking). Explicit permission list per custom tool: `view_cards` (read), `update_card_status` (move within workflow), NO `delete_card`.

**Why it matters early:** The kanban already exists. Retrofitting `created_by` after 100 cards exist is easy (default to `human`), but the UI treatment needs design up front.

### 16. Error Handling Strategy

**The blind spot:** Errors in autonomous runs have three distinct flavors that need distinct handling: (a) user-facing ("agent got stuck, here's why"), (b) debugging ("stack trace for the dev"), (c) programmatic ("machine-readable code for the UI to display the right state").

- **Error classes taxonomy.** Define up front: `SdkError` (SDK-level), `RunError` (run orchestration), `StateError` (RAPID state), `ToolError` (MCP tool), `UserError` (input). Each has user, debug, and programmatic representations.
- **RAPID:RETURN BLOCKED handling.** The CLI stops; the SDK-driven orchestrator needs to (a) mark the run as BLOCKED in the db, (b) surface the blocker reason in the UI, (c) optionally offer user-guided resolution (the user answers the blocker in chat, the agent resumes). The resume story is non-trivial -- budget for it or scope it out explicitly.
- **Graceful degradation.** If the SDK is unreachable, the webUI should fall back to "show CLI command to run manually". Do not hang.
- **Error messages aimed at the browser.** Node's `process.stderr.write('[RAPID ERROR] ...')` is terminal-friendly but the web UI needs structured JSON. The SDK captures stderr in `ToolResultBlock` as strings; the UI extraction layer needs to parse `[RAPID ERROR] ` prefix and surface cleanly.
- **Cancellation.** Users will hit cancel. The SDK supports session abort; test that aborting mid-tool-call leaves state consistent (no partial commits, no half-locked resources).

**Why it matters early:** Error handling that is retrofitted is always inconsistent. Designing the taxonomy once and applying it through the new code is much cheaper than fixing 15 different error formats later.

## Recommendations (Roadmap-Ready)

Ordered list of concerns that should be explicit roadmap items early, not afterthoughts:

1. **Foundation: SDK session dispatcher + run mutex.** A backend module that owns "start a run for skill X on set Y", enforces the per-set lock via a new `runs` table with a uniqueness constraint on `(project_id, set_id, status='running')`, sets SDK `cwd` to the worktree path, threads a `run_id` through every log line. Blocks: all other autonomous features.

2. **Foundation: logging substrate.** Decide: hybrid (SQLite metadata + file-backed raw stream) or all-SQLite. Define `runs`, `run_messages`, `run_tool_calls` tables. Set up Alembic migration. Expose SSE endpoint for `/api/projects/{id}/runs/{run_id}/stream`. Concrete first step: design schema PR, review, migrate.

3. **Foundation: tool shim compatibility audit.** Enumerate every skill's `allowed-tools`. For each, determine behavior under SDK: works as-is / needs shim / needs skill-level change. Produce a matrix in `docs/SKILL_SDK_MATRIX.md`. Concrete first step: write the script that reads all SKILL.md frontmatter and extracts tool lists.

4. **Foundation: authentication + credential model.** Default: localhost-only bind, no auth. Network mode: opt-in bearer token (generated by `rapid-web register-web`). Anthropic credential: reuse Claude Code OAuth if SDK supports it; otherwise `ANTHROPIC_API_KEY` from `~/.rapid/config.toml`. Document in `docs/SECURITY.md`.

5. **Foundation: kanban schema v2.** Add `created_by` (human/agent), `locked_by_run_id`, `completed_by_run_id` columns. Write migration + backfill. UI indicator for agent-created cards. Concrete first step: model changes + migration.

6. **Foundation: error taxonomy + BLOCKED resume skeleton.** Define `SdkError`, `RunError`, `StateError`, `ToolError`, `UserError` as Pydantic models. When a run returns BLOCKED, show the blocker message in UI and offer a text-area to respond; typing a response resumes the SDK session with the input as the next message. MVP is: surface the blocker, offer cancel. Full resume can ship in v7.1.

7. **Foundation: test infrastructure.** Mock SDK harness for unit tests; real-SDK integration tests gated on `ANTHROPIC_API_KEY` secret in CI; one Playwright test per new tab. Budget a line in `pyproject.toml` for `respx` or similar HTTP mocking.

8. **Early: token budget + cost cap UI.** Per-project per-day cost cap (default: $10, user-configurable); per-run token budget (default: 100k, user-configurable). When exceeded, the run is rejected at dispatch. Show budget remaining prominently in the UI.

9. **Early: additive-only constraint + CLI parity check.** Add to PR template: "Does this change break any CLI flow?" Automate with a smoke test that runs `/rapid:execute-set` via CLI against a fixture project every CI run.

10. **Early: documentation stubs.** Create `docs/AUTONOMOUS_MODE.md`, `docs/SDK_INTEGRATION.md`, `docs/MIGRATION_FROM_v6.md`, `docs/SECURITY.md` as empty-but-outlined files on day one. Each feature PR fills in the relevant section. Prevents the "docs written the night before release" anti-pattern.

11. **During UI work: accessibility pass per component.** Live region for log stream, keyboard DnD for kanban, focus management in chat. Add an axe-core smoke test to Playwright.

12. **Post-launch: data retention + export.** Stub `rapid-web prune` and `rapid-web export` commands with "not yet implemented" messages. Monitor db growth. Ship real implementations in v7.1 based on actual usage.
