# Pitfalls Research -- v4.1.0 Polish & Fixes

## Critical Pitfalls

### Pitfall: Zod 3.25.x Silently Bundles Zod 4 Internals
- **Category:** compatibility / correctness
- **Stack:** Zod ^3.25.76
- **Description:** Starting from Zod 3.25.x, the package internally bundles Zod 4 code alongside Zod 3. This doubles the package size and can cause unexpected behavioral differences, particularly with `.optional()`, `.default()`, and `.catch()` on object properties. In Zod 4, properties with `.catch()` or `.default()` always return caught values even when absent from input, diverging from Zod 3 behavior where they were ignored.
- **Trigger:** Any schema using `.optional()` combined with `.default()` or `.catch()` on object properties -- relevant to `ProjectState`, `ReviewIssue`, and `ScoperOutput` schemas in `state-schemas.cjs` and `review.cjs`. The `ReviewIssue` schema uses `.default(false)` on `autoFixAttempted` and `.default('open')` on `status`, which are vulnerable to this behavioral shift.
- **Impact:** Silent data mutation during validation. State objects could gain unexpected default values on fields that were previously undefined, corrupting STATE.json or REVIEW-ISSUES.json integrity. Severity: HIGH.
- **Prevention:** Pin Zod to `3.24.4` (last known stable pre-Zod4 bundling) or explicitly test all schemas with `.safeParse()` on inputs that omit optional fields to verify the expected behavior. Add regression tests for schema parsing of partial objects.
- **Detection:** Unit tests that parse partial objects and assert that omitted optional fields remain `undefined` (not their default values). If these tests break after a `npm update`, the Zod 4 behavioral shift has leaked in.
- **References:** [Zod Issue #4923 -- Zod3 now includes Zod4](https://github.com/colinhacks/zod/issues/4923), [Zod Issue #4883 -- Breaking change in optional properties](https://github.com/colinhacks/zod/issues/4883), [Zod Issue #5239 -- Publish new zod@3.x with zod@4.1.x](https://github.com/colinhacks/zod/issues/5239)

### Pitfall: Init Step 4D/5 Ordering Overwrites REQUIREMENTS.md
- **Category:** data-loss
- **Stack:** RAPID init skill (`skills/init/SKILL.md`, `src/lib/init.cjs`)
- **Description:** Issue #7 confirms that Step 4D writes user-crafted acceptance criteria to `.planning/REQUIREMENTS.md`, then Step 5 calls `init scaffold` which calls `scaffoldProject()` in `init.cjs`. In `fresh` and `reinitialize` modes, `scaffoldProject()` unconditionally writes all files including a blank `REQUIREMENTS.md` template, destroying the user's acceptance criteria. Only `upgrade` mode has an `existsSync` guard that skips existing files.
- **Trigger:** Running `/rapid:init` on a new project (fresh mode) or reinitializing -- the most common paths.
- **Impact:** Complete loss of user-authored acceptance criteria. The user must re-enter all requirements. The verifier agent downstream has nothing meaningful to validate against. Severity: CRITICAL.
- **Prevention:** Two options: (1) Reorder so scaffold runs BEFORE 4D, then 4D writes/overwrites REQUIREMENTS.md with real content. (2) Add an `existsSync` guard in `scaffoldProject()` for REQUIREMENTS.md across all modes, consistent with upgrade mode's additive-only pattern.
- **Detection:** After running init, read `.planning/REQUIREMENTS.md` and verify it contains user-specific acceptance criteria, not the blank template (`## v1 Requirements`).
- **References:** [GitHub Issue #7](https://github.com/pragnition/RAPID/issues/7)

### Pitfall: State Transition Map Allows `pending -> planned` Bypass of Discussion
- **Category:** correctness
- **Stack:** `src/lib/state-transitions.cjs`
- **Description:** The `SET_TRANSITIONS` map allows `pending -> ['discussed', 'planned']`. This means a set can skip the `discussed` state entirely and go straight to `planned`, bypassing the discuss-set phase. While this might be intentional for `--skip` workflows, it creates a dangerous path where sets can enter planning without any CONTEXT.md, leaving the planner agent without implementation vision.
- **Trigger:** Any code path that transitions directly from `pending` to `planned` without first running discuss-set or its `--skip` equivalent.
- **Impact:** The planner receives no CONTEXT.md, leading to poor or generic plans that do not reflect user intent. If the discuss-set enhancement (todo item #4) adds richer context generation, this bypass becomes even more costly. Severity: HIGH.
- **Prevention:** When modifying the discuss-set flow, ensure all transition paths either (a) produce CONTEXT.md or (b) the planner has a guard that checks for CONTEXT.md existence before planning. Consider removing `planned` from `pending`'s allowed transitions if the bypass is unintentional.
- **Detection:** Check for sets in `planned` status that lack a corresponding `.planning/sets/{setId}/CONTEXT.md` file.
- **References:** `src/lib/state-transitions.cjs` line 4

## High-Risk Pitfalls

### Pitfall: AskUserQuestion 4-Option Limit Creates Fragile Gray Area Scaling
- **Category:** correctness / compatibility
- **Stack:** Claude Code AskUserQuestion API, `skills/discuss-set/SKILL.md`
- **Description:** The todo item #4 specifies increasing gray areas beyond 4, but keeping the count as a multiple of 4 because "each ask user question can only have 4 options." This is architecturally fragile: AskUserQuestion actually supports 2-4 options per question, and multi-select mode. The constraint is that option COUNT must be 2-4, not that question COUNT must be a multiple of 4. Confusing "number of gray areas" with "number of options per question" will produce broken prompts.
- **Trigger:** Attempting to present 8 gray areas as 2 AskUserQuestion calls with 4 options each. If any gray area title exceeds the character limits or the prompt structure is wrong, the question silently fails or returns empty.
- **Impact:** The discuss-set skill stops working correctly. Users get empty responses or broken selection flows. The AskUserQuestion tool has a [known bug](https://github.com/anthropics/claude-code/issues/29547) where it silently auto-completes with empty answers when called inside plugin skills. Severity: HIGH.
- **Prevention:** Design the gray area selection as a single multi-select AskUserQuestion (using `multiSelect: true`) rather than splitting into multiple sequential questions. Test the exact prompt format in the Claude Code environment before deploying. Keep option descriptions concise (under 80 chars).
- **Detection:** After discuss-set runs, check if the CONTEXT.md contains actual user decisions or only "Claude's Discretion" entries (indicating questions were not presented).
- **References:** [AskUserQuestion empty answers bug](https://github.com/anthropics/claude-code/issues/29547), [AskUserQuestion returns empty results](https://github.com/anthropics/claude-code/issues/10229)

### Pitfall: RAPID_ROOT Removal May Break Active Skills That Still Reference It
- **Category:** correctness
- **Stack:** Shell preambles across 24+ skill files
- **Description:** Issue #3 calls for sweeping and removing `RAPID_ROOT` references. The Grep results show 103 files reference `RAPID_ROOT`. However, many of these are in `.planning/archive/` (historical, should not be touched) and `skills/*/SKILL.md` (active, must be updated carefully). The skill preamble pattern `RAPID_ROOT="${CLAUDE_SKILL_DIR}/../.."` is used in every single skill file as the first line of every bash block. If `RAPID_ROOT` is removed but `.env` loading still references `$RAPID_ROOT/.env`, all skills break.
- **Trigger:** Removing the `RAPID_ROOT` variable assignment without updating the `.env` loading line that depends on it: `if [ -f "$RAPID_ROOT/.env" ]`.
- **Impact:** Every skill becomes unable to load RAPID_TOOLS from .env, making the entire plugin non-functional. Severity: HIGH.
- **Prevention:** The sweep must distinguish between (a) `RAPID_ROOT` assignments that are actually used by subsequent lines in the same bash block, and (b) truly orphaned references. If `RAPID_ROOT` is used only for `.env` loading, replace the two-line pattern with a single-line equivalent: `if [ -f "${CLAUDE_SKILL_DIR}/../../.env" ]`. Do not touch archive files.
- **Detection:** After the sweep, run `grep -r "RAPID_ROOT" skills/ src/modules/` -- result should be zero. Then run each skill's environment preamble in isolation to verify `RAPID_TOOLS` is still set.
- **References:** [GitHub Issue #3](https://github.com/pragnition/RAPID/issues/3)

### Pitfall: Removing Unit-Test 5-Group Limit Without Token Budget Awareness
- **Category:** performance
- **Stack:** `skills/unit-test/SKILL.md`, Claude Agent SDK
- **Description:** Issue #29 removes the 5-group ceiling on concern-group dispatching. The current ceiling exists because each concern group spawns a separate `rapid-unit-tester` agent. With 10+ concern groups (as was observed in a protobuf set), this spawns 10+ parallel agents, each consuming significant token budgets. The cumulative token usage can exceed the conversation limit or cause rate limiting.
- **Trigger:** A set with many files and diverse concern groups (e.g., 10+ concerns from a large microservice set).
- **Impact:** Conversation token exhaustion, rate limiting, or incomplete test coverage because agents fail silently when hitting limits. API costs scale linearly with group count. Severity: HIGH.
- **Prevention:** Replace the hard limit with a soft budget: (1) Allow unlimited concern groups but batch them into rounds of N (e.g., 5 per round). (2) Track cumulative token usage and warn when approaching limits. (3) Allow the user to approve the dispatch plan before spawning agents (the current Step 4 approval flow already exists).
- **Detection:** Monitor the total number of agents spawned and their completion status. If agents return BLOCKED or fail to return RAPID:RETURN markers, the limit was likely hit.
- **References:** [GitHub Issue #29](https://github.com/pragnition/RAPID/issues/29)

### Pitfall: DAG Re-introduction May Conflict With Existing State Structure
- **Category:** correctness
- **Stack:** `src/lib/dag.cjs`, `src/lib/state-machine.cjs`, `src/lib/state-schemas.cjs`
- **Description:** Todo item #3 adds back the DAG for set dependency resolution. The DAG module (`dag.cjs`) already exists with full toposort, wave assignment, and cycle detection. However, the current `ProjectState` schema in `state-schemas.cjs` has no field for DAG edges or set dependencies. Adding DAG data requires a schema migration. If the DAG is stored in a separate file (`DAG.json` at `.planning/sets/DAG.json`, per `DAG_CANONICAL_SUBPATH`), it can desync from STATE.json -- e.g., a set is deleted from state but remains in DAG edges.
- **Trigger:** Adding a set via `plan create-set` without updating DAG.json, or removing/renaming a set without pruning its edges from the DAG.
- **Impact:** Stale DAG edges cause cycle detection failures, incorrect wave ordering, or crashes in the command center dashboard when rendering a graph with orphaned nodes. Severity: MEDIUM-HIGH.
- **Prevention:** Make DAG updates atomic with state mutations: whenever a set is added/removed in STATE.json, validate and update DAG.json in the same transaction. Add a DAG validation step that cross-references DAG node IDs against STATE.json set IDs.
- **Detection:** Add a `dag validate` CLI command that checks (1) all DAG node IDs exist in STATE.json, (2) all STATE.json set IDs exist in DAG nodes, (3) no cycles.
- **References:** `src/lib/dag.cjs` (existing module), `DAG_CANONICAL_SUBPATH` constant at line 25

## Medium-Risk Pitfalls

### Pitfall: `review log-issue` CLI/Stdin Interface Mismatch Creates Silent Failures
- **Category:** correctness
- **Stack:** `src/commands/review.cjs`, `skills/unit-test/SKILL.md`
- **Description:** Issue #27 documents that skill docs show CLI flags (`--set-id`, `--type`, `--severity`) but the implementation reads a JSON object from stdin. Agents following the documented CLI flags silently fail -- they get a "No data on stdin" error. The fix must choose one interface and update both the implementation and all consuming skills.
- **Trigger:** Any agent or skill that invokes `review log-issue` with CLI flags instead of piping JSON to stdin.
- **Impact:** Issues are not logged, meaning the review pipeline loses bug findings. The bug-hunt and unit-test stages appear to complete successfully but produce no issue records. Severity: MEDIUM.
- **Prevention:** If switching to CLI flags: implement `parseArgs()` for all fields and auto-generate `id` and `createdAt`. If keeping stdin JSON: update all skill docs and agent prompts to use the pipe format. Either way, add a clear error message that explains the expected input format.
- **Detection:** After running the review pipeline, check that `.planning/sets/{setId}/REVIEW-ISSUES.json` contains the expected number of issues.
- **References:** [GitHub Issue #27](https://github.com/pragnition/RAPID/issues/27)

### Pitfall: Making Unit-Test Framework-Agnostic Breaks Existing Test File Generation
- **Category:** compatibility
- **Stack:** `skills/unit-test/SKILL.md`, `agents/rapid-unit-tester.md`
- **Description:** Issue #31 requires making the test runner auto-detected based on the project stack. The current unit-test skill hardcodes `node --test` with `node:test` + `node:assert/strict` in Step 5's agent prompt template. Changing this affects: (1) the agent prompt templates in Steps 3 and 5, (2) the test file naming convention (`.test.cjs` may not apply for Rust/Python/Go), (3) the retry/fix logic in Step 5a which assumes `node --test {testFile}` syntax, and (4) REVIEW-UNIT.md formatting which assumes test file naming conventions.
- **Trigger:** Running unit-test on a non-JavaScript project (Python, Rust, Go, etc.) after making the skill framework-agnostic.
- **Impact:** Test files are generated with wrong naming, wrong runner invocation, or wrong assertion library. The retry loop tries to "fix test code" but does not know which test framework's idioms to use. Severity: MEDIUM.
- **Prevention:** Add test framework detection to the init research phase (as planned). Store detected test framework in `.planning/config.json`. The unit-test skill reads this config and selects the appropriate prompt template, file naming pattern, and runner command. Provide fallback templates for at least: `node:test`, `jest`, `pytest`, `cargo test`, `go test`.
- **Detection:** After execution, verify test files were created with the correct extension and runner. Run `node --test` (or equivalent) on generated files to confirm they execute without import/syntax errors.
- **References:** [GitHub Issue #31](https://github.com/pragnition/RAPID/issues/31)

### Pitfall: Git Worktree Hooks Are Shared With Main Repository
- **Category:** correctness
- **Stack:** Git worktrees, `.rapid-worktrees/`
- **Description:** Git worktrees share the `.git/hooks` directory with the main repository. If a pre-commit hook exists in the main repo, it runs in all worktrees. Conversely, scripts that try to install hooks per-worktree will fail because the worktree's `.git` is a file (not a directory) pointing back to the main `.git/`. The RAPID `rapid-task-completed.sh` hook lives in `src/hooks/` but its installation path may not work correctly in worktree contexts.
- **Trigger:** A pre-commit hook that validates paths or checks for files relative to the repository root may break when run from a worktree at `.rapid-worktrees/{set-name}/` because the working directory differs.
- **Impact:** Commits in worktrees may be blocked by hooks that assume the main checkout structure, or hooks may silently skip validation. Severity: MEDIUM.
- **Prevention:** Ensure hooks use `git rev-parse --show-toplevel` to find the actual working tree root rather than hardcoding paths. Test hook execution from within a worktree directory.
- **Detection:** Create a worktree, make a commit in it, and verify hook output references the correct paths.
- **References:** [Git worktree hook issues](https://github.com/pre-commit/pre-commit/issues/808), [Using Git Hooks When Creating Worktrees](https://mskelton.dev/bytes/using-git-hooks-when-creating-worktrees)

### Pitfall: `node --test` Glob Pattern Expansion Differences Across Platforms
- **Category:** compatibility
- **Stack:** Node.js built-in test runner, `package.json` test script
- **Description:** The `package.json` test script uses `node --test 'src/**/*.test.cjs'`. Glob expansion behavior differs: on Linux, the shell expands the glob before Node sees it (if unquoted), while the quoted version relies on Node's internal glob handling which has known inconsistencies (Issue nodejs/node#52191). On Windows, shell glob expansion does not occur at all. The single quotes in the npm script prevent shell expansion, relying entirely on Node's test runner glob implementation.
- **Trigger:** Running `npm test` on Windows or with Node.js versions where the test runner glob matching has bugs (some versions only match direct children, not nested files).
- **Impact:** Tests silently do not run. `npm test` reports success with 0 tests executed, giving false confidence. Severity: MEDIUM.
- **Prevention:** Verify the glob pattern works with the project's minimum Node.js version (18+). Consider using a test runner wrapper that handles glob expansion independently, or switch to double quotes for cross-platform compatibility.
- **Detection:** After running `npm test`, check that the test count is greater than zero. Add a CI step that compares discovered test files against `glob('src/**/*.test.cjs')` output.
- **References:** [Node Issue #50658 -- glob patterns with npm](https://github.com/nodejs/node/issues/50658), [Node Issue #52191 -- incorrect glob execution](https://github.com/nodejs/node/issues/52191)

### Pitfall: Discuss-Set Enhancement May Break the --skip Auto-Context Path
- **Category:** correctness
- **Stack:** `skills/discuss-set/SKILL.md` Step 4 and Step 5
- **Description:** Todo item #4 calls for richer CONTEXT.md output, more gray areas, and deferred decisions. The `--skip` path (Step 4) spawns a `rapid-research-stack` agent that produces a minimal auto-context. - **Impact:** Plans generated from `--skip` context are missing critical sections, causing the planner to hallucinate or skip aspects of the set. Severity: MEDIUM.
- **Prevention:** Define the CONTEXT.md schema once (either as a template in the skill or as a Zod schema) and reference it from both the interactive path and the --skip agent prompt. Any change to the schema must update both paths.
- **Detection:** Compare CONTEXT.md files generated by interactive vs --skip modes and verify they have the same section headers.
- **References:** `skills/discuss-set/SKILL.md` Steps 4 and 7

## Anti-Patterns for This Stack

### Anti-Pattern: Modifying SKILL.md Prompt Structure Without Testing AskUserQuestion Rendering
- **What people do:** Change question text, option count, or option descriptions in SKILL.md without testing in Claude Code's actual UI.
- **Why it fails:** AskUserQuestion has strict constraints (2-4 options, character limits on headers). Exceeding these causes silent failures or empty responses. The tool also behaves differently when called from plugin skills vs direct conversation.
- **Correct approach:** After any SKILL.md change that modifies an AskUserQuestion call, manually test the exact flow in Claude Code. Verify the question renders, options are selectable, and the response is captured.
- **Example:** Adding a 5th option to a gray area question silently drops to freeform mode, losing structured selection.

### Anti-Pattern: Using `execSync` With String Interpolation for Git Commands
- **What people do:** Build git command strings with template literals: `` execSync(`git ${args.join(' ')}`, ...) ``
- **Why it fails:** If any argument contains spaces, quotes, or shell metacharacters (e.g., set names with special characters), the command breaks or enables shell injection. `worktree.cjs` line 21 uses this pattern.
- **Correct approach:** Use `execFileSync('git', args, ...)` which bypasses shell interpretation entirely, or use the existing `gitExec()` wrapper but audit it for shell safety.
- **Example:** A set named `my set` (with space) would cause `git worktree add .rapid-worktrees/my set` to fail.

### Anti-Pattern: Hardcoding State File Paths Across Multiple Modules
- **What people do:** Construct `.planning/STATE.json` or `.planning/sets/{id}/CONTEXT.md` paths manually in each module.
- **Why it fails:** If the directory structure changes (e.g., adding versioned subdirectories), every path construction must be updated. The DAG module correctly uses `DAG_CANONICAL_SUBPATH` as a constant, but most other modules construct paths inline.
- **Correct approach:** Centralize all planning artifact paths in a single `paths.cjs` module that exports functions like `getStatePath(cwd)`, `getSetDir(cwd, setId)`, `getContextPath(cwd, setId)`.
- **Example:** `state-machine.cjs` defines `const STATE_FILE = 'STATE.json'` and `const PLANNING_DIR = '.planning'` separately from similar definitions in other modules.

### Anti-Pattern: Spawning Subagents Without Verifying Their Output Artifacts
- **What people do:** Spawn an agent, check if it returned COMPLETE, then proceed to the next step without verifying the expected file was actually written.
- **Why it fails:** Agents can return COMPLETE without writing all expected artifacts (e.g., the discuss-set --skip path spawns a research agent and only checks `[ -f CONTEXT.md ]` afterwards). If the agent crashes mid-write, a partial file may exist.
- **Correct approach:** After agent completion, read the artifact file and validate its structure (e.g., check for expected section headers, minimum line count, or parse with a schema).
- **Example:** Step 4 of discuss-set checks file existence but not content: `[ -f ".planning/sets/${SET_ID}/CONTEXT.md" ] && echo "CONTEXT.md created"`.

## Version-Specific Gotchas

- **Node.js 18 (minimum):** `node:test` module is experimental in Node 18.x; the `--test` CLI flag and glob support stabilized in Node 20. Running `node --test 'src/**/*.test.cjs'` on Node 18 may not discover all test files.
- **Zod 3.25.76:** Contains bundled Zod 4 internals. The `^3.25.76` semver range will pull in any 3.25.x+ patch, including ones that further change the Zod 4 bundling behavior.
- **Git 2.30+:** `git worktree list --porcelain` output format changed between versions; ensure parsing handles both old and new formats.
- **proper-lockfile 4.1.2:** `realpath: false` (used in `lock.cjs`) disables symlink resolution; this is correct for most cases but can cause double-locking if the same file is accessed via different symlink paths (relevant for NixOS installations per Issue #3).

## Security Pitfalls

1. **Shell injection via `execSync` with string interpolation:** `worktree.cjs` constructs git commands by joining args into a string. Set names, branch names, or file paths containing shell metacharacters (`; rm -rf /`, `$(cmd)`, backticks) could execute arbitrary commands. -- **Prevention:** Use `execFileSync` instead of `execSync` with shell strings, or validate all user-provided names against a strict pattern (alphanumeric + hyphens only).

2. **stdin JSON injection in `review log-issue`:** The `readStdinSync()` function reads arbitrary JSON from stdin and passes it to `JSON.parse()`. While Zod validation catches schema violations, any pre-validation code that accesses parsed fields (e.g., `issue.file` used in path construction) without sanitization could enable path traversal. -- **Prevention:** Validate all file path fields with `path.isAbsolute()` checks and reject absolute paths or `../` sequences.

3. **Unvalidated set names in file system operations:** Set names from user input are used directly in `path.join()` calls to construct worktree paths and planning artifact directories. A set name like `../../../etc` could write files outside the project root. -- **Prevention:** Validate set names against `/^[a-z0-9][a-z0-9-]*$/` before using in path construction. The `plan.cjs` module should reject non-conforming names.

4. **Lock target file race in `lock.cjs`:** Lines 47-52 check `!fs.existsSync(lockTarget)` then `fs.writeFileSync(lockTarget, ...)`. Between the check and write, another process could create the same file (TOCTOU race). -- **Prevention:** Use `fs.writeFileSync(lockTarget, ..., { flag: 'wx' })` (exclusive create) wrapped in a try-catch, or accept that `proper-lockfile` handles this at the locking level.

## Performance Traps

1. **Unbounded agent spawning after removing 5-group limit:** Without a ceiling, the unit-test skill can spawn 10+ agents simultaneously. Each agent generates its own test plan and executes tests, consuming API tokens proportional to the code volume. For a set with 15 concern groups, this could cost 15x the normal token budget. -- **Mitigation:** Batch concern groups into rounds of 5, with a user approval gate between rounds.

2. **Synchronous file operations in hot paths:** `state-machine.cjs` uses `fs.readFileSync` and `fs.writeFileSync` within locked transactions. For projects with large STATE.json files (many milestones, sets, waves), these blocking reads/writes stall the Node.js event loop. -- **Mitigation:** Not immediately critical (STATE.json is typically small), but if the number of milestones grows, consider async file operations within the lock.

3. **DAG validation on every state read:** If DAG validation (cycle detection, node cross-reference) is added as a step in `readState()`, it runs on every CLI command. Kahn's algorithm is O(V+E) which is fast for small graphs, but unnecessary on read-only operations. -- **Mitigation:** Run DAG validation only on mutations (set add/remove/rename), not on reads.

4. **Test runner re-discovery on each skill invocation:** The framework-agnostic test runner (Issue #31) must detect the project's test framework. If detection involves scanning `package.json`, lockfiles, and source files, doing this on every `/rapid:unit-test` invocation adds latency. -- **Mitigation:** Cache the detected framework in `.planning/config.json` during init. Only re-detect if the user explicitly requests it or the config is missing.
