# Features Research

## Feature Inventory

### Feature: Branding Webserver Overhaul

- **Description:** Replace the current manual-refresh branding webserver with an auto-reloading artifact system. The model creates artifacts (logo, theme, wireframe, branding guidelines page) that are browsable on a live-updating page. After the theme is settled, the branding skill offers follow-up artifact generation (logo, site wireframe, branding guidelines page).
- **Complexity:** complex
- **Dependencies:** independent (but the "Integrate /branding into init flow" UX improvement depends on this being at least partially complete)

#### Decomposition

1. **Auto-reload infrastructure** -- Add Server-Sent Events (SSE) endpoint to branding-server.cjs and `fs.watch` on the branding directory. Client-side JS listens to the SSE stream and reloads/updates content when files change.
2. **Artifact registry system** -- Replace the flat file listing with a structured artifact model. Each artifact has a type (theme, logo, wireframe, guidelines), display metadata (title, description, preview thumbnail), and creation timestamp. Registry stored as `artifacts.json` in the branding directory.
3. **Hub page redesign** -- Transform the current static hub page into a browsable artifact gallery. Each artifact renders as a card with preview, title, type badge, and timestamp. Auto-updates via SSE when new artifacts are created.
4. **Extended branding skill flow** -- After the theme interview settles BRANDING.md, offer additional artifact generation steps: logo document generation, site wireframe generation, and a final branding guidelines summary page.
5. **Artifact CRUD API** -- Add HTTP endpoints for creating/listing/deleting artifacts so the agent can programmatically register artifacts as they are generated. Endpoints: `POST /artifacts`, `GET /artifacts`, `DELETE /artifacts/:id`.

#### Implementation Strategies

**Option 1: SSE + fs.watch (zero-dependency)**
- Approach: Use Node.js built-in `fs.watch` to monitor `.planning/branding/` for changes. Add an `/_events` SSE endpoint to the existing `http.createServer` in branding-server.cjs. The hub page includes a small inline `<script>` that opens an `EventSource` connection and triggers DOM updates or full page reload on file change events. Artifact metadata stored in a JSON file that the server reads on each request.
- Pros: Zero new dependencies (fs.watch and SSE are built-in). Consistent with RAPID's minimal dependency philosophy (only 4 production deps). SSE is simpler than WebSocket for unidirectional push. Works in all modern browsers natively.
- Cons: `fs.watch` has platform quirks (duplicate events on some OSes, no filename on some platforms). Need debouncing logic to avoid rapid-fire reloads. SSE connections keep the HTTP connection open, requiring connection tracking for cleanup.
- Libraries/services: None new -- uses `node:http`, `node:fs`, native `EventSource` browser API.

**Option 2: chokidar + ws (library-assisted)**
- Approach: Use `chokidar` for cross-platform file watching (handles fs.watch quirks) and `ws` for WebSocket-based live reload. More reliable than raw fs.watch but adds two dependencies.
- Pros: chokidar handles debouncing, duplicate events, and platform differences out of the box. WebSocket is bidirectional if future needs arise.
- Cons: Adds 2 new production dependencies to a project that currently has only 4. chokidar is a large dependency tree. WebSocket is overkill for unidirectional push. Violates RAPID's minimalist dependency approach.
- Libraries/services: chokidar (~3.6.0), ws (~8.17.0)

**Recommended:** Option 1 (SSE + fs.watch). The zero-dependency approach aligns with RAPID's existing philosophy. The fs.watch quirks are manageable with a simple debounce wrapper (50-100ms timeout). SSE is the right abstraction for server-to-client push of file change notifications.

#### Data Model Impact

- **New entities:** `artifacts.json` file in `.planning/branding/` directory -- array of artifact objects:
  ```json
  [
    {
      "id": "artifact-uuid",
      "type": "theme|logo|wireframe|guidelines",
      "title": "Color Palette & Typography",
      "description": "Visual identity tokens for the project",
      "filename": "index.html",
      "createdAt": "2026-04-07T12:00:00Z"
    }
  ]
  ```
- **Existing entities modified:** The `_generateHubPage()` function in branding-server.cjs transforms from a flat file listing to an artifact-card gallery renderer. The `.server.pid` file format is unchanged.
- **Relationships:** Each artifact references a file in the branding directory by `filename`. The artifact registry is the source of truth for what appears on the hub page; raw files without registry entries are listed in a separate "Other Files" section for backward compatibility.

#### API Surface

- **New endpoints:**
  - `GET /_events` -- SSE endpoint, streams `data: {"type":"change","filename":"..."}` events
  - `GET /artifacts` -- JSON array of all registered artifacts
  - `POST /artifacts` -- Create a new artifact entry `{type, title, description, filename}`
  - `DELETE /artifacts/:id` -- Remove an artifact entry (does not delete the file)
- **Existing endpoints modified:**
  - `GET /` -- Hub page redesigned to render artifact cards with SSE auto-reload script
  - `GET /_health` -- Unchanged
  - `GET /:filename` -- Static file serving unchanged

### Feature: Update Reminder

- **Description:** On `/rapid:install`, record an `updated_on` timestamp. When the timestamp exceeds 1 week, prompt the user to update RAPID when they next invoke any RAPID skill.
- **Complexity:** moderate
- **Dependencies:** independent

#### Decomposition

1. **Timestamp recording** -- On successful `/rapid:install` completion (after Step 4 verification passes), write `RAPID_UPDATED_ON=<ISO-date>` to the RAPID plugin `.env` file. Also write it to the shell config alongside `RAPID_TOOLS`.
2. **Staleness check function** -- Add a `checkUpdateStaleness()` function to `version.cjs` that reads `RAPID_UPDATED_ON` from environment or `.env` file, compares to current date, and returns `{stale: boolean, daysSinceUpdate: number, updatedOn: string}`.
3. **CLI integration** -- Add an `update-check` subcommand to rapid-tools.cjs that outputs the staleness check result as JSON. This allows skills to invoke `node "${RAPID_TOOLS}" update-check` and parse the result.
4. **Skill-level reminder injection** -- Add update reminder logic to high-traffic skills (init, status, execute-set) that checks staleness and displays a non-blocking reminder banner when stale. The banner reads: "RAPID was last updated {N} days ago. Run /rapid:install to get the latest version."

#### Implementation Strategies

**Option 1: .env file timestamp (environment-based)**
- Approach: Store `RAPID_UPDATED_ON` in the RAPID plugin root `.env` file (alongside existing `RAPID_TOOLS`). The `setup.sh` script and `/rapid:install` skill both write/update this timestamp on successful completion. The `checkUpdateStaleness()` function in version.cjs reads from `process.env.RAPID_UPDATED_ON` first, then falls back to parsing the `.env` file (same pattern as web-client.cjs `isWebEnabled()`).
- Pros: Follows existing RAPID patterns (web-client.cjs already reads from `.env`). No new files or storage locations. The `.env` file already exists and is managed by setup.sh. Environment variable is readable from both CLI and skill contexts.
- Cons: User must re-source shell config for env var to take effect in terminal sessions. But .env fallback handles Claude Code sessions automatically.
- Libraries/services: None new.

**Option 2: Dedicated state file**
- Approach: Write a `.rapid-update-state.json` file in the plugin root with `{updatedOn, version, method}`. More structured but adds another file to track.
- Pros: Cleaner separation of concerns. Can store additional metadata (which version was installed, installation method).
- Cons: Another file to manage. Diverges from existing .env pattern. Needs its own read/write helpers.

**Recommended:** Option 1 (.env file timestamp). It follows the established pattern in RAPID (web-client.cjs already parses .env for RAPID_WEB), requires no new files, and integrates naturally with the existing setup.sh and /rapid:install flows.

#### Data Model Impact

- **New entities:** None -- `RAPID_UPDATED_ON` is added as a new key to the existing `.env` file.
- **Existing entities modified:**
  - `.env` file gains `RAPID_UPDATED_ON=<ISO-date>` line
  - `.env.example` gains `RAPID_UPDATED_ON=` documentation line
  - `version.cjs` gains `checkUpdateStaleness()` export
- **Relationships:** The update check depends on `version.cjs` for the check logic and `web-client.cjs`'s `_parseEnvFile()` pattern for reading `.env`.

#### API Surface

- **New endpoints/commands:**
  - `rapid-tools update-check` -- CLI subcommand returning `{stale, daysSinceUpdate, updatedOn}` as JSON
- **Existing endpoints modified:**
  - `setup.sh` Step 5 -- writes `RAPID_UPDATED_ON` alongside `RAPID_TOOLS`
  - `skills/install/SKILL.md` Step 4 -- writes `RAPID_UPDATED_ON` on successful verification

### Feature: Integrate /branding into Init Flow

- **Description:** Insert the branding interview into the project initialization flow at Step 4B (after all discovery areas are complete, before the granularity preference question) to reduce the friction of discovering and running /branding separately.
- **Complexity:** moderate
- **Dependencies:** Soft dependency on "Branding Webserver Overhaul" -- the init integration can work with the existing branding skill, but the experience is better with the overhauled artifact system.

#### Decomposition

1. **Init skill modification** -- Add a new step between the current Step 4B (deep discovery) and Step 4C (granularity preference). This step offers the user the option to run branding now or skip it.
2. **Branding prompt insertion** -- Use AskUserQuestion to ask: "Would you like to set up project branding now? This defines visual identity and style guidelines for your project." with options: "Yes, run branding interview", "Skip for now (run /rapid:branding later)", and for webapp projects additionally "Yes, with full artifact generation" (logo, wireframe, guidelines).
3. **Inline branding execution** -- If the user opts in, invoke the branding skill's interview logic (Steps 2-6 of branding SKILL.md) inline within the init flow. The server is started but the lifecycle question (Step 8 of branding SKILL.md) is deferred -- the server keeps running through the rest of init.
4. **State continuity** -- Ensure the project brief compiled in Step 4B is available to the branding interview for context-aware defaults (e.g., webapp detection already done, project name known).

#### Implementation Strategies

**Option 1: Skill invocation via Agent tool**
- Approach: From within the init SKILL.md, use the Agent tool to invoke the /rapid:branding skill as a sub-agent. Pass the already-detected project type and name as context. The sub-agent runs the full branding flow, then returns control to init.
- Pros: Clean separation -- branding logic stays in its own skill file. No code duplication. The branding skill can still be run independently.
- Cons: Agent tool invocations add overhead and context window usage. The branding skill's Step 8 (server lifecycle question) needs to be skipped when called from init. Requires passing project type context across the agent boundary.
- Libraries/services: None -- uses existing Agent tool.

**Option 2: Inline branding steps in init SKILL.md**
- Approach: Copy the essential branding interview logic (Steps 2-6) directly into the init SKILL.md as a new step. Skip the full branding skill's preamble/postamble since init has already done environment setup and project detection.
- Pros: Tighter control over the flow. Can skip redundant steps (env setup, project type detection already done). Fewer agent invocations.
- Cons: Code duplication between branding SKILL.md and init SKILL.md. Maintenance burden when branding interview changes. Larger init skill file.
- Libraries/services: None.

**Recommended:** Option 1 (Skill invocation via Agent tool). While it adds one agent invocation, it avoids duplicating the branding interview logic across two skill files. The branding skill should be modified to accept an `--inline` flag or `--skip-lifecycle` flag that suppresses the server lifecycle question (Step 8) and the commit step (Step 9) when invoked from init. The init flow handles committing all artifacts together at its own Step 10.

#### Data Model Impact

- **New entities:** None.
- **Existing entities modified:**
  - `skills/init/SKILL.md` gains a new Step 4B.5 (branding offer)
  - `skills/branding/SKILL.md` gains `--inline` / `--skip-lifecycle` mode support
- **Relationships:** The branding step in init consumes the project type detected in Step 4B's codebase analysis and the project name from Step 4A.

#### API Surface

- **New endpoints/commands:** None.
- **Existing endpoints modified:**
  - `skills/init/SKILL.md` -- new step inserted between 4B and 4C
  - `skills/branding/SKILL.md` -- conditional behavior when invoked with `--inline` flag

## Feature Dependency Graph

```
Branding Webserver Overhaul (independent)
  └── Integrate /branding into Init Flow (soft dependency -- benefits from overhaul but works without it)

Update Reminder (independent)
```

The branding init integration has a soft dependency on the webserver overhaul: it functions with the current branding server, but the overhauled artifact system provides a significantly better experience when invoked from init (users can see artifacts appear live as the agent generates them).

The update reminder is fully independent of both branding features.

---

## Implementation Order

1. **Branding Webserver Overhaul** -- This is the highest-complexity feature and the one that enables the best init integration experience. The SSE auto-reload, artifact registry, and hub page redesign form the foundation. The extended branding skill flow (logo, wireframe, guidelines) builds on top.
   - Wave 1: SSE infrastructure + fs.watch + artifact registry (branding-server.cjs changes)
   - Wave 2: Hub page redesign + artifact CRUD API + branding skill extended flow

2. **Integrate /branding into Init Flow** -- Depends on the branding skill having the `--inline` mode. Even if the webserver overhaul is only partially complete, the init integration works with whatever branding experience exists.
   - Wave 1: Init SKILL.md modification + branding SKILL.md `--inline` flag

3. **Update Reminder** -- Fully independent. Can be implemented in parallel with the branding work.
   - Wave 1: Timestamp recording in setup.sh/.env + version.cjs staleness check + CLI subcommand
   - Wave 2: Skill-level reminder injection in high-traffic skills
