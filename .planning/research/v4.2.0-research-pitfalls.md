# Pitfalls Research -- v4.2.0 Discuss & Audit

## Critical Pitfalls

### Pitfall: RAPID_TOOLS Path Resolution Treats .cjs File as Directory
- **Category:** correctness
- **Stack:** skills/init/SKILL.md, src/hooks/rapid-verify.sh, Node.js `require()`
- **Description:** The init skill uses `require('${RAPID_TOOLS}/../lib/context.cjs')` to load modules relative to `RAPID_TOOLS`. When `RAPID_TOOLS` is a file path like `/store/rapid/src/bin/rapid-tools.cjs`, Node's `require()` resolves `rapid-tools.cjs/..` by treating the `.cjs` filename as a directory component. This means `rapid-tools.cjs/../lib/` resolves to `src/bin/lib/` (not `src/lib/`), because Node strips the last path component but `.cjs` is the extension, not a directory. The same pattern appears in `rapid-verify.sh` line 32 with `$(dirname "$RAPID_TOOLS")/../lib/returns.cjs`.
- **Trigger:** Any `require('${RAPID_TOOLS}/../lib/...')` call in shell-embedded Node scripts. Currently at:
  - `skills/init/SKILL.md:564` -- `context.cjs`
  - `skills/init/SKILL.md:1000` -- `web-client.cjs`
  - `src/hooks/rapid-verify.sh:32` -- `returns.cjs`
- **Impact:** Severity CRITICAL. The init skill's test framework detection step (Step 6a) fails silently because the require path is wrong. Web dashboard registration (Step 10.5) also fails. The verify hook cannot parse RAPID:RETURN markers. On some systems (Nix, read-only installs), this is the blocker that prevents init from completing at all.
- **Prevention:** Replace `${RAPID_TOOLS}/../lib/` with `$(dirname "${RAPID_TOOLS}")/../lib/` in shell contexts (which correctly strips the filename), or use `path.resolve(path.dirname(RAPID_TOOLS), '..', 'lib', 'module.cjs')` in Node contexts. The shell `dirname` approach already works correctly in `rapid-verify.sh` line 32, but the skill templates use string interpolation inside Node `require()` calls where `dirname` is not available. The correct fix is to use `path.dirname()` inside the Node `-e` script: `const libDir = require('path').resolve(require('path').dirname('${RAPID_TOOLS}'), '..', 'lib');`
- **Detection:** Run `grep -rn "RAPID_TOOLS}/../lib" skills/ src/hooks/` to find all instances. Test by setting `RAPID_TOOLS` to an absolute `.cjs` path and running the require.
- **References:** GitHub issue [#3](https://github.com/pragnition/RAPID/issues/3), `issues-todo.md` lines 27-28

### Pitfall: Discuss-Set Test Suite Hardcodes "Exactly 4 Gray Areas" Assertions
- **Category:** correctness
- **Stack:** skills/discuss-set/SKILL.test.cjs, skills/discuss-set/SKILL.md
- **Description:** The existing test suite (`SKILL.test.cjs`) contains 5 tests that assert exactly-4 gray area behavior: Test 1 asserts "exactly 4 numbered options", Test 2 asserts "no 5th option", Test 3 asserts Key Principles says "Exactly 4 gray areas", Test 4 asserts Anti-Patterns says "fewer or more than 4", and Test 5 asserts the heading mentions "4". The milestone feature "gray areas consolidation" changes the model from fixed-4 to variable (4n where n=1-3, yielding 4/8/12 gray areas). Updating SKILL.md without updating these tests will break the test suite.
- **Trigger:** Modifying SKILL.md Step 5 to support variable gray area counts (the consolidation feature) without simultaneously updating or replacing all 5 test assertions.
- **Impact:** Severity CRITICAL. The `npm test` suite will fail, blocking CI and any quality gates. The assertions check for literal strings like "Exactly 4 gray areas" and "fewer or more than 4" which must be rewritten to match the new variable-count model.
- **Prevention:** Update the test suite in the same commit as the SKILL.md changes. Replace the hardcoded "4" assertions with assertions that validate the new variable-count model (e.g., check for "multiple of 4", check that the complexity heuristic table exists, check that batched AskUserQuestion pattern is documented). Map each old test to its replacement:
  - Test 1: Assert first batch has exactly 4 options (still true -- each batch is 4)
  - Test 2: Still valid -- no 5th option per batch
  - Test 3: Change to assert "multiples of 4" or the complexity heuristic table
  - Test 4: Change to assert "non-multiple-of-4" anti-pattern
  - Test 5: Change to assert "Gray Areas" heading (without hardcoded "4")
- **Detection:** Run `node --test skills/discuss-set/SKILL.test.cjs` after SKILL.md changes. All 5 hardcoded tests will fail if not updated.
- **References:** `skills/discuss-set/SKILL.test.cjs` lines 13-72

### Pitfall: Audit-Version Has No State Schema Support for Version/Milestone Completion
- **Category:** correctness / compatibility
- **Stack:** src/lib/state-schemas.cjs, src/lib/state-transitions.cjs
- **Description:** The current `MilestoneState` schema only tracks `id`, `name`, and `sets[]`. There is no `status` field on milestones (unlike sets which have a full lifecycle). The audit-version feature needs to determine which versions are "completed" to audit them, but the schema provides no way to mark a milestone as complete, in-progress, or audited. The only signal is whether all sets within a milestone are `merged`, which requires iterating all sets and checking their statuses.
- **Trigger:** Implementing `/rapid:audit-version` and attempting to query "which versions are completed" or marking a version as "audited". There is no transition map for milestones.
- **Impact:** Severity HIGH. Without a milestone status field, the audit command must derive completion from set statuses, which is fragile (what if a set is added after all others are merged?). If the implementer adds a `status` field to `MilestoneState`, existing STATE.json files will fail Zod validation unless the field is `.optional()` or a migration is provided. If the implementer uses `.passthrough()` (already present on MilestoneState), extra fields survive but are not validated.
- **Prevention:** Define milestone completion as "all sets in milestone have status `merged`" rather than adding a new schema field. This avoids migration issues and leverages existing data. If a `status` field is desired, add it as `.optional()` with a default to maintain backward compatibility. Do NOT make it required without a migration step.
- **Detection:** Check if `MilestoneState` has a `status` field in `state-schemas.cjs`. Currently it does not (line 27-31).
- **References:** `src/lib/state-schemas.cjs` lines 27-31, `src/lib/state-transitions.cjs` (no milestone transitions exist)


## High-Risk Pitfalls

### Pitfall: AskUserQuestion Checkbox Consolidation May Exceed UI Limits
- **Category:** compatibility / correctness
- **Stack:** Claude Code AskUserQuestion API, skills/discuss-set/SKILL.md
- **Description:** The gray areas consolidation feature plans to present up to 12 gray areas across multiple AskUserQuestion calls with multiSelect. Each call would have 4 checkbox options. However, AskUserQuestion options include both a label and a description (the `--` part). If gray area titles and descriptions are long (common for architect-level decisions), the UI may truncate or render poorly. Additionally, the sequential nature of 3 separate AskUserQuestion calls (for n=3) means the user loses context from earlier batches when answering later ones.
- **Trigger:** Complex sets with 7+ tasks triggering n=3 (12 gray areas across 3 prompts). Each prompt has 4 options with potentially long descriptions.
- **Impact:** Severity MEDIUM-HIGH. Users may make uninformed selections because they cannot see all 12 areas simultaneously. Long option descriptions may be truncated in the AskUserQuestion UI, hiding important context. The multi-batch approach is necessary (since there's no way to show 12 checkboxes in one prompt cleanly) but creates a fragmented experience.
- **Prevention:** Keep gray area titles under 60 characters. Keep descriptions (the `--` part) under 100 characters. Show a running summary ("You've selected 2 areas so far from batch 1") at the start of each subsequent batch. Consider showing a final confirmation: "You selected areas X, Y, Z for discussion. Proceed?"
- **Detection:** Test with a set that has 7+ tasks and verify all 3 batches render correctly. Check that selected items from batch 1 are preserved when batch 2 is presented.
- **References:** Claude Code plugin API documentation for AskUserQuestion

### Pitfall: Pros/Cons Table Markdown Alignment Depends on Monospace Rendering
- **Category:** compatibility
- **Stack:** skills/discuss-set/SKILL.md (Format A tables), Claude Code UI
- **Description:** The discuss phase uses Markdown tables for pros/cons comparisons (Format A in Step 6). Markdown table alignment depends on the rendering engine: pipes must align for readability, but the content between pipes varies in length. The Claude Code UI renders Markdown, but the table alignment in the AskUserQuestion context block depends on how the LLM generates the table, not on any formatting library. Inconsistent column widths make tables hard to scan.
- **Trigger:** Generating a pros/cons table where one option has much longer pros/cons text than others, causing column misalignment. Also triggered when the LLM does not pad cells to equal width.
- **Impact:** Severity MEDIUM. Tables render as valid Markdown but look uneven. Users struggle to compare options when pros/cons cells have dramatically different widths. This is purely a readability issue, not a functional failure.
- **Prevention:** Two approaches: (1) Constrain cell content to max ~40 characters per cell, using bullet shorthand rather than full sentences. (2) Switch to Format C (key factors list) when any single cell would exceed 40 characters, since lists handle variable-length content better than table cells. (3) Use fixed-width separators in the Markdown table (e.g., pad with spaces to align columns). Note: approach (3) only helps if the rendering engine uses a monospace font for tables.
- **Detection:** Generate a sample table with varying cell lengths and render it in Claude Code's UI. Check if columns align and if content is readable at a glance.
- **References:** `skills/discuss-set/SKILL.md` lines 237-252 (Format A definition)

### Pitfall: Audit-Version Feature Requires Reading Artifacts Across Multiple Milestones
- **Category:** performance / correctness
- **Stack:** .planning/sets/*, STATE.json, new audit-version skill
- **Description:** The audit-version feature needs to compare what was planned (DEFINITION.md, CONTRACT.json, ROADMAP.md) against what was delivered (committed code, CONTEXT.md decisions). This requires reading many files across all sets in a milestone. For milestones with 5+ sets, each having CONTRACT.json, DEFINITION.md, CONTEXT.md, and multiple PLAN.md files, the audit agent needs to load 20-40 files. This can exceed context limits for the agent or cause slow execution.
- **Trigger:** Running `/rapid:audit-version` on a milestone with many sets and complex contracts.
- **Impact:** Severity MEDIUM-HIGH. If the audit agent tries to read all artifacts at once, it may hit context window limits. If it reads them sequentially, execution time balloons. Incomplete artifact loading leads to false "gap" reports (the feature was delivered but the auditor didn't load the right file).
- **Prevention:** Design the audit to work in two passes: (1) a summary pass that reads only DEFINITION.md and set status from STATE.json to build a checklist, then (2) a targeted deep-dive pass that reads CONTRACT.json and source files only for sets flagged as potentially incomplete. Limit the deep-dive to at most 3 sets per agent invocation.
- **Detection:** Test with a milestone containing 6+ sets. Monitor whether the agent truncates its analysis or misses sets.
- **References:** `.planning/sets/*/DEFINITION.md`, `.planning/sets/*/CONTRACT.json`

### Pitfall: path.dirname() in Shell vs Node Contexts Behaves Differently
- **Category:** correctness
- **Stack:** Shell scripts, Node.js inline scripts, RAPID_TOOLS path resolution
- **Description:** The fix for the RAPID_TOOLS path bug must account for the difference between shell `dirname` and Node's `path.dirname()`. In shell: `dirname "/path/to/rapid-tools.cjs"` returns `/path/to`. In Node: `path.dirname("/path/to/rapid-tools.cjs")` also returns `/path/to`. However, the current broken pattern uses string interpolation inside `require()`: `require('${RAPID_TOOLS}/../lib/...')`. When `RAPID_TOOLS=/path/to/rapid-tools.cjs`, this becomes `require('/path/to/rapid-tools.cjs/../lib/...')`. Node's `require()` resolution treats each `..` as a directory traversal, so `rapid-tools.cjs/..` goes up from `rapid-tools.cjs` (treated as a directory name) to `/path/to/`, which accidentally works on most systems. The actual failure occurs when the filesystem check fails because `rapid-tools.cjs` is not a directory.
- **Trigger:** The bug manifests differently across platforms. On Linux with standard filesystems, `require('/path/to/file.cjs/../lib/x.cjs')` often works because `path.resolve` normalizes it before the filesystem check. The bug is most visible on Nix, where symlinked paths and read-only stores prevent the traversal. It also fails when `RAPID_TOOLS` contains spaces or special characters.
- **Impact:** Severity MEDIUM-HIGH. The fix seems simple but has platform-specific edge cases. A naive fix using shell `dirname` inside a `node -e` script may break if the path contains spaces. A fix using Node's `path.dirname()` inside the inline script requires careful quoting of the interpolated variable.
- **Prevention:** Use a two-step pattern in inline Node scripts:
  ```javascript
  const rapidLib = require('path').resolve(require('path').dirname(process.env.RAPID_TOOLS || ''), '..', 'lib');
  const { fn } = require(require('path').join(rapidLib, 'module.cjs'));
  ```
  This reads `RAPID_TOOLS` from the environment at runtime rather than relying on shell interpolation into the require string. Alternatively, add a `resolve-lib` subcommand to rapid-tools.cjs that prints the lib directory path.
- **Detection:** Test with `RAPID_TOOLS` set to paths containing spaces, symlinks, and Nix store paths.
- **References:** GitHub issue [#3](https://github.com/pragnition/RAPID/issues/3), Node.js `require()` resolution algorithm documentation


## Medium-Risk Pitfalls

### Pitfall: Gray Area Count Heuristic Based on Task Count May Be Unreliable
- **Category:** correctness
- **Stack:** skills/discuss-set/SKILL.md, CONTRACT.json
- **Description:** The consolidation feature uses `CONTRACT.json definition.tasks` array length as the primary signal for determining gray area count (n). However, CONTRACT.json may not have a `definition.tasks` field (it is not required by the contract schema), or the tasks array may be empty early in the lifecycle (contracts can be minimal stubs). If the tasks array is missing or empty, the heuristic falls through with no clear default.
- **Trigger:** Running discuss-set on a set whose CONTRACT.json has no `definition.tasks` field, or where the field exists but is empty.
- **Impact:** Severity MEDIUM. The skill may either crash (if it does `tasks.length` on undefined) or default to 0 tasks, yielding n=0 gray areas (skipping discussion entirely). Both outcomes break the user experience.
- **Prevention:** Default to n=1 (4 gray areas) when `definition.tasks` is missing or empty. Document the fallback in SKILL.md. Add a guard: `const taskCount = contract?.definition?.tasks?.length || 0; const n = taskCount <= 3 ? 1 : taskCount <= 6 ? 2 : 3;`
- **Detection:** Test discuss-set with a minimal CONTRACT.json that has no `definition` field.
- **References:** `skills/discuss-set/SKILL.md` Step 5 complexity heuristic table

### Pitfall: Deferred Items from Audit May Conflict with Existing DEFERRED.md Files
- **Category:** data-loss
- **Stack:** .planning/sets/*/DEFERRED.md, audit-version skill
- **Description:** The audit-version feature may identify gaps that should be deferred to future versions. If it writes deferred items, these could overwrite or conflict with existing DEFERRED.md files that were created during the discuss-set phase. Each set already has a DEFERRED.md written in Step 6.5 of discuss-set.
- **Trigger:** Running audit-version on a completed milestone and attempting to write deferred items to existing set directories.
- **Impact:** Severity MEDIUM. Existing deferred items from the original discussion could be lost if the audit overwrites DEFERRED.md. Even if the audit creates a separate file, the planning tools expect DEFERRED.md at a specific path.
- **Prevention:** The audit-version skill should write to a milestone-level deferred file (e.g., `.planning/milestones/{id}/AUDIT-DEFERRED.md`) rather than per-set DEFERRED.md files. This keeps audit output separate from discussion output. Alternatively, append to existing DEFERRED.md files with a clear "Audit Additions" section header.
- **Detection:** Check if `.planning/sets/*/DEFERRED.md` exists before the audit writes. If it does, verify the audit does not overwrite it.

### Pitfall: discuss-set State Transition "discussed" vs "discussing" Naming Inconsistency
- **Category:** correctness
- **Stack:** skills/discuss-set/SKILL.md, state-transitions.cjs
- **Description:** The SKILL.md already uses the correct status `discussed` for the transition (fixed in a previous milestone), and the state machine allows `discussed -> discussed` self-transition for re-discuss scenarios. However, the status check in Step 2 references `discussed` as the re-discuss state while displaying "in progress" language. This is correct but potentially confusing when combined with the new variable gray area feature: a partially-completed multi-batch discussion that is interrupted will leave the set in `pending` (never transitioned), not `discussed`. There is no "discussing" intermediate state.
- **Trigger:** User starts a discuss-set flow with n=3 (3 AskUserQuestion batches), answers batch 1, then cancels or the session dies. The set remains `pending` because the transition happens in Step 8 (after all discussion). The user re-runs discuss-set and gets the "fresh start" flow, losing batch 1 responses.
- **Impact:** Severity LOW-MEDIUM. No data corruption, but the user loses their partial discussion progress. With the new multi-batch approach (up to 3 sequential prompts), the risk of session interruption between batches increases.
- **Prevention:** This is a design tradeoff, not a bug. The current approach (transition at the end) is correct because partial discussions should not be committed. However, the SKILL.md could note that interrupted multi-batch discussions must be restarted from the beginning. No code change needed.
- **Detection:** Interrupt a discuss-set session after answering the first batch. Re-run discuss-set and verify the set is still in `pending`.

## Anti-Patterns for This Stack

### Anti-Pattern: Embedding require() Paths via Shell Variable Interpolation
- **What people do:** `node -e "const x = require('${RAPID_TOOLS}/../lib/module.cjs');"` -- interpolating a shell variable directly into a Node require path.
- **Why it fails:** Shell interpolation happens before Node executes. If `RAPID_TOOLS` contains a file path (not a directory), the `..` traversal treats the filename as a directory component. Also breaks with spaces in paths, single quotes in paths, or symlinks.
- **Correct approach:** Use `process.env.RAPID_TOOLS` inside the Node script and resolve with `path.dirname()`:
  ```javascript
  node -e "
    const p = require('path');
    const lib = p.resolve(p.dirname(process.env.RAPID_TOOLS), '..', 'lib');
    const { fn } = require(p.join(lib, 'module.cjs'));
  "
  ```
- **Example:** The init skill line 564 uses the broken pattern. The verify hook line 32 uses `dirname` in shell but then still passes via interpolation.

### Anti-Pattern: Testing SKILL.md Against Literal Strings Instead of Structural Properties
- **What people do:** Assert that SKILL.md contains the exact string "Exactly 4 gray areas" or "fewer or more than 4".
- **Why it fails:** Any change to the skill's parameters (like changing from 4 to variable 4n) breaks the test, even if the structural intent is preserved. The test becomes a change detector rather than a correctness validator.
- **Correct approach:** Test structural properties: "Step 5 has a complexity heuristic table", "each AskUserQuestion batch has exactly 4 options", "Anti-Patterns warns against non-multiple-of-4 counts". These are stable across parameterization changes.
- **Example:** `SKILL.test.cjs` tests 3-5 all break when the gray area count becomes variable.

### Anti-Pattern: Deriving Milestone Completion from Absence of State Fields
- **What people do:** Check "are all sets merged?" by iterating sets and checking statuses, but miss edge cases like milestones with zero sets, or sets added after the "completion" check.
- **Why it fails:** A milestone with 0 sets would appear "complete" (vacuously true). A set added mid-audit would change the completion status between checks.
- **Correct approach:** Snapshot the set list at audit start. Require at least one set in the milestone. Check `sets.length > 0 && sets.every(s => s.status === 'merged')`. Document that the audit operates on a point-in-time snapshot.
- **Example:** The audit-version feature must handle the zero-sets edge case explicitly.


## Version-Specific Gotchas

- **Node.js 18:** `node:test` `describe` blocks do not support the `timeout` option (added in Node 20.3). Tests for the audit feature that spawn subprocesses may hang indefinitely.
- **Zod 3.25.76:** The `.passthrough()` modifier preserves unknown keys but does not validate them. If `MilestoneState` is extended with an audit-related field via schema changes, existing STATE.json files with the old schema will parse without error (passthrough ignores unknown fields), but new fields will not be validated on old files unless `.default()` is used.
- **proper-lockfile 4.1.2:** The `stale` option defaults to 10000ms. If audit reads take longer than the stale threshold, another process may steal the lock. For read-only audit operations, avoid locking entirely -- use `readState()` instead of `withStateTransaction()`.
- **Git worktrees (Git 2.30+):** `git worktree list` output format changed slightly between Git 2.30 and 2.38 (additional fields in `--porcelain` mode). The audit feature should not rely on worktree state since it audits completed milestones where worktrees may have been cleaned up.


## Security Pitfalls

1. **Shell injection via set names in audit CLI:** If audit-version accepts user-provided version/milestone names and passes them to shell commands without quoting, names containing shell metacharacters (`$(...)`, backticks, semicolons) could execute arbitrary commands. Prevention: always quote variables in shell (`"${MILESTONE_ID}"`) and validate milestone IDs against the schema before use.
2. **Path traversal in artifact loading:** If the audit feature constructs file paths from set IDs (e.g., `.planning/sets/${setId}/DEFINITION.md`), a malicious set ID like `../../etc/passwd` could read arbitrary files. Prevention: validate set IDs match the expected pattern (alphanumeric + hyphens only) before constructing paths. The existing `resolveSet()` function does not perform this validation.


## Performance Traps

1. **Loading all set artifacts for audit:** Reading CONTRACT.json, DEFINITION.md, CONTEXT.md, SET-OVERVIEW.md, and all PLAN.md files for every set in a milestone creates O(n * m) file reads where n is sets and m is artifacts per set. Mitigation: use a two-pass approach (summary pass, then targeted deep-dive) and cache file contents.
2. **Sequential AskUserQuestion calls in multi-batch discuss:** With n=3, the user faces 3 sequential AskUserQuestion prompts before the deep-dive begins, then potentially 4-12 more prompts for the selected areas. This can feel like an interrogation. Mitigation: keep the selection prompts fast (clear titles, short descriptions) and limit deep-dive questions to 1-2 per selected area.
3. **State locking during audit reads:** Using `withStateTransaction()` for read-only audit operations holds the state lock unnecessarily, blocking other concurrent operations (discuss, plan, execute). Mitigation: use `readState()` (no lock) for audit reads. Only lock if the audit needs to write deferred items back to state.


## Summary Priority List

1. **RAPID_TOOLS path resolution (.cjs treated as directory)** -- Blocks init on Nix/immutable installs, silently fails test framework detection on all platforms. Must fix with `path.dirname()` pattern.
2. **Discuss-set test suite hardcodes "exactly 4"** -- Will break the entire test suite when gray area consolidation is implemented. Must update tests in same commit as SKILL.md changes.
3. **Audit-version has no milestone status in schema** -- Must decide on derived-completion vs. new-field approach before implementation begins. Wrong choice causes migration headaches.
4. **CONTRACT.json tasks field may be missing** -- Gray area count heuristic will crash or produce 0 areas if tasks field is absent. Must add a fallback default.
5. **Shell variable interpolation in require() paths** -- Root cause of the path resolution bug. Fix must handle spaces, symlinks, and cross-platform paths. Use `process.env` inside Node scripts instead of shell interpolation.
6. **AskUserQuestion multi-batch UX fragmentation** -- Users lose context across 3 sequential prompts for n=3. Keep titles short and show running selection summaries.
7. **Audit artifact loading may exceed context limits** -- Use two-pass approach (summary then deep-dive) to avoid loading 20-40 files at once.
8. **Deferred items from audit may overwrite discuss-set DEFERRED.md** -- Write audit output to a separate milestone-level file, not per-set DEFERRED.md.
