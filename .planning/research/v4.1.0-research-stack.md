## Compatibility Matrix

### No Blocking Conflicts Detected

The current stack has no version conflicts that would block milestone work. Key compatibility facts:

1. **Zod 3.25.76 + CommonJS:** Works correctly. CJS loading issues existed in 3.25.7-3.25.16 but were fixed in 3.25.17. The installed version (3.25.76) is well past the fix. Confirmed via runtime test: `require('zod')` succeeds.

2. **Node 25 + node:test:** The built-in test runner is fully stable in Node 24+ and works without issues on Node 25. All test features used by RAPID (describe, it, before, after, mock) are stable APIs.

3. **FastAPI + Python 3.14:** FastAPI 0.135 requires Python 3.9+. The system runs Python 3.14.3, which is well ahead. No compatibility issues. Note: FastAPI has deprecated Pydantic v1 support; RAPID uses Pydantic v2 via SQLModel, so this does not apply.

4. **Vite 8 + React 19:** Vite 8 uses Rolldown (replacing esbuild + Rollup). The @vitejs/plugin-react ^6.0.1 is compatible with Vite 8. No migration needed.

5. **Tailwind CSS 4 + Vite:** Already configured via @tailwindcss/vite plugin with CSS-native @theme configuration. No tailwind.config.js to migrate.

6. **React Router 7:** Already in use. No migration from v6 needed. Package consolidation (no react-router-dom) is already adopted.

### Constraints for Issue #31 (Framework-Agnostic Testing)

Making the unit-test skill framework-agnostic requires detecting the target project's test runner. The context.cjs module already detects languages via manifest files. The detection must extend to:

| Language | Manifest | Test Runner Detection |
|----------|----------|----------------------|
| JavaScript/TypeScript | package.json | Check scripts.test, presence of jest.config, vitest.config, .mocharc |
| Python | pyproject.toml, setup.py | Check for pytest in dependencies, tox.ini |
| Rust | Cargo.toml | Always `cargo test` |
| Go | go.mod | Always `go test` |

No new RAPID dependencies are needed for this detection -- it is filesystem-based analysis using existing `context.cjs` patterns.

## Upgrade Paths

### Zod 3.x to Zod 4.x (NOT recommended for this milestone)

- **Current:** 3.25.76
- **Latest:** 4.3.6
- **Breaking changes:**
  - `message` param replaced with `error` param in schema methods
  - String format validators moved to top-level (`z.string().email()` becomes `z.email()`)
  - `z.number()` no longer accepts Infinity
  - Object `.merge()` deprecated in favor of `.extend()`
  - ZodError issue format restructured
  - UUID validation strictened to RFC 9562/4122
- **Migration path:** Zod 4 is available at `zod/v4` subpath. Both versions can coexist in the same package via `require('zod/v3')` and `require('zod/v4')`. An automated codemod exists: [zod-v3-to-v4](https://github.com/nicoespeon/zod-v3-to-v4).
- **CJS compatibility:** Zod 3.25.76 works with CJS. Zod 4's CJS support via `require('zod/v4')` has reported issues (GitHub issue #4372). Given RAPID's all-CJS codebase, this is a risk.
- **Effort estimate:** Moderate. RAPID uses Zod extensively in state-schemas.cjs, state-machine.cjs, and returns.cjs. Every schema would need API adjustments.
- **Recommendation:** Do NOT upgrade in this milestone. The current version is functional and the migration carries CJS risk. Defer to a dedicated milestone. Priority: **low**.

### proper-lockfile (No upgrade available)

- **Current/Latest:** 4.1.2 (last updated 5+ years ago)
- **Status:** Unmaintained but functional. Uses mkdir-based atomic locking which is inherently stable.
- **Alternatives:** `lockfile` (npm), custom implementation using `fs.mkdirSync` with `{recursive: false}` for atomic mkdir.
- **Recommendation:** No action needed. The library is simple, does one thing, and does it reliably. The lack of updates is a signal of completeness, not abandonment. The API surface is tiny and RAPID wraps it in lock.cjs. Priority: **low**.

### cytoscape-dagre (Monitor but no action)

- **Current/Latest:** 2.5.0 (last updated 3+ years ago)
- **Status:** Low maintenance. The core cytoscape library (3.33.1) is active, but the dagre layout plugin has not been updated.
- **Risk:** Relevant to enhancement #3 (add back DAG display in command center). The plugin works and dagre layouts are well-suited for DAG visualization. No functional issues reported.
- **Recommendation:** Use as-is for the DAG visualization feature. If layout issues arise, consider `cytoscape-elk` or `cytoscape-fcose` as alternatives. Priority: **low**.

## Stack Risks

1. **Zod CJS compatibility on future updates (Medium):** The `^3.25.76` caret range could auto-update to a Zod 3.x version that reintroduces CJS issues if published. The 3.25.7-3.25.16 CJS breakage demonstrated this risk. **Mitigation:** The package-lock.json pins the exact version. As long as `npm install` is run with the lock file, this is safe.

2. **RAPID_ROOT proliferation (Low, actively harmful):** Issue #3 identifies 124 RAPID_ROOT references across 26 skill files, plus 6 in src/modules/roles/. These are computed from `CLAUDE_SKILL_DIR` but only used to load `.env`. The variable name is misleading and the computation is fragile. **Mitigation:** This milestone explicitly addresses removal (issue #3).

3. **proper-lockfile abandonment (Low):** 5+ years without updates. If a Node.js version introduces breaking changes to `fs.mkdir` behavior, this library would not be patched. **Mitigation:** RAPID wraps it in lock.cjs, making replacement straightforward if needed.

4. **cytoscape-dagre stale plugin (Low):** Relevant to DAG visualization feature (#3 enhancement). The plugin works but has no maintainer response if issues arise. **Mitigation:** The core cytoscape library is maintained. Alternative layout plugins exist.

5. **Node 25 EOL (Low):** The development machine runs Node 25 (non-LTS, EOL ~June 2026). This does not affect the project since it enforces only Node 18+ minimum. **Mitigation:** Informational only. Consider testing against Node 24 LTS for release validation.

6. **Python 3.14 early adoption (Low):** The backend requires Python 3.12+ but the system runs 3.14.3. Python 3.14 may have minor ecosystem compatibility gaps with some C extensions. **Mitigation:** FastAPI and all listed dependencies support Python 3.12-3.14. No issues detected.

## Recommendations

Ordered by relevance to the v4.1.0 milestone scope:

1. **Add test runner detection to context.cjs** (for #31): Extend the existing manifest detection in `src/lib/context.cjs` to identify the project's test framework. This is a prerequisite for making the unit-test skill framework-agnostic. Priority: **high** (blocks #31).

2. **Sweep RAPID_ROOT from skills and roles** (for #3): Remove all 130 RAPID_ROOT references. Replace the env-loading preamble pattern in skills to derive RAPID_TOOLS directly without the intermediate variable. Priority: **high** (explicit milestone scope).

3. **Remove 5-group limit in unit-test skill** (for #29): The limit at line 113 of `skills/unit-test/SKILL.md` can be removed. Consider adding a soft warning when groups exceed 8-10 to alert about token usage, but do not enforce a hard cap. Priority: **medium**.

4. **Update node --test references in skills and roles** (for #31): Replace hardcoded `node --test` commands in unit-tester, verifier, and bugfix roles with a configurable test runner command sourced from project config or auto-detection. Files to update: 4 role modules, 3 generated agents, 1 skill. Priority: **medium** (core of #31).

5. **Pin Zod version more tightly** (risk mitigation): Consider changing `^3.25.76` to `~3.25.76` to prevent auto-updating to potentially breaking 3.x releases. This is a one-character change with meaningful risk reduction given the CJS breakage history. Priority: **low**.

6. **Do NOT upgrade to Zod 4 this milestone:** The migration is non-trivial, carries CJS risk, and provides no features needed for the polish/fix scope. Defer to a future milestone. Priority: **not applicable**.

7. **Monitor cytoscape-dagre for DAG feature** (for #3 enhancement): The plugin works for DAG visualization but has no active maintainer. If layout quality issues arise during DAG display implementation, evaluate `cytoscape-elk` as an alternative. Priority: **low** (contingency only).
