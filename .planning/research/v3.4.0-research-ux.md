# Domain & UX Research: v3.4.0 Agent Intelligence

## Domain Conventions

### How Similar Products Work

RAPID operates in the emerging domain of **multi-agent development orchestration** -- tools that coordinate multiple AI coding agents working in parallel on a shared codebase. The competitive landscape has grown substantially since RAPID's inception:

1. **Claude Code Agent Teams** (Anthropic, native): Built-in coordination of multiple Claude Code sessions with one lead and multiple teammates. Experimental, conversation-based coordination. No persistent planning state or DAG-driven dependency ordering. RAPID provides significantly more structure. (Source: [Claude Code Agent Teams docs](https://code.claude.com/docs/en/agent-teams))

2. **Agent Orchestrator** (ComposioHQ): Agent-agnostic orchestrator supporting Claude Code, Codex CLI, and Aider. Auto-detects language and platform. Focuses on CI fixes, merge conflicts, and code reviews. Less structured lifecycle than RAPID; no per-set discussion or contract system. (Source: [github.com/ComposioHQ/agent-orchestrator](https://github.com/ComposioHQ/agent-orchestrator))

3. **Claude Code Workflow Orchestration** (barkain): Plugin for multi-step workflow orchestration with automatic task decomposition and parallel agent execution. Closer to RAPID in concept but lacks RAPID's multi-phase lifecycle (discuss -> plan -> execute -> review -> merge). (Source: [github.com/barkain/claude-code-workflow-orchestration](https://github.com/barkain/claude-code-workflow-orchestration))

4. **Claude Flow** (ruvnet/ruflo): Enterprise-grade multi-agent swarm platform with RAG integration. Focuses on distributed swarm intelligence rather than structured parallel development with contracts. (Source: [github.com/ruvnet/ruflo](https://github.com/ruvnet/ruflo))

5. **GitHub Copilot Memory**: GitHub has deployed repository-scoped persistent memory in Copilot CLI. Cross-agent memory reduces re-establishing context at the start of each task. This is directly relevant to RAPID's "memory" milestone item. (Source: [GitHub Blog](https://github.blog/ai-and-ml/github-copilot/building-an-agentic-memory-system-for-github-copilot/))

6. **Deep Agent CLI**: Introduces persistent memory systems that allow AI agents to learn and evolve alongside developers -- accumulating knowledge, remembering project context, and building expertise over time. (Source: [FlowHunt blog](https://www.flowhunt.io/blog/deep-agent-cli-intelligent-coding-assistants-persistent-memory/))

### Standard Terminology and Naming

In this domain, the following terminology has become standard (2025-2026):

| RAPID Term | Industry Standard | Notes |
|------------|-------------------|-------|
| Set | "task", "workstream", "sprint" | RAPID's "set" is unique but understood by users familiar with the tool |
| Wave | "phase", "step", "batch" | "Wave" is distinctive; most tools use "phase" |
| Skill | "slash command", "skill" | Claude Code ecosystem uses both interchangeably |
| Worktree | "workspace", "sandbox" | "Worktree" is technically precise (git term); most tools say "workspace" |
| Hook | "hook", "callback", "lifecycle event" | "Hook" is the universal term; Claude Code's documentation uses it |
| Contract | "interface contract", "API spec" | RAPID's term is clear in context |
| Quick task | "ad-hoc task", "one-off", "hotfix" | Most tools say "quick fix" or "hotfix" |
| Discuss-set | (no standard equivalent) | RAPID's pre-planning discovery phase is unique in this space |

### Common Workflows and User Journeys

The standard workflow in multi-agent development tools follows a pattern RAPID already mirrors:

1. **Init/Setup** -- One-time project onboarding (universal)
2. **Decompose** -- Break work into parallel units (RAPID does this well with DAG)
3. **Isolate** -- Give each unit a sandbox (RAPID: worktrees; others: branches/containers)
4. **Execute** -- Agents implement in isolation (universal)
5. **Verify** -- Quality checks before integration (RAPID: adversarial review pipeline)
6. **Integrate** -- Merge back to main (universal)

Where RAPID is distinctive: the **discuss-set** phase (requirement elicitation before planning) and the **adversarial review pipeline** (hunter/devils-advocate/judge) have no direct equivalent in competing tools. These are competitive advantages.

## Interaction Models

### Primary Interaction Paradigm

RAPID is a **CLI-first, slash-command-driven** tool operating within the Claude Code terminal environment. The interaction model combines:

1. **Slash commands** (`/rapid:*`) -- Explicit user invocations for lifecycle operations
2. **Subagent orchestration** -- Skills spawn specialized agents for autonomous work
3. **Interactive prompts** (`AskUserQuestion`) -- Decision points where user input is required
4. **Fire-and-forget pipelines** -- Multi-agent sequences that run autonomously after initial input (e.g., `/rapid:quick`)

This matches the emerging standard in the Claude Code ecosystem: slash commands for explicit invocation, skills for richer workflows, and CLAUDE.md for always-true conventions. (Source: [alexop.dev](https://alexop.dev/posts/claude-code-customization-guide-claudemd-skills-subagents/))

### Input/Output Patterns Users Expect

**Current RAPID patterns that align with conventions:**

- **Numeric shorthand** (`/rapid:start-set 1`): Matches CLI tool conventions for positional arguments. Users expect this.
- **Progress breadcrumbs** (`init [done] > start-set > discuss-set > ...`): Effective for multi-step workflows. This is a RAPID-original pattern that is well-received.
- **Banner rendering** (ANSI-colored stage headers): Standard for CLI tools that need visual phase separation. Color-coding by phase type (blue=planning, green=execution, red=review) follows traffic-light conventions.
- **Structured JSON to stdout / human text to stderr**: Follows Unix conventions. Machine consumers parse stdout; humans read stderr.

**Patterns users expect that RAPID could improve:**

- **Activity logging/audit trail**: Users expect to be able to see "what happened" after the fact. `/rapid:quick` currently does not log to any persistent state (this is todo item 7). GitHub Copilot and Deep Agent CLI both persist activity records across sessions.
- **Memory of user preferences**: Users expect the tool to remember their decisions from previous sessions. Currently, RAPID's `CONTEXT.md` captures decisions per-set but there is no cross-session preference memory (todo item 5).
- **Undo/rollback visibility**: Users expect to see what can be undone. RAPID's git-based approach provides this implicitly but does not surface it explicitly.

### Feedback and Progress Indication Conventions

The domain standard (2025-2026) for multi-agent tools includes:

1. **Stage banners**: RAPID already does this well with colored ANSI banners.
2. **Progress indicators during long operations**: Spinners/status messages during agent execution. Claude Code supports `statusMessage` on hooks for this purpose.
3. **Completion summaries**: Structured output showing what was accomplished, artifacts produced, and next steps. RAPID does this consistently.
4. **Error breadcrumbs**: RAPID's pattern of `[step done] > [step FAILED: reason] > [pending steps]` is effective and somewhat unique. Most competing tools simply print error messages without workflow context.

### Error Communication Patterns

RAPID's current error pattern is strong:

```
[RAPID ERROR] Quick task failed at: {step name}
What's done: {what completed before failure}
Next: {what to run to recover}
```

This "What happened / What's done / What to do next" triple is the gold standard for CLI error UX. The breadcrumb pattern reinforces this. No changes needed here -- this is a competitive advantage.

## Information Architecture

### How Information Should Be Organized and Presented

RAPID's `.planning/` directory structure is the project's information architecture. The current layout is well-organized:

```
.planning/
  STATE.json          -- Machine-readable truth
  ROADMAP.md          -- Human-readable plan
  PROJECT.md          -- Project description
  REQUIREMENTS.md     -- Feature requirements
  config.json         -- Runtime configuration
  sets/{id}/          -- Per-set artifacts
    CONTRACT.json     -- Interface boundaries
    CONTEXT.md        -- Implementation vision
    DEFINITION.md     -- Set scope
    wave-*-PLAN.md    -- Wave execution plans
  research/           -- Research outputs
  context/            -- Codebase analysis
  quick/              -- Ad-hoc task artifacts
```

**For the v3.4.0 milestone, information architecture impacts include:**

1. **Memory storage** (todo item 5): Needs a new `.planning/memory/` directory or a `MEMORY.json` file at the `.planning/` root. This should store user preferences, decisions, and contextual notes that persist across sessions and milestones. Industry convention (GitHub Copilot, Deep Agent CLI) is to use a structured store (vector DB or JSON) keyed by topic/context.

2. **Quick task logging** (todo item 7): Currently `.planning/quick/{id}-{slug}/` stores per-task artifacts. Users expect a consolidated activity log (e.g., `.planning/quick/ACTIVITY.jsonl`) that serves as a quick-reference history. JSONL format matches the existing convention in `rapid-task-completed.sh`.

3. **UI specs/contracts** (todo item 3): Needs to extend the existing `CONTRACT.json` schema or add a parallel `UI-CONTRACT.json` per set. This should define component interfaces, prop contracts, and layout agreements between sets that produce UI code.

4. **Hooks state** (todo item 4): Hook results should be logged somewhere agents can check. A `.planning/hooks/` directory with JSONL logs per agent completion would match the existing `.planning/teams/` pattern.

### Navigation and Discovery Patterns

Users navigate RAPID through three mechanisms:

1. **`/rapid:status`** -- Dashboard showing all sets and their states. This is the primary discovery point.
2. **`/rapid:help`** -- Command reference.
3. **Progress breadcrumbs** -- Contextual "you are here" indicators shown after every operation.

For the "Agent Intelligence" milestone, navigation should be enhanced with:

- **Memory-aware context**: When a user starts a new session, the tool should surface relevant memory (prior decisions, user preferences) automatically rather than requiring the user to re-state them.
- **Quick task history**: `//rapid:status` or a dedicated command should show recent quick task activity, not just set statuses.

### Content Hierarchy and Progressive Disclosure

RAPID follows good progressive disclosure:

- **Level 1** (always shown): Status dashboard, progress breadcrumbs, next-step suggestions
- **Level 2** (on request): Set details, wave plans, contract contents
- **Level 3** (deep dive): Research outputs, verification reports, conflict resolution details

This hierarchy should be maintained. The new features should slot into existing levels:

- Memory/preferences: Level 1 (auto-surfaced at session start)
- Hook validation results: Level 2 (shown when relevant, e.g., after agent completion)
- Documentation generation: Level 2 (on explicit invocation)
- UI contracts: Level 2 (during plan-set and execute-set)

## Accessibility Considerations

### Key Accessibility Patterns for CLI Interaction

RAPID is a terminal-based CLI tool. Key accessibility patterns:

1. **Color is not the sole indicator**: RAPID uses ANSI colors for banners (blue/green/red backgrounds) but also includes text labels ("INITIALIZING", "EXECUTING SET"). The verb text provides meaning independent of color. This is good practice.

2. **Screen reader compatibility**: Terminal tools inherently work with screen readers since all output is text. RAPID's structured output (tables, breadcrumbs, clear labels) supports this well.

3. **Keyboard-only operation**: All RAPID interactions are keyboard-driven (slash commands, `AskUserQuestion` with numbered options). No mouse interaction required. This is inherent to the CLI paradigm.

4. **Machine-readable output**: RAPID separates JSON (stdout) from human text (stderr). This allows assistive tools to parse output programmatically.

### Potential Improvement

- **`NO_COLOR` support**: The `display.cjs` module notes "Always outputs colors (no NO_COLOR checking)." The `NO_COLOR` convention (see [no-color.org](https://no-color.org/)) is an industry standard for respecting user preferences to disable ANSI color codes. This should be addressed -- it affects users with visual impairments who use high-contrast terminal themes. Priority: medium.

## User Expectations

### Default Behaviors Users Expect

Based on domain research, users of multi-agent development orchestration tools expect:

1. **Idempotent operations**: Running the same command twice should not produce duplicate state. RAPID handles this well with status checks and re-entry detection.

2. **Graceful interruption**: Ctrl+C should not corrupt state. RAPID's file-locking and atomic write patterns support this, but the experience could be smoother with explicit pause/resume semantics (which `/rapid:pause` already provides).

3. **Automatic context restoration**: When resuming work in a new session, the tool should know what was in progress. RAPID's artifact-based re-entry detection (WAVE-COMPLETE.md markers) handles this for execution. The "memory" feature (todo item 5) would extend this to user preferences and decisions.

4. **Explainable agent behavior**: Users want to understand what agents did and why. RAPID's structured returns (RAPID:RETURN protocol) and verification reports provide this. Hooks (todo item 4) would add a post-completion validation layer.

5. **Non-destructive defaults**: Commands should not delete or overwrite without confirmation. RAPID follows this convention (e.g., cleanup prompts, archive-vs-keep options in new-version).

### Terminology Users Use vs Technical Terms

| User Says | RAPID Term | Recommendation |
|-----------|-----------|----------------|
| "What's the status?" | `/rapid:status` | Aligned |
| "Start working on X" | `/rapid:start-set` then `/rapid:discuss-set` then `/rapid:plan-set` then `/rapid:execute-set` | Multiple steps can feel heavy; the `/rapid:quick` shortcut exists for lightweight tasks |
| "I changed my mind" | No equivalent | This is the "memory" todo item -- users expect mid-stream pivots to be captured |
| "What did we do?" | No consolidated view | The "quick logging" todo item addresses this for quick tasks; consider broader activity logging |
| "Run the hooks" | N/A (hooks run automatically) | Users expect hooks to be transparent -- fire automatically, report only on failure |
| "Generate the docs" | `/rapid:documentation` (new) | Should follow the `/rapid:context` pattern of spawning a subagent that analyzes the codebase |
| "Add something extra" | `/rapid:add-set` | Well-named and discoverable |

### Mental Models from Competing Products

Users coming from competing tools will have these mental models:

1. **From GitHub Copilot**: Expect persistent memory that "just works" across sessions. Memory should be automatic, not requiring explicit save/load operations.

2. **From Cursor/Windsurf IDE agents**: Expect inline UI for showing agent progress and diffs. RAPID is CLI-only, so equivalent is clear textual progress indicators (which RAPID already provides).

3. **From CI/CD tools (GitHub Actions, etc.)**: Expect hooks to be configuration-driven (YAML/JSON), not code-driven. Claude Code's hooks system uses JSON configuration, which aligns well. RAPID should use Claude Code's native hook system rather than inventing its own.

4. **From project management tools (Jira, Linear)**: Expect status boards and activity timelines. RAPID's `/rapid:status` dashboard fills this role but could be richer with activity history.

### Onboarding and Learning Curve Expectations

Users expect:

- **5-minute quick start**: `/rapid:install` -> `/rapid:init` -> first set executing. RAPID's current flow is close but the init discovery conversation (4 batch questions) can take 10-15 minutes. This is acceptable for the depth it provides.
- **Command discoverability**: Tab completion for `/rapid:*` commands is handled by Claude Code's native skill discovery. This works well.
- **Progressive complexity**: Start with `/rapid:quick` for simple tasks, graduate to full set lifecycle for complex work. RAPID supports this progression.

## Recommendations

Ordered by priority for the v3.4.0 "Agent Intelligence" milestone:

1. **Use Claude Code's native hook system for agent completion validation (todo item 4)**: Claude Code provides `SubagentStop` and `Stop` lifecycle events with `agent` handler type that can spawn verification subagents. RAPID should register hooks via the plugin's `hooks/hooks.json` that fire on `SubagentStop` to check if the agent updated project state. This is the industry-standard approach and avoids reinventing hook infrastructure. -- Priority: critical

2. **Implement structured memory persistence (todo item 5)**: Create a `.planning/MEMORY.json` (or `.planning/memory/` directory) that stores user decisions, preferences, and contextual notes. This should be: (a) automatically populated during `/rapid:discuss-set` and `/rapid:new-version`, (b) auto-surfaced via `SessionStart` hook or CLAUDE.md injection at session start, (c) queryable by agents during planning and execution. Follow GitHub Copilot's pattern of repository-scoped memory keyed by topic. -- Priority: critical

3. **Add activity logging for /rapid:quick (todo item 7)**: Write a `.planning/quick/ACTIVITY.jsonl` file with one JSON line per quick task completion. Include: task ID, slug, description, timestamp, commit hashes, files modified. This matches the existing JSONL pattern in `rapid-task-completed.sh`. Surface this in `/rapid:status` as a "Recent quick tasks" section. -- Priority: high

4. **Define UI spec/contract schema (todo item 3)**: Extend CONTRACT.json with a `ui` section or create parallel `UI-CONTRACT.json` files per set. UI contracts should define: component names, prop interfaces, layout agreements, shared design tokens, and interaction patterns. This follows the AG-UI protocol's principle of standardizing how agent-generated UIs communicate. During `/rapid:plan-set`, the planner should generate UI contracts when the set involves frontend work. During `/rapid:execute-set`, executors should validate against these contracts. -- Priority: high

5. **Implement richer planning context for code quality (todo item 6)**: Enhance the context provided to planning and execution agents with: (a) deeper codebase analysis (function-level signatures, dependency graphs), (b) prior set implementation patterns (what worked, what was revised), (c) style guide enforcement via hooks. This aligns with the industry trend toward "institutional knowledge encoding" in skills and memory files. -- Priority: high

6. **Build documentation generation skill (todo item 2)**: Create `/rapid:documentation` skill that spawns a documentation agent to analyze the codebase and generate/update docs. Follow the pattern of existing AI documentation tools (Mintlify, aidocs-cli): scan source files, extract public APIs, generate markdown docs. Should produce both user-facing docs (`docs/`) and API reference docs. Best used at end of milestone as the todo suggests. -- Priority: medium

7. **Implement add-set discovery flow (todo item 1)**: The `/rapid:add-set` skill already exists with a 2-question discovery flow. Enhance it with: (a) memory-aware suggestions (suggest sets based on deferred ideas from previous discussions), (b) dependency auto-detection (analyze proposed scope against existing set contracts), (c) position in DAG visualization. -- Priority: medium

8. **Add NO_COLOR support to display.cjs**: Respect the `NO_COLOR` environment variable convention. When set, suppress ANSI escape codes in banner rendering. This is a small change with accessibility impact. -- Priority: low

9. **Consider unified activity timeline**: Beyond quick task logging, a `.planning/ACTIVITY.jsonl` at the project root that logs all significant operations (set transitions, agent completions, merge events, quick tasks) would provide the "what did we do" view users expect. This builds on the existing `rapid-task-completed.sh` hook pattern. -- Priority: low (post-v3.4.0)
