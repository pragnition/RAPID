# Oversights Research -- v4.3.0 Reliability & State

## Cross-Cutting Concerns

### Error Handling Strategy

**Why this matters for v4.3.0:** All three bug fixes (DAG.json not found, STATE.json missing/invalid, plan-set --gaps on merged sets) involve inadequate error recovery. The current error handling strategy is binary: either the operation succeeds or it throws and exits. There is no middle ground for "partially recoverable" states.

**Current state:**
- `errors.cjs` provides `CliError` (throw-based) and `exitWithError` (write JSON + stderr + exit)
- `state-machine.cjs` `withStateTransaction` throws a generic `Error('Cannot mutate: STATE.json is missing or invalid')` without distinguishing between "file missing" and "file present but corrupt"
- `tryLoadDAG` returns `{ dag: null }` on ENOENT but throws on malformed JSON -- callers have no way to distinguish "never created" from "was created but got corrupted"

**What is missing:**
- Error categorization: The STATE.json bug (bug #2) would benefit from distinguishing transient failures (lock contention, temporary I/O) from permanent failures (file deleted, schema mismatch). Currently all failures produce the same generic error message.
- Recovery suggestions: When `getMergeOrder` throws "DAG.json not found," the error message says "Run /rapid:plan first" -- but if the DAG was created and then lost (e.g., archiving moved it), the suggestion is wrong. Error messages should be context-aware.
- Error chaining: When `withStateTransaction` fails, the original cause (JSON parse failure vs ENOENT vs Zod validation error) is lost. The caller only sees "missing or invalid."

### Logging and Observability

**Why this matters for v4.3.0:** The "intermittent" nature of bug #2 (STATE.json missing/invalid during state transitions) suggests a race condition or transient failure that is invisible without structured logging.

**Current state:**
- `core.cjs` has `output()` (stdout) and `error()` (stderr) -- unstructured text
- Lock compromise events write to stderr via `onCompromised` callback in `lock.cjs`
- No logging of lock acquisition/release timing, no logging of state read/write operations
- No structured logging library; no log levels

**What is missing:**
- Transaction audit trail: When `withStateTransaction` is called, there is no log of which agent/process acquired the lock, how long the lock was held, or what mutations were performed. For debugging intermittent STATE.json failures, this is critical.
- Lock contention visibility: `proper-lockfile` retries silently up to 10 times. If a lock is consistently contested, the only signal is eventual failure. There is no way to observe lock contention trends.
- DAG lifecycle logging: When DAG.json is created, read, or found missing, there is no structured record. This makes bug #1 ("DAG.json not found frequently") nearly impossible to diagnose post-hoc.

### Configuration Management

**Current state:**
- `config.json` at project root for runtime settings (lock_timeout_ms, agent_size_warn_kb)
- `.env` file loaded in skill preambles for RAPID_TOOLS path
- Lock stale threshold hardcoded at 300,000ms in `lock.cjs`
- Retry config hardcoded in `lock.cjs`

**What is missing for v4.3.0:**
- The lock retry parameters (retries: 10, factor: 2, minTimeout: 100, maxTimeout: 2000) are hardcoded. If bug #2 is caused by lock contention under heavy parallel agent load, users have no way to tune these without modifying source code. These should be configurable via `config.json`.
- No configuration validation on load -- if `config.json` is malformed, the error is cryptic.

### Testing Strategy

**Why this matters for v4.3.0:** All three bugs require reproducing edge cases that are hard to test in isolation.

**Current state:**
- 21+ test files using Node.js built-in `node:test`
- Co-located test files (`*.test.cjs` alongside source)
- Tests use `fs.mkdtempSync` for temporary directories
- No integration test harness for end-to-end skill flows
- No concurrent test scenarios (multiple agents accessing STATE.json simultaneously)

**What is missing:**
- Concurrency tests for state-machine.cjs: Bug #2 is likely a race condition. There are no tests that simulate two agents calling `withStateTransaction` simultaneously. This is the single most important testing gap for this milestone.
- Lifecycle integration tests: Bug #3 (plan-set --gaps on merged set) spans the full lifecycle (execute -> gaps -> plan -> merged). Current tests are unit-level and do not exercise multi-step flows.
- DAG creation verification tests: Bug #1 suggests DAG.json is sometimes not created during init/plan. There are no tests that verify the full init -> plan -> merge flow ensures DAG.json exists at the canonical path.
- Test fixtures for corrupt/partial states: Tests for `detectCorruption` and `recoverFromGit` exist, but there are no fixtures for the specific corruption patterns that trigger bug #2 (e.g., truncated write, stale .tmp file left behind).

### Documentation Needs

**Current state:**
- 9 docs in `docs/` directory
- `CLAUDE.md` points to `.planning/context/` files
- No troubleshooting guide
- No documentation of state recovery procedures

**What is missing for v4.3.0:**
- Troubleshooting guide for the three bugs: Users encountering "DAG.json not found" or "STATE.json is missing or invalid" have no self-service recovery path. A troubleshooting doc listing symptoms, causes, and recovery steps would reduce support burden.
- State recovery documentation: `recoverFromGit` exists in code but is not exposed through any user-facing command. Users do not know they can recover STATE.json from git history.

### Data Concerns

**Why this matters for v4.3.0:** STATE.json is the single source of truth for all project state. Its integrity is critical.

**Current state:**
- Atomic writes via tmp file + rename (`writeState`, `withStateTransaction`)
- Zod validation on every read and write
- `detectCorruption` function exists
- `recoverFromGit` function exists
- Lock-based concurrency protection via `proper-lockfile`

**What is missing:**
- No backup before mutation: `withStateTransaction` reads, mutates, and writes. If the process crashes between writing the tmp file and renaming it, a `.tmp` file is left behind. The next read ignores it -- but the mutation is lost. There is no backup of the pre-mutation state.
- No `.tmp` file cleanup: If `STATE.json.tmp` exists alongside `STATE.json`, it is silently ignored. This could mask a previous crash-recovery scenario. A startup check for orphaned `.tmp` files would catch this.
- DAG.json has no backup or recovery mechanism: Unlike STATE.json (which has `recoverFromGit`), DAG.json has no recovery path. It can be regenerated via `dag generate`, but the user is not told this when the "DAG.json not found" error appears.
- The archive step in `/new-version` moves set directories (including DAG.json) to the archive. If the archive runs mid-milestone (user error), DAG.json disappears and merge breaks. This may be the root cause of bug #1.

## Recommendations

Ordered by "include in this milestone" urgency:

1. **Error categorization for STATE.json failures:** Replace the generic `Error('Cannot mutate: STATE.json is missing or invalid')` in `withStateTransaction` with specific error types: `StateMissingError` (file ENOENT), `StateCorruptError` (JSON parse failure), `StateSchemaError` (Zod validation failure). Each should carry the original cause. This directly unblocks bug #2's fix and costs roughly 30 minutes of work.

2. **Orphaned .tmp file detection:** Add a check in `readState` or `withStateTransaction` that warns (to stderr) if `STATE.json.tmp` exists. If STATE.json is missing but `.tmp` exists, attempt to recover by renaming `.tmp` to `STATE.json`. This addresses a potential root cause of bug #2 (crash during atomic write).

3. **DAG.json recovery in error messages:** When `getMergeOrder` throws "DAG.json not found," the error should also check whether DAG.json exists in the archive (`.planning/archive/*/sets/DAG.json`). If found, the error message should suggest `dag generate` to recreate it rather than "Run /rapid:plan first." This addresses the likely root cause of bug #1 (archive moved the DAG).

4. **Concurrency tests for withStateTransaction:** Add tests that spawn 5-10 parallel calls to `withStateTransaction` and verify that all mutations are applied without data loss or corruption. Use `Promise.all` with intentional delays to simulate real agent contention. This is the verification gate for bug #2.

5. **Plan-set status guard for merged sets:** Bug #3 requires checking whether a set's status is `merged` before declaring planning complete. The current code likely only checks for `complete` status. The fix needs a test that exercises the full lifecycle: execute -> gaps found -> set transitions to complete -> merge -> attempt plan-set --gaps.

6. **Lock configuration via config.json:** Move the retry parameters from hardcoded constants in `lock.cjs` to optional fields in `config.json` with the current values as defaults. This allows users experiencing bug #2 to increase retry counts or timeouts without source changes.

7. **Troubleshooting section in docs:** Write a `docs/troubleshooting.md` covering: (a) "DAG.json not found" -- when it happens, how to regenerate with `dag generate`; (b) "STATE.json is missing or invalid" -- when it happens, how to use `state recover` (if exposed) or manual git checkout; (c) "plan-set --gaps says planning is complete" -- explain the merged status issue and workaround.

8. **Transaction logging (lightweight):** Add optional stderr logging to `withStateTransaction` and `acquireLock` that records lock name, PID, acquire time, and hold duration. Gate it behind an environment variable (`RAPID_DEBUG=1`) to avoid noise in normal operation. This is the diagnostic tool needed to understand bug #2 in the field.
