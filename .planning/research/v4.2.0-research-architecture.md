# Architecture Research -- v4.2.0 (Discuss & Audit)

## Current Architecture

RAPID is a Claude Code plugin organized as a 5-layer system: Skill Layer (24 skills) -> Agent Layer (26 agents) -> CLI Layer (rapid-tools.cjs) -> Library Layer (21 modules in src/lib/) -> Storage Layer (.planning/, .rapid-worktrees/, git branches). All code is CommonJS (.cjs), validated with Zod schemas, and state is persisted to `.planning/STATE.json` with file-based locking.

### Strengths Relevant to This Milestone

- **Skill-as-specification pattern**: Each skill (SKILL.md) is a self-contained orchestration document that agents follow step-by-step. This is the natural home for the new `audit-version` command -- it follows the exact same pattern as `new-version`, `discuss-set`, etc.
- **State machine is well-defined**: SetStatus enum (`pending -> discussed -> planned -> executed -> complete -> merged`) with explicit transition maps in `state-transitions.cjs`. The audit-version feature can read state without needing new transitions.
- **CLI command dispatch is extensible**: `rapid-tools.cjs` routes commands to handler files in `src/commands/`. Adding a new command handler is straightforward.
- **Display module has banner infrastructure**: `display.cjs` has `STAGE_VERBS` and `STAGE_BG` maps that accept new stage names for branded banners.
- **AskUserQuestion is the established UX channel**: The discuss-set skill already uses AskUserQuestion extensively with multiSelect, preview panels, and option descriptions. Gray area consolidation builds on this existing pattern.

### Areas Needing Extension

- **No audit/verification concept in state schema**: The current `ProjectState` tracks set lifecycle but has no concept of "audited" status at the milestone level. The audit-version feature needs to produce artifacts without requiring schema changes (artifact-based, not state-based).
- **Module path resolution pattern is fragile**: Skills use `require('${RAPID_TOOLS}/../lib/context.cjs')` in `node -e` blocks. Node.js `require()` does not resolve `..` relative to a file path the way a filesystem does -- it treats `rapid-tools.cjs/../lib/context.cjs` as a literal path and fails because `rapid-tools.cjs` is a file, not a directory.
- **Gray area presentation is rigidly batched**: The current discuss-set skill hardcodes a "batch of 4" pattern using separate AskUserQuestion calls. Consolidating to fewer prompts with more checkboxes requires reworking the Step 5 specification, not the underlying code.

## Recommended Architecture Pattern

- **Pattern:** Extend existing modular skill architecture (no new architectural pattern needed)
- **Rationale:** All four items in this milestone (audit-version, discuss table formatting, gray area consolidation, context.cjs path fix) are incremental additions and refinements to existing modules. The current architecture cleanly supports each change. Introducing a new pattern would add unnecessary complexity for what are essentially a new skill, a SKILL.md rewrite, and a bug fix.
- **Evidence:** The existing skill/agent/CLI/library layering has successfully handled 24 skills and 21 library modules. The audit-version feature follows the same structural pattern as `new-version` (skill + agent + CLI helpers). The discuss improvements are SKILL.md-level changes. The path bug is a single-module fix.

## Module Organization

### Module: audit.cjs (NEW)

- **Responsibility:** Core audit logic -- reads milestone state, reads set artifacts (DEFINITION.md, CONTRACT.json, CONTEXT.md), compares against ROADMAP.md/REQUIREMENTS.md goals, computes coverage scores, and identifies gaps.
- **Public interface:**
  - `auditMilestone(cwd, milestoneId)` -- returns structured audit result with scores and gaps
  - `classifyGaps(auditResult)` -- separates critical gaps from deferrable items
  - `formatAuditReport(auditResult)` -- produces markdown report content
- **Dependencies:** `state-machine.cjs` (read state), `resolve.cjs` (resolve set references), `core.cjs` (findProjectRoot, output helpers), `fs`/`path` (read artifacts)
- **Data ownership:** Produces `.planning/v{version}-AUDIT.md` report artifact. Does not own or modify STATE.json.

### Module: audit.cjs command handler (NEW, in src/commands/)

- **Responsibility:** CLI interface for audit operations -- parses arguments, calls audit library, outputs JSON/text results.
- **Public interface:** `handleAudit(args)` dispatched from rapid-tools.cjs
- **Dependencies:** `../lib/audit.cjs`, `../lib/core.cjs`
- **Data ownership:** None -- delegates to library

### Module: discuss-set/SKILL.md (MODIFY)

- **Responsibility:** Captures developer implementation vision for a set via structured discussion.
- **Changes:**
  - Step 5 (Gray Areas): Consolidate batched AskUserQuestion calls. Instead of separate calls per batch of 4, use fewer calls with more checkboxes (e.g., 8 gray areas = 2 groups of 4 in a single multiSelect, or 1 call with 8 checkboxes).
  - Step 6 (Format A tables): Improve pros/cons table alignment guidance in the specification.
- **Dependencies:** No code dependencies -- this is a SKILL.md specification change
- **Data ownership:** `.planning/sets/{set-id}/CONTEXT.md`, `.planning/sets/{set-id}/DEFERRED.md`

### Module: context.cjs path resolution (BUG FIX)

- **Responsibility:** Fix the broken `require('${RAPID_TOOLS}/../lib/context.cjs')` pattern used in SKILL.md files.
- **Root cause:** Node.js `require()` treats the path literally. `require('/path/to/rapid-tools.cjs/../lib/context.cjs')` fails because `rapid-tools.cjs` is a file, and `require` does not strip the filename before resolving `..`. The filesystem `path.resolve()` does handle this, but `require()` does not.
- **Fix pattern:** Replace `require('${RAPID_TOOLS}/../lib/context.cjs')` with `require(require('path').resolve(require('path').dirname('${RAPID_TOOLS}'), '../lib/context.cjs'))` -- or better, add a CLI subcommand (`rapid-tools context require-path`) that outputs the resolved lib path.
- **Affected files:** `skills/init/SKILL.md` (line 564), `skills/register-web/SKILL.md` (lines 22, 44). Also `skills/unit-test/SKILL.md` and `skills/uat/SKILL.md` use the same `.env` pattern but may not hit this specific require path.

## Data Flow

### Audit-Version Request Lifecycle

```
1. User invokes /rapid:audit-version
2. SKILL.md orchestrates:
   a. Read STATE.json -> identify current milestone, all sets, their statuses
   b. Read ROADMAP.md -> extract original goals/requirements per set
   c. Read REQUIREMENTS.md -> extract feature requirement list
   d. For each set in milestone:
      - Read DEFINITION.md, CONTRACT.json, CONTEXT.md
      - Read PLAN.md files (per wave)
      - Check set status (complete/merged vs incomplete)
   e. Spawn rapid-auditor agent with gathered context
3. Agent performs cross-reference analysis:
   - Requirements in ROADMAP.md vs implemented sets
   - CONTRACT.json exports vs actual source file presence
   - DEFINITION.md scope vs what was executed
4. Agent produces v{version}-AUDIT.md with:
   - Coverage scores (requirements satisfied/total)
   - Gap list (missing features, partial implementations)
   - Classification: critical gaps vs tech debt
5. SKILL.md presents results via AskUserQuestion:
   - If gaps found: offer to plan remediation sets
   - If clean: offer to proceed to new-version
   - If deferrable items: capture in DEFERRED.md with carry-forward context
```

### Discuss Gray Area Consolidation Flow

```
Current (n=2, 8 gray areas):
  AskUserQuestion(batch 1: items 1-4) -> response
  AskUserQuestion(batch 2: items 5-8) -> response
  Combine selections

Proposed (n=2, 8 gray areas):
  AskUserQuestion(all 8 items in single multiSelect) -> response
  OR
  AskUserQuestion(group 1: items 1-4, group 2: items 5-8, presented as one prompt) -> response
```

The key constraint: AskUserQuestion supports `multiSelect: true` with arbitrary numbers of options. The current "batch of 4" limit is a SKILL.md design choice, not a technical limitation. The consolidation is a specification change, not a code change.

### State Management

- **Audit-version reads state but does not mutate it.** The audit report is a .planning/ artifact, not a state transition. This is intentional -- auditing is observational, not lifecycle-altering.
- **Discuss improvements do not change state flow.** The state transition `pending -> discussed` remains unchanged. Only the UX of how gray areas are presented changes.
- **Path fix has no state implications.** It is a require() resolution fix in SKILL.md shell blocks.

## API Design

### New CLI Command: `rapid-tools audit`

Following the existing command pattern (see `src/commands/misc.cjs`, `src/commands/state.cjs`):

```
node "${RAPID_TOOLS}" audit run [milestone-id]
  -- Runs full audit, outputs JSON: { status, scores, gaps[], techDebt[] }

node "${RAPID_TOOLS}" audit report [milestone-id]
  -- Generates and writes v{version}-AUDIT.md, outputs { reportPath }

node "${RAPID_TOOLS}" audit lib-path
  -- Outputs resolved path to src/lib/ directory (fixes the require path bug)
  -- Usage: RAPID_LIB=$(node "${RAPID_TOOLS}" audit lib-path)
  --        Then: require("${RAPID_LIB}/context.cjs")
```

Alternative for the path fix (simpler, recommended): Add a general-purpose `resolve lib-path` subcommand to the existing `resolve` command rather than nesting it under `audit`:

```
node "${RAPID_TOOLS}" resolve lib-path
  -- Outputs: /absolute/path/to/src/lib
```

This is more discoverable and reusable across all skills that need to require library modules.

### Naming Conventions

- Skill directory: `skills/audit-version/`
- CLI command: `audit`
- Agent: `rapid-auditor`
- Role module: `role-auditor.md`
- Library: `src/lib/audit.cjs`
- Report artifact: `.planning/v{version}-AUDIT.md`

All follow existing patterns (compare: `new-version` skill, `review` command, `rapid-reviewer` agent).

## Integration Points

| External System | Protocol | Purpose | Error Handling |
|----------------|----------|---------|---------------|
| .planning/STATE.json | File read (JSON) | Read milestone/set statuses | Fail with "Run /rapid:init" message |
| .planning/ROADMAP.md | File read (Markdown) | Extract original goals/requirements | Warn if missing, degrade gracefully |
| .planning/REQUIREMENTS.md | File read (Markdown) | Feature requirement checklist | Optional -- audit works without it |
| .planning/sets/*/DEFINITION.md | File read (Markdown) | Per-set scope definitions | Skip sets without definitions |
| .planning/sets/*/CONTRACT.json | File read (JSON) | Interface contracts, exports | Skip if absent |
| .planning/sets/*/CONTEXT.md | File read (Markdown) | Discussion decisions | Skip if absent |
| AskUserQuestion API | Claude Code plugin API | Present audit results, gather decisions | N/A (host-provided) |
| Git (status/log) | Shell exec | Check for uncommitted changes | Warn but continue |

## Scaling Considerations

Not applicable for this milestone -- all four items are developer-experience improvements with no runtime performance implications. The audit feature scans `.planning/` artifacts (typically <100 files) and is invoked once per milestone lifecycle.

The only consideration is AskUserQuestion prompt length: consolidating 8-12 gray areas into fewer prompts increases the text per prompt. AskUserQuestion has no documented character limit in the Claude Code plugin API, but excessively long prompts may degrade readability. The recommendation is to cap at 8 checkboxes per prompt (current max of 12 areas = 2 prompts of 6, not 1 prompt of 12).

## Architecture Risks

1. **Audit report scope creep**: The GSD audit-milestone is a comprehensive 7-step workflow with Nyquist compliance, integration checking via subagents, and 3-source cross-referencing. RAPID's version should be simpler -- RAPID has fewer artifact types and no phase-level VERIFICATION.md files. Risk: over-engineering the audit to match GSD's complexity. **Mitigation:** Start with requirements-vs-state coverage checking only. Add integration auditing later if needed.

2. **Gray area consolidation may hit UX limits**: Presenting 8+ checkboxes in a single AskUserQuestion prompt could overwhelm users. The current batched approach was arguably intentional UX design. **Mitigation:** Cap at 8 per prompt. For n=3 (12 areas), use 2 prompts of 6 rather than 1 prompt of 12. Test with actual Claude Code AskUserQuestion rendering.

3. **Path fix pattern may proliferate**: If the fix is `require(path.resolve(path.dirname('${RAPID_TOOLS}'), '../lib/context.cjs'))`, every SKILL.md that needs library access repeats this verbose pattern. **Mitigation:** Add a `resolve lib-path` CLI subcommand so skills can do `RAPID_LIB=$(node "${RAPID_TOOLS}" resolve lib-path)` once, then `require("${RAPID_LIB}/context.cjs")` cleanly.

4. **Discuss table formatting is specification-only**: The pros/cons table alignment issue may be caused by LLM generation variability rather than a specification gap. Adding formatting guidance to SKILL.md helps but cannot guarantee perfect alignment. **Mitigation:** Use fixed-width column hints in the specification (e.g., "pad option names to 20 chars, pros to 40 chars") and consider switching to a non-table format (bullet lists) for complex tradeoffs.

## Recommendations

1. **Add `resolve lib-path` CLI subcommand**: Fixes the context.cjs path bug and prevents future occurrences. All SKILL.md files that use `require('${RAPID_TOOLS}/../lib/...')` should migrate to the new pattern. -- **Priority: critical** (blocks init for users hitting this bug)

2. **Create audit-version as a skill + thin library module**: Follow the GSD audit-milestone pattern but simplified for RAPID's artifact structure. The audit reads ROADMAP.md goals, cross-references against set statuses and artifacts, produces a coverage report, and offers gap remediation via AskUserQuestion. The library module (`audit.cjs`) should be kept thin -- most logic lives in the auditor agent's analysis, not in deterministic code. -- **Priority: high** (new feature, primary deliverable)

3. **Consolidate gray areas in discuss-set SKILL.md**: Change Step 5 to present all gray areas in a single AskUserQuestion when count is 8 or fewer, and at most 2 prompts when count is 12. This is a SKILL.md specification change only -- no code changes to discuss-set infrastructure. -- **Priority: medium** (UX improvement)

4. **Improve table formatting in discuss-set SKILL.md**: Add explicit formatting guidance in Step 6 Format A specification. Recommend fixed-width column padding, or switch to a structured list format for complex tradeoffs. Consider adding a "table formatting" example in the specification. -- **Priority: medium** (UX improvement)

5. **Register audit-version in display.cjs**: Add `'audit-version': 'AUDITING'` to STAGE_VERBS and appropriate background color to STAGE_BG. -- **Priority: low** (polish, needed when skill is created)

6. **Do not add audit state to ProjectState schema**: The audit is observational -- it reads state and produces a report artifact. Adding an "audited" status to milestones would complicate the state machine without clear benefit. If audit becomes a required gate in the future, revisit this decision. -- **Priority: low** (deliberate non-action)
