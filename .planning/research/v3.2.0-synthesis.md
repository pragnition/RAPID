# Research Summary -- v3.2.0 General Fixes

## Project Profile
- **Type:** brownfield Claude Code plugin (developer tooling)
- **Stack:** Node.js (CommonJS/.cjs) + Zod + proper-lockfile + node:test, no build step
- **Scale:** medium (8 milestone goals, 21 library modules, 24 skills, 26 agents)
- **Key Constraint:** Status name mismatch between skills (present-tense) and state machine (past-tense) is the root cause of most bugs in this milestone

## Unified Findings

### Stack and Infrastructure

- **Node.js 25.8.0 (Current, non-LTS):** Odd-numbered release, will not receive LTS support. No impact on v3.2.0 fixes; all APIs used (node:test, fs sync, child_process) are stable. Next LTS target is Node 26. (source: STACK)
- **CommonJS module system is correct for this project:** All 21 modules use `.cjs` extension. No ESM migration needed -- plugin consumed by Claude Code runtime, not external bundlers. (source: STACK)
- **All dependencies at latest compatible versions, zero vulnerabilities:** Zod 3.25.76 (latest v3), ajv 8.18.0, ajv-formats 3.0.1, proper-lockfile 4.1.2. npm audit clean. No dependency upgrades needed for v3.2.0. (source: STACK)
- **Zod v4 migration NOT recommended for v3.2.0:** Breaking changes in `.default()` behavior, error customization APIs, and CJS resolution would require auditing all 4 Zod-using files and 21+ test files. Stay on 3.25.76. (source: STACK, PITFALLS)
- **Zod 3.25.x bundles Zod 4 internally:** The `require('zod')` path is safe, but `zod/v4` subpath has known CJS loading issues (issue #4434) and different optional property semantics (issue #4883). Do not use `zod/v4` imports. (source: STACK, PITFALLS)
- **proper-lockfile 4.1.2 is unmaintained (~5 years, 15 open issues):** Functionally complete and works on Node 25, but represents long-term risk. The `onCompromised` callback in lock.cjs only logs instead of throwing, creating a silent race condition under concurrent access. (source: STACK, PITFALLS, OVERSIGHTS)
- **Version mismatch: plugin.json says 3.1.0, package.json says 3.0.0.** Should be reconciled to 3.2.0 upon release. (source: STACK)
- **No linter, formatter, or CI/CD pipeline detected.** Tests run locally via `node --test src/**/*.test.cjs`. (source: STACK, OVERSIGHTS)
- **`readState()` does not acquire a lock:** Creates TOCTOU race in read-then-mutate patterns. Not critical for single-agent use but risky with parallel agents. (source: PITFALLS)
- **Stale lock threshold is 5 minutes:** After a crash, users wait up to 5 minutes for lock cleanup. `cleanStaleLocks()` is not called automatically on CLI startup. (source: PITFALLS)

### Architecture Direction

- **Pattern: Targeted fixes within existing modular monolith.** The 5-layer architecture (Skill -> Agent -> CLI -> Library -> Storage) is sound. No architectural changes needed; all 8 goals are bugs or UX issues within existing modules. (source: ARCHITECTURE)
- **No new directories or modules needed.** All changes target existing files in src/lib/, src/bin/, skills/, and agents/. (source: ARCHITECTURE)
- **Transaction pattern is correct and should be leveraged.** All state mutations go through `withStateTransaction()` which provides lock -> read -> validate -> mutate -> validate -> write -> unlock atomics. The v3.1.0 status-rename migration proved the architecture supports cross-cutting changes. (source: ARCHITECTURE)
- **Resolve.cjs needs data source consolidation:** Switch from filesystem-based `plan.listSets()` to STATE.json-based resolution. Key design choice: pass `state` as parameter (simpler) vs. add `readStateSync()` (invasive) vs. make `resolveSet()` async (cleanest but cascading changes). (source: ARCHITECTURE, FEATURES, OVERSIGHTS)
- **Review-after-merge should NOT modify set status.** Implement as an out-of-band process with a `--post-merge` flag that skips state transitions and generates separate review artifacts. This avoids schema changes to the terminal `merged` status. (source: ARCHITECTURE, FEATURES, OVERSIGHTS)
- **Agent YAML `color:` field is controlled by Claude Code platform, not RAPID.** Available values: blue, red, green, yellow, purple, orange, pink, cyan. There is no "dark purple" -- only `purple`. Banner colors in display.cjs (ANSI codes) and agent YAML colors are two separate systems. (source: UX, OVERSIGHTS)

### User Experience Direction

- **State-driven resolution is the industry standard:** Tools like Kubernetes, Terraform, Docker Compose, and Git all resolve references against authoritative state, not derived filesystem state. RAPID should follow this pattern for set resolution. (source: UX)
- **AskUserQuestion native multi-header format:** Claude Code supports 1-4 questions per call, each with header (max 12 chars), question text, multiSelect boolean, and 2-4 options with label and description. The discuss-set skill currently bypasses this entirely with freeform text blocks. (source: UX, FEATURES)
- **"Autonomy dial" pattern for AskUserQuestion:** Give users structured controls, not open-ended prompts. Pre-filled options with an "I'll answer" escape hatch is the recommended pattern. Every "Claude decides" must be accompanied by concrete alternative options. (source: UX)
- **High-contrast terminal banners:** White text on standard (non-bright) ANSI backgrounds provides better contrast. Standard magenta `\x1b[45m` is a good dark purple approximation with maximum terminal compatibility. (source: UX, PITFALLS)
- **Both pre-merge and post-merge review are valid patterns:** Industry tools (GitHub PRs, GitLab MRs, trunk-based dev) support multiple review timing options. Default should remain pre-merge; post-merge should be opt-in. (source: UX, FEATURES)
- **State consistency builds user trust:** If discuss-set completed, the state should show `discussed`. Silent failures that leave state at `pending` erode trust. Errors during state update should be loud, not suppressed. (source: UX, OVERSIGHTS)
- **"Did you mean?" suggestions for invalid transitions:** When an unknown status name is passed (e.g., `discussing`), suggest the correct one (`discussed`) with a note about the v3.1.0 rename. (source: UX)
- **NO_COLOR standard not respected:** display.cjs always outputs ANSI colors (line 8 comment says "Always outputs colors"). Minor accessibility concern. (source: OVERSIGHTS, UX)

## Contradictions

### Contradiction: Whether to Add "reviewing" Status to SetStatus Enum

- **FEATURES says:** Add `reviewing` to the SetStatus enum between `complete` and `merged`. Transitions: `complete -> ['reviewing', 'merged']` and `reviewing -> ['merged']`. This gives the review pipeline its own trackable state.
- **ARCHITECTURE says:** Review-after-merge should NOT modify set status. Implement as out-of-band process. For "reviewing," either add it to the enum or have review operate without state transition (just check that status is `complete`). The architecture leans toward option (b) -- no state change for review.
- **OVERSIGHTS says:** Three options presented -- add `reviewed` status, use a boolean flag on set object, or track review separately for post-merge. Notes that adding to Zod enum is a breaking change for existing STATE.json.
- **PITFALLS says:** The review skill's `reviewing` status was "intentionally removed in v3.1.0" (per state-schemas.test.cjs). Removing the transition from the review skill is the correct fix.
- **Resolution:** This needs a user decision. The two viable paths are:
  - **Path A (add reviewing):** Add `reviewing` to the enum. Provides explicit state tracking for review-in-progress. More work but more visibility.
  - **Path B (no reviewing):** Remove the transition from the review skill. Review operates on `complete` sets without state change. Simpler but loses review-in-progress tracking. Post-merge review would need a separate mechanism.
  - The `status-rename.test.cjs` explicitly checks that `reviewing` is NOT in the enum, suggesting Path B aligns with the v3.1.0 intentional removal.

### Contradiction: 16-Color vs 256-Color ANSI for Dark Purple

- **STACK says:** Use 256-color `\x1b[48;5;54m` for dark purple. Notes this breaks the "maximum terminal compatibility" design goal in display.cjs header comment.
- **FEATURES says:** Use 256-color `\x1b[48;5;53m` for dark purple. Notes this is widely supported by modern terminals.
- **ARCHITECTURE says:** Use standard 16-color `\x1b[45m` (magenta) for maximum compatibility, or 256-color with feature detection.
- **PITFALLS says:** Use standard 16-color `\x1b[45m` for maximum compatibility. 256-color may not work on all terminals.
- **UX says:** Use standard 16-color `\x1b[45m` for high contrast with white text; standard variants (4Xm) are darker and more readable than bright variants (10Xm).
- **Resolution:** Consensus leans toward `\x1b[45m` (standard magenta, 16-color) as the safer choice. It provides a dark purple approximation with maximum terminal compatibility. If a more precise purple is needed, `\x1b[48;5;53m` (256-color) works on all modern terminals but fails on very old ones. The conservative recommendation is `\x1b[45m`.

### Contradiction: Sync vs Async resolveSet() When Switching to STATE.json

- **ARCHITECTURE says:** Three options: (a) pass state as parameter, (b) add `readStateSync()`, (c) make `resolveSet()` async. Notes option (a) is simplest but requires callers to pre-load state. Option (c) is cleanest but most invasive.
- **FEATURES says:** Option 2 "Hybrid Resolution" -- resolveSet() reads state internally via `readState()` or accepts pre-loaded state. This implies option (a) with internal fallback.
- **OVERSIGHTS says:** Resolution should use STATE.json. Notes that `resolveWave()` and `execute.cjs` also use `plan.listSets()`, so the fix must cascade.
- **Resolution:** The recommended approach is option (a) -- accept optional `state` parameter. The CLI `resolve set` handler reads state and passes it. Internal library callers that already have state pass it directly. Fallback to filesystem when no state is available (pre-init). This avoids async cascade and is backward compatible.

## Recommended Approach

The v3.2.0 milestone is fundamentally a consistency and correctness release. The single most impactful finding across all 6 research outputs is the **status name mismatch between skills and the state machine**. When the v3.1.0 status-rename migration changed set statuses from present-tense (`discussing`, `planning`, `executing`) to past-tense (`discussed`, `planned`, `executed`), the `.cjs` source files were updated and a test (`status-rename.test.cjs`) was added to enforce the new names -- but the 24 skill Markdown files and 26 agent Markdown files were never updated. This single oversight cascades into multiple bugs: state transitions fail silently (because discuss-set suppresses errors with `2>/dev/null || true`), sets remain stuck in `pending`, downstream skills encounter unexpected state, and users see confusing "discussing -> complete" error messages.

The recommended approach is to treat this milestone in three phases. **Phase 1 (Critical Fixes)** addresses the three data-correctness bugs: fix all status literals across skills and agents (Feature 8), remove ghost wave-plan commands from tool-docs.cjs and skills (Feature 5), and switch set resolution from filesystem to STATE.json (Feature 1). These three fixes are independent of each other (different files, different modules) and can be parallelized. Phase 1 should also include a design decision on the "reviewing" status -- whether to add it to the enum or remove it from the review skill. **Phase 2 (State Enforcement)** builds on Phase 1 by adding verification after every state transition (Feature 7), removing the `2>/dev/null || true` error suppression pattern, and adding a CLI command validation test to prevent ghost commands from recurring. **Phase 3 (UX and New Features)** addresses the user experience improvements: banner colors (Feature 3), discuss-set AskUserQuestion rewrite (Feature 2), AskUserQuestion option audit (Feature 4), and review-after-merge support (Feature 6).

The architecture is sound and requires no structural changes. All fixes target existing files within the established 5-layer architecture. The transaction pattern, schema validation, and test infrastructure provide the safety net for surgical fixes. The key design decisions that need to be made upfront are: (1) whether `reviewing` becomes a valid set status, (2) whether review-after-merge uses a `--post-merge` flag or a separate command, and (3) whether banner colors use 16-color `\x1b[45m` or 256-color `\x1b[48;5;53m`. These decisions do not block Phase 1 work.

### Suggested Set Boundaries

Based on feature dependencies, file ownership, and parallelization potential:

- **Set A: State Machine Consistency** -- Features 8 (state transitions fix) + 7 (state update enforcement). Scope: state-schemas.cjs, state-transitions.cjs, all 24 SKILL.md files (status literals only), state-transitions.test.cjs. Rationale: Feature 7 depends on Feature 8; both modify the same state machine files and skill transition calls. These two form a natural unit of work.

- **Set B: CLI Command Audit and Cleanup** -- Feature 5 (invalid command audit). Scope: tool-docs.cjs, tool-docs.test.cjs, plan-set/SKILL.md (wave-plan reference), review/SKILL.md (wave-plan reference), role-roadmapper.md, role-plan-verifier.md, agents/*.md (rebuild). Rationale: Self-contained audit-and-remove operation across tool-docs and skill files. Independent of state machine changes.

- **Set C: Set Resolution Fix** -- Feature 1 (set name resolution). Scope: resolve.cjs, resolve.test.cjs, plan.cjs (optional refactor). Rationale: Self-contained library-level fix. Independent of state machine and command audit. Requires design decision on state parameter passing.

- **Set D: UX Improvements** -- Features 2 (batched questions) + 4 (never Claude decides without options) + 3 (banner colours). Scope: discuss-set/SKILL.md (Steps 5-6 rewrite), display.cjs, display.test.cjs, all skills (AskUserQuestion audit), core-conventions.md. Rationale: All three are UX-focused with no state machine or CLI router changes. Features 2 and 4 overlap on the discuss-set skill. Feature 3 is trivial and fits here as a visual polish item.

- **Set E: Review After Merge** -- Feature 6 (review-after-merge support). Scope: review.cjs, review/SKILL.md, merge/SKILL.md, possibly state-schemas.cjs (if "reviewing" status is added). Rationale: Complex new feature that should be isolated. Depends on Set A's decision about "reviewing" status but can be started in parallel if the interface is agreed upon.
