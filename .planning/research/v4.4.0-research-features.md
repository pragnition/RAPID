# Features Research -- v4.4.0 (Polish & Documentation)

## Feature Inventory

### Feature: Review State Management

- **Description:** Store the state of the review pipeline (which stages have been completed: scope, unit-test, bug-hunt, uat) so that subsequent commands do not re-ask the user to run earlier stages and can skip or resume intelligently.
- **Complexity:** moderate
- **Dependencies:** independent

#### Current State Analysis

The review pipeline currently has no persistent state tracking. Each skill (`/rapid:review`, `/rapid:unit-test`, `/rapid:bug-hunt`, `/rapid:uat`) operates independently by reading `REVIEW-SCOPE.md` and `REVIEW-ISSUES.json` as file-based artifacts. There is no mechanism to record which review stages have been completed for a set. The set status enum (`pending -> discussed -> planned -> executed -> complete -> merged`) has no review-related states -- review is a "non-mutating observation step" per the SKILL.md.

Key files:
- `src/lib/state-schemas.cjs` -- Zod schemas for SetState (no review fields)
- `src/lib/state-transitions.cjs` -- SET_TRANSITIONS map (complete -> merged, no review substates)
- `src/lib/review.cjs` -- Review library (issues, scoping, summaries; no pipeline state)
- `src/commands/review.cjs` -- CLI handler (scope, log-issue, list-issues, lean, summary)
- `skills/review/SKILL.md`, `skills/bug-hunt/SKILL.md`, `skills/uat/SKILL.md`, `skills/unit-test/SKILL.md`

#### Decomposition

1. **REVIEW-STATE.json schema and persistence** -- Define a Zod-validated schema for review pipeline state, with read/write functions in `src/lib/review.cjs`
2. **Stage completion recording** -- Each review skill writes its completion status to REVIEW-STATE.json after finishing
3. **Stage prerequisite checking** -- Each skill checks REVIEW-STATE.json on startup to determine whether prerequisites are met
4. **CLI subcommand for review state** -- `rapid-tools review state <set-id>` to query current review pipeline state
5. **Skill prompt updates** -- Update all 4 review skill SKILL.md files to read/write review state

#### Implementation Strategies

**Option 1: File-based REVIEW-STATE.json (alongside REVIEW-ISSUES.json)**

- Approach: Create a `REVIEW-STATE.json` file at `.planning/sets/{setId}/REVIEW-STATE.json` (and `.planning/post-merge/{setId}/REVIEW-STATE.json` for post-merge). Schema:
  ```json
  {
    "setId": "auth-core",
    "pipeline": {
      "scope": { "completedAt": "2024-01-01T...", "postMerge": false },
      "unitTest": { "completedAt": "2024-01-01T...", "issuesFound": 3 },
      "bugHunt": { "completedAt": null },
      "uat": { "completedAt": null }
    },
    "lastUpdatedAt": "2024-01-01T..."
  }
  ```
- Pros: Follows existing pattern (REVIEW-ISSUES.json, REVIEW-SCOPE.md are file-based); no schema migration needed for STATE.json; naturally scoped to set; easy to inspect manually
- Cons: One more file to manage; not centralized in STATE.json
- Libraries/services: Zod for validation (already used)

**Option 2: Extend SetState in STATE.json with review substates**

- Approach: Add a `reviewPipeline` field to SetState schema:
  ```javascript
  const SetState = z.object({
    id: z.string(),
    status: SetStatus,
    waves: z.array(WaveState).default([]),
    reviewPipeline: z.object({
      scope: z.string().nullable().default(null),
      unitTest: z.string().nullable().default(null),
      bugHunt: z.string().nullable().default(null),
      uat: z.string().nullable().default(null),
    }).optional(),
  }).passthrough();
  ```
- Pros: Centralized state; atomic with other state changes via `withStateTransaction`
- Cons: Schema migration required; STATE.json mutations need locking; tighter coupling between review and core state

**Recommended:** Option 1 (file-based REVIEW-STATE.json). It follows the established pattern where review artifacts live alongside the set directory, avoids STATE.json schema migration, and keeps the review pipeline decoupled from core lifecycle state. The `.passthrough()` on SetState means Option 2 would technically work without migration, but it mixes concerns.

#### Data Model Impact

- New entities: `REVIEW-STATE.json` schema (ReviewPipelineState)
- Existing entities modified: None (STATE.json untouched)
- Relationships: REVIEW-STATE.json references setId; read by all review skills

#### API Surface

- New CLI: `rapid-tools review state <set-id> [--post-merge]` -- returns JSON of pipeline state
- New CLI: `rapid-tools review mark-complete <set-id> <stage> [--post-merge]` -- marks a stage as done
- Existing endpoints modified: None (skills will call these new commands)

### Feature: Human-Readable Documentation for Open-Sourcing

- **Description:** Prepare comprehensive documentation covering the canonical workflow (`/install` through `/merge` and beyond), plus auxiliary commands (`/quick`, `/bug-fix`, `/add-set`). Target audience: developers evaluating or adopting RAPID for the first time.
- **Complexity:** moderate
- **Dependencies:** independent

#### Current State Analysis

Documentation exists in multiple places:
- `README.md` -- Good overview with 60-second quickstart, architecture section, but stops short of detailed workflow walkthrough
- `DOCS.md` -- Technical reference (version 3.5.0, outdated), command listing, architecture overview
- `docs/` directory -- 9 template-scaffolded files (setup, planning, execution, agents, configuration, merge-and-cleanup, review, state-machines, troubleshooting) plus CHANGELOG.md. Most contain placeholder text from the scaffolding system (`src/lib/docs.cjs`)
- `skills/help/SKILL.md` -- Lists available commands with descriptions

The docs are fragmented and partially generated. The scaffolded docs in `docs/` have real content for agents.md (11KB), configuration.md (5.5KB), and troubleshooting.md (5.3KB), but others are thin. DOCS.md references version 3.5.0 and is out of date.

#### Decomposition

1. **Canonical workflow guide** -- A single narrative document walking through install -> init -> branding -> scaffold -> start-set -> discuss-set -> plan-set -> execute-set -> review -> merge -> audit-version -> new-version
2. **Auxiliary commands guide** -- Document /quick, /bug-fix, /add-set with examples and when to use each
3. **README.md refresh** -- Update to current version, ensure quickstart is accurate, add links to detailed docs
4. **DOCS.md consolidation** -- Either update or deprecate in favor of docs/ directory structure
5. **docs/ directory content fill** -- Replace placeholder content in thin docs with substantive material

#### Implementation Strategies

**Option 1: Single comprehensive guide (docs/workflow.md)**

- Approach: Create one long-form guide (`docs/workflow.md`) that walks through the entire RAPID lifecycle with annotated examples. Keep the existing docs/ structure for reference material (agents, configuration, state machines, troubleshooting). Update README.md to link to workflow.md as the primary getting-started resource.
- Pros: Single entry point for new users; narrative flow; easy to maintain one file
- Cons: Can become very long; harder to link to specific sections from SKILL.md files
- Libraries/services: None (pure markdown)

**Option 2: Structured docs/ with index (current pattern, filled in)**

- Approach: Fill in the existing scaffolded docs with real content. Add a `docs/workflow.md` for the canonical path. Update `docs/setup.md` with current install instructions. Create `docs/auxiliary-commands.md` for /quick, /bug-fix, /add-set. Update DOCS.md to serve as an index pointing into docs/.
- Pros: Modular; each topic standalone; matches existing scaffold structure; easier to cross-reference
- Cons: More files to keep in sync; user must navigate between files
- Libraries/services: None (pure markdown)

**Recommended:** Option 2 (structured docs with index). The scaffolding system already created the structure. Filling in content and adding workflow.md gives both narrative flow and modular reference. DOCS.md becomes the hub.

#### Data Model Impact

- New entities: `docs/workflow.md`, `docs/auxiliary-commands.md`
- Existing entities modified: `README.md`, `DOCS.md`, `docs/setup.md`, `docs/planning.md`, `docs/execution.md`, `docs/review.md`
- Relationships: DOCS.md links to all docs/ files

#### API Surface

- No new endpoints. Documentation only.

### Feature: Better Colouring (Dark Terminal Colors)

- **Description:** Change bright agent colors to dark colors that are readable with white text on dark terminal backgrounds. The current bright backgrounds (bright blue `\x1b[104m`, bright green `\x1b[102m`, bright red `\x1b[101m`) produce hard-to-read white text on many dark terminal themes.
- **Complexity:** trivial
- **Dependencies:** independent

#### Current State Analysis

Colors are defined in two places:

1. **Banner system** (`src/lib/display.cjs`):
   - `STAGE_BG` maps stage names to ANSI bright background codes
   - Uses bright variants: `\x1b[104m` (bright blue bg), `\x1b[102m` (bright green bg), `\x1b[101m` (bright red bg)
   - Text is always `\x1b[97m` (bright white) + `\x1b[1m` (bold)
   - Tests in `src/lib/display.test.cjs`

2. **Agent frontmatter colors** (`src/commands/build-agents.cjs`):
   - `ROLE_COLORS` maps agent roles to Claude Code color names: blue, green, red, yellow, purple, cyan
   - These are Claude Code's agent color labels (not raw ANSI) -- they control the sidebar/badge color in the Claude Code UI, not terminal output
   - 27 agents across 7 colors

The banner colors are the terminal-facing issue. The agent frontmatter colors are Claude Code UI colors and may or may not need changing depending on whether "bright agent colours" refers to terminal banners or the Claude Code agent badge.

#### Decomposition

1. **Replace bright ANSI backgrounds with dark variants** -- Change `\x1b[10Xm` codes to `\x1b[4Xm` equivalents in display.cjs
2. **Update tests** -- Adjust display.test.cjs expectations
3. **Optionally review agent frontmatter colors** -- If the user also wants Claude Code badge colors changed, update ROLE_COLORS in build-agents.cjs

#### Implementation Strategies

**Option 1: Switch to standard (dark) ANSI background codes**

- Approach: Replace bright background escapes with standard dark backgrounds:
  - `\x1b[104m` (bright blue) -> `\x1b[44m` (dark blue)
  - `\x1b[102m` (bright green) -> `\x1b[42m` (dark green)
  - `\x1b[101m` (bright red) -> `\x1b[41m` (dark red)
  Keep text as bright white bold (`\x1b[97m\x1b[1m`) for maximum contrast against dark backgrounds.
- Pros: Minimal change; dark bg + white text = high contrast on dark terminals; 16-color palette maintained for compatibility
- Cons: May look washed out on light terminal themes (edge case; dark terminals are target audience)
- Libraries/services: None (raw ANSI)

**Option 2: Use 256-color palette for finer control**

- Approach: Use `\x1b[48;5;Nm` for more precise color selection (e.g., deep navy, forest green, dark red).
- Pros: More aesthetically pleasing; finer control
- Cons: Not all terminals support 256-color; breaks the "basic 16-color palette for maximum terminal compatibility" design principle stated in display.cjs
- Libraries/services: None

**Recommended:** Option 1 (standard dark ANSI backgrounds). It maintains the 16-color compatibility principle already documented in display.cjs, and switching from `10Xm` to `4Xm` is a surgical change.

#### Data Model Impact

- New entities: None
- Existing entities modified: `src/lib/display.cjs` (STAGE_BG values), `src/lib/display.test.cjs` (test expectations)
- Relationships: None

#### API Surface

- No new endpoints. Display change only.

## Implementation Order

1. **Better Colouring** -- Trivial complexity, self-contained in 2-3 files, quick win that improves DX immediately. Ship first to validate the pattern.
2. **Review State Management** -- Moderate complexity but high user-facing value. The core feature of this milestone. Requires schema design, CLI commands, and skill prompt updates.
3. **Documentation for Open-Sourcing** -- Moderate complexity, content-heavy. Best done last because it can reference the final state of review state management and colouring. Writing docs after features are implemented ensures accuracy.
