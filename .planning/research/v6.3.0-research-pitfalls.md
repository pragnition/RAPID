# Pitfalls Research -- v6.3.0 Mission Control & Fixes

## Critical Pitfalls

### Pitfall: Cytoscape.js Memory Leak on Graph Replacement
- **Category:** performance / data-loss
- **Stack:** cytoscape ^3.33.1, React 19
- **Description:** The current `KnowledgeGraphPage.tsx` creates a new cytoscape instance but only destroys it when the component unmounts (via the useEffect cleanup). When `data` changes, the effect runs again -- it calls `cy.elements().remove()` and `cy.add(elements)` on the existing instance, which is correct. However, the overhaul to a full codebase graph will likely involve replacing the `useDagGraph` hook with a much larger dataset. If the graph is rebuilt from scratch (new cytoscape instance per data change) without properly destroying the old one, each instance leaks DOM event listeners, canvas memory, and internal data structures. Cytoscape's `cy.destroy()` must be called before creating a replacement instance.
- **Trigger:** Navigating between projects or refreshing codebase graph data without cleanup; creating new `cytoscape()` instances inside a useEffect without guarding against the previous instance.
- **Impact:** High -- browser tab memory grows unbounded with each graph re-render. On large codebases (500+ files), a single cytoscape instance with hundreds of nodes already consumes significant memory. Two or three leaked instances can cause the tab to become unresponsive.
- **Prevention:** (1) Always call `cyRef.current.destroy()` before creating a new instance. (2) Use `cy.batch()` for bulk element updates instead of recreating instances. (3) Ensure the useEffect cleanup function calls `cy.destroy(); cyRef.current = null;` -- the current code already does this correctly, so any refactor must preserve this pattern.
- **Detection:** Chrome DevTools Memory tab: take heap snapshots before and after navigating away and back; check for retained cytoscape Core objects. Performance monitor showing steadily increasing JS heap size.
- **References:** https://js.cytoscape.org/#cy.destroy

### Pitfall: Synchronous Tree-Sitter Parsing Blocks FastAPI Event Loop
- **Category:** performance
- **Stack:** FastAPI >=0.135, tree-sitter >=0.24, Python 3.12+
- **Description:** The `/api/projects/{id}/codebase` endpoint calls `get_codebase_tree()` which walks the filesystem and parses up to 500 files with tree-sitter synchronously. The route handler `get_codebase_view` is defined with `def` (not `async def`), which means FastAPI runs it in a threadpool worker -- this is correct. However, overhauling the codebase graph to be a "full codebase graph" will likely increase `max_files` or add new endpoints. Any new endpoint that calls tree-sitter parsing MUST be defined with `def` (not `async def`) to avoid blocking the event loop. If defined with `async def`, the synchronous tree-sitter C extension calls will block the entire uvicorn event loop, freezing all concurrent requests.
- **Trigger:** Defining a new endpoint with `async def` that calls `get_codebase_tree()` or any tree-sitter parsing function.
- **Impact:** Critical -- all concurrent HTTP requests stall until parsing completes. On a project with 500 files, parsing can take 2-10 seconds, during which the health check, SSE connections, and all other API calls are blocked.
- **Prevention:** (1) Always use `def` (not `async def`) for endpoints that call tree-sitter parsing. FastAPI automatically runs `def` functions in a threadpool. (2) Alternatively, explicitly use `asyncio.to_thread(get_codebase_tree, ...)` inside `async def` endpoints. (3) Add a comment guard near the endpoint: `# IMPORTANT: Must be sync def -- tree-sitter blocks event loop`.
- **Detection:** If the `/api/health` endpoint becomes unresponsive during codebase parsing, the event loop is being blocked.
- **References:** https://fastapi.tiangolo.com/async/ ("Very Technical Details > Path operation functions")

### Pitfall: Hardcoded Version String in Multiple Locations
- **Category:** correctness
- **Stack:** Frontend (React/Sidebar.tsx), Backend (app/__init__.py), pyproject.toml
- **Description:** The version "4.2.1" is hardcoded in at least three locations: `web/frontend/src/components/layout/Sidebar.tsx` (lines 123-128, both full and compact variants), `web/backend/app/__init__.py` (line 1), and `web/backend/pyproject.toml` (line 3). The bug report says "Version display stuck at v4.2.1" -- fixing this requires updating ALL locations. Missing any one location means the version is inconsistent between the health endpoint (which reads `__version__`), the frontend sidebar, and the package metadata.
- **Trigger:** Updating the version in one file but forgetting others. There is no single source of truth or automated version sync mechanism.
- **Impact:** High -- users see stale version numbers, making it impossible to verify which release is deployed. The health check returns a wrong version to the Node.js `web-client.cjs` which may confuse diagnostics.
- **Prevention:** (1) Establish a single source of truth (e.g., `pyproject.toml`) and read from it at runtime. (2) Or add a build step that injects the version. (3) For the frontend, fetch the version from `/api/health` instead of hardcoding it. (4) Add a test that asserts all version strings match.
- **Detection:** `grep -r "4.2.1" web/` should return zero results after the fix. A version consistency test can compare all locations.
- **References:** Milestone bug report: "Mission Control RAPID version outdated"

## High-Risk Pitfalls

### Pitfall: Shell Config Detection Only Finds First Match (Break-on-First)
- **Category:** correctness
- **Stack:** Bash (setup.sh, install SKILL.md)
- **Description:** The install skill's shell config detection loop (SKILL.md lines 81-86) iterates over config files and `break`s on the first match: `if grep -qF "RAPID_TOOLS" "$f" ...; then echo "ALREADY_CONFIGURED=$f"; break; fi`. If RAPID_TOOLS is configured in BOTH `~/.bashrc` AND `~/.config/fish/config.fish`, only the first alphabetically iterated file is detected. The user is prompted to update only that one file. This is the exact bug described: "When RAPID_TOOLS env var is set in 2 different shells, the bash command only shows one, so the agent only updates one."
- **Trigger:** User has multiple shell configs (e.g., uses both bash and fish) and both contain RAPID_TOOLS exports.
- **Impact:** High -- after updating, one shell has the correct path and the other has a stale path pointing to an old version. RAPID works in one terminal but fails silently in another with confusing "command not found" or version mismatch errors.
- **Prevention:** (1) Remove the `break` and collect ALL files containing RAPID_TOOLS into a list. (2) Present all found files to the user and update all of them (or let the user choose which to update). (3) The detection loop should output each match, not just the first.
- **Detection:** Run the detection loop manually on a system with multiple shells configured. Check if all config files are reported.
- **References:** Milestone bug report: "Shell config update incomplete"

### Pitfall: Ctrl+Click Conflicts with dnd-kit PointerSensor on Kanban Cards
- **Category:** correctness / compatibility
- **Stack:** @dnd-kit/core ^6.3.1, @dnd-kit/sortable ^10.0.0, React 19
- **Description:** The Kanban card's `onClick` handler (KanbanCard.tsx line 36) opens the detail modal, while the same element has dnd-kit's `{...listeners}` spread on it (line 35) for drag-and-drop. The `PointerSensor` has an `activationConstraint: { distance: 5 }` to distinguish clicks from drags. Adding a Ctrl+click save shortcut is risky because: (1) Ctrl+click on macOS triggers a context menu; (2) Ctrl+click on Linux may or may not propagate depending on browser behavior; (3) The dnd-kit listener intercepts pointer events before they reach the onClick handler. The `CardDetailModal` (line 14-21) only listens for `Escape` key -- there is no keyboard shortcut for saving.
- **Trigger:** Implementing Ctrl+click save on the CardDetailModal without accounting for the event propagation chain from dnd-kit sensors.
- **Impact:** Medium-High -- if implemented on the card itself (not the modal), the save action may conflict with drag initiation or context menu. If implemented on the modal, Ctrl+click must be distinguished from Ctrl+Enter or a dedicated keyboard shortcut.
- **Prevention:** (1) Implement Ctrl+S (or Ctrl+Enter) as the save shortcut in CardDetailModal, NOT Ctrl+click. Ctrl+S is the universal "save" shortcut and avoids all pointer event conflicts. (2) Add the keydown listener alongside the existing Escape listener in the modal's handleKeyDown callback. (3) Call `e.preventDefault()` to prevent the browser's native save-page dialog on Ctrl+S.
- **Detection:** Manual testing: open card detail modal, press Ctrl+S -- verify it saves and does not trigger browser's "Save As" dialog. Test on both Linux and macOS.
- **References:** Milestone bug report: "Kanban board save shortcut -- Ctrl+click should save when adding card details"

### Pitfall: Codebase Graph Performance at Scale with Cytoscape.js
- **Category:** performance
- **Stack:** cytoscape ^3.33.1, cytoscape-dagre ^2.5.0
- **Description:** The current knowledge graph renders the DAG (typically 5-20 nodes). The overhaul to a "full codebase graph" means rendering potentially hundreds of nodes (files) and thousands of edges (imports/dependencies). Cytoscape.js with the dagre layout algorithm has O(V+E) complexity for layout but O(V^2) for rendering with the default Canvas renderer when many edges overlap. The dagre layout also becomes very slow (>5 seconds) above ~500 nodes.
- **Trigger:** Loading a codebase graph with more than ~200 nodes using the dagre layout algorithm.
- **Impact:** High -- the graph page becomes unresponsive. The 2-second `refetchInterval` in `useCodebaseTree` / `useDagGraph` hooks will re-trigger layout recalculations repeatedly, causing the page to freeze in a loop.
- **Prevention:** (1) Disable `refetchInterval` for the codebase graph endpoint or increase it to 30+ seconds. (2) Use `cy.batch()` for all element updates (the current code already does this -- preserve it). (3) Consider using Cytoscape's WebGL renderer (`cytoscape-webgl`) for large graphs. (4) Implement lazy/progressive loading -- show top-level modules first, expand on click. (5) Add a `max_nodes` parameter and paginate or cluster nodes above the threshold. (6) Set `animate: false` on layout for large graphs (>100 nodes).
- **Detection:** Load a project with 200+ source files and observe graph rendering time. Check if the browser's main thread is blocked for >1 second.
- **References:** https://js.cytoscape.org/#performance

### Pitfall: Bug-Fix Multi-Agent Wave Sequencing May Lose Context
- **Category:** correctness
- **Stack:** Claude Code Agent SDK, RAPID agent pipeline
- **Description:** The milestone feature "Multi-agent Bug-Fix Support" plans to split bugfixing into waves with one agent per wave sequentially. The current `bug-fix` SKILL.md (UAT mode, step UAT-d) already dispatches one executor per failure sequentially. The risk with wave-based splitting is that Agent B (wave 2) may need to fix code that Agent A (wave 1) modified. If each agent gets a fresh context without seeing the previous agent's commits, Agent B may: (a) attempt to fix already-resolved issues, (b) conflict with Agent A's changes, or (c) miss that a dependency was already updated.
- **Trigger:** Wave 2 bug-fix agent modifies a file that wave 1 already changed, without awareness of wave 1's commits.
- **Impact:** High -- merge conflicts between wave outputs, or regression bugs where wave 2 reverts wave 1's fix.
- **Prevention:** (1) Ensure each wave's agent starts from a git state that includes all previous waves' commits (sequential on the same branch). (2) Pass the list of modified files from wave N to wave N+1 in the agent prompt. (3) Include the commit log from previous waves in the context. (4) Use the existing `RAPID:RETURN` `modifiedFiles` field to build awareness between waves.
- **Detection:** After all waves complete, run `git log --oneline` to verify commits are sequential on the same branch. Check for any merge conflicts.

## Medium-Risk Pitfalls

### Pitfall: React 19 + dnd-kit@6 Event System Compatibility
- **Category:** compatibility
- **Stack:** React ^19.2.4, @dnd-kit/core ^6.3.1, @dnd-kit/sortable ^10.0.0
- **Description:** The project uses React 19 with dnd-kit. React 19 changed the event delegation model and removed the legacy event system. While @dnd-kit/core 6.x and @dnd-kit/sortable 10.x are the latest versions and should be compatible, there is a known interaction issue: React 19's `Suspense` boundaries can cause dnd-kit sensors to lose their event registration when components suspend and resume. The KnowledgeGraphPage is already lazy-loaded with Suspense, and if the new codebase graph also uses Suspense for progressive loading, drag-and-drop on the graph could break.
- **Trigger:** Wrapping cytoscape graph components in React Suspense boundaries while also using dnd-kit interactions on the same page.
- **Impact:** Medium -- drag-and-drop stops working after a Suspense fallback renders and resolves.
- **Prevention:** (1) Keep dnd-kit and cytoscape on separate pages/routes (currently already the case -- kanban vs graph). (2) If combining them, ensure the DndContext is outside any Suspense boundary. (3) Test drag interactions after lazy-loaded components finish loading.
- **Detection:** Open kanban board, then navigate to graph and back -- verify drag-and-drop still works.

### Pitfall: Tree-Sitter JavaScript Grammar Treats .ts/.tsx as JavaScript
- **Category:** correctness
- **Stack:** tree-sitter-javascript >=0.23
- **Description:** In `codebase_service.py` (line 25), `.ts` and `.tsx` extensions are mapped to the "javascript" language which uses `tree_sitter_javascript`. The tree-sitter JavaScript grammar does NOT support TypeScript syntax (type annotations, interfaces, enums, generics). When parsing TypeScript files, the JavaScript parser will produce parse errors or incorrect ASTs. For the current DAG-only view this is acceptable (symbols are still partially extracted), but for a "full codebase graph" that needs accurate symbol resolution and import/export tracking, this will produce incorrect or incomplete data.
- **Trigger:** Parsing any `.ts` or `.tsx` file that uses TypeScript-specific syntax (type annotations, interfaces, generics, enums, `as` casts).
- **Impact:** Medium -- symbol extraction is incomplete or incorrect for TypeScript files. Import/export edges in the codebase graph may be missing, leading to disconnected nodes.
- **Prevention:** (1) Add `tree-sitter-typescript` as a dependency and create separate language mappings for `.ts` and `.tsx`. (2) The tree-sitter-typescript package provides both `tree_sitter_typescript.language_typescript()` and `tree_sitter_typescript.language_tsx()`. (3) Update `SUPPORTED_LANGUAGES` and `SYMBOL_QUERIES` with TypeScript-specific node types.
- **Detection:** Parse a TypeScript file with type annotations and check `parse_errors` in the response. Count symbols in a known `.ts` file and compare against actual definitions.
- **References:** https://github.com/tree-sitter/tree-sitter-typescript

### Pitfall: Aggressive Polling Interval on Codebase Endpoint
- **Category:** performance
- **Stack:** @tanstack/react-query ^5.91.3, FastAPI
- **Description:** All view hooks in `useViews.ts` use `refetchInterval: 2000` (2 seconds) and `staleTime: 1000`. The codebase tree endpoint parses up to 500 files with tree-sitter on every call. Even with the mtime-based cache in `_parse_cache`, the `os.walk()` + `os.path.getmtime()` calls for 500 files happen on every request. With a 2-second polling interval, the backend is performing a full directory walk 30 times per minute, even when the user is just viewing the page.
- **Trigger:** Opening the codebase page -- the 2-second refetch starts immediately and continues indefinitely.
- **Impact:** Medium -- unnecessary CPU and I/O load on both frontend (re-renders) and backend (directory walks). On networked filesystems or large projects, each walk can take 100-500ms.
- **Prevention:** (1) Increase `refetchInterval` for the codebase endpoint to 30-60 seconds (codebase changes are infrequent). (2) Use `refetchOnWindowFocus: true` instead of polling for the codebase view. (3) Keep the 2-second interval only for state/DAG views where changes are frequent during active development. (4) Add `staleTime: 30000` to the codebase query.
- **Detection:** Open browser DevTools Network tab on the codebase page -- observe one request every 2 seconds to the codebase endpoint.

### Pitfall: No File Content Viewing in Current Backend Architecture
- **Category:** correctness
- **Stack:** FastAPI, Python, tree-sitter
- **Description:** The milestone requires "click-into-file viewing capability" for the codebase graph. The current backend has no endpoint for reading file contents -- only `get_codebase_tree()` which returns symbols. Adding a file-content endpoint requires careful security consideration: path traversal attacks, reading files outside the project directory, and serving binary files could expose sensitive data or crash the server.
- **Trigger:** Implementing a `/api/projects/{id}/file?path=...` endpoint without proper path validation.
- **Impact:** Medium -- path traversal vulnerability (e.g., `?path=../../../etc/passwd`) could expose system files. Serving large binary files without size limits could crash the backend.
- **Prevention:** (1) Resolve the requested path and verify it starts with the project's registered path (`Path.resolve()` then `is_relative_to()`). (2) Reject paths containing `..` or absolute paths. (3) Enforce a maximum file size (e.g., 1MB, matching `_MAX_FILE_SIZE`). (4) Return `text/plain` content type only. (5) Reject binary files (check for null bytes in the first 8KB).
- **Detection:** Attempt to read `?path=../../../../etc/passwd` -- should return 403 or 400, not file contents.

## Anti-Patterns for This Stack

### Anti-Pattern: Creating cytoscape instance inside useEffect without ref guard
- **What people do:** Call `cytoscape({ container })` inside useEffect every time data changes, without checking if `cyRef.current` already exists.
- **Why it fails:** Multiple cytoscape instances are created on the same container, causing visual artifacts, doubled event handlers, and memory leaks. Each instance creates its own WebGL/Canvas context.
- **Correct approach:** Check `if (cyRef.current)` before creating; update elements on the existing instance with `cy.batch()`. Only create a new instance when the container ref changes or on initial mount.
- **Example:** The current `KnowledgeGraphPage.tsx` correctly implements this pattern (lines 128-133 vs 146-232). Any refactor must preserve this guard.

### Anti-Pattern: Using async def for CPU-bound FastAPI endpoints
- **What people do:** Define a FastAPI endpoint with `async def` and call synchronous, CPU-bound code (like tree-sitter parsing or filesystem walks) directly.
- **Why it fails:** `async def` endpoints run on the main event loop. Synchronous calls block the entire event loop, making the server unresponsive to ALL requests until the call completes.
- **Correct approach:** Use regular `def` (FastAPI automatically runs it in a threadpool) or use `asyncio.to_thread()` for long-running sync calls inside `async def`.
- **Example:** Current `get_codebase_view` correctly uses `def` (not `async def`).

### Anti-Pattern: Hardcoding version strings in multiple files
- **What people do:** Copy the version string into `__init__.py`, `pyproject.toml`, frontend components, and other locations independently.
- **Why it fails:** When bumping versions, one or more locations are invariably missed. The system reports conflicting versions in different contexts.
- **Correct approach:** Use a single source of truth. Python: read from `pyproject.toml` using `importlib.metadata.version()`. Frontend: fetch from the `/api/health` endpoint at runtime rather than embedding at build time.

### Anti-Pattern: Implementing "save on Ctrl+click" instead of "save on Ctrl+S"
- **What people do:** Add a click handler that checks for `e.ctrlKey` to implement a "quick save" shortcut.
- **Why it fails:** Ctrl+click has platform-specific behavior (context menu on macOS, opens link in new tab in browsers). It conflicts with dnd-kit's pointer sensors. It is not a recognized UX pattern for "save."
- **Correct approach:** Use Ctrl+S (Cmd+S on macOS) or Ctrl+Enter as the save shortcut. These are universally recognized keyboard shortcuts. Always call `e.preventDefault()` to suppress browser defaults.

### Anti-Pattern: Breaking on first shell config match when multiple exist
- **What people do:** Use `break` after finding the first shell config file containing the env var, assuming there is only one.
- **Why it fails:** Users with multiple shells (bash + fish, zsh + bash) have RAPID_TOOLS in multiple config files. Updating only one leaves the other stale.
- **Correct approach:** Collect ALL matching files into a list, present them all to the user, and update all of them.

## Version-Specific Gotchas

- **React 19.2.4**: The React Compiler (if enabled) will flag `ref.current` access during render as an error. If the cytoscape graph overhaul reads `cyRef.current` outside of effects or event handlers, the compiler will reject it. Keep all ref access inside useEffect, useCallback, or event handlers.
- **Vite 8.0.1**: Vite 8 changed the default module resolution. The `resolve.tsconfigPaths: true` setting replaces the old `vite-tsconfig-paths` plugin. Ensure `tsconfig.json` paths are correctly configured if adding new path aliases for graph components.
- **react-router 7.13.1**: React Router v7 merged with Remix. If adding new routes for file viewing or graph detail views, the loader/action API is different from v6. `useLoaderData` now requires typed loaders via `Route.LoaderArgs`. For this project, since no data loaders are currently used (all data fetching via react-query), this is low risk unless someone adds loaders.
- **FastAPI >=0.135**: Pydantic v2 is required. `response_model` validation is stricter -- returning a dict that does not match the model's field types will raise a validation error (not silently coerce). Any new response models for the codebase graph must exactly match the returned data structure.
- **tree-sitter 0.24+**: The Python bindings changed their API from 0.23 to 0.24. The `Language()` constructor now takes a `language()` function pointer from grammar modules, not a shared library path. The current `codebase_service.py` uses the new API correctly. Do not copy patterns from older tree-sitter tutorials.
- **Tailwind CSS 4.2.2**: Tailwind v4 removed the `tailwind.config.js` file in favor of CSS-based configuration. Custom theme values (colors, spacing) must be defined in CSS using `@theme`. If adding new graph-specific theme colors, they go in the CSS file, not a JS config.

## Security Pitfalls

1. **Path traversal in file content endpoint:** Any new endpoint that reads file contents based on a user-supplied path MUST validate the resolved path stays within the project directory. Use `Path.resolve().is_relative_to(project_path)` as the guard. Without this, an attacker can read any file on the server.
2. **Unsanitized project paths in codebase service:** `get_codebase_tree()` accepts a `Path` directly from the registered project and follows symlinks (`followlinks=False` is already set -- keep this). If a project path itself is a symlink, `os.walk` could escape the project directory. Verify the resolved path at registration time.
3. **CORS origins limited to localhost:** The FastAPI app allows CORS only from `http://127.0.0.1:5173` and `http://localhost:5173`. This is correct for development but if the overhaul adds any new frontend origin (e.g., a different port for graph debugging), it must be added to the allowlist.
4. **No authentication on API endpoints:** All endpoints are unauthenticated. The service binds to `127.0.0.1` by default (safe), but if anyone changes `RAPID_WEB_HOST` to `0.0.0.0`, the entire API including file reading is exposed to the network. The file content endpoint amplifies this risk.
5. **MD5 checksums in SyncEngine:** `sync_engine.py` line 84 uses MD5 for file checksums. While this is marked with `# noqa: S324`, MD5 is cryptographically broken. For integrity checking it is adequate, but if checksums are ever used for security decisions (e.g., verifying file authenticity), switch to SHA-256.

## Performance Traps

1. **2-second polling on heavyweight endpoints:** All view hooks poll every 2 seconds. The codebase endpoint walks the filesystem and parses files on every call. This should be differentiated -- lightweight endpoints (state, DAG) can poll frequently; heavyweight endpoints (codebase) should poll rarely or on-demand.
2. **Cytoscape dagre layout on every data change:** If the codebase data refetches every 2 seconds and the graph re-layouts each time, the CPU will be consumed by continuous dagre calculations. Cache the layout positions and only re-layout when the node/edge set actually changes (compare element counts or checksums).
3. **Module-level `_parse_cache` grows unbounded:** The `_parse_cache` dict in `codebase_service.py` (line 13) maps `(filepath, mtime)` to symbol lists. When files are modified, new cache entries are added but old entries are never evicted. Over long server uptime with active development, this cache grows indefinitely. Add an LRU eviction strategy or cap cache size.
4. **`os.walk` on every codebase request:** Even with cached parsing results, the full `os.walk` traversal happens on every request to discover which files exist. For a project with thousands of files (skipping node_modules etc.), this is 50-200ms of pure I/O per request. Cache the file list with a short TTL (10-30 seconds).
5. **DragOverlay re-renders on every pointer move:** The kanban DragOverlay renders a clone of the active card. During drag operations, React re-renders this component on every pointer move event. For complex cards with markdown descriptions, this can cause frame drops. Keep the DragOverlay content minimal.
