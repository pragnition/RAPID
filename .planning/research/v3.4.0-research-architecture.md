# Architecture Research: v3.4.0 Agent Intelligence

## Current Architecture

### Existing Pattern: Layered Plugin Architecture

RAPID uses a 5-layer architecture (Skill > Agent > CLI > Library > Storage) with clear separation of concerns. The current strengths and how they relate to the v3.4.0 milestone:

**Strengths that support the new requirements:**
- **Modular library layer (src/lib/):** 21 independent modules with clean public APIs make it straightforward to add new modules (memory, hooks, documentation) without disturbing existing functionality.
- **State machine with Zod validation:** The `withStateTransaction` pattern and schema validation at every mutation boundary means extending STATE.json (for memory, quick-task logs, hooks metadata) is safe -- invalid mutations are rejected at write time.
- **Skill-level orchestration:** Skills (SKILL.md files) are self-contained orchestration scripts that compose agents and CLI calls. New skills (/documentation, enhanced /quick, /add-set improvements) fit naturally into this pattern.
- **Agent build pipeline:** The `src/modules/roles/*.md` -> `agents/*.md` assembly pipeline means adding or enriching agent context is a matter of editing role modules and rebuilding. This directly supports the "code quality through richer context" goal.
- **Contract-first planning:** CONTRACT.json already defines function/type exports and cross-set imports. Extending this to include UI specs is architecturally coherent.

**Areas needing extension or modification:**
- **No persistent memory system:** The current architecture is stateless between milestones -- agents have no recall of user preferences, mid-course corrections, or accumulated decisions. STATE.json tracks progress but not intent.
- **No hook execution framework:** The single `rapid-task-completed.sh` hook is a bare shell script with no integration into the agent lifecycle. There is no hook registry, no pre/post event system, and no mechanism for hooks to validate agent behavior.
- **No documentation generation pipeline:** The context generation system (`src/lib/context.cjs`) detects codebases but does not generate user-facing documentation. No module exists for documentation synthesis.
- **Quick task logging is absent:** `/quick` creates artifacts in `.planning/quick/` but these are never indexed, never surfaced in status commands, and never available as memory.
- **CONTRACT.json lacks UI dimensions:** The contract meta-schema only covers function exports, type exports, imports, and behavioral invariants. There is no schema for UI component specifications, layout contracts, or visual consistency rules.
- **Planning context is thin:** CONTEXT.md and PLAN.md contain scope and task descriptions, but lack enriched metadata like complexity estimates, test coverage targets, acceptance criteria detail, or architectural decision records per task.

## Recommended Architecture Pattern

- **Pattern:** Event-driven module extension with persistence layer
- **Rationale:** The milestone's 7 features all require extending the existing layered architecture rather than replacing it. The common thread is: capture more information at lifecycle transition points (events), persist that information durably (storage), and make it available to downstream agents (context enrichment). An event/hook system at lifecycle transitions is the minimal architectural addition that enables hooks, memory, logging, and state-update enforcement -- without restructuring the existing command flow.
- **Evidence:** The existing `state transition set` mechanism already implements a primitive version of this pattern (transition = event, validation = hook). The `rapid-task-completed.sh` demonstrates the hook concept at the Claude Code platform level. Extending this into a first-class event system within RAPID's own lifecycle is the natural evolution.

**Rationale for structure decisions:**
- New modules follow the existing `src/lib/{module}.cjs` + `src/commands/{handler}.cjs` pattern -- no new architectural concepts needed.
- Memory is stored as append-only JSONL files for durability (no data loss on crash) and simplicity (no database dependency).
- UI specs extend CONTRACT.json rather than creating a parallel specification system -- agents already know how to read and validate contracts.
- Hook configuration lives in `.planning/hooks/` alongside other planning artifacts, keeping the `.planning/` directory as the single source of truth for project metadata.

## Module Organization

### Module: hooks.cjs (NEW)

- **Responsibility:** Register, configure, and execute lifecycle hooks that run at agent completion points. Hooks verify that agents have fulfilled their obligations (state updates, artifact creation, contract compliance).
- **Public interface:**
  - `registerHook(cwd, hookConfig)` -- add a hook to the registry
  - `executeHooks(cwd, event, context)` -- run all hooks for a lifecycle event
  - `getHookConfig(cwd)` -- read hook registry
  - `validateAgentCompletion(cwd, setId, phase)` -- specialized hook: check if agent updated state
- **Dependencies:** state-machine.cjs (for reading state), core.cjs (for project root resolution), lock.cjs (for safe config writes)
- **Data ownership:** `.planning/hooks/config.json`, hook execution logs

### Module: memory.cjs (NEW)

- **Responsibility:** Persistent storage and retrieval of user decisions, mid-course corrections, and accumulated project knowledge. Provides both structured query (by set, by topic, by date) and digest generation for agent context injection.
- **Public interface:**
  - `recordDecision(cwd, decision)` -- append a decision record to DECISIONS.jsonl
  - `recordCorrection(cwd, correction)` -- append a correction record to CORRECTIONS.jsonl
  - `queryMemory(cwd, filters)` -- query memory by set, topic, date range
  - `generateMemoryDigest(cwd, scope)` -- produce a concise digest for agent context
  - `getRelevantMemory(cwd, setId)` -- get memory entries relevant to a specific set
- **Dependencies:** core.cjs (project root), lock.cjs (append safety)
- **Data ownership:** `.planning/memory/` directory (DECISIONS.jsonl, CORRECTIONS.jsonl, CONTEXT-NOTES.md)

### Module: documentation.cjs (NEW)

- **Responsibility:** Generate and update project documentation by analyzing the codebase, existing docs, and planning artifacts. Produces structured documentation output suitable for user-facing consumption.
- **Public interface:**
  - `analyzeDocCoverage(cwd)` -- assess which areas need documentation
  - `generateDocs(cwd, options)` -- produce documentation for specified scope
  - `updateExistingDocs(cwd, docPath)` -- update a specific doc file based on current codebase state
  - `listDocArtifacts(cwd)` -- enumerate existing documentation
- **Dependencies:** context.cjs (codebase detection), core.cjs (project root), plan.cjs (set/contract info for API docs)
- **Data ownership:** Writes to `docs/` directory and `.planning/` doc artifacts

### Module: quick-log.cjs (NEW)

- **Responsibility:** Index and persist quick task execution history for memory and auditability. Maintains a machine-readable log that other modules can query.
- **Public interface:**
  - `logQuickTask(cwd, taskRecord)` -- append a task record to INDEX.json
  - `getQuickTaskLog(cwd)` -- read the full quick task log
  - `getRecentQuickTasks(cwd, count)` -- get the N most recent tasks
- **Dependencies:** core.cjs (project root), lock.cjs (safe writes)
- **Data ownership:** `.planning/quick/INDEX.json`

### Module: ui-spec.cjs (NEW)

- **Responsibility:** Define, validate, and enforce UI specification contracts across sets. Ensures visual and behavioral consistency when multiple sets touch UI components.
- **Public interface:**
  - `compileUISpec(cwd, setId)` -- compile UI specs from a set's contract
  - `validateUIConsistency(cwd, specs)` -- check cross-set UI consistency
  - `generateUISpecSection(components)` -- generate the UI section for CONTRACT.json
  - `mergeUISpecs(baseSpec, setSpec)` -- merge set-level specs into project-level UI spec
- **Dependencies:** contract.cjs (contract loading/validation), core.cjs (project root)
- **Data ownership:** UI spec sections within CONTRACT.json, `.planning/UI-SPEC.md` (project-level UI specification)

### Module: state-machine.cjs (EXTENDED)

- **Responsibility:** (Existing) State CRUD, transactions, locking. (New) Emit lifecycle events during state transitions so hooks can execute.
- **Public interface:** Existing API unchanged. Internal addition: hook execution calls within `withStateTransaction` and `transitionSetStatus`.
- **Dependencies:** (New) hooks.cjs for post-transition hook execution
- **Data ownership:** `.planning/STATE.json` (unchanged)

### Module: execute.cjs (EXTENDED)

- **Responsibility:** (Existing) Execution context prep and prompt assembly. (New) Inject memory digests and enriched context into agent prompts.
- **Public interface:** Existing API unchanged. `prepareSetContext` now includes memory digest. New: `assembleEnrichedContext(cwd, setId)` for richer prompt content.
- **Dependencies:** (New) memory.cjs for digest generation, compaction.cjs for token budgeting
- **Data ownership:** No new data ownership

### Module: contract.cjs (EXTENDED)

- **Responsibility:** (Existing) Interface contract management. (New) UI specification section in the contract meta-schema.
- **Public interface:** Existing API unchanged. Meta-schema extended with optional `ui` section.
- **Dependencies:** No new dependencies
- **Data ownership:** CONTRACT.json schema extended (backward compatible via optional field)

### Module: context.cjs (EXTENDED)

- **Responsibility:** (Existing) Language detection and project analysis. (New) Richer context assembly including doc coverage analysis, complexity metrics, and architectural annotations.
- **Public interface:** Existing API plus `generateEnrichedAnalysis(cwd)` for deeper project understanding.
- **Dependencies:** No new external dependencies
- **Data ownership:** `.planning/context/` files (richer content)

## Data Flow

### Memory Recording Flow
1. User makes a decision during `/discuss-set` or `/new-version` -> Skill captures decision
2. Skill calls `memory.cjs recordDecision()` -> Appends to `.planning/memory/DECISIONS.jsonl`
3. User changes mind during `/discuss-set` (re-discuss) -> Skill calls `memory.cjs recordCorrection()` -> Appends to `.planning/memory/CORRECTIONS.jsonl`
4. During `/plan-set` or `/execute-set`, `execute.cjs prepareSetContext()` calls `memory.cjs generateMemoryDigest()` -> Memory digest injected into agent prompt

### Hook Execution Flow
1. Agent finishes execution -> Returns RAPID:RETURN marker
2. Skill orchestrator parses RAPID:RETURN -> Extracts status, artifacts, commits
3. Skill calls `hooks.cjs executeHooks(cwd, 'agent-complete', context)` -> Hooks registry is loaded
4. Each registered hook executes:
   - `state-check` hook: Reads STATE.json, verifies set status was updated if the phase requires it
   - If hook fails: Skill prompts agent to fix the omission or reports to user
5. Skill proceeds with state transition and commit

### Quick Task Logging Flow
1. `/quick` Step 6 (completion) -> In addition to writing SUMMARY.md, calls `quick-log.cjs logQuickTask()`
2. Log record includes: task ID, slug, description, date, status, commits, artifacts
3. `.planning/quick/INDEX.json` is updated atomically
4. Future `/quick` invocations can reference past tasks via `getRecentQuickTasks()`
5. Memory system can include recent quick tasks in agent context via `generateMemoryDigest()`

### Documentation Generation Flow
1. User invokes `/documentation` -> Skill spawns rapid-documentation agent
2. Agent calls `documentation.cjs analyzeDocCoverage()` -> Identifies gaps
3. Agent reads existing docs in `docs/`, planning artifacts, source code
4. Agent generates/updates documentation -> Writes to `docs/`
5. Skill commits documentation changes

### UI Spec Contract Flow
1. During `/plan-set`, planner agent identifies UI components in the set
2. Planner generates UI spec section -> Included in CONTRACT.json `ui` field
3. During `/execute-set`, executor reads UI specs from its own + dependent set contracts
4. Cross-set UI consistency validated via `ui-spec.cjs validateUIConsistency()` during review
5. Review pipeline includes UI consistency check alongside existing contract validation

### Enriched Context Flow (Code Quality)
1. During `/plan-set`, set planner receives enriched context:
   - Memory digest (user decisions, corrections)
   - Existing code patterns from context analysis
   - Previous wave summaries (compacted)
   - UI spec constraints (if applicable)
2. Planner generates richer PLAN.md with:
   - Explicit acceptance criteria per task
   - Complexity estimates
   - Test coverage targets
   - Architectural notes
3. During `/execute-set`, executor receives the enriched PLAN.md plus:
   - Same memory digest
   - Contract + UI specs
   - Inter-set dependency context
4. Result: agents have more descriptive, actionable context -> higher code quality

### State Management

- **Where state lives:**
  - Primary state: `.planning/STATE.json` (project/milestone/set hierarchy, Zod-validated)
  - Memory state: `.planning/memory/*.jsonl` (append-only event log, no schema migration needed)
  - Quick task log: `.planning/quick/INDEX.json` (atomic JSON updates)
  - Hook config: `.planning/hooks/config.json` (static configuration, rarely changes)
  - UI specs: Within `CONTRACT.json` per set (existing validation pipeline)
  - Documentation: `docs/` directory (versioned with git)

- **How state is read and written:**
  - STATE.json: `withStateTransaction()` (lock -> read -> validate -> mutate -> validate -> write -> unlock)
  - Memory JSONL: Append-only with file lock (no read-modify-write needed)
  - Quick log: Read full -> modify -> atomic write (same pattern as STATE.json but simpler schema)
  - Hook config: Read-only at runtime; written by CLI commands or init

- **Consistency guarantees:**
  - STATE.json: Zod validation at every boundary, file lock prevents concurrent mutation
  - Memory: Append-only design means no lost writes; JSONL entries are independently parseable
  - Git commits: State transitions + artifacts committed together (atomic from git's perspective)

## API Design

### CLI Extensions (rapid-tools.cjs)

New commands follow existing kebab-case convention:

```
# Memory commands
memory record-decision <json>      Record a user decision
memory record-correction <json>    Record a mid-course correction
memory query [--set <id>] [--topic <t>] [--since <date>]  Query memory
memory digest [--set <id>]         Generate memory digest for agent context

# Hook commands
hooks list                         List registered hooks
hooks run <event> [--set <id>]     Manually trigger hooks for an event
hooks config                       Show hook configuration

# Documentation commands
docs analyze                       Analyze documentation coverage
docs generate [--scope <s>]        Generate documentation
docs update <path>                 Update specific documentation file

# Quick task log
quick-log list                     List all quick tasks
quick-log recent [--count N]       Show recent quick tasks
```

**Conventions applied:**
- All commands return JSON to stdout, status messages to stderr
- Error format: `[RAPID ERROR] message\n` to stderr
- Success format: `JSON.stringify(result) + '\n'` to stdout
- Commands accept `--json` for machine-readable output (default is JSON for new commands)

### CONTRACT.json UI Extension

```json
{
  "exports": { ... },
  "imports": { ... },
  "behavioral": { ... },
  "ui": {
    "components": [
      {
        "name": "SetStatusBadge",
```

The `ui` section is optional -- sets that do not touch UI simply omit it. Validation is backward-compatible: existing CONTRACT.json files without `ui` remain valid.

## Integration Points

| External System | Protocol | Purpose | Error Handling |
|----------------|----------|---------|---------------|
| Git (worktrees) | CLI exec | Branch isolation, commit tracking | Retry on lock contention, error message with recovery steps |
| Claude Code Plugin API | Skill/Agent definitions | Agent spawning, skill routing | RAPID:RETURN protocol for agent failure reporting |
| Node.js fs | File I/O | State persistence, artifact management | Atomic writes (write tmp + rename), lock protection |
| proper-lockfile | File locks | Concurrent state access prevention | Timeout (300s), stale lock cleanup, automatic retry |
| Zod | Runtime validation | State schema enforcement | Fail-fast with descriptive error messages |
| Ajv | JSON Schema | Contract validation | Schema compilation errors surfaced at validation time |
| jq (optional) | CLI exec | Hook input parsing (shell hooks) | Graceful degradation if jq unavailable |

## Scaling Considerations

- **Context window pressure:** The enriched context (memory, UI specs, richer plans) increases token usage per agent prompt. The existing compaction system (`compaction.cjs`, 120k token budget) must be extended to budget across all context sources, not just wave artifacts. Memory digests should be pre-compacted to a fixed token budget (e.g., 2000 tokens for memory, 1000 tokens for UI specs) before injection.
- **Memory growth:** JSONL files grow unboundedly over the project lifetime. A compaction/archival strategy is needed: archive memory older than 2 milestones, keep only the digest. The `generateMemoryDigest()` function should summarize rather than concatenate.
- **Hook execution time:** Hooks run synchronously after agent completion. If hook logic becomes complex (e.g., full contract validation), it could add latency to the agent pipeline. Hooks should be lightweight checks (file existence, state field verification) not heavy computation.
- **Documentation generation cost:** Generating docs from a large codebase requires reading many files. The documentation module should use the existing `SKIP_DIRS` list from `context.cjs` and limit file scanning depth. Consider generating docs incrementally (per-set changes) rather than full-project regeneration.
- **Quick task log size:** INDEX.json grows with each quick task but typically slowly (quick tasks are infrequent). No scaling concern for foreseeable usage. Cap at 500 entries with archival rotation if needed.

## Architecture Risks

1. **Memory context pollution (high impact):** If the memory digest becomes too large or contains irrelevant decisions, it could confuse agents rather than help them. **Mitigation:** Scope memory queries by set relevance; apply token budget caps; use the compaction/digest pattern (summaries, not raw logs); validate that memory improves output quality via /review pipeline.

2. **Hook execution failure blocking the pipeline (medium impact):** If a hook throws an unhandled error, it could prevent the skill from completing its state transition. **Mitigation:** Wrap all hook executions in try/catch; log hook failures but do not block the main pipeline; hooks report warnings, not hard failures; provide `--skip-hooks` escape hatch for emergencies.

3. **UI spec scope creep (medium impact):** UI specifications could become arbitrarily detailed, duplicating work that belongs in the implementation rather than the contract. **Mitigation:** Define a narrow schema for UI specs (component names, prop types, layout patterns) not pixel-level design; enforce maximum spec size at the schema level.

4. **Backward compatibility with existing projects (medium impact):** Projects initialized with v3.3.0 do not have memory directories, hook config, or UI spec sections. Running v3.4.0 commands against old projects must not break. **Mitigation:** All new `.planning/` subdirectories are created on first use (lazy initialization); new CONTRACT.json fields are optional; the existing `migrate.cjs` module can handle schema evolution.

5. **Circular dependency between hooks and state-machine (low impact):** hooks.cjs depends on state-machine.cjs for reading state; state-machine.cjs would depend on hooks.cjs for executing post-transition hooks. **Mitigation:** Use dependency injection -- state-machine emits events via a callback function; the CLI layer wires the hook executor as the callback. No direct circular require() needed.

6. **Documentation freshness (low impact):** Generated documentation can drift from the actual code between generation runs. **Mitigation:** Documentation is versioned via git and regenerated at milestone boundaries (the intended workflow). The analyze-coverage command can detect drift.

## Recommendations

1. **Implement the hook system first (priority: critical):** The hook system is the foundation for the state-update enforcement feature (goal 4). It is also architecturally foundational -- memory recording (goal 5) and quick-task logging (goal 7) are effectively hooks at specific lifecycle points. Build `hooks.cjs` with an event-callback pattern, then wire state-check as the first built-in hook.

2. **Build the memory module second (priority: critical):** Memory (goal 5) underpins code quality (goal 6) because richer agent context requires remembering what the user said. Implement `memory.cjs` with append-only JSONL storage and a digest generator. Wire recording into `/discuss-set`, `/new-version`, and `/plan-set` skills.

3. **Extend CONTRACT.json for UI specs (priority: high):** UI specs (goal 3) are a schema extension to the existing contract system. This is a targeted change -- extend the Ajv meta-schema in `contract.cjs`, add `ui-spec.cjs` for compilation/validation, and update the planner/executor roles to consume UI specs. Do this before the documentation module because doc generation benefits from knowing UI contracts.

4. **Add quick-task logging to /quick (priority: high):** Quick-task logging (goal 7) is the most contained feature -- add `quick-log.cjs` module, update `/quick` SKILL.md to call it at completion, and expose via CLI. Small blast radius, high value for operational memory.

5. **Enrich planning context for code quality (priority: high):** Code quality (goal 6) requires enriching PLAN.md generation and executor context injection. This means updating `role-set-planner.md` and `role-executor.md` to demand/produce richer task descriptions (acceptance criteria, test targets, complexity). Also update `execute.cjs` to inject memory digest + contract summaries.

6. **Enhance /add-set with memory awareness (priority: medium):** The add-set feature (goal 1) already exists as a skill. Enhancements: make it memory-aware (surface relevant past decisions when creating a new set), and ensure the new set gets properly indexed in the hook system. This is an incremental improvement, not a new system.

7. **Build the documentation generation pipeline last (priority: medium):** Documentation (goal 2) is the most standalone feature -- it does not block or enable other features. Implement `documentation.cjs`, create the `/documentation` skill and `role-documentation.md` agent role. This can be the last set in the milestone without affecting any other deliverable.

8. **Use dependency injection for hook-state coupling (priority: critical -- architectural decision):** Avoid circular requires between hooks.cjs and state-machine.cjs. The CLI layer (`rapid-tools.cjs` or the skill orchestrator) should wire `hooks.executeHooks` as a callback passed to `state-machine.withStateTransaction`. This keeps modules independently testable and avoids Node.js circular dependency issues.

9. **Apply token budgets to enriched context (priority: high -- architectural decision):** The compaction system currently budgets wave artifacts at 120k tokens. With memory digests, UI specs, and enriched plans all competing for context window space, implement a budget allocation system: e.g., 100k for wave context, 10k for memory, 5k for UI specs, 5k for contract summaries. Make this configurable in `.planning/config.json`.

10. **Lazy initialization for new .planning/ subdirectories (priority: high -- architectural decision):** All new directories (`.planning/memory/`, `.planning/hooks/`, `.planning/quick/INDEX.json`) must be created on first use, not during init. This ensures backward compatibility with existing v3.3.0 projects and avoids breaking the scaffold module. Each new module's public functions should call an internal `_ensureDir()` helper before any file operation.
