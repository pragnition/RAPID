### Feature 1: DAG.json as Central Concept

- **Description:** Make DAG.json the single source of truth for dependency tracking, team group assignment, and workflow ordering throughout RAPID. Currently, DAG.json stores nodes (with id, wave, status), edges (from/to), waves (set groupings), and metadata. The goal is to elevate it so that every subsystem (merge ordering, scaffold grouping, status display, execution scheduling) reads from DAG.json rather than reconstructing dependency information independently.
- **Complexity:** complex
- **Dependencies:** independent (foundational -- other features build on this)

#### Decomposition

1. **Extend DAG node schema** -- Add optional fields to DAG nodes: `group` (developer team assignment), `priority` (ordering hint), `description` (human-readable scope). The existing `wave` and `status` fields remain. Maintain backward compatibility with v1 DAGs that lack these fields.
2. **DAG as canonical execution driver** -- Refactor `getExecutionOrder()` and `getMergeOrder()` to read exclusively from DAG.json rather than also consulting STATE.json set ordering. Ensure merge.cjs, execute.cjs, and status display all derive ordering from DAG.
3. **DAG status synchronization** -- Implement a `syncDAGStatus(cwd)` function that reads set statuses from STATE.json and updates DAG node statuses, called after every state transition. Currently DAG nodes have a `status` field that is set to `'pending'` at creation and never updated.
4. **DAG CLI enhancements** -- Extend `dag show` to display group annotations, dependency arrows, and status colors. Add `dag validate` subcommand that checks DAG/STATE.json consistency.
5. **DAG documentation and migration** -- Write migration logic for existing DAGs that lack group/priority fields (default group=null, priority=wave).

#### Implementation Strategies

**Option 1: Extend v1 DAG format in-place**
- Approach: Add optional fields to the existing v1 `createDAG()` function and its output schema. New fields are nullable/optional so existing DAGs validate without migration.
- Pros: No breaking change; existing projects continue working; simpler migration story.
- Cons: v1 and v2 functions coexist (there is already `createDAGv2` in dag.cjs); may accumulate technical debt if v1 is kept indefinitely.
- Libraries/services: None additional -- pure Node.js data structure changes.

**Option 2: Migrate fully to DAGv2 with group support**
- Approach: Deprecate v1 `createDAG()`, extend `createDAGv2()` with group annotations, and add auto-migration that converts v1 DAGs on load. All consumers use a single `loadDAG()` function that normalizes to v2 format internally.
- Pros: Clean single format; v2 already exists with typed nodes; reduces code paths.
- Cons: Requires migration of all existing project DAGs; potential breakage if edge cases missed.
- Libraries/services: None additional.

**Recommended:** Option 2. The v2 DAG format already exists in `createDAGv2()` and `validateDAGv2()`. Extending it with group annotations and deprecating v1 avoids maintaining two parallel systems. Auto-migration on `tryLoadDAG()` minimizes user friction.

#### Data Model Impact

- Modified entity: DAG node object gains `group: string|null`, `priority: number` fields
- Modified entity: DAG metadata gains `groups: { [groupName]: string[] }` summary object
- New function: `syncDAGStatus(cwd)` in dag.cjs
- New function: `migrateDAGv1toV2(dagV1)` in dag.cjs
- Modified function: `tryLoadDAG(cwd)` auto-detects version and migrates

#### API Surface

- Modified: `dag show` -- displays group annotations, dependency arrows
- New: `dag validate` -- checks DAG/STATE.json consistency
- Modified: `dag generate` -- produces v2 DAGs with group annotations
- Internal: `syncDAGStatus()` called by state-machine.cjs after transitions

### Feature 2: DAG-driven Set Grouping

- **Description:** The roadmapper auto-assigns sets to developer groups (DAG annotations). Cross-group dependencies are shown explicitly. Developers are free to adapt the assignments. Groups represent logical team assignments -- "Team A works on sets X and Y while Team B works on Z."
- **Complexity:** moderate
- **Dependencies:** Depends on Feature 1 (DAG.json as central concept)

#### Decomposition

1. **Group assignment algorithm** -- Implement a `assignGroups(dag, teamSize)` function that partitions sets into N groups (where N = teamSize) minimizing cross-group dependencies. Use a greedy graph coloring heuristic: process sets in topological order, assign each to the group with the fewest conflicting edges.
2. **Roadmapper integration** -- Extend the roadmapper agent's output schema to include group assignments. The roadmapper proposes groups; the user can override during approval.
3. **DAG group persistence** -- Store group assignments in DAG.json node objects and in a top-level `groups` summary. Provide `dag regroup --team-size N` CLI command for re-assignment.
4. **Cross-group dependency visualization** -- In `dag show`, highlight edges that cross group boundaries with distinct formatting. Show a group summary table: "Group A: [set-1, set-2] | Group B: [set-3, set-4]".

#### Implementation Strategies

**Option 1: Algorithmic assignment with greedy graph coloring**
- Approach: Process nodes in topological order. For each node, count edges to nodes already assigned per group. Assign to the group with the fewest existing cross-edges. Tie-break by group size (prefer smaller groups).
- Pros: Fast (O(V+E)), deterministic, reasonable quality for typical RAPID project sizes (5-15 sets).
- Cons: Not globally optimal -- greedy heuristics can produce suboptimal partitions for adversarial graphs.
- Libraries/services: None -- pure algorithm in dag.cjs.

**Option 2: Roadmapper-driven semantic grouping**
- Approach: Let the LLM roadmapper agent assign groups based on semantic understanding of set scopes (e.g., "frontend sets together, backend sets together"). Algorithmic assignment as fallback.
- Pros: Semantically aware -- groups make intuitive sense to developers. Can consider file ownership patterns.
- Cons: Non-deterministic; requires re-running roadmapper to regroup; harder to test.
- Libraries/services: None -- uses existing agent pipeline.

**Recommended:** Hybrid -- Option 2 for initial assignment (roadmapper proposes groups during init), Option 1 as the `dag regroup` CLI fallback for re-assignment without running agents.

#### Data Model Impact

- New field on DAG nodes: `group: string` (e.g., "team-a", "team-b")
- New top-level DAG field: `groups: { [groupName]: { sets: string[], description?: string } }`
- New CLI flag: `dag regroup --team-size N`

#### API Surface

- New: `dag regroup --team-size <N>` -- algorithmic group re-assignment
- Modified: `dag show` -- group-aware display with cross-group edge highlighting
- Modified: `dag generate` -- includes group assignments from roadmapper output

### Feature 3: /scaffold Command Overhaul

- **Description:** Split sets into N developer groups minimizing file conflicts; scaffold high-fidelity stubs for cross-group testing; optional foundational set #0 as shared work before groups branch; stubs live on shared branch, replaced by real implementations.
- **Complexity:** very-complex
- **Dependencies:** Depends on Feature 1 (DAG.json) and Feature 2 (DAG-driven grouping)

#### Decomposition

1. **Group-aware set splitting** -- After roadmap generation, assign sets to developer groups (N groups for N developers). Algorithm minimizes file ownership overlap between groups using OWNERSHIP.json data. Output: per-group set lists stored in DAG.json.
2. **Foundational set #0 (optional)** -- Allow the user to designate a "set-0" or "foundation" set that all groups depend on. This set is executed first on main before groups branch. Contains shared infrastructure, types, interfaces that all groups need.
3. **High-fidelity stub generation** -- Overhaul `stub.cjs` to generate stubs that go beyond "throw not implemented". New stubs include: realistic return values (from CONTRACT.json type signatures), JSDoc types, mock data fixtures, and import/export shapes that match the real API. Stubs are placed on a shared branch accessible to all groups.
4. **Shared stub branch management** -- Create a `rapid/stubs` branch from main after set-0 completes. Each group's worktree branches from `rapid/stubs` instead of main. When a set completes and merges, its stubs are automatically replaced by real implementations.
5. **Stub lifecycle hooks** -- Detect when a stub is still referenced at merge time and warn. Provide `scaffold verify-stubs` command that reports which stubs have been replaced and which remain.
6. **Scaffold report v2** -- Extend `scaffold-report.json` with group assignments, stub file locations, and set-0 designation.

#### Implementation Strategies

**Option 1: Extend existing scaffold with group-aware stub generation**
- Approach: Build on the current `stub.cjs` and `scaffold.cjs` modules. Add a new `generateHighFidelityStub()` function that uses CONTRACT.json type information to produce realistic stubs. Use existing worktree branching with a new `rapid/stubs` base branch.
- Pros: Builds on existing infrastructure; incremental development possible.
- Cons: The current stub format (throw-on-call) is simple; high-fidelity stubs require parsing CONTRACT.json type signatures into mock return values, which adds complexity.
- Libraries/services: None -- extends existing pure-JS stub generation.

**Option 2: Agent-generated stubs**
- Approach: Spawn a stub-generation agent that reads CONTRACT.json and generates semantically appropriate stubs using LLM intelligence. More flexible than rule-based generation.
- Pros: Can produce very realistic stubs; handles edge cases that rule-based code would miss.
- Cons: Non-deterministic; slower; costs API tokens; harder to test.
- Libraries/services: Existing agent pipeline.

**Recommended:** Option 1 for the stub generation (deterministic, fast, testable), with the `rapid/stubs` branch management built into the existing worktree module. Agent-generated stubs could be a future enhancement.

#### Data Model Impact

- New entity: "Foundation set" (set-0) -- a set with `foundation: true` flag in DAG.json
- Modified entity: Stub files gain realistic return values, types, and mock data
- New field: DAG nodes gain `group` assignment (from Feature 2)
- New file: `rapid/stubs` branch with generated stub modules
- Modified file: `scaffold-report.json` gains `groups`, `stubs`, `foundationSet` sections

#### API Surface

- Modified: `scaffold run` -- gains `--groups <N>` and `--foundation` flags
- New: `scaffold verify-stubs` -- reports stub replacement status
- New: `scaffold generate-stubs` -- generates high-fidelity stubs from contracts
- Modified: `worktree create` -- can branch from `rapid/stubs` instead of main
- Modified: `stub.cjs` -- `generateHighFidelityStub()` replaces simple throw-stubs

### Feature 4: /init Spec File Integration

- **Description:** Allow users to provide spec files during `/rapid:init` just as they can during `/rapid:new-version`. Research agents build on existing findings rather than starting fresh, reducing redundant work and improving research quality for users who have pre-existing design documents.
- **Complexity:** moderate
- **Dependencies:** independent

#### Decomposition

1. **Spec file argument parsing in init skill** -- Add `--spec <path>` argument support to the init SKILL.md, mirroring the new-version implementation (Step 0.5 pattern).
2. **Spec-aware discovery bypass** -- When a spec file is provided, extract structured information (vision, features, tech stack, etc.) from the spec and pre-populate the project brief. Present extracted content for user confirmation rather than running the full 13-area discovery conversation. Offer "Review individually" / "Accept all" / "Add more" options matching the new-version pattern.
3. **Spec content injection into research agents** -- Pass spec file content as additional context to all 6 research agents. Research agents receive a `## Existing Research / Spec` section in their task prompt, instructing them to build on rather than duplicate the spec's findings.
4. **Acceptance criteria extraction from spec** -- If the spec contains acceptance criteria (matching the `CATEGORY-NNN:` pattern), auto-extract them for Step 4D rather than generating from scratch. Present for user review and modification.

#### Implementation Strategies

**Option 1: Mirror the new-version spec pattern**
- Approach: Copy the spec file handling pattern from `/rapid:new-version` (Step 0.5 + Spec-Aware Goal Extraction) into `/rapid:init`. Adapt for init's 4-batch discovery flow by replacing batches with spec extraction when specContent is available.
- Pros: Proven pattern; consistent UX across init and new-version; minimal new code.
- Cons: Init's discovery flow is more complex (13 areas vs. 5 categories) so mapping spec content to all areas requires more semantic parsing.
- Libraries/services: None -- uses existing LLM capabilities in the skill prompt.

**Recommended:** Option 1. The new-version SKILL.md already has a working spec-aware extraction flow. Adapt it for init's broader discovery conversation, using LLM semantic mapping to fill the 13 discovery areas from spec content.

#### Data Model Impact

- No new entities or tables
- Modified: Init SKILL.md gains `--spec` argument and spec-aware discovery bypass
- Research agent task prompts gain optional `## Existing Research / Spec` section

#### API Surface

- Modified: `/rapid:init --spec <path>` -- new optional argument
- No CLI (rapid-tools.cjs) changes required -- spec file is handled entirely in the SKILL.md orchestration layer

### Feature 5: /init Meta Principles

- **Description:** During init, capture architectural and style decisions as "meta principles." Store a summary in CLAUDE.md (read by all agents) and the full version in `.planning/PRINCIPLES.md`. Agents read both as context, ensuring consistent decision-making across all sets.
- **Complexity:** moderate
- **Dependencies:** independent

#### Decomposition

1. **Principles capture during init** -- Add a new step after Step 4D (Summary Confirmation) in the init SKILL.md. Present the user with common architectural principle categories and ask them to define their preferences. Categories include: code style (functional vs. OOP), error handling philosophy, testing strategy, dependency management, naming conventions, and project-specific constraints.
2. **PRINCIPLES.md generation** -- Write a detailed `.planning/PRINCIPLES.md` file containing all captured principles with rationale. Format as structured markdown with categories, each principle having a statement and reasoning.
3. **CLAUDE.md summary injection** -- Append a concise "Meta Principles" section to the project's root CLAUDE.md. This section contains one-line summaries of each principle (e.g., "Prefer composition over inheritance", "All public functions must have JSDoc"). The section includes a pointer: "Full principles: .planning/PRINCIPLES.md".
4. **Agent context integration** -- Modify `generateScopedClaudeMd()` in worktree.cjs to include PRINCIPLES.md content (or a token-budgeted excerpt) in worktree-scoped CLAUDE.md files. Agents operating in worktrees automatically receive principles context.
5. **Principles evolution** -- Allow principles to be updated via a new `/rapid:principles` skill or by editing PRINCIPLES.md directly. Changes to PRINCIPLES.md should be reflected in CLAUDE.md on the next `worktree generate-claude-md` run.

#### Implementation Strategies

**Option 1: Interactive principles interview**
- Approach: During init, present structured questions about architectural preferences using AskUserQuestion. Cover 5-7 categories (code style, error handling, testing, naming, dependencies, deployment, documentation). Map answers to principle statements. Write both PRINCIPLES.md (full) and CLAUDE.md summary.
- Pros: Guided experience; ensures consistency; structured output.
- Cons: Adds 5-7 more questions to an already long init flow; may cause user fatigue.
- Libraries/services: None -- pure skill orchestration.

**Option 2: Spec-derived principles with optional interview**
- Approach: If a spec file is provided (Feature 4), extract principles from it. If not, present a concise 2-3 question interview focused on the most impactful decisions (code style, testing, error handling). Offer "Use sensible defaults" option for users who want to skip.
- Pros: Shorter flow; leverages spec file when available; respects user's time.
- Cons: Fewer principles captured by default; "sensible defaults" may not match user preferences.
- Libraries/services: None.

**Recommended:** Option 2. Init is already a long process. A short focused interview for the most impactful principles (with "Use sensible defaults" escape hatch) avoids user fatigue. The spec file integration (Feature 4) provides a natural source for detailed principles when available.

#### Data Model Impact

- New file: `.planning/PRINCIPLES.md` -- full architectural principles document
- Modified file: `CLAUDE.md` -- gains "Meta Principles" summary section
- Modified function: `generateScopedClaudeMd()` in worktree.cjs -- reads and injects PRINCIPLES.md

#### API Surface

- Modified: `/rapid:init` -- new step for principles capture after discovery
- New (optional): `/rapid:principles` -- skill for updating principles post-init
- Modified: `worktree generate-claude-md` -- includes PRINCIPLES.md in scoped output

## Bug Fix Analysis

### Bug 1: REQUIREMENTS.md Overwrite

- **Root Cause:** In Step 5 of the init SKILL.md, `scaffoldProject()` is called which runs `generateRequirementsMd()` to create REQUIREMENTS.md with empty template content. This overwrites the user-approved acceptance criteria from Step 4D, because the scaffold function always regenerates REQUIREMENTS.md in `fresh` and `reinitialize` modes.
- **Affected Code:** `src/lib/init.cjs` lines 236-242 (`fileGenerators` includes `'REQUIREMENTS.md'`), and the init SKILL.md Step 5 calling scaffold before writing criteria.
- **Fix Approach:** Either (a) remove REQUIREMENTS.md from `fileGenerators` in `scaffoldProject()` and have it written exclusively by Step 4D, or (b) reorder init so Step 4D writes happen AFTER scaffold, or (c) have scaffold skip REQUIREMENTS.md if it already contains non-trivial content (matching the Edit tool logic already in Step 4D). Recommended: option (a) -- remove REQUIREMENTS.md from scaffold entirely since it is always populated by the discovery process.

### Bug 2: --desc vs --description Flag Mismatch

- **Root Cause:** The init SKILL.md Step 5 uses `--description` in the scaffold command (`node "${RAPID_TOOLS}" init scaffold --name "{name}" --description "{short description}" --team-size {N} --mode {mode}`), but the CLI parser in `src/commands/init.cjs` line 32 only recognizes `--desc`.
- **Affected Code:** `skills/init/SKILL.md` line 481 and `src/commands/init.cjs` line 32.
- **Fix Approach:** Either (a) update the SKILL.md to use `--desc`, or (b) add `--description` as an alias in the CLI parser alongside `--desc`. Recommended: option (b) -- add `--description` as an alias (`case '--description':`) in the CLI switch statement to accept both forms, plus update the SKILL.md to use the canonical `--desc` for consistency with the help text.

### Bug 3: Roadmapper State Overwrite

- **Root Cause:** In Step 9 of the init SKILL.md, after the roadmapper returns its proposed state, the skill writes the roadmapper's `state` content directly to STATE.json using the Write tool. This overwrites the full STATE.json that was created by scaffold in Step 5 (which includes `version`, `rapidVersion`, `projectName`, `createdAt`, `lastUpdatedAt` fields). The roadmapper only returns `{ milestones, currentMilestone }` without the envelope fields required by the ProjectState Zod schema.
- **Affected Code:** Init SKILL.md Step 9 section "c) Write STATE.json" (line ~894-896) uses Write tool to overwrite STATE.json with incomplete roadmapper output.
- **Fix Approach:** Instead of overwriting STATE.json, use `withStateTransaction()` to merge the roadmapper's milestones/sets into the existing STATE.json. This preserves the envelope fields (version, rapidVersion, projectName, createdAt, lastUpdatedAt) while updating only the milestones array. Alternatively, the SKILL.md can use the Read tool to read existing STATE.json, merge in the roadmapper's milestones, and Write the merged result. Recommended: Add a new CLI command `state set-milestones` (or extend `state add-milestone`) that atomically merges roadmapper output into existing STATE.json using `withStateTransaction()`.

## Tech Debt Analysis

### Agent Subagent Restrictions

- **Current State:** The `core-identity.md` module (line 41) includes a "Namespace Isolation" section instructing agents to only use `rapid:*` skills. However, this is a prompt-level instruction with no enforcement mechanism. Agents can still call non-rapid skills if the LLM ignores the instruction.
- **Approach:** This is inherently a prompt engineering problem since Claude Code's Agent tool does not support skill namespace filtering at the API level. Improvements: (a) strengthen the prompt language in core-identity.md with explicit examples of what NOT to call, (b) add a post-hook that validates agent outputs for non-rapid skill references, (c) add the restriction to the `<conventions>` module so it appears in both the identity and conventions sections.
