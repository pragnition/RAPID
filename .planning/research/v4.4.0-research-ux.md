# Domain & UX Research -- v4.4.0 Polish & Documentation

## Domain Conventions

RAPID is a Claude Code plugin for AI-assisted parallel development orchestration. The domain is **developer workflow tooling** -- comparable to Nx, Turborepo, and Bazel for task orchestration, but focused on AI agent coordination rather than build systems.

### Standard Terminology and Naming Conventions

- **Lifecycle stages** follow a linear pipeline pattern common in CI/CD and build tools: init, plan, execute, review, merge. RAPID already uses this convention consistently.
- **Set/Wave/Job hierarchy** is a RAPID-specific decomposition. Competing tools use "projects/tasks" (Nx), "packages/scripts" (Turborepo). The RAPID terminology is sufficiently distinctive but may confuse newcomers who expect "project" or "task" as top-level units.
- **Slash-command invocation** (`/rapid:command`) follows Claude Code plugin conventions. Other Claude Code plugins use the same `/<plugin>:<skill>` pattern (source: [Claude Code Plugins Reference](https://code.claude.com/docs/en/plugins-reference)).
- **Numeric shorthand** for set references (e.g., `/rapid:review 1` instead of `/rapid:review auth-system`) is a strong UX pattern seen in git interactive rebase and GitHub CLI (`gh pr view 123`). RAPID already implements this.

### Common Workflows and User Journeys

Developer workflow tools consistently present:
1. A **linear progression** from setup to execution to verification (RAPID: install -> init -> start-set -> ... -> merge)
2. A **dashboard/status view** showing overall progress (RAPID: `/rapid:status`)
3. **Next-action guidance** after each step completes -- telling users what to do next (RAPID: partially implemented via completion banners)
4. **Re-entry/resume** from interruptions without lost progress (RAPID: artifact-based re-entry in execute-set)

Source: Nx CLI, Turborepo CLI, GitHub Actions runner conventions.

### Industry Standards

- **Conventional Commits** for commit messages -- RAPID follows this with `type(scope): description`.
- **Semantic Versioning** for milestones -- RAPID uses `v4.x.0` format.
- **README-first documentation** for open-source projects. The conventional structure is: overview, quickstart, installation, detailed guides, API reference, contributing guide. RAPID currently has docs/ with 9 topic-specific guides but no single-entry-point README or quickstart.

## Interaction Models

### Primary Interaction Paradigm

RAPID uses a **conversational CLI** paradigm -- users invoke slash commands in a Claude Code chat session, and agents execute multi-step pipelines autonomously. This is distinct from traditional CLIs (synchronous command-response) and also from pure chat interfaces (freeform conversation).

Key expectations in this paradigm:
- **Commands should be self-documenting.** Users cannot `--help` a slash command easily; `/rapid:help` serves as the manual.
- **Progress indication happens through banners and text output.** There are no progress bars -- the agent streams its work in real-time.
- **Errors should suggest remediation.** RAPID already does this well (e.g., "Run `/rapid:execute-set` first").

### Input/Output Patterns

- **Input:** Slash commands with positional arguments and optional flags (e.g., `/rapid:review 1 --post-merge`). Users expect the argument to be the most common identifier -- RAPID correctly uses set-id as the primary argument.
- **Output:** Agents stream markdown-formatted text. Structured data goes to stdout as JSON; human-readable status goes to stderr. This dual-channel approach follows Unix conventions.
- **Artifacts:** Each pipeline stage produces a named markdown artifact (REVIEW-SCOPE.md, REVIEW-BUGS.md, etc.) that feeds the next stage. This is a strong convention -- comparable to Nx's dependency graph of task outputs.

### Feedback and Progress Indication

- **Stage banners** mark the start of each skill invocation (e.g., `RAPID > REVIEWING`). These use colored backgrounds.
- **Completion banners** at end of each skill show summary and next steps.
- **Missing:** There is no unified progress indicator for multi-stage pipelines. The review pipeline (scope -> unit-test -> bug-hunt -> uat) requires the user to manually invoke each sub-stage. Users of comparable tools (e.g., GitHub Actions, Nx affected pipeline) expect to see which stages are done and which remain.

### Error Communication Patterns

- `[RAPID ERROR]` prefix on stderr for fatal errors -- clear and consistent.
- Error messages include remediation hints (e.g., "Set must be in 'complete' state. Run `/rapid:execute-set` first.").
- Missing: No `[RAPID WARNING]` convention for non-fatal issues. Warnings are currently ad-hoc text.

## Information Architecture

### Documentation for Open-Sourcing

Open-source developer tools follow a documentation hierarchy (source: analysis of Nx docs, Turborepo docs, Claude Code plugin docs):

1. **README.md** -- 1-page overview: what it does, install command, 30-second demo, link to docs
2. **Quickstart guide** -- 5-minute walkthrough of the canonical workflow end-to-end
3. **Concept guides** -- explain mental models (sets, waves, jobs, worktrees, contracts)
4. **Command reference** -- one page per command with arguments, flags, examples, state transitions
5. **Troubleshooting** -- common errors and fixes (RAPID has this: docs/troubleshooting.md)
6. **Contributing guide** -- for open-source contributors

RAPID's current docs/ directory covers setup, planning, execution, review, merge, agents, configuration, state machines, and troubleshooting. The gaps for open-sourcing are:
- No README.md with quick overview and install instructions
- No single quickstart/tutorial walking through the full lifecycle
- No concept/glossary page explaining RAPID-specific terminology (set, wave, job, worktree, contract, solo mode)
- The help command (`/rapid:help`) is good but lives inside a SKILL.md, not as browsable web docs

### Navigation and Discovery

- `/rapid:help` is the entry point. It shows a well-organized command table with a workflow diagram.
- `/rapid:status` is the dashboard. It shows set statuses and suggests next actions.
- **Missing:** No command to show the workflow position of the current set (e.g., "Set auth-system is at step 6/8: Review complete. Next: /rapid:merge").

### Content Hierarchy

The canonical workflow has a clear progressive structure that should be reflected in documentation:

```
Phase 1: Setup (once)        /install, /init, /branding, /scaffold
Phase 2: Per-set (repeat)    /start-set, /discuss-set, /plan-set, /execute-set
Phase 3: Review (per-set)    /review, /unit-test, /bug-hunt, /uat
Phase 4: Integration         /merge
Phase 5: Lifecycle           /audit-version, /new-version
Utilities                    /quick, /bug-fix, /add-set, /status, /help
```

### Help and Documentation Conventions

- **Inline help:** Each skill's SKILL.md contains full documentation, but this is not user-facing. Users see the behavior, not the docs.
- **Error-driven documentation:** RAPID error messages link users to the next command. This is good practice (source: Nx, Turborepo, Docker CLI all use error remediation hints).
- **Recommendation:** Add a `docs/` section to the README pointing to each guide, and ensure the help command's workflow diagram stays in sync with the actual command set.

## Accessibility Considerations

### Terminal Color Accessibility

This is directly relevant to the "better colouring" milestone item.

**Current state:** RAPID uses bright ANSI background codes (10Xm range: 101=bright red, 102=bright green, 104=bright blue) with bold bright white text. The comment in `display.cjs` acknowledges these are "bright background variants for better readability with white text."

**Problem:** Bright backgrounds (10Xm) create poor contrast with white text on many terminals:
- Bright green (102m) background with white text is notoriously low-contrast -- nearly unreadable on default terminal palettes.
- Bright blue (104m) with white text is readable on most themes but problematic on Solarized Light.
- Bright red (101m) with white text works reasonably well.

**Industry convention:** Established CLI tools (source: [Choosing readable ANSI colors for CLIs](https://trentm.com/2024/09/choosing-readable-ansi-colors-for-clis.html)) prefer:
- **Dark/normal backgrounds** (40-47 range) instead of bright backgrounds (100-107 range) when pairing with white or bright white text. Dark blue (44m), dark red (41m), dark green (42m) provide significantly better contrast with white text.
- **16-color palette only** for maximum terminal compatibility -- RAPID already follows this.
- **NO_COLOR environment variable support** -- the de facto standard ([no-color.org](https://no-color.org/)). RAPID explicitly does NOT check NO_COLOR, as noted in display.cjs line 7. This should be addressed for open-sourcing.

**Recommended color mapping (dark backgrounds for white text readability):**
| Stage group | Current (bright) | Recommended (dark) | ANSI code |
|-------------|-------------------|---------------------|-----------|
| Planning    | bright blue (104m) | dark blue (44m)    | `\x1b[44m` |
| Execution   | bright green (102m) | dark green (42m) or dark cyan (46m) | `\x1b[42m` or `\x1b[46m` |
| Review      | bright red (101m)  | dark red (41m)      | `\x1b[41m` |
| Audit       | bright red (101m)  | dark magenta (45m)  | `\x1b[45m` |

Dark cyan (46m) is worth considering for execution stages as dark green (42m) can still be hard to read on some themes; cyan provides better contrast universally.

### NO_COLOR Standard

The `NO_COLOR` environment variable (see [no-color.org](https://no-color.org/)) is supported by 500+ CLI tools. When set, tools must not output ANSI escape codes. RAPID should check `process.env.NO_COLOR` in `display.cjs` before emitting escape codes. This is important for:
- CI/CD environments where color codes corrupt log output
- Accessibility tools (screen readers) that cannot interpret ANSI codes
- Users who pipe RAPID output to files

### Keyboard and Screen Reader Considerations

Not directly applicable -- RAPID runs inside Claude Code's terminal, not a GUI. However, the text-based output should avoid relying solely on color to convey meaning. The current banner includes both the stage verb (e.g., "REVIEWING") and color, which is correct.

## User Expectations

### Review Pipeline State Tracking

**Current behavior:** The review pipeline (scope -> unit-test -> bug-hunt -> uat) has no persistent state tracking. Each sub-stage checks for its prerequisite artifact (REVIEW-SCOPE.md) but does not record whether it has been completed. If a user runs `/rapid:review` (scoping) and then `/rapid:bug-hunt`, the bug-hunt skill does not know whether unit-test was already run. The user must manually remember which review stages they have completed.

**User expectation:** Users of multi-stage pipelines expect the system to track which stages are done (source: CI/CD pipeline conventions, GitHub Actions job status, Nx task status). When running `/rapid:uat`, the system should be able to say "Bug hunt has not been run yet. Continue anyway?" or at minimum display which review stages are complete.

**Current artifacts that could serve as state:**
- `REVIEW-SCOPE.md` -- presence indicates scoping is done
- `REVIEW-UNIT.md` -- presence indicates unit testing is done
- `REVIEW-BUGS.md` -- presence indicates bug hunting is done
- `REVIEW-UAT.md` -- presence indicates UAT is done

The artifact-based detection pattern already exists in execute-set (WAVE-COMPLETE.md markers). Extending this same pattern to review stages would be consistent and low-friction. The review state does not need to be in STATE.json -- it can be purely artifact-based, matching the existing execution re-entry pattern.

### Default Behaviors Users Expect

- `/rapid:review` should show which review sub-stages have been completed and suggest the next one.
- `/rapid:status` should show review progress per set (not just set lifecycle status).
- Running a review sub-stage that has already been done should warn and allow re-running (idempotent).
- The completion banner of each review stage should list remaining stages.

### Terminology: User Language vs Technical Terms

| User says | RAPID technical term | Notes |
|-----------|---------------------|-------|
| "task" | "job" | Users may say "task" -- docs should note both |
| "branch" | "set" | Users think in git branches; RAPID abstracts this |
| "test" | "unit-test / bug-hunt / uat" | Users expect "test" to mean all review stages |
| "step" | "wave" | Sequential ordering concept |
| "project" | "milestone" | Version-scoped work grouping |

### Mental Models from Competing Products

Users coming from CI/CD tools expect:
- **Pipeline visualization** -- a list of stages with pass/fail/pending status
- **Re-run from failure** -- ability to re-run a failed stage without re-running predecessors
- **Skip/force** -- ability to skip optional stages

RAPID's review pipeline matches this model but currently lacks the visualization (which review stages are done) and the skip semantics.

### Onboarding and Learning Curve

For open-source adoption, new users need:
1. A **30-second pitch** -- what RAPID does and why they should use it
2. A **5-minute quickstart** -- install and run through one complete set lifecycle
3. A **concept guide** -- explaining sets, waves, jobs, worktrees, contracts
4. **Progressive disclosure** -- start with `/rapid:init` and `/rapid:quick`, introduce full set lifecycle later

The current docs assume familiarity with RAPID concepts. The `/rapid:help` command is excellent for existing users but insufficient for onboarding.

## Recommendations

Ordered by priority for the v4.4.0 Polish & Documentation milestone:

1. **Switch banners to dark ANSI backgrounds (40-47 range):** Replace bright backgrounds (10Xm) with dark equivalents in `display.cjs` STAGE_BG map. Dark blue (44m), dark cyan (46m), dark red (41m) provide better white-text contrast across terminal themes. -- Priority: **high** (directly addresses milestone goal)

2. **Add NO_COLOR support to display.cjs:** Check `process.env.NO_COLOR` before emitting ANSI codes. Return plain-text banners when set. This is table-stakes for open-source CLI tools. -- Priority: **high** (open-sourcing readiness)

3. **Track review pipeline state via artifact presence:** Detect REVIEW-SCOPE.md, REVIEW-UNIT.md, REVIEW-BUGS.md, REVIEW-UAT.md to determine which review stages have been completed. Display this in the completion banner of `/rapid:review` and each sub-stage. -- Priority: **high** (directly addresses milestone goal for review state management)

4. **Add review stage awareness to `/rapid:status`:** Show per-set review progress (scoped/unit-tested/bug-hunted/uat-complete) alongside the existing set status column. -- Priority: **medium** (enhances the status dashboard)

5. **Write a README.md with quickstart:** Create a top-level README with: one-paragraph pitch, install command, canonical workflow diagram (from `/rapid:help`), and links to docs/. This is the single most important doc for open-sourcing. -- Priority: **high** (directly addresses milestone goal)

6. **Write a concept/glossary guide:** New doc explaining RAPID terminology (set, wave, job, worktree, contract, solo mode, milestone) with short definitions and how they relate to familiar concepts (branches, tasks, projects). -- Priority: **medium** (onboarding improvement)

7. **Document the full canonical workflow:** Create a walkthrough guide showing install -> init -> branding -> scaffold -> start-set -> discuss-set -> plan-set -> execute-set -> review -> merge -> audit-version -> new-version with expected output at each step. Also cover /quick, /bug-fix, /add-set as auxiliary commands. -- Priority: **high** (directly addresses milestone goal)

8. **Add review stage suggestions to completion banners:** When a review sub-stage completes, show which stages remain (e.g., "Unit testing complete. Remaining: /rapid:bug-hunt, /rapid:uat"). -- Priority: **medium** (UX polish)

9. **Standardize warning output:** Add a `[RAPID WARNING]` prefix convention for non-fatal messages, consistent with the existing `[RAPID ERROR]` pattern. -- Priority: **low** (polish)

10. **Consider a `--run-all` flag for review:** Allow `/rapid:review 1 --run-all` to automatically chain scope -> unit-test -> bug-hunt -> uat without manual invocation of each stage. -- Priority: **low** (future enhancement, not required for v4.4.0)
