# Research Summary -- v4.1.0 (Polish & Fixes)

## Project Profile
- **Type:** brownfield CLI plugin + web dashboard (monorepo)
- **Stack:** Node.js (CommonJS) + FastAPI (Python 3.12+) + React 19 (Vite 8 / Tailwind 4)
- **Scale:** medium (5 features, 7 bug fixes, ~51 sub-features across 12 work items)
- **Key Constraint:** All changes are incremental refinements to existing skill SKILL.md files and library modules -- no new architectural patterns required. The existing 5-layer modular architecture (Skills > Agents > CLI > Library > Storage) cleanly supports the scattered enhancement targets.

## Unified Findings

### Stack and Infrastructure

- **Node.js 25 (non-LTS) is the development runtime**, but the project enforces only Node 18+ minimum. Node 25 EOL is ~June 2026. Not a deployment risk but worth testing against Node 24 LTS for release validation. (source: STACK)
- **`node:test` built-in runner is stable and production-ready** since Node 24+. All test features used by RAPID (describe, it, before, after, mock) are stable APIs. However, `node:test` is experimental in Node 18.x -- glob support for `--test` flag stabilized only in Node 20. (sources: STACK, PITFALLS)
- **Zod 3.25.76 bundles Zod 4 internals**, doubling package size and potentially changing `.default()` / `.catch()` behavior on optional fields. The `ReviewIssue` schema uses `.default(false)` on `autoFixAttempted` and `.default('open')` on `status`, which are vulnerable. The `^3.25.76` caret range could pull in patches that further change this behavior. (sources: STACK, PITFALLS)
- **Zod 4 upgrade is NOT recommended for this milestone** due to CJS compatibility risk (`require('zod/v4')` has reported issues) and non-trivial migration effort across state-schemas.cjs, state-machine.cjs, and returns.cjs. (source: STACK)
- **proper-lockfile 4.1.2 is unmaintained** (5+ years, no updates) but functional. Uses mkdir-based atomic locking. RAPID wraps it in lock.cjs, making replacement straightforward if needed. The `stale` default of 0 (never stale) can cause permanent locks from crashed processes -- RAPID already mitigates by setting `stale: 300000`. (sources: STACK, PITFALLS)
- **cytoscape-dagre 2.5.0 is low-maintenance** (3+ years since last release) but functional for DAG visualization. Alternative layout plugins (cytoscape-elk, cytoscape-fcose) exist as fallbacks. (source: STACK)
- **All Python and frontend dependencies are current** with no blocking compatibility issues. FastAPI 0.135 + Python 3.14, Vite 8 + React 19, Tailwind CSS 4 all work together. (source: STACK)
- **No CI/CD pipeline exists**. No `.github/workflows/` or equivalent. Not needed for this polish milestone. (source: STACK)
- **No linter configured for the core CLI** (JavaScript/CJS). Web backend has Ruff; frontend has TypeScript compiler. (source: STACK)
- **New config fields needed:** `testFramework` / `testRunner` (from #31), criteria encoding categories (from #2), DAG enforcement mode (from #3). These should go in `.planning/config.json` (per-project), not the RAPID root config. (source: OVERSIGHTS)

### Architecture Direction

- **Recommended pattern:** Incremental refinement of existing modular skill architecture. No new patterns needed. The existing 5-layer system has been stable from v1.0 through v4.0.0, demonstrating that skill-level changes can be made independently. (source: ARCHITECTURE)
- **No structural changes required.** All work fits within the existing project layout. No new directories or modules are needed. The only new artifact is `.planning/DEFERRED.md` for deferred decisions. (sources: ARCHITECTURE, FEATURES)
- **Module boundaries are clean.** Each skill (branding, discuss-set, init, unit-test, new-version) can be modified independently without cross-skill coupling. Key files per module are well-defined: init touches `skills/init/SKILL.md` + `src/lib/init.cjs`, discuss-set touches only its SKILL.md, review touches `src/commands/review.cjs` + `src/lib/review.cjs`, unit-test touches its SKILL.md + role modules. (sources: ARCHITECTURE, FEATURES)
- **State schema is extensible** via `.passthrough()` on all Zod object types. Adding `deps` on SetState or `testFramework` in config does not require schema migration. This is critical for the DAG dependency resolution (Feature 3) and framework-agnostic testing (Bug #31). (source: ARCHITECTURE)
- **DAG infrastructure is feature-complete.** `dag.cjs` provides toposort, wave assignment, v1/v2 DAG creation, validation, and loading. `add-set.cjs` already accepts `--deps` and calls `recalculateDAG()`. The web dashboard has `KnowledgeGraphPage.tsx` using cytoscape-dagre. The gap is wiring -- no skill currently generates DAG.json during the init/new-version workflow. (sources: ARCHITECTURE, OVERSIGHTS)
- **Test framework detection already exists** in `context.cjs` (`detectCodebase()` detects jest, vitest, mocha, pytest). This provides a foundation for making the unit-test skill framework-agnostic. The detection must extend to: package.json `scripts.test` for runner identification, pyproject.toml for pytest, Cargo.toml for cargo test, go.mod for go test. (sources: ARCHITECTURE, STACK)
- **Data flow is well-defined.** State mutations go through `withStateTransaction()` (lock + read + validate + mutate + validate + write + unlock). Skills write artifact files directly via Write tool. Zod validation on every state write ensures consistency. (source: ARCHITECTURE)
- **No new API endpoints needed for the CLI layer.** All changes are within existing commands. The web backend may need a minor enhancement to serve DAG data if the existing endpoint is not functional. (source: ARCHITECTURE)

### User Experience Direction

- **RAPID operates in the AI-assisted parallel development orchestration domain.** Key conventions: spec-driven development, DAG as primary orchestration view, human-in-the-loop approval gates, structured decision capture via AskUserQuestion. (source: UX)
- **Discuss-set questions need progressive disclosure:** title as summary, 3-5 sentence context for depth, pros/cons table for full analysis. The GSD discuss-phase pattern is the benchmark. Current 1-2 sentence context is insufficient for informed architectural decisions. (sources: UX, OVERSIGHTS, FEATURES)
- **Deferred decisions mechanism is needed.** Current CONTEXT.md has an inert `<deferred>` section with no downstream consumers. No CLI commands exist for managing deferred items. GSD treats deferred decisions as first-class. Recommended: project-level `.planning/DEFERRED.md` with `deferred list` / `deferred promote` CLI commands. (sources: UX, OVERSIGHTS, FEATURES)
- **CLI argument consistency is critical for trust.** The review log-issue mismatch (Issue #27) violates user expectations established by other RAPID commands. Users expect `--flag` syntax to work as documented. (sources: UX, PITFALLS, OVERSIGHTS)
- **DAG should be a first-class navigation element**, not buried in a secondary page. Nodes should be colored by status AND display status labels/shapes for color-blind accessibility. Edge labels should say "depends on" to match user mental models. (source: UX)
- **Numeric shorthand pattern is excellent UX** (`/rapid:start-set 1`). Should extend to any new set-referencing command. (source: UX)
- **Branding skill needs codebase awareness** to avoid asking visual UI/UX questions for CLI-only projects. Detection should use `context.cjs` detectCodebase(). (sources: UX, FEATURES, OVERSIGHTS)
- **Progress breadcrumb should appear at skill START**, not just at completion/errors. Users need to know where they are in the lifecycle at all times. (source: UX)
- **Web dashboard accessibility is above average** (role attributes, tabIndex, aria-labels, keyboard navigation). DAG node status colors need shape/label differentiation for color-blind users. (source: UX)

## Contradictions

### Contradiction: RAPID_ROOT -- "Unused" vs "Actively Used"
- **FEATURES says (Bug #3 description):** "RAPID_ROOT was used in v1.0.0 but should have been deprecated in v2.0.0+. Currently computed in every skill preamble but never used meaningfully."
- **OVERSIGHTS says (concern #10):** "RAPID_ROOT is not unused -- it is actively used in every skill's environment preamble as a transient shell variable for .env loading."
- **Resolution:** OVERSIGHTS is correct. RAPID_ROOT is used -- it serves as a local variable to derive the `.env` path. The removal should replace the two-line pattern (assign RAPID_ROOT, then use it for .env) with a single-line equivalent that inlines the path derivation. It should NOT be treated as a blind deletion. Both sources agree on the replacement pattern: `if [ -f "${CLAUDE_SKILL_DIR}/../../.env" ]`.

### Contradiction: DAG Web Dashboard -- "Already Has Infrastructure" vs "GraphPage is a Stub"
- **FEATURES says:** "The web dashboard already has the infrastructure: `KnowledgeGraphPage.tsx` renders a cytoscape.js DAG."
- **OVERSIGHTS says:** "The web dashboard's GraphPage is a placeholder: `<p>Dependency graph visualization will appear here</p>` (GraphPage.tsx line 3). No backend API endpoint serves DAG data to the frontend."
- **Resolution:** Both are partially correct. The `KnowledgeGraphPage.tsx` exists and has cytoscape-dagre integration code (with NODE_COLORS, layout configuration), but there may be a separate `GraphPage.tsx` that is a stub. The OVERSIGHTS finding about missing backend API endpoints and no DAG.json generation in the workflow is the more important issue -- the visualization infrastructure exists but the data pipeline to feed it does not.

### Contradiction: Frontend DAG Library Choice
- **OVERSIGHTS says (concern #14):** "Graph library options not yet chosen" and recommends starting with `dagre` + custom SVG for a lightweight read-only approach.
- **STACK says:** cytoscape.js 3.33.1 + cytoscape-dagre 2.5.0 are already installed in the frontend.
- **Resolution:** STACK is correct that cytoscape-dagre is already integrated. OVERSIGHTS' recommendation to evaluate new libraries is unnecessary -- the existing cytoscape-dagre setup should be used. No new graph library is needed.

### Contradiction: Discuss-Set Gray Area Selection Mechanism
- **FEATURES says:** "With 8 gray areas and 4 options per AskUserQuestion, the user gets 2 selection prompts (Step 5) to pick which areas to discuss."
- **PITFALLS says:** "AskUserQuestion actually supports 2-4 options per question, and multi-select mode. Confusing 'number of gray areas' with 'number of options per question' will produce broken prompts."
- **OVERSIGHTS says:** "Redesign the selection mechanism. Instead of presenting gray areas as AskUserQuestion options, present a numbered list and ask 'Which areas would you like to discuss?'"
- **Resolution:** PITFALLS and OVERSIGHTS provide the better approach. Splitting gray areas across multiple AskUserQuestion calls fragments the user experience. The numbered-list approach (present all gray areas in the message body, then use AskUserQuestion for action selection: "Discuss selected" / "Discuss all" / "Skip all" / "Pick differently") decouples gray area count from the 4-option limit and is more robust.

## Recommended Approach

This milestone is a "polish and fix" release targeting an existing stable v4.0.0 codebase. The architecture is sound -- the 5-layer system (Skills > Agents > CLI > Library > Storage) with skill-level modularity cleanly supports all planned changes. No new architectural patterns are needed. Every work item maps to an existing module or skill file, and the state schema is forward-compatible via `.passthrough()`.

The critical path begins with three items that must be resolved before dependent work can proceed: (1) the init REQUIREMENTS.md overwrite bug, which is a confirmed data-loss issue on the most common code path, (2) the discuss-set gray area selection mechanism design, which constrains how the entire discuss phase overhaul is implemented, and (3) the criteria encoding schema design, which must be agreed upon before the init enhancement and verifier integration can be built.

The largest risk cluster is around the discuss-set overhaul (Feature 4). It touches AskUserQuestion's 4-option constraint (a hard platform limit), introduces a new DEFERRED.md artifact with no existing infrastructure, changes the CONTEXT.md schema (which must stay compatible with the --skip auto-context path), and significantly expands the discussion duration. The recommended mitigation is to decouple gray area count from AskUserQuestion option count by using a numbered-list presentation with meta-action options, design the CONTEXT.md schema changes as a single specification referenced by both interactive and --skip paths, and implement deferred decisions as a simple project-level file before wiring it into downstream skills.

The DAG re-addition (Feature 3) is architecturally the lowest-risk feature because the infrastructure is already complete -- `dag.cjs` provides full toposort, wave assignment, and validation, and the web dashboard has cytoscape-dagre integrated. The only work is wiring: having the roadmapper output set dependencies, generating DAG.json after roadmap approval, and ensuring the command center reads DAG.json for visualization. The main pitfall is DAG/STATE.json desync, which should be addressed by making DAG updates atomic with state mutations.

The unit-test improvements (Bug #29 + Bug #31) should be treated as a combined effort. Removing the 5-group limit is mechanically simple (one line change) but carries a token budget risk that requires dynamic batching. Making the test runner framework-agnostic is the most complex bug fix in the milestone, requiring detection during init, config storage, skill parameterization, and role module updates. However, the detection foundation already exists in `context.cjs`, and the config schema is extensible. The recommended approach is: remove the limit first, add detection and config second, parameterize the skill third.

The trivial fixes (Bug #2 clone URL, Bug #3 RAPID_ROOT sweep) should be shipped first as a quick hygiene pass. They carry zero risk and improve codebase quality. The RAPID_ROOT sweep requires care -- the two-line env preamble pattern must be replaced atomically (both the variable assignment and the .env loading line that uses it), and archived planning files should be left untouched.
