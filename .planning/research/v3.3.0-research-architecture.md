# Architecture Research -- Developer Experience Milestone (v3.3.0)

## Current Architecture

### Existing Pattern: Layered Monolith with Agent Pipeline

RAPID's architecture is a five-layer stack: Skill Layer (24 SKILL.md files) -> Agent Layer (26 generated agents) -> CLI Layer (rapid-tools.cjs, 2717 lines) -> Library Layer (21 modules in src/lib/) -> Storage Layer (.planning/, .rapid-worktrees/, git branches). This layered design is fundamentally sound for the project's scope. The key strengths are:

- **Transaction pattern for state mutations** -- `withStateTransaction()` provides lock-acquire-read-validate-mutate-validate-write-release semantics. This pattern is well-proven and should be extended (not replaced) for MERGE-STATE and registry mutations.
- **Worktree isolation** -- Physical file isolation via git worktrees eliminates concurrent file access conflicts. This is the correct primitive for parallel development.
- **Contract-first planning** -- Interface contracts (`CONTRACT.json`) define boundaries before implementation. Validated at three gates: after planning, during execution, before merge.
- **Structured returns** -- The `RAPID:RETURN` protocol gives agents a typed communication channel with machine-parseable status (COMPLETE/CHECKPOINT/BLOCKED).

The primary architectural weaknesses this milestone must address:

1. **Monolithic CLI backbone** -- `rapid-tools.cjs` at 2717 lines houses all 18+ command handlers with manual routing and argument parsing. This is the single biggest maintainability bottleneck.
2. **Inconsistent mutation patterns** -- State uses `withStateTransaction()`, but MERGE-STATE uses three different write patterns, and the registry uses `registryUpdate()` (a different callback pattern). The `execute update-phase` command bypasses transactions entirely.
3. **Path resolution ambiguity** -- Mixed use of `path.resolve(cwd, ...)` (7 occurrences) and `path.join(cwd, ...)` (15+ occurrences) in the CLI, plus `.planning/waves/` vs `.planning/sets/` path confusion left over from v2-to-v3 migration.
4. **Tightly coupled review pipeline** -- The monolithic `/review` skill (1023 lines) orchestrates scoper, unit-tester, bug-hunter, devils-advocate, judge, bugfix, and UAT agents in a single session with no ability to re-enter individual stages.

### Where Existing Architecture Supports the Milestone

- **Solo Mode (item 1):** The worktree lifecycle is already factored into `worktree.cjs` with clear create/register/cleanup boundaries. Solo mode can intercept at `createWorktree()` and `registryUpdate()` without restructuring the lifecycle.
- **Review Decomposition (item 3):** The review pipeline already uses discrete agents (scoper, unit-tester, etc.) spawned via the Agent tool. Each stage writes to a separate artifact file (REVIEW-UNIT.md, REVIEW-BUGS.md, REVIEW-UAT.md). The decomposition is already implied by the data flow; the skill just needs splitting.
- **Version Migration (item 5):** The `version.cjs` module already provides `getVersion()` and `versionCheck()`. The `state-schemas.cjs` module uses Zod for schema validation. Migration can leverage Zod's `.safeParse()` to detect schema shape and route to the correct migration path.

### Where Existing Architecture Needs Extension or Modification

- **CLI Split (item 15):** The monolithic `rapid-tools.cjs` must be decomposed into per-command handler files. This is a prerequisite for clean parallel development on multiple features.
- **Mutation Consistency (items 9, 10):** MERGE-STATE needs a `withMergeStateTransaction()` wrapper mirroring `withStateTransaction()`. The `execute update-phase` bypass must be wrapped in the transaction pattern.
- **Scaffold Command (item 2):** No scaffolding infrastructure exists. The `init.cjs` module handles `.planning/` directory creation but has no concept of project-type-aware code scaffolding.
- **Compaction (item 4):** No compaction infrastructure exists. This requires a new module for context size estimation and a RAPID-aware summarization strategy.
- **Argument Parsing (item 16):** All 10+ occurrences of `args.indexOf('--flag')` need replacement with a consistent parser. Node.js 18+ provides `util.parseArgs()` as a built-in (stable since Node.js 20, available in 18.3+).

## Recommended Architecture Pattern

- **Pattern:** Modular monolith with command-handler decomposition
- **Rationale:** RAPID is a CLI tool, not a service. It does not need microservices, event buses, or plugin registries. The current layered architecture is correct in shape but has accumulated organic coupling in the CLI layer. The decomposition should split the CLI into independent command handlers while preserving the existing library layer as shared infrastructure. This is the simplest intervention that resolves the maintainability bottleneck without over-engineering.
- **Evidence:** The existing module structure in `src/lib/` already demonstrates that the project benefits from single-responsibility modules (e.g., `merge.cjs` at 1982 lines is a natural unit; `state-machine.cjs` at 449 lines is well-scoped). Extending this pattern to the CLI layer aligns with the codebase's existing conventions.

### Rationale for Structure Decisions

1. **`src/commands/` directory** -- Each command handler becomes a separate file. The `rapid-tools.cjs` router reduces to a `switch` statement that `require()`s the appropriate handler. This enables per-command testing, per-command git diffs, and parallel development without merge conflicts on the monolithic file.

2. **New library modules** -- `args.cjs` (argument parsing), `errors.cjs` (error standardization), `scaffold.cjs` (project scaffolding), `migrate.cjs` (version migration), and `compaction.cjs` (context management) follow the existing module template pattern: `'use strict'`, named function exports, co-located tests.

3. **Review skill split** -- The existing `/review` SKILL.md (1023 lines) splits into four independent skills. Each skill reads from the shared artifact produced by `/review` (REVIEW-SCOPE.md). The existing `review.cjs` library module does not need splitting -- its functions are already concern-scoped (scoping functions, issue logging functions, summary generation).

4. **Deprecated skill removal** -- Six deprecated skills (`discuss`, `execute`, `plan`, `set-init`, `wave-plan`, `new-milestone`) serve only as redirects. Removing them reduces the skill surface from 24 to 18 (plus 4 new = 22 total) and eliminates maintenance surface.

## Module Organization

### Module: args (NEW)

- **Responsibility:** Parse CLI arguments consistently across all command handlers
- **Public interface:** `parseArgs(args, schema)` where schema defines expected flags with types
- **Dependencies:** `node:util` (Node.js built-in `util.parseArgs()`)
- **Data ownership:** None (stateless utility)
- **Design note:** Should wrap Node.js built-in `util.parseArgs()` (stable since Node.js 20, available since 18.3). Supports `--flag value` and `--flag=value` syntax. Returns typed object. Replaces all 10+ `args.indexOf('--flag')` patterns in the CLI.

### Module: errors (NEW)

- **Responsibility:** Standardize error output format across all command handlers
- **Public interface:** `exitWithError(msg, code)`, `formatError(msg)`, `validateStdinJson(zodSchema)`
- **Dependencies:** `zod` (for stdin validation), `core.cjs` (for `error()` function)
- **Data ownership:** None (stateless utility)

### Module: scaffold (NEW)

- **Responsibility:** Generate project-type-aware foundation code (layout shells, test harnesses, shared types)
- **Public interface:** `detectProjectType(cwd)`, `generateScaffold(cwd, projectType, options)`, `scaffoldManifest(projectType)`
- **Dependencies:** `context.cjs` (language/framework detection), `init.cjs` (for `.planning/` scaffold integration)
- **Data ownership:** Owns scaffold templates and project-type detection heuristics

### Module: migrate (NEW)

- **Responsibility:** Detect RAPID version from state shape, run ordered migration steps, produce migration reports
- **Public interface:** `detectVersion(cwd)`, `planMigration(fromVersion, toVersion)`, `executeMigration(cwd, plan, options)`, `dryRun(cwd)`
- **Dependencies:** `state-machine.cjs` (state read/write), `state-schemas.cjs` (schema validation), `version.cjs` (version comparison), `worktree.cjs` (registry migration)
- **Data ownership:** Owns migration step definitions (versioned functions) and backup management

### Module: compaction (NEW)

- **Responsibility:** Estimate context size, identify safe-to-compact segments, produce structured summaries
- **Public interface:** `estimateTokens(text)`, `identifyCompactable(contextSegments)`, `compactSegments(segments, targetTokens)`, `generateSummary(compactedData)`
- **Dependencies:** `returns.cjs` (parse RAPID:RETURN for completed agent results)
- **Data ownership:** None (transforms in-flight context data)
- **Design note:** This is the highest-risk module in the milestone. Context compaction interacts with Claude Code's internal context management in ways that are difficult to predict. The module should be designed for conservative compaction (prefer keeping too much over losing critical context) and should always preserve artifact file paths as recovery pointers.

### Module: state-machine (MODIFIED)

- **Responsibility:** Existing: state CRUD, transactions, locking. New: solo mode flag awareness.
- **Public interface:** Existing API unchanged. New: `isSoloMode(cwd)` reads config to determine if solo mode is active.
- **Dependencies:** Unchanged
- **Data ownership:** Owns `STATE.json`. Solo mode flag is stored in `.planning/config.json` (not STATE.json, since it is a project-level configuration, not per-milestone state).

### Module: merge (MODIFIED)

- **Responsibility:** Existing: 5-level conflict detection, 4-tier resolution, merge execution. New: unified MERGE-STATE transaction pattern.
- **Public interface:** Existing API plus new `withMergeStateTransaction(cwd, setId, mutationFn)` that mirrors `withStateTransaction()`.
- **Dependencies:** Unchanged
- **Data ownership:** Owns MERGE-STATE.json per set

### Module: worktree (MODIFIED)

- **Responsibility:** Existing: git worktree lifecycle, registry. Modified: rename `loadRegistry()` to `readRegistry()` (returns frozen copy) and `registryUpdate()` to `withRegistryUpdate()` (mutation callback). Add solo mode bypass for worktree creation.
- **Public interface:** `readRegistry(cwd)` (read-only, frozen), `withRegistryUpdate(cwd, updateFn)` (mutation callback). `createWorktree()` gains `{ solo: boolean }` option that skips git worktree creation and registers a virtual entry pointing to the project root.
- **Dependencies:** Unchanged
- **Data ownership:** Owns REGISTRY.json

## Data Flow

### Request Lifecycle (CLI Command)

```
1. User invokes: node rapid-tools.cjs <command> <subcommand> <args>
2. Router (rapid-tools.cjs):
   - Parse command/subcommand
   - Commands not needing project root: prereqs, init, context, display
   - All others: findProjectRoot() -> cwd
   - require() handler from src/commands/<command>.cjs
   - Delegate: handler(cwd, subcommand, args)
3. Handler (src/commands/<command>.cjs):
   - Parse args via args.cjs utility
   - Validate preconditions (state, registry, etc.)
   - Call library functions from src/lib/
   - Output JSON to stdout, status to stderr
   - Exit with appropriate code
4. Library (src/lib/<module>.cjs):
   - Business logic, state mutations, file I/O
   - Transaction wrappers ensure atomicity
   - Zod validation at mutation boundaries
```

### Solo Mode Data Flow

```
Standard path:
  /start-set -> createWorktree() -> git worktree add -> register in REGISTRY.json
  /execute-set -> work in .rapid-worktrees/<set>/
  /merge -> git merge --no-ff -> update STATE.json

Solo mode path:
  /start-set -> isSoloMode(cwd) -> skip createWorktree() -> register virtual entry in REGISTRY.json (path = ".")
  /execute-set -> work directly in project root on main branch
  /merge -> no-op with message ("Solo mode: commits already on main")
```

### Review Pipeline Data Flow (After Decomposition)

```
/review <set>
  -> review scope <set>
  -> writes REVIEW-SCOPE.md
  -> exits

/unit-test <set>
  -> reads REVIEW-SCOPE.md (error if missing: "Run /review first")
  -> spawns rapid-unit-tester agents (per concern group or per chunk)
  -> writes REVIEW-UNIT.md
  -> logs issues via review log-issue
  -> exits

/bug-hunt <set>
  -> reads REVIEW-SCOPE.md (error if missing: "Run /review first")
  -> spawns rapid-bug-hunter + rapid-devils-advocate + rapid-judge pipeline
  -> spawns rapid-bugfix for accepted bugs (up to 3 cycles)
  -> writes REVIEW-BUGS.md
  -> logs issues via review log-issue
  -> exits

/uat <set>
  -> reads REVIEW-SCOPE.md (error if missing: "Run /review first")
  -> spawns rapid-uat agent
  -> writes REVIEW-UAT.md
  -> logs issues via review log-issue
  -> exits
```

### Migration Data Flow

```
/migrate
  -> detectVersion(cwd): read STATE.json shape + .planning/ structure -> infer version
  -> planMigration(detected, current): produce ordered step list [v2->v3, v3->v3.1, ...]
  -> backup: copy .planning/ to .planning/.pre-migrate-backup/
  -> for each step: apply schema migration, relocate artifacts, update config
  -> build-agents: regenerate agent prompts
  -> stamp new version in STATE.json
  -> produce MIGRATION-REPORT.md
```

### State Management

- **Primary state:** `.planning/STATE.json` -- ProjectState Zod schema, protected by `withStateTransaction()`, atomic writes via tmp+rename
- **Merge state:** `.planning/sets/<setId>/MERGE-STATE.json` -- MergeState Zod schema, currently inconsistent write patterns (three different approaches). Recommendation: unify under `withMergeStateTransaction()`.
- **Registry state:** `.planning/worktrees/REGISTRY.json` -- worktree tracking, protected by `registryUpdate()` (lock-based callback). Recommendation: rename to `withRegistryUpdate()` for naming consistency.
- **Config state:** `config.json` (project root) and `.planning/config.json` -- runtime configuration (lock timeout, agent size warning). Solo mode flag should be stored in `.planning/config.json` as `"mode": "solo"` or `"mode": "team"`.
- **Consistency guarantee:** All state writes are protected by file locks (`proper-lockfile`). Locks use the process PID and auto-clean stale locks. Lock timeout is 300 seconds (configurable).

### Argument Parsing Standardization

The new `args.cjs` module should wrap `util.parseArgs()` from Node.js built-in `node:util`:

```javascript
const { parseArgs } = require('node:util');

function rapidParseArgs(args, options) {
  // options: { flags: { name: { type: 'boolean'|'string', short: 'n' } } }
  const result = parseArgs({
    args,
    options: options.flags,
    allowPositionals: true,
    strict: false,
  });
  return {
    flags: result.values,
    positionals: result.positionals,
  };
}
```

This eliminates the 10+ occurrences of `args.indexOf('--flag')` with off-by-one risk and adds `--flag=value` support for free.

### Error Response Standardization

```javascript
function exitWithError(msg, code = 1) {
  process.stdout.write(JSON.stringify({ error: msg }) + '\n');
  process.stderr.write(`[RAPID ERROR] ${msg}\n`);
  process.exit(code);
}
```

Replaces the current three error patterns:
1. `error()` + `process.exit(1)`
2. `{ error: msg }` JSON output + exit
3. Throw and let caller handle

## Integration Points

| External System | Protocol | Purpose | Error Handling |
|----------------|----------|---------|---------------|
| Git | CLI (execSync/execFileSync) | Worktree create/list/remove, branch management, merge --no-ff | Structured `{ ok, stdout, exitCode, stderr }` via `gitExec()` |
| Claude Code Plugin API | SKILL.md + Agent definitions | Skill invocation, agent spawning via Agent tool | RAPID:RETURN protocol with BLOCKED status |
| File system | Node.js `fs` | STATE.json, REGISTRY.json, MERGE-STATE.json, artifacts | Lock-protected transactions, atomic writes via tmp+rename |
| proper-lockfile | npm package | File-based distributed locking | Auto-cleanup of stale locks (PID check), configurable timeout |
| Zod | npm package | Schema validation for state, merge-state, review issues | `safeParse()` for detection, `parse()` for enforcement (throws on invalid) |
| Ajv | npm package | JSON Schema validation for CONTRACT.json | Compile-time schema validation with format extensions |

No new external dependencies are recommended for this milestone. The `util.parseArgs()` function is a Node.js built-in (available since 18.3, stable since 20.0). All other new modules use only existing dependencies.

## Scaling Considerations

### Context Token Budget

The context-aware compaction module (item 4) is the most architecturally significant scaling concern. Key design decisions:

- **Target:** Keep working context below ~120k tokens during orchestration
- **Trigger points:** Between waves during execute-set, between review cycles, after agent returns are processed
- **Safe to compact:** Completed wave results (keep summary, discard full output), resolved discussions, old agent returns, superseded plan versions
- **Must preserve:** Current task context, active contracts, unresolved blockers, in-progress wave data, error context
- **Recovery path:** Artifact files on disk serve as source of truth. Compacted summaries include file path references for re-reading.

### Risk of Compaction

The compaction module fights against Claude Code's built-in context management. The two systems could interact poorly:
- Claude Code may compact what RAPID considers critical, or vice versa
- RAPID's compaction may produce summaries that Claude Code then further compacts, losing fidelity
- A simpler alternative -- always read from disk artifacts rather than relying on conversation context -- may be more reliable

**Recommendation:** Implement compaction conservatively. Focus first on reducing context generation (shorter agent prompts, leaner returns) before adding active compaction. The compaction module should be opt-in via config and should log what it compacts for debugging.

### CLI Split Impact on Performance

Splitting `rapid-tools.cjs` into per-command handlers adds one `require()` call per invocation. Given that Node.js `require()` is synchronous and cached, the overhead is negligible (< 1ms). The benefit (per-command testing, smaller diffs, parallel development) vastly outweighs the cost.

### Solo Mode Performance

Solo mode eliminates git worktree creation/cleanup overhead and merge ceremony overhead. For solo developers, this removes approximately 3-5 seconds per set lifecycle (worktree creation) and the entire merge pipeline. The trade-off is loss of isolation -- concurrent set execution on the same branch is not safe in solo mode (but solo developers do not run concurrent sets).

## Architecture Risks

1. **Compaction/Claude Code interaction (HIGH):** Context compaction may interact unpredictably with Claude Code's internal context management. -- Mitigation: Make compaction opt-in, conservative, and logged. Provide a fallback to disk-based artifact reading. Consider this item as experimental/beta quality in v3.3.0.

2. **CLI split migration risk (MEDIUM):** Extracting 18 handlers from a 2717-line file requires careful testing to ensure no handler's behavior changes. -- Mitigation: Each handler extraction should be a mechanical move (copy function, add module.exports, require in router). Run the full test suite (`rapid-tools.test.cjs`) after each extraction. The test file has 1536+ lines of coverage.

3. **Solo mode / team mode switching (MEDIUM):** Switching from solo to team mid-project (retroactive worktree creation) is complex and error-prone. -- Mitigation: v3.3.0 should treat mode as immutable once set during init. Switching requires explicit migration. Document this limitation.

4. **Review decomposition shared state (LOW):** The four independent review skills (/review, /unit-test, /bug-hunt, /uat) share REVIEW-SCOPE.md as a coordination artifact. If the scope changes between skill invocations (e.g., new commits on the branch), results may be inconsistent. -- Mitigation: Include a scope hash in REVIEW-SCOPE.md; downstream skills warn if the hash does not match current branch state.

5. **Migration atomicity (LOW):** If a migration step fails midway, the project may be in a partially migrated state. -- Mitigation: The backup (`.planning/.pre-migrate-backup/`) provides rollback. Each migration step should be idempotent (safe to re-run).

6. **Path mismatch cleanup (LOW):** The `.planning/waves/` vs `.planning/sets/` path mismatch affects review artifacts, execute artifacts, and skill references. Cleanup touches multiple files across modules and skills. -- Mitigation: This is mechanical but wide-reaching. Search-and-replace with verification across all source and skill files.

7. **Deprecated skill removal (LOW):** Removing 6 deprecated skills may break muscle memory for users who learned the old command names. -- Mitigation: The "skill not found" error from Claude Code is sufficient. The old names are already marked deprecated with `disable-model-invocation: true`.

## Recommendations

Ordered by architectural priority and implementation dependency:

1. **CLI Split into src/commands/ (item 15):** Critical -- This is a prerequisite for clean parallel development on all other items. Split first, then implement features in the decomposed handlers. Mechanical extraction, low risk, high value.

2. **Argument Parsing Utility (item 16):** Critical -- Create `src/lib/args.cjs` wrapping `util.parseArgs()`. Apply across all handlers during the CLI split (natural pairing). Eliminates a class of off-by-one bugs.

3. **Error Response Standardization (item 11):** Critical -- Create `src/lib/errors.cjs` with `exitWithError()` and `validateStdinJson()`. Apply across all handlers during the CLI split (natural pairing).

4. **Resume Command Deduplication (item 8):** Critical -- Delete `execute resume` path, route both entry points to a single `resumeSet()` function in `execute.cjs`. This eliminates an active bug-farm during parallel execution.

5. **State Mutation Bypass Fix (item 9):** Critical -- Wrap `execute update-phase` in `withStateTransaction()`. One-line fix with high impact on data integrity.

6. **MERGE-STATE Transaction Wrapper (item 10):** High -- Create `withMergeStateTransaction()` in `merge.cjs`. Migrate all three write patterns to use it. Follows the proven `withStateTransaction()` pattern.

7. **Path Resolution Consistency (item 12):** High -- Audit all `path.join(cwd, ...)` vs `path.resolve(cwd, ...)` in the CLI and standardize on `path.resolve()`. Mechanical change, prevents bugs when cwd is relative.

8. **Review Artifact Path Fix (item 13):** High -- Migrate all `.planning/waves/` references to `.planning/sets/` across source files and skills. Eight occurrences in source, multiple in skills.

9. **Registry Naming Clarity (item 19):** Medium -- Rename `loadRegistry()` -> `readRegistry()` and `registryUpdate()` -> `withRegistryUpdate()`. Apply across all call sites.

10. **Agent Build Detection Fix (item 14):** Medium -- Remove comment-based `<!-- CORE: Hand-written agent` detection. Use only the `SKIP_GENERATION` list (already exists at line 692 of rapid-tools.cjs).

11. **Deprecated Skills Cleanup (item 18):** Medium -- Remove 6 deprecated skill directories. No code changes needed beyond directory deletion.

12. **Stdin JSON Validation (item 17):** Medium -- Create `validateStdinJson()` in `errors.cjs` using Zod. Apply to all stdin-reading commands: `plan create-set`, `state add-milestone`, `execute pause`, `review log-issue`.

13. **Solo Mode (item 1):** High -- Add `mode` field to `.planning/config.json`. Prompt during `/init`. Check in `worktree.cjs` (skip creation), `execute.cjs` (direct commit), merge handlers (no-op). Requires items 1-3 to be done first (clean command handlers to add solo checks to).

14. **Review Pipeline Decomposition (item 3):** High -- Split `/review` SKILL.md into `/review` (scope), `/unit-test`, `/bug-hunt`, `/uat`. Create 3 new skill directories. Each skill reads REVIEW-SCOPE.md as prerequisite. The existing `review.cjs` library needs no changes.

15. **Missing DEFINITION.md Bug (item 6):** Medium -- Investigate path resolution in `start-set` skill and `worktree.cjs` `generateScopedClaudeMd()`. Likely a `path.join` vs `path.resolve` issue or a set ID mismatch.

16. **Discuss-Set "Let Claude Decide All" Bug (item 7):** Low -- Fix in `skills/discuss-set/SKILL.md` gray area question assembly. Change "Let Claude decide all" from a peer checkbox to a separate meta-prompt before topic selection.

17. **Scaffold Command (item 2):** Medium -- Create `src/lib/scaffold.cjs` and `src/commands/scaffold.cjs`. Integrate with `context.cjs` for project type detection. This is a new feature with no existing code to modify, so it can be developed independently.

18. **Version Migration Command (item 5):** Medium -- Create `src/lib/migrate.cjs` and `src/commands/migrate.cjs`. Implement version detection from STATE.json schema shape. Define migration step registry (ordered functions). Include backup, dry-run, and report generation.

19. **Context-Aware Compaction (item 4):** Low -- This is the highest-risk, highest-complexity item. Create `src/lib/compaction.cjs` as an experimental module. Make it opt-in via config. Focus on reducing context generation first (shorter prompts, leaner returns) before active compaction. Consider deferring to v3.4.0 if the simpler approach (disk-based artifact reading) proves sufficient.
