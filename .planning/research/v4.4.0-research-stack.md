## Zod Upgrade Analysis

The installed `zod@3.25.76` already exposes `zod/v4` as a subpath export. This is relevant because:

1. **Current usage:** All `require('zod')` calls resolve to Zod 3 API. The codebase uses `z.object()`, `z.enum()`, `z.string()`, `z.number()`, `z.array()`, `z.record()`, `.default()`, `.optional()`, `.passthrough()`, `.safeParse()` -- all stable in both v3 and v4.
2. **Breaking changes in Zod 4 that would affect this codebase:**
   - `.default()` inside `.optional()` fields now applies defaults (behavior change). Review `state-schemas.cjs` which uses `.default()` on `status`, `waves`, `jobs`, `sets` fields -- these are inside required fields, not optional, so **no impact**.
   - `.passthrough()` behavior: unchanged in v4. **No impact**.
   - Error customization APIs changed: this codebase uses `.safeParse()` and checks `result.success` / `result.error.issues` -- the issues format is compatible. **Low risk**.
3. **CommonJS compatibility:** Zod 3.25.76 already bundles Zod 4 under the `zod/v4` subpath. The root `require('zod')` still resolves to Zod 3 API. A full upgrade to `zod@4.x` would make `require('zod')` resolve to Zod 4 API -- confirmed CJS support via exports field inspection.
4. **Recommendation:** No upgrade needed for this milestone. The current version is modern and receives updates. A Zod 4 migration is a separate effort and not required by any feature in this milestone.

## Feature-Specific Stack Implications

### Review State Persistence

The milestone requires storing review pipeline state (which stages have been completed: lean review, bug hunt, UAT). Current assessment of the stack:

- **Storage mechanism:** The existing pattern is JSON files in `.planning/sets/<set-id>/` (e.g., `REVIEW-ISSUES.json`, `MERGE-STATE.json`). A new `REVIEW-STATE.json` file following this pattern is the natural approach.
- **Validation:** Zod is already used for review schemas (`ReviewIssue`, `ReviewIssues`, `ScoperOutput` in `review.cjs`). Adding a `ReviewPipelineState` schema is straightforward with the current Zod 3 API.
- **Concurrency:** Review state is per-set and not accessed concurrently (review is a sequential pipeline within a single agent). `proper-lockfile` is not needed for this; simple `fs.writeFileSync` is sufficient, consistent with how `REVIEW-ISSUES.json` is handled.
- **No new dependencies required.**

### Terminal Color Changes

The milestone requires changing bright ANSI background colors to dark variants. Stack implications:

- **Current implementation:** `display.cjs` uses raw ANSI escape codes with bright background variants (`\x1b[10Xm` range: 101=bright red, 102=bright green, 104=bright blue).
- **Target:** Switch to standard (dark) background codes (`\x1b[4Xm` range: 41=red, 42=green, 44=blue) for better readability with white text on dark terminals.
- **ANSI compatibility:** Standard 8-color codes (30-37 fg, 40-47 bg) are universally supported across all terminals. Switching from bright (10X) to standard (4X) backgrounds improves compatibility.
- **NO_COLOR standard:** The codebase does not check `NO_COLOR` or `FORCE_COLOR` environment variables. This is noted but not a blocker -- banners are visual branding and always use color by design (commented in display.cjs line 8).
- **No new dependencies required.** Raw ANSI codes are the correct approach for this project.

## Recommendations

1. **Add `engines` field to package.json:** `"engines": { "node": ">=18" }` -- provides npm install-time warnings for incompatible Node.js versions. **Priority: medium** (good practice for open-sourcing).

2. **Use Zod for review state schema:** Add a `ReviewPipelineState` Zod schema in `review.cjs` following the existing `ReviewIssues` pattern. No new dependencies. **Priority: high** (directly supports the review state feature).

3. **Switch ANSI backgrounds from bright to standard range:** Replace `\x1b[10Xm` codes with `\x1b[4Xm` in `display.cjs`. This is a ~20-line change in the `STAGE_BG` constant. **Priority: high** (directly supports the color improvement feature).

4. **Do not upgrade Zod to v4 in this milestone:** The current version works correctly, the v4 migration involves behavioral changes in `.default()` and error APIs, and no feature in this milestone requires v4 capabilities. **Priority: N/A** (defer).

5. **Do not add new dependencies:** All three features (review state, colors, documentation) can be implemented with existing stack. **Priority: N/A** (constraint).

6. **Consider adding `test:coverage` npm script:** `"test:coverage": "node --test --experimental-test-coverage 'src/**/*.test.cjs'"` -- useful for documentation and contributor onboarding. **Priority: low**.
