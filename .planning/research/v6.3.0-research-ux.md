# Domain & UX Research: v6.3.0 Mission Control & Fixes

## Domain Conventions

RAPID's Mission Control web dashboard operates at the intersection of two domains: **developer tool dashboards** (project management, observability) and **interactive code visualization** (codebase graphs, dependency viewers). The v6.3.0 milestone introduces a major UX surface change -- overhauling the knowledge graph into a full codebase graph -- plus interaction refinements across the kanban board, bug-fix orchestration, and shell configuration.

### Standard Terminology and Naming Conventions

| Current RAPID Term | Domain Standard | Assessment |
|---|---|---|
| Knowledge Graph | Code Graph / Dependency Graph / Codebase Map | "Knowledge Graph" is misleading -- it implies an AI/semantic graph (e.g., FalkorDB CodeGraph, GraphRAG). The current implementation renders the DAG of sets, not a knowledge graph. If the overhaul creates a full codebase graph, "Code Graph" or "Codebase Graph" aligns with Sourcetrail, CodeVisualizer, and Emerge conventions. |
| Mission Control | Dashboard / Control Panel / Command Center | "Mission Control" is distinctive branding that works well for a developer tool. GitHub uses "Copilot Squad" to evoke team coordination; RAPID's space metaphor is similarly evocative. Keep it. |
| Kanban Board | Board / Task Board | Standard Trello/Jira/Linear terminology. No change needed. |
| Bug-fix waves | Fix batches / Fix rounds / Fix passes | "Waves" is already established in RAPID's vocabulary (DAG waves). Extending it to bug-fix orchestration is natural and internally consistent. |

**Source:** FalkorDB CodeGraph blog, Emerge GitHub, Sourcetrail documentation, Copilot Squad GitHub blog.

### Common Workflows and User Journeys

**Code Graph Exploration (Industry pattern from Sourcetrail, CodeVisualizer, AppMap):**

1. User opens graph view -- sees high-level overview of codebase structure
2. User zooms/pans to area of interest
3. User clicks a node (file, function, class) -- detail panel appears
4. Detail panel shows source code or symbol information
5. User can navigate between connected nodes via edges
6. User returns to overview with a "fit" or "reset" action

This is the canonical "overview first, zoom and filter, then details on demand" (Shneiderman's mantra) pattern used by virtually every interactive graph tool.

**Current RAPID Knowledge Graph workflow:**
1. User opens /graph page -- sees DAG of sets (not codebase)
2. Clicks node -- sees set name, status, wave, dependencies
3. No drill-down, no file viewing, no code context

The gap between current state and the codebase graph feature is significant. The overhaul needs to bridge from a simple DAG display to a full codebase exploration tool.

**Multi-agent Bug-Fix (Industry pattern from GitHub Copilot Fleet, Squad, Composio):**

1. Orchestrator receives bug list (or generates from UAT)
2. Bugs are prioritized/grouped into waves by severity or file scope
3. Each wave dispatches agents sequentially or in parallel
4. Results are aggregated and reported as a batch summary
5. Failed fixes are surfaced with blocker details for human triage

RAPID's current `/rapid:bug-fix --uat` already follows this pattern for UAT failures but processes bugs sequentially in a single agent context. The proposed multi-agent wave splitting would reduce context rot by giving each wave a fresh agent context -- this aligns with the "narrow and opinionated agents" pattern advocated by Addy Osmani's "Code Agent Orchestra" analysis.

**Source:** Addy Osmani "The Code Agent Orchestra", GitHub Copilot Fleet blog, GitHub Squad documentation.

### Competitive Landscape Patterns

| Tool | Code Graph Feature | Click-to-File | Multi-Agent Bug-Fix |
|---|---|---|---|
| Sourcetrail | Full codebase graph with symbols, cross-references | Yes -- graph node click highlights code in split panel | N/A |
| CodeVisualizer (VS Code) | Function flowcharts + dependency graphs | Yes -- click navigates to code in editor | N/A |
| Emerge | Interactive dependency graph with metrics | Limited -- file-level only, no source viewing | N/A |
| GitHub Next Repo Visualization | File-level sunburst/treemap | No -- visual only, no source viewing | N/A |
| Copilot Fleet | N/A | N/A | Yes -- parallel agent dispatch with orchestrator |
| Squad | N/A | N/A | Yes -- coordinated agents with role specialization |
| Composio Agent Orchestrator | N/A | N/A | Yes -- task-per-agent with status dashboard |

**Key observation:** No competing tool combines both a codebase graph AND multi-agent orchestration in the same dashboard. RAPID has an opportunity to be the first tool that lets you visualize codebase structure AND see agent activity overlaid on that graph.

## Interaction Models

### Primary Interaction Paradigm

Mission Control is a **web dashboard** (React 19 + Tailwind CSS + Vite) with a sidebar navigation pattern. Users interact via:

- **Mouse-driven exploration** -- click, drag, zoom on graphs; drag-and-drop on kanban
- **Keyboard shortcuts** -- vim-style "g" prefix navigation (gd=dashboard, gk=graph, gb=board)
- **Sidebar project selector** -- dropdown to switch between registered projects

The current graph page uses Cytoscape.js with dagre layout. This is appropriate for the existing DAG visualization but may need evaluation for the codebase graph feature, which will have significantly more nodes (hundreds of files vs. dozens of sets).

### Input/Output Patterns Users Expect

**For codebase graph (based on Sourcetrail/CodeVisualizer conventions):**

- **Search/filter** -- Users expect to type a filename or symbol name and have the graph highlight/navigate to it. Every major graph tool includes a search box. The current Knowledge Graph page has no search.
- **Click-to-view** -- Clicking a file node should open a source code viewer panel. Sourcetrail uses a split view (graph left, code right). CodeVisualizer uses a sidebar panel. For RAPID's single-page layout, a slide-out panel or bottom drawer is most natural.
- **Hover tooltips** -- Hovering a node should show a brief summary (file path, symbol count, language) without requiring a click. The current implementation only shows details on click.
- **Edge semantics** -- Users expect to understand what edges mean. In the DAG, edges are dependency relationships. In a codebase graph, edges could represent imports, function calls, or file references. Edge type should be visually distinguishable (different colors, labels, or dash patterns).
- **Zoom-to-fit on load** -- The graph should auto-fit to show all visible nodes when first rendered. The current implementation relies on dagre layout with fixed padding, which works for small graphs but may not scale.

**For kanban board save shortcut:**

The industry convention for keyboard-driven save in modal contexts:

| Tool | Save Shortcut | Notes |
|---|---|---|
| Trello | No Ctrl+click; Enter submits | Trello uses Enter to save card edits |
| Linear | Cmd/Ctrl+Enter | Saves and closes issue modal |
| Notion | Cmd/Ctrl+Enter | Sends block or completes edit |
| Kanban Tool | Ctrl+Enter | Creates/saves and closes task |
| Kanbased | Keyboard-first; Enter submits | Full keyboard-driven kanban |

**Recommendation:** The bug report says "Ctrl+click should save when adding card details." The standard convention is **Ctrl+Enter** (not Ctrl+click), which saves the current form/modal. Ctrl+click typically opens links in new tabs in web browsers and should not be overridden. The fix should implement **Ctrl+Enter** to save the CardDetailModal, matching Linear/Notion/Kanban Tool conventions.

**Source:** Kanban Tool keyboard shortcuts documentation, Linear keyboard shortcuts, Notion keyboard shortcuts.

### Feedback and Progress Indication Conventions

**For the code graph:**

- **Loading states** -- The current loading skeleton (pulsing rectangle) is adequate for the DAG. For a full codebase parse that may take seconds, a progress bar with "Parsing N files..." text is more appropriate. Backend should support a progress-aware endpoint or the frontend should show the tree-sitter parse status.
- **Empty states** -- "No DAG.json found" is the current empty state. For a codebase graph, the empty state should distinguish between "no project selected," "parsing in progress," and "no supported source files found" (the CodebasePage already handles this well and should be a template).
- **Graph rendering performance** -- Cytoscape.js handles up to ~1000 nodes reasonably. For larger codebases, the frontend should implement progressive loading (show top-level directories first, expand on click) or server-side pagination. The current `max_files=500` limit in the codebase service is a reasonable default.

**For multi-agent bug-fix:**

- **Wave progress** -- When the orchestrator splits bugs into waves, the user needs to see which wave is currently executing. The existing bug-fix skill uses text banners for this; no dashboard integration exists yet.
- **Agent context freshness** -- The key UX benefit of wave-splitting is reducing context rot. The user should understand why bugs are being split into waves (not just that they are). A brief explanation like "Splitting into 3 waves to keep agent context fresh" sets expectations.

### Error Communication Patterns

The current error patterns are consistent across Mission Control pages:

- 404 errors show "No data found" with muted text
- Backend errors show red text with retry button
- Loading states use skeleton animations

These patterns should be maintained for the new code graph feature. One improvement: the current Knowledge Graph 404 message says "No DAG.json found for this project" -- this is an implementation detail. Users should see "No graph data available. Run `/rapid:init` to generate project structure." pointing them toward an actionable next step.

## Information Architecture

### How Information Should Be Organized for the Code Graph

The overhaul from "knowledge graph" (DAG of sets) to "code graph" (full codebase) changes the information architecture significantly:

**Current (Knowledge Graph = Set DAG):**
```
/graph -> Cytoscape.js rendering of sets/DAG.json
          Nodes = sets, Edges = set dependencies
          ~5-20 nodes typical
```

**Proposed (Code Graph = Codebase Graph):**
```
/graph -> Interactive codebase visualization
          Option A: File-level graph (files as nodes, imports as edges)
          Option B: Symbol-level graph (functions/classes as nodes, calls/references as edges)
          Option C: Hierarchical (directory tree with expandable file nodes containing symbols)
          ~50-500+ nodes typical
```

**Recommendation:** Use a **layered approach** (progressive disclosure):

1. **Layer 1: Directory overview** -- treemap or cluster view of directories, colored by language. Quick orientation.
2. **Layer 2: File graph** -- files as nodes, import/dependency edges between them. Click a directory cluster to drill in.
3. **Layer 3: Symbol detail** -- click a file node to see its symbols in a side panel (reusing the existing CodebasePage symbol tree component). Source code viewing via CodeMirror (already available in the project for the Notes editor).

This preserves the existing Codebase page's file tree as the detailed view while the graph provides the exploratory, relationship-focused view. The two pages should be interlinked (click "view in graph" from Codebase page, click "view symbols" from Graph page).

**Key decision for roadmap:** Should the set DAG remain visible somewhere? Options:
- Replace the graph page entirely with the codebase graph (set DAG moves to the State page or Dashboard)
- Add a toggle/tab on the graph page: "Codebase" | "Set Dependencies"
- Keep both as separate nav items (adds a 10th nav item, against UX advice from previous research about cognitive load)

**Recommendation:** Tab-based toggle on the graph page. The DAG is still useful context but is not accessed frequently enough to warrant its own nav item.

### Navigation and Discovery Patterns

The current sidebar has 9 navigation items, which is at the upper bound of "manageable" for developer tools. The v6.0.0 research noted that "17+ commands = cognitive overload" for CLI; the same principle applies to dashboard navigation. Adding more pages should be avoided.

The current navigation structure:
```
Dashboard | Projects | State | Worktrees | Graph | Codebase | Kanban | Notes | Settings
```

The Graph and Codebase pages are conceptually related (both visualize code structure). If the code graph feature subsumes the Codebase page's functionality (graph + click-to-view-source), consider merging them into a single "Code" page with two views: graph and list. This would reduce navigation items from 9 to 8.

### Content Hierarchy and Progressive Disclosure

The code graph should follow **Shneiderman's visual information-seeking mantra**: "Overview first, zoom and filter, then details on demand."

Applied to the RAPID code graph:
1. **Overview** -- full codebase graph with directory clusters, auto-fit zoom
2. **Zoom** -- mouse wheel zoom, click directory to expand files, search filter
3. **Filter** -- language filter (show only Python, only JS, etc.), hide test files toggle
4. **Details** -- click file node to open side panel with: file path, language, symbol tree, source code viewer

### Help and Documentation Conventions

The graph page should include a help tooltip or "?" button explaining:
- What the graph represents (codebase file dependencies / set dependency DAG)
- How to interact (click, drag, zoom, search)
- How to read edge types (import vs. dependency)

The current GraphControls component (Fit, Horizontal/Vertical, Reset) is a good start. Add a "?" button to this control bar for contextual help.

## Accessibility Considerations

### Key Accessibility Patterns for the Code Graph

- **Keyboard navigation** -- Cytoscape.js supports keyboard events but does not provide built-in keyboard navigation between nodes. The graph should implement Tab/Arrow key navigation between nodes, with Enter to select/view details. This is an area where most graph visualization tools fall short.
- **Screen reader announcements** -- When a node is selected, announce its details via aria-live region. The current `selectedNode` detail panel should have `role="status"` or `aria-live="polite"`.
- **Color-independent status indication** -- Node colors currently encode status (pending=muted, executing=orange, complete=green). These should also have shape or pattern variation for colorblind users. The current node shape is uniformly "roundrectangle" -- varying shape per status (circle for pending, diamond for executing, rectangle for complete) would improve accessibility.
- **Focus management in modals** -- The CardDetailModal should trap focus within the modal when open and return focus to the triggering element when closed. The current implementation adds a keydown listener for Escape but does not trap focus, which is an accessibility gap.

### Kanban Board Keyboard Navigation

The current kanban board uses dnd-kit for drag-and-drop but lacks keyboard-driven card creation and editing shortcuts. The bug report requesting Ctrl+click for save suggests users want keyboard-driven workflows. Beyond fixing the save shortcut:

- Arrow keys to navigate between cards within a column
- Enter to open card detail
- Ctrl+Enter to save and close detail modal
- Ctrl+N or "n" to create a new card in the focused column
- Escape to close modals (already implemented)

**Source:** Kanban Tool keyboard shortcuts, Kanbased keyboard-driven design, WAI-ARIA dialog pattern.

## User Expectations

### Default Behaviors Users Expect

**Code graph users expect:**
- The graph renders immediately with sensible defaults (no configuration required)
- File nodes are labeled with filenames (not full paths -- paths shown on hover/click)
- Edge direction indicates dependency direction (A imports B = arrow from A to B)
- Double-click a node to zoom to it (standard Cytoscape.js behavior, not currently enabled)
- The graph remembers zoom/pan position when navigating away and back (React state preservation)
- A search box is prominently placed (top of page or in graph controls bar)

**Kanban users expect:**
- Ctrl+Enter saves the current form (not Ctrl+click, which is browser-reserved for open-in-new-tab)
- Unsaved changes trigger a confirmation dialog on close ("You have unsaved changes. Discard?")
- Card creation is fast -- type title, press Enter, card appears

**Bug-fix orchestration users expect:**
- Clear indication of how many waves and why (e.g., "3 waves, grouped by severity")
- Progress per wave (not just final summary)
- Ability to stop/cancel mid-wave if a critical issue is found
- Each wave's results are visible before the next wave starts (sequential confirmation model)

### Terminology Users Use vs. Technical Terms

| User says | RAPID technical term | Notes |
|---|---|---|
| "Show me the code structure" | Codebase graph / CodebasePage | Users think in terms of "structure" not "graph" or "AST" |
| "What depends on this file?" | Incoming edges in codebase graph | Users think about dependencies as questions, not graph theory |
| "Fix these bugs" | `/rapid:bug-fix --uat <set>` | Users want to say "fix bugs" not "dispatch sequential agents per severity wave" |
| "Save" (in modal context) | `handleSave` / `updateCard.mutate` | Users expect Ctrl+S or Ctrl+Enter, not mouse-only interactions |
| "Version" (in sidebar) | `__version__` in `app/__init__.py` | Users expect this to match the plugin version (6.2.0), not the web backend's independent version (4.2.1) |

### Mental Models from Competing Products

**VS Code users** (majority of RAPID's audience) expect:
- File explorer on the left, code on the right (split panel layout)
- Ctrl+P to search for files by name
- Minimap for code orientation
- Breadcrumb navigation showing file path

These mental models should inform the code graph's click-to-view-file panel design: place the source viewer on the right side of the graph, use a breadcrumb at the top showing the file path, and include a search input that works like Ctrl+P (fuzzy file name matching).

**Jira/Linear users** (for kanban board) expect:
- Cards save automatically or with Ctrl+Enter
- Drag-and-drop for reordering
- Quick-add at the top or bottom of a column
- Card detail as a modal overlay (not a separate page)

RAPID's kanban already follows most of these patterns. The Ctrl+Enter save is the missing piece.

### Onboarding and Learning Curve Expectations

For the code graph overhaul:
- First-time users should see a brief tooltip or overlay: "This is your codebase graph. Click any file to view its source. Drag to pan, scroll to zoom."
- The graph should work immediately without any configuration or additional setup
- If tree-sitter parsing is required, it should happen on first page load with a clear loading indicator

## Recommendations

Ordered by priority for the v6.3.0 roadmap:

1. **Rename "Knowledge Graph" to "Code Graph"**: The current name is misleading. The page title, nav item label, and route should all use "Code Graph" or "Codebase Graph." -- Priority: critical (user-facing terminology must match functionality)

2. **Implement click-to-view-file in code graph**: When a user clicks a file node in the codebase graph, open a side panel with source code viewed via CodeMirror (already available for Notes). This is the core UX feature of the overhaul. Follow the Sourcetrail pattern: graph on left, code panel on right. -- Priority: critical (this is the primary feature)

3. **Add search/filter to code graph**: A search input that filters graph nodes by file name or symbol name, with results highlighting in the graph. Without search, a 200+ node graph is unusable. -- Priority: critical (usability at scale)

4. **Fix Kanban save shortcut to use Ctrl+Enter**: The bug report says "Ctrl+click should save." Implement Ctrl+Enter instead -- it is the industry standard for save-and-close in modals (Linear, Notion, Kanban Tool). Ctrl+click is browser-reserved and should not be overridden. -- Priority: high (bug fix, straightforward)

5. **Fix version display to use dynamic version**: The sidebar hardcodes "RAPID v4.2.1" and the backend `__init__.py` also hardcodes "4.2.1". Both should either read from the main `package.json` (version 6.2.0+) or at minimum the web backend's `pyproject.toml` version should be bumped to match. Best approach: the `/api/health` endpoint already returns a version; the sidebar should fetch and display it dynamically rather than hardcoding. -- Priority: high (bug fix, confuses users about which version they are running)

6. **Fix shell config detection to check all config files**: The install skill's shell detection loop (`for f in ... break`) only finds the first file containing RAPID_TOOLS. It should iterate all files and report/update all of them. -- Priority: high (bug fix, causes misconfigured environments)

7. **Add tabbed view for Code Graph and Set DAG**: Keep the existing set dependency DAG accessible via a tab toggle ("Codebase" | "Dependencies") on the graph page, rather than removing it entirely or adding a new nav item. -- Priority: medium (preserves useful functionality without adding navigation complexity)

8. **Implement progressive disclosure for large graphs**: For codebases with 200+ files, render directory-level clusters first with expand-on-click to show files. This prevents the graph from being an unreadable hairball on initial load. -- Priority: medium (usability at scale)

9. **Add graph performance safeguards**: Cytoscape.js becomes sluggish above ~1000 nodes. Add a warning when the codebase exceeds the renderable limit and offer a "simplified view" that shows directories only. The existing `max_files=500` backend limit is a reasonable starting point. -- Priority: medium (prevents degraded experience on large projects)

10. **Design multi-agent bug-fix wave UX**: The bug-fix skill currently processes all failures sequentially in one agent context. The wave-splitting feature should display a clear progress banner per wave: "Wave 1/3: Fixing 4 critical bugs... Wave 2/3: Fixing 6 high-severity bugs..." with results per wave shown before proceeding. -- Priority: medium (new feature, clear user benefit from reduced context rot)

11. **Add focus trap to CardDetailModal**: The modal does not trap focus, which is an accessibility violation (WAI-ARIA dialog pattern). Add focus trap and return focus to trigger element on close. -- Priority: medium (accessibility requirement)

12. **Add hover tooltips to graph nodes**: Show file path, language, and symbol count on hover without requiring a click. Reduces click-to-discover friction for exploration. -- Priority: low (polish, improves exploration UX)

13. **Add first-time user tooltip for code graph**: A brief, dismissible overlay on first visit: "Your codebase graph. Click files to view source. Scroll to zoom." Stored in localStorage so it only shows once. -- Priority: low (onboarding polish)
