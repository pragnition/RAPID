# Research Summary

## Project Profile
- **Type:** brownfield CLI plugin (Claude Code plugin for multi-agent parallel development)
- **Stack:** Node.js (CommonJS .cjs) + Git worktrees + Zod/Ajv validation + proper-lockfile
- **Scale:** large (5 features, 3 bug fixes, 1 tech debt item; touches DAG, scaffold, init, state, agent prompt systems)
- **Key Constraint:** DAG schema version unification (v1/v2 coexistence) must be resolved before any group-annotation feature can be built, as `recalculateDAG()` silently strips non-core node properties

## Unified Findings

### Stack and Infrastructure

- **Node.js 18+ minimum is EOL.** Node 18 ended support April 2025. The project runs on Node 25 without issues and uses no Node 18-specific APIs. Bump minimum to Node 20+ (maintenance LTS through April 2026) or Node 22+ (through April 2027). Change is documentation-only: update `prereqs.cjs` version check. (sources: STACK, OVERSIGHTS)
- **Zod 3.25.76 is safe; do NOT upgrade to Zod 4.** The `^3.25.76` range is constrained to `<4.0.0` by semver. Zod 4 deprecates `.passthrough()` (used 5 times in `state-schemas.cjs`), has CJS bundle size regression (4x), and offers no features needed for this milestone. (source: STACK)
- **Ajv 8.18.0 is current.** Already includes CVE-2025-69873 fix. CJS import via `require('ajv').default` works correctly. The `CONTRACT_META_SCHEMA` uses `additionalProperties: false`, which will reject any new fields (e.g., `fileOwnership`) added by milestone features -- the schema must be updated first. (sources: STACK, PITFALLS)
- **proper-lockfile 4.1.2 is unmaintained (5+ years).** Uses `fs.mkdir` atomicity which remains stable. No immediate risk, but no future security patches. The `STALE_THRESHOLD` of 300s may cause issues in CI/containerized environments with PID reuse. (sources: STACK, PITFALLS)
- **No build step, no CI pipeline, no linting.** Raw `.cjs` files executed directly. 1832 tests run in ~17s via `node:test`. 7 pre-existing failing tests. No code coverage configured. No CI workflow exists despite v5.0 adding GitHub community infrastructure. (source: STACK)
- **CommonJS module system is deliberate and appropriate.** No ESM migration warranted. All new modules (group.cjs, principles.cjs) follow the same pattern. (source: STACK)

### Architecture Direction

- **Recommended pattern: Extended modular monolith with DAG-driven coordination.** RAPID is already a modular monolith (5-layer stack: Skill > Agent > CLI > Library > Storage) with 21 core library modules. The v6.0.0 changes deepen the DAG's role from "post-hoc artifact" to "central coordination primitive" without altering the fundamental layering. (source: ARCHITECTURE)
- **Module boundaries:** Two new modules recommended: `group.cjs` (developer group partitioning algorithm, separate from `dag.cjs` topology) and `principles.cjs` (principles capture, PRINCIPLES.md generation, CLAUDE.md injection). All other changes extend existing modules. (source: ARCHITECTURE)
- **Data flow redesign:** The init pipeline becomes DAG-driven: Discovery > Scaffold (preserves REQUIREMENTS.md) > Research (spec-aware) > Roadmap (merges into STATE.json instead of overwriting) > DAG Generation (with group annotations) > Group Assignment > Scaffold Stubs (optional). (source: ARCHITECTURE)
- **State management:** All state mutations continue through `withStateTransaction()` with Zod validation and file locking. New `mergeStatePartial()` function for roadmapper output merge replaces the current destructive Write tool pattern. (sources: ARCHITECTURE, OVERSIGHTS)
- **DAG version decision:** Migrate fully to DAGv2 (or v3 with group annotations). Deprecate v1 `createDAG()`. Add auto-migration in `tryLoadDAG()` for existing v1 DAGs. Critical: `recalculateDAG()` in `add-set.cjs` currently calls v1 and strips all non-core properties -- must be updated. (sources: ARCHITECTURE, FEATURES, OVERSIGHTS, PITFALLS)

### User Experience Direction

- **Domain alignment:** RAPID operates in the multi-agent coding orchestration domain alongside Composio Agent Orchestrator, Superset.sh, and Conductor. RAPID's differentiation is its thorough planning phase (discuss-set, plan-set) which competitors skip. The `/rapid:quick` command addresses friction for simple tasks. (source: UX)
- **DAG-centric UX expectation:** Users from Airflow/Argo expect the DAG to be the single source of truth for execution ordering, not a side artifact. The "DAG as central concept" feature aligns perfectly with this expectation. (source: UX)
- **Group assignment should be advisory.** Domain convention (Nx, Airflow team annotations) treats group assignment as a recommendation developers can adapt. RAPID's planned approach ("developers free to adapt") matches this expectation. (source: UX)
- **Flag conventions:** CLI domain standard (clig.dev) recommends supporting both short and long flag forms. The `--desc`/`--description` mismatch violates user expectations. Standardize on full-form flags as canonical with short aliases. (source: UX)
- **Terminology:** "Principles" is natural and should be used in user-facing contexts instead of "meta principles" which is opaque. "Developer group" or simply "group" is intuitive for team assignment. "Set" is non-standard but well-established in RAPID. (source: UX)
- **Progressive disclosure for groups:** Follow the existing pattern: status dashboard shows groups as headers, each group lists sets with statuses, cross-group dependencies shown at group level. (source: UX)
- **Accessibility:** RAPID as a CLI tool is inherently keyboard-navigable. Color is never the sole information carrier. Markdown tables are screen-reader compatible. No specific accessibility actions needed for this milestone. (source: UX)

## Contradictions

### Contradiction: DAG Version for Group Annotations
- **FEATURES says:** "Option 2 (Recommended): Migrate fully to DAGv2 with group support. Deprecate v1 `createDAG()`, extend `createDAGv2()` with group annotations."
- **ARCHITECTURE says:** "Extended with group-annotation support... `createDAGv3(nodes, edges, options)` -- v3 DAG with optional group annotations" (proposing a NEW v3 version)
- **Resolution:** This is a naming disagreement, not a fundamental conflict. Both agree on extending the DAG schema with group annotations and deprecating v1. Whether it is called "extended v2" or "v3" is an implementation choice. The ARCHITECTURE recommendation to create v3 is more explicit about backward compatibility (v2 consumers remain unchanged). **Recommend: v3 with v1-to-v2 auto-migration and v2-to-v3 auto-migration in `tryLoadDAG()`.** Needs implementer decision on naming.

### Contradiction: Where Group Data Lives
- **FEATURES says:** "New field on DAG nodes: `group: string`" and "New top-level DAG field: `groups: { [groupName]: { sets: string[], description?: string } }`"
- **OVERSIGHTS says:** "Extend DAG node schema with optional `group` field. Extend state schema with optional `group` on `SetState`."
- **PITFALLS says:** "Store group annotations outside DAG.json (e.g., in STATE.json per-set metadata, or in a separate GROUPS.json)" as one alternative.
- **Resolution:** FEATURES and OVERSIGHTS agree that group data should be on DAG nodes. PITFALLS raises a valid alternative (storing outside DAG.json) as a workaround for the `recalculateDAG()` stripping problem. The stripping problem should be fixed directly (carry forward annotations during recalc) rather than working around it with separate storage. **Recommend: group data on DAG nodes (primary) with optional mirror on `SetState` for convenience, and fix `recalculateDAG()` to preserve annotations.**

### Contradiction: Spec File Routing to Research Agents
- **FEATURES says:** "Pass spec file content as additional context to all 6 research agents."
- **PITFALLS says:** "Route spec files only to the research-synthesizer agent, not to individual research agents" to avoid anchoring bias.
- **Resolution:** Both identify valid concerns. FEATURES prioritizes research quality (agents build on existing findings), while PITFALLS prioritizes research independence (agents evaluate without bias). **Recommend: PITFALLS' middle-ground option -- provide spec to all 6 agents WITH explicit framing: "The user has provided this spec. Evaluate its assertions critically -- recommend alternatives if the spec's choices have significant drawbacks." Tag spec-derived assertions with `[FROM SPEC]`.** Needs user decision on whether independence or efficiency is higher priority.

### Contradiction: Foundation Set #0 as DAG Node vs. Pre-DAG Step
- **PITFALLS says:** "Make Set #0 a 'scaffold' step that commits to the shared branch BEFORE worktrees are created -- not a tracked set in the DAG."
- **FEATURES says:** Foundation set #0 is a set with `foundation: true` flag in DAG.json, tracked like any other set.
- **OVERSIGHTS says:** "If set #0 is implemented as a regular set, enforce DAG ordering in start-set... If set #0 is a different concept (pre-set scaffold), it needs its own lifecycle status."
- **Resolution:** Making set #0 a regular DAG node is simpler (reuses existing lifecycle) but creates a serial bottleneck. Making it a pre-DAG step is cleaner but requires a separate lifecycle. **Recommend: Regular DAG node with `foundation: true` flag, but with an explicit scope constraint (interfaces and stubs only, no feature implementation) to minimize the bottleneck.** Needs user decision.

## Recommended Approach

The v6.0.0 milestone should be approached in three phases, reflecting the clear dependency structure identified across all six research files.

**Phase 1: Stabilize the Foundation.** Begin by fixing all three known bugs (REQUIREMENTS.md overwrite, `--desc`/`--description` mismatch, roadmapper state overwrite) and resolving the DAG version question. These are prerequisite repairs that affect every subsequent feature. The bug fixes are small (1-10 line changes each) but prevent data loss and state corruption that would undermine all other work. The DAG version resolution -- committing to a unified format (v2/v3 with auto-migration from v1), fixing `recalculateDAG()` to preserve annotations, and adding `fileOwnership` to the contract meta-schema -- establishes the data model foundation that Features 1, 2, and 3 all build on. This phase also includes the Node.js minimum version bump (documentation-only change to `prereqs.cjs`) and the `execSync`-to-`execFileSync` security fix in `worktree.cjs`.

**Phase 2: Build Core Capabilities.** With the foundation stable, implement the five features in dependency order. Features 4 (spec file integration) and 5 (meta principles) are independent and can proceed in parallel with the DAG work. Feature 1 (DAG as central concept) should be built next, followed by Feature 2 (DAG-driven set grouping) which depends on it. Throughout this phase, the developer group data model must be designed early and propagated consistently: group annotations on DAG nodes, optional group field on SetState, display integration in `dag show` and `/rapid:status`. The principles capture step should enforce a strict CLAUDE.md summary length (10-15 lines) to avoid agent context bloat.

**Phase 3: Advanced Scaffold.** Feature 3 (scaffold overhaul) is the most complex and depends on Features 1 and 2. It introduces the shared stub branch, high-fidelity stub generation, foundational set #0, and stub lifecycle management. The stub lifecycle protocol (how stubs are created, consumed, and replaced) should be defined in a design document before implementation begins. Key decisions: where stubs live (dedicated `rapid/stubs` branch vs. main), how replacement works (merge deletes stub, real file takes precedence), and how to auto-resolve the predictable stub-to-real merge conflicts (T1 auto-resolve rule with `RAPID-STUB` marker). The foundational set #0 should be a regular DAG node with a `foundation: true` flag but with scope constraints (interfaces and stubs only) to minimize the serial bottleneck.

The agent subagent namespace enforcement (Tech Debt #1) can be addressed at any point as a prose change to `core-identity.md`. It does not interact with any feature work.
