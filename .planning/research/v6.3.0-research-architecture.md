# Architecture Research -- v6.3.0 Mission Control & Fixes

## Current Architecture

RAPID is a brownfield project with a well-established 5-layer architecture: Skill Layer (30 skills) -> Agent Layer (27 agents) -> CLI Layer (rapid-tools.cjs router + 23 command handlers) -> Library Layer (41 modules in src/lib/) -> Storage Layer (.planning/ state + .rapid-worktrees/). The web dashboard (Mission Control) adds a parallel access path: React frontend (Vite + React 19 + TanStack Query + Tailwind 4) -> Python backend (FastAPI + SQLModel + SQLite) -> same .planning/ storage.

### Strengths of the Existing Architecture

1. **Clean separation between CLI and web paths.** The Node.js CLI operates directly on .planning/ files via the Library Layer, while the Python backend reads those same files through service modules (state_service, dag_service, codebase_service, worktree_service). Neither path blocks the other.

2. **File-watching sync pattern.** The FileWatcherService monitors .planning/ directories with watchdog (inotify with polling fallback) and updates SQLite metadata on STATE.json changes. This decouples the CLI write path from the web read path.

3. **Existing tree-sitter infrastructure.** The codebase_service already parses Python, JavaScript/TypeScript, Go, and Rust files using tree-sitter, extracting functions, classes, methods, and type definitions. This is the foundation the code graph will build on.

4. **Lazy-loaded KnowledgeGraphPage.** The current graph page uses cytoscape.js with the dagre layout plugin, loaded via React.lazy(). It currently renders only the DAG (set dependency graph), not a codebase graph.

5. **Established frontend patterns.** All views follow the same hook pattern: `useQuery` with 2-second refetchInterval, `apiClient.get()` for data fetching, Zustand for project selection state. The CodeMirror editor component provides a reusable code viewer.

### Areas Needing Extension or Modification

1. **Knowledge graph page is DAG-only.** It renders set dependency nodes, not codebase files. The "Rapid Mission Control Code Graph" feature requires replacing this with a full codebase graph that shows files, symbols, and import/dependency relationships.

2. **Backend codebase endpoint lacks relationship data.** `get_codebase_tree()` returns a flat list of files with symbols but no import/dependency edges between files. A code graph needs file-to-file relationships.

3. **No file content viewing endpoint.** The backend has no endpoint to serve individual file contents. The "click-into-file viewing" feature requires a new endpoint.

4. **Version is hardcoded in three places.** The sidebar shows `RAPID v4.2.1` as a string literal (Sidebar.tsx lines 124+126), and the backend `__init__.py` exports `"4.2.1"`. The health endpoint serves this version. The actual RAPID version is `6.2.0` in package.json.

5. **Shell config update is single-shell.** The install skill (SKILL.md Step 2) iterates shell config files with a `for` loop but `break`s after the first match. If RAPID_TOOLS is set in both `~/.bashrc` and `~/.config/fish/config.fish`, only the first match gets reported to the agent, which then only updates that one.

6. **Bug-fix skill is sequential-only.** The current `/rapid:bug-fix` skill dispatches one executor at a time for each failure. The "Multi-agent Bug-Fix Support" feature requires wave decomposition and parallel dispatch.

7. **Kanban CardDetailModal lacks keyboard save.** The modal handles Escape for close but has no Ctrl+click or Ctrl+Enter save shortcut.

## Recommended Architecture Pattern

- **Pattern:** Evolutionary extension of existing modular monolith
- **Rationale:** All six changes in this milestone (2 features + 3 bug fixes + 1 deferred decision) are localized modifications to well-defined modules. The existing architecture cleanly supports these additions without structural changes. The codebase graph is the largest feature but fits within the existing codebase_service -> views router -> React page pattern.
- **Evidence:** Previous milestones (v4.0.0 through v6.2.0) successfully extended this same architecture for the web dashboard, kanban, notes, and branding subsystems.

## Module Organization

### Module: Codebase Graph Backend (codebase_service.py)

- **Responsibility:** Parse project source files with tree-sitter and extract a code graph consisting of files, symbols, and import/dependency edges.
- **Public interface:** `get_codebase_graph(project_path, max_files) -> dict` returning nodes (files + symbols) and edges (imports, function calls); `get_file_content(project_path, relative_path) -> dict` returning file text with optional syntax highlighting metadata.
- **Dependencies:** tree-sitter, tree-sitter-python, tree-sitter-javascript, tree-sitter-go, tree-sitter-rust (all existing).
- **Data ownership:** Owns the in-memory parse cache and graph construction logic. Reads source files from project path (read-only).

### Module: Codebase Graph Frontend (KnowledgeGraphPage.tsx + graph/ components)

- **Responsibility:** Render an interactive codebase graph with cytoscape.js showing files as nodes, imports as edges, and support click-to-view-file.
- **Public interface:** React page component at /graph route.
- **Dependencies:** cytoscape, cytoscape-dagre (existing), apiClient, useCodeGraph hook (new), CodeMirror editor (existing, reusable for file viewing).
- **Data ownership:** Owns the graph visualization state (zoom, pan, selection, layout direction).

### Module: Bug-Fix Wave Orchestrator (bug-fix SKILL.md)

- **Responsibility:** Decompose bug descriptions into severity-grouped waves and dispatch one bugfix agent per wave sequentially (reducing context rot vs. dispatching all at once).
- **Public interface:** `/rapid:bug-fix` skill invocation with optional `--waves` flag.
- **Dependencies:** rapid-executor agent, SKILL.md orchestration, optionally rapid-tools.cjs for state transitions.
- **Data ownership:** Owns the wave decomposition plan and result aggregation. Does not own bug data (that comes from UAT-FAILURES.md or user input).

### Module: Install Shell Config (install SKILL.md)

- **Responsibility:** Detect all shell config files containing RAPID_TOOLS and update all of them.
- **Public interface:** `/rapid:install` skill Step 2 shell detection.
- **Dependencies:** bash for shell detection, AskUserQuestion for user confirmation.
- **Data ownership:** Writes to user shell config files (bashrc, zshrc, config.fish, etc.).

### Module: Version Management (Sidebar.tsx + backend __init__.py)

- **Responsibility:** Display the current RAPID version in the UI and API.
- **Public interface:** Sidebar version text, /api/health version field.
- **Dependencies:** Backend __init__.__version__ string, health endpoint.
- **Data ownership:** Single source of truth is package.json version (6.2.0, soon 6.3.0). Backend __init__.py and sidebar must derive from this or be updated during release.

## Data Flow

### Code Graph Request Lifecycle

1. **Frontend:** User navigates to /graph -> `KnowledgeGraphPage` mounts -> `useCodeGraph(projectId)` fires TanStack Query
2. **API call:** `GET /api/projects/{id}/codebase-graph?max_files=500`
3. **Backend router:** `views.py` resolves project path from DB -> calls `codebase_service.get_codebase_graph()`
4. **Tree-sitter parsing:** Walk project directory, parse each file, extract symbols AND import statements
5. **Graph construction:** Build node list (files) and edge list (import relationships between files)
6. **Response:** Return `{nodes: [...], edges: [...], metadata: {...}}` as JSON
7. **Frontend rendering:** cytoscape.js renders graph with file nodes, import edges, click handlers
8. **File click:** User clicks file node -> `useFileContent(projectId, filePath)` fires -> `GET /api/projects/{id}/file?path=relative/path`
9. **File display:** CodeMirror editor renders file content in read-only mode in a slide-out panel or modal

### Bug-Fix Wave Flow

1. **User invocation:** `/rapid:bug-fix` with multiple bugs or `--uat <set-id>` with many failures
2. **Wave decomposition:** Sort failures by severity, group into waves (e.g., wave 1 = critical, wave 2 = high, wave 3 = medium+low)
3. **Sequential wave dispatch:** For each wave, spawn one rapid-executor agent
4. **Agent execution:** Each agent processes its batch of bugs, commits atomically per fix
5. **Result collection:** Parse RAPID:RETURN from each agent, aggregate results
6. **Summary display:** Show combined results table with per-wave status

### Version Display Flow

1. **Backend startup:** `app/__init__.py` exports `__version__` -> health endpoint serves it
2. **Frontend sidebar:** Currently hardcoded `"RAPID v4.2.1"` -- should fetch from `/api/health` and display dynamically
3. **Dashboard:** Already fetches and displays `data.version` from useHealthCheck()

### State Management

- **Graph state:** Frontend-only, lives in React component state (cytoscape instance) and TanStack Query cache (2-second polling). No persistence needed.
- **File content:** Frontend-only, TanStack Query cache with longer staleTime (files change infrequently during a session).
- **Bug-fix wave state:** Ephemeral, lives within the skill invocation session. No persistent state needed -- results are committed as git commits.
- **Version:** Backend reads from __init__.py at import time. Frontend should fetch once and cache.
- **Shell config:** Filesystem state in user home directory. Read-scan-write pattern during /rapid:install.
- **Kanban save shortcut:** Frontend-only, no state changes -- purely a UX event handler addition.

## API Design

### New Endpoints Required

**GET /api/projects/{id}/codebase-graph?max_files=500**
- Returns `{nodes: [{id, path, language, symbol_count}], edges: [{source, target, type}], metadata: {total_files, languages, parse_errors}}`
- Could extend existing `/codebase` endpoint or be a separate endpoint. Recommendation: separate endpoint because the graph structure (nodes + edges) differs fundamentally from the tree structure (files with nested symbols).

**GET /api/projects/{id}/file?path=relative/path**
- Returns `{path, content, language, line_count}`
- Security: validate that `path` is within the project directory (no path traversal). Reject if resolved path escapes project root.
- Size limit: refuse files larger than 1MB (consistent with existing _MAX_FILE_SIZE in codebase_service).

### Existing Endpoints Unchanged

- `GET /api/health` -- already returns version (just needs correct version value)
- `GET /api/projects/{id}/codebase` -- existing tree view remains for the Codebase page
- `GET /api/projects/{id}/dag` -- existing DAG view remains but the Knowledge Graph page will no longer use it (it will use codebase-graph instead)

### Convention Compliance

All new endpoints follow the existing pattern: `/api/projects/{id}/resource`, FastAPI router in `app/routers/views.py`, Pydantic response models in `app/schemas/views.py`, service logic in `app/services/`.

## Scaling Considerations

### Code Graph Performance

- **Bottleneck:** Tree-sitter parsing of large codebases (500+ files). The existing `_parse_cache` mitigates this with `(filepath, mtime)` keying, so subsequent requests only re-parse changed files.
- **Import edge extraction:** This is new work. Parsing import statements requires walking AST nodes for `import_statement`, `require()` calls, etc. This is O(n) per file and can be done in the same pass as symbol extraction.
- **Graph payload size:** A 500-file project with ~5 edges per file produces ~2500 edges + 500 nodes. At ~200 bytes per element, that is ~600KB JSON -- well within acceptable API response size.
- **Frontend rendering:** cytoscape.js handles 1000+ nodes without performance issues with the dagre layout. The force-directed layout for code graphs may be heavier; consider cose-bilkent layout for better performance on dense graphs.

### File Content Endpoint

- **1MB file size limit** is already enforced in codebase_service. Apply the same limit to file content serving.
- **No persistent caching needed** -- files are served from disk on demand. TanStack Query cache on the frontend handles repeated views within a session.

### Bug-Fix Wave Dispatch

- **Sequential wave execution** is intentional (reduces context rot). No parallelism concerns.
- **Memory:** Each executor agent runs in its own context. Sequential dispatch means only one is active at a time.

## Architecture Risks

1. **Import resolution accuracy across languages**: Extracting import edges requires language-specific AST node matching. Python (`import_statement`, `import_from_statement`), JavaScript (`import_declaration`, `require` calls in `call_expression`), Go (`import_spec`), Rust (`use_declaration`). Each has edge cases (dynamic imports, re-exports, conditional requires). **Impact:** Incomplete or incorrect edges in the code graph. **Mitigation:** Start with static import statements only (the 90% case); skip dynamic imports and log them as warnings.

2. **Path traversal in file content endpoint**: A `path` query parameter that reads arbitrary files is a security risk. **Impact:** Reading files outside the project directory (e.g., `/etc/passwd`). **Mitigation:** Resolve the absolute path, then verify it starts with the project root. Reject symlinks that escape the project boundary. Return 403 for violations.

3. **Version drift between package.json, backend __init__.py, and sidebar**: The version is currently maintained in three places. **Impact:** Version display remains stuck if any source is not updated. **Mitigation:** Either (a) make the backend read version from the Node.js package.json at startup, or (b) add a release checklist step, or (c) have the sidebar fetch version from /api/health (already done on dashboard -- just extend to sidebar). Option (c) is simplest and most reliable.

4. **Shell config detection break-on-first-match**: The existing install skill uses `break` after finding the first shell config file with RAPID_TOOLS. If the user has configurations in multiple shells (common for users who switch between bash and fish), only one gets updated. **Impact:** Stale RAPID_TOOLS path in secondary shell. **Mitigation:** Remove the `break`, collect all matching files, and present a multi-select or batch-update flow.

5. **Ctrl+click ambiguity in kanban**: Ctrl+click in browsers typically opens links in new tabs. Using it for "save" may conflict with default browser behavior. **Impact:** User confusion or unintended tab opening. **Mitigation:** Use `Ctrl+Enter` as the primary keyboard shortcut (standard "submit" pattern in forms), and `Ctrl+S` as alternative. Avoid Ctrl+click if any card elements are links.

## Recommendations

1. **Sidebar version from API**: Replace the hardcoded `"RAPID v4.2.1"` in Sidebar.tsx with a `useHealthCheck()` call that fetches the version from `/api/health`. The dashboard already does this. Also update `web/backend/app/__init__.py` from `"4.2.1"` to match the current release version. -- **Priority: critical** (visible user-facing bug)

2. **Separate codebase-graph endpoint**: Create `GET /api/projects/{id}/codebase-graph` as a new endpoint rather than extending the existing `/codebase` endpoint. The existing `/codebase` endpoint returns a tree structure (files with nested symbols); the graph endpoint returns a flat node+edge structure. Mixing these would complicate both the response schema and the frontend consumption. -- **Priority: critical** (foundation for the code graph feature)

3. **File content endpoint with path traversal guard**: Add `GET /api/projects/{id}/file?path=...` with strict path validation. Resolve the path against the project root, reject if it escapes. Return the file content with language detection for syntax highlighting in CodeMirror. -- **Priority: high** (enables click-into-file viewing)

4. **Import edge extraction in codebase_service**: Extend the tree-sitter parsing pass to extract import/require/use statements and resolve them to relative file paths within the project. Store edges as `{source_file, target_file, type: "import"}`. Start with static imports only. -- **Priority: high** (core data for code graph visualization)

5. **Overhaul KnowledgeGraphPage for code graph**: Replace the DAG-based graph with a codebase graph. Use cytoscape.js with a force-directed layout (cose or cose-bilkent). Files are nodes colored by language, imports are directed edges. Click a node to open a file viewer panel using the existing CodeMirror component. -- **Priority: high** (primary feature of the milestone)

6. **Fix shell config multi-match**: Remove the `break` in the install skill's shell detection loop. Collect all config files containing `RAPID_TOOLS` and update all of them, or present the full list to the user for selection. -- **Priority: high** (bug fix)

7. **Bug-fix wave decomposition**: Extend the `/rapid:bug-fix` skill to group failures into severity-based waves and dispatch one executor per wave sequentially. This reduces context rot by keeping each agent's scope narrow. The UAT-d step already iterates failures sequentially -- the change is to batch them into waves rather than one-at-a-time. -- **Priority: high** (feature request)

8. **Kanban Ctrl+Enter save shortcut**: Add a `keydown` listener in CardDetailModal for `Ctrl+Enter` (and `Cmd+Enter` on macOS) that triggers `handleSave()`. This is the standard "submit" shortcut for modal forms. Avoid Ctrl+click due to browser semantics conflicts. -- **Priority: medium** (UX improvement)

9. **Backend __init__.py version automation**: Consider reading the version from a shared source (e.g., a VERSION file at the repo root, or the Node.js package.json) to eliminate manual synchronization. Short-term: update to match current release. Long-term: automate in the release process. -- **Priority: medium** (technical debt)

10. **Code graph layout performance**: For large codebases (500+ files), the default force-directed layout can be slow. Consider using the `cose-bilkent` layout plugin for cytoscape.js which handles large graphs more efficiently, or offer a hierarchical layout option as alternative. Lazy-loading the graph page (already done) helps with initial bundle size. -- **Priority: low** (performance optimization for scale)
