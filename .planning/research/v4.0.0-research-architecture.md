# Architecture Research: Mission Control Dashboard (v4.0)

## Current Architecture

RAPID is a Claude Code plugin built as a Node.js CommonJS project. The existing system has five layers: Skill -> Agent -> CLI -> Library -> Storage, all operating within a target project's `.planning/` directory. Key characteristics of the current architecture that constrain and inform the dashboard design:

- **Storage is file-based.** All project state lives in `.planning/STATE.json`, validated by Zod schemas. Worktree tracking lives in `.planning/worktrees/REGISTRY.json`. Planning artifacts are Markdown and JSON files in `.planning/`.
- **CLI is the single entry point.** All operations route through `rapid-tools.cjs`, which dispatches to command handlers in `src/commands/` that call library modules in `src/lib/`.
- **No web layer exists today.** The plugin runs entirely within Claude Code's process. There is no HTTP server, no database, no persistent daemon.
- **Projects are discovered by walking up the directory tree** looking for `.planning/`. There is no global registry of RAPID projects.

The dashboard represents an entirely new surface area: a persistent web service living at `~/.rapid/` that watches and aggregates data from multiple RAPID project directories. It does not modify project state -- it is read-only for v4.0.

## Recommended Architecture Pattern

- **Pattern:** Satellite service with shared data contracts
- **Rationale:** The dashboard is architecturally independent from the RAPID CLI plugin. It runs as a separate process (systemd user service) with its own tech stack (Python/FastAPI + Vite/React), its own database (SQLite), and its own lifecycle. The two systems communicate through: (1) a REST API for project registration, and (2) filesystem reads of `.planning/` artifacts from registered projects. This avoids entangling the dashboard's web concerns with the CLI's Node.js/CommonJS concerns.
- **Evidence:** This mirrors the pattern used by tools like Grafana (separate dashboard service reading from data sources), Docker Desktop (separate GUI service wrapping a CLI tool), and GitHub Desktop (separate app reading git state). The spec explicitly states the dashboard is "not meant as a means for the user to edit/work on the project" -- it is a read-only observer, reinforcing the satellite pattern.

### Why FastAPI over Express.js

The spec lists "fastapi/expressjs" as options. FastAPI is the stronger choice here:

1. **Separate runtime prevents coupling.** A Python service cannot accidentally import RAPID's Node.js modules or share process state. This enforces a clean integration boundary.
2. **Built-in OpenAPI docs.** FastAPI auto-generates interactive API documentation (Swagger UI), valuable for a service that will be extended over time.
3. **SQLModel/Pydantic integration.** FastAPI's data validation with Pydantic models pairs naturally with SQLite via SQLModel, providing type-safe database access with minimal boilerplate.
4. **Async-native.** FastAPI's async support maps well to file watching (via `watchfiles`) and WebSocket connections for real-time updates.
5. **Ecosystem separation.** Keeping the dashboard in Python means it can be installed via pip/uv into its own venv at `~/.rapid/venv/`, completely isolated from Node.js dependency concerns.

### Why React (not Svelte)

1. **Ecosystem maturity.** React has broader component library availability for dashboards (charts, kanban boards, markdown editors, graph visualizations).
2. **Team familiarity.** React is more widely known, making future contributions easier.
3. **Library availability.** Specific needs like knowledge graph visualization (react-force-graph, d3-react), kanban (dnd-kit), and markdown editing (MDXEditor, react-markdown) have mature React implementations.

### Rationale for structure decisions

1. **`dashboard/` at repo root** keeps the web service code alongside the plugin but clearly separated. It is its own Python project with its own `pyproject.toml`, not interleaved with the Node.js source.
2. **Nested `dashboard/dashboard/` package** follows Python convention where the outer directory is the project root (with pyproject.toml, frontend/, etc.) and the inner directory is the importable package.
3. **Frontend inside `dashboard/frontend/`** keeps the Vite project co-located with its backend. In production, `vite build` outputs to `dashboard/frontend/dist/` which FastAPI mounts as static files.
4. **`~/.rapid/` as runtime home** separates runtime state (database, config, logs, venv) from source code. The service runs from the installed venv but stores all mutable state in the user's home directory.

## Module Organization

### Module: Project Registry (Backend)

- **Responsibility:** Track which RAPID project directories are registered with the dashboard. Store project metadata, sync state from `.planning/STATE.json`.
- **Public interface:** REST endpoints (`GET /api/projects`, `POST /api/projects`, `DELETE /api/projects/{id}`), `ProjectSyncService` class for background state refresh.
- **Dependencies:** `database.py` for SQLite sessions, `state_reader.py` for parsing RAPID artifacts.
- **Data ownership:** `registered_projects` table in SQLite.

### Module: State Reader (Backend)

- **Responsibility:** Read and parse RAPID's `.planning/` directory structure from registered project paths. Extract STATE.json, ROADMAP.md, set definitions, contracts, worktree registry.
- **Public interface:** `read_project_state(project_path) -> ProjectSnapshot`, `read_worktrees(project_path) -> List[WorktreeSnapshot]`.
- **Dependencies:** None (pure file I/O, no database writes).
- **Data ownership:** None. This is a stateless reader of external RAPID project files.

### Module: File Watcher (Backend)

- **Responsibility:** Monitor registered project directories for changes to `.planning/` files. Trigger state re-sync when STATE.json or REGISTRY.json changes.
- **Public interface:** `FileWatcherService.start(project_paths)`, `FileWatcherService.stop()`, events emitted on change.
- **Dependencies:** `watchfiles` library, `project_sync.py` for triggering refreshes.
- **Data ownership:** None. Produces events consumed by the sync service.

### Module: Kanban Board (Backend + Frontend)

- **Responsibility:** Provide a kanban board for user task tracking. Independent of RAPID's set/wave/job state -- this is the user's personal task management.
- **Public interface:** CRUD endpoints for boards, columns, cards. Drag-and-drop reordering support.
- **Dependencies:** `database.py` for SQLite.
- **Data ownership:** `kanban_boards`, `kanban_columns`, `kanban_cards` tables.

### Module: Notes (Backend + Frontend)

- **Responsibility:** Markdown note editor with tagging. Linked to projects optionally.
- **Public interface:** CRUD endpoints for notes. Tag-based filtering. Full-text search.
- **Dependencies:** `database.py` for SQLite. SQLite FTS5 for search.
- **Data ownership:** `notes`, `note_tags` tables.

### Module: Knowledge Graph (Backend + Frontend)

- **Responsibility:** Visual knowledge graph of project concepts, dependencies, and relationships. Nodes represent modules, sets, dependencies, concepts. Edges represent relationships.
- **Public interface:** Graph query endpoints (`GET /api/knowledge/{project_id}/graph`), node/edge CRUD.
- **Dependencies:** `database.py`, `state_reader.py` for auto-populating graph from project structure.
- **Data ownership:** `knowledge_nodes`, `knowledge_edges` tables.

### Module: Codebase Mapper (Backend + Frontend)

- **Responsibility:** Analyze and visualize project source code structure for compatible codebases. Show file tree, module dependencies, language breakdown.
- **Public interface:** `GET /api/codebase/{project_id}/map`, `POST /api/codebase/{project_id}/scan`.
- **Dependencies:** `state_reader.py` (reads `.planning/context/CODEBASE.md` if present), filesystem access to project directory.
- **Data ownership:** `codebase_snapshots` table (cached analysis results).

### Module: Frontend Shell (Frontend)

- **Responsibility:** Application shell with sidebar navigation, project selector, and page routing.
- **Public interface:** React Router routes, layout components.
- **Dependencies:** All page components, Zustand stores, API client.
- **Data ownership:** Client-side navigation state only.

## Data Flow

### Project Registration Flow

```
1. User runs /rapid:init in a project
2. init command detects RAPID_WEB=true in environment
3. Calls: node rapid-tools.cjs register-web --path /path/to/project --name "My Project"
4. register-web command POSTs to http://localhost:8998/api/projects
   Body: { "path": "/path/to/project", "name": "My Project" }
5. FastAPI endpoint validates path exists and contains .planning/
6. Creates row in registered_projects table
7. Triggers initial state sync (reads STATE.json, ROADMAP.md, REGISTRY.json)
8. File watcher begins monitoring project's .planning/ directory
9. Returns 201 with project ID
```

### State Sync Flow (Background)

```
1. File watcher detects change in {project_path}/.planning/STATE.json
2. Emits change event to ProjectSyncService
3. ProjectSyncService.refresh_project(project_id):
   a. state_reader.read_project_state(path) -> ProjectSnapshot
   b. Update project_snapshots table with new state
   c. If WebSocket clients connected, broadcast update
4. Dashboard frontend receives update via WebSocket or next poll
5. UI reflects new set statuses, wave progress, etc.
```

### Request Lifecycle (API)

```
1. Browser -> GET http://localhost:8998/api/projects/{id}/state
2. localhost_only middleware: verify request is from 127.0.0.1/::1
3. FastAPI router -> projects.get_project_state(id)
4. Service layer -> query SQLite for cached state
5. If stale (>30s old), trigger background re-read from filesystem
6. Return JSON response with project state
7. Browser renders updated UI
```

### Frontend Data Flow

```
1. Page component mounts (e.g., ProjectDetail)
2. useProjects() hook triggers API call via api/projects.ts
3. Response updates Zustand projectStore
4. Components re-render with new data
5. usePolling(interval=5000) schedules periodic refresh
6. WebSocket connection (if available) provides instant updates
```

### State Management

- **Server-side (SQLite):** All persistent state. Projects, kanban items, notes, knowledge graph data, cached project snapshots. Located at `~/.rapid/rapid.db`.
- **Server-side (filesystem reads):** STATE.json and REGISTRY.json from each registered project. Read on demand and cached in SQLite.
- **Client-side (Zustand):** In-memory UI state. Selected project, expanded panels, form state. Not persisted across page reloads (except via URL params for navigation state).
- **Consistency:** The dashboard's copy of project state is eventually consistent. File watcher provides near-real-time updates (sub-second on Linux via inotify). Polling fallback every 5 seconds for platforms without inotify support.

## API Design

### REST Conventions

- **Base path:** `/api/` for all REST endpoints
- **Resource naming:** Plural nouns (`/projects`, `/notes`, `/kanban/boards`)
- **HTTP methods:** GET (read), POST (create), PUT (full update), PATCH (partial update), DELETE (remove)
- **Response format:** JSON with consistent envelope: `{ "data": ..., "error": null }` or `{ "data": null, "error": { "message": "...", "code": "..." } }`
- **Pagination:** Cursor-based for lists: `?cursor=abc&limit=50`
- **Filtering:** Query params: `?project_id=123&status=executing`

### WebSocket Endpoint

```

```

Messages follow a typed envelope:

```json
{
  "type": "state_changed",
  "data": {
    "project_id": "abc123",
    "changed_files": ["STATE.json"],
    "snapshot": { ... }
  }
}
```

### Communication Strategy: Hybrid Polling + WebSocket

For the v4.0 release, the recommended approach is:

1. **Primary: Short polling (5s interval).** Simple, reliable, works everywhere. The dashboard is read-only and state changes are infrequent (a set transition happens maybe once every few minutes during active development). 5-second polling is sufficient.
2. **Enhancement: WebSocket for active views.** When a user is viewing a specific project's detail page, open a WebSocket connection for near-instant updates. This avoids the 5-second delay for the actively viewed project.
3. **File watcher drives server-side cache.** Regardless of how the frontend polls, the backend uses `watchfiles` (Rust-based, uses inotify on Linux) to monitor `.planning/` directories. State is pre-cached in SQLite so API responses are fast regardless of filesystem latency.

This hybrid approach avoids WebSocket complexity for the initial implementation while leaving the door open for real-time features.

## SQLite Schema Design

### Database Location

`~/.rapid/rapid.db` -- single SQLite file for all dashboard data.

### Core Tables

```sql
-- Registered RAPID projects
CREATE TABLE projects (
    id TEXT PRIMARY KEY,             -- UUID
    name TEXT NOT NULL,
    path TEXT NOT NULL UNIQUE,       -- Absolute path to project root
    registered_at TEXT NOT NULL,     -- ISO 8601
    last_synced_at TEXT,             -- Last successful state sync
    is_active INTEGER DEFAULT 1     -- Soft delete / deactivate
);

-- Cached project state snapshots
CREATE TABLE project_snapshots (
    id TEXT PRIMARY KEY,
    project_id TEXT NOT NULL REFERENCES projects(id) ON DELETE CASCADE,
    state_json TEXT NOT NULL,         -- Full STATE.json content
    roadmap_md TEXT,                  -- ROADMAP.md content
    rapid_version TEXT,
    snapshot_at TEXT NOT NULL,
    UNIQUE(project_id)               -- Only keep latest snapshot per project
);

-- Cached worktree status
CREATE TABLE worktree_snapshots (
    id TEXT PRIMARY KEY,
    project_id TEXT NOT NULL REFERENCES projects(id) ON DELETE CASCADE,
    registry_json TEXT NOT NULL,      -- Full REGISTRY.json content
    snapshot_at TEXT NOT NULL,
    UNIQUE(project_id)
);

-- Kanban boards (user-created, optionally linked to project)
CREATE TABLE kanban_boards (
    id TEXT PRIMARY KEY,
    project_id TEXT REFERENCES projects(id) ON DELETE SET NULL,
    name TEXT NOT NULL,
    created_at TEXT NOT NULL,
    updated_at TEXT NOT NULL
);

CREATE TABLE kanban_columns (
    id TEXT PRIMARY KEY,
    board_id TEXT NOT NULL REFERENCES kanban_boards(id) ON DELETE CASCADE,
    name TEXT NOT NULL,
    position INTEGER NOT NULL,
    color TEXT                        -- Optional hex color
);

CREATE TABLE kanban_cards (
    id TEXT PRIMARY KEY,
    column_id TEXT NOT NULL REFERENCES kanban_columns(id) ON DELETE CASCADE,
    title TEXT NOT NULL,
    description TEXT,                 -- Markdown content
    position INTEGER NOT NULL,
    labels TEXT,                      -- JSON array of label strings
    due_date TEXT,                    -- ISO 8601 or null
    created_at TEXT NOT NULL,
    updated_at TEXT NOT NULL
);

-- Notes with markdown content
CREATE TABLE notes (
    id TEXT PRIMARY KEY,
    project_id TEXT REFERENCES projects(id) ON DELETE SET NULL,
    title TEXT NOT NULL,
    content TEXT NOT NULL DEFAULT '', -- Markdown content
    created_at TEXT NOT NULL,
    updated_at TEXT NOT NULL
);

CREATE TABLE note_tags (
    note_id TEXT NOT NULL REFERENCES notes(id) ON DELETE CASCADE,
    tag TEXT NOT NULL,
    PRIMARY KEY (note_id, tag)
);

-- Full-text search index for notes
CREATE VIRTUAL TABLE notes_fts USING fts5(
    title, content,
    content='notes',
    content_rowid='rowid'
);

-- Knowledge graph (per-project)
CREATE TABLE knowledge_nodes (
    id TEXT PRIMARY KEY,
    project_id TEXT NOT NULL REFERENCES projects(id) ON DELETE CASCADE,
    label TEXT NOT NULL,
    node_type TEXT NOT NULL,          -- 'module', 'set', 'concept', 'dependency', 'file'
    properties TEXT,                  -- JSON object for extensible metadata
    x REAL,                           -- Optional layout position
    y REAL,
    created_at TEXT NOT NULL
);

CREATE TABLE knowledge_edges (
    id TEXT PRIMARY KEY,
    project_id TEXT NOT NULL REFERENCES projects(id) ON DELETE CASCADE,
    source_id TEXT NOT NULL REFERENCES knowledge_nodes(id) ON DELETE CASCADE,
    target_id TEXT NOT NULL REFERENCES knowledge_nodes(id) ON DELETE CASCADE,
    edge_type TEXT NOT NULL,          -- 'depends_on', 'imports', 'related_to', 'contains'
    properties TEXT,                  -- JSON object
    created_at TEXT NOT NULL
);

-- Cached codebase analysis results
CREATE TABLE codebase_snapshots (
    id TEXT PRIMARY KEY,
    project_id TEXT NOT NULL REFERENCES projects(id) ON DELETE CASCADE,
    file_tree TEXT NOT NULL,          -- JSON: nested file structure
    language_stats TEXT,              -- JSON: { "javascript": 45, "python": 30, ... }
    module_graph TEXT,                -- JSON: import/export dependency graph
    scanned_at TEXT NOT NULL,
    UNIQUE(project_id)
);

-- Indexes
CREATE INDEX idx_notes_project ON notes(project_id);
CREATE INDEX idx_kanban_cards_column ON kanban_cards(column_id);
CREATE INDEX idx_knowledge_nodes_project ON knowledge_nodes(project_id);
CREATE INDEX idx_knowledge_edges_project ON knowledge_edges(project_id);
CREATE INDEX idx_knowledge_edges_source ON knowledge_edges(source_id);
CREATE INDEX idx_knowledge_edges_target ON knowledge_edges(target_id);
```

### Schema Design Rationale

1. **UUIDs for primary keys** rather than autoincrement. The dashboard may eventually sync across machines and UUIDs avoid collision.
2. **JSON-in-TEXT columns** for extensible metadata (node properties, card labels, file trees). SQLite's `json_extract()` function enables querying into these when needed.
3. **FTS5 for notes search.** SQLite's built-in full-text search is performant for single-user scale and avoids external search dependencies.
4. **Knowledge graph uses adjacency list** (nodes + edges tables) following the simple-graph pattern. This is simpler than recursive CTEs for the expected graph sizes (tens to hundreds of nodes per project).
5. **Snapshot tables cache RAPID state.** Rather than querying the filesystem on every request, state is read into SQLite on change (triggered by file watcher) and served from the database. This decouples API response time from filesystem latency.

## Integration Points

| External System | Protocol | Purpose | Error Handling |
|----------------|----------|---------|---------------|
| RAPID CLI (`rapid-tools.cjs`) | HTTP POST to dashboard API | Register projects during `/init` | Fail silently if RAPID_WEB not set or service unreachable; log warning |
| Project `.planning/` directories | Filesystem read (inotify watch) | Read STATE.json, ROADMAP.md, REGISTRY.json, context files | Graceful degradation: mark project as "unreachable" if path disappears |
| SQLite (`~/.rapid/rapid.db`) | Python SQLModel/SQLAlchemy | All persistent dashboard state | WAL mode for concurrent reads; connection retry on SQLITE_BUSY |
| Vite dev server | HTTP proxy (dev only) | Frontend dev server proxies `/api/*` to FastAPI | Vite proxy config in `vite.config.ts` |
| systemd (user instance) | Unit file at `~/.config/systemd/user/` | Auto-start on login, restart on crash | `Restart=on-failure`, `RestartSec=5` |

### File Watching Strategy

The service uses `watchfiles` (Rust-based, uses inotify on Linux, FSEvents on macOS, ReadDirectoryChangesW on Windows) to monitor registered project directories:

```
For each registered project:
  Watch: {project_path}/.planning/STATE.json
  Watch: {project_path}/.planning/worktrees/REGISTRY.json
  Watch: {project_path}/.planning/ROADMAP.md
  Watch: {project_path}/.planning/context/*.md
```

When a watched file changes:
1. Debounce (500ms) to coalesce rapid successive writes
2. Re-read the changed file
3. Update the corresponding snapshot in SQLite
4. Notify connected WebSocket clients

Fallback for platforms without native file system events: poll every 10 seconds using file mtime comparison.

### Project Discovery

The dashboard does **not** auto-discover projects. Projects are explicitly registered via:

1. **During `/init`:** When `RAPID_WEB=true` is detected, the init flow calls `rapid-tools.cjs register-web --path $(pwd)` which POSTs to the dashboard API.
2. **Manual registration:** A `/register-web` skill for older projects, or the dashboard's own UI ("Add Project" button with directory picker).
3. **CLI registration:** `rapid-tools.cjs register-web --path /path/to/project` can be called directly.

The `register-web` command in rapid-tools.cjs:
- Checks `RAPID_WEB` env var is `true`
- Checks `http://localhost:8998/api/system/health` is reachable
- POSTs project path and name to `/api/projects`
- Prints success/failure message to stderr
- Fails silently (non-blocking) if the service is not running

## Integration with RAPID CLI

### New CLI Command: `register-web`

Added to `rapid-tools.cjs` as a new command:

```
rapid-tools.cjs register-web --path <project-path> [--name <project-name>]
```

Implementation in `src/lib/web-register.cjs`:
- Uses Node.js built-in `http` module (no new dependencies) to POST to `http://localhost:8998/api/projects`
- Reads project name from `.planning/STATE.json` if `--name` not provided
- Returns JSON result to stdout: `{ "registered": true, "project_id": "..." }`
- Exits 0 on success, 0 on service-unreachable (non-blocking), 1 on invalid args

### Init Command Integration

In `src/commands/init.cjs`, after successful scaffolding:

```javascript
// After scaffold completes successfully:
if (process.env.RAPID_WEB === 'true') {
  try {
    const { registerWithDashboard } = require('../lib/web-register.cjs');
    await registerWithDashboard(cwd, projectName);
  } catch (err) {
    process.stderr.write(`[RAPID] Dashboard registration skipped: ${err.message}\n`);
  }
}
```

### How the Dashboard Reads RAPID Data

The dashboard reads these files from each registered project (all read-only):

| File | What It Provides |
|------|-----------------|
| `.planning/STATE.json` | Milestone/set/wave/job hierarchy and statuses |
| `.planning/ROADMAP.md` | Set descriptions, dependency DAG |
| `.planning/worktrees/REGISTRY.json` | Active worktrees, their phases, branches |
| `.planning/sets/*/DEFINITION.md` | Set scope and description |
| `.planning/sets/*/CONTRACT.json` | Interface contracts between sets |
| `.planning/context/CODEBASE.md` | File layout, tech stack, dependencies |
| `.planning/context/ARCHITECTURE.md` | System architecture description |
| `.planning/PROJECT.md` | Project description and team config |

The state_reader service parses these into structured Python objects using Pydantic models that mirror RAPID's Zod schemas (SetStatus, WaveStatus, etc.).

## Security

### Localhost-Only Binding

- **uvicorn binds to `127.0.0.1:8998`** exclusively. Not `0.0.0.0`.
- **Middleware enforcement:** A FastAPI middleware checks `request.client.host` is `127.0.0.1` or `::1` on every request, as defense-in-depth beyond the bind address.
- **No HTTPS needed.** Localhost traffic does not leave the machine. TLS adds complexity with no security benefit for local-only services.

### Authentication Decision

**No authentication required for v4.0.** Rationale:

1. The service only binds to localhost -- no network exposure.
2. It is a single-user tool (one user's projects on one machine).
3. The dashboard is read-only for RAPID state (it cannot modify `.planning/` files).
4. Adding auth to a localhost-only service creates friction (password management, token storage) with no meaningful security gain.
5. The kanban/notes features are personal productivity tools, not shared resources.

**Future consideration:** If the dashboard ever supports remote access or multi-user features, add HTTP Basic Auth with a configurable password stored in `~/.rapid/config.json`. This would be a v4.x addition, not v4.0.

### Filesystem Access Control

- The dashboard process runs as the user, inheriting their filesystem permissions.
- It only reads from registered project paths and `~/.rapid/`.
- Write access is limited to `~/.rapid/rapid.db` and `~/.rapid/logs/`.

## Scaling Considerations

### Expected Load

- **Single user, single machine.** No horizontal scaling needed.
- **1-10 registered projects** typical. Upper bound ~50.
- **API requests:** Low frequency. Dashboard open in browser tab, polling every 5 seconds = ~12 requests/minute per view.
- **WebSocket connections:** 1-3 concurrent (one per open project detail tab).
- **SQLite throughput:** More than sufficient. SQLite handles thousands of queries/second for read-heavy workloads at this scale.

### Bottleneck Areas

1. **Codebase scanning** for large repositories could take seconds. This should be a background task with progress polling, per the project's async task conventions.
2. **Knowledge graph rendering** with hundreds of nodes may stress the browser. Use WebGL-based rendering (e.g., react-force-graph which uses three.js/d3-force).
3. **File watching** with many registered projects could accumulate file descriptors. Cap at reasonable limit (e.g., 100 watched paths per project, 50 projects max).

### Optimization Strategy

- **SQLite WAL mode** for concurrent reads during writes.
- **Debounced file watching** (500ms) to avoid thrashing on rapid successive writes.
- **Lazy loading** on frontend: only load data for the active view/project.
- **Cached snapshots** in SQLite avoid re-reading `.planning/` on every API call.

## Architecture Risks

1. **Project path staleness:** A registered project directory may be moved or deleted. **Impact:** Dashboard shows stale data or errors. **Mitigation:** Periodic health check (every 60s) marks projects as "unreachable" if `.planning/` is not found. UI shows clear status indicator.

2. **SQLite locking under concurrent writes:** The file watcher, API writes (kanban/notes), and snapshot updates all write to the same database. **Impact:** SQLITE_BUSY errors. **Mitigation:** WAL mode + connection retry with exponential backoff. Single-user workload makes true concurrent writes rare.

3. **Python dependency management:** The dashboard introduces Python as a dependency for a Node.js project. Users who do not have Python 3.10+ will not be able to use the dashboard. **Impact:** Installation friction. **Mitigation:** Make the dashboard explicitly optional (RAPID_WEB=true opt-in). Document Python version requirements clearly. Consider `uv` for faster, more reliable Python environment setup.

4. **Cross-platform file watching:** `watchfiles` works on Linux (inotify), macOS (FSEvents), and Windows (ReadDirectoryChangesW), but behavior may differ. Network filesystems (NFS, SMB) may not support inotify. **Impact:** State sync delay or failure on some platforms. **Mitigation:** Polling fallback with 10-second interval when file events are unavailable.

5. **Frontend build integration:** The Vite build output must be bundled with the Python package for production serving. **Impact:** Complex build/install pipeline. **Mitigation:** Include `frontend/dist/` as package data in `pyproject.toml`. Build step: `cd frontend && npm run build` before `pip install`.

6. **Schema migration:** As the dashboard evolves, the SQLite schema will need migrations. **Impact:** Upgrades could break existing databases. **Mitigation:** Use Alembic for schema migrations from day one. Store schema version in database.

## Recommendations

1. **Use FastAPI + SQLModel + SQLite for the backend:** Type-safe, minimal boilerplate, built-in docs, clean separation from Node.js CLI. -- Priority: critical

2. **Use Vite + React + TypeScript for the frontend:** Mature ecosystem for dashboard UIs, strong component library availability. -- Priority: critical

3. **Use Zustand for client-side state management:** Lightweight, no boilerplate, scales well for a dashboard with multiple independent views. Avoids Redux complexity for a single-user app. -- Priority: high

4. **Implement polling-first, WebSocket-optional communication:** 5-second polling is sufficient for the expected update frequency. Add WebSocket for active project views as an enhancement. -- Priority: high

5. **Use `watchfiles` for server-side file monitoring:** Rust-based, cross-platform, async-compatible. Drives the SQLite cache that backs API responses. -- Priority: high

6. **Make the dashboard entirely optional:** Detection via `RAPID_WEB=true` environment variable. The RAPID CLI must function identically with or without the dashboard. Registration failures are non-blocking warnings. -- Priority: critical

7. **Use Alembic for database migrations from day one:** Avoids painful migration retrofitting later. The schema will evolve as features are added. -- Priority: high

8. **Support both systemd (Linux) and launchd (macOS):** Detect OS during install and generate the appropriate service definition. Provide manual start instructions as fallback. -- Priority: medium

9. **Build frontend before pip install:** The `pyproject.toml` build hook should run `npm run build` in `frontend/` and include `dist/` as package data. This keeps the install as a single `pip install .` command. -- Priority: medium

10. **Use `uv` as recommended Python tool manager:** `uv` is significantly faster than pip for venv creation and package installation, and is becoming the standard Python project tool. Recommend it in docs, fall back to pip if unavailable. -- Priority: low
