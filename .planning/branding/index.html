RAPID v4.2.0

Project Branding Guidelines — Technical & Precise | Formal / Third-Person | Detailed with Examples

Project Identity

RAPID is a technical-first developer tool for orchestrating parallel, isolated development workflows. The project presents itself as precise, specification-grade infrastructure — comparable to protocol documentation or compiler references.

Communication conveys exactness, authority, and deep domain knowledge without unnecessary embellishment.

Tone & Voice

Perspective: Third-person Formal. Declarative statements. No colloquialisms.

DO
"The merge subsystem detects five levels of conflict: textual, structural, dependency, API, and semantic."
DON'T
"We check for five kinds of conflicts when merging your code."
DO
"The executor agent transitions the set status from planned to executed upon completion."
DON'T
"Once you're done planning, we'll basically just run the executor and it moves things along."

Status Lifecycle

Sets: pending → discussed → planned → executed → complete → merged
Waves: pending → executing → complete
Jobs: pending → executing → complete

Output Style

Documentation: Detailed with examples. Each non-trivial concept receives a code sample or usage example demonstrating correct application.

Code comments: Explain non-obvious behavior, architectural decisions, and edge cases. Include inline examples where a function signature alone is insufficient.

Commit messages: type(scope): description format. Types: feat, fix, refactor, test, chore, docs.

Planning artifacts: Structured markdown with explicit section headers, tables for tabular data, and numbered lists for sequential procedures.

Anti-Patterns

DO NOT
  • Use emojis in documentation, comments, commit messages, or agent output
  • Use marketing language, superlatives, or promotional tone ("blazing fast", "revolutionary", "game-changing")
  • Use filler words: "basically", "simply", "just", "very", "really", "actually", "literally"
  • Use second-person ("you") or first-person ("we", "I") in documentation or comments
  • Use colloquialisms, slang, or casual abbreviations ("gonna", "wanna", "LGTM" in prose)
  • Substitute domain terms with generic alternatives (e.g., "task" instead of "job", "module" instead of "set")
  • Omit examples when documenting non-trivial interfaces or behaviors
