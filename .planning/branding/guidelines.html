Chat Surface

Chats drive interactive skills (/rapid:discuss-set, /rapid:quick, /rapid:bug-fix, /rapid:branding). Every chat page renders a persistent composer at the bottom. Every run page does not. This is the single load-bearing mode signal.

Message Roles

Three roles: user, assistant, tool. Role label above the body in uppercase monospace. Avatar border color encodes role: border-bg-4 for user, border-accent for assistant, border-info for tool. Timestamp right-aligned in the role row.

Do / Don't

DO

Render a composer on chat pages

The composer is the load-bearing signal. Users know from chrome alone they are in a chat, not an autonomous run.

DON'T

Render a composer on run pages

A run is headless by definition. To talk to it, open a linked chat.

DO

Render tool calls inline in the message flow

Every tool use is a collapsible card within the conversation. State is always visible at a glance.

DON'T

Render tool calls in modals or popovers

Modals break the scroll context. Approval modals are only for blocking can_use_tool prompts that cannot be resolved inline.

DO

Signal elevation with surface tokens

Stack surface-0 → surface-4. Nested cards step one level higher than their parent.

DON'T

Use drop shadows or blur

Shadows break the flat terminal-dashboard aesthetic and fail against light theme variants.

DO

Stop auto-scrolling when user scrolls up

Show the ↓ N new messages pill. User clicks it to re-engage auto-scroll.

DON'T

Yank the viewport back to bottom

Users lose their place in long streams. Violates Chrome DevTools Console conventions.

DO

Ask structured questions for enumerable answers

Render radio/checkbox forms inline. Keyboard shortcut (A/B/C) per option.

DON'T

Parse freeform answers for closed-set questions

Freeform input is reserved for genuinely open-ended questions.

DO

Use domain terminology

Chat thread, Agent run, Set, Wave, Job, Tool call, Contract, Worktree, DAG.

DON'T

Substitute generic alternatives

"conversation", "session", "task", "module", "feature", "function call", "sandbox", "schema".

Chat Surface Accessibility

  • aria-busy="true" on the active assistant message region while tokens stream; flipped to false on completion. Otherwise screen readers re-read the entire partial message on each chunk.
  • aria-live="polite" on the message list so new messages announce without interrupting the user's current reading position.
  • Structured-question form options are a role="radiogroup" with keyboard navigation (Arrow keys, Enter to submit).
  • The composer textarea is labeled; Enter sends, Shift+Enter inserts a newline, Escape clears the input.
  • Tool-call cards expose their collapsed/expanded state via aria-expanded. Running state adds aria-busy="true".
  • Approval modals (can_use_tool) trap focus and return it to the trigger on close.
  • Auto-scroll opt-out respects prefers-reduced-motion: when enabled, the "new messages" pill is always visible rather than hidden behind a scroll threshold.

Keyboard

  • Focus ring: 2px --accent outline with 2px offset on every interactive element.
  • Nav: g{d,p,a,h,k,s,w,c} for top-level surfaces. ⌘K opens command palette.
  • Composer: ↵ send, ⇧↵ newline, / slash-command autocomplete, @ set mention.
  • Run page: ⇧P pause, ⇧S stop, ⇧A approve (when awaiting).

Voice & Tone

Third-person, formal, declarative. Avoid you, we, I, let's. Never use emojis, marketing language, or filler words. Apply RAPID domain terminology consistently — including to agent responses rendered in the chat surface.

Documentation

"The merge subsystem detects five levels of conflict."
"We check for five kinds of conflicts when merging your code!"

Assistant message (chat)

"Reproduced. The L1 detector strips newlines but not BOM markers."
"Oh interesting! I can totally see what's happening here 😊"

Error card

"Merge conflict — level 3 (dependency). package.json version mismatch."
"Oops! Something went wrong. Please try again."

Empty state

"No chat threads. Start one with /rapid:discuss-set."
"Looks like you haven't started any chats yet — let's go!"
