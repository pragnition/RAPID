  <name>Task 2: Rewrite merge SKILL.md pipeline orchestrator</name>

  <action>
    Completely rewrite skills/merge/SKILL.md for the v2.0 merge pipeline. The skill orchestrates the full merge flow using CLI subcommands and spawns the merger agent for semantic analysis.

    **Frontmatter:**
    ```yaml
    ---
    description: Merge completed sets into main -- 5-level conflict detection, 4-tier resolution, DAG-ordered merging, bisection recovery, rollback
    allowed-tools: Read, Write, Bash, Agent, AskUserQuestion
    ---
    ```

    **Step 1: Load Merge Plan** (preserved from v1.0, enhanced)
    - Run `node "${RAPID_TOOLS}" merge order` to get DAG-ordered wave groups
    - Run `node "${RAPID_TOOLS}" merge status` to check current progress
    - Show merge plan with wave structure
    - AskUserQuestion: "Start merge" / "Cancel"
    - If specific set provided, merge only that set and its unmerged dependencies

    **Step 2: Process Each Wave** (enhanced)
    - For each wave, run Steps 3-8 for each set. Sequential within wave.
    - Before wave starts, record pre-wave commit: `node "${RAPID_TOOLS}" merge merge-state` to get baseline

    **Step 3: Detection Pipeline** (NEW)
    - Update MERGE-STATE: `node "${RAPID_TOOLS}" merge update-status {set} detecting`
    - Run 5-level detection: `node "${RAPID_TOOLS}" merge detect {set}`
    - Parse JSON result. Display detection summary:
      > Detection Report for {set}:
      > - L1 Textual: {count} conflicts
      > - L2 Structural: {count} conflicts
      > - L3 Dependency: {count} conflicts
      > - L4 API: {count} conflicts
      > - L5 Semantic: pending (requires merger agent)
    - If zero conflicts across all levels, skip to Step 6 (merge directly)
    - If conflicts found, proceed to Step 4

    **Step 4: Resolution Cascade** (NEW)
    - Update MERGE-STATE: `node "${RAPID_TOOLS}" merge update-status {set} resolving`
    - Run resolution: `node "${RAPID_TOOLS}" merge resolve {set}`
    - Parse JSON result. Display resolution summary:
      > Resolution Summary for {set}:
      > - Tier 1 (deterministic): {count} resolved
      > - Tier 2 (heuristic): {count} resolved
      > - Remaining for AI: {count}
    - If all resolved by T1/T2, proceed to Step 5 (programmatic gate before merge)
    - If unresolved conflicts remain, proceed to Step 4b (merger agent)

    **Step 4b: Merger Agent** (NEW)
    - Spawn merger agent via Agent tool with:
      - Set context (CONTEXT.md, plans)
      - Contexts of other sets already merged in this wave
      - Detection report (L1-L4 results)
      - Unresolved conflicts from resolution cascade
      - Interface contracts for the set
    - Parse RAPID:RETURN from agent:
      - semantic_conflicts: display any L5 findings
      - resolutions: show how many resolved with what confidence
      - escalations: show conflicts needing human input
    - For each escalation (confidence < threshold):
      > Conflict in {file}: {description}
      > Confidence: {score} (below threshold {threshold})

      AskUserQuestion:
      - "Accept AI resolution" -- description: "Apply the AI's resolution despite low confidence"
      - "Resolve manually" -- description: "Open the file and resolve this conflict yourself"
      - "Skip conflict" -- description: "Leave conflict unresolved and continue"

    **Step 5: Programmatic Gate** (preserved from v1.0)
    - Run `node "${RAPID_TOOLS}" merge review {set}`
    - If passed: proceed to merge
    - If failed (ownership violations, contract failures): block with AskUserQuestion options

    **Step 6: Merge Set** (preserved from v1.0, enhanced with MERGE-STATE)
    - `node "${RAPID_TOOLS}" merge execute {set}`
    - If merged: show commit hash, update MERGE-STATE to complete
    - If conflict: AskUserQuestion with Resolve/Show diff/Abort options (same as v1.0)
    - If error: AskUserQuestion with Retry/Skip/Abort

    **Step 7: Post-Wave Integration Gate** (preserved from v1.0, enhanced with auto-bisection)
    - After all sets in wave merged, run `node "${RAPID_TOOLS}" merge integration-test`
    - If passed: show gate pass, continue to next wave
    - If failed: **auto-trigger bisection** (per user decision: "triggers automatically on post-wave integration gate failure")
      - Display: "Integration tests failed. Running bisection recovery automatically..."
      - `node "${RAPID_TOOLS}" merge bisect {waveNum}`
      - Display result: "Breaking set: {set} (found in {n} iterations)"
      - THEN present options via AskUserQuestion:
        - "Rollback breaking set" -- description: "Revert {set}'s merge commit"
        - "Investigate manually" -- description: "Pause for manual investigation of the breaking set"
        - "Abort pipeline" -- description: "Exit without reverting"

    **Step 7b: Rollback** (NEW)
    - If rollback requested for a set:
      - `node "${RAPID_TOOLS}" merge rollback {set}`
      - Parse cascade impact. If hasCascade=true:
        > Dependent sets affected: {list}
        AskUserQuestion:
        - "Proceed with rollback" -- description: "Revert {set}, dependent sets may need manual fixing"
        - "Cancel rollback" -- description: "Keep the merge, investigate instead"
      - If reverted successfully: show revert commit, update status
      - Run integration tests again to verify rollback fixed the issue

    **Step 8: Pipeline Complete** (preserved from v1.0, enhanced)
    - Show final summary including detection/resolution stats from MERGE-STATE
    - AskUserQuestion for next steps: cleanup, status, done

    **Key conventions:**
    - RAPID_ROOT and RAPID_TOOLS loading pattern (same as v1.0 SKILL.md)
    - Progress banners with wave/set context
    - All AskUserQuestion gates are blocking
    - Merger agent is the ONLY subagent spawned (reviewer subagent from v1.0 is REMOVED -- review is now done in Phase 22's review module)
    - Never use git add -A or git add .
    - Idempotent re-entry: check MERGE-STATE.json to skip already-merged sets
    - Bisection runs automatically on integration gate failure -- NO pre-bisection user prompt (user locked decision)
