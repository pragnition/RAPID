<task type="auto">
  <name>Task 1: Rewrite execute SKILL.md for Mark II job-level execution</name>
  <files>skills/execute/SKILL.md</files>
  <action>
    Rewrite `skills/execute/SKILL.md` completely. Use the v1.0 skill as structural inspiration but replace all set-level logic with job-level logic. The skill takes `<set-id>` as input and executes all waves in that set sequentially.

    **Frontmatter:**
    ```yaml
    ---
    description: Execute jobs within waves -- dispatches parallel subagents per job, tracks progress, reconciles per wave
    allowed-tools: Read, Write, Bash, Agent, AskUserQuestion
    ---
    ```

    **Structure the skill as sequential steps:**

    **Step 0: Environment + Precondition Check**
    - Load RAPID_TOOLS env (same .env fallback pattern as v1.0 skill)
    - Parse `<set-id>` argument from the user's command. If not provided, use AskUserQuestion to ask which set to execute
    - Read STATE.json to find the set and its waves: `node "${RAPID_TOOLS}" state get set <milestone> <set-id>`
    - Verify at least one wave has JOB-PLAN.md files: `node "${RAPID_TOOLS}" wave-plan list-jobs <set-id> <wave-id>` for each wave
    - If no JOB-PLAN.md files exist for ANY wave, inform user: "No job plans found. Run /rapid:discuss and /rapid:wave-plan first." Use AskUserQuestion with "Run discuss" / "Cancel" options. Do NOT auto-trigger those commands.

    **Step 1: Detect Execution Mode**
    - Same agent teams detection as v1.0: `node "${RAPID_TOOLS}" execute detect-mode`
    - AskUserQuestion if agent teams available (same options as v1.0)
    - Mode locked for entire run

    **Step 2: Smart Re-entry**
    - Read all job statuses: `node "${RAPID_TOOLS}" execute job-status <set-id>`
    - Parse JSON output. For each wave, classify jobs:
      - `complete` -> skip (show "already complete")
      - `failed` -> re-execute (show "retrying, previously failed")
      - `executing` -> treat as failed/stale (show "stale state, re-executing")
      - `pending` -> normal execution
    - Show summary of what will be executed vs skipped
    - If ALL jobs are complete, inform user and exit
    - Use AskUserQuestion: "Begin execution" / "Cancel"

    **Step 3: Process Each Wave (Sequential)**
    For each wave in order (Wave 1, Wave 2, etc.):

    **Step 3a: Transition wave to 'executing'**
    ```bash
    node "${RAPID_TOOLS}" state transition wave <milestone> <set-id> <wave-id> executing
    ```

    **Step 3b: Load job plans for this wave**
    ```bash
    node "${RAPID_TOOLS}" wave-plan list-jobs <set-id> <wave-id>
    ```
    Read each JOB-PLAN.md file.

    **Step 3c: Print initial progress banner**
    ```
    --- RAPID Execute ---
    Wave {waveId} (1/N)
      {jobId}: Pending
      {jobId}: Pending
    [HH:MM]
    ---------------------
    ```

    **Step 3d: Dispatch parallel jobs**

    *Subagent mode:* For each job that needs execution (pending, failed, stale-executing):
    1. Transition job to 'executing': `node "${RAPID_TOOLS}" state transition job <m> <s> <w> <j> executing`
    2. Print updated progress showing this job as "Executing"
    3. Spawn subagent using Agent tool with job executor prompt:

    **Job executor subagent prompt template:**
    ```
    You are implementing job '{jobId}' in set '{setId}'.

    ## Your JOB-PLAN
    {Full content of {jobId}-PLAN.md}

    ## File Ownership
    You may ONLY modify these files:
    {File list from WAVE-PLAN.md or JOB-PLAN.md file assignments}

    ## Commit Convention
    After each implementation step, commit with:
      git add <specific files>
      git commit -m "type({setId}): description"
    Where type is feat|fix|refactor|test|docs|chore

    ## Working Directory
    {worktreePath -- from the set's worktree}

    ## Completion
    When all steps complete, emit RAPID:RETURN with status COMPLETE.
    If context window limit reached, emit CHECKPOINT with handoff details.
    If blocked, emit BLOCKED with category.
    ```

    All jobs in a wave are spawned in parallel (multiple Agent tool calls). If rate limits are hit (Agent tool errors with rate-limit-like messages), fall back to sequential execution within the wave and inform the user.

    *Agent teams mode:* Same prompt but dispatched via agent teams API. One team per wave (`rapid-{setId}-{waveId}`), one teammate per job. If teams fail, fall back to subagent mode for the entire wave with a visible warning.

    **Step 3e: Collect returns and transition jobs**
    After each subagent returns:
    - Parse the RAPID:RETURN from the output
    - If COMPLETE: transition job to 'complete', print updated progress
    - If CHECKPOINT: generate job handoff file at `.planning/waves/{setId}/{waveId}/{jobId}-HANDOFF.md`, transition job status stays as-is (the skill notes it as paused)
    - If BLOCKED: transition job to 'failed', print updated progress with error detail
    - If no RAPID:RETURN marker: transition job to 'failed', log warning about unparseable return

    **Step 3f: Commit STATE.json after all jobs in wave resolve**
    ```bash
    node "${RAPID_TOOLS}" execute commit-state "chore({setId}): complete wave {waveId} execution"
    ```

    **Step 3g: Reconcile wave**
    ```bash
    node "${RAPID_TOOLS}" execute reconcile-jobs <set-id> <wave-id> --branch main --mode "{executionMode}"
    ```
    Parse JSON output for overall result (PASS/PASS_WITH_WARNINGS/FAIL).

    **Step 3h: Transition wave status**
    ```bash
    node "${RAPID_TOOLS}" state transition wave <milestone> <set-id> <wave-id> reconciling
    ```
    Then after reconciliation display:
    ```bash
    node "${RAPID_TOOLS}" state transition wave <milestone> <set-id> <wave-id> complete
    ```
    (If reconciliation FAIL, leave as 'reconciling' for retry)

    **Step 3i: User decision after wave reconciliation**
    Use AskUserQuestion with dynamic options based on result:

    If PASS:
    - "Continue to next wave" / "Pause here"

    If PASS_WITH_WARNINGS:
    - "Continue anyway" / "Retry failed jobs" / "Pause"

    If FAIL:
    - "Retry failed jobs" / "Cancel execution"

    If "Retry failed jobs": re-execute only failed jobs in this wave (go back to Step 3d for those jobs)
    If "Pause": commit state, print resume instructions, exit
    If "Cancel": commit state, exit

    **Step 4: Final Summary**
    After all waves complete:
    ```
    --- RAPID Execute Complete ---
    Set: {setId}
    Waves: {completed}/{total}
    Jobs: {completed}/{total}
    Execution mode: {mode}
    --------------------------------
    ```
    Use AskUserQuestion:
    - "View status" / "Start review" / "Done"

    **Important Notes section** (at bottom of SKILL.md):
    - Agent tool spawns subagents in isolated context windows. Subagents cannot spawn sub-subagents.
    - Rate limiting: if parallel spawns cause errors, reduce to sequential within the wave.
    - Smart re-entry: on every invocation, read STATE.json first. Skip complete, retry failed, re-execute stale.
    - File ownership enforcement: each job only modifies files from its JOB-PLAN.md. Violations are caught in reconciliation.
    - Git commits: parallel job agents commit to the same branch. Each stages only its own files. If commit fails with HEAD race, retry once.
    - STATE.json committed at workflow boundaries (after each wave completes), not on every job transition.
    - Existing role-executor.md stays for v1.0 backward compatibility. New job-level execution uses role-job-executor.md.
    - Discuss and plan steps are NOT part of this skill. If missing, prompt user to run /rapid:discuss and /rapid:wave-plan.
