  <action>
Create the skills/review/ directory and write SKILL.md as the /rapid:review orchestrator. Follow the same structural patterns as skills/execute/SKILL.md (frontmatter, numbered steps, AskUserQuestion at every gate, CLI commands for state and data access).

**Frontmatter:**
```yaml
---
description: Review completed waves/sets -- orchestrates unit test, bug hunt, and UAT pipeline
allowed-tools: Read, Write, Bash, Agent, AskUserQuestion
---
```

**Skill structure (follow this exact order):**

## Step 0: Environment + Set Resolution

### 0a: Load environment
- Load RAPID_TOOLS from .env or environment (same pattern as execute SKILL.md)
- Resolve project root

### 0b: Parse arguments
- Accept: `/rapid:review <set-id>` (all waves) or `/rapid:review <set-id> <wave-id>` (specific wave)
- Validate set exists in STATE.json: `node "${RAPID_TOOLS}" state get`
- Validate set status is 'executing' or 'reviewing' (cannot review a set that hasn't been executed)

### 0c: Transition set to 'reviewing'
- If set is in 'executing' state, transition: `node "${RAPID_TOOLS}" state transition set <milestone> <set-id> reviewing`
- If already 'reviewing', skip transition (idempotent re-entry)

## Step 1: Stage Selection

Use AskUserQuestion to let user choose review stages:
- **question:** "Which review stages to run for set '{set-id}'?"
- **options:**
  - "All stages" -- description: "Run unit test, bug hunt, and UAT in order (recommended for first review)"
  - "Unit test only" -- description: "Generate test plan, write tests, run and report"
  - "Bug hunt only" -- description: "Static analysis with adversarial verification (hunter/advocate/judge)"
  - "UAT only" -- description: "Acceptance testing with browser automation"
  - "Unit test + Bug hunt" -- description: "Testing and bug hunting, skip UAT"
  - "Bug hunt + UAT" -- description: "Bug hunting and acceptance testing, skip unit tests"

Record selected stages as a list.

## Step 2: Resolve Waves

Determine which waves to review:
- If user specified a wave-id, review only that wave
- Otherwise, list all waves for the set: `node "${RAPID_TOOLS}" state get` and iterate through set's waves
- Only review waves in 'complete' or 'reconciling' status (skip pending/executing waves)

## Step 3: For Each Wave (Sequential)

For each wave, compute review scope:
```bash
node "${RAPID_TOOLS}" review scope <set-id> <wave-id>
```

Print banner:
```
--- RAPID Review ---
Set: {setId}
Wave: {waveId} ({index}/{total})
Scope: {totalFiles} files ({changedFiles} changed + {dependentFiles} dependents)
Stages: {selected stages}
--------------------
```

### Step 3a: Unit Test Stage (if selected)

1. Read JOB-PLAN.md files for this wave to extract acceptance criteria:
   ```bash
   node "${RAPID_TOOLS}" wave-plan list-jobs <set-id> <wave-id>
   ```
   Read each JOB-PLAN.md file.

2. Spawn unit-tester subagent via Agent tool. Pass:
   - Scoped file list (from review scope)
   - Acceptance criteria extracted from JOB-PLAN.md files
   - Instruction to generate test plan first (CHECKPOINT), then write tests after approval

3. Parse RAPID:RETURN from subagent. If status=CHECKPOINT (test plan ready):
   - Display the test plan to the user
   - Use AskUserQuestion:
     - **question:** "Approve test plan for wave {waveId}?"
     - **options:**
       - "Approve" -- description: "Proceed with writing and running tests"
       - "Modify" -- description: "Describe changes to the test plan"
       - "Skip unit tests" -- description: "Skip testing for this wave"
   - If "Approve": re-invoke the subagent with approval signal to proceed
   - If "Modify": re-invoke with user feedback
   - If "Skip": move to next stage

4. After tests complete, parse COMPLETE return data: { testsWritten, testsPassed, testsFailed, output, testFiles }
5. Write REVIEW-UNIT.md to `.planning/waves/{setId}/{waveId}/REVIEW-UNIT.md` with test results
6. Log any test failures as issues: `echo '<issue-json>' | node "${RAPID_TOOLS}" review log-issue <set-id> <wave-id>`

### Step 3b: Bug Hunt Stage (if selected)

Run the adversarial pipeline with 3-cycle bugfix iteration:

**Cycle loop (max 3 iterations):**

For cycle = 1 to 3:

1. **Determine scope:**
   - Cycle 1: full review scope (all changed files + dependents)
   - Cycle 2+: narrowed to only files modified by previous bugfix cycle

2. **Spawn bug-hunter subagent.** Pass scoped file list with explicit instruction: "Only report bugs in these files." Parse COMPLETE return: { findings, totalFindings }.

3. **If no findings:** break the cycle loop (codebase is clean). Print: "Bug hunt cycle {N}: no findings. Codebase clean."

4. **Spawn devils-advocate subagent.** Pass hunter findings + scoped file list. The advocate is read-only and challenges each finding with evidence. Parse COMPLETE return: { assessments }.

5. **Spawn judge subagent.** Pass hunter findings + advocate assessments. Judge produces rulings: ACCEPTED, DISMISSED, DEFERRED. Parse COMPLETE return: { rulings, accepted, dismissed, deferred }.

6. **Handle DEFERRED rulings.** For each DEFERRED ruling, present hunter evidence and advocate evidence to the developer:
   - Use AskUserQuestion per finding:
     - **question:** "Finding {id}: {description}"
     - Show hunter evidence and advocate counter-evidence
     - **options:**
       - "Accept (real bug)" -- description: "Bug is real, should be fixed"
       - "Dismiss (false positive)" -- description: "Not a real bug, skip it"
       - "Defer (fix later)" -- description: "Log for later review"

7. **Write REVIEW-BUGS.md** to `.planning/waves/{setId}/{waveId}/REVIEW-BUGS.md` with all rulings from all cycles.

8. **Collect ACCEPTED bugs.** If none: break (clean). If accepted bugs exist:
   - Spawn bugfix subagent with accepted bugs array
   - Parse COMPLETE return: { fixed, unfixable, modifiedFiles }
   - Log unfixable bugs as issues
   - Record modifiedFiles for scope narrowing in next cycle
   - Update issue status for fixed bugs

9. **After cycle 3 with remaining unfixed bugs:**
   Present remaining bugs to user:
   - Use AskUserQuestion per bug:
     - "Fix manually" -- description: "You will fix this yourself"
     - "Defer" -- description: "Log and move on"
     - "Dismiss" -- description: "Not worth fixing"
   - Update issue statuses accordingly

### Step 3c: UAT Stage (if selected)

1. Read JOB-PLAN.md acceptance criteria and WAVE-CONTEXT.md decisions for test scenario derivation.

2. Determine browser automation tool:
   - Check `.planning/config.json` for `browserAutomation` field
   - If not set, use AskUserQuestion:
     - **question:** "Which browser automation tool to use for UAT?"
     - **options:**
       - "Chrome DevTools MCP" -- description: "Use Chrome DevTools Protocol (default, recommended)"
       - "Playwright MCP" -- description: "Use Playwright for browser automation"
       - "Skip automated steps" -- description: "Run only human-verified steps"

3. Spawn UAT subagent via Agent tool. Pass:
   - Acceptance criteria from JOB-PLAN.md
   - User decisions from WAVE-CONTEXT.md
   - Browser automation tool choice
   - Instruction to generate test plan first (CHECKPOINT)

4. Parse CHECKPOINT return (test plan with automated/human tagged steps). Present to user:
   - Use AskUserQuestion:
     - "Approve" -- description: "Run the test plan as tagged"
     - "Modify tags" -- description: "Change which steps are automated vs human"
     - "Skip UAT" -- description: "Skip acceptance testing for this wave"

5. After approval, re-invoke UAT subagent to execute the plan:
   - [automated] steps: executed via configured MCP tool
   - [human] steps: subagent returns CHECKPOINT with description of what to verify. Display to user with AskUserQuestion: "Pass" / "Fail (describe issue)"

6. Write REVIEW-UAT.md to `.planning/waves/{setId}/{waveId}/REVIEW-UAT.md`
7. Log failed UAT steps as issues

## Step 4: Generate Review Summary

After all waves have been reviewed:

```bash
node "${RAPID_TOOLS}" review summary <set-id>
```

Print the summary to the user:
```
--- RAPID Review Complete ---
Set: {setId}
Waves reviewed: {N}
Unit tests: {passed} passed, {failed} failed
Bug hunt: {accepted} accepted, {dismissed} dismissed, {deferred} deferred
UAT: {passed} passed, {failed} failed, {skipped} skipped
Open issues: {count}

Review artifacts:
  .planning/waves/{setId}/REVIEW-SUMMARY.md
  .planning/waves/{setId}/{waveId}/REVIEW-UNIT.md
  .planning/waves/{setId}/{waveId}/REVIEW-BUGS.md
  .planning/waves/{setId}/{waveId}/REVIEW-UAT.md
-----------------------------
```

## Step 5: Next Action

Use AskUserQuestion:
- **question:** "Review complete for set '{set-id}'. What's next?"
- **options:**
  - "Proceed to merge" -- description: "Set is ready. Run /rapid:merge {set-id}"
  - "Re-run review" -- description: "Run another review cycle on this set"
  - "Fix issues first" -- description: "Run /rapid:execute {set-id} --fix-issues to address remaining issues"
  - "Done for now" -- description: "Exit. Review artifacts saved."

**Important Notes section (at bottom):**
- This skill uses the Agent tool to spawn subagents. Each subagent runs in its own context. Subagents cannot spawn sub-subagents.
- The bug hunt adversarial pipeline is the quality gate -- the hunter finds, the advocate challenges, the judge rules. This is not a rubber-stamp process.
- DEFERRED rulings ALWAYS require human input. The skill pauses and presents evidence from both sides.
- Iteration limit is 3 bugfix cycles. After 3 cycles, remaining bugs go to the user.
- Re-hunts (cycles 2+) narrow scope to only files the bugfix agent modified in the previous cycle. This prevents infinite loops.
- UAT browser automation depends on MCP tool availability. If the tool is not available, automated steps should be converted to human steps.
- Review state is set-level: the set transitions executing -> reviewing. Wave states are unchanged.
- All review artifacts are committed at the end alongside STATE.json.
