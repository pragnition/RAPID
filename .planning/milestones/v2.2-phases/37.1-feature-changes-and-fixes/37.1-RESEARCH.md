# Phase 37.1: Feature Changes and Fixes - Research

**Researched:** 2026-03-11
**Domain:** Claude Code plugin skill/agent architecture, UX workflow redesign, new command implementation
**Confidence:** HIGH

## Summary

Phase 37.1 is a brownfield refactoring and feature addition phase for the RAPID Claude Code plugin (v2.x). The codebase is a Node.js CJS module system with 18 skills (in `skills/*/SKILL.md`), 31 agents (in `agents/*.md` built from `src/modules/`), and a CLI toolchain (`src/bin/rapid-tools.cjs` backed by `src/lib/*.cjs`). All changes target Markdown skill files, Markdown agent modules, and JavaScript source files -- no new dependencies are needed.

The phase encompasses 7 distinct work areas: (1) hiding waves behind set-level UX, (2) redesigning the discussion flow to single-round set-level, (3) adding `/migrate` command, (4) adding `/quick` command, (5) improving workflow awareness after init, (6) fixing tool-calling reliability in agents, and (7) ensuring STATE.md is updated after execute completes. Each area touches different files but some share integration points (e.g., `/rapid:help` and next-step suggestions are affected by multiple changes).

**Primary recommendation:** Decompose into 3-4 waves ordered by dependency: Wave 1 covers foundational changes (discussion redesign, wave hiding in status/help, workflow awareness banner, state update after execute), Wave 2 covers new commands (/migrate, /quick), and Wave 3 covers the tool-calling audit and cross-cutting cleanup. Planning optimization can be folded into whichever wave touches `plan-set/SKILL.md`.

## User Constraints (from CONTEXT.md)

### Locked Decisions

#### Workflow Restructuring (Waves Invisible to Users)
- Waves become an internal implementation detail -- users interact at the **set** level only
- New user-facing lifecycle: init -> set-init -> discuss -> plan -> execute -> review -> merge -> cleanup
- `/rapid:init` now also decomposes the project into sets (absorbs part of what set-init planned)
- `/rapid:set-init` stays for worktree creation and scoped CLAUDE.md generation only

- Discuss outputs per-wave WAVE-CONTEXT.md files internally (splits decisions to relevant waves)
- `/rapid:plan` means plan-set -- plans all waves at once; `/rapid:wave-plan` becomes internal-only

- `/rapid:status` default view shows sets only; `--detail` flag expands to show wave-level breakdown

- Single round per gray area (one question with concrete options, no Round 2 follow-up)
- "Looks good" appears only once at the very end, after ALL gray areas are done
- No "Revise" option -- the built-in "Other" freeform handles revisions
- At the final confirmation: no summary shown, just ask "Looks good / Revise"
- If user selects "Revise" at the end, they provide freeform text describing what to change (not a multi-select of areas)
- Discussion focuses on **product/UX decisions and high-level developer preferences** -- not technical implementation details

- After `/rapid:init` completes, show the full lifecycle as a simple numbered list:
  1. set-init  2. discuss  3. plan  4. execute  5. review  6. merge  7. cleanup

- No visual flair -- flat numbered list format

- Heuristic detection: scan for planning directories and patterns (STATE.md, ROADMAP.md, phases/), adapt to any framework structure with user confirmation at each step

### Pattern 1: Skill File Structure
**What:** Every RAPID command is a Markdown file in `skills/{name}/SKILL.md` with YAML frontmatter and ordered steps.
**When to use:** All new commands (/migrate, /quick) and all modified commands.
**Example:**
```markdown
---
description: Short description of what the command does
allowed-tools: Bash(rapid-tools:*), Read, Write, AskUserQuestion, Agent
---

# /rapid:{name} -- Human-Readable Title

## Environment Setup
[standard env preamble]

## Display Stage Banner
[standard banner display]

## Step N: [Action]
[implementation steps]
```

### Pattern 2: CLI Command Registration
**What:** New CLI commands are registered in `rapid-tools.cjs` dispatch and backed by library modules in `src/lib/`.
**When to use:** /migrate and /quick both need new CLI subcommands.
**Example flow:**
1. Create `src/lib/migrate.cjs` with business logic
2. Create `src/lib/migrate.test.cjs` with unit tests
3. Register `migrate` command in `rapid-tools.cjs` dispatch
4. Create `skills/migrate/SKILL.md` that calls `node "${RAPID_TOOLS}" migrate <subcommand>`

### Pattern 3: State Schema Extension
**What:** State is validated by Zod schemas in `state-schemas.cjs`. Any new fields must be added there.
**When to use:** Quick-task tracking needs new fields in STATE.json.
**Example:** Add `quickTasks` array to ProjectState schema with counter, description, date, commitHash fields.

### Pattern 4: Next-Step Suggestion Protocol
**What:** Every skill ends with a `> **Next step:** /rapid:{command} {args}` suggestion using numeric shorthand.
**When to use:** When updating flow to remove wave-level suggestions.
**Current references to update:**

- `set-init/SKILL.md` line 171: currently suggests `/rapid:discuss {setIndex}.1` -- must change to `/rapid:discuss {setIndex}`

### Anti-Patterns to Avoid
- **Do NOT create new agent files directly.** Edit role modules in `src/modules/roles/` and rebuild with `build-agents`.
- **Do NOT modify STATE.json directly in skills.** Always use `node "${RAPID_TOOLS}" state transition` CLI.
- **Do NOT add new npm dependencies.** The migration and quick-task features are implementable with Node.js built-ins.
- **Do NOT break the env preamble pattern.** Every Bash command in a skill must start with the RAPID_ROOT/RAPID_TOOLS loading block.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| State file locking | Manual lockfile | `proper-lockfile` via `acquireLock()` | Race conditions with parallel agents |
| State validation | Manual JSON checks | `zod` schemas in `state-schemas.cjs` | Existing pattern, type safety |
| Set/wave resolution | Manual directory listing | `resolve.cjs` functions | Handles numeric/string/dot-notation |
| Framework detection in /migrate | Hard-coded patterns | Heuristic scanner with user confirmation | Must be framework-agnostic per decision |
| Agent file generation | Manual .md editing | `build-agents` CLI from role modules | Source of truth is `src/modules/` |

**Key insight:** The existing CLI toolchain (`rapid-tools.cjs`) provides a comprehensive API surface. New features should add subcommands to this existing CLI rather than creating separate scripts.

## Common Pitfalls

### Pitfall 1: Skill Referencing Wave-Level Commands in Next Steps
**What goes wrong:** After restructuring, skills still suggest `/rapid:wave-plan` or `/rapid:discuss {setIndex}.{waveIndex}` to users.
**Why it happens:** 8+ skill files contain next-step suggestions, and it is easy to miss one.
**How to avoid:** Systematic grep for all `rapid:wave-plan` and `rapid:discuss.*\.` references across all `skills/*/SKILL.md` files. Replace every instance.
**Warning signs:** User sees wave-plan or wave-level discuss suggestions after the changes.

### Pitfall 2: Discussion Skill Scope Change (Wave -> Set)
**What goes wrong:** The discuss skill currently resolves a single wave, reads wave-level artifacts, and writes a single WAVE-CONTEXT.md. Changing to set-level means it must: resolve a set, read all waves in the set, conduct a unified discussion, then split decisions into per-wave WAVE-CONTEXT.md files.
**Why it happens:** The current skill architecture is deeply wave-scoped (Steps 2-7 all reference single wave IDs).
**How to avoid:** Restructure discuss/SKILL.md from scratch rather than patching. The input resolution (Step 2) changes from `resolve wave` to `resolve set`. The context gathering (Step 3) must iterate all waves. The output (Step 7) must write N WAVE-CONTEXT.md files.
**Warning signs:** Discuss produces a single WAVE-CONTEXT.md when there are multiple waves in the set.

### Pitfall 3: State Transition Compatibility
**What goes wrong:** The discuss skill currently transitions a single wave from `pending -> discussing`. When operating at set level, all waves in the set need to transition.
**Why it happens:** The state machine `transitionWave()` operates on one wave at a time.
**How to avoid:** Call `transitionWave()` in a loop for all waves in the set. The existing state machine already supports this -- no schema changes needed. But the set transition to `planning` must only happen after ALL waves are discussed.
**Warning signs:** Some waves stay in `pending` after set-level discuss completes.

### Pitfall 4: /quick Command State Schema Addition
**What goes wrong:** Adding `quickTasks` to STATE.json requires a schema change in `state-schemas.cjs`. Existing STATE.json files without this field will fail validation.
**Why it happens:** Zod strict parsing rejects unknown fields, and missing required fields fail.
**How to avoid:** Make `quickTasks` an optional field with `.optional().default([])` in the schema. Existing STATE.json files without it will parse correctly.
**Warning signs:** `readState()` returns `{ valid: false }` after the schema update on existing projects.

### Pitfall 5: /migrate Heuristic Detection False Positives
**What goes wrong:** The migrate command detects a `.planning/` directory but it belongs to an unrelated system, or detects `phases/` that are not from GSD/openspec.
**Why it happens:** Heuristic detection by file patterns can match unrelated project structures.
**How to avoid:** Per the locked decision, require user confirmation at each step. The detection presents findings and asks the user to confirm before any modifications. Backup to `.planning.bak/` before any changes.
**Warning signs:** /migrate silently overwrites non-RAPID planning files.

### Pitfall 6: Agent Tool-Calling Env Preamble
**What goes wrong:** Agents spawned by skills fail to run `node "${RAPID_TOOLS}" ...` commands because RAPID_TOOLS env var is not set in their execution context.
**Why it happens:** Skills set RAPID_TOOLS in their own Bash environment, but spawned agents start fresh. The agent's `core-state-access.md` module has the preamble, but some agents may not include that core module, or the CLAUDE_SKILL_DIR-based fallback may not resolve correctly in agent context.
**How to avoid:** Audit all 31 agents to verify they include `core-state-access` module. For agents that need Bash tool access but don't have it, add it to their `tools:` frontmatter. For agents that have Bash but no env loading, ensure their core module includes the standard preamble.
**Warning signs:** Agents emit `[RAPID ERROR] RAPID_TOOLS is not set` during execution.

### Pitfall 7: Help Command is Static (disable-model-invocation)
**What goes wrong:** The help skill has `disable-model-invocation: true` in its frontmatter -- it outputs static text without any intelligence. Updating help means literally changing the text content.
**Why it happens:** Help was designed as a static reference card.
**How to avoid:** Simply update the literal text in `skills/help/SKILL.md`. No logic changes needed. But ensure the static text is 100% accurate -- no model will catch errors at runtime.
**Warning signs:** /help shows outdated commands or missing new commands.

## Open Questions

1. **Planning optimization specifics**
   - What we know: Planning (plan-set) takes significant time and tokens. The user mentioned thinking of optimizations.
   - What's unclear: Which specific optimizations are most impactful -- reducing agent count, shortening prompts, caching research, skipping verification for simple waves?
   - Recommendation: The planner should analyze the plan-set pipeline and identify the highest-token-cost steps. Likely candidates: wave research (can be made optional for simple waves), job planner prompt size (reduce context duplication), verification (skip for single-job waves). This is under Claude's discretion per CONTEXT.md.

2. **Agent tool-calling audit scope**
   - What we know: Tool calling is "very patchy" per todo.md. Some agents may lack Bash tool access or have incorrect env preambles.
   - What's unclear: Which specific agents fail and in which scenarios. The audit needs to be systematic.
   - Recommendation: The planner should categorize agents by their tool needs: (a) agents that need Bash for RAPID_TOOLS (state access), (b) agents that are pure Read/Write (no CLI needed), (c) agents that need Bash for git operations. Verify each category has correct `tools:` frontmatter and correct core modules. This is under Claude's discretion per CONTEXT.md.

3. **STATE.md vs STATE.json update after execute**
   - What we know: STATE.md is a human-readable Markdown file that summarizes project state. STATE.json is the machine-readable state file. The execute skill commits STATE.json but STATE.md is a separate artifact.
   - What's unclear: Whether "STATE.md should be updated" means the Markdown file or the JSON file. The existing `execute commit-state` CLI only commits STATE.json.
   - Recommendation: Add a step to the execute skill's final summary (Step 4) that updates STATE.md with the execution results. This ensures the human-readable state reflects completion even if /rapid:review is not run afterward.
