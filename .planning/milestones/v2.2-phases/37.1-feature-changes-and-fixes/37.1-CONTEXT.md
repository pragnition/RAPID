<decisions>
## Implementation Decisions

### Workflow Restructuring (Waves Invisible to Users)
- Waves become an internal implementation detail — users interact at the **set** level only
- New user-facing lifecycle: init → set-init → discuss → plan → execute → review → merge → cleanup
- `/rapid:init` now also decomposes the project into sets (absorbs part of what set-init planned)
- `/rapid:set-init` stays for worktree creation and scoped CLAUDE.md generation only
- `/rapid:discuss` operates at **set level** (not wave level), replacing the old per-wave discussion
- Discuss outputs per-wave WAVE-CONTEXT.md files internally (splits decisions to relevant waves)
- `/rapid:plan` means plan-set — plans all waves at once; `/rapid:wave-plan` becomes internal-only
- `/rapid:wave-plan` and wave-level discuss commands **removed entirely** as user-facing commands (not in /help, not in next-step suggestions)
- `/rapid:status` default view shows sets only; `--detail` flag expands to show wave-level breakdown

### Discussion Flow Redesign
- Single round per gray area (one question with concrete options, no Round 2 follow-up)
- "Looks good" appears only once at the very end, after ALL gray areas are done
- No "Revise" option — the built-in "Other" freeform handles revisions
- At the final confirmation: no summary shown, just ask "Looks good / Revise"
- If user selects "Revise" at the end, they provide freeform text describing what to change (not a multi-select of areas)
- Discussion focuses on **product/UX decisions and high-level developer preferences** — not technical implementation details

### Workflow Awareness
- After `/rapid:init` completes, show the full lifecycle as a simple numbered list:
  1. set-init  2. discuss  3. plan  4. execute  5. review  6. merge  7. cleanup
- Only shown after project-level init (not after set-init or other commands)
- No visual flair — flat numbered list format

### /migrate Command
- Full migration: state, roadmap, plans, context files, and directory restructuring to RAPID conventions
- Heuristic detection: scan for planning directories and patterns (STATE.md, ROADMAP.md, phases/), adapt to any framework structure with user confirmation at each step
- Backup before migrate: copy old planning directory to .planning.bak/ before restructuring
- After migration completes, automatically run /rapid:status to show the project in RAPID format

### /quick Command
- Works directly on current branch (no worktree isolation)
- Tracked in RAPID state (STATE.md) with quick-task counter, description, and commit hash
- No guardrails on task size — trust the developer
- Auto-commit when task is done with descriptive commit message

### Claude's Discretion
- Tool calling audit: which agents need fixes and how to correct them (planner determines scope)
- Planning optimization: approach for reducing token usage during planning (planner decides)
- State update after execute: how to ensure STATE.md is updated when execute completes even if review doesn't run (planner decides)

</decisions>

<specifics>
## Specific Ideas

- "Waves should basically be invisible to the user" — the set is the primary unit of interaction
- "Discuss should discuss application details and not technical details" — product/UX focus
- Discussion should only suggest "looks good" at the final question, not per-area
- No "revise" option that just prompts for revision text — let the "Other" function handle freeform input directly

</specifics>
