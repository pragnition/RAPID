    **3. Update role-planner.md for project-level set decomposition:**

    Replace the current role-planner.md content with an expanded version that instructs the planner subagent on how to decompose a project into sets. The updated module should cover:

    **Role description:**
    - You decompose project work into parallelizable sets -- independent workstreams that different developers or Claude instances can work on simultaneously
    - Each set goes through its own full discuss/plan/execute lifecycle in an isolated worktree
    - Your output is a structured JSON proposal that the `/rapid:plan` skill will persist

    **Responsibilities:**
    - Analyze REQUIREMENTS.md to identify natural boundaries (modules, features, layers)
    - Analyze codebase structure (from brownfield detection) to respect existing architecture
    - Propose sets with explicit boundaries, file ownership, and acceptance criteria
    - Define interface contracts (JSON Schema) specifying what each set exposes and consumes
    - Build dependency edges and wave assignments to maximize parallelism
    - Assign shared-file ownership to exactly one set, with CONTRIBUTIONS.json for cross-set needs

    **Set Decomposition Strategy:**
    - Start with requirements groups: cluster related requirements into candidate sets
    - Validate against codebase structure: does the proposed set boundary align with directory/module boundaries?
    - Check for shared resources: identify files needed by multiple sets (package.json, configs, shared types)
    - Assign shared resources to the most logical owner; create CONTRIBUTIONS.json entries for others
    - Look for natural layering: core/shared types set at wave 1, feature sets at wave 2, integration set at wave 3
    - Check set size: minimum 3-5 owned files and 2+ tasks. Merge tiny sets into neighbors

    **Contract Design Guidance:**
    - exports.functions: list every function this set will expose that other sets might consume
    - exports.types: list every data type/shape this set defines that other sets reference
    - imports.fromSets: list every other set this set depends on and which functions/types it consumes
    - behavioral.invariants: conditions that must always hold (e.g., "authenticate returns null, never throws, for invalid tokens")
    - behavioral.sideEffects: observable effects (e.g., "writes session to disk", "emits auth:login event")
    - Keep contracts at the interface level -- don't specify implementation details

    **Output Format:**
    Return a JSON array of set definition objects. Each object has:
    ```json
    {
      "name": "set-name-kebab-case",
      "scope": "What this set does in 1-3 sentences",
      "ownedFiles": ["src/module/**", "tests/module/**", "specific-file.js"],
      "tasks": [
        { "description": "Task description", "acceptance": "How to verify it's done" }
      ],
      "acceptance": ["Overall set acceptance criterion 1", "..."],
      "wave": 1,
      "parallelWith": ["other-set-name"],
      "contract": {
        "exports": {
          "functions": [{ "name": "fn", "file": "src/module/index.cjs", "params": [{"name": "p", "type": "string"}], "returns": "object" }],
          "types": [{ "name": "TypeName", "file": "src/module/types.cjs", "shape": { "type": "object", "properties": {} } }]
        },
        "imports": {
          "fromSets": [{ "set": "other-set", "functions": ["otherFn"], "types": ["OtherType"] }]
        },
        "behavioral": {
          "invariants": ["condition that must always hold"],
          "sideEffects": ["observable effect"]
        }
      },
      "contributions": [
        { "file": "package.json", "owner": "core-set", "intent": "Add express dependency", "section": "dependencies", "priority": 2 }
      ]
    }
    ```

    **Constraints:**
    - Never assign the same file to two sets (ownership must be exclusive)
    - Every dependency between sets must be an explicit edge (no implicit "oh, this set also needs that")
    - Contract exports must match what imports reference (if set A imports fn from set B, set B must export that fn)
    - No circular dependencies between sets (the DAG must be acyclic)
    - Prefer more small sets over fewer large ones, but not so small that contract overhead exceeds work
    - Every set must have at least one task with a testable acceptance criterion
