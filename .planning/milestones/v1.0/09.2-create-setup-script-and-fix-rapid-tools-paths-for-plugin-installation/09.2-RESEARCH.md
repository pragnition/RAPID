# Phase 09.2: Create Setup Script and Fix RAPID_TOOLS Paths - Research

**Researched:** 2026-03-05
**Domain:** Shell scripting, Claude Code plugin installation, environment variable persistence
**Confidence:** HIGH

## Summary

This phase creates a setup/install system for RAPID and removes all hardcoded fallback paths. The current codebase uses `${RAPID_TOOLS:-$HOME/RAPID/src/bin/rapid-tools.cjs}` (or the tilde variant `~/RAPID/...`) across 13 source files with 70+ occurrences. The user has decided to remove all fallbacks, replacing them with bare `${RAPID_TOOLS}`, and to create both a `setup.sh` script and a `/rapid:install` skill to ensure the env var is set.

A critical discovery from official Claude Code docs: `${CLAUDE_SKILL_DIR}` is a string substitution available in SKILL.md files that resolves to the skill's directory path. However, it is NOT an environment variable available in bash -- it is a text replacement performed before Claude reads the content. This means bash commands in SKILL.md files will see the substituted path literally. Core agent modules (`.md` files in `src/modules/core/`) do NOT have access to `${CLAUDE_SKILL_DIR}` since they are assembled separately. Therefore, `RAPID_TOOLS` as a bash environment variable remains the correct approach for all paths.

**Primary recommendation:** Create `setup.sh` at repo root that auto-detects plugin location, sets RAPID_TOOLS, persists to shell config, runs npm install, and validates. Create `/rapid:install` skill that wraps setup.sh. Update all 13 files to use bare `${RAPID_TOOLS}` with no fallback.

### Pattern 1: Self-Locating Setup Script
**What:** setup.sh determines its own location using `dirname` and `readlink`, then derives RAPID_TOOLS path from it.
**When to use:** Always -- the script must work regardless of where it is invoked from.
**Example:**
```bash
#!/usr/bin/env bash
set -euo pipefail

# Resolve script's actual directory (handles symlinks)
SCRIPT_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"
RAPID_TOOLS_PATH="$SCRIPT_DIR/src/bin/rapid-tools.cjs"

# For marketplace installs, the structure is the same
# ~/.claude/plugins/cache/<hash>/src/bin/rapid-tools.cjs
# For git clone installs:
# ~/RAPID/src/bin/rapid-tools.cjs
```

### Pattern 2: Shell Config Detection and Persistence
**What:** Detect user's shell, append export to appropriate config file.
**When to use:** During setup to persist RAPID_TOOLS.
**Example:**
```bash
detect_shell_config() {
    case "$SHELL" in
        */bash)
            # Prefer .bashrc for interactive shells, .bash_profile for login
            if [[ -f "$HOME/.bashrc" ]]; then
                echo "$HOME/.bashrc"
            else
                echo "$HOME/.bash_profile"
            fi
            ;;
        */zsh)
            echo "$HOME/.zshrc"
            ;;
        */fish)
            echo "$HOME/.config/fish/config.fish"
            ;;
        *)
            echo "$HOME/.profile"
            ;;
    esac
}

persist_env_var() {
    local config_file="$1"
    local rapid_tools_path="$2"
    local shell_name
    shell_name="$(basename "$SHELL")"

    if [[ "$shell_name" == "fish" ]]; then
        local export_line="set -gx RAPID_TOOLS \"$rapid_tools_path\""
    else
        local export_line="export RAPID_TOOLS=\"$rapid_tools_path\""
    fi

    # Idempotent: check if already present
    if grep -qF "RAPID_TOOLS" "$config_file" 2>/dev/null; then
        echo "RAPID_TOOLS already configured in $config_file"
        return 0
    fi

    echo "" >> "$config_file"
    echo "# RAPID plugin for Claude Code" >> "$config_file"
    echo "$export_line" >> "$config_file"
}
```

### Pattern 5: Error Message on Missing RAPID_TOOLS
**What:** When `$RAPID_TOOLS` is not set, commands should fail with a clear error.
**When to use:** This pattern applies to the SKILL.md files where Claude reads bash code blocks and then runs them. Claude will see the error handling and execute it.
**Example:**
```bash
# Guard at top of SKILL.md bash blocks
if [[ -z "${RAPID_TOOLS:-}" ]]; then
    echo "ERROR: RAPID_TOOLS is not set."
    echo "Run setup: https://github.com/fishjojo1/RAPID#installation"
    echo "Or use: /rapid:install"
    exit 1
fi
```
**Important note:** This guard pattern needs to appear in each SKILL.md's first bash block. However, since Claude reads the SKILL.md and interprets the commands, a simpler approach is to change `node "${RAPID_TOOLS:-...}"` to `node "${RAPID_TOOLS}"` -- bash will error naturally with "RAPID_TOOLS: unbound variable" if `set -u` is active, or simply try to run `node ""` which will fail. A more explicit check can be added as a Step 0 in each skill.

### Anti-Patterns to Avoid
- **Hardcoded fallback paths:** `${RAPID_TOOLS:-$HOME/RAPID/...}` breaks on marketplace installs. Use bare `${RAPID_TOOLS}`.
- **$HOME in paths for node:** Per CLAUDE.md convention, use `~` not `$HOME` for shell commands. However, in env var exports, `$HOME` is correct since `~` is not expanded in all contexts.
- **Requiring terminal restart:** Set the var in the current session with `export RAPID_TOOLS=...` before persisting to config file.
- **Writing to shell config without checking:** Always check for existing RAPID_TOOLS entry first (idempotent).
- **Using readlink -f on macOS:** macOS `readlink` does not support `-f`. Use `cd ... && pwd` pattern instead.

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Plugin registration | Custom plugin registry | `claude plugin add .` | Official CLI command handles all registration |

## Common Pitfalls

### Pitfall 1: macOS readlink Compatibility
**What goes wrong:** `readlink -f` does not work on macOS (BSD readlink). Scripts that use it silently fail or produce wrong paths.
**Why it happens:** Linux uses GNU readlink which supports `-f` (canonicalize). macOS does not.
**How to avoid:** Use `SCRIPT_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"` instead.
**Warning signs:** Script works on Linux but fails on macOS.

### Pitfall 2: Fish Shell Syntax Differences
**What goes wrong:** `export VAR=value` is bash/zsh syntax. Fish uses `set -gx VAR value`. Writing bash syntax to `config.fish` breaks the user's shell.
**Why it happens:** Fish has a fundamentally different syntax from POSIX shells.
**How to avoid:** Detect fish shell and generate fish-compatible syntax: `set -gx RAPID_TOOLS "/path/to/rapid-tools.cjs"`.
**Warning signs:** Users report "broken shell" after running setup.

### Pitfall 3: RAPID_TOOLS Points to Wrong File
**What goes wrong:** Setting RAPID_TOOLS to the directory instead of the specific `rapid-tools.cjs` file, or to a non-existent path.
**Why it happens:** Confusion about what RAPID_TOOLS should point to.
**How to avoid:** RAPID_TOOLS must point to the full path of `src/bin/rapid-tools.cjs`. Validate the file exists before persisting.
**Warning signs:** `node "${RAPID_TOOLS}" prereqs` fails with "Cannot find module".

### Pitfall 4: Plugin Cache Path Changes Between Versions
**What goes wrong:** The marketplace plugin cache path (`~/.claude/plugins/cache/<hash>/`) includes a hash that changes when the plugin is updated. If RAPID_TOOLS is set to a specific cache path, it breaks on update.
**Why it happens:** Claude Code copies plugins to a versioned cache directory.
**How to avoid:** For marketplace installs, the setup.sh/install skill should set RAPID_TOOLS to the CURRENT cache path and note that it may need re-running after plugin updates. Alternatively, investigate whether there is a stable symlink or a way to reference the plugin by name.
**Warning signs:** Plugin works after install, breaks after update.
**Confidence:** MEDIUM -- Need to verify exact cache path structure.

### Pitfall 5: Overwriting Existing RAPID_TOOLS Configuration
**What goes wrong:** Re-running setup appends a second `export RAPID_TOOLS=...` line to shell config.
**Why it happens:** No check for existing configuration.
**How to avoid:** grep for `RAPID_TOOLS` in the config file before appending. If found, update in place or skip.

### Pitfall 6: SKILL.md Bash Blocks and Unset RAPID_TOOLS
**What goes wrong:** `node "${RAPID_TOOLS}"` when RAPID_TOOLS is empty resolves to `node ""` which gives an unhelpful error.
**Why it happens:** Without the fallback `:-...`, bash expands to empty string.
**How to avoid:** Two approaches: (1) Add a guard check as Step 0 in each SKILL.md, or (2) rely on Claude being intelligent enough to recognize the error and guide the user. Approach 1 is more robust. A single guard block at the start of each SKILL.md is recommended:
```bash
if [ -z "${RAPID_TOOLS}" ]; then echo "[RAPID ERROR] RAPID_TOOLS not set. Run /rapid:install or setup.sh"; exit 1; fi
```

## Code Examples

### setup.sh Structure
```bash
#!/usr/bin/env bash
# Source: Custom for RAPID, following Claude Code plugin patterns
set -euo pipefail

SCRIPT_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"
RAPID_TOOLS_PATH="$SCRIPT_DIR/src/bin/rapid-tools.cjs"

echo "=== RAPID Setup ==="
echo ""

# Step 1: Check prerequisites
echo "[1/5] Checking prerequisites..."
if ! command -v node &>/dev/null; then
    echo "  ERROR: Node.js is required (18+). Install from https://nodejs.org"
    exit 1
fi
NODE_VERSION=$(node -v | sed 's/v//' | cut -d. -f1)
if [[ "$NODE_VERSION" -lt 18 ]]; then
    echo "  ERROR: Node.js 18+ required, found v$(node -v)"
    exit 1
fi
echo "  OK: Node.js $(node -v)"

if ! command -v git &>/dev/null; then
    echo "  ERROR: git is required (2.30+)"
    exit 1
fi
echo "  OK: git $(git --version | awk '{print $3}')"

# Step 2: Install npm dependencies
echo "[2/5] Installing dependencies..."
if [[ -d "$SCRIPT_DIR/node_modules" ]]; then
    echo "  [skip] Already installed"
else
    (cd "$SCRIPT_DIR" && npm install --production 2>&1) || {
        echo "  ERROR: npm install failed"
        exit 1
    }
    echo "  OK: Dependencies installed"
fi

# Step 3: Validate rapid-tools.cjs works
echo "[3/5] Validating RAPID tools..."
if ! node "$RAPID_TOOLS_PATH" prereqs --json &>/dev/null; then
    echo "  ERROR: rapid-tools.cjs validation failed"
    exit 1
fi
echo "  OK: RAPID tools functional"

# Step 4: Persist RAPID_TOOLS env var
echo "[4/5] Setting RAPID_TOOLS environment variable..."
export RAPID_TOOLS="$RAPID_TOOLS_PATH"

# Detect shell and persist
SHELL_NAME="$(basename "$SHELL")"
case "$SHELL_NAME" in
    fish)
        CONFIG_FILE="$HOME/.config/fish/config.fish"
        EXPORT_LINE="set -gx RAPID_TOOLS \"$RAPID_TOOLS_PATH\""
        ;;
    zsh)
        CONFIG_FILE="$HOME/.zshrc"
        EXPORT_LINE="export RAPID_TOOLS=\"$RAPID_TOOLS_PATH\""
        ;;
    bash)
        CONFIG_FILE="$HOME/.bashrc"
        [[ ! -f "$CONFIG_FILE" ]] && CONFIG_FILE="$HOME/.bash_profile"
        EXPORT_LINE="export RAPID_TOOLS=\"$RAPID_TOOLS_PATH\""
        ;;
    *)
        CONFIG_FILE="$HOME/.profile"
        EXPORT_LINE="export RAPID_TOOLS=\"$RAPID_TOOLS_PATH\""
        ;;
esac

if grep -qF "RAPID_TOOLS" "$CONFIG_FILE" 2>/dev/null; then
    echo "  [skip] Already configured in $CONFIG_FILE"
else
    echo "" >> "$CONFIG_FILE"
    echo "# RAPID plugin for Claude Code" >> "$CONFIG_FILE"
    echo "$EXPORT_LINE" >> "$CONFIG_FILE"
    echo "  OK: Added to $CONFIG_FILE"
fi

# Step 5: Register as Claude Code plugin
echo "[5/5] Registering Claude Code plugin..."
if command -v claude &>/dev/null; then
    (cd "$SCRIPT_DIR" && claude plugin add . 2>&1) || {
        echo "  WARNING: Plugin registration failed (non-fatal)"
    }
    echo "  OK: Registered"
else
    echo "  [skip] Claude Code CLI not found -- register manually with: claude plugin add $SCRIPT_DIR"
fi

echo ""
echo "=== Setup Complete ==="
echo "RAPID_TOOLS=$RAPID_TOOLS_PATH"
echo ""
echo "NOTE: Run 'source $CONFIG_FILE' or open a new terminal to activate."
```

## Open Questions

1. **Plugin cache path stability**
   - What we know: Marketplace plugins are cached at `~/.claude/plugins/cache/<hash>/`. The hash likely changes on update.
   - What's unclear: Whether there is a stable path or symlink for installed plugins. Whether RAPID_TOOLS needs re-setting after plugin updates.
   - Recommendation: For marketplace installs, detect current cache path and set RAPID_TOOLS to it. Document that `/rapid:install` should be re-run after plugin updates. Consider adding a hook that auto-updates RAPID_TOOLS on plugin update.

2. **`claude plugin add .` behavior**
   - What we know: This registers a local directory as a plugin. The exact behavior (copy to cache vs reference in-place) differs from marketplace install.
   - What's unclear: Whether `claude plugin add .` creates a cached copy or uses the directory in-place.
   - Recommendation: Test both `claude plugin add .` and `claude --plugin-dir .` during implementation. The `--plugin-dir` flag loads in-place (no cache).

3. **RAPID_TOOLS for agents spawned in worktrees**
   - What we know: Agents spawned by RAPID execute in different worktree directories. RAPID_TOOLS as an env var persists across spawned processes.
   - What's unclear: Whether subagents/agent teams inherit env vars from the parent process.
   - Recommendation: Since env vars are inherited by child processes in Unix, this should work. Validate during implementation.
