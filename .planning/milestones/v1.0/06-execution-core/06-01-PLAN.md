    Follow 'use strict' + CommonJS pattern (`module.exports = { ... }`).

    `generateStub(contractJson, setName)`:
    - Extract `exports.functions` and `exports.types` from contractJson (default to empty arrays)
    - Build lines array starting with:
      ```
      // AUTO-GENERATED stub for set: {setName}
      // Generated from CONTRACT.json -- DO NOT EDIT
      'use strict';
      ```
    - For each type in exports.types, generate a JSDoc @typedef block:
      ```
      /**
      ```

      ```
      /**
       * @param {{paramType}} {paramName}
       * @returns {{returnType}}
       */
      function {fnName}({params joined by comma}) {
        throw new Error('Stub: {fnName} not yet implemented by set {setName}');
      }
      ```
    - End with `module.exports = { {function names joined by comma} };`
    - Return lines.join('\n')

    - Return array of absolute file paths written

    - Validate phase is one of 'discuss', 'plan', 'execute'
    - Call prepareSetContext to get the set's artifacts
    - Build prompt sections array based on phase:

    For 'discuss' phase:
    ```
    # Set: {setName} -- Discussion Phase

    You are reviewing the '{setName}' set before implementation begins. Your job is to surface any questions or ambiguities about the implementation approach.

    ## Contract
    {contractStr in fenced json block}

    ## Definition
    {definition content}

    ## Instructions
    Review the contract and definition above. Ask the developer clarifying questions about:
    - Implementation approach and technology choices
    - Edge cases or error handling not covered in the contract
    - Integration points with other sets that need clarification

    If everything is clear, state that you have no questions and summarize the implementation approach you would take.
    ```

    For 'plan' phase:
    ```
    # Set: {setName} -- Planning Phase

    You are creating an implementation plan for the '{setName}' set.

    ## Contract
    {contractStr}

    ## Definition
    {definition}

    ## Discussion Decisions
    {priorContext or 'No prior discussion -- proceed with contract and definition as given.'}

    ## Instructions
    Create a step-by-step implementation plan. For each step:
    1. What files to create or modify
    2. What to implement
    3. How to verify it works
    4. What to commit (one commit per logical task)

    Organize steps so each commit leaves the codebase in a working state.
    ```

    For 'execute' phase:
    ```
    # Set: {setName} -- Execution Phase

    {scopedMd}

    ## Implementation Plan
    {priorContext or 'No plan provided -- implement according to the contract and definition.'}

    ## Commit Convention
    After each task, commit with: type({setName}): description
    Where type is feat|fix|refactor|test|docs|chore

    Use `git add <specific files>` -- NEVER `git add .` or `git add -A`.
    Each commit must leave the codebase in a working state.
    ```
