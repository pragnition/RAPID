<objective>
Build the structured return protocol and filesystem verification system.

Purpose: AGNT-02 requires all agents use a machine-parseable return protocol, and AGNT-03 requires independent filesystem verification of agent work. This plan creates the return parser/generator (hybrid Markdown table + JSON comment format) and the two-tier verification system (lightweight during execution, heavyweight at merge time). Together these ensure agents communicate results reliably and their claims are independently verified.

Output: `returns.cjs` for parsing/generating structured returns, `verify.cjs` for tiered filesystem verification, CLI subcommands `parse-return` and `verify-artifacts`, all with passing tests.
</objective>

  <behavior>
    - parseReturn(text with RAPID:RETURN marker): extracts JSON, returns { parsed: true, data: {...} }
    - parseReturn(text without marker): returns { parsed: false, error: 'No RAPID:RETURN marker found' }
    - parseReturn(text with malformed JSON): returns { parsed: false, error: 'Invalid JSON: ...' }
    - parseReturn(text with unclosed marker): returns { parsed: false, error: 'Unclosed RAPID:RETURN marker' }
    - generateReturn({ status: 'COMPLETE', artifacts: [...], ... }): produces string with Markdown table + JSON comment
    - generateReturn COMPLETE output: contains `## COMPLETE` heading, table with Status/Artifacts/Commits/Tasks/Duration/Next/Notes rows, and `<!-- RAPID:RETURN {...} -->` comment
    - generateReturn CHECKPOINT output: contains handoff fields (Done/Remaining/Decisions/Blockers/Resume)
    - generateReturn BLOCKED output: contains Category/Blocker/Resolution fields
    - validateReturn(data): returns { valid: true } for complete data, { valid: false, errors: [...] } for missing required fields
    - validateReturn: COMPLETE requires status + artifacts + tasks_completed + tasks_total
    - validateReturn: BLOCKED requires status + blocker_category + blocker + resolution
    - validateReturn: CHECKPOINT requires status + handoff_done + handoff_remaining + handoff_resume
    - Round-trip: parseReturn(generateReturn(data)).data deep-equals data (consistency guarantee)
    - rapid-tools.cjs `parse-return <file>`: reads file, parses return, prints JSON result
  </behavior>
  <action>
    1. Create `rapid/src/lib/returns.cjs`:
       - Constants: `RETURN_MARKER = '<!-- RAPID:RETURN'`, `RETURN_END = '-->'`.
       - `parseReturn(agentOutput)`:
         a. Find `RETURN_MARKER` in text. If not found, return `{ parsed: false, error: 'No RAPID:RETURN marker found' }`.
         b. Find `RETURN_END` after marker. If not found, return `{ parsed: false, error: 'Unclosed RAPID:RETURN marker' }`.
         c. Extract JSON string between marker and end. Trim whitespace.
         d. Parse JSON. If parse fails, return `{ parsed: false, error: 'Invalid JSON: {message}' }`.
         e. Return `{ parsed: true, data }`.
       - `generateReturn(data)`:
         a. Validate data has required fields for its status (call validateReturn internally, throw if invalid).
         b. Build Markdown table rows based on status type:
            - COMPLETE: Status, Artifacts (backtick-wrapped), Commits, Tasks (X/Y), Duration, Next, Warnings, Notes
            - CHECKPOINT: Status, Artifacts, Tasks, Duration, Done, Remaining, Decisions, Blockers, Resume
            - BLOCKED: Status, Category, Blocker, Resolution, Artifacts (partial), Tasks
            - Only include rows for non-null, non-empty fields.
         c. Assemble: `## {STATUS}\n\n{table}\n\n<!-- RAPID:RETURN {JSON.stringify(data)} -->`
         d. Return the assembled string.
       - `validateReturn(data)`:
         a. Check `data.status` exists and is one of COMPLETE, CHECKPOINT, BLOCKED.
         b. Per-status required fields:
            - COMPLETE: artifacts (array), tasks_completed (number), tasks_total (number)
            - CHECKPOINT: handoff_done (string), handoff_remaining (string), handoff_resume (string)
            - BLOCKED: blocker_category (one of DEPENDENCY, PERMISSION, CLARIFICATION, ERROR), blocker (string), resolution (string)
         c. Return `{ valid: true }` or `{ valid: false, errors: [...specific missing/invalid fields] }`.
       - Export: `{ parseReturn, generateReturn, validateReturn }`.

    2. Update `rapid/src/bin/rapid-tools.cjs` to add `parse-return` subcommand:
       - `parse-return <file>`: Read file, call parseReturn on its contents, print result as JSON to stdout.
       - `parse-return --validate <file>`: Read file, parse return, then validate the parsed data, print validation result.

    3. Write tests in `rapid/src/lib/returns.test.cjs` using `node:test` and `node:assert`:
       - parseReturn with valid COMPLETE return extracts correct data
       - parseReturn with valid BLOCKED return extracts correct data
       - parseReturn with valid CHECKPOINT return extracts correct data
       - parseReturn with no marker returns parsed: false
       - parseReturn with malformed JSON returns parsed: false with error
       - generateReturn COMPLETE produces correct Markdown table + JSON
       - generateReturn BLOCKED includes category/blocker/resolution in table
       - generateReturn CHECKPOINT includes handoff fields in table
       - validateReturn COMPLETE with missing artifacts returns errors
       - validateReturn BLOCKED with invalid category returns errors
       - Round-trip test: parseReturn(generateReturn(data)).data deep-equals original data for each status type

    IMPORTANT: Use `~` for home directory paths in shell commands, NOT `$HOME`.
  </action>

  <behavior>
    - verifyLight(artifacts, commits): returns { passed: [...], failed: [...] }
    - verifyLight: existing file produces { type: 'file_exists', target: path } in passed
    - verifyLight: missing file produces { type: 'file_missing', target: path } in failed
    - verifyLight: valid git commit hash produces { type: 'commit_exists', target: hash } in passed
    - verifyLight: invalid git commit hash produces { type: 'commit_missing', target: hash } in failed
    - verifyHeavy(artifacts, testCommand): includes all verifyLight checks PLUS test execution and substantive content checks
    - verifyHeavy: when testCommand passes, adds { type: 'tests_pass', target: command } to passed
    - verifyHeavy: when testCommand fails, adds { type: 'tests_fail', target: command, error: stderr } to failed
    - verifyHeavy: file with < 50 chars produces { type: 'stub_content', target: path } in failed
    - verifyHeavy: file containing 'TODO' or 'placeholder' produces { type: 'stub_content', target: path } in failed
    - verifyHeavy: substantive file (> 50 chars, no TODOs) passes content check
    - generateVerificationReport(results, tier): produces Markdown VERIFICATION.md content with frontmatter (pass_count, fail_count, tier) and detailed results table
    - rapid-tools.cjs `verify-artifacts <file1> [file2...]`: runs verifyLight and prints results
    - rapid-tools.cjs `verify-artifacts --heavy --test "command" <file1> [file2...]`: runs verifyHeavy and prints results
  </behavior>
  <action>
    1. Create `rapid/src/lib/verify.cjs`:
       - Import `fs`, `{ execSync }` from `child_process`, `path`.
       - `verifyLight(artifacts, commits)`:
         a. Initialize `results = { passed: [], failed: [] }`.
         b. For each artifact path: check `fs.existsSync()`. Push `{ type: 'file_exists', target }` to passed or `{ type: 'file_missing', target }` to failed.
         c. For each commit hash: try `execSync('git cat-file -t ' + hash, { stdio: 'pipe' })`. Push `{ type: 'commit_exists', target }` to passed. On error, push `{ type: 'commit_missing', target }` to failed.
         d. Return results.
       - `verifyHeavy(artifacts, testCommand)`:
         a. Start with `verifyLight(artifacts, [])` results.
         b. If testCommand provided: try `execSync(testCommand, { stdio: 'pipe', timeout: 60000 })`. Push `{ type: 'tests_pass', target: testCommand }` to passed. On error, push `{ type: 'tests_fail', target: testCommand, error: stderr.toString() }` to failed.
         c. For each existing artifact: read content, check `content.length < 50` OR `content.includes('TODO')` OR `content.includes('placeholder')`. If any, push `{ type: 'stub_content', target }` to failed. Otherwise push `{ type: 'content_substantive', target }` to passed.
         d. Return results.
       - `generateVerificationReport(results, tier)`:
         a. Produce a Markdown string with YAML frontmatter: `pass_count`, `fail_count`, `tier` (light or heavy), `timestamp`.
         b. Body: `## Passed` with table of passed checks, `## Failed` with table of failed checks.
         c. Each row: `| {type} | {target} | {error if any} |`.
         d. Summary line: `**Result:** {PASS if no failures, FAIL if any failures}`.
         e. Return the string.
       - Export: `{ verifyLight, verifyHeavy, generateVerificationReport }`.

    2. Update `rapid/src/bin/rapid-tools.cjs` to add `verify-artifacts` subcommand:
       - `verify-artifacts <file1> [file2...]`: Call verifyLight with provided file paths and empty commits array. Print results JSON to stdout.
       - `verify-artifacts --heavy --test "<command>" <file1> [file2...]`: Call verifyHeavy with file paths and test command. Print results JSON.
       - `verify-artifacts --report <file1> [file2...]`: Call verifyLight, then generateVerificationReport, print the Markdown report.
       - Update usage/help text.

    3. Write tests in `rapid/src/lib/verify.test.cjs` using `node:test` and `node:assert`:
       - verifyLight: existing file passes (create temp file)
       - verifyLight: missing file fails
       - verifyLight: valid git commit passes (use HEAD commit from current repo)
       - verifyLight: invalid commit hash fails
       - verifyHeavy: passing test command produces tests_pass
       - verifyHeavy: failing test command produces tests_fail with error
       - verifyHeavy: stub file (< 50 chars) produces stub_content failure
       - verifyHeavy: file with TODO produces stub_content failure
       - verifyHeavy: substantive file passes content check
       - generateVerificationReport: produces Markdown with frontmatter containing counts
       - generateVerificationReport: PASS result when no failures
       - generateVerificationReport: FAIL result when failures exist

    IMPORTANT: Use `~` for home directory paths in shell commands, NOT `$HOME`. For tests that need git commit hashes, use `git rev-parse HEAD` to get a valid hash from the current repo. For test command tests, use `node -e "process.exit(0)"` (pass) and `node -e "process.exit(1)"` (fail).
