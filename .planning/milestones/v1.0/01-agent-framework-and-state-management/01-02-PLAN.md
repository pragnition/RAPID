Build the composable agent module system and assembly engine.

Output: Module files in `rapid/src/modules/`, assembly engine in `rapid/src/lib/assembler.cjs`, config in `rapid/config.json`, and `assemble-agent` CLI subcommand.

  <action>
    Create the `rapid/src/modules/` directory structure with `core/` and `roles/` subdirectories.

    **Core modules** (shared by ALL agents):

    1. `core-identity.md` (~15 lines): Base RAPID agent identity. Include:
       - "You are a RAPID agent -- part of a team-based parallel development system for Claude Code."
       - "You operate within a project that has been decomposed into independent sets, each executing in its own git worktree."
       - "All project state lives in `.planning/`. You interact with state via the `rapid-tools.cjs` CLI."
       - "You MUST use the structured return protocol to report your results (see returns section)."

    2. `core-returns.md` (~50 lines): Full structured return protocol specification. Include:
       - The hybrid format explanation (Markdown table + `<!-- RAPID:RETURN {...} -->` JSON comment)
       - Template for COMPLETE return with all standard fields (status, artifacts, commits, tasks_completed, tasks_total, duration_minutes, next_action, warnings, notes)
       - Template for CHECKPOINT return with handoff fields (handoff_done, handoff_remaining, handoff_decisions, handoff_blockers, handoff_resume)
       - Template for BLOCKED return with blocker fields (blocker_category: DEPENDENCY|PERMISSION|CLARIFICATION|ERROR, blocker, resolution)
       - Critical rule: "Generate the JSON payload FIRST, then render the Markdown table FROM the JSON. Never generate them independently."
       - Example of each status type.

    3. `core-state-access.md` (~20 lines): How to interact with `.planning/` state. Include:
       - CLI commands: `node rapid/src/bin/rapid-tools.cjs state get [field]`, `state update <field> <value>`
       - Lock commands: `node rapid/src/bin/rapid-tools.cjs lock acquire <name>`, `lock status <name>`
       - Rule: "Reads are safe without locking. Writes MUST go through the CLI (which acquires locks automatically)."
       - Rule: "Never write directly to `.planning/` files. Always use the CLI tool."

    4. `core-git.md` (~15 lines): Atomic commit conventions. Include:
       - Commit message format: `type(scope): description` where type is feat/fix/refactor/test/docs
       - Rule: "Each task produces exactly one commit. Do not batch multiple tasks into one commit."
       - Rule: "Commit only files you modified. Use `git add <specific files>`, never `git add .` or `git add -A`."
       - Rule: "Verify your commit landed: `git log -1 --oneline` after committing."

    5. `core-context-loading.md` (~15 lines): Progressive context loading strategy. Include:
       - "Start with the minimum context needed for your task. Load additional files only when needed."
       - "Read PLAN.md and SUMMARY.md first. These contain the task spec and what's already been built."
       - "For state: use `rapid-tools.cjs state get` rather than reading STATE.md directly."
       - "For codebase exploration: use Grep and Glob to find relevant files before reading them."
       - "Never load more than 5 files speculatively. Each file costs context budget."

    **Role modules** (one per agent type):

    6. `role-planner.md` (~25 lines): Planning-specific behavior. Include:
       - "You decompose work into parallelizable sets with explicit boundaries."
       - "Each set gets a DEFINITION.md (scope, files owned, acceptance criteria) and a CONTRACT.md (interfaces exposed/consumed)."
       - "You produce dependency graphs showing which sets run in parallel vs sequential."
       - "You assign shared-file ownership to prevent merge conflicts."
       - "Your output is a set of PLAN.md files, one per set."

    7. `role-executor.md` (~25 lines): Execution-specific behavior. Include:
       - "You implement tasks defined in PLAN.md within your assigned worktree."
       - "You follow the exact specifications in the plan. If a plan is ambiguous, report BLOCKED with category CLARIFICATION."
       - "After each task, commit atomically: one commit per task, bisectable history."
       - "You work within your set's boundaries. Never modify files outside your set's ownership."
       - "If you encounter a dependency on another set, report BLOCKED with category DEPENDENCY."

    8. `role-reviewer.md` (~25 lines): Review-specific behavior. Include:
       - "You perform deep code review before sets merge to main."
       - "Check: code style consistency, correctness, contract compliance, test coverage."
       - "Validate all interface contracts are satisfied between sets."
       - "If issues are fixable (style violations, missing tests), spawn a cleanup agent."
       - "If contracts are violated, BLOCK the merge with specific violation details."

    9. `role-verifier.md` (~25 lines): Verification-specific behavior. Include:
       - "You verify agent task completion by checking filesystem artifacts -- never trust self-reports."
       - "Lightweight checks (during execution): file existence + git commit verification."
       - "Heavyweight checks (at merge time): tests pass + contract compliance + content is substantive (not stubs)."
       - "Produce VERIFICATION.md with pass/fail results and evidence for each check."
       - "On verification failure: the working agent gets one auto-retry. If second attempt fails, report BLOCKED."

    10. `role-orchestrator.md` (~25 lines): Orchestration-specific behavior. Include:
        - "You coordinate the overall RAPID workflow: planning, execution, verification, merge."
        - "You spawn subagents (planner, executor, reviewer, verifier) as needed."
        - "You enforce sync gates: all sets must finish planning before any begins execution."
        - "You parse structured returns from subagents using the `<!-- RAPID:RETURN -->` marker."
        - "You are the MAIN THREAD agent -- you cannot be a subagent yourself (subagents cannot spawn subagents)."

    Each module should be focused, authoritative, and not contradict other modules. Core modules set baseline behavior; role modules extend without contradicting.

    2. Create `rapid/src/lib/assembler.cjs`:
       - Import `fs`, `path`, and `loadConfig` from `./core.cjs`.
       - `assembleAgent({ role, coreModules, context, outputPath })`:
         a. Generate YAML frontmatter using `generateFrontmatter(role)` (see RESEARCH.md Pattern 1 for tool configs per role).
         b. Read each core module from `rapid/src/modules/core/`, wrap in XML tags: `<{tag_name}>\n{content}\n</{tag_name}>` where tag_name is derived from filename (strip `core-` prefix and `.md` suffix, e.g., `core-identity.md` -> `<identity>`).

         f. If `outputPath` provided, write to file and return path. If not, return the assembled string.
         g. Warn (via `core.output()`) if assembled size exceeds `agent_size_warn_kb` from config.
