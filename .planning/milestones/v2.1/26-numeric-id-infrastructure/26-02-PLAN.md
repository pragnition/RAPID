    Add a resolver step to each skill that accepts a set or wave argument. The resolver is called ONCE at the argument boundary -- before any other operations. After resolution, the skill uses the string ID for all subsequent CLI calls.

    **For set-accepting skills** (set-init, execute, review, merge, pause, resume, cleanup, assumptions):

    Add a "Resolve Set Reference" step immediately after environment setup and BEFORE the existing argument handling logic. Insert this resolver block wherever the skill currently receives or parses the set name argument:

    ```markdown
    ### Resolve Set Reference

    If the user provided a set argument (e.g., `/rapid:set-init 1` or `/rapid:set-init auth-system`), resolve it through the numeric ID resolver:

    \```bash
    # (env preamble here)
    RESOLVE_RESULT=$(node "${RAPID_TOOLS}" resolve set "<user-input>" 2>&1)
    RESOLVE_EXIT=$?
    if [ $RESOLVE_EXIT -ne 0 ]; then
      echo "$RESOLVE_RESULT"
      # Display the error message from the JSON and STOP
    fi
    SET_NAME=$(echo "$RESOLVE_RESULT" | node -e "d=JSON.parse(require('fs').readFileSync(0,'utf-8')); console.log(d.resolvedId)")
    \```

    Use `SET_NAME` for all subsequent operations. The numeric input has been resolved to a string ID.
    ```

    Key specifics per skill:
    - **set-init**: In Step 1 "Determine Set to Initialize", if a set name was provided as argument, resolve it first. If no argument, the existing AskUserQuestion flow remains (but selected option is still resolved -- though it will be a string ID, which the resolver handles).
    - **execute**: In Step 0b "Parse set-id argument", resolve the provided set-id before state lookup.
    - **review**: In Step 0b "Parse arguments", resolve set-id. If optional wave-id also provided, resolve wave reference too using `resolve wave`.
    - **merge**: In Step 1c, if a specific set name was provided, resolve it first.
    - **pause**: In Step 2, if set name was provided as argument, resolve it first.
    - **resume**: In Step 2, if set name was provided as argument, resolve it first.
    - **cleanup**: In Step 3, if user provided set name as argument, resolve it first.
    - **assumptions**: In Step 1, if set name was provided, resolve it first.

    **For wave-accepting skills** (discuss, wave-plan):

    Replace the existing `wave-plan resolve-wave` call with the new `resolve wave` command. The key change:

    OLD pattern (in both discuss and wave-plan):
    ```bash
    WAVE_RESULT=$(node "${RAPID_TOOLS}" wave-plan resolve-wave <waveId>)
    ```

    NEW pattern:
    ```bash
    RESOLVE_RESULT=$(node "${RAPID_TOOLS}" resolve wave "<user-input>" 2>&1)
    RESOLVE_EXIT=$?
    if [ $RESOLVE_EXIT -ne 0 ]; then
      # Parse error from JSON and display to user, STOP
    fi
    ```

    For discuss and wave-plan, the user may invoke with:
    - Numeric: `/rapid:discuss 1.1` (new) or `/rapid:wave-plan 1.1` (new)
    - String: `/rapid:discuss wave-1` (existing) or `/rapid:wave-plan auth wave-1` (existing)

    The new resolver handles both cases. For the "set + wave" two-argument case (e.g., `/rapid:discuss auth wave-1`), the skill should first resolve the set, then resolve the wave within that set context. If the user provides dot notation (1.1), a single resolve call handles both.

    Important: After resolution, all downstream operations continue to use the resolved string IDs exactly as before. The resolver is a translation layer at the boundary only.

    AVOID: Do not remove the existing AskUserQuestion fallback flows (when no argument provided). Those remain for interactive discovery. Only add the resolver step for when an argument IS provided.

    AVOID: Do not change any CLI handler functions in rapid-tools.cjs. Only SKILL.md prompt files are modified in this task.

  <action>
    Update the `/rapid:status` skill to show numeric indices inline with set and wave listings.

    In Step 3 "Display Dashboard", modify the display instructions to prefix sets and waves with their numeric index:

    **Sets**: Show as `{N}: {set-id} [{status}]` where N is the 1-based alphabetical index.
    Example: `1: set-01-foundation [executing]`

    **Waves**: Show indented with dot notation `  {setIndex}.{waveIndex}: {wave-id} [{status}]`
    Example: `  1.1: wave-01 [complete]`

    **Jobs**: Show with triple dot notation (display only, not functional): `    {setIndex}.{waveIndex}.{jobIndex}: {job-id} [{status}]`
    Example: `    1.1.1: job-setup [complete]`

    The actual data comes from the `status-v2` command which returns `table`, `actions`, `milestone`. The skill currently renders the `table` field directly. Update the rendering instructions to:

    1. After receiving the status-v2 JSON, also run:
       ```bash
       SETS_LIST=$(node "${RAPID_TOOLS}" plan list-sets 2>/dev/null)
       ```
       to get the alphabetically-sorted set list for index mapping.

    2. When rendering each set in the table, compute its 1-based index from the sorted set list and prefix the display.

    3. When rendering waves within a set, use `{setIndex}.{waveIndex}` format (1-based within the set's waves array).

    4. When rendering jobs within a wave, use `{setIndex}.{waveIndex}.{jobIndex}` format.

    **Next-step suggestions**: Update the actions display in Step 4 to use numeric shorthand. When the `actions` array contains commands like `/set-init auth` or `/wave-plan auth wave-1`, replace the set/wave references with their numeric equivalents:
    - `/rapid:set-init auth` becomes `/rapid:set-init 1` (where auth is set index 1)
    - `/rapid:wave-plan auth wave-1` becomes `/rapid:wave-plan 1.1`

    Add a note after the dashboard that explains the numbering: "Tip: Use numeric shorthand (e.g., `/rapid:set-init 1`, `/rapid:wave-plan 1.1`) instead of full set/wave names."

    AVOID: Do not modify the status-v2 CLI handler or worktree module. Only the SKILL.md display rendering instructions change.
