# Phase 27.1: Skill-to-Agent Overhaul - Context

**Gathered:** 2026-03-09
**Status:** Ready for planning

<domain>
## Phase Boundary

Populate the `agents/` directory with all 26 RAPID subagent types so orchestrator skills spawn RAPID-namespaced agents instead of falling through to other plugins (e.g., `gsd-review:*`). Refactor skills to reference registered agent types. Replace the runtime assembler with a build-time agent generation command.

Currently `agents/` has only 1 file (`rapid-planner.md`) out of 26 role modules. Skills build prompts by reading role module files inline. This phase fixes the agent registration gap and streamlines the skill-to-agent interface.

</domain>

<decisions>
## Implementation Decisions

### Agent granularity
- All 26 role modules become registered agents in `agents/`
- Skills (user-invocable commands like `/rapid:execute`) remain as skills in `skills/`
- Agents are everything spawned by skills as subagents
- No consolidation — each role module maps 1:1 to an agent file
- Research agents: 5 separate files (rapid-research-stack.md, rapid-research-features.md, etc.)
- Review agents: 5 separate files (rapid-unit-tester.md, rapid-bug-hunter.md, rapid-devils-advocate.md, rapid-judge.md, rapid-bugfix.md)
- No "legacy" pruning — register everything, remove later if truly unused

### Prompt assembly strategy
- Agent .md files contain full pre-assembled prompts (core modules + role module baked in)
- Source of truth stays in `src/modules/` (core/ and roles/)
- A build script generates `agents/` files by concatenating core modules + role module + frontmatter
- Build command: `node rapid-tools.cjs build-agents` (new CLI subcommand)
- Auto-build runs during `setup.sh` / install so agents/ is always up-to-date
- Per-role core module selection: build script has internal mapping of which core modules each role needs (not all agents get all 5 core modules)

### Skill modifications
- All 18 skills updated (not just the ~6 that spawn agents — normalize patterns across the board)
- Skills reference agents by name (e.g., "Spawn rapid-wave-researcher agent") instead of reading role modules
- Hybrid context passing: skills pass essential IDs and small context (set, wave, job IDs). Agents read large artifacts (plans, contracts, source files) themselves via rapid-tools CLI
- Skills pass worktree path to agents so they work correctly within worktrees

### Worktree awareness
- Skills explicitly pass worktree path when spawning agents
- Agents know they may be operating in a worktree context
- Claude's Discretion: whether rapid-tools.cjs also needs worktree-aware path resolution

### Assembler fate
- Remove assembler.cjs entirely and rewrite as build-agents command in rapid-tools.cjs
- config.json loses the `agents` section — all agent metadata lives in agent .md frontmatter (tools, model, color)
- config.json keeps only non-agent settings (lock_timeout_ms, agent_size_warn_kb)

### Claude's Discretion
- Exact naming convention for agent files (e.g., `rapid-wave-researcher.md` vs `wave-researcher.md`)
- Whether rapid-tools.cjs needs internal worktree-aware path resolution beyond what skills pass
- How the build script maps roles to core modules (inline mapping vs separate config)
- Whether to add a `.gitignore` entry for `agents/` (generated files) or commit them
- Test strategy for the build command

</decisions>

## Specific Ideas

- The root cause of namespace contamination: skills spawn agents via the Agent/Task tool, but without registered agent types in `agents/`, Claude Code falls through to other installed plugins (like `gsd-review:*`) for matching agent types
- The existing `agents/rapid-planner.md` shows the correct format — YAML frontmatter (name, description, tools, model, color) + full prompt body
- User specifically wants agents to work in worktree contexts — currently rapid-tools.cjs doesn't resolve paths correctly in worktrees (e.g., shows no wave-plans even though the worktree contains them)
