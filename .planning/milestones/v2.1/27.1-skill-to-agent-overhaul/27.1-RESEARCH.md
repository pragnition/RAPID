# Phase 27.1: Skill-to-Agent Overhaul - Research

**Researched:** 2026-03-09
**Domain:** Claude Code agent registration, build-time prompt assembly, skill-to-agent interface
**Confidence:** HIGH

## Summary

This phase converts RAPID's 26 role modules into 26 pre-assembled agent files in the `agents/` directory, replaces the runtime assembler with a build-time `build-agents` CLI command, and updates all 17 skills to reference registered agent types by name instead of reading role module files inline.

Currently, `agents/` contains only 1 file (`rapid-planner.md`) out of 26 needed. Skills spawn subagents by `cat`-ing role module files directly (e.g., `cat src/modules/roles/role-bug-hunter.md`) and pasting the content into the Agent tool prompt. This means Claude Code cannot match the spawned agent to a registered agent type, causing namespace fallthrough to other plugins. The fix is straightforward: generate all 26 agent .md files at build time, then update skills to reference agents by their registered `name` field.

The existing `assembleAgent()` function in `assembler.cjs` already does the correct concatenation (frontmatter + core modules + role module + context). The `build-agents` command reuses this logic with a hardcoded per-role core module mapping. The skill modifications are largely mechanical: replace inline `cat` + prompt construction with `"Spawn the rapid-{role} agent"` patterns.

**Primary recommendation:** Build the `build-agents` command first (reusing assembler internals), generate all 26 agents, verify with tests, then do a systematic skill update pass.

### Pattern 1: Claude Code Agent File Format
**What:** Each agent is a `.md` file with YAML frontmatter + markdown body
**When to use:** For every registered agent type
**Format (verified from official docs and existing rapid-planner.md):**
```markdown
---
name: rapid-{role}
description: {role description}
tools: {comma-separated tool list}
model: inherit
color: {color}
---

{full assembled prompt content -- core modules wrapped in XML tags + role module}
```

**Key facts from official Claude Code docs (HIGH confidence):**
- Agents stored in plugin's `agents/` directory are loaded at session start
- `name` and `description` are the only required fields
- `tools` field restricts which tools the agent can use (inherits all if omitted)
- `model` supports: `sonnet`, `opus`, `haiku`, `inherit`
- `color` sets background color for UI identification
- Claude uses the `description` to decide when to delegate tasks
- Subagents cannot spawn other subagents -- the skill orchestrates
- Plugin agents use `plugin-name:agent-name` namespace

### Pattern 2: Build-Time Agent Generation
**What:** A `build-agents` CLI subcommand that reads source modules and writes assembled agents
**When to use:** During setup, after module edits, in CI

```javascript
// In rapid-tools.cjs handleBuildAgents():

  'planner': ['core-identity.md', 'core-returns.md', 'core-state-access.md', 'core-git.md', 'core-context-loading.md'],
  'executor': ['core-identity.md', 'core-returns.md', 'core-state-access.md', 'core-git.md'],
  'reviewer': ['core-identity.md', 'core-returns.md', 'core-state-access.md'],
  // ... per-role mapping for all 26 roles
};

for (const [role, coreModules] of Object.entries(ROLE_CORE_MAP)) {
  const assembled = assembleAgent({ role, coreModules, context: {} });
  fs.writeFileSync(path.join(agentsDir, `rapid-${role}.md`), assembled);
}
```

### Anti-Patterns to Avoid
- **Duplicating role instructions in skills AND agents:** The whole point is that agents contain their own system prompt. Skills should NOT re-paste role module content.
- **Passing entire file contents via skill prompt:** Use hybrid approach -- pass IDs and small context, let agents read large artifacts via rapid-tools CLI.
- **Leaving `cat src/modules/roles/` references in skills:** Every such reference is a missed conversion.
- **Building agents at runtime:** The decision is build-time generation. Do not assemble on-the-fly.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| YAML frontmatter generation | Custom string builder | Existing `generateFrontmatter()` from assembler.cjs | Already handles all 16+ roles, tools, colors, descriptions correctly |
| Core module concatenation | Custom file reader | Existing `assembleAgent()` logic from assembler.cjs | Already wraps modules in XML tags, handles ordering, checks sizes |
| Agent file format | Custom template engine | Direct string concatenation of frontmatter + sections | The format is simple: YAML header + markdown body. No templating needed |

**Key insight:** The assembler.cjs already contains 95% of the build-agents logic. The new command just iterates over all roles with a per-role core module mapping instead of reading from config.json.

## Common Pitfalls

### Pitfall 1: Incomplete ROLE_TOOLS/ROLE_COLORS/ROLE_DESCRIPTIONS Maps
**What goes wrong:** 10 of 26 roles currently lack entries in ROLE_TOOLS, ROLE_COLORS, and ROLE_DESCRIPTIONS (the init/context/plan/set-init roles: codebase-synthesizer, context-generator, research-architecture, research-features, research-oversights, research-pitfalls, research-stack, research-synthesizer, roadmapper, set-planner).
**Why it happens:** These roles were added for skills that read modules inline and never needed assembler metadata.
**How to avoid:** Add all 10 missing roles to ROLE_TOOLS, ROLE_COLORS, and ROLE_DESCRIPTIONS BEFORE building agents. The `generateFrontmatter()` function falls back to defaults for unknown roles, but the defaults are wrong (e.g., `description: RAPID {role} agent` instead of a proper description).
**Warning signs:** Generated agents with `color: default` or generic description text.

### Pitfall 2: Agent Size Bloat
**What goes wrong:** Some assembled agents exceed the 15KB warning threshold, consuming excessive subagent context.
**Why it happens:** The planner role module alone is 11KB. Adding all 5 core modules (9.7KB total) creates a 20KB+ agent.
**How to avoid:** Use per-role core module selection. Not all agents need all 5 core modules. Research agents don't need `core-git.md` or `core-state-access.md`. Review read-only agents (devils-advocate, bug-hunter) don't need `core-git.md`.
**Warning signs:** Size warnings during `build-agents`, agents > 15KB.

### Pitfall 3: Skills Still Reading Modules Inline
**What goes wrong:** After generating agents, some skills still `cat` role modules and paste them into prompts, leading to duplicated instructions (agent system prompt + inline paste) and doubled context usage.
**Why it happens:** Mechanical update misses a skill or a branch within a skill.
**How to avoid:** After all skill edits, grep for `cat src/modules/roles/` and `role-.*\.md` in all skill files. Zero matches means clean conversion.
**Warning signs:** `grep -r "cat src/modules/roles" skills/` returns any results.

### Pitfall 4: Worktree Path Not Passed to Agents
**What goes wrong:** Agents spawned in worktree context operate in the wrong directory, can't find wave plans, produce incorrect file paths.
**Why it happens:** Skills forget to pass the worktree path when spawning agents.
**How to avoid:** Every skill that spawns an agent MUST include the working directory in the agent task prompt. Standardize a `## Working Directory` section in every spawn prompt.
**Warning signs:** Agents report "file not found" for paths that exist in the worktree.

### Pitfall 5: config.json agents Section Removed Too Early
**What goes wrong:** The existing `assemble-agent` CLI command reads from config.json's `agents` section. Removing it before the build-agents command is working breaks the existing flow.
**Why it happens:** Eager cleanup before replacement is ready.
**How to avoid:** Phase the work: (1) build-agents command working, (2) skills updated, (3) then remove config.json agents section and assembler.cjs.
**Warning signs:** `assemble-agent` command errors about missing config.

### Pitfall 6: Test Count Assertion in assembler.test.cjs
**What goes wrong:** The existing test expects 25 role modules but there are 26 on disk (role-merger.md was added without updating the test).
**Why it happens:** Pre-existing test drift.
**How to avoid:** Fix the test expectation when adding the 10 missing roles to the maps, or replace the test entirely with build-agents tests.
**Warning signs:** `node --test src/lib/assembler.test.cjs` already fails.

### Example 4: build-agents CLI Command Structure

```javascript
// In rapid-tools.cjs switch/case dispatcher:
case 'build-agents':
  handleBuildAgents(cwd, subArgs);
  break;

// Handler function:
function handleBuildAgents(cwd, args) {
  const fs = require('fs');
  const path = require('path');
  const { resolveRapidDir } = require('../lib/core.cjs');

  const rapidDir = resolveRapidDir();
  const agentsDir = path.join(rapidDir, 'agents');
  const modulesDir = path.join(rapidDir, 'src', 'modules');

  // Ensure agents/ exists
  if (!fs.existsSync(agentsDir)) {
    fs.mkdirSync(agentsDir, { recursive: true });
  }

  let built = 0;
  for (const [role, coreModules] of Object.entries(ROLE_CORE_MAP)) {
    const sections = [];

    // 1. Frontmatter
    sections.push(generateFrontmatter(role));

    // 2. Core modules
    for (const mod of coreModules) {
      const content = fs.readFileSync(path.join(modulesDir, 'core', mod), 'utf-8').trim();
      const tag = mod.replace('.md', '').replace('core-', '');
      sections.push(`<${tag}>\n${content}\n</${tag}>`);
    }

    // 3. Role module
    const roleContent = fs.readFileSync(path.join(modulesDir, 'roles', `role-${role}.md`), 'utf-8').trim();
    sections.push(`<role>\n${roleContent}\n</role>`);

    const assembled = sections.join('\n\n');
    fs.writeFileSync(path.join(agentsDir, `rapid-${role}.md`), assembled);
    built++;
  }

  output(`Built ${built} agents in ${agentsDir}`);
}
```

## Discretion Recommendations

### Naming Convention: Use `rapid-` prefix
**Recommendation:** `rapid-{role}.md` (e.g., `rapid-wave-researcher.md`)
**Reasoning:** The existing `rapid-planner.md` already uses this convention. The `name` frontmatter field uses `rapid-{role}`. Consistency is more important than brevity. The `rapid-` prefix prevents namespace collisions with other plugins.

### Role-to-Core Module Mapping: Inline mapping in handleBuildAgents
**Recommendation:** Hardcode the `ROLE_CORE_MAP` object directly in the build function, not in a separate config file.
**Reasoning:** The mapping changes rarely (only when new roles are added). Keeping it inline means one file to edit, one place to read. A separate config file adds indirection without benefit for 26 static entries. The existing ROLE_TOOLS/ROLE_COLORS/ROLE_DESCRIPTIONS maps use the same inline pattern.

### .gitignore for agents/: Do NOT gitignore -- commit generated agents
**Recommendation:** Commit the generated `agents/` files to git.
**Reasoning:** Claude Code loads agent .md files from the plugin's `agents/` directory at session start. If agents are gitignored, anyone cloning the repo must run `setup.sh` before agents work. Committing them means the plugin works immediately after clone/install. The source of truth (`src/modules/`) is also committed, so there's no information loss. Add a comment at the top of each generated file: `<!-- GENERATED by build-agents -- do not edit directly. Edit src/modules/ instead. -->`.

### Worktree-Aware Path Resolution in rapid-tools.cjs: Defer
**Recommendation:** Do NOT add worktree-aware path resolution to rapid-tools.cjs in this phase.
**Reasoning:** Skills already pass the worktree path explicitly when spawning agents. The agent receives the path and operates within it. Adding worktree awareness to rapid-tools.cjs itself is a separate concern (and noted as an existing issue in CONTEXT.md). The skill-to-agent interface change does not require it.

### Test Strategy: Unit tests for build-agents + integration verification
**Recommendation:**
1. Unit test `handleBuildAgents()` -- verify it produces 26 files, each starting with valid frontmatter, each containing the role's XML-tagged content
2. Verify each generated agent is under `agent_size_warn_kb` (15KB)
3. Verify all skills have zero `cat src/modules/roles/` references after conversion
4. Keep existing assembler.test.cjs tests running until assembler is removed (fix the 25->26 count)

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| Skills read role modules inline via `cat` | Skills reference registered agent types by name | This phase | Eliminates namespace fallthrough, reduces skill complexity |
| Runtime agent assembly via `assemble-agent` CLI | Build-time agent generation via `build-agents` CLI | This phase | Faster agent loading, no runtime assembly overhead |

## Open Questions

1. **Agent tool delegation syntax**
   - What we know: Claude Code matches agent types by the `name` field in `.md` frontmatter. Skills use the Agent tool to spawn subagents.
   - What's unclear: The exact syntax/mechanism for a skill to tell Claude "use the rapid-bug-hunter agent type for this delegation" vs. generic agent spawning. The current pattern puts the full prompt in the Agent tool call. With registered agents, the skill may need to say "Spawn agent of type rapid-bug-hunter" in its instructions.
   - Recommendation: Test with the existing `rapid-planner.md` agent to verify how Claude Code routes Agent tool calls to registered agent types. The skill instructions should simply say "Use the rapid-bug-hunter agent" and Claude Code should match on the registered name.

2. **Context files injection**
   - What we know: The current assembler can inject `.planning/context/` files (STYLE_GUIDE.md, CONVENTIONS.md, etc.) into agent prompts at assembly time.
   - What's unclear: Whether build-time assembly should include context files or if agents should load them at runtime.
   - Recommendation: Do NOT bake context files into agents at build time. Context files are project-specific and change per-project. Build-time agents should contain only core + role. Context should be passed at spawn time by the skill or loaded by the agent via rapid-tools CLI.
