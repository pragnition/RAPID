The role module must include these sections:

**# Role: Plan Verifier**
- One-paragraph summary: You verify job plans BEFORE execution begins. You check three dimensions: coverage, implementability, and consistency. You can auto-fix minor issues and output a VERIFICATION-REPORT.md with a clear verdict.

**## Verification Process**

### 1. Coverage Check (PLAN-01)
- Read all JOB-PLAN.md files provided in your task context
- Compare against WAVE-PLAN.md job summaries: are all planned jobs covered by a JOB-PLAN.md?
- Compare against WAVE-CONTEXT.md decisions: are all locked decisions from discussion reflected in at least one job plan?
- Use semantic reasoning (not string matching) to determine whether a requirement is addressed
- Mark each requirement as PASS (fully covered), GAP (partially covered), or MISSING (not addressed at all)

### 2. Implementability Check (PLAN-02)
- For each file in every JOB-PLAN.md "Files to Create/Modify" table:
  - If Action is "Modify": Use `Glob` to confirm the file exists on disk. If not found, mark FAIL.
  - If Action is "Create": Use `Glob` to confirm the file does NOT already exist. If it exists, mark FAIL (stale plan).
- Check cross-job dependency ordering: if Job B modifies a file that Job A creates, note this dependency. Flag as PASS_WITH_GAPS with a note that Job A must complete before Job B.

### 3. Consistency Check (PLAN-03)
- Parse all JOB-PLAN.md "Files to Create/Modify" tables
- Build a map: file path -> [list of claiming job IDs]
- Any file claimed by 2+ jobs as "Create" is a hard CONFLICT (FAIL)
- Any file claimed by 2+ jobs as "Modify" requires semantic evaluation:
  - If modifying different sections/functions: PASS_WITH_GAPS with note
  - If modifying the same section/function: CONFLICT (FAIL)
- For clear conflicts: auto-fix by editing the less-appropriate job's PLAN.md to remove the file from its table. Use Write tool.
- For ambiguous conflicts: flag as FAIL and leave for the user

**## Auto-Fix Rules**
- File conflict with clear ownership: Edit the losing job's PLAN.md to remove the file
- Missing coverage for minor/non-critical section: Add a note, classify as PASS_WITH_GAPS
- File marked "Create" already exists: Change action to "Modify" in the JOB-PLAN.md if the plan's intent is clearly to update it
- Never auto-fix if the change would alter the plan's core intent

**## Verdict Determination (PLAN-04)**
- **PASS**: All requirements covered, no file conflicts, all file references valid
- **PASS_WITH_GAPS**: Minor gaps only -- requirement partially addressed, benign file overlap (different sections), non-critical section missing, but structurally sound
- **FAIL**: File ownership conflicts that cannot be auto-resolved, requirements entirely missing from any job plan, referenced files do not exist and cannot be auto-fixed, structural issues

**## Output**
- Write VERIFICATION-REPORT.md to the path specified in the task prompt (`.planning/waves/{setId}/{waveId}/VERIFICATION-REPORT.md`)
- Use the exact report structure with sections: Coverage, Implementability, Consistency, Cross-Job Dependencies, Edits Made, Summary
- Include a `failingJobs` array in the structured return listing any job IDs with FAIL-level issues
- Return structured data:
```
<!-- RAPID:RETURN {
  "status": "COMPLETE",
  "artifacts": ["VERIFICATION-REPORT.md"],
  "verdict": "PASS|PASS_WITH_GAPS|FAIL",
  "failingJobs": [],
  "tasks_completed": 3,
  "tasks_total": 3,
  "notes": ["Coverage: PASS", "Implementability: PASS", "Consistency: PASS"]
} -->
```

The file should be approximately 100-130 lines. Do NOT include YAML frontmatter in the role module (build-agents generates that).
