  <action>
1. Add `ScoperOutput` Zod schema to review.cjs (in the Schemas section):
   ```javascript
   const ConcernFile = z.object({
     file: z.string(),
     rationale: z.string(),
   });
   const ConcernGroup = z.object({
     name: z.string(),
     files: z.array(z.string()),
     rationale: z.record(z.string(), z.string()),
   });
   const ScoperOutput = z.object({
     concerns: z.array(ConcernGroup),
     crossCutting: z.array(ConcernFile),
     totalFiles: z.number(),
     concernCount: z.number(),
     crossCuttingCount: z.number(),
   });
   ```

2. Add optional `concern` field to ReviewIssue schema: `concern: z.string().optional()` (after originatingWave)

3. Add `normalizedLevenshtein(a, b)` function -- returns 0-1 similarity score using standard Levenshtein distance matrix. See RESEARCH.md code example for reference implementation.

4. Add `scopeByConcern(scoperOutput, allFiles)` function:
   - Accepts parsed scoper RAPID:RETURN data and full file list
   - Checks cross-cutting fallback: if crossCuttingFiles.length > allFiles.length * 0.5, return { concernGroups: [], fallback: true, warning: "Cross-cutting files (N/M) exceed 50% threshold..." }
   - Otherwise builds concern groups, each group's files array = concern's own files + ALL cross-cutting files
   - Returns { concernGroups: [{concern, files}], fallback: false }

5. Add `deduplicateFindings(findings)` function:
   - Compares all pairs: same file + normalizedLevenshtein(description) >= 0.7 = duplicate
   - Higher severity wins (critical>high>medium>low). Equal severity: longer evidence/codeSnippet wins
   - Returns filtered array with dominated findings removed
   - Preserves concern tags on surviving findings

6. Export all new functions and schemas: scopeByConcern, deduplicateFindings, normalizedLevenshtein, ScoperOutput

7. Write comprehensive tests in review.test.cjs covering all behaviors listed above
