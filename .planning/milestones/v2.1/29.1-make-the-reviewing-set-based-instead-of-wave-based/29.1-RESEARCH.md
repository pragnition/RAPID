# Phase 29.1: Make the Reviewing Set-Based Instead of Wave-Based - Research

**Researched:** 2026-03-10
**Domain:** RAPID review pipeline refactoring (wave-scoped -> set-scoped)
**Confidence:** HIGH

## Summary

This phase transforms the RAPID review pipeline from iterating per-wave (running the full unit test / bug hunt / UAT pipeline N times, once per wave) to a single set-level pass across all changed files in the set branch. The change affects 5 main integration points: the review SKILL.md orchestrator, the review.cjs library, the rapid-tools.cjs CLI commands, 6 review agent role modules, and 6 generated agent files.

The core technical change is straightforward: replace `scopeWaveForReview` (which diffs a wave worktree vs base) with `scopeSetForReview` (which diffs the set branch vs main to get ALL changed files across all waves). The review pipeline then works on this combined scope, chunked by directory for large scopes, with wave attribution derived from JOB-PLAN.md file lists. The lean wave-level review (auto-run during reconciliation in /rapid:execute) is unaffected.

**Primary recommendation:** Replace the per-wave loop in SKILL.md Step 3 with a single set-level scope pass, refactor review.cjs to drop wave parameters from scoping/issue functions, update CLI commands to accept set-id only, and update all 6 agent role modules to reference set/chunk context instead of wave context.

## Architecture Patterns

### Current Architecture (What Exists)

The review pipeline currently follows a per-wave iteration pattern:

```
/rapid:review <set-id> [<wave-id>]
  Step 0: Resolve set + optional wave
  Step 1: Select stages (unit test / bug hunt / UAT)
  Step 2: Resolve reviewable waves (filter by complete/reconciling status)
  Step 3: FOR EACH WAVE (sequential):
    3.0: Scope wave files (diff wave worktree vs base)
    3a: Unit test stage (plan -> approve -> execute -> report)
    3b: Bug hunt stage (hunter -> advocate -> judge, up to 3 cycles)
    3c: UAT stage (plan -> approve -> execute -> report)
    3d: Wave review complete
  Step 4: Generate review summary
  Step 5: Display next steps
```

**Key files:**
- `skills/review/SKILL.md` (779 lines) -- orchestration instructions
- `src/lib/review.cjs` (433 lines) -- scoping, issue management, summary generation
- `src/bin/rapid-tools.cjs` (review section, ~190 lines) -- CLI command handlers
- `src/lib/review.test.cjs` (442 lines) -- unit tests
- 6 agent role modules: `role-bug-hunter.md`, `role-devils-advocate.md`, `role-judge.md`, `role-bugfix.md`, `role-unit-tester.md`, `role-uat.md`
- 6 generated agent files: `rapid-bug-hunter.md`, `rapid-devils-advocate.md`, `rapid-judge.md`, `rapid-bugfix.md`, `rapid-unit-tester.md`, `rapid-uat.md`

### Target Architecture (What to Build)

```
/rapid:review <set-id>
  Step 0: Resolve set (NO wave argument)
  Step 1: Select stages (unchanged)
  Step 2: Scope set files (diff set branch vs main, one-hop dependents)
  Step 3: Chunk scope by directory (if above threshold)
  Step 4: Build wave attribution map (from JOB-PLAN.md file lists)
  Step 5: Run selected stages on set scope:
    5a: Unit test (chunked by directory, parallel per chunk)
    5b: Bug hunt (hunters per chunk in parallel, merge, one advocate, one judge)
    5c: UAT (once on full set scope)
  Step 6: Generate review summary (with wave attribution)
  Step 7: Display next steps
```

### Directory Chunking Pattern

**Recommendation for threshold (Claude's Discretion):** 15 files triggers chunking. Below 15, a single agent pass. This keeps individual agent context windows manageable while avoiding over-chunking for small scopes.

**How it works:**
1. Collect all scoped files (changed + dependents)
2. Group by parent directory (e.g., `src/lib/`, `src/modules/roles/`, `skills/review/`)
3. If total files <= threshold: single agent pass, no chunking
4. If total files > threshold: each directory group becomes a chunk
5. Small groups (< 3 files) merge into the nearest sibling directory chunk

### Wave Attribution Pattern

**How to map findings to originating waves:**
1. Read all JOB-PLAN.md files across all waves: `.planning/waves/{setId}/{waveId}/*-PLAN.md`
2. Extract "Files to Create/Modify" table from each JOB-PLAN.md
3. Build a map: `{ filePath -> waveId }` (first match wins)
4. For findings: look up `finding.file` in the map to get originating wave

**Handling attribution ambiguity (Claude's Discretion):** Files appearing in multiple waves' job plans should be attributed to the LAST wave that modified them (highest wave index). This aligns with the sequential execution model where later waves see earlier waves' changes. Files not in any job plan get tagged as `"unattributed"`.

### Chunk Merging Strategy for Bug Hunt (Claude's Discretion)

**Recommendation:** Simple concatenation with deduplication by finding ID.

1. Each hunter chunk produces `findings[]` with IDs like `BUG-001`, `BUG-002`
2. Prefix chunk index to prevent collisions: `C1-BUG-001`, `C2-BUG-001`
3. Merge all findings into a single array
4. Pass merged array to ONE advocate agent and ONE judge agent

This is simpler than any ranking or filtering strategy and preserves the adversarial review integrity.

### Unit Test Plan Presentation (Claude's Discretion)

**Recommendation:** Group by directory chunk in the test plan display, but present as a flat approval:

```
--- Unit Test Plan ---
[src/lib/] (5 tests)
  [1] review.cjs: scopeSetForReview returns correct file list
  [2] review.cjs: logIssue writes to set-level REVIEW-ISSUES.json
  ...
[src/modules/roles/] (3 tests)
  [6] role-bug-hunter.md: contains set/chunk context references
  ...
Approve all? [Approve | Modify | Skip]
---------------------
```

One approval for the entire test plan, not per-chunk.

### `review scope` CLI Output Change (Claude's Discretion)

**Recommendation:** Change from:
```
rapid-tools review scope <set-id> <wave-id>
-> { changedFiles: [...], dependentFiles: [...], totalFiles: N }
```

To:
```
rapid-tools review scope <set-id>
-> { changedFiles: [...], dependentFiles: [...], totalFiles: N, chunks: [{dir: "src/lib/", files: [...]}], waveAttribution: { "file.cjs": "wave-1" } }
```

The output includes chunking information and wave attribution so the SKILL.md orchestrator does not need to recompute them.

## Common Pitfalls

### Pitfall 1: Breaking Lean Review
**What goes wrong:** The lean review (`review lean <set-id> <wave-id>`) is called from /rapid:execute during wave reconciliation. If wave-id parameters are removed from ALL review CLI commands, lean review breaks.
**Why it happens:** Over-aggressive refactoring -- removing wave-id from commands that still need it.
**How to avoid:** Keep `review lean` and `review scope` (with wave-id) as-is. The lean review is explicitly out of scope for this change. Only the full review pipeline changes to set-level.
**Warning signs:** Tests for lean review fail after refactoring.

### Pitfall 2: REVIEW-ISSUES.json Schema Break
**What goes wrong:** Existing tests expect `waveId` as a required field in the `ReviewIssues` container schema. The new schema uses `originatingWave` as an optional field on individual issues.
**Why it happens:** Zod schema validation rejects existing test data.
**How to avoid:** Update ALL test fixtures simultaneously with schema changes. The `ReviewIssues` container schema drops `waveId` (required) entirely; individual `ReviewIssue` objects gain `originatingWave` (optional string).
**Warning signs:** `node --test src/lib/review.test.cjs` fails on schema tests.

### Pitfall 3: CLI Backward Compatibility
**What goes wrong:** Users (or other skills like merge) call `review log-issue <set-id> <wave-id>` with the old 2-arg format.
**Why it happens:** Not all callers are updated.
**How to avoid:** Search ALL callers of review CLI commands across SKILL.md files: `/rapid:review`, `/rapid:execute` (lean review), `/rapid:merge`. Update each caller. The lean review in execute STILL uses wave-id -- do not change it.
**Warning signs:** grep for `review log-issue` and `review update-issue` across all skill files.

### Pitfall 4: Agent Prompt Mismatch
**What goes wrong:** Updated SKILL.md passes set/chunk context to agents, but agent role modules still mention "wave" and expect wave-level context variables.
**Why it happens:** Forgetting to update agent prompts alongside orchestrator changes.
**How to avoid:** Update role modules FIRST, rebuild agents, THEN update SKILL.md to pass the new context format.
**Warning signs:** Agent returns reference wave-specific fields that no longer exist.

### Pitfall 5: Empty Chunk Edge Case
**What goes wrong:** A set with all changes in a single directory produces a single chunk. The "chunking" code path runs but spawns only 1 agent. This is wasteful overhead.
**Why it happens:** Threshold check only counts total files, not chunk count.
**How to avoid:** After chunking, if there's only 1 chunk, fall back to single-agent pass regardless of file count. The chunking threshold should be a file count trigger, but the decision to parallelize should also consider chunk count.

### chunkByDirectory (new function)

```javascript
// Source: new addition to src/lib/review.cjs
const CHUNK_THRESHOLD = 15;

function chunkByDirectory(files) {
  if (files.length <= CHUNK_THRESHOLD) {
    return [{ dir: '.', files }];
  }

  const groups = new Map();
  for (const file of files) {
    const dir = path.dirname(file);
    if (!groups.has(dir)) groups.set(dir, []);
    groups.get(dir).push(file);
  }

  // Merge small groups (< 3 files) into nearest sibling
  const chunks = [];
  let overflow = [];
  for (const [dir, dirFiles] of groups) {
    if (dirFiles.length < 3) {
      overflow.push(...dirFiles);
    } else {
      chunks.push({ dir, files: dirFiles });
    }
  }
  if (overflow.length > 0) {
    if (chunks.length > 0) {
      chunks[chunks.length - 1].files.push(...overflow);
    } else {
      chunks.push({ dir: '.', files: overflow });
    }
  }

  return chunks;
}
```

### buildWaveAttribution (new function)

```javascript
// Source: new addition to src/lib/review.cjs
function buildWaveAttribution(cwd, setId) {
  const attribution = {};
  const wavesDir = path.join(cwd, '.planning', 'waves', setId);

  if (!fs.existsSync(wavesDir)) return attribution;

  const waveEntries = fs.readdirSync(wavesDir, { withFileTypes: true })
    .filter(e => e.isDirectory())
    .sort((a, b) => a.name.localeCompare(b.name));

  for (const waveEntry of waveEntries) {
    const waveDir = path.join(wavesDir, waveEntry.name);
    const planFiles = fs.readdirSync(waveDir).filter(f => f.endsWith('-PLAN.md'));

    for (const planFile of planFiles) {
      const content = fs.readFileSync(path.join(waveDir, planFile), 'utf-8');
      // Reuse the table regex from the lean review implementation
      const tableRegex = /\|\s*`?([^`|]+?)`?\s*\|\s*(Create|Modify)\s*\|/gi;
      let match;
      while ((match = tableRegex.exec(content)) !== null) {
        const filePath = match[1].trim();
        if (filePath && filePath !== 'File' && !filePath.startsWith('---')) {
          // Last wave wins (sequential execution model)
          attribution[filePath] = waveEntry.name;
        }
      }
    }
  }

  return attribution;
}
```

### Updated logIssue (set-level)

```javascript
// Source: refactored src/lib/review.cjs
function logIssue(cwd, setId, issue) {
  const setDir = path.join(cwd, '.planning', 'waves', setId);
  const issuesPath = path.join(setDir, 'REVIEW-ISSUES.json');

  const validatedIssue = ReviewIssue.parse(issue);

  fs.mkdirSync(setDir, { recursive: true });

  let existing = { setId, issues: [], lastUpdatedAt: '' };
  if (fs.existsSync(issuesPath)) {
    existing = JSON.parse(fs.readFileSync(issuesPath, 'utf-8'));
  }

  existing.issues.push(validatedIssue);
  existing.lastUpdatedAt = new Date().toISOString();

  fs.writeFileSync(issuesPath, JSON.stringify(existing, null, 2), 'utf-8');
}
```
