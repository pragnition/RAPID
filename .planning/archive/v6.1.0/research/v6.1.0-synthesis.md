# Research Summary

## Project Profile
- **Type:** brownfield CLI plugin (Claude Code agent harness)
- **Stack:** JavaScript (CommonJS) + Node.js >=20 + Git worktrees
- **Scale:** medium (4 core features, 3 deferred decisions, ~24 skill files affected by cross-cutting changes)
- **Key Constraint:** All state must survive `/clear` -- any data needed after a context reset must be persisted to `.planning/` as a file artifact (source: ARCHITECTURE, PITFALLS, UX)

## Unified Findings

### Stack and Infrastructure

- Node.js minimum version needs updating: `package.json` says `>=20`, `setup.sh` checks `>=18`, README badge says "20+". Node 20 reaches EOL on April 30, 2026 (24 days). All three locations must be synchronized to `>=22`. (source: STACK)
- Zod 3.25.76 bundles both Zod 3 and Zod 4, doubling package size (~288KB CJS). No runtime breakage with current `.cjs` extension and no `type: module`, but the dual-bundle is unnecessary bloat. A future migration to Zod 4 proper requires changing 5 `.passthrough()` calls to `z.looseObject()` -- mechanical but deferred. (source: STACK, PITFALLS)
- proper-lockfile 4.1.2 is dormant (last published 2021) but functional with zero CVEs. PID recycling on Linux can cause stale locks to appear held after SIGKILL; RAPID's 300-second stale threshold mitigates this. No suitable drop-in replacement exists for cross-process file locking. (source: STACK, PITFALLS)
- 346 `execSync` call sites across 22 files vs. 28 `execFileSync` in 5 files. Shell injection risk is theoretical (all current inputs are hardcoded strings) but the pattern is fragile. Deferred from v6.0.0; not in scope for this milestone. (source: STACK, PITFALLS)
- No CI/CD pipeline detected. Contributors cannot verify changes automatically. Outside scope but relevant to onboarding. (source: STACK)
- No linter or formatter configured (`npm run lint` / `npm run format` do not exist). Minor contributor onboarding gap but out of scope. (source: STACK)
- All 4 dependencies (zod, ajv, ajv-formats, proper-lockfile) have zero known CVEs. `npm audit` clean. (source: STACK)

### Feature Landscape

**Core features (this milestone):**

1. **Audit-to-Set Handoff Mechanism** -- Persist structured remediation context from `/rapid:audit-version` to disk so `/rapid:add-set` can auto-discover and pre-populate scope after `/clear`. Creates `.planning/pending-sets/{set-name}.json` (or `.planning/v{VERSION}-REMEDIATION.json`). Touches audit-version, add-set, and status skills. (source: FEATURES, ARCHITECTURE, PITFALLS, OVERSIGHTS, UX)

2. **Clear-Before-Next-Command Guidance** -- Add standardized `/clear` reminder footer to all lifecycle skill completion outputs. Zero of 28 skills currently include this. Requires defining a clear policy on which skill transitions need `/clear` (lifecycle phase boundaries: yes; review sub-steps: no). Recommended implementation: `renderFooter()` utility in `display.cjs` as single source of truth. (source: FEATURES, ARCHITECTURE, PITFALLS, OVERSIGHTS, UX)

3. **Beginner-Friendly README.md** -- Rewrite with problem statement (context rot), single primary install path, `/clear` mental model explanation, annotated quickstart, and "First Project" walkthrough. Current README has two contradictory install paths (marketplace vs. git clone), no problem statement, and a 12-step quickstart that assumes AI workflow familiarity. (source: FEATURES, ARCHITECTURE, PITFALLS, OVERSIGHTS, UX)

4. **General UX Audit** -- Systematic review of error messages, breadcrumb consistency, command discoverability, and first-run experience. Scoped to a predefined checklist, not open-ended exploration. Depends on the `/clear` footer pattern being established first. (source: FEATURES, OVERSIGHTS, UX)

**Deferred decisions (carry-forward, not implemented this milestone):**

5. **LLM-Based build-agents Validation** -- Static regex scan for namespace violations in generated agent files. Independent, low priority. (source: FEATURES)

6. **execSync to execFileSync Migration** -- Production-only migration of 346 call sites. Significant effort. Deferred from v6.0.0. (source: FEATURES, STACK, PITFALLS)

7. **Auto-Regroup on DAG Structural Changes** -- Wire `partitionIntoGroups()` call after `recalculateDAG()` in `state add-set`. Trivial 3-5 line change. Deferred from dag-central-grouping set. (source: FEATURES, PITFALLS)

### Architecture Direction

- **Recommended pattern:** Targeted extension of the existing modular monolith. No new modules, layers, or structural reorganization needed. (source: ARCHITECTURE)
- **Handoff mechanism:** File-based artifact in `.planning/` following the proven pattern (DEFERRED.md, CONTRACT.json, CONTEXT.md all use write-file-then-read-file across `/clear` boundaries). The REMEDIATION artifact is NOT part of STATE.json -- standalone file, no schema migration, no Zod changes, no withStateTransaction complexity. (source: ARCHITECTURE, FEATURES)
- **Display extension:** `display.cjs` currently handles only banner rendering. Extend with `renderFooter(nextCommand)` for standardized completion output. Skills call this at completion for consistent `/clear` guidance. (source: ARCHITECTURE, FEATURES, PITFALLS)
- **Skill prompt updates vs. code changes:** The `/clear` guidance and breadcrumb standardization are primarily prompt-level changes (SKILL.md edits), not code changes. Keep UX additions minimal in prompt text to avoid growing already-large skill files (init: ~1300 lines, execute-set: ~600 lines). Prefer runtime display utilities over prompt-embedded text. (source: ARCHITECTURE, PITFALLS)
- **Backward compatibility:** add-set must fall through to standard interactive flow when no REMEDIATION artifact exists. Simple `fs.existsSync` guard clause. (source: ARCHITECTURE)
- **State management:** No changes to STATE.json, state-schemas.cjs, or the state machine. The REMEDIATION artifact operates outside the transaction system. (source: ARCHITECTURE)

### User Experience Direction

**Domain conventions (from competitor analysis):**

- All competing products (GSD, OpenSpec) explicitly recommend `/clear` / fresh context between phases. This is the industry standard for Claude Code agent harnesses. RAPID's architecture is perfectly aligned with this pattern but does not communicate it to users. (source: UX)
- Users expect three things at the end of every command: (1) what was accomplished, (2) what to do next with exact command, (3) how to get there (context management). RAPID provides (1) and (2) but not (3). (source: UX)
- GSD's README structure is the benchmark for onboarding: problem statement -> one-command install -> 3-step quickstart -> deeper explanation. RAPID's 12-step quickstart is too many steps. (source: UX)

**Terminology alignment:**

- RAPID's "set" is unique -- competitors use "phase" (GSD) or "spec" (OpenSpec). Users coming from GSD will call sets "phases." No action needed but README/docs should note the mapping. (source: UX)
- Users may fear `/clear` loses progress. Documentation must explicitly state: "Your progress is safe -- RAPID saves everything to disk. `/clear` only resets the conversation, not your project state." (source: UX)

**Information architecture:**

- Progressive disclosure model is well-structured (help -> DOCS.md -> docs/*.md -> technical_documentation.md). The gap is the entry point (README) which skips "why" and jumps to "how." (source: UX, OVERSIGHTS)
- First-run experience needs improvement: install -> init transition assumes user understands RAPID's mental model. A 30-second orientation before init would reduce confusion. (source: OVERSIGHTS)

**Accessibility:**

- NO_COLOR support is correctly implemented. (source: UX)
- Color scheme uses semantic meaning (blue=planning, green=execution, red=review) with textual stage verbs as fallback. Good accessible design. (source: UX)
- No significant accessibility gaps identified for a CLI tool. (source: UX)

## Contradictions

### Contradiction: Handoff Artifact Naming and Location

- **FEATURES says:** Create `.planning/pending-sets/{set-name}.json` -- one file per remediation proposal in a new directory.
- **ARCHITECTURE says:** Create `.planning/v{VERSION}-REMEDIATION.json` -- a single file per audit containing all remediation entries, placed directly in `.planning/`.
- **OVERSIGHTS says:** Three options: `.planning/REMEDIATION-QUEUE.md`, extend DEFERRED.md format, or `.planning/sets/{set-name}/HANDOFF.md` pre-creation files.
- **PITFALLS says:** `.planning/v{VERSION}-REMEDIATION.json` or `.planning/AUDIT-HANDOFF.json`.
- **Resolution:** All sources agree on the principle (file-based persistence in `.planning/`). The disagreement is on naming and granularity. FEATURES recommends per-set files in a new directory; ARCHITECTURE recommends a single aggregated file. Both approaches work. The single-file approach (ARCHITECTURE) is simpler (one file to read/write, no new directory), while the per-set approach (FEATURES) is more granular (individual consumption/cleanup). **Needs design decision during discuss-set.** FEATURES' recommendation of Option 1 (dedicated pending-sets directory) has the strongest rationale: it cleanly separates "deferred" items from "remediation" items and follows existing artifact patterns.

### Contradiction: CLI Utility vs. Prompt-Level Implementation for Footer

- **FEATURES says:** Create `renderFooter(nextCommand)` in `display.cjs` exposed via `node rapid-tools.cjs display footer "{next-command}"` -- code-based single source of truth.
- **ARCHITECTURE says:** "Since skills are prompt-driven (not code-driven), the more practical approach is standardizing the footer pattern in skill prompts directly." But also acknowledges drift risk and suggests a display CLI subcommand.
- **Resolution:** Both approaches have merit. FEATURES' CLI utility approach prevents drift across 24 skills but requires a code change. ARCHITECTURE's prompt-only approach is simpler but has higher maintenance burden. **The PITFALLS research resolves this:** the anti-pattern "Modifying Multiple Skill Files Without a Shared Module" explicitly warns against copy-pasting UX improvements into each SKILL.md independently. The CLI utility approach (FEATURES) is recommended.

### Contradiction: Scope of `/clear` Guidance (Targeted vs. Universal)

- **ARCHITECTURE says:** "Strategy A (targeted, ~10 lifecycle skills) is recommended" -- skip skills like status and quick that do not have a natural next step.
- **PITFALLS says:** Adding `/clear` guidance to some skills but not all creates worse UX; the guidance must be systematic with a clear policy for which transitions need clearing.
- **OVERSIGHTS says:** Apply to all 24 skills mechanically for consistency.
- **UX says:** "Every skill should end its output with" the `/clear` line.
- **Resolution:** PITFALLS provides the most nuanced answer. The guidance should NOT be universal (status, quick, branding are not lifecycle transitions) but MUST follow a documented policy. Specifically: add to all skills that end with a "Next step: /rapid:X" suggestion AND where the next step is a different lifecycle phase. Do NOT add between review sub-steps (review -> unit-test -> bug-hunt -> uat). **The policy must be defined before implementation begins.**

## Recommended Approach

This milestone is a UX and documentation overhaul, not a structural or architectural change. The existing five-layer modular monolith architecture (skills, agents, CLI, library, storage) is sound and requires no reorganization. All four core features can be achieved through: (a) a new persistent artifact type for audit-to-set handoff, (b) a small extension to the display library for standardized completion footers, (c) systematic prompt-level updates across lifecycle skills, and (d) a README rewrite.

The critical insight from the research is that the `/clear` guidance and the audit-to-set handoff are two manifestations of the same design principle: **any data that must survive a context reset must be persisted to disk, and the system must tell users when to reset context.** The audit handoff solves the persistence side; the `/clear` footer solves the communication side. Both should be designed together since they share the same underlying principle and will be referenced by the README rewrite.

Implementation should proceed in two phases. First, establish the foundational patterns: define the `/clear` policy (which transitions require it), build the `renderFooter()` display utility, design the REMEDIATION artifact schema, and update the Node.js version requirements. Second, apply those patterns broadly: update all lifecycle skills with the footer, implement the audit-version writer and add-set reader for the handoff artifact, rewrite the README, and conduct the targeted UX audit against a predefined checklist. The deferred decisions (execSync migration, Zod 4, build-agents validation, auto-regroup) should be captured as carry-forward context for the next milestone but not implemented.

A key risk to manage is consistency across the ~24 skill file updates. The PITFALLS research emphatically warns against incremental or patchwork skill updates. The structural test recommendation from OVERSIGHTS (grep all SKILL.md files for required patterns) should be implemented BEFORE the bulk changes, so regressions are caught immediately. The display utility approach (single source of truth in code) further mitigates drift.

The README rewrite deserves particular care. Research from UX and OVERSIGHTS converges on one point: the README must answer "why should I use RAPID?" before "how do I use RAPID?" The problem statement (context rot in long AI sessions) and the solution (artifact-based state that survives `/clear`) must be the first things a new user reads. GSD's README structure (problem -> install -> 3-step quickstart) is the benchmark, not RAPID's current 12-step quickstart.

### Suggested Set Boundaries

- **Set A: clear-guidance-and-display** -- Implement `renderFooter()` utility in display.cjs, define `/clear` policy, add structural test, apply footer to all lifecycle skills (~10-12 skills). Rationale: foundation that all other UX work depends on; self-contained code+prompt changes; owned by display.cjs and SKILL.md files.

- **Set B: audit-handoff** -- Design and implement the REMEDIATION artifact (audit-version writer, add-set reader, status display integration). Rationale: core feature request; touches audit-version, add-set, and status skills; independent of Set A (different files).

- **Set C: readme-and-onboarding** -- README rewrite, install instructions fix, Node.js version update (package.json, setup.sh, README badge), first-run orientation, help command update. Rationale: documentation-focused set; depends on Set A (references the `/clear` pattern) and should come after the footer pattern is established. Pure documentation/prompt changes, no library code.

- **Set D: ux-audit** -- Breadcrumb standardization, error message improvements, remaining UX polish from predefined checklist. Rationale: depends on Sets A and B being complete so the audit validates new patterns rather than flagging known gaps. Cross-cutting but scoped to a checklist.

- **Set E (deferred): deferred-decisions** -- Auto-regroup wiring (trivial), execSync migration tracking, LLM build-agents validation, Zod 4 migration tracking. Rationale: carry-forward items that should be captured in planning artifacts but not necessarily implemented this milestone. If capacity allows, the auto-regroup wiring (3-5 lines) could be included in Set B since it touches add-set.

**Parallelism:** Sets A and B can execute in parallel (no file overlap). Set C depends on A. Set D depends on A and B. Set E is independent.
