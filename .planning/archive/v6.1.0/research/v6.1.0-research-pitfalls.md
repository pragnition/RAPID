# Pitfalls Research: v6.1.0 UX & Onboarding

## Critical Pitfalls

### Pitfall: Audit-to-Set Context Loss on /clear
- **Category:** data-loss
- **Stack:** Claude Code plugin API, RAPID skill layer
- **Description:** After `/rapid:audit-version` completes and suggests creating remediation sets via `/rapid:add-set`, the user must `/clear` before running the next command (per RAPID's own guidance). When they do, all audit context -- gap descriptions, severity ratings, remediation recommendations, and the specific scope that the auditor agent determined -- is lost from the conversation. The `/rapid:add-set` skill then asks the user to describe the set's scope from scratch via its 2-question discovery flow, even though this information was just generated by the audit pipeline.
- **Trigger:** User runs `/rapid:audit-version` -> selects "Create remediation set" -> receives a suggested `/rapid:add-set {name}` command -> runs `/clear` -> runs `/rapid:add-set {name}` -> is asked "What should this new set accomplish?" with no pre-populated context.
- **Impact:** High. Users must manually re-describe scope that was already machine-generated. Information fidelity degrades as users paraphrase from memory. This is the primary handoff gap identified in the milestone brief.
- **Prevention:** Implement a persistent handoff mechanism. The audit skill already writes `.planning/v{VERSION}-AUDIT.md` and `.planning/v{VERSION}-DEFERRED.md` to disk. A handoff file (e.g., `.planning/AUDIT-HANDOFF.json`) could store structured remediation context (set name, scope, severity, remediation description) that `/rapid:add-set` reads on startup to pre-populate its discovery flow. Alternatively, extend the `--spec` pattern already used by `/rapid:new-version` to let `add-set` accept a `--from-audit` flag.
- **Detection:** Run `/rapid:audit-version` -> create a remediation set -> `/clear` -> run `/rapid:add-set` and observe that no audit context is available.
- **References:** Milestone brief "audit-to-set handoff mechanism" feature; `skills/audit-version/SKILL.md` Step 4e prints commands but writes no machine-readable handoff artifact; `skills/add-set/SKILL.md` Step 2 has no mechanism to read pre-existing context.

### Pitfall: Inconsistent /clear Guidance Across Skills
- **Category:** correctness
- **Stack:** RAPID skill layer (28 skills)
- **Description:** The milestone brief identifies a need to tell users to `/clear` before the next RAPID command. However, 0 of the 28 existing skills currently include any `/clear` guidance in their completion output. Adding this guidance inconsistently -- to some skills but not all -- would create a worse UX than having none at all, because users would not know which commands require clearing and which do not. Furthermore, some skills chain naturally (e.g., `/rapid:review` then `/rapid:unit-test` then `/rapid:bug-hunt`) and telling users to `/clear` between every step of the review pipeline would be counterproductive.
- **Trigger:** Implementing "/clear before next command" guidance in only a subset of skills, or implementing it between skills that intentionally share context (review pipeline).
- **Impact:** High. Users either over-clear (losing useful context between review substeps) or under-clear (experiencing context rot mid-session). Inconsistent guidance erodes trust.
- **Prevention:** Define a clear policy: which skill transitions require `/clear` and which do not. At minimum, `/clear` should be recommended (a) between major lifecycle phases (plan -> execute, execute -> review, review -> merge, merge -> audit, audit -> new-version) but NOT (b) between review sub-steps (review -> unit-test -> bug-hunt -> uat). The guidance text must be identical across all skills that emit it, sourced from a single template or display utility function.
- **Detection:** Grep all SKILL.md files for `/clear` mentions after implementation. Verify that review pipeline skills do NOT include the guidance.
- **References:** No existing `/clear` mentions found in any SKILL.md files (confirmed via grep). Review pipeline chains: `skills/review/SKILL.md` suggests next steps of unit-test/bug-hunt/uat without clearing.

## High-Risk Pitfalls

### Pitfall: README Overhaul Breaking Existing Users' Mental Model
- **Category:** correctness
- **Stack:** Documentation (README.md)
- **Description:** The current README leads with value proposition and "60-Second Quickstart" showing the full lifecycle as a linear sequence of commands. Beginners may interpret this as "run all 10 commands in one session." A README rewrite that makes the install process clearer but does not also clarify that each command is a separate session (with `/clear` between them) will make the onboarding worse, not better. Additionally, the README currently shows two different install methods (marketplace plugin via `/plugin install` and git clone) with inconsistent command syntax, and the marketplace install uses a path format (`rapid@pragnition/pragnition-public-plugins`) that is not self-explanatory.
- **Trigger:** Rewriting the README to be "beginner-friendly" without explicitly addressing the session-per-command workflow and the context rot problem that RAPID solves via `/clear`.
- **Impact:** Medium-High. New users attempt to run the entire lifecycle in a single Claude Code session, hit context rot around execute-set, and blame RAPID for the failure.
- **Prevention:** The README must explicitly state: (1) each command runs in its own Claude Code session, (2) use `/clear` between commands to avoid context rot, (3) RAPID's artifacts on disk (STATE.json, PLAN.md, CONTRACT.json) carry context between sessions -- you do not need conversation memory. Include a visual flow diagram showing session boundaries.
- **Detection:** Have a new user follow the README from scratch and observe where they get confused.
- **References:** Current README at `/home/kek/Projects/RAPID/README.md` lines 44-61; no mention of `/clear` or sessions anywhere in the quickstart.

### Pitfall: Progress Breadcrumb Inconsistency
- **Category:** correctness
- **Stack:** RAPID skill layer
- **Description:** Only 7 of 28 skills include progress breadcrumbs (init, start-set, discuss-set, plan-set, execute-set, add-set, new-version). The remaining 21 skills -- including high-traffic ones like review, merge, cleanup, audit-version, unit-test, bug-hunt, uat -- have no breadcrumbs. If the UX audit adds breadcrumbs to some but not all skills, users will encounter an inconsistent experience where sometimes they see where they are in the workflow and sometimes they do not.
- **Trigger:** UX audit adds `/clear` guidance and breadcrumbs to some skills but misses others, creating a patchwork experience.
- **Impact:** Medium. Users lose orientation in the workflow at exactly the points where they need it most (post-merge, post-review).
- **Prevention:** Create a canonical breadcrumb template in `display.cjs` or a shared constant that all skills can reference. Systematically add breadcrumbs to ALL skills that represent lifecycle steps. Skills that are not lifecycle steps (help, status, install, migrate) can omit breadcrumbs.
- **Detection:** Grep all SKILL.md files for "breadcrumb" after implementation. Cross-reference against the full skill list.
- **References:** Confirmed via grep: `skills/review/SKILL.md`, `skills/merge/SKILL.md`, `skills/audit-version/SKILL.md`, `skills/cleanup/SKILL.md` have zero breadcrumb mentions.

### Pitfall: Zod 3.25.x Bundled Zod 4 Package Size and CJS Compatibility
- **Category:** compatibility
- **Stack:** Zod 3.25.76
- **Description:** Starting in Zod 3.25.0, the package bundles both Zod 3 and Zod 4, effectively doubling the package size. While RAPID's `.cjs` extension and lack of `"type": "module"` in package.json means CJS loading works correctly, the bloated package size (Zod included twice) increases install time and disk usage unnecessarily. If a future contributor adds `"type": "module"` to package.json or changes the import pattern, CJS loading could break (as documented in colinhacks/zod#4434).
- **Trigger:** Running `npm install` pulls Zod 3.25.76 which contains both Zod 3 and Zod 4 source trees. No immediate breakage, but the attack surface increases.
- **Impact:** Medium. No runtime breakage currently. Risk is (a) wasted disk space / install time, and (b) future breakage if module configuration changes.
- **Prevention:** Pin Zod to 3.24.x if the Zod 4 features are not needed. Alternatively, accept the current version but add a comment in package.json noting the dual-bundle behavior. If upgrading to Zod 4 proper (the `zod/v4` subpath), plan a migration since error map precedence and optional property behavior changed.
- **Detection:** Check `node_modules/zod` size. If it exceeds ~500KB, it contains the dual bundle.
- **References:** [Zod issue #4923](https://github.com/colinhacks/zod/issues/4923) -- "Latest Zod3 now includes Zod4, which causes breaking changes on minor zod upgrade"; [Zod issue #4434](https://github.com/colinhacks/zod/issues/4434) -- "CJS loading 3.25.7 broken"

### Pitfall: Git Worktree Concurrent Operation Corruption
- **Category:** data-loss
- **Stack:** Git 2.30+, git worktree
- **Description:** Git's internal design assumes single-process access to the `.git` directory. When multiple RAPID agents run concurrent git operations across worktrees (e.g., parallel `execute-set` waves committing simultaneously), the shared `.git` directory can experience silent corruption. RAPID mitigates this with sequential commit serialization in `execute-set`, but if the UX improvements lead to users running multiple RAPID commands in parallel terminal sessions against the same repo (e.g., executing set-1 in one terminal while planning set-2 in another), git operations can collide.
- **Trigger:** User opens two terminal sessions, runs `/rapid:execute-set 1` in one and `/rapid:plan-set 2` in another, both performing git operations against the same `.git` directory.
- **Impact:** Medium-High. Silent corruption of git index or objects. Failure mode is data loss that only surfaces later during merge.
- **Prevention:** Document in the README and onboarding that parallel RAPID commands should use the RAPID parallelism model (DAG-ordered execution within a single command invocation), not manual terminal parallelism. Consider adding a repo-level lock check at skill startup.
- **Detection:** Run `git fsck` after parallel operations. Check for "broken link" or "dangling" object warnings.
- **References:** [auto-worktree issue #176](https://github.com/kaeawc/auto-worktree/issues/176) -- "git's single-process limitation with concurrent worktree operations"; [Jon Roosevelt blog](https://jonroosevelt.com/blog/git-worktrees-broke-dedicated-machines-fixed-it) -- "shared .git directory is the fundamental problem"

## Medium-Risk Pitfalls

### Pitfall: execSync Shell Injection in prereqs.cjs
- **Category:** security
- **Stack:** Node.js child_process, prereqs.cjs
- **Description:** `prereqs.cjs` uses `execSync()` (3 call sites) which spawns a shell and is vulnerable to command injection if any user-controlled input is interpolated into the command string. While the current usage passes hardcoded command strings (e.g., `'git rev-parse --show-toplevel'`), the deferred decision to migrate `execSync` to `execFileSync` across the codebase (346 total `execSync` occurrences across 22 files vs only 28 `execFileSync` occurrences across 5 files) means the vast majority of shell command execution still uses the injection-vulnerable API. Any UX change that introduces user-provided paths or arguments into these call sites creates an injection vector.
- **Trigger:** A future change passes user-supplied input (e.g., a project path, set name) into an `execSync` call without sanitization.
- **Impact:** Medium. No current exploitation path, but the pattern is fragile. Severity escalates if RAPID is used in CI/CD environments where input comes from untrusted sources.
- **Prevention:** This is explicitly a deferred decision from the bug-fixes-foundation set. Do NOT attempt to fix this within the UX & Onboarding milestone -- it would expand scope beyond the milestone's goals. Document the risk in the pitfalls section of the audit report.
- **Detection:** Grep for `execSync` calls that interpolate variables: `execSync(\`...${...}...\`)`.
- **References:** Deferred context in milestone brief: "Migrate execSync to execFileSync in prereqs.cjs for consistency" and "Comprehensive execSync audit across entire codebase"

### Pitfall: Auto-Regroup Not Triggered After add-set
- **Category:** correctness
- **Stack:** DAG.json, state add-set CLI
- **Description:** When `/rapid:add-set` creates a new set and calls `state add-set`, the DAG is recalculated (wave assignments) but developer groups are NOT automatically regrouped. This means a newly added set may end up in a suboptimal group or no group at all, and the `dag show` display may not reflect the new set's position correctly. This is a deferred decision from the dag-central-grouping set.
- **Trigger:** User runs `/rapid:add-set` to create a remediation set from audit results. The set appears in STATE.json and DAG.json but group assignments are stale.
- **Impact:** Low-Medium. The set still works correctly for execution; only the developer group partitioning display is inaccurate.
- **Prevention:** This is explicitly a deferred decision. Do NOT fix within UX & Onboarding. Document in the add-set completion output that users should run `dag regroup` if they want updated group assignments.
- **Detection:** After `add-set`, run `node rapid-tools.cjs dag show` and check if the new set appears in a group.
- **References:** Deferred context in milestone brief: "Auto-regroup on DAG structural changes after add-set"

### Pitfall: AUDIT.md Remediation Commands Printed But Not Persisted
- **Category:** data-loss
- **Stack:** audit-version SKILL.md
- **Description:** In Step 4e of `audit-version`, remediation commands are printed to the console (e.g., `/rapid:add-set fix-auth-validation`) but are NOT written to any persistent file. The `v{VERSION}-AUDIT.md` report does not include the remediation command list. The `v{VERSION}-DEFERRED.md` file only covers deferred items, not items the user chose to remediate. When the user `/clear`s, the remediation commands are gone.
- **Trigger:** User completes audit -> sees "Run these commands to create remediation sets" in console -> `/clear` -> cannot remember the exact set names or scopes.
- **Impact:** Medium. Users must re-read the AUDIT.md report and manually reconstruct what they decided, or re-run the audit.
- **Prevention:** Write a `.planning/v{VERSION}-REMEDIATION.json` file containing structured remediation data (set names, scopes, severities). Have `/rapid:add-set` check for this file and offer to pre-populate from it.
- **Detection:** Run audit, choose "Create remediation set" for a gap, then check what files were written. Only AUDIT.md and DEFERRED.md exist -- no remediation artifact.
- **References:** `skills/audit-version/SKILL.md` Step 4e -- prints remediation commands to stdout only.

### Pitfall: proper-lockfile Stale Lock with Default stale=0
- **Category:** correctness
- **Stack:** proper-lockfile 4.1.2
- **Description:** proper-lockfile's default `stale` option is 0, meaning locks never auto-expire. RAPID configures a 300-second (5-minute) stale threshold, which is correct. However, if a developer process is killed with SIGKILL (not SIGTERM), the PID-based staleness detection in proper-lockfile may not recognize the lock as stale immediately, since it checks if the PID is still alive. On Linux, PID recycling can cause a stale lock to appear "held" by an unrelated process.
- **Trigger:** RAPID agent process receives SIGKILL (e.g., OOM killer, `kill -9`), leaving a lock. A new process inherits the same PID within the stale window.
- **Impact:** Low-Medium. Lock contention blocks all state mutations. The troubleshooting guide documents manual cleanup (`rm -rf .planning/.locks/*.target.lock`), but beginners may not know to look there.
- **Prevention:** The existing 300-second stale threshold is appropriate. Consider adding lock age display to `/rapid:status` output so users can see if stale locks exist. The troubleshooting doc already covers manual cleanup.
- **Detection:** Check for `.planning/.locks/` directory contents when commands hang.
- **References:** [proper-lockfile npm page](https://www.npmjs.com/package/proper-lockfile) documents stale detection limitations; `docs/troubleshooting.md` covers manual cleanup.

## Anti-Patterns for This Stack

### Anti-Pattern: Printing Transient Data That Must Survive /clear
- **What people do:** Print remediation commands, next-step suggestions, or structured context to the console and expect users to copy them before clearing.
- **Why it fails:** `/clear` erases the conversation. Users forget, mistype, or lose the output. This is the fundamental problem the audit-to-set handoff mechanism must solve.
- **Correct approach:** Any data that must survive a `/clear` must be written to a file in `.planning/`. Use structured JSON for machine consumption and Markdown for human readability. The `/rapid:pause` and `/rapid:resume` skills already demonstrate this pattern with `HANDOFF.md`.
- **Example:** `audit-version` should write `.planning/v{VERSION}-REMEDIATION.json` alongside the existing AUDIT.md and DEFERRED.md artifacts.

### Anti-Pattern: Inconsistent "Next Step" Guidance Patterns
- **What people do:** Some skills show next steps as `> **Next step:** /rapid:foo`, others show a `Next steps:` block with multiple options, others show a progress breadcrumb, and some show nothing at all.
- **Why it fails:** Users cannot develop muscle memory for the workflow. They do not know what to do next, or they miss that they should `/clear` first.
- **Correct approach:** Standardize on a single completion output format used by ALL lifecycle skills. The format should include: (1) completion summary, (2) progress breadcrumb, (3) explicit next command, (4) `/clear` guidance when appropriate. Implement this as a shared function in `display.cjs` to prevent drift.
- **Example:** A `renderCompletion(stage, nextStep, clearRequired)` function that all skills call.

### Anti-Pattern: README That Assumes Knowledge of AI Agent Workflows
- **What people do:** Write quickstart sections that list commands without explaining that each is a separate AI agent invocation requiring a fresh context window.
- **Why it fails:** Users who are new to Claude Code plugins do not understand that `/rapid:execute-set` is an autonomous agent session that can take 10-30 minutes and consume 100K+ tokens. They may try to interact with it mid-execution or run subsequent commands in the same session.
- **Correct approach:** Lead with a conceptual overview: "Each RAPID command launches an AI agent that runs autonomously. When it finishes, clear your context with `/clear` before running the next command." Include estimated session durations for each command.

### Anti-Pattern: Modifying Multiple Skill Files Without a Shared Module
- **What people do:** Copy-paste the same UX improvement (e.g., `/clear` guidance, breadcrumb template) into each SKILL.md independently.
- **Why it fails:** The improvements inevitably diverge as different agents make slightly different edits. When the template needs to change, 28 files must be updated instead of 1.
- **Correct approach:** Create a shared display utility (extend `display.cjs`) or a shared Markdown template that skills include by reference. For prompt-level content (SKILL.md), define the canonical text in one place and reference it.

## Version-Specific Gotchas

- **Node.js 20+**: The `node:test` runner added `--experimental-test-snapshots` in v22. If any UX tests need snapshot testing, they will not work on Node 20. RAPID's minimum is 20, but CI should test on both 20 and latest LTS.
- **Zod 3.25.76**: Contains Zod 4 at `zod/v4` subpath. Do not accidentally `require('zod/v4')` -- it has different error map precedence and optional property semantics.
- **Git 2.30+**: Worktree `--orphan` flag was added in Git 2.38. RAPID does not use orphan worktrees, but if any UX improvement creates branches for documentation or context storage, ensure compatibility with Git 2.30.
- **proper-lockfile 4.1.2**: The `update` option interval must be less than `stale` to prevent false-positive stale detection. RAPID's configured values are correct; do not change one without the other.

## Security Pitfalls

1. **execSync shell injection surface**: 346 `execSync` calls across 22 files. Any interpolation of user input creates a command injection vector. Deferred to future security hardening set -- do NOT expand scope to fix in this milestone.
2. **AUDIT.md and DEFERRED.md may contain sensitive requirement descriptions**: If the project handles PII or security-critical features, audit reports could leak requirement details. These files are in `.planning/` which is typically committed to git. Consider adding a `.gitignore` pattern for sensitive audit artifacts.
3. **Lock files in `.planning/.locks/` are world-readable**: On shared systems, another user could read lock metadata (PID, timestamps). Low risk for single-developer setups, but relevant for team environments.

## Performance Traps

1. **Audit on large milestones triggers TWO_PASS threshold at 5 sets**: The audit-version skill uses a two-pass analysis when `setCount >= 5`. This spawns the auditor agent twice -- once for summary scan and once for deep-dive. With 10+ sets, the summary table alone can consume significant context. The threshold is appropriate but should not be lowered.
2. **Progress breadcrumb rendering in display.cjs**: If breadcrumbs are added to all 28 skills by modifying the prompt text, each skill's prompt grows. For skills that already approach the model's context limit (execute-set with large plans), even small prompt increases can push execution context into compaction. Implement breadcrumbs as a post-execution display, not embedded in the prompt template.
3. **SKILL.md file sizes**: Several skill files are already large (init: ~1300 lines, execute-set: ~600 lines, merge: ~660 lines, new-version: ~780 lines). Adding UX boilerplate to each increases agent prompt sizes, reducing the context budget available for actual work. Keep UX additions minimal in prompt text; prefer runtime display.

## Summary Priority List

1. **Audit-to-set context loss on /clear** -- Remediation commands are printed but not persisted; `/add-set` cannot read audit context after `/clear`. This is the milestone's headline feature and the most impactful pitfall to solve.
2. **Inconsistent /clear guidance across skills** -- Adding `/clear` guidance to some skills but not all creates worse UX than the current state. Requires a systematic policy defining which transitions need clearing.
3. **README assumes AI workflow knowledge** -- Quickstart shows commands as a linear list without explaining session boundaries, context rot, or `/clear`. Beginners will run everything in one session and hit context rot.
4. **Progress breadcrumb inconsistency** -- Only 7/28 skills have breadcrumbs. Review, merge, audit, and cleanup -- the phases where users most need orientation -- have none.
5. **Remediation commands not persisted to disk** -- audit-version prints `/rapid:add-set` commands to console but writes no machine-readable file. Context lost on `/clear`.
6. **Git worktree concurrent operation risk** -- README/onboarding should warn against manual terminal parallelism; RAPID's internal parallelism model is safe but users may bypass it.
7. **Zod 3.25.76 dual-bundle bloat** -- Package is twice the expected size due to bundled Zod 4. Not a runtime issue but affects install time and could cause future CJS breakage.
8. **Auto-regroup not triggered after add-set** -- Deferred decision; document in add-set output rather than attempting to fix.
9. **execSync shell injection surface** -- Deferred to future security set; do not expand scope.
10. **SKILL.md size growth from UX additions** -- Large prompt files reduce context budget. Keep UX additions minimal and prefer runtime display over prompt-embedded text.
