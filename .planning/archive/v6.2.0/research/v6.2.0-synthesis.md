# Research Summary -- DX Refinements (v6.2.0)

## Project Profile
- **Type:** brownfield CLI plugin (Claude Code plugin for parallel development)
- **Stack:** Node.js (>=22, CommonJS) + Zod 3.25.76 + node:http + node:test
- **Scale:** small milestone (3 features, moderate complexity) on a mature codebase (21 core modules, 26 agents, 24 skills)
- **Key Constraint:** Zero new production dependencies -- all features must be implementable with Node.js built-in APIs (source: STACK, FEATURES, PITFALLS, OVERSIGHTS)

## Unified Findings

### Stack and Infrastructure

- Node.js `fs.watch({ recursive: true })` is confirmed working on Node 22+ across platforms. Sufficient for watching the small `.planning/branding/` directory (~20 files). No need for `chokidar`. (source: STACK)
- Server-Sent Events (SSE) are natively supported via `node:http` with `text/event-stream` content type. No WebSocket library needed. (source: STACK, FEATURES, ARCHITECTURE, UX)
- All 4 production dependencies (zod 3.25.76, ajv 8.18.0, ajv-formats 3.0.1, proper-lockfile 4.1.2) are at their latest stable versions within their respective major lines. No updates required for this milestone. (source: STACK)
- Zod v4 (4.3.6) exists but is NOT relevant to this milestone. The `.passthrough()` deprecation and CJS loading instabilities make it a risk. Deferred to a future migration set. (source: STACK, PITFALLS)
- The `^3.25.76` semver range for Zod allows automatic patch upgrades to potentially broken 3.25.x versions that bundle Zod 4 internally. Pinning the exact version is recommended. (source: PITFALLS)
- The existing branding server (`branding-server.cjs`) already provides working HTTP infrastructure: PID tracking, health probes, static file serving, hub page generation, and path traversal protection. (source: STACK, ARCHITECTURE)
- `version.cjs` already exports `getVersion()` and `versionCheck()` with semver comparison. Extending it for update staleness is natural. (source: STACK)
- No CI pipeline, no linter, no formatter, no code coverage. Not relevant to this milestone but noted. (source: STACK)

### Architecture Direction

- **Pattern:** Extended modular monolith (unchanged from current). No new architectural boundaries needed. (source: ARCHITECTURE)
- **Module boundaries:**
  - `branding-server.cjs` -- HTTP server with SSE, artifact API, and live-reload. Public API gains `notifyClients(event, data)`. (source: ARCHITECTURE)
  - `branding-artifacts.cjs` (NEW) -- Artifact type registry, manifest CRUD, file operations. Owns `artifacts.json` and `artifacts/` subdirectory. (source: ARCHITECTURE)
  - `version.cjs` -- Gains `readInstallTimestamp()`, `writeInstallTimestamp()`, `isUpdateStale()`. (source: ARCHITECTURE)
  - Skill files modified in-place for init integration and update reminder injection. (source: ARCHITECTURE, FEATURES)
- **Data flow:** Agent writes artifact via POST `/_artifacts` -> `branding-artifacts.cjs` creates file + updates manifest -> `branding-server.cjs` pushes SSE event -> browser updates DOM. (source: ARCHITECTURE)
- **State management:** Branding artifacts managed via `artifacts.json` manifest (NOT STATE.json). Install timestamp stored in a dedicated file at plugin root (NOT STATE.json, NOT config.json). SSE connections held in-memory, ephemeral. (source: ARCHITECTURE, PITFALLS, OVERSIGHTS)

### User Experience Direction

**Autoreload expectations (from UX):**
- Industry standard: automatic browser refresh on file changes is baseline for any dev preview server (Vite, webpack-dev-server, BrowserSync). Manual refresh is perceived as a tooling deficiency.
- SSE-based auto-reload with injected inline `<script>` (~15 lines) is the right pattern: zero config, zero dependencies, sub-500ms latency, graceful reconnection via EventSource built-in retry.
- Hub page should evolve from flat file listing to artifact card gallery with type badges, timestamps, and preview thumbnails. Each artifact appears as it is created via SSE push.
- A "Reconnecting..." indicator should appear if the SSE connection drops for >2 seconds.
(source: UX, FEATURES)

**Update reminder expectations (from UX):**
- Follow the `update-notifier` convention: deferred display (AFTER command output), non-blocking, one-liner with exact command to run, suppressible via `NO_UPDATE_NOTIFIER` env var.
- TTY-only display: do not show in piped/scripted contexts.
- Show on `/rapid:install` and `/rapid:status` as primary trigger points; avoid adding latency to every skill invocation.
- Respect `NO_COLOR` convention in all new terminal output.
(source: UX, OVERSIGHTS)

**Init branding integration expectations (from UX):**
- Core questions first, optional enrichment later. Branding is enrichment, not prerequisite.
- Single opt-in question with one-sentence explanation and prominent Skip option. "Skip" must have zero negative consequences.
- Context-aware framing: "brand guidelines" for webapps, "output style and communication guidelines" for CLIs/libraries.
- Cap at 1 question during init (opt-in gate only); defer full interview to standalone `/rapid:branding`.
(source: UX, OVERSIGHTS)

**Artifact browsing model (from UX):**
- Progressive generation sequence: BRANDING.md -> index.html -> logo.svg -> wireframe.html -> guidelines.html
- Each artifact appears on the hub page as it is created, with auto-reload ensuring no manual intervention.
- Agent generates sequentially; user reviews each before proceeding to the next.
(source: UX, FEATURES)

**Accessibility (from UX):**
- `NO_COLOR` must be respected in all new terminal output (existing `display.cjs` pattern).
- Hub page keyboard navigation: all artifact links and interactive elements focusable via Tab/Enter.
- Existing color contrast (#58a6ff on #0d1117) meets WCAG AA (~5.3:1). Maintain this in redesign.
- Auto-reload should include a subtle visual indicator so the page change is not disorienting.
(source: UX)

## Contradictions

### Contradiction: Update Timestamp Storage Location

- **STACK says:** Store `updated_on` in root `config.json` rather than `.env`, because `.env` is overwritten by `setup.sh` (cat > not cat >>). (source: STACK, recommendation #2)
- **FEATURES says:** Store `RAPID_UPDATED_ON` in the `.env` file using the existing `_parseEnvFile()` pattern from `web-client.cjs`. The `.env` file already exists and is managed by `setup.sh`. (source: FEATURES, option 1)
- **ARCHITECTURE says:** Store in a dedicated `.rapid-install-timestamp` JSON file at plugin root to avoid `.env` parsing fragility and `config.json` git noise. (source: ARCHITECTURE)
- **PITFALLS says:** Store in plugin-root `.env` file because it is the natural location for plugin-level metadata, but warns about storage location ambiguity. (source: PITFALLS)
- **OVERSIGHTS says:** Store in a dedicated `.rapid-meta.json` at the plugin root. Avoid `.env` (sourced as env vars, fragile parsing) and `config.json` (committed to git, causes noise). (source: OVERSIGHTS)
- **Resolution:** Three distinct recommendations exist: `.env` (FEATURES, PITFALLS), `config.json` (STACK), and dedicated file (ARCHITECTURE, OVERSIGHTS). The strongest argument is for a dedicated file: `.env` has the `setup.sh` overwrite risk (STACK) and fragile parsing (OVERSIGHTS), while `config.json` is committed to git (OVERSIGHTS). A dedicated plugin-root JSON file (e.g., `.rapid-install-meta.json`) avoids both problems. **Recommend: dedicated file.** However, the STACK research also notes that `config.json` "already stores runtime configuration and survives reinstalls" -- this is a valid counterpoint if the git noise concern is acceptable. This may need a user decision if `.env` simplicity is preferred.

### Contradiction: Branding Init Integration -- Inline vs Sub-Skill Invocation

- **FEATURES says:** Use Agent tool to invoke `/rapid:branding` as a sub-skill from init. Clean separation, no code duplication, branding skill stays independent. Add `--inline` / `--skip-lifecycle` flag. (source: FEATURES, recommended option 1)
- **ARCHITECTURE says:** Run the branding interview inline within the init skill using the same AskUserQuestion budget. Avoids context-switching overhead and nested skill context. (source: ARCHITECTURE, recommendation #3)
- **PITFALLS says:** The branding skill expects scaffolded project state that does not exist during init step 4B. Must use a SUBSET of the branding skill (interview rounds only), skipping env setup, codebase detection, and server start. (source: PITFALLS)
- **OVERSIGHTS says:** Keep branding-in-init as a single opt-in question, not a full interview. If "Yes", invoke branding as a sub-flow. (source: OVERSIGHTS)
- **Resolution:** Both approaches are viable. The inline approach (ARCHITECTURE) avoids agent overhead but requires duplicating branding interview logic in init. The sub-skill approach (FEATURES) avoids duplication but adds agent spawn overhead and requires the branding skill to handle pre-scaffolding state gracefully. PITFALLS confirms that naive insertion will fail because `.planning/` does not exist yet. **Recommend: inline approach with a minimal question set (1-2 questions).** The full branding interview (4-5 questions) is too heavy for init context (OVERSIGHTS), and the sub-skill approach adds complexity for handling the pre-scaffolding state. A condensed inline variant sidesteps both issues.

### Contradiction: Whether to Start Branding Server During Init

- **FEATURES says:** Start the server during init; the server keeps running through the rest of init so users can see artifacts appear. (source: FEATURES, decomposition step 3)
- **ARCHITECTURE says:** Start the server during branding step in init; it is non-blocking. (source: ARCHITECTURE, data flow)
- **PITFALLS says:** Do NOT start the branding server during init. It creates a background process that outlives the conversation, adds error handling complexity, and mixes server lifecycle into init. (source: PITFALLS, anti-pattern #3)
- **OVERSIGHTS says:** `server.unref()` is needed if server starts during init, otherwise the process hangs. Also notes that if the user closes Claude Code, the server becomes orphaned. (source: OVERSIGHTS)
- **Resolution:** PITFALLS makes the strongest argument. Starting the server during init adds significant complexity (port conflicts without user prompts, orphaned processes, lifecycle management in a flow that already has 10+ steps). The init flow should only generate BRANDING.md + index.html and display a message: "Run `/rapid:branding` to preview." **Recommend: do NOT start server during init.**

## Recommended Approach

The v6.2.0 DX Refinements milestone contains three features of varying complexity that map cleanly to two parallel workstreams. The branding webserver overhaul is the most complex feature and should drive the primary set, while the update reminder is fully independent and can proceed in parallel.

The branding overhaul should be implemented bottom-up: first the infrastructure layer (SSE endpoint, `fs.watch` with debouncing, artifact manifest schema and `branding-artifacts.cjs` module), then the presentation layer (hub page redesign with artifact cards and injected auto-reload script), then the API layer (artifact CRUD endpoints), and finally the skill layer (extended branding flow with sequential artifact generation). This ordering ensures each layer has a stable foundation. The `fs.watch` implementation must include robust debouncing (200-500ms) because the API is notoriously flaky across platforms, and a manual refresh fallback button should be included as a safety valve. SSE is the correct transport -- all six research files converge on this recommendation -- and the injected client script should be minimal (~15 lines) with automatic reconnection.

The init flow branding integration should be kept deliberately minimal: one opt-in question at step 4B.5, and if accepted, a condensed inline interview (2-3 questions maximum) rather than invoking the full branding skill. The branding server should NOT be started during init -- only BRANDING.md and index.html should be generated, with a message directing users to `/rapid:branding` for the full preview experience. This avoids the circular dependency problem (`.planning/` may not exist at step 4B), the process lifecycle complexity, and the AskUserQuestion budget exhaustion risk that multiple research agents flagged.

The update reminder should store its timestamp in a dedicated plugin-root JSON file (e.g., `.rapid-install-meta.json`) rather than `.env` or `config.json`. The `.env` approach has an overwrite risk from `setup.sh`, and `config.json` is committed to git. The reminder should follow the `update-notifier` convention: deferred display after command output, non-blocking, actionable (includes exact command), and suppressible. It should appear on `/rapid:status` and `/rapid:install` only, not on every skill invocation, to avoid adding latency.

All features must maintain the zero-dependency constraint. No `chokidar`, no `ws`, no `livereload`. Node.js built-in `fs.watch`, `http`, and `EventSource` are sufficient. The Zod version should be pinned to exactly `3.25.76` to prevent the known CJS loading instability in the 3.25.x line.

### Suggested Set Boundaries

- **Set A: `branding-overhaul`** -- SSE auto-reload infrastructure, `branding-artifacts.cjs` module, artifact manifest schema, hub page redesign, artifact CRUD API, extended branding skill flow, branding-server.cjs tests.
  - Rationale: All branding server changes are tightly coupled (SSE depends on artifact model, hub page depends on manifest, skill flow depends on CRUD API). Keeping them in one set avoids cross-set coordination.
  - Wave 1: SSE + fs.watch + artifact manifest + branding-artifacts.cjs
  - Wave 2: Hub page redesign + artifact CRUD API + client-side auto-reload script

- **Set B: `init-branding-integration`** -- Init SKILL.md step 4B.5 insertion, branding SKILL.md inline mode support, context passing from init discovery.
  - Rationale: Modifying the init flow is a separate concern from the server overhaul. This set has a soft dependency on Set A (better experience with overhaul complete) but functions independently. It is primarily skill-markdown changes, not library code.
  - Wave 1: Init SKILL.md modification + branding SKILL.md inline mode

- **Set C: `update-reminder`** -- Timestamp recording in setup.sh and install skill, staleness check in version.cjs, CLI subcommand, reminder banner injection in status/install skills.
  - Rationale: Fully independent of both branding sets. Touches different files (version.cjs, setup.sh, install/status skill markdowns). Can execute in parallel with Sets A and B.
  - Wave 1: Timestamp storage + staleness check function + CLI subcommand + tests
  - Wave 2: Skill-level reminder injection + opt-out mechanism

- **Set D: `hygiene`** (optional) -- Pin Zod exact version, update SKILL.md version strings, update `.env.example`, `server.unref()` addition.
  - Rationale: Small mechanical fixes that do not fit cleanly in the feature sets. Can be a quick cleanup set or folded into the final wave of any feature set.
