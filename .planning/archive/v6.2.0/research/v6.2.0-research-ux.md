# Domain & UX Research -- DX Refinements (v6.2.0)

## Domain Conventions

RAPID operates in the **developer tooling / CLI plugin** domain. The primary interaction surface is the Claude Code terminal, with a secondary browser-based preview for branding artifacts. Conventions from this domain that apply to the three milestone features:

### Auto-Reload Dev Servers

Industry-standard developer preview servers (Vite, webpack-dev-server, BrowserSync, LiveReload) all provide automatic browser reload on file changes. The dominant implementation patterns are:

- **Server-Sent Events (SSE):** Lightweight, unidirectional push from server to browser. Used by LiveReload.js modernized implementations and `serve-reload-replace`. No external dependencies required -- Node.js `http` module plus `fs.watch` suffices. (Source: [Live Reloading From Scratch in Node.JS](https://www.alexander-morse.com/blog/live-reloading-from-scratch-in-nodejs/), [serve-reload-replace](https://github.com/ahmadnassri/node-serve-reload-replace))
- **WebSocket-based:** Full-duplex, used by Vite HMR and webpack-dev-server. More complex but supports bidirectional communication. Requires a WebSocket library or `ws` dependency. (Source: [webpack HMR docs](https://webpack.js.org/concepts/hot-module-replacement/))
- **Polling-based:** Simplest implementation -- client periodically fetches a version/hash endpoint. Higher latency but zero client-side dependencies. Used as a fallback in many implementations.

For RAPID's branding server (which serves static artifacts, not a live application), SSE is the appropriate pattern: it requires zero new dependencies, integrates naturally with `fs.watch` on the `.planning/branding/` directory, and the injected client script is minimal (~15 lines of JavaScript). Full HMR is unnecessary because branding artifacts are complete file replacements, not module-level patches.

### CLI Update Reminders

The `update-notifier` pattern (used by npm, yarn, Angular CLI, Gatsby CLI, and 3000+ npm packages) establishes the canonical UX for CLI update reminders:

1. **Timestamp-based staleness:** Record installation/update time; check against a configurable interval (typically 1 day to 1 week). (Source: [update-notifier npm](https://www.npmjs.com/package/update-notifier))
2. **Deferred display:** Show the update notification AFTER the command output completes, not before or during. This prevents interrupting the user's primary workflow. (Source: [Trabe -- Improve UX of CLI tools](https://medium.com/trabe/improve-the-ux-of-cli-tools-with-version-update-warnings-23eb8fcb474a))
3. **Non-blocking check:** The update check runs in a background/unref'd process so it never slows command execution. (Source: [update-notifier GitHub](https://github.com/sindresorhus/update-notifier))
4. **Opt-out mechanism:** Users can suppress notifications via environment variable (`NO_UPDATE_NOTIFIER`) or config flag. (Source: [update-notifier docs](https://www.npmjs.com/package/update-notifier))
5. **TTY-only display:** Notifications only render in interactive terminals, not in piped/scripted contexts. (Source: [simple-update-notifier](https://www.npmjs.com/package/simple-update-notifier))

RAPID should follow this pattern but with a simpler implementation: since RAPID does not check a remote registry, it only needs a local `updated_on` timestamp compared against `Date.now()`. No background process or network call is required.

### Init Flow Branding Integration

Developer tool initialization wizards (Yeoman, create-react-app, Nx, Turborepo) follow a consistent progressive disclosure pattern:

- **Core questions first, optional enrichment later.** The critical path (name, stack, features) completes before optional steps (themes, linting config, CI setup). (Source: [UX patterns for CLI tools](https://lucasfcosta.com/2022/06/01/ux-patterns-cli-tools.html))
- **Explicit skip option.** Every optional step offers a clearly-labeled skip path. Users who decline should not see degraded behavior elsewhere. (Source: [Appcues onboarding UX guide](https://www.appcues.com/blog/user-onboarding-ui-ux-patterns))
- **Brief context before the question.** One sentence explaining what the optional step does and why, before asking the user to commit time to it. (Source: [Choosing the right onboarding UX pattern](https://www.appcues.com/blog/choosing-the-right-onboarding-ux-pattern))

The existing RAPID init flow has steps 4A (logistics) -> 4B (discovery) -> 4C (granularity) -> 4D (confirmation) -> 4E (principles). Branding belongs between 4B and 4C because it requires project context from discovery (project type, visual vs CLI nature) but should not block the structural decisions in granularity/confirmation.

## Interaction Models

### Primary: Claude Code Terminal (CLI)

RAPID is a Claude Code plugin. All user interaction occurs through slash commands (`/rapid:init`, `/rapid:branding`, `/rapid:install`) in the Claude Code terminal. The interaction model is:

- **Command-driven with AskUserQuestion prompts:** Users invoke a skill, the agent asks structured questions with prefilled options, the user selects or provides freeform input.
- **Structured return protocol:** Agent outputs end with machine-parseable `RAPID:RETURN` markers indicating COMPLETE, CHECKPOINT, or BLOCKED status.
- **Banner + footer framing:** Each skill displays an ANSI-colored stage banner at start and a box-drawn footer with next-action guidance at end.

### Secondary: Browser Preview (Branding Server)

The branding server is a secondary interaction surface, used only during and after `/rapid:branding`:

- **Current pattern:** User must manually refresh browser after each artifact update. This is friction that the auto-reload feature eliminates.
- **Expected pattern (post-milestone):** Artifacts appear on a browsable hub page. When the agent writes a new file to `.planning/branding/`, the browser reloads automatically via injected SSE client. The user passively observes artifact creation in real time.

### Artifact Browsing Model

The milestone brief describes an "artifact system" where the model creates artifacts (logo, theme, wireframe, branding guidelines page) on a browsable page. This maps to a pattern seen in:

- **Claude Artifacts:** Side-panel rendering of generated content (HTML, SVG, code) with preview/code toggle. (Source: [Claude Artifacts guide](https://albato.com/blog/publications/how-to-use-claude-artifacts-guide))
- **Storybook:** Browsable component library with live preview per artifact.
- **Design system documentation sites:** Browsable pages with color swatches, typography samples, spacing demos.

The RAPID branding hub page already implements a basic version of this (file listing with links). The enhancement is: each artifact gets its own preview card on the hub, the page auto-updates when new artifacts are created, and the agent sequentially generates artifacts (theme -> logo -> wireframe -> guidelines) with the user reviewing each one.

## Information Architecture

### Branding Artifact Hierarchy

The branding artifacts should follow a progressive generation sequence that matches the design process:

1. **BRANDING.md** -- Core guidelines (colors, typography, terminology). Generated first because all subsequent artifacts derive from it.
2. **index.html** -- Visual preview of the guidelines. Generated second for human verification.
3. **logo.svg** (or similar) -- Brand mark. Optional, generated after guidelines are confirmed.
4. **wireframe.html** -- Site layout wireframe using the brand tokens. Optional, offered after logo.
5. **guidelines.html** -- Comprehensive branding guidelines page combining all artifacts. Generated last as a summary.

Each artifact should appear on the hub page as it is created, with the SSE-based auto-reload ensuring the hub page updates without user intervention.

### Update Reminder Placement

The update reminder should appear at a natural pause point in the CLI interaction:

- **After command completion, before the footer.** This mirrors the `update-notifier` convention of appending the notice after output.
- **Not during command execution.** The reminder must never interrupt an active workflow.
- **Contextually relevant.** Show on `/rapid:install` (where the user is already in a setup/maintenance mindset) and optionally on `/rapid:status` (where staleness information is contextually relevant).

### Init Flow Information Architecture

The current init flow steps:

```
Step 1: Prerequisites
Step 2: Git check
Step 3: Existing project detection
Step 4A: Logistics (name, team, model)
Step 4B: Deep discovery (13 areas in 4 batches)
--- Proposed insertion: Step 4B.5: Branding offer ---
Step 4C: Granularity preference
Step 4D: Summary confirmation + acceptance criteria
Step 4E: Principles capture
Step 5: Scaffold
...
```

The branding step should be framed as: "Based on the project context gathered, would you like to define brand guidelines now? This takes ~2 minutes and shapes how all agents communicate and style output." Options: "Yes, run branding interview" / "Skip for now (run /rapid:branding later)".

## Accessibility Considerations

### Terminal Output

- **NO_COLOR support:** The existing `display.cjs` respects the `NO_COLOR` environment variable (https://no-color.org). The update reminder and any new terminal output must also respect this convention.
- **Screen reader compatibility:** ANSI escape codes are stripped by assistive technology. The existing fallback format (`--- RAPID > VERB ---`) is already screen-reader-friendly.
- **Box-drawing characters:** The footer uses Unicode box-drawing (`=`, `|`). The NO_COLOR fallback uses ASCII (`+`, `-`, `|`), which is more universally compatible.

### Browser Preview

- **Keyboard navigation:** The branding hub page should ensure all artifact links and interactive elements are focusable and navigable with Tab/Enter.
- **Color contrast:** The existing hub page uses `#58a6ff` on `#0d1117` (blue on dark background). This meets WCAG AA contrast ratio requirements (approximately 5.3:1).
- **Auto-reload announcement:** The SSE-triggered reload should not cause disorientation. A brief visual indicator (e.g., a subtle flash or "Updated" toast) helps users understand why the page changed.

### Not Applicable

- **Internationalization:** RAPID is an English-only developer tool. i18n patterns do not apply.
- **Touch/mobile interfaces:** The browser preview is designed for desktop development contexts.

## User Expectations

### Branding Server Auto-Reload

Users who have worked with Vite, webpack-dev-server, or any modern dev server expect:

- **Automatic browser refresh when files change.** Manual refresh is perceived as a tooling deficiency, not a feature.
- **No configuration required.** The auto-reload should work out of the box. No flags, no browser extensions, no separate watcher process.
- **Sub-second latency.** The delay between file write and browser update should be imperceptible (< 500ms).
- **Graceful reconnection.** If the SSE connection drops (server restart, network hiccup), the client should automatically reconnect.

### Update Reminder

Users of CLI tools (npm, pip, brew, rustup) expect update reminders to be:

- **Non-intrusive.** A boxed notice after command output, not a modal dialog or blocking prompt.
- **Actionable.** The reminder should include the exact command to run (e.g., "Run /rapid:install to update").
- **Suppressible.** Power users expect an environment variable or config flag to disable reminders.
- **Accurate.** The reminder should not appear when the tool is already up-to-date. The staleness check should compare against a concrete timestamp, not a heuristic.

### Branding in Init Flow

Users running `/rapid:init` for the first time expect:

- **The init flow to complete core setup quickly.** Branding is perceived as an enhancement, not a prerequisite. Inserting it mid-flow risks frustrating users who want to get to coding.
- **Clear skip option.** The offer must clearly communicate that skipping has zero negative consequences.
- **Context awareness.** If the project is a CLI tool or library (not a webapp), the branding step should adjust its framing -- "brand guidelines" resonates more with webapp developers than CLI tool authors. For CLI projects, frame it as "output style and communication guidelines."

### Mental Models from Competing Products

- **Vite/webpack users** expect any preview server to auto-reload. The current manual-refresh pattern in RAPID's branding server is below the expected baseline.
- **npm/yarn users** expect update notifications to appear after command output with a box-drawn border, similar to npm's yellow update notice box.
- **Yeoman/create-react-app users** expect optional init steps to be clearly marked and skippable without consequence.

## Recommendations

1. **Implement SSE-based auto-reload for branding server:** Use Node.js `fs.watch` on `.planning/branding/` and inject a minimal SSE client into served HTML pages. No new dependencies required. The `_generateHubPage()` function should include a `<script>` block with an EventSource listener that triggers `location.reload()` on file change events. -- Priority: **high**

2. **Use the deferred-display update reminder pattern:** Record `RAPID_UPDATED_ON` timestamp in `.env` during `/rapid:install`. On subsequent CLI invocations, compare against current date. If >7 days stale, append a non-blocking reminder after command output. Respect `NO_UPDATE_NOTIFIER` env var and TTY-only display. -- Priority: **high**

3. **Insert branding as optional Step 4B.5 in init flow:** After discovery (4B) completes and before granularity (4C), offer branding with a one-sentence explanation and a prominent "Skip" option. If accepted, delegate to the existing `/rapid:branding` skill inline. If skipped, continue to 4C with no side effects. -- Priority: **high**

4. **Frame the branding offer based on detected project type:** For webapp projects, use "brand guidelines" framing. For CLI/library projects, use "output style and communication guidelines" framing. This matches the existing project-type detection in the branding skill's Step 2. -- Priority: **medium**

5. **Add artifact sequencing to branding flow:** After initial BRANDING.md + index.html generation, offer sequential artifact creation: logo -> wireframe -> guidelines page. Each artifact appears on the auto-reloading hub page as it is generated. The user reviews each before the agent proceeds to the next. -- Priority: **medium**

6. **Implement SSE reconnection in the client script:** Use EventSource's built-in reconnection with a small backoff. Display a brief "Reconnecting..." indicator if the connection drops for >2 seconds. -- Priority: **low**

7. **Add opt-out mechanism for update reminders:** Support `NO_UPDATE_NOTIFIER=1` environment variable and a `config.json` flag (`"updateReminder": false`) for users who want to suppress the notice permanently. -- Priority: **low**

8. **Preserve the current NO_COLOR and accessibility patterns:** All new terminal output (update reminder box, branding offer prompt) must respect `NO_COLOR` and use the existing ASCII fallback conventions from `display.cjs`. -- Priority: **critical**
