# Architecture Research

## Current Architecture

RAPID follows a 5-layer stack (Skill -> Agent -> CLI -> Library -> Storage) with 21 core library modules, 26 generated agents, and 24 skill definitions. The current v6.2.0 architecture is a mature modular monolith that has proven stable across 20+ milestones. This DX Refinements milestone does not challenge the fundamental architecture -- it extends three specific subsystems: the branding webserver, the init flow, and the install lifecycle.

### Strengths Relevant to This Milestone

1. **Branding already cleanly isolated.** The branding subsystem (`branding-server.cjs`, `role-branding.md`, `skills/branding/SKILL.md`) is a self-contained module with no inbound dependencies from core. Branding context is optionally injected into execution via `buildBrandingContext()` in `execute.cjs`. This isolation makes it safe to overhaul the server without cascading changes.
2. **Init flow is step-sequenced.** The init skill (`skills/init/SKILL.md`) follows a numbered step sequence (0.5 through 9+) with clear sub-phases (4A logistics, 4B discovery, 4C granularity, 4D confirmation, 4E principles). Inserting a new step is straightforward -- the architecture supports it.
3. **Install is skill-only.** The install skill is a `disable-model-invocation: true` procedural skill with no library-layer dependencies beyond `prereqs.cjs` and `setup.sh`. Adding timestamp tracking is a minimal change.
4. **Well-tested server module.** `branding-server.cjs` has comprehensive unit tests covering start/stop lifecycle, stale PID cleanup, path traversal prevention, and hub page generation. The test harness uses ephemeral ports and temp directories.

### Weaknesses Exposed by This Milestone

1. **Static-only server.** The current branding server is a read-only static file server with no client-side interactivity. It generates the hub page on each request by scanning the filesystem (`_generateHubPage()`), but the browser must be manually refreshed to see new files. There is no push mechanism (SSE, WebSocket, or polling) for live updates.
2. **Artifact model is implicit.** Branding artifacts are "whatever files exist in `.planning/branding/`." There is no manifest, no metadata per artifact, and no concept of artifact types (logo, theme, wireframe, guidelines). The hub page treats all files equally.
3. **Init flow has no branding integration point.** Branding is currently invoked separately via `/rapid:branding`. The init flow has no hook, callback, or optional-step mechanism for inserting auxiliary skills mid-flow.
4. **No install timestamp infrastructure.** The `.env` file currently stores only `RAPID_TOOLS=<path>`. There is no timestamp field, no version field, and no mechanism to read/check staleness.

## Recommended Architecture Pattern

- **Pattern:** Extended modular monolith (unchanged from current)
- **Rationale:** All three features (webserver overhaul, init integration, update reminder) are incremental extensions to existing modules. No new architectural boundaries are needed. The branding server stays in-process, the init flow gains one optional step, and the install skill gains one check at the start.
- **Evidence:** The v6.0.0 milestone successfully extended the same monolith pattern with 6 new modules (group.cjs, principles.cjs, etc.) without stability issues. This milestone's scope is smaller.

## Project Structure

The existing directory layout requires minimal additions. Changes are contained within established locations:

```
src/
  lib/
    branding-server.cjs        [OVERHAUL] Add SSE-based autoreload, artifact registry,
                                           POST endpoint for artifact creation
    branding-artifacts.cjs     [NEW]      Artifact type definitions, manifest management,
                                           file write hooks for SSE notification
    version.cjs                [EXTEND]   Add installTimestamp read/write/check helpers
  commands/
    init.cjs                   [NO CHANGE] No library-layer changes needed for init integration
  modules/
    roles/
      role-branding.md         [EXTEND]   Multi-phase branding flow (theme -> logo -> wireframe -> guidelines)

skills/
  branding/SKILL.md            [OVERHAUL] Expanded multi-phase flow with artifact creation
  init/SKILL.md                [EXTEND]   Insert branding as Step 4B.5 (after discovery, before granularity)
  install/SKILL.md             [EXTEND]   Record timestamp in Step 1, check staleness in Step 0

.planning/
  branding/
    .server.pid                [EXISTS]   PID tracking (unchanged)
    BRANDING.md                [EXISTS]   Authoritative branding guidelines (unchanged)
    index.html                 [EXISTS]   Visual preview (now autoreloaded)
    artifacts.json             [NEW]      Artifact manifest with types and metadata
    artifacts/                 [NEW]      Subdirectory for generated artifacts (logos, wireframes)
```

Rationale: New functionality lives in new files (`branding-artifacts.cjs`, `artifacts.json`) rather than overloading existing ones. The server module is overhauled in-place because its public API (`start`, `stop`, `status`) remains the same -- only the internal request handler and client-side HTML change.

## Module Organization

### Module: branding-server.cjs (OVERHAUL)

- **Responsibility:** HTTP server for branding artifacts with live-reload capability
- **Public interface:** `start(projectRoot, port)`, `stop(projectRoot)`, `status(projectRoot)`, `notifyClients(event, data)` (new)
- **Dependencies:** `node:http`, `node:fs`, `node:path`, `branding-artifacts.cjs` (new)
- **Data ownership:** `.planning/branding/.server.pid`, SSE client connections
- **Key changes:**
  1. Add SSE endpoint (`/_events`) that pushes file-change notifications to connected browsers
  2. Add POST endpoint (`/_artifacts`) for creating new artifacts from the branding skill
  3. Inject autoreload `<script>` tag into served HTML pages (EventSource-based)
  4. Replace static hub page with a dynamic artifact browser showing typed artifacts

### Module: branding-artifacts.cjs (NEW)

- **Responsibility:** Artifact type registry, manifest management, file-system operations for branding artifacts
- **Public interface:** `createArtifact(projectRoot, { type, name, content })`, `listArtifacts(projectRoot)`, `loadManifest(projectRoot)`, `saveManifest(projectRoot, manifest)`
- **Dependencies:** `node:fs`, `node:path`
- **Data ownership:** `.planning/branding/artifacts.json`, `.planning/branding/artifacts/*`
- **Design notes:**
  - Artifact types: `theme` (color palette, typography), `logo` (SVG/PNG), `wireframe` (HTML mockup), `guidelines` (compiled branding page)
  - Manifest format: `{ artifacts: [{ id, type, name, path, createdAt, description }] }`
  - File operations are synchronous (matching codebase convention) with atomic writes

### Module: version.cjs (EXTEND)

- **Responsibility:** Version checking and install timestamp management
- **Public interface:** `getVersion()`, `versionCheck()` (existing), `readInstallTimestamp(pluginRoot)` (new), `writeInstallTimestamp(pluginRoot)` (new), `isUpdateStale(pluginRoot, thresholdDays)` (new)
- **Dependencies:** `node:fs`, `node:path`, `prereqs.cjs`
- **Data ownership:** Reads `.env` or `.rapid-install-timestamp` for timestamp data
- **Design notes:**
  - Timestamp stored in a dedicated file (`.rapid-install-timestamp`) at plugin root rather than in `.env`, because `.env` is a flat key-value format and adding structured data to it introduces parsing fragility
  - File contains JSON: `{ "updatedOn": "ISO-8601", "version": "6.2.0" }`
  - `isUpdateStale()` returns `{ stale: boolean, daysSinceUpdate: number, installedVersion: string }` -- pure check with no side effects

### Module: skills/init/SKILL.md (EXTEND)

- **Responsibility:** Project initialization flow orchestration
- **Public interface:** Skill definition (no code API)
- **Dependencies:** CLI commands, AskUserQuestion, Agent tool
- **Changes:** Insert optional branding step as Step 4B.5 between discovery (4B) and granularity preference (4C)

### Module: skills/install/SKILL.md (EXTEND)

- **Responsibility:** Plugin installation and configuration
- **Public interface:** Skill definition (no code API)
- **Dependencies:** setup.sh, CLI commands, AskUserQuestion
- **Changes:** Write timestamp after successful install (Step 1 extension), check staleness at entry (Step 0 extension)

### Module: skills/branding/SKILL.md (OVERHAUL)

- **Responsibility:** Branding interview and artifact generation
- **Public interface:** Skill definition (no code API)
- **Dependencies:** branding-server.cjs, AskUserQuestion, Write tool
- **Changes:** Multi-phase flow: theme -> optional logo generation -> optional wireframe -> branding guidelines page. Each phase creates an artifact via the server's POST endpoint, triggering autoreload.

## Data Flow

### Branding Server Autoreload Flow

```
1. Agent writes artifact file via branding-server POST /_artifacts
   -> branding-artifacts.cjs creates file + updates artifacts.json
   -> branding-server.cjs calls notifyClients('artifact-created', { id, type, name })

2. Browser has open EventSource connection to /_events
   -> Receives SSE message: event: artifact-created, data: { id, type, name }
   -> Client-side JS inserts new artifact card into the hub page DOM
   -> If artifact is HTML (wireframe, guidelines), optionally auto-navigates or shows preview

3. On file modification (theme update, logo regen):
   -> POST /_artifacts with overwrite flag
   -> SSE pushes 'artifact-updated' event
   -> Browser re-fetches the specific artifact and updates its preview
```

### Init Flow with Branding Integration

```
Step 4A: Logistics (name, team size, model)
Step 4B: Deep discovery (13 areas, 4 batches)
    |
    v
Step 4B.5: Optional Branding [NEW]
    -> AskUserQuestion: "Set up project branding now?"
       - "Yes" -> Run abbreviated branding interview (4 questions)
                  -> Generate BRANDING.md + index.html
                  -> Start branding server, display URL
       - "Skip" -> Continue silently
    |
    v
Step 4C: Granularity preference
Step 4D: Summary confirmation
Step 4E: Principles capture
Step 5:  Scaffold
...
```

Key constraint: The branding step must be NON-BLOCKING. If the user skips it, zero additional time is added. If they accept it, the interview runs inline (same session, same AskUserQuestion budget). The server starts but does not block the flow -- the user can view it in a browser tab while the init continues.

### Update Reminder Flow

```
/rapid:install execution:
    Step 0 (new): Read .rapid-install-timestamp
       -> If file exists AND daysSince > 7:
          Display: "RAPID was last updated {N} days ago. Consider running /rapid:install to update."
       -> If file missing: Skip (first install)
    ...
    Step 1: Run setup.sh (existing)
    Step 1.5 (new): Write .rapid-install-timestamp
       -> { "updatedOn": "ISO-8601", "version": "6.2.0" }
    ...
```

### State Management

- **Branding artifacts:** Managed by `branding-artifacts.cjs` via `artifacts.json` manifest. No integration with STATE.json (branding remains independent of project lifecycle state).
- **Install timestamp:** Stored in `.rapid-install-timestamp` at plugin root. Read-only from skills; written only by install. Not part of project state.
- **SSE connections:** Held in memory by the branding server process. No persistence -- connections are ephemeral. Server cleanup on `stop()` closes all SSE connections.

## API Design

### Branding Server HTTP API (Extended)

| Endpoint | Method | Purpose | Response |
|----------|--------|---------|----------|
| `/_health` | GET | Health probe | `{"status":"ok"}` |
| `/` | GET | Artifact hub page (dynamic) | HTML with SSE client script |
| `/_events` | GET | SSE event stream | `text/event-stream` |
| `/_artifacts` | POST | Create/update artifact | `{"id","type","name","path"}` |
| `/_artifacts` | GET | List all artifacts | `{"artifacts":[...]}` |
| `/{filename}` | GET | Static file serving | File content with MIME type |

SSE event types:
- `artifact-created` -- New artifact added to manifest
- `artifact-updated` -- Existing artifact content changed
- `connected` -- Initial connection acknowledgement with current artifact list

POST `/_artifacts` body:
```json
{
  "type": "logo|theme|wireframe|guidelines",
  "name": "Project Logo v1",
  "filename": "logo.svg",
  "content": "<svg>...</svg>",
  "overwrite": false
}
```

### CLI Commands (No Changes)

The existing CLI command surface (`node rapid-tools.cjs ...`) does not need new subcommands for this milestone. All branding artifact creation flows through the HTTP API. The install timestamp flows through file I/O in `version.cjs`.

## Integration Points

| External System | Protocol | Purpose | Error Handling |
|----------------|----------|---------|---------------|
| Browser (local) | HTTP + SSE | Branding artifact preview with live reload | Graceful: server start failures are non-fatal, SSE disconnects auto-reconnect |
| `.env` file | File I/O | RAPID_TOOLS path reading | Already handled via env preamble |
| `.rapid-install-timestamp` | File I/O | Install freshness tracking | Missing file = first install, parse errors = skip check |
| AskUserQuestion API | Claude Code tool | User interaction during init/branding | Fallback to defaults if tool unavailable |
| Agent tool | Claude Code tool | Subagent spawning for branding in init flow | Standard RAPID error handling |

## Scaling Considerations

This milestone's features are all local-only (localhost HTTP server, local file I/O, single-user skill invocations). Scaling is not a primary concern. However:

- **SSE connection limit:** The branding server should cap SSE connections at a reasonable limit (e.g., 10) to prevent resource exhaustion from accidental browser tab accumulation. Each SSE connection holds an open HTTP response.
- **Artifact directory size:** The `artifacts/` subdirectory could grow if logo generation produces many variants. The manifest should track artifact count and the hub page should paginate or virtualize if count exceeds ~50.
- **Install timestamp check latency:** Reading a single JSON file on install startup adds negligible overhead (<1ms).

## Architecture Risks

1. **SSE complexity in a CJS module:** Server-Sent Events require holding response objects open and managing client disconnection. The current `branding-server.cjs` closes all connections cleanly. Adding SSE requires careful connection lifecycle management (tracking active connections, handling `close` events, avoiding memory leaks from abandoned connections). -- **Mitigation:** Keep a `Set` of active SSE response objects. Remove on `close` event. Clear all on `stop()`.

2. **Init flow length:** Adding branding as Step 4B.5 makes the already-long init flow (up to 13 discovery areas + 4 logistics + principles + confirmation) even longer. -- **Mitigation:** Make branding strictly opt-in with a single AskUserQuestion gate. Default option should be "Skip" to avoid adding friction for users who do not want branding. Consider offering an abbreviated 2-question interview variant for the init context (vs. the full 4-question standalone branding interview).

3. **Branding server port conflicts during init:** If the user accepts branding during init, the server starts on port 3141. If another RAPID session is running branding separately, port conflicts occur. -- **Mitigation:** The existing `port_in_use` error handling already prompts for an alternative port. No additional work needed.

4. **Timestamp file permissions/locations:** The `.rapid-install-timestamp` file lives at the plugin root. For marketplace installs, the plugin root is in `~/.claude/plugins/...` which may have restricted write permissions. -- **Mitigation:** Use `try/catch` around timestamp writes; skip gracefully if write fails. The update reminder is advisory, not blocking.

5. **Artifact POST endpoint security:** The POST `/_artifacts` endpoint creates files on disk. While the server binds to `127.0.0.1` only (localhost), any local process could create artifacts. -- **Mitigation:** The server already binds to `127.0.0.1`. Add path sanitization to prevent writing outside `.planning/branding/`. Reject filenames with path separators or `..` components.

## Recommendations

1. **Use SSE over WebSocket for autoreload:** SSE is simpler (unidirectional server-to-client), requires no additional dependencies (no `ws` package), and is natively supported by all modern browsers via `EventSource`. The branding server only needs to push notifications to the browser, not receive messages from it. -- **Priority: critical** (core of the webserver overhaul feature)

2. **Store install timestamp in a dedicated file, not .env:** The `.env` file is a flat key-value format parsed by shell `export` commands. Adding structured data (JSON timestamps) would break the existing `grep -v '^#' .env | xargs` parsing pattern used throughout skills. A separate `.rapid-install-timestamp` JSON file is cleaner. -- **Priority: critical** (prevents breaking existing .env consumers)

3. **Insert branding as a gated optional step in init, not an inline skill invocation:** Rather than invoking `/rapid:branding` as a sub-skill (which would create a nested skill context), run the branding interview inline within the init skill using the same AskUserQuestion budget. This avoids context-switching overhead and keeps the init flow as a single conversational session. -- **Priority: high** (architectural decision for init integration)

4. **Separate artifact storage from BRANDING.md:** Generated artifacts (logos, wireframes) should live in `.planning/branding/artifacts/` subdirectory, not alongside `BRANDING.md` and `index.html`. This keeps the root branding directory clean and makes the manifest unambiguous. `BRANDING.md` remains the authoritative text artifact; everything else is visual/generated. -- **Priority: high** (clean separation of concerns)

5. **Add autoreload script injection rather than requiring a custom HTML template:** The server should inject a `<script>` tag before `</body>` in any served HTML file that establishes the EventSource connection. This way, the existing `index.html` (and any future HTML artifacts) get autoreload for free without modifying their source. -- **Priority: high** (ergonomic design -- artifacts should not need to know about the server)

6. **Cap the branding interview at 2-3 questions when invoked from init:** The standalone `/rapid:branding` skill uses 4-5 AskUserQuestion calls. When embedded in the init flow (which already uses 8-12 calls), budget is constrained. Use 2 questions for visual identity + terminology, and defer component style and interaction patterns to a later standalone branding run. -- **Priority: medium** (UX discipline -- init should not exceed 15 total AskUserQuestion calls)

7. **Make the update reminder non-intrusive:** The staleness check should produce a single-line advisory message, not a blocking prompt. Format: `"RAPID was installed {N} days ago (v{X}). Run /rapid:install to check for updates."` Do not use AskUserQuestion for this -- it should be informational only. -- **Priority: medium** (UX -- update nagging is a common anti-pattern)

8. **Add connection heartbeat to SSE endpoint:** Browsers and proxies may close idle SSE connections. Send a `:keepalive` comment every 30 seconds to maintain the connection. -- **Priority: low** (robustness -- only matters for long branding sessions)

9. **Implement deferred decisions as backlog items, not architecture changes:** The 6 deferred items (backlog --list, context regen, execSync migration, auto-regroup, multi-language stubs) require no architectural preparation in this milestone. They are future work with no structural dependencies on this milestone's features. -- **Priority: low** (scope containment)
