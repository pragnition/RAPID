# Pitfalls Research -- v6.0.0 (Scale & Quality)

## Critical Pitfalls

### Pitfall: STATE.json Overwrite by Roadmapper Agent (Known Bug)
- **Category:** data-loss
- **Stack:** RAPID init pipeline (skills/init/SKILL.md, role-roadmapper.md)
- **Description:** Step 9 of the init skill writes `STATE.json` from the roadmapper's `state` output using the Write tool, which performs a full file overwrite. If the roadmapper returns an incomplete state structure (missing `version`, `rapidVersion`, `createdAt`, or `lastUpdatedAt` fields), the resulting STATE.json fails Zod validation on next read. More critically, any valid state data written between scaffold (Step 5) and roadmap acceptance (Step 9) is silently replaced.
- **Trigger:** The roadmapper agent returns `state` JSON that lacks required ProjectState fields. The init SKILL.md writes this directly without passing through `ProjectState.parse()` validation. Additionally, if any intermediate step writes to STATE.json between Steps 5 and 9, that work is lost.
- **Impact:** Critical -- the entire project state is replaced with potentially incomplete data. All sets written by `scaffoldProject()` are lost. Subsequent `readState()` calls may fail with `STATE_VALIDATION_ERROR`.
- **Prevention:** (1) Validate the roadmapper's `state` output against `ProjectState.parse()` before writing. (2) Use `withStateTransaction()` to merge roadmapper output into existing state rather than replacing it. (3) The init SKILL.md Step 9c should use a merge strategy: read existing STATE.json, overlay only `milestones[].sets` and `currentMilestone` from the roadmapper output, and preserve `version`, `rapidVersion`, `createdAt`, `lastUpdatedAt`.
- **Detection:** `node "${RAPID_TOOLS}" state get` fails after init completes. `readState()` returns `{ valid: false }`.
- **References:** Milestone bug fix #3 "Roadmapper state overwrite"

### Pitfall: REQUIREMENTS.md Overwrite During Scaffold (Known Bug)
- **Category:** data-loss
- **Stack:** RAPID init pipeline (init.cjs, skills/init/SKILL.md)
- **Description:** Step 5 calls `init scaffold`, which runs `scaffoldProject()` in `init.cjs`. The function `scaffoldProject()` generates `REQUIREMENTS.md` from `generateRequirementsMd()` -- a blank template. In `fresh` and `reinitialize` modes, this overwrites any user-approved acceptance criteria written in Step 4D. Step 4D's instructions do say to check if the file exists and append, but Step 5 runs after 4D and the scaffold `fresh` mode unconditionally writes all files.
- **Trigger:** Running `/rapid:init` on a fresh project. Step 4D writes acceptance criteria to `.planning/REQUIREMENTS.md`, then Step 5 calls `scaffoldProject(cwd, opts, 'fresh')` which overwrites `REQUIREMENTS.md` with the blank template.
- **Impact:** Critical -- all user-approved acceptance criteria from the discovery phase are silently lost. The roadmapper in Step 9 receives a blank REQUIREMENTS.md, producing a roadmap with no criteria traceability.
- **Prevention:** (1) Remove `REQUIREMENTS.md` from `scaffoldProject()` file generators -- it should only be written by Step 4D. (2) Or reorder: run scaffold before 4D. (3) Or add `REQUIREMENTS.md` to a skip-list when Step 4D has already written it.
- **Detection:** After init completes, `.planning/REQUIREMENTS.md` contains the blank template instead of the acceptance criteria from the discovery conversation.
- **References:** Milestone bug fix #1 "REQUIREMENTS.md overwrite"

### Pitfall: DAG v1/v2 Schema Confusion When Adding Group Annotations
- **Category:** correctness
- **Stack:** dag.cjs (`createDAG` vs `createDAGv2`), add-set.cjs (`recalculateDAG`)
- **Description:** The codebase has two DAG versions: v1 (`createDAG`) and v2 (`createDAGv2`). The v2 format supports typed nodes (set/wave/job) with a `version: 2` field, while v1 has no version field. The `recalculateDAG()` function in `add-set.cjs` calls `createDAG` (v1), not `createDAGv2`. Adding developer group annotations to DAG nodes requires extending the DAG schema, but if the extension targets v2 while `recalculateDAG` still calls v1, all group annotations are silently dropped on every DAG recalculation.
- **Trigger:** Adding group annotations to DAG.json (for the "DAG-driven set grouping" feature), then performing any operation that calls `recalculateDAG()` -- including `dag generate` and `addSetToMilestone()`. The v1 `createDAG()` strips any properties not in its output template.
- **Impact:** Critical -- developer group assignments are silently lost whenever the DAG is recalculated, which happens on every `add-set` operation and on `dag generate`. The user sees their groups disappear without warning.
- **Prevention:** (1) Decide on one DAG version and use it consistently. (2) If extending v1, ensure `createDAG()` preserves extra node properties (currently it does not -- it spreads `...node` then overwrites `wave` and `status`). Actually, looking at the code, `createDAG()` line 155 does `{ ...node, wave, status }` which DOES preserve extra properties. But `recalculateDAG()` at line 99 creates nodes as `{ id: s.id }` only, stripping any existing annotations. (3) `recalculateDAG()` must read existing DAG annotations and carry them forward.
- **Detection:** After running `dag generate`, check if DAG.json nodes still contain group annotations. If they are gone, the preservation logic is missing.
- **References:** DAG.json feature ("DAG as central concept"), DAG-driven set grouping feature

## High-Risk Pitfalls

### Pitfall: `--desc` vs `--description` Flag Mismatch (Known Bug)
- **Category:** correctness
- **Stack:** CLI (rapid-tools.cjs line 57), skills/init/SKILL.md (line 481)
- **Description:** The CLI usage string documents the flag as `--desc` and the argument parser accepts `--desc`. However, the init SKILL.md instructs the orchestrating agent to pass `--description`. Since the CLI uses manual argument parsing (not a flag library), unrecognized flags are silently ignored or cause misaligned argument indexing.
- **Trigger:** Every `/rapid:init` invocation that reaches Step 5 scaffold. The agent passes `--description "{text}"` but the CLI expects `--desc`.
- **Impact:** High -- the project description is silently dropped during scaffolding. PROJECT.md gets an empty or undefined description field. Downstream research agents receive no project description.
- **Prevention:** (1) Standardize on one flag name and update both the CLI parser and the SKILL.md. (2) Add flag aliases in the CLI parser so both `--desc` and `--description` are accepted. (3) Add integration tests that verify the init scaffold command round-trips the description.
- **Detection:** After `init scaffold`, check PROJECT.md for the presence of the expected description text.
- **References:** Milestone bug fix #2 "--desc vs --description flag mismatch"

### Pitfall: `fileOwnership` Not in CONTRACT_META_SCHEMA Causes Silent Validation Bypass
- **Category:** correctness
- **Stack:** contract.cjs (CONTRACT_META_SCHEMA), add-set.cjs (recalculateDAG)
- **Description:** The `CONTRACT_META_SCHEMA` in `contract.cjs` uses `additionalProperties: false`, which means any `fileOwnership` field in CONTRACT.json would be REJECTED by `compileContract()`. However, `recalculateDAG()` in `add-set.cjs` reads `contractJson.fileOwnership` to build OWNERSHIP.json. This creates a contradiction: contracts that include `fileOwnership` fail meta-schema validation, but contracts without it produce empty ownership maps.
- **Trigger:** Any CONTRACT.json that includes a `fileOwnership` field will fail `compileContract()` validation. Contracts that pass validation will have no `fileOwnership`, causing `recalculateDAG()` to produce empty ownership maps.
- **Impact:** High -- either contract validation is broken (rejecting valid contracts) or ownership tracking is broken (always empty). The scaffold overhaul feature depends on ownership for minimizing file conflicts between developer groups.
- **Prevention:** Add `fileOwnership` to the `CONTRACT_META_SCHEMA` as an optional array of strings. Alternatively, use a separate field path that the meta-schema already allows (e.g., within `exports`).
- **Detection:** Run `compileContract()` on a CONTRACT.json that includes `fileOwnership` -- it will return `{ valid: false }`.
- **References:** Feature #3 "/scaffold command overhaul" depends on correct file ownership tracking

### Pitfall: `passthrough()` in Zod Schemas Allows Silent State Corruption
- **Category:** correctness / data-loss
- **Stack:** Zod 3.25.76, state-schemas.cjs
- **Description:** Every schema in `state-schemas.cjs` uses `.passthrough()`, which allows any extra properties to pass through validation without error. While this provides flexibility (e.g., preserving unknown fields from newer RAPID versions), it means that malformed data (typos in field names, wrong nesting, extra junk) is silently accepted and persisted. The "DAG as central concept" feature will add new fields to state objects (e.g., group assignments, DAG metadata); `.passthrough()` means validation will never catch misspelled or misplaced fields.
- **Trigger:** Any state mutation that includes a typo (e.g., `set.stauts = 'pending'` instead of `set.status = 'pending'`) passes validation. The correct field retains its default value, the typo is persisted, and downstream code reads the wrong value.
- **Impact:** High -- bugs from state corruption are extremely difficult to diagnose because the state "looks" valid but has unexpected extra properties and default values for fields that were meant to be set.
- **Prevention:** (1) For new fields added in this milestone (group annotations, DAG metadata), use explicit schema fields rather than relying on passthrough. (2) Consider adding a `strict()` validation mode for development/testing that rejects unknown properties. (3) Add focused unit tests that verify state round-trips preserve expected fields and ONLY expected fields.
- **Detection:** After a state mutation, read STATE.json and check for unexpected properties at each level. Unit tests with `Object.keys()` assertions can catch this.
- **References:** Zod documentation on `.passthrough()` vs `.strict()`: https://zod.dev/?id=passthrough

### Pitfall: Shell Injection via `execSync` Template String in worktree.cjs
- **Category:** security
- **Stack:** Node.js child_process, worktree.cjs (line 21)
- **Description:** The `gitExec()` function in `worktree.cjs` constructs a shell command using template string interpolation: `` execSync(`git ${args.join(' ')}`) ``. If any argument contains shell metacharacters (`;`, `|`, `$()`, backticks), they will be interpreted by the shell. Set names flow through this function, and with the scaffold overhaul allowing user-specified group names, untrusted input may reach this code path.
- **Trigger:** A set name or branch name containing shell metacharacters, e.g., `my-set; rm -rf /` or `$(malicious-command)`. While RAPID currently uses sanitized set names from the roadmapper, the scaffold overhaul feature introduces user-specified group names that could be less controlled.
- **Impact:** High -- arbitrary command execution on the developer's machine. In practice, mitigated by the fact that inputs are typically generated by AI agents, but the code path exists.
- **Prevention:** (1) Use `execFileSync('git', args, opts)` instead of `` execSync(`git ${args.join(' ')}`) `` -- `execFileSync` does not invoke a shell, so metacharacters are passed literally. (2) The `state-machine.cjs` module already uses `execFileSync` correctly (lines 493, 508, 511) -- apply the same pattern to `worktree.cjs`. (3) Validate set/group names against a strict regex (e.g., `/^[a-z0-9][a-z0-9-]*$/`).
- **Detection:** Code review of all `execSync` callsites. `grep -n "execSync\(\`" src/` will find template-string command construction.
- **References:** Node.js docs on `execFileSync` vs `execSync`: https://nodejs.org/api/child_process.html#child_processexecfilesyncfile-args-options

### Pitfall: DAG Group Assignment Race Condition During Parallel Set Operations
- **Category:** correctness
- **Stack:** add-set.cjs, dag.cjs, lock.cjs
- **Description:** The "DAG-driven set grouping" feature assigns sets to developer groups based on DAG analysis. `recalculateDAG()` reads STATE.json and all CONTRACT.json files, computes the DAG, and writes DAG.json and OWNERSHIP.json. These file writes are NOT protected by the state lock -- only STATE.json writes use `withStateTransaction()`. If two agents are working in parallel (one adding a set, another running `dag generate`), the DAG.json can be overwritten with stale data.
- **Trigger:** Two concurrent operations: (1) `addSetToMilestone()` which calls `recalculateDAG()` after its state transaction, and (2) another agent calling `dag generate` at the same time. Both read STATE.json at different moments and write DAG.json without coordination.
- **Impact:** High -- DAG.json ends up with stale or inconsistent data. Group assignments may reference sets that no longer exist or miss newly added sets.
- **Prevention:** (1) Protect DAG.json writes with a dedicated lock (e.g., `acquireLock(cwd, 'dag')`). (2) Or include DAG recalculation within the state transaction itself, so DAG and state are always consistent. (3) At minimum, read STATE.json inside a lock before computing the DAG.
- **Detection:** After parallel set operations, verify that DAG.json contains exactly the sets present in STATE.json.

### Pitfall: Spec File Integration Clobbers Research Agent Independence
- **Category:** correctness
- **Stack:** init pipeline (Step 7), research agent roles
- **Description:** The "/init spec file integration" feature allows users to provide pre-existing spec files during init. Research agents are designed to be independent -- they "do NOT read another research agent's output" (SKILL.md Step 7 comment). If spec files are injected into all 6 research agents as additional context, and those spec files contain opinionated architectural or stack decisions, the research agents will anchor on the spec's decisions rather than independently evaluating alternatives. This defeats the purpose of parallel independent research.
- **Trigger:** User provides a spec file that includes specific technology choices (e.g., "use PostgreSQL with Prisma ORM"). All 6 research agents receive this as context. The stack agent, architecture agent, and features agent all anchor on PostgreSQL/Prisma instead of independently evaluating options.
- **Impact:** High -- research quality degrades because agents confirm the spec rather than critically evaluating it. The synthesis step produces a biased summary.
- **Prevention:** (1) Route spec files only to the research-synthesizer agent, not to individual research agents. Let the synthesizer reconcile spec assertions with independent research findings. (2) Or provide spec content to research agents with an explicit framing: "The user has provided this spec. Evaluate its assertions critically -- recommend alternatives if the spec's choices have significant drawbacks." (3) Tag spec-derived assertions differently from research-derived findings in the synthesis output.
- **Detection:** After init, review research outputs to check if they independently evaluate alternatives or merely echo the spec file's decisions.

## Medium-Risk Pitfalls

### Pitfall: `proper-lockfile` Stale Threshold Too Long for CI Environments
- **Category:** performance / correctness
- **Stack:** proper-lockfile 4.1.2, lock.cjs
- **Description:** The `STALE_THRESHOLD` is set to 300,000ms (5 minutes). In CI/CD or containerized environments where processes may be killed abruptly (OOM killer, timeout), stale locks persist for 5 minutes before being auto-cleared. The `cleanStaleLocks()` function checks PID liveness, but in containerized environments, PID reuse is common and a new process may have the same PID as the dead lock owner.
- **Trigger:** A RAPID process is killed mid-transaction in a Docker container. A new RAPID process starts, gets the same PID (container PID namespaces start from 1), and `cleanStaleLocks()` incorrectly concludes the lock owner is still alive.
- **Impact:** Medium -- lock is stuck for up to 5 minutes. In interactive use this is a minor annoyance; in CI pipelines it can cause timeouts or pipeline failures.
- **Prevention:** (1) Add a timestamp check to `cleanStaleLocks()`: if the lock target's timestamp is older than `STALE_THRESHOLD`, force-clean regardless of PID status. (2) Make `STALE_THRESHOLD` configurable via `config.json` for CI environments. (3) Store additional metadata (hostname, start time) in the lock target file.
- **Detection:** After a process crash, observe whether subsequent RAPID commands hang waiting for the lock.
- **References:** proper-lockfile stale detection: https://github.com/moxystudio/node-proper-lockfile#options

### Pitfall: `createDAG()` Spreads Extra Properties But `recalculateDAG()` Strips Them
- **Category:** correctness
- **Stack:** dag.cjs (line 155), add-set.cjs (line 99)
- **Description:** The `createDAG()` function preserves extra node properties via `{ ...node, wave, status }`. This means if you pass `{ id: 'set-01', group: 'team-alpha' }`, the output node will include `group: 'team-alpha'`. However, `recalculateDAG()` creates nodes as `milestone.sets.map(s => ({ id: s.id }))` -- stripping all properties except `id`. Even if DAG annotations exist in the current DAG.json, `recalculateDAG()` does not read the existing DAG before rebuilding.
- **Trigger:** (1) DAG.json is annotated with group assignments. (2) Any operation calls `recalculateDAG()` (e.g., `addSetToMilestone`, `dag generate`). (3) The new DAG.json has nodes with only `id`, `wave`, `status` -- group assignments are gone.
- **Impact:** Medium-high -- this is a design flaw that becomes critical when the "DAG-driven set grouping" feature is implemented. Group annotations must survive DAG recalculation.
- **Prevention:** (1) Store group annotations outside DAG.json (e.g., in STATE.json per-set metadata, or in a separate GROUPS.json). (2) Or modify `recalculateDAG()` to read the existing DAG.json and merge annotations into the new nodes before writing. (3) Add a dedicated `metadata` or `annotations` field to DAG nodes that is preserved across recalculations.
- **Detection:** Unit test: create a DAG with annotated nodes, call `recalculateDAG()`, verify annotations persist.

### Pitfall: PRINCIPLES.md and CLAUDE.md Agents Context Size Bloat
- **Category:** performance
- **Stack:** Agent prompt assembly (execute.cjs, worktree.cjs)
- **Description:** The "/init meta principles" feature captures architectural/style decisions in both `CLAUDE.md` (summary) and `PRINCIPLES.md` (full version). The `generateScopedClaudeMd()` function in `worktree.cjs` already assembles context from multiple sources (project CLAUDE.md, contracts, definitions). Adding PRINCIPLES.md content to every agent's context may push prompt sizes past the `agent_size_warn_kb` threshold (in `config.json`), especially for agents that already receive substantial context (e.g., executors receiving plan + contract + definition + scoped CLAUDE.md).
- **Trigger:** A project with extensive principles (20+ architectural decisions). Each agent receives the full PRINCIPLES.md on every invocation, adding 5-15KB to every prompt.
- **Impact:** Medium -- increases token cost, may cause context window pressure for smaller model configurations, and dilutes the signal-to-noise ratio of agent prompts.
- **Prevention:** (1) Include only the CLAUDE.md summary in agent prompts, not the full PRINCIPLES.md. (2) Make PRINCIPLES.md opt-in per agent role (e.g., only planners and architects read it, not unit testers). (3) Use the existing `compaction.cjs` module to summarize principles when they exceed a threshold.
- **Detection:** After implementing the feature, check agent prompt sizes with `wc -c` on generated agent context. Compare before and after.

### Pitfall: Scaffold Overhaul Stub Replacement on Shared Branch Creates Merge Conflicts
- **Category:** correctness
- **Stack:** stub.cjs, worktree.cjs, merge.cjs
- **Description:** The "/scaffold command overhaul" feature specifies "stubs live on shared branch, replaced by real implementations." When a stub on the shared branch is replaced by a real implementation in a set branch, every other set branch that also has the shared branch as a parent will see a merge conflict at the stub file location. The 5-level conflict detection in `merge.cjs` will flag these as L1 (textual) conflicts, but the resolution is always "take the real implementation" -- a pattern that should be auto-resolvable.
- **Trigger:** (1) Scaffold creates stubs on the shared branch for 3 sets. (2) Set A implements its real code, replacing stub-A. (3) When Set B merges, it encounters a conflict at stub-A's location because the shared branch has the stub but Set A's branch has real code. (4) This compounds with every set.
- **Impact:** Medium -- merge conflicts are expected but can be numerous. Without auto-resolution logic for stub-to-real replacements, every merge requires manual or AI-assisted resolution.
- **Prevention:** (1) Add a stub marker comment at the top of generated stubs: `// RAPID-STUB: auto-generated, safe to overwrite`. (2) Add a T1 (deterministic auto-resolve) rule in merge.cjs that detects stub markers and always takes the non-stub version. (3) Or use a different strategy: instead of replacing stubs, have real implementations live at different paths and update imports via CONTRACT.json.
- **Detection:** During merge, check if conflicted files contain the `RAPID-STUB` marker. If so, auto-resolution should handle it.

### Pitfall: Foundation Set #0 Creates Implicit Dependency for All Other Sets
- **Category:** correctness
- **Stack:** DAG (dag.cjs), init pipeline, scaffold skill
- **Description:** The scaffold overhaul includes an "optional foundational set #0 as shared work before groups branch." If Set #0 is a DAG node with all other sets depending on it, the DAG becomes a star graph with Set #0 at the root. This means: (a) no sets can start until Set #0 completes, eliminating early parallelism; (b) `assignWaves()` puts Set #0 alone in Wave 1 and everything else in Wave 2+; (c) Set #0 changes require re-merging into every active worktree.
- **Trigger:** Using the scaffold overhaul with the foundation set option. The roadmapper adds Set #0 as a dependency for all other sets.
- **Impact:** Medium -- significantly reduces parallelism in the early phases. For a 10-set project, Wave 1 has only 1 set instead of the potential 3-4. Developer groups sit idle waiting for Set #0 to complete.
- **Prevention:** (1) Make Set #0 a "scaffold" step that commits to the shared branch BEFORE worktrees are created -- not a tracked set in the DAG. (2) If it must be a DAG set, minimize its scope to the absolute minimum shared infrastructure. (3) Allow partial completion: sets can start once Set #0's CONTRACT.json is satisfied, even if Set #0 has not fully completed.
- **Detection:** After roadmap generation, check the DAG for a single root node with all other nodes depending on it. If found, warn the user about the parallelism impact.

### Pitfall: Ajv `additionalProperties: false` Conflicts with Schema Evolution
- **Category:** compatibility
- **Stack:** Ajv 8.18.0, contract.cjs
- **Description:** The `CONTRACT_META_SCHEMA` in `contract.cjs` uses `additionalProperties: false`. This means adding any new field to CONTRACT.json (e.g., `fileOwnership`, `group`, `dagMetadata`, `principles`) will cause existing contracts to fail validation. Each feature in this milestone that needs to add contract metadata must first update the meta-schema, or the feature will be blocked.
- **Trigger:** Any code that calls `compileContract()` on a CONTRACT.json with new fields not yet in the meta-schema.
- **Impact:** Medium -- development velocity is blocked until the schema is updated. In parallel development, one team's schema change can break another team's validation.
- **Prevention:** (1) Update `CONTRACT_META_SCHEMA` first as a foundational change before implementing features that add contract fields. (2) Consider using `additionalProperties: true` for forward-compatibility, with explicit validation of known fields. (3) Add schema versioning to CONTRACT.json so older validators can gracefully handle newer contracts.
- **Detection:** `compileContract()` returns `{ valid: false }` for contracts with new fields.

### Pitfall: Set Status Schema Does Not Include `discussing` or `executing` Transitions
- **Category:** correctness
- **Stack:** state-schemas.cjs, state-transitions.cjs
- **Description:** The `SetStatus` enum in `state-schemas.cjs` is `['pending', 'discussed', 'planned', 'executed', 'complete', 'merged']`. The `SET_TRANSITIONS` map allows `pending -> discussed` and `planned -> executed`. However, neither `discussing` nor `executing` appear as valid statuses in the Zod enum -- these intermediate states (referenced in the architecture docs) are not representable in the schema. If any code attempts to set `status: 'discussing'` or `status: 'executing'`, Zod will reject the mutation.
- **Trigger:** Attempting to set a set status to `discussing` or `executing` -- which the lifecycle diagram in ARCHITECTURE.md shows as valid states.
- **Impact:** Medium -- the lifecycle diagram is misleading. Actual transitions go directly from `pending` to `discussed` and from `planned` to `executed`. This is not a bug per se, but any new feature that tries to use `discussing` or `executing` as set statuses will fail at validation.
- **Prevention:** Document this design decision clearly. If the milestone adds new status transitions (e.g., for the DAG-driven workflow), ensure they use existing valid statuses only.
- **Detection:** Attempting `transitionSet(cwd, m, s, 'discussing')` throws a Zod validation error.

## Anti-Patterns for This Stack

### Anti-Pattern: Writing Files Then Validating (Instead of Validate-Then-Write)
- **What people do:** Write generated data to disk (STATE.json, DAG.json, REQUIREMENTS.md), then read it back and validate.
- **Why it fails:** If validation fails, the file already contains bad data. Other processes may read the file between write and validation. The init SKILL.md Step 9c writes STATE.json directly from roadmapper output without validation -- this is the root cause of the roadmapper state overwrite bug.
- **Correct approach:** Always validate in memory first, then write atomically. The `writeState()` and `withStateTransaction()` functions already implement this pattern for STATE.json -- extend it to DAG.json and other critical files.
- **Example:** `ProjectState.parse(roadmapperOutput.state)` should be called BEFORE `fs.writeFileSync()`.

### Anti-Pattern: Manual Argument Parsing Without Validation
- **What people do:** Parse CLI flags with `args.indexOf('--flag')` and `args[i+1]` without checking for missing values or typos.
- **Why it fails:** Unknown flags are silently ignored. Missing values cause index-out-of-bounds reads that return `undefined`. The `--desc` vs `--description` mismatch is a direct consequence of this pattern.
- **Correct approach:** Use a minimal argument parser that (a) validates known flags, (b) reports unknown flags as errors, and (c) checks that required values are present. Even a simple `parseArgs()` helper would catch the `--description` typo.
- **Example:** Current: `const desc = args[args.indexOf('--desc') + 1]` silently returns `undefined` if `--desc` is not present. Better: validate the flag exists before indexing.

### Anti-Pattern: Agent Output as Direct File Content
- **What people do:** Take an AI agent's JSON output and write it directly to disk as the canonical state file.
- **Why it fails:** Agent output may be malformed, incomplete, or use different field names than expected. JSON parsing succeeds but schema validation is skipped.
- **Correct approach:** Always pass agent output through the relevant schema validator before writing. Treat agent output as untrusted input.
- **Example:** The init SKILL.md Step 9c writes `state` from the roadmapper without `ProjectState.parse()`. This should be: parse, validate, then merge into existing state.

### Anti-Pattern: Silent Graceful Degradation in Data Pipelines
- **What people do:** Use `try { ... } catch { continue; }` when reading CONTRACT.json or other data files, logging nothing.
- **Why it fails:** `recalculateDAG()` silently skips malformed contracts (lines 112-114 of add-set.cjs). If a CONTRACT.json has a JSON syntax error, the set gets no edges in the DAG and no ownership -- but no error is reported. The user has no idea their contract was ignored.
- **Correct approach:** Log warnings to stderr for skipped files so the user can investigate. Use `process.stderr.write()` (already the project convention) to report skipped contracts.
- **Example:** `catch { continue; }` should be `catch (e) { process.stderr.write(\`[RAPID] Skipping malformed CONTRACT.json for ${set.id}: ${e.message}\n\`); continue; }`

## Version-Specific Gotchas

- **Node.js 18+**: `fs.cpSync()` is used in `scaffoldProject()` (line 258 of init.cjs). This function was added in Node.js 16.7.0 and marked stable in 20.x. In Node.js 18.x, it is experimental-stage and may produce warnings in some builds. Not a blocker, but test on Node.js 18 specifically.
- **Zod 3.25.76**: The `.passthrough()` method preserves unknown keys during `parse()` but NOT during `safeParse().data` in some edge cases with nested objects. Always use `parse()` for round-trip preservation.
- **Ajv 8.18.0**: The CJS import path is `require('ajv').default` (line 17 of contract.cjs). This is correct for Ajv 8.x but will break if Ajv ever drops the ESM-first packaging. Pin the major version.
- **proper-lockfile 4.1.2**: The `realpath: false` option (used in lock.cjs) is necessary because `.planning/.locks/` may be on a symlinked filesystem. Without this option, proper-lockfile resolves symlinks and can create locks at unexpected paths.
- **Git 2.30+**: The `git worktree add` command requires Git 2.5+, but `git worktree list --porcelain` (used for registry validation) requires Git 2.15+. The `--orphan` flag for `git worktree add` requires Git 2.36+.

## Security Pitfalls

1. **Shell injection via `execSync` template strings**: `worktree.cjs` line 21 uses `` execSync(`git ${args.join(' ')}`) `` which passes through a shell. Set names containing metacharacters could execute arbitrary commands. Use `execFileSync` instead.
2. **Lock target file contains PID**: The `.planning/.locks/*.target` files contain process PIDs. While not a direct vulnerability, PID information aids process enumeration in multi-tenant environments. Not a concern for local development but worth noting for hosted RAPID deployments.
3. **No input validation on set/group names**: Set names flow through file paths (`path.join(cwd, '.planning', 'sets', set.id)`). A set name containing `../` could escape the planning directory. Validate set names against a strict pattern before use in path construction.

## Performance Traps

1. **O(n^2) ownership overlap checking**: `createOwnershipMap()` in `contract.cjs` (lines 313-338) uses a double nested loop to check for directory overlaps. For N file ownership entries across all sets, this is O(N^2). With the scaffold overhaul increasing the number of owned files per set, this could become noticeable for large projects (50+ sets with 20+ files each = 1000+ entries).
2. **Synchronous file I/O in recalculateDAG**: `recalculateDAG()` reads every set's CONTRACT.json synchronously using `fs.readFileSync()`. For 15+ sets, this is 15+ synchronous disk reads. Not a bottleneck for SSDs but could be slow on network filesystems used in some CI setups.
3. **DAG recalculation on every addSetToMilestone**: Each `addSetToMilestone()` call recalculates the entire DAG. Adding 10 sets in sequence means 10 full DAG recalculations. Consider batching: add all sets first, then recalculate once.
4. **Full state file read on every transaction**: `withStateTransaction()` reads the entire STATE.json on every mutation. For projects with many milestones and sets, STATE.json can grow to 50-100KB. This is not a bottleneck yet but becomes one with frequent concurrent access.

## Summary Priority List

1. **STATE.json overwrite by roadmapper** -- Known bug, silently replaces valid state with potentially incomplete data. Fix before anything else in the init pipeline.
2. **REQUIREMENTS.md overwrite during scaffold** -- Known bug, silently destroys user-approved acceptance criteria. Fix by reordering or excluding from scaffold generators.
3. **`fileOwnership` not in CONTRACT_META_SCHEMA** -- Architectural contradiction that blocks the scaffold overhaul and ownership-based group splitting. Fix the schema first.
4. **DAG annotations stripped by `recalculateDAG()`** -- Blocks the "DAG-driven set grouping" feature. Any group assignment will be lost on recalculation.
5. **`--desc` vs `--description` flag mismatch** -- Known bug, causes project description to be silently dropped during init.
6. **Shell injection via `execSync` in worktree.cjs** -- Security issue that becomes higher risk when the scaffold overhaul introduces user-specified group names.
7. **DAG v1/v2 schema confusion** -- Must decide which version to use before adding group annotations. Currently `recalculateDAG()` uses v1 while v2 exists unused.
8. **Spec file integration bias risk** -- Spec files injected into all research agents will suppress independent research. Route carefully.
9. **Stub replacement merge conflicts** -- The scaffold overhaul's shared-branch stub strategy will generate predictable merge conflicts that need T1 auto-resolution rules.
10. **`passthrough()` schema permissiveness** -- Silent state corruption risk when adding new fields. Consider explicit schema fields for all milestone additions.
