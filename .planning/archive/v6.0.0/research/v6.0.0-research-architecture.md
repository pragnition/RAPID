# Architecture Research

## Current Architecture

RAPID follows a 5-layer stack (Skill -> Agent -> CLI -> Library -> Storage) with 21 core library modules, 26 generated agents, and 24 skill definitions. The architecture has proven effective across 19 milestones for single-developer workflows, but the v6.0.0 milestone reveals structural gaps in multi-developer coordination, DAG centrality, and init-time knowledge reuse.

### Strengths of the Existing Architecture

1. **Clean separation of concerns.** Skills orchestrate, agents execute, the CLI routes, and library modules hold business logic. No layer leaks into another.
2. **Worktree isolation model.** Physical file isolation via git worktrees prevents concurrent file access conflicts between sets.
3. **Contract-first planning.** CONTRACT.json defines boundaries before implementation, validated at three gates (post-plan, during execution, pre-merge).
4. **Transactional state management.** All state mutations go through `withStateTransaction()` with Zod validation and file locking.
5. **DAG infrastructure already exists.** `dag.cjs` provides toposort, wave assignment, v1/v2 DAG creation, and validation -- a solid foundation to build on.

### Weaknesses Exposed by This Milestone

1. **DAG is peripheral, not central.** DAG.json is generated after roadmapping and used during merge ordering, but it does not drive set grouping, developer assignment, or workflow ordering at init-time. The roadmapper produces sets without consulting the DAG for parallel-group optimization.
2. **Scaffold is disconnected from planning.** The scaffold command generates foundation files but has no concept of developer groups, cross-group stubs, or foundational set #0. It operates independently from the roadmapper.
3. **Init pipeline is stateless between steps.** Step 5 (scaffold) overwrites artifacts created in Step 4D (REQUIREMENTS.md). Step 9 (roadmapper output) overwrites valid STATE.json from Step 5 with an incomplete structure. These are symptoms of a pipeline that does not track which artifacts are authoritative at each stage.
4. **No spec file ingestion path.** Research agents start fresh every time. There is no mechanism to seed them with user-provided findings, design documents, or prior analysis.
5. **No meta-principles capture.** Architectural and style decisions from the discovery conversation are not persisted anywhere. Agents cannot reference them during execution.
6. **Namespace enforcement is advisory.** The core-identity.md tells agents to only use `rapid:*` skills, but there is no enforcement mechanism -- agents can still invoke non-RAPID subagents.

## Recommended Architecture Pattern

- **Pattern:** Extended modular monolith with DAG-driven coordination
- **Rationale:** RAPID is already a modular monolith (single process, well-bounded modules). The v6.0.0 changes do not require a fundamentally different pattern -- they require deepening the DAG's role from "post-hoc artifact" to "central coordination primitive" and extending the scaffold system to be group-aware. The monolith pattern is ideal because all coordination is local (no network boundaries), state is unified in .planning/, and the agent build pipeline already composes modules into agents at build time.
- **Evidence:** The existing 19-milestone track record demonstrates this pattern works. The proposed changes add new library modules and extend existing ones without altering the fundamental layering.

### Rationale for Structure Decisions

- **No new top-level directories.** All changes fit within the existing `src/lib/`, `src/commands/`, `src/modules/`, `.planning/`, and `skills/` structure.
- **New `group.cjs` module.** Developer group partitioning is a distinct concern (graph partitioning) that deserves its own module rather than being folded into `dag.cjs`. The dag module handles topology; the group module handles multi-developer assignment on top of that topology.
- **New `principles.cjs` module.** Principles capture and injection is a cross-cutting concern accessed by init, discuss, plan, and execute skills. A dedicated module keeps it clean.
- **PRINCIPLES.md in `.planning/` root.** Follows the convention of PROJECT.md, ROADMAP.md, and REQUIREMENTS.md as top-level planning artifacts.

## Module Organization

### Module: group.cjs (NEW)

- **Responsibility:** Partition sets into developer groups that minimize cross-group file conflicts, annotate DAG nodes with group assignments, and surface cross-group dependencies explicitly.
- **Public interface:**
  - `partitionIntoGroups(dag, contracts, numDevelopers)` -- Returns `{ groups: Array<{id, sets: string[], sharedDeps: string[]}>, crossGroupEdges: Array<{from, to, sharedFiles}> }`
  - `annotateDAGWithGroups(dag, groups)` -- Returns modified DAG with `group` field on each node
  - `generateGroupReport(groups, crossGroupEdges)` -- Returns markdown summary of group assignments
- **Dependencies:** `dag.cjs` (topology), `contract.cjs` (file ownership), `plan.cjs` (set definitions)
- **Data ownership:** Owns the group annotation on DAG nodes and the cross-group dependency report. Does not own DAG.json itself (that remains with dag.cjs).

### Module: principles.cjs (NEW)

- **Responsibility:** Capture architectural/style principles from user discovery, persist to PRINCIPLES.md, and generate CLAUDE.md summary sections.
- **Public interface:**
  - `generatePrinciplesMd(principlesData)` -- Returns markdown content for PRINCIPLES.md
  - `generateClaudeMdSection(principlesData)` -- Returns compact summary suitable for appending to CLAUDE.md
  - `loadPrinciples(cwd)` -- Reads and parses PRINCIPLES.md from .planning/
- **Dependencies:** `core.cjs` (path resolution)
- **Data ownership:** Owns PRINCIPLES.md content. Contributes a section to CLAUDE.md but does not own the full file.

### Module: dag.cjs (EXTENDED)

- **Responsibility:** Directed acyclic graph operations -- topological sort, wave assignment, DAG creation/validation, execution order extraction, and DAG persistence. Extended with group-annotation support.
- **Public interface (additions):**
  - `createDAGv3(nodes, edges, options)` -- v3 DAG with optional group annotations
  - `validateDAGv3(dag)` -- Validates v3 structure including group fields
  - Existing `toposort`, `assignWaves`, `createDAG`, `createDAGv2`, `validateDAG`, `validateDAGv2`, `getExecutionOrder`, `tryLoadDAG` remain unchanged.
- **Dependencies:** `core.cjs` (DAG_SUBPATH)
- **Data ownership:** Owns DAG.json structure and persistence.

### Module: stub.cjs (EXTENDED)

- **Responsibility:** Contract stub generation for cross-set and cross-group testing. Extended with framework-polymorphic stub generation and group-aware stub placement.
- **Public interface (additions):**
  - `generateGroupStubs(cwd, groupId, allGroups, contracts)` -- Generates stubs for all sets that a group depends on from other groups
  - `generateFoundationalStubs(cwd, scaffoldReport, groupAssignments)` -- Generates foundational stubs (basic UI, basic API) for cross-group testing
- **Dependencies:** `plan.cjs`, `worktree.cjs`, `contract.cjs`, `group.cjs` (new)
- **Data ownership:** Owns stub files in `.rapid-stubs/` directories.

### Module: init.cjs (EXTENDED)

- **Responsibility:** Project scaffolding, detection, and initial file generation. Extended with REQUIREMENTS.md preservation logic and state merge helper.
- **Public interface (additions):**
  - `scaffoldProject()` updated: skip overwriting REQUIREMENTS.md when it contains user-approved content
  - `mergeRoadmapperState(cwd, roadmapperOutput)` -- Merges roadmapper output into existing STATE.json instead of overwriting it
- **Dependencies:** `state-machine.cjs`, `version.cjs`
- **Data ownership:** Owns scaffold file generation. Shares STATE.json ownership with state-machine.cjs.

### Module: state-machine.cjs (EXTENDED)

- **Responsibility:** State CRUD, transactions, locking, Zod validation. Extended with partial state merge capability.
- **Public interface (additions):**
  - `mergeStatePartial(cwd, partial)` -- Merges a partial state update (e.g., milestones/sets from roadmapper) into an existing valid STATE.json, preserving fields the partial does not include (version, projectName, createdAt, etc.)
- **Dependencies:** `lock.cjs`, `state-schemas.cjs`, `state-transitions.cjs`, `dag.cjs`
- **Data ownership:** Owns STATE.json reads and writes.

### Module: core-identity.md (EXTENDED)

- **Responsibility:** Shared agent identity, namespace isolation, tool invocation patterns. Extended with stronger subagent enforcement.
- **Public interface (additions):**
  - New section: explicit deny-list pattern for non-RAPID subagent invocation with concrete enforcement language (e.g., "If you spawn a subagent, the agent name MUST start with `rapid-`. If you are about to call a skill, the skill name MUST start with `rapid:`. Any deviation is a critical error.")
- **Dependencies:** None (Markdown prompt module)
- **Data ownership:** Owns agent identity prompt content.

## Data Flow

### DAG-Driven Init Pipeline (New)

The redesigned init pipeline makes the DAG the central coordination artifact:

```
Discovery (Steps 1-4)
  |
  v
Scaffold (Step 5)  -----> Writes: base files, config.json
  |                        DOES NOT overwrite: REQUIREMENTS.md (if user-approved)
  v
Research (Steps 6-8) ----> Reads: spec files (if provided), CODEBASE-ANALYSIS.md
  |                        Writes: research/*.md, SUMMARY.md
  v
Roadmap (Step 9) --------> Reads: SUMMARY.md, REQUIREMENTS.md, project brief
  |                        Writes: ROADMAP.md, set definitions, CONTRACT.json
  |                        MERGES INTO (not overwrites): STATE.json
  v
DAG Generation ----------> Reads: STATE.json, CONTRACT.json files
  |                        Writes: DAG.json (with group annotations)
  v
Group Assignment --------> Reads: DAG.json, CONTRACT.json (file ownership)
  |                        Writes: DAG.json (updated with group fields)
  v
Scaffold Stubs (optional)-> Reads: DAG.json groups, CONTRACT.json exports
                            Writes: foundational stubs for cross-group testing
```

### State Preservation Flow (Bug Fix)

The key architectural change is replacing the "overwrite" pattern in Step 9 with a "merge" pattern:

```
Step 5: scaffoldProject() creates full valid STATE.json
  { version: 1, projectName, currentMilestone, milestones: [], createdAt, lastUpdatedAt }

Step 9: roadmapper returns partial state
  { milestones: [...], currentMilestone }

Step 9 (new): mergeRoadmapperState() merges partial into existing
  Preserves: version, projectName, createdAt, rapidVersion
  Updates: milestones, currentMilestone, lastUpdatedAt
  Result: complete valid STATE.json
```

### Spec File Flow (New)

### DAG-Driven Group Assignment Flow (New)

```
DAG.json (with edges)
  |
  v
partitionIntoGroups(dag, contracts, numDevelopers)
  |
  +--> Graph partitioning:
  |    1. Build file-ownership conflict graph between sets
  |    2. Weight edges by number of shared files
  |    3. Use greedy balanced partitioning to minimize cross-group shared-file weight
  |    4. Respect DAG ordering (dependencies stay in same group when possible)
  |
  v
{ groups: [{id: "group-1", sets: ["auth", "api"]}, ...], crossGroupEdges: [...] }
  |
  v
annotateDAGWithGroups() --> DAG.json nodes get `group` field
  |
  v
generateGroupReport() --> human-readable summary of group assignments
```

### Request Lifecycle (CLI)

Unchanged from current architecture:

```
User invokes /rapid:init (or other skill)
  -> Skill SKILL.md orchestrates
  -> Bash calls node "${RAPID_TOOLS}" <command>
  -> rapid-tools.cjs routes to src/commands/<command>.cjs
  -> Command handler calls src/lib/*.cjs modules
  -> Library modules read/write .planning/ with locking
  -> JSON output returned to skill
  -> Skill presents results to user
```

### State Management

- **Where state lives:** `.planning/STATE.json` (machine-readable), `.planning/ROADMAP.md` (human-readable), `.planning/sets/DAG.json` (dependency graph)
- **How state is read and written:** All reads go through `readState()` with Zod validation. All writes go through `withStateTransaction()` with lock -> validate -> mutate -> validate -> write -> unlock. The new `mergeStatePartial()` function follows the same transactional pattern but merges rather than replaces.
- **Consistency guarantees:** File locks prevent concurrent writes. Zod schemas enforce structural validity on every write. Atomic rename ensures no partial writes.

## Scaling Considerations

### Multi-Developer Coordination (New Focus)

The primary scaling concern for v6.0.0 is supporting teams of 2-6 developers working in parallel. The DAG-driven group assignment addresses this:

- **File conflict minimization.** The group partitioning algorithm minimizes cross-group shared-file weight. Sets that modify the same files are placed in the same developer group.
- **Cross-group dependency visibility.** Explicit cross-group edges in the DAG report tell developers which group's work their group depends on, enabling proactive coordination.
- **Foundational set #0.** A shared foundational set (implemented before groups branch) establishes shared interfaces, reducing the surface area of merge conflicts.
- **Stub-based testing.** Cross-group stubs allow each group to test their work against stable interfaces before the final merge, catching integration issues earlier.

### Expected Load Characteristics

- **State file size:** STATE.json grows proportionally with sets (tens of KB for 10-15 sets). No scaling concern.
- **DAG complexity:** Kahn's algorithm is O(V + E). Even with 50+ sets, DAG operations complete in milliseconds.
- **Group partitioning:** The greedy balanced partitioning is O(V^2) in the worst case, which is acceptable for the expected set count (5-20 sets).
- **Stub generation:** One stub file per cross-group dependency import. For a 4-group project with 3 cross-group dependencies, this is 3-12 stub files -- trivial.

### Bottleneck Areas

1. **Agent spawn latency.** The 6-parallel research agent spawn in init is the primary bottleneck. This is unchanged by v6.0.0 work.
2. **Roadmapper single-pass constraint.** The roadmapper generates all sets and contracts in a single agent invocation. For very large projects (20+ sets), this pushes context window limits. The group assignment running post-roadmap avoids adding to this constraint.
3. **Merge complexity.** Cross-group merges remain the hardest scaling challenge. The group-aware stub system and foundational set #0 reduce but do not eliminate merge conflicts.

## Architecture Risks

1. **Group partitioning quality depends on file-ownership accuracy.** If CONTRACT.json `ownedFiles` lists are incomplete or wrong, the partitioning algorithm will produce suboptimal groups. **Impact:** Merge conflicts that grouping was supposed to prevent. **Mitigation:** Validate file ownership completeness during contract generation; warn when ownedFiles lists are empty.

2. **State merge logic is more complex than state overwrite.** The `mergeStatePartial()` function must correctly handle all combinations of existing vs incoming fields. **Impact:** State corruption if merge logic has bugs. **Mitigation:** Comprehensive unit tests covering all merge scenarios; Zod validation catches structural issues post-merge.

3. **Foundational set #0 creates a serial bottleneck.** All developer groups depend on set #0 completing first. If set #0 is large, it delays the entire project. **Impact:** Reduced effective parallelism. **Mitigation:** Keep set #0 minimal (interfaces and stubs only, no feature implementation); make set #0 optional based on project structure.

4. **Spec file format is unconstrained.** Users can provide any Markdown file as a spec. Research agents must handle arbitrary formats gracefully. **Impact:** Low-quality research if specs are poorly structured. **Mitigation:** Document recommended spec structure; research agents treat specs as supplementary context, not authoritative source.

5. **PRINCIPLES.md drift over milestones.** Principles captured during init may become stale as the project evolves. **Impact:** Agents following outdated guidelines. **Mitigation:** new-version skill should prompt for principles review; principles.cjs should support incremental updates.

6. **DAG v3 backward compatibility.** Extending DAG.json with group annotations must not break existing DAG consumers (merge, status, execute-set). **Impact:** Runtime errors in existing commands. **Mitigation:** Group fields are optional in DAGv3 schema; all existing consumers use `getExecutionOrder()` which reads `waves` (unchanged); validate backward compatibility in dag.test.cjs.

## Recommendations

Ordered by implementation priority:

1. **Fix REQUIREMENTS.md overwrite (Bug #1):** Modify `scaffoldProject()` to skip REQUIREMENTS.md when it already contains user-approved content. Check for the encoded criteria regex pattern (`/^[A-Z]+-\d{3}:/`) as the signal that the file has been user-approved. -- **Priority: critical** (data loss bug, blocking correct init flow)

2. **Fix --desc vs --description flag mismatch (Bug #2):** Add `--description` as an alias for `--desc` in `src/commands/init.cjs` line 32. This is a 1-line fix: add `case '--description':` above the existing `case '--desc':`. -- **Priority: critical** (init fails for every user following skill instructions)

3. **Fix roadmapper state overwrite (Bug #3):** Implement `mergeStatePartial()` in state-machine.cjs or init.cjs. Step 9 in SKILL.md must use this merge function instead of Write to update STATE.json. The merge preserves `version`, `projectName`, `createdAt`, `rapidVersion`, `lastUpdatedAt` from the existing state and updates only `milestones` and `currentMilestone` from the roadmapper output. -- **Priority: critical** (DAG generation fails after init, blocking entire workflow)

4. **DAG.json as central concept:** Extend DAG node schema to support optional `group` field. Create DAGv3 creation and validation functions. Ensure backward compatibility with v1/v2 consumers. This is the foundation that all group-related features build on. -- **Priority: high**

5. **Developer group partitioning (group.cjs):** Implement the graph partitioning algorithm that assigns sets to developer groups. This is the core algorithmic work of the milestone. -- **Priority: high**

6. **DAG-driven roadmapper integration:** Modify the roadmapper role to consider developer group boundaries when designing set boundaries. The roadmapper should produce group annotations as part of its output. Cross-group dependencies should be explicitly surfaced in the roadmap. -- **Priority: high**

7. **Scaffold overhaul for cross-group stubs:** Extend the scaffold skill to generate framework-polymorphic stubs for cross-group testing. Add support for a foundational set #0 that provides shared interfaces. Stubs should live on a shared branch and be replaced by real implementations during execution. -- **Priority: high**

8. **Spec file integration in init:** Add spec file path argument to `/rapid:init` and `/rapid:new-version`. Research agents receive spec content as prior findings and build on them rather than starting fresh. -- **Priority: medium**

9. **Meta-principles capture:** Add a principles discovery step (4E) to the init flow. Create `principles.cjs` for generation and persistence. Inject a summary into CLAUDE.md so all agents inherit the project's architectural decisions. -- **Priority: medium**

10. **Agent subagent namespace enforcement:** Strengthen the namespace isolation section in `core-identity.md` with explicit deny-list language and concrete examples. Consider adding a lint check in `build-agents` that flags non-`rapid-` agent references in role modules. -- **Priority: low** (advisory enforcement is largely sufficient; violations are rare in practice)
