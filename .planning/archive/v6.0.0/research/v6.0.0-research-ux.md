# Domain & UX Research: v6.0.0 Scale & Quality

## Domain Conventions

RAPID operates in the **multi-agent coding orchestration** domain -- a category that solidified in 2025-2026 with tools like Composio's Agent Orchestrator, Superset.sh, Conductor, and Claude Code Workflow. The conventions below reflect both this emerging domain and the broader CLI developer tools space.

### Standard Terminology

| RAPID Term | Domain Standard | Notes |
|------------|----------------|-------|
| Set | Task / Workstream / Agent task | "Set" is non-standard but established in RAPID. Composio uses "task", Superset uses "agent session". RAPID's term is fine since it is well-defined internally. |
| Wave | Wave / Tier / Stage | "Wave" aligns well with DAG-level terminology. Airflow uses "priority_weight" and TaskGroups, but "wave" is more intuitive for developers. |
| DAG | DAG | Universal term across Airflow, Argo Workflows, Prefect, and build tools (Nx, Bazel). No change needed. |
| Worktree | Worktree / Sandbox / Isolation context | Git worktree is the correct term. Composio and Superset both use "worktree" explicitly. |
| Developer group | Team / Squad / Agent fleet | This is a new concept for RAPID. Domain tools use "agent fleet" (Composio), "parallel sessions" (Superset). For human developers, "group" or "team" are natural. |

**Source:** Composio Agent Orchestrator README, Superset.sh documentation, Airflow DAG concepts.

### Common Workflows and User Journeys

The standard user journey in multi-agent orchestration tools follows this pattern:

1. **Define scope** -- describe what needs building (spec file, issue tracker, freeform description)
2. **Decompose** -- tool splits work into parallelizable units (automatic or guided)
3. **Assign** -- units assigned to agents/developers (auto or manual)
4. **Execute** -- parallel isolated execution with progress monitoring
5. **Review** -- diff-first review of completed work
6. **Merge** -- orchestrated integration with conflict resolution

RAPID's lifecycle (`init > start-set > discuss-set > plan-set > execute-set > review > merge`) maps cleanly to this pattern. The discuss-set step is more thorough than competitors -- most tools skip the interactive vision capture step entirely (Composio auto-detects everything, Superset expects a prompt).

**Industry standard (Composio Agent Orchestrator):** "Spawn and walk away. The system handles isolation, feedback routing, and status tracking. You review PRs and make decisions -- the rest is automated."

**RAPID's differentiation:** RAPID is more opinionated about the planning phase, which is a strength for complex projects but adds friction for quick tasks. The `/rapid:quick` command addresses this gap.

### Competitive Landscape Patterns

| Tool | Initialization | Decomposition | Group Assignment | Progress |
|------|---------------|---------------|------------------|----------|
| Composio Agent Orchestrator | `ao setup` auto-detects language, SCM, branch | Orchestrator splits tasks | Agent-per-task, auto-assigned | Dashboard + tmux sessions |
| Superset.sh | Open workspace, dispatch agents | Manual per-agent prompts | One agent per worktree, manual | Built-in terminal + diff viewer |
| Conductor | Config file | Manual task list | Agent-per-task | Visual dashboard |
| RAPID | `/rapid:init` multi-step interview | Roadmapper agent + DAG | Manual per-set (new: group assignment) | `/rapid:status` dashboard |

**Source:** Composio Agent Orchestrator GitHub, Superset.sh website, Addy Osmani "The Code Agent Orchestra".

## Interaction Models

### Primary Interaction Paradigm

RAPID is a **CLI-embedded agent orchestrator** operating through Claude Code's slash command system. The interaction model is:

- **Slash commands** as entry points (`/rapid:init`, `/rapid:execute-set 1`)
- **AskUserQuestion** structured prompts for decision points (233 occurrences across 28 skills)
- **Subagent delegation** for autonomous work (agents work in background)
- **Numeric shorthand** for referencing sets (`/rapid:start-set 1` instead of `/rapid:start-set auth-system`)

This is consistent with the Claude Code plugin ecosystem where skills define user-facing commands and agents handle autonomous work.

### Input/Output Patterns Users Expect

**Input conventions (CLI domain standard, per clig.dev):**
- Long flags should be guessable: `--description` not `--desc` (current bug: init CLI accepts `--desc` but skill says `--description`)
- Both short and long forms should exist for frequent flags
- Positional arguments for the most common parameter (RAPID does this: `execute-set 1`)
- Flags should be order-independent (RAPID does this: `discuss-set --skip 1` and `discuss-set 1 --skip` both work)

**Output conventions:**
- Progress breadcrumbs showing pipeline position (RAPID implements this: `init [done] > start-set > discuss-set...`)
- Structured JSON to stdout for machine consumption, human text to stderr (RAPID follows this)
- Next-step suggestions after every command (RAPID implements this consistently)
- Visual hierarchy with banners and color coding (RAPID has branded banners per skill)

**Source:** clig.dev (Command Line Interface Guidelines), Heroku CLI Style Guide, Atlassian CLI design principles.

### Feedback and Progress Indication Conventions

The domain standard for long-running agent operations (which RAPID's subagent spawns are) follows these conventions:

- **Immediate acknowledgment:** Confirm the command was received before starting work (RAPID does this with banners)
- **Stage indicators:** Show which stage of a multi-stage process is active (RAPID's breadcrumb does this)
- **Completion notification:** Clear signal when work finishes with artifacts listed (RAPID:RETURN protocol handles this)

**Gap identified:** RAPID lacks real-time progress during subagent execution. Competing tools (Composio's dashboard, Superset's terminal view) show live agent output. RAPID's current model is "spawn and wait" -- the user sees nothing until the agent completes. The Mission Control web dashboard partially addresses this but requires separate setup.

### Error Communication Patterns

RAPID follows the domain standard well:
- `[RAPID ERROR]` prefix for errors on stderr
- Descriptive error messages with context (template literals with variable interpolation)
- Error breadcrumbs showing exactly where failure occurred
- Next-step recovery suggestions after every error

This aligns with the "12 Rules of Great CLI UX" which emphasizes that error messages should tell the user what went wrong AND what to do about it.

**Source:** clig.dev error guidelines, "The 12 Rules of Great CLI UX" (dev.to).

## Information Architecture

### How Information Should Be Organized

RAPID organizes information across three tiers:

1. **State tier** (machine-readable): STATE.json, DAG.json, REGISTRY.json, CONTRACT.json
2. **Planning tier** (human+machine): ROADMAP.md, DEFINITION.md, CONTEXT.md, PLAN.md
3. **Execution tier** (human-readable): REVIEW-SCOPE.md, GAPS.md, DEFERRED.md

This three-tier model is sound and mirrors how Airflow separates DAG definitions (code), metadata (database), and logs (human-readable).

### Navigation and Discovery Patterns

**Current pattern:** Users navigate via slash commands and numeric set references. The `/rapid:status` command serves as the primary dashboard.

**Domain convention for DAG visualization:** Tools like Airflow, Argo Workflows, and Nx all provide visual DAG representations. Users expect to see:
- Nodes (sets) with status coloring
- Edges (dependencies) with directional arrows
- Wave/level grouping as horizontal bands
- The ability to see which groups of work can run in parallel

**Relevance to DAG-driven set grouping feature:** When RAPID adds developer group annotations to the DAG, users will expect:
- Groups visually distinguished (by color, label, or section)
- Cross-group dependencies explicitly marked
- A clear mapping from "which developer works on what"

**Source:** Airflow Graph View, Argo Workflows DAG visualization, Nx project graph.

### Content Hierarchy and Progressive Disclosure

RAPID's progressive disclosure model works well:
1. `/rapid:status` -- high-level overview (set names + statuses)
2. Set-level details -- CONTRACT.json, DEFINITION.md, SET-OVERVIEW.md
3. Wave-level details -- per-wave PLAN.md files
4. Execution details -- commit history, WAVE-COMPLETE.md markers

**Recommendation for developer groups:** Follow the same progressive disclosure:
1. Status dashboard shows groups as headers (like wave headers today)
2. Each group lists its sets with statuses
3. Cross-group dependencies shown at group level, not set level

### Help and Documentation Conventions

RAPID has comprehensive documentation in `docs/` (9 guides) and contextual help via next-step suggestions. The domain convention is:
- `--help` flag on every command (RAPID skills are slash commands, so this is less applicable)
- Built-in examples in help text (RAPID's status command shows the "Tip: Use numeric shorthand" line)
- Error messages that link to relevant docs

## Accessibility Considerations

### Key Accessibility Patterns for CLI Interaction

- **Screen reader compatibility:** Markdown tables in terminal output are well-handled by modern screen readers. RAPID's table-based status dashboard is accessible.
- **Color is not the sole information carrier:** RAPID uses color for branding (banners, agent colors) but always includes text labels alongside color. Status values are text strings, not color-only indicators.
- **Keyboard navigation:** As a CLI tool, RAPID is inherently keyboard-navigable. AskUserQuestion prompts use numbered options which are accessible.

### Not Applicable

- **Touch targets, responsive layouts, WCAG contrast ratios:** Not applicable -- RAPID is a terminal-based tool, not a web UI. The Mission Control web dashboard would need these, but that is a separate concern outside this milestone's scope.

**Rationale:** CLI accessibility is primarily about text clarity, structured output, and not relying solely on color -- all of which RAPID already handles. No specific action items for this milestone.

## User Expectations

### Default Behaviors Users Expect

Based on domain research, users of multi-agent orchestration tools expect:

1. **Idempotent commands:** Running the same command twice should be safe. RAPID partially implements this (scaffold is additive-only, discuss-set offers re-discuss) but some commands could be more robust (e.g., start-set with existing branch).

2. **Spec file as input shortcut:** Tools like Composio auto-detect from issue trackers, Cursor uses `.cursorrules`, and Airflow reads DAG files. Users expect to be able to provide structured input rather than re-entering everything interactively. RAPID's planned `--spec` flag for `/rapid:init` aligns with this expectation.

3. **DAG drives everything:** In Airflow, the DAG is the central concept -- task ordering, scheduling, monitoring, and retry logic all derive from it. Users of DAG-based tools expect the DAG to be the single source of truth for execution order, not a side artifact. RAPID's planned "DAG as central concept" feature matches this expectation exactly.

4. **Group assignment is advisory, not mandatory:** In team-based tools (Nx, Airflow with team annotations), group/team assignment is a recommendation that developers can adapt. Users expect to be able to move sets between groups without breaking the system. RAPID's planned approach ("developers free to adapt") aligns with this.

5. **Flag consistency:** Users expect `--description` and `--desc` to both work, or for one to be the canonical form with the other as an alias. The current mismatch where the skill uses `--description` but CLI accepts `--desc` violates this expectation and should be resolved.

**Source:** Airflow DAG documentation, Composio Agent Orchestrator setup, clig.dev flag conventions.

### Terminology Users Use vs. Technical Terms

| What users say | What RAPID calls it | Alignment |
|---------------|--------------------|-----------| 
| "task" or "work item" | Set | Divergent -- but RAPID's term is well-documented |
| "dependency graph" | DAG | Aligned |
| "team" or "group" | Developer group (new) | Aligned -- "group" is intuitive |
| "scaffold" or "bootstrap" | Scaffold | Aligned |
| "spec" or "requirements" | Spec file (new) / REQUIREMENTS.md | Aligned |
| "principles" or "style guide" | Meta principles (new) / PRINCIPLES.md | "Principles" is natural; "meta principles" is more opaque |
| "stub" | Stub | Aligned -- standard term |

### Mental Models from Competing Products

Users coming from these tools will have specific expectations:

- **From Airflow/Argo:** DAG-centric thinking. Expect visual DAG representation, automatic execution ordering, retry on failure, and the DAG to be the single source of truth.
- **From Composio Agent Orchestrator:** "Spawn and walk away" model. Expect minimal configuration, auto-detection, and a dashboard for monitoring. May find RAPID's init interview verbose.
- **From Superset.sh:** Agent-per-worktree isolation. Expect to see agent output in real-time and review diffs inline. May expect a visual diff viewer.
- **From Nx/Turborepo:** Monorepo tooling users expect caching, incremental execution, and project graph visualization. RAPID's DAG is conceptually similar but operates at a higher level.

### Onboarding and Learning Curve Expectations

The init flow is the onboarding experience. Current expectations based on domain research:

- **Quick start in under 5 minutes** for simple projects (Composio's `ao setup` takes seconds)
- **Guided interview for complex projects** (RAPID's 4-batch discovery is appropriate here)
- **Spec file import to skip interview** for users who already know what they want (planned feature)
- **Clear progress indicators** showing how far along the init process is (RAPID's breadcrumbs do this)

**Source:** Composio Agent Orchestrator setup flow, garden.io developer onboarding guide.

## Recommendations

Ordered by priority for the Scale & Quality milestone:

### Critical

1. **Fix `--desc` vs `--description` flag mismatch**: The init skill (line 481) instructs agents to use `--description` but `src/commands/init.cjs` (line 32) only accepts `--desc`. Either add `--description` as an alias in the CLI parser or update the skill to use `--desc`. Domain convention (clig.dev) recommends supporting both short and long forms. -- Priority: critical

2. **Fix REQUIREMENTS.md overwrite in init scaffold**: Step 5 of the init skill calls `init scaffold` which may overwrite user-approved criteria from Step 4D. The skill already has logic to check for existing content (added in the "Need to change something" path at line 458-461), but the scaffold step at line 481 runs before this protection takes effect for a reinitialize. The scaffold step must respect existing REQUIREMENTS.md content. -- Priority: critical

3. **Fix roadmapper state overwrite**: Step 9 of init overwrites valid STATE.json with roadmapper output. The roadmapper should merge its output into existing state rather than replacing it wholesale. This is a state management boundary issue -- the roadmapper should only add/update milestone and set entries, never replace the entire state object. -- Priority: critical

### High

4. **Make DAG.json the central concept**: The DAG currently exists as a side artifact -- it is generated during roadmapping but not consistently used for execution ordering, status display, or merge sequencing. All consumers of set ordering (status, execute-set, merge, scaffold) should read from DAG.json rather than STATE.json insertion order. The DAG should be the authoritative source for "what depends on what" and "what can run in parallel." -- Priority: high

5. **DAG-driven developer group annotations**: Add a `group` field to DAG nodes (e.g., `{"id": "auth-system", "wave": 1, "group": "A", "status": "pending"}`). The roadmapper should auto-assign groups based on file ownership overlap minimization. Group labels should be simple (A, B, C or Team 1, Team 2, Team 3) and advisory. Display groups in the status dashboard as a column or grouping header. -- Priority: high

6. **Spec file integration for `/rapid:init`**: Allow `--spec path/to/spec.md` on init (not just new-version). Users who already have a requirements document should not need to re-enter everything interactively. The spec file should pre-populate the discovery conversation, with confirmation rather than re-entry. Follow the same pattern already implemented in new-version's Step 0.5 and Spec-Aware Goal Extraction flow. -- Priority: high

7. **Meta principles capture during init**: Add a step after discovery (Step 4D) to capture architectural and style preferences. Use the terminology "Principles" (not "meta principles") in user-facing contexts -- "meta principles" is technical jargon. Write a summary to CLAUDE.md (under a `## Principles` section) and a full version to `.planning/PRINCIPLES.md`. Agents should read both as context. Pattern: similar to how BRANDING.md works -- captured once, referenced everywhere. -- Priority: high

### Medium

8. **Scaffold command overhaul for developer groups**: When DAG includes group annotations, scaffold should be able to split stub generation per group. The foundational set #0 concept (shared work before groups branch) aligns with the scaffold "additive-only" convention -- set #0 produces the baseline that all groups build on. Stubs should live on a shared branch and be replaced by real implementations, following the contract-first pattern RAPID already uses. -- Priority: medium

9. **Enforce agent subagent restrictions**: Agents calling non-`rapid:` prefixed subagents is a namespace pollution issue. The convention should be enforced in the agent build pipeline -- add a validation check during `build-agents` that scans role modules for non-rapid: agent references. This is a developer-facing convention, not a user-facing one. -- Priority: medium

10. **Align set ordering with user mental model**: The feedback note in memory states that sets should display in ROADMAP.md order, not alphabetically. When DAG.json becomes the central concept, the display order should be: DAG wave order first, then ROADMAP.md insertion order within each wave. The numeric index users see in `/rapid:status` should match what they see in the roadmap. -- Priority: medium

### Low

11. **Consider real-time progress during subagent execution**: Competing tools (Composio dashboard, Superset terminal view) show live agent output. RAPID currently has no progress indication during subagent work. This is a UX gap but is complex to implement within the Claude Code plugin model. The Mission Control web dashboard partially addresses this. -- Priority: low

12. **Terminology documentation**: Create a glossary section in DOCS.md mapping RAPID terms to domain equivalents (set = workstream, wave = DAG level, etc.) for users coming from other tools. -- Priority: low
