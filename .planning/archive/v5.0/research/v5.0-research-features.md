## Feature Inventory

### Feature: Banner SVG

- **Description:** A 1280x320 SVG banner for the GitHub repository header, using the Everforest dark palette with serif title typography. Replaces the existing pixel-art wave banner for GitHub display purposes.
- **Complexity:** trivial
- **Dependencies:** independent

#### Decomposition

1. **Banner SVG creation** -- Hand-craft an SVG at 1280x320 with three centered text elements: `/rapid:init` (monospace, grey), `RAPID` (Georgia serif, bold, foreground), and the tagline (serif, italic, grey) on Everforest dark background (`#2d353b`).
2. **Visual verification** -- Open in browser to confirm text centering, font rendering, and palette correctness.

#### Implementation Strategies

**Option 1: Hand-crafted inline SVG (Recommended)**
- Approach: Write the SVG directly using `<text>` elements with explicit font-family, fill colors, and text-anchor centering. No external tooling required.
- Pros: Zero dependencies, deterministic output, trivially diffable in git, renders identically on GitHub.
- Cons: Font rendering depends on the viewer having Georgia installed (GitHub's SVG renderer does have it). No complex typography control.
- Libraries/services: None -- raw SVG.

**Option 2: Design tool export (Figma/Inkscape)**
- Approach: Design in a visual tool and export SVG.
- Pros: WYSIWYG editing, precise positioning.
- Cons: Exports bloated SVG with tool-specific metadata, harder to maintain in version control, introduces a tool dependency for future edits.

**Recommended:** Option 1. The banner is three text elements on a solid background -- a visual tool is overkill. The existing `branding/banner.svg` (236 lines of hand-crafted SVG rects) demonstrates that hand-crafted SVGs are the established pattern in this project.

#### Data Model Impact

- New entities/tables needed: None
- Existing entities modified: None
- Relationships: None

#### API Surface

- No API changes. Static file creation only.

### Feature: Social Preview PNG

- **Description:** A 1280x640 PNG raster image for GitHub's Open Graph social preview (og:image). Same G1 design as the banner, vertically expanded.
- **Complexity:** moderate
- **Dependencies:** Depends on Banner SVG (uses the same design, scaled to 640px height)

#### Decomposition

1. **Temporary tall SVG creation** -- Create a 1280x640 variant of the banner SVG with vertically centered text elements.
2. **SVG-to-PNG rasterization** -- Convert the SVG to a PNG at exact 1280x640 pixel dimensions.
3. **Cleanup** -- Remove the temporary SVG after rasterization. The PNG is the deliverable.
4. **Manual upload** -- Document that the PNG must be manually uploaded via GitHub repo Settings > General > Social preview.

#### Implementation Strategies

**Option 1: ImageMagick `convert` (Recommended)**
- Approach: Use `convert -size 1280x640 branding/social-preview.svg branding/social-preview.png` (ImageMagick 7.1.2 is installed on this system). For text rendering fidelity, may need `-font Georgia` and `-density 96` flags.
- Pros: Already installed on the system (`/usr/bin/convert`, ImageMagick 7.1.2-18), no new dependencies, single command.
- Cons: SVG text rendering quality depends on ImageMagick's SVG delegate (librsvg or Inkscape). Georgia font must be available on the system. Text positioning may differ slightly from browser rendering.
- Libraries/services: ImageMagick (system-installed).

**Option 2: Playwright browser screenshot**
- Approach: Load the SVG in a headless Chromium browser at 1280x640 viewport and take a screenshot. This guarantees pixel-perfect rendering matching what browsers display.
- Pros: Exact browser-quality rendering, handles all CSS/font features.
- Cons: Requires Playwright + Chromium download (~200MB), heavier than needed for a simple text SVG.
- Libraries/services: Playwright (npm).

**Option 3: Sharp library (Node.js)**
- Approach: Use the `sharp` npm package to convert SVG to PNG. Sharp uses librsvg internally.
- Pros: Fast, Node.js native, good SVG support.
- Cons: Requires adding a dependency. librsvg text rendering may not match browser output for system fonts.
- Libraries/services: sharp (npm).

**Recommended:** Option 1 (ImageMagick) for simplicity since it is already installed. If text rendering quality is inadequate, fall back to Option 2 (Playwright) for browser-accurate output. The social preview is a one-time generation, not a recurring build step, so heavier tooling is acceptable as a fallback.

#### Data Model Impact

- New entities/tables needed: None
- Existing entities modified: None

#### API Surface

- No API changes. Static file creation only.

2. **SVG construction** -- Hand-craft SVG with `<rect>` nodes (stroke-only, rounded corners), `<text>` labels (command in foreground, artifact in grey), `<line>` arrows with `<marker>` arrowheads.
3. **Visual verification** -- Confirm all 7 nodes render without overlap, arrows point left-to-right, and text is legible at the specified font sizes.

#### Implementation Strategies

**Option 1: Hand-crafted SVG with calculated coordinates (Recommended)**
- Approach: Pre-calculate all x/y positions for the 7 nodes and 6 arrows. Use SVG `<marker>` for arrowheads, `<rect>` with `rx` for rounded node boxes, `<text>` with `text-anchor="middle"` for centered labels. The spec already provides exact coordinates.
- Pros: Deterministic, maintainable, matches the existing hand-crafted SVG convention in `branding/`.
- Cons: Manual coordinate math (already done in the spec).
- Libraries/services: None.

**Recommended:** Option 1. The coordinates are already computed in the spec (`oss_brainstorm/plan.md`). This is a direct transcription task.

#### Data Model Impact

### Feature: GitHub Issue Templates

- **Description:** Three YAML files in `.github/ISSUE_TEMPLATE/`: `bug_report.yml` (structured form for bug reports), `feature_request.yml` (structured form for feature proposals), and `config.yml` (disables blank issues, points to templates).
- **Complexity:** trivial
- **Dependencies:** independent

#### Decomposition

1. **Directory creation** -- Create `.github/ISSUE_TEMPLATE/` directory (does not exist yet).
2. **bug_report.yml** -- YAML form with fields: description (textarea, required), repro steps (textarea, required), expected behavior (textarea, required), RAPID version (input, required), Claude Code version (input, required), OS (input), error logs (textarea, render: shell). Labels: `["bug"]`.
3. **feature_request.yml** -- YAML form with fields: description (textarea, required), use case (textarea, required), proposed solution (textarea, optional). Labels: `["enhancement"]`.
4. **config.yml** -- Set `blank_issues_enabled: false` with a `contact_links` array pointing contributors to documentation or discussions.

#### Implementation Strategies

**Option 1: GitHub YAML issue forms (Recommended)**
- Approach: Use GitHub's native YAML form schema (`.yml` extension). Top-level keys: `name`, `description`, `labels`, `body`. Body uses typed form elements: `textarea`, `input`, `dropdown`, `checkboxes`, `markdown`. This produces structured forms in the GitHub UI rather than freeform Markdown templates.
- Pros: Better UX for reporters (form fields with validation), structured data, GitHub-native. The YAML form syntax supports `validations.required`, `attributes.placeholder`, `attributes.render` for code blocks.
- Cons: Only works on GitHub.com (not self-hosted GitLab/Gitea).

**Option 2: Legacy Markdown templates (.md)**
- Approach: Use `ISSUE_TEMPLATE/bug_report.md` with YAML frontmatter and Markdown body.
- Pros: Works everywhere.
- Cons: No form validation, reporters often delete template sections, less structured.

**Recommended:** Option 1. The spec explicitly requires YAML forms. GitHub YAML issue forms are the modern standard and provide better signal-to-noise ratio on incoming issues.

**Key syntax notes for implementation:**
- Top-level keys: `name`, `description`, `title` (prefill), `labels`, `assignees`, `body`
- Body element types: `markdown`, `input`, `textarea`, `dropdown`, `checkboxes`
- Each element has `type`, `id`, `attributes` (label, description, placeholder, value, render), `validations` (required)
- `render: shell` on textarea produces syntax-highlighted code blocks in the submitted issue
- `blank_issues_enabled: false` in config.yml forces users through templates

#### Data Model Impact

- None

#### API Surface

- None (GitHub platform configuration only)

### Feature: Reference Migration

- **Description:** Change all occurrences of `fishjojo1/RAPID` to `pragnition/RAPID` across the active codebase. Historical archives may be left as-is.
- **Complexity:** moderate
- **Dependencies:** independent (but should run after README overhaul to avoid double-editing)

#### Decomposition

1. **Scope analysis** -- Identify which files contain `fishjojo1/RAPID`. There are 46 files total with 100 occurrences. Categorize into: active source (must change), planning/context (should change), and archive (leave as-is for historical accuracy).
2. **Active file migration** -- The critical files requiring updates:
   - `.claude-plugin/plugin.json` (2 occurrences: homepage, repository)
   - `.planning/PROJECT.md` (1 occurrence)
   - `.planning/context/CODEBASE.md` (1 occurrence)
   - `.planning/STATE.md` (1 occurrence)
   - `oss_brainstorm/plan.md` (1 occurrence)
3. **Planning/milestone files** -- Files under `.planning/milestones/` and `.planning/sets/` that reference the old org. These are semi-active planning artifacts.
4. **Archive exclusion** -- Files under `.planning/archive/` (35+ files) should be left untouched as historical records.
5. **Verification** -- After migration, grep to confirm no active-path references remain.

#### Implementation Strategies

**Option 1: Targeted sed replacement on active files only (Recommended)**
- Approach: Run `sed -i 's|fishjojo1/RAPID|pragnition/RAPID|g'` on the curated list of active files. Explicitly exclude `.planning/archive/` from the migration scope.
- Pros: Precise, auditable, preserves archive history.
- Cons: Requires manual file list curation.

**Option 2: Bulk replacement with archive exclusion**
- Approach: Run a recursive find-and-replace excluding the archive directory.
- Pros: Catches everything.
- Cons: May change planning artifacts that were historically accurate, harder to review.

**Recommended:** Option 1. The active file list is small (5-8 files). Targeted replacement is safer and produces a cleaner, reviewable diff. Archive files should be treated as immutable historical records.

**Scope classification:**

| Category | File Count | Action |
|----------|-----------|--------|
| Active source (`.claude-plugin/`, root `.md`) | 2-3 files | Migrate |
| Active planning (`.planning/PROJECT.md`, `.planning/context/`, `.planning/STATE.md`) | 3 files | Migrate |
| Semi-active planning (`.planning/milestones/`, `.planning/sets/`) | ~8 files | Migrate (these are referenced by active workflows) |
| Archive (`.planning/archive/`) | ~35 files | Leave as-is |
| Brainstorm (`oss_brainstorm/`) | 1 file | Migrate |

#### Data Model Impact

- `.claude-plugin/plugin.json` -- The `homepage` and `repository` fields change from `fishjojo1/RAPID` to `pragnition/RAPID`. This affects how the Claude Code plugin marketplace resolves the plugin's source.
- No database or state schema changes.

#### API Surface

- No API changes. Plugin identity metadata changes only.

## Implementation Order

1. **Banner SVG** -- Foundation visual asset. All other branding assets and the README header depend on the design language established here. Trivial complexity, quick win.

2. **Lifecycle Flow SVG** -- Independent diagram, no prerequisites. The SVG source is already drafted in `oss_brainstorm/plan.md`, making this a transcription task.

3. **Agent Dispatch SVG** -- Independent diagram, no prerequisites. Also pre-drafted in the plan. Can be done in parallel with Lifecycle Flow.

4. **Social Preview PNG** -- Requires the Banner SVG design to be finalized. Rasterization is a single command. Quick follow-up after Banner.

5. **GitHub Issue Templates** -- Independent of all branding work. YAML forms are self-contained files. Can be done in parallel with SVG work.

6. **GitHub PR Template** -- Same as issue templates. Independent, trivial.

7. **CONTRIBUTING.md** -- Benefits from issue templates existing (to link to them), but not strictly dependent. Short document.

8. **README.md Overhaul** -- The capstone feature. Must come after all SVG assets are created (referenced by `<img>` tags). Requires careful surgical editing to preserve existing content.

9. **Reference Migration** -- Last step. Run after README overhaul to avoid editing the same file twice. Mechanical find-and-replace on a curated file list.

**Parallelism opportunities:**
- Steps 1-3 (all SVG creation) can run in parallel.
- Steps 5-7 (community files) can run in parallel with SVG creation.
- Step 4 depends only on Step 1.
- Step 8 depends on Steps 1-3.
- Step 9 depends on Step 8.

This decomposition naturally suggests **3 waves**:
- **Wave 1:** Banner SVG + Lifecycle Flow SVG + Agent Dispatch SVG + Issue Templates + PR Template + CONTRIBUTING.md (all independent)
- **Wave 2:** Social Preview PNG + README.md Overhaul (depends on Wave 1 SVGs)
- **Wave 3:** Reference Migration (depends on Wave 2 README)

Alternatively, a tighter **2-wave** split is feasible since the README overhaul agent can verify SVG files exist before referencing them:
- **Wave 1:** All SVG creation + all community files
- **Wave 2:** README overhaul + Reference Migration (sequential within wave)

## Ambiguity Notes

1. **Version badge number.** The spec uses `4.4.0` in the badge URL, but the active milestone is v4.5. The badge should reflect whatever version is current at merge time. Implementation should use the version from `.claude-plugin/plugin.json`.

2. **Attribution line.** The spec shows `Built with love by fishjojo1` in the README header. Since references are migrating to `pragnition/RAPID`, the attribution link target (`github.com/fishjojo1`) should be confirmed -- is `fishjojo1` still the correct individual attribution, or should it change to the organization?

3. **Archive file migration scope.** The spec says "change fishjojo1/RAPID -> pragnition/RAPID across the codebase" without explicitly excluding archives. This research recommends excluding `.planning/archive/` for historical accuracy, but the decision should be confirmed.

4. **Existing banner.svg disposition.** The current `branding/banner.svg` (pixel-art wave) and `branding/logo.svg` are not mentioned in the spec. They should be preserved as-is (not deleted) since the spec only adds `banner-github.svg` as a new file. The 17 design iterations in `branding/designs/` should similarly be left untouched.

5. **Social preview rasterization quality.** The SVG uses Georgia (a system serif font). If Georgia is not installed on the rasterization host, the fallback font will produce different typography. The implementation should verify font availability before rasterization, or embed the font as SVG text paths.

6. **Doc links section.** The spec references individual doc files (`docs/setup.md`, `docs/planning.md`, etc.) that all exist in the current codebase. The arrow-prefixed link list should use these exact paths. The current README's "Further Reading" section (line 231-232) links to `DOCS.md` -- the new format adds granular per-topic links alongside the full reference link.
