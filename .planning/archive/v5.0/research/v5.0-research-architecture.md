# Architecture Research: OSS Presentation Milestone

## Current Architecture (Brownfield Analysis)

### What Exists

RAPID is a 5-layer Claude Code plugin (Skill > Agent > CLI > Library > Storage) with 21 core library modules, 26 generated agents, and 24 skills. The existing architecture is exclusively concerned with orchestrating parallel development workflows -- state machines, worktree isolation, contract validation, merge pipelines.

The OSS Presentation milestone introduces a fundamentally different concern: **static asset production and repository presentation**. This is not a feature extension of the existing architecture; it is a parallel concern that touches the repository surface area (README, branding, community files, GitHub config) without modifying any runtime behavior.

### Existing Architecture Strengths for This Milestone

1. **Clear directory conventions.** The project already has a `branding/` directory with 17 design iterations (banner + icon SVGs) and an `index.html` preview gallery. The pattern of storing visual assets in `branding/` is established.
2. **Separation of concerns.** Source code lives in `src/`, docs in `docs/`, agents in `agents/`, planning in `.planning/`. The new files (`CONTRIBUTING.md`, `.github/` templates) follow existing conventions of placing config at repo root and platform-specific files in dot-directories.
3. **README already has strong content.** The current README covers the problem statement, quickstart, architecture, command reference, and a real-world example. The milestone restructures presentation without rewriting substance.

### Areas Requiring Extension

1. **No `.github/` directory exists.** Issue templates, PR template, and config.yml are entirely new. This is a new directory boundary that did not exist before.
2. **Branding palette inconsistency.** The existing `branding/banner.svg` uses a custom ocean/blue palette (`#0a0e27`, `#0d2b45`, `#2878a8`, etc.). The milestone specifies Everforest dark palette (`#2d353b`, `#d3c6aa`, `#859289`, `#475258`). The new assets represent a complete visual rebrand.
3. **Reference fragmentation.** `fishjojo1/RAPID` appears in 52 files across the codebase. Most are in `.planning/archive/` (historical artifacts), but active files include `.claude-plugin/plugin.json`, `.planning/PROJECT.md`, `.planning/context/CODEBASE.md`, and `.planning/STATE.md`. The README already uses `pragnition/RAPID` for the install command, creating an inconsistency.
4. **No rasterization pipeline.** The social preview PNG requires converting SVG to raster format. The project has no existing image processing toolchain -- this is a one-off operation, not a recurring build step.

## Recommended Architecture Pattern

- **Pattern:** Static asset overlay with reference normalization
- **Rationale:** This milestone adds static files (SVGs, Markdown, YAML) alongside the existing codebase without modifying runtime architecture. The work decomposes into independent file-creation tasks with a single cross-cutting concern (reference migration). No new modules, no new runtime dependencies, no architectural changes to the 5-layer system.
- **Evidence:** The existing codebase already follows this pattern -- `branding/` coexists with `src/` without coupling. GitHub community files (`.github/`, `CONTRIBUTING.md`) are a standard convention that every major OSS project uses without architectural overhead.

## Module Organization

This milestone does not introduce code modules. Instead, the deliverables decompose into four independent concern areas:

- **Responsibility:** Produce Everforest-themed SVG/PNG files for GitHub repository presentation
- **Files:** `branding/banner-github.svg`, `branding/social-preview.png`, `branding/lifecycle-flow.svg`, `branding/agent-dispatch.svg`
- **Dependencies:** None (standalone static files)
- **Constraints:** Must use exact Everforest dark palette. SVGs must be self-contained (no external font/image references) for GitHub rendering. Social preview PNG must be exactly 1280x640.

### Concern: Community Files

- **Responsibility:** Restructure README with visual header stack, SVG diagram replacements, collapsible architecture section, and doc links
- **Files:** `README.md`
- **Dependencies:** Depends on branding assets being created first (references `branding/banner-github.svg`, `branding/lifecycle-flow.svg`, `branding/agent-dispatch.svg`)
- **Constraints:** Must preserve all existing prose and command tables. Only presentation layer changes -- no content rewrites.

### Concern: Reference Migration
- **Responsibility:** Change `fishjojo1/RAPID` to `pragnition/RAPID` across the codebase
- **Files:** `.claude-plugin/plugin.json`, `.planning/PROJECT.md`, `.planning/context/CODEBASE.md`, `.planning/STATE.md`, and potentially other active files
- **Dependencies:** None
- **Constraints:** Archive files (`.planning/archive/`) should NOT be modified -- they are historical records. Only active/current files should be updated. The `author.name` field in `plugin.json` may need to remain as-is or change to an org name.

## Scaling Considerations

Not applicable in the traditional sense. These are static files served by GitHub's CDN. However:

- **SVG file size.** Hand-crafted SVGs with text elements and simple shapes will be <20KB each. The existing pixel-art `banner.svg` is the largest at ~30KB due to dense rect elements. No optimization needed.
- **PNG size.** The social preview at 1280x640 with a simple text layout on a solid background will be <50KB. Well within GitHub's limits.
- **Font rendering.** SVGs using `font-family="Georgia, serif"` rely on the viewer's system fonts. GitHub's SVG renderer strips custom fonts. Georgia is available on macOS/Windows but not all Linux distributions. The fallback to generic `serif` is acceptable -- the visual difference is minimal for large text.
- **Future branding updates.** If the Everforest palette or typography changes, all four SVGs and the social preview need manual updates. There is no build pipeline to regenerate them. For a project of this size, manual updates are appropriate.

## Architecture Risks

1. **GitHub SVG sanitization breaks rendering** (Medium Impact) -- GitHub's SVG sanitizer strips certain elements (`<script>`, `<foreignObject>`, some CSS). Mitigation: use only basic SVG elements (`<rect>`, `<text>`, `<line>`, `<circle>`, `<g>`) with inline `fill`/`font-*` attributes. Avoid `<style>` blocks, CSS classes, and `<use>` references. Test by pushing to a branch and viewing on GitHub before merging.

2. **Font rendering inconsistency across platforms** (Low Impact) -- Georgia serif may render differently on Linux (where it is often missing) vs macOS/Windows. Mitigation: use `font-family="Georgia, 'Times New Roman', serif"` as a more robust fallback chain. Accept that minor typographic differences across platforms are inherent to web fonts in SVG.

3. **Reference migration misses active files** (Medium Impact) -- The grep for `fishjojo1` found 52 files total, but most are archives. Risk is missing an active file that still references the old org. Mitigation: grep for `fishjojo1` across the entire repo, categorize each hit as archive (skip) or active (update), and verify zero active-file hits after migration.

4. **README link breakage during restructuring** (Low Impact) -- The README overhaul moves sections, replaces diagrams, and adds new elements. Risk is breaking existing anchor links or image paths. Mitigation: verify all relative paths (`branding/`, `docs/`, `DOCS.md`, `LICENSE`) resolve correctly after editing.

5. **Archive file modification during reference migration** (Medium Impact) -- A bulk find-and-replace could accidentally modify `.planning/archive/` files that should remain as historical records. Mitigation: explicitly exclude archive directories from any automated replacement. The reference migration concern must be implemented as targeted file edits, not a global sed.

6. **Social preview PNG generation dependency** (Low Impact) -- Rasterizing SVG to PNG requires either Playwright (already available in the project's tool environment) or Inkscape. If neither is available, the PNG cannot be generated. Mitigation: Playwright is already installed as an MCP server tool. Alternatively, open the SVG in any browser and use DevTools screenshot at exact dimensions.

## Recommendations

1. **Create branding SVGs before README overhaul** (Priority: critical) -- The README references `branding/banner-github.svg`, `branding/lifecycle-flow.svg`, and `branding/agent-dispatch.svg`. These files must exist before the README edit, otherwise the README will contain broken image links. Implementation order: SVGs first, README last.

2. **Use inline SVG attributes, not CSS classes** (Priority: critical) -- GitHub's SVG sanitizer is aggressive. The existing `branding/banner.svg` uses `<style>` blocks with CSS classes, which may be stripped on GitHub. New SVGs should use `fill="..."`, `font-family="..."`, `font-size="..."` directly on elements. This guarantees rendering regardless of sanitizer behavior.

3. **Scope reference migration to active files only** (Priority: high) -- The 46+ files containing `fishjojo1` are overwhelmingly in `.planning/archive/` and `.planning/milestones/`. Only 4 active files need updating: `.claude-plugin/plugin.json`, `.planning/PROJECT.md`, `.planning/context/CODEBASE.md`, and `.planning/STATE.md`. Modifying archives would corrupt historical records.

4. **Keep community files concise** (Priority: high) -- The spec calls for "brief, no-nonsense" CONTRIBUTING.md. The project's existing voice (visible in README and docs) is direct and technical. Issue templates should use YAML forms with minimal required fields -- overly detailed templates discourage contributions.

5. **Treat SVGs as hand-crafted assets, not generated output** (Priority: medium) -- The lifecycle-flow and agent-dispatch diagrams encode project-specific information (command names, agent relationships). These will need manual updates when commands or agents change. This is acceptable for a project with 18 milestones of history -- diagram updates happen at most once per major version.

6. **Collapsible architecture section in README** (Priority: medium) -- The agent dispatch tree is detailed and tall (~40 lines). Wrapping it in `<details><summary>Architecture</summary>` keeps the README scannable while preserving the information for interested readers. The lifecycle diagram stays visible (it is compact at ~200px height).

7. **Commit atomically per concern area** (Priority: medium) -- Four natural commit boundaries: (a) branding SVGs + PNG, (b) community files, (c) README overhaul, (d) reference migration. This enables clean git bisection and independent review of each concern.

8. **Validate GitHub rendering on a branch before merging** (Priority: medium) -- Push the branch to GitHub and verify: SVGs render in README, badges load, issue templates appear in the "New Issue" dropdown, PR template appears when opening a PR. GitHub's local preview tools (like `grip`) do not fully replicate GitHub's SVG/YAML handling.

9. **Do not modify `oss_brainstorm/` directory** (Priority: low) -- The `oss_brainstorm/spec.md` and `oss_brainstorm/plan.md` files are input artifacts for this milestone. They should remain as-is for traceability, even though they reference `fishjojo1`. They are working documents, not published content.

10. **Consider version bump in plugin.json** (Priority: low) -- The current version is `4.4.0` in both `plugin.json` and `package.json`. If this milestone ships as v5.0, both files should be updated. However, version bumping may be handled by a separate process (new-version skill). Flag for decision during planning.
