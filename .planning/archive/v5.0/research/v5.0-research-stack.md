### SVG (Hand-Crafted)

This milestone introduces hand-crafted SVG files as branding assets. No SVG library or build tool is used -- SVGs are authored as raw XML.

- **Banner SVG** (`branding/banner-github.svg`): 1280x320, Everforest palette, text elements with serif typography
- **Lifecycle Flow SVG** (`branding/lifecycle-flow.svg`): ~1280x200, horizontal flow diagram
- **Agent Dispatch SVG** (`branding/agent-dispatch.svg`): ~1280x600, vertical tree diagram
- **Social Preview PNG** (`branding/social-preview.png`): 1280x640 rasterization of SVG source

**Key constraint:** GitHub renders SVGs in README via its Camo proxy with sanitization. The sanitizer strips `<script>` tags, external resources (`@import`, `<use>` with external hrefs), and certain CSS properties. The `font-family`, `text-anchor`, and basic SVG text attributes (`fill`, `font-size`, `font-weight`, `letter-spacing`) are preserved. Confirmed: SVG `<text>` elements with `font-family="Georgia, serif"` will render, but the actual font used depends on the viewer's browser/OS. GitHub does not guarantee Georgia availability -- it falls back to the browser's default serif.

### Markdown / GitHub Flavored Markdown

README.md, CONTRIBUTING.md, and PR template all use GitHub Flavored Markdown (GFM). Relevant GFM features for this milestone:

- `> [!TIP]` callout syntax (stable since GitHub 2024)
- `<details><summary>` collapsible sections (stable)
- `<p align="center">` for centered content (stable, used for banner/badges)
- `<picture><source>` for responsive images (stable since GitHub 2022)
- Shields.io badge embedding via `<img>` tags

### GitHub Issue Template Forms (YAML)

GitHub YAML issue forms (`.yml` extension in `.github/ISSUE_TEMPLATE/`) are the target format. Key syntax:

- Top-level keys: `name`, `description`, `title` (optional), `labels` (optional), `assignees` (optional), `body` (required)
- Body element types: `markdown`, `textarea`, `input`, `dropdown`, `checkboxes`
- Each element has `type`, `id`, `attributes` (with `label`, `description`, `placeholder`, `value`, `options`, `required`)
- `config.yml` at `.github/ISSUE_TEMPLATE/config.yml` controls blank issue behavior and external links
- PR templates go at `.github/PULL_REQUEST_TEMPLATE.md` (plain Markdown, not YAML)

**Compatibility note:** YAML issue forms require a GitHub.com public or GitHub Enterprise repository. They do not work with GitHub Free private repos. Since RAPID is a public open-source repo, this is not a concern.

### Zod Version Analysis

Zod 3.25.76 (installed) vs Zod 4.3.6 (latest). This is the only dependency with a significant version gap. Breaking changes in Zod v4:

1. **Error customization APIs**: Unified under a single `error` param (fragments removed)
2. **Default value behavior**: Defaults now apply even within optional fields
3. **Object methods**: `.strict()` and `.passthrough()` are legacy; use `z.strictObject()` and `z.looseObject()`
4. **Refinement handling**: `.pick()` / `.omit()` on refined objects now throws instead of silently dropping refinements
5. **String format validators**: `z.string().email()` becomes `z.email()` (top-level)
6. **Number validation**: `z.number().safe()` deprecated

**Impact on RAPID:** The project uses Zod extensively in `src/lib/state-schemas.cjs` for state validation. A Zod v4 upgrade would require reviewing all schema definitions. However, this is out of scope for the OSS Presentation milestone -- no source code changes are planned.

### proper-lockfile Status

Last release was v4.1.2 in 2022. The package has 5.8M weekly downloads and functions correctly. It uses `mkdir` atomic locking which is a well-understood, stable pattern. No CVEs reported. The lack of new releases reflects stability, not abandonment. No action needed.

## Tooling Assessment

### SVG Rasterization (for social-preview.png)

The social preview PNG (1280x640) requires rasterizing an SVG source. Three tools are available on this system:

| Tool | Version | Suitability | Notes |
|------|---------|-------------|-------|
| `rsvg-convert` | 2.61.4 (librsvg, Pango, Cairo) | Best option | Native, fast, reliable text rendering via Pango/fontconfig. Command: `rsvg-convert -w 1280 -h 640 input.svg -o output.png` |
| `magick` (ImageMagick 7) | Installed | Fallback | `convert` is deprecated in IMv7; use `magick` instead. SVG rendering quality depends on delegate library (uses librsvg internally on most Linux systems). |
| Playwright (via npx) | 1.58.2 | Overkill | Requires launching a full Chromium browser instance. Not installed as local dep. Works but unnecessarily heavy for static SVG rasterization. |

**Recommendation:** Use `rsvg-convert` as primary tool. It is already installed, renders SVG text correctly using system fonts via Pango/fontconfig, and produces high-quality PNGs.

**Font availability for rasterization:** Georgia (the typeface, not the script) is NOT installed on this system. `fc-match "Georgia"` resolves to "Noto Serif Regular". The SVG spec declares `font-family="Georgia, serif"` -- at rasterization time via rsvg-convert, this will render as Noto Serif. On GitHub (browser rendering), it will use whatever serif font the viewer's browser provides. The visual difference is minor since both are traditional serif faces. This is acceptable for the use case.

## Stack Risks

### 1. SVG Font Rendering Inconsistency Across Platforms

**Risk:** SVGs using `font-family="Georgia, serif"` will render differently on different platforms. GitHub's Camo proxy serves SVGs as images, so the viewer's browser renders the text. Windows/macOS users with Georgia installed see one look; Linux users and GitHub's preview see a fallback serif.

**Impact:** Low -- visual inconsistency is cosmetic only. The Everforest palette and layout remain consistent regardless of font. Serif fallback (e.g., Noto Serif, Times New Roman) preserves the design intent.

**Mitigation:** Accept the fallback behavior. The design uses simple centered text with generous spacing, so font metric differences are absorbed. Alternatively, convert text to paths in SVG for pixel-perfect rendering (trade-off: loses text selectability and increases file size).

### 2. GitHub SVG Sanitization Breaking Future Diagram Complexity

**Risk:** If future SVGs use advanced features (`<foreignObject>`, CSS animations, `<use>` with external refs, embedded JavaScript), GitHub's sanitizer will strip them.

**Impact:** Low for this milestone -- all planned SVGs use basic elements (`<rect>`, `<text>`, `<line>`, `<path>`, `<circle>`). Could affect future diagram additions.

**Mitigation:** Keep SVGs simple. Test each SVG in a GitHub README before finalizing. Avoid `<foreignObject>`, `<script>`, external `<use>` references, and `@import` in `<style>`.

### 3. Reference Migration Scope Underestimation

**Risk:** The `fishjojo1/RAPID` to `pragnition/RAPID` migration touches active files beyond just README and DOCS. Active non-archive files still containing `fishjojo1`: `.claude-plugin/plugin.json`, `.planning/context/CODEBASE.md`, `.planning/PROJECT.md`, `.planning/STATE.md`, and `.planning/post-merge/hygiene-sweep/REVIEW-SCOPE.md`.

**Impact:** Medium -- the `.claude-plugin/plugin.json` is the plugin manifest and controls how Claude Code discovers the plugin. The `homepage` and `repository` fields reference `fishjojo1/RAPID`. Incorrect references could cause confusion for users installing via `claude plugin add`.

**Mitigation:** Enumerate all non-archive files with `fishjojo1` references during planning. Prioritize `.claude-plugin/plugin.json` and `.planning/PROJECT.md` as mandatory. Archive files (`.planning/archive/`, `.planning/milestones/`) should be left as-is -- they are historical records.

### 4. Existing Banner Assets Conflict

**Risk:** The `branding/` directory already contains 17 design variants (`branding/designs/1-wave/` through `17-minimal/`), a `banner.svg` (pixel-art ocean wave style), a `logo.svg`, and an `index.html` preview page. The milestone creates `banner-github.svg` alongside the existing `banner.svg`. The coexistence is fine but the existing `index.html` references the old designs and would need updating to include new assets.

**Impact:** Low -- no breakage, but the `branding/index.html` becomes stale if not updated.

**Mitigation:** Either update `branding/index.html` to include the new Everforest assets, or defer since the branding server (from v4.5 milestone) may handle this.


## Recommendations

### 1. Use `rsvg-convert` for PNG Rasterization

**Rationale:** Already installed (v2.61.4), uses Pango/fontconfig for reliable text rendering, fast, no browser dependency. Command: `rsvg-convert -w 1280 -h 640 source.svg -o branding/social-preview.png`.

**Priority:** High -- blocking for the social-preview.png deliverable.

### 2. Keep SVG Text as Live Text (Do Not Convert to Paths)

**Rationale:** Converting text to `<path>` elements produces pixel-perfect rendering but sacrifices text selectability, increases file size significantly, and makes future text updates harder. The banner text ("RAPID", tagline, command name) is simple enough that serif fallback differences are negligible.

**Priority:** Medium -- design decision for the branding set.

### 3. Scope Reference Migration to Active Files Only

**Rationale:** 52 files contain `fishjojo1` references, but 47 are in `.planning/archive/`, `.planning/milestones/`, or `oss_brainstorm/`. Only 5 active files need updating: `.claude-plugin/plugin.json`, `.planning/context/CODEBASE.md`, `.planning/PROJECT.md`, `.planning/STATE.md`, `.planning/post-merge/hygiene-sweep/REVIEW-SCOPE.md`. Archive files are historical records and should remain untouched.

**Priority:** High -- the plugin manifest (`.claude-plugin/plugin.json`) is user-facing.

### 4. Update `.claude-plugin/plugin.json` Author Field

**Rationale:** The `author.name` field is `"fishjojo1"` and `homepage`/`repository` point to `fishjojo1/RAPID`. These should become `"pragnition"` and point to `pragnition/RAPID` respectively.

**Priority:** High -- affects plugin marketplace listing.

### 5. Declare the Everforest G1 Palette as Canonical for All Branding

**Rationale:** The existing `branding/banner.svg` uses a different palette (`#0a0e27` background, blue-ocean pixel-art style). The web frontend uses the full Everforest dark theme (`#2D353B` background, `#D3C6AA` foreground). The OSS spec defines a minimal "G1" subset. Standardizing on the G1 subset for all branding assets ensures consistency. Values confirmed from the web frontend's `everforest-dark.css`:

| Role | Hex | CSS Variable |
|------|-----|-------------|
| Background | `#2D353B` | `--th-bg-0` |
| Foreground | `#D3C6AA` | `--th-fg` |
| Grey/Muted | `#859289` | `--th-muted` |
| Border | `#475258` | `--th-bg-3` / `--th-border` |
| Accent (green) | `#A7C080` | `--th-accent` |
| Link (aqua) | `#83C092` | `--th-link` |

**Priority:** Medium -- design consistency item.

### 6. Test SVG Rendering in a GitHub README Before Finalizing

**Rationale:** GitHub's SVG sanitizer behavior is documented but edge cases exist (e.g., `dominant-baseline` attribute stripping reported in github/markup#1160). Testing in a scratch repo or branch before merging prevents surprises.

**Priority:** Medium -- quality assurance step.

### 7. Defer Zod v4 Upgrade to a Separate Milestone

**Rationale:** Zod 3.25.76 is the terminal v3 release and is fully functional. Zod v4 has significant breaking changes that would require auditing all state schemas. This is out of scope for a presentation/branding milestone and should be a dedicated set in a future version.

**Priority:** Low -- informational only, not actionable in this milestone.

### 8. Use ImageMagick v7 `magick` Command (Not `convert`)

**Rationale:** If ImageMagick is used as a fallback for rasterization, note that `convert` is deprecated in IMv7. The correct command is `magick input.svg output.png` or `magick convert input.svg output.png`. The system has IMv7 installed.

**Priority:** Low -- only relevant if rsvg-convert is unavailable.
