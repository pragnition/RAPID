# Oversights Research

## Cross-Cutting Concerns

### Reference Migration Completeness

The milestone calls for changing `fishjojo1/RAPID` to `pragnition/RAPID` "across the codebase." This is the single highest-risk oversight in the entire milestone because partial migration creates a broken project for new users.

**What exists today:**
- 46 files reference `fishjojo1/RAPID` (searched via grep)
- 18 files already reference `pragnition/RAPID`
- Key active files still pointing to fishjojo1: `.claude-plugin/plugin.json` (homepage, repository, author), `.planning/PROJECT.md`, `.planning/context/CODEBASE.md`, `.planning/STATE.md`
- Git remote `origin` still points to `https://github.com/fishjojo1/RAPID`

**What gets forgotten:**
1. **plugin.json author field** -- the `"author": { "name": "fishjojo1" }` field is distinct from the repo URL. The spec's README keeps `fishjojo1` as personal attribution ("Built with love by fishjojo1") while changing the org. This dual identity (personal author vs org repo) needs clear rules: which references become `pragnition` and which stay `fishjojo1`?
2. **Archive files** -- Roughly 30 of the 46 fishjojo1 references are in `.planning/archive/` and `.planning/milestones/`. These are historical records. Changing them rewrites history; leaving them creates grep noise. The decision needs to be explicit.
3. **Git remote itself** -- `git remote set-url origin` is not mentioned in any set overview. If the deploy target is `pragnition/RAPID`, the origin remote needs updating on the developer's machine.
4. **package.json** -- Currently missing `repository` and `homepage` fields entirely. These should be added pointing to `pragnition/RAPID`.
5. **DOCS.md version string** -- Hardcoded `4.4.0` in DOCS.md and multiple other places. The milestone name is "OSS Presentation" with no explicit version bump, but the version badge in the spec says `4.4.0`. If this ships as a new version, all version strings need updating.

### Version Number Synchronization

Version `4.4.0` appears in at least 4 locations that must stay synchronized:
- `package.json` ("version": "4.4.0")
- `.claude-plugin/plugin.json` ("version": "4.4.0")
- `DOCS.md` ("Version: 4.4.0")
- `README.md` (version badge URL: `version-4.4.0-d3c6aa`)

The spec hardcodes `4.4.0` in the badge. If the current milestone (v4.5) ships before or concurrently with the OSS presentation work, the badge will be wrong immediately. There is no automated version bump mechanism -- every version string is manually maintained. This is a coordination gap that creates stale version badges on the very first release after this work lands.

### SVG Cross-Platform Rendering

The banner SVG spec calls for `font-family="Georgia, serif"`. SVG text rendering depends on the viewer having the specified font installed. This matters in three distinct contexts:

1. **GitHub README rendering** -- GitHub sanitizes SVGs served from repositories. It strips `<script>`, wraps in a sandboxed `<img>` tag, and may alter font rendering. Georgia is not guaranteed to be available in GitHub's SVG renderer.
2. **npm README rendering** -- npm also renders SVGs differently from browsers.
3. **Social preview PNG** -- The plan calls for Playwright-based rasterization, which uses the system's font stack. If Georgia is not installed on the build machine, the fallback serif font will look different.

The existing `branding/banner.svg` sidesteps this entirely by using pixel-art rectangles (no text elements relying on fonts). The new banner introduces font-dependent rendering -- a regression in cross-platform consistency.

**Mitigations commonly forgotten:**
- Convert text to `<path>` elements (guarantees identical rendering everywhere, but loses editability)
- Embed the font as base64 within the SVG (increases file size, but renders consistently)
- Use a web-safe fallback stack and accept minor differences

### Social Preview PNG Generation Reproducibility

The spec says to create `branding/social-preview.png` via Playwright screenshot. This creates a build-time dependency that is easy to forget:

1. **Playwright is not in the project's dependencies** -- RAPID has zero dev dependencies. Adding Playwright just for one PNG is a heavy addition (Playwright downloads Chromium, ~100MB+).
2. **The PNG is a one-time artifact** -- Once generated, it gets committed and manually uploaded to GitHub settings. But if someone needs to regenerate it (palette change, version bump), the steps are not documented anywhere permanent.
3. **The temporary SVG (`branding/social-preview.svg`)** -- The plan creates this for rasterization then says nothing about whether to keep or delete it. Leftover temp files in committed directories create confusion.

### GitHub Issue Template YAML Validation

The spec calls for `.github/ISSUE_TEMPLATE/bug_report.yml` and `feature_request.yml` using GitHub's YAML form syntax. This syntax is strict and easy to get subtly wrong:

1. **`type` field constraints** -- GitHub issue forms support `input`, `textarea`, `dropdown`, `checkboxes`, and `markdown`. Using anything else silently breaks the form.
2. **Required field `name`** -- Every template YAML must have a top-level `name` key. Missing it makes the template invisible in the issue picker.
3. **`config.yml` blank_issues_enabled** -- If set to `false`, users cannot create issues without a template. This is aggressive for a new OSS project where contributors might have questions that do not fit templates. Consider keeping it `true` or adding a "General" template.
4. **Testing** -- YAML forms only work on the default branch. You cannot test them from a feature branch. This means the first real validation happens after merge to main.

### README Section Ordering and Anchor Links

The spec restructures the README significantly. Two concerns:

1. **DOCS.md cross-references** -- `DOCS.md` links back to `README.md` sections. If section headings change (e.g., "Further Reading" becomes "Docs"), those anchors break silently. GitHub does not report broken anchor links.
2. **External links to the README** -- If anyone has linked to `#architecture` or `#how-it-works` from blog posts, docs, or other repos, heading changes break those links. Since this is a pre-OSS-release polish, the risk is low but worth noting.

### Branding Directory Organization

The `branding/` directory currently contains:
- `banner.svg` (existing pixel-art wave)
- `logo.svg` (existing)
- `index.html` (preview page from branding-server set)
- `designs/` (17 design variants from exploration)

The milestone adds:
- `banner-github.svg` (new Everforest banner)
- `social-preview.png` (rasterized version)
- `lifecycle-flow.svg` (diagram)
- `agent-dispatch.svg` (diagram)

**What gets forgotten:**
1. **Old banner.svg** -- The new `banner-github.svg` replaces the pixel-art `banner.svg` for GitHub use, but `banner.svg` is not deleted. Having both `banner.svg` and `banner-github.svg` in the same directory without clear naming conventions creates confusion about which is canonical.
2. **The `designs/` directory** -- 17 design exploration variants (34 SVG files) ship to consumers on `claude plugin add`. These are design artifacts, not runtime assets. They inflate the install size unnecessarily. A `.npmignore` or equivalent exclusion pattern is missing.
3. **The `index.html`** -- This preview file was created by the branding-server set. After OSS release, it will be visible to the public. Is it useful documentation or internal tooling noise?

### .gitattributes for Binary Files

The project has no `.gitattributes` file. The milestone adds `social-preview.png`, which is a binary file. Without `.gitattributes`:
- Git may attempt text-mode line ending conversion on PNG files
- `git diff` will show binary gibberish for PNG changes
- Git LFS is not configured (likely unnecessary for a single PNG, but worth noting)

A minimal `.gitattributes` with `*.png binary` and `*.svg linguist-language=SVG` prevents these issues and also tells GitHub's language statistics to categorize SVGs correctly (rather than inflating the "SVG" language percentage).

### CONTRIBUTING.md Tone and Scope for Claude Code Plugin

The spec says "brief, no-nonsense" CONTRIBUTING.md with "dev install, bug reports, feature proposals, code style." For a Claude Code plugin specifically, contributors face unique onboarding friction:

1. **Plugin development requires Claude Code** -- Unlike a normal npm package, you cannot `npm test` and see results. Contributors need Claude Code installed to test their changes in context. This is a barrier worth addressing explicitly.
2. **Agent build step** -- After changing role modules in `src/modules/roles/`, contributors must run `node src/bin/rapid-tools.cjs build-agents` to regenerate the `agents/` directory. This is non-obvious and easy to forget.
3. **Testing without worktrees** -- Solo mode exists but is not documented for contributors. Running the full lifecycle requires a project to test against.
4. **The `.planning/` directory** -- Contributors will see this massive directory and wonder what it is. A brief note explaining it is project state (not source code) saves confusion.

### README Badge Staleness

The spec hardcodes the version badge: `https://img.shields.io/badge/version-4.4.0-d3c6aa?style=flat-square`. This is a static badge that must be manually updated with every release. The spec itself notes this: "update it manually when bumping versions, or switch to a GitHub release tag badge once releases are published."

However, there is no GitHub Releases workflow yet. The project has shipped 20+ milestones without ever creating a GitHub Release. If the badge stays hardcoded, it will be wrong within days of the next milestone. If switching to a release-based badge, that requires creating the first GitHub Release -- which is not in any set's scope.

### Documentation Link Consistency

The README restructure changes doc links to arrow-prefixed format:
```
-> **[Setup](docs/setup.md)**: installation and configuration
```

But `docs/setup.md` currently links to `pragnition/RAPID` for the install command, while other doc files might still reference `fishjojo1/RAPID`. The reference migration needs to cover all files in `docs/`, not just the top-level README and planning files.

Checking docs/: `docs/setup.md` already uses `pragnition/RAPID` for installation, but `DOCS.md` may have stale references.

### Accessibility of SVG Diagrams

The lifecycle-flow and agent-dispatch SVGs replace existing Mermaid/ASCII diagrams. The current ASCII text-based representations are inherently accessible (screen readers can read them). Replacing them with SVGs without `<title>`, `<desc>`, and `aria-label` attributes makes the architecture section inaccessible to screen reader users.

For an OSS project presentation, accessibility signals quality. At minimum:
- Each SVG should include `<title>` and `<desc>` elements
- The `<img>` tags in the README should include meaningful `alt` text (not just "lifecycle diagram")
- Consider keeping the ASCII dispatch tree inside a `<details>` block as an accessible alternative

### GitHub-Specific Markdown Rendering

The README overhaul uses several GitHub-flavored Markdown features:
1. `> [!TIP]` callout syntax -- This renders as a styled callout on GitHub but appears as a plain blockquote on npm, PyPI, or any non-GitHub Markdown renderer.
2. `<details><summary>` -- Works on GitHub but not all Markdown renderers.
3. `<picture><source srcset>` -- Used for the banner. GitHub supports this, but npm's README renderer does not.

Since RAPID is a Claude Code plugin installed via `claude plugin add` (not npm), the npm rendering concern is minimal. But if the project ever publishes to npm or other registries, the README will look broken.

## Recommendations

Ordered by cost of adding later (most expensive to retrofit first):

1. **SVG font rendering strategy**: Decide text-to-path vs embedded font vs accept-inconsistency BEFORE creating the banner. Redoing the SVG after it is designed and approved is pure waste. The spec's approach (Georgia serif) will render differently on every platform. Convert title text to `<path>` elements for the banner (editability is not needed for a banner), and use system fonts for diagram labels where minor variation is acceptable.

2. **Reference migration scope document**: Before executing, produce a definitive list of files to change and files to leave unchanged. Explicitly mark archive files as "do not touch" or "update too." The decision affects 30+ files and must be consistent. The `author` field in plugin.json is a distinct decision from the `repository` URL.

3. **.gitattributes file**: Add before the first PNG commit. Two lines (`*.png binary`, `*.svg text`) prevent git from corrupting the social preview. Trivial to add now, confusing to debug later.

4. **CONTRIBUTING.md with plugin-specific onboarding**: Address the build-agents step, the need for Claude Code to test, and the `.planning/` directory explanation. Generic OSS contribution guides do not cover these.

5. **SVG accessibility attributes**: Add `<title>`, `<desc>`, and meaningful `alt` text to every SVG and `<img>` tag during creation. Retrofitting accessibility across committed SVGs and README markup is tedious.

6. **Version badge strategy decision**: Either create the first GitHub Release (enabling dynamic badges) or document the manual update process in CONTRIBUTING.md. The worst outcome is a badge that says 4.4.0 for six months.

7. **Issue template testing on a fork**: Create a temporary fork, push templates there, and verify the issue picker renders correctly before merging to main. YAML form syntax errors are invisible until someone tries to file an issue.

8. **Branding directory hygiene**: Decide whether `designs/` ships with the plugin or is excluded. If shipping, add a README to explain what it is. If excluding, add it to a packaging exclusion list (the project does not currently have `.npmignore` or `files` in package.json, but the Claude Code plugin marketplace may have its own packaging rules).

9. **Document social preview regeneration**: One section in CONTRIBUTING.md explaining how to regenerate the PNG. Without this, the knowledge of how to update the social preview dies with the original implementor.

10. **Anchor link audit**: After the README restructure, grep `DOCS.md` and `docs/*.md` for any `README.md#` anchor references and update them. This takes 5 minutes but is always forgotten.
