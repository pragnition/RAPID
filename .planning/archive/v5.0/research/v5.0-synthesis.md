# Research Summary

## Project Profile
- **Type:** brownfield Claude Code plugin (developer tool)
- **Stack:** Node.js (CommonJS .cjs) + Zod + Git worktrees -- this milestone is static assets only (SVG, Markdown, YAML, PNG)
- **Scale:** medium (9 features, 34 sub-features, all static file creation/modification)
- **Key Constraint:** GitHub's SVG sanitizer and camo proxy strip font declarations and advanced CSS, making font-dependent SVG text rendering unreliable for the banner and diagrams (sources: STACK, PITFALLS, OVERSIGHTS)

## Unified Findings

### Stack and Infrastructure

- **Node.js 25.8.0 detected, 18+ required.** No runtime code changes in this milestone; version is irrelevant. (source: STACK)
- **Git 2.53.0 detected, 2.30+ required.** Only standard commit/push operations needed. (source: STACK)
- **Zod 3.25.76 installed; Zod 4.3.6 is latest.** Breaking changes in v4 (error APIs, defaults, object methods, string validators). Upgrade is explicitly out of scope for this milestone. (sources: STACK, ARCHITECTURE)
- **proper-lockfile 4.1.2** is stable, no action needed. (source: STACK)
- **SVG rasterization tooling:** `rsvg-convert` 2.61.4 (best option, uses Pango/fontconfig), ImageMagick 7 `magick` (fallback), Playwright via npx (overkill). Georgia font is NOT installed on the system; `fc-match "Georgia"` resolves to Noto Serif. (source: STACK)
- **GitHub YAML issue forms** require public repo (satisfied) and have strict validation rules. YAML boolean parsing of bare `yes`/`no`/`true`/`false` silently breaks templates. (sources: STACK, PITFALLS, OVERSIGHTS)
- **GitHub Markdown features** used: `> [!TIP]` callout, `<details><summary>`, `<picture><source>`, shields.io badges -- all stable and supported. (source: STACK)
- **No `.github/` directory exists yet.** Issue templates, PR template, and config.yml are entirely new. (source: ARCHITECTURE)
- **No `.gitattributes` file exists.** PNG binary files risk git text-mode corruption without `*.png binary` declaration. (source: OVERSIGHTS)
- **package.json** is missing `repository` and `homepage` fields; these should be added pointing to `pragnition/RAPID`. (source: OVERSIGHTS)
- **Version 4.4.0** is hardcoded in 4 locations (package.json, plugin.json, DOCS.md, README badge). No automated version bump mechanism exists. (sources: STACK, OVERSIGHTS)

### Architecture Direction

- **Pattern:** Static asset overlay with reference normalization. No runtime architecture changes. No new code modules, dependencies, or build steps. (source: ARCHITECTURE)
- **Directory structure:** New SVGs go in existing `branding/`. New community files go in `.github/` (new) and repo root. README is surgically edited, not rewritten. (source: ARCHITECTURE)
- **Existing branding assets preserved:** `banner.svg` (pixel-art wave), `logo.svg`, `designs/` (17 variants), `index.html` all remain untouched. New `banner-github.svg` coexists alongside old `banner.svg`. (sources: ARCHITECTURE, FEATURES)
- **Palette standardization:** The milestone establishes the Everforest dark "G1" subset as canonical for all branding. Existing `banner.svg` uses a different ocean/blue palette. New assets use: bg `#2D353B`, fg `#D3C6AA`, grey `#859289`, border `#475258`, accent `#A7C080`, link `#83C092`. (sources: STACK, UX)
- **SVG authoring convention:** Hand-crafted inline SVG with calculated coordinates, no external tooling. Uses `<rect>`, `<text>`, `<line>`, `<path>`, `<circle>` primitives with inline `fill`/`font-*` attributes (not CSS classes). This matches the existing convention in `branding/banner.svg`. (sources: FEATURES, ARCHITECTURE)
- **Commit strategy:** Atomic commits per concern area (branding, community files, README, references) for clean bisection and independent review. (source: ARCHITECTURE)
- **State management:** No runtime state changes. All changes are filesystem artifacts. Git is the only state management concern. (source: ARCHITECTURE)

### User Experience Direction

**Domain conventions:** (source: UX)
- The header stack pattern (centered banner + badge row + attribution) is the dominant layout across high-quality OSS repos. RAPID's spec matches this convention exactly.
- Progressive disclosure (visual hook -> install -> quickstart -> architecture -> reference -> contributing) matches the standard top-down funnel. Developers spend 10-30 seconds deciding whether to star a repo.
- Static SVGs over Mermaid is the convention for high-polish projects. SVGs allow palette-matched branding and consistent cross-platform rendering.

**Interaction models:** (source: UX)
- README IS the product interface for this milestone. Every UX decision optimizes for the first-impression read.
- Issue forms are interaction design -- structured forms guide reporters through needed information. Standard patterns: description, repro steps, expected vs actual, version/environment.
- PR templates must be short (10-20 lines). Long templates get deleted.

**Terminology:** (source: UX)
- Use "plugin" (not "extension" or "package")
- Use `/namespace:command` syntax for skills
- Install path is `claude plugin add`, not npm/pip/brew
- "set" is RAPID-specific terminology (not "workstream" or "feature branch" for users)

**Accessibility:** (sources: UX, OVERSIGHTS, PITFALLS)
- SVGs need solid dark backgrounds (not transparent) -- GitHub renders in both light and dark mode
- Foreground `#D3C6AA` on bg `#2D353B` = 7.4:1 contrast (exceeds WCAG AAA)
- Grey `#859289` on bg `#2D353B` = ~3.4-4.1:1 contrast (borderline WCAG AA; use only for large/decorative text)
- SVGs should include `<title>` and `<desc>` elements; `<img>` tags need meaningful `alt` text
- `<details>` elements are natively accessible (screen readers announce as disclosure widgets)
- Heading hierarchy (H1 > H2 > H3) enables proper screen reader navigation

**Key UX signals:** (source: UX)
- 3 badges = minimum maturity signal ("this is maintained software")
- Issue templates signal "we accept contributions"
- Short CONTRIBUTING.md signals "we respect your time"
- `pragnition/RAPID` (org ownership) signals longevity over individual `fishjojo1/RAPID`
- Georgia serif typography communicates: minimalist, intentional, confident

## Contradictions

### Contradiction: SVG Font Rendering Severity

- **PITFALLS says:** SVG text/font rendering on GitHub is "Critical" impact -- "the entire branding effort fails visually" if Georgia renders incorrectly. Recommends converting ALL title text to `<path>` elements as the primary approach.
- **STACK says:** SVG font rendering inconsistency is "Low" impact -- "visual inconsistency is cosmetic only." Recommends keeping live text and accepting fallback behavior. Notes that "the design uses simple centered text with generous spacing, so font metric differences are absorbed."
- **UX says:** Georgia is "a web-safe serif font available on all platforms" and "will render reliably without font embedding" -- recommends using it directly in SVG text elements.
- **OVERSIGHTS says:** The new banner "introduces font-dependent rendering -- a regression in cross-platform consistency" compared to the existing pixel-art banner. Recommends converting text to `<path>` elements.
- **Resolution:** The severity depends on the context. For the banner SVG title ("RAPID"), which is the first visual impression, text-to-path conversion is the safer approach (PITFALLS/OVERSIGHTS are correct). For diagram labels in lifecycle-flow and agent-dispatch SVGs, where minor font variation is acceptable and editability is valuable, live text with fallback chains is appropriate (STACK is correct). UX's claim that Georgia "will render reliably" on GitHub is factually incorrect -- GitHub's camo proxy does not guarantee font availability. **Needs user decision on banner text: paths vs live text.**

### Contradiction: Social Preview Rasterization Tool

- **FEATURES says:** Recommends ImageMagick `convert` as primary (Option 1), Playwright as fallback (Option 2). Notes ImageMagick is "already installed."
- **STACK says:** Recommends `rsvg-convert` as primary (best text rendering via Pango/fontconfig), ImageMagick as fallback. Notes `convert` is deprecated in IMv7 -- use `magick` instead.
- **ARCHITECTURE says:** References "Playwright-based rasterization" in the data flow.
- **OVERSIGHTS says:** Playwright is not in project dependencies, and adding it for one PNG is heavy. Raises concern about reproducibility.
- **Resolution:** STACK's recommendation is most technically accurate. `rsvg-convert` 2.61.4 is installed and provides the best text rendering via Pango/fontconfig. FEATURES incorrectly references `convert` (deprecated in IMv7). The correct fallback chain is: rsvg-convert (primary) -> magick (fallback) -> Playwright (last resort). **Use `rsvg-convert`.**

### Contradiction: Everforest Grey Contrast Ratio

- **PITFALLS says:** Grey `#859289` on `#2D353B` is "approximately 3.4:1" -- fails WCAG AA for normal text.
- **UX says:** Grey `#859289` on `#2D353B` is "approximately 4.1:1" -- "meets WCAG AA."
- **Resolution:** The actual contrast ratio is approximately 3.4:1 (PITFALLS is correct based on standard WCAG calculation). The UX research may have used a different calculation method or made an error. At 3.4:1, the grey passes WCAG AA for large text (18pt+/14pt bold) but fails for normal text. **Use grey only for large/decorative text; use `#9DA9A0` (grey2) or `#D3C6AA` (fg) for smaller readable text.**

### Contradiction: `<picture>` Element Usage

- **FEATURES says:** The spec uses `<picture><source srcset>` for the banner.
- **PITFALLS says:** The `<picture>` wrapper is redundant since only one variant exists and adds complexity without benefit. Recommends plain `<img>` tag instead.
- **UX says:** `<picture>` with `media="(prefers-color-scheme: dark)"` can provide mode-specific images, but "for a consistently dark-branded project, a single dark SVG is the conventional choice."
- **Resolution:** PITFALLS and UX agree -- a single dark variant on a solid background works in both modes. Use plain `<img>` unless separate light/dark variants are planned. **Use `<img>` tag.**

## Recommended Approach

This milestone is a **presentation polish** operation on a mature brownfield codebase. No runtime code, no dependencies, no state changes. All deliverables are static files (SVG, PNG, Markdown, YAML) that enhance the repository's GitHub presentation and community infrastructure. The existing 5-layer architecture remains completely untouched.

The work decomposes cleanly into four independent concern areas: branding assets, community files, README overhaul, and reference migration. The first two have zero interdependencies and can execute in full parallel. The README overhaul depends on branding assets (it references SVG file paths via `<img>` tags). Reference migration runs last because it touches the README and should not create double-edit conflicts.

The single most important technical decision to make before any work begins is the **SVG font rendering strategy**. Four of six research files flag this as a risk. The banner title "RAPID" should use `<path>` elements (converted from Georgia serif) to guarantee pixel-perfect rendering on GitHub's camo proxy. Diagram labels in the lifecycle-flow and agent-dispatch SVGs can use live `<text>` elements with a fallback font chain (`Georgia, 'Times New Roman', serif`) because minor variation in diagram labels is acceptable and editability is valuable. This hybrid approach balances visual consistency (banner) with maintainability (diagrams).

For rasterization of the social preview PNG, use `rsvg-convert` (v2.61.4, already installed) as the primary tool. It renders SVG text via Pango/fontconfig, producing Noto Serif where Georgia is unavailable -- a visually acceptable substitution for a serif title on a solid background. If text-to-path conversion is done for the banner, the font dependency is eliminated entirely for the PNG as well.

The reference migration from `fishjojo1/RAPID` to `pragnition/RAPID` must be scoped carefully. Only 5-8 active files need changes; the remaining 35+ files in `.planning/archive/` and `.planning/milestones/` are historical records that should not be modified. The `plugin.json` manifest is the highest-priority file because it controls plugin discovery via `claude plugin add`. The `author.name` field is a distinct decision from the repository URL -- the spec maintains personal attribution ("Built with love by fishjojo1") while changing the organizational ownership.

### Suggested Set Boundaries

- **Set A: Branding Assets** (Banner SVG, Lifecycle Flow SVG, Agent Dispatch SVG, Social Preview PNG, `.gitattributes`) -- Rationale: All SVG/PNG creation is a single visual design concern. These files must exist before the README can reference them. The `.gitattributes` file is trivial and must precede the PNG commit. This set has zero dependencies on other sets and can start immediately.

- **Set B: Community Infrastructure** (CONTRIBUTING.md, Issue Templates, PR Template) -- Rationale: All community files are independent Markdown/YAML that do not depend on branding assets or README changes. Can execute fully in parallel with Set A. Self-contained concern with its own validation requirements (YAML form testing on a fork).

- **Set C: README Overhaul + Reference Migration** (README.md restructure, `fishjojo1` -> `pragnition` migration across active files) -- Rationale: The README overhaul depends on Set A (references SVG file paths). Reference migration should run after the README is finalized to avoid double-editing. These two features share the same files and should be in one set to prevent ownership conflicts. This set must run after Set A completes.

This yields a DAG of: `{Set A, Set B} -> Set C`, where A and B are fully parallel and C depends on A. Set B has no downstream dependents and can merge independently at any time.
