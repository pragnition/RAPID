## Domain Conventions

### How Similar Products Handle the Patterns Relevant to This Milestone

**1. Solo Mode vs Team Mode in Agentic Dev Tools**

The developer tool ecosystem has converged on a spectrum model rather than a binary solo/team switch:

- **Claude Code Agent Teams** (Anthropic): No explicit "solo mode." Solo developers simply don't use the team coordination features. The tool itself doesn't change behavior -- overhead features (like branch isolation) are opt-in by default. (Source: [Claude Code Agent Teams docs](https://code.claude.com/docs/en/agent-teams))
- **Cursor 2.0**: Multi-agent mode is always on. Solo developers see a single-agent grid in Mission Control. No ceremony reduction for solo use -- the system is lightweight enough that a team-of-one doesn't feel the overhead. (Source: [Cursor's Third Era](https://www.latent.space/p/cursor-third-era))
- **Zed**: Collaboration features "shine when your team leans into them; if you're mostly solo and offline, some of the magic will feel less critical" -- features are present but not forced. (Source: [Agentic CLI Tools comparison](https://getstream.io/blog/agentic-cli-tools/))
- **Terraform Workspaces**: Uses a single `default` workspace for solo developers, multiple named workspaces for teams. The command surface is identical; the difference is configuration, not behavior. (Source: Terraform docs)

**Convention:** The industry standard is "team features that gracefully degrade for solo use" rather than an explicit mode toggle. The key user expectation is that solo mode should remove ceremony, not capabilities. Users expect identical commands with fewer steps, not a separate command set.

**2. Scaffold / Foundation Layer Commands**

- **WordPress CLI** (`wp scaffold`): Generates plugin/theme boilerplate as a subcommand. Foundation scaffolding runs once, is committed to main, and all subsequent feature work builds on it. Scaffolding is project-type-aware (plugin vs theme vs block-theme). (Source: [WP-CLI scaffold docs](https://developer.wordpress.org/cli/commands/scaffold/plugin/))
- **dotnet scaffold**: Interactive CLI tool that asks the user to select the scaffolding category from a list of supported categories. Generates layered project structures: Business, Data, HTML, Help, Test packages in sequence. (Source: [dotnet scaffold announcement](https://devblogs.microsoft.com/dotnet/introducing-dotnet-scaffold/))
- **Angular CLI** (`ng generate`): Scaffold runs at project init or on-demand. Generated code follows framework conventions and integrates with existing project structure. Always idempotent -- re-running doesn't break existing code.
- **Create React App / Vite**: Foundation scaffolding is the first step. All feature development assumes the scaffold exists. These tools explicitly separate "project foundation" from "feature development."

**Convention:** Scaffold commands are project-type-aware, run once (early in the project lifecycle), commit to the main branch, and are the implicit dependency of all feature work. Interactive selection of scaffold components is the dominant pattern over fully automatic scaffolding. The "set-0" concept -- an implicit dependency for all real sets -- aligns well with how Angular, Rails, and .NET treat their scaffold output.

**3. Review Pipeline Decomposition**

- **TDFlow** (2025 research): Decomposes repository-scale software engineering into four sub-agent components with specialized roles. Achieves 88.8% pass rate on SWE-Bench Lite by "reducing long-context burden on any individual sub-agent" and "focusing each sub-agent on specific, pre-defined sub-tasks." (Source: [TDFlow paper](https://arxiv.org/html/2510.23761v1))
- **Baz.co Agentic Code Review**: Breaks reviews into phases -- context mapping, intent inference, Socratic questioning, and targeted investigations. Targeted investigation spawns independent sub-agents that traverse the repo independently and report back. (Source: [Architecture of Agentic Code Review](https://baz.co/resources/engineering-intuition-at-scale-the-architecture-of-agentic-code-review))
- **Zencoder Agentic Pipeline**: Unit tests, build failure detection/fix, and code reviews are explicitly modeled as independent pipeline stages. (Source: [Zencoder agentic pipeline](https://zencoder.ai/product/agentic-pipeline))
- **GitHub Actions**: CI/CD stages (lint, test, build, deploy) have always been independently invocable and composable. The pattern of "scope first, then run stages" is universal in CI/CD.

**Convention:** The industry is converging on independent, composable review stages rather than monolithic review commands. The key insight from TDFlow is that decomposition is not just a UX improvement -- it measurably improves output quality by constraining sub-agent scope. Users expect each stage to be independently re-runnable and to detect its own prerequisites (e.g., "run /review first to generate scope").

**4. Context Window Management and Compaction**

- **Claude Code built-in** (2026): Server-side compaction at ~167K tokens (33K buffer). Users can customize compaction behavior in CLAUDE.md: "When compacting, always preserve the full list of modified files and any test commands." (Source: [Claude Code context buffer management](https://claudefa.st/blog/guide/mechanics/context-buffer-management))
- **Claude Code best practice**: "Subagents explore in a separate context, keeping your main conversation clean for implementation." Structured prompts consume 30% fewer tokens than narrative prompts. (Source: [Claude Code best practices](https://code.claude.com/docs/en/best-practices))
- **Opus 4.6** (March 2026): Introduces adaptive reasoning and context compaction for long-running agents. 1M context window available on Max/Team/Enterprise plans. (Source: [InfoQ Opus 4.6 announcement](https://www.infoq.com/news/2026/03/opus-4-6-context-compaction/))

**Convention:** The dominant pattern is "write to disk, read from disk" rather than relying on in-conversation context. RAPID already follows this pattern with STATE.json and artifact files. The compaction concern is best addressed by reducing context generation (shorter prompts, leaner returns) and delegating to subagents rather than building a custom compaction layer that might conflict with Claude Code's built-in system.

**5. Version Migration Commands**

- **AWS CLI v1-to-v2 Migration Tool**: `--dry-run` flag detects breaking changes without modifying scripts. Reports what would change in a structured format. (Source: [AWS CLI migration docs](https://docs.aws.amazon.com/cli/latest/userguide/cli-migration-tool.html))
- **GitHub Actions Importer**: Dry-run subcommand converts pipelines and outputs YAML without applying changes. Sequential migration steps (audit -> dry-run -> migrate). (Source: [GitHub Actions Importer docs](https://docs.github.com/en/actions/tutorials/migrate-to-github-actions/automated-migrations/use-github-actions-importer))
- **Entity Framework Core**: `dotnet ef database update --dry-run` shows SQL that would run without executing it. (Source: [EF Core CLI docs](https://learn.microsoft.com/en-us/ef/core/cli/dotnet))
- **Django migrations**: `python manage.py migrate --plan` shows what migrations would run. `python manage.py showmigrations` shows applied vs pending. Migrations are sequential and never skip intermediate steps.

**Convention:** Migration commands universally support `--dry-run`, produce structured reports, back up existing state, and run versioned steps sequentially (never skipping intermediate versions). The audit-then-migrate pattern (show what will change, get approval, apply) mirrors RAPID's existing propose-then-approve pattern for roadmaps.

**6. Monolithic CLI Decomposition**

- **kubectl**: Subject-verb structure (`kubectl get pods`, `kubectl apply -f`). Commands are organized into groups (basic, deploy, debug, etc.). Each command is a self-contained module. (Source: [clig.dev](https://clig.dev/))
- **Docker CLI**: Two levels of subcommand -- one noun, one verb (`docker container create`, `docker image build`). Consistent flag names across subcommands. (Source: [Atlassian CLI principles](https://www.atlassian.com/blog/it-teams/10-design-principles-for-delightful-clis))
- **git**: Modular architecture where each subcommand (`git-commit`, `git-diff`, etc.) is a separate executable. The main `git` binary is a thin router. (Source: git internals)

**Convention:** The standard pattern for splitting a monolithic CLI is: thin router file that dispatches to `src/commands/{command}.cjs` files. Subcommand consistency (same flags, same output format, same error handling) matters more than subcommand count. Node.js tools typically avoid heavyweight argument parsing libraries (commander, yargs) in plugin contexts and use a lightweight custom `parseArgs()` helper.

## Interaction Models

### Primary Interaction Paradigm

RAPID is a **Claude Code slash-command plugin**. The interaction model is:
1. User types `/rapid:{command}` in the Claude Code terminal
2. The skill orchestrates subagent spawns and CLI calls
3. Progress is communicated via markdown banners in the conversation
4. User decisions are collected via `AskUserQuestion` (multiple-choice or freeform)
5. Results persist to filesystem artifacts (STATE.json, PLAN.md, etc.)

This paradigm is well-established and should not change for this milestone.

### Input/Output Patterns Users Expect

**Input patterns:**
- Numeric shorthand for set references (`/rapid:start-set 1` instead of `/rapid:start-set auth-system`)
- Flags for behavior modification (`--skip`, `--solo`, `--dry-run`, `--post-merge`)
- Flag formats: `--flag value` and `--flag=value` both expected (currently only `--flag value` is supported)
- No interactive multi-step wizards (anti-pattern identified in v3.0 research)

**Output patterns:**
- Branded banners at command start (already implemented)
- Progress breadcrumbs showing workflow position (already implemented)
- Single next-action suggestion at command end (already implemented)
- Structured JSON to stdout for machine consumption, human text to stderr (already implemented)
- Review stage results displayed as formatted tables with counts

### Feedback and Progress Indication

- Stage banners at major boundaries (init, discuss, plan, execute, review, merge)
- Agent count and current step name during multi-agent operations
- Elapsed time for long operations (not yet implemented, would be valuable for execute-set)
- Cycle count and iteration progress during bug hunt loops (already implemented)

### Error Communication Patterns

RAPID's existing "Errors Are Navigation" pattern (progress breadcrumbs showing what is done, what failed, what to run next) is the correct approach and should be preserved. The primary improvement needed is **standardizing the error output format** across all CLI handlers (item #11 in the milestone: Error Response Format Standardization).

Current state: three different error patterns coexist:
- `error()` + `process.exit(1)`
- `{ error: msg }` JSON output + exit
- Throw and let caller handle

**User expectation:** Every error should produce both machine-readable JSON on stdout AND human-readable text on stderr, consistently. The JSON should include the error message, the current state (what is done), and the suggested recovery command.

## Information Architecture

### How Information Should Be Organized and Presented

RAPID's information architecture follows a hierarchy:

```
Project Level:   /status         (dashboard -- where am I?)
Set Level:       /start-set N    (claim set, create isolation)
                 /discuss-set N  (capture vision)
                 /plan-set N     (decompose into waves/jobs)
                 /execute-set N  (implement)
                 /review N       (quality gate)
                 /merge N        (integrate)
Auxiliary:       /scaffold       (foundation layer)
                 /migrate        (version upgrade)
                 /quick          (one-off fix)
```

**For the Developer Experience milestone, the key IA changes are:**

1. **Review command decomposition** creates a second level of navigation within the review stage:
   - `/review N` -> scope only (produces REVIEW-SCOPE.md)
   - `/unit-test N` -> reads scope, runs unit tests
   - `/bug-hunt N` -> reads scope, runs adversarial pipeline
   - `/uat N` -> reads scope, runs acceptance tests

   This follows the "progressive disclosure" principle: users discover review substages as needed. The entry point (`/review`) produces the prerequisite artifact (REVIEW-SCOPE.md) that other stages consume.

2. **Solo mode** should be a configuration flag, not a separate information architecture. Users should not need to learn a different command set. The same `/status`, `/start-set`, `/execute-set` commands should work identically in solo mode with reduced ceremony.

3. **Deprecated skills cleanup** (item #18) simplifies the information architecture by removing 7 old command names that currently redirect. "Skill not found" is a better experience than a redirect message that adds cognitive load.

### Navigation and Discovery Patterns

- `/status` remains the primary discovery surface (shows all sets and their next actions)
- `/help` should be a static reference card (implemented, not changed by this milestone)
- Each command validates its own prerequisites and tells the user what to run first
- Numeric shorthand is the primary navigation mechanism for sets

### Content Hierarchy and Progressive Disclosure

The milestone adds two new entry points to the command hierarchy:

| Level | Commands | When Discovered |
|-------|----------|-----------------|
| Core workflow | /init, /start-set, /discuss-set, /plan-set, /execute-set, /review, /merge | During first project setup |
| Review substages | /unit-test, /bug-hunt, /uat | After first /review run |
| Foundation | /scaffold | After /init, before /start-set |
| Maintenance | /migrate, /status, /quick | When needed |

This keeps the core 7-command workflow intact and adds new commands as progressive disclosure layers.

## Accessibility Considerations

### Key Accessibility Patterns for CLI Interaction

- **Screen reader compatibility:** All output uses plain text and markdown. Banners use ASCII characters, not Unicode box-drawing (verified in current implementation). Status indicators use text labels, not color-only indicators.
- **Keyboard-only navigation:** All AskUserQuestion interactions are keyboard-navigable (handled by Claude Code's built-in UI).
- **Color and contrast:** RAPID uses agent name colors in banners. These are decorative and always accompanied by text labels. No information is conveyed by color alone.
- **Cognitive accessibility:** The "7 core commands + progressive disclosure" structure respects Miller's Law. Single next-action suggestions reduce decision fatigue.

### Known AskUserQuestion Accessibility Issues

The AskUserQuestion multiselect mode has a known bug where "Enter key acts like Tab instead of selecting options" (Source: [GitHub issue #12030](https://github.com/anthropics/claude-code/issues/12030)). This is relevant to the "Let Claude Decide All" fix (bug #7) -- the multiselect interaction model is fundamentally fragile. The fix should move toward single-select or sequential questions rather than relying on multiselect behavior.

## User Expectations

### What Users Will Assume Based on Domain Research

**1. Solo mode users expect zero overhead, not "team mode with features disabled."**

Users choosing solo mode have a mental model from tools like Cursor (single-agent editing), Aider (single conversation), and basic Claude Code usage (no isolation). They expect:
- No branch creation, no worktree setup, no merge ceremonies
- Commits landing directly on their current branch
- `/merge` either not existing or being a no-op with a friendly message
- The same lifecycle steps (discuss, plan, execute, review) working identically
- The ability to switch to team mode later without losing work

**Risk:** If solo mode still creates branches "just without worktree isolation," users will perceive it as broken. The expectation is zero git overhead beyond normal commits.

**2. Scaffold command users expect a foundation they can build on, not a replacement for framework CLIs.**

Users will have already run `create-react-app`, `cargo init`, or similar before invoking `/scaffold`. They expect RAPID's scaffold to add:
- Shared test configuration
- Common directory structure conventions
- Routing skeleton / API client setup / state management boilerplate
- Shared types/interfaces that multiple sets will import

They do NOT expect it to replace their framework's scaffolding. The scaffold should detect what already exists and fill gaps, not overwrite.

**3. Review pipeline users expect stage independence.**

After decomposition, users will expect to run `/bug-hunt 1` without running `/unit-test 1` first. The only hard prerequisite should be `/review 1` (which produces REVIEW-SCOPE.md). Users will also expect to re-run a single stage without re-running others (idempotent overwrite behavior).

**4. Context compaction users expect transparency.**

If context is compacted, users want to know:
- What was compacted (which completed stages)
- Where the full information lives on disk
- That nothing they need was lost

The risk is that custom compaction fights with Claude Code's built-in compaction. Users will blame RAPID for quality degradation even if the built-in system caused it. Recommendation: lean into "write to disk, read from disk" rather than building a parallel compaction system.

**5. Migration command users expect safety and predictability.**

Users updating RAPID on an existing project expect:
- `--dry-run` that shows exactly what will change
- Automatic backup of pre-migration state
- Sequential version steps (no skipping)
- Clear report of what changed
- The tool to work even on partially-migrated state (crash recovery)

**6. "Let Claude decide all" users expect a two-step interaction.**

The current bug (item #7) conflates a meta-action with topic selection. Users expect either:
- A preliminary question: "Would you like to discuss specific areas or let Claude decide everything?" (if "Claude decides" -> skip topic selection entirely)
- Or: topic selection with an implicit "Claude decides all unselected topics" behavior (no explicit "Let Claude decide all" checkbox)

The Claude Code AskUserQuestion multiselect has known bugs (Enter acts like Tab). Using single-select or sequential prompts is more reliable.

### Terminology Users Use vs Technical Terms

| User Term | RAPID Internal Term | Notes |
|-----------|-------------------|-------|
| "project" | milestone | Users think in projects, not milestones |
| "feature" or "task" | set | "Set" is RAPID-specific jargon |
| "plan" | wave / PLAN.md | Users don't distinguish waves from jobs |
| "run tests" | /unit-test or /review with unit test stage | Users expect a direct "test" command |
| "find bugs" | /bug-hunt or /review with bug hunt stage | Self-explanatory name, good |
| "code review" | /review | Users may expect this to produce PR-like output |
| "ship it" / "merge" | /merge | Natural alignment |
| "upgrade" / "update" | /migrate | Users expect "upgrade" more than "migrate" |

### Mental Models from Competing Products

Users coming from:
- **Cursor**: Expect immediate agent execution without lifecycle ceremonies. Solo mode should feel this lightweight.
- **Copilot Workspace**: Expect specification-then-plan-then-implementation. RAPID's discuss-then-plan-then-execute maps well.
- **Devin**: Expect autonomous operation with occasional steering. RAPID's "strong defaults, rare questions" principle aligns.
- **Plain Claude Code**: Expect slash commands that work after `/clear`. Every command must bootstrap from filesystem state.

### Onboarding and Learning Curve Expectations

Users expect to learn RAPID in this order:
1. `/rapid:install` (one-time)
2. `/rapid:init` (project setup)
3. `/rapid:status` (where am I?)
4. The lifecycle loop: start-set -> discuss-set -> plan-set -> execute-set -> review -> merge

The Developer Experience milestone should not change this learning curve. New commands (/scaffold, /migrate, /unit-test, /bug-hunt, /uat) are discovered through the workflow, not upfront.

## Recommendations

Ordered by priority within the context of this milestone's goals. Each recommendation is tied to specific milestone items.

### Critical Priority

1. **Fix "Let Claude Decide All" as two-step interaction (Bug #7):** Replace the multiselect option with a preliminary single-select question: "Would you like to discuss specific areas or let Claude decide everything?" This avoids the AskUserQuestion multiselect bug and matches user expectations for meta-actions. -- Priority: critical

2. **Standardize error output format across all CLI handlers (Item #11):** Create `exitWithError(msg, code)` helper producing JSON on stdout and human text on stderr. Apply to all 18 handlers in rapid-tools.cjs. This is the foundation for every other improvement -- inconsistent errors undermine trust. -- Priority: critical

3. **Fix resume command duplication (Item #8):** Route both `resume` and `execute resume` to a single `resumeSet()` function. Two code paths for the same operation is a data corruption vector during parallel execution. -- Priority: critical

4. **Fix state mutation bypass in `execute update-phase` (Item #9):** Wrap in `withStateTransaction()`. Every state mutation must use the transaction pattern -- no exceptions. -- Priority: critical

5. **Fix MERGE-STATE mutation inconsistency (Item #10):** Create `withMergeStateTransaction()` mirroring the state-machine pattern. Three different write patterns for the same data structure is a consistency hazard. -- Priority: critical

### High Priority

6. **Implement Solo Mode as configuration-driven ceremony reduction (Feature #1):** Model as a boolean `solo` flag in `config.json` (set during /init) and a `--solo` override on `/start-set`. In solo mode: skip worktree creation (work on current branch), skip merge (commits land directly), skip conflict detection. Preserve discuss/plan/execute/review lifecycle. `/merge` in solo mode should no-op with message "Solo mode: commits already on main. Nothing to merge." -- Priority: high

7. **Decompose /review into independent skills (Feature #3):** Split into `/review` (scope only, writes REVIEW-SCOPE.md), `/unit-test`, `/bug-hunt`, `/uat`. Each substage reads REVIEW-SCOPE.md as prerequisite. Each is independently re-runnable. Remove the "which stages?" AskUserQuestion from the current monolithic /review -- the decomposition itself eliminates the need for stage selection. Add judge leaning display with confidence indicator as specified in the user requirements. -- Priority: high

8. **Split monolithic CLI into src/commands/ (Item #15):** Transform rapid-tools.cjs from ~3500-line monolith into thin router that imports from `src/commands/{command}.cjs`. Each command file exports a handler function. This reduces merge conflicts during parallel development and enables per-command testing. Follow git's architecture: main binary is a dispatcher, subcommands are self-contained modules. -- Priority: high

9. **Fix review artifact path mismatch (Item #13):** Migrate review artifacts from `.planning/waves/{setId}/` to `.planning/sets/{setId}/`. Update all references in skills and library code. This is a v2-to-v3 migration leftover that causes confusion. -- Priority: high

10. **Fix missing DEFINITION.md on start-set (Bug #6):** Investigate path resolution, timing, and roadmapper generation. The most likely root cause is a path resolution mismatch between the set ID format used during init (by the roadmapper) and the set ID format resolved by `resolve set` during start-set. -- Priority: high

### Medium Priority

11. **Implement /scaffold command (Feature #2):** New skill that runs after /init, acts as "set-0" implicit dependency. Should be project-type-aware (detect from init research), interactive (ask what to include via AskUserQuestion single-select), and commit to main before any set branches are created. The roadmapper should be aware of scaffold existence. -- Priority: medium

12. **Implement /migrate command (Feature #5):** Support `--dry-run`, automatic backup, sequential version steps, structured migration report. Detect version from STATE.json schema shape. Run schema migrations, artifact relocation, agent rebuild, config updates. Stamp new version after successful migration. The audit-then-migrate pattern (show -> approve -> apply) matches RAPID's existing propose-then-approve convention. -- Priority: medium

13. **Replace manual argument parsing with parseArgs helper (Item #16):** Write a lightweight `parseArgs(args, schema)` helper (~30 lines) handling `--flag value` and `--flag=value`. Apply across all CLI handlers. This eliminates off-by-one flag parsing bugs. -- Priority: medium

14. **Add stdin JSON validation with Zod (Item #17):** Create `readAndValidateStdin(zodSchema)` utility for all stdin-reading commands (plan create-set, state add-milestone, execute pause, review log-issue). Rejects malformed input with descriptive errors instead of silent propagation. -- Priority: medium

15. **Fix agent build comment-based detection fragility (Item #14):** Use only the SKIP_GENERATION list for gating core agents. Remove `<!-- CORE: Hand-written agent` comment-based detection path. -- Priority: medium

16. **Resolve path resolution inconsistency (Item #12):** Standardize on `path.resolve()` everywhere (safer with relative cwd). Audit all `path.join()` calls in execute and worktree handlers. -- Priority: medium

### Low Priority

17. **Implement context-aware internal compaction (Feature #4):** Lean into "write to disk, read from disk" rather than building a custom compaction layer. Focus on reducing context generation: shorter agent prompts, leaner RAPID:RETURN payloads, summary-only wave results between execution steps. Add CLAUDE.md instructions for what to preserve during built-in compaction. Defer a full custom compaction system -- it risks fighting with Opus 4.6's built-in adaptive context compaction. -- Priority: low

18. **Clean up 7 deprecated skills (Item #18):** Remove skills/discuss, skills/execute, skills/plan, skills/set-init, skills/wave-plan, skills/new-milestone, skills/pause (if deprecated). "Skill not found" is a better UX than a redirect message. -- Priority: low

19. **Rename registry read/write functions (Item #19):** Rename `loadRegistry()` to `readRegistry()` (returns frozen copy) and `registryUpdate()` to `withRegistryUpdate()` (mutation callback pattern). Makes intent explicit at call sites. -- Priority: low
