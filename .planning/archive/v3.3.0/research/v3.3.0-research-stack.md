# Stack Research -- Developer Experience Milestone (v3.3.0)

## Core Stack Assessment

### Node.js Runtime

| Property | Value |
|----------|-------|
| Detected minimum | 18+ (setup.sh enforces `>= 18`) |
| Development machine | v25.8.0 (Current track) |
| Current LTS | v24.x (Active LTS), v22.x (Maintenance LTS) |
| Module system | CommonJS (`.cjs` throughout) |

**Key findings:**

- **Node.js 18 hit EOL on April 30, 2025.** The `setup.sh` minimum version check (`NODE_VERSION -lt 18`) allows users to run RAPID on a dead runtime that no longer receives security patches. This is a concrete risk for the milestone since the `/migrate` command (item 5) will need to validate runtime compatibility.
- Node.js 20 reached Maintenance LTS and will EOL on April 30, 2026 -- one month from now. Minimum should be bumped to Node.js 22 (Maintenance LTS until April 2027) or Node.js 24 (Active LTS until October 2026, Maintenance until April 2028).
- The project uses `node:test` built-in test runner (stable since Node.js 20). All 30 test files use it. This is consistent and appropriate -- zero-dependency testing.
- `util.parseArgs()` is stable since Node.js 20. This is directly relevant to milestone item 16 (Manual Argument Parsing replacement). The built-in can replace all 10 `args.indexOf('--flag')` and 12 `args.includes('--flag')` call sites in `rapid-tools.cjs` without adding a dependency.

**Recommendation:** Bump minimum Node.js version to 22 in `setup.sh` and document in CLAUDE.md. Node.js 22 provides stable `node:test`, stable `util.parseArgs()`, and has support until April 2027.

## Upgrade Paths

### Zod 3.25.76 -> Zod 4.x

**Current situation:** Zod 3.25.x is a bridge release that contains both Zod 3 and Zod 4 at different subpaths. The default `require('zod')` import returns Zod 3 behavior. Zod 4 is available at `require('zod/v4')`.

**Breaking changes in Zod 4:**
- `invalid_type_error` and `required_error` options replaced with unified `error` function
- Schema-level error maps take precedence over contextual error maps (reversed from Zod 3)
- Issue type renames: `ZodInvalidTypeIssue` -> `$ZodIssueInvalidType`, etc.
- Some issue types merged or removed (e.g., `ZodInvalidEnumValueIssue` -> `$ZodIssueInvalidValue`)
- Community-maintained codemod `zod-v3-to-v4` available

**Impact on RAPID:** The project uses Zod conservatively in `state-schemas.cjs` -- basic `z.object()`, `z.string()`, `z.enum()`, `z.array()`, `z.literal()`, and `.default()`. No custom error messages, no error maps, no advanced features. Migration effort would be trivial.

**Recommendation:** Not urgent for this milestone. Zod 3 via the 3.25.x bridge is stable and functional. Consider upgrading to Zod 4 in a future milestone when there is a clear feature need (e.g., Zod Mini for smaller bundle, improved TypeScript inference).

**Migration effort:** Trivial -- the schemas used are basic and unchanged between Zod 3 and 4.

### proper-lockfile 4.1.2 -> ???

**Current situation:** No newer version. Package has been unmaintained for approximately 5 years. 15 open issues on GitHub. No known CVEs at present.

**Alternatives evaluated:**
- **async-lock**: In-process only, not inter-process. Not a replacement for file-based locking.
- **lockfile** (npm): Uses `O_EXCL` flag, which fails on network filesystems. Inferior to proper-lockfile's `mkdir` strategy.
- **Custom implementation**: The `mkdir`-based atomic locking strategy is ~50 lines of code. RAPID already wraps proper-lockfile in `src/lib/lock.cjs`.

**Recommendation:** Keep proper-lockfile for now. The lock.cjs wrapper already abstracts the dependency, so replacing it later is straightforward. The package has no known CVEs and its functionality is simple and battle-tested. If a Node.js signal handling change breaks it, the wrapper layer limits blast radius. Flag for replacement in a future maintenance milestone.

**Migration effort:** Moderate -- requires implementing and testing a custom mkdir-based locking mechanism, plus stale lock detection.

### Node.js Minimum Version: 18 -> 22

**Current situation:** setup.sh allows Node.js 18 (EOL since April 2025).

**Why Node 22:**
- LTS Maintenance until April 2027 (13 months of support remaining)
- `node:test` fully stable
- `util.parseArgs()` fully stable
- All current dependencies work on Node 22
- Node 20 EOL is April 2026 (next month), so it would be a poor choice

**Breaking changes for users on Node 18/20:** None from RAPID's codebase. All code uses standard CommonJS and built-in APIs available since Node 18. The bump is purely about ensuring users get security patches from the Node.js project.

**Recommendation:** Bump minimum to Node 22 in this milestone. Update `setup.sh` version check, `CLAUDE.md` documentation, and the `prereqs` command output.

**Migration effort:** Trivial -- one-line change in setup.sh, documentation updates.

## Stack Risks

Prioritized by severity and likelihood.

### 1. Unmaintained File Locking Dependency
- **Risk:** proper-lockfile has not been updated in ~5 years. Future Node.js changes to signal handling or filesystem APIs could break locking behavior silently.
- **Impact:** Corrupted STATE.json in concurrent execution scenarios. Data loss.
- **Mitigation:** The `lock.cjs` wrapper abstracts the dependency. Monitor for breakage. Plan custom replacement in a future milestone. Current Node.js versions (22-25) work correctly.
- **Priority:** Medium (no immediate breakage, but long-term liability)

### 2. EOL Node.js Minimum Version
- **Risk:** Users on Node.js 18 (EOL) or 20 (EOL next month) receive no security patches. setup.sh allows these versions.
- **Impact:** Security vulnerabilities in the Node.js runtime. Not RAPID-specific, but RAPID should not encourage running on unsupported runtimes.
- **Mitigation:** Bump minimum to Node.js 22 in this milestone.
- **Priority:** High (easily fixable, should not be deferred)

### 3. Version Detection Gap for /migrate
- **Risk:** STATE.json has `version: 1` (Zod schema literal) but no `rapidVersion` field tracking the RAPID plugin version (currently 3.2.0 in plugin.json, 3.0.0 in package.json). The `/migrate` command (item 5) needs reliable version detection.
- **Impact:** Migration command cannot determine source version accurately. May apply wrong migration steps.
- **Mitigation:** Add `rapidVersion` field to STATE.json schema. Detect legacy versions by schema shape (absence of field = pre-3.3.0). The version.cjs module already reads package.json version -- extend this for state stamping.
- **Priority:** High (prerequisite for item 5)

### 4. Plugin Version Mismatch
- **Risk:** `package.json` says version 3.0.0 while `plugin.json` says 3.2.0. No single source of truth for the RAPID version.
- **Impact:** `version.cjs` reads from `package.json` (3.0.0), which is stale. The `/migrate` command would use the wrong version number when stamping STATE.json.
- **Mitigation:** Synchronize `package.json` and `plugin.json` versions. Consider making `plugin.json` the single source of truth (it is what Claude Code reads) and having `version.cjs` read from there instead.
- **Priority:** High (must fix before /migrate lands)

### 5. Monolithic CLI Splitting Complexity
- **Risk:** `rapid-tools.cjs` is 2,717 lines with 18 handlers, tightly coupled argument parsing, and one exported function (`migrateStateVersion`). Splitting (item 15) requires careful extraction without breaking the test file (`rapid-tools.test.cjs`).
- **Impact:** If done carelessly, introduces regressions across all CLI commands.
- **Mitigation:** Extract handlers into `src/commands/{command}.cjs` with co-located tests. Keep `rapid-tools.cjs` as a thin router. Move tests alongside their handlers. Use `node:test` `--test` glob to discover split test files.
- **Priority:** Medium (structural, not a runtime risk)

### 6. Context Compaction Platform Dependency
- **Risk:** Item 4 (Context-Aware Internal Compaction) depends on Claude Code's context management internals, which are undocumented and may change between Claude Code releases.
- **Impact:** Compaction logic may conflict with Claude Code's built-in compression. Behavior may change without notice across Claude Code updates.
- **Mitigation:** Design compaction as file-based summarization (write summaries to disk, reference from prompts) rather than attempting to manipulate Claude Code's internal context window. This makes the feature platform-resilient.
- **Priority:** Medium (design risk, not a dependency risk)

## Recommendations

Ordered by priority for the milestone roadmap.

### 1. Synchronize Version Numbers
- **Action:** Align `package.json` version with `plugin.json` version (both to 3.3.0 for this milestone). Make `version.cjs` read from a single authoritative source.
- **Rationale:** Prerequisite for `/migrate` command (item 5). Prevents version confusion across the codebase.
- **Priority:** Critical -- must be done before any version-aware features ship.

### 2. Bump Node.js Minimum to 22
- **Action:** Change `setup.sh` check from `-lt 18` to `-lt 22`. Update documentation. Update `prereqs` command output text.
- **Rationale:** Node 18 is EOL, Node 20 EOL is April 2026. Node 22 is Maintenance LTS until April 2027. This also unlocks stable `util.parseArgs()` for item 16.
- **Priority:** High -- straightforward change, eliminates EOL risk.

### 3. Use util.parseArgs() for Argument Parsing Replacement
- **Action:** For item 16, use the Node.js built-in `util.parseArgs()` (stable since Node 20) instead of adding a dependency like `commander` or `yargs`. Write a thin `parseArgs(args, schema)` wrapper.
- **Rationale:** Zero new dependencies. Handles `--flag value` and `--flag=value` consistently. Supports `boolean` and `string` types. Fits the project's zero-devdeps philosophy.
- **Priority:** High -- directly supports item 16 with no dependency cost.

### 4. Add rapidVersion Field to STATE.json Schema
- **Action:** Add `rapidVersion: z.string().optional()` to the `ProjectState` Zod schema. Stamp on creation and migration. Use absence to detect pre-3.3.0 states.
- **Rationale:** Prerequisite for `/migrate` command (item 5). The `version: z.literal(1)` field tracks schema version, not RAPID version -- both are needed.
- **Priority:** High -- schema change that should land early to establish the pattern.

### 5. Keep Zod 3 for This Milestone
- **Action:** Do not upgrade to Zod 4. The current Zod 3 via 3.25.x bridge is stable and the schemas are simple.
- **Rationale:** Zod 4 migration effort is trivial but adds risk with no benefit for this milestone's features. The bridge release (3.25.x) ensures forward compatibility.
- **Priority:** Low -- defer to a future milestone.

### 6. Keep proper-lockfile for This Milestone
- **Action:** Do not replace proper-lockfile. The `lock.cjs` abstraction layer limits exposure. No known CVEs.
- **Rationale:** Replacement requires implementing custom mkdir-based locking with stale detection. The current solution works. Replacement effort is better spent on milestone features.
- **Priority:** Low -- flag for future maintenance milestone.

### 7. Plan CLI Split to Align with New Commands
- **Action:** For item 15, split into `src/commands/{command}.cjs` files. Route new commands (scaffold, migrate, solo-mode flags) through the split architecture from the start, rather than adding them to the monolith and splitting later.
- **Rationale:** Avoids adding to the 2,717-line file. New features built in the split structure serve as proof of the pattern.
- **Priority:** Medium -- structural improvement that makes all other items easier. Should be sequenced early.

### 8. Consider Native Test Coverage
- **Action:** Add `--experimental-test-coverage` to the test runner invocation for the new modules created in this milestone.
- **Rationale:** Zero-cost (built into Node 22+), provides visibility into coverage of new skill/command code.
- **Priority:** Low -- nice-to-have, not blocking.
