# Research Summary -- Developer Experience Milestone (v3.3.0)

## Project Profile
- **Type:** brownfield CLI plugin (Claude Code extension)
- **Stack:** Node.js (CommonJS) + Zod + proper-lockfile + Git worktrees
- **Scale:** large (19 items: 5 features, 2 bugs, 4 critical fixes, 4 structural improvements, 4 design improvements)
- **Key Constraint:** The monolithic `rapid-tools.cjs` (2717 lines, 18+ handlers) is the single file that almost every milestone item must touch, making sequencing and conflict avoidance the dominant planning concern.

---

## Unified Findings

### Stack and Infrastructure

- **Node.js minimum version is dangerously stale.** setup.sh allows Node.js 18 (EOL April 2025) and Node.js 20 (EOL April 2026). The effective minimum is already Node.js 20 (required for stable `node:test` and `util.parseArgs()`). Bumping to Node.js 22 (Maintenance LTS until April 2027) is trivial and eliminates EOL risk. (sources: STACK, OVERSIGHTS)
- **Plugin version numbers are split.** `package.json` says 3.0.0, `plugin.json` says 3.2.0. `version.cjs` reads from `package.json`, meaning it returns the stale value. The `/migrate` command cannot stamp accurate versions until this is resolved. (source: STACK)
- **STATE.json lacks a `rapidVersion` field.** Only `version: z.literal(1)` exists (schema version, not RAPID version). Version detection from schema shape is fragile. Adding `rapidVersion` to the schema is a prerequisite for `/migrate`. (sources: STACK, OVERSIGHTS)
- **The `z.literal(1)` schema version is a migration barrier.** If `/migrate` bumps the version to 2, all existing `readState()` calls fail Zod validation because `z.literal(1)` rejects anything other than 1. The schema versioning strategy must be redesigned before building migrations. (source: OVERSIGHTS)
- **Zod 3.25.76 bundles Zod 4 internally.** The dual-bundling has caused documented CJS loading failures (GitHub #4434). While 3.25.56+ patched the worst breakage, the risk persists on `npm update`. Recommendation: pin exact version or stay on 3.25.76; do not upgrade to Zod 4 this milestone. (sources: STACK, PITFALLS)
- **Zod `.parse()` strips unknown fields by default.** Adding new fields to STATE.json without using `.passthrough()` on schemas will cause silent data loss -- new fields are written, then silently stripped on the next `withStateTransaction()` call. This is critical for both migration and solo mode. (source: PITFALLS)
- **proper-lockfile is unmaintained (~5 years stale).** 15 open GitHub issues, no known CVEs. The `lock.cjs` wrapper abstracts the dependency, limiting blast radius. The `onCompromised` callback logs but does not abort, meaning lock compromise leads to silent state corruption. (sources: STACK, PITFALLS)
- **No `npm test` script exists.** package.json has zero scripts. Tests are run ad-hoc with `node --test src/lib/*.test.cjs`. After CLI split adds `src/commands/*.test.cjs`, a unified test command is needed. (source: OVERSIGHTS)
- **`writeRegistry()` is not atomic.** Unlike `writeState()` (which uses tmp+rename), `writeRegistry()` does direct `fs.writeFileSync()`. Concurrent reads can see partially-written JSON. (source: OVERSIGHTS)
- **All production dependencies are healthy except proper-lockfile.** ajv 8.18.0, ajv-formats 3.0.1, zod 3.25.76 are all current. Total dependency tree: 11 packages. Zero dev dependencies. (source: STACK)

### Feature Landscape

The 19 items decompose into these functional groups:

**Group A -- Data Integrity Fixes (items 8, 9, 10):**
Resume command duplication, state mutation bypass, MERGE-STATE inconsistency. These address active data corruption and race condition risks. Must be fixed before any structural changes.
(source: FEATURES)

**Group B -- Structural Foundation (items 11, 12, 14, 15, 16, 17, 18, 19):**
CLI split, error standardization, argument parsing, stdin validation, path resolution, agent build detection fix, deprecated skills cleanup, registry naming. These are mechanical refactoring tasks that reduce technical debt and enable clean parallel development.
(source: FEATURES)

**Group C -- Bug Fixes (items 6, 7):**
DEFINITION.md path mismatch bug, discuss-set "Let Claude Decide All" UX bug. Quick wins that improve daily developer experience.
(source: FEATURES)

**Group D -- Major Features (items 1, 3, 5, 13):**
Solo mode, review pipeline decomposition, version migration, review artifact path fix. These deliver user-visible value and require completed structural foundation.
(source: FEATURES)

**Group E -- New Capabilities (items 2, 4):**
Project scaffold command, context-aware compaction. Highest complexity, lowest urgency. Scaffold is independent; compaction is experimental and risky.
(source: FEATURES)

**Dependency chain:**
- Items 11 + 15 + 16 should be done together or in sequence (error format -> CLI split -> arg parsing)
- Item 13 (review artifact path fix) should precede item 3 (review decomposition)
- Item 13 should also inform item 5 (migration includes path relocation)
- Items 8, 9, 10 should precede item 15 (fix bugs before splitting the file they live in)
(sources: FEATURES, ARCHITECTURE)

**Third-party services:** None required. All 19 items can be implemented using the existing stack with zero new dependencies. `util.parseArgs()` is a Node.js built-in. (source: FEATURES)

### Architecture Direction

- **Recommended pattern: modular monolith with command-handler decomposition.** RAPID is a CLI tool, not a service. The existing layered architecture (Skill -> Agent -> CLI -> Library -> Storage) is correct. The decomposition should split the CLI into independent command handlers while preserving the library layer as shared infrastructure. (source: ARCHITECTURE)
- **Proposed structure:** `src/commands/{command}.cjs` files with a thin `rapid-tools.cjs` router (~200 lines). Each command exports `async function handle(cwd, args)`. New library modules: `args.cjs`, `errors.cjs`, `scaffold.cjs`, `migrate.cjs`, `compaction.cjs`. (source: ARCHITECTURE)
- **Transaction pattern should be extended, not replaced.** `withStateTransaction()` is proven. `withMergeStateTransaction()` and `withRegistryUpdate()` should follow the same lock-read-validate-mutate-validate-write-release pattern. (sources: ARCHITECTURE, FEATURES)
- **Solo mode data flow:** `/start-set` skips `createWorktree()` and registers a virtual entry (`path: ".", branch: "main"`). `/execute-set` works directly in the project root. `/merge` no-ops with a friendly message. Solo mode is configuration-driven ceremony reduction, not a separate execution model. (sources: ARCHITECTURE, UX)
- **Review decomposition data flow:** `/review` writes REVIEW-SCOPE.md (or .json). `/unit-test`, `/bug-hunt`, `/uat` each read the scope artifact as their prerequisite. Existing `review.cjs` library module is already concern-scoped and needs no splitting. (sources: ARCHITECTURE, FEATURES)
- **Error handling strategy for CLI split:** Command handlers should throw typed errors. The router in `rapid-tools.cjs` catches and calls `exitWithError()` in one place. `exitWithError()` lives in `core.cjs` (shared). This makes command handlers testable in-process. (sources: ARCHITECTURE, OVERSIGHTS)

### User Experience Direction

- **Solo mode should remove ceremony, not capabilities.** Industry convention (Cursor, Zed, Claude Code native) is "team features that gracefully degrade for solo use." Users expect identical commands with fewer steps, not a different command set. Zero git overhead beyond normal commits. (source: UX)
- **Review decomposition aligns with industry convergence on composable stages.** TDFlow research shows decomposition measurably improves output quality by constraining sub-agent scope. Users expect each stage to be independently re-runnable with prerequisite detection. (source: UX)
- **Scaffold command should detect and fill gaps, not replace framework CLIs.** Users will have already run `create-react-app` or `cargo init`. RAPID's scaffold adds shared test config, directory conventions, and shared types. Interactive component selection is preferred over fully automatic scaffolding. (source: UX)
- **Migration commands universally support `--dry-run`, backup, and sequential steps.** The audit-then-migrate pattern matches RAPID's existing propose-then-approve convention. (source: UX)
- **Context compaction should be "write to disk, read from disk" rather than custom compaction.** Opus 4.6's built-in adaptive context compaction handles the hard problem. RAPID should focus on reducing generation rather than building a parallel system. (source: UX)
- **"Let Claude Decide All" should be a two-step interaction** -- preliminary single-select meta-question, then topic selection. Avoids AskUserQuestion multiselect bugs (GitHub #12030). (source: UX)
- **Error communication should follow "Errors Are Navigation" pattern** -- show what is done, what failed, and what to run next. JSON on stdout + human text on stderr consistently. (source: UX)
- **Progressive disclosure preserves the 7-command core workflow.** New commands (/scaffold, /migrate, /unit-test, /bug-hunt, /uat) are discovered through the workflow, not upfront. (source: UX)

## Contradictions

### Contradiction: CLI Split vs Feature Sequencing

- **ARCHITECTURE says:** CLI split should happen first (Wave 1) because it is a prerequisite for clean parallel development on all other items.
- **PITFALLS says:** Data integrity fixes (items 8, 9, 10) should be done BEFORE the CLI split, because splitting the file preserves bugs permanently in separate modules and makes them harder to fix later.
- **FEATURES says:** Critical fixes first (items 8, 9, 10), then structural fixes including CLI split.
- **Resolution:** PITFALLS and FEATURES are correct. Data integrity fixes (resume duplication, state mutation bypass, MERGE-STATE inconsistency) should land in the monolith first, then the CLI split extracts the already-fixed code. The architecture recommendation to split first is optimistic about doing both simultaneously but the risk of preserving bugs in split files is real.

### Contradiction: Deprecated Skills -- Remove vs Redirect

- **PITFALLS says:** Do not remove deprecated skills entirely. Replace their SKILL.md content with a single-line redirect message. "Skill not found" is confusing; a one-line redirect file is zero maintenance and provides actionable guidance.
- **UX says:** "Skill not found" is a better UX than a redirect message that adds cognitive load. Remove the skills entirely.
- **FEATURES says:** Delete the directories. Claude Code's "skill not found" error is sufficient redirection.
- **Resolution:** This is a genuine UX judgment call. The PITFALLS argument (redirect messages are zero cost and guide users) is slightly stronger than the UX argument (clean removal reduces surface). Recommend keeping single-line redirect SKILL.md files for one more major version, then removing in v3.4.0. This is a low-stakes decision either way.

### Contradiction: Compaction Strategy

- **FEATURES says:** Hybrid approach (lean generation + lightweight monitor). Create `tokenEstimate()` function, add context budget tracking, write compaction module.
- **ARCHITECTURE says:** Implement compaction conservatively. Focus on reducing generation first. The compaction module should be opt-in and logged. Consider deferring to v3.4.0.
- **UX says:** Do not build a custom compaction layer. Lean into "write to disk, read from disk." Opus 4.6 has built-in adaptive context compaction. Defer full custom system.
- **PITFALLS says:** Custom compaction fights Claude Code's built-in system. Double-compaction causes unpredictable quality degradation.
- **Resolution:** Consensus across 3 of 4 sources is that active compaction is high-risk and should be minimized. Implement the lean generation approach (shorter prompts, disk-first artifacts) and context budget monitoring. Defer the active compaction module to a future milestone. Mark as experimental/optional if implemented at all in v3.3.0.

### Contradiction: Path Resolution Strategy

- **FEATURES says:** Standardize on `path.resolve()` everywhere. Replace `path.join(cwd, ...)` with `path.resolve(cwd, ...)`.
- **PITFALLS says:** Normalize `cwd` to absolute at CLI entry point, then use `path.join()` consistently (more explicit about what it does).
- **OVERSIGHTS says:** The real fix is defensive: add a guard in `findProjectRoot()` asserting the returned path is absolute. This makes join vs resolve moot since `cwd` is always absolute.
- **Resolution:** OVERSIGHTS has the most pragmatic solution. Add an assertion in `findProjectRoot()` that the returned path is absolute, then the join-vs-resolve distinction is academic. The assertion is a one-line change that eliminates the entire class of bugs. Use `path.resolve()` for new code as a convention, but do not churn all existing `path.join()` call sites.

## Priority Rankings

### Critical Path Items (must do first)

1. **Synchronize version numbers** (package.json and plugin.json to 3.3.0): Prerequisite for every version-aware feature. The `/migrate` command, version stamping in STATE.json, and `version.cjs` all depend on a single source of truth. (sources: STACK, FEATURES)
2. **Fix data integrity bugs (items 8, 9, 10):** Resume duplication, state mutation bypass, and MERGE-STATE inconsistency are active data corruption vectors. Must be fixed in the monolith before the CLI split extracts them. (sources: FEATURES, PITFALLS, ARCHITECTURE)
3. **Add `.passthrough()` to Zod state schemas:** Without this, any new field added to STATE.json (rapidVersion, soloMode) is silently stripped on the next state mutation. Blocks every feature that adds state fields. (sources: PITFALLS, OVERSIGHTS)
4. **Resolve schema versioning strategy:** Change `version: z.literal(1)` to support version evolution. Without this, `/migrate` cannot stamp new versions. (source: OVERSIGHTS)
5. **Add `npm test` script:** Before any structural changes, ensure there is a single command to verify nothing is broken. Takes 30 seconds, prevents regressions. (source: OVERSIGHTS)

1. **CLI split (item 15) + error standardization (item 11) + argument parsing (item 16):** The structural foundation that makes all subsequent work cleaner. Split the CLI into `src/commands/`, establish error handling pattern, and replace manual flag parsing. Natural to do together. (sources: ARCHITECTURE, FEATURES, OVERSIGHTS)
2. **Bump Node.js minimum to 22:** Eliminates EOL runtime risk, unlocks stable `util.parseArgs()` for item 16. One-line change in setup.sh + documentation updates. (source: STACK)
3. **Bug fixes (items 6, 7):** DEFINITION.md path fix and discuss-set UX fix. Quick wins that improve daily developer experience immediately. (source: FEATURES)
4. **Review artifact path fix (item 13):** Migrate `.planning/waves/` to `.planning/sets/`. Must be done before review decomposition (item 3). (sources: FEATURES, ARCHITECTURE, UX)
5. **Trivial structural fixes (items 12, 14, 18, 19):** Path resolution guard, agent build detection fix, deprecated skills handling, registry naming. Mechanical refactoring that reduces technical debt. (sources: FEATURES, ARCHITECTURE)

### Medium Priority (schedule as capacity allows)

1. **Solo mode (item 1):** High-value feature for solo developers. Requires careful implementation across 8+ skill files and 4+ library modules. Should be built on the already-split CLI for clean per-command solo checks. (sources: FEATURES, ARCHITECTURE, OVERSIGHTS, UX)
2. **Review pipeline decomposition (item 3):** High-value feature that gives users control over review stages. TDFlow research shows decomposition improves output quality. Depends on review path fix (item 13). (sources: FEATURES, ARCHITECTURE, UX)
3. **Version migration (item 5):** Important for upgrade path. Requires resolved schema versioning, `rapidVersion` field, and stable version numbers. Build after schema changes settle. (sources: FEATURES, STACK, OVERSIGHTS)
4. **Stdin JSON validation (item 17):** Apply after CLI split for cleaner per-command schemas. Use `.passthrough()` mode to avoid rejecting agent-generated JSON with extra fields. (sources: FEATURES, PITFALLS)

### Low Priority (nice to have)

1. **Project scaffold (item 2):** Most complex new feature. Entirely independent of other items. Build last when the platform is stable. The agent-driven approach (not template-based) is recommended. (sources: FEATURES, UX)
2. **Context compaction (item 4):** Highest-risk item. Consensus across PITFALLS, ARCHITECTURE, and UX is that active compaction should be deferred or minimized. Implement lean generation and context budget monitoring only. (sources: FEATURES, ARCHITECTURE, PITFALLS, UX)
3. **Shell injection fix (`execSync` -> `execFileSync`):** Low likelihood but trivial to fix. Can land at any time as a hardening measure. (source: PITFALLS)
4. **Make `writeRegistry()` atomic:** One-line fix (write-temp+rename). Low urgency but prevents a real concurrent-access edge case. (source: OVERSIGHTS)

## Recommended Approach

This milestone is large (19 items, ~21-30 days of effort) and structurally challenging because most items touch the same monolithic CLI file. The primary risk is not any individual item's complexity, but the coordination cost of making 19 changes to interconnected modules without introducing regressions or merge conflicts.

The recommended approach is **foundation-first, features-second**, organized into five waves. Wave 1 addresses prerequisites that block everything else: version synchronization, schema passthrough/versioning fixes, data integrity bugs, the `npm test` script, and test helper extraction. These are small, high-value changes that harden the codebase before structural refactoring begins. Wave 2 is the CLI split with error standardization and argument parsing -- the single biggest structural change, done as one coordinated effort. Wave 3 builds features (solo mode, review decomposition, migration) on the clean foundation. Wave 4 handles new capabilities (scaffold) and cleanup (deprecated skills, documentation). Wave 5 is experimental work (context compaction) that may be deferred.

The critical insight from the research is that the CLI split is both urgent and risky. Every other item in the milestone touches `rapid-tools.cjs`. If the split is done early, all other changes target clean, independent command files with minimal merge conflict risk. If the split is done late, every change lands in the monolith and must be re-extracted. However, the PITFALLS research correctly identifies that data integrity bugs (items 8, 9, 10) must be fixed in the monolith first, because splitting the file before fixing the bugs preserves them in separate modules. The resolution is: fix bugs in the monolith (Wave 1), then split immediately (Wave 2).

The second critical insight is that Zod schema management is a hidden prerequisite for multiple features. Adding `.passthrough()` to state schemas, resolving the `z.literal(1)` versioning barrier, and establishing the `rapidVersion` field must happen before solo mode (which adds config fields), version migration (which stamps versions), and any other feature that touches STATE.json. These schema changes are small individually but collectively gate the entire feature pipeline.

Context compaction (item 4) is the consensus riskiest item. Four of six research sources recommend against building active compaction. The built-in adaptive compaction in Opus 4.6 and Claude Code's native context management handle the hard problem. RAPID should focus on being a good citizen of the context window (shorter prompts, disk-first artifacts, leaner returns) rather than building a parallel compaction system. Recommend implementing context budget monitoring and lean generation practices only, deferring any active compaction module to v3.4.0 or later.

### Suggested Set Boundaries

Based on feature dependencies, module boundaries, and the goal of minimizing merge conflicts between parallel sets:

- **Set A: Foundation Hardening** -- Version synchronization (package.json/plugin.json), Node.js minimum bump, `npm test` script, Zod `.passthrough()` addition, schema versioning strategy, `rapidVersion` field addition, `writeRegistry()` atomicity, shell injection fix. Rationale: all prerequisite changes that touch state-schemas.cjs, setup.sh, package.json, and core.cjs. No overlap with CLI handler code.

- **Set B: Data Integrity Fixes** -- Resume deduplication (item 8), state mutation bypass fix (item 9), MERGE-STATE transaction wrapper (item 10), `onCompromised` lock-loss fix. Rationale: all three bugs live in the CLI handlers and library modules. Fixing them together ensures consistent transaction patterns across all state types. Must complete before CLI split.

- **Set C: CLI Restructuring** -- CLI split (item 15), error standardization (item 11), argument parsing replacement (item 16), stdin validation (item 17), test helper extraction. Rationale: these are the structural refactoring tasks that transform the monolith into `src/commands/`. Natural to do as one coordinated effort since they all modify the same file and establish the patterns for all future command development.

- **Set D: Structural Cleanup** -- Path resolution guard (item 12), review artifact path fix (item 13), agent build detection fix (item 14), deprecated skills handling (item 18), registry naming (item 19). Rationale: mechanical refactoring across library modules and skills. Independent of CLI restructuring -- targets library code and skill files.

- **Set E: Solo Mode** -- Solo mode implementation (item 1) across config, init, worktree, execute, merge, and status modules. Rationale: pervasive feature that touches 8+ skill files and 4+ library modules. Must be its own set to manage the cross-cutting complexity. Depends on Set A (config strategy) and Set C (clean command handlers).

- **Set F: Review Pipeline** -- Review decomposition (item 3), DEFINITION.md bug fix (item 6), discuss-set UX fix (item 7). Rationale: all three touch skill files. Review decomposition is the primary work; the two bug fixes are quick additions that touch adjacent files.

- **Set G: New Commands** -- Version migration (item 5), project scaffold (item 2). Rationale: both are new commands/skills with new library modules. Largely independent of existing code. Scaffold is the most complex; migration depends on schema changes from Set A.

- **Set H: Context Optimization (optional)** -- Context compaction (item 4): lean generation practices, context budget monitoring, CLAUDE.md compaction instructions. Rationale: experimental, highest-risk work. Should only proceed if the lean generation approach proves insufficient during execution of other sets. Consider deferring to v3.4.0.
