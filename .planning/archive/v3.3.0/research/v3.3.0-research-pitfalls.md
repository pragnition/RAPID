# Pitfalls Research -- Developer Experience Milestone (v3.3.0)

## Critical Pitfalls

### Pitfall: Zod 3.25.76 Bundles Zod 4 Internally -- CJS Breakage Risk
- **Category:** compatibility
- **Stack:** Zod 3.25.76 (state validation)
- **Description:** Zod 3.25.x versions bundle Zod 4 code internally alongside v3. This doubles the package size and has caused documented CJS loading failures (GitHub issue #4434) where `require('zod')` produces ESM-cannot-be-required errors. While Zod 3.25.56+ patched the worst CJS breakage, the dual-bundling remains. Any future patch release in the 3.25.x range could re-introduce CJS incompatibility, and the Version Migration feature (item 5) needs to handle this if schema validation code is touched.
- **Trigger:** Running `npm update` or `npm install` resolves to a newer 3.25.x patch with regression; or new code accidentally imports from `zod/v4` subpath.
- **Impact:** Total application failure -- STATE.json cannot be read or written, all RAPID commands fail. No graceful degradation path exists.
- **Prevention:** Pin exact version in package.json (`"zod": "3.25.76"` without caret) or downgrade to `"zod": "~3.24.4"` which is the last stable v3-only release. If staying on 3.25.76, add a startup smoke test that validates `require('zod').z.string().parse('ok')` succeeds.
- **Detection:** Any `require()` error mentioning "ESM module cannot be imported with require" from a zod path. Also detectable by checking if `node_modules/zod/v4/` directory exists (it should not in a clean v3 install).
- **References:** [CJS loading broken #4434](https://github.com/colinhacks/zod/issues/4434), [Zod3 now includes Zod4 #4923](https://github.com/colinhacks/zod/issues/4923), [Cannot find module 'zod/v4' #4372](https://github.com/colinhacks/zod/issues/4372)

### Pitfall: State Mutation Bypass in update-phase Creates Unvalidated Registry Entries
- **Category:** data-loss / correctness
- **Stack:** rapid-tools.cjs (execute update-phase handler, line ~1834)
- **Description:** The `execute update-phase` handler writes directly to the worktree REGISTRY.json via `registryUpdate()` but never touches STATE.json via `withStateTransaction()`. It also creates registry entries for sets that may not have worktrees (line 1852: "Create entry if not present"). This means: (a) a registry entry can exist with no corresponding worktree on disk, (b) the registry phase and STATE.json status can diverge silently, and (c) the fallback creation bypasses all the validation that `createWorktree` → `setInit` normally performs.
- **Trigger:** Calling `execute update-phase` on a set that has no worktree yet, or calling it when STATE.json is in an inconsistent state.
- **Impact:** Phantom registry entries cause subsequent commands (cleanup, merge, resume) to fail with confusing errors. State/registry divergence means `/status` shows incorrect information.
- **Prevention:** When fixing this (item 9), the update-phase handler must: (1) verify the registry entry already exists before updating, (2) optionally cross-check with STATE.json status, (3) never create registry entries as a side effect. Solo Mode (item 1) must also respect this pattern -- skipping worktrees means the registry will be empty, so all code paths that read registry entries need a solo-mode fallback.
- **Detection:** Run `worktree reconcile` and check for entries marked `orphaned`. Compare registry phase values against STATE.json set statuses.
- **References:** todo-explore.md item 2 (State Mutation Bypass)

### Pitfall: Resume Command Duplication -- Two Code Paths With Different Registry Logic
- **Category:** correctness
- **Stack:** rapid-tools.cjs (handleResume ~line 1613, execute resume ~line 1917)
- **Description:** Two independent implementations of resume exist. `handleResume()` (the top-level `resume` command) reads HANDOFF.md and updates the registry phase. `execute resume` (the subcommand) also reads HANDOFF.md but has slightly different registry update logic and validation. When one path is fixed/updated, the other is not, creating behavioral divergence. During the CLI split (item 15), if both paths are migrated independently, the bug is preserved permanently in separate files.
- **Trigger:** Invoking `rapid-tools resume <set>` vs `rapid-tools execute resume <set>` produces different behavior.
- **Impact:** Inconsistent state after resume -- one path may update registry but not STATE.json, or vice versa. Parallel execution after resume may operate on stale context.
- **Prevention:** Fix the duplication (item 8) BEFORE the CLI split (item 15). Route both entry points to a single `resumeSet()` function in `execute.cjs`. The CLI split should import from the same library function, not copy-paste handler code.
- **Detection:** Diff the two code paths. After a resume, verify both REGISTRY.json and STATE.json agree on the set's status.
- **References:** todo-explore.md item 1 (Resume Command Duplication)

## High-Risk Pitfalls

### Pitfall: Solo Mode Skipping Worktrees Breaks All Registry-Dependent Code Paths
- **Category:** correctness
- **Stack:** worktree.cjs, execute.cjs, merge.cjs, rapid-tools.cjs
- **Description:** Solo Mode (item 1) skips worktree creation, meaning REGISTRY.json will have no entry for the set. However, at least 15 code paths in rapid-tools.cjs call `loadRegistry()` and look up `registry.worktrees[setName]` -- getting `undefined` when no worktree exists. Functions like `generateScopedClaudeMd()`, `prepareSetContext()`, `execute prepare-context`, `execute verify`, and the entire merge pipeline assume a registry entry with `branch`, `path`, and `phase` fields.
- **Trigger:** Any lifecycle command after `start-set` in solo mode.
- **Impact:** Crashes or incorrect behavior for every command that touches the registry. In solo mode, `execute prepare-context` fails because `entry.path` is undefined; merge commands fail because `entry.branch` is undefined.
- **Prevention:** Implement a `getSoloEntry(cwd, setName)` helper that returns a synthetic registry-like object with `{ branch: 'main', path: '.', phase: '...', status: 'active' }`. Every `loadRegistry()` consumer must either go through a wrapper that injects solo entries, or check `isSoloMode()` first. Identify all call sites with: `grep -n 'loadRegistry\|registry\.worktrees\|entry\.path\|entry\.branch' src/`.
- **Detection:** Run any lifecycle command in solo mode after implementing -- will crash immediately if not handled.
- **References:** user-todo.md item 1 (Solo Mode)

### Pitfall: CLI Monolith Split Can Break All Test Fixtures and External Callers
- **Category:** compatibility
- **Stack:** rapid-tools.cjs (~2700 lines), all SKILL.md files, all agent .md files
- **Description:** The monolithic CLI split (item 15) changes the internal structure of the entry point that every SKILL.md invokes via `node "${RAPID_TOOLS}" <command>`. While the routing should remain in rapid-tools.cjs, if the split introduces any behavioral change (different error codes, different JSON output format, different stderr handling), it will break all 24 skills and 26 agents that shell out to the CLI. Additionally, rapid-tools.test.cjs has ~400 lines of integration tests that invoke the CLI directly.
- **Trigger:** Moving handler code to `src/commands/*.cjs` and introducing even subtle differences in output format, exit codes, or error handling.
- **Impact:** Silent failures in agent orchestration -- agents parse CLI output and make decisions based on JSON structure. A changed field name or missing newline breaks the parsing chain.
- **Prevention:** (1) Keep rapid-tools.cjs as the single entry point -- only move handler implementations, not the routing. (2) Write a CLI contract test that captures the exact JSON output shape for every command before splitting. (3) Run the full test suite after each handler extraction. (4) Do not change error format (item 4) simultaneously with the split -- do one, stabilize, then the other.
- **Detection:** Run `node src/bin/rapid-tools.cjs <every-command> --help` before and after, diff the outputs. Run the full test suite.
- **References:** todo-explore.md item 8 (Monolithic CLI Split)

### Pitfall: Argument Parser Replacement Can Silently Change Flag Behavior
- **Category:** correctness
- **Stack:** rapid-tools.cjs (10 occurrences of `args.indexOf`)
- **Description:** Replacing manual `args.indexOf('--flag')` with a `parseArgs()` utility (item 16) seems safe but introduces subtle behavior changes. The current code does `args[args.indexOf('--branch') + 1]` which returns `undefined` if `--branch` is the last argument (indexOf returns N, N+1 is out of bounds). A parseArgs utility that supports `--branch=value` syntax would handle this differently. If the replacement changes how missing values are handled (undefined vs empty string vs error), commands that previously "worked by accident" with missing flags will now fail.
- **Trigger:** Any command invocation that relied on the old behavior of silently getting `undefined` for missing flag values.
- **Impact:** Commands that "worked" in specific agents' usage patterns suddenly error, but only in that specific usage context -- hard to reproduce.
- **Prevention:** Before replacing, catalog every `args.indexOf` call site with its current behavior for edge cases (missing value, flag at end, duplicate flags). Write tests that capture each edge case. The new `parseArgs()` must handle missing values identically to the old pattern unless intentionally changing behavior (and updating callers).
- **Detection:** Full integration test suite. Specifically test each command with `--flag` at the end of args with no value.
- **References:** todo-explore.md item 9 (Manual Argument Parsing)

### Pitfall: proper-lockfile onCompromised Swallows Lock Loss Silently
- **Category:** data-loss
- **Stack:** proper-lockfile 4.1.2 (lock.cjs)
- **Description:** The `onCompromised` callback in lock.cjs (line 60) logs to stderr but does NOT abort the operation: `process.stderr.write(...); // Log but don't crash -- let the operation complete`. This means if a lock is compromised (stale detection triggers, another process steals the lock), the current operation continues writing to STATE.json while another process also writes. The atomic rename provides some safety, but two concurrent `withStateTransaction()` calls can now both succeed, with one silently overwriting the other's changes.
- **Trigger:** Two RAPID processes (e.g., parallel set execution) both attempt state mutations within the 5-minute stale window. If one process is slow (agent thinking time), its lock can be stolen.
- **Impact:** Silent state data loss -- one set's status transition is overwritten by another's. No error is surfaced.
- **Prevention:** Change `onCompromised` to set a flag that causes the write phase of `withStateTransaction` to abort with an error rather than silently proceeding. Alternatively, throw inside `onCompromised` to crash the operation immediately. For the context-aware compaction feature (item 4), long-running operations must refresh locks or use shorter stale thresholds.
- **Detection:** Nearly impossible to detect after the fact -- the state looks valid but contains stale data. Can be caught by comparing `lastUpdatedAt` timestamps before and after a transaction and verifying the timestamp changed as expected.
- **References:** [proper-lockfile compromise handling](https://github.com/moxystudio/node-proper-lockfile#oncompromised)

### Pitfall: Zod .parse() Strips Unknown Fields -- Schema Migration Data Loss
- **Category:** data-loss
- **Stack:** Zod 3.25.76, state-schemas.cjs, state-machine.cjs
- **Description:** Zod's default behavior strips unrecognized keys during `.parse()`. The `ProjectState` schema in state-schemas.cjs defines only `version`, `projectName`, `currentMilestone`, `milestones`, `lastUpdatedAt`, and `createdAt`. If the Version Migration command (item 5) adds new fields to STATE.json (e.g., `rapidVersion`, `soloMode`), these fields will be silently stripped by ANY code path that runs `ProjectState.parse(state)` until the schema is also updated. Conversely, if an older RAPID version reads a STATE.json with new fields, it will silently strip them -- the migration command must account for this.
- **Trigger:** Adding a new field to STATE.json without simultaneously updating state-schemas.cjs. Or an older RAPID version reading a STATE.json from a newer version.
- **Impact:** Silent data loss. The new field is written, appears in the file briefly, then is removed on the next `withStateTransaction()` call. The user never sees an error.
- **Prevention:** Use `.passthrough()` on the ProjectState schema: `z.object({...}).passthrough()` to preserve unknown keys. This is essential for forward compatibility. The migration command must validate with passthrough mode to avoid stripping fields it doesn't know about yet. Add a startup check that warns if STATE.json contains keys not in the schema.
- **Detection:** Diff STATE.json before and after any state mutation. Unknown fields disappearing is the symptom.
- **References:** [Zod strip vs passthrough](https://zod.dev/?id=strip)

## Medium-Risk Pitfalls

### Pitfall: git worktree Shell Injection via execSync String Interpolation
- **Category:** security
- **Stack:** worktree.cjs (gitExec function, line 21)
- **Description:** `gitExec()` builds the command string via template literal: `` execSync(`git ${args.join(' ')}`) ``. If any element of `args` contains shell metacharacters (`;`, `|`, `` ` ``, `$()`), arbitrary commands can execute. Set names are validated to disallow whitespace, but not shell metacharacters. A set name like `my-set;rm -rf /` would pass the whitespace check but execute the injected command.
- **Trigger:** A set name or branch name containing shell metacharacters passed through `gitExec()`.
- **Impact:** Arbitrary command execution with the user's permissions.
- **Prevention:** Replace `execSync(\`git ${args.join(' ')}\`)` with `execFileSync('git', args)` which does not invoke a shell and treats all args as literal values. This is a one-line change per call site. The scaffold command (item 2) and solo mode (item 1) introduce new code paths that call `gitExec()` -- all must use the safe variant.
- **Detection:** Static analysis for `execSync` with template literals. grep for `` execSync(` `` in all source files.
- **References:** [Node.js child_process security](https://nodejs.org/api/child_process.html#child_processexecfilesyncfile-args-options)

### Pitfall: Review Pipeline Decomposition Loses Cross-Stage Context
- **Category:** correctness
- **Stack:** review.cjs, SKILL.md files for review pipeline
- **Description:** The current monolithic `/review` passes context between stages (scoper -> unit-tester -> bug-hunter -> devil's-advocate -> judge) within a single conversation. Decomposing into independent skills (item 3) means each stage runs in a fresh context. The bug-hunter will not have the unit-tester's findings in context unless explicitly loaded from disk. The devil's-advocate will not know what the bug-hunter considered and rejected. If cross-stage context is not serialized to disk artifacts between stages, the decomposed pipeline will produce lower-quality results than the monolith.
- **Trigger:** Running `/bug-hunt` after `/unit-test` without an intermediate artifact that captures unit-test findings.
- **Impact:** Degraded review quality -- duplicate findings, missed context, devil's advocate arguing against already-validated findings.
- **Prevention:** Each decomposed stage must write a structured artifact (not free-form markdown) that the next stage reads. Define a shared schema for review findings (file, line, severity, finding, evidence, status). The `/review` skill (scope only) must generate REVIEW-SCOPE.md with enough context that downstream stages can operate independently.
- **Detection:** Compare monolithic review output against decomposed review output on the same codebase. Count unique findings and duplicates.
- **References:** user-todo.md item 3 (Review Pipeline Decomposition)

### Pitfall: Context Compaction Conflicts With Claude Code's Built-in Context Management
- **Category:** correctness / performance
- **Stack:** Claude Code runtime, RAPID orchestration layer
- **Description:** Claude Code already has built-in context window management that summarizes and compacts conversation history when approaching limits. If RAPID implements its own compaction (item 4) that restructures or removes messages, the two systems can fight: RAPID compacts a section, Claude Code's system doesn't know the compaction happened and may re-summarize already-compacted content, or RAPID's compaction removes something that Claude Code's system considers critical context.
- **Trigger:** A long-running execute-set session where RAPID triggers compaction between waves, then Claude Code's built-in system also triggers shortly after.
- **Impact:** Double-compaction causes excessive context loss. Agent output quality degrades silently mid-execution. Subtle: the compaction looks like it worked but critical details are gone.
- **Prevention:** Do not attempt to modify the conversation context directly. Instead, implement compaction as a "context budget" strategy: (1) write completed wave results to disk immediately, (2) instruct agents to read from disk rather than relying on conversation history, (3) keep conversation prompts lean by referencing artifact paths rather than inlining content. This works WITH Claude Code's context management rather than against it.
- **Detection:** Monitor token usage across waves. If usage does not decrease after compaction, the two systems are fighting.
- **References:** user-todo.md item 4 (Context-Aware Internal Compaction)

### Pitfall: path.resolve() vs path.join() During CLI Split Creates Inconsistent Paths
- **Category:** correctness
- **Stack:** rapid-tools.cjs, all handler code
- **Description:** The codebase mixes `path.resolve(cwd, entry.path)` and `path.join(cwd, entry.path)`. These behave identically only when `cwd` is absolute. If `cwd` comes from `process.cwd()` it is always absolute, but if it comes from a registry entry's relative path or a user-provided argument, it may not be. During the CLI split (item 15), if handlers are moved to separate files that receive `cwd` as a parameter, and the parameter is sometimes relative (e.g., from a test fixture), paths will break on `path.join()` but work on `path.resolve()`.
- **Trigger:** Running RAPID from a directory different from the project root, or test fixtures that use relative paths.
- **Impact:** File-not-found errors that only occur in specific invocation contexts. Tests pass but production usage fails (or vice versa).
- **Prevention:** Normalize `cwd` to absolute at the CLI entry point: `cwd = path.resolve(cwd)`. Then use `path.join()` consistently everywhere (it is more explicit about what it does). Fix this (item 12) BEFORE the CLI split.
- **Detection:** Run tests from different working directories. Check if any test uses a relative cwd.
- **References:** todo-explore.md item 5 (Path Resolution Inconsistency)

### Pitfall: Deprecated Skills Removal Breaks User Muscle Memory and Existing Scripts
- **Category:** compatibility
- **Stack:** skills/ directory (7 deprecated skills)
- **Description:** Removing deprecated skills entirely (item 18) means users who type `/discuss` (old name) or `/execute` (old name) get a generic "skill not found" error with no guidance on the new name. If users have external scripts or documentation referencing old names, they break without explanation.
- **Trigger:** Any user invoking a removed skill name.
- **Impact:** User confusion, broken external automation, support burden.
- **Prevention:** Instead of removing deprecated skills entirely, replace their SKILL.md content with a single line that says "This command has been renamed to /discuss-set. Please use the new name." This is a 1-line file, zero maintenance, and provides actionable guidance. Only remove after 2+ major versions.

- **References:** todo-explore.md item 11 (Deprecated Skills Cleanup)

### Pitfall: Stdin JSON Validation With Zod Rejects Valid Input Due to Strict Mode
- **Category:** correctness
- **Stack:** Zod, rapid-tools.cjs stdin reading (5 call sites)
- **Description:** Adding Zod validation to stdin JSON (item 17) using `.parse()` will strip unknown fields (Zod's default) and reject input with extra keys if `.strict()` is used. Agents generate JSON output that may include extra diagnostic fields not in the schema. If stdin validation is too strict, it breaks agent -> CLI communication. If too loose (passthrough), it provides no real validation.
- **Trigger:** An agent sends JSON with an extra field (e.g., `confidence`, `reasoning`) that the schema does not define.
- **Impact:** Commands fail with cryptic Zod validation errors when invoked by agents that include extra context.
- **Prevention:** Use `.passthrough()` mode for stdin validation schemas. Validate that required fields exist and have correct types, but do not reject unknown fields. Document the expected schema for each stdin-reading command.
- **Detection:** Run each stdin-reading command with extra fields in the JSON payload. Verify it succeeds.

## Anti-Patterns for This Stack

### Anti-Pattern: Using execSync With Template Literals for Git Commands
- **What people do:** `` execSync(`git ${args.join(' ')}`) `` -- concatenates user-provided values into a shell command string.
- **Why it fails:** Shell metacharacters in arguments (set names, branch names) can execute arbitrary commands. Even without malicious intent, special characters in project names (`my-project (copy)`) will cause parse errors.
- **Correct approach:** Use `execFileSync('git', args, { cwd, encoding: 'utf-8' })` which bypasses the shell entirely and treats arguments as literal values.
- **Example:** Replace `execSync(\`git checkout ${branch}\`)` with `execFileSync('git', ['checkout', branch])`.

### Anti-Pattern: Reading Registry Without Null-Checking in a Multi-Mode System
- **What people do:** `const entry = registry.worktrees[setName]; doSomething(entry.path)` -- assumes entry always exists.
- **Why it fails:** In solo mode, no registry entries exist. In team mode after cleanup, entries may be removed. The `undefined` propagates into path operations, causing cryptic `Cannot read property 'path' of undefined` errors.
- **Correct approach:** Always validate: `const entry = registry.worktrees[setName]; if (!entry) { /* handle missing */ }`. For solo mode, provide a synthetic entry or use a `getSetEntry(cwd, setName)` abstraction that handles both modes.

### Anti-Pattern: Fixing Multiple Coupled Issues in the Same PR
- **What people do:** Fix the CLI split (item 15), argument parsing (item 16), and error standardization (item 4) simultaneously because they all touch rapid-tools.cjs.
- **Why it fails:** Each change has a large blast radius. Combined, they make it impossible to isolate which change caused a regression. Merge conflicts between parallel sets are maximized because every set touches the same file.
- **Correct approach:** Strict ordering: (1) Fix data-integrity bugs (items 8, 9, 10) first, (2) split the CLI, (3) then refactor individual command files. Each step should be independently testable and deployable.

### Anti-Pattern: Implementing Schema Migration Without Forward Compatibility
- **What people do:** Add new required fields to STATE.json schema, deploy, and expect the migration command to handle everything.
- **Why it fails:** An older RAPID version reading a STATE.json with new fields will strip them via `ProjectState.parse()`. Users who downgrade or use different RAPID versions across machines will silently lose state data.
- **Correct approach:** (1) Use `.passthrough()` on all Zod schemas to preserve unknown keys. (2) New fields should always have defaults (`z.string().default('')`). (3) The `version` field in STATE.json must be checked before any mutation, and mutations must refuse to operate on newer-version state files.

## Version-Specific Gotchas

- **Zod 3.25.76**: `.parse()` strips unknown keys by default -- use `.passthrough()` for forward compatibility. This is a v3 behavior, not new, but becomes critical when adding migration features.
- **Zod 3.25.76**: TypeScript type inference may break if `tsconfig` targets ES5/ES2015 due to bundled v4 code using `const` type parameters. Not relevant for this project (pure JS/CJS) but blocks adding TypeScript in the future.
- **Node.js 25.8.0**: `module.parent` is deprecated; code should use `require.main` instead. No impact on current RAPID code but relevant if adding module detection logic.
- **Node.js 25.8.0**: `node:test` mock timers are stable but mock module loaders remain experimental. Complex mocking of CJS modules requires dependency injection patterns rather than `mock.module()`.
- **proper-lockfile 4.1.2**: The `realpath: false` option (used in lock.cjs) disables symlink resolution. If `.planning/` is a symlink (e.g., in some CI setups), locks will target the symlink, not the real path, allowing two processes to lock different files thinking they hold the same lock.
- **Git 2.30+**: `git worktree remove` does not automatically delete the branch. The branch must be deleted separately with `git branch -d`. If `worktree cleanup` is called but branch deletion fails (e.g., branch has unmerged changes), orphan branches accumulate. The cleanup must handle this explicitly.

## Security Pitfalls

1. **Shell injection via gitExec()**: Set names containing shell metacharacters execute arbitrary commands -- replace `execSync` with `execFileSync` across all git command invocations.
2. **Stdin JSON injection**: Five stdin-reading code paths use `JSON.parse()` without schema validation. While agents are trusted callers, a malformed JSON payload (e.g., deeply nested objects) can cause stack overflow or excessive memory consumption. Add Zod validation with depth limits.
3. **Lock file PID spoofing**: `cleanStaleLocks()` reads PID from lock target files and checks `process.kill(pid, 0)`. If a different process reuses the PID, the stale lock is not cleaned. Low risk in practice but can cause lock deadlocks on long-running CI systems where PID space wraps quickly.
4. **Atomic write race on crash**: `writeState()` writes to `STATE.json.tmp` then renames. If the process crashes between `writeFileSync` and `renameSync`, the `.tmp` file remains. The next `readState()` reads the old `STATE.json`, but the `.tmp` file contains newer data that is lost. The migration command should check for and recover `.tmp` files.

## Performance Traps

1. **Synchronous file operations in hot paths**: All state reads/writes use `fs.readFileSync` and `fs.writeFileSync`. For single-user CLI usage this is fine, but if context compaction (item 4) triggers frequent state reads during execution, synchronous I/O blocks the event loop. Consider async I/O for any high-frequency read paths.
2. **Full state reparse on every mutation**: `withStateTransaction()` reads, parses, validates the full ProjectState on every mutation. For projects with many milestones and sets, this becomes expensive. The migration command (item 5) may process state many times during migration steps. Consider caching the parsed state within a single transaction.
3. **loadRegistry() is called without caching**: Multiple handlers call `loadRegistry()` independently within the same command invocation, re-reading and re-parsing REGISTRY.json each time. During the CLI split, this can worsen if each command module independently loads the registry.
4. **Agent build iterates all files on every invocation**: `build-agents` reads every module file, assembles every agent, and writes all 26 output files even if only one module changed. For the agent build comment-based detection fix (item 14), consider incremental builds based on file modification timestamps.

## Summary Priority List

1. **Solo Mode registry assumption** (HIGH likelihood x HIGH impact) -- Every registry-consuming code path will crash without null checks; this is the most pervasive risk in the milestone.
2. **Zod field stripping in migration** (HIGH likelihood x HIGH impact) -- Adding fields to STATE.json without `.passthrough()` guarantees silent data loss on the next state mutation.
3. **Resume command duplication before CLI split** (HIGH likelihood x MEDIUM impact) -- If not fixed first, the split preserves the bug in two separate files, making it harder to fix later.
4. **CLI split breaking agent communication** (MEDIUM likelihood x HIGH impact) -- Any JSON output format change breaks 24 skills and 26 agents silently.
5. **onCompromised lock loss swallowed** (LOW likelihood x CRITICAL impact) -- Rare but causes undetectable state corruption when it occurs.
6. **Argument parser edge case divergence** (MEDIUM likelihood x MEDIUM impact) -- Subtle behavioral changes affect specific agent usage patterns.
7. **Review decomposition context loss** (HIGH likelihood x MEDIUM impact) -- Decomposed stages produce worse results without explicit artifact serialization.
8. **Shell injection via gitExec** (LOW likelihood x HIGH impact) -- Agent-generated set names are usually safe, but the vector exists and is trivial to close.
9. **Context compaction vs Claude Code double-compaction** (MEDIUM likelihood x MEDIUM impact) -- Two compaction systems fighting causes unpredictable quality degradation.
10. **Deprecated skill removal without redirect** (HIGH likelihood x LOW impact) -- Users hit confusing errors; easy to prevent with 1-line redirect files.
