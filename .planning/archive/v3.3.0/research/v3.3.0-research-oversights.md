# Oversights Research -- Developer Experience Milestone (v3.3.0)

## Cross-Cutting Concerns

### 1. Solo Mode Pervasive Branching Through Lifecycle

Solo mode (item #1) is the most architecturally pervasive feature in this milestone. It is not a flag that can be checked in one place -- it fundamentally alters the control flow in at least 8 different skill files and 4 library modules.

**Commonly overlooked touch points:**
- **Registry semantics change:** In solo mode, `REGISTRY.json` still needs entries to track set phase/status, but `path` and `branch` fields become meaningless since there is no worktree. The `loadRegistry()` / `registryUpdate()` APIs assume every entry has a valid `path`. Code that calls `path.resolve(cwd, entry.path)` (e.g., `execute.cjs:488`) will construct a path to a non-existent worktree directory.
- **`generateScopedClaudeMd()`** in `worktree.cjs` references worktree paths to scope the CLAUDE.md. In solo mode this must fall back to the project root, but it needs to avoid scoping to files from other sets (which are all on the same branch now).
- **Git operations:** `worktree.cjs:gitExec()` passes `cwd` to `execSync`. In team mode, `cwd` is the worktree path. In solo mode, `cwd` must be the project root, but git commands that assume branch isolation (e.g., `git diff` between set branch and main) become nonsensical.
- **`execute prepare-context`** reads from worktree path. Solo mode must redirect to project root.
- **`merge execute`** does `git merge --no-ff rapid/{setName}`. Solo mode has no branches to merge. This must no-op cleanly, but the state machine still needs to transition the set to `merged`.
- **Status display:** `worktree status` and the `/status` skill need a distinct rendering for solo mode. The "worktree" column in status tables makes no sense when there are no worktrees.

**What gets missed:** The flag needs to be propagated through both the skill layer (SKILL.md files read by Claude) AND the CLI layer (rapid-tools.cjs commands). If only one layer is updated, behavior diverges silently. The config must be stored where both layers can read it -- likely `config.json` at project root and/or a `soloMode` field in `STATE.json`.

**Switching from solo to team mid-project:** This is explicitly called out in user-todo.md as an edge case, but the solution is non-trivial. Retroactively creating worktrees for sets that already have commits on main requires cherry-picking or branch creation from specific commit points. Consider forbidding this transition (solo -> team is destructive) or requiring a clean milestone boundary.

### 2. CLI Split Coordination with Test Infrastructure

The monolithic CLI split (item #15) into `src/commands/` is the highest-coordination task because every other fix in this milestone touches `rapid-tools.cjs`. If the split happens first, all other changes target new files. If it happens last, all other changes land in the monolith and then get split out, doubling the work.

**Overlooked concerns:**
- **Test file co-location:** The existing test file `rapid-tools.test.cjs` is 990+ lines testing the monolith. Splitting the CLI means splitting the test file too. Each `src/commands/{command}.cjs` needs a co-located `{command}.test.cjs`. The current test file tests CLI integration (spawns `node rapid-tools.cjs`), which still works after the split, but unit tests for individual command handlers need new homes.
- **Shared test helpers:** `rapid-tools.test.cjs` creates test repos with `git init`, STATE.json, REGISTRY.json, etc. These helpers are duplicated per test block. A CLI split is a natural point to extract `src/test-helpers.cjs` -- otherwise each command test file re-invents the same fixture creation.
- **Module require chains:** `rapid-tools.cjs` currently does lazy requires (`const fs = require('fs')` inside handler functions). When split into separate files, each command file needs its own imports. The current pattern of requiring `../lib/worktree.cjs` etc. from `src/bin/` must become `../lib/worktree.cjs` from `src/commands/`, changing the relative path.
- **No `npm test` script:** package.json has zero scripts. There is no `test` script. Tests are run ad-hoc with `node --test src/lib/*.test.cjs`. After a CLI split, the test surface area grows. The absence of a `test` script means there is no single command to verify nothing is broken. This is the time to add `"test": "node --test src/**/*.test.cjs"` to package.json.

### 3. State Schema Versioning Gap

The current `ProjectState` schema uses `version: z.literal(1)`. The `/migrate` command (item #5) needs a way to detect schema version, run migrations, and stamp new versions. But:

- **`z.literal(1)` is a strict barrier:** If `/migrate` bumps the version to 2, all existing `readState()` calls will fail Zod validation because `z.literal(1)` rejects anything that is not exactly `1`. The migration system must update the schema to accept the new version, but that means the schema definition itself becomes migration-aware. Either `version` changes to `z.number().min(1)` (losing strictness) or the schema is versioned alongside the state.
- **Dual state files:** There are *two* state files -- `STATE.json` (Zod-validated via `state-schemas.cjs`) and `STATE.md` (legacy, YAML frontmatter with `rapid_state_version`). The `migrateStateVersion()` function at line 99 of `rapid-tools.cjs` already handles a `gsd_state_version` -> `rapid_state_version` rename in STATE.md, but nobody validates STATE.md's version against STATE.json's version. These can drift.
- **No `rapidVersion` in STATE.json:** The user-todo mentions detecting version "from STATE.json schema shape, artifact directory structure, or an explicit `rapidVersion` field." Currently STATE.json has no `rapidVersion` field. Detecting version from schema shape is fragile. The migration system's first step should be stamping a `rapidVersion` field.
- **MergeStateSchema also has implicit version:** `merge.cjs` defines `MergeStateSchema` with its own structure. If MERGE-STATE evolves between RAPID versions, the migration system must handle it too -- but it is not in `state-schemas.cjs`, it is in `merge.cjs`. The migration system needs to know about both schemas.

### 4. Error Handling Strategy During Refactor

Three items touch error handling: Error Response Format Standardization (#11), the `exitWithError()` helper, and the overall pattern of 128 `process.exit(1)` calls in the CLI. The overlooked concern is **error handling in library code vs CLI code.**

**Current confusion:**
- Library modules (src/lib/*.cjs) throw errors. They never call `process.exit()`.
- The CLI (rapid-tools.cjs) catches those errors and calls `process.exit(1)`.
- But some CLI handlers have try/catch blocks that swallow errors and call `error()` + `process.exit(1)`, while others let errors propagate to the main() catch handler.
- The new `exitWithError()` helper addresses CLI-level standardization, but does not solve the problem of library functions that return inconsistent error shapes (some throw Error objects, some return `{ valid: false, errors: [...] }`, some return null).

**What gets missed in a CLI split:**
- If command files call `process.exit()`, they become untestable in-process (you cannot assert behavior after exit).
- The better pattern is: command handlers throw typed errors, and the router in `rapid-tools.cjs` catches and calls `process.exit()` in one place. This must be designed before the split, not after.
- The `exitWithError(msg, code)` helper must live in `core.cjs` (shared by all commands), not in the CLI router.

### 5. Backward Compatibility of Review Pipeline Decomposition

The review decomposition (item #3) splits `/review` into `/review`, `/unit-test`, `/bug-hunt`, `/uat`. This creates a backward-compatibility gap:

- **Existing review artifacts:** Projects mid-review have `REVIEW.md` files written by the old monolithic `/review`. The new `/review` (scope-only) writes `REVIEW-SCOPE.md`. If a user updates RAPID mid-project, old REVIEW.md files sit alongside new REVIEW-SCOPE.md files, creating confusion.
- **Agent references:** The existing agents `rapid-scoper`, `rapid-unit-tester`, `rapid-bug-hunter`, `rapid-devils-advocate`, `rapid-judge`, `rapid-bugfix`, `rapid-uat` already exist. But they are currently spawned by the monolithic `/review` skill. After decomposition, each new skill needs to know which agents to spawn. The agent definitions themselves may not need changes, but the spawning logic is embedded in SKILL.md files.
- **Status tracking:** The review pipeline currently tracks progress in the SKILL.md conversation (implicit). The decomposed pipeline needs explicit state: "has this set been scoped? unit tested? bug hunted?" Without explicit state, there is no way to prevent a user from running `/uat` before `/review` except by checking for `REVIEW-SCOPE.md` file existence.
- **Documentation:** `docs/review.md` describes the monolithic pipeline. After decomposition, it must explain the new composable workflow. Users who read old docs will be confused by the new commands.

### 6. Stdin JSON Validation Cascade

Item #17 (Missing Stdin JSON Validation) intersects with several other items. Currently 5+ commands read JSON from stdin without schema validation: `plan create-set`, `plan decompose`, `plan write-dag`, `state add-milestone`, `execute pause`, `review log-issue`.

**What gets missed:**
- The `readAndValidateStdin(zodSchema)` utility needs to handle the case where stdin is a TTY (interactive terminal). Currently, some commands check `!process.stdin.isTTY` (line 391 of rapid-tools.cjs) while others just call `fs.readFileSync(0, 'utf-8')` which blocks forever if stdin is a terminal. The utility must unify this behavior.
- If the CLI is split into commands, each command file needs access to this utility. It should live in `core.cjs` or a new `src/lib/cli-utils.cjs`.
- The Zod schemas for stdin validation do not exist yet for most commands. `plan create-set` accepts freeform JSON. Defining schemas for all stdin-reading commands is additional scope.

### 7. Argument Parsing Migration Risk

Item #16 (Manual Argument Parsing) proposes a `parseArgs(args, schema)` helper. What gets overlooked:

- **Node.js 18.11+ has `util.parseArgs()`** built-in. Writing a custom helper duplicates built-in functionality. However, `util.parseArgs()` has limitations (no `--flag=value` shorthand in early versions). The decision between custom vs built-in must be made explicitly.
- **Every handler's argument positions change:** The current `args[0]`, `args[1]` pattern has implicit contracts. `handleExecute(cwd, subcommand, args.slice(2))` means the execute handler receives args with the subcommand already stripped. A `parseArgs()` call needs to know this stripping already happened. If the stripping changes during the CLI split, argument positions shift.
- **Flag ordering assumptions:** Current code like `const branch = args[args.indexOf('--branch') + 1]` fails silently if `--branch` is the last argument (returns `undefined`). This is the kind of edge case that must be tested in the new parser. The migration must include test cases for every existing flag across all commands.

### 8. Deprecated Skills Cleanup and User Migration

Item #18 (Deprecated Skills Cleanup) removes 7 deprecated skills: `discuss`, `execute`, `plan`, `set-init`, `wave-plan`, `new-milestone`, and one more. These currently have `disable-model-invocation: true` and redirect users.

**What gets missed:**
- Users with muscle memory for `/rapid:execute` or `/rapid:plan` will get "skill not found" errors with no guidance on the new name. The error message is generic Claude Code behavior, not a RAPID-controlled message.
- The `docs/*.md` files may still reference old skill names. A grep for old skill names across all docs/ and CLAUDE.md files is needed.
- Some agent prompts in `src/modules/roles/` may reference old skill names in their instructions (e.g., "invoke `/execute` to...").
- External user documentation, blog posts, or READMEs outside the repo may reference old names. This is beyond RAPID's control but worth noting in release notes.

### 9. Concurrent Registry Updates in CLI Split

The CLI split puts command handlers in separate files, but the concurrency model stays the same: multiple Claude Code sessions can invoke different commands simultaneously. The locking strategy must be reviewed:

- **Lock granularity:** Currently, `state` and `worktree-registry` are separate lock names. After the CLI split, the same locks are acquired from different files. This works, but the lock timeout (300 seconds / 5 minutes) may be too long for quick status queries that only need read access.
- **Read-only operations:** Commands like `state get`, `worktree list`, `worktree status`, `merge status` do NOT acquire locks. They read files directly. This is fine for eventual consistency, but during a concurrent mutation, a reader can see partially-written JSON. The `writeState()` function uses atomic writes (write-to-temp + rename), which is safe on most filesystems, but `writeRegistry()` does a direct `fs.writeFileSync()` which is NOT atomic.
- **Lock contention during execute-set:** When execute-set runs waves in parallel, multiple agents call `execute update-phase` concurrently. Each call acquires the `worktree-registry` lock. With the current retry config (10 retries, 100-2000ms exponential backoff), agents could wait up to ~20 seconds for a registry update that takes <10ms. This is acceptable but not ideal.

### 10. Testing Strategy Gaps

The codebase has 21 test files with co-located tests, but several concerns are untested or under-tested:

- **No integration tests:** All tests are unit tests that mock or stub git operations. There are no tests that create a real git repo, run the full lifecycle (init -> plan -> execute -> merge), and verify end-to-end correctness. This means the solo mode implementation could pass all unit tests while being fundamentally broken in practice.
- **No test runner script:** `package.json` has no `"test"` script. Tests are discovered by running `node --test src/lib/*.test.cjs`. After the CLI split adds `src/commands/*.test.cjs`, the glob pattern must be updated. Adding a `"test"` script is cheap and prevents "which files do I test?" confusion.
- **Test cleanup:** Tests create temp directories via `os.tmpdir()` and attempt cleanup in `afterEach`. But if a test throws before `afterEach`, temp directories leak. This is not a problem in CI (ephemeral runners) but slowly fills `/tmp` on developer machines during development.
- **Snapshot testing absence:** Agent prompt assembly (`build-agents`) produces large markdown strings. Changes to core modules silently alter all 26 agent prompts. There are no snapshot tests to catch unintended prompt changes. The `build-agents.test.cjs` tests structure but not content.

### 11. Configuration Management for New Features

Multiple items in this milestone introduce new configuration:

- Solo mode needs `soloMode: true/false` persisted somewhere
- Migration needs `rapidVersion` in STATE.json
- Review decomposition may need review stage tracking state
- Context compaction needs threshold configuration

**What gets missed:**
- **`config.json` vs `STATE.json` vs `.planning/config.json`:** There are already three configuration locations: `config.json` at project root (lock timeout, agent size warning), `.planning/STATE.json` (project state), and potentially `.planning/config.json` (planning config). Adding solo mode to one but not the other creates confusion about which file to read.
- **Schema validation for config.json:** `STATE.json` has Zod validation. `config.json` has NO schema validation -- `loadConfig()` in `core.cjs` just parses JSON and merges with defaults. New config fields from this milestone should be validated.
- **Per-set vs per-project config:** Solo mode is described as both a project-wide setting (configured during `/init`) and a per-set override (`--solo` flag). This creates precedence questions: does the per-set flag override the project config? Where is the per-set solo flag stored? In STATE.json's set object? In REGISTRY.json?

### 12. Documentation Needs Post-Refactor

This milestone changes the user-facing command surface significantly:

- `/review` becomes 4 commands (`/review`, `/unit-test`, `/bug-hunt`, `/uat`)
- `/scaffold` is a new command
- `/migrate` is a new command
- Solo mode changes the behavior of `/init`, `/start-set`, `/execute-set`, `/merge`
- 7 deprecated commands are removed

**What gets missed:**
- **`docs/review.md`** must be rewritten for the decomposed pipeline
- **`docs/setup.md`** must mention the solo mode prompt during `/init`
- **`docs/troubleshooting.md`** needs entries for common solo mode confusion
- **Help text in USAGE string** (lines 7-91 of rapid-tools.cjs) must be updated for all new commands. This string is 85 lines long and manually maintained.
- **Agent prompts:** The `rapid-executor` agent prompt references "the orchestrator will manage worktree..." -- this language is wrong in solo mode. Agent prompts need conditional sections or solo-mode-aware wording.
- **CLAUDE.md project instructions** mention "26 specialized agents" -- after deprecated cleanup and possible new agents for decomposed review, this count changes.

### 13. Resume Command Consolidation Side Effects

Item #8 (Resume Command Duplication) seems straightforward -- merge two code paths into one. But:

- **`handleResume()` (line 1613)** reads STATE.json for wave/job progress context. **`execute resume` (line 1917)** only checks registry phase and loads HANDOFF.md. They provide different data to the caller. The consolidated function must include both behaviors.
- **The `--info-only` flag** exists on `handleResume()` but not on `execute resume`. Skills that call `execute resume` may not expect `--info-only` support. If the consolidated function always returns info, skill logic may break.
- **Entry point routing:** `resume` is a top-level command AND an `execute` subcommand. After consolidation, both routes must dispatch to the same function. If the CLI split puts `execute` in `src/commands/execute.cjs` and `resume` in `src/commands/resume.cjs`, the shared implementation must be in a library module (`execute.cjs` already has `parseHandoff()`).

### 14. Path Resolution Audit Scope

Item #12 (Path Resolution Inconsistency) says "pick `path.resolve()` everywhere." But the fix scope is larger than it appears:

- There are **650 total occurrences** of `path.resolve` or `path.join` across all source and test files. Auditing all of them is a significant effort.
- **Test files use `path.join` for temp directories** -- changing these to `path.resolve` would work but is unnecessary churn.
- **Registry stores relative paths** (e.g., `.rapid-worktrees/set-name`). These are intentionally relative so the project can be moved. Using `path.resolve(cwd, relativePath)` is correct when resolving them, but `path.join(cwd, relativePath)` is also correct when `cwd` is absolute (which it always is, since `findProjectRoot()` returns `path.resolve()`). The actual risk is only when `cwd` is somehow relative, which should never happen.
- **The real fix is defensive:** Add a guard in `findProjectRoot()` that asserts the returned path is absolute, then document that all library functions receive absolute `cwd`. This makes `join` vs `resolve` moot.

### 15. Build-Agents Reliability During Concurrent Development

Item #14 (Agent Build Comment-Based Detection) intersects with the CLI split and with concurrent set development:

- **`build-agents` is called during `setup.sh`** and can be called manually. It reads from `src/modules/` and writes to `agents/`. If two developers run setup simultaneously, they could race on writing agent files.
- **The `SKIP_GENERATION` list** currently skips core agents (`planner`, `executor`, `merger`, `reviewer`). If the list gets out of sync with actual hand-written agents, `build-agents` could overwrite a hand-written agent with a generated one. This is the "fragile detection" problem the item describes.
- **Agent files are committed to git.** If `build-agents` changes an agent's content (e.g., due to a module edit), the generated file change shows up in `git status` as a dirty file in the main repo. During RAPID execution (where sets run in worktrees), an unexpected dirty file in the main repo can confuse merge detection.

## Recommendations

Ordered by urgency -- what to include in the roadmap early:

1. **Add `npm test` script immediately:** Before any refactoring, add `"test": "node --test src/**/*.test.cjs"` to package.json. This takes 30 seconds and provides a safety net for all subsequent changes. Also consider `"test:watch"` for development feedback.

2. **Split the CLI first (item #15):** This is the prerequisite for clean parallel development of all other items. Extract command handlers into `src/commands/`, keep `rapid-tools.cjs` as a thin router with centralized error handling. Design the error flow (throw from commands, catch+exit in router) before the split.

3. **Define configuration strategy before solo mode:** Decide where solo mode lives (STATE.json `soloMode` field? config.json? both?) and add Zod validation for config.json. Then implement solo mode with the config location settled. Add the `rapidVersion` field to STATE.json at the same time -- both are schema changes.

4. **Resolve state schema versioning before `/migrate`:** Change `version: z.literal(1)` to a versioned approach (e.g., `z.union([v1Schema, v2Schema])` or `version: z.number().min(1)` with runtime schema selection). Without this, `/migrate` cannot stamp a new version without breaking all existing readState() calls.

5. **Make `writeRegistry()` atomic:** One-line fix -- write to temp file, then `fs.renameSync()`. Prevents corrupt reads during concurrent access. Do this before any feature that increases concurrent registry access (like solo mode removing worktree isolation).

6. **Design review artifact migration before decomposition:** Before splitting `/review`, define where old REVIEW.md files go and what the new file layout looks like (REVIEW-SCOPE.md, REVIEW-UNIT.md, REVIEW-BUGS.md, REVIEW-UAT.md). Add the file-existence check for REVIEW-SCOPE.md into the utility library, not just the skill files.

7. **Extract test helpers:** Before splitting test files alongside the CLI split, create `src/lib/test-helpers.cjs` with shared functions for creating temp git repos, writing test STATE.json, writing test REGISTRY.json, etc. This prevents copy-paste of fixture code across the new command-level test files.

8. **Audit skill files for old command names:** Before removing deprecated skills (item #18), grep all `src/modules/roles/*.md`, `skills/*/SKILL.md`, `docs/*.md`, and `CLAUDE.md` for references to the old names (`/execute`, `/plan`, `/discuss`, `/set-init`, `/wave-plan`, `/new-milestone`). Fix references first, then delete.

9. **Document the solo mode behavioral contract:** Before implementing, write a behavioral spec: "In solo mode, the following commands behave differently: ..." with a table of command + team behavior + solo behavior. This prevents partial implementation where some commands are solo-aware and others are not.

10. **Plan for context compaction cautiously (#4):** This is the riskiest feature. It fights Claude Code's built-in context management, requires understanding future agent needs, and subtle bugs silently degrade quality. Consider deferring to a later milestone or starting with the simpler approach (shorter agent prompts, disk-based context recovery) rather than active compaction.
