# Architecture Research: v4.4.0 Polish & Documentation

## Current Architecture

### Existing Architectural Pattern and Strengths

RAPID follows a layered modular monolith: Skill Layer -> Agent Layer -> CLI Layer -> Library Layer -> Storage Layer. This is well-suited for a Claude Code plugin because the entire system runs as a single process (or spawned subprocesses) and does not need network-based service decomposition.

**Strengths relevant to this milestone:**
- The CLI layer (`rapid-tools.cjs`) provides a clean command router that dispatches to library modules. Adding new subcommands (e.g., `review pipeline-state`) requires only a new handler function and a dispatch case.
- The library layer (`src/lib/`) keeps business logic separate from command parsing. The `review.cjs` module already owns Zod schemas for review issues and scoper output, making it the natural home for review pipeline state.
- The display module (`src/lib/display.cjs`) centralizes all ANSI color handling in a single 99-line file with two lookup tables (`STAGE_BG`, `STAGE_VERBS`). Color changes are isolated to this one file plus `src/commands/build-agents.cjs` for agent frontmatter colors.
- Documentation lives in `docs/` (9 files) and `DOCS.md` (consolidated reference). The structure is topic-based, matching the canonical workflow stages.

**Strengths preserved:**
- Zod validation at every mutation boundary ensures schema changes propagate failures immediately.
- `.passthrough()` on all state schemas allows adding new fields without breaking existing data (critical for review state addition).
- The file-based artifact pattern (REVIEW-SCOPE.md, REVIEW-ISSUES.json, REVIEW-BUGS.md, etc.) is already established and works for inter-skill communication.

### Areas Needing Extension

1. **No review pipeline state tracking.** Currently, each review skill (review, unit-test, bug-hunt, uat) operates independently. There is no persistent record of which stages have completed. If a user runs `/rapid:bug-hunt` and then `/rapid:uat`, the UAT skill has no way to know bug-hunt already ran. Each skill redundantly asks "have you done X first?"

2. **Agent colors are hardcoded bright variants.** The `ROLE_COLORS` map in `build-agents.cjs` uses Claude Code's color names (`blue`, `green`, `red`, `yellow`, `purple`, `cyan`). These render as bright/vivid colors that are hard to read on dark terminals with white text. The display module's `STAGE_BG` uses bright ANSI backgrounds (`\x1b[10Xm` series) which have the same readability issue.

3. **Documentation is incomplete for open-sourcing.** The `docs/` directory covers individual topics but lacks a comprehensive getting-started guide covering the full canonical workflow. The 10 existing files are: setup, planning, execution, agents, review, merge-and-cleanup, troubleshooting, state-machines, configuration, and CHANGELOG.

## Recommended Architecture Pattern

- **Pattern:** Extend the existing layered modular monolith with a new `review-state` data module
- **Rationale:** The project already has a well-established pattern for state tracking (Zod-validated JSON files in `.planning/`). Adding review pipeline state follows the same pattern rather than introducing a new architectural concept. No structural change is needed -- only a new data schema and persistence path.
- **Evidence:** The merge pipeline already solved the same problem with `MERGE-STATE.json` -- a per-set JSON file tracking which merge stages completed. The review pipeline should follow an identical approach.

## Module Organization

### Module: Review Pipeline State (extension of review.cjs)

- **Responsibility:** Track which review stages (scope, unit-test, bug-hunt, uat) have completed for each set, with timestamps and summary metadata
- **Public interface:**
  - `readReviewPipelineState(cwd, setId, postMerge?)` -- Read current pipeline state
  - `writeReviewPipelineState(cwd, setId, stageId, result, postMerge?)` -- Record a stage completion
  - `isReviewStageComplete(cwd, setId, stageId, postMerge?)` -- Check if a specific stage is done
  - `ReviewPipelineStateSchema` -- Zod schema for validation
- **Dependencies:** `fs`, `path`, `zod` (no dependency on state-machine.cjs -- review state is independent of set lifecycle state)
- **Data ownership:** Owns `REVIEW-PIPELINE-STATE.json` per set directory

### Module: Display (modification of display.cjs)

- **Responsibility:** Render branded ANSI banners and stage indicators
- **Public interface:** `renderBanner(stage, target)`, `STAGE_BG`, `STAGE_VERBS` (unchanged API)
- **Dependencies:** None (zero-dependency module)
- **Data ownership:** Owns ANSI color mapping tables

### Module: Build Agents (modification of build-agents.cjs)

- **Responsibility:** Generate agent .md files from role modules with frontmatter
- **Public interface:** `ROLE_COLORS` map (values change, keys unchanged)
- **Dependencies:** `fs`, `path`
- **Data ownership:** Owns agent generation pipeline and color/tool/description maps

### Module: Documentation (new file + modifications)

- **Responsibility:** Comprehensive user-facing documentation for open-source consumption
- **Public interface:** Markdown files in `docs/`
- **Dependencies:** References skills, commands, and workflow concepts
- **Data ownership:** Owns all user-facing documentation content

## Data Flow

### Review Pipeline State Lifecycle

```
/rapid:review <set-id>
  -> writes REVIEW-SCOPE.md
  -> writes REVIEW-PIPELINE-STATE.json { scope: { completed: true, at: "..." } }

/rapid:unit-test <set-id>
  -> reads REVIEW-PIPELINE-STATE.json (checks scope is done)
  -> runs unit tests
  -> updates REVIEW-PIPELINE-STATE.json { unitTest: { completed: true, at: "...", summary: {...} } }

/rapid:bug-hunt <set-id>
  -> reads REVIEW-PIPELINE-STATE.json (checks scope is done)
  -> runs adversarial pipeline
  -> updates REVIEW-PIPELINE-STATE.json { bugHunt: { completed: true, at: "...", cycles: N, summary: {...} } }

/rapid:uat <set-id>
  -> reads REVIEW-PIPELINE-STATE.json (checks scope + bug-hunt are done, or warns)
  -> runs UAT
  -> updates REVIEW-PIPELINE-STATE.json { uat: { completed: true, at: "...", summary: {...} } }
```

### State Management

- **Where state lives:** `.planning/sets/{setId}/REVIEW-PIPELINE-STATE.json` (standard) or `.planning/post-merge/{setId}/REVIEW-PIPELINE-STATE.json` (post-merge)
- **How state is read/written:** Direct JSON file read/write with Zod validation. No locking needed because review stages run sequentially (one skill at a time per set).
- **Consistency guarantees:** Zod parse-on-read ensures corrupted state is caught immediately. Each write is atomic (write to temp file, rename). The schema uses `.passthrough()` for forward compatibility.

### Review State Schema (proposed)

```javascript
const ReviewStageResult = z.object({
  completed: z.boolean(),
  at: z.string(),         // ISO timestamp
  summary: z.record(z.unknown()).optional(),
});

const ReviewPipelineState = z.object({
  setId: z.string(),
  postMerge: z.boolean().default(false),
  scope: ReviewStageResult.optional(),
  unitTest: ReviewStageResult.optional(),
  bugHunt: ReviewStageResult.optional(),
  uat: ReviewStageResult.optional(),
  lastUpdatedAt: z.string(),
}).passthrough();
```

### Color Change Flow

```
build-agents.cjs ROLE_COLORS
  -> agents/*.md frontmatter (color: field)
  -> Claude Code renders agent output with specified color

display.cjs STAGE_BG
  -> renderBanner() calls
  -> Terminal output via process.stdout.write
```

Color changes propagate immediately on next `build-agents` run (for agent colors) or next skill invocation (for banner colors). No state migration needed.

## Integration Points

| External System | Protocol | Purpose | Error Handling |
|----------------|----------|---------|---------------|
| Claude Code Plugin API | SKILL.md + agents/*.md | Skill invocation, agent spawning | Skills catch errors and display user-friendly messages |
| Git | CLI (execSync) | Worktree management, merge, diff | Try/catch with fallback behavior |
| File system | Node.js `fs` | State persistence, artifact I/O | Zod validation on read, atomic writes |
| Terminal | ANSI escape codes | Colored output | 16-color basic palette for max compatibility |

No new integration points are introduced in this milestone. All features operate within existing boundaries.

## Scaling Considerations

This milestone's features do not introduce scaling concerns:

- **Review pipeline state** is per-set, per-file JSON. No aggregation or cross-set queries needed.
- **Color changes** are static configuration with zero runtime cost.
- **Documentation** is read-once at development time, not processed by the system.

The only consideration is that REVIEW-PIPELINE-STATE.json adds one more file per set to the `.planning/` directory tree. Given sets are typically 3-8 per milestone, this is negligible.

## Architecture Risks

1. **Review state schema rigidity:** If review stages are added or renamed later, existing REVIEW-PIPELINE-STATE.json files become stale. **Impact:** Low -- `.passthrough()` on the schema and optional fields mean old files validate fine with new schemas. **Mitigation:** Use optional fields for every stage; never require a stage to be present.

2. **Skill-level changes without library support:** Each review skill (SKILL.md) would need to call the pipeline state functions. Since skills are markdown-based prompts, they cannot directly call JavaScript. **Impact:** Medium -- skills must invoke the CLI (`rapid-tools review pipeline-state ...`), adding CLI surface area. **Mitigation:** Add CLI subcommands `review pipeline-state read <set-id>` and `review pipeline-state update <set-id> <stage> <result-json>` that delegate to the library functions.

3. **Color accessibility regression:** Changing from bright to dark ANSI colors could reduce contrast on light-themed terminals. **Impact:** Low -- most developer terminals use dark themes. **Mitigation:** Use standard (non-bright) foreground colors with backgrounds that have sufficient contrast on both light and dark themes. Test both.

4. **Documentation drift:** Comprehensive docs created now may drift as features evolve. **Impact:** Medium over time. **Mitigation:** Structure docs to reference SKILL.md files for details rather than duplicating instruction content. Keep the getting-started guide procedural (step 1, step 2...) so it is easy to maintain.

## Recommendations

1. **Add REVIEW-PIPELINE-STATE.json with CLI interface:** Follow the MERGE-STATE.json precedent exactly -- a per-set JSON file managed through CLI subcommands. Library functions in `review.cjs`, CLI handlers in `src/commands/review.cjs`. -- **Priority: high** (directly addresses the primary feature request and existing user pain point)

2. **Make pipeline state advisory, not blocking:** When a review skill reads pipeline state and finds a prerequisite incomplete, it should WARN the user (e.g., "Bug hunt has not been run. It is recommended before UAT.") but NOT block execution. This preserves the current flexibility where users can run any review stage independently. -- **Priority: high** (architectural decision that affects all skill modifications)

3. **Use dark ANSI background codes (40-47 series) for stage banners:** Replace bright backgrounds (`\x1b[10Xm`) with standard dark backgrounds (`\x1b[4Xm`). Specifically: bright blue `\x1b[104m` -> dark blue `\x1b[44m`, bright green `\x1b[102m` -> dark green `\x1b[42m`, bright red `\x1b[101m` -> dark red `\x1b[41m`. Combined with `\x1b[1m` (bold) and `\x1b[97m` (bright white text), these maintain readability while being less harsh. -- **Priority: medium** (UX improvement, simple change)

4. **Map agent ROLE_COLORS to darker Claude Code color variants:** The Claude Code plugin API accepts color values. Check if it supports extended color names (e.g., `darkblue`, `darkgreen`). If only the basic 8 names are supported, the options are limited. Test which color values Claude Code actually renders in dark terminals. -- **Priority: medium** (depends on Claude Code's supported color palette)

5. **Create a single canonical getting-started guide:** One file `docs/getting-started.md` that walks through the full workflow: install -> init -> branding -> scaffold -> start-set -> discuss-set -> plan-set -> execute-set -> review -> merge -> audit-version -> new-version. Also cover quick, bug-fix, and add-set as side workflows. Reference existing docs files for deep dives. -- **Priority: high** (directly addresses documentation goal for open-sourcing)

6. **Restructure docs/ as a documentation site:** Consider adding a `docs/README.md` or `docs/index.md` that serves as a table of contents, linking all other docs in logical order. This helps GitHub rendering and any future documentation site generation. -- **Priority: low** (nice-to-have for open-sourcing)

7. **Defer the continuous audit hook:** The milestone brief explicitly defers this to a future milestone focused on automated quality gates. The current hooks system (`src/lib/hooks.cjs`) supports post-task verification checks but does not have a post-merge trigger point. Adding one would require changes to the merge pipeline that are out of scope. -- **Priority: deferred** (confirmed deferral from milestone brief)

8. **Defer new-version DEFERRED.md integration:** The milestone brief defers this to the next milestone touching the new-version skill. The current new-version researcher pipeline already reads project context; adding DEFERRED.md discovery is a targeted addition to the researcher briefs. -- **Priority: deferred** (confirmed deferral from milestone brief)
