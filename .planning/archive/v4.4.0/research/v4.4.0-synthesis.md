# Research Summary -- v4.4.0 Polish & Documentation

## Project Profile
- **Type:** brownfield plugin (Claude Code CLI plugin)
- **Stack:** Node.js (v18+) + CommonJS (.cjs) + Zod 3.25.x + proper-lockfile 4.1.2
- **Scale:** small (3 in-scope features, all independent, touching ~15 files total)
- **Key Constraint:** All features must ship with zero new dependencies and no STATE.json schema migration

## Unified Findings

### Stack and Infrastructure

- Node.js v18+ is required; host runs v25.8.0. No `engines` field in package.json -- users on older Node get cryptic errors instead of npm install-time warnings (sources: STACK, OVERSIGHTS)
- Entire codebase is 100% CommonJS (.cjs). No ESM migration pressure. Node.js 18-25 all support CJS without issue (source: STACK)
- Zod 3.25.76 is installed; Zod 4 is available but not needed. Current API usage (z.object, z.enum, .safeParse, .passthrough, .default) is compatible with both versions. Migration deferred (sources: STACK, PITFALLS)
- proper-lockfile 4.1.2 is effectively unmaintained (5 years, no commits) but functional. Its transitive dep signal-exit is pinned at v3 (v4 is ESM-only). No bugs encountered (source: STACK)
- No build step, no linter, no CI/CD pipeline. Tests use built-in `node:test`. No test coverage reporting configured (source: STACK)
- No known CVEs for any dependency (source: STACK)
- Lock stale threshold is 5 minutes (300,000ms). If an agent crashes mid-transaction during review pipeline, all subsequent skills are blocked for 5 minutes (source: PITFALLS)

### Feature Landscape

Three in-scope features, all fully independent with zero file overlap:

**1. Review State Management** (moderate complexity)
- Core problem: review pipeline (scope -> unit-test -> bug-hunt -> uat) has no persistent state. Sessions that end mid-review lose all progress tracking (sources: FEATURES, ARCHITECTURE, PITFALLS, OVERSIGHTS, UX)
- Recommended storage: standalone REVIEW-STATE.json per set (not a SetState schema extension) to avoid state migration (sources: FEATURES, ARCHITECTURE, OVERSIGHTS)
- Existing precedent: MERGE-STATE.json follows the same per-set JSON file pattern (source: ARCHITECTURE)
- Compaction hook `review-stage-complete` already exists as a no-op placeholder in compaction.cjs -- should be wired up (source: OVERSIGHTS)
- Sub-features: schema + persistence, stage completion recording, prerequisite checking, CLI subcommand, skill prompt updates (source: FEATURES)
- File ownership: `src/lib/review.cjs`, `src/commands/review.cjs`, 4 review skill SKILL.md files (source: FEATURES)

**2. Documentation for Open-Sourcing** (moderate complexity)
- Current state: 9 files in docs/ (some thin/placeholder), outdated DOCS.md (references v3.5.0), README.md exists but needs refresh (sources: FEATURES, OVERSIGHTS, UX)
- Missing docs: no quickstart/tutorial, no concept/glossary guide, no docs for /branding, /scaffold, /audit-version, /new-version, /quick, /bug-fix, /add-set (sources: OVERSIGHTS, UX)
- Recommended approach: structured docs/ with index -- fill existing scaffolded files, add workflow.md and auxiliary-commands.md, update DOCS.md as hub (source: FEATURES)
- README.md is the open-source front door -- must be created/updated first (sources: OVERSIGHTS, UX)
- Version references must be bumped from v4.3.0 to v4.4.0 in help/SKILL.md, CHANGELOG.md, version.cjs, package.json (source: OVERSIGHTS)
- File ownership: README.md, DOCS.md, docs/* (source: FEATURES)

**3. Better Colouring** (trivial complexity)
- Current bright ANSI backgrounds (10Xm range) are hard to read with white text on dark terminals, especially bright green (102m) (sources: FEATURES, ARCHITECTURE, PITFALLS, UX)
- Recommended: switch to standard dark backgrounds (4Xm range): 104m->44m, 102m->42m, 101m->41m. Keep bright white bold foreground (sources: FEATURES, STACK, ARCHITECTURE, UX)
- display.test.cjs has 6+ hardcoded ANSI assertions that must be updated in the same task (source: OVERSIGHTS)
- Missing banner registrations: unit-test, bug-hunt, uat, bug-fix are not in STAGE_VERBS/STAGE_BG -- existing bug, cheap to fix now (source: OVERSIGHTS)
- Agent frontmatter ROLE_COLORS are Claude Code UI colors (not terminal ANSI) -- separate system, may or may not need changing (sources: FEATURES, OVERSIGHTS)
- File ownership: `src/lib/display.cjs`, `src/lib/display.test.cjs` (source: FEATURES)

**Deferred features** (NOT in scope):
- Continuous audit hook (post-merge auto-audit) -- deferred to future milestone (sources: FEATURES, ARCHITECTURE, OVERSIGHTS)
- New-version DEFERRED.md integration -- deferred to next milestone touching new-version (sources: FEATURES, ARCHITECTURE, OVERSIGHTS)

### Architecture Direction

- **Pattern:** Extend the existing layered modular monolith (Skill -> Agent -> CLI -> Library -> Storage). No architectural changes needed (source: ARCHITECTURE)
- **Review state module:** Add ReviewPipelineState schema and read/write functions to existing `src/lib/review.cjs`. Add CLI handlers to `src/commands/review.cjs`. Follow MERGE-STATE.json precedent exactly (source: ARCHITECTURE)
- **State isolation:** Review pipeline state is decoupled from core lifecycle state (SetState). Uses `.passthrough()` and optional fields for forward compatibility. No locking needed -- review stages run sequentially per set (source: ARCHITECTURE)
- **Display module:** Remains a zero-dependency, single-file color module. API unchanged (renderBanner, STAGE_BG, STAGE_VERBS) (source: ARCHITECTURE)
- **No new integration points** introduced. All features operate within existing boundaries (source: ARCHITECTURE)

### Cross-Cutting Requirements

**Must address early:**
- Review state storage design (REVIEW-STATE.json vs SetState extension) -- wrong choice forces migration of all review artifacts and 4+ skill files (source: OVERSIGHTS)
- Review state idempotency/re-entry rules -- must be designed alongside schema, not retrofitted (source: OVERSIGHTS)
- Display color + test update atomicity -- guaranteed test failures if not coordinated (source: OVERSIGHTS)

**Can address later:**
- NO_COLOR support -- important for open-sourcing but not structurally entangled (sources: PITFALLS, UX)
- Missing banner registrations (unit-test, bug-hunt, uat, bug-fix) -- existing bug, additive fix (source: OVERSIGHTS)
- Documentation testing/command verification -- manual validation step after docs are written (source: OVERSIGHTS)
- Version bump coordination (v4.3.0 -> v4.4.0) -- execute last but plan early (source: OVERSIGHTS)
- Solo set + review state edge case -- design decision needed but not complex implementation (source: OVERSIGHTS)

### User Experience Direction

- RAPID uses a **conversational CLI** paradigm -- slash commands in Claude Code chat, not traditional CLI (source: UX)
- Users expect **pipeline visualization** for multi-stage review: which stages are done, which remain. Comparable to CI/CD pipeline status views (source: UX)
- Each review sub-stage completion banner should list remaining stages (e.g., "Unit testing complete. Remaining: /rapid:bug-hunt, /rapid:uat") (source: UX)
- Review state should be **advisory, not blocking**: warn about missing prerequisites but allow proceeding (sources: ARCHITECTURE, UX, OVERSIGHTS)
- Documentation must follow **README-first** convention: overview, quickstart, installation, detailed guides. Current docs assume familiarity with RAPID concepts (source: UX)
- New users need a **concept/glossary guide** mapping RAPID terminology to familiar concepts (set=branch, job=task, wave=step, milestone=project) (source: UX)
- The `/rapid:help` command is strong but insufficient for onboarding -- external browsable docs are needed (source: UX)
- Consider progressive disclosure: start with `/rapid:init` and `/rapid:quick`, introduce full lifecycle later (source: UX)
- Missing `[RAPID WARNING]` prefix convention for non-fatal messages -- currently ad-hoc (source: UX)
- Artifact presence detection (REVIEW-SCOPE.md, REVIEW-UNIT.md, etc.) could serve as lightweight state tracking, matching the existing WAVE-COMPLETE.md pattern (source: UX)

## Contradictions

### Contradiction: Review State Storage Mechanism

- **FEATURES says:** Use file-based REVIEW-STATE.json (Option 1, recommended) -- avoids STATE.json schema migration, follows established pattern
- **PITFALLS says:** Store review state in STATE.json as a structured field on SetState (anti-pattern section says "Correct approach: Store review pipeline state in STATE.json")
- **OVERSIGHTS says:** Use standalone REVIEW-STATE.json to avoid triggering a state migration (agrees with FEATURES)
- **ARCHITECTURE says:** Use standalone REVIEW-PIPELINE-STATE.json following MERGE-STATE.json precedent (agrees with FEATURES)
- **UX says:** Artifact presence detection (REVIEW-UNIT.md, REVIEW-BUGS.md) could serve as state, no explicit JSON file needed
- **Resolution:** Consensus favors standalone JSON file (3 of 5 agents agree). PITFALLS' recommendation for STATE.json integration is inconsistent with its own warning about schema migration risks. UX's artifact-presence approach is the lightest-weight but lacks metadata (timestamps, summary counts). **Recommend: REVIEW-STATE.json per set**, combining the structured metadata benefits with migration-free deployment.

### Contradiction: File Name for Review State

- **FEATURES says:** `REVIEW-STATE.json`
- **ARCHITECTURE says:** `REVIEW-PIPELINE-STATE.json`
- **OVERSIGHTS says:** `REVIEW-STATE.json`
- **Resolution:** Minor naming difference. `REVIEW-STATE.json` is shorter and consistent with `MERGE-STATE.json` (not `MERGE-PIPELINE-STATE.json`). **Recommend: REVIEW-STATE.json**.

### Contradiction: NO_COLOR -- Blocker or Nice-to-Have?

- **PITFALLS says:** NO_COLOR support is a high-priority blocker for open-source readiness (ranked #3 in priority list)
- **UX says:** NO_COLOR is "table-stakes for open-source CLI tools" (high priority)
- **STACK says:** NO_COLOR is "noted but not a blocker -- banners are visual branding and always use color by design"
- **Resolution:** STACK's assessment is specific to RAPID's current internal-only use. For open-sourcing (a milestone goal), NO_COLOR support is important. **Recommend: include NO_COLOR support in the color change task, not as a separate feature**.

## Recommended Approach

This milestone has three fully independent features with zero file overlap, making it ideal for parallel execution as three separate sets. The features vary in complexity but none require new dependencies or architectural changes -- all extend existing patterns within the established layered modular monolith.

The **review state management** feature is the most technically nuanced. The key design decision -- using a standalone REVIEW-STATE.json file per set rather than extending the SetState schema in STATE.json -- avoids triggering a state migration and follows the established MERGE-STATE.json precedent. The schema should be designed and implemented first (Zod schema, read/write helpers, CLI subcommands), followed by wiring the 4 review skill SKILL.md files to read/write the state. The idempotency rule should be "advisory, not blocking": prerequisite stages that are incomplete produce a warning, not an error. The existing compaction hook placeholder `review-stage-complete` should be connected. A roundtrip unit test is essential to catch the Zod `.passthrough()` silent-typo risk.

The **colouring** feature is trivial in scope but has a testing trap: display.test.cjs contains 6+ hardcoded ANSI escape code assertions that will break unless updated alongside display.cjs. The fix is straightforward -- replace bright backgrounds (10Xm) with dark equivalents (4Xm) in all 17 STAGE_BG entries, and add the 4 missing banner registrations (unit-test, bug-hunt, uat, bug-fix) since the file is already being modified. Adding NO_COLOR support (check `process.env.NO_COLOR` before emitting ANSI codes) is a low-cost addition that significantly improves open-source readiness and should be included in this task.

The **documentation** feature is content-heavy but mechanically simple. README.md is the highest-priority artifact -- it is the open-source front door and should be written first. The structured docs/ approach (filling existing scaffolded files, adding workflow.md and auxiliary-commands.md) provides both narrative flow and modular reference. Documentation should be written last among the three features so it can reference the final state of review state management and colouring. Version references must be bumped to v4.4.0 in help/SKILL.md, CHANGELOG.md, version.cjs, and package.json as part of this set. Documentation must cross-reference every `/rapid:X` command against the actual skills/ directory to avoid stale command name references.

### Suggested Set Boundaries

- **Set A: colouring** -- `src/lib/display.cjs`, `src/lib/display.test.cjs`, optionally `src/commands/build-agents.cjs`. Rationale: trivial complexity, self-contained, quick win that validates the development pipeline. Includes dark ANSI backgrounds, missing banner registrations, and NO_COLOR support.

- **Set B: review-state** -- `src/lib/review.cjs`, `src/lib/review.test.cjs`, `src/commands/review.cjs`, `src/commands/review.test.cjs`, `skills/review/SKILL.md`, `skills/unit-test/SKILL.md`, `skills/bug-hunt/SKILL.md`, `skills/uat/SKILL.md`. Rationale: the core feature of this milestone, moderate complexity, touches the most files but all within the review domain.

- **Set C: documentation** -- `README.md`, `DOCS.md`, `docs/*`, `skills/help/SKILL.md`, `package.json` (version bump + engines field), `src/lib/version.cjs`. Rationale: content-heavy, benefits from being last so it can document the final state of the other two features. Includes version bump to v4.4.0.

All three sets have zero file overlap and can execute as fully parallel sets with no dependency ordering.
