# Pitfalls Research -- v4.4.0 Polish & Documentation

## Critical Pitfalls

### Pitfall: Review Pipeline State Not Persisted Across Sessions
- **Category:** correctness
- **Stack:** STATE.json / state-schemas.cjs / review.cjs
- **Description:** The current `SetState` schema has no field for tracking which review stages (scoping, unit-test, bug-hunt, UAT) have been completed. Review pipeline progress exists only in the conversation context. If a session ends mid-review, there is no way to know which stages ran successfully. Adding a `reviewStages` field to `SetState` requires a schema migration since `ProjectState` is Zod-validated at every read/write boundary.
- **Trigger:** User runs `/rapid:review`, then `/rapid:unit-test`, then the session disconnects. Next session, `/rapid:bug-hunt` has no way to know unit-test already passed.
- **Impact:** Critical -- users re-run expensive review stages or skip stages they believe already ran, producing incomplete reviews.
- **Prevention:**
  1. Add a `reviewProgress` field to `SetState` (e.g., `{ scoped: boolean, unitTested: boolean, bugHunted: boolean, uatPassed: boolean }`).
  2. Use `.passthrough()` already on `SetState` to allow forward-compatible additions without breaking existing state files -- but downstream consumers that validate strictly may reject unknown keys.
  3. Write a state migration function that adds default values (`false`) for existing sets.
  4. Update each review skill to check and set the corresponding flag via `withStateTransaction`.
- **Detection:** After adding the feature, run all review skills in sequence and verify flags persist in STATE.json between separate CLI invocations.
- **References:** `src/lib/state-schemas.cjs:21-25` (SetState uses `.passthrough()`), `src/lib/state-transitions.cjs` (no review substates)

### Pitfall: Zod `.passthrough()` Masks Schema Drift Silently
- **Category:** correctness / data-loss
- **Stack:** Zod 3.25.x / state-schemas.cjs
- **Description:** Every state schema (`SetState`, `WaveState`, `MilestoneState`, `ProjectState`) uses `.passthrough()`, which allows any extra keys to survive validation without error. This is intentional for forward compatibility, but it means typos in new field names (e.g., `reviewProgres` instead of `reviewProgress`) will silently persist in STATE.json and never be caught. When adding review pipeline state tracking, a misspelled key will appear to work but will never be read back correctly.
- **Trigger:** A developer adds a new field name to state but misspells it in either the write or read path. Zod validation passes on both sides because `.passthrough()` keeps the misspelled key and the correctly-named default key.
- **Impact:** High -- state data is written but never read, causing features to appear broken with no error messages.
- **Prevention:**
  1. For any new state fields, add them explicitly to the Zod schema rather than relying on passthrough.
  2. Write unit tests that roundtrip state through parse/serialize and assert the new fields survive.
  3. Consider adding a `.strict()` mode for development/testing that rejects unknown keys.
- **Detection:** Unit tests that assert specific field values after state mutation. Integration test that writes review progress, reads it back, and asserts equality.
- **References:** Zod docs on `.passthrough()` vs `.strict()`, `src/lib/state-schemas.cjs`

## High-Risk Pitfalls

### Pitfall: ANSI Color Codes Hardcoded Without NO_COLOR Support
- **Category:** compatibility
- **Stack:** display.cjs / ANSI escape codes
- **Description:** `display.cjs` explicitly states "Always outputs colors (no NO_COLOR checking)" (line 7). The `NO_COLOR` standard (https://no-color.org/) is widely adopted. When preparing for open-source, users with `NO_COLOR=1` set (common in CI environments, screen readers, and piped output) will see raw escape codes like `\x1b[104m` in their output instead of clean text.
- **Trigger:** Any environment where `NO_COLOR` is set, or where stdout is piped to a file/another program, or where the terminal does not support ANSI codes.
- **Impact:** High -- broken output for CI pipelines, accessibility tools, and log files. Bad first impression for open-source users.
- **Prevention:**
  1. Check `process.env.NO_COLOR` and `!process.stdout.isTTY` before emitting ANSI codes.
  2. Fall back to plain text (no escape codes) when color is disabled.
  3. Optionally support `FORCE_COLOR=1` to override the TTY check.
- **Detection:** Run `NO_COLOR=1 node src/bin/rapid-tools.cjs display banner init` and verify no escape codes in output.
- **References:** https://no-color.org/, display.cjs line 7

### Pitfall: Bright Background Colors Unreadable on Light Terminals
- **Category:** compatibility
- **Stack:** display.cjs STAGE_BG map
- **Description:** The current STAGE_BG mapping uses bright background ANSI codes (`\x1b[104m` bright blue, `\x1b[102m` bright green, `\x1b[101m` bright red) with bright white (`\x1b[97m`) foreground text. On dark terminals, bright backgrounds with white text work well. On light terminals (common on macOS default, some IDE terminals), bright green and bright blue backgrounds make white text nearly invisible. The milestone goal is to change bright colors to dark colors -- but dark backgrounds with white text may also have contrast issues on certain terminal themes.
- **Trigger:** User has a light-colored terminal theme (e.g., Solarized Light, macOS default Terminal.app light mode).
- **Impact:** High -- banners become unreadable, undermining the professional appearance for open-source.
- **Prevention:**
  1. Use dark/standard background codes (`\x1b[44m` blue, `\x1b[42m` green, `\x1b[41m` red) instead of bright variants (`\x1b[10Xm`).
  2. Keep foreground as bright white + bold (`\x1b[1m\x1b[97m`) for maximum contrast against dark backgrounds.
  3. Test against both dark and light terminal themes before shipping.
  4. Document supported terminal configurations in the open-source docs.
- **Detection:** Visual inspection on at least 3 terminal configs: dark theme, light theme, and default macOS Terminal.
- **References:** `src/lib/display.cjs:55-73` (STAGE_BG map), ANSI escape code reference

### Pitfall: Agent Color Values Not Validated Against Claude Code API
- **Category:** correctness
- **Stack:** build-agents.cjs / Claude Code plugin API
- **Description:** Agent frontmatter `color:` values (blue, green, red, yellow, purple, cyan) are assigned in `ROLE_COLORS` in `build-agents.cjs`. These values are passed directly to Claude Code's agent definition format. If Claude Code only accepts a specific set of color names (or hex values, or a different format), an invalid color will either be silently ignored or cause an agent load failure. There is no validation that the chosen color values are in the set that Claude Code actually supports.
- **Trigger:** Changing color values (as planned in this milestone) to names not in Claude Code's allowed set.
- **Impact:** Medium-high -- agents silently fall back to default color or fail to load entirely, with no clear error.
- **Prevention:**
  1. Check Claude Code plugin documentation for the exact set of supported color values.
  2. Add a validation step in `build-agents` that rejects unknown color names.
  3. Test the built agents by loading them in Claude Code and verifying color rendering.
- **Detection:** After changing colors, run `/rapid:install` and verify each agent loads without warnings.
- **References:** `src/commands/build-agents.cjs:49-77` (ROLE_COLORS)

### Pitfall: Documentation Referencing Non-Existent or Renamed Commands
- **Category:** correctness
- **Stack:** docs/*.md / skills/*/SKILL.md
- **Description:** The milestone requires documenting the canonical workflow including 14+ commands. The project has evolved through multiple versions with command renames (e.g., `set-init` to `start-set`, `wave-plan` to `plan-set`). The `STAGE_VERBS` map in `display.cjs` still contains legacy entries (`set-init`, `discuss`, `wave-plan`, `execute`). Documentation that references stale command names will confuse new open-source users.
- **Trigger:** Copy-pasting from existing internal docs or code comments that still reference old command names.
- **Impact:** High -- new users follow documentation, commands fail, immediate loss of trust in the project.
- **Prevention:**
  1. Cross-reference every command name in documentation against the actual `skills/*/` directory listing.
  2. Remove or mark legacy `STAGE_VERBS` entries to prevent confusion.
  3. Add a documentation validation test that checks every `/rapid:X` reference resolves to an existing skill directory.
- **Detection:** `grep -r '/rapid:' docs/ | sort -u` and verify each matches a directory in `skills/`.
- **References:** `src/lib/display.cjs:26-44` (STAGE_VERBS with legacy entries), `skills/` directory listing (28 skills)

### Pitfall: Lock Stale Threshold Too Long for Interactive Use
- **Category:** performance
- **Stack:** proper-lockfile 4.1.2 / lock.cjs
- **Description:** The stale lock threshold is 5 minutes (300,000ms) in `lock.cjs`. If a Claude Code agent crashes mid-transaction (OOM, timeout, network disconnect), the lock remains held for 5 minutes. During review pipeline tracking, multiple skills may need to update state in sequence. A stale lock from a crashed review stage blocks all subsequent pipeline stages for 5 minutes.
- **Trigger:** Agent crash, session timeout, or `Ctrl+C` during a state transaction. Subsequent command within 5 minutes hits lock contention.
- **Impact:** Medium-high -- user waits 5 minutes with no feedback, or gets a cryptic lock error. Particularly painful during review pipeline where multiple skills run in sequence.
- **Prevention:**
  1. Consider reducing stale threshold to 60-90 seconds for interactive commands.
  2. Add a `--force-unlock` CLI command for manual recovery.
  3. Ensure review pipeline state writes are fast (sub-second) to minimize the window where a crash can leave a stale lock.
- **Detection:** Kill a process mid-transaction, then immediately try another state-modifying command. Verify recovery time.
- **References:** `src/lib/lock.cjs:8` (STALE_THRESHOLD = 300000)

## Medium-Risk Pitfalls

### Pitfall: State Migration Not Handled for New Review Fields
- **Category:** correctness
- **Stack:** state-schemas.cjs / state-machine.cjs
- **Description:** Adding `reviewProgress` to `SetState` creates a schema change. Existing STATE.json files from previous milestones will not have this field. While `.passthrough()` means old files will still validate, code that reads `state.sets[i].reviewProgress` will get `undefined` on unmigrated state. If the code does not handle `undefined` gracefully, it will throw or behave incorrectly.
- **Trigger:** Upgrading from v4.3.0 to v4.4.0 with an existing project that has sets in various states.
- **Impact:** Medium -- existing projects break on upgrade unless migration is handled.
- **Prevention:**
  1. Use Zod `.default({})` on the new field so it auto-populates on parse.
  2. Add a migration function in the `/rapid:migrate` skill.
  3. Defensive coding: always access with `state.reviewProgress ?? {}`.
- **Detection:** Copy a v4.3.0 STATE.json, run the new code against it, verify no crashes.
- **References:** `src/lib/state-schemas.cjs` (current SetState schema), `skills/migrate/SKILL.md`

### Pitfall: Review Scope File Coupling Creates Fragile Pipeline
- **Category:** correctness
- **Stack:** review.cjs / skills/review, unit-test, bug-hunt, uat
- **Description:** The review pipeline is file-coupled: `/rapid:review` writes `REVIEW-SCOPE.md`, which `/rapid:unit-test`, `/rapid:bug-hunt`, and `/rapid:uat` read as input. If the review scope format changes (e.g., new sections for review pipeline state), downstream skills will fail to parse it. There is no versioning or schema validation on `REVIEW-SCOPE.md`.
- **Trigger:** Modifying REVIEW-SCOPE.md format in the review skill without updating all three downstream consumers.
- **Impact:** Medium -- downstream review skills fail or produce incomplete results.
- **Prevention:**
  1. Add a `<!-- SCOPE-META ... -->` version field and check it in downstream skills.
  2. If adding review pipeline tracking, keep it in STATE.json (structured, validated) rather than in REVIEW-SCOPE.md (markdown, unvalidated).
  3. Integration test that runs the full review pipeline end-to-end.
- **Detection:** Change REVIEW-SCOPE.md format, run all downstream skills, verify they still work.
- **References:** `skills/review/SKILL.md:10-53` (SCOPE-META format)

### Pitfall: execSync in review.cjs Blocks Event Loop
- **Category:** performance
- **Stack:** Node.js / review.cjs
- **Description:** `scopeSetPostMerge()` uses `execSync` for `git log`, `git cat-file`, and `git diff` operations. In the context of adding review state persistence, if state writes are also synchronous, the entire review scoping operation becomes a chain of blocking calls. This is fine for CLI usage but could cause issues if the code is ever called from a long-running process.
- **Trigger:** Large repos with many changed files where git operations take seconds.
- **Impact:** Low-medium -- noticeable delay during review scoping on large changesets.
- **Prevention:**
  1. For this milestone, this is acceptable since all usage is CLI-based.
  2. If review pipeline state introduces async state writes (via `withStateTransaction` which is already async), ensure the callers properly await.
  3. Do not mix sync git calls with async state mutations in the same function without proper sequencing.
- **Detection:** Profile review scoping on a set with 50+ changed files.
- **References:** `src/lib/review.cjs:113-182` (scopeSetPostMerge with execSync)

### Pitfall: Documentation Markdown Links Break in Different Viewers
- **Category:** compatibility
- **Stack:** Markdown / GitHub / local viewers
- **Description:** When writing open-source documentation, relative links between docs (e.g., `[setup](./setup.md)`) work on GitHub but may break in local Markdown viewers, npm package viewers, or MkDocs-style documentation sites. Anchor links (`#section-name`) are case-sensitive on some platforms and case-insensitive on others.
- **Trigger:** Users reading docs outside of GitHub (local editors, VS Code preview, documentation sites).
- **Impact:** Medium -- broken navigation in docs is a usability issue for new contributors.
- **Prevention:**
  1. Use consistent relative paths from the docs/ root.
  2. Test links in both GitHub rendering and at least one local viewer.
  3. Avoid special characters in heading text (parentheses, colons) that create inconsistent anchor slugs.
  4. Use lowercase-only anchor references.
- **Detection:** Run a link checker (e.g., `markdown-link-check`) over the docs/ directory.

## Anti-Patterns for This Stack

### Anti-Pattern: Storing Pipeline State in Markdown Files
- **What people do:** Track which review stages have run by writing status markers into REVIEW-SCOPE.md or similar markdown artifacts.
- **Why it fails:** Markdown files have no schema validation. Parsing status from markdown is fragile (regex-based). Multiple agents writing to the same markdown file creates merge conflicts. The state is invisible to the CLI (`node rapid-tools.cjs state get`).
- **Correct approach:** Store review pipeline state in STATE.json as a structured field on `SetState`. This integrates with existing Zod validation, file locking, and CLI state queries.

### Anti-Pattern: Using Bright ANSI Colors for Both Background and Foreground
- **What people do:** Pair bright background (`\x1b[10Xm`) with bright foreground (`\x1b[9Xm`) for "vivid" output.
- **Why it fails:** Bright-on-bright has insufficient contrast. Many terminals map bright colors differently (e.g., Solarized remaps bright colors to dim variants). The result is invisible or near-invisible text.
- **Correct approach:** Use standard/dark backgrounds (`\x1b[4Xm`) with bold bright white foreground (`\x1b[1m\x1b[97m`). This provides high contrast on both dark and light terminal themes.

### Anti-Pattern: Documentation That Describes Architecture Instead of Usage
- **What people do:** Write open-source docs that explain internal state machines, Zod schemas, and agent pipelines.
- **Why it fails:** New users want to know "how do I use this?" not "how does this work internally?" Architecture docs are useful for contributors but are a barrier for first-time users.
- **Correct approach:** Lead with a quick-start guide showing the canonical workflow (`/install` through `/merge`). Put architecture details in a separate CONTRIBUTING.md or ARCHITECTURE.md.

### Anti-Pattern: Hard-Coding Color Names Without Enum Validation
- **What people do:** Use string literals like `'blue'`, `'green'` in ROLE_COLORS without any validation that these are valid Claude Code agent colors.
- **Why it fails:** A typo (`'pruple'` instead of `'purple'`) silently produces an invalid agent definition. No error until runtime.
- **Correct approach:** Define a `VALID_COLORS` array or Set, and validate each entry in ROLE_COLORS against it during build-agents.

## Version-Specific Gotchas

- **Node.js 18+**: `node:test` runner behavior changed between 18.x and 20.x (test isolation, `--test` glob patterns). Documentation should specify minimum Node version clearly.
- **Zod 3.25.76**: This is a very recent version. If users install with `^3.25.76`, they could get a patch that changes `.passthrough()` behavior. Pin exact version in documentation for reproducibility.
- **proper-lockfile 4.1.2**: The `update` interval (half of stale threshold) means lock refresh fires every 2.5 minutes. On NFS or network filesystems, this can race with stale detection. Not an issue for local development but worth noting for CI environments.
- **Git 2.30+**: `git worktree` has improved significantly in recent versions. Git 2.30 is the minimum but some features (like `git worktree repair`) require 2.36+. Document the recommended Git version.

## Security Pitfalls

1. **STATE.json injection via project name**: The `projectName` field in STATE.json is user-supplied (from `/rapid:init`). If documentation generators or display functions interpolate this value into shell commands or HTML without escaping, it creates an injection vector. -- Prevention: Sanitize `projectName` on input (alphanumeric + hyphens only) or escape on output.
2. **Lock target files contain PID**: `lock.cjs` writes `process.pid` to lock target files. While these files are gitignored, if the `.gitignore` in `.locks/` is accidentally deleted, PIDs could leak into version control. -- Prevention: The current `.gitignore` approach is correct; add a pre-commit hook check.
3. **execSync with user-derived arguments in review scoping**: `scopeSetPostMerge` interpolates `setId` into a `git log --grep` command. If `setId` contains shell metacharacters, this could be exploited. -- Prevention: The current code uses `execSync` with string arguments (not arrays), making it shell-interpreted. Use `execFileSync` with array arguments for git commands that include user input.

## Performance Traps

1. **walkDir in findDependents scans entire project tree**: `review.cjs:findDependents()` calls `walkDir(cwd, ...)` which recursively scans all source files to find import/require references. On large repos, this is O(files * changedFiles). -- Mitigation: Cache the dependency graph per session, or use `git grep` instead of filesystem walk.
2. **State file re-read on every CLI invocation**: Each `node rapid-tools.cjs` call reads, validates, and parses STATE.json from scratch. For the review pipeline where 4+ skills run in sequence, this means 4+ full parse cycles. -- Mitigation: Acceptable for current scale; would need caching for larger projects.
3. **Agent build regenerates all 26 agents**: `build-agents` rebuilds every agent file even when only one role module changed. After color changes, all 26 files will be modified, creating a large diff. -- Mitigation: Consider incremental builds (check mtime), or accept the rebuild cost since it is a one-time operation per change.

## Summary Priority List

1. **Review pipeline state not persisted** -- Adding state tracking is the core feature; getting the schema migration wrong will break all existing projects.
2. **Zod passthrough masks schema drift** -- New fields added without explicit schema entries will silently fail, making review state tracking appear broken.
3. **NO_COLOR standard not respected** -- Immediate blocker for open-source readiness; CI environments and accessibility tools will produce garbled output.
4. **Documentation referencing stale command names** -- Legacy command names in code and docs will confuse new open-source users on day one.
5. **Bright ANSI colors unreadable on light terminals** -- The color change must be tested against multiple terminal themes or the fix becomes a new bug.
6. **Agent color values not validated** -- Changing colors without knowing Claude Code's valid set risks breaking agent definitions.
7. **State migration for existing projects** -- Upgrading from v4.3.0 must not break existing STATE.json files.
8. **Lock stale threshold too long** -- 5-minute lockout after crash is painful during iterative review pipeline usage.
9. **Review scope file coupling** -- Changes to REVIEW-SCOPE.md format without updating downstream skills breaks the pipeline.
10. **execSync with user-derived arguments** -- Shell injection risk in review scoping, low probability but high severity.
