# Oversights Research -- v4.0.0 Mission Control

## Cross-Cutting Concerns

### 1. Data Ownership and Source-of-Truth Boundaries

**Why this matters for this project specifically:** The Mission Control dashboard introduces a second persistence layer (SQLite at `~/.rapid/`) alongside the existing per-project `.planning/` directories. This creates a dual-source-of-truth problem that will cause confusion about where data canonically lives.

**Specific boundary tensions:**

| Data | Current Owner | Dashboard Candidate | Conflict Risk |
|------|--------------|--------------------|----|
| Project STATE.json | `.planning/STATE.json` per project | Dashboard reads it | Read-only -- low risk |
| Worktree REGISTRY.json | `.planning/worktrees/REGISTRY.json` | Dashboard reads it | Read-only -- low risk |
| Kanban board items | Does not exist yet | `~/.rapid/rapid.db` | **HIGH** -- kanban items have no project-side representation |
| Markdown notes | Does not exist yet | `~/.rapid/rapid.db` or filesystem | **HIGH** -- notes about a project stored outside the project |
| Project registry | Does not exist yet | `~/.rapid/rapid.db` | **MEDIUM** -- project paths stored centrally |
| Knowledge graph | Does not exist yet | `~/.rapid/rapid.db` | **MEDIUM** -- derived data, but expensive to regenerate |
| Codebase mapping | Does not exist yet | `~/.rapid/rapid.db` | **MEDIUM** -- derived from analysis, cached centrally |

**The critical question:** If a user deletes and re-clones a project, should their kanban items and notes survive? If yes, the dashboard DB is the source of truth and project data lives outside the project -- which means it is not version-controlled, not backed up with the project, and not portable. If no, then notes should live in `.planning/` and the dashboard merely renders them.

**Recommendation:** Establish a clear rule before implementation:
- **Derived/cached data** (knowledge graph, codebase mapping, worktree state snapshots) belongs in `~/.rapid/rapid.db` because it can be regenerated from the project.
- **User-created data** (kanban items, notes) should live in `.planning/notes/` and `.planning/kanban/` within each project, with the dashboard DB holding only an index/cache. This ensures portability and version control.
- **Registry data** (which projects exist and their paths) belongs exclusively in `~/.rapid/rapid.db` since it is machine-specific.

### 2. Stale Data Detection and Race Conditions

**Why this matters for this project specifically:** The RAPID CLI uses file-based locking (`proper-lockfile` with mkdir-based atomic locks) and writes STATE.json atomically. The dashboard will be reading these same files from a separate process. There is no IPC channel between the CLI and the dashboard.

**Race condition scenarios:**

1. **CLI writes STATE.json while dashboard is reading it:** The CLI uses atomic rename (write temp, rename), so the dashboard should see either the old or new file, never a partial write. However, the dashboard could read between the temp-write and the rename, seeing stale data. This is a very narrow window but real.

2. **Multiple CLI instances modifying state while dashboard polls:** The CLI lock system serializes writes, but the dashboard has no participation in this lock protocol. If the dashboard reads STATE.json between two CLI transactions, it sees an intermediate state that may be stale milliseconds later.

3. **File watcher event ordering:** If using `watchdog` or `watchfiles` to detect STATE.json changes, filesystem event delivery is not guaranteed to be ordered or deduplicated. Rapid successive writes (e.g., transitioning a set through multiple states in a script) could cause the dashboard to process events out of order or trigger excessive re-reads.

4. **Network filesystem delays:** For projects on NFS, SMB, or FUSE mounts, inotify events may be delayed or missing entirely. The dashboard would show stale data indefinitely.

**Recommended approach:**

- Use **polling with mtime comparison** as the primary mechanism rather than filesystem watchers. Poll every 2-5 seconds, compare file mtime, re-read only on change. This is robust across all filesystem types.
- Use filesystem watching (`watchfiles` library, not `watchdog` -- it is faster and lower resource) as an optimization layer that triggers immediate re-reads, but never as the sole mechanism.
- The dashboard should **never acquire RAPID file locks** -- it is a read-only consumer. It should tolerate reading slightly stale data gracefully.
- Display a "last synced" timestamp in the UI so users understand when data was last refreshed.

### 3. Project Lifecycle Edge Cases

**Why this matters for this project specifically:** The dashboard maintains a registry of project paths. Unlike a CI system that manages its own repos, RAPID projects are user-managed directories that can be moved, deleted, renamed, or corrupted independently of the dashboard.

**Edge cases to handle:**

| Scenario | Detection | Recommended Behavior |
|----------|-----------|---------------------|
| Project directory deleted | `os.path.exists(path)` returns False | Mark as "unavailable" in registry, show grayed-out in UI, do not auto-remove |
| Project moved to new path | Old path gone, no way to detect new path | Show as unavailable, provide "relocate" action in UI |
| `.planning/` directory deleted but project dir exists | `os.path.exists(path/.planning)` fails | Show as "uninitialized" -- project exists but RAPID state is gone |
| Git repo corrupted | `git status` exits non-zero | Show project with warning badge, still display last-known state from cache |
| STATE.json has invalid JSON | JSON parse fails | Show project with error badge, display raw parse error |
| STATE.json fails Zod validation | Schema validation fails | Show project with warning, display partial data where possible |
| Project on unmounted drive | Path traversal hangs or returns ENOENT | Use **timeout on all filesystem probes** (2-second max). Never block the event loop. |
| Symlinked project directory | `os.path.realpath()` differs from registered path | Resolve symlinks at registration time, store canonical path |
| Project directory has restrictive permissions | `os.access(path, os.R_OK)` fails | Show as "permission denied", do not crash |

**Critical implementation detail:** All filesystem probes for registered projects must be **async with timeouts**. A single hung NFS mount must not block the dashboard from serving requests for other projects. Use `asyncio.wait_for()` with a 2-second timeout on any operation that touches a project path.

### 4. Service Lifecycle Management (systemd)

**Why this matters for this project specifically:** The spec calls for the dashboard to "start on boot" via systemd. This introduces several concerns that are easy to overlook for a developer tool.

**Systemd user unit requirements:**

1. **Lingering must be enabled:** Without `loginctl enable-linger <user>`, user services stop when the user logs out. The install process must either enable linger or clearly warn the user. Enabling linger requires root on some distributions.

2. **XDG_RUNTIME_DIR availability:** User services depend on `/run/user/<uid>/` existing. Before linger is enabled, this directory only exists while the user has an active session. If the service starts before login (after reboot with linger), it must not assume `XDG_RUNTIME_DIR` is set.

3. **Unit file location:** Must be `~/.config/systemd/user/rapid-dashboard.service`. The install skill should create this file, not the setup.sh script, because it requires the Python virtualenv path which may vary.

4. **Environment variable propagation:** The systemd user service does NOT inherit the user's shell environment. `RAPID_TOOLS`, `PATH`, and other env vars set in `.bashrc`/`.zshrc`/`config.fish` are NOT available. The unit file must explicitly set `Environment=` or use `EnvironmentFile=`.

5. **Restart policy:** Use `Restart=on-failure` with `RestartSec=5s`. Do NOT use `Restart=always` -- if the port is already in use, the service would restart-loop.

6. **Port conflict detection:** Port 8998 could be in use by another application. The service should detect this at startup and fail with a clear error message, not silently crash. Consider making the port configurable via `~/.rapid/config.toml` or an env var.

7. **Graceful shutdown:** The service must handle SIGTERM cleanly -- close SQLite connections, flush pending writes, release any file watchers. FastAPI with uvicorn handles SIGTERM by default, but any background tasks (file watchers, polling loops) need explicit cleanup.

8. **macOS and non-systemd systems:** The spec assumes systemd, but RAPID supports macOS. For macOS, the equivalent is a launchd plist (`~/Library/LaunchAgents/com.rapid.dashboard.plist`). For systems without either (WSL, containers), provide instructions for manual startup. The install skill should detect the init system and generate the appropriate config.

**Recommended unit file template:**

```ini
[Unit]
Description=RAPID Mission Control Dashboard
After=network.target

[Service]
Type=simple
ExecStart=/path/to/venv/bin/uvicorn rapid_dashboard.main:app --host 127.0.0.1 --port 8998
WorkingDirectory=%h/.rapid
Restart=on-failure
RestartSec=5s
EnvironmentFile=%h/.rapid/.env

[Install]
WantedBy=default.target
```

### 5. Resource Consumption and Background Service Overhead

**Why this matters for this project specifically:** This is a developer tool. Developers are sensitive to background processes consuming resources -- especially memory, CPU, and disk I/O. A dashboard that consumes 500MB of RAM or spins a CPU core during idle will be uninstalled immediately.

**Baseline resource budget for a background developer tool:**

| Resource | Acceptable | Concerning | Unacceptable |
|----------|-----------|------------|--------------|
| Idle RAM | < 50 MB | 50-100 MB | > 100 MB |
| Idle CPU | < 0.1% | 0.1-1% | > 1% |
| Disk I/O (idle) | < 1 KB/s | 1-10 KB/s | > 10 KB/s |
| Startup time | < 3s | 3-10s | > 10s |

**Specific resource concerns:**

1. **Python process overhead:** A FastAPI + uvicorn process with SQLite, even idle, typically consumes 30-60 MB of RAM. This is within budget but leaves little headroom. Avoid importing heavy libraries (pandas, numpy, etc.) unless absolutely needed.

2. **File watching overhead:** Watching multiple project directories recursively with `watchdog` can consume significant CPU and memory on large directory trees. Each watched project adds overhead. With 10+ projects registered, this could exceed the budget.
   - **Recommendation:** Do NOT recursively watch entire project directories. Only watch specific files: `STATE.json`, `REGISTRY.json`, and `config.json` within each project's `.planning/` directory. Use polling-based watching (compare mtime every 5 seconds) rather than inotify for cross-platform compatibility and lower resource usage.

3. **SQLite journal/WAL files:** With WAL mode enabled (which is recommended for concurrent read/write), SQLite creates `-wal` and `-shm` files alongside the database. The WAL file can grow unbounded if the service never checkpoints. Configure `PRAGMA wal_autocheckpoint = 1000;` and run periodic manual checkpoints during idle periods.

4. **Frontend build size:** Vite produces optimized bundles, but a knowledge graph visualization library (like D3, Cytoscape, or vis.js) can add 200-500 KB to the bundle. This is fine for a local service but should be lazy-loaded to keep initial page load fast.

5. **No hot-reload in production:** The systemd service should run uvicorn WITHOUT `--reload`. File watching for code changes in production wastes CPU and can cause unexpected restarts.

### 6. Configuration Management

**Why this matters for this project specifically:** The dashboard introduces a new configuration surface (`~/.rapid/`) that is separate from both the RAPID plugin config (`config.json` in plugin root) and per-project config (`.planning/config.json`). Users now have three places where RAPID-related configuration lives.

**Configuration locations after v4.0:**

| Location | Scope | Contents |
|----------|-------|----------|
| `{plugin-root}/config.json` | RAPID plugin global | `lock_timeout_ms`, `agent_size_warn_kb` |
| `{project}/.planning/config.json` | Per-project | `model`, `solo`, `max_parallel_sets` |
| `~/.rapid/config.toml` (new) | Dashboard global | `port`, `db_path`, `log_level`, `poll_interval` |
| `~/.rapid/.env` (new) | Dashboard secrets | `LANGFUSE_*` keys if observability is added |

**Overlooked questions:**

1. Should the dashboard port be changeable without editing a file? An env var `RAPID_PORT` would be more ergonomic than editing config.toml, but env vars do not persist across service restarts unless added to the unit file.

2. The `~/.rapid/` directory must be created during install. What permissions should it have? Default `0755` is fine for single-user, but if the directory contains an SQLite database with project paths, it could leak information on shared machines. Consider `0700`.

3. Configuration file format: The existing RAPID config uses JSON. The spec does not specify. TOML is more human-friendly for a config file that users might edit. But introducing a second format (TOML vs JSON) is a minor consistency concern. Recommendation: use JSON for consistency with the rest of RAPID, or TOML with a clear rationale documented.

### 7. Error Handling Strategy

**Why this matters for this project specifically:** The dashboard is a long-running background service, unlike the CLI which runs and exits. Error handling patterns that work for CLI tools (crash with error message, let user retry) do not work for background services where crashes mean silent unavailability.

**Error categories and handling:**

| Error Category | Example | CLI Approach | Dashboard Approach (Needed) |
|---------------|---------|-------------|---------------------------|
| Invalid project state | STATE.json fails Zod parse | Throw, exit 1 | Log warning, serve partial data, show error badge in UI |
| Filesystem unavailable | Project on unmounted drive | Throw, exit 1 | Mark project unavailable, continue serving others |
| Database corruption | SQLite integrity check fails | N/A (no DB) | Auto-backup, attempt recovery with `.recover`, fall back to fresh DB |
| Port conflict | Port 8998 in use | N/A | Fail to start with clear error in systemd journal, suggest port change |
| Unexpected exception | Unhandled error in endpoint | Crash | FastAPI exception handlers, log to `~/.rapid/logs/`, return 500 with error ID |

**Critical patterns for the dashboard:**

1. **Global exception handler:** FastAPI allows `@app.exception_handler(Exception)` to catch unhandled errors. Log full traceback to file, return sanitized error to client. Never expose file paths or system info in HTTP error responses.

2. **Per-project isolation:** A failure in reading one project's data must never affect other projects. Each project's data refresh should be wrapped in try/except independently.

3. **Structured logging:** Use Python's `logging` module with JSON formatter for machine-readable logs. Log to `~/.rapid/logs/dashboard.log` with rotation (max 10 MB, keep 3 files). This makes debugging straightforward without filling disk.

4. **Health check endpoint:** Expose `GET /health` that returns 200 with a JSON body containing uptime, registered project count, and last successful poll time. This enables monitoring and debugging.

### 8. SQLite Database Schema and Migration Strategy

**Why this matters for this project specifically:** SQLite databases evolve with the application. The dashboard will ship with a schema, and future RAPID versions will need to modify it. Without a migration strategy from day one, users upgrading from v4.0 to v4.1 will need to delete their database and lose all kanban/notes data.

**Recommended approach:**

1. **Schema version table:** Create a `schema_version` table with a single row tracking the current version number. Check this at startup.

2. **Migration files:** Store migrations as numbered Python files (`001_initial.py`, `002_add_notes_table.py`) or SQL files. Apply unapplied migrations at startup, in order.

3. **Backup before migration:** Before applying any migration, copy `rapid.db` to `rapid.db.backup.{timestamp}`. If migration fails, log error and continue with old schema (graceful degradation) or restore backup.

4. **Avoid ORM magic migrations:** Do NOT use Alembic or similar auto-migration tools for a simple SQLite database. Hand-written SQL migrations are more predictable and debuggable. The schema is small enough to manage manually.

5. **WAL mode at creation:** Set `PRAGMA journal_mode=WAL;` when creating the database. This allows concurrent reads while the dashboard writes. Also set `PRAGMA busy_timeout=5000;` to handle brief lock contention gracefully.

### 9. Testing Strategy

**Why this matters for this project specifically:** This is the first time RAPID has a web service component. The existing test infrastructure (Node.js `node:test` with `assert/strict`) does not apply to a Python FastAPI service. A parallel testing strategy must be established.

**Testing concerns:**

1. **Python test framework:** Use `pytest` with `pytest-asyncio` for async endpoint testing. FastAPI has built-in `TestClient` (backed by httpx) that makes API testing straightforward. Establish this in the initial set, not later.

2. **Frontend testing:** For a Vite SPA, use `vitest` for unit tests and optionally Playwright for E2E. Given the dashboard is read-only, E2E tests are lower priority than API tests.

3. **Integration test challenge:** Testing the dashboard reading from real `.planning/` directories requires creating fixture projects with known STATE.json content. Create a `tests/fixtures/` directory with sample project structures.

4. **Service lifecycle tests:** Test that the service starts, serves the health endpoint, and shuts down gracefully. Test startup with missing `~/.rapid/`, with corrupted DB, with unavailable projects. These must run in isolated temp directories.

5. **Cross-language test gap:** RAPID's existing Node.js tests verify CLI behavior. Dashboard Python tests verify API behavior. Nobody tests the integration between them (e.g., "CLI writes state, dashboard reads it correctly"). Consider a small set of integration tests that run both the CLI and dashboard against a shared fixture project.

### 10. Security Considerations

**Why this matters for this project specifically:** The dashboard binds to a network port (8998). Even though it binds to localhost, this introduces an attack surface that does not exist in the current CLI-only architecture.

**Specific concerns:**

1. **Bind address:** The spec says port 8998 but does not specify bind address. MUST bind to `127.0.0.1` (IPv4 localhost) or `::1` (IPv6 localhost), NOT `0.0.0.0`. Binding to all interfaces would expose project data to the local network.

2. **No authentication needed (but protect against CSRF):** Since this is a local-only service, authentication is not needed. However, a malicious website could issue requests to `localhost:8998` via JavaScript (CSRF). FastAPI should set CORS headers to reject requests from origins other than `http://localhost:8998` and `http://127.0.0.1:8998`.

3. **Path traversal in API:** If the API exposes endpoints like `GET /api/projects/{project_id}/files/{path}`, the `path` parameter must be sanitized to prevent reading arbitrary files. Since the dashboard is read-only, the risk is information disclosure, not modification.

4. **SQLite injection:** If using raw SQL queries (not an ORM), always use parameterized queries. Never interpolate user input into SQL strings.

5. **Project paths in responses:** API responses will contain filesystem paths (project locations). These are not secrets per se, but they reveal the user's directory structure. For a local-only service this is acceptable, but document the assumption that the service is never exposed to the network.

### 11. Developer Experience (Local Development)

**Why this matters for this project specifically:** RAPID is a developer tool built by developers. The development workflow for the dashboard must be smooth, or contributors will struggle. This is complicated by the mixed tech stack (Python backend + JavaScript frontend + Node.js CLI).

**Setup requirements for contributors:**

1. **Python environment:** Developers need Python 3.10+ with a virtualenv. The setup process must be documented. Should the dashboard have its own `requirements.txt` / `pyproject.toml`, or share the RAPID root `package.json`? Recommendation: separate `pyproject.toml` in a `dashboard/` subdirectory.

2. **Frontend development:** Vite dev server (`npm run dev`) needs to proxy API requests to FastAPI. Configure `vite.config.js` with a proxy for `/api/*` to `http://localhost:8998`. This allows hot-reload on the frontend while the backend runs separately.

3. **Simultaneous processes:** During development, a contributor needs to run: (a) FastAPI with `--reload`, (b) Vite dev server, and optionally (c) the RAPID CLI for testing registration. A `Makefile` or `justfile` with a `dev` target that starts both processes would help.

4. **Database reset:** During development, the schema will change frequently. Provide a `make reset-db` command that deletes `~/.rapid/rapid.db` and recreates it with the current schema.

5. **Hot reload caveat:** FastAPI's `--reload` flag uses `watchfiles` internally. If the dashboard source code lives inside the RAPID project directory, and the RAPID project is also a registered project in the dashboard, the file watcher could trigger infinite reload loops. Place the dashboard source outside the watched project directories or configure the reload watcher to exclude `.planning/` and `.rapid-worktrees/`.

### 12. Migration Path for Existing Users

**Why this matters for this project specifically:** RAPID already has users (the project itself has gone through v1.0 to v3.6.0 with many milestones). The dashboard must be purely opt-in and not break existing workflows.

**Migration concerns:**

1. **RAPID_WEB environment variable:** The spec says future commands detect the dashboard via `RAPID_WEB=true` in the user's shell config. The `/rapid:install` skill must be updated to offer dashboard installation as an optional step. This is a change to an existing skill, not just new code.

2. **Backward compatibility of /init:** The spec says `/init` should register with the dashboard if `RAPID_WEB=true`. This must be a non-blocking, best-effort operation. If the dashboard service is not running or the registration API returns an error, `/init` must proceed normally without failing. Add a 2-second timeout on the registration HTTP call.

3. **Old projects without registration:** The spec mentions a `/register_web` command for old projects. This should be a simple skill that sends the project path to the dashboard API. It does not need to modify the project's `.planning/` directory.

4. **Version detection:** When the dashboard reads a project's STATE.json, it should handle all schema versions gracefully. The `version` field in STATE.json is currently always `1`, but future versions may change. The dashboard should use the same Zod/validation approach as the CLI (or a Python equivalent like Pydantic) to validate and coerce project state.

5. **Uninstall path:** If a user decides they do not want the dashboard, they should be able to disable it cleanly: stop the systemd service, remove the unit file, remove `RAPID_WEB` from shell config, and optionally delete `~/.rapid/`. Provide a `/rapid:uninstall-web` command or add an option to `/rapid:install`.

### 13. Large Project Handling and UX at Scale

**Why this matters for this project specifically:** The current RAPID STATE.json for this project already has 13 milestones with dozens of sets. Some milestones have 10 sets. A production project could easily have 50+ sets across milestones, 30+ worktree entries, and hundreds of plan files. The knowledge graph for a large codebase could have thousands of nodes.

**Performance at scale:**

1. **API response size:** An endpoint that returns the full STATE.json for all projects could return megabytes of data if many projects are registered. Paginate project listings and return summaries (project name, current milestone, active set count) for the list view, with full details fetched per-project on demand.

2. **Knowledge graph rendering:** A knowledge graph with 1000+ nodes will overwhelm most browser-based graph visualization libraries. Implement server-side graph reduction: show only top-level modules by default, expand on click. Use WebGL-based rendering (e.g., Sigma.js or deck.gl) instead of SVG-based (D3 force layout) if node count exceeds 500.

3. **Kanban board with many items:** If notes and kanban items accumulate over months, the board could have hundreds of cards. Implement virtual scrolling and archiving for completed items.

4. **SQLite query performance:** With WAL mode and proper indexes, SQLite handles millions of rows efficiently. But without indexes on frequently queried columns (project_id, status, created_at), simple queries can degrade. Define indexes in the initial schema migration, not as an afterthought.

5. **Codebase mapping for large repos:** Analyzing a codebase with 10,000+ files takes time. This analysis must run in a background task, not in an API request handler. Use FastAPI's `BackgroundTasks` or a simple task queue. The frontend should poll for completion status, never wait on a long-running HTTP request (per CLAUDE.md instructions about long-running processes).

### 14. Logging and Observability

**Why this matters for this project specifically:** Per the user's CLAUDE.md, external AI API calls should use Langfuse for observability. While the dashboard may not make AI API calls initially, establishing an observability pattern from the start prevents costly retrofitting.

**Recommended approach:**

1. **Structured logging from day one:** Use Python `logging` with a JSON formatter. Include request_id, project_id, and operation in every log entry. Log to `~/.rapid/logs/dashboard.log`.

2. **Log rotation:** Use `logging.handlers.RotatingFileHandler` with `maxBytes=10*1024*1024` (10 MB) and `backupCount=3`. This prevents log files from filling disk.

3. **Request tracing:** Add middleware that assigns a UUID to each request and includes it in the response headers (`X-Request-ID`). This makes debugging specific requests straightforward.

4. **Metrics to collect from day one:**
   - Request count and latency per endpoint (use FastAPI middleware)
   - Number of registered projects and their availability status
   - SQLite database size
   - File watcher event count (to detect thrashing)
   - Service uptime

5. **Langfuse integration (deferred):** If AI features are added later (e.g., AI-powered project insights), Langfuse keys should be loaded from `~/.rapid/.env` as specified in the user's CLAUDE.md. Prepare the `.env.example` file with commented-out Langfuse variables now.

### 15. Browser Compatibility and Frontend Concerns

**Why this matters for this project specifically:** The spec mentions vim keybindings in the markdown editor, which have browser-specific behavior. The dashboard is served locally, so the user controls their browser, but the expected browser should be documented.

**Considerations:**

1. **Vite default targets:** Vite 7 targets Baseline Widely Available browsers (Chrome 111+, Edge 111+, Firefox 114+, Safari 16.4+). Since this is a local developer tool, this is more than sufficient. Do not add `@vitejs/plugin-legacy`.

2. **Vim keybindings in markdown editor:** Libraries like CodeMirror 6 with vim mode handle keybindings consistently across browsers. Avoid building custom keybinding handlers -- use an established library. CodeMirror 6 is the recommended choice for a markdown editor with vim support.

3. **WebSocket vs polling for real-time updates:** If the dashboard needs to show state changes in near-real-time, WebSocket is more efficient than polling. FastAPI supports WebSocket natively. However, for a read-only dashboard that refreshes every few seconds, simple polling with `fetch()` is simpler and sufficient. Start with polling; add WebSocket only if the polling approach causes noticeable lag or excessive requests.

4. **Offline/disconnected state:** If the user opens the dashboard and the backend service is stopped, the SPA should show a clear "Dashboard service is not running" message with instructions to start it, not a generic network error. Implement a frontend health check that polls `/health` and shows a connection status indicator.

### 16. Multi-User and Shared Machine Scenarios

**Why this matters for this project specifically:** On shared development machines (university labs, shared servers), multiple users might each install RAPID and the dashboard. Each user gets their own `~/.rapid/` directory and systemd user service, so there is no direct conflict. However:

1. **Port collision:** If two users both run the dashboard on port 8998, only one succeeds. Since these are per-user services binding to localhost, they do not conflict across users (each user gets their own loopback namespace... actually, no -- localhost port space is shared across all users on the same machine). The second user's service will fail to bind.
   - **Recommendation:** Make the port configurable and document the conflict scenario. Alternatively, use a port derived from the user's UID (e.g., `8998 + (uid % 1000)`) as a default, but this is fragile. Better to just make it configurable and fail with a clear error.

2. **`~/.rapid/` permissions:** Default directory creation uses the user's umask (typically 022, resulting in 755). This means other users can read the dashboard database, which contains project paths. Use `os.makedirs(mode=0o700)` to restrict access.

3. **Shared project directories:** Two users could register the same project directory. The dashboard should handle this gracefully -- it is just a read-only view. But if both users' dashboards poll the same STATE.json simultaneously, there is no contention since reads do not lock.

## Cross-Cutting Concern: Accessibility

Limited applicability. The dashboard is a developer tool used by technical users in their chosen browser. However, basic accessibility should not be ignored:

- Ensure keyboard navigation works for all interactive elements (kanban card movement, note editing).
- Use semantic HTML elements (`<nav>`, `<main>`, `<article>`) for screen reader compatibility.
- Ensure sufficient color contrast ratios (WCAG AA) for status badges and graph node colors.
- The markdown editor's vim mode is inherently keyboard-first, which is good for accessibility.

No specific WCAG compliance target is needed, but gross accessibility violations (e.g., all-mouse-only interactions, insufficient contrast) should be avoided.

## Cross-Cutting Concern: Internationalization

Not applicable. RAPID is an English-only developer tool. The dashboard UI should be in English with no i18n framework. If internationalization is ever needed, the cost to retrofit a Vite + React/Vue SPA is moderate (string externalization), but there is no indication this will be needed.

## Cross-Cutting Concern: CI/CD Pipeline

The dashboard introduces a Python component to a previously JavaScript-only project. CI needs to be extended:

1. **Python test step:** Add a `pytest` step to CI alongside the existing `node --test` step. This requires Python 3.10+ in the CI environment.
2. **Frontend build step:** Add a `vite build` step to verify the frontend compiles without errors.
3. **Lint step:** Add `ruff` (Python linter/formatter) and `eslint` (frontend JS/TS linter) to CI.
4. **No deployment pipeline needed:** The dashboard runs locally, not on a server. There is no deploy step.

## Recommendations

Ordered by retrofit cost and urgency:

1. **Data ownership boundaries (concern #1):** Decide BEFORE implementation where each data type lives. User-created data (kanban, notes) should live in `.planning/` for portability and version control; derived data (knowledge graph cache, codebase mapping) belongs in `~/.rapid/rapid.db`. Document this decision as an architecture decision record. The cost of moving data between storage locations after launch is extremely high because it requires both data migration and API changes.

2. **SQLite schema with versioned migrations (concern #8):** Ship v4.0.0 with a `schema_version` table and numbered migration files. Even if the initial schema is trivial, having the migration infrastructure from day one means future schema changes are painless. Without this, the first schema change after launch will require users to delete their database.

3. **Systemd/launchd service configuration (concern #4):** Design the service lifecycle before writing the install skill. Key decisions: enable linger automatically or warn the user? Support macOS launchd? Make the port configurable? These affect the install skill, the unit file template, and the server startup code. Ensure `EnvironmentFile=` is used in the unit file so `~/.rapid/.env` is loaded.

4. **Security defaults (concern #10):** Bind to `127.0.0.1` only. Set CORS to reject non-localhost origins. Use parameterized SQL queries. These are one-line configurations that prevent real vulnerabilities. Getting them wrong at launch and fixing later is embarrassing and potentially harmful.

5. **Error handling patterns (concern #7):** Establish the global exception handler, per-project isolation pattern, and health check endpoint in the first set. Every subsequent endpoint benefits from these patterns. Without them, error handling will be inconsistent across endpoints and difficult to fix later.

6. **Resource-aware architecture (concern #5):** Choose polling-based state sync over recursive file watching. Set a resource budget (< 50 MB idle RAM, < 0.1% idle CPU) and verify it with benchmarks before release. Switching sync mechanisms after launch requires rearchitecting the data layer.

7. **Structured logging (concern #14):** Configure Python `logging` with JSON formatter and rotation before writing any endpoints. Every subsequent `logger.info()` call automatically gets the right format. Retrofitting structured logging after 50 unstructured log statements is tedious busywork.

8. **API pagination from day one (concern #13):** Design list endpoints with `limit` and `offset` parameters from the start, even if v4.0.0 only has a few items. Adding pagination to an existing API is a breaking change for any frontend code consuming the unpaginated response.

9. **Long-running process pattern (concern #13, CLAUDE.md requirement):** Codebase analysis and knowledge graph generation must follow the pattern specified in the user's CLAUDE.md: backend immediately responds with a task ID, frontend polls for status. Never block an HTTP request for more than 10 seconds. Implement this pattern in the first background task; all subsequent tasks reuse it.

10. **Testing infrastructure (concern #9):** Set up `pytest`, `TestClient`, and fixture projects in the first set. Require tests for every endpoint from the start. The cross-language integration gap (Node.js CLI + Python dashboard) should have at least one smoke test that exercises the full path.
