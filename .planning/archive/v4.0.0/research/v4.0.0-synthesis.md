# Research Summary -- Mission Control Dashboard (v4.0.0)

## Project Profile
- **Type:** Greenfield web service (satellite companion to existing brownfield CLI plugin)
- **Stack:** Python 3.12+ (FastAPI + SQLModel + SQLite) backend, JavaScript (Vite + frontend framework) SPA, Node.js CLI integration via HTTP
- **Scale:** Large -- 8 features, 66 sub-features, dual runtime (Node.js + Python), full-stack web dashboard
- **Key Constraint:** The dashboard must be purely optional and read-only -- it must never block or modify the existing RAPID CLI workflow, and all CLI-to-dashboard communication must be non-blocking with graceful failure

## Unified Findings

### Stack and Infrastructure

- **Python 3.12+ as minimum, 3.14.3 on dev system.** FastAPI 0.135.x and SQLModel 0.0.37 both require Python >= 3.10. Targeting 3.12 balances modern features with compatibility. (source: STACK)
- **FastAPI 0.135.x as backend framework.** Auto-generated OpenAPI docs, Pydantic v2 integration, async-native, dependency injection for database sessions. The `lifespan` context manager must be used exclusively -- `@app.on_event("startup")` is deprecated and silently fails when `lifespan` is defined. (sources: STACK, ARCHITECTURE, PITFALLS)
- **SQLModel 0.0.37 (pre-1.0) for ORM.** Same author as FastAPI, combines SQLAlchemy ORM with Pydantic validation. Pre-1.0 versioning means API may shift; pin exact version. Fallback path to raw SQLAlchemy is straightforward. (sources: STACK, FEATURES, ARCHITECTURE)
- **SQLite at `~/.rapid/rapid.db` with WAL mode.** Single-file database, no server process needed. WAL mode enables concurrent reads while a single writer operates. Must set `PRAGMA journal_mode=WAL`, `PRAGMA busy_timeout=5000`, and `PRAGMA synchronous=NORMAL` at database creation. `render_as_batch=True` required for Alembic migrations due to SQLite `ALTER TABLE` limitations. (sources: STACK, ARCHITECTURE, PITFALLS, OVERSIGHTS)
- **Uvicorn 0.42.x as ASGI server.** Installed via `fastapi[standard]`. Will be the `ExecStart` target in the systemd unit. Must NOT use `--reload` in production. (sources: STACK, ARCHITECTURE)
- **Vite 8.0.x for frontend build tooling.** Uses Rolldown + Oxc internally for faster builds. Built-in dev proxy for routing API calls to FastAPI during development. (source: STACK)
- **Tailwind CSS 4.2.x with Everforest dark palette.** CSS-first config, OKLCH colors, fast incremental builds. Custom CSS variables map Everforest hex values to design tokens. (sources: STACK, UX)
- **Cytoscape.js for knowledge graph visualization.** Most mature graph library, framework-agnostic, built-in dagre layout for DAG visualization, handles 100K+ nodes. (sources: STACK, UX, FEATURES)
- **CodeMirror 6 for markdown editor.** Best-in-class code/markdown editor engine with vim keybinding support. Framework-specific wrappers available. (sources: STACK, UX)
- **Python venv at `~/.rapid/.venv/` for isolation.** Keeps Python dependencies separate from system. `uv` recommended for speed; fallback to pip+venv. (sources: STACK, ARCHITECTURE, OVERSIGHTS)
- **systemd user unit for service management on Linux.** No root/sudo needed. Must enable lingering (`loginctl enable-linger $USER`) for boot start. macOS requires launchd plist equivalent. (sources: STACK, ARCHITECTURE, PITFALLS, OVERSIGHTS)
- **Two-runtime architecture (Node.js + Python) with HTTP-only communication.** Node.js CLI calls `localhost:8998/api/...` to register projects. No shared database access, no shared process state. Registration must fail silently if service is unreachable. (sources: STACK, ARCHITECTURE, PITFALLS)
- **Langfuse integration should be optional.** Only load when AI features are active; do not require Langfuse keys for the basic dashboard. Prepare `.env.example` with commented-out Langfuse variables. (sources: STACK, OVERSIGHTS)
- **ruff for Python linting/formatting.** Single Rust-based tool replacing flake8+black+isort. (source: STACK)

### Feature Landscape

**Feature groups with dependencies:**

```
Service Management (foundational, independent)
  +-- Project Registry (requires running web service)
        +-- Project State View (reads STATE.json)
        +-- Worktree Tracking (reads REGISTRY.json)
        +-- Knowledge Graph (reads DAG.json, uses graph visualization)
        +-- Kanban Board (own data model, project-scoped)
        +-- Markdown Note Editor (own data model, project-scoped)
        +-- Codebase Mapping (reuses graph infrastructure, tree-sitter analysis)
```

(source: FEATURES)

**Recommended implementation order:**

1. Service Management -- foundational infrastructure (FastAPI scaffold, SQLite init, systemd unit, install flow)
2. Project Registry -- core data backbone; all other features depend on it
3. Project State View -- simplest read-only view, validates the full stack end-to-end
4. Worktree Tracking -- similar pattern to State View, adds git subprocess enrichment
5. Knowledge Graph -- introduces graph visualization infrastructure reused by Codebase Mapping
6. Kanban Board -- complex but self-contained, can partially parallelize with Knowledge Graph
7. Markdown Note Editor -- straightforward CRUD, lower priority since users already have editors
8. Codebase Mapping -- most complex feature, requires tree-sitter, background task pattern, and graph reuse

(source: FEATURES)

**Third-party service dependencies:** All dependencies are MIT/BSD-licensed, free, and self-hosted. No external paid services required. tree-sitter-language-pack is the heaviest dependency (native binary, 170+ language parsers). (source: FEATURES)

### Architecture Direction

- **Pattern: Satellite service with shared data contracts.** The dashboard is architecturally independent from the RAPID CLI plugin -- separate process, separate tech stack, separate database, separate lifecycle. Communication through REST API (project registration) and filesystem reads (`.planning/` artifacts). This mirrors Grafana (separate dashboard reading data sources) and Docker Desktop (separate GUI wrapping CLI). (source: ARCHITECTURE)
- **Module boundaries:** Project Registry, State Reader, File Watcher, Kanban, Notes, Knowledge Graph, Codebase Mapper, and Frontend Shell are distinct modules with explicit responsibilities and data ownership. (source: ARCHITECTURE)
- **Data flow:** Projects register via HTTP POST from CLI. File watcher (using `watchfiles`, Rust-based) monitors `.planning/` files and updates SQLite cache. Frontend polls API (5s interval) with optional WebSocket for active views. State is eventually consistent. (source: ARCHITECTURE)
- **State management:** Server-side state in SQLite (persistent), filesystem reads for RAPID project data (cached in SQLite), client-side state in Zustand or similar (ephemeral). (source: ARCHITECTURE)
- **API design:** REST with `/api/` prefix, plural noun resources, cursor-based pagination, JSON envelope responses. WebSocket at `/ws/projects/{id}/updates` for real-time push. (source: ARCHITECTURE)
- **Security:** Bind to `127.0.0.1` only, localhost-only middleware as defense-in-depth, no authentication for v4.0 (local-only single-user tool), CORS restricted to localhost origins, parameterized SQL queries only. (sources: ARCHITECTURE, PITFALLS, OVERSIGHTS)
- **Dashboard source in `dashboard/` at repo root** with nested `dashboard/dashboard/` Python package and `dashboard/frontend/` Vite project. Runtime state at `~/.rapid/`. (source: ARCHITECTURE)

### Cross-Cutting Requirements

**Must address early (high retrofit cost):**
- Data ownership boundaries -- deciding where each data type lives (dashboard DB vs `.planning/`) before writing any code. Moving data between storage locations after launch requires both data migration and API changes. (source: OVERSIGHTS)
- SQLite schema with versioned migrations from day one -- shipping without migration infrastructure means the first schema change forces users to delete their database. (sources: OVERSIGHTS, ARCHITECTURE)
- Error handling patterns -- global exception handler, per-project isolation, health check endpoint established in the first set. Without them, error handling is inconsistent across all endpoints. (source: OVERSIGHTS)
- Security defaults -- `127.0.0.1` bind address, CORS restriction, parameterized queries. Getting these wrong at launch is both embarrassing and harmful. (sources: OVERSIGHTS, PITFALLS)
- Resource-aware architecture -- choosing polling-based state sync over recursive file watching. Switching sync mechanisms after launch requires rearchitecting the data layer. Target: < 50 MB idle RAM, < 0.1% idle CPU. (source: OVERSIGHTS)
- Structured logging with JSON formatter and rotation before writing any endpoints. (source: OVERSIGHTS)
- API pagination (`limit`/`offset` or cursor) from day one -- adding pagination to existing endpoints is a breaking API change. (source: OVERSIGHTS)
- Long-running process pattern per CLAUDE.md -- backend immediately responds with task ID, frontend polls for status. Never block HTTP requests for > 10 seconds. (sources: OVERSIGHTS, FEATURES)

**Can address later (low retrofit cost):**
- Developer experience tooling (Makefile, proxy config, database reset) -- can be added incrementally. (source: OVERSIGHTS)
- Migration path for existing users (`/register_web` command, backward-compatible `/init`) -- can be done independently. (source: OVERSIGHTS)
- Browser compatibility -- Vite defaults are sufficient. (source: OVERSIGHTS)
- Multi-user/shared machine port collision -- simple configuration option. (source: OVERSIGHTS)
- macOS launchd support -- can be added after systemd is working. (sources: OVERSIGHTS, ARCHITECTURE)
- Internationalization -- not applicable for an English-only developer tool. (source: OVERSIGHTS)

### User Experience Direction

- **Command palette (Ctrl+K and `:` triggers) is the single most expected UX pattern.** De facto standard across VS Code, Linear, Raycast, Notion, Slack, and GitHub. Must show recent items, fuzzy search, and shortcut hints. (source: UX)
- **Everforest dark theme as default.** Complete palette with surface elevation model (lighter = higher). bg_dim (#232A2E) for sidebar, bg0 (#2D353B) for main content, bg1 (#343F44) for cards/panels. Foreground #D3C6AA (warm cream) passes WCAG AAA at 8.2:1 contrast. Light theme toggle available but dark is default. (source: UX)
- **Vim-style J/K/H/L navigation with mode indicator.** Follow Linear's proven model. Normal mode (default) for single-key navigation, Insert mode for text inputs, Command mode via `:` or Ctrl+K. Mode indicator badge in status bar. Must disable shortcuts when focus is in input/textarea elements. (sources: UX, PITFALLS)
- **Collapsible sidebar navigation (three states: open, icon-only, docked).** Follow Grafana's pattern. Contains project selector with status dots and view navigation. `Ctrl+B` to toggle. (source: UX)
- **Information architecture: three-level progressive disclosure.** Overview (project list with summaries) -> Project (dashboard with set status, access to kanban/graph/notes) -> Detail (individual set details, note content). Use expandable sections and slide-out panels rather than full page navigations where possible. (source: UX)
- **Keyboard shortcut overlay on `?` key.** GitHub's convention, organized by category with `<kbd>` styling. Progressive discovery: passive tooltips -> on-demand cheatsheet -> first-time banner -> contextual tips (max one per session). (source: UX)
- **Status visualization with Everforest semantic colors.** Merged=green (#A7C080), Complete=aqua (#83C092), Executing=yellow (#DBBC7F), Planning/Discussing=blue (#7FBBB3), Pending=grey (#7A8478), Blocked=red (#E67E80). Multi-segment progress bars for glanceable project status. Never rely solely on color -- pair with icons or text labels. (source: UX)
- **Accessibility baseline.** WCAG AA contrast ratios (Everforest palette passes for primary text and accent colors). Semantic HTML (`<nav>`, `<main>`, `<aside>`). ARIA live regions for dynamic updates. Visible focus indicators (aqua #83C092 outline). Support `prefers-reduced-motion` and `prefers-color-scheme`. Provide vim-mode disable option for assistive technology users. (source: UX)
- **Typography.** System sans-serif for headings/body, monospace (JetBrains Mono, Fira Code, or system) for code/technical content. (source: UX)
- **Kanban board as read-only status view initially.** Columns map to RAPID lifecycle statuses (Pending, Discussing, Planning, Executing, Complete, Merged). No drag-and-drop needed for v1 since the dashboard is read-only. (sources: UX, FEATURES)
- **Markdown editor with split-pane and vim keybindings.** Source/preview split with synchronized scrolling, auto-save (debounced 1-2s), Notion-style slash commands. (source: UX)
- **Persist UI state in localStorage.** Panel sizes, sidebar state, selected project, last viewed section, dismissed tips. (source: UX)

## Contradictions

### Contradiction: Frontend Framework Choice (Svelte 5 vs React 19)

- **STACK says:** Svelte 5 is the better fit. Smaller bundle (~1.6 KB vs ~42 KB), simpler component model, compiler-based approach aligns with a lightweight always-on local service. The ecosystem gap is manageable because this is not a complex enterprise app.
- **ARCHITECTURE says:** React is the stronger choice. Broader component library availability for dashboards (charts, kanban, graph visualization, markdown editors). More widely known for future contributions. Specific needs like react-force-graph, dnd-kit, and MDXEditor have mature React implementations.
- **FEATURES says:** Recommends React-specific libraries throughout (React Flow + dagre for knowledge graph, dnd-kit for kanban, @uiw/react-md-editor for notes, TanStack Query for data fetching, React Router for navigation, Zustand for state management).
- **UX says:** References framework-agnostic solutions (Cytoscape.js, CodeMirror 6) alongside both React and Svelte options. Recommendations are framework-neutral.
- **Resolution:** The FEATURES and ARCHITECTURE research both recommend React and have selected React-specific libraries for every major component. The STACK research acknowledges React is "fully viable" and recommends Svelte only slightly. Given the library ecosystem alignment and the number of features requiring component libraries, **React 19 is the pragmatic choice** unless the team specifically values Svelte's smaller bundle size over ecosystem breadth. This needs a user decision.

### Contradiction: File Watching Library

- **ARCHITECTURE says:** Use `watchfiles` (Rust-based, uses inotify on Linux, FSEvents on macOS). Async-compatible, cross-platform.
- **PITFALLS says:** Documents memory leaks and inotify exhaustion in `watchdog`, recommends `PollingObserver` as fallback, or custom `inotify_simple` wrapper.
- **OVERSIGHTS says:** Use polling-based watching (compare mtime every 5 seconds) as primary mechanism, filesystem watching as optimization layer. Recommends `watchfiles` over `watchdog` if watching is used.
- **Resolution:** No real contradiction -- all three agree `watchfiles` is preferred over `watchdog`. The recommendation converges on: use `watchfiles` for specific file watches (not recursive directories), with polling as fallback. OVERSIGHTS' suggestion to make polling the primary mechanism is the most conservative and resource-friendly approach.

### Contradiction: Alembic vs Hand-Written Migrations

- **STACK says:** Use Alembic with `render_as_batch=True` for SQLite migrations. Well-documented with SQLModel.
- **OVERSIGHTS says:** "Do NOT use Alembic or similar auto-migration tools for a simple SQLite database. Hand-written SQL migrations are more predictable and debuggable."
- **ARCHITECTURE says:** Use Alembic for schema migrations from day one.
- **Resolution:** STACK and ARCHITECTURE agree on Alembic; OVERSIGHTS disagrees. For a project with 9+ tables that will evolve across versions, Alembic provides significant value (auto-generating diffs, rollback support, batch mode for SQLite). Hand-written migrations are simpler but do not scale. **Alembic is recommended**, but migration files should be reviewed and edited rather than auto-applied blindly.

### Contradiction: State Management Library

- **ARCHITECTURE says:** Use Zustand for client-side state management. Lightweight, no boilerplate, scales well.
- **FEATURES says:** Use TanStack Query for server state (polling, caching, background refetch). React context for UI state.
- **Resolution:** These are complementary, not conflicting. TanStack Query handles server data fetching/caching, Zustand handles client-side UI state. Both can coexist. Use TanStack Query for API data and Zustand for navigation/UI state.

### Contradiction: Kanban Data Model Location

- **FEATURES says:** Kanban data (boards, columns, cards) stored in `~/.rapid/rapid.db` with its own SQLite tables.
- **OVERSIGHTS says:** User-created data (kanban items, notes) should live in `.planning/kanban/` within each project for portability and version control. Dashboard DB holds only an index/cache.
- **Resolution:** This is a genuine architectural tension. Storing in `.planning/` enables version control and portability but requires the RAPID CLI to understand kanban schemas. Storing in `~/.rapid/rapid.db` is simpler but data is lost on project re-clone. **This needs a user decision** before implementation. The OVERSIGHTS recommendation is architecturally cleaner but adds complexity to the CLI side.

## Recommended Approach

The Mission Control dashboard is a greenfield web service that sits alongside the existing RAPID CLI plugin as a read-only companion. It introduces a second runtime (Python) and a second persistence layer (SQLite) to an established Node.js/CommonJS codebase. The core architectural insight from all six research files is that clean separation is essential: the dashboard must be a true satellite service that communicates with the CLI exclusively via HTTP and reads project data from the filesystem without participating in the CLI's file-locking protocol.

Two decisions must be made before any code is written. First, the frontend framework: React 19 has stronger library ecosystem support for the specific components needed (graph visualization, kanban drag-and-drop, markdown editing), while Svelte 5 offers a smaller footprint more befitting a lightweight always-on local service. The research leans React based on feature implementation needs, but this should be an explicit team decision. Second, data ownership: where user-created data (kanban boards, notes) canonically lives. The OVERSIGHTS research argues convincingly that user-created data should live in `.planning/` within each project for portability and version control, while the FEATURES and ARCHITECTURE research assume it lives in the dashboard's SQLite database for simplicity. This decision affects every data model and API design.

Implementation should proceed in a layered approach. The first layer is pure infrastructure: FastAPI scaffold with security defaults (127.0.0.1 binding, CORS, error handling, health check), SQLite database with WAL mode and Alembic migrations, systemd user unit, and the install flow. The second layer is the data backbone: Project Registry with CLI registration hook, file watcher for `.planning/` artifacts, and the State Reader service that parses RAPID's Zod-validated JSON into Pydantic models. The third layer is the frontend shell: application skeleton with Everforest theme, sidebar navigation, routing, and the command palette. The fourth layer is feature implementation: State View and Worktree Tracking first (simple read-only views that validate the full stack), then Knowledge Graph and Kanban (complex interactive views), then Notes, and finally Codebase Mapping as a stretch goal.

Resource constraints are important: this is a background developer tool that must idle below 50 MB RAM and 0.1% CPU. File watching must be targeted (specific files, not recursive directories) to avoid inotify exhaustion and memory leaks. All long-running operations (codebase analysis, knowledge graph generation) must follow the task-ID-plus-polling pattern required by the project's CLAUDE.md. The dashboard must degrade gracefully when projects become unavailable, when STATE.json is malformed, or when the service itself is unreachable from the CLI.

The dual-runtime installation complexity is the highest practical risk. Users who installed RAPID for CLI functionality now face Python dependency requirements. Mitigation is strong: the dashboard is entirely opt-in via `RAPID_WEB=true`, Python setup is bundled in the install script using `uv`, and a `rapid-dashboard doctor` command validates all prerequisites. The install flow should create `~/.rapid/` with `0700` permissions, set up the venv, initialize the database, generate the systemd unit, and add `RAPID_WEB=true` to shell config -- all as an optional step that can be skipped entirely.

### Suggested Set Boundaries

- **Set A: Infrastructure & Service Shell** -- FastAPI scaffold, SQLite setup with WAL and migrations, systemd unit, install flow, health check, structured logging, security defaults. Rationale: foundational infrastructure that everything else depends on. No frontend work, pure backend and ops.

- **Set B: Project Registry & Data Pipeline** -- Registration endpoint, `register-web` CLI command, init hook integration, State Reader service, file watcher with polling fallback, project snapshot caching. Rationale: the data backbone that all features consume. Bridges the CLI and dashboard.

- **Set D: Read-Only Views** -- Project State View (milestone/set/wave hierarchy with progress bars), Worktree Tracking view (REGISTRY.json reader with git status enrichment), Project Dashboard (summary cards). Rationale: simplest features that validate the full stack end-to-end.

- **Set E: Interactive Features** -- Knowledge Graph (Cytoscape.js/React Flow with dagre layout), Kanban Board (CRUD + drag-and-drop or read-only status mapping), Markdown Note Editor (CodeMirror 6 with vim mode, auto-save, tagging). Rationale: complex UI features that can be built in parallel once the shell and data pipeline are stable.

- **Set F: Advanced Analysis** -- Codebase Mapping (tree-sitter parsing, background task queue, module dependency graph, file treemap visualization). Rationale: most complex feature, benefits from all preceding infrastructure being stable. Stretch goal for v4.0.
