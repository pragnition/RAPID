# Features Research -- v6.3.0 Mission Control & Fixes

## Feature Inventory

### Feature 1: Rapid Mission Control Code Graph

- **Description:** Replace the current DAG-based "Knowledge Graph" page in Mission Control with a full codebase graph. The new graph should visualize actual source files, their relationships (imports/exports, module dependencies), and allow click-to-view file content. Modern, sleek design with interactive navigation.
- **Complexity:** very-complex
- **Dependencies:** Independent (builds on existing Mission Control web infrastructure)

#### Current State Analysis

The Mission Control web dashboard currently has two related pages:

1. **Knowledge Graph page** (`web/frontend/src/pages/KnowledgeGraphPage.tsx`) -- Uses Cytoscape.js with cytoscape-dagre layout to render the RAPID set dependency DAG (`DagGraph`). Nodes represent sets, edges represent set dependencies. Clicking a node shows set metadata (status, wave, dependencies). This is a project-management graph, not a code graph.

2. **Codebase page** (`web/frontend/src/pages/CodebasePage.tsx`) -- A flat file tree with tree-sitter symbol extraction. Shows files with expandable symbols (functions, classes, methods). No graph visualization, no inter-file relationships, no file content viewing.

The backend already has:
- `web/backend/app/services/codebase_service.py` -- tree-sitter parsing service that extracts symbols from Python, JavaScript/TypeScript, Go, and Rust files. Has mtime-based caching.
- `GET /api/projects/{id}/codebase` endpoint returning `CodebaseTree` (files, symbols, languages).
- `GET /api/projects/{id}/dag` endpoint returning `DagGraph` (nodes, edges, waves).

#### Decomposition

1. **Backend: Code dependency graph extraction** -- New service that builds an import/export dependency graph between files. Parse `import`/`require`/`from...import` statements to create edges between files.
2. **Backend: File content API** -- New endpoint `GET /api/projects/{id}/files/{path}` that returns file content (with syntax highlighting metadata or raw text). Must sanitize path to prevent directory traversal.
3. **Backend: Code graph API** -- New endpoint `GET /api/projects/{id}/code-graph` returning a graph structure with file nodes and dependency edges, optionally filtered by language or directory.
4. **Frontend: Code graph visualization** -- New page component replacing/augmenting KnowledgeGraphPage. Interactive graph with file nodes colored by language, dependency edges, zoom/pan, and layout controls.
5. **Frontend: Click-to-view file panel** -- Side panel or modal that displays file content when clicking a node. Syntax-highlighted code view with line numbers. Should show the file's symbols in a sidebar outline.
6. **Frontend: Graph styling and UX** -- Modern design with smooth animations, search/filter, minimap, responsive layout. Dark theme integration with existing theme system.

#### Implementation Strategies

**Option 1: Cytoscape.js enhancement (extend existing)**
- Approach: Keep Cytoscape.js (already in use for DAG graph) but build a new graph data source. Add a new backend service to extract import/require relationships via tree-sitter or regex. Render file nodes instead of set nodes.
- Pros: No new graph library needed; team already has Cytoscape patterns established; layout algorithms (dagre, breadthfirst) already working.
- Cons: Cytoscape.js can struggle with large graphs (500+ nodes); styling options are more limited than modern alternatives; no built-in file content rendering.
- Libraries/services: cytoscape (already installed, v3.33.1), cytoscape-dagre (already installed), plus potentially cytoscape-fcose for force-directed layout.

**Option 2: React Flow (modern graph library)**
- Approach: Replace Cytoscape.js with React Flow for the code graph page. React Flow provides native React node rendering (HTML/CSS inside nodes), built-in minimap, controls, and smooth animations. Keep Cytoscape for the existing DAG page.
- Pros: Native React integration; each node is a React component (can embed file previews, language badges, symbol counts directly in nodes); built-in minimap and controls; better performance with virtualization; excellent TypeScript support; large community.
- Cons: New dependency (~50KB gzipped); learning curve for the team; two graph libraries in the bundle.
- Libraries/services: @xyflow/react (React Flow v12), @xyflow/minimap, @xyflow/controls, @xyflow/background.

**Option 3: D3.js force-directed graph**
- Approach: Build a custom force-directed graph using D3.js. Full control over every visual aspect.
- Pros: Maximum flexibility; no opinionated layout constraints; widely known.
- Cons: Significantly more implementation effort; must build zoom/pan/minimap from scratch; poor React integration (imperative DOM manipulation vs declarative); hard to maintain.
- Libraries/services: d3, d3-force.

**Recommended:** Option 2 (React Flow). The ability to render React components inside nodes is a major advantage for a code graph -- file nodes can show language badges, symbol counts, and file previews directly. The built-in minimap and controls eliminate significant development effort. React Flow is purpose-built for interactive node graphs with modern web aesthetics.

#### Data Model Impact

- New entities/tables needed: None in the database. The code graph is derived dynamically from file system analysis (similar to existing codebase_service.py).
- Existing entities modified: None.
- New backend data structures:
  - `CodeGraphNode` -- `{ id: string, path: string, language: string, symbol_count: number, line_count: number }`
  - `CodeGraphEdge` -- `{ source: string, target: string, type: "import"|"require"|"from", symbols: string[] }`
  - `CodeGraph` -- `{ nodes: CodeGraphNode[], edges: CodeGraphEdge[], languages: string[], total_files: number }`

#### API Surface

- New endpoints:
  - `GET /api/projects/{id}/code-graph` -- Returns `CodeGraph` with file dependency nodes and edges
  - `GET /api/projects/{id}/code-graph?language=python&dir=src/` -- Filtered variant
  - `GET /api/projects/{id}/files?path=src/lib/state.cjs` -- Returns file content with metadata (language, line count, symbols)
- Existing endpoints modified:
  - `GET /api/projects/{id}/codebase` -- May need expansion to include import/dependency data alongside symbols

### Feature 2: Multi-agent Bug-Fix Support

- **Description:** Extend the `/rapid:bug-fix` orchestrator to support splitting bug investigation and fixing across multiple waves. Each wave spawns one bugfix agent sequentially, reducing context rot by giving each agent a fresh context window focused on a subset of bugs.
- **Complexity:** moderate
- **Dependencies:** Independent (modifies existing bug-fix skill only)

#### Current State Analysis

The current `/rapid:bug-fix` skill (`skills/bug-fix/SKILL.md`) has two modes:

1. **Normal mode (Steps 1-5):** User describes a bug, the skill investigates, presents findings, then dispatches a single `rapid-executor` agent to apply the fix.

2. **UAT mode (`--uat`, Steps UAT-a through UAT-Results):** Reads `UAT-FAILURES.md`, sorts failures by severity, and dispatches one `rapid-executor` per failure sequentially. This is already a multi-dispatch pattern but processes all failures in a single orchestrator context window.

The problem with both modes when there are many bugs: the orchestrator accumulates context from each executor dispatch (investigation + fix + RAPID:RETURN parsing). After 5+ bugs, the orchestrator's context window fills up, leading to "context rot" -- degraded quality in later fixes.

The `rapid-bugfix` agent role (`src/modules/roles/role-bugfix.md`) is a leaf agent that fixes accepted bugs from the review pipeline. It is distinct from the `/rapid:bug-fix` skill -- the skill is user-facing, the agent is pipeline-internal.

#### Decomposition

1. **Wave splitting logic in SKILL.md** -- Add a wave decomposition step that groups bugs/failures into waves of configurable size (default 3-5 per wave). Each wave gets its own agent spawn.
2. **Wave-sequential dispatch** -- Execute waves sequentially: spawn agent for wave 1, collect results, spawn agent for wave 2 with fresh context plus a summary of wave 1 results, etc.
3. **Cross-wave state tracking** -- Maintain a running tally of fixed/blocked/remaining bugs across waves. Pass a minimal summary (not full context) between waves.
4. **Configurable wave size** -- Add `--wave-size <n>` flag to control how many bugs per wave (default: 3 for normal bugs, 5 for UAT failures).
5. **Results aggregation** -- Combine results from all waves into the existing results display format.

#### Implementation Strategies

**Option 1: SKILL.md orchestration (extend existing pattern)**
- Approach: Modify `skills/bug-fix/SKILL.md` to add wave-splitting logic between bug sorting (Step UAT-b) and dispatch (Step UAT-d). The SKILL.md orchestrator groups failures into waves, dispatches one executor per wave with a condensed context summary from prior waves.
- Pros: No new code files needed; follows existing SKILL.md orchestration pattern; the skill already has sequential dispatch infrastructure (UAT mode); minimal code surface change.
- Cons: SKILL.md is already 300 lines; adding wave logic increases complexity; skill-level orchestration has limited programmatic control.
- Libraries/services: None new.

**Option 2: CLI command + library module**
- Approach: Add a `src/lib/bugfix-waves.cjs` module that handles wave decomposition, context summarization, and result aggregation. The SKILL.md calls the CLI to compute waves, then dispatches per wave. This separates orchestration logic from prompt text.
- Pros: Testable wave logic (unit tests for grouping, summarization); cleaner separation of concerns; reusable for other multi-wave patterns.
- Cons: More files to create; adds a library module for what is currently a skill-only feature; the skill still needs to handle agent dispatch.
- Libraries/services: None new.

**Recommended:** Option 1 (SKILL.md orchestration). The wave splitting is fundamentally an orchestration concern that belongs in the skill definition. The logic is straightforward (chunk an array, iterate chunks, pass summaries). Adding a library module creates unnecessary indirection for what is essentially: `waves = chunk(sortedBugs, waveSize); for each wave: spawn(executor, wave, priorSummary)`.

#### Data Model Impact

- New entities/tables needed: None.
- Existing entities modified: None.
- No state machine changes -- bug-fix operates outside the set lifecycle.

#### API Surface

- New endpoints/commands needed: None for backend API.
- Existing CLI surface changes:
  - `/rapid:bug-fix --uat <set-id>` gains optional `--wave-size <n>` flag
  - `/rapid:bug-fix` normal mode gains optional `--wave-size <n>` flag for multi-bug scenarios
- No RAPID CLI command changes needed (bug-fix does not use rapid-tools.cjs commands).

### Bug Fix 1: Shell Config Update Incomplete

- **Description:** When `RAPID_TOOLS` env var is already configured in multiple shell config files (e.g., both `~/.bashrc` and `~/.config/fish/config.fish`), the install skill's detection loop (`for f in ~/.bashrc ...`) only finds and reports the first match. The agent then only updates one config file, leaving the other stale.
- **Complexity:** trivial
- **Dependencies:** Independent

#### Decomposition

1. **Fix detection loop to collect ALL matches** -- Change the `for f in ...` loop in SKILL.md Step 2 from `break` on first match to collecting all matches.
2. **Present all matches to user** -- Show all config files that contain `RAPID_TOOLS` and let the user choose which to update (or update all).
3. **Update all selected config files** -- In Step 3, write the new export line to each selected config file, not just one.

#### Implementation Strategies

**Recommended approach:**
- Modify `skills/install/SKILL.md` Step 2: Replace the `break` after first match with continued accumulation. Change `ALREADY_CONFIGURED=$f` + `break` to append all matching files.
- In the AskUserQuestion for "RAPID_TOOLS already configured", show all matching files and offer "Update all" as the primary option.
- In Step 3, iterate over all selected config files for the write operation.

#### Data Model Impact
- None.

#### API Surface
- No changes. This is a SKILL.md-only modification.

### Bug Fix 2: Kanban Board Save Shortcut

- **Description:** When editing card details in the Kanban board modal, Ctrl+click (or Ctrl+Enter) should save the card instead of requiring the mouse to click the Save button.
- **Complexity:** trivial
- **Dependencies:** Independent

#### Decomposition

1. **Add Ctrl+Enter keyboard shortcut to CardDetailModal** -- Listen for Ctrl+Enter (or Cmd+Enter on macOS) in the modal and trigger `handleSave`.

#### Implementation Strategies

**Recommended approach:**
- Modify `web/frontend/src/components/kanban/CardDetailModal.tsx`: Extend the existing `handleKeyDown` callback (which already listens for Escape) to also handle `Ctrl+Enter` / `Meta+Enter` to trigger `handleSave()`.
- The current `handleKeyDown` is: `if (e.key === "Escape") { onClose(); }`. Add: `if ((e.ctrlKey || e.metaKey) && e.key === "Enter") { handleSave(); }`.
- This is a ~3 line change in a single file.

#### Data Model Impact
- None.

#### API Surface
- No changes. Frontend-only modification.

### Bug Fix 3: Mission Control RAPID Version Outdated

- **Description:** The sidebar in Mission Control displays "RAPID v4.2.1" (hardcoded in `Sidebar.tsx`) instead of the actual current version. The backend `__version__` in `app/__init__.py` is also stuck at `"4.2.1"`.
- **Complexity:** trivial
- **Dependencies:** Independent

#### Decomposition

1. **Update backend version** -- Change `web/backend/app/__init__.py` from `__version__ = "4.2.1"` to dynamically read from `package.json` or update to `"6.3.0"`.
2. **Make frontend version dynamic** -- Replace the hardcoded "RAPID v4.2.1" in `Sidebar.tsx` with a value fetched from the `/api/health` endpoint (which already returns `version`).

#### Implementation Strategies

**Option 1: Dynamic version from health API (recommended)**
- Approach: The `/api/health` endpoint already returns the version. The frontend already has the `HealthResponse` type defined in `types/api.ts`. Add a `useHealth` hook that fetches `/api/health` and extract the version. In `Sidebar.tsx`, replace the hardcoded string with the fetched version.
- For the backend, either: (a) read version from the root `package.json` at startup, or (b) hardcode but ensure it is updated during the release process. Option (a) is more maintainable.
- Pros: Version always matches the running backend; single source of truth.
- Cons: Requires the backend to be running to show version (but Mission Control requires it anyway).

**Option 2: Build-time version injection**
- Approach: Inject the version at frontend build time via Vite's `define` config, reading from `package.json`.
- Pros: No runtime API call needed.
- Cons: Version reflects build time, not running backend version; could mismatch if backend is newer.

**Recommended:** Option 1 (Dynamic from health API). The backend must already be running for Mission Control to function. The health endpoint is already available and returns the version. This ensures the displayed version always matches the running service.

Additionally, the backend `__init__.py` should be updated to read from `package.json` or at minimum be set to `"6.3.0"` for this release. The recommended long-term fix is to have the backend read version from the root `package.json` to maintain a single source of truth.

#### Data Model Impact
- None.

#### API Surface
- No new endpoints. The existing `/api/health` endpoint already returns version.
- Frontend adds a new `useHealth()` hook.

## Implementation Order

1. **Bug Fix 3 (Version display):** Rationale -- Trivial change, immediate user-visible improvement, unblocks accurate version reporting across the dashboard.
2. **Bug Fix 2 (Kanban save shortcut):** Rationale -- Trivial UX improvement, single-file ~3-line change.
3. **Bug Fix 1 (Shell config update):** Rationale -- Trivial but slightly more nuanced (SKILL.md modification with user-facing prompt changes). Fixes a real installation friction point.
4. **Feature 2 (Multi-agent Bug-Fix):** Rationale -- Moderate complexity, high value for reducing context rot in bug-fix workflows. No frontend changes needed.
5. **Feature 1 (Code Graph):** Rationale -- Very complex, requires coordinated backend + frontend work across multiple sub-features. Highest effort but also highest visual impact.
