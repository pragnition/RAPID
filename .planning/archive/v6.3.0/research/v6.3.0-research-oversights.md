# Oversights Research -- v6.3.0 Mission Control & Fixes

## Cross-Cutting Concerns

### Version Synchronization Strategy

The milestone has a version bug (Mission Control showing "v4.2.1") but the problem is deeper than a single hardcoded string. There are currently **three independent version sources** that must stay in sync:

1. **Node.js plugin:** `package.json` `"version": "6.2.0"` -- read by `src/lib/version.cjs` via `getVersion()`
2. **Python backend:** `web/backend/pyproject.toml` `version = "4.2.1"` -- read by `app/__init__.py` (`__version__ = "4.2.1"`)
3. **Frontend sidebar:** `web/frontend/src/components/layout/Sidebar.tsx` -- hardcoded strings `"RAPID v4.2.1"` and `"v4"`

The backend and frontend versions were never updated when the plugin moved from v4.2.1 to v6.x. Fixing the version display in the sidebar is Bug #3, but the **oversight is that there is no single source of truth mechanism** to prevent this from happening again.

**What gets forgotten:** After bumping the fix, someone must remember to update `pyproject.toml`, `app/__init__.py`, and the Sidebar component. The `/rapid:new-version` skill bumps `package.json` but has no knowledge of the web backend or frontend version strings.

**Recommendation:** The version should be derived from a single source at build/startup time:
- Frontend: inject version from `package.json` at Vite build time via `define` config or environment variable
- Backend: read version from the Node.js `package.json` at startup, or share it via a common root-level version file
- The `/rapid:new-version` skill should be updated to bump all three locations, or better, only one canonical location

### Multi-Shell Configuration Management

Bug #1 (shell config update incomplete) exposes a systemic oversight: the install skill in `skills/install/SKILL.md` has a **find-first-break pattern** in its shell detection loop:

```bash
for f in ~/.bashrc ~/.bash_profile ~/.zshrc ~/.config/fish/config.fish ~/.profile; do
    if grep -qF "RAPID_TOOLS" "$f" 2>/dev/null; then
        echo "ALREADY_CONFIGURED=$f"
        break    # <-- stops at first match, ignores other shells
    fi
done
```

When a user has RAPID_TOOLS configured in multiple shell configs (common when they use both bash and fish, or bash and zsh), only the first detected config gets updated. The fix requires:

1. Collecting **all** config files containing RAPID_TOOLS (not just the first)
2. Updating all of them during `/rapid:install`
3. Presenting the user with a list of found configs rather than a single recommendation

**Beyond the immediate bug:** The setup.sh script itself writes only to `.env` and does not touch shell configs at all (Step 5). Shell config is entirely delegated to the install skill's agent-driven logic. This means the fix must be in the SKILL.md orchestration, not in setup.sh.

### Code Graph Overhaul -- File Extension Coverage Gap

The existing `codebase_service.py` tree-sitter parser maps extensions to languages, but **`.cjs` and `.mjs` files are not included** in the supported extensions:

```python
"javascript": ("tree_sitter_javascript", [".js", ".jsx", ".ts", ".tsx"]),
```

This means RAPID's own codebase (which is 100% `.cjs`) would show **zero files** in the code graph when scanning itself. For the code graph overhaul to be useful, the extension list must include `.cjs`, `.mjs`, `.cts`, `.mts` at minimum. This is especially important because the feature targets "full codebase graph" -- missing the project's own files would be immediately visible and embarrassing.

### Code Graph Overhaul -- Performance and Caching

The current `/api/projects/{id}/codebase` endpoint calls `get_codebase_tree()` synchronously on every request. The `useCodebaseTree` hook has `refetchInterval: 2000` (polls every 2 seconds). For large projects:

- Tree-sitter parsing is CPU-bound and blocking the FastAPI event loop (the endpoint is a sync function, not async)
- The `_parse_cache` in `codebase_service.py` is a module-level dict keyed on `(filepath, mtime)` -- it grows unbounded with no eviction
- With 500 files and 2-second polling, this creates unnecessary load even when nothing changed

For the graph overhaul (which will add more visual rendering and click-into-file), the backend needs:
- Response-level caching with mtime-based invalidation (not per-file, per-tree)
- The 2-second polling interval should be increased to 30+ seconds for codebase data (it changes infrequently)
- Consider making the endpoint async or running parsing in a thread pool

### Multi-Agent Bug-Fix -- Agent Context Isolation

The new feature adds wave-based splitting to `/rapid:bug-fix`, spawning one bugfix agent per wave. Several concerns:

1. **State handoff between waves:** Each agent runs sequentially, but later agents need to know what earlier agents changed. If wave-2 bugs are in files modified by wave-1 fixes, the agent may read stale code. There is no mechanism in the current bug-fix skill to pass modified-files context between waves.

2. **Commit conflict potential:** Each wave's agent commits independently to the same branch. If wave-1 modifies `file-A.cjs` and wave-2 also needs to modify `file-A.cjs`, the second agent will see wave-1's commits in the working tree, which is correct. But the skill needs to ensure agents see each other's commits (sequential execution handles this, but the skill must not use `--no-ff` or branch per wave).

3. **Error propagation between waves:** If wave-1 fails to fix a critical bug, should wave-2 still attempt its fixes? The current bug-fix skill (UAT mode) continues on BLOCKED results, which is appropriate. But the new wave-splitting needs the same resilience pattern.

4. **Context window management:** The stated goal of wave-splitting is to "reduce context rot." This only works if each wave's agent starts with a fresh context. The skill must spawn separate agent instances (not accumulate context within one session).

### Kanban Save Shortcut -- Keyboard Event Handling Architecture

Bug #2 (Ctrl+click should save) requires changing the `CardDetailModal` component. Currently the modal only listens for Escape via `document.addEventListener("keydown", ...)`. Several cross-cutting concerns:

1. **Platform-specific modifier keys:** On macOS, the convention is Cmd+Enter (or Cmd+click), not Ctrl+click. The fix should handle both `e.ctrlKey` and `e.metaKey`.

2. **Conflict with existing keyboard shortcuts:** The app has a `KeyboardContext` and `useKeyboardNav` hook. The save shortcut must be registered through the existing keyboard infrastructure rather than adding ad-hoc event listeners, or at minimum must not conflict with the command palette (`CommandPalette.tsx`) bindings.

3. **Save confirmation UX:** Currently the Save button triggers `handleSave` which calls `onSave` then `onClose`. A keyboard shortcut should follow the same path, not bypass validation. The Ctrl+Enter (standard for "submit" in modals) is a safer pattern than Ctrl+click.

4. **Focus context:** The shortcut should only fire when focus is within the modal (title input or description textarea), not when the modal's backdrop is focused.

### Error Handling -- Frontend API Error Boundaries

The frontend `apiClient.ts` throws `ApiError` for non-OK responses, but there is no global error boundary or toast notification system. Each page handles errors individually with inline error states. For the code graph overhaul, which involves heavier API calls:

- Large codebase parsing may timeout (the default 2-second polling timeout from the view hooks is too aggressive for initial parse)
- There is no retry logic in the `apiClient` itself -- retry is page-specific (only KanbanBoard has a Retry button)
- Network errors (backend down) show as raw error messages rather than user-friendly notifications

**Recommendation:** Add an error toast/notification system before building more complex visualizations.

### Logging and Observability

**Backend (good):** Structured JSON logging is already in place via `pythonjsonlogger` with rotation and stderr output. The `get_logger()` function provides namespaced loggers.

**Frontend (missing):** No client-side error logging or telemetry. For the code graph overhaul with interactive features (click-into-file), frontend errors in rendering or data parsing will be invisible.

**CLI/Plugin (adequate):** Uses `process.stderr.write('[RAPID ...]')` convention consistently. But the multi-agent bug-fix feature spawning multiple sequential agents has no aggregate log -- individual agent returns are parsed but there is no persistent log of the wave-by-wave execution.

**Recommendation:** For this milestone specifically, add a summary log file for multi-agent bug-fix runs (e.g., `.planning/logs/bug-fix-{timestamp}.json`).

### Testing Strategy Gaps

**Backend tests exist** (`web/backend/tests/`) with pytest, including API tests and service tests.

**Frontend has zero test infrastructure.** The `web/frontend/package.json` has no test runner -- no vitest, no jest, no testing-library. With the code graph overhaul adding interactive features (click-into-file viewing, graph navigation), there is no way to regression-test the UI.

**Plugin tests are solid** (`src/**/*.test.cjs` with `node:test`), but any new multi-agent bug-fix logic in the skill SKILL.md is untestable (skill files are markdown orchestration instructions, not code).

For this milestone:
- The code graph overhaul should not proceed without at least a vitest setup in the frontend, even if coverage starts at 0
- The multi-agent bug-fix skill changes are inherently untestable through unit tests -- manual UAT or integration tests are the only option

### Configuration Management -- CORS Origins

The backend hardcodes CORS origins:
```python
allow_origins=["http://127.0.0.1:5173", "http://localhost:5173"],
```

This only allows the Vite dev server. When the frontend is served statically (production mode via `serve_spa`), CORS is not needed because it is same-origin. However, if someone runs the backend on a different host/port or behind a reverse proxy, CORS will block requests. The code graph overhaul may increase interest in deploying Mission Control for team-wide access.

Not applicable to this milestone's immediate scope, but worth noting.

### Documentation Needs

1. **Code graph API documentation:** The `/api/projects/{id}/codebase` endpoint has no OpenAPI description beyond the response model. The overhaul should document the query parameters (currently just `max_files`) and the expected response structure changes.

2. **Multi-agent bug-fix usage:** The new wave-splitting behavior needs documentation in the skill description and in `docs/`. Users need to understand when wave-splitting activates and how to control it.

3. **Version bump procedure:** After fixing the version desync, document the procedure for bumping versions across all three locations (or the single-source-of-truth mechanism if implemented).

### Accessibility

**Current state:** The frontend uses semantic HTML in some places (buttons, labels, aria-labels on the Header and Sidebar) but not consistently. The Kanban board and Knowledge Graph page have no ARIA roles or keyboard navigation for the graph itself.

**For this milestone:** The code graph overhaul (cytoscape-based interactive graph) is inherently difficult for screen readers. At minimum:
- The graph should have an `aria-label` describing its purpose
- A text-based alternative (the existing file tree in CodebasePage) should remain accessible
- The click-into-file feature must be keyboard-accessible (Tab to focus nodes, Enter to open)
- Not applicable: color-only status indicators already have text labels

The Kanban save shortcut (Bug #2) actually improves accessibility by adding keyboard interaction.

### Internationalization

Not applicable for this project. RAPID is a developer tool with a technical English-speaking audience. String externalization would be overhead without benefit.

### CI/CD Pipeline

No CI/CD pipeline exists. Tests run locally only (`npm test` for plugin, `pytest` for backend). There are no GitHub Actions workflows.

For this milestone, this is acceptable since the team is small. However, the code graph overhaul introduces a frontend build step that could silently break. At minimum:
- The `tsc -b && vite build` command should be run before merging frontend changes
- Backend tests should be run before merging backend changes

### Developer Experience

**Local development is multi-step:**
1. Run `./setup.sh` (plugin + backend + frontend)
2. Start backend: `rapid-web` (requires systemd or manual uvicorn)
3. Start frontend dev: `cd web/frontend && npm run dev`
4. Register project: `/rapid:register-web`

For the code graph overhaul, developers will need all three running simultaneously. A single `npm run dev:all` or docker-compose would help, but is out of scope for this milestone.

**Hot reload:** Frontend has Vite HMR. Backend has no auto-reload (uvicorn `--reload` flag is not set in the CLI entry point). Plugin changes require `build-agents` re-run.

### Data Concerns

**Codebase parsing data:** The `get_codebase_tree` function returns tree-sitter AST data including file paths, symbol names, and line numbers. This is read-only and ephemeral (not stored in the database). No privacy concerns, but the code graph overhaul should not store parsed ASTs in the SQLite database -- they belong in a response-level cache with TTL.

**Kanban board data:** Stored in SQLite via SQLModel. No backup strategy exists. The kanban data is project-specific and could be lost if the `~/.rapid/rapid.db` file is deleted. Not a concern for this milestone's scope (the save shortcut fix does not change the storage model).

**State file integrity:** The multi-agent bug-fix wave-splitting operates on `.planning/` state. Since each agent runs sequentially and commits to the same branch, there is no concurrent write risk. Adequate for this feature.

## Recommendations

1. **Version single-source-of-truth:** Before fixing the sidebar display, establish a mechanism so the version is derived from one file. Cheapest approach: Vite `define` for frontend, and a startup read of the Node.js `package.json` for the Python backend. Update `/rapid:new-version` to handle all locations.

2. **Add `.cjs`/`.mjs`/`.cts`/`.mts` to codebase_service.py SUPPORTED_LANGUAGES:** This is a one-line change that unblocks the code graph from being useful on RAPID's own codebase and any other CommonJS project. Do this before any graph overhaul work.

3. **Multi-shell detection in install skill:** Change the `for...break` loop to collect all matching configs. Present them as a checklist to the user. Update all selected configs.

4. **Set up vitest in `web/frontend/`:** Even if no tests are written immediately, having the infrastructure means graph component tests can be added incrementally. Cost: 10 minutes. Value: prevents an untestable frontend from growing larger.

5. **Increase codebase polling interval:** Change `refetchInterval: 2000` in `useCodebaseTree` to `refetchInterval: 30000` or higher. Codebase structure changes infrequently -- 2-second polling is appropriate for state/kanban but not for tree-sitter parsing.

6. **Multi-agent bug-fix: pass modified-files between waves:** The sequential agent dispatch should include a "files modified by previous waves" field in the prompt for subsequent waves, so later agents can re-read those files.

7. **Platform-aware keyboard shortcuts for kanban:** Use `e.metaKey || e.ctrlKey` in the Ctrl+Enter save handler. Register through the existing keyboard infrastructure if possible.

8. **Make codebase endpoint async:** Wrap the tree-sitter parsing in `asyncio.to_thread()` or use a `run_in_executor` call so it does not block the FastAPI event loop. This is important when the code graph overhaul increases the max_files limit or adds more detailed parsing.

9. **Add aggregate logging for multi-wave bug-fix runs:** Write a JSON summary to `.planning/logs/` after all waves complete, capturing per-wave status, commit hashes, and blockers.

10. **Document version bump procedure:** Whether automated or manual, write down the steps to update version across all three locations. Add a test or CI check that verifies version consistency.
