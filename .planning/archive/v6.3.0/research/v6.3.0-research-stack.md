## Upgrade Paths

### TypeScript 5.9.3 -> 6.0.x (frontend devDependency)
- **Target:** TypeScript 6.0.2
- **Breaking changes:** TypeScript 6.0 removes deprecated APIs, changes some emit behavior, and introduces stricter type checking. A migration guide is available.
- **Recommendation:** Defer to a future housekeeping milestone. The `^5.9.3` range protects against accidental upgrade. TypeScript 5.9 is the last 5.x release and will continue receiving patches.
- **Migration effort:** Moderate -- requires running `tsc --strict` and fixing any new errors.
- **Priority:** Low (no functionality blocked by staying on 5.9).

### Zod 3.25.76 -> Zod 4.x (plugin core dependency)
- **Target:** Zod 4.3.6 (importable from `zod/v4` subpath)
- **Breaking changes:** Optional field defaults, error customization API changes, `z.record()` requires two arguments, `z.string().email()` moves to `z.email()`, stricter UUID validation, error `.errors` becomes `.issues`.
- **Recommendation:** Defer. Zod 3 at the root import is still supported within the same package. The plugin's 41+ library modules use Zod extensively for state validation. Migration is non-trivial.
- **Migration effort:** Significant -- every Zod schema definition would need review. The `z.record()` two-argument requirement alone affects multiple modules.
- **Priority:** Low (Zod 3 is stable and supported).

### proper-lockfile 4.1.2 (unmaintained)
- **Status:** No updates since 2021. 1.1M weekly downloads. Functional but no active maintainer.
- **Known issues:** None critical. The library is simple and works correctly.
- **Alternatives:** `lockfile` (npm), or a custom mkdir-based locking strategy.
- **Recommendation:** Monitor. The library works and RAPID has tests covering its usage. Only replace if a vulnerability is discovered.
- **Migration effort:** Moderate -- would need to rewrite `src/lib/lock.cjs` and update all callers.
- **Priority:** Low (functional, no CVEs).

### react-router 7.13.1 -> 7.14.0 (frontend dependency)
- **Migration effort:** Trivial -- patch bump within same major.
- **Priority:** Low (routine `npm update`).

### @tanstack/react-query 5.91.3 -> 5.96.2 (frontend dependency)
- **Migration effort:** Trivial -- minor bump within same major.
- **Priority:** Low (routine `npm update`).

### Vite 8.0.1 -> 8.0.7 (frontend devDependency)
- **Migration effort:** Trivial -- patch bump.
- **Priority:** Medium (patch releases typically contain bug fixes and security patches).

## Milestone-Specific Stack Implications

### Feature 1: Rapid Mission Control Code Graph Overhaul

**Current state:** Two separate views exist:
1. `KnowledgeGraphPage.tsx` -- DAG view of sets using Cytoscape.js + cytoscape-dagre (reads `.planning/DAG.json`)
2. `CodebasePage.tsx` -- File/symbol tree view using tree-sitter parsing (reads from backend `/projects/{id}/codebase` endpoint)

**Stack implications for the overhaul:**
- **Cytoscape.js** is well-suited for the existing DAG graph and can handle a full codebase dependency graph. It supports up to tens of thousands of nodes/edges with good performance. For a codebase-scale graph, this is adequate.
- **tree-sitter** backend already parses Python, JavaScript, Go, Rust files and extracts symbols. To build a "full codebase graph" with dependency/import relationships, the `codebase_service.py` will need to be extended to extract import statements and build an edge list (not just symbol extraction).
- **Missing grammar:** `tree-sitter-typescript` is not in pyproject.toml dependencies. JavaScript grammar handles `.ts`/`.tsx` files but may miss TypeScript-specific constructs (decorators, namespace, type-only imports). Consider adding it.
- **Click-into-file viewing:** The frontend will need either a file content viewer component or integration with an existing code viewer. CodeMirror is already installed (used by the notes editor). It can be reused for read-only file viewing with syntax highlighting.
- **Backend API extension needed:** Currently the backend serves codebase structure as a flat file list. The graph overhaul will need a new endpoint (or modification of existing) that returns graph edges representing imports/dependencies between files.

### Feature 2: Multi-Agent Bug-Fix Support

**Stack implications:**
- The `/rapid:bug-fix` skill (SKILL.md) already dispatches to `rapid-executor` agents. The enhancement to split into waves and spawn one agent per wave sequentially is purely a skill/orchestration change.
- No new dependencies needed. This is a `.md` skill file change plus potentially a new `.cjs` library module for wave decomposition.
- The existing DAG infrastructure (`src/lib/dag.cjs`) provides wave assignment logic that could be reused or adapted.

### Bug Fix: Shell Config Update Incomplete

**Root cause analysis from code:**
- `skills/install/SKILL.md` (lines 81-86) uses a `for` loop that `break`s after finding the FIRST shell config with RAPID_TOOLS:
  ```bash
  for f in ~/.bashrc ~/.bash_profile ~/.zshrc ~/.config/fish/config.fish ~/.profile; do
      if grep -qF "RAPID_TOOLS" "$f" 2>/dev/null; then
          echo "ALREADY_CONFIGURED=$f"
          break  # <-- BUG: stops after first match
      fi
  done
  ```
- This means if RAPID_TOOLS is set in both `~/.bashrc` and `~/.config/fish/config.fish`, only the first one is detected and updated.
- **Fix scope:** Skill file change only (SKILL.md). No library code changes needed.

### Bug Fix: Kanban Board Save Shortcut

**Root cause from code:**
- `CardDetailModal.tsx` only listens for `Escape` key (line 15). No Ctrl+click handler exists.
- The save button is a standard `onClick` handler. There is no keyboard shortcut for saving.
- **Fix scope:** Frontend component change. Add `Ctrl+Enter` or `Ctrl+S` keyboard handler to `CardDetailModal.tsx`.

### Bug Fix: Mission Control RAPID Version Outdated

**Root cause confirmed from code:**
- `web/backend/app/__init__.py` has hardcoded `__version__ = "4.2.1"` (line 1)
- `web/backend/pyproject.toml` has `version = "4.2.1"` (line 3)
- `web/frontend/src/components/layout/Sidebar.tsx` has hardcoded `"RAPID v4.2.1"` (lines 124, 126)
- The backend's health endpoint serves this version to the dashboard (DashboardPage.tsx line 96).
- **Three locations need updating:** `__init__.py`, `pyproject.toml`, and `Sidebar.tsx`.
- A test (`test_init.py:test_version_matches_pyproject`) verifies `__init__.py` matches `pyproject.toml`, so both must be updated in sync.
- **Root fix:** These should be dynamically derived from `package.json` (the single source of truth at v6.2.0) rather than hardcoded. At minimum, the three hardcoded values need to be updated to match the current plugin version.

## Stack Risks

1. **Hardcoded version strings (3 locations):** Multiple places hardcode "4.2.1" instead of reading from a single source. Risk of version drift persists even after fixing. Impact: user confusion, incorrect version display. Mitigation: centralize version to a single source, or automate version sync during `setup.sh` or `build-agents`.

2. **No frontend test coverage:** The code graph overhaul is a significant UI change with zero automated test coverage. Impact: regressions go undetected. Mitigation: add basic component tests with vitest, or add Playwright smoke tests.

3. **Pre-existing test failure:** `status-rename.test.cjs` fails due to stale `executing` literals in `review.test.cjs` and `dag.cjs`. Impact: blocks clean test runs, masks new test failures. Mitigation: fix the 2 occurrences before starting milestone work.

4. **proper-lockfile unmaintained (5 years):** No active development. Impact: if a vulnerability is found, there is no upstream to patch it. Mitigation: monitor, have replacement plan ready.

5. **tree-sitter-typescript missing:** The codebase parser classifies `.ts`/`.tsx` as JavaScript. Impact: for TypeScript-heavy projects, the code graph may miss or misparse constructs. Mitigation: add `tree-sitter-typescript` to pyproject.toml dependencies.

6. **Frontend dependency ranges (^):** Frontend uses caret ranges, which means `npm install` can produce different versions across environments. Impact: "works on my machine" class of bugs. Mitigation: commit `package-lock.json` (already done) and consider using `npm ci` in setup.

## Recommendations

1. **Fix pre-existing test failure in `status-rename.test.cjs`:** Update the 2 stale `executing` literals in `review.test.cjs` and `dag.cjs` before starting milestone work. Priority: **critical** (blocks clean test baseline).

2. **Centralize version string for Mission Control:** Replace the 3 hardcoded "4.2.1" locations with a mechanism to read from `package.json` or a shared constant. Priority: **high** (directly addresses a milestone bug fix, prevents recurrence).

3. **Add `tree-sitter-typescript` to backend dependencies:** The code graph overhaul will need accurate TypeScript parsing. The current JavaScript grammar is a rough approximation. Priority: **high** (directly supports Feature 1).

4. **Extend `codebase_service.py` with import/dependency extraction:** The existing service only extracts symbols. A code graph needs import edges. This is a backend implementation task, not a dependency change. Priority: **high** (core requirement of Feature 1).

5. **Reuse CodeMirror for file viewing in code graph:** CodeMirror is already installed for notes. It can render read-only syntax-highlighted file content without adding new dependencies. Priority: **medium** (supports click-into-file feature).

6. **Add Playwright or vitest for frontend smoke tests:** At minimum for the new code graph UI. Priority: **medium** (risk mitigation for major UI feature).

7. **Run `npm update` on frontend dependencies:** react-router 7.14.0, @tanstack/react-query 5.96.2, vite 8.0.7 are available as non-breaking updates. Priority: **low** (routine maintenance).

8. **Defer TypeScript 6.0 upgrade and Zod 4 migration:** Neither blocks this milestone. Both involve significant effort. Priority: **low** (future milestone candidates).

9. **Establish devDependencies pin policy (deferred decision):** The deferred decision about pinning devDependencies should be resolved when a future hardening set is planned. Currently only runtime deps are exact-pinned, which is the higher-risk surface. Priority: **low** (carry forward).
