# Research Summary -- v6.3.0 Mission Control & Fixes

## Project Profile
- **Type:** brownfield plugin + web dashboard (CLI + React frontend + FastAPI backend)
- **Stack:** Node.js (CommonJS) + Python 3.12+ (FastAPI/SQLModel) + React 19 (Vite/TypeScript/Tailwind 4)
- **Scale:** medium (2 features, 3 bug fixes; the code graph overhaul is the dominant item)
- **Key Constraint:** The code graph overhaul spans the full stack (backend tree-sitter parsing, new API endpoints, frontend graph visualization) and has no existing test coverage on the frontend

## Unified Findings

### Stack and Infrastructure

- **Node.js 22+ / CommonJS:** Stable runtime, no changes needed. Project runs on Node 25.8.0 locally but targets >=22. All plugin code uses `.cjs` extension. (source: STACK)
- **Python backend (FastAPI + tree-sitter):** Core parsing infrastructure already exists in `codebase_service.py` for Python, JavaScript, Go, Rust. Backend uses sync `def` endpoints which FastAPI runs in threadpool -- critical pattern to preserve for any new endpoints calling tree-sitter. (sources: STACK, ARCHITECTURE, PITFALLS)
- **React 19 + Vite 8 + Tailwind 4 frontend:** Modern stack, all dependencies current or near-current. TypeScript 5.9.3 pinned (6.0 available but deferred). (source: STACK)
- **Cytoscape.js 3.33.1:** Currently used for DAG visualization. Adequate for codebase graph up to ~500-1000 nodes. Performance degrades above that with dagre layout. (sources: STACK, ARCHITECTURE, PITFALLS)
- **CodeMirror (already installed):** Can be reused for read-only file viewing in the code graph click-to-view feature. No new dependency needed. (sources: STACK, FEATURES)
- **tree-sitter-typescript MISSING:** `.ts`/`.tsx` files are parsed with the JavaScript grammar, which misses TypeScript-specific constructs (type annotations, interfaces, generics). The code graph overhaul needs accurate TypeScript parsing. (sources: STACK, PITFALLS, OVERSIGHTS)
- **`.cjs`/`.mjs` extensions MISSING from parser:** `codebase_service.py` does not include `.cjs`, `.mjs`, `.cts`, `.mts` in supported extensions. RAPID's own codebase (100% `.cjs`) would show zero files in the code graph. (source: OVERSIGHTS)
- **proper-lockfile unmaintained (5 years):** Functional but no active maintainer since 2021. No CVEs. Monitor only. (source: STACK)
- **No CI/CD pipeline:** Tests run locally only. No GitHub Actions. Pre-existing test failure in `status-rename.test.cjs` has persisted uncaught. (sources: STACK, OVERSIGHTS)
- **No frontend test infrastructure:** Zero test coverage on the frontend -- no vitest, jest, or testing-library in devDependencies. (sources: STACK, OVERSIGHTS)
- **Frontend dependency updates available:** react-router 7.14.0, @tanstack/react-query 5.96.2, vite 8.0.7 -- all non-breaking. (source: STACK)

### Architecture Direction

- **Pattern:** Evolutionary extension of existing modular monolith (source: ARCHITECTURE)
- **Rationale:** All changes are localized modifications to well-defined modules. The existing 5-layer architecture (Skill -> Agent -> CLI -> Library -> Storage) with parallel web access path (React -> FastAPI -> Storage) cleanly supports these additions without structural changes. (source: ARCHITECTURE)
- **Module boundaries preserved:** Backend services extend `codebase_service.py` and `views.py`; frontend extends existing page/hook/type patterns; skill changes are self-contained in SKILL.md files. (source: ARCHITECTURE)
- **Data flow for code graph:** Frontend useCodeGraph hook -> GET /api/projects/{id}/codebase-graph -> codebase_service.get_codebase_graph() -> tree-sitter parse + import extraction -> JSON response -> Cytoscape.js render (source: ARCHITECTURE)
- **State management:** Graph state is frontend-only (React component state + TanStack Query cache). Bug-fix wave state is ephemeral within skill invocation. No persistent state changes needed for any feature. (source: ARCHITECTURE)
- **Separate endpoint recommended for code graph:** New `GET /api/projects/{id}/codebase-graph` rather than extending existing `/codebase`. The graph structure (nodes + edges) differs fundamentally from the tree structure (files with nested symbols). (source: ARCHITECTURE)
- **File content endpoint with path traversal guard:** New `GET /api/projects/{id}/file?path=...` with strict path validation using `Path.resolve().is_relative_to()`. Reject paths with `..`, enforce 1MB size limit, reject binary files. (sources: ARCHITECTURE, PITFALLS)

### Cross-Cutting Requirements

**Must address early (high retrofit cost):**
- Version single-source-of-truth mechanism -- without automation, the 3-location version desync will recur on every release (sources: OVERSIGHTS, PITFALLS)
- `.cjs`/`.mjs` extension support in codebase_service -- blocks the code graph from being useful on RAPID's own codebase (source: OVERSIGHTS)
- Frontend test infrastructure (vitest setup) -- adding tests after the graph overhaul is complete requires retroactive coverage with no baseline (source: OVERSIGHTS)
- Codebase endpoint polling interval tuning -- 2-second polling on tree-sitter parsing creates unnecessary load from day one (sources: OVERSIGHTS, PITFALLS)
- Multi-agent context handoff -- modified-files list must propagate between waves to avoid regressions (source: OVERSIGHTS)
- Sync `def` convention for new endpoints -- using `async def` for tree-sitter endpoints blocks the entire event loop (sources: PITFALLS, OVERSIGHTS)

**Can address later (low retrofit cost):**
- Frontend error toast/notification system -- improves UX but does not block core functionality (source: OVERSIGHTS)
- CI/CD pipeline -- valuable but high setup cost, not blocking this milestone (source: OVERSIGHTS)
- CORS configuration for non-localhost deployments -- only relevant if Mission Control is deployed for team access (source: OVERSIGHTS)
- Aggregate logging for multi-wave bug-fix runs -- cheap to add but not functionally blocking (source: OVERSIGHTS)
- Backend hot-reload for development -- developer convenience, not a blocker (source: OVERSIGHTS)

### User Experience Direction

- **Rename "Knowledge Graph" to "Code Graph":** The current name implies an AI/semantic graph. The overhaul creates a codebase dependency graph -- "Code Graph" or "Codebase Graph" aligns with Sourcetrail, CodeVisualizer, and Emerge conventions. (source: UX)
- **Follow Shneiderman's mantra:** "Overview first, zoom and filter, then details on demand." Progressive disclosure: directory clusters -> file nodes -> symbol detail panel with source code. (source: UX)
- **Click-to-view-file pattern:** Graph on left, code panel on right (Sourcetrail pattern). Use CodeMirror for syntax-highlighted read-only viewing. Show breadcrumb file path, symbol outline sidebar. (source: UX)
- **Search/filter is critical:** A 200+ node graph is unusable without search. Add a search input for filename/symbol filtering with graph highlighting. (source: UX)
- **Kanban save shortcut must be Ctrl+Enter (not Ctrl+click):** Ctrl+click is browser-reserved for opening links in new tabs. Ctrl+Enter is the industry standard for save-and-close in modals (Linear, Notion, Kanban Tool). Handle both `e.ctrlKey` and `e.metaKey`. (sources: UX, PITFALLS)
- **Tabbed view for Code Graph and Set DAG:** Keep the existing set dependency DAG accessible via a tab toggle ("Codebase" | "Dependencies") on the graph page, rather than removing it or adding a new nav item. The sidebar already has 9 items -- at the upper bound. (source: UX)
- **Performance safeguards:** For codebases with 200+ files, render directory-level clusters first. Add warning when exceeding renderable limit. Offer "simplified view" option. (source: UX)
- **Accessibility:** Add keyboard navigation between graph nodes (Tab/Arrow), screen reader announcements via aria-live for selected nodes, focus trap in CardDetailModal, color-independent status indication. (sources: UX, OVERSIGHTS)
- **Error communication:** Replace implementation-detail errors ("No DAG.json found") with actionable messages ("No graph data available. Run `/rapid:init` to generate project structure."). (source: UX)
- **Multi-agent bug-fix UX:** Display wave progress ("Wave 1/3: Fixing 4 critical bugs..."), explain why splitting is happening ("Splitting into 3 waves to keep agent context fresh"), show per-wave results before proceeding. (source: UX)

## Contradictions

### Contradiction: Graph Library Choice (Cytoscape.js vs React Flow)
- **FEATURES says:** Recommends React Flow (Option 2) for the code graph. Native React node rendering, built-in minimap, better performance with virtualization. "The ability to render React components inside nodes is a major advantage."
- **ARCHITECTURE says:** Recommends Cytoscape.js with force-directed layout (cose or cose-bilkent). "Use cytoscape.js with a force-directed layout. Files are nodes colored by language, imports are directed edges."
- **UX says:** Discusses Cytoscape.js performance limits but does not explicitly recommend React Flow. Suggests progressive loading and performance safeguards for Cytoscape.
- **Resolution:** Needs user decision. React Flow offers richer node rendering (React components inside nodes) and better React integration, but adds a new ~50KB dependency and introduces a second graph library. Cytoscape.js is already installed and has established patterns in the codebase. The tradeoff is developer experience and visual richness (React Flow) vs. consistency and zero new dependencies (Cytoscape.js). Both are viable.

### Contradiction: Bug-Fix Wave Dispatch Model
- **FEATURES says:** Recommends SKILL.md-only orchestration (Option 1). "The wave splitting is fundamentally an orchestration concern that belongs in the skill definition."
- **ARCHITECTURE says:** Describes the wave orchestrator as a module with a `--waves` flag, suggesting slightly more structure than pure SKILL.md orchestration.
- **Resolution:** Minor difference in framing, not a true conflict. Both agree the core logic belongs in the skill file. ARCHITECTURE's `--waves` flag naming differs from FEATURES' `--wave-size <n>` flag. The `rapid-roadmapper` should standardize on one flag name.

### Contradiction: File Content Endpoint Path Structure
- **FEATURES says:** `GET /api/projects/{id}/files?path=src/lib/state.cjs` (query parameter)
- **ARCHITECTURE says:** `GET /api/projects/{id}/file?path=relative/path` (query parameter, singular "file")
- **Resolution:** Minor naming difference. Both agree on query parameter approach. Standardize on one name during implementation (singular `file` is more RESTful for a single-file endpoint).

## Recommended Approach

The v6.3.0 milestone is dominated by the code graph overhaul, which spans the full stack and accounts for approximately 70% of the total effort. The remaining 30% consists of a moderate-complexity feature (multi-agent bug-fix waves) and three trivial bug fixes. The architecture is well-suited for evolutionary extension -- no structural changes are needed, only targeted additions to existing modules.

The critical-path pre-work should be done first: fix the pre-existing test failure, add missing file extensions to the parser, add `tree-sitter-typescript` as a dependency, and establish a version single-source-of-truth mechanism. These are low-effort items that unblock or de-risk the main feature work.

The code graph feature should be built backend-first: extend `codebase_service.py` with import extraction and add the new API endpoints (`/code-graph` and `/file`), then build the frontend visualization. A key decision point is whether to use Cytoscape.js (already installed, established patterns) or React Flow (richer React integration, built-in minimap). This decision should be made before frontend work begins. Regardless of library choice, performance safeguards (increased polling interval, layout animation controls, progressive loading for large codebases) must be designed into the initial implementation rather than retrofitted.

The bug fixes and multi-agent feature are fully independent of the code graph and of each other, making them ideal for parallel execution in separate sets. The three bug fixes are small enough to group into a single set, while the multi-agent bug-fix feature is self-contained as a SKILL.md modification. The code graph, being the largest item with frontend and backend components, should be its own set (or potentially split into backend and frontend sub-sets if parallel execution is desired).

Frontend test infrastructure (vitest) should be established before code graph UI work begins, even if initial test coverage is zero. This prevents the pattern of building a major UI feature with no regression testing capability.

### Suggested Set Boundaries

- **Set A: "Code Graph Backend"** -- Extend `codebase_service.py` with import extraction, add `.cjs`/`.mjs`/`.cts`/`.mts` extensions, add `tree-sitter-typescript` dependency, create `/code-graph` and `/file` API endpoints with path traversal guards, increase codebase polling interval. Rationale: Backend work is the foundation and has no frontend dependencies. Can be started and completed independently.

- **Set B: "Code Graph Frontend"** -- Overhaul KnowledgeGraphPage.tsx into code graph visualization (Cytoscape.js or React Flow), add click-to-view file panel with CodeMirror, add search/filter, implement tabbed view for Code Graph + Set DAG, rename "Knowledge Graph" to "Code Graph", add graph performance safeguards. Depends on Set A backend APIs. Rationale: All frontend graph work is tightly coupled and should be developed together.

- **Set C: "Bug Fixes"** -- Fix version display (update 3 locations + establish single-source-of-truth), fix Kanban Ctrl+Enter save shortcut, fix shell config multi-match detection, fix pre-existing test failure in `status-rename.test.cjs`. Rationale: All trivial-complexity fixes that are independent of each other and of the code graph feature. Can be parallelized with Set A.

- **Set D: "Multi-Agent Bug-Fix Waves"** -- Add wave decomposition to `/rapid:bug-fix` SKILL.md, implement cross-wave context handoff (modified-files list), add `--wave-size` flag, implement results aggregation. Rationale: Self-contained SKILL.md feature with no backend/frontend dependencies. Can be parallelized with Sets A, B, and C.

**DAG ordering:** Sets A, C, and D can execute in parallel (wave 1). Set B depends on Set A (wave 2).
