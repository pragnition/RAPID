### Feature 1: Branding Skill Refocus (UI/UX + Codebase Awareness)

- **Description:** Refocus the branding skill from documentation-only concerns to visual UI/UX brand guidelines. Make it codebase-aware so it adapts its interview and output based on whether the project has a webapp, CLI, API-only backend, etc.
- **Complexity:** moderate
- **Dependencies:** independent

#### Decomposition

1. **Codebase detection probe** -- Add a codebase scan step (before the interview) that detects project type: webapp (React/Vue/Angular/etc.), CLI tool, API-only, mobile, library/SDK, or mixed. Use `context detect` output or inspect `package.json`/`Cargo.toml`/etc. for framework indicators.
2. **Conditional interview branches** -- Adapt interview questions based on detected project type. For webapps: ask about color palettes, typography hierarchy, component theming, spacing/density, animation/transition style. For CLI tools: ask about terminal output formatting, color usage in terminal, progress indicator style. For API-only: reduce scope to documentation and error message tone.
3. **BRANDING.md output expansion** -- Add new sections to BRANDING.md output: `<visual>` (colors, typography, spacing -- webapp only), `<components>` (component style guide -- webapp only), `<terminal>` (CLI output style -- CLI only). Keep existing sections (identity, tone, terminology, output, anti-patterns) for all project types.
4. **HTML preview enhancement** -- Update the generated `index.html` to render visual branding elements (color swatches, typography samples) when webapp branding data exists.

#### Implementation Strategies

**Option 1: Extend existing SKILL.md with conditional branches**
- Approach: Add a Step 2.5 that runs codebase detection, sets a `PROJECT_TYPE` variable, then conditionally includes/excludes interview rounds in Step 3 based on project type.
- Pros: Single file change, preserves existing flow, backward-compatible
- Cons: SKILL.md grows longer, conditional logic in markdown instructions can be fragile
- Libraries/services: Uses existing `context detect` CLI command

**Option 2: Split into sub-skills per project type**
- Approach: Create `branding-webapp`, `branding-cli`, `branding-api` sub-skills, with a router skill that detects type and dispatches.
- Pros: Cleaner separation of concerns, each sub-skill stays focused
- Cons: Skill proliferation (3 new skills), shared logic duplication, harder to maintain
- Libraries/services: None needed

**Recommended:** Option 1. The branding skill is already compact (290 lines). Adding conditional branches keeps it under the 500-line threshold and avoids fragmenting the user-facing command surface. The branching is straightforward: detect type, then include/exclude specific AskUserQuestion rounds.

#### Data Model Impact

- New entities/tables needed: None
- Existing entities modified: BRANDING.md output schema gains optional `<visual>`, `<components>`, `<terminal>` sections
- Relationships: BRANDING.md continues to be consumed by `execute.cjs` prompt assembly

#### API Surface

- New endpoints/commands needed: None (uses existing `context detect`)
- Existing endpoints modified: None
- Skill changes: `skills/branding/SKILL.md` updated

### Feature 2: Init Command -- Encoded Acceptance Criteria

- **Description:** Scope the init command's acceptance criteria output into encoded, machine-verifiable criteria (e.g., UIUX-001, PERF-001, SEC-001) so the verifier agent can systematically check each criterion.
- **Complexity:** moderate
- **Dependencies:** independent

#### Decomposition

1. **Criteria taxonomy definition** -- Define a fixed set of criterion categories with encoding prefixes: FUNC (functional), UIUX (UI/UX), PERF (performance), SEC (security), A11Y (accessibility), I18N (internationalization), DATA (data model), INTG (integration), TEST (testing), DEPL (deployment).
2. **REQUIREMENTS.md format update** -- Change the freeform acceptance criteria format to structured encoded entries: `- [ ] UIUX-001: Dashboard loads within 2 seconds on 3G`. Each entry gets a unique ID, category prefix, and testable description.
3. **Init Step 4D generation logic** -- Update the init SKILL.md Step 4D to auto-generate encoded IDs based on the category of each criterion derived from the discovery conversation.
4. **Verifier integration** -- Update the verifier agent role (`src/modules/roles/role-verifier.md` or equivalent) to parse encoded criteria IDs and produce a per-criterion pass/fail report.
5. **REQUIREMENTS.md template update** -- Update `generateRequirementsMd()` in `src/lib/init.cjs` to produce the new encoded format as a template.

#### Implementation Strategies

**Option 1: Category-prefixed sequential IDs**
- Approach: Each criterion gets `{CATEGORY}-{NNN}` format. Categories derived from the discovery batch that produced the requirement. The verifier reads REQUIREMENTS.md, parses the encoding, and produces a structured report.
- Pros: Simple, human-readable, easy to reference in discussions
- Cons: IDs may need renumbering if criteria are reordered (minor -- only cosmetic)
- Libraries/services: None -- pure string formatting

**Option 2: Hierarchical requirement IDs with traceability matrix**
- Approach: Requirements get hierarchical IDs (e.g., FUNC-001.A, FUNC-001.B for sub-criteria) and a traceability matrix mapping each to a set.
- Pros: More precise traceability, supports sub-requirements
- Cons: Over-engineering for the current scope, adds complexity to the init flow
- Libraries/services: None

**Recommended:** Option 1. Simple prefixed IDs cover the use case. Traceability to sets can be handled by the roadmapper (which already receives REQUIREMENTS.md content) without needing hierarchical sub-IDs.

#### Data Model Impact

- New entities/tables needed: None
- Existing entities modified: `.planning/REQUIREMENTS.md` format changes from freeform checkboxes to encoded entries
- Relationships: REQUIREMENTS.md consumed by verifier agent, roadmapper, and plan-set

#### API Surface

- New endpoints/commands needed: None
- Existing endpoints modified: `init scaffold` template generation in `src/lib/init.cjs` (`generateRequirementsMd`)
- Skill changes: `skills/init/SKILL.md` Step 4D updated

### Feature 3: DAG Re-addition (Set Dependency Resolution + Web Dashboard)

- **Description:** Re-add the DAG for set dependency resolution and ordering. Generate DAG.json during roadmap creation. Display the DAG in the command center web dashboard.
- **Complexity:** complex
- **Dependencies:** independent (web dashboard display depends on DAG.json existing, but both can be built simultaneously)

#### Decomposition

1. **DAG generation during init/new-version** -- Update the roadmapper agent to output set dependencies alongside the roadmap. After roadmap approval, generate DAG.json using the existing `createDAG()` or `createDAGv2()` from `src/lib/dag.cjs`.
2. **DAG generation CLI command** -- Add a `dag generate` or `dag create` CLI subcommand that reads STATE.json sets and their declared dependencies, builds the DAG, and writes `.planning/sets/DAG.json`.
3. **Set ordering by DAG** -- Update `start-set`, `execute-set`, `merge`, and `status` commands to respect DAG wave ordering when displaying or processing sets.
4. **Web dashboard integration** -- The web dashboard already has the infrastructure: `KnowledgeGraphPage.tsx` renders a cytoscape.js DAG, `dag_service.py` reads DAG.json, and `views.py` exposes the API endpoint. The gap is that DAG.json is never generated in the current workflow. Once sub-feature 1 or 2 produces DAG.json, the dashboard visualization works automatically.
5. **DAG status sync** -- Keep DAG node statuses in sync with STATE.json set statuses. When a set transitions status, update the corresponding DAG node.

#### Implementation Strategies

**Option 1: Roadmapper-generated dependencies + CLI DAG builder**
- Approach: The roadmapper agent includes a `dependencies` array per set in its output. After roadmap approval, the init/new-version SKILL.md calls a new `dag create` CLI command that reads STATE.json sets and their dependencies, builds the DAG using `createDAGv2()`, and writes DAG.json. Status sync happens via a hook or state transition post-hook.
- Pros: Leverages existing `dag.cjs` library fully, clean separation between dependency declaration (roadmapper) and DAG construction (CLI)
- Cons: Requires roadmapper prompt update and new CLI subcommand
- Libraries/services: Existing `src/lib/dag.cjs` (toposort, assignWaves, createDAGv2), existing web dashboard (cytoscape.js + dagre layout)

**Option 2: Inline DAG construction in SKILL.md**
- Approach: The SKILL.md directly constructs DAG.json by calling `createDAG()` inline after roadmap approval, without a separate CLI command.
- Pros: Fewer moving parts
- Cons: SKILL.md becomes responsible for JSON construction logic that belongs in the CLI layer; harder to test

**Recommended:** Option 1. The CLI command approach follows the existing architecture pattern where SKILL.md files orchestrate and CLI commands handle business logic. The `dag.cjs` library is already feature-complete for this purpose.

#### Data Model Impact

- New entities/tables needed: None -- DAG.json already has a well-defined schema in `dag.cjs`
- Existing entities modified: STATE.json sets may gain an optional `dependencies` array field; roadmapper output gains a `dependencies` key per set
- Relationships: DAG.json references set IDs from STATE.json; web dashboard reads DAG.json via `dag_service.py`

#### API Surface

- New endpoints/commands needed:
  - `node "${RAPID_TOOLS}" dag create` -- Build DAG.json from STATE.json dependencies
  - `node "${RAPID_TOOLS}" dag show` -- Display DAG in terminal (optional)
- Existing endpoints modified: Roadmapper agent prompt to include dependency outputs
- Web API already exists: `GET /api/projects/{id}/dag` via `views.py`

### Feature 4: Discuss Phase Overhaul

- **Description:** Overhaul the discuss-set phase: ask architect-level and UI/UX questions instead of coding details; increase gray areas from 4 to 8 (or another multiple of 4); add richer inline context with pros/cons and recommendations; output a richer CONTEXT.md; add deferred decisions file/folder management.
- **Complexity:** complex
- **Dependencies:** independent

#### Decomposition

1. **Gray area count increase** -- Change from exactly 4 gray areas to 8 (or configurable multiples of 4). With 8 gray areas and 4 options per AskUserQuestion, the user gets 2 selection prompts (Step 5) to pick which areas to discuss.
2. **Question reframing** -- Update the gray area identification instructions to focus on architectural and UI/UX decisions rather than implementation details. Add explicit guidance: "Ask about WHAT the user wants, not HOW to code it. Frame questions as system architect / application designer decisions."
3. **Richer inline context per question** -- For each question in Step 6, require: (a) 3-5 sentence context with concrete research findings, (b) pros/cons table for each option, (c) a tagged "(Recommended)" option with brief rationale. This aligns with issue #19.
4. **Pre-discussion research step** -- Add a lightweight research substep before gray area identification. The skill should scan relevant source files, read existing contracts, and gather technical context to inform better question generation. This addresses issue #19's dependency on #18.
5. **Deferred decisions output** -- Create `.planning/sets/{SET_ID}/DEFERRED.md` for decisions that are out of scope but noted during discussion. Track deferred items with unique IDs and rationale.
6. **CONTEXT.md enrichment** -- Expand the CONTEXT.md output template to include: canonical references (full file paths mentioned), trade-off summaries for each decision, confidence levels per decision, and structured recommendations section.
7. **AskUserQuestion batching** -- With 8 gray areas and multiple questions per area, the flow needs careful batching. Present 2 AskUserQuestion calls for area selection (4 areas each), then deep-dive selected areas. Keep question count per area to 2-4 questions.

#### Implementation Strategies

**Option 1: In-place SKILL.md rewrite with research substep**
- Approach: Rewrite `skills/discuss-set/SKILL.md` to: (a) add a Step 4.5 lightweight research scan, (b) expand Step 5 to identify 8 gray areas across 2 AskUserQuestion selection prompts, (c) update Step 6 question format to include richer context, (d) add Step 7.5 for deferred decisions output, (e) enrich CONTEXT.md template in Step 7.
- Pros: Single-file change, preserves existing workflow, backward-compatible with --skip mode
- Cons: SKILL.md becomes significantly longer (potentially 500+ lines)
- Libraries/services: None -- all prompt engineering changes

**Option 2: Agent-assisted research before discussion**
- Approach: Spawn a lightweight research agent before the discussion to gather context, then use that context to generate better gray areas and richer inline recommendations. Inspired by the `--advisor` mode in get-shit-done.
- Pros: Higher quality gray areas and recommendations due to dedicated research pass
- Cons: Adds agent spawn overhead, increases token consumption, may slow the discuss phase
- Libraries/services: Existing `rapid-research-stack` or a new lightweight research agent

**Recommended:** Option 1 with selective elements from Option 2. The research substep should be inline (Glob + Grep codebase scanning within the SKILL.md) rather than a separate agent spawn, keeping the flow fast. The richer question format and CONTEXT.md enrichment are pure prompt engineering changes.

#### Data Model Impact

- New entities/tables needed: `.planning/sets/{SET_ID}/DEFERRED.md` -- new artifact
- Existing entities modified: CONTEXT.md format gains new sections (canonical refs, trade-offs, confidence levels)
- Relationships: DEFERRED.md may be consumed by future milestone planning; CONTEXT.md continues to feed into plan-set

#### API Surface

- New endpoints/commands needed: None
- Existing endpoints modified: None
- Skill changes: `skills/discuss-set/SKILL.md` major rewrite

### Feature 5: /new-version Comprehensiveness

- **Description:** Make the /new-version command's planning pipeline more comprehensive so it does not miss user requirements. The issue is that the milestone goals conversation is too shallow, leading to incomplete research and roadmaps.
- **Complexity:** moderate
- **Dependencies:** Feature 2 (encoded criteria format, if adopted, should be used here too)

#### Decomposition

1. **Deep discovery interview** -- Replace the single freeform "Describe the goals" question in Step 2C with a multi-batch discovery conversation similar to the init command's Step 4B. Cover: vision delta (what changed since last milestone), new features, technical debt, performance goals, and explicit non-goals.
2. **Requirements cross-check** -- After gathering goals, cross-reference against the previous milestone's REQUIREMENTS.md and ROADMAP.md to identify: (a) criteria not yet met, (b) deferred features from discuss-set DEFERRED.md files, (c) known issues from review pipeline.
3. **Goal completeness validation** -- Before spawning research agents, present the compiled goals back to the user with a structured summary (similar to init Step 4D) and ask for confirmation. Include a "What am I missing?" prompt.
4. **Research agent brief enrichment** -- Pass the full structured milestone brief (not just a single paragraph) to all 6 research agents. Include prior milestone context, unresolved deferred decisions, and carried-forward requirements.

#### Implementation Strategies

**Option 1: Mirror init's deep discovery in new-version**
- Approach: Copy the multi-batch discovery pattern from `skills/init/SKILL.md` Step 4B into `skills/new-version/SKILL.md` Step 2. Adapt batches for milestone-scoped context (previous milestone learnings, delta requirements, technical debt backlog).
- Pros: Proven pattern from init, consistent UX across commands
- Cons: Adds 3-5 more AskUserQuestion calls to new-version, making it longer
- Libraries/services: None -- prompt engineering

**Option 2: Auto-aggregate context from previous milestone artifacts**
- Approach: Automatically read DEFERRED.md files, REVIEW-ISSUES.json, and REQUIREMENTS.md from the previous milestone. Present a pre-populated goals draft to the user for review and amendment rather than asking from scratch.
- Pros: Less user effort, catches forgotten items, data-driven
- Cons: May present irrelevant items from large milestone backlogs
- Libraries/services: None

**Recommended:** Combination of both. Auto-aggregate deferred decisions and unmet requirements (Option 2) into a draft, then present for user review and allow additions via multi-batch discovery (Option 1). This catches forgotten items while letting the user add new goals.

#### Data Model Impact

- New entities/tables needed: None
- Existing entities modified: New-version milestone brief becomes richer (structured like init's project brief)
- Relationships: Reads from previous milestone's DEFERRED.md files and REQUIREMENTS.md

#### API Surface

- New endpoints/commands needed: None
- Existing endpoints modified: None
- Skill changes: `skills/new-version/SKILL.md` Steps 2-3 substantially expanded

### Bug Fix 3: #7 -- Init Step 4D Writes REQUIREMENTS.md Before Step 5 Scaffold Overwrites It

- **Description:** In the init flow, Step 4D writes acceptance criteria to `.planning/REQUIREMENTS.md`, then Step 5 runs `init scaffold` which calls `scaffoldProject()` which creates a blank template REQUIREMENTS.md, overwriting the user's criteria.
- **Complexity:** low
- **Dependencies:** independent

#### Decomposition

1. **Verify the bug** -- Confirmed by code inspection. `scaffoldProject()` in `src/lib/init.cjs` (line 237) always generates REQUIREMENTS.md via `generateRequirementsMd()`. In `reinitialize` mode, it deletes and recreates all files including REQUIREMENTS.md. In `upgrade` mode (line 276+), it skips existing files -- so the bug only manifests in `fresh` and `reinitialize` modes.
2. **Fix: Reorder init steps** -- Move Step 5 (scaffold) to run BEFORE Step 4D (acceptance criteria generation). This ensures the scaffold creates the template first, then Step 4D overwrites it with actual criteria.
3. **Alternative fix: Guard in scaffoldProject()** -- Add a content check in `scaffoldProject()`: if REQUIREMENTS.md already exists and contains non-template content, skip it. This is the "additive-only" approach.

#### Implementation Strategies

**Option 1: Reorder steps (SKILL.md change only)**
- Approach: Move Step 5 scaffold to occur between Steps 4A/4B (logistics/discovery) and Step 4D (acceptance criteria). The scaffold needs name and description which are available after Step 4A.
- Pros: Simple SKILL.md reorder, no library changes
- Cons: Step numbering changes require careful review of all cross-references

**Option 2: Guard in init.cjs (library change)**
- Approach: In `scaffoldProject()`, before writing REQUIREMENTS.md, check if the file exists and has content beyond the template. Skip if non-template content detected.
- Pros: Defensive fix that prevents data loss regardless of SKILL.md ordering
- Cons: Content detection heuristic may be fragile

**Recommended:** Both. Reorder steps in SKILL.md (primary fix) AND add a guard in `scaffoldProject()` (defense in depth). The guard ensures that even if future SKILL.md changes reintroduce the ordering issue, user data is not lost.

#### Data Model Impact

- None

#### API Surface

- Modified: `scaffoldProject()` in `src/lib/init.cjs` gains a guard check for existing REQUIREMENTS.md content
- Skill changes: `skills/init/SKILL.md` step reordering

### Bug Fix 7: #31 -- Make Unit-Test Framework-Agnostic

- **Description:** Unit-test skill hardcodes `node --test` with `node:test` + `node:assert/strict` for all projects regardless of stack. This breaks for Rust (cargo test), Python (pytest), Go (go test), etc.
- **Complexity:** complex (high)
- **Dependencies:** independent (but benefits from Feature 2's init research additions)

#### Decomposition

1. **Test framework detection** -- Add a detection step that identifies the project's test framework from: package.json (jest, vitest, mocha), pytest.ini/pyproject.toml (pytest), Cargo.toml (cargo test), go.mod (go test), etc. Store the detected framework in `.planning/config.json` under a `testFramework` key.
2. **Init research integration** -- Add test framework detection to the `rapid:init` research phase. The research-stack agent should identify and recommend a test framework as part of its output.
3. **Generic test runner abstraction** -- Replace the hardcoded `node --test` instructions in SKILL.md Step 5 with a parameterized runner: `{TEST_COMMAND} {TEST_FILE}`. The test command is derived from the detected framework.
4. **Framework-specific test file templates** -- Provide test file templates for common frameworks: node:test (.test.cjs), jest (.test.js), vitest (.test.ts), pytest (test_.py), cargo test (tests/), go test (_test.go).
5. **Result parsing abstraction** -- Each framework has different output formats. Create a result parser that normalizes test output into the standard `{ passed, failed, errors }` structure regardless of framework.
6. **SKILL.md instruction rewrite** -- Rewrite Step 5 agent prompt to be framework-aware. The prompt should include the detected test framework and use framework-appropriate patterns.

#### Implementation Strategies

**Option 1: Config-driven test framework with detection**
- Approach: Detect framework during init, store in config.json, read during unit-test. Agent prompts receive the framework name and use appropriate commands/patterns. The core value proposition (plan-approve-execute-report pipeline) is preserved regardless of framework.
- Pros: Clean separation between detection and execution, extensible to new frameworks
- Cons: Detection heuristics may miss edge cases; some projects use multiple test frameworks
- Libraries/services: None -- detection uses file existence checks

**Option 2: Per-language skill variants**
- Approach: Create `unit-test-node`, `unit-test-python`, `unit-test-rust` skill variants.
- Pros: Framework-specific instructions can be very precise
- Cons: Massive skill proliferation, maintenance burden, duplicated pipeline logic

**Recommended:** Option 1. A single skill with framework detection is far more maintainable. The detection step stores framework info in config.json, and the agent prompt template is parameterized. New frameworks can be added by extending the detection map and adding a template.

#### Data Model Impact

- New entities/tables needed: None
- Existing entities modified: `.planning/config.json` gains optional `testFramework` field (e.g., `{ "testFramework": "node:test" }`)
- Relationships: Config consumed by unit-test skill

#### API Surface

- New endpoints/commands needed:
