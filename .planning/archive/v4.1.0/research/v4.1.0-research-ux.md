# Domain & UX Research -- v4.1.0 (Polish & Fixes)

## Domain Conventions

RAPID operates in the **AI-assisted parallel development orchestration** domain. This is a rapidly maturing space where the user is a developer acting as an orchestrator of autonomous AI agents. Key conventions observed across competing products and industry trends:

### Standard Terminology

| Term in Domain | RAPID Equivalent | Notes |
|---|---|---|
| Skill / Workflow | Skill (SKILL.md) | Universal SKILL.md format is cross-compatible with Claude Code, Cursor, Gemini CLI, Codex CLI, Antigravity (source: Claude Code docs, Composio) |
| Agent / Subagent | Agent (agents/*.md) | Industry standard; Cursor 2.0 and Antigravity both use "agent" |
| Spec-driven development | CONTEXT.md + PLAN.md | AWS Kiro popularized spec-driven mode alongside vibe coding (source: RedMonk 2025 survey) |
| Decision Record (ADR) | Gray areas in discuss-set | Architecture Decision Records are the industry standard for capturing decisions with context, options, and consequences (source: adr.github.io, Microsoft Azure Well-Architected Framework) |
| DAG / Dependency Graph | DAG.json + set ordering | Apache Airflow, dbt, and Prefect all use DAG visualization as the primary orchestration view (source: Airflow docs, dbt blog) |
| Progress indicator | Progress breadcrumb | CLI tools standardize on steppers for linear workflows and breadcrumbs for hierarchical navigation (source: Evil Martians CLI UX patterns, Carbon Design System) |
| Checkpoint / Rollback | RAPID:RETURN CHECKPOINT | Developers expect instant undo to known-good states (source: RedMonk "10 Things Developers Want") |
| Human-in-the-loop | AskUserQuestion | Approval gates before destructive actions are an industry expectation, not a nice-to-have (source: RedMonk, Anthropic best practices) |

### Competitive Landscape Patterns

1. **Cursor 2.0**: Multi-agent orchestration with dashboard showing which agents are working on what; pause/redirect/terminate controls (source: RedMonk 2025)
2. **AWS Kiro**: Dual spec-driven and vibe coding modes; requirements files serve as agent contracts (source: RedMonk 2025)
3. **Google Antigravity**: Autonomous agents at center of development; parallel agent dispatch (source: Faros AI reviews)
4. **Zencoder Zenflow**: Orchestration layer coordinating multiple AI agents with workflow visualization (source: The New Stack)
5. **GSD (Get Shit Done)**: Phase-based workflow with rich gray area discussion, deferred decisions mechanism, canonical references, and advisor-backed research for each question (source: gsd-build/get-shit-done GitHub)

RAPID is differentiated by its worktree isolation, contract-first planning, and adversarial review pipeline. These are unique in the space and should be preserved as first-class UX elements.

## Interaction Models

### Primary Interaction Paradigm

RAPID uses a **hybrid CLI-skill + web dashboard** model:
- **CLI skills** (slash commands via Claude Code) for the primary lifecycle workflow
- **Web dashboard** (React + Tailwind) for monitoring and visualization (command center)
- **AskUserQuestion** for structured decision capture during skills

This matches the industry trajectory: developers want CLI-first agent control with visual dashboards for monitoring (source: RedMonk, The New Stack "Agentic CLI Era").

### Input/Output Patterns Users Expect

1. **Structured multiple-choice prompts**: AskUserQuestion is the standard interaction unit. Each question should present 2-4 concrete options with descriptions. A "(Recommended)" tag on the suggested option is now a recognized convention across Claude Code plugins (source: ClaudeLog, SmartScope). RAPID already follows this pattern but should extend it with richer context per option (see Recommendations).

2. **CLI argument consistency**: The review log-issue API mismatch (Issue #27) violates user expectations. Users expect CLI flag syntax (`--type`, `--severity`) to work as documented. When stdin JSON is the actual mechanism, the docs must say so explicitly, or the implementation should accept flags. Mixed conventions within a single tool erode trust. The `parseArgs` utility already exists in `src/lib/args.cjs` and should be applied consistently.

3. **Numeric shorthand**: RAPID's pattern of allowing `/rapid:start-set 1` instead of `/rapid:start-set auth-system` is excellent UX. This aligns with how git uses numeric refs (HEAD~1) and how Airflow uses task indices. This convention should extend to any new command.

4. **JSON to stdout, human text to stderr**: RAPID already follows this convention (source: STYLE_GUIDE.md). This is industry standard for composable CLI tools (source: 12-Factor CLI patterns).

### Feedback and Progress Indication Conventions

1. **Progress breadcrumbs**: RAPID uses a text-based breadcrumb (`init [done] > start-set [done] > discuss-set > ...`). This is a linear stepper pattern, which is the correct choice for sequential workflows (source: Carbon Design System, Evil Martians). However, the breadcrumb only appears at completion and on errors. Industry practice is to show the stepper at the START of each skill invocation as well, so users always know where they are in the lifecycle.

2. **Stage banners**: The ANSI-colored banners (`display.cjs`) with color-coded stages (blue=planning, green=execution, red=review) follow terminal UI conventions well. The three-color grouping provides good affordance.

3. **Long-running operations**: The web dashboard should poll for state changes rather than using long-lived HTTP connections (per project CLAUDE.md guidance). This matches how Airflow and dbt dashboards work -- periodic refresh of DAG state.

### Error Communication Patterns

1. **Error breadcrumbs** showing `[FAILED: {reason}]` at the failure point are good UX. This pattern mirrors CI/CD pipeline visualizations (GitHub Actions, GitLab CI).
2. **Descriptive errors with next steps**: RAPID's convention of "Show error + suggest next command" aligns with best practices from the Anthropic Claude Code best practices guide: "address root causes, not symptoms."
3. **Graceful degradation**: The pattern where optional features (DAG generation, branding) fail without stopping the workflow is correct. Users expect non-critical failures to warn, not block.

## Information Architecture

### How Information Should Be Organized

RAPID's information architecture follows a clear hierarchy:

```
Project > Milestone > Set > Wave > Job
```

This maps to the web dashboard's navigation:
- Dashboard (project overview)
- State (milestone + sets)
- Kanban (sets as cards across status columns)
- Knowledge Graph / DAG (dependency visualization)
- Worktrees (active working copies)
- Notes (freeform developer notes)

**DAG visualization placement**: When DAG is re-added to the command center (todo item #3), it should be prominently positioned. In Airflow, the Graph view is the PRIMARY view, not a secondary page. For RAPID, the DAG should be visible from the Dashboard page or accessible with one click, showing sets as nodes colored by status with dependency edges. The existing `KnowledgeGraphPage.tsx` using cytoscape-dagre is a solid foundation.

### Navigation and Discovery Patterns

1. **Command palette** (`CommandPalette.tsx`): Already implemented. This is the standard power-user discovery mechanism (VS Code Ctrl+Shift+P, Raycast, etc.).
2. **Sidebar navigation**: Current sidebar with collapsible states (full/compact/hidden) follows standard patterns.
3. **Keyboard shortcuts**: Already supported (`useKeyboardNav.ts`, `KeyboardContext.tsx`). This is expected for developer tools.

### Content Hierarchy and Progressive Disclosure

For the **discuss-set improvement** (todo item #4), the current gray area presentation lacks progressive disclosure. The GSD discuss-phase pattern provides a better model:

**Current RAPID pattern** (too sparse):
```
Context: {1-2 sentences explaining this specific tradeoff}
```

**Recommended enriched pattern** (from GSD and Issue #19):
```
Context: {3-5 sentences with research findings and concrete references}
Pros/Cons table for each option
(Recommended) tag with rationale
```

This is progressive disclosure: the question title is the summary, the context provides depth, and the pros/cons table provides full analysis. Users can make informed decisions without external research.

### Help and Documentation Conventions

The `/rapid:help` skill provides a static command reference table, which is the standard pattern for CLI help. The `disable-model-invocation: true` flag ensures no AI overhead for a simple reference lookup -- this is correct.

## Accessibility Considerations

### Key Accessibility Patterns

1. **Terminal/CLI accessibility**: RAPID's ANSI banners use the 16-color palette for maximum terminal compatibility (noted in `display.cjs` comments). This is correct -- 256-color or RGB would break on some terminals and screen readers.

2. **Web dashboard accessibility**: The sidebar already implements `role="button"`, `tabIndex`, `aria-label`, and keyboard event handlers (`Escape` to close). This is above-average for developer tools.

3. **AskUserQuestion accessibility**: Multiple-choice prompts with numbered options are inherently accessible -- they work with screen readers and keyboard-only navigation.

### Color and Contrast Considerations

- The web dashboard uses a theme system (`useTheme.ts`, `theme.ts`) with semantic color tokens (`text-fg`, `text-muted`, `bg-surface-0`, `bg-surface-1`, `text-accent`, `text-error`). This supports dark/light mode switching, which is an industry expectation for developer tools.
- Node status colors in the DAG visualization (`NODE_COLORS` in `KnowledgeGraphPage.tsx`) should be distinguishable by shape or label, not just color, for color-blind users. Currently they are only differentiated by color.

### Inclusive Design Patterns

- The `--skip` flag on `discuss-set` provides a non-interactive path for CI/automation contexts where AskUserQuestion is unavailable. This is important for accessibility in headless environments.
- Not applicable: RAPID is a developer tool, not a consumer-facing product. Advanced accessibility patterns (high contrast mode, font scaling, reduced motion) are nice-to-have but not industry-expected for this domain.

## User Expectations

### Default Behaviors Users Expect

1. **Idempotent re-runs**: When a user runs `/rapid:discuss-set` on an already-discussed set, RAPID correctly asks what to do (re-discuss, view existing, cancel). This matches user expectations from git (`git init` in an existing repo warns gracefully).

2. **Numeric shorthand everywhere**: Users who learn `/rapid:start-set 1` will expect `1` to work for ALL set-referencing commands. RAPID already provides this via `resolve set`.

3. **Non-destructive defaults**: The archive step in `/rapid:new-version` is optional (user chooses). This matches the industry expectation that destructive operations require explicit consent (source: RedMonk "Human-in-the-Loop Controls").

4. **Consistent flag behavior**: When `--post-merge` works as a flag in one command, users expect ALL flags to work as documented. The log-issue stdin-JSON mismatch (Issue #27) violates this expectation.

### Terminology Users Use vs Technical Terms

| User Says | RAPID Internal Term | Alignment |
|---|---|---|
| "task" or "work item" | Set | Low -- "set" is abstract. Users may initially confuse with mathematical sets. However, this is established terminology and changing it would break existing workflows. |
| "dependency" | DAG edge | Good -- but users think in terms of "X depends on Y," not "edge from X to Y." DAG visualization should label edges as "depends on." |
| "discussion" or "interview" | discuss-set | Good -- Claude Code best practices explicitly recommend "interview" terminology. |
| "plan" | PLAN.md / wave plan | Good. |
| "review" | review + unit-test + bug-hunt + uat | Users may not expect 4 separate commands. The pipeline nature should be clear from the status dashboard. |

### Mental Models from Competing Products

1. **From Airflow/dbt**: Users expect DAG as the central navigation element with status-colored nodes and click-to-inspect detail.
2. **From Jira/Linear**: Users expect kanban board with drag-and-drop status transitions. RAPID's KanbanBoard page aligns with this.
3. **From GitHub Actions**: Users expect a pipeline view showing which step is running, which passed, which failed.
4. **From GSD**: Users familiar with GSD expect rich discussion phases with pros/cons tables, recommended options, and deferred decisions capture.

### Onboarding and Learning Curve Expectations

- `/rapid:help` provides a complete command reference. This is the minimum.
- The progress breadcrumb at skill completion helps users learn the lifecycle sequence.
- The `DOCS.md` with correct URLs (Issue #2) is important -- broken URLs during onboarding are a high-frustration pain point.
- The init skill's prereq checker with install guides is excellent onboarding UX.

## Recommendations

Ordered by relevance to the v4.1.0 Polish & Fixes milestone:

1. **Enrich discuss-set gray area questions with research context** (Issue #19): Each question should include 3-5 sentence context with concrete references, a pros/cons table per option, and a "(Recommended)" tag with rationale. This directly mirrors the GSD discuss-phase pattern and addresses the reported issue where users "pick Claude decides for everything." Increase gray area count from 4 to a multiple of 4 (8 or 12) to provide richer coverage while maintaining AskUserQuestion's 4-option constraint. -- Priority: critical

2. **Add deferred decisions mechanism to discuss-set**: When users mention ideas outside the current set's scope, capture them in a `DEFERRED.md` file (or `.planning/sets/{set-id}/DEFERRED.md`). The GSD pattern of responding "that would be a new capability -- want me to note it for the backlog?" prevents scope creep while preserving good ideas. This was explicitly requested in todo item #4. -- Priority: critical

3. **Make branding skill codebase-aware**: The current branding skill asks about tone/voice/terminology but does not inspect the codebase to determine if the project has a webapp, CLI, API, or library. It should detect the project type (via `context.cjs` language detection or package.json inspection) and tailor questions accordingly. A CLI-only project should not be asked about "visual brand" in the same way a webapp would. -- Priority: high

4. **Scope init criteria into encoded categories**: The init command should generate specific, encoded acceptance criteria (UIUX-001, PERF-001, SEC-001, etc.) that the verifier agent can systematically check. This aligns with the ADR pattern where each decision gets a unique identifier for traceability. Categories should include: UIUX (user experience), PERF (performance), SEC (security), FUNC (functionality), COMPAT (compatibility), TEST (test coverage). -- Priority: high

5. **Add DAG visualization to the web dashboard**: Re-add the DAG as a first-class element. The existing `KnowledgeGraphPage.tsx` with cytoscape-dagre is a solid foundation. The DAG should be accessible from the Dashboard page (not buried in a secondary route). Nodes should be colored by status AND display status labels/shapes for color-blind accessibility. Edge labels should say "depends on" to match user mental models. -- Priority: high

6. **Fix CLI argument consistency for review log-issue** (Issue #27): Either implement proper CLI flag parsing using the existing `parseArgs` from `args.cjs` (so `--type`, `--severity` flags work as documented), or update all skill docs to show the stdin JSON format. The former is preferred because it matches the user expectation established by other RAPID commands. Auto-generate `id` and `createdAt` fields server-side to reduce caller burden. -- Priority: high

7. **Elevate discuss-set questions to architect/designer level**: Questions should focus on "how should this LOOK and BEHAVE" rather than "how should this be CODED." This means asking about user journeys, information hierarchy, error states, and loading states rather than implementation patterns like "should I use a factory pattern." This directly addresses todo item #4. -- Priority: high

8. **Make unit-test framework-agnostic** (Issue #31): Auto-detect the project's test framework from package.json, Cargo.toml, pyproject.toml, go.mod, etc. The core value is the plan-approve-execute-report pipeline, not the specific runner. Users expect RAPID to work with their existing test setup, not force `node --test`. -- Priority: medium

9. **Remove 5-group limit in unit-test skill** (Issue #29): Replace the fixed limit with dynamic grouping based on the actual number of concerns. When more than 5 groups exist, merge related concerns or run multiple dispatch rounds. Watch token usage as noted in the issue. -- Priority: medium

10. **Show progress breadcrumb at skill START**: Currently breadcrumbs only appear at completion and on errors. Display the stepper with the current stage highlighted at the beginning of each skill invocation so users always know where they are in the lifecycle. This is standard for multi-step workflow UIs (source: Carbon Design System stepper pattern). -- Priority: low

11. **Improve new-version comprehensiveness** (todo item #5): When gathering milestone goals (Step 2, Question C), use a structured interview rather than a single freeform question. Ask about feature additions, bug fixes, performance goals, and scope boundaries in separate AskUserQuestion calls. This reduces the chance of missing user requirements. -- Priority: medium

12. **Add edge labels to DAG visualization**: Use "depends on" or "blocks" directional labels on edges in the cytoscape graph. Users think in dependency terms, not graph theory terms. -- Priority: low
