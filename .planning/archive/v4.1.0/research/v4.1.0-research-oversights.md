# Oversights Research -- v4.1.0 Polish & Fixes

## Cross-Cutting Concerns

### 1. Discuss-Set Rework: Question Count Constraint vs AskUserQuestion Batching

**Why this matters for this project specifically:** The milestone brief item #4 asks for more gray areas (beyond the current fixed 4), questions that are multiples of 4 (to match AskUserQuestion's 4-option constraint), and architect/UI-UX level questions instead of coding-detail questions. These three requirements interact in non-obvious ways.

**The constraint chain:**
- AskUserQuestion supports exactly 4 options per call
- The current skill hardcodes "exactly 4 gray areas" (SKILL.md line 331, 348)
- The brief says "keep questions as multiples of 4" and "if we have 4n gray areas, we should have n ask user questions"
- But the gray area *selection* step (Step 5, lines 164-174) presents gray areas AS the 4 options, meaning the selection mechanism itself is limited to 4

**Overlooked interactions:**
1. If the number of gray areas increases to 8 or 12, the selection mechanism in Step 5 must change. Currently, gray areas ARE the AskUserQuestion options. With 8 gray areas, you would need 2 selection prompts, and the user experience becomes fragmented -- they cannot see all areas at once before choosing.
2. The current Step 5 says "unselected areas default to Claude's discretion." With multi-prompt selection, how does the user indicate "none of these"? Each prompt's 4 options would need a "skip all in this batch" option, consuming one of the 4 slots.
3. The Step 6 deep-dive (lines 185-203) already uses one AskUserQuestion per question within a gray area. If gray areas expand from 4 to 8+, and each area has 2-3 questions, total AskUserQuestion calls could balloon to 16-24+. This significantly extends the discussion duration.
4. The anti-pattern on line 348 says "Do NOT ask about fewer or more than 4 gray areas." This must be explicitly removed and replaced with the new constraint.

**Recommendation:** Redesign the selection mechanism. Instead of presenting gray areas as AskUserQuestion options, present a numbered list in the message body and ask "Which areas would you like to discuss? (enter numbers, e.g. 1,3,5)" with options like "Discuss selected areas" / "Discuss all" / "Skip all (Claude decides)" / "Let me pick differently". This decouples the gray area count from the 4-option limit.

### 2. Discuss-Set: Deferred Decisions File/Folder -- No Infrastructure Exists

**Why this matters:** The milestone brief item #4 says "allow any out of scope user questions/decisions to be outputted to a deferred decisions file/folder and then manage that." The current CONTEXT.md template has a `<deferred>` section (SKILL.md lines 263-266) but it is inert -- nothing downstream reads it, no management commands exist, and no skill consumes or surfaces deferred items.

**What exists today:**
- CONTEXT.md has a `<deferred>## Deferred Ideas` section (lines 263-266)
- In --skip mode, the deferred section is explicitly set to "Empty" (line 134)
- No CLI command exists for managing deferred items (listing, promoting, archiving)
- No downstream skill reads the deferred section
- No `.planning/deferred/` directory or file exists

**What "manage that" requires:**
1. A storage location: either a `.planning/deferred/DEFERRED.md` file or a `.planning/sets/{setId}/DEFERRED.md` per set
2. A way to add items during discussion (already partially covered by the `<deferred>` section)
3. A way to surface deferred items when starting a new version or new set (the `/new-version` skill should read deferred items to feed into the roadmapper)
4. A way to promote a deferred item into actual scope (either into an existing set or a new set)
5. The get-shit-done reference workflow treats deferred ideas as first-class: "Out-of-scope suggestions and reviewed-but-not-folded todos" with cross-referencing

**Overlooked interaction with /new-version:** When a new milestone is created, deferred items from the previous milestone should be surfaced to the user as potential scope. Currently /new-version (Step 2, Question C) asks for goals freeform but does not reference any previously deferred decisions.

**Recommendation:** Start with a simple `.planning/DEFERRED.md` file at the project level (not per-set, since deferred items by definition transcend set scope). Add a `deferred list` and `deferred promote` CLI command. Have /new-version read DEFERRED.md and present items to the user. This is low-cost to implement and high-value for project continuity.

### 3. Init Criteria Encoding: No Schema, Verifier, or Traceability Path Exists

**Why this matters:** The milestone brief item #2 says the init command should "scope into encoded criteria (UIUX-001, PERF-001, etc.) for the verifier to check against." This requires changes across three separate systems that must be coordinated.

**Current state of acceptance criteria:**
- Init Step 4D generates "formal acceptance criteria" as markdown checkboxes (SKILL.md lines 422-442)
- Written to `.planning/REQUIREMENTS.md` as unstructured markdown
- The roadmapper receives them as freeform text (SKILL.md line 758-760)
- The verifier agent (role-verifier.md) has no concept of encoded criteria -- it checks filesystem artifacts, git commits, and test results, not requirement codes
- The plan-verifier agent (role-plan-verifier.md) checks coverage against wave/job plans, not against encoded requirements

**What needs to be built:**
1. A criteria encoding scheme (e.g., `UIUX-001: Landing page matches Figma mockup`)
2. A structured format in REQUIREMENTS.md that can be machine-parsed (not just markdown checkboxes)
3. An update to the verifier role to check encoded criteria
4. A traceability mechanism: which set addresses which criteria?
5. A CLI command or verification step that checks criteria fulfillment

**Overlooked complexity:**
- The verifier currently runs at merge time on filesystem artifacts (verify.cjs lines 23, 56). Encoded criteria like "UIUX-001: Landing page is responsive" cannot be verified by checking file existence -- they require subjective evaluation. The verifier would need a hybrid approach: some criteria are auto-verifiable (file exists, test passes), others require agent judgment.
- If criteria are encoded during init, they must survive across milestones. The REQUIREMENTS.md gets overwritten by scaffold (the #7 bug). Even after fixing #7, the criteria encoding must be stable across /new-version calls.
- The roadmapper needs to emit traceability: "Set auth-system addresses FUNC-001, FUNC-002, SEC-001." This is a new output format for the roadmapper agent.

**Recommendation:** Design the encoding schema first as a standalone task before implementing. A simple approach: structured YAML frontmatter in REQUIREMENTS.md with a `criteria:` array, each having `id`, `category`, `description`, `verifiable` (boolean), and `verification_method` fields. This allows the verifier to programmatically check verifiable criteria while flagging subjective ones for agent review.

### 4. DAG Reintroduction: Existing Module vs Skill/Dashboard Integration Gap

**Why this matters:** The milestone brief item #3 says "add back the DAG" for set dependency resolution and display in the command center web dashboard. The DAG module (`src/lib/dag.cjs`) already exists with full functionality (toposort, wave assignment, validation). But the integration layer between the DAG module and the skills/dashboard is broken or missing.

**What already exists:**
- `src/lib/dag.cjs`: Complete DAG implementation with `toposort()`, `assignWaves()`, `createDAG()`, `validateDAG()`, `tryLoadDAG()`
- `DAG_CANONICAL_SUBPATH`: Defined as `.planning/sets/DAG.json`
- The merge skill (`skills/merge/SKILL.md`) references DAG for merge ordering
- The add-set skill (`skills/add-set/SKILL.md`) references DAG
- 17 source files reference `DAG.json` or `DAG_CANONICAL`

**What is missing or broken:**
1. No skill or step in `/rapid:init` or `/rapid:new-version` creates the initial DAG.json. The roadmapper outputs sets with dependencies, but nobody serializes the dependency graph to DAG.json.
2. The web dashboard's GraphPage is a placeholder: `<p>Dependency graph visualization will appear here</p>` (GraphPage.tsx line 3)
3. No backend API endpoint serves DAG data to the frontend
4. The `/rapid:start-set` skill does not enforce DAG ordering (it does not check whether a set's dependencies are met before allowing it to start)
5. The init SKILL.md Step 9 sends `Set dependencies (which other sets it depends on)` to the roadmapper, but does not instruct it to output a DAG.json file

**Overlooked interactions:**
- If DAG enforcement is added to start-set (blocking a set from starting until its dependencies are complete), this changes the UX significantly. Users accustomed to starting any set in any order will be blocked. Need to decide: is DAG advisory (displayed but not enforced) or mandatory (blocks start-set)?

- DAG.json needs to be updated when sets are added via `/rapid:add-set` or carried forward via `/rapid:new-version`. Currently add-set.cjs references DAG but the update path needs verification.

**Recommendation:** Implement in phases: (1) Generate DAG.json during roadmap creation (init/new-version), (2) Display DAG in the web dashboard, (3) Optionally enforce ordering in start-set with a `--force` override.

### 5. Branding Skill: Scope Expansion Without Codebase Awareness

**Why this matters:** The milestone brief item #1 says the branding skill should be "refocused from documentation to UI/UX" and be "codebase-aware (not all projects have webapps)." The current branding skill (SKILL.md) is entirely about tone, voice, terminology, and documentation style -- it has zero UI/UX capability.

**Current branding skill scope (what it actually does):**
- Round 1: Project personality (professional/friendly/technical)
- Round 2: Communication style (formal/conversational/terse)
- Round 3: Terminology and naming conventions
- Round 4: Documentation and code comment structure
- Final: Anti-patterns to avoid
- Output: BRANDING.md (50-150 lines of text style guidelines) + index.html (static page)

**What "refocus from documentation to UI/UX" implies:**
- Visual identity: colors, typography, spacing, component styling
- UI component patterns: card layouts, navigation patterns, form styles
- UX patterns: loading states, error handling, empty states, micro-interactions
- Design tokens: CSS variables, theme configuration
- Accessibility defaults: contrast ratios, focus indicators

**Codebase awareness gap:** The current skill does not read any source files. It does not detect whether the project has a frontend, what CSS framework is used (Tailwind, styled-components, CSS modules), or whether a design system already exists. Adding codebase awareness means:
1. Running `context detect` or equivalent to identify frontend presence
2. Scanning for CSS/styling frameworks
3. Checking for existing design tokens or theme files
4. Adapting questions based on findings (skip visual questions for CLI-only projects)

**Overlooked scope creep risk:** Expanding branding to cover visual UI/UX essentially creates a "design system generator" skill. This is a substantially larger undertaking than the current 5-question interview. The skill could easily balloon from 290 lines to 600+ lines, violating the "keep the interview quick, under 2 minutes" principle (line 278).

**Recommendation:** Split into two phases: (1) Add codebase detection to skip UI/UX questions for non-webapp projects, (2) Add a separate `/rapid:design-system` or `/rapid:branding --visual` mode that covers visual identity. Keep the core branding skill focused on communication tone but add an optional visual extension.

### 6. Unit-Test Skill: Framework Detection Requires Init Research Pipeline Change

**Why this matters:** Issues #29 (remove 5-group limit) and #31 (make framework-agnostic) interact. Removing the limit is straightforward (change one line), but making the test runner framework-agnostic requires changes across three layers.

**Current hardcoding in unit-test SKILL.md:**
- Line 193: "Write test files using `node --test` framework (node:test + node:assert/strict)"
- Line 195: "Run tests with: node --test {testFile}"
- Line 237: "Re-run each fixed test with: node --test {testFile}"
- Line 333: "Uses `node --test` framework. All tests use Node.js built-in test runner"
- The agent role for unit-tester likely reinforces this

**What framework-agnostic requires:**
1. **Detection at init time:** The init research phase or `context detect` must identify the project's test framework (jest, vitest, pytest, cargo test, go test, etc.) and store it in config
2. **Config storage:** `.planning/config.json` needs a `testFramework` or `testRunner` field that skills can read
3. **Skill parameterization:** The unit-test SKILL.md must read the config and inject the correct test command into agent prompts instead of hardcoding `node --test`
4. **Agent role update:** The rapid-unit-tester role module must be told to use the project's native test framework, not node:test

**Overlooked interactions:**
- The RAPID project itself uses `node --test` for its own tests. If the unit-test skill becomes framework-agnostic, RAPID's self-testing should still work. But the skill instructions must distinguish between "RAPID's internal tests" and "the target project's tests."
- The retry mechanism (Step 5a, lines 202-247) spawns a `rapid-test-fixer` agent. This agent's instructions also hardcode `node --test`. It needs the same parameterization.
- Test file naming conventions differ by framework: `.test.cjs` (node:test), `.spec.ts` (jest/vitest), `_test.go` (go test), `test_*.py` (pytest). The SKILL.md hardcodes `{originalFile}.test.cjs` (line 195). This must be parameterized.
- The 5-group limit removal (#29) is technically simple but has a token usage implication: each concern group spawns a separate agent. With 10+ groups, that is 10+ agent spawns. Need to add a soft cap or dynamic batching strategy.

**Recommendation:** Implement in order: (1) Remove the 5-group limit and add dynamic batching (merge small groups, split large ones), (2) Add test framework detection to init/context detect, (3) Parameterize the unit-test SKILL.md and unit-tester role to read from config.

### 8. Init Step 4D/5 REQUIREMENTS.md Overwrite: Verified Bug

**Why this matters:** Issue #7 reports that Step 4D writes REQUIREMENTS.md before Step 5's scaffold overwrites it. This is confirmed by code inspection.

**Evidence chain:**
1. Step 4D (SKILL.md line 456): "Write the acceptance criteria to `.planning/REQUIREMENTS.md` using the Write tool."
2. Step 5 (SKILL.md line 475): Runs `init scaffold --mode {mode}`
3. `init.cjs` line 237: `'REQUIREMENTS.md': () => generateRequirementsMd(opts.name)` is in the file generators
4. `init.cjs` lines 263-264 (reinitialize mode): Unconditionally writes all files
5. `init.cjs` lines 300-304 (fresh mode): Unconditionally writes all files
6. Only upgrade mode (lines 284-291) checks if a file exists before writing

**The fix has two viable approaches:**
- **Approach A (reorder):** Move scaffold (Step 5) to before Step 4D. Scaffold creates the template, then Step 4D overwrites it with real content. Simple but requires rearranging skill steps.
- **Approach B (guard in scaffold):** Have scaffold check if REQUIREMENTS.md already has non-template content before overwriting. This is the "additive-only" approach suggested in the issue. But detecting "non-template content" requires comparing against the template output, which is fragile.
- **Approach C (remove from scaffold):** Remove REQUIREMENTS.md from the scaffold's file generators entirely. Let Step 4D be the sole creator. This is cleanest but means REQUIREMENTS.md will not exist if Step 4D is skipped or fails.

**Recommendation:** Approach A (reorder) is safest. Move scaffold to run before the acceptance criteria write. If the skill flow makes this awkward, Approach C (remove from scaffold generators) is the second choice, paired with having Step 4D always create the file even with a template if no user criteria exist.

### 10. RAPID_ROOT Sweep: Active Use in Every Skill Preamble

**Why this matters:** Issue #3 says to "sweep/remove unused RAPID_ROOT references from codebase." But RAPID_ROOT is not unused -- it is actively used in every skill's environment preamble.

**The pattern in every skill preamble (103 files reference RAPID_ROOT):**
```bash
RAPID_ROOT="${CLAUDE_SKILL_DIR}/../.."
if [ -z "${RAPID_TOOLS:-}" ] && [ -f "$RAPID_ROOT/.env" ]; then
  export $(grep -v '^#' "$RAPID_ROOT/.env" | xargs)
fi
```

RAPID_ROOT serves as a local variable to locate the `.env` file. It is not used as a persistent environment variable or stored in config. The issue's original complaint was about RAPID_ROOT being "computed but unused" -- but it IS used, just as a transient shell variable for .env loading.

**The actual problem:** RAPID_ROOT is computed from `CLAUDE_SKILL_DIR` using `/../..`, which assumes a fixed directory depth. This is fragile if the skill directory structure changes. However, it currently works and is consistent across all skills.

**What "sweep and remove" actually means:**
- The issue says "remove RAPID_ROOT if not used anywhere." Since it IS used (as a .env locator), removing it requires an alternative .env loading mechanism.
- The alternative would be to derive the .env path from RAPID_TOOLS: `dirname "$(dirname "${RAPID_TOOLS}")"` gives the project root from `src/bin/rapid-tools.cjs`.
- OR: load .env from a well-known location (e.g., `~/.rapid/.env` or the project root detected at setup time).

**Recommendation:** Do NOT blindly remove RAPID_ROOT. Instead, rename the variable or replace the .env loading pattern with a RAPID_TOOLS-derived approach. The sweep should focus on removing any references to RAPID_ROOT as a persistent env var (e.g., in setup.sh or config), not removing the transient shell variable from skill preambles.

### 14. Web Dashboard DAG Display: Frontend Library Selection

**Why this matters:** The DAG display in the command center (milestone brief item #3) requires a graph visualization library. The web frontend currently uses React (with Vite, TailwindCSS, zustand, react-query). The GraphPage is a stub.

**Frontend tech stack (from web/frontend):**
- React + TypeScript + Vite
- TailwindCSS for styling
- Zustand for state management
- React Query (@tanstack/react-query) for server state
- React Router for navigation

**Graph library options not yet chosen:**
- `react-flow` / `@xyflow/react`: Full-featured, interactive, drag-and-drop. Heavyweight (~150KB). Best for editable graphs.
- `dagre` + custom SVG: Lightweight layout engine. Requires manual rendering. Best for read-only display.
- `d3-dag`: D3-based DAG layout. Moderate weight. Good for algorithmic layouts but requires D3 knowledge.
- `elkjs`: Eclipse Layout Kernel for JS. Good automatic layouts. Can work with React.

**Overlooked considerations:**
- The DAG is relatively small (typically 3-15 nodes). A heavyweight library like react-flow may be overkill.
- The dashboard is already a separate deployment (Python backend + React frontend). Adding a large graph library increases bundle size.
- The DAG should show set status (color coding: pending=gray, executing=blue, complete=green, merged=purple). This requires the backend to merge DAG.json with STATE.json data.
- Interactive features (clicking a set node to see details, expanding to show waves) would benefit from react-flow but are not strictly necessary for v1.

**Recommendation:** Start with `dagre` for layout + custom SVG/React components for rendering. Keep it lightweight and read-only for v1. Add react-flow later if interactive editing is needed.

## Cross-Cutting Concern: Developer Experience

The expanded flow (8+ gray areas, richer context, deferred decisions) could take 5-10 minutes. This needs to be balanced against the "keep the interview quick" principle. The `--skip` flag provides an escape hatch, but the interactive path should still be efficient.

## Cross-Cutting Concern: Data Concerns

The REQUIREMENTS.md overwrite bug (#7) is a data loss concern. No other data integrity issues are introduced by this milestone. Deferred decisions (new feature) create new data that needs to be preserved across milestones -- the archival step in /new-version should NOT archive DEFERRED.md (or should only archive resolved items).

## Recommendations

Ordered by retrofit cost and dependency chain:

1. **Discuss-set question count redesign (concern #1):** Design the new selection mechanism FIRST before implementing any other discuss-set changes. The question count constraint (multiples of 4) and the AskUserQuestion 4-option limit create a fundamental UX design challenge. Get this right before layering on richer context (#12) or deferred decisions (#2). -- *First step: Prototype the selection UI flow on paper/pseudocode.*

2. **Init criteria encoding schema (concern #3):** Design the `UIUX-001` encoding format and storage schema before implementing. This touches init, verifier, and roadmapper -- all need to agree on the format. -- *First step: Write a schema proposal (YAML in REQUIREMENTS.md with category-prefixed IDs).*

3. **Init REQUIREMENTS.md overwrite fix (concern #8):** Fix this immediately since it is trivial and prevents data loss. Either reorder Steps 4D/5 or remove REQUIREMENTS.md from scaffold generators. -- *First step: Move scaffold (Step 5) before Step 4D in init SKILL.md.*

4. **DOCS.md and fishjojo1 clone URL sweep (concern #11):** Fix the URL in DOCS.md, README.md, plugin.json, and context/CODEBASE.md. Do not modify archive files. -- *First step: Find-and-replace in the 4 active files.*

5. **Deferred decisions infrastructure (concern #2):** Create `.planning/DEFERRED.md` and add basic CLI commands. Then wire into /new-version and discuss-set. -- *First step: Add `deferred list` and `deferred add` CLI subcommands.*

6. **DAG integration (concern #4):** Wire the existing dag.cjs module into the init/new-version roadmap pipeline to generate DAG.json. Then build the dashboard display. -- *First step: Add DAG.json generation to the roadmapper output processing.*

7. **Review log-issue CLI flags (concern #7):** Update `review.cjs` to use `parseArgs` for the `log-issue` subcommand. Auto-generate `id` and `createdAt`. Update skill docs (unit-test, bug-hunt, uat) to match. -- *First step: Add parseArgs to the log-issue handler.*

8. **Unit-test framework-agnostic (concern #6):** Remove the 5-group limit first (simple). Then add test framework detection to init/context detect. Then parameterize the unit-test skill. -- *First step: Remove "up to 5 concern groups maximum" from unit-test SKILL.md line 113.*

9. **New-version comprehensiveness (concern #9):** Add structured goal capture (deferred items, open issues, confirmation step) after deferred decisions infrastructure is in place. -- *First step: Add deferred items surfacing to Step 2 of new-version SKILL.md.*

10. **Discuss-set question quality and enrichment (concern #12):** Add pre-discussion research substep after the question count redesign is stable. Layer in code context, prior decisions, and pros/cons tables per issue #19. -- *First step: Add a "gather prior context" substep that reads previous sets' CONTEXT.md files.*

11. **Branding skill codebase awareness (concern #5):** Add `context detect` check to determine if the project has a frontend. Skip visual UI/UX questions for non-webapp projects. -- *First step: Add a codebase detection step before the interview rounds.*

12. **RAPID_ROOT sweep (concern #10):** Replace the `RAPID_ROOT` pattern in skill preambles with a `RAPID_TOOLS`-derived approach. Do not remove the .env loading capability. -- *First step: Create a reusable preamble snippet that derives project root from RAPID_TOOLS.*
