# Features Research -- v3.1.0 Polish & Cleanup

## Feature Inventory

### Feature: Set Status Naming Fix

- **Description:** Rename `-ing` status suffixes to past tense across the entire codebase: `discussing` becomes `discussed`, `planning` becomes `planned`, `executing` becomes `executed`. This is a grammatical consistency fix -- the `-ing` form suggests ongoing action, while the status actually represents a completed phase gate (i.e., the set has been discussed, not that it is actively being discussed right now).
- **Complexity:** moderate
- **Dependencies:** independent (no other features depend on this, but should be implemented before other features to avoid merge conflicts)

#### Decomposition

1. **Schema and transition core rename** -- Update `state-schemas.cjs` (Zod enum) and `state-transitions.cjs` (transition map keys/values). These two files are the single source of truth for all status values.
2. **State machine references** -- Update `state-machine.cjs` which has 11 occurrences in `validateDiskArtifacts` and related functions checking status strings like `['planning', 'executing', 'complete', 'merged']`.
3. **Library module updates** -- Update references across all `src/lib/*.cjs` files: `worktree.cjs` (20 occurrences: phase labels, `formatWaveSummary`, `STATUS_SORT_ORDER`, `deriveNextActions`), `execute.cjs` (11 occurrences in `assembleExecutorPrompt` and verification), `display.cjs` (3 occurrences in `STAGE_VERBS`/`STAGE_BG`), `merge.cjs` (20 occurrences), `review.cjs` (6 occurrences), `init.cjs` (25 occurrences), `plan.cjs` (14 occurrences), `context.cjs` (1 occurrence), `contract.cjs` (1 occurrence), `core.cjs` (5 occurrences), `lock.cjs` (2 occurrences), `dag.cjs` (1 occurrence), `resolve.cjs` (2 occurrences), `tool-docs.cjs` (4 occurrences).
4. **CLI tool updates** -- Update `src/bin/rapid-tools.cjs` (36 occurrences) which passes status values to transition commands and displays them.
5. **Test suite updates** -- Update all `*.test.cjs` files. Key files: `state-transitions.test.cjs` (38 occurrences -- validates all transitions), `state-machine.test.cjs` (34 occurrences), `state-machine.lifecycle.test.cjs` (31 occurrences), `worktree.test.cjs` (45 occurrences), `merge.test.cjs` (44 occurrences), `plan.test.cjs` (37 occurrences), `init.test.cjs` (66 occurrences), `review.test.cjs` (17 occurrences), `display.test.cjs` (6 occurrences), `state-schemas.test.cjs` (6 occurrences), `rapid-tools.test.cjs` (65 occurrences), and others.
6. **Skill file updates** -- Update all 19 skill `SKILL.md` files that reference status values in instructions, validation checks, and display text. Key skills: `discuss-set` (4 direct refs), `execute-set` (6 direct refs), `pause` (8 refs to `executing`), `resume` (1 ref), `status` (3 refs), `new-version` (1 ref), `plan-set` (refs), `start-set` (refs).
7. **Agent module updates** -- Update 13 files in `src/modules/` that reference status values in agent identity/role descriptions.
8. **Hook updates** -- Update `src/hooks/rapid-task-completed.sh` (5 occurrences).
9. **STATE.json migration** -- Handle existing `STATE.json` files that contain old status values. Need a one-time migration in `migrateStateVersion()` or a new migration function in the CLI entry point.

#### Implementation Strategies

**Option 1: Big-Bang Find-and-Replace**
- Approach: Use `replace_all` across all files in a single pass, starting from the schema/transition core outward. Run the full test suite after to validate.
- Pros: Simple, fast, atomic -- no intermediate states where some files use old names and others use new.
- Cons: Large diff, harder to review. If any edge case is missed, tests catch it immediately.
- Libraries/services: None needed; pure string replacement.

**Option 2: Backwards-Compatible Migration with Aliasing**
- Approach: Add new status values alongside old ones, accept both in Zod schema via `z.enum([...]).or(z.enum([...]))`, write a coercion layer, then deprecate old values.
- Pros: Backwards compatible, existing STATE.json files work without migration.
- Cons: Significantly more complex, adds temporary code that must be removed later. Doubles the validation surface area.

**Recommended:** Option 1. This is an internal tool with no external API consumers. A clean find-and-replace with a migration function for existing STATE.json is the simplest and most correct approach. The existing test suite (662 total occurrences across 51 files) provides comprehensive coverage to catch any missed renames.

#### Data Model Impact
- Modified entities: `SetStatus` enum in `state-schemas.cjs` (values change)
- Modified entities: `SET_TRANSITIONS` map in `state-transitions.cjs` (keys change)
- No new entities
- Existing STATE.json files need one-time migration (value replacement in JSON)

#### API Surface
- CLI commands affected: `state transition set <m> <s> <status>` -- the `<status>` argument values change
- No new endpoints
- All existing `state get` outputs will show new status names

### Feature: Parallel Wave Execution

- **Description:** Enable independent waves within a set to execute concurrently instead of sequentially. Currently, `execute-set` spawns one `rapid-executor` agent per wave and processes waves in strict order (1, 2, 3...). This feature allows waves that have no inter-wave dependencies (i.e., waves at the same DAG level) to run simultaneously using the Agent tool's parallel invocation capability.
- **Complexity:** complex
- **Dependencies:** independent (but benefits from Status Naming Fix being done first to avoid merge conflicts)

#### Decomposition

1. **Wave dependency analysis** -- Add a function to determine which waves within a set can run in parallel. Currently, the DAG operates at the set level (sets grouped into waves), not at the intra-set level. Intra-set waves are numbered sequentially and assumed to be sequential. Need to either: (a) add intra-set dependency edges, or (b) assume wave N depends on wave N-1 (current model) and only parallelize within a single wave's tasks.
2. **Parallel agent dispatch** -- Modify `execute-set/SKILL.md` to spawn multiple `rapid-executor` agents concurrently when waves are independent. The Agent tool supports parallel invocations -- multiple Agent calls in the same message execute concurrently.
3. **Git commit serialization** -- When multiple executors work in the same worktree concurrently, git operations will conflict. Need either: (a) separate worktrees per wave, or (b) a commit queue/lock mechanism.
4. **Completion detection and barrier** -- All parallel executors must complete before advancing to dependent waves. Parse `RAPID:RETURN` from each parallel executor and handle mixed results (some COMPLETE, some BLOCKED).
5. **Error handling for partial completion** -- If one parallel executor fails/blocks while others succeed, the system needs to handle partial wave completion gracefully. Write partial completion markers, allow re-entry to execute only failed portions.
6. **Progress tracking** -- Update the progress banner and re-entry detection to handle multiple concurrent executors within a wave.

#### Implementation Strategies

**Option 1: Intra-Wave Task Parallelism (Same Worktree)**
- Approach: Within a single wave, identify independent tasks from the PLAN.md and dispatch them to parallel agents. All agents work in the same worktree with a commit lock.
- Pros: No additional worktree overhead. Works within the existing wave-level plan structure.
- Cons: Git conflicts are likely when parallel agents modify the same worktree. Lock contention serializes git operations, reducing the parallelism benefit. Complex error recovery.
- Libraries/services: Would use existing `lock.cjs` for git serialization.

**Option 2: Independent Wave Groups (Separate Worktrees per Executor)**
- Approach: Create temporary sub-worktrees for each parallel executor within a wave. Each executor gets its own copy. After all complete, merge sub-worktrees back into the set's main worktree.
- Pros: No git conflicts between parallel executors. Clean isolation.
- Cons: Significant disk overhead. Complex merge step. Worktree creation/cleanup adds latency. Git worktrees within worktrees may not work reliably.
- Libraries/services: Git worktree commands, merge tooling.

**Option 3: Sequential Waves, Parallel Sets (Status Quo Enhancement)**
- Approach: Keep waves sequential within a set, but improve the existing DAG-level set parallelism. The current system already supports running independent sets in parallel (different worktrees). Enhance `execute-set` to detect when multiple sets are at the "ready to execute" state and prompt the user to launch parallel execution sessions.
- Pros: Leverages existing isolation model (one worktree per set). No git conflict issues. Already partially implemented.
- Cons: Does not address the user's specific request for intra-set wave parallelism.
- Libraries/services: None new.

**Option 4: Wave-Level Plan Splitting with Parallel Dispatch**
- Approach: During `plan-set`, detect independent task groups within each wave and emit separate sub-plan files (e.g., `wave-1a-PLAN.md`, `wave-1b-PLAN.md`). During `execute-set`, dispatch independent sub-plans to parallel executors using separate worktrees (one per sub-plan). After all sub-plan executors complete, merge their branches into the set's worktree.
- Pros: Clean separation at the planning level. Executors have full isolation. Dependency analysis happens once at plan time.
- Cons: Requires changes to both planning and execution phases. Merging sub-plan branches adds complexity. Sub-plan interdependency detection may be imperfect.
- Libraries/services: `dag.cjs` for dependency analysis, git branch merge.

**Recommended:** Option 4 (Wave-Level Plan Splitting). This approach cleanly separates the dependency analysis concern (at plan time) from the execution concern (at execute time). It maintains full git isolation between parallel executors, avoiding the thorny commit serialization problem. The merge step can reuse existing merge tooling from the `merge.cjs` module.

However, the pragmatic first step might be Option 3 -- documenting and improving the existing set-level parallelism -- if true intra-set parallelism proves too complex for this milestone. The skill instructions already say "one executor per wave, sequential waves" as a key principle; changing this requires careful architectural consideration.

#### Data Model Impact
- New entities: Sub-plan files (`wave-{N}-{group}-PLAN.md`) if using Option 4
- New entities: Sub-wave completion markers (`WAVE-{N}-{group}-COMPLETE.md`)
- Modified entities: PLAN.md structure may need dependency annotations
- No schema changes to STATE.json (set-level state only)

#### API Surface
- Modified: `execute-set` skill to support parallel dispatch
- Modified: `plan-set` skill to emit dependency-aware sub-plans (Option 4)
- New: Potentially a `rapid-tools plan analyze-deps` CLI subcommand for task dependency analysis
- Modified: `execute-set` re-entry detection to handle partial sub-wave completion

### Feature: Old Command Cleanup

- **Description:** Remove the deprecation stubs for old v2 commands that were kept during the v3.0 migration. These stubs currently exist as thin skill files that display a deprecation message directing users to the v3 equivalent. Now that v3 is established, remove these stubs entirely to clean up the command/skill list.
- **Complexity:** trivial
- **Dependencies:** independent

#### Decomposition

1. **Identify deprecated skills** -- Remove 5 deprecated skill directories:
   - `skills/discuss/SKILL.md` (points to `discuss-set`)
   - `skills/execute/SKILL.md` (points to `execute-set`)
   - `skills/plan/SKILL.md` (points to `plan-set`)
   - `skills/set-init/SKILL.md` (points to `start-set`)
   - `skills/wave-plan/SKILL.md` (points to `plan-set`)
   - `skills/new-milestone/SKILL.md` (points to `new-version`)
2. **Identify deprecated command files** -- Remove the `commands/` directory entirely or selectively. Currently contains 6 files: `assumptions.md`, `context.md`, `plan.md`, `help.md`, `install.md`, `init.md`. These appear to be older command definitions that have been superseded by skill equivalents in `skills/`. Need to verify which are truly redundant vs still active.
3. **Update help skill** -- Remove the "Deprecated Commands" table from `skills/help/SKILL.md` (lines 95-107).
4. **Update display module** -- Remove legacy stage names from `STAGE_VERBS` and `STAGE_BG` in `src/lib/display.cjs` (entries for `set-init`, `discuss`, `wave-plan`, `execute`).
5. **Verify no remaining references** -- Grep for any remaining references to the old command names in active skills and modules.

#### Implementation Strategies

**Option 1: Full Removal**
- Approach: Delete all deprecated skill files and command stubs. Remove references in help. Remove legacy display entries.
- Pros: Clean, simple, reduces maintenance burden.
- Cons: Users who have muscle memory for old commands will get "unknown command" instead of a helpful redirect. However, this is acceptable for an internal tool.
- Libraries/services: None needed.

**Option 2: Graceful Removal with Runtime Redirect**
- Approach: Instead of static stub files, add a runtime check in the CLI that detects old command names and prints a redirect message. Remove the skill/command files but keep the redirect logic centralized.
- Pros: Users still get a helpful message when using old names. Single point of maintenance.
- Cons: Adds code rather than removing it. The old commands were skill-level (not CLI-level), so CLI redirects wouldn't catch `/rapid:discuss` invocations.

**Recommended:** Option 1 (Full Removal). The deprecated commands have been in place since v3.0. Users should have transitioned by now. If anyone uses an old command, they will see it is not available and can check `/rapid:help` for current commands. Keeping stubs adds visual noise to the skill list.

#### Data Model Impact
- No data model changes
- No schema changes

#### API Surface
- Removed: 6 deprecated skills (`discuss`, `execute`, `plan`, `set-init`, `wave-plan`, `new-milestone`)
- Potentially removed: 6 command files in `commands/` (need to verify which are active)
- Modified: `help` skill -- removal of deprecated commands table

### Feature: Install Command Flexibility

- **Description:** Make the install command version-aware so it can detect when RAPID_TOOLS points to an outdated version and offer to reinstall/update. Currently, the install skill checks if RAPID_TOOLS is configured in shell config files and skips if found (`ALREADY_CONFIGURED`). It does not check whether the configured path points to the current version or an older installation.
- **Complexity:** moderate
- **Dependencies:** independent

#### Decomposition

1. **Version detection mechanism** -- Add a way to determine the installed RAPID version. Options: (a) read `package.json` version from the configured `RAPID_TOOLS` path, (b) add a `rapid-tools version` CLI subcommand, or (c) embed version in the `.env` file.
2. **Version comparison** -- Compare installed version vs current version. The current RAPID root is known (`${CLAUDE_SKILL_DIR}/../..`); the installed RAPID root can be derived from the configured `RAPID_TOOLS` path.
3. **Update detection in install skill** -- Modify `skills/install/SKILL.md` Step 2 to check not just "is RAPID_TOOLS configured?" but also "does it point to the current installation?" and "is it the current version?". If outdated, prompt the user to update.
4. **Update flow** -- When an update is detected, offer to: (a) update the shell config to point to the new installation path, (b) re-run `setup.sh` for the new version, (c) skip (keep old version).
5. **CLI version subcommand** -- Add `rapid-tools version` to the CLI that outputs the version from `package.json`. This is useful for diagnostics and version comparison.
6. **Setup script version check** -- Modify `setup.sh` to handle the case where `.env` already exists with a different RAPID_TOOLS path (different installation directory or version).

#### Implementation Strategies

**Option 1: Path-Based Detection**
- Approach: Compare the `RAPID_TOOLS` path in shell config with the current plugin's `rapid-tools.cjs` path. If they differ, the user has a different (possibly older) installation configured. Prompt to update.
- Pros: Simple, no version parsing needed. Detects both version mismatches and path mismatches (e.g., user moved the repo).
- Cons: Does not distinguish between "outdated" and "different installation" (user might intentionally have two copies).
- Libraries/services: None; pure path comparison.

**Option 2: Semantic Version Comparison**
- Approach: Add a `rapid-tools version` CLI command. During install, run the existing `RAPID_TOOLS` to get its version, compare with current `package.json` version using semver comparison. Offer update only if current is newer.
- Pros: Precise version comparison. Can distinguish between upgrade, downgrade, and same-version.
- Cons: Requires the old installation to be functional enough to run `rapid-tools version`. Adds a CLI subcommand.
- Libraries/services: Could use Node.js built-in version parsing (no semver library needed for basic comparison).

**Recommended:** Option 2 (Semantic Version Comparison). This is more robust and gives the user clear information about what version they have vs what version is being installed. The `rapid-tools version` command is also independently useful for diagnostics. For basic semver comparison of MAJOR.MINOR.PATCH strings, no external library is needed.

#### Data Model Impact
- No schema changes
- No new entities
- `.env` file may be updated during version migration

#### API Surface
- New CLI subcommand: `rapid-tools version` -- outputs `{ "version": "3.1.0", "path": "/path/to/rapid-tools.cjs" }`
- Modified: `skills/install/SKILL.md` -- adds version detection and update flow in Step 2
- Modified: `setup.sh` -- handles existing installations more gracefully

## Feature Dependency Graph

```
Set Status Naming Fix (independent, do first to reduce merge conflicts)
Old Command Cleanup (independent)
Install Command Flexibility (independent)
Parallel Wave Execution (independent, but complex -- do last)
```

All four features are independent -- none depends on another's data model or API. However, there is a practical ordering concern: the Status Naming Fix touches nearly every file in the codebase (662 occurrences across 51 source files, 19 skill files, 13 module files). If other features are developed in parallel branches, the status rename will cause merge conflicts with any branch that references status values.

---

## Implementation Order

1. **Set Status Naming Fix** -- Do first. This is a cross-cutting rename that touches 80+ files. Doing it first avoids merge conflicts with other features. It is purely mechanical (find-and-replace) and the test suite provides immediate validation.

2. **Old Command Cleanup** -- Do second. This is the simplest feature (delete 6-12 files, update one help file, remove 4 display entries). Quick win that reduces noise.

3. **Install Command Flexibility** -- Do third. Moderate complexity, self-contained changes to the install skill and setup script. Adds a small CLI subcommand. Well-scoped.

4. **Parallel Wave Execution** -- Do last. This is the most complex feature, touching the core execution architecture. It benefits from the other three features being complete and stable. May need further scoping discussion before implementation.
