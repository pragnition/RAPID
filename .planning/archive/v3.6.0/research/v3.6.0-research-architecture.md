# Architecture Research: v3.6.0 Workflow & UX Polish

## Current Architecture

RAPID is a Claude Code plugin organized in five layers: Skill (25 user-facing SKILL.md command definitions), Agent (26 specialized AI workers built from role modules), CLI (rapid-tools.cjs command router at ~3500 lines), Library (21+ core modules in src/lib/), and Storage (.planning/ state files + .rapid-worktrees/ git worktrees). The architecture uses CommonJS throughout, validates state with Zod schemas, and coordinates parallel work via file-based locking (proper-lockfile).

### Strengths Relevant to This Milestone

1. **Transactional state management.** All mutations flow through `withStateTransaction()` with Zod validation. The state schema hierarchy (ProjectState > milestones > sets > waves > jobs) provides a solid foundation. The SET_TRANSITIONS map (`pending -> discussed -> planned -> executed -> complete -> merged`) is well-defined with self-transitions (`executed -> executed`, `discussed -> discussed`) for idempotent re-entry.

2. **Modular skill architecture.** Each skill is a self-contained SKILL.md file with clear step-by-step instructions. Skills orchestrate by spawning agents and calling CLI commands. This separation makes it straightforward to modify individual workflows (init, execute-set, merge, review) without affecting others.

3. **DAG library is feature-complete.** `dag.cjs` provides `createDAG()`, `validateDAG()`, `assignWaves()`, and `getExecutionOrder()` with both v1 and v2 DAG formats. `plan.cjs` provides `writeDAG()` for persistence. `add-set.cjs` provides `recalculateDAG()` for rebuilding from state + contracts. The algorithms are correct; the issue is that DAG.json is not created/updated at all the right lifecycle points.

4. **Solo mode foundation exists.** `worktree.cjs` already has `setInitSolo()`, `isSoloMode()`, `getSetDiffBase()`, and solo detection in `createWorktree`, `removeWorktree`, `formatStatusTable`, and `reconcileRegistry`. The registry marks solo entries with `solo: true` and `startCommit`. The merge command detects solo sets and fast-paths them. The gaps are in post-execution state management and merge messaging.

5. **Display system is extensible.** `display.cjs` uses a stage-verb/stage-bg pattern for branded banners. Adding new stages (like `branding`) requires only two map entries. The ANSI color system is basic 16-color for terminal compatibility.

### Areas Requiring Modification

1. **DAG.json creation is scattered.** DAG.json is currently created/written in three places: (a) `plan.writeDAG()` called from `add-set.recalculateDAG()`, (b) `plan.decomposeIntoSets()` during planning orchestration, and (c) `plan write-dag` CLI command that writes stdin JSON to disk. However, the init flow (roadmapper) does NOT create DAG.json -- it writes STATE.json, ROADMAP.md, CONTRACT.json, and DEFINITION.md but skips DAG generation. The `merge order` CLI reads DAG.json and crashes with ENOENT if it does not exist. The `execute prepare-context` CLI also reads DAG.json.

2. **Init flow uses prose-based conversation.** The current init skill (Step 4B) uses batched freeform AskUserQuestion calls -- the agent asks 3-4 broad topic questions and the user responds in prose. The milestone requires restructured AskUserQuestion with 4 sub-questions per call, iterating until satisfied, then summary + confirmation, plus a granularity preference (number of sets).

3. **Execute-set does not reliably transition to complete.** The execute-set skill (Step 6) calls `state transition set "${MILESTONE}" "${SET_ID}" complete` after all waves finish. When this Bash call fails (CWD drift, worktree vs project root context), the set stays in `executed` state. The review skill requires `complete` or `executed` status, but historically only accepted `complete`. The v3.5.0 fix relaxed review to accept `executed` too, but the root cause (transition failure) persists.

4. **Solo sets do not auto-transition to merged.** After solo execution completes, the set transitions to `complete` but never to `merged`. Since solo sets have no worktree branch to merge, the user must manually run `/merge` which should immediately recognize the solo set and mark it merged. The v3.5-todo explicitly requests: "when we execute a solo set, we also need to mark the state as merged in the end."

5. **Review cycle skills run automatically.** The bug-hunt skill runs up to 3 cycles (hunter -> advocate -> judge -> bugfix) without user confirmation between cycles. The unit-test skill similarly proceeds through plan -> execute phases without inter-cycle confirmation. The milestone requires user confirmation before starting each new cycle.

6. **No branding system exists.** There is no `/branding` command, no branding guideline storage, no mechanism for agents to reference UI branding decisions. This is a new feature requiring a new skill, potentially a new agent role, a storage location for branding artifacts, and integration points in existing execution/planning agents.

## Recommended Architecture Pattern

- **Pattern:** Incremental extension with defensive hardening
- **Rationale:** The six milestone goals are a mix of bug fixes (3 items), workflow refinements (2 items), and one new feature. The existing 5-layer architecture is sound. Each change should operate within its layer boundaries: skills control orchestration flow, the CLI handles command dispatch, and libraries provide reusable logic. No new layers or fundamental structural changes are needed.
- **Evidence:** All prior milestones (v3.0 through v3.5.0) have successfully extended this architecture. The v3.5.0 milestone addressed similar robustness issues (state transition alignment, DEFINITION.md gaps, untracked file collisions) using the same incremental approach.

## Module Organization

### Module: DAG Management (dag.cjs + plan.cjs + add-set.cjs)

- **Responsibility:** Create, validate, persist, and recalculate dependency graphs from state + contracts
- **Public interface:** `createDAG()`, `validateDAG()`, `assignWaves()`, `getExecutionOrder()`, `writeDAG()`, `recalculateDAG()`
- **Dependencies:** plan.cjs (writeDAG), state-machine.cjs (readState), contract.cjs (createOwnershipMap)
- **Data ownership:** `.planning/sets/DAG.json`
- **Change needed:** DAG.json must be created at init time (post-roadmap acceptance) and updated whenever sets are added/removed. The `recalculateDAG()` function in add-set.cjs is the correct entry point for recalculation from state -- it should also be called during init after STATE.json is written.

### Module: Init Flow (init.cjs + skills/init/SKILL.md)

- **Responsibility:** Project scaffolding, discovery conversation, research orchestration, roadmap generation
- **Public interface:** `scaffoldProject()`, `detectExisting()`, `generateProjectMd()`, `generateConfigJson()`
- **Dependencies:** state-machine.cjs (createInitialState), version.cjs
- **Data ownership:** `.planning/` root files (PROJECT.md, STATE.json, config.json, etc.)
- **Change needed:** (1) Step 4B restructured from prose batches to structured AskUserQuestion with 4 sub-questions per call. (2) Add granularity preference step asking user to choose set count. (3) After roadmap acceptance (Step 9), call recalculateDAG or equivalent to generate DAG.json. (4) Summary + confirmation before proceeding.

### Module: Execute & State (execute.cjs + skills/execute-set/SKILL.md)

- **Responsibility:** Wave execution orchestration, context assembly, verification, state transitions
- **Public interface:** `prepareSetContext()`, `assembleExecutorPrompt()`, `verifySetExecution()`, `getChangedFiles()`
- **Dependencies:** worktree.cjs, plan.cjs, verify.cjs, contract.cjs, compaction.cjs
- **Data ownership:** WAVE-COMPLETE.md markers, PLAN-DIGEST.md files, HANDOFF.md files
- **Change needed:** (1) Make state-to-complete transition more robust (pre-flight CWD check, absolute path for RAPID_TOOLS). (2) For solo sets, auto-transition to `merged` after `complete` transition succeeds.

### Module: Solo Mode (worktree.cjs)

- **Responsibility:** Solo set initialization, diff-base tracking, registry management
- **Public interface:** `setInitSolo()`, `isSoloMode()`, `getSetDiffBase()`
- **Dependencies:** lock.cjs (acquireLock)
- **Data ownership:** `.planning/worktrees/REGISTRY.json` solo entries
- **Change needed:** (1) After solo execution completes and set transitions to `complete`, auto-transition to `merged`. This can be done either in the execute-set skill (after the `complete` transition, check if solo, then transition to `merged`) or as a post-execution hook. (2) Merge skill should detect solo sets early and display "Solo set -- no merge needed, already merged" message.

### Module: Merge Pipeline (merge.cjs + skills/merge/SKILL.md)

- **Responsibility:** DAG-ordered merging, conflict detection, resolution, integration gates
- **Public interface:** `mergeSet()`, `detectConflicts()`, `resolveConflicts()`, `checkCascadeImpact()`
- **Dependencies:** worktree.cjs, dag.cjs, state-machine.cjs
- **Data ownership:** `.planning/MERGE-STATE.json`
- **Change needed:** Solo set detection in Step 3a-solo already exists and fast-paths with `{ merged: true, solo: true }`. The skill needs enhanced messaging: "Set '{setName}' is a solo set and was already merged during execution. No merge operation needed." If the set is already in `merged` state, the merge skill should display this and skip gracefully.

### Module: Review Cycle Skills (bug-hunt/SKILL.md + unit-test/SKILL.md)

- **Responsibility:** Adversarial bug hunting (hunter-advocate-judge-bugfix pipeline), unit test generation/execution
- **Public interface:** Skills only (SKILL.md orchestration)
- **Dependencies:** review.cjs (deduplicateFindings, extractAcceptanceCriteria, serializeReviewScope)
- **Data ownership:** REVIEW-BUGS.md, REVIEW-UNIT.md, REVIEW-ISSUES.json
- **Change needed:** Both skills currently run cycles automatically. Add AskUserQuestion confirmation before each new cycle (cycle 2+). The bug-hunt already has `MAX_BUGFIX_CYCLES = 3` and a cycle loop (Step 3). The confirmation should be inserted at the start of Step 3 for cycles > 1.

### Module: Branding System (NEW)

- **Responsibility:** Capture and formalize UI branding guidelines through user discussion
- **Public interface:** `/rapid:branding` skill, `rapid-branding` agent (new)
- **Dependencies:** display.cjs (stage banner), core.cjs (output utilities)
- **Data ownership:** `.planning/branding/BRANDING.md` (new artifact)
- **Change needed:** Create new skill, new agent role, new storage location. Other execution agents need awareness: the executor prompt assembly should check for `.planning/branding/BRANDING.md` and include it as context if present. This parallels the existing `qualityContext` and `uiContext` optional injection in `enrichedPrepareSetContext()`.

### Module: Display System (display.cjs)

- **Responsibility:** Branded ANSI terminal banners
- **Public interface:** `renderBanner()`, `STAGE_VERBS`, `STAGE_BG`
- **Dependencies:** None (zero dependencies)
- **Data ownership:** None (stateless utility)
- **Change needed:** Add `branding` stage verb and background color entries.

## Data Flow

### DAG.json Lifecycle (current vs. proposed)

**Current flow (gaps marked with X):**
```
1. /init -> roadmapper produces state JSON -> STATE.json written  X (no DAG.json created)
2. /add-set -> recalculateDAG() -> DAG.json written               OK
3. /plan-set -> decompose into waves -> (no DAG update)            X (DAG should reflect wave structure)
4. /execute-set -> reads DAG.json for wave ordering                X (ENOENT if never created)
5. /merge -> reads DAG.json for merge ordering                     X (ENOENT if never created)
```

**Proposed flow:**
```
1. /init -> after roadmap acceptance + STATE.json written -> call recalculateDAG() -> DAG.json created
2. /add-set -> recalculateDAG() -> DAG.json updated                (no change)
3. /plan-set -> after wave decomposition -> optionally update DAG   (nice-to-have, not required)
4. /execute-set -> reads DAG.json (now guaranteed to exist)         (no change needed)
5. /merge -> reads DAG.json (now guaranteed to exist)               (no change needed)
6. Defensive: commands that read DAG.json should handle missing file gracefully with clear error
```

### Init Flow Lifecycle (proposed)

```
Step 1-3: Prerequisites, git check, existing detection        (no change)
Step 4A:  Logistics (name, team size, model)                   (no change)
Step 4B:  Discovery -- RESTRUCTURED:
          Loop:
            Ask 4 structured sub-questions via AskUserQuestion
            Parse responses, identify gaps
            If gaps remain -> loop
          Summary display + AskUserQuestion confirmation
Step 4C:  Granularity preference -- NEW:
          AskUserQuestion: "How many sets should the project be decomposed into?"
          Options: suggested numbers (3-5 small, 5-8 medium, 8-12 large, custom)
          Pass preference to roadmapper
Step 5:   Scaffold                                             (no change)
Step 6:   Brownfield detection                                 (no change)
Step 7:   Research agents                                      (no change)
Step 8:   Synthesis                                            (no change)
Step 9:   Roadmap generation                                   (pass granularity preference)
Step 9b:  DAG.json creation -- NEW:
          After roadmap accepted + STATE.json written:
          Call recalculateDAG(cwd, milestoneId)
Step 10:  Auto-commit (includes DAG.json)                      (no change)
Step 11-13: Completion                                         (no change)
```

### Solo Set Lifecycle (proposed)

```
1. /start-set N --solo      -> setInitSolo() registers in REGISTRY.json with solo:true
2. /discuss-set N            -> normal discussion (works on main branch)
3. /plan-set N               -> normal planning
4. /execute-set N            -> executes on main branch
   Step 6 (Complete):
     a. state transition set <m> <s> complete
     b. IF solo mode detected from registry:
        state transition set <m> <s> merged     <- NEW
        Display: "Solo set auto-merged (no branch merge needed)"
5. /merge N                  -> detect solo, display: "Solo set already merged"
                                Skip all merge operations
                                Display: "Set '{name}' is a solo set. It was auto-merged after execution."
```

### Review Cycle Lifecycle (proposed)

```
Bug Hunt:
  Cycle 1: Run automatically (no confirmation needed for first cycle)
  Cycle 2: AskUserQuestion: "Cycle 1 found {N} bugs, {M} fixed. Start cycle 2?"
            Options: "Continue" / "Stop here"
  Cycle 3: AskUserQuestion: "Cycle 2 found {N} bugs, {M} fixed. Start cycle 3 (final)?"
            Options: "Continue" / "Stop here"

Unit Test:
  Plan phase: Run automatically + approval step (already exists)
  Execute phase: Runs once (no multi-cycle currently)
  No change needed for unit-test cycles (it is single-cycle by design)
```

### State Management

- **Where state lives:** `.planning/STATE.json` (Zod-validated JSON), `.planning/worktrees/REGISTRY.json` (worktree tracking), `.planning/sets/DAG.json` (dependency graph)
- **How state is read and written:** Always through `withStateTransaction()` for STATE.json, `withRegistryUpdate()` for REGISTRY.json, `writeDAG()` for DAG.json
- **Consistency guarantees:** File-based locking via proper-lockfile with 300s timeout and PID-based stale lock cleanup. Atomic writes via Zod validation at every mutation boundary.
- **New state for branding:** `.planning/branding/BRANDING.md` (markdown artifact, not JSON state). Read by execution agents as optional context. Not part of the state machine.

## API Design

### New CLI Commands

**Branding:**
```
node rapid-tools.cjs branding generate    # Output branding context for agent injection
```

This follows the existing pattern of `context detect`, `review scope`, etc. -- a CLI command that reads artifacts and outputs structured data for skill/agent consumption.

### Modified CLI Behavior

**DAG.json creation at init:**
No new CLI command needed. The init skill should call `node "${RAPID_TOOLS}" plan recalculate-dag <milestone-id>` after writing STATE.json. This maps to the existing `recalculateDAG()` function. A new CLI subcommand under `plan` may be needed:

```
node rapid-tools.cjs plan recalculate-dag <milestone-id>    # Rebuild DAG.json from state + contracts
```

This wraps `add-set.recalculateDAG()` which already exists as a library function but has no direct CLI entry point (it is called internally by `add-set` command).

**Solo auto-merge in execute-set:**
No new CLI command needed. The execute-set skill adds a conditional call after `state transition set ... complete`:

```bash
# Check if solo
SOLO_CHECK=$(node "${RAPID_TOOLS}" worktree status --json | node -e "...")
if [ "$IS_SOLO" = "true" ]; then
  node "${RAPID_TOOLS}" state transition set "${MILESTONE}" "${SET_ID}" merged
fi
```

## Scaling Considerations

- **Set count granularity preference:** The new granularity preference in init affects the roadmapper's output, not the architecture. The DAG, state machine, and worktree system already support arbitrary set counts. The primary concern is that very high set counts (15+) create proportionally more worktrees, branches, and merge operations. The init skill should guide users toward reasonable numbers (3-12) based on project complexity.

- **Review cycle confirmation:** Adding user confirmation between bug-hunt cycles does not affect scalability. The cycles are already bounded (max 3). The change reduces unnecessary computation when early cycles find no bugs.

- **DAG.json creation at init:** `recalculateDAG()` reads all CONTRACT.json files and builds the graph. For the typical RAPID project (3-12 sets), this is instantaneous. For edge cases with 20+ sets, it would still be fast (O(V+E) toposort).

## Architecture Risks

1. **CWD drift in execute-set state transition (existing bug, high impact).** The execute-set skill runs the state transition in a Bash command that may execute in the worktree directory rather than the project root. Since `rapid-tools.cjs` uses `findProjectRoot()` which walks up the directory tree looking for `.planning/`, this works IF the worktree has a `.planning/` symlink or the CWD is correct. But solo mode runs on main where `.planning/` exists, so solo mode is not affected. The fix should use absolute paths for `RAPID_TOOLS` and explicit CWD in Bash calls.

   **Mitigation:** In the execute-set skill, always use `cd {projectRoot} &&` before state transition calls, or pass `--cwd` to the CLI.

2. **Solo auto-merge race condition (low impact).** If the `complete` transition succeeds but the `merged` transition fails (e.g., lock contention), the set is in `complete` but the skill has displayed "auto-merged." The user would see an inconsistent state.

   **Mitigation:** Use a retry loop (max 2 attempts) for the `merged` transition. If both fail, warn the user explicitly: "Auto-merge failed. Run `/rapid:merge {set}` to complete."

3. **Init flow AskUserQuestion restructuring scope creep (medium impact).** The current init skill is 824 lines of carefully sequenced instructions. Restructuring the discovery conversation (Step 4B) from prose batches to structured sub-questions risks breaking the flow if not done carefully. The agent must still gather the same 10 discovery areas -- just through a different interaction pattern.

   **Mitigation:** Keep the same 10 discovery areas and 4-batch grouping structure. Change only the question format (from freeform prose to structured options with 4 sub-questions per AskUserQuestion call). The completion check logic stays the same.

4. **Branding integration surface area (medium impact).** The branding system needs to be referenced by execution agents to be useful. This means modifying `enrichedPrepareSetContext()` in execute.cjs to load branding context, and ensuring the branding file path is standard. If the branding context is too large, it may push execution prompts over context limits.

   **Mitigation:** Keep BRANDING.md concise (target: 50-150 lines). Add it as an optional context section in `enrichedPrepareSetContext()` following the same pattern as `qualityContext` and `uiContext`.

5. **DAG.json ENOENT on older projects (medium impact).** Projects initialized before this fix will not have DAG.json. Commands that read DAG.json (execute prepare-context, merge order, worktree status) will fail.

   **Mitigation:** Add defensive DAG.json checks in every command that reads it. If missing, either (a) auto-generate from STATE.json + contracts using `recalculateDAG()`, or (b) display a clear error: "DAG.json not found. Run `node rapid-tools plan recalculate-dag <milestone>` to generate it."

## Recommendations

Ordered by implementation priority:

1. **Create DAG.json at init time** (Critical): After roadmap acceptance and STATE.json write in the init skill, call `recalculateDAG(cwd, milestoneId)` to generate DAG.json. Add a `plan recalculate-dag` CLI subcommand that wraps the existing library function. Also add defensive DAG.json-missing handling in `commands/execute.cjs` and `commands/merge.cjs` -- attempt auto-regeneration before failing.

2. **Fix execute-set state-to-complete transition** (Critical): In the execute-set skill Step 6, ensure the `state transition set ... complete` command runs with explicit CWD set to project root. Use `cd "${PROJECT_ROOT}" && node "${RAPID_TOOLS}" state transition set ...` or pass an absolute path. Add error checking after the transition and a recovery path if it fails (retry once, then warn user).

3. **Implement solo set auto-merge** (Critical): In the execute-set skill Step 6, after the `complete` transition succeeds, check the registry for `solo: true`. If solo, immediately transition to `merged` and display "Solo set auto-merged." In the merge skill Step 3a-solo, enhance the message to explain why no merge is needed. Ensure the merge skill gracefully handles sets already in `merged` state.

4. **Restructure init discovery to structured AskUserQuestion** (High): Rewrite init skill Step 4B to use structured AskUserQuestion with 4 sub-questions per call. Each call covers one topic batch. The agent iterates until gaps are filled, then displays a summary for user confirmation. Add Step 4C for granularity preference (number of sets). Pass the preference to the roadmapper agent.

5. **Add review cycle confirmation** (High): In the bug-hunt skill Step 3, add an AskUserQuestion gate before cycles 2 and 3. Present cycle results so far and ask "Continue to cycle {N}?" with "Continue" and "Stop here" options. If "Stop here," skip to Step 4 (completion). The unit-test skill is single-cycle by design and does not need this change.

6. **Build branding system** (Medium): Create `skills/branding/SKILL.md` as a new optional post-init skill. Create `src/modules/roles/role-branding.md` for the branding discussion agent. Store output at `.planning/branding/BRANDING.md`. Add branding context loading to `enrichedPrepareSetContext()` in execute.cjs. Add `branding` stage to display.cjs. Update CLAUDE.md or relevant agent roles to mention the branding artifact when it exists.

7. **Add new display stage for branding** (Low): Add `'branding': 'BRANDING'` to `STAGE_VERBS` and `'branding': '\x1b[105m'` (bright magenta) to `STAGE_BG` in display.cjs. Also add `'bug-hunt': 'BUG HUNTING'` and `'unit-test': 'UNIT TESTING'` if not already present (these stages are referenced in skills but may not have banner entries).

8. **Defensive DAG.json auto-generation** (Low): In any command that reads DAG.json and gets ENOENT, attempt `recalculateDAG()` automatically before failing. This provides backward compatibility for projects initialized before this fix.
