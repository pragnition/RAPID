# Pitfalls Research -- v3.6.0 Workflow & UX Polish

## Critical Pitfalls

### Pitfall: DAG.json Path Inconsistency Between Merge and Other Commands
- **Category:** correctness
- **Stack:** RAPID CLI (src/commands/merge.cjs vs src/commands/execute.cjs)
- **Description:** The merge command reads DAG.json from `.planning/DAG.json` (line 266 of merge.cjs) while every other command reads from `.planning/sets/DAG.json`. This means the merge resolver's heuristic context silently fails to load the DAG on any project that was initialized after the path was standardized. The error is swallowed by an empty catch block (`catch { /* no DAG data */ }`), so the user never sees a warning -- merge resolution just proceeds without DAG ordering context, which can produce incorrect merge priority decisions.
- **Trigger:** Any project using the current init flow where DAG.json is written to `.planning/sets/DAG.json` via `plan.writeDAG()`. The merge resolve subcommand then reads from the wrong path.
- **Impact:** Merge conflict resolution uses incorrect or empty DAG ordering, potentially choosing the wrong winner in structural conflicts. Silent failure -- no user-visible error.
- **Prevention:** Consolidate all DAG.json reads to use a single `loadDAG(cwd)` function in `plan.cjs` that encapsulates the path. Update `merge.cjs` line 266 to use `.planning/sets/DAG.json`. The migrate skill already documents this migration (line 171 of migrate/SKILL.md), but the code was never updated.
- **Detection:** Search for `'.planning', 'DAG.json'` vs `'.planning', 'sets', 'DAG.json'` across all source files. Any path.join with only two segments after `.planning` is suspect.
- **References:** `src/commands/merge.cjs:266`, `src/lib/plan.cjs:258`, `skills/migrate/SKILL.md:171`

### Pitfall: Execute Agent Does Not Transition State to Complete
- **Category:** correctness
- **Stack:** RAPID set lifecycle (execute-set SKILL.md, state-transitions.cjs)
- **Description:** The milestone brief explicitly identifies this bug: "When the execute agent is done, it does not update the state to complete and therefore the review agent refuses to review." The execute-set skill's Step 6 transitions to `complete` and the review skill's Step 0c requires `complete` or `executed` status. If the state transition in Step 6 fails silently (e.g., the CLI call errors but the skill continues), the set remains in `executed` state, which should be accepted by review but may not be if the status check is strict.
- **Trigger:** When the execute-set skill's Step 6 state transition command fails (e.g., state lock contention, malformed milestone ID variable, or the skill being interrupted between wave completion and Step 6). Also possible if an executor returns COMPLETE but the orchestrator fails to reach Step 6 due to context window exhaustion.
- **Impact:** The review pipeline refuses to operate on the set. The user is stuck without understanding why. The workaround requires manual state manipulation via `rapid-tools state transition`.
- **Prevention:** (1) Add `executed` as an explicitly accepted status in the review skill's status check (it already is in the skill text, but verify the CLI command does not reject it). (2) Consider making the state transition more resilient -- retry on lock contention. (3) Add a final confirmation message after the transition so the user can see if it succeeded.
- **Detection:** After running `/rapid:execute-set`, check `node "${RAPID_TOOLS}" state get --all` and verify the target set shows `complete`. If it shows `executed`, the transition failed.
- **References:** `skills/execute-set/SKILL.md:380-387`, `skills/review/SKILL.md:122-124`, `src/lib/state-transitions.cjs:4-9`

### Pitfall: Solo Mode State Not Transitioning to Merged After Execution
- **Category:** correctness
- **Stack:** RAPID solo mode (worktree.cjs, state-transitions.cjs, merge SKILL.md)
- **Description:** Solo sets execute directly on main without a worktree branch. The milestone requires that solo sets be automatically marked as `merged` after execution completes, since there is no separate merge step needed. Currently, the state machine requires the sequence `executed -> complete -> merged`, but solo execution only reaches `complete`. The merge skill detects solo sets and fast-paths them, but if the user never runs `/merge`, the set permanently remains in `complete` state. This blocks milestone completion tracking and DAG-dependent sets that wait for `merged` status.
- **Trigger:** User runs a solo project through `/execute-set` but never runs `/merge` (because they rightly expect that solo sets on main do not need merging).
- **Impact:** DAG-dependent sets may wait indefinitely for the solo set to reach `merged`. The milestone progress display shows incomplete even though all work is done.
- **Prevention:** Two options: (1) At the end of execute-set Step 6, if the set is solo, automatically transition from `complete` to `merged`. This requires adding a solo-detection check in the execute-set skill. (2) Have the merge skill proactively inform the user at start that solo sets are auto-merged, and auto-transition them without requiring user intervention.
- **Detection:** Check STATE.json after solo execution. If status is `complete` but registry entry has `solo: true`, the auto-merge transition was missed.
- **References:** `src/lib/worktree.cjs:441-450`, `skills/merge/SKILL.md:134-141`, `src/lib/state-transitions.cjs:8`

## High-Risk Pitfalls

### Pitfall: AskUserQuestion 60-Second Timeout in Init Flow Redesign
- **Category:** correctness / compatibility
- **Stack:** Claude Code Plugin API (AskUserQuestion tool)
- **Description:** AskUserQuestion has a documented 60-second timeout. The init flow redesign calls for "structured AskUserQuestion with 4 sub-questions at a time" in iterative rounds. If any round takes the user more than 60 seconds to respond (which is likely for thoughtful project description), the tool call times out and the agent may fail or retry, losing context from previous rounds.
- **Trigger:** User takes more than 60 seconds to read and respond to a batch of 4 sub-questions during `/init`.
- **Impact:** Init flow crashes or loses accumulated context. User has to restart initialization from scratch.
- **Prevention:** (1) Keep each AskUserQuestion call focused on 2-3 questions maximum (not 4). The current init skill already uses topic batches of 2-3 -- do not increase batch size. (2) Use freeform mode for open-ended questions so the user can type quickly. (3) Persist intermediate answers to a scratch file (e.g., `.planning/.init-progress.json`) so context survives timeouts. (4) Test the timeout behavior explicitly during development.
- **Detection:** If AskUserQuestion returns an error or empty response after a delay, the timeout was hit. Check the agent's error output for timeout-related messages.
- **References:** [Claude Code AskUserQuestion documentation](https://claudelog.com/faqs/what-is-ask-user-question-tool-in-claude-code/)

### Pitfall: AskUserQuestion Cannot Be Used From Sub-Agents
- **Category:** compatibility
- **Stack:** Claude Code Plugin API (AskUserQuestion tool)
- **Description:** AskUserQuestion can only be called from the main Claude context, not from spawned sub-agents. Several milestone features require user confirmation from contexts that may run as sub-agents: bug-hunt cycle confirmation, unit-test cycle confirmation, and the branding discussion. If these are implemented as sub-agent interactions, the AskUserQuestion calls will silently fail or throw errors.
- **Trigger:** Implementing cycle confirmation prompts inside the `rapid-bug-hunter` or `rapid-unit-tester` agent roles instead of in the orchestrating skill (SKILL.md).
- **Impact:** The confirmation prompt is never shown to the user, and the agent either errors out or proceeds without confirmation (defeating the purpose).
- **Prevention:** All user-facing confirmation prompts MUST be placed in the skill orchestrator (SKILL.md), not in the agent role definitions (role-*.md). The bug-hunt and unit-test skills already orchestrate cycles in their SKILL.md files -- add the confirmation prompt at Step 3 (before starting a new cycle iteration) in the skill, not in the agent.
- **Detection:** Grep for `AskUserQuestion` in `src/modules/roles/` files. Any occurrence there is a potential failure point since roles run as sub-agents.
- **References:** [Claude Code AskUserQuestion limitations](https://www.atcyrus.com/stories/claude-code-ask-user-question-tool-guide)

### Pitfall: DAG.json ENOENT When No Sets Exist Yet
- **Category:** correctness
- **Stack:** RAPID CLI (src/commands/execute.cjs, src/commands/worktree.cjs, src/commands/merge.cjs)
- **Description:** Multiple commands attempt to read `.planning/sets/DAG.json` with `fs.readFileSync()` and throw ENOENT when the file does not exist. The error from the user's log shows: `ENOENT: no such file or directory, open '/home/kek/Projects/TRust/.planning/sets/DAG.json'`. This happens when commands that read DAG.json are invoked before the planning stage creates it. The milestone goal states "DAG.json should be updated/created by all the relevant agents," which means multiple agents will need to write DAG.json, increasing the surface area for this error.
- **Trigger:** Running `/merge`, `/status`, or any command that reads DAG.json before `/plan` has been run, or after a milestone change that wipes the sets directory.
- **Impact:** Cryptic ENOENT error that doesn't tell the user what to do. Blocks the merge pipeline entirely.
- **Prevention:** (1) Wrap all DAG.json reads in a helper function that returns `null` or a default empty DAG when the file doesn't exist (the `worktree.cjs` status command already does this gracefully with `catch { /* no DAG available */ }`, but `execute.cjs` throws a hard error). (2) When implementing "DAG.json created by all relevant agents," ensure `start-set` creates a minimal DAG.json with just the newly started set if one doesn't exist yet. (3) Standardize error messages to suggest the correct recovery action.
- **Detection:** Run any DAG-consuming command on a project without `.planning/sets/DAG.json`. If you get ENOENT instead of a helpful message, this pitfall is present.
- **References:** `v3.5-todo.md:5-6`, `src/commands/execute.cjs:87-93`

### Pitfall: Concurrent DAG.json Writes From Multiple Agents
- **Category:** data-loss / correctness
- **Stack:** RAPID CLI (plan.cjs writeDAG, add-set.cjs recalculateDAG)
- **Description:** The milestone goal requires DAG.json to be "updated/created by all the relevant agents." Currently, `writeDAG()` in `plan.cjs` uses bare `fs.writeFileSync()` with no locking. If multiple agents or skills attempt to update DAG.json concurrently (e.g., two parallel `start-set` or `add-set` operations), the last write wins and the other's changes are lost. This is different from STATE.json which uses `withStateTransaction()` for lock-protected atomic writes.
- **Trigger:** Two agents running `add-set` or `recalculateDAG` at the same time for different sets in the same milestone.
- **Impact:** DAG.json loses nodes or edges from one of the concurrent writes. The lost set becomes invisible to the DAG-ordered merge pipeline.
- **Prevention:** (1) Protect DAG.json writes with the same locking mechanism used for STATE.json -- add a `withDAGTransaction()` wrapper that acquires a `dag` lock before reading, mutating, and writing. (2) Alternatively, make DAG.json a derived artifact that is always regenerated from STATE.json + CONTRACT.json files (which is what `recalculateDAG` already does), and never written by individual agents.
- **Detection:** Run two `add-set` commands in rapid succession. Check if both sets appear in DAG.json afterward.
- **References:** `src/lib/plan.cjs:257-259`, `src/lib/add-set.cjs:71`

### Pitfall: Review Cycle Confirmation Breaking Existing Automatic Flow
- **Category:** correctness
- **Stack:** RAPID review pipeline (skills/bug-hunt/SKILL.md, skills/unit-test/SKILL.md)
- **Description:** The milestone requires bug-hunt and unit-test to "prompt the user for confirmation upon starting a new cycle" instead of running cycles automatically. The current bug-hunt skill has a hard-coded `MAX_BUGFIX_CYCLES = 3` loop (Step 3) with automatic advancement. Adding a confirmation prompt between cycles introduces a new failure mode: if the user declines to continue, the skill must cleanly exit with partial results. Currently, the cycle loop has no early-exit-with-results path -- it either completes all cycles or hits zero findings.
- **Trigger:** User selects "Stop" or "No more cycles" after cycle 1, but the skill has no logic to write REVIEW-BUGS.md with partial cycle data and exit gracefully.
- **Impact:** If not handled correctly: (1) REVIEW-BUGS.md may not be written, losing all findings from completed cycles. (2) The skill may crash trying to continue the loop without valid cycle data. (3) Issues may not be logged for findings from completed cycles.
- **Prevention:** (1) Add an explicit early-exit path after the confirmation prompt that writes REVIEW-BUGS.md with all findings accumulated so far (`allAcceptedBugs`), logs issues, and prints the completion banner. (2) Make sure the `modifiedFiles` and `allAcceptedBugs` arrays persist across the prompt boundary. (3) Test the "decline at cycle 2" path explicitly.
- **Detection:** Run bug-hunt, let cycle 1 complete and find bugs, then decline cycle 2 at the confirmation prompt. Verify REVIEW-BUGS.md exists and contains cycle 1 findings.
- **References:** `skills/bug-hunt/SKILL.md:112-338`

## Medium-Risk Pitfalls

### Pitfall: Solo Mode Working on Main Branch Risks Corrupting Production Code
- **Category:** data-loss
- **Stack:** Git (solo mode, worktree.cjs)
- **Description:** Solo mode executes directly on the current branch (typically main). Unlike worktree mode where failed executions can be discarded by deleting the worktree, solo mode changes are committed directly to main. If an executor produces broken code, the commits are on main and require manual `git revert` or `git reset` to undo.
- **Trigger:** An executor agent produces incorrect or partial code during solo execution, commits it to main, then the skill errors out before verification.
- **Impact:** main branch contains broken code. Other workflows (CI/CD, other developers) see broken state.
- **Prevention:** (1) Record the `startCommit` hash before execution begins (already done in `setInitSolo`). (2) Before transitioning to `complete`, run verification. If verification fails, warn the user and provide the `git reset --hard {startCommit}` command but do NOT run it automatically. (3) Consider creating a temporary branch even in solo mode for safety, merging to main only on success.
- **Detection:** After solo execution, run `git log --oneline {startCommit}..HEAD` to see all commits made. If any test or verification fails, the user needs to manually inspect.
- **References:** `src/lib/worktree.cjs:441-450`, `skills/execute-set/SKILL.md:194`

### Pitfall: Branding Skill Creating Circular Dependency With Init
- **Category:** correctness
- **Stack:** RAPID skill pipeline (init, branding)
- **Description:** The milestone introduces a new `/branding` command that runs "optionally after /init" and generates UI branding guidelines that "naturally the other agents need to know about." If branding guidelines are stored in a location that the init flow reads (e.g., `.planning/BRANDING.md`), re-running `/init` could overwrite or conflict with branding data. Additionally, if agents reference branding guidelines during execution but the user skipped `/branding`, every agent that tries to load branding guidelines needs a graceful fallback.
- **Trigger:** (1) User runs `/init` -> `/branding` -> re-runs `/init` (reinitialize mode) -> loses branding data. (2) User skips `/branding` but agents try to load branding config.
- **Impact:** (1) Lost branding work on re-init. (2) Agents crash or produce errors when branding file is missing.
- **Prevention:** (1) Store branding output outside the files that `/init` overwrites (e.g., `.planning/BRANDING.md` is safe if init's `scaffoldProject` does not include it in `fileGenerators`). (2) When implementing agent references to branding, always use try/catch or existence checks. (3) The reinitialize mode's backup mechanism preserves old data in `.planning.backup.{timestamp}/`, but the user would need to manually restore branding -- document this.
- **Detection:** Run `/init` reinitialize after creating branding guidelines. Check if `.planning/BRANDING.md` survives (it should, since `scaffoldProject` only creates PROJECT.md, STATE.md, ROADMAP.md, REQUIREMENTS.md, config.json, STATE.json).
- **References:** `src/lib/init.cjs:233-240`, `v3.5-todo.md:18`

### Pitfall: State Machine Missing "discussed" -> "executed" Shortcut
- **Category:** correctness
- **Stack:** RAPID state machine (state-transitions.cjs)
- **Description:** The state transition map allows: `pending -> discussed -> planned -> executed -> complete -> merged`. The init flow redesign asks for granularity preferences (number of sets). For very simple projects, users might want to skip detailed planning and go straight from discussion to execution. The current transition map does not allow `discussed -> executed` -- it requires going through `planned` first. If the granularity system tries to skip the planning step for very simple single-set projects, the transition will be rejected.
- **Trigger:** Implementing a "minimal granularity" option where a single set skips detailed wave planning and goes directly to execution.
- **Impact:** State transition throws `Invalid transition: "discussed" -> "executed"`. The user's chosen granularity preference cannot be honored.
- **Prevention:** If the granularity system needs a skip-planning path, add `executed` to the allowed transitions from `discussed`. Alternatively, auto-generate a minimal wave-1-PLAN.md and transition through `planned` programmatically without requiring user interaction.
- **Detection:** Attempt `node "${RAPID_TOOLS}" state transition set {milestone} {set} executed` when set status is `discussed`. If it fails, this transition is missing.
- **References:** `src/lib/state-transitions.cjs:3-5`

### Pitfall: proper-lockfile Compromised Lock Silently Continues
- **Category:** data-loss
- **Stack:** proper-lockfile 4.1.2 (src/lib/lock.cjs)
- **Description:** The lock module's `onCompromised` handler logs a warning to stderr but allows the operation to continue. A compromised lock means another process may have acquired the lock, meaning two processes could be mutating STATE.json simultaneously. The comment says "Log but don't crash -- let the operation complete," but this is dangerous because the operation may be writing stale data that overwrites another process's changes.
- **Trigger:** Long-running state transactions (>5 minutes, which is the `STALE_THRESHOLD`) or process slowdowns that cause the lock to be reclaimed by another process.
- **Impact:** Two concurrent state mutations can produce a corrupted or partially overwritten STATE.json. The second process's changes silently win.
- **Prevention:** (1) Consider crashing on compromised lock instead of continuing -- throw an error that tells the user to re-run the command. (2) Alternatively, re-read state after detecting compromise and re-apply the mutation. (3) Reduce the likelihood by increasing `STALE_THRESHOLD` for long operations or adding heartbeat checks.
- **Detection:** Check stderr for `[RAPID] Lock "state" compromised` messages. If these appear during execution, data integrity may be compromised.
- **References:** `src/lib/lock.cjs:59-62`, [proper-lockfile documentation](https://github.com/moxystudio/node-proper-lockfile)

### Pitfall: Init Flow Granularity Preference Not Propagated to Roadmap Agent
- **Category:** correctness
- **Stack:** RAPID init pipeline (init SKILL.md, roadmapper agent)
- **Description:** The milestone adds a granularity preference (user chooses number of sets) during init. This preference needs to flow from the init skill to the roadmapper agent, which generates the roadmap and set decomposition. If the granularity preference is stored only in the init skill's context (as a variable) and not persisted to a file the roadmapper can read (e.g., `config.json` or `PROJECT.md`), the roadmapper agent will ignore the user's preference and decompose into however many sets it deems appropriate.
- **Trigger:** User specifies "3 sets" during init, but the roadmapper agent generates 7 sets because it never received the granularity constraint.
- **Impact:** User frustration -- they explicitly requested a certain granularity but got something different.
- **Prevention:** (1) Persist the granularity preference in `.planning/config.json` (e.g., `planning.target_sets: 3`). (2) Pass it explicitly in the roadmapper agent's prompt context. (3) After the roadmapper returns, validate that the number of sets matches the user's preference (within +/- 1) and ask for confirmation.
- **Detection:** During init, choose a specific number of sets. After roadmap generation, count the actual sets produced. If they differ significantly from the preference, this propagation failed.
- **References:** `v3.5-todo.md:14`, `src/lib/init.cjs:177-191`

## Anti-Patterns for This Stack

### Anti-Pattern: Swallowing File Read Errors With Empty Catch Blocks
- **What people do:** Wrap `fs.readFileSync()` in try/catch with empty catch blocks (e.g., `catch { /* no DAG data */ }`) to make code "resilient."
- **Why it fails:** The code silently proceeds without critical data, producing incorrect results. The DAG.json path inconsistency bug went undetected because the wrong-path read was swallowed by `catch { /* no DAG data */ }`.
- **Correct approach:** If the file is optional, use `fs.existsSync()` first and log a debug message when absent. If the file is required, throw a descriptive error with recovery instructions.
- **Example:** `src/commands/merge.cjs:265-271` silently ignores DAG.json read failure, causing merge resolution to proceed without DAG ordering context.

### Anti-Pattern: Hardcoding File Paths Instead of Using Centralized Path Constants
- **What people do:** Construct paths like `path.join(cwd, '.planning', 'sets', 'DAG.json')` inline in multiple files.
- **Why it fails:** Path changes require finding and updating every inline occurrence. The `.planning/DAG.json` vs `.planning/sets/DAG.json` inconsistency is a direct result of this pattern.
- **Correct approach:** Define path-building functions in a single module (e.g., `plan.cjs` already has `writeDAG` that knows the path). Add a `getDAGPath(cwd)` function and use it everywhere.
- **Example:** DAG.json path is constructed inline in `execute.cjs:90`, `execute.cjs:263`, `worktree.cjs:124`, `worktree.cjs:138`, `merge.cjs:266` -- five different locations, one using a different path.

### Anti-Pattern: Relying on Skill Orchestrator Context Survival for Multi-Step Flows
- **What people do:** Store intermediate state (accumulated findings, cycle counters, modified file lists) in the skill's agent context memory, assuming the context window will persist across long-running sub-agent dispatches.
- **Why it fails:** Claude Code's context window can fill up during long review cycles with multiple sub-agent returns. When context compaction occurs, intermediate state stored only in the conversation may be lost.
- **Correct approach:** Persist intermediate state to files (e.g., `.planning/sets/{setId}/.cycle-state.json`) and reload from disk at the start of each cycle iteration. This also enables re-entry after interruptions.
- **Example:** Bug-hunt SKILL.md stores `modifiedFiles`, `allAcceptedBugs`, and `cycleNumber` as in-memory variables (Step 2, line 108-110).

### Anti-Pattern: Not Distinguishing Between "Status Check" and "Status Transition"
- **What people do:** Check set status by reading STATE.json, then separately run a transition command, without handling the race condition between the check and the transition.
- **Why it fails:** Between the status check and the transition call, another process may have already transitioned the set, causing the transition to fail with an invalid transition error.
- **Correct approach:** Use `withStateTransaction()` for check-and-transition in a single atomic operation. Alternatively, handle transition failures gracefully with a re-read and retry.
- **Example:** The execute-set skill checks status in Step 1, then transitions in Step 3, with potential concurrent `/execute-set` invocations on the same set.

## Version-Specific Gotchas

- **Node.js 18+**: `fs.cpSync()` used in init.cjs reinitialize mode (`line 254`) was added in Node.js 16.7 but only became stable in Node.js 20. On Node.js 18, it works but may emit experimental warnings.
- **proper-lockfile 4.1.2**: The `update` interval (150,000ms = 2.5 min in RAPID config) must be less than `stale` (300,000ms = 5 min). If these values are misconfigured, locks become prematurely stale.
- **Git 2.30+**: `git worktree add -b` fails if the branch already exists. RAPID does not handle this error -- if a previous worktree was manually deleted without running `git worktree prune`, the stale branch reference blocks re-creation.
- **Zod 3.25.76**: The `.passthrough()` modifier is needed on schemas to preserve unknown fields during STATE.json round-trips. If a future milestone adds fields to STATE.json without updating the Zod schema, `ProjectState.parse()` will silently strip them.

## Security Pitfalls

1. **Shell injection in git commands via set names:** `worktree.cjs` uses `execSync(\`git ${args.join(' ')}\`)` which concatenates arguments into a shell string. A malicious set name containing shell metacharacters (e.g., `; rm -rf /`) could execute arbitrary commands. The `createWorktree` function validates against whitespace but not against shell metacharacters. -- **Prevention:** Use `execFileSync('git', [...args])` instead of `execSync` with string concatenation, or validate set names against a strict `[a-z0-9-]` pattern.
2. **Unvalidated user input from AskUserQuestion in init flow:** The init discovery conversation accepts freeform text that is interpolated into prompts sent to sub-agents. If the user input contains RAPID:RETURN markers or prompt injection patterns, it could confuse downstream agents. -- **Prevention:** Sanitize or escape user input before interpolating into agent prompts. Strip any `<!-- RAPID:RETURN` patterns from user-provided text.
3. **Environment variable leakage via .env loading:** The env preamble `export $(grep -v '^#' "$RAPID_ROOT/.env" | xargs)` exports ALL variables from .env into the shell environment, including potentially sensitive ones. If a sub-process or error message logs the environment, secrets could leak. -- **Prevention:** Only export the specific variables needed (e.g., `RAPID_TOOLS`), not all variables from .env.

## Performance Traps

1. **Full STATE.json re-read on every CLI command:** Every `state get`, `state transition`, and `withStateTransaction` call reads the entire STATE.json file. As milestones accumulate and sets grow, STATE.json can become large (>100KB). With frequent CLI calls during execution (wave status checks, transitions), this creates I/O overhead. -- **Mitigation:** Consider adding a lightweight status cache file that is updated alongside STATE.json, or use targeted reads that extract only the needed set/wave data.
2. **Synchronous git operations in worktree.cjs:** `gitExec()` uses `execSync()` with a 30-second timeout. During merge operations that process multiple sets, each set triggers multiple synchronous git commands in series. A slow filesystem or large repo can cause the merge pipeline to take minutes per set. -- **Mitigation:** Where possible, batch git operations (e.g., `git diff --name-only` for multiple branches in one call). For the merge pipeline, consider parallelizing the merge-tree fast-path checks.
3. **Bug-hunt cycle spawning up to 5 parallel sub-agents per cycle:** Each bug-hunt cycle spawns one agent per concern group (up to 5), plus one devil's advocate and one judge -- up to 7 agents per cycle, times 3 cycles = 21 agent spawns. Each spawn has overhead. Adding a confirmation prompt between cycles also adds user-wait latency, stretching the total wall time significantly. -- **Mitigation:** Allow the user to choose "run all cycles without asking" as a confirmation option (opt-in auto-mode alongside the new confirmation default).
