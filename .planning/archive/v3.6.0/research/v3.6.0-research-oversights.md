# Oversights Research -- v3.6.0 Workflow & UX Polish

## Cross-Cutting Concerns

### 1. DAG.json Lifecycle Management Gap

**Why this matters for this project specifically:** The milestone explicitly calls out a DAG.json ENOENT error (`.planning/sets/DAG.json` not found) and states that DAG.json should be "updated/created by all relevant agents." Currently, DAG.json is only created in two places:

1. `add-set.cjs:recalculateDAG()` -- called after `state add-set` during `/rapid:add-set`
2. `plan.cjs:writeDAG()` -- called via CLI `plan write-dag` (reads from stdin)

However, the init flow (`/rapid:init`) never calls either. The roadmapper agent returns a JSON proposal with `roadmap`, `state`, and `contracts` keys -- but no `dag` key. The init SKILL.md (Step 9) writes STATE.json, ROADMAP.md, CONTRACT.json, and DEFINITION.md but never generates or writes DAG.json. The first time DAG.json exists is when `/rapid:add-set` is called (which triggers `recalculateDAG()`), but if the user follows the standard flow (init -> start-set -> discuss-set -> plan-set -> execute-set), DAG.json is never created.

**Where DAG.json is consumed (and will fail without it):**
- `merge.cjs:getMergeOrder()` -- hard fails with ENOENT (no try/catch)
- `commands/execute.cjs` `wave-status` subcommand -- hard fails with a CliError
- `commands/execute.cjs` `reconcile` subcommand -- hard fails with a CliError
- `commands/merge.cjs` `bisect` subcommand -- calls `getMergeOrder()` which hard fails
- `commands/worktree.cjs` `status` -- gracefully ignores missing DAG
- `commands/merge.cjs` `resolve` -- gracefully ignores missing DAG (uses existsSync)

**Path inconsistency:** Most code reads DAG.json from `.planning/sets/DAG.json`, but `commands/merge.cjs` line 266 reads from `.planning/DAG.json` (no `sets/` prefix). This dual-path lookup is partially handled by the migrate skill but could still cause confusion.

**Agents that should create/update DAG.json but do not:**
- The roadmapper agent (during init) -- should produce DAG data in its return JSON
- The init SKILL.md (Step 9) -- should write DAG.json after writing STATE.json
- The plan-set SKILL.md -- after wave decomposition, dependency information changes
- After milestone operations (new-version, add-set) -- add-set already handles this

**Recommendation:** Add DAG.json creation to the init flow immediately after STATE.json is written. The `recalculateDAG()` function in `add-set.cjs` already does exactly this -- it reads STATE.json and CONTRACT.json to build the DAG. Call it from the init SKILL.md after writing STATE.json and contracts. Also fix the path inconsistency in `commands/merge.cjs` line 266.

### 2. Solo Mode State Lifecycle Incompleteness

**Why this matters:** The milestone requires that solo sets "mark the state as merged after solo execution" and that the merge agent should "inform the user that it is a solo set that doesn't require merging." The current implementation is partially there but has gaps.

**Current solo mode behavior:**
- `start-set` with `--solo`: Creates a registry entry with `solo: true`, records `startCommit`, no branch or worktree created. Works correctly.
- `execute-set`: Detects solo from registry. Uses project root as working directory. Executor commits directly to the current branch. State transitions `planned -> executed`. At the end, transitions `executed -> complete`. This works.
- `merge.cjs:mergeSet()`: Correctly detects `solo: true` and returns `{ merged: true, solo: true }` immediately.
- `merge SKILL.md`: Has solo set detection in Step 1d and fast-path in Step 3a-solo.

**What is missing:**
1. **Auto-mark as merged after solo execution:** The execute-set SKILL.md transitions to `complete` but NOT to `merged`. The milestone says solo sets should be marked `merged` after execution. This requires either (a) the execute-set skill doing `complete -> merged` transition for solo sets, or (b) a post-execution hook.
2. **The merge skill still processes solo sets.** Even though `mergeSet()` returns immediately for solo sets, the merge SKILL.md still includes them in the merge plan, dispatches the fast-path check, and runs integration tests. The milestone wants the merge agent to simply "inform the user that it is a solo set that doesn't require merging" -- implying solo sets should be skipped entirely from the merge plan display, not just fast-pathed.
3. **State transition gap:** SET_TRANSITIONS allows `complete -> merged` but does NOT allow `executed -> merged` directly. If the desired behavior is "execute solo set -> auto-merged", there are two options: (a) execute transitions `planned -> executed`, then skill does `executed -> complete`, then skill does `complete -> merged`, or (b) add `executed -> merged` to the transition map. Option (a) is safer since it uses existing transitions.
4. **Review skip for solo merged sets:** If a solo set is auto-marked as `merged` after execution, the review step is bypassed entirely. The user should be informed: "Solo set already merged -- review is optional. Run /rapid:review {setIndex} --post-merge if desired."
5. **Worktree cleanup is N/A:** Solo sets have no worktree to clean up, but the cleanup skill may still list them. This should be handled gracefully.

**Recommendation:** In the execute-set SKILL.md, after all waves complete and verification passes, check if the set is solo. If solo, perform the additional transitions: `executed -> complete -> merged`. Display a message: "Solo set '{SET_ID}' executed and auto-merged (work is already on main)." In the merge SKILL.md, when building the merge plan, filter out sets that are already `merged` AND have `solo: true` in the registry. Display: "Set '{SET_ID}' is a solo set (already merged after execution). No merge action needed."

### 3. Init Flow Redesign: AskUserQuestion Constraint and Discovery Quality

**Why this matters:** The milestone requires replacing "prose-based Q&A" with "structured AskUserQuestion (4 sub-questions at a time)" during the init discovery phase. The current init SKILL.md (Step 4B) uses freeform AskUserQuestion batches for open-ended discovery. The milestone vision is fundamentally different: structured questions with options.

**Current init Step 4B behavior:**
- 4 topic batches, each using a SINGLE freeform AskUserQuestion with 2-4 embedded sub-questions
- User responds in prose (free text)
- Agent analyzes responses and asks follow-ups

**The tension:**
- AskUserQuestion with pre-filled options works well for CLOSED questions ("Which framework?", "Solo or team?")
- The current discovery covers OPEN questions ("What are you building and why?", "Must-have features for v1?")
- Forcing open-ended discovery into 4-option-per-question format risks either: (a) shallow discovery because options constrain responses, or (b) too many rounds because each question covers less ground

**Overlooked UX implications:**
1. **Question fatigue:** If each discovery area becomes 4+ individual AskUserQuestion calls, the total count could reach 15-20 questions. Users may abandon the init flow.
2. **Option quality:** Pre-filled options for "What are you building?" make no sense -- these must remain freeform. The milestone likely means: where options ARE possible, use them; where they are not, continue with freeform but in smaller batches.
3. **The summary + confirmation step:** The milestone says "then summary + confirmation." This is already partially present (the project brief in Step 4B), but it should be a formal AskUserQuestion with "Accept summary / Add more info / Start over" options.
4. **Granularity preference:** The milestone wants the user to "suggest number of sets and let the user choose." This needs to be passed to the roadmapper agent and affects set decomposition. The roadmapper currently has no input for this. It will need a new field in its input context (e.g., `targetSetCount: 5`) and logic to aim for that number of sets.

**The missing interaction flow:**
```
1. Quick logistics (name, team size, model) -- structured options [ALREADY EXISTS]
2. Structured discovery rounds (4 questions per round, with options where possible, freeform where needed) [NEW]
3. Agent self-assessment: "Do I have enough context?" -> if no, more questions [NEW]
4. Summary display + confirmation AskUserQuestion [NEW]
5. Granularity preference: "How many sets?" with suggested options [NEW]
```

**Recommendation:** Redesign Step 4B into a hybrid approach: use AskUserQuestion with options for questions that have enumerable answers (architecture pattern, database choice, deployment target, auth strategy) and freeform for open-ended questions (vision, features, success criteria). Add a new Step 4C for granularity preference that presents 3-4 options like: "3-5 sets (coarse, faster)", "6-8 sets (balanced)", "9-12 sets (fine-grained, more parallel)". Pass the selected count to the roadmapper.

### 4. Review Cycle Confirmation: Behavioral Change Propagation

**Why this matters:** The milestone requires bug-hunt and unit-test to "prompt the user for confirmation before starting a new cycle (instead of running many cycles automatically)." This changes the control flow in two skills.

**Current bug-hunt behavior (SKILL.md Step 3):**
- Cycles 1-3 execute automatically
- `MAX_BUGFIX_CYCLES = 3` (from `review.cjs`)
- No user confirmation between cycles
- Only stops if: cycle limit reached, no findings, modifiedFiles empty, or all findings dismissed/deferred

**Current unit-test behavior:**
- No cycle loop at all -- unit-test runs once (plan -> approve -> execute -> report)
- The milestone may mean that unit-test should also confirm before writing/running tests (currently it already has plan approval in Step 4)

**What needs to change for bug-hunt:**
1. After Step 3.9 (bugfix agent completes), before incrementing cycleNumber and looping back to 3.1, insert an AskUserQuestion:
   - "Bug hunt cycle {N} complete. {fixedCount} bugs fixed, {modifiedFiles.length} files modified."
   - Options: "Continue to cycle {N+1}" / "Stop here"
2. If "Stop here": skip to Step 4 (completion)
3. The first cycle should NOT require confirmation (nothing to confirm yet)
4. Only cycles 2 and 3 require confirmation (they are triggered by bugfix agent having modified files)

**Overlooked implications of this change:**
1. **UAT skill:** The UAT skill does NOT have cycles (runs once on full scope). No changes needed there. But if the user expects similar confirmation behavior across all review skills, this should be documented.
2. **Existing MAX_BUGFIX_CYCLES constant:** Should this remain at 3 or become configurable? If the user always says "Stop here" after cycle 1, the constant is moot. Consider making it a config option in `.planning/config.json`.
3. **Agent return parsing:** The cycle confirmation happens in the SKILL.md orchestrator, not in any agent. No agent changes needed -- only skill file changes.
4. **Progress context between cycles:** When confirming, the user needs context about what happened. Show: total findings, accepted/dismissed counts, files modified, whether new bugs were found in bugfixed files.

### 5. Branding System: Integration Surface Area

**Why this matters:** The milestone introduces a new `/branding` command that runs after `/init` to formalize UI branding guidelines. This is a new cross-cutting concern because the branding output needs to be consumed by multiple downstream agents.

**What does not exist yet:**
1. No `skills/branding/SKILL.md`
2. No branding artifact format (e.g., `.planning/BRANDING.md` or `.planning/branding.json`)
3. No agent role for branding discussion
4. No mechanism for downstream agents to discover and consume branding guidelines

**Integration points where branding would be consumed:**
- `execute.cjs:enrichedPrepareSetContext()` -- currently loads quality and UI contract context, would need to also load branding context
- `ui-contract.cjs:buildUiContext()` -- already provides UI context to executors; branding could extend this
- Agent role modules that produce UI code (executor, bugfix) -- need branding context injected
- The scoper agent (during review) -- could flag branding violations as concerns

**Overlooked design questions:**
1. **Branding vs UI Contract:** The existing `UI-CONTRACT.json` system (from v3.4.0) already defines UI conventions. Is branding a SEPARATE artifact or an extension of UI-CONTRACT? If separate, agents need to load both. If integrated, the existing `buildUiContext()` function needs updating.
2. **Scope of branding:** Does "branding" mean just colors/fonts/spacing, or does it extend to tone-of-voice, iconography, component library choices, animation style? The depth affects how much discussion the skill needs to capture.
3. **Discovery ordering:** The milestone says `/branding` runs "optionally after /init." But branding decisions could affect the roadmap (e.g., adding a "design system" set). Should the roadmapper be aware of branding? If so, branding should run BEFORE roadmap generation (Step 9 of init), not after.
4. **Persistence location:** Following existing conventions, this would be `.planning/BRANDING.md` or `.planning/branding/GUIDELINES.md`. It should be committed alongside other planning artifacts.
5. **Brownfield vs greenfield:** For brownfield projects, the branding skill should analyze existing CSS/design tokens/theme files and present them to the user as starting points rather than starting from scratch.

**Recommendation:** Create branding as a separate artifact at `.planning/BRANDING.md`. Extend `buildUiContext()` in `ui-contract.cjs` to also read BRANDING.md when it exists. This way, all downstream agents that already receive UI context will automatically receive branding context too, with zero changes to executor/review/merge pipelines. The branding skill should be a standalone discussion-based skill similar to discuss-set but focused on visual identity.

### 6. Error Log Fix: DAG.json ENOENT Error Path

**Why this matters:** The specific error in the milestone brief is:
```
[RAPID ERROR] ENOENT: no such file or directory, open '/home/kek/Projects/TRust/.planning/sets/DAG.json'
```

This comes from `merge.cjs:getMergeOrder()` (line 1561):
```javascript
const dagPath = path.join(cwd, '.planning', 'sets', 'DAG.json');
const dagJson = JSON.parse(fs.readFileSync(dagPath, 'utf-8'));
```

No try/catch, no existence check, no graceful fallback. This function is called by:
- `commands/merge.cjs` `order` subcommand (Step 1b of merge SKILL.md)
- `commands/merge.cjs` `bisect` subcommand

**Overlooked error handling patterns across DAG.json consumers:**

| Location | Behavior on Missing DAG.json |
|----------|------------------------------|
| `merge.cjs:getMergeOrder()` | **CRASH** -- no error handling |
| `commands/execute.cjs` `wave-status` | Throws CliError with helpful message |
| `commands/execute.cjs` `reconcile` | Throws CliError with helpful message |
| `commands/worktree.cjs` `status` | Graceful -- sets dagJson to null |
| `commands/merge.cjs` `resolve` | Graceful -- uses existsSync guard |

**The fix is not just error handling.** The real fix is ensuring DAG.json exists. But even with that fix, `getMergeOrder()` should have a defensive guard:

```javascript
function getMergeOrder(cwd) {
  const dagPath = path.join(cwd, '.planning', 'sets', 'DAG.json');
  if (!fs.existsSync(dagPath)) {
    throw new Error('DAG.json not found. Run /rapid:init or /rapid:add-set to generate the dependency graph.');
  }
  const dagJson = JSON.parse(fs.readFileSync(dagPath, 'utf-8'));
  return dag.getExecutionOrder(dagJson);
}
```

### 7. Execute-Set State Transition to Complete: Timing and Reliability

**Why this matters:** The milestone explicitly states: "When the execute agent is done, it does not update the state to complete." The execute-set SKILL.md (Step 6) does have the transition:

```bash
node "${RAPID_TOOLS}" state transition set "${MILESTONE}" "${SET_ID}" complete
```

But the transition is from `executed` to `complete`, and the initial transition (Step 3) is from `planned` to `executed`. The question is: why does the `executed -> complete` transition fail silently?

**Potential root causes:**
1. **The skill ends prematurely.** If the verifier agent (Step 5) encounters an error or timeout, the skill may never reach Step 6. The verifier is non-blocking in design ("verification is non-blocking"), but if the agent itself crashes, the entire skill execution may terminate.
2. **Re-entry confusion.** If the user re-runs execute-set after partial completion, the set is already in `executed` state. Step 3 says "skip transition (re-entry)." After all waves complete, Step 6 tries `executed -> complete`. But if the set was previously transitioned to `complete` in a failed run, the re-entry finds status `complete` and Step 1 says "Already done. STOP." -- so it never re-executes.
3. **The verifier gap.** Step 5 spawns a `rapid-verifier` agent. If this agent returns BLOCKED or its RAPID:RETURN is malformed, the SKILL.md does not define error recovery for the verifier. The skill says "parse RAPID:RETURN from verifier" but has no BLOCKED/CHECKPOINT handling for it. A silent failure here would skip Step 6.
4. **Git commit failure.** Step 6 commits marker files. If `git add` or `git commit` fails (e.g., nothing to commit because markers were already committed in a previous run), the bash block may exit non-zero, potentially halting the skill before the state transition runs.

**Recommendation:** Restructure Step 6 so the state transition runs FIRST (before the git commit). If the state transition succeeds but the git commit fails, the state is correct even though artifacts are uncommitted. Also add explicit error handling for the verifier step: if the verifier fails or returns BLOCKED, skip verification and proceed to Step 6 with a warning. The state transition to `complete` should be robust against verifier failures.

### 8. AskUserQuestion Option Count Constraint

**Why this matters:** Multiple milestone goals involve AskUserQuestion interaction patterns. The Claude Code AskUserQuestion tool has a practical limit on how many options can be displayed clearly. The discuss-set skill already constrains to "exactly 4 gray areas" because of this.

**Cross-cutting places where option count matters:**
1. **Init discovery (goal 3):** "4 sub-questions at a time" -- this is achievable
2. **Granularity preference (goal 3):** "suggest number of sets" -- 3-4 options for granularity levels
3. **Bug-hunt cycle confirmation (goal 6):** 2 options (continue/stop) -- simple
4. **Unit-test cycle confirmation (goal 6):** 2-3 options (continue/modify/stop)
5. **Branding discussion (goal 5):** Multiple rounds of structured questions about colors, typography, spacing

**Overlooked constraint:** The discuss-set skill's "exactly 4 gray areas" rule is a UX decision, not an AskUserQuestion limitation. AskUserQuestion CAN handle more options. But user attention drops with more options. The milestone's "4 sub-questions at a time" aligns with cognitive load research (Miller's Law: 7+/-2 items).

**Recommendation:** Establish a project-wide convention: AskUserQuestion calls should have 2-5 options maximum. Questions with more than 5 possible answers should use freeform mode. Document this in CONVENTIONS.md or a shared module so all skills follow the same pattern.

### 9. Configuration Propagation for New Features

**Why this matters:** The solo mode config (`solo: true` in `.planning/config.json`) is already read by start-set. New features in this milestone may need additional config entries.

**Current config locations and their consumers:**
- `/home/kek/Projects/RAPID/config.json` (runtime): `lock_timeout_ms`, `agent_size_warn_kb` -- read by `lock.cjs`, `build-agents`
- `.planning/config.json` (per-project): `model`, `teamSize`, `name`, `solo` -- read by `start-set`, `init`

**New config entries that may be needed:**
1. `targetSetCount: number` -- granularity preference from init (goal 3)
2. `reviewCycleConfirmation: boolean` -- whether bug-hunt/unit-test prompt between cycles (goal 6) -- or this could be hardcoded as always-on
3. `brandingEnabled: boolean` -- whether branding guidelines exist (goal 5) -- or just check for BRANDING.md file existence
4. `soloAutoMerge: boolean` -- whether solo sets auto-transition to merged after execution (goal 4)

**Overlooked questions:**
- Should `targetSetCount` be stored in config or just passed directly to the roadmapper? If stored, it persists for add-set operations (the add-set flow could warn "adding this set exceeds your target of N sets").
- Is `reviewCycleConfirmation` needed at all? The milestone says "should now prompt" -- this implies it becomes the default behavior, not a toggle. If so, no config entry needed; just change the skill files.
- Config schema validation: `.planning/config.json` is currently read with bare `JSON.parse` and no schema validation. Adding new fields without a schema means old projects might break when reading config with new code.

**Recommendation:** For this milestone, avoid adding new config entries unless absolutely necessary. `targetSetCount` should be a runtime input to the roadmapper (passed in the agent prompt), not persisted config. `reviewCycleConfirmation` should be hardcoded as always-on (the milestone says "should now prompt," not "should optionally prompt"). Solo auto-merge should be the default behavior for solo sets (no config toggle needed).

### 11. Testing Strategy for Behavioral Skill Changes

**Why this matters:** Multiple goals in this milestone modify SKILL.md files (init, execute-set, merge, bug-hunt, unit-test, discuss-set). SKILL.md files are agent instructions, not executable code. They have no automated test coverage because they are natural language instructions interpreted by AI agents.

**Current testing landscape:**
- Library modules (`src/lib/*.cjs`) have co-located unit tests via `node:test`
- SKILL.md files have NO automated testing
- Agent role modules (`src/modules/roles/*.md`) have NO automated testing
- The only validation is `build-agents` which checks syntax, not behavior

**What could go wrong with untested SKILL.md changes:**
1. A skill says "transition to `discussed`" but the code expects `discussing` (this was the v3.2.0 bug)
2. A skill references a CLI subcommand that does not exist (the `wave-plan` incident)
3. A skill's flow has a dead branch that is never reached
4. New steps are added but error handling breadcrumbs are not updated
5. AskUserQuestion option text does not match the action taken

**Recommendation:** While full integration testing of skills is impractical, add a lightweight validation step:
- A test that scans all SKILL.md files for `node "${RAPID_TOOLS}"` patterns and verifies each command exists
- A test that scans for `state transition set` calls and verifies the target status is in the Zod enum
- A test that verifies all skills reference existing agent names (e.g., `rapid-verifier` exists in `agents/`)

This was also recommended in the v3.2.0 oversights research (concern #10). The fact that it was recommended but not implemented makes it even more important now, as this milestone adds more skill changes.

### 12. Cross-Skill State Consistency After Solo Auto-Merge

**Why this matters:** If solo sets are auto-marked as `merged` after execution (goal 4), this creates a state that downstream skills do not expect.

**Skills that check set status and their expectations:**
| Skill | Expected Status | Behavior if `merged` |
|-------|----------------|---------------------|
| review | `complete` or `executed` | "Cannot review -- wrong status" (STOP) |
| bug-hunt | Reads REVIEW-SCOPE.md | Would work if REVIEW-SCOPE.md exists |
| unit-test | Reads REVIEW-SCOPE.md | Would work if REVIEW-SCOPE.md exists |
| uat | Reads REVIEW-SCOPE.md | Would work if REVIEW-SCOPE.md exists |
| merge | `complete` | "Already merged" (SKIP) |
| cleanup | Any | Would report "no worktree" for solo sets |

**The problem:** The review skill (Step 0c) validates that status is `complete` or `executed`. If a solo set is auto-merged to `merged`, review CANNOT run on it at all (standard mode). The `--post-merge` flag exists but requires the user to know about it. This creates a jarring UX where: solo set executes -> auto-merged -> user runs /rapid:review 1 -> "Cannot review, status is merged" -> user is confused.

**Recommendation:** Either:
- (a) Add `merged` to the set of valid statuses for review (alongside `complete` and `executed`), OR
- (b) The review skill should detect solo+merged sets and automatically switch to `--post-merge` mode, OR
- (c) After solo auto-merge, display: "Solo set merged. To review: /rapid:review {setIndex} --post-merge"

Option (b) is the best UX. The user should not need to know about `--post-merge` for solo sets.

## Cross-Cutting Concern: Data Concerns

**DAG.json integrity:** DAG.json is a derived artifact (built from STATE.json + CONTRACT.json). If it becomes stale (e.g., a set is added but DAG.json is not rebuilt), merge order and wave-status will use outdated information. The `recalculateDAG()` function should be called whenever STATE.json sets change, not just during `add-set`. Currently there is no staleness detection.

**STATE.json backup on destructive transitions:** The `merged` status is terminal. Once a set is marked `merged`, it cannot be un-merged through the state machine. If solo auto-merge marks a set as `merged` incorrectly (e.g., execution actually failed but the skill proceeded), there is no undo path short of manually editing STATE.json. This is an existing concern, not new to this milestone, but auto-merge amplifies the risk.
