# Stack Research -- v3.6.0 Workflow & UX Polish

## Core Stack Assessment

### Node.js Runtime

| Property | Value |
|----------|-------|
| Detected version | 25.8.0 (Current) |
| Latest stable | 25.8.1 (Current, released 2026-03-10) |
| Active LTS | 24.x (Krypton, LTS until 2028-04-30) |
| Maintenance LTS | 22.x (Jod, maintenance until 2027-04-30) |
| Minimum required by RAPID | 18+ (specified in prereqs.cjs) |

**Key findings:**

- Node.js 18 reached end-of-life on 2025-04-30. RAPID's minimum version requirement of `18` is outdated -- it references an EOL runtime. The prereqs check in `prereqs.cjs` (line 118) sets `minVersion: '18'`. This should be bumped to at least 20, ideally 22.
- The development environment runs Node.js 25.8.0, which is a Current (non-LTS) release. Node.js 25.x EOLs on 2026-06-01. For production plugin users, Node.js 22 or 24 LTS would be the typical runtime.
- Starting with Node.js 27 (April 2027), every release becomes LTS -- the odd/even distinction is being removed.
- Confirmed via: `node --version` output and nodejs.org release schedule.

**Milestone impact:** The milestone work is purely plugin logic (state management, agent prompts, UX flows). No Node.js version-specific APIs are introduced. The existing `node:test` runner, `child_process`, and `fs` APIs used throughout are stable across all supported versions (18-25). No compatibility risk for this milestone.

### Module System (CommonJS)

- RAPID uses CommonJS exclusively (`.cjs` extension, `require()` calls throughout).
- All 21+ library modules, 50+ test files, and the CLI backbone use `require()`.
- No ESM migration is planned or needed for this milestone.
- The `node:test` built-in test runner works correctly with CommonJS in all supported Node.js versions (18+).

**Milestone impact:** None. All milestone features (init flow redesign, solo mode, branding command, review cycle confirmation) are pure CommonJS additions or modifications.

### Git

| Property | Value |
|----------|-------|
| Minimum required | 2.30 |
| Feature used | Worktrees, branches, rev-parse |

- Git 2.30+ requirement is appropriate. Worktree support has been stable since Git 2.5, and 2.30 is reasonable for the `--abbrev-ref` and branch management features used.

**Milestone impact:** Solo mode improvements (goal 4) specifically bypass worktree creation, operating directly on main. The existing `setInitSolo()` function in `worktree.cjs` already captures `startCommit` for diff-base tracking. No new Git features are required.

### Zod 3.25.76 -> 4.3.6

**Status:** Active development. Zod v4 was released mid-2025 with significant improvements (14x faster parsing, 57% smaller core).

**Breaking changes relevant to RAPID:**
1. `.passthrough()` is deprecated in favor of `z.looseObject()`. RAPID uses `.passthrough()` on `JobState`, `WaveState`, `SetState`, `MilestoneState`, and `ProjectState` in `state-schemas.cjs`.
2. `.deepPartial()` is removed entirely (not used by RAPID).
3. `z.enum()` API changes: `.Enum` and `.Values` removed, only `.enum` remains. RAPID uses `z.enum()` for `SetStatus`, `WaveStatus`, `JobStatus`, and review issue types.
4. Error customization API changes: `message` parameter renamed to `error`.
5. `z.record()` now requires two arguments (RAPID may use single-argument form in contract.cjs).
6. **CommonJS concerns:** There are reported issues with Zod v4 CommonJS bundles -- 4x bundle size increase and module resolution problems. Since RAPID is 100% CommonJS, this is a risk factor.

**Assessment:** Do NOT upgrade Zod for this milestone. Zod 3.25.76 is functional and the v4 migration requires touching every schema definition. This is a separate, dedicated effort.

### proper-lockfile 4.1.2

**Status:** Last published ~5 years ago. No active maintainer. 856 dependents on npm. No known CVEs.

**Alternatives evaluated:**
- `async-lock`: In-process only, not inter-process. Not a replacement.
- `lockfile` (npm): Also unmaintained (deprecated by npm).
- `async-mutex`: In-process only, not suitable.

**Assessment:** proper-lockfile remains the best option for file-based inter-process locking in Node.js. Its API surface is small and stable. The lack of updates reflects maturity rather than abandonment -- file locking semantics do not change. No action needed for this milestone.

### ajv 8.18.0 + ajv-formats 3.0.1

**Status:** Both up to date. ajv 8.18.0 is the latest release. ajv-formats 3.0.1 is compatible.

**Assessment:** No action needed.

## Stack Risks

1. **Zod v4 CommonJS regression** (Medium): If Zod v4 is inadvertently installed (e.g., `npm update` with `^3.25.76` pulling v4), the CommonJS bundle size increase and potential module resolution issues could break RAPID. **Mitigation:** The `^3.25.76` range in package.json will NOT pull v4 (semver major boundary). Safe.

2. **proper-lockfile abandonment** (Low): No maintainer activity for 5 years. If a Node.js core change breaks `graceful-fs` or `retry` (its dependencies), there is no upstream fix path. **Mitigation:** proper-lockfile's codebase is small (~200 lines of core logic). RAPID could vendor it or fork if needed. Not a concern for this milestone.

3. **Node.js 18 EOL documentation** (Low): RAPID's documented minimum of Node.js 18 references an EOL runtime. Users on Node 18 may encounter unpatched security vulnerabilities. **Mitigation:** Bump minimum to 20 or 22 in prereqs.cjs. Trivial change.

4. **No test coverage metrics** (Low): Without coverage reporting, it is difficult to verify that new milestone features (init flow, solo mode, branding, review cycle) are adequately tested. **Mitigation:** Consider adding `--experimental-test-coverage` flag to test script. Not blocking.

## Recommendations

1. **Bump Node.js minimum to 20 (or 22):** Node.js 18 is EOL. Update `prereqs.cjs` minVersion and documentation. -- **Priority: medium** (not blocking for this milestone, but reflects current reality)

2. **Pin Zod version range:** The current `^3.25.76` range is safe (won't cross the v4 major boundary), but consider documenting that Zod v4 migration is a known future task with specific breaking changes identified above. -- **Priority: low**

3. **Do NOT upgrade Zod for this milestone:** The CommonJS compatibility issues in Zod v4 and the 6+ API changes affecting core schemas make this a separate effort. -- **Priority: deferred**

4. **No dependency changes needed for milestone features:** All six milestone goals (bug fixes, DAG.json management, init flow redesign, solo mode improvements, branding system, review cycle confirmation) are achievable with the current stack. The work is in application logic, agent prompts, and state management -- not in infrastructure or dependencies.

5. **Consider test coverage flag:** Add `--experimental-test-coverage` to the npm test script for visibility into test gaps when implementing new features. -- **Priority: low**

6. **Monitor proper-lockfile:** No action needed now, but if Node.js 26+ introduces breaking changes to `graceful-fs` internals, be prepared to vendor the lockfile logic. -- **Priority: low**

## Milestone-Specific Stack Notes

### DAG.json Management (Goal 2)

The `dag.cjs` library provides `createDAG()`, `createDAGv2()`, `validateDAG()`, and `validateDAGv2()`. The `plan.cjs` module has a `writeDAG()` function. The DAG.json ENOENT issue (goal 1) likely stems from agents expecting `.planning/sets/DAG.json` to exist before it has been created. The fix is in application flow logic, not stack-level.

### Init Flow Redesign (Goal 3)

The `init.cjs` module handles scaffolding (`scaffoldProject()`). The `stdin.cjs` module provides `readStdinSync()` and `readAndValidateStdin()` for piped input. The "AskUserQuestion" pattern referenced in the milestone is a Claude Code plugin API feature (structured question prompts), not a library dependency. No stack changes needed -- this is agent prompt and skill redesign.

### Solo Mode (Goal 4)

`worktree.cjs` already has `isSoloMode()`, `setInitSolo()`, and `getSetDiffBase()` functions. The solo mode flag is stored in `REGISTRY.json` as `solo: true`. The state machine in `state-transitions.cjs` has no solo-specific transitions -- the same `SET_TRANSITIONS` map applies. The merge agent informing users about solo sets and auto-marking as merged will be state machine logic, not stack changes.

### Branding System (Goal 5)

The `display.cjs` module handles banner rendering with raw ANSI escape codes. The `ui-contract.cjs` module handles UI context for execution. A new `/branding` command would be a new skill definition (`.md` file in `skills/`) plus potentially a new library module. No external dependencies needed.

### Review Cycle Confirmation (Goal 6)

The `review.cjs` module defines `MAX_BUGFIX_CYCLES: 3`. The confirmation prompt before starting a new cycle is an agent prompt change (in `role-bug-hunter.md`, `role-unit-tester.md`), not a library change. No stack impact.
