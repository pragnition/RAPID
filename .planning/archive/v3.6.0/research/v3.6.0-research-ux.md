**Domain:** AI-powered developer tool (Claude Code plugin for parallel development)
**Milestone:** v3.6.0 -- Workflow & UX Polish
**Researched:** 2026-03-19
**Overall confidence:** HIGH (milestone goals target concrete, observed UX friction points with well-established patterns from the ecosystem)

## Domain Conventions

### 1. Init Flow: Structured Questions vs. Prose-Based Discovery

**Current state:** The init flow (SKILL.md Step 4B) conducts a deep adaptive discovery conversation using freeform AskUserQuestion prompts across 4 topic batches. Users respond in prose paragraphs, which the agent must then parse for requirements. This creates two problems: (a) users must compose lengthy written responses, and (b) the agent may miss nuances or require follow-ups.

**Milestone goal:** Replace prose-based Q&A with structured AskUserQuestion (4 sub-questions at a time) until the agent is satisfied, then summary + confirmation, plus granularity preference.

**Industry convention: wizard-style staged disclosure outperforms open-ended interviews for requirements gathering.**

| Tool | Init/Setup Pattern | Source |
|------|-------------------|--------|
| Yeoman | Structured prompts with defaults, one question at a time | [yeoman.io](https://yeoman.io) |
| Create React App | Zero-config with escape hatches | [create-react-app.dev](https://create-react-app.dev) |
| Nx | Interactive wizard with choices, then summary confirmation | [nx.dev](https://nx.dev) |
| Terraform | `terraform init` with structured provider/backend selection | [terraform.io](https://terraform.io) |
| Vite | Multiple-choice scaffold wizard, then writes config | [vitejs.dev](https://vitejs.dev) |
| Claude Code Plan Mode | AskUserQuestion with 1-4 structured sub-questions per call | [Claude Code docs](https://code.claude.com/docs) |

**Key UX principles from research:**

1. **Progressive disclosure via staged batches** -- Show one step of the process at a time, dividing information into equally important fragments with linear progression. This improves simplicity ([NN/g Progressive Disclosure](https://www.nngroup.com/articles/progressive-disclosure/)).

2. **Structured choices reduce cognitive load** -- Pre-filled options with safe defaults prevent analysis paralysis. Users select rather than compose. The AskUserQuestion tool natively supports 1-4 questions per call, each with 2-4 pre-filled options, headers, and descriptions.

3. **Summary + confirmation before committing** -- After gathering all inputs, present a compiled summary and ask the user to confirm or modify. This is the "dry run" pattern from clig.dev guidelines: "confirm for users that everything looks good before they do something scary" ([clig.dev](https://clig.dev/)).

4. **Granularity preference** -- Let users choose project decomposition granularity (number of sets). This maps to the "first-run wizard" pattern: "a few high-signal prompts with safe defaults and a clear escape hatch" ([Medium: Top 8 CLI UX Patterns](https://medium.com/@kaushalsinh73/top-8-cli-ux-patterns-users-will-brag-about-4427adb548b7)).

**Recommended init flow structure:**

- **Phase 1 (Logistics):** Keep existing structured questions for name, team size, model selection. Add a granularity preference question: "How many sets should the project have?" with options like "Few (3-5 sets) -- Coarser grained, faster to execute", "Medium (6-10 sets) -- Balanced granularity", "Many (11-15 sets) -- Fine grained, more parallelizable", "Claude decides -- Let the agent determine optimal set count".

- **Phase 2 (Discovery):** Replace freeform batches with structured AskUserQuestion calls. Each call presents 4 sub-questions with pre-filled options. For example, instead of asking "What are you building and why?" as freeform, present structured choices: project type (web app, CLI, API, library, mobile), primary problem domain (e-commerce, SaaS, dev tooling, data pipeline, etc.), differentiation (speed, UX, cost, integration, other), and scale target (prototype, small, medium, large). Continue rounds of 4 questions until the agent has sufficient context.

- **Phase 3 (Summary):** Present a compiled project brief and ask: "Does this look correct? Anything to add or change?" with options "Looks good -- proceed", "Add more details", "Start over".

**AskUserQuestion constraints to respect:**
- Maximum 4 questions per tool call
- Each question: header (max ~12 chars), question text, 2-4 options
- Each option: label (1-5 words), description
- Users can always type freeform via "Other" option
- 60-second timeout per call -- keep questions focused
- Cannot be used from sub-agents

**Source:** [clig.dev](https://clig.dev/), [NN/g Progressive Disclosure](https://www.nngroup.com/articles/progressive-disclosure/), [UX Database: Progressive vs Staged Disclosure](https://www.uxdatabase.io/newsletter-issue/04-staged-vs-progressive-disclosure), [ClaudeLog AskUserQuestion](https://claudelog.com/faqs/what-is-ask-user-question-tool-in-claude-code/), [SmartScope AskUserQuestion Guide](https://smartscope.blog/en/generative-ai/claude/claude-code-askuserquestion-tool-guide/)

### 2. Review Cycle Confirmation: Human-in-the-Loop Guardrails

**Current state:** The bug-hunt skill (Step 3) runs up to 3 cycles automatically (`MAX_BUGFIX_CYCLES = 3`). Each cycle spawns hunter, devil's advocate, judge, and bugfix agents. The unit-test skill similarly runs plan-then-execute without inter-cycle confirmation. Users have no control over whether to proceed to cycle 2 or 3 -- the pipeline auto-advances.

**Milestone goal:** Bug-hunt and unit-test should prompt the user for confirmation before starting a new cycle.

**Industry convention: human-in-the-loop confirmation for iterative automated processes.**

| Product | Cycle Control Pattern | Source |
|---------|----------------------|--------|
| GitHub Copilot Agent | Requires approval before applying changes | [GitHub Copilot](https://github.com/features/copilot) |
| Claude Code | Auto-run mode with allow/deny lists for command guardrails | [Claude Code docs](https://code.claude.com/docs) |
| Codacy Guardrails | Risk-tiered review: auto-approve low risk, human-gate high risk | [codacy.com/guardrails](https://www.codacy.com/guardrails) |
| OpenAI Agents SDK | Guardrails with `max_turns` and human approval gates | [openai.github.io/openai-agents-python](https://openai.github.io/openai-agents-python/guardrails/) |
| CI/CD Pipelines | Manual approval gates between stages (e.g., staging -> production) | Industry standard |

**Key UX principles:**

1. **Rate limiting prevents runaway costs and confusion** -- "Set a limit like 'Maximum 20 tool calls per conversation,' which prevents runaway costs and infinite loops" ([Towards AI: Guide to Guardrails](https://towardsai.net/p/machine-learning/the-complete-guide-to-guardrails-building-ai-agents-that-wont-go-rogue)). For RAPID, cycles are the natural unit of rate limiting.

2. **Confirm before escalating resource usage** -- clig.dev: "confirm for users that everything looks good before they do something scary." Each bug-hunt cycle spawns 3+ agents. Asking before cycle 2+ prevents unnecessary agent spawning when cycle 1 found nothing significant.

3. **Show results before asking to continue** -- Present cycle N results summary, then ask whether to proceed to cycle N+1. This follows the "present intermediate state" pattern from [clig.dev](https://clig.dev/).

**Recommended pattern:**

After each bug-hunt cycle completes (Step 3.9 in current SKILL.md), before incrementing `cycleNumber`, present results and ask:

```
"Bug hunt cycle {N} complete. {accepted} accepted, {dismissed} dismissed, {deferred} deferred. {fixedCount} bugs fixed, {modifiedFiles.length} files modified.

Start another cycle? (Cycle {N+1} will re-scan the {modifiedFiles.length} modified files for regressions.)"

Options:
- "Continue to cycle {N+1}" -- "Re-scan modified files for regressions introduced by fixes"
- "Stop here" -- "End bug hunt with current results"
```

For unit-test, the existing Step 4 already presents the test plan for approval. The confirmation should be added after Step 5 (execute) if there are failures: "Tests completed with {failed} failures. Would you like to fix and re-run, or accept current results?"

**Source:** [clig.dev](https://clig.dev/), [Propel: Agentic Engineering Code Review Guardrails](https://www.propelcode.ai/blog/agentic-engineering-code-review-guardrails), [Agno: Guardrails for AI Agents](https://www.agno.com/blog/guardrails-for-ai-agents)

### 3. Solo Mode: Main Branch Development Without Worktrees

**Current state:** Solo mode is partially implemented. The start-set skill (Step 3) creates a registry entry with `solo: true` and records a `startCommit`. The execute-set skill detects solo entries and uses the project root as the working directory. However, after execution, the set transitions to `complete` (Step 6 of execute-set) but NOT to `merged`. The merge skill (Step 3a-solo) fast-paths solo sets through `merge execute` which returns `{ merged: true, solo: true }`, but the user still has to manually invoke `/merge`.

**Milestone goal:** Solo sets should work on main branch (no worktree), merge agent should inform user that solo sets don't need merging, and state should be marked as merged after solo execution.

**Industry convention: solo developer workflows should be simple and direct.**

| Workflow | Solo Pattern | Source |
|----------|-------------|--------|
| Trunk-based development | Commit directly to main/trunk | [trunkbaseddevelopment.com](https://trunkbaseddevelopment.com/) |
| Git for solo developers | "Commit to master on solo projects" -- common practice | [DEV Community discussion](https://dev.to/daveskull81/git-workflow-do-you-commit-to-master-on-your-solo-projects-hi4) |
| Solo git workflow | "The best solo git workflow is the one simple enough that you actually follow it" | [caffeinatedprogrammer.com](https://caffeinatedprogrammer.com/blog/git-workflow-for-solo-developers/) |
| GitHub Desktop | No branch required for personal repos | [desktop.github.com](https://desktop.github.com/) |

**Key UX principles:**

1. **Eliminate unnecessary ceremony** -- Solo developers working on main branch should not be asked to run `/merge` when there is nothing to merge. The set already committed directly to main.

2. **Auto-transition when intent is clear** -- After solo execution completes, the set is on main and there is no branch to merge. Transitioning directly to `merged` is both correct and expected. The state machine should support `complete -> merged` transition for solo sets.

3. **Inform, don't block** -- When the user invokes `/merge` on a milestone with solo sets, the merge agent should display "Set '{name}' is a solo set (committed directly to main). No merge needed -- marking as merged." and auto-transition the state.

**Recommended implementation:**

- **execute-set change:** After Step 6 state transition to `complete`, check if the set is solo. If solo, immediately transition to `merged`:
  ```
  node "${RAPID_TOOLS}" state transition set "${MILESTONE}" "${SET_ID}" merged
  ```
  Display: "Solo set '{SET_ID}' executed on main branch. Automatically marked as merged (no worktree merge needed)."

- **merge skill change:** In Step 1d, when detecting solo sets, if a solo set is in `complete` state, auto-transition to `merged` and display the informational message. If it is already `merged`, skip with "already merged (solo)".

- **review scoping for solo sets:** The review scope command should use `startCommit` from the registry to diff against HEAD, since there is no branch divergence point. This already works per the current review SKILL.md (Step 1 "Solo set scoping").

**Source:** [DEV Community: Git Workflow Solo](https://dev.to/daveskull81/git-workflow-do-you-commit-to-master-on-your-solo-projects-hi4), [Caffeinated Programmer: Solo Git](https://caffeinatedprogrammer.com/blog/git-workflow-for-solo-developers/), [Atlassian: Comparing Workflows](https://www.atlassian.com/git/tutorials/comparing-workflows)

### 4. DAG.json Lifecycle: Creation and Maintenance by All Relevant Agents

**Current state:** DAG.json is created by the planner agent (`plan write-dag`) and read by multiple agents (execute, merge, worktree status). However, there is an ENOENT error when agents try to read DAG.json before it has been created (observed in the merge skill). The path is inconsistent: some code reads from `.planning/sets/DAG.json`, other code from `.planning/DAG.json`.

**Milestone goal:** DAG.json should be updated/created by all relevant agents.

**Industry convention: dependency graphs should be generated alongside their dependents and kept in sync.**

| Tool | Dependency Graph Lifecycle | Source |
|------|---------------------------|--------|
| npm/yarn | `package-lock.json` updated by install, update, remove | [docs.npmjs.com](https://docs.npmjs.com) |
| Terraform | State file updated by plan, apply, destroy, import | [terraform.io](https://terraform.io) |
| Make | DAG implicitly rebuilt from Makefile on every invocation | UNIX standard |
| Bazel | BUILD graph rebuilt on every build command | [bazel.build](https://bazel.build) |
| Nx | Project graph regenerated automatically | [nx.dev](https://nx.dev) |

**Key UX principles:**

1. **Single canonical path** -- The DAG.json path must be consistent. Currently it varies between `.planning/sets/DAG.json` and `.planning/DAG.json`. Pick one canonical location and update all references.

2. **Create on first need, update on every mutation** -- DAG.json should be created during init (roadmapper output), updated when sets are added (`/add-set`), and regenerated when set dependencies change. Every agent that mutates the set structure should also update DAG.json.

3. **Graceful degradation on missing DAG** -- Instead of ENOENT crash, agents should attempt to regenerate DAG.json from STATE.json set data when the file is missing. This is a recoverable error, not a fatal one.

4. **Path should be `.planning/sets/DAG.json`** -- This is where `plan.cjs` writes it (line 258: `path.join(cwd, '.planning', 'sets', 'DAG.json')`). The merge code that reads from `.planning/DAG.json` (line 1888) is the bug.

**Agents that should create/update DAG.json:**
- **Roadmapper** (init): Creates initial DAG from set dependencies
- **Plan-set**: Updates DAG when sets are decomposed into waves
- **Add-set**: Updates DAG when new sets are added
- **Merge**: Reads DAG for merge ordering (should not crash if missing)
- **Execute**: Reads DAG for execution ordering (should regenerate if missing)

**Source:** Observed error from v3.5-todo.md, code analysis of `src/commands/merge.cjs` line 266 vs `src/lib/plan.cjs` line 258

### 5. Execute Agent State Completion Bug

**Current state:** The execute-set SKILL.md Step 6 calls `state transition set "${MILESTONE}" "${SET_ID}" complete`, but per the v3.5-todo.md, the execute agent sometimes finishes without updating state to complete. This causes the review agent to refuse to review because the set is not in `complete` state.

**Milestone goal:** Fix the execute agent to reliably set state to complete.

**Industry convention: state transitions should be atomic and verified.**

| Pattern | Approach | Source |
|---------|----------|--------|
| Two-phase commit | Write intent, execute, confirm | Database standard |
| Post-condition verification | Check state after transition | [clig.dev](https://clig.dev/) |
| Idempotent transitions | Allow re-transition to same state | REST API conventions |

**Key UX principles:**

1. **Verify after transition** -- After calling `state transition set`, read STATE.json back and confirm the status is now `complete`. If not, retry once.

2. **Transition in the skill, not the agent** -- The SKILL.md orchestrator (not the spawned rapid-executor agent) should be responsible for the final state transition. This is already the design intent in the current execute-set SKILL.md, but the issue may be that the skill exits early on CHECKPOINT or BLOCKED without reaching Step 6.

3. **Handle all exit paths** -- Ensure that if the last wave executor returns COMPLETE but the verifier fails or times out, the skill still transitions to `complete` (verification is already non-blocking per Step 5: "continue to Step 6 -- verification is non-blocking").

**Source:** [Atlassian CLI Design Principles](https://www.atlassian.com/blog/it-teams/10-design-principles-for-delightful-clis), code analysis of execute-set SKILL.md

## Interaction Models

### Primary Interaction Paradigm

RAPID is a **CLI-orchestrated multi-agent system** embedded in Claude Code. The user interacts through slash commands (`/rapid:init`, `/rapid:execute-set 1`), and the system responds with a mix of:

1. **Structured prompts** (AskUserQuestion) for decisions
2. **Progress banners** (ANSI-colored stage indicators)
3. **Progress breadcrumbs** (workflow position indicators)
4. **Agent-generated output** (from spawned subagents)

This is not a traditional CLI where users type commands and get results. It is a **conversational CLI** where the user and system alternate between commands, structured questions, and output.

### Input/Output Patterns

**Input patterns users expect:**
- Numeric shorthand for set references (`/rapid:review 1` instead of `/rapid:review auth-system`)
- Flag-based modifiers (`--solo`, `--post-merge`, `--skip`, `--gaps`)
- Freeform text when "Other" is selected in AskUserQuestion

**Output patterns users expect:**
- Stage banners indicating current phase (ANSI-colored)
- Workflow breadcrumbs showing progress position
- Tables for structured data (set dashboards, merge summaries)
- Completion banners with "Next steps" suggestions
- JSON output for machine consumption (separated to stdout vs stderr for humans)

### Feedback and Progress Indication

The current system uses:
- **Banners:** Colored ANSI stage banners (`display.cjs renderBanner()`)
- **Breadcrumbs:** Text-based position indicators (`init [done] > start-set > ...`)
- **Per-wave progress:** "Wave N: complete" / "Wave N: pending"
- **Next-step hints:** Suggesting the next command to run

**Missing patterns that should be considered:**
- **Time estimates** -- No indication of how long operations will take. For multi-agent operations (research, bug-hunt), users have no expectation of duration.
- **Agent spawn indicators** -- When subagents are spawned, the user sees text output but no structured progress indicator showing "agent 3/6 complete."

### Error Communication

Error patterns are well-established:
- `[RAPID ERROR]` prefix for errors
- Breadcrumb with failure point: `init [done] > start-set [FAILED: reason]`
- Structured recovery via AskUserQuestion (Retry/Skip/Cancel)
- Guard checks with clear remediation ("Run `/rapid:review {id}` first")

## Information Architecture

### Command Hierarchy

RAPID uses a flat command namespace with semantic grouping:

```
/rapid:init          -- Project setup
/rapid:start-set N   -- Set lifecycle
/rapid:discuss-set N
/rapid:plan-set N
/rapid:execute-set N
/rapid:review N      -- Quality gate
/rapid:unit-test N
/rapid:bug-hunt N
/rapid:uat N
/rapid:merge         -- Integration
/rapid:status        -- Observation
/rapid:help          -- Documentation
```

This flat structure with numeric shorthand is intuitive for the target audience (developers using Claude Code). The proposed `/rapid:branding` command fits naturally as an optional post-init step.

### Navigation and Discovery

- **Status command** serves as the central dashboard with next-action routing
- **Breadcrumbs** show workflow position at every step
- **"Next step" suggestions** at completion of every skill
- **AskUserQuestion** for routing decisions

### Progressive Disclosure

The init flow currently reveals all questions upfront through batch prompting. The proposed structured AskUserQuestion pattern improves this by revealing questions 4 at a time, each with pre-filled options.

### Help and Documentation

- `docs/` directory with 9 guides
- `skills/help/SKILL.md` for command help
- In-context breadcrumbs and next-step suggestions

## Accessibility Considerations

### Terminal Accessibility

- **Color:** The display system uses 16-color ANSI palette for maximum terminal compatibility. However, there is no `NO_COLOR` environment variable support (noted in `display.cjs` line 7: "Always outputs colors (no NO_COLOR checking)"). The [NO_COLOR standard](https://no-color.org/) is widely adopted.
- **Screen readers:** Terminal screen readers work with plain text output. The ANSI color codes may cause issues. The structured JSON output (stdout) is screen-reader compatible.
- **Keyboard navigation:** Fully keyboard-driven as a CLI tool. No mouse interaction required.

### Cognitive Accessibility

- **Structured choices over freeform:** The milestone's init flow redesign directly improves cognitive accessibility by replacing open-ended prose with multiple-choice questions.
- **Consistent patterns:** Every skill follows the same structure: banner, resolve, validate, execute, complete with breadcrumb.
- **Error recovery:** Structured Retry/Skip/Cancel options prevent user from being stuck.

### Inclusive Design

- **AskUserQuestion "Other" option:** Always available for users who think outside the provided options.
- **`--skip` flag:** For power users who want to bypass interactive flows.
- **Solo mode:** Simpler workflow for single developers who do not need parallel isolation.

**Not applicable:** Visual accessibility (contrast, font size) -- RAPID is a terminal tool where the terminal emulator controls visual presentation. Internationalization -- RAPID targets English-speaking developer audience and uses technical domain terminology.

## User Expectations

### Default Behaviors Users Expect

1. **Solo sets should "just work" on main** -- Users expect that choosing "Solo (1 developer)" means no worktree ceremony. After execution, the set should be done -- no merge step needed.

2. **Review cycles should not run away** -- Users expect to be asked before expensive multi-agent cycles restart. Running 3 cycles of hunter + advocate + judge + bugfix without confirmation is surprising.

3. **Missing files should not crash agents** -- Users expect graceful handling when DAG.json does not exist yet. A missing dependency graph is a recoverable state, not a fatal error.

4. **Init should be interactive, not essay-writing** -- Users in Claude Code's conversational interface expect to click options, not write paragraphs. The structured AskUserQuestion pattern matches the mental model of a setup wizard.

5. **Post-execution state should reflect reality** -- If execution completed successfully, the state should show `complete` (and `merged` for solo). Users should not have to diagnose why the review agent refuses to work.

### Terminology Users Use

| User Term | RAPID Term | Notes |
|-----------|-----------|-------|
| "task" | "set" | Users may say "task" when they mean "set" |
| "branch" | "worktree" | Users think in git branches, RAPID thinks in worktrees |
| "run tests" | `/rapid:unit-test` or `/rapid:bug-hunt` | Users do not distinguish between test types initially |
| "merge" | `/rapid:merge` | Direct mapping |
| "start" | `/rapid:start-set` | Users may just say "start" |
| "what's next" | `/rapid:status` | Users want to know the current state and next action |

### Mental Models from Competing Products

- **Cursor/Copilot:** Users expect AI to do work autonomously but confirm before destructive or expensive operations.
- **GitHub Actions:** Users expect workflows with approval gates between stages.
- **CI/CD pipelines:** Users expect to see progress, results, and a "continue?" prompt before advancing to the next stage.

### Onboarding and Learning Curve

The init flow is the onboarding experience. Current pain points:
- Too many open-ended questions (addressed by structured AskUserQuestion redesign)
- No granularity control (addressed by set count preference)
- Branding discussion mixed into general init (addressed by separate `/branding` command)

## Branding System: /branding Command UX

### Domain Conventions for Branding/Design System Tools

**Industry convention: branding guidelines are gathered through structured discussion, not generated automatically.**

| Tool | Branding/Design System Approach | Source |
|------|-------------------------------|--------|
| Shopify Polaris | Comprehensive style guide: colors, typography, spacing, tone of voice, content guidelines | [polaris.shopify.com](https://polaris.shopify.com) |
| Material Design | Design tokens, component specs, accessibility guidelines, motion guidelines | [material.io](https://material.io) |
| Tailwind CSS | Utility-first config file (`tailwind.config.js`) with theme customization | [tailwindcss.com](https://tailwindcss.com) |
| Storybook | Living style guide generated from component code | [storybook.js.org](https://storybook.js.org) |
| Standards.site | Guided brand guideline creation with structured templates | [standards.site](https://standards.site/) |

**Recommended /branding command structure:**

1. **Optionality** -- The command runs AFTER init and is explicitly optional. Not every project needs branding guidelines (CLI tools, APIs, backend services may not have user-facing UI).

2. **Detection** -- Before starting, check if the project has user-facing components. If it is purely backend/CLI, suggest skipping or focusing only on output formatting conventions.

3. **Structured interview** -- Use AskUserQuestion to gather branding preferences across categories:
   - **Color palette:** Primary, secondary, accent, semantic (success/error/warning/info)
   - **Typography:** Font families, heading hierarchy, body text
   - **Spacing/Layout:** Grid system, spacing scale, breakpoints
   - **Tone of voice:** Formal/casual, technical/approachable, error message style
   - **Component patterns:** Button styles, form patterns, card layouts
   - **Accessibility:** WCAG level target, contrast requirements

4. **Output artifact** -- Write `.planning/BRANDING.md` with structured guidelines that downstream agents (executor, reviewer) can reference. The file should be readable by both humans and AI agents.

5. **Agent awareness** -- The executor and reviewer agents need to know about BRANDING.md. Add a reference in the worktree-scoped CLAUDE.md so agents can read it when implementing user-facing features.

**Source:** [Mockplus: UI Style Guide Examples](https://www.mockplus.com/blog/post/ui-style-guide), [Toptal: UI Style Guide](https://www.toptal.com/designers/ui/ui-styleguide-better-ux), [Creately: Design System Generator](https://creately.com/lp/design-system-generator/)

## Recommendations

Ordered by priority for the v3.6.0 milestone:

1. **Fix execute-set state completion bug:** Verify state transition to `complete` after execution, handle all exit paths (COMPLETE, CHECKPOINT, BLOCKED), add post-transition verification. -- Priority: critical

2. **Fix DAG.json ENOENT:** Standardize path to `.planning/sets/DAG.json`, fix the inconsistent `.planning/DAG.json` reference in merge.cjs line 1888, add graceful regeneration from STATE.json when DAG.json is missing. -- Priority: critical

3. **Solo mode auto-merge:** After solo set execution completes (state = `complete`), auto-transition to `merged` without requiring the user to run `/merge`. Display informational message. In merge skill, detect solo sets and auto-transition with explanation. -- Priority: high

4. **Review cycle confirmation:** Add AskUserQuestion gate between bug-hunt cycles (after Step 3.9, before incrementing cycleNumber). Present cycle results summary and "Continue/Stop" choice. Add similar gate to unit-test after execution with failures. -- Priority: high

5. **Init flow restructure:** Replace freeform prose discovery (Step 4B) with structured AskUserQuestion rounds of 4 sub-questions each. Add granularity preference to logistics (Step 4A). Add summary confirmation step before proceeding to scaffold. -- Priority: high

6. **DAG.json lifecycle management:** Ensure DAG.json is created by roadmapper during init, updated by plan-set, and updated by add-set. All consumers should handle missing DAG gracefully. -- Priority: high

7. **Branding command:** Implement `/rapid:branding` as optional post-init command. Use structured AskUserQuestion interview across branding categories. Output `.planning/BRANDING.md`. Update agent awareness so executors reference branding guidelines when implementing UI. -- Priority: medium

8. **Error log fixes:** Address the specific error logs identified in v3.5-todo.md (ENOENT for DAG.json, execute state not updating). These are the user-visible symptoms of recommendations 1 and 2 above. -- Priority: medium (symptoms of critical bugs above)
