# Domain & UX Research -- v4.3.0 Reliability & State

## Domain Conventions

RAPID operates in the **developer tooling / build orchestration** domain, specifically as a Claude Code plugin for parallel AI-assisted development. The closest comparable products are:

- **Nx** -- Monorepo task orchestrator with DAG-based execution, caching, and parallel task running
- **Turborepo** -- Task runner with dependency-aware parallel execution and error continuation modes
- **Lerna** -- Monorepo management with topological task ordering
- **GitHub Actions / CI orchestrators** -- Pipeline state machines with step-level failure handling

### Standard Terminology and Naming Conventions

RAPID's terminology aligns well with domain conventions:

| RAPID Term | Domain Standard | Assessment |
|-----------|-----------------|------------|
| Set | Nx "project", Turborepo "package" | Unique to RAPID -- acceptable since sets are a novel concept (parallel workstreams, not packages) |
| Wave | Nx "task group", CI "stage" | Good -- clearly implies sequential ordering |
| DAG | Universal term | Correct -- standard graph theory terminology |
| Merge | Git standard | Correct |
| Worktree | Git standard | Correct |
| State transitions | State machine standard | Correct -- `pending -> discussed -> planned -> executed -> complete -> merged` is clear |

### Common Workflows and User Journeys

The canonical lifecycle (`init -> start-set -> discuss-set -> plan-set -> execute-set -> review -> merge`) follows a linear pipeline model similar to CI/CD systems. This is well understood by developers.

**Industry expectation:** Tools in this space provide clear breadcrumb-style progress indicators showing where the user is in the pipeline. RAPID already does this with its progress breadcrumb pattern (e.g., `init [done] > start-set [done] > plan-set [FAILED] > ...`). This is a strong convention match.

### Competitive Landscape Patterns

1. **Nx** provides `nx graph` for visual DAG inspection and `nx affected` for targeted execution. RAPID's DAG is opaque to users -- it exists as a JSON file but has no visual inspection command.
2. **Turborepo** offers `--continue=dependencies-successful` for graceful partial failure, allowing independent tasks to proceed. RAPID's merge pipeline has this (sequential within waves, skip on failure), but state transition failures are all-or-nothing.
3. **GitHub Actions** uses step-level `if: always()` / `if: failure()` for conditional execution after failures. RAPID's SKILL.md files encode this logic but it is not user-visible.

## Interaction Models

### Primary Interaction Paradigm

RAPID uses a **slash-command CLI within Claude Code**, not a standalone terminal CLI. This creates a unique interaction model:

- **Input:** Natural language + slash commands (`/rapid:execute-set 1`)
- **Output:** Structured text blocks within Claude Code's chat interface
- **Feedback:** Progress banners, breadcrumbs, AskUserQuestion prompts
- **State:** Persistent on disk (STATE.json, MERGE-STATE.json, DAG.json)

### Input/Output Patterns Users Expect

1. **Commands should be idempotent or clearly state why they cannot retry.** When `merge order` fails with "DAG.json not found," the user has no way to diagnose whether the DAG was never created (init bug) or the path resolution is wrong (code bug). The error should distinguish between these cases.

2. **Partial failure should not block unrelated operations.** When STATE.json becomes temporarily invalid (e.g., during a concurrent write), subsequent operations fail with a generic "missing or invalid" error. Users expect either automatic retry or a diagnostic that explains the transient nature of the failure.

3. **Status checks should reflect ground truth.** When `/plan-set --gaps` sees a merged set and says "planning is complete," the user's mental model is violated -- they just saw gaps and are trying to fix them. The tool should recognize the gap-closure intent.

### Feedback and Progress Indication Conventions

RAPID's feedback conventions are well-designed:

- **`[RAPID]` prefix** for informational output (source: `core.cjs:output()`)
- **`[RAPID ERROR]` prefix** for errors (source: `core.cjs:error()`)
- **Progress breadcrumbs** showing pipeline position (e.g., `init [done] > start-set [done] > ...`)
- **Stage banners** via `display banner <stage>`

**Gap identified:** Error messages lack diagnostic context. Compare:

- **Current:** `"DAG.json not found at /path. Run /rapid:plan first to create sets and DAG."`
- **Better (Nx-style):** `"DAG.json not found at /path. This usually means /rapid:init did not complete successfully, or the DAG was not written during roadmap generation. Check that .planning/sets/DAG.json exists. If missing, re-run /rapid:init or /rapid:new-version."`

### Error Communication Patterns

The three bugs in this milestone reveal a pattern: **errors describe the symptom but not the cause or recovery path.**

| Bug | Current Error | What Users Need |
|-----|--------------|-----------------|
| DAG.json not found | "Run /rapid:plan first" | Diagnostic: Was DAG created? Is cwd correct? Is STATE.json present? |
| STATE.json missing/invalid | "Cannot mutate: STATE.json is missing or invalid" | Diagnostic: Does file exist? Is it valid JSON? Is it locked by another process? |
| --gaps on merged set | "Planning is complete" | Context-aware: "Set is merged but has unresolved gaps. Use /rapid:plan-set --gaps --force to re-plan gap closure." |

**Domain convention (Rust/Cargo-style):** The best CLI tools provide multi-line errors with:
1. **What happened** (the error)
2. **Why it happened** (the cause, if diagnosable)
3. **What to do** (actionable recovery command)

## Information Architecture

### How Information Should Be Organized and Presented

RAPID's information architecture follows a **layered disclosure** model:

1. **Surface layer:** Slash commands with brief output (`/rapid:status`)
2. **Detail layer:** Planning artifacts (STATE.json, PLAN.md, CONTEXT.md)
3. **Deep layer:** Merge state, DAG internals, lock files

This is appropriate for the domain. However, the bugs in this milestone expose a gap in the **diagnostic layer** -- when things go wrong, users must manually inspect `.planning/` files to understand the system state.

### Navigation and Discovery Patterns

- **`/rapid:status`** -- primary dashboard command (exists)
- **`/rapid:help`** -- command listing (exists)
- **Progress breadcrumbs** -- pipeline position (exists)

**Missing pattern:** A diagnostic/troubleshooting command (e.g., `rapid-tools doctor` or `rapid-tools diagnose`) that checks for common issues:
- STATE.json exists and is valid
- DAG.json exists and is valid
- All referenced sets exist in both STATE.json and DAG.json
- No stale locks
- Worktree registry is consistent

This is a convention in developer tools: `brew doctor`, `flutter doctor`, `npm doctor`, `rustup check`.

### Content Hierarchy and Progressive Disclosure

RAPID correctly uses progressive disclosure:
- Brief confirmation after plan-set (wave count, verification status)
- Full details available in PLAN.md files
- Error breadcrumbs show context without overwhelming

### Help and Documentation Conventions

RAPID has 9 documentation files in `docs/` and inline SKILL.md instructions. The `/rapid:help` skill provides command listing. This is adequate for the domain.

## Accessibility Considerations

### Key Accessibility Patterns for the Interaction Model

RAPID operates entirely within Claude Code's text-based chat interface. Traditional web accessibility concerns (screen readers, color contrast, keyboard navigation) are handled by the Claude Code host application.

**Relevant accessibility considerations for this domain:**

1. **Error messages should be parseable by both humans and machines.** RAPID already does this with dual stdout (JSON) and stderr (human-readable) output via `exitWithError()`. This is good practice.

2. **Progress indicators should work without visual formatting.** RAPID's breadcrumb pattern (`init [done] > start-set [done] > ...`) is text-based and works well in any terminal context.

3. **Output should not rely solely on color.** RAPID's `[RAPID]` and `[RAPID ERROR]` prefixes provide semantic meaning without requiring color support. The banner rendering may use ANSI colors but the prefix convention ensures readability without them.

### Inclusive Design Patterns

- **Abbreviation avoidance:** RAPID uses full words in user-facing output ("planning", "executing", not "plnng", "exec"). Good.
- **Consistent terminology:** Status names are consistent across all commands and artifacts. Good.
- **Error codes:** RAPID does not use numeric error codes. For a plugin of this complexity, named error categories (not numeric codes) are appropriate.

## User Expectations

### Default Behaviors Users Expect

Based on domain conventions from comparable tools:

1. **Idempotent retries.** Running the same command twice should either succeed (idempotent) or explain why it cannot proceed. Currently, `/plan-set --gaps` on a merged set does not handle the gap-closure use case.

2. **Transient failure recovery.** When STATE.json is temporarily locked or mid-write, users expect automatic retry with backoff, not immediate failure. The `proper-lockfile` library supports this, but the error path may not be using it consistently for reads.

3. **Spec-as-context in /new-version.** When users provide a spec file to `/new-version`, they expect the tool to pre-populate or skip categories already covered by the spec. This follows the principle of least surprise -- if I gave you a document that answers the questions, do not ask them again. This is analogous to `npm init --yes` skipping interactive prompts when defaults are available.

4. **DAG availability after init.** Users expect that after `/rapid:init` or `/rapid:new-version` completes, all downstream commands work. The DAG.json-not-found error violates this expectation because the user believes they completed initialization.

### Terminology Users Use vs Technical Terms

| User Says | Technical Term | RAPID's Term |
|-----------|---------------|--------------|
| "my project broke" | STATE.json validation failure | "STATE.json is missing or invalid" |
| "it's stuck" | Lock contention or stale lock | No user-facing term |
| "I already did that" | Idempotent re-entry | Status check in SKILL.md |
| "there are still gaps" | PASS_WITH_GAPS verification | "Planning is complete" (misleading for merged+gaps) |

### Mental Models from Competing Products

1. **CI/CD pipeline model:** Users think of sets as pipeline stages that can be re-run. A merged set with gaps should be re-runnable for gap closure, similar to how CI systems allow re-running a failed stage.

2. **Git model:** Users understand branches and merges. The "solo set auto-merge" pattern may confuse users who expect explicit merge steps.

3. **Package manager model:** Users expect `install` to work after `init`. The DAG.json-not-found error violates this expectation.

### Onboarding and Learning Curve Expectations

RAPID has a relatively steep learning curve (26 agents, 24 skills, 7-step lifecycle). The structured slash-command interface mitigates this by guiding users through the pipeline. The progress breadcrumb convention is particularly effective for orientation.

**Key user expectation:** When something goes wrong mid-pipeline, the recovery path should be obvious. Currently, error messages suggest re-running earlier commands but do not diagnose whether that will actually fix the problem.

## Recommendations

Ordered by priority for the v4.3.0 Reliability & State milestone:

1. **Enrich error messages with diagnostic context for STATE.json and DAG.json failures**: When `withStateTransaction` fails with "missing or invalid," check whether the file exists, whether it contains valid JSON, and whether a lock is held. Report the specific cause. -- Priority: critical (directly addresses Bug #1 and Bug #2)

2. **Add gap-aware status logic to /plan-set**: When `--gaps` flag is present and set is in `merged` status, check for GAPS.md existence. If gaps exist, allow re-planning instead of declaring "planning is complete." This requires either a state transition exception or a parallel "gap-closure" workflow that does not depend on set status. -- Priority: critical (directly addresses Bug #3)

3. **Add DAG.json existence validation to init/new-version completion gate**: Before declaring init or new-version complete, verify DAG.json was written. If missing, report the failure immediately rather than letting downstream commands discover it. -- Priority: critical (directly addresses Bug #1)

4. **Make /new-version spec-aware**: When a spec file argument is provided, parse it before the goal-gathering loop. Pre-populate category answers from spec content. For categories already covered by the spec, display "Pre-filled from spec: {summary}" and skip the AskUserQuestion prompt. Allow the user to override at the completeness confirmation step. -- Priority: high (directly addresses UX improvement)

5. **Add automatic retry with backoff for STATE.json read failures**: When `readState()` returns invalid (possibly due to concurrent write), retry 2-3 times with short delays before failing. This follows the convention set by `proper-lockfile`'s own retry mechanism. -- Priority: high (addresses intermittent Bug #2)

6. **Add deferred-item auto-discovery to /new-version research pipeline**: During Step 2C-v (Deferred Decisions), scan not only DEFERRED.md files but also GAPS.md files, VERIFICATION-REPORT.md "PASS_WITH_GAPS" entries, and todo comments in source. Present discovered items alongside explicit deferrals. -- Priority: medium (addresses Deferred Decision)

7. **Implement a `rapid-tools doctor` diagnostic command**: Check STATE.json validity, DAG.json existence, lock file staleness, worktree registry consistency, and set status coherence. Output a pass/fail checklist. This follows the `brew doctor` / `flutter doctor` convention adopted by most mature developer tools. -- Priority: medium (improves overall reliability UX)

8. **Improve error message format to include cause and recovery**: Adopt a three-part error format: (1) what happened, (2) likely cause, (3) recovery command. Example: `"[RAPID ERROR] DAG.json not found at .planning/sets/DAG.json\n  Cause: DAG was not generated during initialization.\n  Fix: Re-run /rapid:new-version or manually create the DAG with /rapid:init"`. -- Priority: medium (improves all error UX)
