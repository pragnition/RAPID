# Architecture Research: v4.3.0 Reliability & State

## Current Architecture

### Existing Strengths

The project uses a well-structured 5-layer architecture (Skill > Agent > CLI > Library > Storage) with clear separation of concerns. Key architectural patterns that work well:

- **Transaction pattern** in `state-machine.cjs`: lock -> read -> validate -> mutate -> validate -> write -> unlock via `withStateTransaction()`. This provides atomic state mutations with Zod validation.
- **Canonical path constants**: `DAG_CANONICAL_SUBPATH` in `dag.cjs` centralizes the DAG path definition.
- **File-based locking** via `proper-lockfile` with stale detection and retry configuration.
- **Atomic writes** via tmp-file + `fs.renameSync()` for crash safety.

### Existing Weaknesses Relevant to This Milestone

1. **DAG.json path inconsistency**: `src/commands/merge.cjs:274` uses the old path `.planning/DAG.json` instead of the canonical `.planning/sets/DAG.json`. This is a hardcoded path that bypasses `tryLoadDAG()` and `DAG_CANONICAL_SUBPATH`. The heuristic merge context silently fails to load DAG data because it looks at the wrong path, though it swallows the error.

2. **DAG.json not created by /new-version**: The `/init` skill explicitly calls `dag generate` and verifies the file exists (with retry logic). The `/new-version` skill does NOT call `dag generate` at all after writing STATE.json and CONTRACT.json files in Step 8. This means any project that enters via `/new-version` will have no DAG.json, causing `merge order` to fail.

3. **readState() lacks lock protection**: `readState()` reads STATE.json without acquiring a lock. While `writeState()` and `withStateTransaction()` both acquire the `state` lock, a concurrent read during the brief window between `writeFileSync(tmpFile)` and `renameSync(tmpFile, stateFile)` could theoretically read stale data. More critically, if `readState()` is called while another process holds the lock and is mid-mutation, it could read a partially-written or just-renamed file.

4. **withStateTransaction error message is opaque**: The error "Cannot mutate: STATE.json is missing or invalid" (line 162) conflates two distinct failure modes -- file missing vs. Zod validation failure. The `readResult` contains `errors` when validation fails, but these are discarded in the error message.

5. **plan-set --gaps does not exist**: `execute-set/SKILL.md:374` tells users to run `/rapid:plan-set --gaps` but `plan-set/SKILL.md` has zero handling for a `--gaps` flag. The plan-set skill's Step 2 validates preconditions by checking set status, and for a `merged` set it would correctly identify it as "already complete" -- but there is no gap-closure planning path.

6. **Set status blocks gap re-entry**: After `/execute-set` in solo mode, the set transitions to `complete` -> `merged`. The `plan-set` skill checks status and blocks any set in `complete` or `merged` state from re-planning. The state transition map (`SET_TRANSITIONS`) allows `merged -> []` (terminal), so there is no valid transition back to `planning` for gap closure.

## Recommended Architecture Pattern

- **Pattern:** Targeted fixes within existing modular monolith architecture
- **Rationale:** The existing architecture is sound. The bugs are localized to specific modules and skill definitions, not systemic architecture failures. The right approach is surgical fixes rather than structural refactoring.
- **Evidence:** All three bugs have clear, traceable root causes within 1-2 files each. The UX improvements are additive to existing skill definitions.

## Module Organization

- **Public interface:** `tryLoadDAG(cwd)`, `createDAG()`, `validateDAG()`, `getExecutionOrder()`, `DAG_CANONICAL_SUBPATH`
- **Dependencies:** `fs`, `path` (Node built-ins only)
- **Data ownership:** `.planning/sets/DAG.json`
- **Status for this milestone:** Well-designed. `tryLoadDAG()` is the correct entry point. The bug is in consumers not using it.

### Module: state-machine.cjs
- **Responsibility:** State CRUD, transactions, locking, corruption detection
- **Public interface:** `readState()`, `writeState()`, `withStateTransaction()`, `transitionSet()`, `findSet()`, `detectCorruption()`, `recoverFromGit()`
- **Dependencies:** `lock.cjs`, `state-schemas.cjs`, `state-transitions.cjs`, `dag.cjs`
- **Data ownership:** `.planning/STATE.json`
- **Status for this milestone:** Needs improved error diagnostics in `withStateTransaction()`.

### Module: state-transitions.cjs
- **Responsibility:** Defines valid state transitions for sets, waves, and jobs
- **Public interface:** `SET_TRANSITIONS`, `WAVE_TRANSITIONS`, `JOB_TRANSITIONS`, `validateTransition()`
- **Dependencies:** None
- **Data ownership:** Transition rules (in-memory constants)
- **Status for this milestone:** May need modification if gap-closure requires re-entering planning state from merged.

### Module: merge.cjs
- **Responsibility:** 5-level conflict detection, 4-tier resolution, merge ordering, feature regression checks
- **Public interface:** `getMergeOrder()`, `mergeSet()`, `detectConflicts()`, etc.

- **Status for this milestone:** `getMergeOrder()` correctly uses `tryLoadDAG()`. The hardcoded wrong path at line 274 is in the heuristic context for conflict resolution, not the merge ordering itself.

### Module: lock.cjs
- **Responsibility:** File-based distributed locking
- **Public interface:** `acquireLock()`, `isLocked()`, `cleanStaleLocks()`

- **Status for this milestone:** No changes needed.

## Data Flow

### Bug 1: DAG.json Not Found During Merge Order

```
User runs: /rapid:merge (or merge order)
  -> commands/merge.cjs dispatches to merge.getMergeOrder(cwd)
  -> lib/merge.cjs:getMergeOrder calls dag.tryLoadDAG(cwd)
  -> dag.cjs:tryLoadDAG reads .planning/sets/DAG.json
  -> File not found -> returns { dag: null }
  -> getMergeOrder throws "DAG.json not found"
```

Root cause flow:
```
/rapid:new-version
  -> Step 8: Writes STATE.json + CONTRACT.json files
  -> Step 9: Completion summary
  -> MISSING: No `dag generate` step
  -> DAG.json never created
```

### Bug 2: STATE.json Missing/Invalid During Transitions

```
Agent calls: state transition set <milestone> <set> <newStatus>
  -> commands/state.cjs dispatches to sm.transitionSet()
  -> state-machine.cjs:transitionSet calls withStateTransaction()
  -> withStateTransaction acquires lock, calls readState()
  -> readState returns null (missing) or { valid: false } (invalid)
  -> Throws "Cannot mutate: STATE.json is missing or invalid"
```

Possible causes:
1. **Race condition during concurrent agent writes**: Two agents call `withStateTransaction()` near-simultaneously. The lock should serialize them, but if the lock is stale (>5min), proper-lockfile may consider it stale and grant a second lock.
2. **Incomplete init**: STATE.json was never written or was corrupted during a prior failed write.
3. **Worktree path confusion**: Agent runs from a worktree but `cwd` resolves to the worktree root (which has no `.planning/` directory). The `.planning/` directory only exists at the project root, not in worktrees.

### Bug 3: plan-set --gaps on Merged Set

```
User runs: /rapid:plan-set 1 --gaps
  -> plan-set/SKILL.md Step 2: Resolve set
  -> Step 2: Load state, check set status
  -> Set status is "merged" (terminal state)
  -> Skill says "planning is complete" and STOPs
  -> User cannot re-plan for gap closure
```

### State Management

- **Where state lives:** `.planning/STATE.json` (persisted to disk)
- **How state is read:** `readState(cwd)` -- synchronous fs.readFileSync, Zod validation
- **How state is written:** `writeState(cwd, state)` or `withStateTransaction(cwd, fn)` -- lock-protected, atomic rename
- **Consistency guarantees:** File-level locking via `proper-lockfile` with 5-minute stale threshold. Atomic write via tmp+rename. Zod validation before and after mutation.

## Integration Points

| External System | Protocol | Purpose | Error Handling |

| Git (worktrees) | CLI (execFileSync) | Branch isolation per set | Try/catch with error propagation |

## Architecture Risks

1. **readState() is lock-free (Medium Impact)**: While atomic rename means reads won't see partial writes, `readState()` can return stale data if called outside a transaction while a write is in progress. In `withStateTransaction()`, the read IS lock-protected because the lock is acquired first. The risk is when `readState()` is called standalone (e.g., in `addMilestone()`, `validateDiskArtifacts()`). Mitigation: For the specific bug (#2), the real issue is likely path resolution, not lock contention. Still, `readState()` should include diagnostic information when it fails.

2. **Stale lock threshold of 5 minutes (Low Impact)**: If an agent operation takes longer than 5 minutes, `proper-lockfile` considers the lock stale and may grant a new lock. This is by design (prevents deadlocks from crashed processes) but could cause rare concurrent mutations. Mitigation: The `onCompromised` callback logs the event, and the 5-minute threshold is generous. No change needed.

3. **Terminal "merged" state blocks gap closure (High Impact)**: The state machine treats "merged" as terminal (`merged -> []`). There is no mechanism to re-enter the planning/execution cycle for a merged set. This is a design decision that needs revisiting for gap closure workflows. Mitigation: Either (a) allow `merged -> planning` transition for gap closure, or (b) create a new set for gap closure work rather than re-opening the merged set.

4. **No DAG.json generation in /new-version (High Impact)**: Projects initialized via `/new-version` will systematically fail on any command that reads DAG.json (merge order, execute prepare-context, etc.). Mitigation: Add `dag generate` step to `/new-version` Step 8, matching the pattern already in `/init`.

5. **Hardcoded wrong DAG path in merge.cjs (Low Impact)**: Line 274 uses `.planning/DAG.json` instead of `.planning/sets/DAG.json`. This is in a try/catch that silently swallows the error, so it does not crash -- but it means heuristic merge context never has DAG data. Mitigation: Use `tryLoadDAG()` or `DAG_CANONICAL_SUBPATH`.

## Recommendations

Ordered by priority and dependency:

1. **Add `dag generate` to /new-version Step 8**: After writing STATE.json and CONTRACT.json files, call `node "${RAPID_TOOLS}" dag generate` with verification logic identical to /init Step 9d. This directly fixes Bug #1 for projects entering via /new-version. -- Priority: **critical**

2. **Fix hardcoded DAG path in merge.cjs**: Replace the manual path construction at line 274 with `tryLoadDAG(cwd)` or use `DAG_CANONICAL_SUBPATH`. While this is a low-impact cosmetic fix (the code swallows the error), it ensures consistency and prevents future confusion. -- Priority: **medium**

3. **Enhance withStateTransaction error diagnostics**: When `readState()` returns null, report "STATE.json does not exist at {path}". When it returns `{ valid: false }`, report "STATE.json failed validation: {first error message}". Include the `cwd` in the error for worktree path debugging. -- Priority: **high**

4. **Implement gap-closure workflow for plan-set**: Two approaches:

   **Option A (Recommended): New waves on existing set.** Add `--gaps` flag support to `plan-set/SKILL.md`. When `--gaps` is specified: (a) Allow sets in `complete` or `merged` status to be re-planned. (b) Read GAPS.md to understand what needs closure. (c) Spawn the planner to create new gap-closure waves (e.g., `wave-gap-01-PLAN.md`). (d) Transition set back to `planned` status (requires adding `merged -> planned` and `complete -> planned` to SET_TRANSITIONS). (e) Execute only gap-closure waves via `execute-set --gaps`.

   **Option B: Create a new set for gaps.** Instead of re-opening a merged set, create a new "gap-closure" set in the same milestone. This avoids modifying the state transition map but requires more orchestration. Less desirable because it fragments related work across sets.

   -- Priority: **critical**

5. **Add `complete -> planned` and `merged -> planned` transitions for gap closure**: Modify `SET_TRANSITIONS` in `state-transitions.cjs` to allow: `complete: ['merged', 'planned']` and `merged: ['planned']`. Guard these transitions behind a `--gaps` flag or similar mechanism so they are not accidentally used in normal workflows. -- Priority: **critical** (prerequisite for recommendation 4)

6. **Make /new-version spec-aware in goal gathering**: Modify Step 2C of `new-version/SKILL.md` to accept an optional spec file argument. When provided, pre-populate the goals categories from the spec content and present them as defaults in the AskUserQuestion prompts, allowing the user to confirm or adjust rather than entering from scratch. -- Priority: **high**

7. **Auto-discover deferred items in researchers**: Extend the 6-researcher spawn prompts in `/new-version` Step 5 to include any DEFERRED.md content from the previous milestone as additional context. This is already partially done for the deferred-decisions category in Step 2C-v, but the research agents themselves do not receive this context. -- Priority: **medium**

8. **Add readState lock protection for standalone calls**: Consider making `readState()` acquire a shared/read lock to prevent reading during writes. However, this is low priority because (a) atomic rename makes partial reads impossible, and (b) the `withStateTransaction()` pattern already serializes read-modify-write cycles. -- Priority: **low**
