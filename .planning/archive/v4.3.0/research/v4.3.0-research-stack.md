## Upgrade Paths

### Zod 3.25.76 -> 4.x (NOT recommended for this milestone)

Zod 4 is a major rewrite with breaking changes relevant to this project:
- **Issue type renames:** `ZodInvalidTypeIssue` -> `$ZodIssueInvalidType`, etc. The project uses `safeParse()` and `.error.issues` in state-machine.cjs -- these would need updating.
- **`errorMap` replaced by `error` function:** Any custom error maps would break.
- **CJS support status:** Zod 4 ships as ESM-first. CJS compatibility via `require('zod')` needs verification before any upgrade. The project is 100% CJS, making this a significant migration risk.
- **Migration effort:** Moderate. The project uses `z.object()`, `z.string()`, `z.array()`, `z.enum()`, `z.union()`, `z.optional()`, `.parse()`, `.safeParse()` -- all of which have Zod 4 equivalents, but the error format changes would require test updates.
- **Recommendation:** Defer to a dedicated milestone. Not relevant to reliability/state bug fixes.

### proper-lockfile 4.1.2 (current is latest)

No upgrade available. The library is in maintenance mode but is stable and well-tested for its purpose (mkdir-based file locking). The `onCompromised` callback, stale detection, and retry configuration used by RAPID's `lock.cjs` are all working correctly.

**Risk note:** If proper-lockfile ever becomes abandoned, the only alternative with similar semantics is `lockfile` (npm), which uses a different mechanism. No action needed now.

## Stack-Specific Findings for Milestone Bugs

### Bug 1: DAG.json not found during merge order

**Stack finding (confirmed via code):** `src/commands/merge.cjs` line 274 uses the **wrong path** `.planning/DAG.json` instead of the canonical `.planning/sets/DAG.json`. This is a hardcoded path that bypasses the `tryLoadDAG()` helper and `DAG_CANONICAL_SUBPATH` constant defined in `dag.cjs`.

Meanwhile, `src/lib/merge.cjs` line 1580 correctly uses `dag.tryLoadDAG(cwd)` via the canonical path. The bug is isolated to the `merge resolve` command handler in `src/commands/merge.cjs`.

**Stack implication:** Pure code bug, no dependency or version issue. The `tryLoadDAG()` helper exists and works correctly -- the command handler just doesn't use it.

### Bug 2: STATE.json missing/invalid during state transitions

**Stack finding:** The state-machine uses proper-lockfile with:
- 5-minute stale threshold (300,000ms)
- Exponential backoff retry (10 retries, 100ms-2000ms)
- Atomic write via `writeFileSync` to `.tmp` then `renameSync`

**Potential race condition:** `readState()` (line 47-67) does NOT acquire a lock before reading. It reads the file directly. If another process is mid-write (between `writeFileSync` to `.tmp` and `renameSync`), the read should still see consistent data because `renameSync` is atomic on POSIX. However, if a process crashes between creating the `.tmp` file and renaming, the original `STATE.json` remains valid.

**More likely cause:** The `.planning/` directory or `STATE.json` may not exist in worktree contexts if the worktree was created before `STATE.json` was written, or if the `.planning/` symlink/copy is stale. This is a path resolution issue, not a locking issue. The `readState()` function returns `null` when the file doesn't exist, and the caller treats `null` as "missing or invalid."

**Stack implication:** proper-lockfile is working as designed. The issue is likely in how worktrees access `.planning/` state, or in timing between init and first state access.

### Bug 3: /plan-set --gaps treats merged sets as complete

**Stack finding:** This is a pure logic bug in the plan module's gap detection. No dependency or version implications. The gap detection likely checks set status but doesn't distinguish between "merged with remaining gaps" and "fully complete."

## Stack Risks

1. **Zod 3.x end-of-life timeline:** Zod 3 is in maintenance mode with Zod 4 released. No immediate risk -- v3 still gets patches -- but long-term the project should plan a migration. **Impact:** Low (deferred). **Mitigation:** Track Zod 3 security patches; plan migration in a future milestone.

2. **proper-lockfile maintenance status:** Last significant update was 2022. The library works well but is not actively developed. **Impact:** Low. **Mitigation:** The locking behavior is simple and correct; if the library becomes truly abandoned, the code is small enough to vendor or replace.

3. **No CI pipeline:** Test regressions can only be caught by manual `npm test` runs. **Impact:** Medium for reliability milestone. **Mitigation:** Run full test suite before and after each bug fix. Consider adding CI in a future milestone.

4. **Node.js version floor (18+) vs system version (25.x):** The project claims Node 18+ support but is only tested on Node 25. Some `node:test` behaviors differ between Node 18 and 25. **Impact:** Low for this milestone. **Mitigation:** No action needed unless distributing to users on Node 18.

## Recommendations

1. **No dependency changes for this milestone:** All bugs are code-level issues (wrong paths, logic errors), not dependency problems. Keep the stack stable. -- **Priority: critical** (do not introduce unnecessary risk)

2. **Use `tryLoadDAG()` consistently:** The canonical path helper exists in `dag.cjs` and should be used everywhere instead of hardcoded paths. Fix `src/commands/merge.cjs:274` to use `tryLoadDAG()`. -- **Priority: critical** (directly fixes Bug 1)

3. **Audit all `.planning/` path constructions:** Search for any other hardcoded `.planning/DAG.json` references that bypass the canonical subpath. The merge command has at least one; there may be others. -- **Priority: high**

4. **Add integration tests for worktree state access:** The STATE.json bug likely stems from path resolution in worktree contexts. Adding tests that verify `readState()` works correctly from within a worktree would prevent regressions. -- **Priority: high**

5. **Defer Zod 4 migration:** The breaking changes (ESM-first, renamed issue types, error API changes) make this a non-trivial migration. Schedule for a dedicated future milestone. -- **Priority: low**

6. **Consider adding code coverage:** Running `node --test --experimental-test-coverage` would help identify untested code paths in state-machine.cjs and merge.cjs where the bugs live. -- **Priority: medium**
