# Features Research -- v4.3.0 Reliability & State

## Feature Inventory

### Feature: DAG.json Not Found During Merge Order
- **Description:** `node "${RAPID_TOOLS}" merge order` fails with "DAG.json not found" when DAG.json exists at the canonical path `.planning/sets/DAG.json`. The issue may be that DAG.json was never created (init/plan failure), or the `cwd` passed to `getMergeOrder` does not resolve correctly (e.g., called from a worktree path where `tryLoadDAG` looks relative to `cwd` rather than the project root).
- **Complexity:** moderate
- **Dependencies:** independent

#### Root Cause Analysis

Three distinct failure modes have been identified:

1. **Path resolution in worktrees:** `getMergeOrder(cwd)` in `src/lib/merge.cjs:1579` calls `dag.tryLoadDAG(cwd)` which builds the path as `path.join(cwd, '.planning', 'sets', 'DAG.json')`. If `cwd` is a worktree path (`.rapid-worktrees/some-set/`), `.planning/sets/` does not exist there -- it only exists at the project root. Unlike `plan.cjs` which has `resolveProjectRoot(cwd)`, `merge.cjs` does not resolve back to the real project root before loading the DAG.

2. **DAG not created during /new-version:** The `/new-version` skill (SKILL.md) does NOT call `dag generate` after writing STATE.json and CONTRACT.json files. Only `/init` calls `dag generate` explicitly. When a new milestone is created via `/new-version`, DAG.json from the previous milestone may be archived or stale, and no new DAG.json is generated.

3. **Wrong path in merge resolve:** `src/commands/merge.cjs:274` uses the OLD path `.planning/DAG.json` instead of the canonical `.planning/sets/DAG.json` when loading the DAG for heuristic context in the resolve subcommand.

#### Decomposition
1. **Add `resolveProjectRoot` to merge order path** -- Use `resolveProjectRoot(cwd)` in `getMergeOrder()` before calling `tryLoadDAG`, consistent with how `plan.cjs` handles worktree paths.
2. **Add `dag generate` to /new-version** -- After the roadmapper writes STATE.json and CONTRACT.json in Step 8, call `dag generate` to ensure DAG.json exists for the new milestone.
3. **Fix wrong DAG path in merge resolve** -- Change `src/commands/merge.cjs:274` from `.planning/DAG.json` to use `DAG_CANONICAL_SUBPATH` or `tryLoadDAG`.
4. **Add unit tests for worktree DAG resolution** -- Test `getMergeOrder` when called with a worktree path as `cwd`.

#### Implementation Strategies

**Option 1: Centralize project root resolution in merge.cjs**
- Approach: Import `resolveProjectRoot` from `plan.cjs` and use it at the top of `getMergeOrder()` and `detectCascadeImpact()`. Fix the hardcoded path in merge resolve to use `tryLoadDAG`.
- Pros: Minimal change, consistent with existing pattern in plan.cjs
- Cons: Creates cross-module dependency (merge.cjs importing from plan.cjs)
- Libraries/services: none needed

**Option 2: Move `resolveProjectRoot` to core.cjs as shared utility**
- Approach: Extract `resolveProjectRoot` from `plan.cjs` to `core.cjs` (or a new `resolve-root.cjs`). Update both `plan.cjs` and `merge.cjs` to import from the shared location.
- Pros: Eliminates cross-module coupling, single source of truth, any future module needing root resolution gets it for free
- Cons: Slightly more files changed, need to update imports in plan.cjs

**Recommended:** Option 2 -- Moving `resolveProjectRoot` to a shared location is the cleaner long-term fix and prevents the same bug from recurring in other modules.

#### Data Model Impact
- New entities/tables needed: none
- Existing entities modified: none
- Relationships: none

#### API Surface
- New endpoints/commands needed: none
- Existing endpoints modified:
  - `merge order` -- internal path resolution fix
  - `merge resolve` -- fix DAG path from `.planning/DAG.json` to `.planning/sets/DAG.json`

### Feature: STATE.json Missing/Invalid During State Transitions
- **Description:** `state transition set` fails intermittently with "Cannot mutate: STATE.json is missing or invalid". The error comes from `withStateTransaction()` in `src/lib/state-machine.cjs:161` when `readState()` returns null or `{ valid: false }`.
- **Complexity:** complex
- **Dependencies:** independent

#### Root Cause Analysis

Several potential causes:

1. **Race condition with concurrent agents:** Multiple agents running in parallel may compete to read/write STATE.json. While `withStateTransaction` acquires a lock before reading, the lock acquisition itself could fail silently if the lock target file is corrupted or if the `.planning/.locks/` directory does not exist yet.

2. **Stale lock files:** If an agent process crashes or is killed, it may leave a stale lock. The current stale threshold is 5 minutes (300,000ms). During that window, another agent calling `acquireLock` will retry 10 times with exponential backoff but may still fail if the stale lock persists.

3. **Atomic write failure:** `writeState` uses a tmp+rename pattern (`STATE.json.tmp` -> `STATE.json`), which should be atomic on most filesystems. However, if a crash occurs between `writeFileSync(tmpFile)` and `renameSync(tmpFile, stateFile)`, the `.tmp` file exists but `STATE.json` may be missing or corrupted.

4. **Zod validation failure on read:** If STATE.json was manually edited or written by a non-RAPID tool, `ProjectState.safeParse()` may fail on valid-looking JSON that does not match the Zod schema exactly (e.g., missing `lastUpdatedAt`, wrong type for a field).

#### Decomposition
1. **Improve error diagnostics in `withStateTransaction`** -- When `readState` returns null or invalid, include the specific reason (file missing vs parse error vs schema validation error) in the thrown error.
2. **Add recovery path for corrupt STATE.json** -- Before throwing, check if STATE.json.tmp exists (partial write) and if it is valid; also check git for the last committed version.
3. **Ensure lock directory exists before acquiring** -- `ensureLocksDir` is already called in `acquireLock`, but verify it does not fail silently.
4. **Add startup stale lock cleanup** -- Call `cleanStaleLocks` at the start of state mutation operations, not just at startup.
5. **Add unit tests for concurrent access patterns** -- Test `withStateTransaction` under simulated concurrent access.

#### Implementation Strategies

**Option 1: Enhanced error reporting + auto-recovery**
- Approach: Wrap the error in `withStateTransaction` with diagnostic info. Add a recovery attempt: if STATE.json is missing, check for `.tmp` file; if invalid, attempt `recoverFromGit()` before throwing.
- Pros: Transparent to callers, self-healing for common failure modes
- Cons: Auto-recovery could mask deeper issues; `recoverFromGit` changes the working tree
- Libraries/services: none (uses existing `recoverFromGit` in state-machine.cjs)

**Option 2: Defensive read with retries**
- Approach: Add a retry loop around `readState()` inside `withStateTransaction` (2-3 attempts with 100ms delay). This handles transient filesystem issues (NFS, FUSE, etc.) and race conditions where the file is being written.
- Pros: Handles transient failures without user intervention
- Cons: Adds latency on failure, may mask bugs
- Libraries/services: none

**Recommended:** Option 1 with elements of Option 2 -- Enhanced error diagnostics are essential regardless. Add a single retry of `readState()` after a brief delay (100ms) before attempting recovery. If retry succeeds, proceed normally. If both fail, attempt `recoverFromGit()` as a last resort, logging the recovery action.

#### Data Model Impact
- New entities/tables needed: none
- Existing entities modified: none
- Relationships: none

#### API Surface
- New endpoints/commands needed:
  - `state recover` -- Manual recovery command that runs diagnostics and attempts repair
- Existing endpoints modified:
  - `state transition` -- Enhanced error messages with diagnostic info

### Feature: /plan-set --gaps Treats Merged Sets as Complete
- **Description:** After running `/execute-set` with gaps found, then completing the merge, attempting `/plan-set --gaps` on the merged set says "planning is complete" instead of allowing gap-closure planning. The issue is in the plan-set SKILL.md status validation logic.
- **Complexity:** moderate
- **Dependencies:** independent

#### Root Cause Analysis

The plan-set SKILL.md (line 75) has this status check:

```
- If `planning` or later: Display: "Set '{SET_ID}' is already in '{status}' state. Planning is complete."
  Suggest: /rapid:execute-set {SET_INDEX}
  STOP.
```

The phrase "or later" encompasses ALL statuses after `discussed`: `planned`, `executed`, `complete`, and `merged`. When `--gaps` is passed, the set has already been through the full lifecycle (discussed -> planned -> executed -> complete -> merged), so its status is `merged`. The SKILL.md does not have any special handling for `--gaps` mode -- it treats all post-discussed statuses identically and stops.

Additionally, the state transition map (`SET_TRANSITIONS` in state-transitions.cjs) does not allow transitioning from `merged` back to any earlier state. The valid transitions from `merged` are `[]` (terminal state). So even if the skill allowed `--gaps` on a merged set, the state transition at Step 8 (`discussed -> planned`) would fail because the set is in `merged` status.

The execute-set SKILL.md (line 63) defines `--gaps` as: "only execute waves without WAVE-COMPLETE.md markers (gap closure mode)". But there is no corresponding `--gaps` handling in plan-set at all -- the feature is only referenced in the execute-set skill output message (line 374): "To close gaps: /rapid:plan-set {SET_INDEX} --gaps then /rapid:execute-set {SET_INDEX} --gaps".

#### Decomposition
1. **Add `--gaps` flag support to plan-set SKILL.md** -- When `--gaps` is passed, bypass the normal status validation and allow re-planning regardless of current status.
2. **Define gap-closure planning behavior** -- In `--gaps` mode, read GAPS.md, plan only the gap-closure waves (new wave numbers beyond existing), do NOT transition set state (it stays at its current status).
3. **Handle state transition correctly for gaps** -- In `--gaps` mode, skip the state transition step entirely (Step 8) since the set is already past `planned`.
4. **Add --gaps to plan-set SKILL.md documentation** -- Update the command invocation examples and key principles.

#### Implementation Strategies

**Option 1: Skill-only fix (no state machine changes)**
- Approach: Modify plan-set SKILL.md to accept `--gaps` flag. When present: (a) allow any non-pending status, (b) read GAPS.md as additional planner context, (c) planner produces new wave PLAN.md files with higher wave numbers, (d) skip state transition in Step 8. The set status stays as-is.
- Pros: No code changes to state-machine.cjs or state-transitions.cjs; contained within the SKILL.md
- Cons: The skill becomes more complex with conditional logic; no CLI-level validation of the flag

**Option 2: Add a re-planning transition to the state machine**
- Approach: Add `merged -> planned` and `complete -> planned` transitions to `SET_TRANSITIONS`. Add `--gaps` to plan-set SKILL.md with the same behavior as Option 1, but also transition the set back to `planned` status.
- Pros: State machine accurately reflects that the set is being re-planned
- Cons: Complicates the linear lifecycle model; `merged` is supposed to be terminal; could confuse other tooling that expects merged to be final

**Recommended:** Option 1 -- The gap closure flow should be a lightweight overlay that does not disrupt the state machine. The set remains in its terminal state (`merged` or `complete`), and gap-closure waves are additional planning artifacts layered on top. This keeps the state machine simple and the lifecycle linear.

#### Data Model Impact
- New entities/tables needed: none
- Existing entities modified: none (GAPS.md already exists as an artifact)
- Relationships: gap-closure wave PLANs reference GAPS.md content

#### API Surface
- New endpoints/commands needed: none
- Existing endpoints modified:
  - `plan-set` skill -- add `--gaps` flag handling (SKILL.md only, no CLI changes)
  - `execute-set` skill -- `--gaps` already supported, no changes needed

### Feature: /new-version Should Be Spec-Aware
- **Description:** When a spec file is provided as an argument to `/new-version` (e.g., `/rapid:new-version spec.md`), the agent should use it as context for the goal-gathering questions rather than asking from scratch. Currently, `/new-version` always runs the full 5-category goal-gathering sequence regardless of whether a spec file was provided.
- **Complexity:** moderate
- **Dependencies:** independent

#### Root Cause Analysis

The `/new-version` SKILL.md has no argument parsing beyond what is needed for the milestone ID. It does not accept a file path argument. The goal-gathering in Step 2C is always the full interactive sequence.

#### Decomposition
1. **Add spec file argument parsing** -- Parse the first positional argument as a file path. If it exists, read its contents.
2. **Pre-populate goals from spec** -- Parse the spec file to extract goals by category (features, bugs, tech debt, UX, deferred). Present the extracted goals to the user for confirmation/modification instead of asking from scratch.
3. **Modify goal-gathering flow** -- When spec is provided: display extracted goals, ask user to confirm or modify each category (streamlined flow). When no spec: run the existing interactive flow unchanged.
4. **Pass spec content to researchers** -- Include the raw spec file content in the milestone brief sent to all 6 research agents for additional context.

#### Implementation Strategies

**Option 1: Simple spec injection**
- Approach: Read the spec file, display its contents to the user, then pre-fill the goals categories. Skip the initial "Enter features/Enter bug fixes/..." prompts and go straight to the completeness confirmation (Step 2C-vi) with pre-filled values. User can "Add more" to modify.
- Pros: Minimal SKILL.md changes, reuses existing completeness confirmation loop
- Cons: Requires the spec file to be structured in a parseable way for accurate category extraction

**Option 2: Spec as context with interactive refinement**
- Approach: Read the spec file and present it as context alongside each category question. Instead of "What new features should this milestone deliver?" ask "Based on the spec, these features were identified: {list}. Confirm, modify, or add more?" for each category.
- Pros: More guided, handles ambiguous specs better
- Cons: Still requires 5 prompts, just with pre-filled context; more SKILL.md complexity

**Recommended:** Option 1 -- Simple spec injection is the better UX. Read the spec file, use AI to extract goals by category, present the summary at the completeness confirmation step. The user gets to review everything at once and make adjustments. This reduces the 5-prompt sequence to a 1-prompt confirmation.

#### Data Model Impact
- New entities/tables needed: none
- Existing entities modified: none
- Relationships: none

#### API Surface
- New endpoints/commands needed: none
- Existing endpoints modified:
  - `/rapid:new-version` skill -- add optional spec file argument

### Feature: /new-version Researchers Should Auto-Discover Deferred Items
- **Description:** The research agents spawned by `/new-version` should automatically discover and incorporate deferred items from DEFERRED.md files without relying solely on the user selecting them during goal-gathering. Currently, deferred items are only surfaced in Step 2C-v (user selection), and the researchers do not independently scan for deferred items.
- **Complexity:** trivial
- **Dependencies:** Feature: /new-version Should Be Spec-Aware (shares SKILL.md modification surface area)

#### Root Cause Analysis

The deferred items flow is:
1. Step 2C-v: `/new-version` finds DEFERRED.md files, user selects which to carry forward
2. Selected items are included in `goals.deferredDecisions`
3. This text is passed to researchers as part of the category-tagged goals string

The gap is that researchers only see the user-selected deferred items, not all deferred items. If the user skips deferred items or misses some, the researchers have no independent way to discover them.

#### Decomposition
1. **Add deferred item discovery to researcher brief** -- Include all DEFERRED.md contents (not just user-selected items) in the milestone brief passed to each research agent.
2. **Tag deferred items as "auto-discovered"** -- Distinguish between user-selected deferred items (in goals) and auto-discovered ones (in researcher context) so the synthesizer can reconcile.

#### Implementation Strategies

**Option 1: Include raw DEFERRED.md in researcher brief**
- Approach: Before spawning researchers in Step 5, read all DEFERRED.md files and append their full contents as a "## Deferred Items from Previous Milestone" section in each researcher's brief, independent of which items the user selected.
- Pros: Simple, researchers get full context
- Cons: Researchers may recommend work the user explicitly chose to skip
- Libraries/services: none

**Recommended:** Option 1 -- Including the raw deferred content gives researchers full context. The synthesizer and roadmapper can filter based on user selections. The risk of recommending skipped items is low and can be handled at the roadmap approval step.

#### Data Model Impact
- New entities/tables needed: none
- Existing entities modified: none
- Relationships: none

#### API Surface
- New endpoints/commands needed: none
- Existing endpoints modified:
  - `/rapid:new-version` skill -- add DEFERRED.md reading before researcher spawning

## Implementation Order

1. **STATE.json Missing/Invalid During State Transitions:** This is the most impactful bug -- intermittent state mutation failures can corrupt project state and block all other operations. Fixing this first improves reliability for all downstream work.

2. **DAG.json Not Found During Merge Order:** Second priority because merge ordering is a critical workflow step. The fix is well-scoped (move `resolveProjectRoot` to shared module, fix wrong path in merge resolve).

3. **/plan-set --gaps Treats Merged Sets as Complete:** Third priority. This is a workflow gap that blocks the gap-closure loop. The fix is entirely within SKILL.md (no code changes).

4. **/new-version Spec-Aware:** UX improvement that streamlines milestone creation. Lower priority than bug fixes but high value for users with existing spec documents.

5. **/new-version Deferred Auto-Discovery:** Smallest change, least impact. Can be combined with the spec-aware feature since both modify the same SKILL.md file.
