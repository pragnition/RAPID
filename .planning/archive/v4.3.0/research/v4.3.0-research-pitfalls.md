# Pitfalls Research -- v4.3.0 Reliability & State

## Critical Pitfalls

### Pitfall: DAG.json Path Resolution Fails in Worktrees

- **Category:** correctness
- **Stack:** Node.js, git worktrees, dag.cjs / merge.cjs
- **Description:** `getMergeOrder()` calls `tryLoadDAG(cwd)` which constructs the path as `path.join(cwd, '.planning', 'sets', 'DAG.json')`. When `cwd` is the CLI-resolved project root via `findProjectRoot()`, this works. But when the caller is inside a git worktree (e.g., `.rapid-worktrees/set-name/`), `findProjectRoot()` walks up and finds the worktree's `.planning/` directory IF one exists there, or fails entirely if it does not. The `resolveProjectRoot()` function in plan.cjs uses `git rev-parse --git-common-dir` to resolve back to the true root, but this function is NOT used by the CLI backbone (`rapid-tools.cjs` uses `findProjectRoot()` from `core.cjs` instead).
- **Trigger:** Running `node "${RAPID_TOOLS}" merge order` from within a worktree directory, or when the shell's `cwd` is inside `.rapid-worktrees/`. Also triggered when `.planning/sets/` directory exists but DAG.json was never written (init completed but plan/roadmap was not run).
- **Impact:** Merge workflow completely blocked. The user sees "DAG.json not found" and cannot determine merge order. This is reported as a frequent failure.
- **Prevention:** Ensure `getMergeOrder()` (and all DAG consumers) use `resolveProjectRoot()` from plan.cjs rather than relying on the raw `cwd` passed from the CLI. Alternatively, make the CLI backbone's `findProjectRoot()` worktree-aware by incorporating the `git rev-parse --git-common-dir` logic. A single canonical root resolution function should be used everywhere.
- **Detection:** Run `merge order` from a worktree directory. If it fails with "DAG.json not found" but the file exists at the project root, this is the bug.
- **References:** Milestone bug report #1; `src/lib/dag.cjs:294-306` (`tryLoadDAG`); `src/lib/core.cjs:30-48` (`findProjectRoot`); `src/lib/plan.cjs:39-75` (`resolveProjectRoot`)

### Pitfall: STATE.json Race Between Lock Acquisition and File Read

- **Category:** data-loss / correctness
- **Stack:** Node.js, proper-lockfile 4.1.2, state-machine.cjs
- **Description:** `withStateTransaction()` acquires a lock, then calls `readState()`. But `readState()` is an async function that checks `fs.existsSync()` and then reads synchronously. Between the existence check and the read, another process could theoretically be writing a `.tmp` file and renaming it. More importantly, if STATE.json is deleted or corrupted between when the lock is acquired and when `readState()` runs (e.g., by a git operation in another terminal), the transaction throws "Cannot mutate: STATE.json is missing or invalid" with no recovery path. The error message is opaque -- it does not distinguish between "file missing" and "file exists but fails Zod validation."
- **Trigger:** Running state transitions while another terminal/agent is performing git operations (checkout, merge, reset) that modify `.planning/STATE.json`. Also triggered when STATE.json has been corrupted by a prior interrupted write (though atomic rename should prevent this).
- **Impact:** State transitions fail intermittently with an unhelpful error. The user must manually investigate whether the file is missing, has invalid JSON, or fails schema validation. Reported as milestone bug #2.
- **Prevention:** Split the error message in `withStateTransaction()` into three distinct cases: (1) file does not exist, (2) file exists but is not valid JSON, (3) file exists with valid JSON but fails schema validation. For case 1, check if a `.tmp` file exists (indicating interrupted write) and recover. For cases 2-3, use `detectCorruption()` which already exists but is not called during transaction failures.
- **Detection:** The error "Cannot mutate: STATE.json is missing or invalid" appears in agent output. Check if STATE.json exists on disk and whether it passes `JSON.parse()`.
- **References:** Milestone bug report #2; `src/lib/state-machine.cjs:157-181` (`withStateTransaction`); `src/lib/state-machine.cjs:375-397` (`detectCorruption`)

### Pitfall: plan-set --gaps Flag Is Referenced but Never Implemented

- **Category:** correctness
- **Stack:** SKILL.md (plan-set), SKILL.md (execute-set)
- **Description:** After execution verification finds gaps, execute-set SKILL.md (line 374) tells the user: "To close gaps: /rapid:plan-set {SET_INDEX} --gaps then /rapid:execute-set {SET_INDEX} --gaps". However, plan-set SKILL.md has NO handling for a `--gaps` flag. The status validation in plan-set Step 2 checks `if planning or later: "Planning is complete"` and STOPs. A merged set (status `merged`) will hit this guard. This means the entire gap-closure workflow for merged or completed sets is broken -- the user follows the suggested command and gets told planning is already done.
- **Trigger:** Execute a set. Verification finds gaps. Follow the suggested gap-closure command `/rapid:plan-set N --gaps` on a set whose status is `planned`, `executed`, `complete`, or `merged`.
- **Impact:** Gap closure workflow is completely non-functional for sets past the `discussed` state. This is milestone bug #3.
- **Prevention:** Add `--gaps` flag handling to plan-set SKILL.md: (1) parse for the flag, (2) when present, skip the status validation guard for sets in `planned`/`executed`/`complete`/`merged` states, (3) load GAPS.md as additional context for the planner, (4) instruct the planner to create only gap-closure wave plans (new waves appended after existing ones). Also need to ensure `state-transitions.cjs` allows re-entering `planned` from `complete` or `merged` if gap planning is initiated (currently `merged` is terminal with no valid transitions).
- **Detection:** Run `/rapid:plan-set N --gaps` on any set past `discussed` state. The skill will print "Planning is complete" and stop.
- **References:** Milestone bug report #3; `skills/plan-set/SKILL.md:75`; `skills/execute-set/SKILL.md:374`; `src/lib/state-transitions.cjs:3-10` (SET_TRANSITIONS)

## High-Risk Pitfalls

### Pitfall: Two Different Project Root Resolution Functions with Different Behavior

- **Category:** correctness
- **Stack:** Node.js, git, core.cjs, plan.cjs
- **Description:** The codebase has TWO project root resolution functions with incompatible behavior: (1) `findProjectRoot()` in `core.cjs` walks up directories looking for `.planning/`, and (2) `resolveProjectRoot()` in `plan.cjs` uses `git rev-parse --git-common-dir` to handle worktrees. The CLI backbone uses `findProjectRoot()`, but plan operations use `resolveProjectRoot()`. This means any CLI command run from within a worktree may resolve to the wrong root, while plan operations correctly resolve to the true root.
- **Trigger:** Any CLI command executed from within a worktree directory that needs to access shared state (STATE.json, DAG.json, etc.).
- **Impact:** Commands silently operate on the wrong directory, leading to "not found" errors or (worse) writing state to the wrong location. Different commands may resolve to different roots depending on which function they use.
- **Prevention:** Consolidate to a single `resolveProjectRoot()` function that handles both worktree and non-worktree cases. Replace all uses of `findProjectRoot()` in the CLI with the worktree-aware version. Add a unit test that verifies root resolution from inside a worktree.
- **Detection:** Run any rapid-tools command from inside `.rapid-worktrees/*/` and observe whether it finds the correct `.planning/` directory.
- **References:** `src/lib/core.cjs:30-48`; `src/lib/plan.cjs:39-75`; `src/bin/rapid-tools.cjs:200`

### Pitfall: proper-lockfile onCompromised Callback Silently Continues

- **Category:** data-loss
- **Stack:** proper-lockfile 4.1.2, lock.cjs
- **Description:** The `onCompromised` callback in `acquireLock()` writes to stderr but does NOT throw or abort the operation. If a lock becomes compromised (another process takes it due to stale detection), the current operation continues to read/mutate/write state without the lock's protection. This means two concurrent mutations can corrupt STATE.json.
- **Trigger:** A state transaction takes longer than 2.5 minutes (half the 5-minute stale threshold), causing proper-lockfile to consider the lock stale. Another process then acquires the lock, and the original process's `onCompromised` fires but continues operating.
- **Impact:** Silent data corruption of STATE.json. Two concurrent writes produce a last-write-wins race condition that can lose state transitions.
- **Prevention:** The `onCompromised` callback should set a flag that causes the transaction to abort (throw) before writing. Alternatively, throw directly in the callback. Also consider whether the 5-minute stale threshold is appropriate -- long-running agent spawns during set transitions could exceed this.
- **Detection:** Check stderr for `[RAPID] Lock "state" compromised` messages. If these appear alongside state corruption, this is the cause.
- **References:** `src/lib/lock.cjs:59-62`; proper-lockfile docs on `onCompromised`

### Pitfall: Merged State Is Terminal -- No Re-entry for Gap Closure

- **Category:** correctness
- **Stack:** state-transitions.cjs, state-machine.cjs
- **Description:** The `SET_TRANSITIONS` map defines `merged: []` (empty array = terminal state). Once a set is merged, no state transition is valid. The gap-closure workflow requires transitioning a merged set back to `planned` or `executing`, but this is impossible without modifying the transition map. Even `complete: ['merged']` only allows forward movement to merged.
- **Trigger:** User follows the gap-closure workflow on a set that has already been merged.
- **Impact:** The state machine will throw "merged is a terminal state with no valid transitions" when attempting to re-plan a merged set.
- **Prevention:** Add reverse transitions for gap closure: either `merged -> planned` and `complete -> planned`, or introduce a new `gap-planning` status. The cleaner approach is adding `planned` to the allowed transitions from both `complete` and `merged` states, gated by a `--gaps` flag at the CLI level.
- **Detection:** Attempt `state transition set <milestone> <set> planned` on a merged set.
- **References:** `src/lib/state-transitions.cjs:3-10`; `src/lib/state-transitions.cjs:32-51` (`validateTransition`)

### Pitfall: Atomic Rename May Fail on Network Filesystems

- **Category:** data-loss
- **Stack:** Node.js `fs.renameSync`, state-machine.cjs
- **Description:** `writeState()` and `withStateTransaction()` use `fs.writeFileSync(tmpFile)` followed by `fs.renameSync(tmpFile, stateFile)` for atomic writes. On local filesystems this is atomic (POSIX guarantee). On NFS, CIFS, or Docker-mounted volumes, `rename()` may NOT be atomic and can fail mid-operation, leaving STATE.json in an inconsistent state (zero-length file or partial write).
- **Trigger:** Running RAPID from a network-mounted directory or certain Docker volume configurations.
- **Impact:** STATE.json becomes zero-length or contains partial JSON, causing all subsequent state operations to fail. The `detectCorruption()` function would catch this, but `recoverFromGit()` requires the state to have been committed.
- **Prevention:** After rename, verify the file was written correctly by reading it back and checking size/validity. Keep a backup of the previous state (e.g., STATE.json.bak) before overwriting.
- **Detection:** STATE.json is 0 bytes or contains truncated JSON after a write operation.
- **References:** `src/lib/state-machine.cjs:84-91` (`writeState`); `src/lib/state-machine.cjs:172-176` (`withStateTransaction`)

## Medium-Risk Pitfalls

### Pitfall: readState Returns null vs Object -- Callers Must Handle Both

- **Category:** correctness
- **Stack:** state-machine.cjs
- **Description:** `readState()` returns `null` when the file is missing, `{ valid: false, errors }` when parse/validation fails, and `{ valid: true, state }` on success. This three-way return type is error-prone for callers. The `withStateTransaction()` checks `!readResult || !readResult.valid` but conflates missing file and invalid file into the same error message. Other callers (e.g., `addMilestone`) have slightly different error handling patterns.
- **Trigger:** Any code path that calls `readState()` and does not handle all three return shapes correctly.
- **Impact:** Confusing error messages ("STATE.json is missing or invalid") and potential null pointer exceptions if callers assume the return is always an object.
- **Prevention:** When fixing bug #2, replace the opaque error message with specific diagnostics. Consider adding a `readStateOrThrow()` helper that throws typed errors for missing vs invalid cases.
- **Detection:** Search for all `readState()` call sites and verify each handles null, invalid, and valid cases.
- **References:** `src/lib/state-machine.cjs:47-67`

### Pitfall: Lock Target File PID Written Once, Never Updated

- **Category:** correctness
- **Stack:** lock.cjs, proper-lockfile
- **Description:** When `acquireLock()` creates a lock target file, it writes the current `process.pid`. But if multiple different processes acquire/release the same lock over time, the target file retains the PID of whichever process FIRST created it (due to `if (!fs.existsSync(lockTarget))`). The `cleanStaleLocks()` function uses this PID to determine if the owning process is alive, but the PID in the file may belong to a long-dead process while the actual lock holder is a different process.
- **Trigger:** Process A creates the lock target (PID 1000), releases the lock. Process B acquires the lock. Process A dies. `cleanStaleLocks()` sees PID 1000 is dead and removes the lock artifacts, even though Process B holds the lock.
- **Impact:** `cleanStaleLocks()` can remove lock artifacts for actively held locks, causing concurrent access violations.
- **Prevention:** Update the PID in the target file each time the lock is acquired, not just on first creation. Change `if (!fs.existsSync(lockTarget))` to always write.
- **Detection:** Run two concurrent processes that both use the state lock. Kill the first, then run `cleanStaleLocks()`. The second process's lock may be incorrectly cleaned.
- **References:** `src/lib/lock.cjs:47-52`; `src/lib/lock.cjs:110-131`

### Pitfall: execSync for Git Commands in resolveProjectRoot Blocks Event Loop

- **Category:** performance
- **Stack:** Node.js, plan.cjs
- **Description:** `resolveProjectRoot()` uses `execSync('git rev-parse --git-common-dir')` which blocks the Node.js event loop. In normal CLI usage this is fine, but if this function is called from within an async transaction or during lock holding, it extends the lock hold time unnecessarily.
- **Trigger:** Any code path that calls `resolveProjectRoot()` while holding a lock.
- **Impact:** Lock hold times increase, raising the risk of stale detection by proper-lockfile.
- **Prevention:** Use `execFileSync` (already used elsewhere in the codebase) or cache the resolved root within a process. The root does not change during a single CLI invocation.
- **Detection:** Profile lock hold times during merge operations.
- **References:** `src/lib/plan.cjs:41-43`

### Pitfall: Zod .passthrough() Allows Schema Drift

- **Category:** correctness
- **Stack:** Zod 3.25.76, state-schemas.cjs
- **Description:** All state schemas use `.passthrough()`, which means extra properties in STATE.json are preserved through parse-and-rewrite cycles. This is intentional (forward compatibility), but it means that typos in property names (e.g., `stauts` instead of `status`) are silently preserved rather than rejected. A corrupted or hand-edited STATE.json with extra garbage keys will never be flagged.
- **Trigger:** Manual editing of STATE.json, migration bugs that add misspelled properties, or agent bugs that write extra fields.
- **Impact:** Silent accumulation of garbage data in STATE.json. Required fields still get validated, but spurious data persists indefinitely.
- **Prevention:** Add a periodic validation mode (e.g., `state validate --strict`) that uses `.strict()` parsing to flag unknown keys. Do not change the default `.passthrough()` behavior as it serves forward compatibility.
- **Detection:** Inspect STATE.json for unexpected keys beyond the schema-defined ones.
- **References:** `src/lib/state-schemas.cjs` (all schemas use `.passthrough()`)

## Anti-Patterns for This Stack

### Anti-Pattern: Two Root Resolution Functions

- **What people do:** Add a new root resolution function in a module that needs worktree awareness, while leaving the old one in place for the CLI.
- **Why it fails:** Different commands resolve to different roots, causing "file not found" errors that only appear in worktree contexts. Hard to reproduce because it depends on the shell's cwd.
- **Correct approach:** Single canonical `resolveProjectRoot()` that handles all cases, imported everywhere.
- **Example:** `core.cjs:findProjectRoot()` vs `plan.cjs:resolveProjectRoot()` -- these should be one function.

### Anti-Pattern: Opaque Error Messages for Multi-Cause Failures

- **What people do:** Check two conditions in one guard (`!readResult || !readResult.valid`) and throw a single error message.
- **Why it fails:** Users cannot distinguish between "file is missing" (run init) and "file is corrupted" (recover from git) without manual investigation. This wastes time and leads to wrong recovery actions.
- **Correct approach:** Separate error paths with specific messages and recovery suggestions for each failure mode.

### Anti-Pattern: Telling Users to Run Commands That Do Not Exist

- **What people do:** Reference `--gaps` flag in execute-set's gap report without implementing it in the target skill.
- **Why it fails:** Users follow the suggested workflow and hit a dead end. This is worse than no suggestion because it creates false confidence.
- **Correct approach:** Never reference a flag or command in user-facing output unless it is fully implemented. Use feature flags or conditional output.

### Anti-Pattern: Terminal States Without Escape Hatches

- **What people do:** Define `merged: []` as a terminal state, then add workflows that require re-entering earlier states from merged.
- **Why it fails:** The state machine correctly blocks the transition, but the higher-level workflow assumes it is possible. The bug is in the gap between the state machine's rules and the skill's assumptions.
- **Correct approach:** When adding a new workflow that requires re-entry, update the state transition map first. Design state machines with known escape hatches for maintenance operations.

## Version-Specific Gotchas

- **Node.js 18+**: `fs.rmSync` with `{ recursive: true, force: true }` is stable, but `fs.rm` (async) with the same options can throw on ENOENT in some 18.x point releases. The codebase uses `rmSync` so this is not currently an issue, but switching to async would require testing.
- **proper-lockfile 4.1.2**: The `realpath: false` option is required for worktree paths (symlinks). Without it, proper-lockfile resolves symlinks before creating lock directories, which can cause locks to be created in unexpected locations.
- **Zod 3.25.x**: The `.default()` method on enums works correctly for creating new objects but does NOT fill in defaults when parsing existing objects that are missing the field -- it only applies to `undefined` values, not missing keys in `.passthrough()` objects. This is relevant to state schema evolution.
- **Git 2.30+**: `git rev-parse --path-format=absolute` was added in Git 2.31. Systems with Git 2.30 exactly will fail with an unknown option error. The codebase declares Git 2.30+ as a requirement but uses a 2.31+ feature.

## Security Pitfalls

1. **Lock target file contains PID**: The PID in `.planning/.locks/state.target` could be used to target a process with `kill`. Mitigated by the `.gitignore` in the locks directory preventing these files from being committed, but a shared filesystem could expose them. Low risk for a development tool.
2. **execSync/execFileSync shell injection**: `resolveProjectRoot()` uses `execSync()` with a hardcoded command string, not `execFileSync()`. If `cwd` contained shell metacharacters (unlikely but possible), this could be exploited. The fix is straightforward: use `execFileSync('git', ['rev-parse', ...])` as done elsewhere in the codebase.
3. **STATE.json written with world-readable permissions**: `fs.writeFileSync()` uses the default umask. If the project directory has permissive permissions, STATE.json (containing project structure metadata) is readable by all users. Not a significant risk for a dev tool.

## Performance Traps

1. **Synchronous file I/O during lock hold**: `withStateTransaction()` holds the lock while performing synchronous reads and writes. This is correct for atomicity but means lock contention is bounded by disk I/O speed. On slow filesystems, this can cause lock timeouts for concurrent agents.
2. **readState reads entire file every transaction**: Each `withStateTransaction()` call reads and parses the full STATE.json. For projects with many milestones and sets, this file can grow large. Consider adding a size check or capping the number of historical milestones in the active state file.
3. **toposort uses O(V+E) but with array scans**: The cycle detection in `toposort()` uses `sorted.includes(id)` which is O(n) per check, making the overall complexity O(V*(V+E)) instead of O(V+E). For small DAGs (typical RAPID use: 5-20 sets) this is negligible, but it would become noticeable with hundreds of sets.

## Summary Priority List

1. **plan-set --gaps not implemented** (Critical) -- Gap closure workflow is completely broken. execute-set directs users to a nonexistent feature.
2. **Merged state is terminal with no re-entry** (Critical) -- Even if --gaps is implemented, the state machine blocks the transition back from merged/complete.
3. **DAG.json path resolution in worktrees** (Critical) -- Two competing root resolution functions cause merge order to fail when cwd is inside a worktree.
4. **Opaque "missing or invalid" STATE.json error** (High) -- Users cannot diagnose whether the file is missing, has bad JSON, or fails schema validation.
5. **onCompromised callback silently continues** (High) -- Lock compromise during long operations can cause silent state corruption.
6. **Lock target PID never updated** (Medium) -- cleanStaleLocks can remove actively held locks.
7. **execSync in resolveProjectRoot** (Medium) -- Blocks event loop and is vulnerable to shell injection; should use execFileSync.
8. **Zod .passthrough() allows schema drift** (Medium) -- Garbage keys in STATE.json accumulate silently.
9. **Git 2.30 vs 2.31 feature mismatch** (Low) -- `--path-format=absolute` requires Git 2.31+, not 2.30+ as documented.
