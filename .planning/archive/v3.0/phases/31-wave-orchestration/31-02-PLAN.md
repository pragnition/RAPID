  <action>
    Create `skills/plan-set/SKILL.md` following the established skill structure. The skill must implement the following steps:

    **Frontmatter:**
    ```yaml
    ---
    description: Plan all waves in a set with automatic sequencing -- spawns wave analyzer for dependency detection, then runs wave-plan pipeline per wave
    allowed-tools: Bash(rapid-tools:*), Agent, AskUserQuestion, Read, Write, Glob, Grep
    ---
    ```

    **Step 1: Environment Setup**
    Standard RAPID env preamble (RAPID_ROOT, .env loading, RAPID_TOOLS check).
    Display stage banner: `node "${RAPID_TOOLS}" display banner plan-set`

    **Step 2: Resolve Set and Validate Preconditions**
    - Accept set identifier: `/rapid:plan-set <set-id>` (e.g., `/rapid:plan-set 1` or `/rapid:plan-set auth-system`)
    - Resolve via: `node "${RAPID_TOOLS}" resolve set "<user-input>"`
    - Load full state: `node "${RAPID_TOOLS}" state get --all`
    - Find all waves in the resolved set
    - **Fail fast:** If ANY wave has status `pending`, abort with message listing undiscussed waves and suggesting `/rapid:discuss` for each
    - **Smart re-entry:** Classify waves:
      - `pending` -> ABORT (undiscussed, fail fast)
      - `discussing` -> plan this wave
      - `planning` or later -> skip (already planned)
    - If all waves are already planned (none in `discussing`), inform user "All waves already planned" and STOP
    - Display summary: N waves to plan, M already planned, wave IDs

    **Step 3: Wave Dependency Analysis (for 2+ waves to plan)**
    - If only 1 wave to plan: skip analyzer, treat as single sequential batch
    - If 2+ waves to plan: Spawn the **rapid-wave-analyzer** agent
      - Read ALL WAVE-CONTEXT.md files for waves being planned (from `.planning/waves/{setId}/{waveId}/WAVE-CONTEXT.md`)
      - Provide all WAVE-CONTEXT.md contents to the agent
      - Parse the RAPID:RETURN JSON from agent output
      - Convert dependency edges to planning batches:
        - Use the `assignWaves`-style BFS leveling algorithm (implemented inline in Bash/Node since dag.cjs is a module)
        - Nodes = waves to plan, Edges = dependency pairs from analyzer
        - Group by level: Level 1 waves plan first (parallel), Level 2 next, etc.
      - If analyzer fails or returns no RAPID:RETURN: fall back to sequential planning (each wave in its own batch, safest default)
    - **Re-entry note:** On re-entry (some waves already planned), skip the analyzer entirely. Derive batch structure from remaining discussing waves -- treat them all as a single sequential batch.

    **Step 4: Plan Waves in Batches**
    For each batch (in order), plan all waves in the batch. Within a batch, plan waves in parallel (multiple Agent tool calls in a single response if batch size > 1).

    **For each wave in a batch, run the full wave-plan pipeline inline:**
    This is the critical section. Plan-set CANNOT invoke the wave-plan skill. It must replicate the pipeline steps directly. For each wave:

    1. Display wave transition banner: `node "${RAPID_TOOLS}" display banner plan-set "Wave {waveIndex}/{totalWaves}: {waveId}"`
    2. Read context: WAVE-CONTEXT.md, CONTRACT.json, SET-OVERVIEW.md (if exists), targeted source files
    3. Spawn **rapid-wave-researcher** agent (same prompt as wave-plan Step 3)
    4. Read WAVE-RESEARCH.md output, read OWNERSHIP.json (if exists)
    5. Spawn **rapid-wave-planner** agent (same prompt as wave-plan Step 4)
    6. Read WAVE-PLAN.md output, extract job IDs
    7. Spawn **rapid-job-planner** agents -- parallel if 3+ jobs, sequential if 1-2 (same prompt as wave-plan Step 5)
    8. Spawn **rapid-plan-verifier** agent (same as wave-plan Step 5.5)
       - If FAIL: Use AskUserQuestion with "Re-plan failing jobs" / "Override" / "Cancel"
       - If Cancel: stop the entire chain (per locked decision)
       - If re-plan: retry once, if second FAIL offer only Override/Cancel
    9. Run contract validation: `node "${RAPID_TOOLS}" wave-plan validate-contracts "${SET_ID}" "${WAVE_ID}"`
       - Handle violations same as wave-plan Step 6
    10. Transition wave: `node "${RAPID_TOOLS}" state transition wave "${MILESTONE_ID}" "${SET_ID}" "${WAVE_ID}" planning`

    **For parallel batches (batch size > 1):**
    - Spawn one Agent per wave in the batch simultaneously. Each agent receives the full pipeline instructions (steps 1-10 above) for its specific wave.
    - Wait for all agents in the batch to complete before proceeding to the next batch.
    - IMPORTANT: These per-wave pipeline agents are orchestrator-like. They need to be able to spawn sub-agents. But Claude Code does NOT allow sub-sub-agents. Therefore, for parallel batches, the plan-set skill must run each wave's pipeline itself in sequence within the parallel batch (use multiple sequential Agent calls, NOT nested agents). Actually, the correct pattern is: for parallel batches, the plan-set skill itself dispatches agents for each wave-pipeline-step across all waves in the batch, interleaving them. For example: spawn wave-researcher for wave-A and wave-B in parallel, wait, then spawn wave-planner for wave-A and wave-B in parallel, wait, etc.
    - For sequential batches (batch size = 1), run the full pipeline for the single wave before moving to the next batch.

    **Step 5: Commit All Artifacts and Present Results**
    - Git add and commit all planning artifacts across all waves:
      ```bash
      git add ".planning/waves/${SET_ID}/"
      git commit -m "plan(${SET_ID}): plan-set complete -- ${WAVES_PLANNED} waves planned"
      ```
    - Display summary: waves planned (listing each), parallel batches used, any overrides
    - Print next step (no AskUserQuestion): `> **Next step:** /rapid:execute {setIndex}`

    **Error Handling (per locked decisions):**
    - If ANY wave's planning fails and user cancels re-plan: entire chain stops, no skip-and-continue
    - Parallel batches: if one wave fails, siblings continue. Only subsequent dependent batches are blocked.
    - Commit state for completed waves before stopping (partially planned is valid for re-entry)

    **Anti-patterns to avoid:**
    - Do NOT invoke `/rapid:wave-plan` skill (skills cannot call other skills)
    - Do NOT write persistent analyzer artifacts (RAPID:RETURN is ephemeral per user decision)
    - Do NOT attempt sub-sub-agent spawning (Claude Code hard constraint)
    - Do NOT skip undiscussed waves (fail fast per user decision)
    - Do NOT transition wave state before verification passes (deferred transition pattern per Phase 30)
