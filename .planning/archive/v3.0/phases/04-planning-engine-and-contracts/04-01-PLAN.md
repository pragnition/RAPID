  <files>rapid/src/lib/dag.cjs, rapid/src/lib/dag.test.cjs</files>
  <behavior>
    - toposort(nodes, edges) returns node IDs in dependency order for a simple linear chain (A -> B -> C returns [A, B, C])
    - toposort(nodes, edges) returns valid order where all dependencies appear before dependents for a diamond DAG
    - toposort(nodes, edges) throws Error with message containing involved node names when cycle detected (A -> B -> A)
    - toposort(nodes, edges) returns all nodes for a DAG with no edges (all independent)
    - toposort(nodes, edges) throws Error for edges referencing unknown node IDs
    - assignWaves(nodes, edges) returns wave 1 for all nodes when no edges exist
    - assignWaves(nodes, edges) assigns wave 1 to nodes with no incoming edges, wave 2 to nodes depending only on wave 1, etc.
    - assignWaves(nodes, edges) handles diamond patterns correctly (A,B -> C: A=1, B=1, C=2)
    - assignWaves(nodes, edges) handles deep chains (A -> B -> C -> D: waves 1, 2, 3, 4)
    - createDAG(nodes, edges) returns full DAG.json structure with nodes (including wave), edges, waves (with checkpoints), and metadata
    - createDAG(nodes, edges) rejects duplicate node IDs with descriptive error
    - createDAG(nodes, edges) rejects edges where from or to references an unknown node
    - createDAG(nodes, edges) rejects cycles with descriptive error
    - validateDAG(dag) returns { valid: true } for a well-formed DAG object
    - validateDAG(dag) returns { valid: false, errors: [...] } for missing required fields
    - getExecutionOrder(dag) returns array of arrays (waves), each inner array contains set IDs that can run in parallel
  </behavior>
  <action>
    Create `rapid/src/lib/dag.cjs` as a CommonJS module following the established library pattern (like state.cjs, verify.cjs).

    **toposort(nodes, edges):**
    - Implement Kahn's algorithm (BFS-based) as shown in 04-RESEARCH.md code examples
    - `nodes` is an array of `{ id: string }` objects (may have additional properties)
    - `edges` is an array of `{ from: string, to: string }` directed edges (from depends-on-nothing, to depends-on-from)
    - IMPORTANT: Edge direction convention -- `{ from: "auth", to: "api" }` means "auth" must complete before "api" can start. "from" is the dependency, "to" is the dependent
    - Before running sort, validate all edge endpoints exist in nodes. Throw Error listing unknown IDs if not
    - Returns sorted array of node IDs
    - If sorted length < nodes length, cycle detected -- throw Error listing remaining unsorted node IDs

    **assignWaves(nodes, edges):**
    - BFS-based level assignment as shown in 04-RESEARCH.md code examples
    - Returns object mapping nodeId -> wave number (1-indexed)
    - Nodes with no incoming edges get wave 1
    - Each subsequent level increments wave number

    **createDAG(nodes, edges):**
    - Validate inputs: check for duplicate node IDs, check edge endpoint validity
    - Run toposort to detect cycles (will throw on cycle)
    - Run assignWaves to compute wave assignments
    - Build the full DAG.json structure:
      - `nodes`: original nodes with `wave` and `status: "pending"` added
      - `edges`: as provided
      - `waves`: grouped by wave number, each with `sets` array and `checkpoint` object
      - `checkpoint.contracts`: array of `rapid://contracts/{setId}` URIs
      - `checkpoint.artifacts`: array of `.planning/sets/{setId}/CONTRACT.json` paths
      - `metadata`: `{ created, totalSets, totalWaves, maxParallelism }` where `maxParallelism` is the largest wave size
    - Returns the complete DAG object

    **validateDAG(dag):**
    - Check required top-level fields: nodes, edges, waves, metadata
    - Check each node has `id`, `wave`, `status`
    - Check each edge has `from`, `to`
    - Check metadata has `totalSets`, `totalWaves`
    - Returns `{ valid: true }` or `{ valid: false, errors: string[] }`

    **getExecutionOrder(dag):**
    - Returns array of arrays representing execution waves
    - `[[wave-1-sets], [wave-2-sets], ...]`
    - Each inner array contains set IDs from that wave (can run in parallel)
    - Ordered by wave number

    Export all five functions. Use only `require('fs')` and `require('path')` (zero new dependencies).
  </action>

    **1. Define CONTRACT_META_SCHEMA:**

    Create a JSON Schema that validates the structure of CONTRACT.json files. This schema enforces:
    - Top-level `type: "object"` with `properties` containing `exports`, `imports`, `behavioral`
    - `exports` is required, contains `functions` (array of `{ name, file, params: [{ name, type }], returns }`) and `types` (array of `{ name, file, shape }`)
    - `imports` is optional, contains `fromSets` array of `{ set, functions?, types? }`
    - `behavioral` is optional, contains `invariants` (string array) and `sideEffects` (string array)
    - `$schema`, `$id`, `title`, `description` are optional metadata fields

    **2. compileContract(contractJson):**
    - Validate `contractJson` against CONTRACT_META_SCHEMA using Ajv
    - If valid, return `{ valid: true, validate: compiledValidator }` where compiledValidator is the Ajv-compiled function
    - If invalid, return `{ valid: false, errors: ajv.errors }` with human-readable error messages

    **3. generateContractTest(setName, contractJson):**
    - Generate a complete `.test.cjs` file as a string
    - Include AUTO-GENERATED header with set name and regeneration instructions
    - Use `require('node:test')` and `require('node:assert/strict')` (matching project pattern)
    - For each function in `exports.functions`: generate `it()` that checks file exists and function is exported
    - For each type in `exports.types`: generate `it()` that checks file exists and Ajv can compile the shape schema
    - Use template literals for code generation. Properly escape any user-provided strings in the output
    - Include `require('ajv')` in the generated test for type shape validation

    **4. createManifest(sets):**
    - `sets` is array of `{ name, contractPath, contract, wave }`
    - Build MANIFEST.json object with `version: 1`, `generated` (ISO date string), and `contracts` array
    - Each contract entry: `{ set, path, schemaId, wave, exports, consumers }`
    - `schemaId` follows `rapid://contracts/{setName}` convention
    - `exports` extracted from `contract.exports.functions[].name`
    - `consumers` computed by scanning all other sets' `imports.fromSets` for references to this set

    **5. createOwnershipMap(sets):**
    - `sets` is array of `{ name, ownedFiles: string[] }` where ownedFiles are paths or glob-like patterns (e.g., `src/auth/**`)
    - Build OWNERSHIP.json with `version: 1`, `generated`, and `ownership` map
    - Detect conflicts: if two sets claim the same file, throw Error listing both sets and the conflicting path
    - Detect overlap: if set A claims `src/auth/**` and set B claims `src/auth/login.js`, throw Error (the specific file falls under the directory pattern)
    - For overlap detection: use simple `startsWith` check -- if path A ends with `/**` and path B starts with A's prefix, they overlap

    **6. checkOwnership(ownershipMap, filePath):**
    - `ownershipMap` is the `ownership` object from OWNERSHIP.json (path -> set mapping)
    - First try exact match: `ownershipMap[filePath]`
    - Then try directory pattern match: for each entry ending in `/**`, check if `filePath.startsWith(entry.replace('/**', '/'))`
    - Return owning set name or null if unowned

    **7. createContribution(setName, contributions):**
    - `contributions` is array of `{ file, owner, intent, section?, priority? }`
    - Returns CONTRIBUTIONS.json object: `{ set: setName, contributesTo: contributions }`
    - Validates each contribution has required fields (file, owner, intent)

    Export all functions plus CONTRACT_META_SCHEMA. Module depends on `ajv` (installed in step 0).
