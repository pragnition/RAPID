  <action>
    Create `skills/discuss/SKILL.md` following the GSD discuss-phase pattern adapted for RAPID waves. Use the set-init/SKILL.md as the structural template for environment setup and AskUserQuestion patterns.

    **Frontmatter:**
    ```
    ---
    name: discuss
    description: Capture developer implementation vision for a wave via structured discussion
    allowed-tools: Bash(rapid-tools:*), Read, AskUserQuestion, Glob, Grep
    ---
    ```

    **Steps:**

    **Step 1: Environment Setup** -- Standard RAPID_TOOLS env loading preamble (same as set-init).

    **Step 2: Resolve Wave** -- Accept wave ID argument (e.g., `/rapid:discuss wave-1` or `/rapid:discuss <set-id> <wave-id>`).
    - Run `node "${RAPID_TOOLS}" wave-plan resolve-wave <waveId>` to find the wave
    - If ambiguous (multiple matches), use AskUserQuestion to let user pick the set
    - If wave not found, display error with available waves and STOP
    - Verify wave status is 'pending' (ready for discussion). If already discussed/planned, use AskUserQuestion: "Wave already in {status}. Re-discuss? / View existing context / Cancel"

    **Step 3: Gather Context** -- Read the wave's jobs, set contract, and relevant source files:
    - Read STATE.json to get wave jobs list (IDs and titles from SET-OVERVIEW.md if available)
    - Read `.planning/sets/{setId}/CONTRACT.json` for interface contracts
    - Read `.planning/sets/{setId}/DEFINITION.md` for set scope
    - Read `.planning/sets/{setId}/SET-OVERVIEW.md` if it exists for preliminary wave breakdown
    - Use Grep/Glob to read target source files that the wave's jobs will modify (based on CONTRACT.json file references)
    - Display wave summary: "Wave {waveId} in set {setId}: {N} jobs"

    **Step 4: Identify Gray Areas** -- Analyze the wave context and identify 5-8 gray areas that need developer input. Gray areas are facets of implementation where:
    - Multiple valid approaches exist
    - Edge cases need clarification
    - Integration points are ambiguous
    - Performance/quality tradeoffs exist
    - User experience decisions are needed
    Present gray areas using AskUserQuestion with `multiSelect: true`:
    ```
    "I've identified these areas that would benefit from your input. Select which to discuss (or select none to let me decide all):"
    Options: [list of gray areas with brief descriptions]
    ```

    **Step 5: Deep-Dive Selected Areas** -- For each selected gray area, run a 4-question discussion loop:
    - Question 1: Open-ended "How do you want to handle {area}?"
    - Question 2: Follow-up probing uncovered facets or edge cases
    - Question 3: Clarification on specifics (file structure, naming, patterns)
    - Question 4: Confirmation or revision of the approach
    Every question includes a "Claude decides" option: "Let me make this call based on the codebase and contracts"
    If "Claude decides" is selected, document the rationale and move on.

    **Step 6: State Transitions** -- After discussion is complete:
    - Transition wave: `node "${RAPID_TOOLS}" state transition wave {milestoneId} {setId} {waveId} discussing`
    - Transition set to planning (first discuss triggers this): `node "${RAPID_TOOLS}" state transition set {milestoneId} {setId} planning 2>/dev/null || true` (catch error if set already in 'planning')

    **Step 7: Write WAVE-CONTEXT.md** -- Using the wave-plan CLI:
    - `node "${RAPID_TOOLS}" wave-plan create-wave-dir {setId} {waveId}`
    - Write WAVE-CONTEXT.md to `.planning/waves/{setId}/{waveId}/WAVE-CONTEXT.md` containing:
      - Wave boundary (scope)
      - Decisions (all locked decisions from the discussion)
      - Claude's Discretion items (areas where Claude decides was selected)
      - Deferred Ideas (anything out of wave scope)
      - Code context insights (relevant patterns discovered)

    **Step 8: Commit and Next Steps**
    - Commit WAVE-CONTEXT.md: `git add .planning/waves/{setId}/{waveId}/WAVE-CONTEXT.md && git commit -m "discuss({setId}): capture wave {waveId} implementation vision"`
    - Use AskUserQuestion for next steps:
      - "Run /rapid:wave-plan to start planning this wave" -- proceed to research + planning
      - "Discuss another wave" -- run /rapid:discuss for a different wave
      - "View /rapid:status" -- see current project state

    **Important patterns to follow:**
    - Use the tilde ~ for node paths in shell commands per CLAUDE.md
    - Every AskUserQuestion has a "Claude decides" / "Let Claude decide" option within deep-dive loops (Step 5 only -- NOT at the gray area selection level)
    - Read prior artifacts (CONTRACT.json, SET-OVERVIEW.md) to avoid re-asking settled questions
    - Discussion is wave-scoped: jobs are context but NOT individually discussed
    - The skill should be ~150-200 lines to be thorough but not bloated
