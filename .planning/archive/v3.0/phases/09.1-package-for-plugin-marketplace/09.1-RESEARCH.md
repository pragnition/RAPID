### Pattern 1: Plugin Manifest Schema

**What:** The `.claude-plugin/plugin.json` file defines plugin identity and component paths.
**When to use:** Every plugin must have this file for marketplace distribution.

Current RAPID plugin.json:
```json
{
  "name": "rapid",
  "version": "0.2.0",
  "description": "RAPID - Agentic Parallelizable and Isolatable Development for Claude Code",
  "author": { "name": "fishjojo1" },
  "homepage": "https://github.com/fishjojo1/RAPID",
  "repository": "https://github.com/fishjojo1/RAPID",
  "license": "MIT",
  "keywords": ["rapid", "parallel", "development", "worktree", "multi-agent", "orchestration"]
}
```

Source: https://code.claude.com/docs/en/plugins-reference#plugin-manifest-schema

**Required fields:** Only `name` is strictly required. But for marketplace quality, ALL of the above fields plus `description`, `version`, `author`, `homepage`, `repository`, `license`, and `keywords` should be present. RAPID already has all of these.

**Optional component path fields:** `commands`, `agents`, `skills`, `hooks`, `mcpServers`, `lspServers`, `outputStyles`. RAPID uses default directory locations, so these are auto-discovered and do NOT need to be declared in plugin.json.

### Pattern 2: DOCS.md / README.md for Plugin Documentation

**What:** Plugin documentation that users see when evaluating and using the plugin.
**When to use:** Every plugin needs comprehensive documentation.

The official Claude Code plugin structure uses `README.md` as the standard documentation file. RAPID uses `DOCS.md` instead, which is fine -- Claude Code does not enforce a specific documentation filename. However, the user specifically requested a DOCS.md file, so we should use that.

Source: https://code.claude.com/docs/en/plugins, https://github.com/anthropics/claude-code/blob/main/plugins/README.md

**DOCS.md must cover:**
1. Overview and value proposition
2. Installation instructions (marketplace install command + manual)
3. All available commands/skills with descriptions and usage examples
4. Agent descriptions and when they are used
5. Architecture overview (how the plugin works)
6. Prerequisites/requirements
7. Getting started workflow
8. Configuration options

### Pattern 4: Plugin Submission to Official Directory

**What:** Submitting plugin for inclusion in Anthropic's official plugin directory.
**When to use:** For maximum discovery. The official directory has 9000+ plugins and prominent placement.

**Submission process:**
1. Use the in-app submission form at `claude.ai/settings/plugins/submit` or `platform.claude.com/plugins/submit`
2. Anthropic performs basic automated review before adding to directory
3. "Anthropic Verified" badge requires additional quality/safety review

Source: https://code.claude.com/docs/en/plugins#submit-your-plugin-to-the-official-marketplace

### Anti-Patterns to Avoid

- **Putting components inside .claude-plugin/**: Only plugin.json goes in `.claude-plugin/`. Commands, agents, skills, hooks must be at the plugin root. RAPID already follows this correctly.
- **Using absolute paths in plugin**: All paths must be relative and start with `./`. Use `${CLAUDE_PLUGIN_ROOT}` for hook/MCP paths. RAPID's hook already uses relative paths.
- **Referencing files outside plugin directory**: Plugins are copied to cache on install. External paths break. RAPID uses `~/RAPID/rapid/...` paths in SKILL.md files -- this is a CRITICAL issue that needs attention (see Pitfalls section).
- **Version mismatch between plugin.json and package.json**: plugin.json says 0.2.0, package.json says 0.1.0. Must synchronize.

## Common Pitfalls

### Pitfall 1: Hardcoded Paths in SKILL.md Files
**What goes wrong:** RAPID's SKILL.md files use `~/RAPID/rapid/src/bin/rapid-tools.cjs` as the path to the CLI. When installed via marketplace, the plugin is copied to `~/.claude/plugins/cache/`. The `~/RAPID/rapid/` path will not exist on other users' machines.
**Why it happens:** During development, the plugin is loaded from a local path. Paths that work in development break when the plugin is cached elsewhere.
**How to avoid:** Use `${CLAUDE_PLUGIN_ROOT}/src/bin/rapid-tools.cjs` instead of hardcoded paths. The `${CLAUDE_PLUGIN_ROOT}` variable resolves to the actual plugin install location. However, this variable is documented only for hooks and MCP server configs -- SKILL.md files may need to use a different approach (relative paths from the skill's location, or detect the plugin root at runtime).
**Warning signs:** Any path starting with `~/RAPID/` or absolute paths in SKILL.md files.
**Confidence:** MEDIUM -- Need to verify whether `${CLAUDE_PLUGIN_ROOT}` works in SKILL.md bash blocks or if an alternative approach is needed. The official docs only show it used in hooks and MCP configs.

### Pitfall 2: "Coming Soon" Labels on Implemented Skills
**What goes wrong:** The `/rapid:help` command shows 7 commands as "Coming Soon" that are actually fully implemented (plan, assumptions, execute, status, pause, merge, cleanup, context). Users think the plugin is incomplete.
**Why it happens:** The help command was written in Phase 02 when only init and help existed. It was never updated as later phases added skills.
**How to avoid:** Update the help command to reflect actual available skills. Mark all 10 skills as "Available."
**Warning signs:** help.md contains static text that doesn't match the actual skills directory.

### Pitfall 3: Version Number Mismatch
**What goes wrong:** `plugin.json` declares version `0.2.0` but `package.json` declares `0.1.0`. Claude Code uses the plugin.json version for cache/update detection. If version isn't bumped between releases, users won't see updates.
**Why it happens:** Two files track version independently with no synchronization.
**How to avoid:** Synchronize both to the same version. For marketplace release, bump to `1.0.0` (all 34 v1 requirements are complete).
**Warning signs:** `grep -r '"version"' plugin.json package.json` shows different values.

### Pitfall 4: Missing LICENSE File
**What goes wrong:** plugin.json declares `"license": "MIT"` but no LICENSE file exists in the plugin directory. This is a packaging gap that marketplace reviewers may flag.
**Why it happens:** License metadata was added to plugin.json but the actual license text file was never created.
**How to avoid:** Add a standard MIT LICENSE file to the plugin root.
**Warning signs:** `ls rapid/LICENSE` returns "not found."

### Pitfall 5: Plugin Caching and node_modules
**What goes wrong:** When Claude Code copies the plugin to cache, it includes `node_modules/` (3.3MB). This is expected behavior but means the plugin carries its runtime dependencies. If a dependency is missing from package.json, it will fail in cache.
**Why it happens:** Claude Code copies the entire plugin directory tree.
**How to avoid:** Ensure all runtime dependencies are in package.json (currently correct: ajv, ajv-formats, proper-lockfile). The `.gitignore` already excludes `node_modules/` from git but the directory must exist for the plugin to work. Consider whether `npm install` needs to run post-install.
**Warning signs:** Plugin works in development but crashes after marketplace install.
**Confidence:** MEDIUM -- Need to verify whether Claude Code runs npm install for cached plugins or if node_modules must be bundled.

### Pitfall 6: Private Package.json
**What goes wrong:** `package.json` has `"private": true`. While this prevents accidental npm publish, it may interfere with npm-based marketplace sources.
**Why it happens:** Standard safety practice for non-npm packages.
**How to avoid:** Keep `"private": true` if distributing via git source (which RAPID does). Only remove if distributing as npm package.
**Warning signs:** npm publish fails. Not an issue for git-based distribution.

## Open Questions

1. **`${CLAUDE_PLUGIN_ROOT}` in SKILL.md bash blocks**
   - What we know: `${CLAUDE_PLUGIN_ROOT}` is documented for hooks and MCP server configs. SKILL.md files contain bash code blocks that Claude reads and executes.
   - What's unclear: Whether Claude expands `${CLAUDE_PLUGIN_ROOT}` in SKILL.md bash blocks, or whether an alternative approach is needed (e.g., detecting plugin root at runtime via `__dirname` or a wrapper script).
   - Recommendation: Test with `claude --plugin-dir` from a different location. If `${CLAUDE_PLUGIN_ROOT}` doesn't work in SKILL.md, implement a runtime detection approach where the CLI tool resolves its own path relative to the script location.

2. **npm install for cached plugins**
   - What we know: Plugins are copied to `~/.claude/plugins/cache/`. RAPID depends on ajv, ajv-formats, and proper-lockfile in node_modules.
   - What's unclear: Whether Claude Code runs `npm install` after copying to cache, or if node_modules must be included in the plugin directory.
   - Recommendation: Ensure node_modules is NOT gitignored for the actual plugin distribution (it's in .gitignore for the git repo, but it must be present at install time). Alternatively, investigate bundling dependencies into a single file.

3. **Official directory review timeline**
   - What we know: Submissions go through basic automated review. "Anthropic Verified" requires additional review.
   - What's unclear: How long the review process takes, what specific criteria are applied.
   - Recommendation: Submit early, self-host marketplace as immediate distribution channel.
