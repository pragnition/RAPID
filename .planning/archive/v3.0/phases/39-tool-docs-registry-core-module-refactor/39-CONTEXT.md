<decisions>
## Implementation Decisions

### YAML tool doc format
- Hybrid one-liner format with inline arg type hints: `state-get: state get <entity:milestone|set> <id:str> -- Read entity`
- Subcommand only -- no `node "${RAPID_TOOLS}"` prefix per entry (agents already know the invocation pattern from core-identity.md)
- `getToolDocsForRole()` returns raw YAML string, ready to embed -- no structured object serialization
- 1,000-token budget per agent enforced as build-time warning, not error (allows temporary overages during development)

### XML prompt schema
- 5 top-level tags allowed: `<identity>`, `<role>`, `<tools>`, `<returns>`, `<context>`
- All tags are top-level -- no nesting required
- Content inside tags is Markdown
- Schema document lives at `src/modules/PROMPT-SCHEMA.md` (co-located with core and role modules)
- Build-time validation: check that required tags (`identity`, `role`, `returns`) are present -- warn if missing
- Schema is a build-time reference only -- not embedded in agent prompts at runtime

### Core module consolidation
- Consolidate from 5 modules to 3:
  - `core-identity.md` -- absorbs context-loading guidance + RAPID_TOOLS setup prerequisite
  - `core-conventions.md` (NEW) -- git commit conventions only (from core-git.md). Code style is project-specific and belongs in CLAUDE.md
  - `core-returns.md` -- unchanged
- Retired modules deleted completely -- no stubs or redirect notices. Git history preserves old content
- `ROLE_CORE_MAP` updated in this phase to reflect new 3-module set

### Build pipeline integration
- `assembleAgentPrompt()` directly injects tool docs -- calls `getToolDocsForRole()`, wraps in `<tools>` tags, inserts between identity and role sections
- No template file (`core-tools.md`) -- direct injection is simpler
- Roles with no CLI commands (e.g., research-synthesizer) get no `<tools>` section -- omitted entirely
- `ROLE_TOOL_MAP` is a static explicit object in `tool-docs.cjs` -- no magic derivation from frontmatter
- One agent proof-of-concept: rebuild `rapid-executor` with new XML structure to validate the pipeline. Full rebuild of all 31 agents deferred to Phase 41

### Claude's Discretion
- Exact ordering of XML sections in assembled prompt
- Which specific CLI commands each role needs in ROLE_TOOL_MAP
- Content and structure of PROMPT-SCHEMA.md beyond the tag vocabulary
- How to restructure core-identity.md content when absorbing context-loading and state-access guidance

</decisions>

<specifics>
## Specific Ideas

- TOOL_REGISTRY stores each command as a string: `'state get <entity:milestone|set> <id:str> -- Read entity'`
- ROLE_TOOL_MAP is `{ executor: ['state-get', 'state-transition', ...], planner: [...] }` -- explicit, auditable
- assembleAgentPrompt() section order: identity, conventions, tools, role, returns (context injected at runtime by skills)
- Token budget warning fires during `handleBuildAgents()` for any role exceeding 1,000 tokens in tool docs
