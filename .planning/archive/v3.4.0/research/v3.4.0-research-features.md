# Features Research -- v3.4.0 Agent Intelligence

## Feature Inventory

### Feature: add-set Enhancement

- **Description:** Allow users to inject a new set at the end of a milestone for spontaneous work that does not fit the simple bugfix/feature pattern of `/quick`. This is for larger, multi-wave work items that the user discovers mid-milestone.
- **Complexity:** trivial
- **Dependencies:** independent

#### Current State

The `/rapid:add-set` skill already exists at `skills/add-set/SKILL.md`. It performs a 2-question discovery flow, creates DEFINITION.md and CONTRACT.json, updates STATE.json and ROADMAP.md, and suggests `/rapid:start-set`. The CLI already has `state add-milestone` support. The state-machine.cjs `withStateTransaction` supports atomic set insertion. The flow is functional end-to-end.

#### Decomposition

1. **Audit existing add-set** -- Verify the existing SKILL.md covers edge cases: duplicate detection, DAG re-computation after injection, dependency declaration on existing sets.
2. **DAG recalculation** -- After adding a set, re-run `dag.createDAG` and persist updated DAG.json so `/status` and `/execute-set` reflect the new topology.
3. **OWNERSHIP.json update** -- Regenerate OWNERSHIP.json when a new set is added to prevent ownership conflicts during execution.
4. **Position control** -- Allow the user to specify whether the new set should depend on specific existing sets (currently only captures freeform text about dependencies, does not wire up CONTRACT.json imports).

#### Implementation Strategies

**Option 1: Incremental Enhancement of Existing Skill**
- Approach: Add DAG recalculation and OWNERSHIP.json regeneration steps to the existing add-set SKILL.md. Add a Step 4.5 that wires CONTRACT.json imports based on user-stated dependencies.
- Pros: Minimal code change; leverages existing `dag.createDAG`, `contract.createOwnershipMap`.
- Cons: Skill file grows longer; logic is in SKILL.md (agent instructions) rather than CLI.
- Libraries/services: Existing `dag.cjs`, `contract.cjs`, `plan.cjs`

**Option 2: CLI-backed add-set command**
- Approach: Add a `node rapid-tools.cjs set add` CLI command that handles state mutation, DAG recalculation, and ownership update atomically. The SKILL.md becomes a thin orchestrator calling the CLI.
- Pros: Testable; atomic; consistent with other CLI-first patterns.
- Cons: More code to write; adds another CLI subcommand.

**Recommended:** Option 2 -- moving logic into the CLI is consistent with the project's pattern of keeping SKILL.md as orchestrator and CLI as business logic.

#### Data Model Impact

- New entities: None
- Existing entities modified: STATE.json (new set entry), DAG.json (recomputed), OWNERSHIP.json (recomputed)
- Relationships: New set may declare imports from existing sets

#### API Surface

- New CLI command: `node rapid-tools.cjs set add --id <set-id> --milestone <milestone-id> [--depends-on <set1,set2>]`
- Existing commands modified: None (SKILL.md updated to call new CLI command)

### Feature: Documentation Generation

- **Description:** Generate or update project documentation at the end of a version/milestone. Produces comprehensive docs from codebase state, planning artifacts, and context files.
- **Complexity:** complex
- **Dependencies:** independent

#### Decomposition

1. **Doc generation engine** -- New library module `src/lib/docs.cjs` that collects sources (ROADMAP.md, set DEFINITIONs, CONTRACTs, CODEBASE.md, ARCHITECTURE.md, CONVENTIONS.md, STYLE_GUIDE.md) and assembles structured documentation.
2. **Template system** -- Define documentation templates (API reference, architecture overview, getting started, changelog) that the engine populates.
3. **Diff-aware update** -- When docs already exist in `docs/`, compare generated content against existing and produce a merge rather than overwrite.
4. **Documentation agent** -- New agent role `role-documentation.md` that is spawned by the skill to perform the generation with codebase awareness.
5. **SKILL.md orchestrator** -- New `/rapid:docs` skill that orchestrates the documentation pipeline.
6. **Changelog extraction** -- Parse git log across the milestone's set branches to produce a structured changelog.

#### Implementation Strategies

**Option 1: Agent-Driven Generation**
- Approach: A documentation agent reads planning artifacts, source code, and existing docs, then generates/updates Markdown files in `docs/`. The agent uses Read/Write/Glob tools to access the codebase.
- Pros: Leverages LLM for natural language quality; can understand code context; handles ambiguity.
- Cons: Non-deterministic output; may produce inconsistent formatting across runs.
- Libraries/services: Existing agent infrastructure (`build-agents`, role modules)

**Option 2: Template + Agent Hybrid**
- Approach: A `docs.cjs` library module produces structured data (function signatures, set summaries, architecture diagrams), then an agent fills in narrative sections using templates as scaffolding.
- Pros: Deterministic structure; agent adds value where needed; repeatable.
- Cons: More implementation effort; template maintenance overhead.

**Recommended:** Option 2 -- the hybrid approach gives consistent structure while leveraging the agent for prose quality. The deterministic scaffolding ensures docs are always complete.

#### Data Model Impact

- New entities: `.planning/docs-config.json` (optional, controls which doc sections to generate)
- Existing entities modified: None (writes to `docs/` directory)
- Relationships: Reads from `.planning/context/`, `.planning/sets/`, `src/`

#### API Surface

- New skill: `/rapid:docs` (triggers documentation generation pipeline)
- New CLI command: `node rapid-tools.cjs docs generate [--sections <list>]`
- New CLI command: `node rapid-tools.cjs docs changelog --milestone <id>`

### Feature: UI Specs/Contracts

- **Description:** Define UI specifications and contracts across phases/sets so that the UI is created coherently across multiple agents working in parallel. Currently only API/function contracts exist; UI layout, component hierarchy, and interaction patterns are not captured.
- **Complexity:** complex
- **Dependencies:** independent

#### Decomposition

1. **UI contract schema** -- Define a JSON Schema for `UI-CONTRACT.json` that captures component hierarchy, layout constraints, styling conventions, interaction patterns, and shared state.
2. **UI contract validation** -- Extend `contract.cjs` (or new `ui-contract.cjs`) with validation against the UI contract meta-schema using Ajv.
3. **Cross-set UI consistency checks** -- During planning, compare UI-CONTRACT.json files across sets to detect conflicting layout assumptions, duplicate components, or incompatible interaction patterns.
4. **UI contract generation during plan-set** -- Extend the planning pipeline so sets that touch UI files also produce UI-CONTRACT.json.
5. **UI contract injection into executor context** -- When executing a set that produces UI, inject relevant UI-CONTRACT.json files from sibling and dependent sets into the executor prompt.
6. **Wireframe/mockup spec format** -- Define a text-based format (ASCII wireframe or structured Markdown) that agents can produce and consume for layout specifications.

#### Implementation Strategies

**Option 1: Structured JSON Contract Extension**
- Approach: Extend the existing CONTRACT.json schema to include a `ui` section with component declarations, layout constraints, and styling tokens. Reuse the existing Ajv validation pipeline.
- Pros: Single contract file per set; leverages existing validation; no new file format.
- Cons: CONTRACT.json becomes very large for UI-heavy sets; mixes API and UI concerns.
- Libraries/services: Ajv (existing), no new deps

**Option 2: Separate UI-CONTRACT.json**
- Approach: New `UI-CONTRACT.json` file per set, validated by a separate meta-schema. Includes: component tree, layout grid system, styling tokens (colors, spacing, typography), interaction states, shared state contracts. Cross-set validation aggregates all UI contracts into a MANIFEST.
- Pros: Clean separation; UI teams can evolve independently; does not bloat API contracts.
- Cons: Two contract files to maintain; more validation code; agents need to reference both.
- Libraries/services: Ajv (existing), no new deps

**Option 3: Design System Specification**
- Approach: A single project-level `.planning/UI-SPEC.md` Markdown document that defines the design system (components, tokens, layout rules). Individual sets reference this spec rather than duplicating UI contracts per set. Sets declare which components they implement.
- Pros: Single source of truth; no per-set UI contract overhead; human-readable.
- Cons: Less machine-validatable; harder to detect violations automatically.

**Recommended:** Option 2 -- separate UI-CONTRACT.json gives clean separation from API contracts while remaining machine-validatable. The per-set granularity aligns with RAPID's set isolation model.

#### Data Model Impact

- New entities: `.planning/sets/{set-id}/UI-CONTRACT.json` per UI-touching set, `.planning/UI-SPEC.md` (project-level design system reference)
- Existing entities modified: CONTRACT_META_SCHEMA in `contract.cjs` would NOT change (kept separate)
- Relationships: UI contracts reference shared component names; cross-set validation during plan-set

#### API Surface

- New CLI command: `node rapid-tools.cjs contract validate-ui <set-id>`
- New CLI command: `node rapid-tools.cjs contract ui-manifest` (aggregate all UI contracts)
- Modified: `plan-set` SKILL.md extended to produce UI-CONTRACT.json for UI-touching sets
- Modified: `execute-set` SKILL.md extended to inject UI contract context

### Feature: Hooks (State Update Verification)

- **Description:** Add hooks that run on completion of a RAPID agent to verify that project state has been updated if it needs to be. Agents often forget to update STATE.json after completing work; this hook serves as a guardrail.
- **Complexity:** moderate
- **Dependencies:** independent

#### Current State

There is an existing hook at `src/hooks/rapid-task-completed.sh` that writes completion records to `.planning/teams/` JSONL files. The hook infrastructure exists in the Claude Code plugin system. The plugin manifest at `.claude-plugin/plugin.json` can declare hooks.

#### Decomposition

1. **Hook expansion** -- Extend `rapid-task-completed.sh` or create a new hook `rapid-state-check.sh` that verifies STATE.json was updated when expected.
2. **Expected state inference** -- Logic to determine what state transitions SHOULD have happened based on the completed task's context (which set, which phase, what the return status was).
3. **RAPID:RETURN parsing in hook** -- Parse the agent's RAPID:RETURN marker to determine if the task completed successfully and what transitions were expected.
4. **Remediation prompt** -- If state was not updated, the hook should output a clear message instructing the agent (or user) to update state.
5. **Hook registration** -- Ensure the hook is registered in the plugin manifest and skill definitions.

#### Implementation Strategies

**Option 1: Bash Hook with jq Parsing**
- Approach: Extend the existing bash hook to also read STATE.json and compare against expected state based on the task context. Use `jq` for JSON parsing.
- Pros: Simple; consistent with existing hook pattern; no new dependencies.
- Cons: Complex state logic in bash is fragile; limited error handling.
- Libraries/services: jq (system dependency)

**Option 2: Node.js Hook Script**
- Approach: Create `src/hooks/rapid-state-check.cjs` that uses the existing `state-machine.cjs` readState and findSet functions to verify state consistency. Called from a thin bash wrapper.
- Pros: Reuses existing state management code; type-safe; testable with node:test.
- Cons: Adds Node.js execution overhead to hook invocation.
- Libraries/services: Existing `state-machine.cjs`, `state-transitions.cjs`

**Option 3: In-Skill Verification Step**
- Approach: Instead of a hook, add a mandatory final step to each SKILL.md (execute-set, plan-set, discuss-set) that verifies state was updated before displaying the completion breadcrumb.
- Pros: Runs in agent context (can fix issues); no hook system dependency.
- Cons: Must be added to every skill; agents can still skip it; duplicated logic.

**Recommended:** Option 2 -- Node.js hook reuses existing state management code and is testable. The bash wrapper keeps it compatible with the Claude Code hook system.

#### Data Model Impact

- New entities: `src/hooks/rapid-state-check.cjs`
- Existing entities modified: `src/hooks/rapid-task-completed.sh` (may be extended or replaced)
- Relationships: Reads STATE.json; compares against REGISTRY.json phase

#### API Surface

- New hook script: `src/hooks/rapid-state-check.cjs`
- Modified: Plugin manifest or skill definitions to register the hook
- New CLI command (optional): `node rapid-tools.cjs state verify <set-id>` (manual state consistency check)

### Feature: Memory System

- **Description:** Better "memory" of what the user requests during `/rapid:new-version` and when the user changes their mind about implementation halfway through execution. Currently, user decisions from discuss-set are captured in CONTEXT.md, but mid-execution changes are lost.
- **Complexity:** complex
- **Dependencies:** independent

#### Decomposition

1. **Decision log** -- New file `.planning/DECISIONS.md` (or `.planning/decisions.jsonl`) that records every user decision with timestamp, context, and the question that was asked.
2. **CONTEXT.md versioning** -- When the user changes their mind during execution, create a new version of CONTEXT.md (e.g., `CONTEXT-v2.md`) rather than overwriting, preserving the decision history.
3. **Decision capture middleware** -- Intercept all AskUserQuestion responses in skill orchestrators and log them to the decision log before continuing.
4. **Mid-execution change flow** -- New mechanism for the user to modify set scope/decisions during execution. Could be a `/rapid:amend-set` command that re-runs discuss-set targeting the active worktree.
5. **Memory injection into agent context** -- When spawning agents, inject relevant decisions from the log into the prompt so agents are aware of user preferences and changes.
6. **Cross-milestone memory** -- Persist user preferences (coding style, framework choices, naming conventions) across milestones in a `.planning/PREFERENCES.md` file.

#### Implementation Strategies

**Option 1: JSONL Decision Log**
- Approach: Append-only `.planning/decisions.jsonl` file. Each entry: `{ timestamp, skill, question, answer, setId?, milestoneId }`. Skills log decisions after each AskUserQuestion. Agent prompts include relevant decisions as context.
- Pros: Simple; append-only (no conflicts); easy to query; machine-readable.
- Cons: Skills must be modified to log; JSONL grows over time; needs filtering logic for relevance.
- Libraries/services: No new deps (fs.appendFileSync)

**Option 2: Structured DECISIONS.md**
- Approach: Human-readable Markdown file with sections per set/phase. Updated via Write tool after each decision. Injected into agent prompts as-is.
- Pros: Human-readable; easy to review; agents can parse naturally.
- Cons: Harder to query programmatically; concurrent writes may conflict; less structured.

**Option 3: Memory Module with Embeddings**
- Approach: A `memory.cjs` module that stores decisions and retrieves relevant ones using keyword matching. No vector DB needed -- simple text similarity via shared terms.
- Pros: Smart retrieval; only injects relevant context; scales well.
- Cons: More complex; keyword matching may miss relevant decisions.

**Recommended:** Option 1 (JSONL) for the decision log, combined with a human-readable summary in `.planning/DECISIONS.md` that is regenerated from the JSONL on demand. This gives both machine queryability and human readability.

#### Data Model Impact

- New entities: `.planning/decisions.jsonl` (append-only log), `.planning/PREFERENCES.md` (cross-milestone user preferences)
- Existing entities modified: CONTEXT.md (may gain versioning suffix)
- Relationships: Skills write to decisions.jsonl; agents read filtered decisions from prompt injection

#### API Surface

- New CLI command: `node rapid-tools.cjs memory log --skill <name> --question <q> --answer <a> [--set <set-id>]`
- New CLI command: `node rapid-tools.cjs memory query [--set <set-id>] [--milestone <id>] [--last <N>]`
- New CLI command: `node rapid-tools.cjs memory summarize` (regenerate DECISIONS.md from JSONL)
- New library module: `src/lib/memory.cjs`
- Modified: All interactive skills (discuss-set, new-version, add-set) to call memory log after AskUserQuestion

### Feature: Code Quality (Richer Planning/Context/State)

- **Description:** Improve code quality by providing agents with richer, more descriptive planning context, state information, and coding guidelines. Currently, agents get CONTRACT.json, DEFINITION.md, and scoped CLAUDE.md -- but lack detailed implementation guidelines, coding patterns, and quality expectations.
- **Complexity:** complex
- **Dependencies:** Memory (partially -- code quality benefits from decision history)

#### Decomposition

1. **Enhanced CONTEXT.md** -- Add sections for coding patterns, error handling expectations, testing requirements, and performance considerations specific to each set.
2. **Quality gates in PLAN.md** -- Extend wave PLAN.md templates to include explicit quality criteria per task: test coverage expectations, error handling requirements, documentation requirements.
3. **Pattern library injection** -- During execution, inject relevant code patterns from existing codebase into the executor prompt (e.g., "here is how error handling is done in this project").
4. **Enhanced scoped CLAUDE.md** -- Extend `generateScopedClaudeMd` to include style guide excerpts, testing conventions, and architectural constraints relevant to the set's files.
5. **Post-execution quality checks** -- Extend the verifier agent to check for code quality indicators: consistent error handling, test presence, documentation comments.
6. **State enrichment** -- Add wave-level and job-level metadata to STATE.json: expected files, quality criteria, test expectations.

#### Implementation Strategies

**Option 1: Enhanced Context Injection**
- Approach: Enrich the existing `assembleExecutorPrompt` in `execute.cjs` and `generateScopedClaudeMd` in `worktree.cjs` with additional context sections. Pull from STYLE_GUIDE.md, CONVENTIONS.md, and codebase patterns automatically.
- Pros: No new files; enhances existing flow; agents get better context transparently.
- Cons: Increases prompt size (token cost); must be careful about budget (compaction.cjs DEFAULT_BUDGET_TOKENS = 120000).
- Libraries/services: Existing execute.cjs, worktree.cjs, compaction.cjs

**Option 2: Quality Profile System**
- Approach: Define `.planning/QUALITY-PROFILE.md` with project-wide quality expectations (testing strategy, error handling patterns, documentation standards). During plan-set, the planner references this profile. During execute-set, the executor gets relevant excerpts injected.
- Pros: Single source of truth; configurable per project; does not bloat individual set artifacts.
- Cons: Another file to maintain; must be generated during init.

**Recommended:** Option 2 combined with Option 1 -- create a QUALITY-PROFILE.md during init (or context generation), and enhance the existing context injection to include relevant profile sections based on the set's scope.

#### Data Model Impact

- New entities: `.planning/QUALITY-PROFILE.md`, enhanced wave PLAN.md template sections
- Existing entities modified: `generateScopedClaudeMd` output (richer), `assembleExecutorPrompt` output (richer), set CONTEXT.md template (additional sections)
- Relationships: QUALITY-PROFILE.md referenced during plan-set and execute-set

#### API Surface

- New CLI command: `node rapid-tools.cjs context quality-profile` (generate QUALITY-PROFILE.md from codebase analysis)
- Modified: `execute prepare-context` enhanced with quality context
- Modified: `worktree.cjs generateScopedClaudeMd` to include quality directives
- Modified: plan-set SKILL.md to reference quality profile

### Feature: Quick Task Logging

- **Description:** The `/rapid:quick` command should log completed tasks to a state file or persistent location to serve as a memory of what ad-hoc work was done. Currently, quick tasks create artifacts in `.planning/quick/{id}-{slug}/` but are not tracked in any aggregated index.
- **Complexity:** trivial
- **Dependencies:** Memory (benefits from shared logging infrastructure, but can be independent)

#### Current State

The `/rapid:quick` SKILL.md already creates a directory at `.planning/quick/{id}-{slug}/` with PLAN.md, SUMMARY.md, and VERIFICATION-REPORT.md. It commits these artifacts. However, there is no index or queryable log of all quick tasks. The SKILL.md explicitly states "Do NOT add to STATE.json sets array."

#### Decomposition

1. **Quick task log file** -- Create `.planning/quick/LOG.jsonl` that records each quick task with id, slug, description, date, status, commits, and files modified.
2. **Log append in SKILL.md** -- Add a step to the quick SKILL.md (after Step 6) that appends to the log file.
3. **Quick task query CLI** -- New CLI command to list/search quick task history.
4. **Integration with memory system** -- If the memory feature is built, quick task log entries should also appear in the decision/activity log.

#### Implementation Strategies

**Option 1: JSONL Log File**
- Approach: Append a JSON object to `.planning/quick/LOG.jsonl` at the end of each quick task. Contains: `{ id, slug, description, date, status, commits, filesModified }`.
- Pros: Simple; append-only; consistent with proposed memory JSONL pattern; no state conflicts.
- Cons: Requires parsing JSONL for queries.
- Libraries/services: No new deps (fs.appendFileSync)

**Option 2: Quick Tasks Section in STATE.json**
- Approach: Add a `quickTasks` array to STATE.json at the milestone level. Each entry: `{ id, slug, description, completedAt }`.
- Pros: Queryable via existing state get; visible in /status.
- Cons: Contradicts current explicit anti-pattern ("Do NOT add to STATE.json"); STATE.json schema change required; Zod schema update needed.

**Recommended:** Option 1 -- JSONL log file keeps quick tasks separate from set lifecycle state (respecting the existing design decision) while providing persistence and queryability.

#### Data Model Impact

- New entities: `.planning/quick/LOG.jsonl`
- Existing entities modified: quick SKILL.md (additional step)
- Relationships: Optional integration with memory system

#### API Surface

- New CLI command: `node rapid-tools.cjs quick list [--last <N>]`
- New CLI command: `node rapid-tools.cjs quick log --id <id> --slug <slug> --description <desc> --status <status> --commits <hashes>`
- Modified: `/rapid:quick` SKILL.md (Step 6.5: append to LOG.jsonl)

## Feature Dependency Graph

```
add-set (independent -- existing feature, enhancement only)

documentation (independent)

ui-specs (independent)

hooks (independent)

memory (independent)
  \__ code-quality (benefits from memory's decision history, but not hard dependency)

quick-logging (independent, soft synergy with memory)
```

All seven features are structurally independent -- none has a hard blocking dependency on another. However, there are beneficial synergies:

- **memory + code-quality:** Code quality agents benefit from knowing user preferences stored in the memory system.
- **memory + quick-logging:** Quick task log entries could feed into the unified memory system.
- **hooks + memory:** Hooks could log state verification events to the memory log.
- **documentation + code-quality:** Better code quality context improves documentation generation accuracy.

---

## Implementation Order

1. **quick-logging:** Rationale -- smallest scope (trivial), provides immediate value, establishes the JSONL logging pattern that memory will reuse. Estimated 1-2 tasks.

2. **add-set:** Rationale -- enhancement to existing feature, trivial scope. Primarily adds DAG recalculation and CLI command. Estimated 2-3 tasks.

3. **hooks:** Rationale -- moderate complexity, high impact on reliability. Prevents the common failure mode of agents forgetting state updates. Standalone implementation with no dependencies. Estimated 3-4 tasks.

4. **memory:** Rationale -- foundational for both code-quality and cross-milestone continuity. Establishes the decision logging infrastructure that other features benefit from. Estimated 5-7 tasks.

5. **code-quality:** Rationale -- builds on memory system for decision injection. Requires changes to multiple existing modules (execute.cjs, worktree.cjs, compaction.cjs) so benefits from stability of earlier features. Estimated 5-7 tasks.

6. **ui-specs:** Rationale -- complex but self-contained. Does not block or depend on other features. Can be developed in parallel with code-quality. Estimated 6-8 tasks.

7. **documentation:** Rationale -- best done last because it benefits from all other features being in place. The documentation agent can then document the new features (hooks, memory, UI specs, etc.) as part of its generation pass. Estimated 6-8 tasks.
