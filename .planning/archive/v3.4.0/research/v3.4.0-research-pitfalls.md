# Pitfalls Research -- v3.4.0 Agent Intelligence

## Critical Pitfalls

### Pitfall: Zod 3.25 Ships Zod 4 Internally -- Package Size and CJS Breakage
- **Category:** compatibility
- **Stack:** zod ^3.25.76
- **Description:** Starting in v3.25.0, the `zod` npm package bundles *both* Zod 3 and Zod 4 at their respective subpaths. This doubles the package size and has caused CJS loading failures for projects with `type: "module"` in package.json. While RAPID does not set `type: "module"`, any future dependency or consumer that does will trigger breakage. Additionally, `npm install` may silently resolve to a newer 3.25.x that ships different Zod 4 internals, causing test failures without any code change.
- **Trigger:** Running `npm install` and getting a newer patch of zod 3.25.x, or a downstream tool importing `zod/v4` by mistake.
- **Impact:** Build failures, doubled bundle size, silent schema behavior changes if Zod 4 APIs leak in.
- **Prevention:**
  1. Pin zod to an exact version in package.json (e.g., `"zod": "3.25.76"`) rather than using `^3.25.76`.
  2. Add a lockfile (`package-lock.json`) and commit it to prevent drift.
  3. Never import from `zod/v4` -- always use `require('zod')` which resolves to the v3 API.
  4. If schema validation behaves unexpectedly after `npm install`, diff `node_modules/zod/package.json` against the pinned version.
- **Detection:** CI test failures after clean install; unexpectedly large `node_modules/zod` directory (>2MB).
- **References:**
  - [GitHub Issue #4923: Latest Zod3 now includes Zod4](https://github.com/colinhacks/zod/issues/4923)
  - [GitHub Issue #4434: CJS loading 3.25.7 broken](https://github.com/colinhacks/zod/issues/4434)
  - [Zod v4 Migration Guide](https://zod.dev/v4/changelog)

### Pitfall: passthrough() on State Schemas Allows Silent Data Corruption
- **Category:** data-loss
- **Stack:** zod 3.25.76 + state-schemas.cjs
- **Description:** Every Zod schema in `state-schemas.cjs` uses `.passthrough()`, which means *any* unrecognized key injected into STATE.json will be silently preserved through parse/validate/write cycles. This is by design for extensibility, but it means a buggy agent or skill that writes a malformed field (e.g., `"stauts": "merged"` instead of `"status": "merged"`) will persist the typo indefinitely. For the v3.4.0 milestone, multiple new features (memory, quick logging, hooks, code quality enrichment) will add new fields to state objects. If those fields are misspelled or structured incorrectly, `.passthrough()` will let them through without any validation.
- **Trigger:** Agent writes `state.sets[0].memories = [...]` (a new field that has no schema enforcement). A bug writes `state.sets[0].memorys = [...]` -- both pass validation identically.
- **Impact:** Accumulation of garbage keys in STATE.json that confuse agents reading state; state bloat over time; no error signal when the wrong field name is used.
- **Prevention:**
  1. When adding new fields to state objects for this milestone (e.g., memory entries, quick-task logs, UI contracts), add them to the Zod schema with explicit types rather than relying on passthrough.
  2. Consider a `state detect-corruption` enhancement that warns on unexpected top-level keys.
  3. Write unit tests that assert specific field names are present after mutation operations.
- **Detection:** Run `Object.keys()` diff between expected schema shape and actual STATE.json keys. If unexpected keys appear, they were introduced by a passthrough leak.

### Pitfall: Hook State-Check Can Deadlock on STATE.json Lock
- **Category:** correctness
- **Stack:** proper-lockfile 4.1.2 + state-machine.cjs + hooks system
- **Description:** The milestone goal "hooks that run on completion of a rapid agent to check if project state has been updated" requires reading STATE.json from within a hook handler. If the hook fires during a `withStateTransaction()` that has not yet released its lock (e.g., a `TaskCompleted` hook fires while the agent's final state transition is still in progress), the hook's attempt to `acquireLock(cwd, 'state')` will either deadlock (same process) or spin on retries (different process). The lock retry config allows 10 retries with exponential backoff up to 2s, but if the lock holder is the same Node.js process, no amount of retrying will succeed because Node.js is single-threaded.
- **Trigger:** `TaskCompleted` hook fires synchronously or in the same process as the agent that is mid-transaction on STATE.json.
- **Impact:** Hook hangs for 5 minutes (stale threshold), then either crashes or proceeds with stale state. In the worst case, the lock is marked compromised and two writers corrupt STATE.json simultaneously.
- **Prevention:**
  1. Hook handlers must use `readState()` (which does NOT acquire a lock) rather than `withStateTransaction()` for read-only state checks.
  2. If the hook needs to *write* state, it should shell out to `node "${RAPID_TOOLS}" state transition ...` as a separate process so the lock is acquired independently.
  3. Never acquire a lock inside a callback that may be invoked while the same lock is held.
  4. Add a `readStateNoLock()` utility if one does not already exist (current `readState()` does not lock, which is correct).
- **Detection:** Hook execution exceeds 10 seconds; `[RAPID] Lock "state" compromised` message in stderr.
- **References:**
  - [proper-lockfile: Error: Lock file is already being held](https://github.com/moxystudio/node-proper-lockfile/issues/92)

### Pitfall: add-set Writes STATE.json Directly, Bypassing withStateTransaction
- **Category:** data-loss
- **Stack:** skills/add-set/SKILL.md + state-machine.cjs
- **Description:** The existing `add-set` skill (SKILL.md Step 5) instructs the agent to "Write the updated STATE.json using the Write tool (same pattern as /init roadmapper writes STATE.json directly)." This means the agent reads STATE.json, modifies it in memory, and writes it back via the `Write` tool -- completely bypassing the lock-protected `withStateTransaction()` function. If another agent or process writes STATE.json between the read and write, the second write will silently overwrite the first agent's changes. With the v3.4.0 milestone adding richer state and more concurrent features, this race window becomes more likely.
- **Trigger:** Two agents running concurrently: one doing `add-set` and another finishing `execute-set` that transitions state. The `add-set` agent's stale read loses the state transition.
- **Impact:** Lost state transitions, sets reverting to previous statuses, milestone regression.
- **Prevention:**
  1. Refactor `add-set` to use `node "${RAPID_TOOLS}" state add-set <setId>` CLI command that internally uses `withStateTransaction()`.
  2. At minimum, use `state commit-state` after the write to detect if the file was modified by someone else.
  3. Add a CLI command `state add-set` that atomically adds a set to the current milestone within a transaction.
- **Detection:** `state detect-corruption` after add-set; git diff shows unexpected state changes.

## High-Risk Pitfalls

### Pitfall: Memory Storage Without Size Bounds Causes Context Bloat
- **Category:** performance
- **Stack:** .planning/ filesystem + agent context assembly
- **Description:** The "memory" feature (items 5 and 7 from the todo list) will store user requests, mid-implementation changes, and /quick task logs. If this storage has no size limits, the accumulated memory will grow unboundedly. Since RAPID agents load planning context into their prompts, unbounded memory files will eventually exceed Claude's context window or cause the `agent_size_warn_kb` threshold (currently 15KB) to trigger on every invocation. Quick tasks alone already generate 8+ task directories in past milestones.
- **Trigger:** Over 20+ quick tasks or 10+ memory entries across a milestone's lifetime.
- **Impact:** Agent context exceeds useful limits; prompt assembly becomes slow; agents get confused by excessive historical context.
- **Prevention:**
  1. Implement a ring buffer or capped-size strategy: keep only the last N memory entries (e.g., 20) or cap total memory file size at 10KB.
  2. Use the existing compaction pattern (see `compaction.cjs`) to create digest/summary versions of old memory entries.
  3. Separate "active memory" (current milestone) from "archive memory" (previous milestones), and only load active memory into agent context.
  4. For /quick logging, store only task ID, slug, date, and status -- not the full plan or execution details.
- **Detection:** Monitor `.planning/memory/` directory size; warn in `execute prepare-context` if memory context exceeds a threshold.

### Pitfall: Hook Command Scripts Receive JSON on stdin -- jq Dependency Not Guaranteed
- **Category:** compatibility
- **Stack:** Claude Code hooks API + bash scripts
- **Description:** The existing `rapid-task-completed.sh` hook uses `jq` to parse JSON from stdin (`echo "$INPUT" | jq -r '.task_id'`). The v3.4.0 milestone plans to add more hooks for state checking. However, `jq` is not a guaranteed dependency -- it is not listed in the project's prerequisites, and the `prereqs` command only checks for git, Node.js, and jq. While jq happens to be checked, the hook scripts themselves do not fail gracefully if jq is missing. Additionally, Claude Code hook commands receive JSON on stdin and must return valid JSON on stdout -- mixing shell text output with JSON will cause hook failures.
- **Trigger:** User system without jq installed; hook script that echo's debug text alongside JSON output.
- **Impact:** Hook silently fails (exit code 0 with no JSON = no effect); or hook crashes with parse error.
- **Prevention:**
  1. Write new hooks in Node.js rather than bash to avoid jq dependency entirely: `"command": "node /path/to/hook.cjs"` -- the hook can `JSON.parse()` stdin natively.
  2. If bash hooks are used, check for jq at the top of the script and fail gracefully with exit 0 (not exit 2, which would block the action).
  3. Hook stdout must be pure JSON or empty -- never mix with debug output. Use stderr for debug logging.
  4. Consider the `type: "prompt"` or `type: "agent"` hook handlers from the Claude Code API instead of command hooks for complex state-checking logic.
- **Detection:** Hook does not fire; no `.planning/teams/` files created; stderr shows "jq: command not found".
- **References:**
  - [Claude Code Hooks Reference](https://code.claude.com/docs/en/hooks)

### Pitfall: UI Specs/Contracts Feature Has No Existing Infrastructure
- **Category:** correctness
- **Stack:** contract.cjs + plan.cjs
- **Description:** The existing CONTRACT.json schema (`CONTRACT_META_SCHEMA` in contract.cjs) defines exports/imports for *functions* and *types* only. There is no concept of UI components, layouts, design tokens, or visual contracts. The v3.4.0 milestone goal "define UI specs/contracts across phases to ensure coherent UI creation" requires either extending the existing contract schema or creating a parallel system. If the existing schema is extended, all existing Ajv validation, `checkOwnership()`, and contract test generation must be updated. If a parallel system is created, the two systems may diverge and confuse agents about which contract to check.
- **Trigger:** Agent tries to validate a UI contract against the existing `CONTRACT_META_SCHEMA` and it fails because the schema does not define UI fields.
- **Impact:** UI contract validation silently passes (because unknown fields in JSON Schema default to valid) or crashes depending on `additionalProperties` settings.
- **Prevention:**
  1. Decide upfront: extend `CONTRACT_META_SCHEMA` with a `ui` section, or create a separate `UI-CONTRACT.json` with its own schema.
  2. If extending, add Ajv schema properties for UI components *before* any set uses them.
  3. Add contract validation gate that explicitly checks for UI contract presence in sets that touch frontend files.
  4. Write a schema migration for existing CONTRACT.json files that adds the new `ui` section as empty/optional.
- **Detection:** Contract validation returns valid for a CONTRACT.json that should have UI specs but does not.

### Pitfall: Documentation Generation May Overwrite User-Edited Files
- **Category:** data-loss
- **Stack:** docs/*.md + DOCS.md + generated documentation
- **Description:** The "documentation" feature will generate/update existing documentation. If the generator overwrites files that the user has manually edited (e.g., DOCS.md, README.md, or files in docs/), those manual edits are lost. The current codebase has 9 documentation files in `docs/` and a top-level `DOCS.md`. Without a diff/merge strategy, a documentation generator that writes these files will destroy customizations.
- **Trigger:** User runs documentation generation after manually editing `docs/architecture.md` with project-specific notes.
- **Impact:** User's manual documentation edits are silently overwritten.
- **Prevention:**
  1. Use a "generated sections" pattern: documentation files have marked regions (e.g., `<!-- GENERATED START -->` / `<!-- GENERATED END -->`) and only those regions are rewritten.
  2. Before overwriting, check git diff -- if the file has uncommitted changes, warn and skip or prompt for confirmation.
  3. Keep generated documentation in a separate directory (e.g., `docs/generated/`) distinct from user-authored documentation.
  4. Store a hash of the last generated content and only regenerate if source has changed.
- **Detection:** `git diff` shows documentation files changed after generation; user reports lost notes.

## Medium-Risk Pitfalls

### Pitfall: proper-lockfile onCompromised Silently Continues
- **Category:** data-loss
- **Stack:** proper-lockfile 4.1.2 + lock.cjs
- **Description:** In `lock.cjs`, the `onCompromised` callback logs to stderr but does not throw or abort: `process.stderr.write(...)`. This means if a lock is compromised (e.g., another process steals it), the current operation continues as if it still holds the lock. For single-writer scenarios this rarely matters, but with the v3.4.0 milestone adding hooks that may read/write state concurrently, a compromised lock could lead to two writers updating STATE.json simultaneously.
- **Trigger:** Lock stale timeout (5 minutes) expires during a long-running agent operation; another process acquires the "stale" lock; original process's lock is compromised but it continues writing.
- **Impact:** STATE.json contains a partial merge of two independent mutations.
- **Prevention:**
  1. Change `onCompromised` to set a flag that `withStateTransaction` checks before writing.
  2. Consider shortening the stale threshold for hooks (which should complete in seconds, not minutes).
  3. Add a `state detect-corruption` check after any operation that took longer than the stale threshold.
- **Detection:** `[RAPID] Lock "state" compromised` in stderr; STATE.json contains unexpected field values.

### Pitfall: Quick Task ID Generation Uses Directory Count -- Brittle to Deletions
- **Category:** correctness
- **Stack:** skills/quick/SKILL.md
- **Description:** The quick task SKILL.md computes the next ID as `EXISTING=$(ls .planning/quick/ | wc -l); NEXT_ID=$((EXISTING + 1))`. If a user deletes a quick task directory, the count decreases and the next ID will collide with an existing directory. For the v3.4.0 "quick logging" feature, this means log entries could be associated with the wrong task.
- **Trigger:** User deletes `.planning/quick/3-some-task/`, then runs another quick task. The new task gets ID 3, colliding with any references to the original task 3.
- **Impact:** Task log references become ambiguous; summary files overwrite each other.
- **Prevention:**
  1. Use a high-water-mark file (`.planning/quick/.next-id`) that only increments, never decreases.
  2. Or parse existing directory names to find `max(id) + 1` instead of counting.
  3. For the logging feature, include a timestamp or UUID in the log entry to disambiguate.
- **Detection:** Two quick task directories with the same numeric prefix; log file references non-existent task.

### Pitfall: Agent State Transition Forgetfulness Is a Prompt Problem, Not a Code Problem
- **Category:** correctness
- **Stack:** Agent prompts (src/modules/roles/*.md) + hooks
- **Description:** The todo item "agents often forget to update the project state" suggests adding hooks as a guardrail. However, the root cause is that agent role prompts do not always include explicit instructions to call `state transition` at the end of their work. A hook that *checks* after the fact can detect the omission but cannot fix it (the agent has already terminated). If the hook blocks via exit code 2, the TaskCompleted event prevents the task from being marked complete, but the agent's work is already done -- it just was not recorded in state.
- **Trigger:** Agent completes work but does not call `state transition set` before emitting RAPID:RETURN.
- **Impact:** STATE.json shows stale status; subsequent commands fail because they expect a different status.
- **Prevention:**
  1. Fix the root cause: audit all role prompts in `src/modules/roles/` and ensure every role that performs state transitions has an explicit checklist item for state update.
  2. Use a `Stop` hook (not `TaskCompleted`) which fires when the main agent finishes responding -- the hook can inject `additionalContext` telling the agent to check state before truly stopping.
  3. Use a `PostToolUse` hook matching the `Agent` tool to check if the subagent's RAPID:RETURN includes state transition evidence.
  4. The hook should *prompt* the agent to fix the issue (using `decision: "block"` with a `reason`), not just log it.
- **Detection:** `state get set` shows status that does not match the expected status after a skill run.
- **References:**
  - [Claude Code Hooks: Stop event](https://code.claude.com/docs/en/hooks)

### Pitfall: Memory Feature Creates Circular Context -- Agents Read Their Own Previous Mistakes
- **Category:** correctness
- **Stack:** Agent context assembly + memory storage
- **Description:** If memory stores "what the user requested" and "what agents decided", then future agents will read both the good decisions and the bad ones. If an agent made a wrong implementation choice that was later corrected, the memory of the wrong choice could confuse a future agent into repeating it. This is especially problematic for the "user changes their mind mid-implementation" scenario -- the memory will contain both the original request AND the changed request, potentially causing agents to implement a hybrid of both.
- **Trigger:** User says "use SQLite" during discuss-set, then changes to "use PostgreSQL" during execute-set. Memory contains both directives.
- **Impact:** Future agents attempt to use SQLite because it appeared first in memory, even though the user switched to PostgreSQL.
- **Prevention:**
  1. Memory entries must be timestamped and explicitly marked as "superseded" when the user changes direction.
  2. Use a structured format: `{ "directive": "...", "superseded_by": "entry-id", "timestamp": "..." }`.
  3. When loading memory into agent context, filter out superseded entries or present them with clear "[SUPERSEDED]" markers.
  4. Consider a "decisions log" pattern where only the *latest* decision for each topic is loaded by default.
- **Detection:** Agent output references a decision that was explicitly reversed by the user.

### Pitfall: Ajv require('ajv').default Import Pattern Is Fragile
- **Category:** compatibility
- **Stack:** ajv ^8.17.1 + contract.cjs
- **Description:** The contract module uses `require('ajv').default` which is the correct CJS import for Ajv v8 (ESM-first packaging). However, this pattern breaks if: (a) Ajv publishes a v9 with a different export shape, (b) a bundler processes the import differently, or (c) the project ever migrates to ESM. The comment in contract.cjs acknowledges this fragility but does not guard against it. For v3.4.0, if UI contracts are validated with Ajv, this import will be duplicated in more files.
- **Trigger:** Ajv major version bump; or copying the import pattern incorrectly in new code.
- **Impact:** `TypeError: Ajv is not a constructor` at runtime.
- **Prevention:**
  1. Pin Ajv to exact version in package.json.
  2. Centralize the Ajv instantiation in a single module (e.g., `ajv-instance.cjs`) that all contract-related code imports.
  3. Add a startup smoke test that verifies `new Ajv()` works.
- **Detection:** Runtime crash with "Ajv is not a constructor" or "Cannot read properties of undefined".
- **References:**
  - [Ajv ESM/CJS Issue #2569](https://github.com/ajv-validator/ajv/issues/2569)

### Pitfall: Git Worktree Shared Index Lock Under Concurrent Operations
- **Category:** correctness
- **Stack:** git worktrees + worktree.cjs
- **Description:** Git worktrees share the `.git` directory of the main repository. When multiple sets are being worked on in parallel, concurrent git operations (commits, status checks, branch operations) can cause `index.lock` contention. The existing worktree.cjs uses `execSync` with a 30-second timeout, which will crash if git cannot acquire its own internal lock within that window. With v3.4.0 adding hooks that may run git operations on TaskCompleted, the concurrency window widens.
- **Trigger:** Two agents in different worktrees both running `git commit` simultaneously; or a hook running `git status` while an agent is mid-commit.
- **Impact:** `fatal: Unable to create '/path/.git/worktrees/set-name/index.lock': File exists` -- agent crashes.
- **Prevention:**
  1. Wrap git operations in retry logic with backoff (the existing `gitExec` does not retry).
  2. Clean stale `index.lock` files if they are older than the git operation timeout.
  3. Ensure hooks that run git commands use separate worktree paths, not the main repo.
  4. Consider using `--no-optional-locks` flag for read-only git operations in hooks.
- **Detection:** Git error messages containing "index.lock" in stderr; agent BLOCKED returns with git errors.
- **References:**
  - [Git Worktree Documentation](https://git-scm.com/docs/git-worktree)
  - [Fixing Git Lock Errors](https://dev.to/rijultp/fixing-common-git-lock-errors-understanding-and-recovering-from-gitindexlock-47ej)

## Anti-Patterns for This Stack

### Anti-Pattern: Writing STATE.json Directly from Agent Prompts
- **What people do:** Skill instructions tell agents to "Read STATE.json, modify it, write it back" using the Write tool, bypassing the CLI's transaction system.
- **Why it fails:** No lock protection; race conditions with concurrent agents; Zod validation skipped; `lastUpdatedAt` not updated.
- **Correct approach:** Always use `node "${RAPID_TOOLS}" state transition set ...` or `withStateTransaction()` for state mutations. For new state operations (like add-set), create a CLI command that wraps the transaction.
- **Example:** `add-set` SKILL.md currently writes STATE.json directly -- this should be refactored to use a `state add-set` CLI command.

### Anti-Pattern: Using .passthrough() for Extensibility Without Field Allowlists
- **What people do:** Schema uses `.passthrough()` to allow future fields without schema changes.
- **Why it fails:** Typos in field names are silently accepted; garbage accumulates in state over time; no way to distinguish intentional extensions from bugs.
- **Correct approach:** Add new fields to the Zod schema explicitly when they are needed. Use `.passthrough()` sparingly, and complement it with runtime checks for known-unexpected fields.

### Anti-Pattern: Counting Files to Generate Sequential IDs
- **What people do:** `ls directory/ | wc -l` to determine the next numeric ID.
- **Why it fails:** Deletions cause ID reuse; race conditions if two quick tasks start simultaneously; hidden files (dotfiles) affect count.
- **Correct approach:** Use a monotonic counter stored in a file (`.next-id`), or use timestamp-based IDs, or parse existing names to find `max + 1`.

### Anti-Pattern: Shell Hooks That Mix stdout Formats
- **What people do:** Write bash hooks that `echo` debug messages and also output JSON.
- **Why it fails:** Claude Code expects *only* valid JSON on stdout from command hooks (on exit 0). Any non-JSON text causes parse failures.
- **Correct approach:** Use stderr for all debug/status output. Return only JSON or nothing on stdout. Better yet, write hooks in Node.js to avoid shell quoting issues entirely.

### Anti-Pattern: Loading All Memory Into Every Agent Context
- **What people do:** Concatenate all memory files and inject them into the agent prompt preamble.
- **Why it fails:** Context window fills up with irrelevant historical data; agents get confused by contradictory historical entries; token cost increases.
- **Correct approach:** Load only memory relevant to the current set/phase. Use the compaction pattern (digests for old entries, full content for recent ones). Cap total memory context at a fixed token budget.

## Version-Specific Gotchas

- **Zod 3.25.76:** `.passthrough()` combined with `.omit()` causes omitted keys to pass through as "unknown" keys rather than being stripped. Do not combine these methods.
- **proper-lockfile 4.1.2:** The `update` interval (half of `stale` by default) must be shorter than `stale` or locks will appear stale to other processes while the holder is active. Current config: stale=300000ms, update=150000ms -- this is correct.
- **Node.js 18+:** `fs.renameSync()` is atomic on Linux/macOS but NOT on Windows (cross-filesystem). RAPID's atomic write pattern (tmp+rename) is safe on the target platforms but would fail on Windows if `.planning/` is on a different filesystem than the temp file.
- **Ajv 8.17.1:** The `require('ajv').default` import path is specific to v8. If Ajv is bumped to v9, this will likely break. Pin the version.

## Security Pitfalls

1. **Hook command injection:** Hook commands receive JSON via stdin. If the JSON contains shell metacharacters and the hook uses `echo "$INPUT" | jq ...`, improperly quoted variables can lead to command injection. Prevention: use `cat` instead of `echo` for stdin, or write hooks in Node.js.
2. **STATE.json contains no authentication:** Any process that can read `.planning/STATE.json` can modify it. Hooks and agents all share the same file with no access control. Prevention: rely on filesystem permissions; do not store secrets in state.
3. **Git branch deletion without confirmation:** `worktree delete-branch` with `--force` can destroy uncommitted work. Prevention: the existing code checks for dirty worktrees, but hooks or automated cleanup could bypass this.
4. **Memory files may store sensitive user input:** If the "memory" feature records verbatim user requests during discuss-set or new-version, it could capture API keys, passwords, or internal URLs the user mentions. Prevention: never store raw user input in memory; sanitize or summarize.

## Performance Traps

1. **Large STATE.json with history of all milestones:** STATE.json currently contains 10 milestones with all their sets. As more milestones are added, every `readState()` call parses the entire history. Prevention: archive completed milestones out of STATE.json (the archive pattern already exists in `.planning/archive/`).
2. **Synchronous git operations in hooks:** `execSync` in worktree.cjs blocks the Node.js event loop. If hooks trigger git operations, they add latency to every agent task completion. Prevention: use async git operations or run hooks with `async: true` in the hooks configuration.
3. **Context assembly reads every .planning/ file sequentially:** `prepareSetContext()` and `assembleCompactedWaveContext()` read multiple files synchronously. Adding memory files, UI contracts, and richer context multiplies the I/O. Prevention: read files in parallel with `Promise.all()`; cache frequently-read files.
4. **Unbounded quick task directory growth:** Each quick task creates a directory with PLAN.md, VERIFICATION-REPORT.md, and SUMMARY.md. Over time, `.planning/quick/` can contain hundreds of directories. Prevention: archive completed quick tasks when starting a new version; implement a cleanup command.

## Summary Priority List

Ordered by likelihood x impact:

1. **Hook state-check deadlock** -- Hooks reading/writing STATE.json while a transaction lock is held will hang or corrupt state. Most likely to occur because the hooks feature is new and touches the most critical shared resource.
2. **add-set direct STATE.json writes** -- Already exists in production. Every `add-set` invocation risks a race condition. Impact grows with v3.4.0's richer concurrent features.
3. **passthrough() silent data corruption** -- New fields added without schema enforcement will accumulate typos and garbage. Affects every feature in this milestone that extends state.
4. **Memory context bloat** -- Unbounded memory storage will degrade agent performance within a single milestone lifecycle. Both the memory and quick-logging features contribute.
5. **Zod 3.25 package instability** -- A clean `npm install` can pull a different patch version that breaks tests. Low effort to prevent (pin version), but high impact if not addressed.
6. **Memory circular context** -- Superseded decisions confusing future agents. Subtle and hard to detect after the fact.
7. **UI contract schema gap** -- No infrastructure exists for UI contracts. Building on the wrong abstraction will cause rework.
8. **Hook jq dependency** -- Existing hook uses jq; new hooks should prefer Node.js to avoid this.
9. **Quick task ID collision** -- File-counting ID generation will break after deletions.
10. **Documentation overwrite** -- Generator must respect user-edited sections or it destroys manual work.
