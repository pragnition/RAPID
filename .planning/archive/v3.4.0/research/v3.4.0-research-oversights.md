# Oversights Research -- Agent Intelligence Milestone (v3.4.0)

## Cross-Cutting Concerns

### 1. Memory Persistence Architecture: No Unified Storage Layer

The milestone includes three distinct "memory" features (item 5: user request memory, item 6: richer planning context, item 7: /quick logging), but there is no existing memory infrastructure to build on. Each will likely be implemented independently unless a shared storage pattern is established early.

**What gets overlooked:**
- **Three separate storage locations:** /quick logging will naturally write to `.planning/quick/`, user request memory during /new-version likely goes in `.planning/` or `STATE.json`, and richer planning context updates `.planning/context/`. Without a unified approach, agents reading "what has happened so far" must check three unrelated locations.
- **Memory file format inconsistency:** The existing `rapid-task-completed.sh` hook writes JSONL (append-only). Quick task summaries are Markdown. STATE.json is structured JSON. A new memory system that mixes formats will be harder for agents to parse at context-assembly time. The `execute.cjs:prepareSetContext()` function currently reads only `scopedMd`, `definition`, and `contractStr` -- it has no concept of "project memory."
- **Memory accumulation across milestones:** The `/new-version` skill archives old milestone artifacts (Step 4.5). If memory is stored per-milestone (e.g., in `.planning/sets/`), it gets archived. If it is stored globally (e.g., `.planning/MEMORY.md`), it accumulates unboundedly. Neither behavior is obviously correct, and the decision affects whether agents in v3.5.0 can reference v3.4.0 decisions.
- **Context window budget for memory:** The compaction system (`compaction.cjs`) has a `DEFAULT_BUDGET_TOKENS = 120000` limit. Adding memory content to agent prompts without budgeting means agent prompts could exceed context limits. Memory needs its own compaction/summarization strategy, or it needs to be integrated into the existing compaction pipeline.

**Why this matters for RAPID specifically:** RAPID agents already struggle with context overload -- the `agent_size_warn_kb: 15` config exists precisely because agent prompts grew too large. Adding memory to prompts without a budget will exacerbate this. The v3.3.0 context-optimization set added digest/compaction infrastructure; memory should leverage that same pattern rather than creating a parallel one.

### 2. Hook System Has No Programmatic Interface

The milestone introduces hooks (item 4) to ensure agents update project state on completion. The only existing hook is `rapid-task-completed.sh` -- a bash script that reads JSON from stdin and writes JSONL. There is no hook registry, no hook lifecycle management, and no way for hooks to communicate results back to the caller.

**What gets overlooked:**
- **Hook discovery and registration:** The existing hook is hard-coded in `.claude-plugin/plugin.json` (or discovered by Claude Code's plugin framework). RAPID has no mechanism for registering custom hooks -- the current hook relies on Claude Code's TaskCompleted event. If the new hooks need to fire at RAPID-specific lifecycle events (e.g., "after set transitions to complete"), they cannot use Claude Code's hook infrastructure at all. They need RAPID's own hook dispatch.
- **Hook return values:** The current `rapid-task-completed.sh` is fire-and-forget (exits 0). But the proposed hooks need to "check if project state has been updated" -- this implies the hook must be able to report a finding (state was/was not updated) and potentially prompt the agent to fix it. That requires a request-response pattern, not fire-and-forget.
- **Hook timing in the agent lifecycle:** Claude Code's TaskCompleted hook fires *after* the agent has already finished its response. At that point, the agent cannot retroactively update state. The hook either needs to fire *before* the agent emits its RAPID:RETURN (as an inline check within the role module instructions), or it needs to be a separate post-agent step in the SKILL.md orchestration.
- **Hook and state machine coupling:** If hooks validate state transitions, they need access to the state machine. But hooks currently run as shell scripts, not Node.js modules. They would need to call `node "${RAPID_TOOLS}" state get` to check state -- adding a shell-to-Node.js bridge cost on every hook invocation.
- **Existing hook's project root detection is fragile:** Lines 20-25 of `rapid-task-completed.sh` use relative path traversal (`../`) to find `.planning/`. This breaks when the working directory is a worktree (`~/.rapid-worktrees/set-name/`), which is 2-3 levels below `.planning/`, not 1. The existing hook may silently fail in worktree contexts.

**Why this matters for RAPID specifically:** The real problem hooks solve is that agents forget to call `state transition set` after completing work. The simplest solution may not be hooks at all -- it may be adding a state-check step to the SKILL.md orchestrators (skills/execute-set, skills/plan-set, etc.) that runs after parsing RAPID:RETURN. This is more reliable than a separate hook system because the SKILL.md already has access to the state and the transition logic. The hook system adds complexity that may not pay off if SKILL.md-level checks suffice.

### 3. /quick Logging Has No Query Interface

Item 7 asks /quick to log to a state file for memory. But logging without retrieval is useless. The oversight is that nobody will query the quick task log unless there is an explicit read path.

**What gets overlooked:**
- **Where does the log get read?** Currently, quick task artifacts live in `.planning/quick/{N}-{slug}/`. The SKILL.md writes a `{N}-SUMMARY.md` file (Step 6) and commits it. But no other skill or agent reads these summaries. The `/status` command does not display quick task history. The `/init` and `/new-version` research pipelines do not include quick task history as input. The log exists but is never consumed.
- **Structured vs unstructured logging:** The current SUMMARY.md is Markdown (human-readable but hard to query programmatically). If agents need to search quick task history (e.g., "did we already fix the lock timeout issue?"), they need either structured JSON or a searchable index. The JSONL pattern from `rapid-task-completed.sh` would be more appropriate for programmatic consumption.
- **Quick task count/ID collision:** The current ID generation (`ls .planning/quick/ | wc -l`) counts directories. If a quick task directory is deleted or archived, the count decreases and the next ID may collide with an existing one. This is a latent bug that becomes visible once quick tasks accumulate. Use a monotonic counter stored in a file, not directory count.
- **Quick task state not in STATE.json:** The SKILL.md explicitly says "Do NOT add to STATE.json sets array." This means quick task history is invisible to the state machine, status display, and any state-aware tooling. If /quick logging is meant to serve as memory, it needs to be queryable from the same place agents check project state.

### 4. UI Specs/Contracts Extend an Existing System That Has No UI Concept

Item 3 proposes UI specs/contracts across phases for coherent UI creation. The existing CONTRACT.json schema (defined in `contract.cjs`) only supports `exports.functions` and `exports.types` -- there is no notion of UI components, layouts, screens, or visual specs.

**What gets overlooked:**
- **CONTRACT.json schema must be extended:** Adding UI specs means either extending the existing JSON Schema (`CONTRACT_META_SCHEMA` in contract.cjs) with new top-level keys (e.g., `uiComponents`, `screens`, `layouts`), or creating a separate artifact (e.g., `UI-CONTRACT.json`). Extending the existing schema is cleaner but requires updating all validation paths. Creating a separate artifact avoids schema migration but doubles the number of contract files to manage.
- **Ajv validation gates affected:** Contracts are validated at 3 gates: after planning, during execution, before merge. If UI specs are added to contracts, all three gates must handle the new fields. The `checkOwnership()` function in contract.cjs and the stub generation in `stub.cjs` also need updates.
- **UI specs presuppose a UI project:** RAPID is used for any project type (library, CLI, API, webapp). UI contracts are only relevant for webapps. The contract system needs a way to mark UI specs as optional/conditional based on project type. The `scaffold.cjs` classifier already detects project types (`webapp`, `api`, `library`, `cli`), but contracts do not reference project type.
- **No rendering or validation for UI specs:** Code contracts can be validated by checking that exported functions exist in the source code. UI specs have no equivalent validation -- there is no way to programmatically verify that a component renders a specified layout. The "contract validation" gate for UI specs is fundamentally different from code contracts.
- **Cross-set UI consistency:** The whole point is ensuring "coherent UI creation" across sets. This means UI contracts must be shared/visible across sets, not scoped to individual set directories. The existing contract system is set-scoped (`.planning/sets/{set-id}/CONTRACT.json`). A project-wide `UI-SPECS.md` or `UI-CONTRACTS.json` in `.planning/` would be more appropriate for cross-cutting UI rules.

### 5. add-set Breaks DAG Invariants

The add-set skill (item 1) adds a new set to an existing milestone. The existing SKILL.md (already implemented in `skills/add-set/SKILL.md`) writes the set directly to STATE.json with `pending` status and appends to ROADMAP.md. However, it does not update the dependency graph.

**What gets overlooked:**
- **No dependency edge declaration:** The add-set SKILL.md asks about dependencies (Step 2, Question 2: "Are there dependencies on existing sets?") but records the answer only in DEFINITION.md as text. It does not update the DAG edges in ROADMAP.md or STATE.json. The `dag.cjs` module uses Kahn's topological sort to determine wave groupings, but it gets its edges from the roadmapper output, not from DEFINITION.md text.
- **Wave ordering disruption:** If a user adds a set that depends on sets already in execution, the DAG wave assignment changes. Sets that were in Wave 2 might need to move to Wave 3 to accommodate the new dependency. The existing `assignWaves()` function recomputes waves from edges, but nobody calls it after add-set.
- **Merge ordering:** The merge pipeline merges sets in DAG order. A set added mid-milestone without proper dependency edges could be merged before its dependencies, causing integration failures.
- **Contract cross-references:** The add-set SKILL.md creates an empty CONTRACT.json with no imports/exports. But if the new set depends on existing sets, those sets' contracts need an `imports.fromSets` entry pointing to the existing sets. This is deferred to /plan-set, but the gap between add-set and plan-set means the DAG is temporarily inconsistent.

### 6. Documentation Generation Has No Source-of-Truth Tracking

Item 2 proposes generating/updating project documentation at version end. But there is no mechanism to track what documentation exists, what it covers, and whether it is stale.

**What gets overlooked:**
- **No doc manifest:** The 9 files in `docs/` are not registered anywhere. There is no manifest listing which docs exist, what they cover, or when they were last updated. A documentation generation command needs to know what to update. Without a manifest, it must re-scan all docs every time.
- **Docs drift from implementation:** `docs/state-machines.md` describes state transitions. If state transitions change in `state-transitions.cjs`, the docs become stale. There is no linkage between source files and their documentation. A doc generation tool that blindly regenerates risks overwriting manual edits; one that only appends misses stale content.
- **Generated vs hand-written docs:** Some docs may be fully generated (API reference, command list), while others are hand-written (troubleshooting, setup guides). The generation tool must distinguish between these. A `<!-- GENERATED: DO NOT EDIT -->` marker pattern would prevent overwriting hand-written content.
- **CLAUDE.md instructions:** The project's CLAUDE.md references `.planning/context/` for generated context files. If documentation generation updates these context files, it changes the instructions agents receive. This is powerful but dangerous -- a bad generation run could corrupt agent instructions.
- **Documentation scope:** Does "documentation" mean user-facing docs (docs/*.md), developer docs (code comments, JSDoc), project context (`.planning/context/*.md`), or all of the above? Each has different generation strategies and different staleness risks.

### 7. Code Quality Improvements Lack Measurable Baselines

Item 6 asks for "richer planning/context/state to ensure agents get a more descriptive view." But there is no way to measure whether agents' output quality actually improves.

**What gets overlooked:**
- **No quality metrics exist today:** There are no code coverage targets, no lint rules, no complexity thresholds. The CONVENTIONS.md mentions `node:test` but has no coverage requirements. Without baselines, "richer context" is a subjective improvement with no way to verify it worked.
- **Agent prompt size vs quality tradeoff:** More context does not automatically mean better output. The v3.3.0 milestone added compaction specifically because prompts were too large. Adding more planning/state detail increases prompt size. The question is not "add more" but "add the right things and remove the wrong things."
- **Which agents benefit?** The executor agent's quality depends on PLAN.md specificity. The planner agent's quality depends on CONTEXT.md richness. The reviewer agents' quality depends on knowing what to look for. Each agent needs different kinds of enriched context. A blanket "more state" approach helps some agents while adding noise for others.
- **Feedback loop from review pipeline:** The review pipeline (scoper -> unit-tester -> bug-hunter -> devils-advocate -> judge) generates detailed findings. These findings are never fed back into the planning context for future sets. If a bug pattern is found in set-01, set-02's planner has no way to know about it. A "lessons learned" feedback mechanism from review to planning would improve code quality more than richer initial context.

### 8. Testing Strategy for SKILL.md Files

None of the milestone items have automated tests for SKILL.md behavior. This is a systemic blind spot.

**What gets overlooked:**
- **SKILL.md files are untestable:** SKILL.md files are Markdown instructions interpreted by Claude Code. They cannot be unit tested. The only testing is manual invocation. This means add-set, hooks, /quick logging, documentation generation, and memory are all implemented as SKILL.md changes with no regression safety net.
- **SKILL.md instructions can contradict library behavior:** The add-set SKILL.md writes STATE.json directly using the Write tool (Step 5), bypassing `state-machine.cjs:writeState()`. This means the Zod validation in `writeState()` is not applied. If the SKILL.md constructs invalid state, it will not be caught until the next CLI command reads the state. The same risk exists for any new SKILL.md that manipulates state.
- **Integration test gap:** There are 21+ unit test files in `src/lib/`, but no integration tests that simulate the full lifecycle (init -> discuss -> plan -> execute -> review -> merge). The only integration test is `scaffold.integration.test.cjs`. This means cross-cutting features like hooks or memory, which affect multiple lifecycle stages, cannot be tested end-to-end.
- **Test runner has no `npm test` script:** package.json has zero scripts. Tests are run ad-hoc. Previous oversights research (v3.3.0) identified this same gap. It has not been addressed.

### 9. Configuration Fragmentation

The project has configuration in multiple locations that are not synchronized:

- `config.json` (project root): `lock_timeout_ms`, `agent_size_warn_kb`
- `.planning/config.json`: `project.name`, `project.version`, `model`, `planning.max_parallel_sets`
- `.claude-plugin/plugin.json`: `name`, `version`, `description`
- `package.json`: `name`, `version`, `description`
- `.planning/STATE.json`: `projectName`, `rapidVersion`

**What gets overlooked:**
- **Version is stored in 3 places:** `.claude-plugin/plugin.json` has version `3.3.0`, `.planning/config.json` has `3.0.0`, and `package.json` has `3.3.0`. The `.planning/config.json` version is already stale.
- **New features need new config:** Hooks need a config entry (enabled/disabled, which hooks to run). Memory needs a config entry (storage location, max size). UI specs need a config entry (whether UI contracts are enabled for this project type). Without a clear configuration strategy, each feature adds its own config in its own location.
- **Config reads are scattered:** Library modules read `config.json` (project root) for lock timeouts. Skills read `.planning/config.json` for project settings. There is no unified config reader. The `core.cjs:findProjectRoot()` function finds the root, but there is no `loadConfig()` that merges all config sources.

### 10. Richer Planning Context Needs a Consumption Path, Not Just Generation

Item 6 asks for "richer planning/context/state" but focuses on generation. The overlooked question is: which agents read this richer context, and through what mechanism?

**What gets overlooked:**
- **Context files are loaded explicitly, not discovered:** The `prepareSetContext()` function in `execute.cjs` loads exactly three things: scopedMd, definition, contractStr. It does not scan `.planning/context/` for additional files. Adding new context files (ARCHITECTURE.md, CONVENTIONS.md, etc.) has zero effect unless the prompt assembly code is updated to include them.
- **Agent role modules are static:** Agent prompts are built from `src/modules/roles/role-*.md` files at build time (`build-agents`). Richer context cannot be injected at build time because it depends on the project being analyzed. It must be injected at runtime, but the runtime injection path (`execute prepare-context`) has a fixed output format.
- **Context is set-scoped but quality is project-scoped:** Code quality context (e.g., "use 2-space indentation, always add tests") applies to all sets. But context is injected per-set via CONTEXT.md and CONTRACT.json. Project-wide context either goes in CLAUDE.md (which is already long) or needs a new injection path.
- **Planners vs executors need different context richness:** The planner needs architectural context (how modules relate, what patterns to follow). The executor needs implementation context (existing function signatures, test patterns, style conventions). Sending both to both wastes context budget. The richer context system should be role-aware.

### 11. Memory During /new-version: Conversation State Loss

Item 5 specifically mentions "better memory on what the user requests during new-version/when the user changes their mind." The fundamental problem is that /new-version is a single Claude conversation. User decisions within that conversation (Step 2-3 answers) exist only in the conversation context.

**What gets overlooked:**
- **Mid-conversation changes are not tracked:** If the user says "Actually, I want to add a CLI tool instead of a web app" after the roadmapper has already generated sets, the original request and the correction are both in the conversation. But only the final roadmap is persisted. The change of mind is lost.
- **No decision log artifact:** The /discuss-set skill produces CONTEXT.md with a `<decisions>` section capturing user choices. The /new-version skill produces no equivalent. User answers to Step 2 (milestone goals) and Step 3 (carry-forward decisions) are consumed but never written to an artifact.
- **Cross-conversation memory:** When the user runs /new-version to create v3.5.0, they cannot reference decisions made during v3.4.0 creation. The v3.4.0 research, synthesis, and roadmap are archived, but the user's *reasoning* behind those choices is not. A `DECISIONS.md` or `MILESTONE-NOTES.md` artifact would bridge this gap.
- **Session-to-session continuity:** Claude Code does not persist conversation history between sessions. If the user runs /new-version, is interrupted, and returns in a new session, all context is lost. The `/pause` and `/resume` skills exist for set execution but not for /new-version. The new-version flow has no checkpoint/resume mechanism.

### 12. Interaction Between add-set and In-Progress Execution

Item 1 (add-set) allows users to inject a set mid-milestone. But the add-set skill has no guardrails for what happens when other sets are currently executing.

**What gets overlooked:**
- **Worktree state conflicts:** If set-01 is in `executing` status with an active worktree, adding set-02 at this point is fine (it creates a separate worktree). But if the user wants set-02 to depend on set-01's output, set-01 is not yet merged to main. The new set's worktree branches from main, which lacks set-01's changes. The dependency is declared but unsatisfiable until set-01 merges.
- **Merge ordering after add-set:** The merge pipeline processes sets in DAG order. If add-set creates a set without edges, it defaults to Wave 1 (no dependencies). But if it should logically come after existing sets, the absence of edges means it could be merged first. The user's textual description of dependencies (in DEFINITION.md) is not machine-readable.
- **Status display becomes confusing:** The `/status` command shows sets in the current milestone. A set added mid-execution appears alongside sets at various lifecycle stages. Users may be confused about why some sets are in "executing" while a new set is in "pending." The status display needs to convey the temporal ordering (original roadmap sets vs ad-hoc additions).

### 13. /quick Tasks on Non-Main Branches

The /quick SKILL.md says "in-place execution on the current branch." But if the user runs /quick while on a worktree branch (e.g., `rapid/set-01`), the quick task modifies the set's branch, not main.

**What gets overlooked:**
- **Quick task commits on wrong branch:** If the user is in a worktree directory and runs /quick, commits land on `rapid/{set-name}`. When the set is merged, the quick task changes come along with the set changes, muddling the set's commit history.
- **No branch check:** The /quick SKILL.md does not check which branch is active. It should verify the user is on main (or at least warn if they are not). The `worktree.cjs:detectMainBranch()` function exists but is not called by /quick.
- **.planning/quick/ in worktree vs project root:** Worktrees share the same `.planning/` directory (they are separate git worktrees of the same repo, and `.planning/` is tracked). Quick task artifacts committed in a worktree branch appear in that branch but not in main until merge. If two sets run /quick simultaneously, they could create conflicting entries in `.planning/quick/`.

## Recommendations

Ordered by "should be in the roadmap early" priority:

1. **Memory persistence architecture:** Define a `.planning/memory/` directory with a standard format (JSONL for structured events, Markdown for human-readable summaries). Create a `memory.cjs` library module with `append(event)`, `query(filter)`, and `summarize(timeRange)` functions. All three memory features (user requests, /quick logs, richer context) should write through this module. Integrate memory reads into `execute.cjs:prepareSetContext()` so agents automatically receive relevant memory. **Concrete first step:** Create `src/lib/memory.cjs` with JSONL append/read and a `DEFAULT_MEMORY_DIR = '.planning/memory'` constant, before any feature writes its own storage.

2. **Hook implementation strategy decision:** Decide between (a) Claude Code TaskCompleted hooks (shell scripts, fire-and-forget, limited to post-agent events), (b) RAPID SKILL.md-level post-checks (inline state validation after RAPID:RETURN parsing, full access to state machine), or (c) a hybrid (SKILL.md checks for critical paths, hooks for optional telemetry). Recommendation: (b) for state update enforcement (add a "Step N: Validate State Updated" to execute-set, plan-set, discuss-set SKILL.md files), and (a) only for optional logging. **Concrete first step:** Add a "Step: Post-Execution State Validation" to `skills/execute-set/SKILL.md` that checks the set's status after parsing RAPID:RETURN.

3. **Unified config reader:** Create `src/lib/config.cjs` that reads `config.json` (root) and `.planning/config.json` and merges them with a defined precedence. Add a `getConfigValue(key, defaultValue)` API. New features register their config keys in one place. **Concrete first step:** Create `config.cjs` with `loadConfig(cwd)` that returns a merged config object from both files.

4. **/quick logging with structured format:** Change the /quick SKILL.md to write a JSONL append to `.planning/memory/quick-tasks.jsonl` (or whatever the memory module defines) alongside the existing Markdown summary. Add a `rapid-tools.cjs` subcommand `memory list quick` that reads and displays the log. **Concrete first step:** Define the JSONL schema for quick task events.

5. **UI contract design decision:** Decide whether UI specs are (a) an extension to CONTRACT.json (new `ui` top-level key), (b) a separate `UI-CONTRACT.json` file per set, or (c) a project-wide `.planning/UI-SPECS.md`. Recommendation: (c) because UI consistency is cross-cutting, not per-set. Individual sets reference the global UI spec and declare which components they implement. **Concrete first step:** Define the UI-SPECS.md format and create a template.

6. **Context consumption path:** Update `execute.cjs:prepareSetContext()` to accept an optional `contextSources` parameter that lists additional files to include (from `.planning/context/`). Add role-aware filtering so planners receive architectural context and executors receive implementation context. Integrate with the compaction system so additional context respects token budgets. **Concrete first step:** Add a `projectContext` field to the return value of `prepareSetContext()` that reads ARCHITECTURE.md and CONVENTIONS.md from `.planning/context/`.

7. **/new-version decision log:** Add a Step 2.5 to `skills/new-version/SKILL.md` that writes user answers (milestone goals, carry-forward decisions) to `.planning/memory/milestone-decisions.jsonl` before spawning the research pipeline. This captures the "why" alongside the "what." **Concrete first step:** Add a Write tool call to new-version SKILL.md Step 2 output.

8. **add-set DAG recomputation:** After add-set writes the new set to STATE.json, call `node "${RAPID_TOOLS}" plan recompute-dag` (or equivalent) to re-run Kahn's toposort with the new edges. This requires the add-set SKILL.md to also capture dependency edges as structured data in the set's CONTRACT.json `imports.fromSets`, not just as text in DEFINITION.md. **Concrete first step:** Add a Step 4.5 to add-set SKILL.md that asks for structured dependency edges and writes them to CONTRACT.json imports.

9. **npm test script:** Add `"test": "node --test src/lib/*.test.cjs"` to package.json. This has been identified as a gap in two consecutive milestone oversights reports and still has not been addressed. It costs 1 line of JSON.

10. **Documentation staleness tracking:** Create `.planning/context/DOC-MANIFEST.json` listing each doc file, its source-code dependencies, and last-generation timestamp. The documentation generation command compares source file mtimes against doc timestamps to identify stale docs. **Concrete first step:** Write the manifest schema and populate it from the existing 9 docs.

11. **Code quality feedback loop:** After the review pipeline runs for a set, extract top findings into `.planning/memory/review-findings.jsonl`. When the planner runs for subsequent sets, include recent findings in its context. This creates a learning loop where the system improves over time. **Concrete first step:** Add a findings-extraction step to `skills/review/SKILL.md` that writes to the memory module.

12. **/quick branch guard:** Add a branch detection check to Step 0 of the /quick SKILL.md that warns if the user is not on the main branch. Use `git rev-parse --abbrev-ref HEAD` and compare against main/master. If on a worktree branch, display a warning and offer to continue or cancel. **Concrete first step:** Add 5 lines of bash to Step 0 of `/quick` SKILL.md.
