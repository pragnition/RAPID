**Key finding:** The project declares Node 18+ as the minimum (in `setup.sh` line 39) but Node 18 has been EOL for nearly a year. ## Zod 3 to 4 Migration Analysis (Critical)

This is the most significant stack decision for this milestone. Zod 4 was published as the default `npm install zod` target (zod@4.0.0 on npm). The project currently pins `^3.25.76` which will not auto-upgrade to v4.

### Current Zod Usage in the Project

The project uses Zod in 4 source files:
- `src/lib/state-schemas.cjs` -- Defines `ProjectState`, `MilestoneState`, `SetState`, `WaveState`, `JobState` schemas
- `src/lib/returns.cjs` -- Parses RAPID:RETURN markers
- `src/lib/review.cjs` -- Review data validation
- `src/lib/merge.cjs` -- Merge conflict data validation

Key Zod features currently in use:
- `z.object()`, `z.string()`, `z.number()`, `z.array()`, `z.enum()` -- all compatible with v4
- `.passthrough()` on all 5 object schemas -- **deprecated in Zod 4** (use `z.looseObject()` instead)
- `.safeParse()`, `.parse()` -- API unchanged in v4
- `.default()`, `.optional()` -- compatible, but `.default()` behavior changed on optional properties
- `const { z } = require('zod')` -- **CommonJS import has known issues with Zod 4**

### Zod 4 Breaking Changes Affecting This Project

1. **`.passthrough()` deprecated** -- All 5 state schemas use `.passthrough()`. In Zod 4, this is replaced by `z.looseObject()` which allows extra properties to pass through. The old `.passthrough()` still works for backward compatibility but is marked legacy.

2. **CommonJS require path** -- The standard `require('zod')` now returns Zod 4 if you install zod@4.x. However, there have been reported CJS loading issues. The safe migration path for CJS projects is `require('zod/v3')` (pinned v3 subpath) or fully migrating to the v4 API.

3. **`.enum()` overloaded** -- Now supports native TypeScript enums directly. The current `z.enum([...])` array syntax still works.

4. **Bundle size regression for CJS** -- Zod 4 CJS bundles are reportedly 4x larger than v3 CJS bundles. For a CLI tool this is acceptable but notable.

### Migration Decision Matrix

| Option | Effort | Risk | Benefit |
|--------|--------|------|---------|
| **Stay on zod@^3.25.76** | None | Low (v3 receives security fixes via `zod/v3` subpath) | No disruption |
| **Pin to `require('zod/v3')`** | Trivial (4 file changes) | Very low | Future-proofs against accidental v4 upgrade |
| **Migrate to Zod 4** | Moderate (5 schema files, test validation) | Medium (CJS issues reported) | Performance gains, modern API |

**Recommendation:** For this milestone, **stay on zod@^3.25.76** (no changes). The v3.25.x line includes the `zod/v3` subpath for future safety. Zod 4's CJS story is still maturing and the project has no pain points that v4 solves. If the semver range is ever broadened, update the 4 require statements to `require('zod/v3')` as a defensive measure. A full Zod 4 migration can be a dedicated set in a future milestone when CJS support stabilizes.

## proper-lockfile Assessment

| Property | Value |
|----------|-------|
| Version | 4.1.2 |
| Last published | ~5 years ago |
| npm dependents | 856 |
| GitHub stars | ~250 |
| Known CVEs | None |

The project uses proper-lockfile in a single file (`src/lib/lock.cjs`) for file-based distributed locking of `.planning/STATE.json`. The lock mechanism uses `mkdir`-based atomicity which is the correct approach for local filesystem locking (avoids the `O_EXCL` race condition that affects the older `lockfile` package).

**Alternatives evaluated:**
- `async-lock` -- In-memory only, not suitable for cross-process file locking
- `lockfile` (npm) -- Older, uses `O_EXCL` (less reliable on network filesystems), also unmaintained (8 years)
- `async-mutex` -- In-memory only

**Recommendation:** No change needed. proper-lockfile is functionally correct and stable. "Unmaintained" for a lock library that works correctly is not a risk -- the locking semantics are not evolving. The project wraps it in `lock.cjs` which provides a clean abstraction boundary if replacement is ever needed.

## Tooling Assessment

### Build Tools

- **Current:** No build step. Source files are `.cjs` and run directly. Agent `.md` files are assembled by `build-agents` command.
- **Assessment:** This is the right approach for a CLI plugin. No bundler needed.
- **Gap for milestone:** None.

### Test Framework

- **Current:** Node.js built-in `node:test` with `node:assert/strict`. 40 test files, ~44 `require('node:test')` imports. Tests run via `node --test 'src/**/*.test.cjs'`.
- **Assessment:** The built-in test runner is mature and sufficient. Node 22+ adds automatic subtest lifecycle management (no need to manually await subtests), JSON module mocking, and improved watch mode.
- **Gap for milestone:** The `hooks` feature (item 4) will need integration testing that validates hook exit codes and JSON stdout parsing. The built-in test runner supports mocking (`test.mock`) which is used in ~10 test files already. No external test framework needed.
- **Opportunity:** Consider using `--test-reporter spec` in the npm test script for better developer experience during this milestone's development.

### Linting/Formatting

- **Current:** None configured. No ESLint, Prettier, or similar tools.
- **Assessment:** For a 21-module CJS project maintained primarily by AI agents, the lack of a linter is notable but not blocking. The codebase follows consistent conventions documented in CONVENTIONS.md.
- **Gap for milestone:** The "code quality" item (item 6) could benefit from adding ESLint with minimal rules, but this should be a separate concern from the milestone's feature work.

### CI/CD

- **Current:** None detected (no `.github/workflows/`, no `.gitlab-ci.yml`).
- **Assessment:** Tests run locally via `npm test`. For a Claude Code plugin, this is adequate -- the deployment model is `git clone` + `setup.sh`.
- **Gap for milestone:** None for this milestone's scope.

## Stack Implications for Milestone Features

### 1. add-set (already implemented as skill)
- **Stack impact:** None. Uses existing state-machine.cjs and CLI patterns.

### 2. documentation
- **Stack impact:** Minimal. Will read source files and generate markdown. No new dependencies needed.
- **Consideration:** May want a markdown AST library if structured doc generation is needed, but simple string concatenation suffices for the current codebase scale.

### 3. UI specs/contracts
- **Stack impact:** Extends the existing CONTRACT.json (Ajv-validated JSON Schema) pattern. Ajv 8.18.0 supports JSON Schema 2020-12 which is more than sufficient for UI contract definitions.
- **Consideration:** May need to add UI-specific schema properties to the contract validation in `contract.cjs`.

### 4. hooks
- **Stack impact:** The Claude Code plugin hooks API (documented at code.claude.com/docs/en/hooks) supports 17 hook event types. The project currently has one hook (`rapid-task-completed.sh` -- a bash script). The milestone needs to add hooks that verify state updates.
- **Key technical detail:** Hooks can be `command` (shell), `http`, `prompt`, or `agent` type. The `TaskCompleted` and `SubagentStop` events are the most relevant. Exit code 2 blocks the action and feeds stderr back to the model. Hook configuration lives in `hooks/hooks.json` within the plugin.
- **No `hooks/hooks.json` currently exists** in the plugin -- only the shell script. This is a gap: the existing `rapid-task-completed.sh` appears to be a standalone script not wired into the plugin hook system via `hooks.json`.

### 5. memory
- **Stack impact:** Requires persistent storage for user decisions and mid-conversation pivots. Options:
  - Extend STATE.json with a `memory` or `decisions` field (validated by Zod)
  - Separate JSONL file (like the existing teams completion tracking pattern)
  - **Recommendation:** JSONL append-only log is simpler and avoids lock contention on STATE.json for frequent writes.

### 6. code quality (richer planning/context/state)
- **Stack impact:** Extends existing Zod schemas in `state-schemas.cjs`. The `.passthrough()` on all schemas means extra fields can be added without breaking existing data. This is a strength for incremental enrichment.
- **Consideration:** If schemas become significantly more complex, consider splitting `state-schemas.cjs` into multiple files (e.g., `schemas/set.cjs`, `schemas/wave.cjs`).

### 7. /quick logging
- **Stack impact:** Minimal. The `.planning/quick/` directory does not yet exist (created on first use). The SKILL.md already writes `{NEXT_ID}-SUMMARY.md` files. The gap is that these are not aggregated into a queryable log.
- **Recommendation:** Add a JSONL append log at `.planning/quick/HISTORY.jsonl` (consistent with the teams JSONL tracking pattern). No new dependencies needed -- `fs.appendFileSync()` suffices.
