# Oversights Research -- v4.5 Developer Experience II

## Cross-Cutting Concerns

### UAT Failure File Format and Schema

The milestone replaces the current automation+human hybrid UAT with a fully human-driven step-by-step workflow. The current `REVIEW-UAT.md` format (defined in `skills/uat/SKILL.md` and `src/modules/roles/role-uat.md`) is structured around automated scenario results with pass/fail/skipped statuses. The new human-only workflow needs a different data model:

- **Step-by-step human verdicts need a stable schema.** The current REVIEW-UAT.md format assumes automated execution with step-level `pass/fail/skipped` statuses and `evidence` fields like screenshots. A human-only workflow needs: step description, success criteria, human verdict (pass/fail), and failure reason (freeform text from the human). This schema must be defined upfront because bug-fix `--uat` will parse it.
- **The UAT failure file must be machine-parseable.** Bug-fix `--uat` needs to read failures programmatically. The current REVIEW-UAT.md uses Markdown tables which are fragile to parse. Consider embedding a JSON metadata block (like `<!-- SCOPE-META {...} -->` used in REVIEW-SCOPE.md) or switching to a dedicated `UAT-FAILURES.json` file alongside the human-readable Markdown.
- **Review pipeline state tracking (`review mark-stage`) needs updating.** The current `review state` command tracks `scope`, `unit-test`, `bug-hunt`, and `uat` stages. If the UAT workflow changes from agent-driven to human-driven, the stage completion logic in `src/commands/review.cjs` and `src/lib/review.cjs` must still work. The `ReviewIssue` Zod schema (line 39-53 of `review.cjs`) uses `source: z.enum(['lean-review', 'unit-test', 'bug-hunt', 'uat'])` -- this should remain compatible, but the issue content will differ.

### Bug-Fix `--uat` Integration with Existing Review Infrastructure

The bug-fix skill (`skills/bug-fix/SKILL.md`) currently has **no set association** -- it works from any branch in any directory. Adding `--uat` that reads a set-specific UAT failure file introduces a dependency on set resolution:

- **Set resolution is needed.** The `--uat` flag implies a set ID (from the todo: `/bug-fix <set_no> --uat`). The bug-fix skill must now import the set resolution pattern used by review skills (`node "${RAPID_TOOLS}" resolve set "<set-input>"`). This is a design change -- bug-fix goes from "branch-agnostic" to "optionally set-aware."
- **Working directory ambiguity.** Should bug-fix `--uat` operate in the worktree (if the set is pre-merge) or in the project root (if post-merge)? The UAT failure file lives at `.planning/sets/{setId}/REVIEW-UAT.md` or `.planning/post-merge/{setId}/REVIEW-UAT.md`. Bug-fix needs the same dual-path resolution logic that the UAT skill already implements in Step 1.
- **Failure-to-fix mapping.** Each UAT failure describes a human-observed problem, not a code location. The bug-fix investigation flow (Step 2) works well for user-described bugs but may need adjustment when the input is structured failure data rather than freeform text. Consider whether bug-fix `--uat` should iterate through failures one at a time or batch-analyze them.

### Branding Webserver Process Lifecycle

The branding webserver feature replaces static `index.html` file generation with a background Python webserver. This introduces process management concerns that RAPID has not previously dealt with:

- **Process cleanup on exit.** If the Python webserver is spawned in the background, what kills it? Claude Code sessions end without cleanup hooks. The process will be orphaned. RAPID needs a mechanism to track the PID and clean up -- potentially a `.planning/branding/.server-pid` file and a cleanup check at the start of the next branding invocation.
- **Port collision.** If the user runs `/rapid:branding` twice (or in two different projects), two servers will try to bind the same port. The server needs either a configurable port or automatic port selection with the chosen port reported to the user.
- **Python availability.** The existing codebase is Node.js-only. Python is assumed available but not verified. The branding skill should check for Python availability (`python3 --version`) before attempting to spawn the server, with a clear error message if missing.
- **Cross-platform port opening.** The current branding skill already handles `xdg-open` (Linux) and `open` (macOS) for the HTML file. With a webserver, the opened URL changes from `file://...` to `http://localhost:{port}/`. This is actually simpler (no file:// CORS issues) but the detection logic needs updating.
- **Index page as artifact directory.** The `index.html` must now serve as an index of all generated artifacts (SVGs, design directories, etc.). Looking at the existing `branding/` directory structure (which contains `banner.svg`, `logo.svg`, `designs/` with 17 subdirectories), the index page needs to dynamically list these. This is a significant change from the current single-page generation.

### State Compatibility Between UAT Versions

The UAT workflow is changing from agent-driven to human-driven. Projects mid-review will have artifacts in the old format:

- **Migration of existing REVIEW-UAT.md files.** If a project has REVIEW-UAT.md files from the old format, will the new bug-fix `--uat` be able to parse them? The schema change should be backward-compatible or the migration path should be documented.
- **Review pipeline gating.** The current UAT skill enforces ordering: scope must be complete, unit-test must be complete before UAT runs. If UAT becomes human-only and simpler, should these prerequisites change? A human doing step-by-step testing does not strictly need unit tests to have run first.

### New-Version Roadmapper Set Count Passthrough

The `new-version` skill (`skills/new-version/SKILL.md`) spawns the roadmapper agent in Step 7 but **never passes `targetSetCount`**. The `/init` skill (in `skills/init/SKILL.md`) asks the user for granularity preference in Step 4C and passes it to the roadmapper. The new-version skill skips this entirely -- the roadmapper falls back to its default behavior, which tends toward minimal decomposition.

- **The fix is straightforward but easy to forget.** Add an AskUserQuestion in the new-version flow (before Step 7) that mirrors the init flow's granularity question. Pass the result as `targetSetCount` in the roadmapper prompt.
- **Default should be generous.** The todo specifically says "the agent should be more generous with sets." Rather than defaulting to "auto" (which lets the roadmapper minimize), the default should be "6-10" (standard) or even "11-15" (granular) unless the user chooses otherwise.
- **Carry-forward sets affect the count.** If sets are carried forward from the previous milestone, the new set count should be for NEW work only. The roadmapper prompt should clarify that carried-forward sets are pre-existing and do not count toward the target.

### Error Handling Strategy

- **Webserver spawn failures.** The branding webserver could fail for many reasons (port in use, Python missing, permission denied). Each failure mode needs a distinct, actionable error message rather than a generic "server failed to start."
- **UAT human interaction failures.** In the new human-only UAT, the human might abandon the session mid-way (close the terminal, network timeout). The skill should save partial progress so that re-running continues from where the user left off rather than restarting from step 1.
- **Bug-fix `--uat` with empty failure file.** If all UAT steps passed, the failure file may be empty or not exist. Bug-fix `--uat` should handle this gracefully: "No UAT failures found for set '{setId}'. All steps passed."

### Testing Strategy

- **UAT skill tests must change.** The current role test (`src/modules/roles/role-uat.md` tested by structural validation in `role-branding.test.cjs` pattern) validates the presence of sections like "automated" and "browser automation." A human-only workflow removes these concepts. Tests need updating to validate the new step-by-step format.
- **Webserver integration tests.** Testing that a Python webserver starts and serves files requires spawning a process and making HTTP requests. This is a new testing pattern for RAPID -- all existing tests are pure-function unit tests or structural validation. Consider whether this needs an integration test or can be validated structurally.
- **Bug-fix `--uat` needs parser tests.** The new UAT failure file format needs parser unit tests in the same co-located `*.test.cjs` pattern used throughout the codebase. These tests should cover: valid failure file parsing, empty file, malformed file, old-format file (backward compatibility).

### Documentation Needs

- **Updated UAT workflow documentation in `docs/`.** The existing documentation describes the agent-driven UAT flow. With the switch to human-only, the user-facing documentation must explain the new step-by-step workflow clearly.
- **Bug-fix `--uat` usage documentation.** This is a new flag on an existing command. The help text, skill description, and command documentation all need updating.
- **Branding webserver usage.** Users need to know: how to start the server, how to stop it, what port it uses, and how to access the artifact index.

### Developer Experience

- **UAT session continuity.** The human-only UAT should support interruption and resumption. If the user completes 5 of 10 steps and exits, re-running `/rapid:uat` should offer to continue from step 6 rather than restarting. This requires persisting intermediate state.
- **Branding server auto-refresh.** If the user generates new branding artifacts while the server is running, the index page should reflect the changes without requiring a server restart. Python's `http.server` serves files dynamically, so this works by default, but the index.html must be regenerated when artifacts change.
- **Bug-fix `--uat` iteration loop.** After fixing one UAT failure, the user may want to re-run the UAT step to verify. Consider whether bug-fix `--uat` should automatically re-run the specific failed step after applying the fix, or leave re-verification to a manual `/rapid:uat` re-run.

### Configuration Management

- **Webserver port configuration.** The branding webserver port should be configurable via `.planning/config.json` (consistent with how `browserAutomation` is stored there). Default to a sensible port (e.g., 8765) that is unlikely to conflict with common dev servers.
- **Python path configuration.** Some systems have `python3` but not `python`, or vice versa. The branding skill should try both and allow override via config.

### Accessibility

Not applicable for this milestone. RAPID is a CLI/agent tool with no direct UI. The branding webserver serves developer reference pages, not user-facing applications.

### Internationalization

Not applicable. RAPID's output is English-only and targets developer audiences.

### CI/CD Pipeline

Not applicable for this milestone. The changes are to the RAPID plugin itself, which is tested locally via `node:test`. No deployment pipeline changes are needed.

### Logging and Observability

- **UAT step progress logging.** The human-only UAT should log each step's verdict as it happens (to stderr or a progress file), not just at the end. This supports debugging if the session is interrupted.
- **Webserver startup logging.** When the branding webserver starts, log the PID, port, and URL to stderr so the user has the information even if the banner scrolls off screen.

### Data Concerns

- **UAT failure data privacy.** UAT failures may contain descriptions of user-observed behavior that reference project-specific details. The failure file is stored in `.planning/` which is typically gitignored. Ensure the new UAT failure format does not accidentally expose sensitive information if `.planning/` is committed.
- **Branding artifact serving.** The webserver serves files from `.planning/branding/`. Ensure it does not serve parent directories or other `.planning/` contents (use a restricted document root).

## Recommendations

1. **UAT failure file schema**: Define the machine-parseable UAT failure format FIRST, before implementing either the UAT workflow or bug-fix `--uat`. Both features depend on it. Use the embedded JSON metadata pattern (`<!-- UAT-META {...} -->`) already established by REVIEW-SCOPE.md. This prevents the two features from diverging on format expectations.

2. **Webserver PID tracking**: Design the branding webserver with process lifecycle management from day one. Write the PID to `.planning/branding/.server-pid`, check for stale processes on startup, and kill the old server before starting a new one. This pattern is cheap to implement upfront and expensive to retrofit after users encounter orphaned processes.

3. **New-version `targetSetCount`**: Add the granularity question to `skills/new-version/SKILL.md` Step 7 (before spawning the roadmapper). Mirror the pattern from `skills/init/SKILL.md` Step 4C. Default to "6-10" (standard) rather than "auto" to fulfill the "more generous" requirement from the project goals.

4. **UAT session continuity**: Persist step-by-step progress to a `.planning/sets/{setId}/UAT-PROGRESS.json` file after each step verdict. On re-run, offer to resume from the last incomplete step. This is significantly harder to add after the human-only workflow is built because it requires inserting state checkpoints into every step.

5. **Bug-fix `--uat` as set-aware extension**: Model the `--uat` flag as an alternative to Step 1 (Gather Bug Description) in the existing bug-fix flow. Instead of asking the user to describe a bug, read the UAT failure file and present failures for selection. The rest of the flow (investigation, fix, dispatch executor) remains unchanged.

6. **Branding webserver port and Python checks**: Add `brandingServerPort` to `.planning/config.json` schema. Check Python availability with `python3 --version || python --version` before spawning. Report clear errors for each failure mode (port in use, Python not found, permission denied).

7. **Document the branding artifact directory structure**: The `index.html` served by the webserver needs to enumerate artifacts from the `branding/` directory dynamically. Since Python's `http.server` is static, the index must be regenerated when artifacts change. Consider generating the index as part of each branding artifact creation rather than as a separate step.
