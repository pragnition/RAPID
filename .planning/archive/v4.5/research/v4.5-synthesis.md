# Research Summary -- v4.5 Developer Experience II

## Project Profile
- **Type:** brownfield plugin (Claude Code plugin for parallel development orchestration)
- **Stack:** Node.js 18+ (CommonJS), Git 2.30+, Zod 3.25.x state validation, Claude Code v3.0.0 plugin API
- **Scale:** medium (4 features, ~18 sub-tasks, modifications to 6 existing files)
- **Key Constraint:** The UAT failure file format is a shared contract between two features (UAT workflow and bug-fix --uat) and must be defined before either is implemented (sources: FEATURES, ARCHITECTURE, OVERSIGHTS, PITFALLS)

## Unified Findings

### Stack and Infrastructure

- **Node.js 25.8.0 is runtime, Node 18+ is minimum.** All code uses stable APIs; no practical risk from running on Current channel. No `engines` field in package.json to enforce the minimum. (source: STACK)
- **Python 3 is NOT a current dependency.** The branding webserver feature would introduce a soft dependency on `python3` being on the system PATH. Python 3.6+ is sufficient for `http.server`, but the `--directory` flag requires 3.7+. (sources: STACK, PITFALLS, OVERSIGHTS)
- **Strong consensus: use Node.js `http` module instead of Python.** Three research files independently recommend avoiding the Python dependency. Node.js `http.createServer` is ~20 lines, keeps RAPID pure Node.js, and eliminates cross-platform availability concerns. (sources: STACK, PITFALLS, ARCHITECTURE)
- **Dependencies are lean (4 production deps) and current.** `zod@3.25.76` works fine; Zod 4 upgrade is NOT for this milestone (cross-cutting concern touching many modules). `proper-lockfile@4.1.2` is stale (5+ years) but functional. `ajv` and `ajv-formats` are current. (source: STACK)
- **No build step, no CI/CD, no linting/formatting configured.** Tests run locally via `node --test`. All outside scope for this milestone. (source: STACK)
- **Background process management is a new pattern for RAPID.** The codebase currently uses `execSync` for synchronous shell commands. The branding webserver is the first long-lived background process. A clean pattern using `child_process.spawn` with PID tracking needs to be established. (sources: STACK, PITFALLS, OVERSIGHTS)

### Feature Landscape

Four features with a clear dependency graph:

```
Human-Only UAT Workflow (independent, complex)
  \-- Bug-Fix --uat Flag (depends on UAT-FAILURES file, moderate)

Branding Webserver (independent, moderate)

Generous Set Planning in new-version (independent, trivial)
```

**Recommended implementation order** (source: FEATURES):
1. Generous Set Planning -- trivial change, immediate DX improvement
2. Human-Only UAT Workflow -- foundational, blocks bug-fix --uat
3. Bug-Fix --uat Flag -- depends on UAT workflow completion
4. Branding Webserver -- independent, quality-of-life, can parallel with 2-3

**Third-party dependencies:** None required. Python `http.server` is optional and consensus recommends against it in favor of Node.js `http` module. (sources: FEATURES, STACK, PITFALLS)

### Architecture Direction

- **Pattern: extend the existing modular monolith.** No new architectural layers needed. All four features are extensions to existing modules/skills. Previous milestones (v3.2 through v4.4) followed this pattern successfully. (source: ARCHITECTURE)
- **No new directories needed.** All changes fit within existing file locations: `src/modules/roles/`, `skills/`, `src/lib/`. (source: ARCHITECTURE)
- **UAT workflow control inversion.** The current agent-driven UAT (agent classifies steps, executes automated ones, CHECKPOINTs for human ones) becomes skill-driven (agent generates test plan only, SKILL.md drives human step-by-step loop via AskUserQuestion). This is the highest-impact architectural change. (sources: ARCHITECTURE, FEATURES)
- **Bug-fix becomes optionally set-aware.** Currently branch-agnostic; the `--uat` flag introduces set resolution (borrowed from review skill pattern). (sources: ARCHITECTURE, OVERSIGHTS)
- **STATE.json is not structurally modified** by any feature in this milestone. (source: ARCHITECTURE)
- **UAT-FAILURES.md is a new planning artifact** using embedded JSON metadata (`<!-- UAT-META {...} -->`) following the REVIEW-SCOPE.md precedent. Stored at `.planning/sets/{setId}/UAT-FAILURES.md`. (sources: ARCHITECTURE, FEATURES, OVERSIGHTS, UX)

### Cross-Cutting Requirements

**Must address early (high retrofit cost):**
- UAT failure file schema with machine-parseable metadata -- both UAT workflow and bug-fix `--uat` depend on this format. Changing it later means updating two consumers. (sources: OVERSIGHTS, PITFALLS, FEATURES, ARCHITECTURE)
- Webserver process lifecycle design (PID tracking, cleanup-on-startup) -- orphaned processes are hard to debug; must be designed in from the start. (sources: OVERSIGHTS, PITFALLS)
- UAT session continuity (resume from step N) -- adding resume support after the workflow is built requires rethinking state persistence. (source: OVERSIGHTS)
- UAT control flow decision (skill-driven vs agent-driven) -- determines the entire implementation approach for the UAT rewrite. (sources: PITFALLS, ARCHITECTURE, FEATURES)

**Can address later (low retrofit cost):**
- New-version `targetSetCount` passthrough -- simple prompt addition, no library changes. (source: OVERSIGHTS)
- Python availability check -- one-line check. (source: OVERSIGHTS)
- Test updates for changed UAT role -- co-located tests, straightforward to update alongside the role. (source: OVERSIGHTS)
- Documentation updates -- can follow implementation. (source: OVERSIGHTS)
- `engines` field in package.json -- trivial. (source: STACK)
- Cross-reference banners between features (UAT mentions bug-fix, etc.) -- improves discoverability but not blocking. (source: UX)

### User Experience Direction

- **UAT interaction model: guided step-by-step evaluation.** Present ONE step at a time with step number, total count (`[3/12]`), action to perform, and expected outcome. Human marks Pass/Fail/Skip. On failure, brief comment prompt. Summary table at end. This matches the Azure DevOps Test Plans pattern and is the industry standard for manual test execution. (sources: UX, FEATURES, ARCHITECTURE)
- **Terminology: keep "UAT" as the command, describe mode as "guided human testing" in help text.** The term "guided" signals the tool presents steps rather than automating them. (source: UX)
- **Bug-fix `--uat` should auto-discover the failure file** at the conventional path. Multiple failures presented as a selectable list, not auto-fixed in bulk. The flag should be mentioned in the UAT completion banner for discoverability. (sources: UX, FEATURES)
- **Branding server: use "preview server" terminology.** Print URL clearly, auto-open browser, provide stop mechanism. Index page should list all artifacts with clickable links. (sources: UX, FEATURES)
- **Set planning: default to "Standard (6-10)" rather than "auto".** Present the same 4 options as `/rapid:init`. The "Let Claude decide" option should bias toward comprehensiveness. Show proposed count before committing. (sources: UX, FEATURES, OVERSIGHTS)
- **Progressive disclosure:** UAT shows one step at a time. Bug-fix shows failure summary first, details on selection. Set planning shows recommendation first, then offers adjustment. (source: UX)
- **Keep UAT step count focused: 5-15 per set.** Human-only model requires more effort per step. Generate fewer, higher-value steps mapped directly to acceptance criteria. (source: UX)

## Contradictions

### Contradiction: Python vs Node.js for branding webserver

- **FEATURES says:** "Recommended: Option 1 (Python `http.server`). Python is universally available, requires zero installation, and is sufficient for serving static branding assets."
- **STACK says:** "Recommendation #1: Use Node.js `http` module instead of Python. This eliminates the cross-runtime dependency and keeps RAPID as a pure Node.js project. The implementation is ~20 lines of code."
- **PITFALLS says:** "Anti-Pattern: Adding Python as a runtime dependency to a Node.js project. Correct approach: Use Node.js `http.createServer`."
- **ARCHITECTURE says:** "Python 3 dependency is safe because it is pre-installed on macOS and virtually all Linux distributions. The fallback preserves existing behavior."
- **Resolution:** The weight of evidence favors **Node.js `http` module**. Three of four research files recommend it (STACK, PITFALLS explicitly; ARCHITECTURE provides a fallback plan but does not strongly advocate for Python). Using Node.js eliminates the cross-language dependency, avoids platform availability issues, and aligns with RAPID's pure-Node.js stack. The implementation effort is comparable (~20 lines).

### Contradiction: Branding server PID tracking necessity

- **ARCHITECTURE says:** "Branding webserver is ephemeral -- no state persisted. No PID tracking needed since it is a dev-time tool."
- **PITFALLS says:** "Write PID to `.planning/branding/.server.pid` on startup. Check for and kill existing PID on re-run."
- **OVERSIGHTS says:** "Design the branding webserver with process lifecycle management from day one. Write the PID to `.planning/branding/.server-pid`."
- **Resolution:** PID tracking IS needed. ARCHITECTURE's assessment that no tracking is needed underestimates the orphaned process risk. When Claude Code sessions terminate unexpectedly, the server process persists and blocks the port. PID file management is cheap to implement and prevents a frustrating user experience. Follow PITFALLS and OVERSIGHTS recommendation.

### Contradiction: UAT session continuity (resume from step N)

- **OVERSIGHTS says:** "Persist step-by-step progress to `UAT-PROGRESS.json`. On re-run, offer to resume from the last incomplete step. Cost to retrofit: HIGH."
- **FEATURES and ARCHITECTURE say:** No mention of resume capability; the workflow is described as a single-pass sequential flow.
- **Resolution:** Resume capability is a valuable enhancement but may be out of scope for the core feature. **Recommend implementing it** given the high retrofit cost flagged by OVERSIGHTS. If the UAT flow has 10+ steps and the user's session drops at step 7, restarting from step 1 is frustrating. Design the step loop with progress persistence from the start.

## Recommended Approach

This milestone is a focused developer experience improvement with four features that fit cleanly into RAPID's existing modular monolith architecture. No new architectural layers, directories, or dependencies are needed. The key implementation insight is that the UAT failure file format is the critical shared contract: it connects the UAT workflow (producer) with the bug-fix `--uat` flag (consumer), and must be defined before either feature is built. The format should follow the established REVIEW-SCOPE.md pattern -- structured Markdown with an embedded JSON metadata comment block for machine parsing.

The UAT workflow rewrite is the most complex and highest-impact change. The consensus across all research files is to use a skill-driven approach: the `rapid-uat` agent generates a focused test plan (5-15 steps per set, each with action and success criteria), then the SKILL.md orchestrator drives the human through each step via sequential AskUserQuestion calls. This eliminates the fragile CHECKPOINT relay pattern of the current hybrid model. Progress should be persisted after each step verdict to support session resumption, given the high cost of retrofitting this later.

For the branding webserver, the research strongly favors using Node.js `http.createServer` over Python `http.server`. This keeps RAPID as a pure Node.js project, eliminates a cross-language dependency, and avoids platform availability issues. The server needs PID file management (write on start, check-and-kill on re-run) to handle orphaned processes from terminated Claude Code sessions. Port auto-detection (starting at a default, incrementing on conflict) prevents the most common failure mode.

The generous set planning feature is trivially implemented by copying the granularity AskUserQuestion from `/rapid:init` Step 4C into `/rapid:new-version` before the roadmapper spawn. The default should be "Standard (6-10)" rather than "auto" to address the user's complaint about the roadmapper producing too few sets. The bug-fix `--uat` feature is a straightforward extension of the existing bug-fix flow, replacing Step 1 (user describes bug) with reading from the UAT failure file. It should present failures as a selectable list and process them sequentially through the existing investigation and fix pipeline.

### Suggested Set Boundaries

- **Set A: UAT Workflow + Failure Format** -- Rewrite `role-uat.md` and `skills/uat/SKILL.md` for human-only step-by-step flow. Define and implement UAT-FAILURES.md format with embedded metadata. Add progress persistence for session continuity. Rationale: the failure format is tightly coupled to the UAT workflow and should be defined by the same set that produces it.

- **Set B: Bug-Fix --uat Flag** -- Extend `skills/bug-fix/SKILL.md` with `--uat` argument parsing, set resolution, failure file reader, and sequential fix iteration. Rationale: depends on Set A's failure format but touches a different skill file, so it can be planned and executed after Set A merges.

- **Set C: Branding Preview Server** -- Replace branding Steps 6-7 with Node.js HTTP server, implement PID lifecycle management, generate artifact index page, update `role-branding.md`. Rationale: fully independent of Sets A and B; touches entirely different files.

- **Set D: New-Version Granularity Prompt** -- Add `targetSetCount` AskUserQuestion to `skills/new-version/SKILL.md`, pass to roadmapper. Rationale: trivial change to a single file; fully independent of all other sets. Can be done first for immediate benefit.

Note: Sets C and D are fully independent and can execute in parallel with each other and with Set A. Set B depends on Set A. This gives a DAG of: `{A, C, D} -> {B}` with A, C, D all in wave 1 and B in wave 2.
