# Features Research

## Feature Inventory

### Feature: Human-Only UAT Workflow

- **Description:** Replace the current automation+human hybrid UAT with a fully human-driven step-by-step workflow. The UAT agent presents each step with clear success criteria, the human passes or fails each step interactively, and failures are logged to a structured UAT file for downstream consumption (bug-fix --uat).
- **Complexity:** complex
- **Dependencies:** independent (builds on existing UAT infrastructure)

#### Current State Analysis

The existing UAT system (`role-uat.md`, `skills/uat/SKILL.md`) is a hybrid model:
- The UAT agent classifies steps as `[automated]` or `[human]`
- Automated steps execute via browser automation MCP (Playwright or Chrome DevTools)
- Human steps use CHECKPOINT returns to pause for user input
- The SKILL.md orchestrator handles CHECKPOINT relay between agent and user
- Results are written to `REVIEW-UAT.md` and issues logged via `review log-issue`

The desired change removes automated classification entirely -- every step becomes human-verified.

#### Decomposition

1. **Rewrite role-uat.md to remove automation classification** -- Remove the `[automated]`/`[human]` tagging system. All steps become human-verified. Remove browser automation execution logic from the role. The agent becomes a test plan generator and human verification guide only.
2. **Rewrite skills/uat/SKILL.md orchestration** -- Remove Step 4 (browser automation tool selection). Simplify Step 5 to generate human-only test plans. Replace Step 7 execution with a step-by-step human verification loop. Remove Step 7a retry logic (no automated tests to retry).
3. **Step-by-step verification loop in SKILL.md** -- For each test step: present the step with success criteria, use AskUserQuestion with Pass/Fail/Skip options, record result. Continue to next step. No agent spawning during execution phase.
4. **UAT failure file generation** -- Write failures to a structured file (e.g., `.planning/sets/{setId}/UAT-FAILURES.md` or JSON) that can be consumed by `bug-fix --uat`.
5. **Update REVIEW-UAT.md format** -- Adapt the output format to reflect human-only results (remove automated/human type distinction).

#### Implementation Strategies

**Option 1: Minimal role rewrite, SKILL.md handles everything**
- Approach: Keep the UAT agent for test plan generation only. The SKILL.md skill handles the entire step-by-step verification loop using AskUserQuestion calls directly (no agent CHECKPOINT relay). The agent generates the plan, SKILL.md executes the human verification loop.
- Pros: Simpler agent, no CHECKPOINT relay complexity, direct user interaction via AskUserQuestion
- Cons: Longer SKILL.md, more logic in the orchestrator layer
- Libraries/services: None -- uses existing AskUserQuestion and Write tools

**Option 2: Agent-driven with CHECKPOINT per step**
- Approach: The UAT agent presents each step and emits CHECKPOINT. SKILL.md relays the CHECKPOINT to the user via AskUserQuestion, then resumes the agent with the result. Agent aggregates results and produces final output.
- Pros: Agent maintains test context across steps, can adapt later steps based on earlier results
- Cons: CHECKPOINT relay is fragile (each round-trip risks context loss), agent session must be maintained across many pauses
- Libraries/services: None

**Recommended:** Option 1. The UAT agent generates a test plan (one COMPLETE return), then the SKILL.md skill drives the human verification loop directly. This avoids the fragility of multi-CHECKPOINT agent sessions and keeps the interaction model simple.

#### Data Model Impact

- New entities: `UAT-FAILURES.md` or `UAT-FAILURES.json` in `.planning/sets/{setId}/` -- structured list of failed steps with criterion, description, expected outcome, and human notes
- Existing entities modified: `REVIEW-UAT.md` format simplified (remove type column from scenario results)
- Relationships: `UAT-FAILURES` file is consumed by `bug-fix --uat` feature

#### API Surface

- Modified skill: `skills/uat/SKILL.md` -- complete rewrite of Steps 4-7a
- Modified role: `src/modules/roles/role-uat.md` -- remove automation classification, simplify to plan-only
- New artifact: `.planning/sets/{setId}/UAT-FAILURES.md` (or `.json`)
- No CLI command changes needed -- `review log-issue` still works for logging individual failures

### Feature: Bug-Fix `--uat` Flag

- **Description:** Extend `/rapid:bug-fix` to accept a `--uat` argument that reads the UAT failure file and fixes reported issues. Instead of the user manually describing a bug, the failures are read from the structured UAT output.
- **Complexity:** moderate
- **Dependencies:** Human-only UAT workflow (needs UAT-FAILURES file to exist)

#### Decomposition

1. **Argument parsing in SKILL.md** -- Detect `--uat` flag in bug-fix skill invocation. When present, skip Step 1 (user bug description gathering).
2. **UAT failure file reader** -- Read UAT-FAILURES from `.planning/sets/{setId}/` (the set must be resolvable). Parse the structured failure data.
3. **Failure-to-bug-description translation** -- Convert each UAT failure into a bug description format compatible with the existing investigation flow (Step 2). Each failure includes: criterion text, step description, expected vs actual outcome, relevant files.
4. **Multi-bug iteration** -- If multiple UAT failures exist, present them as a list. User selects which to fix (or "fix all sequentially"). For each selected failure, run Steps 2-4 of the existing bug-fix flow.
5. **Set context injection** -- When `--uat` is used, also load the set's CONTEXT.md and DEFINITION.md for richer investigation context.

#### Implementation Strategies

**Option 1: Sequential single-bug flow**
- Approach: Parse UAT failures, present list to user, fix one at a time using existing bug-fix Steps 2-4. After each fix, ask if user wants to fix the next failure.
- Pros: Reuses all existing bug-fix infrastructure, minimal new code
- Cons: Slower for many failures, user must approve each individually
- Libraries/services: None -- reuses existing skill structure

**Option 2: Batch bug description with prioritization**
- Approach: Parse all UAT failures, generate a single unified bug description that groups related failures, then dispatch a single executor agent to fix all related issues in one pass.
- Pros: Faster for related failures, single executor session has full context
- Cons: Higher risk of partial fixes, harder to verify which failures were addressed
- Libraries/services: None

**Recommended:** Option 1. Sequential single-bug flow is safer and reuses existing infrastructure. The user maintains control over which failures to address and can verify each fix individually. For a future iteration, batch mode could be added as an optimization.

#### Data Model Impact

- New entities: None (reads UAT-FAILURES file created by the UAT feature)
- Existing entities modified: None
- Relationships: Reads from `.planning/sets/{setId}/UAT-FAILURES.md` (produced by human-only UAT)

#### API Surface

- Modified skill: `skills/bug-fix/SKILL.md` -- add `--uat` argument parsing, add failure file reader, add set resolution step
- No new CLI commands
- No new library modules needed

### Feature: Branding Webserver

- **Description:** Instead of generating standalone `index.html` files that clutter folders, the branding agent spawns a lightweight Python webserver in the background and serves `index.html` as an index page for all generated artifacts (SVGs, design variants, etc.).
- **Complexity:** moderate
- **Dependencies:** independent

#### Current State Analysis

The branding skill (`skills/branding/SKILL.md`) currently:
- Generates `.planning/branding/BRANDING.md` (authoritative branding guidelines)
- Generates `.planning/branding/index.html` (self-contained visual reference)
- Opens the HTML file via `xdg-open` / `open` in Step 7

The `branding/` directory at project root contains logo SVGs, banner SVGs, and 17 design variant directories, plus an `index.html` that renders all designs as a gallery.

The problem: Each branding generation creates standalone HTML files opened via `file://` protocol. Accumulating HTML files clutters the folder, and `file://` has limitations (no relative path resolution quirks, no hot reload).

#### Decomposition

1. **Python webserver launcher** -- Create a small utility that spawns `python3 -m http.server <port>` in the background, serving a specified directory. Kill previous instances on re-run. Store PID for cleanup.
2. **Index page generator** -- Generate an `index.html` that acts as a directory listing/gallery for all branding artifacts. Auto-discovers SVGs, design variants, and branding guidelines. Replaces individual standalone HTML files.
3. **Modify branding SKILL.md Step 6-7** -- Instead of generating and xdg-opening an HTML file, generate the index.html and spawn the webserver. Open `http://localhost:<port>` in the browser.
4. **Server lifecycle management** -- Store server PID in `.planning/branding/.server-pid`. On re-run, kill existing server before starting a new one. Provide a cleanup command or auto-cleanup on skill exit.
5. **Update role-branding.md** -- Remove the self-contained HTML constraint. The index.html can reference relative paths to SVGs and other assets since it is served via HTTP.

#### Implementation Strategies

**Option 1: Python http.server (simplest)**
- Approach: Use `python3 -m http.server <port> --directory <dir>` as a background process. Write a shell snippet that checks for Python3, kills any existing server on the same port, spawns a new one, stores PID.
- Pros: Zero dependencies (Python is ubiquitous), single command, works on Linux/macOS/WSL
- Cons: No auto-reload, basic directory listing only, single-threaded
- Libraries/services: Python stdlib `http.server` module

**Option 2: Node.js serve or similar**
- Approach: Use `npx serve` or a minimal Node HTTP server to serve the branding directory.
- Pros: Stays in the Node ecosystem, can add hot-reload via livereload
- Cons: Adds npm dependency or npx invocation, RAPID currently has zero dev dependencies
- Libraries/services: `serve` npm package (or custom ~20 lines of Node http)

**Recommended:** Option 1. Python `http.server` is universally available, requires zero installation, and is sufficient for serving static branding assets. The project already requires Node but Python3 is a safe assumption on development machines. The simplicity matches the use case.

#### Data Model Impact

- New entities: `.planning/branding/.server-pid` -- stores PID of running webserver
- Existing entities modified: `.planning/branding/index.html` -- becomes a gallery/index page rather than standalone reference
- Relationships: The index.html references all files in the `branding/` directory tree

#### API Surface

- Modified skill: `skills/branding/SKILL.md` -- Steps 6-7 rewritten for webserver approach
- Modified role: `src/modules/roles/role-branding.md` -- relax self-contained HTML constraint
- No new CLI commands (server management is internal to the branding skill)
- Optional: `skills/branding-stop/SKILL.md` or a `--stop` flag on the branding skill to kill the server

### Feature: Generous Set Planning in new-version

- **Description:** The roadmapper agent currently plans minimal sets when invoked from `/rapid:new-version` (inheriting project settings without a `targetSetCount` parameter). It should be more generous with set decomposition and allow the user to choose how many sets they want, mirroring the granularity question from `/rapid:init`.
- **Complexity:** trivial
- **Dependencies:** independent

#### Current State Analysis

The `/rapid:init` skill (Step 4C) asks the user a granularity question with 4 options (Compact 3-5, Standard 6-10, Granular 11-15, Let Claude decide) and passes `targetSetCount` to the roadmapper. The roadmapper role (`role-roadmapper.md`) already supports this parameter as input #6.

The `/rapid:new-version` skill (Step 7) spawns the roadmapper but does NOT pass a `targetSetCount` parameter. The roadmapper instruction block in new-version says "Decompose the milestone into sets" and "Output SETS ONLY" but provides no granularity guidance. This means the roadmapper defaults to its own judgment, which tends toward minimal decomposition.

The fix is straightforward: add the same granularity question from init's Step 4C into new-version's Step 2 (goal gathering), and pass the result to the roadmapper in Step 7.

#### Decomposition

1. **Add granularity question to new-version Step 2** -- After Step 2C-vi (completeness confirmation) or as a new Step 2D, ask the same AskUserQuestion as init Step 4C with the 4 granularity options.
2. **Pass targetSetCount to roadmapper** -- In Step 7 roadmapper prompt, add a `## Target Set Count` section with the user's selection and guidance text (matching init's format).
3. **Display granularity in summary** -- Show the selected granularity in the Step 9 completion summary.

#### Implementation Strategies

**Option 1: Copy init's granularity question verbatim**
- Approach: Add the exact same AskUserQuestion from init Step 4C into new-version as a new Step 2D (between completeness confirmation and unfinished sets handling). Pass the result to the roadmapper in Step 7.
- Pros: Consistent UX between init and new-version, minimal implementation effort, reuses proven UI pattern
- Cons: One more AskUserQuestion call in the new-version flow
- Libraries/services: None -- AskUserQuestion is built-in

**Recommended:** Option 1. This is the only sensible approach. It is a direct copy of existing, working code from init.

#### Data Model Impact

- New entities: None
- Existing entities modified: None (targetSetCount is a runtime parameter, never persisted)
- Relationships: None

#### API Surface

- Modified skill: `skills/new-version/SKILL.md` -- add Step 2D (granularity question), modify Step 7 (roadmapper prompt)
- No role changes needed (roadmapper already supports targetSetCount)
- No CLI changes
- No library changes

## Feature Dependency Graph

```
Human-Only UAT Workflow (independent)
  +-- Bug-Fix --uat Flag (depends on UAT-FAILURES file from UAT workflow)

Branding Webserver (independent)

Generous Set Planning in new-version (independent)
```

## Implementation Order

1. **Generous Set Planning in new-version**: Rationale -- trivial change (add one question + one prompt section), immediately improves DX for all subsequent milestones. Can be done in under an hour.
2. **Human-Only UAT Workflow**: Rationale -- foundational change that the bug-fix --uat feature depends on. Moderate complexity due to SKILL.md rewrite, but no new libraries or infrastructure needed.
3. **Bug-Fix --uat Flag**: Rationale -- depends on UAT workflow being complete (needs the UAT-FAILURES file format). Moderate complexity but largely reuses existing bug-fix infrastructure.
4. **Branding Webserver**: Rationale -- independent quality-of-life feature. Can be done in parallel with items 2-3. Placed last because it has the least impact on the core development workflow.
