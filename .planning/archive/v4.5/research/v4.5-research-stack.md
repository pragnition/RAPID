## Upgrade Paths

### Zod 3.25.76 -> Zod 4.x
- **Recommended target:** Zod 4.3.6 (latest stable)
- **Breaking changes:**
  - Import path changes: can use `zod/v4` subpath for incremental migration
  - Issue type renames (e.g., `ZodInvalidTypeIssue` -> `$ZodIssueInvalidType`)
  - Defaults now applied inside optional fields (behavioral change)
  - Some error types merged or removed
  - Community codemod available: `zod-v3-to-v4`
- **Benefits:** 14x faster string parsing, 7x faster array parsing, 6.5x faster object parsing, 57% smaller core bundle, built-in `.toJSONSchema()` conversion
- **Migration effort:** Moderate. RAPID uses Zod extensively in state-schemas.cjs, review.cjs, returns.cjs, and other modules. The schema definitions themselves are largely compatible, but error handling code and any direct references to Zod issue types would need updating.
- **Recommendation:** NOT for this milestone. Zod upgrade is a cross-cutting concern that touches many modules and should be its own dedicated set in a future milestone. The current Zod 3.25.76 is fully functional and stable.

### proper-lockfile 4.1.2
- **No upgrade available.** The package is at its final version.
- **Alternative consideration:** If locking issues emerge on newer Node.js versions, alternatives include `lockfile` (npm), `fs-ext` (flock-based), or a custom implementation using `fs.mkdir` atomicity.
- **Migration effort:** Significant if replacing. The locking primitives are used in `lock.cjs` and `state-machine.cjs`.
- **Recommendation:** No action needed for this milestone. Monitor for issues on newer Node versions.

## Stack Risks

1. **Python dependency for branding webserver (Medium):** The branding webserver feature introduces a runtime dependency on `python3` being available on the user's system PATH. Unlike Node.js (which is guaranteed since RAPID runs on Node), Python is not universally present on all development machines. A fallback or clear error message is needed when Python is not found.
   - **Mitigation:** Check for `python3` availability at runtime before attempting to spawn the server. Provide a clear error message suggesting installation if not found. Alternatively, consider using Node.js `http` module (stdlib) instead of Python to avoid the external dependency entirely.

2. **proper-lockfile staleness (Low):** The package has not been updated in 5+ years. While it works correctly today, it is not being maintained against newer Node.js versions. If a breaking change in Node.js internals affects its `fs.open`/`fs.rename` based locking, there would be no upstream fix.
   - **Mitigation:** No action needed now. If issues arise, the project already has `lock.cjs` as an abstraction layer, making replacement straightforward.

3. **Background process lifecycle management (Medium):** The branding webserver feature requires spawning a Python HTTP server in the background and managing its lifecycle (start, serve, eventual cleanup). The codebase currently uses `execSync` for synchronous shell commands but does not have patterns for long-lived background processes.
   - **Mitigation:** Use `child_process.spawn` with `detached: true` and `unref()` for fire-and-forget background servers. Store the PID for later cleanup. This is standard Node.js, no new dependencies needed.

4. **No `engines` field in package.json (Low):** Users on Node < 18 will get cryptic errors from `require('node:test')` failures rather than a clear version mismatch message.
   - **Mitigation:** Add `"engines": { "node": ">=18" }` to package.json. Trivial fix, can be done as part of any set.

## Recommendations

1. **Use Node.js `http` module instead of Python for branding webserver (Priority: high):** Rather than introducing a Python dependency, use Node.js built-in `http` module (or `http.createServer` with `fs.readFileSync`) to serve static files. This eliminates the cross-runtime dependency and keeps RAPID as a pure Node.js project. The implementation is ~20 lines of code.

2. **Add Python fallback detection if Python approach is kept (Priority: high):** If the Python `http.server` approach is preferred for simplicity, add a check at the start of the branding skill: `which python3 || which python`. Provide a clear error message if neither is found.

3. **Do NOT upgrade Zod to v4 in this milestone (Priority: critical -- avoid):** Zod 4 is a major version with breaking changes affecting error types, import paths, and default behavior. The upgrade touches every module that uses Zod schemas. This should be a separate, dedicated milestone effort, not bundled with feature work.

4. **Add `engines` field to package.json (Priority: low):** Add `"engines": { "node": ">=18" }` for better error messages on incompatible Node versions.

5. **Establish background process patterns (Priority: medium):** The branding webserver is the first background process in RAPID. Establish a clean pattern using `child_process.spawn` with detached mode, PID tracking, and cleanup on exit. This pattern may be reused by future features.

6. **Monitor proper-lockfile compatibility (Priority: low):** No action now, but track Node.js release notes for changes to `fs` internals that could affect this package. The `lock.cjs` abstraction layer makes future replacement manageable.
