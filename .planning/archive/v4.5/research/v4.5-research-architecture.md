# Architecture Research -- v4.5 Developer Experience II

## Current Architecture

RAPID follows a 5-layer architecture: Skill -> Agent -> CLI -> Library -> Storage. The system is a Claude Code plugin built on Node.js CommonJS modules, with Zod-validated state, file-based locking, and git worktree isolation for parallel development.

### Strengths Relevant to This Milestone

1. **Clear separation between Skills and Agents.** Skills (SKILL.md) orchestrate user interaction and agent spawning. Agents (.md role definitions) perform specialized work. This separation is well-suited for the UAT workflow redesign -- the skill handles the human interaction loop while the agent handles test plan generation.

2. **Existing review pipeline structure.** The review pipeline (`scoper -> unit-tester -> bug-hunter -> devils-advocate -> judge -> bugfix -> uat`) already has a UAT stage with REVIEW-UAT.md output format. The current UAT agent already classifies steps as `[automated]` vs `[human]`. The architecture supports CHECKPOINT returns for pausing execution for human input.

3. **Modular CLI routing.** The CLI backbone (`rapid-tools.cjs`) dispatches to library modules via a command router. Adding new CLI subcommands (e.g., `bug-fix --uat`) follows established patterns.

4. **Web-client module exists.** `src/lib/web-client.cjs` already provides HTTP helpers for Python web backend integration, with `.env`-based configuration and graceful failure. This provides a foundation for the branding webserver feature.

### Areas Requiring Extension

1. **UAT workflow is agent-driven, not human-driven.** The current `role-uat.md` has the agent classify steps and drive execution, with human steps handled via CHECKPOINT. The milestone requires inverting this: the human drives the workflow step-by-step, with the agent presenting steps and recording results.

2. **Bug-fix skill has no review artifact awareness.** `bug-fix/SKILL.md` explicitly states it does NOT read REVIEW-UNIT.md, REVIEW-BUGS.md, or any review artifacts. Adding `--uat` support requires connecting it to REVIEW-UAT.md (or a new UAT failure file).

3. **Branding generates static HTML files.** The current branding skill writes `index.html` to `.planning/branding/` and opens it via `xdg-open`/`open`. There is no server component. The milestone requires replacing this with a lightweight webserver serving an index of all generated artifacts.

4. **new-version skill lacks granularity prompt.** The `init` skill has a `targetSetCount` prompt with options (compact 3-5, standard 6-10, granular 11-15, auto). The `new-version` skill does NOT pass `targetSetCount` to the roadmapper agent at all. The milestone requires making new-version more generous with set decomposition and giving the user control.

## Recommended Architecture Pattern

- **Pattern:** Extend the existing modular monolith with targeted feature additions. No new architectural layers needed.
- **Rationale:** All four features are extensions to existing modules/skills. The codebase is well-structured with clear module boundaries. Introducing new patterns (e.g., a separate webserver framework) would add unnecessary complexity for what amounts to a Python one-liner.
- **Evidence:** Previous milestones (v3.2, v3.6, v4.0, v4.2, v4.4) have all followed this extension pattern successfully. The 5-layer architecture accommodates new features without structural changes.

## Module Organization

### Module: UAT Skill + Agent (modified)

- **Responsibility:** Present a step-by-step human-driven acceptance test workflow for completed sets.
- **Public interface:** `/rapid:uat <set-id>` skill invocation (unchanged command surface). `REVIEW-UAT.md` output artifact (unchanged format). New: writes a `UAT-FAILURES.md` file containing structured failure records when tests fail.
- **Dependencies:** `review.cjs` (scope loading, issue logging), `REVIEW-SCOPE.md` (input), `state-machine.cjs` (review state tracking).
- **Data ownership:** Owns `REVIEW-UAT.md` and `UAT-FAILURES.md` per set.

### Module: Bug-Fix Skill (modified)

- **Responsibility:** Investigate and fix bugs. Extended to accept `--uat` flag that reads UAT failure records.
- **Public interface:** `/rapid:bug-fix [description]` (existing). New: `/rapid:bug-fix --uat [set-id]` reads `UAT-FAILURES.md` and fixes reported issues.
- **Dependencies:** `rapid-executor` agent (for applying fixes). New: reads `UAT-FAILURES.md` from `.planning/sets/{setId}/` or `.planning/post-merge/{setId}/`.
- **Data ownership:** None (operates on current branch, delegates code changes to executor).

### Module: Branding Skill (modified)

- **Responsibility:** Conduct branding interview and generate artifacts. Extended to serve artifacts via lightweight webserver.
- **Public interface:** `/rapid:branding` skill invocation (unchanged). New behavior: spawns `python3 -m http.server` in background instead of writing standalone HTML and opening via file://.
- **Dependencies:** None beyond existing (Write, AskUserQuestion, Bash).
- **Data ownership:** Owns `.planning/branding/BRANDING.md` and `.planning/branding/index.html` (index page listing all artifacts).

### Module: New-Version Skill (modified)

- **Responsibility:** Create new milestone with research pipeline and roadmap generation. Extended with granularity prompt.
- **Public interface:** `/rapid:new-version [--spec path]` (existing). New behavior: prompts user for set granularity before roadmapper, passes `targetSetCount` to roadmapper agent, defaults to generous decomposition.
- **Dependencies:** All 6 research agents, synthesizer, roadmapper (unchanged).
- **Data ownership:** Owns STATE.json milestone creation, ROADMAP.md content (unchanged).

## Data Flow

### Feature 1: Human-Only UAT Workflow

```
1. /rapid:uat <set-id>
2. SKILL loads REVIEW-SCOPE.md + acceptance criteria (unchanged)
3. SKILL spawns rapid-uat agent for test plan generation (unchanged)
4. Agent returns test plan -- ALL steps marked as [human] (changed)
5. SKILL presents each step to user via AskUserQuestion loop (changed)
   For each step:
   a. Display step description + success criteria
   b. AskUserQuestion: "Pass" | "Fail" | "Skip"
   c. If Fail: AskUserQuestion freeform for failure description
   d. Record result
6. SKILL writes REVIEW-UAT.md (unchanged format)
7. If failures exist: SKILL writes UAT-FAILURES.md (new)
8. SKILL logs issues via review log-issue (unchanged)
```

Key change: The CHECKPOINT-based agent-human handoff loop is replaced by a SKILL-driven sequential AskUserQuestion loop. The agent generates the plan but does NOT execute tests. The human evaluates every step.

### Feature 2: Bug-Fix --uat Flag

```
1. /rapid:bug-fix --uat [set-id]
2. SKILL resolves set-id (new: set resolution logic, borrowed from review skill pattern)
3. SKILL reads UAT-FAILURES.md from set directory (new)
4. SKILL presents failure summary to user (reuse existing Step 3 format)
5. For each failure (or batched):
   a. SKILL investigates relevant files (reuse existing Step 2)
   b. SKILL presents findings + proposed fix (reuse existing Step 3)
   c. User approves -> SKILL dispatches rapid-executor (reuse existing Step 4)
6. SKILL displays results (reuse existing Step 5)
```

### Feature 3: Branding Webserver

```
1. /rapid:branding (interview steps 1-5 unchanged)
2. Step 6: Generate index.html that lists all branding artifacts (changed)
   - index.html serves as an artifact index, not a standalone branding page
   - Lists: BRANDING.md preview, color palette, typography preview, etc.
3. Step 7: Spawn Python webserver instead of xdg-open (changed)
   a. Find available port (default 8765, increment if busy)
   b. python3 -m http.server {port} -d .planning/branding/ &
   c. Open http://localhost:{port}/ in browser
   d. Display: "Branding server running at http://localhost:{port}/ -- press Ctrl+C to stop"
4. Step 8: Commit (unchanged)
```

### Feature 4: Generous Set Planning in New-Version

```
1. /rapid:new-version (Steps 0-6 unchanged)
2. Step 6.5 (new): Granularity prompt before roadmapper
   a. AskUserQuestion with same options as /init:
      "Compact (3-5)" | "Standard (6-10)" | "Granular (11-15)" | "Let Claude decide"
   b. Default highlighted: "Standard (6-10)" (more generous than current implicit minimal)
3. Step 7: Roadmapper spawned with targetSetCount parameter (changed)
   - Roadmapper prompt now includes "## Target Set Count" section
   - Same format as /init's roadmapper invocation
4. Steps 8-9 unchanged
```

### State Management

- **UAT-FAILURES.md** is a new planning artifact written to `.planning/sets/{setId}/UAT-FAILURES.md` (standard) or `.planning/post-merge/{setId}/UAT-FAILURES.md` (post-merge). It is a structured Markdown file, not JSON, matching the review artifact convention.
- **Branding webserver** is ephemeral -- no state persisted. The `python3` process runs in background and terminates when the user exits or kills it. No PID tracking needed since it is a dev-time tool.
- **targetSetCount** remains a runtime parameter (not persisted to config.json), consistent with the `/init` pattern.
- **STATE.json** is not structurally modified by any feature in this milestone.

## API Design

No new CLI commands are added. All changes are flag extensions or behavioral modifications to existing commands:

| Command | Change | Type |
|---------|--------|------|
| `/rapid:uat <set-id>` | Human-only step loop instead of agent execution | Behavioral |
| `/rapid:bug-fix --uat [set-id]` | New `--uat` flag reads UAT failures | Flag addition |
| `/rapid:branding` | Webserver instead of file:// open | Behavioral |
| `/rapid:new-version` | Granularity prompt before roadmapper | Step addition |

Conventions preserved: kebab-case commands, AskUserQuestion for all user interaction, structured returns for agent communication, idempotent file writes.

## Integration Points

| External System | Protocol | Purpose | Error Handling |
|----------------|----------|---------|---------------|
| Python 3 (http.server) | Spawned subprocess | Serve branding artifacts | Check `python3` availability; fall back to `xdg-open` if not found |
| Git | CLI | Branch/worktree operations (unchanged) | Existing error handling in worktree.cjs |
| Claude Code Plugin API | Agent spawning | UAT plan generation, executor dispatch | Existing RAPID:RETURN parsing |
| Browser | HTTP (localhost) | View branding artifacts | Open via xdg-open/open; non-fatal if browser unavailable |

### Python 3 Dependency

The branding webserver introduces a soft dependency on Python 3. This is safe because:
1. Python 3 is pre-installed on macOS and virtually all Linux distributions
2. The fallback (direct file:// open) preserves existing behavior if Python is unavailable
3. No pip packages required -- `http.server` is in the Python standard library

## Scaling Considerations

Not applicable for this milestone. All features are single-user, local-machine operations. No concurrency, network scaling, or performance concerns.

## Architecture Risks

1. **AskUserQuestion call budget in human-only UAT.** A set with many acceptance criteria could generate dozens of test steps, each requiring an AskUserQuestion call. Risk: user fatigue, long sessions.
   - **Impact:** Medium -- poor UX if too many steps.
   - **Mitigation:** Batch related steps into groups. Present a summary table after each scenario rather than after each individual step. Allow "Pass all remaining" shortcut.

2. **UAT-FAILURES.md format stability.** This is a new artifact consumed by the bug-fix skill. If the format changes, the bug-fix parser breaks.
   - **Impact:** Low -- internal artifact, both producer and consumer owned by RAPID.
   - **Mitigation:** Define a clear schema upfront. Use a structured Markdown format with parseable sections (similar to REVIEW-SCOPE.md's `<!-- SCOPE-META {...} -->` pattern).

3. **Port conflicts for branding webserver.** `python3 -m http.server` on a fixed port may conflict with other local services.
   - **Impact:** Low -- dev-time tool, easily recoverable.
   - **Mitigation:** Probe for available port starting at 8765, incrementing up to 5 times. Display the actual port in the output.

4. **Roadmapper may ignore targetSetCount guidance.** The roadmapper agent receives `targetSetCount` as a soft constraint. It may still produce minimal sets if it determines the work does not warrant more.
   - **Impact:** Low -- the user sees the proposed roadmap and can request revision.
   - **Mitigation:** Strengthen the roadmapper prompt to emphasize generous decomposition. Add a note: "When in doubt, prefer more smaller sets over fewer large ones."

## Recommendations

1. **Rewrite UAT skill as human-driven AskUserQuestion loop, remove agent execution phase.** The agent generates the test plan (kept), but execution is replaced by a sequential skill-driven pass/fail prompt per step. This is the highest-impact change.
   - **Rationale:** The current hybrid model (agent executes automated steps, pauses for human on others) creates complexity around CHECKPOINT handling and browser automation configuration. A fully human-driven flow is simpler, more reliable, and matches the milestone goal exactly.
   - **Priority:** Critical

2. **Define UAT-FAILURES.md format with embedded metadata block.** Use `<!-- UAT-FAILURES-META {...} -->` JSON for machine-parseable metadata, followed by structured Markdown sections per failure. Each failure entry should include: step name, criterion, failure description (user-provided), relevant files.
   - **Rationale:** Enables the bug-fix skill to parse failures without fragile regex. Follows the REVIEW-SCOPE.md precedent.
   - **Priority:** Critical

3. **Add --uat flag to bug-fix skill with set resolution.** Reuse the set resolution pattern from the review/uat skills (`node "${RAPID_TOOLS}" resolve set`). Read UAT-FAILURES.md and iterate through failures, presenting each as a bug to investigate and fix.
   - **Rationale:** Straightforward extension of existing bug-fix flow. The investigation and fix steps are identical; only the input source changes (user description vs UAT failure file).
   - **Priority:** High

4. **Replace branding Step 6-7 with Python webserver.** After generating `index.html` as an artifact index, spawn `python3 -m http.server {port} -d .planning/branding/` in background. Fall back to `xdg-open` if Python 3 is not available.
   - **Rationale:** Prevents folder clutter from multiple standalone HTML files. A webserver serves the index cleanly and can be extended later to serve additional artifacts.
   - **Priority:** High

5. **Add granularity prompt to new-version skill between Step 6 (synthesis) and Step 7 (roadmapper).** Copy the prompt format from `/init` Step 4C. Pass `targetSetCount` to the roadmapper in the same format as `/init`. Default to "Standard (6-10)" rather than implicit minimal.
   - **Rationale:** Direct gap: `/init` has this feature, `/new-version` does not. The roadmapper already supports `targetSetCount` as an input parameter. This is a prompt-only change in the skill definition.
   - **Priority:** Medium

6. **Strengthen roadmapper prompt for generous decomposition.** In both `/init` and `/new-version`, add guidance to the roadmapper prompt: "Prefer more, smaller sets that can be executed in parallel. When a set covers multiple distinct concerns, consider splitting it."
   - **Rationale:** Addresses the UX goal of allowing users to choose granularity while also nudging the roadmapper toward generous decomposition by default.
   - **Priority:** Medium

7. **Add "Pass all remaining" shortcut to UAT human loop.** After a configurable number of individual pass/fail prompts (e.g., 5), offer an additional option: "Pass all remaining steps in this scenario." This mitigates the call budget risk for large test plans.
   - **Rationale:** User experience safeguard. Without this, a 20-step test plan requires 20 AskUserQuestion calls, which is tedious.
   - **Priority:** Medium
