# Domain & UX Research -- v4.5.0 Developer Experience II

## Domain Conventions

RAPID is a Claude Code plugin for AI-assisted parallel development orchestration. The domain is **developer workflow tooling** -- comparable to Nx, Turborepo, and Bazel for build orchestration, but focused on AI agent coordination. This milestone targets four UX improvements that span manual testing workflows, dev server patterns, bug-fix flag conventions, and planning granularity controls.

### Standard Terminology and Naming Conventions

- **UAT (User Acceptance Testing)** is well-established terminology in the software industry. However, "human-only UAT" versus "automated UAT" is a meaningful distinction. Competing products use:
  - "Manual test run" (Azure DevOps Test Plans) -- industry-standard term for human-driven step-through testing
  - "Exploratory testing" -- unscripted human testing (not applicable here; RAPID's UAT is structured)
  - "Walkthrough" or "guided testing" -- less formal terms for step-by-step human verification
  - **Recommendation:** Keep "UAT" as the command name but describe the mode as "guided human testing" in help text and banners. The term "guided" signals that the tool presents steps rather than automating them.

- **`--uat` flag** on bug-fix follows the convention of source-specific flags seen in other developer tools:
  - `gh pr create --recoverable` (GitHub CLI uses flags for input sources)
  - `npm audit fix --force` (npm uses flags to modify behavior scope)
  - `jest --testPathPattern` (Jest uses flags to scope input)
  - **Recommendation:** `--uat` is concise and follows RAPID's existing flag conventions (`--post-merge`, `--spec`). Consider also accepting `--uat <set-id>` for disambiguation when multiple UAT failure files exist.

- **Dev server** terminology is universal for lightweight HTTP servers in development contexts:
  - `npm start` / `vite dev` / `next dev` -- all use "dev server" as the concept
  - `python -m http.server` -- the standard one-liner for serving static files
  - **Recommendation:** Use "preview server" rather than "dev server" since the branding server serves generated artifacts for preview, not a development application.

### Common Workflows and User Journeys

**Manual Testing Workflow (Azure DevOps Test Plans pattern):**
1. Present a numbered list of test steps, each with an action and expected result
2. Human marks each step as Pass/Fail/Skip
3. On failure, prompt for a comment explaining why
4. After all steps, summarize results and log failures
5. Generate a failure report file for downstream consumption

Source: [Azure DevOps Test Plans - Run Manual Tests](https://learn.microsoft.com/en-us/azure/devops/test/run-manual-tests?view=azure-devops)

**Dev Server Pattern (Python http.server / Vite):**
1. Spawn server in background on an available port
2. Print the URL to stdout/stderr
3. Auto-open browser (platform-detected)
4. Server keeps running until explicitly stopped or parent process exits
5. Serve index.html as directory listing

Source: [Python http.server documentation](https://docs.python.org/3/library/http.server.html), [Real Python HTTP Server Guide](https://realpython.com/python-http-server/)

**Bug-Fix from Failure File (CI/CD repair pattern):**
1. Read structured failure data from a known file path
2. Present failures as a prioritized list
3. User selects which failures to address (or "all")
4. Tool investigates and applies fixes sequentially
5. Re-run relevant tests to verify

Source: Common pattern in CI/CD tools like GitHub Actions retry mechanisms and TestCollab CLI result uploads.

### Industry Standards

- **Step-by-step manual test execution** is a mature pattern. Azure DevOps, TestRail, and Zephyr all present test steps with Action + Expected Result + Pass/Fail controls. The key convention is that each step has explicit success criteria visible to the human before they evaluate. (Source: [Azure DevOps Test Plans overview](https://learn.microsoft.com/en-us/azure/devops/test/overview?view=azure-devops))
- **Background process management** in CLI tools follows the pattern of printing PID and port, writing a PID file for cleanup, and providing a stop/kill command. (Source: [CLIG.dev](https://clig.dev/))
- **Granularity selection** in planning tools uses a spectrum approach -- Nx uses "affected" scope (implicit), Turborepo uses explicit pipeline definitions, and RAPID already has the 3-tier targetSetCount system from `/rapid:init`.

## Interaction Models

### Primary Interaction Paradigm

RAPID uses a **conversational CLI** paradigm where slash commands trigger multi-step agent pipelines. The four features in this milestone each introduce distinct interaction sub-patterns:

| Feature | Interaction Model | User Role |
|---------|------------------|-----------|
| Human-only UAT | Step-by-step guided evaluation | Active evaluator (pass/fail each step) |
| Bug-fix `--uat` | Flag-driven source selection | Passive reviewer (approve/reject fix) |
| Branding webserver | Background service + browser preview | Observer (view artifacts in browser) |
| Generous set planning | Choice-driven planning control | Decision-maker (select set count) |

### Input/Output Patterns Users Expect

**For human-only UAT:**
- Users expect to see ONE step at a time (not all at once) with clear success criteria
- Each step should show: step number, total steps, action to perform, expected outcome
- After marking pass/fail, the next step appears immediately
- On failure, a brief comment prompt (1-2 sentences) is expected -- not a long form
- At the end, a summary table showing pass/fail counts and a list of failures

**For bug-fix `--uat`:**
- Users expect `--uat` to be optional -- the default behavior (freeform bug description) must remain unchanged
- The flag should read the UAT failure file automatically without requiring the user to specify its path
- If no UAT failure file exists, a clear error message pointing to `/rapid:uat` is expected
- Multiple failures should be presented as a selectable list, not auto-fixed in bulk

**For branding webserver:**
- Users expect the server URL to be printed clearly (e.g., "Preview server running at http://localhost:8421")
- Auto-opening the browser is expected (RAPID already does this for the current branding HTML)
- Users expect a way to stop the server (Ctrl+C in terminal, or a `/rapid:branding stop` subcommand)
- The index page should list all artifacts with clickable links -- not just render one file

**For generous set planning:**
- Users expect to be asked about set count preference DURING new-version, not only during init
- The default should be more generous than the current behavior (bias toward more sets)
- The "Let Claude decide" option should still exist but with a documented bias toward comprehensiveness
- Users should see the proposed count and be able to adjust before proceeding

### Feedback and Progress Indication Conventions

- **UAT step progress:** Show `[3/12]` style progress counters. This is universal in step-by-step CLI tools. (Source: [CLI Guidelines](https://clig.dev/))
- **Server lifecycle:** Print server PID and port on start. Print "Server stopped" on cleanup. This follows Unix daemon conventions.
- **Planning feedback:** Show the proposed set count alongside a brief justification before the user commits. Nx and Turborepo show what will be affected before executing.

### Error Communication Patterns

RAPID already uses `[RAPID ERROR]` prefix consistently. For the new features:

- **UAT with no steps generated:** "No acceptance criteria found. Ensure wave PLAN.md files contain acceptance criteria."
- **`--uat` with no failure file:** "No UAT failure file found for set '{setId}'. Run `/rapid:uat {setId}` first."
- **Server port conflict:** "Port {port} is in use. Trying port {port+1}..." (auto-retry with incrementing ports)
- **Set count outside valid range:** Should not error -- clamp to valid range with a note.

## Information Architecture

### How Information Should Be Organized and Presented

**UAT failure file (new artifact):**
The UAT failure file should be a structured Markdown file (consistent with RAPID's artifact convention) stored at `.planning/sets/{setId}/UAT-FAILURES.md` or equivalent. Key design decisions:

- **Markdown over JSON:** RAPID's human-facing artifacts are Markdown; machine-facing ones are JSON. Since both humans (reading failures) and machines (`--uat` flag parsing) consume this file, use Markdown with a machine-parseable metadata comment block -- the same pattern as REVIEW-SCOPE.md's `<!-- SCOPE-META {...} -->`.
- **Failure granularity:** Each failure should include: step number, step description, expected outcome, actual outcome (human's comment), severity (derived from criterion importance).

**Branding index page:**
The index.html should serve as a directory listing with:
- Project name and branding summary at the top
- A card or list entry for each generated artifact (BRANDING.md rendered, color swatches, terminal mockup)
- Each artifact should be a separate HTML page linked from the index
- This follows the pattern of documentation sites (e.g., Storybook's sidebar navigation)

### Navigation and Discovery Patterns

- **UAT steps:** Linear navigation (step 1 -> step 2 -> ... -> summary). No jumping ahead. Users can go back to re-evaluate a step (change a pass to a fail).
- **Branding server:** Flat navigation (index page -> artifact page -> back to index). No deep hierarchy.
- **Bug-fix from UAT:** Prioritized list navigation. Show highest-severity failures first. User selects which to fix.

### Progressive Disclosure

- **UAT:** Show one step at a time (not all steps upfront). This reduces cognitive load and keeps the human focused on the current evaluation.
- **Set planning:** Show the recommended count first, then offer adjustment. Do not present all options equally -- lead with a recommendation.
- **Bug-fix `--uat`:** Show failure summary first, then details on demand when the user selects a failure to fix.

## Accessibility Considerations

### Key Accessibility Patterns for the Interaction Model

- **UAT step presentation:** Each step should be self-contained with all context needed to evaluate it. The human should not need to scroll back or reference external files to understand what to test.
- **Keyboard navigation:** All AskUserQuestion interactions in RAPID are already keyboard-navigable (Claude Code's built-in UI). No additional work needed.
- **Color usage in branding server:** The index.html and artifact pages should not rely solely on color to convey information. Use labels alongside color swatches.

### Screen Reader and Keyboard Navigation

Not directly applicable -- RAPID operates within Claude Code's terminal UI, which handles accessibility at the platform level. The branding webserver HTML output should include semantic HTML (`<nav>`, `<main>`, `<article>`) and ARIA labels for the artifact listing.

### Inclusive Design Patterns

- **UAT failure comments:** Allow freeform text input, not just predefined options. Human evaluators may need to describe unexpected behaviors that do not fit categories.
- **Server port selection:** Auto-detect available port rather than requiring manual configuration. This prevents "port already in use" errors that frustrate users.

## User Expectations

### Default Behaviors They Expect

1. **UAT should be quick.** Users switching from automated+human hybrid to human-only will expect fewer total steps (since they are doing all the work). The system should generate focused, high-value steps -- not exhaustive checklists. Target 5-15 steps per set, not 30+.

2. **`--uat` should auto-discover the failure file.** Users will not want to specify file paths. The convention in RAPID is that artifacts live at known paths (`.planning/sets/{setId}/...`), so `--uat` should follow this pattern.

3. **Branding server should be ephemeral.** Users expect a dev server to start quickly, serve content, and die when they are done. It should not require manual cleanup or leave orphan processes.

4. **Set count in new-version should default to "generous."** The user complaint is that the current system plans too few sets, causing scope to be missed. The new default should bias toward more sets (e.g., "Standard 6-10" instead of "auto" which may minimize).

### Terminology They Use vs Technical Terms

| User Says | RAPID Technical Term | Notes |
|-----------|---------------------|-------|
| "test it myself" | "human-only UAT" | Users think in terms of manual verification |
| "fix the UAT failures" | `--uat` flag on bug-fix | Users connect UAT output to bug-fix input |
| "show me the branding" | branding preview server | Users want to see, not open files |
| "more sets" / "don't miss things" | generous set decomposition | Users care about coverage, not set count |

### Mental Models from Competing Products

- **Kiro (AWS)** uses spec-driven development with acceptance criteria that map to task lists. Users of spec-driven tools expect criteria to be testable and traceable. RAPID's UAT steps should directly reference acceptance criteria from wave PLANs. (Source: [Kiro spec-driven dev tools comparison](https://dev.to/filiksyos/i-tested-the-top-spec-driven-dev-tools-in-2026-4gdm))
- **Azure DevOps Test Plans** presents manual tests as "test runs" with step-by-step execution. Users familiar with this pattern will expect a similar step-action-result-verdict flow. (Source: [Azure DevOps manual test runner](https://learn.microsoft.com/en-us/azure/devops/test/run-manual-tests?view=azure-devops))
- **Vite/Next.js dev server** starts instantly, shows a URL, and auto-reloads. While auto-reload is not needed for branding artifacts, the "instant start + URL + browser open" pattern is deeply ingrained. (Source: common developer mental model)

### Onboarding and Learning Curve Expectations

- **Human-only UAT:** Users already know `/rapid:uat`. The change is behavioral (less automation, more human input). The command invocation should remain identical -- the UX change is in what happens after invocation.
- **`--uat` flag:** Flags are discoverable via `/rapid:help`. The flag should be mentioned in the UAT completion banner ("To fix failures: `/rapid:bug-fix {setId} --uat`").
- **Branding server:** Replaces the current auto-open-file behavior. Users will notice the change immediately (browser opens to localhost instead of file://). No new commands to learn.
- **Set planning in new-version:** Users already encounter set count selection in `/rapid:init`. Bringing the same UX to `/rapid:new-version` is a consistency improvement, not a new concept.

## Recommendations

Ordered by priority, with rationale:

1. **Redesign UAT as a step-by-step human evaluation flow:** Replace the agent-driven test plan + execution model with a guided walkthrough where the UAT agent generates numbered steps (each with action + success criteria), and the human passes or fails each step via AskUserQuestion. Failures are logged to a structured UAT failure file. -- Priority: **critical** (core feature request; the existing hybrid model is the primary pain point)

2. **Add `--uat` flag to bug-fix:** Extend `/rapid:bug-fix` to accept `--uat [set-id]` that reads the UAT failure file, presents failures as a selectable list, and feeds selected failures into the existing investigation+fix pipeline. Auto-discover the failure file at the conventional path. -- Priority: **critical** (direct dependency on feature 1; completes the UAT-to-fix feedback loop)

3. **Add set count selection to new-version roadmapper flow:** Port the `targetSetCount` AskUserQuestion from `/rapid:init` Step 4C into `/rapid:new-version` Step 7 (before spawning the roadmapper). Change the default from "auto" to "Standard (6-10 sets)" to bias toward generous decomposition. Pass the selection to the roadmapper agent prompt. -- Priority: **high** (addresses the user's complaint about missed scope; low implementation risk since the mechanism already exists in init)

4. **Replace branding HTML auto-open with a Python preview server:** Instead of generating and opening `index.html` via `xdg-open`/`open`, spawn `python3 -m http.server {port} --directory .planning/branding/` in the background. Convert `index.html` from a single-artifact page to an index listing all branding artifacts. Print the server URL and auto-open the browser. Write the PID to `.planning/branding/.server-pid` for cleanup. -- Priority: **high** (improves the branding experience; prevents folder clutter from multiple generated HTML files)

5. **Design UAT failure file format:** Use Markdown with embedded SCOPE-META-style JSON comment for machine parsing. Store at `.planning/sets/{setId}/REVIEW-UAT-FAILURES.md`. Include step number, description, expected result, human comment, and severity. This artifact bridges UAT output and bug-fix `--uat` input. -- Priority: **high** (shared artifact format must be designed before both UAT and bug-fix features are implemented)

6. **Add cross-references in completion banners:** UAT completion banner should mention `--uat` flag on bug-fix. Bug-fix completion should mention re-running UAT. Branding server start should mention how to stop it. These cross-references reduce the learning curve for new workflows. -- Priority: **medium** (improves discoverability but is not blocking)

7. **Implement port auto-detection for branding server:** Try the default port (e.g., 8421), and if occupied, increment until an available port is found. Print the actual port used. This prevents "address already in use" errors when the user re-runs branding without stopping the previous server. -- Priority: **medium** (defensive UX; prevents a common frustration pattern)

8. **Keep UAT step count focused (5-15 per set):** The human-only model requires more human effort per step than the hybrid model. Generate fewer, higher-value steps that map directly to acceptance criteria. Avoid exhaustive checklists. -- Priority: **medium** (UX quality concern; prevents user fatigue in the new workflow)

9. **Add `--stop` subcommand or cleanup hook for branding server:** Provide a way to stop the background server cleanly. Options: `/rapid:branding stop`, a cleanup hook that kills the PID on session end, or simply documenting `kill $(cat .planning/branding/.server-pid)`. -- Priority: **low** (nice to have; the server is lightweight and ephemeral)

10. **Document the "generous planning" rationale in help text:** When the user selects a set count in new-version, briefly explain why more sets improve coverage: "More sets = finer-grained parallel work = less chance of scope being missed by a single agent." -- Priority: **low** (educational; helps users make informed choices)
