# Pitfalls Research -- v4.5 Developer Experience II

## Critical Pitfalls

### Pitfall: Python webserver process orphaning
- **Category:** correctness / data-loss
- **Stack:** Python `http.server`, Node.js `child_process`, Claude Code plugin runtime
- **Description:** The branding webserver feature spawns a background Python process. If the parent Claude Code session terminates unexpectedly (user closes terminal, timeout, crash), the Python process becomes orphaned and continues binding the port. On next invocation, port-in-use errors will block the webserver from starting. Since Claude Code agents have no session-level cleanup hooks, there is no guaranteed teardown path.
- **Trigger:** User closes Claude Code mid-session, agent timeout, or OS signal kills the parent process without propagating to child.
- **Impact:** High -- port remains bound, subsequent `/rapid:branding` invocations fail with `EADDRINUSE`, user must manually find and kill the Python process.
- **Prevention:**
  1. Write the PID to a known file (e.g., `.planning/branding/.server.pid`) on startup.
  2. Before starting a new server, check for and kill any existing PID from the file.
  3. Use a high ephemeral port (e.g., 0 for OS-assigned) rather than a fixed port to reduce collision risk.
  4. Bind to `127.0.0.1` only (not `0.0.0.0`) to avoid security exposure.
  5. Consider adding a `--stop` flag to the branding skill to explicitly kill the server.
- **Detection:** `lsof -i :<port>` or `ss -tlnp | grep <port>` shows a Python process owned by nobody/the user.
- **References:** Python docs on `http.server` module; Node.js `child_process.spawn` detached option.

### Pitfall: UAT human-only workflow loses CHECKPOINT/agent resume semantics
- **Category:** correctness
- **Stack:** RAPID agent protocol (CHECKPOINT returns), Claude Code Agent tool
- **Description:** The current UAT skill uses a multi-phase agent-driven workflow with CHECKPOINT returns for human verification steps. Replacing this with a purely human-driven step-by-step flow fundamentally changes the control flow. If the new implementation still spawns a `rapid-uat` agent that emits CHECKPOINTs, but the skill no longer knows how to resume the agent after each human answer, the workflow will stall or produce incomplete results. Conversely, if CHECKPOINTs are removed entirely, the UAT agent role module and its RAPID:RETURN protocol become inconsistent with the rest of the system.
- **Trigger:** Spawning an agent that expects to be resumed after CHECKPOINT, but the orchestrating skill has been rewritten to not handle agent resumption.
- **Impact:** Critical -- UAT workflow hangs indefinitely or silently drops human verification results.
- **Prevention:**
  1. Decide upfront whether the new UAT workflow still uses a spawned agent or is entirely skill-driven (SKILL.md orchestrates directly with AskUserQuestion, no agent spawn).
  2. If skill-driven: remove or repurpose the `rapid-uat` agent; the skill itself walks the user through steps.
  3. If agent-driven: preserve the CHECKPOINT protocol but ensure the skill's Step 7 handler properly relays human responses back.
  4. Update `agents/rapid-uat.md` and `src/modules/roles/role-uat.md` to match whichever design is chosen.
- **Detection:** Manual testing -- invoke `/rapid:uat` and verify each human step completes and records results.

### Pitfall: Bug-fix `--uat` flag reads stale or non-existent UAT failure file
- **Category:** correctness
- **Stack:** RAPID CLI argument parsing, review pipeline file layout
- **Description:** The new `--uat` flag on `/rapid:bug-fix` reads a UAT failure file to determine what to fix. If the UAT workflow changes its output format or file location (which it will -- the milestone explicitly changes UAT to human-only), the bug-fix skill may read an old-format `REVIEW-UAT.md` or look in the wrong directory. Additionally, if the user runs `/rapid:bug-fix --uat` before running the new UAT workflow, there is no file to read.
- **Trigger:** Running `/rapid:bug-fix 1 --uat` when REVIEW-UAT.md does not exist, or when it was written by the old UAT workflow format.
- **Impact:** High -- bug-fix either crashes with a file-not-found error or misparses the UAT results and proposes incorrect fixes.
- **Prevention:**
  1. Define the UAT failure file format before implementing either feature (UAT rewrite and bug-fix `--uat`). Make this a shared contract.
  2. Add a guard check in bug-fix: if the file does not exist, display a clear error with next steps ("Run /rapid:uat first").
  3. Include a format version marker in the UAT output file (e.g., `<!-- UAT-FORMAT:v2 -->`) so bug-fix can detect incompatible formats.
  4. Ensure the file path resolution logic matches between UAT write and bug-fix read (standard vs. post-merge paths).
- **Detection:** Run `/rapid:bug-fix --uat` without having run UAT first; verify error message quality.

## High-Risk Pitfalls

### Pitfall: Branding webserver `file://` to `http://` transition breaks relative paths
- **Category:** correctness
- **Stack:** Python `http.server`, HTML, browser file access
- **Description:** The current branding flow generates `index.html` opened via `file://` protocol (using `xdg-open`). Moving to an HTTP server changes the security context. Relative paths that worked under `file://` (e.g., linking to `./designs/banner.svg`) may break if the server's document root is not set correctly. Additionally, `file://` access to local SVGs is unrestricted, but HTTP serving from a different directory may fail if the server root does not contain the referenced assets.
- **Trigger:** The `index.html` references files in `branding/designs/` but the Python server is started with a different working directory, or the HTML uses absolute `file://` paths.
- **Impact:** Medium-high -- branding preview page shows broken images/links, degrading the user experience that this feature is meant to improve.
- **Prevention:**
  1. Start the Python server with its working directory set to the branding assets root (e.g., `.planning/branding/` or `branding/`).
  2. Ensure all asset references in generated HTML use relative paths (not absolute filesystem paths).
  3. Make `index.html` an actual index page that links to all generated artifacts with relative URLs.
  4. Test that the server serves both `index.html` and referenced assets (SVGs, sub-pages).
- **Detection:** Open `http://localhost:<port>/` and verify all images and links load.

### Pitfall: Roadmapper set count override conflicts with existing `targetSetCount` parameter
- **Category:** correctness / compatibility
- **Stack:** RAPID roadmapper agent, `new-version` skill
- **Description:** The roadmapper already accepts a `targetSetCount` parameter (documented in `role-roadmapper.md` lines 13-14) with values "3-5", "6-10", "11-15", or "auto". The milestone goal says "allow the user to choose how many sets they wish" and make the roadmapper "more generous." If the implementation adds a new parameter or changes the default without updating the existing `targetSetCount` handling, the two mechanisms will conflict or one will silently override the other.
- **Trigger:** New user-facing set count selection in `new-version` that passes a value the roadmapper does not recognize, or changing the default from "auto" to a specific range without updating the roadmapper's logic.
- **Impact:** Medium-high -- roadmapper produces unexpected number of sets, or ignores the user's preference entirely.
- **Prevention:**
  1. Reuse the existing `targetSetCount` parameter rather than inventing a new one.
  2. In the `new-version` skill (Step 7), pass the user's chosen set count as the `targetSetCount` in the roadmapper prompt.
  3. Change the default from "auto" (which inherits project settings and tends to be minimal) to a more generous default like "6-10".
  4. Add an AskUserQuestion in `new-version` Step 7 or Step 2 to let the user pick from the existing range options.
- **Detection:** Run `/rapid:new-version` and verify the proposed roadmap set count matches the user's selection.

### Pitfall: AskUserQuestion budget exhaustion in new UAT step-by-step flow
- **Category:** performance / correctness
- **Stack:** Claude Code `AskUserQuestion` tool, RAPID skill orchestration
- **Description:** The new human-only UAT workflow requires one `AskUserQuestion` call per test step (pass/fail). If a UAT plan has 15-20 steps, this could hit Claude Code's per-skill tool call limits or context window constraints. The current UAT skill already uses multiple AskUserQuestion calls for plan approval, retry confirmation, etc. Adding per-step human interaction multiplies the count significantly.
- **Trigger:** UAT test plan with more than ~10 steps, each requiring a separate AskUserQuestion call.
- **Impact:** Medium-high -- skill hits internal limits, truncates remaining steps, or the conversation becomes unwieldy for the user.
- **Prevention:**
  1. Batch steps into groups (e.g., present 3-5 steps at once, ask "pass all / fail specific ones").
  2. Allow the user to mark all remaining steps as "pass" with a single action.
  3. Keep the total step count manageable by designing UAT plans with fewer, broader acceptance criteria rather than many micro-steps.
  4. Track remaining steps so the user can see progress (e.g., "Step 3/8").
- **Detection:** Run UAT on a complex set with many acceptance criteria; observe whether all steps complete.

### Pitfall: Python not available on all platforms
- **Category:** compatibility
- **Stack:** Python 3, cross-platform CLI
- **Description:** The branding webserver feature assumes Python is installed and available as `python3` in PATH. On some systems (Windows without Python, minimal Docker containers, certain CI environments), Python may not be available. The RAPID project is a Node.js project with zero Python dependencies today.
- **Trigger:** Running `/rapid:branding` on a machine without Python 3 installed.
- **Impact:** Medium -- branding webserver fails to start; feature is unusable. Fallback behavior must be defined.
- **Prevention:**
  1. Check for Python availability before attempting to spawn the server: `which python3 || which python`.
  2. If Python is not available, fall back to the current `xdg-open`/`open` behavior (direct file opening).
  3. Consider using Node.js built-in `http` module instead of Python, since the project already requires Node.js 18+. This eliminates the external dependency entirely.
  4. If sticking with Python, document the dependency clearly.
- **Detection:** `which python3` returns non-zero exit code.
- **References:** Node.js `http.createServer` as a zero-dependency alternative.

## Medium-Risk Pitfalls

### Pitfall: UAT failure file format ambiguity between old and new workflows
- **Category:** correctness
- **Stack:** RAPID review pipeline, markdown parsing
- **Description:** The current `REVIEW-UAT.md` format (documented in `skills/uat/SKILL.md` Step 8) uses structured markdown with scenario results, criteria coverage tables, and pass/fail/skip metrics. The new human-only UAT will likely produce a different format (step-by-step results rather than scenario-based results). If both formats coexist in the same codebase (e.g., old UAT results from a previous review), downstream consumers like `compaction.cjs` (line 267) and the review summary generator may choke on the format mismatch.
- **Trigger:** Running the new UAT on a set that previously had old-format UAT results, or vice versa.
- **Impact:** Medium -- review summary generation fails or produces garbled output.
- **Prevention:**
  1. Include a format version header in REVIEW-UAT.md (e.g., `<!-- UAT-VERSION: 2 -->`).
  2. Update `compaction.cjs` to handle both formats or at least not crash on unrecognized format.
  3. The new format should be a strict superset or clearly distinguishable from the old one.
- **Detection:** Run review summary after new-format UAT; check for parse errors.

### Pitfall: Branding webserver serves entire `.planning/` directory
- **Category:** security
- **Stack:** Python `http.server`
- **Description:** Python's `http.server` serves all files in its document root directory. If the server is started from `.planning/branding/` and the index links to `../`, it could serve `STATE.json`, contracts, and other planning artifacts. If started from the project root, the entire codebase including `.env` files becomes accessible over HTTP.
- **Trigger:** Starting the Python server from the wrong directory, or the index.html containing `../` links.
- **Impact:** Medium -- sensitive project data exposed on localhost. Low blast radius (localhost only) but still a bad practice.
- **Prevention:**
  1. Always start the server from a specific, restricted directory containing only branding artifacts.
  2. Bind to `127.0.0.1` only (never `0.0.0.0`).
  3. Use a custom request handler that rejects path traversal (`../`) attempts.
  4. Better yet, use Node.js `http` module with explicit file allowlisting.
- **Detection:** Attempt to access `http://localhost:<port>/../STATE.json` and verify it returns 403/404.

### Pitfall: Concurrent branding webserver instances on same port
- **Category:** correctness
- **Stack:** Python `http.server`, TCP port binding
- **Description:** If a user runs `/rapid:branding` twice (e.g., to update styles), and the first server is still running, the second invocation will fail with `EADDRINUSE` unless the skill properly detects and manages the existing server.
- **Trigger:** Running `/rapid:branding` when a previous branding server is still running.
- **Impact:** Medium -- second invocation fails with a confusing port error.
- **Prevention:**
  1. Store PID and port in `.planning/branding/.server.pid`.
  2. On startup, check if the PID file exists and if the process is still alive.
  3. If alive, either reuse the existing server (just open the browser) or kill-and-restart.
  4. Use port 0 for OS-assigned ports to avoid fixed-port conflicts.
- **Detection:** Run `/rapid:branding` twice in quick succession.

### Pitfall: Roadmapper "generous" set planning produces too many sets for solo developers
- **Category:** correctness
- **Stack:** RAPID roadmapper agent, set lifecycle
- **Description:** Making the roadmapper "more generous" with sets could overcorrect -- producing 10-15 sets for a milestone that a solo developer will execute sequentially. Each set carries overhead (start-set, discuss-set, plan-set, execute-set, review, merge = 6 lifecycle steps minimum). For a solo dev, 12 sets means 72+ lifecycle invocations, which is exhausting.
- **Trigger:** Default "generous" set count applied to a small project or solo developer workflow.
- **Impact:** Medium -- user abandons the tool due to excessive ceremony, or merges become a bottleneck as 12 branches need sequential merging.
- **Prevention:**
  1. Present the set count options with clear descriptions of tradeoffs (e.g., "3-5 sets (compact, less overhead)" vs "6-10 (balanced)" vs "11-15 (fine-grained, more parallelizable)").
  2. Make the user's choice explicit rather than changing the default silently.
  3. Consider the project's actual team size when presenting recommendations.
  4. The roadmapper should justify its set count in the proposal so the user can make an informed decision during the approval step.
- **Detection:** Run `/rapid:new-version` for a small milestone and check if the proposed set count is reasonable.

## Anti-Patterns for This Stack

### Anti-Pattern: Spawning background processes without lifecycle management
- **What people do:** Use `child_process.spawn` with `detached: true` and forget about the process.
- **Why it fails:** Orphaned processes accumulate, bind ports, consume resources. No cleanup on parent exit.
- **Correct approach:** Write PID to a known file, check on startup, implement explicit stop mechanism. Consider `unref()` only after PID tracking is in place.
- **Example:** Write PID to `.planning/branding/.server.pid`; on next run, read PID, check `kill(pid, 0)`, kill if alive, then start new server.

### Anti-Pattern: Mixing automation-level and human-level abstractions in UAT
- **What people do:** Build a UAT workflow that tries to be both automated (agent-driven) and manual (human-driven), resulting in a confusing hybrid where neither the agent nor the human has clear ownership of each step.
- **Why it fails:** The agent generates test plans expecting to execute them, but the human is supposed to verify. Neither party has full context, steps get skipped or duplicated.
- **Correct approach:** Pick one paradigm: either the skill drives the human through steps (presenting criteria, collecting pass/fail), or the agent executes automated checks. The v4.5 goal is explicitly human-only. Implement it as a skill-driven flow where the SKILL.md orchestrates with AskUserQuestion, not an agent with CHECKPOINT returns.

### Anti-Pattern: Adding Python as a runtime dependency to a Node.js project
- **What people do:** Use Python for a "quick" webserver because `python3 -m http.server` is a one-liner.
- **Why it fails:** Introduces a cross-language dependency, breaks on systems without Python, complicates installation and documentation.
- **Correct approach:** Use Node.js `http.createServer` or a simple static file server in ~20 lines of JavaScript. The project already requires Node.js 18+.
- **Example:**
```javascript
const http = require('http');
const fs = require('fs');
const path = require('path');
const server = http.createServer((req, res) => {
  const filePath = path.join(docRoot, req.url === '/' ? 'index.html' : req.url);
  // serve file with appropriate content-type
});
server.listen(0, '127.0.0.1', () => { /* log port */ });
```

### Anti-Pattern: Changing default behavior without a migration path
- **What people do:** Change the roadmapper's default set count from "auto" to "generous" and deploy, breaking expectations for existing users who relied on the old default.
- **Why it fails:** Users who were happy with "auto" suddenly get 10+ sets when they expected 3-5.
- **Correct approach:** Add an explicit user choice (AskUserQuestion) in the `new-version` flow. The default remains "auto" but the user is prompted to choose. The "generous" option is opt-in.

## Version-Specific Gotchas

- **Node.js 18+**: `http.createServer` works fine for static file serving. No need for `http2` for this use case.
- **Python 3.x `http.server`**: The `--directory` flag (e.g., `python3 -m http.server 8080 --directory ./branding`) was added in Python 3.7. Do not assume Python 3.6 compatibility.
- **Claude Code Agent tool**: Spawning more than ~6 agents in parallel can hit platform rate limits. The new-version skill already spawns 6 researchers; adding any more in the same flow risks failures.
- **proper-lockfile 4.1.2**: No version-specific gotchas for v4.5 features, but lock contention could increase if the UAT flow makes many rapid `review log-issue` calls.

## Security Pitfalls

1. **Localhost webserver exposure**: Binding to `0.0.0.0` instead of `127.0.0.1` exposes branding assets (and potentially `.planning/` contents) to the local network -- always bind to `127.0.0.1`.
2. **Path traversal in webserver**: Python's `http.server` is vulnerable to path traversal on some platforms. Use `--directory` flag to restrict the document root, or use a custom handler that normalizes paths.
3. **PID file race condition**: If two branding invocations race to read/write the PID file, both may start servers. Use file locking (the project already has `proper-lockfile`) for the PID file.

## Performance Traps

1. **AskUserQuestion overhead in step-by-step UAT**: Each AskUserQuestion call involves a round-trip with the user. A 20-step UAT plan becomes tedious. Batch steps or allow bulk pass to keep the flow under 2-3 minutes.
2. **Webserver startup latency**: Python's `http.server` takes ~500ms to start. If the skill waits synchronously for the server to be ready before opening the browser, this adds perceived latency. Spawn detached and open the browser after a brief delay, or poll for port availability.
3. **Roadmapper re-invocation on revision**: If the user rejects the roadmap proposal and requests revision, the entire roadmapper agent is re-spawned with full context. For large milestones with extensive research synthesis, this can hit context window limits. Keep the revision prompt concise.

## Summary Priority List

1. **UAT CHECKPOINT/agent resume semantics mismatch** -- Rewriting UAT to human-only without updating the agent protocol will cause the workflow to hang or silently drop results. Design decision (skill-driven vs. agent-driven) must be made first.
2. **Python webserver process orphaning** -- Background Python processes with no lifecycle management will accumulate and block ports. Use PID file management or switch to Node.js `http` module.
3. **Bug-fix `--uat` reads wrong file format** -- UAT format changes in tandem with bug-fix must be coordinated. Define the shared contract first.
4. **Roadmapper set count override conflicts** -- Reuse existing `targetSetCount` parameter; do not introduce a parallel mechanism.
5. **Python dependency in a Node.js project** -- Prefer Node.js `http.createServer` to avoid cross-language dependency.
6. **AskUserQuestion budget in step-by-step UAT** -- Batch steps to avoid exhausting tool call limits or user patience.
7. **Branding webserver serves sensitive files** -- Restrict document root, bind to localhost only, reject path traversal.
8. **Solo developer overhead from "generous" sets** -- Present set count as an explicit user choice with clear tradeoff descriptions.
