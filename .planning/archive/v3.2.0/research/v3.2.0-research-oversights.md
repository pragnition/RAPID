# Oversights Research -- v3.2.0 General Fixes

## Cross-Cutting Concerns

### 1. State Terminology Mismatch: Present-Tense vs Past-Tense Status Names

**Why this matters for this project specifically:** The milestone brief lists "discussing -> complete" transition errors and invalid state transitions as explicit bugs. Investigation reveals the root cause is deeper than individual transition fixes: there is a systematic mismatch between the status names used in skills (present-tense) and the status names defined in the state machine (past-tense).

**Current state machine (state-transitions.cjs line 3-10):**
```
SET_TRANSITIONS = {
  pending:     ['discussed', 'planned'],
  discussed:   ['planned'],
  planned:     ['executed'],
  executed:    ['complete'],
  complete:    ['merged'],
  merged:      [],
}
```

**SetStatus Zod enum (state-schemas.cjs line 4):**
```
['pending', 'discussed', 'planned', 'executed', 'complete', 'merged']
```

**Skills that use WRONG status names:**
| Skill | Transition Attempted | Valid Status Would Be |
|-------|---------------------|----------------------|
| discuss-set (SKILL.md:268) | `state transition set ... discussing` | `discussed` |
| execute-set (SKILL.md:148) | `state transition set ... executing` | `executed` |
| execute-set (SKILL.md:332) | `state transition set ... complete` | `complete` (correct) |
| review (SKILL.md:80) | `state transition set ... reviewing` | No such status exists |
| plan-set (SKILL.md:297) | `state transition set ... planning` | `planned` |

The discuss-set skill masks the failure with `2>/dev/null || true` (line 268), which means the set NEVER actually transitions from `pending` to `discussed`. It stays `pending`, then plan-set expects status `discussing` (which also does not exist). The entire lifecycle chain is broken for any set that goes through the interactive discuss path.

**Hidden impact:** Because these status names fail Zod validation before they even reach the transition map, the Zod error is thrown at `ProjectState.parse()` inside `withStateTransaction()`. The entire transaction is rolled back, but the skill continues because errors are suppressed. This means STATE.json may be inconsistent with actual progress.

**What to fix:** Either (a) change all skills to use past-tense statuses matching the Zod enum and transition map, or (b) add present-tense statuses to the enum and transition map. Option (a) is lower risk since it only changes skill documentation, not the state machine.

### 2. Set Name Resolution via Filesystem vs STATE.json Divergence

**Why this matters:** The milestone brief's #1 goal is the set name resolution bug: "archiving sets 4,5 then adding 6,7,8,9 makes set 6 resolve to set 8."

**Root cause analysis:** `resolveSet()` in resolve.cjs (line 19-41) calls `plan.listSets(cwd)` which reads `.planning/sets/` directory entries alphabetically. Numeric indices (e.g., "set 6") are resolved as 1-based indices into this directory listing. But when sets are archived (their directories removed from `.planning/sets/`), the remaining directories re-compact: what was set 6 is now at index 4 if sets 4 and 5 were removed.

**The double-source-of-truth problem:** Set definitions live on disk in `.planning/sets/{name}/` while set state lives in STATE.json under `milestones[].sets[]`. After archival:
- STATE.json still contains entries for all sets (including archived ones, possibly with status `merged`)
- `.planning/sets/` only contains directories for non-archived sets
- The numeric index maps to the filesystem, not to STATE.json

**Overlooked edge cases for any fix:**
1. If the fix switches to STATE.json-based resolution, the order may differ from alphabetical disk order (STATE.json preserves insertion order)
2. If some sets are carried forward across milestones via `addMilestone()` (state-machine.cjs line 282), they may appear in both the old and new milestone's set arrays
3. The `resolveWave()` function (resolve.cjs line 74) also uses `plan.listSets()` for its set resolution (line 166), so any fix must cascade there
4. The `execute.cjs` module also uses `plan.listSets()` for bleed checking (line 154)

**Recommendation:** Resolution should use STATE.json as the source of truth for current milestone sets, not the filesystem. The numeric index should map to the current milestone's `sets[]` array order. `plan.listSets()` should remain for plan-time operations, but `resolveSet()` should optionally accept STATE.json data for resolution.

### 4. Review State Machine Gap

**Why this matters:** The review skill (SKILL.md:69-80) expects a `reviewing` status in the set lifecycle, but this status does not exist in the Zod schema (`SetStatus`) or the transition map (`SET_TRANSITIONS`). The skill attempts `state transition set <milestone> <set-id> reviewing` which will fail.

**Current lifecycle:** `pending -> discussed -> planned -> executed -> complete -> merged`

**Expected lifecycle with review:** `pending -> discussed -> planned -> executed -> complete -> reviewed -> merged` (or similar)

**Overlooked implications:**
1. The review skill currently cannot track its own state -- there is no way to know if a set has been reviewed vs just completed
2. "Review after merge" (milestone brief #6) would need yet another status or a different tracking mechanism
3. The `complete -> merged` transition in the merge skill would need to be updated if a `reviewed` status is inserted
4. If `reviewing` is added as a status, the transition map needs both `complete -> reviewing` (pre-merge review) and potentially `merged -> reviewing` (post-merge review)
5. Adding a status to the Zod enum is a breaking change for any STATE.json that has already been written with the current schema

**Options:**
- **Add `reviewed` status between `complete` and `merged`:** Clean but requires all users to review before merging. Could make `complete -> merged` AND `complete -> reviewed -> merged` both valid paths.
- **Track review state separately:** Use a `reviewed: boolean` flag on the set object rather than a lifecycle status. Avoids breaking the existing transition chain.
- **For "review after merge":** This cannot use the set lifecycle at all since `merged` is terminal. Would need a separate tracking mechanism (e.g., a `reviewStatus` field, or a review artifact existence check).

### 5. AskUserQuestion Batching and "Claude Decides" Pattern

**Why this matters:** The milestone brief goals #2 and #4 both address UX issues with how questions are presented to users. These are cross-cutting because multiple skills use AskUserQuestion with varying patterns.

**Current inconsistencies found across skills:**

1. **discuss-set (SKILL.md:156-167):** Uses AskUserQuestion with structured options for gray area selection. Includes "Let Claude decide all" as option 1. Gray area questions (Step 6, line 179-199) are described as "freeform AskUserQuestion" with 2-3 questions in the body text, but NO pre-filled options -- the user must type their answer.

2. **review (SKILL.md:89-98):** Uses AskUserQuestion with proper structured options for stage selection.

3. **merge (SKILL.md:82-87):** Uses AskUserQuestion with structured options for merge confirmation.

**The gap:** The discuss-set Step 6 deep-dive questions (line 179-199) describe asking 2-3 questions per gray area as a single text block with a trailing "Answer all questions above, or type 'Claude decides'." This is NOT using AskUserQuestion's option batching -- it is a free-text prompt. The user has to type "Claude decides" as text rather than selecting a pre-filled option.

**Cross-cutting fix needed:**
- Every place that says "type 'Claude decides'" should instead be a pre-filled option in AskUserQuestion
- The discuss-set deep-dive should present per-question options where possible, or at minimum include "I'll answer in my own words" and "Let Claude decide this area" as explicit options
- This pattern should be documented in a shared convention so new skills follow the same approach

### 6. State Update Enforcement After Lifecycle Operations

**Why this matters:** The milestone brief goal #7 says "Agents MUST update state after merging/executing/discussing." This reveals a systemic gap: skills do not have a verification step to confirm state transitions succeeded.

**Current pattern in skills:**
```bash
node "${RAPID_TOOLS}" state transition set ... <status> 2>/dev/null || true
```

The `2>/dev/null || true` pattern in discuss-set suppresses all errors including transition failures. Other skills (execute-set, plan-set) do NOT suppress errors, but they also do not verify the transition succeeded by re-reading state.

**Overlooked enforcement mechanisms:**
1. **No post-transition verification:** No skill reads STATE.json after a transition to confirm the new status was written
2. **No commit-with-state pattern:** The discuss-set skill commits CONTEXT.md and STATE.json together, but if the transition silently failed, STATE.json in the commit may still show `pending`
3. **No idempotency check:** Skills check precondition status but do not verify postcondition status
4. **Agent role files have no mandate:** None of the role modules (src/modules/roles/) instruct agents to verify state transitions after operations

**Recommendation:** Add a `state verify` CLI command that reads STATE.json and confirms a set is in the expected status. Skills should call this after every transition, and the command should exit non-zero if the expected status does not match. This makes failures loud and prevents cascading incorrect states.

### 7. Error Handling Across Agent Boundaries

**Why this matters for this project:** When a skill spawns an agent (e.g., discuss-set spawns rapid-research-stack in --skip mode), the agent's failure is communicated via RAPID:RETURN with status BLOCKED. But not all skills have comprehensive handling for every failure mode.

**Observed gaps:**
1. **discuss-set --skip mode:** If the rapid-research-stack agent fails to write CONTEXT.md, the skill checks for file existence (line 137-139) but still proceeds to state transition on line 264-268. The state transition is suppressed with `2>/dev/null || true`, so even if it fails, the skill shows "Next step: /rapid:plan-set" -- misleading the user into thinking discussion succeeded.

2. **plan-set:** If the rapid-plan-verifier agent returns a malformed RAPID:RETURN, there is no explicit fallback. The skill says "Parse RAPID:RETURN for verdict" but does not specify what happens if parsing fails.

3. **execute-set:** If a rapid-executor agent returns CHECKPOINT but the skill fails to parse the checkpoint data, there is no explicit recovery path.

4. **review skill:** Multiple subagent spawns (scoper, unit-tester, bug-hunter, devils-advocate, judge, bugfix, uat) each have their own return format. A parsing failure in any one could cascade or silently corrupt the review pipeline state.

**Cross-cutting recommendation:** Establish a standard "agent return handling" pattern in a shared module or convention:
- Parse RAPID:RETURN with a try/catch
- On parse failure: log error, treat as BLOCKED status
- On unexpected status: log warning, escalate to user via AskUserQuestion
- Verify expected side effects (file writes, state transitions) after agent completion

### 8. Banner Color System Limitations

**Why this matters:** Milestone brief #3 asks to change banner color from pink to dark purple. The current display.cjs uses basic 16-color ANSI codes for terminal compatibility.

**Current color system (display.cjs):**
- Uses `\x1b[10Xm` bright background codes
- Planning stages = bright blue `\x1b[104m`
- Execution stages = bright green `\x1b[102m`
- Review/merge stages = bright red `\x1b[101m`
- No pink/magenta in the current mapping

**Separate from display.cjs:** Agent banner colors are set in YAML frontmatter via the `color:` field (rapid-tools.cjs line 482-508). Current palette: blue, green, red, yellow, purple, cyan, default. The milestone brief says "pink" looks bad with white text -- this likely refers to `purple` in the agent frontmatter, which Claude Code renders differently than expected.

**Overlooked considerations:**
1. The `color` field in agent YAML frontmatter is interpreted by Claude Code's rendering engine, not by RAPID's ANSI code. The available color options and their exact rendering are controlled by Claude Code SDK, not by RAPID. Need to verify what colors Claude Code actually supports and how they render.
2. If "dark purple" is not a supported Claude Code color value, the fix may require a different approach (e.g., custom hex color, different named color).
3. The display.cjs banner colors (ANSI codes) and the agent YAML colors are two separate systems. Changes to one do not affect the other.
4. `NO_COLOR` environment variable is not respected (display.cjs line 8 comment says "Always outputs colors"). This is a minor accessibility concern.

### 9. Testing Strategy for State Machine Changes

**Why this matters:** Multiple milestone fixes (goals #1, #7, #8) modify the state machine or state transitions. The current test suite covers the existing transition chain but may not cover the new paths being added.

**Current test coverage (from files found):**
- `state-transitions.test.cjs` -- Tests the full chain and individual transitions
- `state-machine.lifecycle.test.cjs` -- Tests end-to-end lifecycle transitions
- `state-schemas.test.cjs` -- Tests Zod schema validation
- `resolve.test.cjs` -- Tests set and wave resolution

**Gaps that should be tested:**
1. **Set resolution with archived sets:** No tests verify behavior when `.planning/sets/` has fewer directories than STATE.json has sets
2. **Invalid status in transition call:** Tests exist for invalid transitions between valid statuses, but may not test what happens when a completely invalid status string (like `discussing`) is passed to `transitionSet()`
3. **Review state tracking:** If a `reviewed` status is added, the full lifecycle tests need updating
4. **Concurrent state mutations:** The lock system is tested in lock.test.cjs, but there are no tests for concurrent `withStateTransaction()` calls from multiple agents
5. **Error recovery:** No tests verify what happens if a state transition fails mid-write (e.g., disk full, permission error)

### 10. Documentation and Discoverability for CLI Commands

**Why this matters:** The milestone brief goal #5 (invalid command calls) and #8 (unknown "wave-plan" command) both stem from agents not knowing which CLI commands exist. The USAGE string in rapid-tools.cjs (line 7-91) is the single source of truth for available commands, but agents learn their commands from role modules and skill files, not from the USAGE string.

**Current discovery chain:**
1. Agent role modules (src/modules/roles/*.md) reference CLI commands in their instructions
2. Skill files (skills/*/SKILL.md) reference CLI commands in bash code blocks
3. The tool-docs system (src/lib/tool-docs.cjs) provides YAML documentation for tools
4. The USAGE string in rapid-tools.cjs defines the actual available commands

**Gaps:**
- There is no automated check that commands referenced in skills/agents actually exist in rapid-tools.cjs
- The tool-docs system (tool-docs.cjs) generates YAML tool documentation, but only for agents that have tools injected via the build-agents process. Skills do not consume tool-docs.
- When a command is removed (like `wave-plan`), there is no migration script or grep that finds and updates all references

**Recommendation:** Add a CI/test step that extracts all `node "${RAPID_TOOLS}" <command>` patterns from skills and agents, then verifies each command exists in rapid-tools.cjs. This prevents command drift.

### 11. Configuration Management for New Features

**Why this matters:** The "review after merge" feature (goal #6) needs a configuration option. Currently, `.planning/config.json` exists but is minimally used. The config.json at project root has `lock_timeout_ms` and `agent_size_warn_kb`. Adding new user-configurable behavior (like "when to review") needs a clear config schema.

**Current config locations:**
- `/home/kek/Projects/RAPID/config.json` -- Runtime config (lock settings)
- `.planning/config.json` -- Planning config (per-project, currently sparse)

**Overlooked questions:**
- Should "review after merge" be a global RAPID config or per-project planning config?
- Should it be a boolean toggle or a per-set override?
- How does the user discover and set this config? No CLI command exists for config management.
- The review skill already has stage selection via AskUserQuestion -- could this simply be offered as a step in the merge pipeline rather than a config option?

### 12. Worktree Lifecycle After Review/Merge

**Why this matters:** The merge skill (SKILL.md:609) states "Merged worktrees are NOT auto-removed. Developer runs /rapid:cleanup explicitly." But there is no guidance on when worktrees should be cleaned up relative to review.

If "review after merge" (goal #6) is implemented:
1. The set's worktree is still needed during post-merge review (for running tests, reading files)
2. But the merge has already happened on main, so the worktree's branch is stale
3. The cleanup skill would need to know not to remove worktrees for sets pending post-merge review

This is an interaction between three separate features (review, merge, cleanup) that could easily be overlooked when implementing any one of them independently.

## Recommendations

Ordered by retrofit cost and urgency:

1. **State terminology mismatch (concern #1):** Fix this FIRST. Decide whether to use past-tense statuses everywhere (lower risk) or add present-tense statuses (larger change). Update all 5 affected skills and add tests for the corrected transitions. This unblocks goals #7 and #8.

2. **Set name resolution (concern #2):** Change `resolveSet()` to resolve against STATE.json's current milestone sets array instead of filesystem directories. Preserve `plan.listSets()` for backward compatibility in plan-time operations. Update `resolveWave()` and `execute.cjs` callers. Add test cases for post-archival resolution.

3. **Remove invalid CLI references (concern #3):** Audit every skill and agent role file for `node "${RAPID_TOOLS}"` calls. Either add the missing subcommands to rapid-tools.cjs or rewrite the skill steps to use alternative approaches (direct file reads). Specifically: add `plan validate-contracts` and remove all `wave-plan` references.

4. **Review state machine gap (concern #4):** Add `reviewed` as an optional status. Make both `complete -> merged` (skip review) and `complete -> reviewed -> merged` valid paths. For "review after merge," add `merged -> post-merge-reviewed` or track separately with a boolean flag.

5. **CLI command validation test (concern #10):** Write a test that extracts command patterns from all `.md` files under `skills/` and `agents/`, then verifies each exists in rapid-tools.cjs USAGE string. Run this in the existing test suite. Estimated effort: 1-2 hours. Prevents recurrence of goals #5 and #8.

6. **State transition verification (concern #6):** Remove all `2>/dev/null || true` suppressions from state transition calls. Add explicit error handling. Add a `state verify <milestoneId> <setId> <expectedStatus>` CLI command. Update all skills to verify after transition.

7. **AskUserQuestion pattern (concern #5):** Update discuss-set Step 6 to use pre-filled options instead of freeform "type Claude decides." Establish a convention: every AskUserQuestion MUST have at least one pre-filled option. Document this in CONVENTIONS.md or a shared convention module.

8. **Test coverage for new paths (concern #9):** Write tests before or alongside each fix. Specifically: resolution with missing directories, invalid status strings, lifecycle with `reviewed` status, and concurrent transaction safety.
