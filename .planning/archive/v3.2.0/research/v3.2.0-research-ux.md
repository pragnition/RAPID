## Domain Conventions

### 1. Set Name Resolution After Archiving: Index-Based Resolution Must Be State-Aware

**The bug:** When sets are archived and new sets are added, `resolveSet()` in `src/lib/resolve.cjs` uses `plan.listSets(cwd)` which reads `.planning/sets/` directory entries sorted alphabetically. After archiving sets 4 and 5, then adding sets 6-9, the directory listing changes but the alphabetical sort produces a different 1-based index mapping than the user expects.

**Root cause:** `listSets()` reads the filesystem (`fs.readdirSync`) and sorts alphabetically. It has no awareness of STATE.json's milestone context or archived sets. The numeric index resolution (`resolveSet("6", cwd)`) assumes the filesystem ordering matches the user's mental model of set numbering within the current milestone.

**Industry convention: state-driven resolution, not filesystem-driven.**

| Tool | How References Resolve | Source of Truth |
|------|----------------------|-----------------|
| Kubernetes | Pod names resolve via API server state, not filesystem | etcd state store |
| Terraform | Resource references resolve via state file, not directory listing | terraform.tfstate |
| GitHub Actions | Job references resolve via workflow YAML + run state | GitHub API |
| Docker Compose | Service names resolve via compose file definition order | docker-compose.yml |
| Git | Branch/tag references resolve via ref database | .git/refs |

**The pattern is clear:** references should resolve against the authoritative state (STATE.json), not derived filesystem state. The filesystem is a cache; the state file is the source of truth.

**Source:** [Terraform State -- HashiCorp](https://developer.hashicorp.com/terraform/language/state), [Kubernetes API Conventions](https://github.com/kubernetes/community/blob/master/contributors/devel/sig-architecture/api-conventions.md)

**Specific recommendation:** `resolveSet()` should resolve numeric indices against `state.milestones[currentMilestone].sets[]` order, not `fs.readdirSync('.planning/sets/')` alphabetical sort. The STATE.json set list is the canonical ordering. Filesystem-based resolution should be the fallback for backward compatibility only.

### 2. Batched Questions UX: AskUserQuestion Native Multi-Header Pattern

**The bug:** During the discuss phase, the agent sometimes presents multiple questions as freeform text in a single AskUserQuestion prompt, overwhelming the user. The current discuss-set SKILL.md (Step 6, line 179-191) instructs agents to batch 2-3 questions per gray area into a single freeform prompt with a "type 'Claude decides'" escape.

**What AskUserQuestion actually supports (native capabilities):**

Per Claude Code's tool specification ([source: Piebald-AI/claude-code-system-prompts](https://github.com/Piebald-AI/claude-code-system-prompts/blob/main/system-prompts/tool-description-askuserquestion.md), [source: bgauryy/gist](https://gist.github.com/bgauryy/0cdb9aa337d01ae5bd0c803943aa36bd)):

- **1-4 questions per tool call** (the `questions` array)
- Each question has: `header` (max 12 chars), `question` (full text), `multiSelect` (boolean), `options` (2-4 choices)
- Each option has: `label` (1-5 words), `description` (explanation)
- Users can navigate between question headers in the UI

**The mismatch:** The discuss-set skill instructs agents to present 2-3 questions as freeform text in one prompt. This bypasses the native multi-header capability where each question gets its own header with pre-filled options. The result is a wall of text that overwhelms users.

**Industry convention: structured choices reduce cognitive load.**

**Source:** [Designing For Agentic AI -- Smashing Magazine](https://www.smashingmagazine.com/2026/02/designing-agentic-ai-practical-ux-patterns/) -- "Autonomy dial" pattern: give users structured controls, not open-ended prompts. Pre-filled options with an escape hatch ("I'll answer my own way") is the recommended pattern.

**Source:** [Laws of UX -- Cognitive Load](https://lawsofux.com/cognitive-load/) -- Structured multiple-choice questions impose less cognitive load than open-ended prompts requiring users to compose answers from scratch.

**Source:** [Inquirer.js patterns](https://www.npmjs.com/package/@inquirer/prompts) -- The standard CLI prompting library uses per-question prompts with pre-filled choices, not multi-question freeform blocks.

**Specific recommendation:** Rewrite discuss-set Step 6 to use one AskUserQuestion call with 1-4 questions (matching the native API limit). Each gray area becomes one question with `header` (area name, max 12 chars), the full question in `question`, and 2-4 pre-filled option approaches plus an "I'll answer" option. This leverages the native tab-between-headers navigation in Claude Code's UI.

### 3. Never Show "Claude Decides" Without Options: Always Provide Pre-Filled Choices

**The bug (from todo_new.md):** Some AskUserQuestion prompts present a "Claude decides" option alongside a freeform "Type something" option, with no structured alternatives. This violates the principle that every decision gate should offer pre-filled choices.

**Bad pattern observed:**
```
Answer all, or 'Claude decides'.

> 1. Claude decides
     Let Claude handle auto-advance behavior decisions
  2. I'll answer
     I'll provide my thoughts on auto-advance behavior
  3. Type something.
  4. Chat about this
```

**Industry convention: default-with-opt-out, never bare delegation.**

Every AskUserQuestion call should present 2-4 concrete pre-filled options representing the most likely choices, plus an "I'll answer" escape hatch. The "Claude decides" option is fine as the first option, but it must be accompanied by specific alternatives the user can evaluate, not just "Type something."

**Source:** [RAPID v3.0 UX Research, Principle 3: Strong Defaults, Rare Questions](../research/DOMAIN-UX.md) -- "The 'Autonomy Dial' from Smashing Magazine's agentic UX patterns. Not every decision needs developer input." The autonomy dial works because the user sees the AI's proposed action and can accept or override -- not because the user is given an empty box.

**Source:** [Claude Code AskUserQuestion Guide -- SmartScope](https://smartscope.blog/en/generative-ai/claude/claude-code-askuserquestion-tool-guide/) -- Pre-filled options with clear descriptions are the recommended pattern for reducing ambiguity.

**Specific recommendation:** Audit all AskUserQuestion calls across all skills and agent roles. Every call must have 2-4 pre-filled option objects with `label` and `description`. The "Claude decides" option is acceptable as option 1 but must always be accompanied by specific concrete alternatives. Remove any "Type something" or freeform-only patterns.

## Interaction Models

### Banner Colors: Terminal Color Readability

**The bug:** Subagent banner colors look bad. The user specifically reports pink banners with white text have poor readability, and requests dark purple instead.

**Current state in display.cjs:** Banners use `STAGE_BG` with 16-color ANSI codes:
- Planning stages: `\x1b[104m` (bright blue background)
- Execution stages: `\x1b[102m` (bright green background)
- Review stages: `\x1b[101m` (bright red background)

**Current state in agent YAML:** Agent frontmatter `color:` field controls the Claude Code subagent banner. Current assignments include blue (most agents), green (executor, merger, bugfix, set-merger), red (reviewer, judge), cyan (unit-tester, uat), yellow (bug-hunter, conflict-resolver), purple (devils-advocate).

**Note:** The `color` field in agent YAML is a Claude Code platform property that controls the subagent's UI banner color. Available values per Claude Code documentation ([source: Claude Code Sub-agents Docs](https://code.claude.com/docs/en/sub-agents)): `blue`, `red`, `green`, `yellow`, `purple`, `orange`, `pink`, `cyan`. There is no "dark purple" option -- only `purple`.

**Industry convention: high-contrast text on colored backgrounds.**

**Source:** [WCAG 2.1 Success Criterion 1.4.3](https://www.w3.org/WAI/WCAG21/Understanding/contrast-minimum.html) -- Minimum contrast ratio of 4.5:1 for normal text. White text on bright pink fails this threshold. White text on darker backgrounds (purple, dark blue) passes.

**Source:** [ANSI Escape Codes Reference](https://gist.github.com/fnky/458719343aabd01cfb17a3a4f7296797) -- For the 16-color palette, `\x1b[45m` is standard magenta/purple background (dark), `\x1b[105m` is bright magenta (lighter). For 256-color, `\x1b[48;5;53m` gives a dark purple/magenta background. RAPID currently uses the 16-color palette for maximum terminal compatibility.

**For display.cjs stage banners:** The `STAGE_BG` colors (bright blue, bright green, bright red) are not pink. The "pink" complaint likely refers to either:
1. The Claude Code platform banner color for agents with `color: pink` in their YAML frontmatter (not controlled by display.cjs)
2. How `\x1b[105m` (bright magenta) renders on some terminals

**For agent YAML colors:** Change any `color: pink` agents to `color: purple`. Currently no agents use `color: pink` -- the closest is `color: purple` (devils-advocate only). If the user sees pink, it may be the platform's rendering of `purple` or the banner colors on certain terminal themes.

**Specific recommendation:**
1. In display.cjs, consider using `\x1b[45m` (standard magenta, darker) instead of bright variants where dark purple is desired. For 256-color terminals, `\x1b[48;5;53m` gives a true dark purple.
2. In agent YAML frontmatter, audit `color:` values. Replace any `pink` with `purple`. Consider a more deliberate color scheme: blue for research/planning, green for execution, red for review, purple for merge/conflict, yellow for adversarial.
3. Add a `subagentColor` config option to display.cjs or config.json so the color scheme can be customized without editing source.

### Review After Merge: Post-Merge Review Pattern

**The request:** Users sometimes want to review after merging, not just after each set completes.

**Current flow:** `execute-set` -> `complete` -> `/rapid:review` -> `reviewed`(?) -> `/rapid:merge` -> `merged`. Review is a pre-merge gate.

**Industry convention: both pre-merge and post-merge review are valid patterns.**

| Workflow | Pre-Merge Review | Post-Merge Review | When Used |
|----------|------------------|-------------------|-----------|
| GitHub PR | Required (branch protection) | Optional (post-deploy checks) | Most teams |
| GitLab MR | Required (merge request pipelines) | Post-merge pipeline | Standard |
| Trunk-based dev | Not required | Post-commit review | High-trust teams |
| RAPID current | Required (review -> merge) | Not supported | Always |

**Source:** [Post-Commit Reviews -- Cindy Sridharan](https://copyconstruct.medium.com/post-commit-reviews-b4cc2163ac7a) -- "A heavy investment in automation and monitoring is required for fostering a successful 'post-commit' review culture." Post-commit review works when paired programming or high automation confidence exists.

**Source:** [The role of code review in CI/CD pipelines -- Graphite](https://graphite.dev/guides/role-code-review-ci-cd) -- Code review can happen at multiple points: pre-merge (PR review), post-merge (integration validation), and post-deploy (production monitoring).

**Use case for RAPID:** When a user trusts the executor agents and wants to merge quickly (e.g., small sets with high isolation), forcing a review before merge adds friction. They may prefer to batch-review all sets after merge, catching integration issues rather than per-set issues.

**Specific recommendation:**
1. Make `/rapid:review` accept a `--post-merge` flag or allow reviewing sets in `merged` state (not just `complete` state).
2. Add a "skip review" option to the workflow: allow `complete` -> `merged` transition without requiring review. The review step becomes optional but recommended.
3. In the merge skill, add a step that asks whether to review before or after merge when the set status is `complete` (not yet reviewed).
4. Update state-transitions.cjs to allow `complete` -> `merged` (this already exists). Ensure `merged` -> (no further transitions) is preserved -- post-merge review does not change state, it just generates review artifacts.

### State Update Enforcement: Agents Must Update State

**The bug:** Agents sometimes fail to update STATE.json after completing their phase (merging, executing, discussing). This leaves the project state stale.

**Industry convention: state transitions are the agent's responsibility, enforced at the orchestration layer.**

| Tool | Enforcement Mechanism |
|------|----------------------|
| Kubernetes | Controller reconciliation loops -- if a pod reports ready but the state is wrong, the controller fixes it |
| GitHub Actions | Runner reports status back to API -- if status update fails, job is marked failed |
| Terraform | State file is updated atomically at the end of each apply -- if the update fails, the apply is considered failed |
| Airflow | Task status is updated in the metadata database -- DAG runner treats missing updates as failures |

**Source:** [Kubernetes Controllers](https://kubernetes.io/docs/concepts/architecture/controller/) -- The controller pattern ensures desired state matches actual state through reconciliation loops.

**RAPID's situation:** Skills (SKILL.md files) instruct agents to call `state transition set` as a specific step. But agents can skip steps, fail mid-execution, or encounter errors that prevent the state update. No reconciliation or verification occurs.

**Specific recommendation:**
1. Each SKILL.md should have state transition as the FINAL mandatory step, with verification (`state get --all` to confirm the transition succeeded).
2. The `/rapid:status` skill should include a "state reconciliation" check: compare filesystem artifacts (e.g., CONTEXT.md exists -> set should be at least `discussed`) against STATE.json and warn about mismatches.
3. Add a post-execution hook in the task-completed hook (`src/hooks/rapid-task-completed.sh`) that verifies state transitions occurred. If the set's filesystem artifacts suggest a phase is complete but state was not updated, log a warning.
4. Consider making state transition a PRECONDITION of the next phase (already partially done -- plan-set checks for `discussed` status) rather than relying on the current phase to transition.

### Invalid Command Calls: Agents Reference Non-Existent CLI Commands

**The bug:** Agents call `wave-plan` and other commands that do not exist in `rapid-tools.cjs`. Specific findings:

**Commands referenced in skills but NOT in rapid-tools.cjs:**
- `wave-plan` -- referenced in:
  - `skills/plan-set/SKILL.md` line 280: `wave-plan validate-contracts`
  - `skills/review/SKILL.md` line 190: `wave-plan list-jobs`
  - `src/lib/tool-docs.cjs` lines 66-69: 4 wave-plan subcommands registered
  - `src/modules/roles/role-roadmapper.md` line 160: references `/rapid:wave-plan`

**Commands referenced in tool-docs.cjs but NOT in rapid-tools.cjs:**
- `wave-plan resolve-wave`
- `wave-plan create-wave-dir`
- `wave-plan validate-contracts`
- `wave-plan list-jobs`

**Status name mismatches between skills and state-transitions.cjs:**
The state-transitions.cjs was updated to use past-tense status names (`discussed`, `planned`, `executed`), but multiple skills still reference the old present-progressive names:
- `skills/discuss-set/SKILL.md` line 268: transitions to `discussing` (should be `discussed`)
- `skills/plan-set/SKILL.md` line 74: checks for `discussing` (should be `discussed`)
- `skills/plan-set/SKILL.md` line 293: transitions from `discussing` to `planning` (should be `discussed` to `planned`)
- `skills/execute-set/SKILL.md` line 80: checks for `discussing` (should be `discussed`)
- `skills/status/SKILL.md` line 30: lists old status names
- `skills/new-version/SKILL.md` line 47: lists old status names

**This is the root cause of the `"discussing" -> "complete"` error in todo_new.md.** The discuss-set skill transitions to `discussing`, but the state machine only accepts `discussed` as the target. The error cascades: subsequent skills check for the old name, fail, and agents try to force transitions that do not exist.

**Industry convention: single source of truth for command and status registries.**

**Source:** [Command Line Interface Guidelines (clig.dev)](https://clig.dev/) -- "The help text of your command is part of your API. Don't change it without thought." Commands and statuses should be documented in one canonical place and all references should derive from it.

**Specific recommendation:**
1. Audit ALL skills and role modules for `wave-plan` references. Replace with the correct equivalent commands or remove if the functionality was intentionally removed in v3.
2. Audit ALL skills for old status names (`discussing`, `planning`, `executing`). Replace with new past-tense names (`discussed`, `planned`, `executed`).
3. Update `src/lib/tool-docs.cjs` to remove the 4 `wave-plan-*` entries and add any replacement commands.
4. Consider generating a "valid commands" reference from rapid-tools.cjs that skills and agents can be validated against (a linting step in `build-agents`).

## Information Architecture

### Workflow Flexibility: Optional Review Before Merge

The current fixed workflow is:
```
init > start-set > discuss-set > plan-set > execute-set > review > merge
```

With review-after-merge support, the workflow becomes:
```
init > start-set > discuss-set > plan-set > execute-set > [review] > merge > [review]
```

Where `[review]` indicates an optional step. The user chooses when to review:
- **Pre-merge review** (default, recommended): Catches bugs before they enter main
- **Post-merge review** (opt-in): Faster workflow, catches integration issues
- **Both**: Maximum coverage for critical changes

The progress breadcrumb should reflect this flexibility:
```
init [done] > start-set [done] > discuss-set [done] > plan-set [done] > execute-set [done] > merge [done] > review [pending]
```

### Error Messages for Invalid State Transitions

When an agent attempts an invalid transition (e.g., `discussing -> complete` when the valid path is `discussed -> planned`), the error message should:

1. Show the CURRENT valid transition map (not just the error)
2. Suggest the correct transition
3. Identify if the status name is from a previous version

**Current error:**
```
Invalid transition: "discussing" -> "complete". Valid transitions from "discussing": [planning]
```

**This error is itself wrong** because `discussing` is no longer a valid status. The actual error should be:
```
Unknown status "discussing". Valid statuses: pending, discussed, planned, executed, complete, merged.
Did you mean "discussed"? (Status names were updated to past tense in v3.1.0)
```

**Specific recommendation:** Add a "did you mean?" suggestion to `validateTransition()` when the status is unknown but similar to a valid one (e.g., Levenshtein distance <= 3 or suffix match like `-ing` -> `-ed`).

## Accessibility Considerations

### Terminal Color Readability

The banner color issue (pink with white text) is fundamentally an accessibility problem. White text on bright/light backgrounds fails WCAG contrast requirements.

**Current display.cjs palette:**
- Bright blue background (`\x1b[104m`) with bright white text: **PASS** (good contrast)
- Bright green background (`\x1b[102m`) with bright white text: **MARGINAL** (may fail on some terminals)
- Bright red background (`\x1b[101m`) with bright white text: **MARGINAL** (depends on terminal theme)

**Recommendation:** Use standard (non-bright) background colors for higher contrast with white text:
- `\x1b[44m` (standard blue) -- high contrast with white
- `\x1b[42m` (standard green) -- high contrast with white
- `\x1b[41m` (standard red) -- high contrast with white
- `\x1b[45m` (standard magenta/purple) -- high contrast with white

The bright variants (10Xm) are lighter and have worse contrast with white text. The standard variants (4Xm) are darker and more readable.

### AskUserQuestion Accessibility

The native multi-header format in AskUserQuestion is more accessible than freeform text blocks because:
- Screen readers can navigate between question headers
- Each question has a bounded set of options (keyboard-navigable)
- Pre-filled options reduce typing burden
- The 12-character header limit enforces concise labeling

## User Expectations

### What Users Expect from Set Resolution

1. **"Set 6 means the 6th set in my current milestone."** Users think of set numbers as their order within the active milestone context, not as filesystem directory indices.
2. **Archiving should not change live set numbering.** If sets 1-5 existed and 4-5 were archived, then sets 6-9 were added, users expect sets 6-9 to be numbered 1-4 in the current context (or 6-9 if preserving original numbering).
3. **Explicit is better than magic.** If numbering changes after archiving, the `/rapid:status` display should clearly show the numeric index alongside the set name so users can verify.

### What Users Expect from Discussion Questions

1. **One question at a time, with options.** Users expect structured prompts like a form, not essay questions. Each gray area should be its own question with pre-filled options.
2. **Navigation between questions.** The native AskUserQuestion multi-header format lets users tab between questions. Users expect this navigation.
3. **"I'll answer" as an escape hatch, not the default.** Every question should have concrete options. "I'll answer" is for edge cases where the pre-filled options do not match.
4. **Minimal questions overall.** The 4-gray-area limit in discuss-set is correct. Users do not want more than 4 decision points per set.

### What Users Expect from Review Timing

1. **Optional review is a sign of tool maturity.** Mature CI/CD tools let users configure which quality gates are required and which are optional. RAPID forcing review before merge feels rigid.
2. **Post-merge review catches different bugs.** Pre-merge review catches per-set issues. Post-merge review catches integration issues between merged sets. Both are valuable.
3. **The default should remain pre-merge.** Most users benefit from catching bugs before merge. The opt-out should require explicit action (flag or prompt).

### What Users Expect from State Consistency

1. **The tool's state should always reflect reality.** If I ran discuss-set and it completed, the state should show `discussed`. Finding the state still at `pending` is confusing and erodes trust.
2. **Errors during state update should be loud.** If the state transition fails, the user should see a clear error, not a silent success with stale state.
3. **The next command should validate the previous command completed.** If plan-set requires `discussed` status, it should fail clearly if the status is still `pending` (because discuss-set failed to update state).
