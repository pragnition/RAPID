# Pitfalls Research -- v3.2.0 General Fixes

## Critical Pitfalls

### Pitfall: Status Name Mismatch Between Skills and State Machine
- **Category:** correctness
- **Stack:** state-transitions.cjs, state-schemas.cjs, all SKILL.md files
- **Description:** The state machine uses past-tense status names (`discussed`, `planned`, `executed`) for set transitions, but multiple SKILL.md files pass present-tense names (`discussing`, `planning`, `executing`) to the `state transition set` CLI command. This causes every state transition call in the affected skills to fail with "Invalid transition" errors.
- **Trigger:** Any skill calling `node "${RAPID_TOOLS}" state transition set` with the wrong status name. Specifically:
  - `discuss-set/SKILL.md:268` passes `discussing` (should be `discussed`)
  - `plan-set/SKILL.md:297` passes `planning` (should be `planned`)
  - `execute-set/SKILL.md:148` passes `executing` (should be `executed`)
  - `review/SKILL.md:80` passes `reviewing` (does not exist at all)
  - `execute-set/SKILL.md:332` passes `complete` (this one is correct -- `complete` is the actual status name)
  - `merge/SKILL.md:364` passes `merged` (also correct)
- **Impact:** Severity HIGH. The discuss-set skill silently suppresses the error with `2>/dev/null || true`, meaning the state is never actually transitioned but appears to succeed. The plan-set and execute-set skills will fail loudly. The review skill tries to transition to `reviewing`, which does not exist in the schema at all. This is the root cause of the "discussing -> complete" error in the milestone brief -- the set remains stuck in `pending` (discuss-set silently fails), then later skills that check for the expected status find the wrong state.
- **Prevention:** Audit every `state transition set` call in all skills and agents against the canonical `SET_TRANSITIONS` map in `state-transitions.cjs`. The valid status names are: `pending`, `discussed`, `planned`, `executed`, `complete`, `merged`. Use a grep-based CI check or a linting step in `build-agents` to verify that all transition targets match the schema enum.
- **Detection:** Search for `state transition set` in all `.md` files and compare the target status against `SetStatus` in `state-schemas.cjs`. A simple grep for `transition set.*discussing|transition set.*planning|transition set.*executing|transition set.*reviewing` will find all violations.
- **References:** `src/lib/state-transitions.cjs` lines 3-10, `src/lib/state-schemas.cjs` line 4

### Pitfall: Set Name Resolution Uses Disk Directory Listing, Not State
- **Category:** correctness
- **Stack:** resolve.cjs, plan.cjs (listSets)
- **Description:** `resolveSet()` in `resolve.cjs` calls `plan.listSets(cwd)` which reads `.planning/sets/` directory entries via `fs.readdirSync`. Numeric indices (e.g., "set 6") map to the Nth alphabetically-sorted directory on disk. When sets are archived (directories removed from `.planning/sets/`), the numeric mapping shifts -- set 6 might now map to what was set 8. The state machine (STATE.json) has its own authoritative list of sets per milestone, but `resolveSet` does not consult it.
- **Trigger:** Archive or delete set directories from `.planning/sets/` while sets still exist in STATE.json. Then use numeric indexing (e.g., `/rapid:execute-set 6`). The resolution will point to the wrong set because the directory listing has gaps.
- **Impact:** Severity CRITICAL. Users operate on the wrong set without realizing it. Commands will execute against unintended targets, potentially corrupting state or modifying wrong files.
- **Prevention:** The resolver should use STATE.json as the source of truth for set listing, not the filesystem. Specifically, `resolveSet` should read sets from the current milestone in STATE.json, falling back to disk only when state is unavailable. Alternatively, provide a reconciliation step that keeps `.planning/sets/` in sync with STATE.json.
- **Detection:** Compare `plan.listSets(cwd)` output with the set IDs in `state.milestones[current].sets`. If they differ, resolution is broken. Users will notice when the resolved set name doesn't match what they expected.
- **References:** `src/lib/resolve.cjs` lines 19-53, `src/lib/plan.cjs` lines 169-181

### Pitfall: `wave-plan` CLI Command Does Not Exist
- **Category:** correctness
- **Stack:** rapid-tools.cjs, skills/plan-set/SKILL.md, skills/review/SKILL.md, tool-docs.cjs
- **Description:** The CLI router in `rapid-tools.cjs` has no `wave-plan` case in the main `switch(command)` block (lines 153-214). The valid commands are: `lock`, `state`, `parse-return`, `verify-artifacts`, `plan`, `assumptions`, `worktree`, `execute`, `merge`, `set-init`, `review`, `resume`, `resolve`, `build-agents`. However, `tool-docs.cjs` registers four `wave-plan` commands (lines 66-69), and `plan-set/SKILL.md` calls `node "${RAPID_TOOLS}" wave-plan validate-contracts` (line 280), and `review/SKILL.md` calls `wave-plan list-jobs` (line 190). These are ghost commands from the v2 era that were never wired in v3.
- **Impact:** Severity HIGH. The plan-set skill's contract validation step (Step 7) will always fail with "Unknown command: wave-plan". Since the skill treats this as advisory, it may not block the flow, but the validation gate is completely non-functional. The review skill's job listing also fails, potentially breaking acceptance criteria extraction.
- **Prevention:** Remove all `wave-plan` entries from `tool-docs.cjs`. Replace `wave-plan validate-contracts` calls in skills with either a new subcommand under the `plan` command group or remove the call entirely if contract validation is not yet implemented in v3. Audit `ROLE_TOOL_MAP` in `tool-docs.cjs` for any roles referencing `wave-plan-*` keys (currently `plan-verifier` references `wave-plan-validate`).
- **Detection:** Run `grep -r "wave-plan" skills/ src/modules/ src/lib/tool-docs.cjs` and cross-reference against the command router's switch cases.
- **References:** `src/bin/rapid-tools.cjs` lines 153-214 (no wave-plan case), `src/lib/tool-docs.cjs` lines 66-69, `skills/plan-set/SKILL.md` line 280, `skills/review/SKILL.md` line 190


## High-Risk Pitfalls

### Pitfall: Review Skill References Non-Existent `reviewing` Status
- **Category:** correctness
- **Stack:** skills/review/SKILL.md, state-transitions.cjs
- **Description:** The review skill (line 80) attempts to transition a set to `reviewing` status, but `reviewing` is not in the `SetStatus` enum (`pending`, `discussed`, `planned`, `executed`, `complete`, `merged`). The Zod schema validation will reject this status, and the transition will throw an error. There is no concept of a `reviewing` lifecycle state in the v3 state machine.
- **Impact:** Severity HIGH. Review cannot be started because the state transition fails. The review pipeline is completely broken unless the user manually bypasses this step.
- **Prevention:** Either remove the `reviewing` transition entirely (reviews operate on `complete` sets and the set moves to `merged` after merge, not after review), or add `reviewing` to the enum and transition map if a review state is architecturally desired. Based on the current lifecycle (`complete -> merged`), removing the transition is the correct fix.
- **Detection:** `grep -r "reviewing" skills/ src/lib/state-transitions.cjs src/lib/state-schemas.cjs` shows the mismatch immediately.
- **References:** `skills/review/SKILL.md` lines 75-83, `src/lib/state-schemas.cjs` line 4

### Pitfall: Discuss-Set Silently Swallows State Transition Errors
- **Category:** data-loss / correctness
- **Stack:** skills/discuss-set/SKILL.md
- **Description:** Line 268 uses `2>/dev/null || true` after the state transition call. This was intended to handle the "already discussing" re-entry case gracefully, but it also silently swallows legitimate errors like wrong status names (the `discussing` vs `discussed` mismatch), invalid JSON, or lock failures. The state never actually transitions, but the skill proceeds as if it did.
- **Impact:** Severity HIGH. The set remains in `pending` status while CONTEXT.md is written. Downstream skills (plan-set) check for `discussing` status and may reject the set or encounter unexpected behavior. State and artifacts drift apart.
- **Prevention:** Parse the transition command output and check the exit code explicitly. Handle the specific "already in discussing state" case, but surface all other errors. Use stderr capture instead of `/dev/null` to log transition details.
- **Detection:** After `discuss-set` completes, read STATE.json and verify the set's status actually changed from `pending`. If it still says `pending`, the transition silently failed.
- **References:** `skills/discuss-set/SKILL.md` line 268

### Pitfall: Execute-Set Uses Mismatched Transition Names in Step 6
- **Category:** correctness
- **Stack:** skills/execute-set/SKILL.md
- **Description:** Step 3 (line 148) transitions to `executing` (wrong -- should be `executed`), and Step 6 (line 332) transitions to `complete` (correct). However, the expected precondition in the transition map is: `planned -> executed`, not `planned -> executing`. The skill will fail at Step 3 because `executing` is not a valid target for set transitions (it's only valid for wave/job transitions).
- **Impact:** Severity HIGH. Set execution cannot start because the initial state transition fails. The entire execute-set flow is broken at the first CLI call.
- **Prevention:** Change `executing` to `executed` at line 148. Note the semantic disconnect: the skill transitions to `executed` at the start of execution (Step 3), which semantically means "done executing" even though execution is just beginning. This is an architectural naming issue -- the past-tense names are confusing for intermediate states. Consider whether the transition should happen at the start or end of execution.
- **Detection:** Run `execute-set` -- it will fail with "Invalid transition" error at Step 3.
- **References:** `skills/execute-set/SKILL.md` lines 148, 332, `src/lib/state-transitions.cjs` lines 3-10

### Pitfall: proper-lockfile Compromised Lock Callback Only Logs
- **Category:** data-loss
- **Stack:** proper-lockfile 4.1.2, lock.cjs
- **Description:** In `lock.cjs` line 59-62, the `onCompromised` callback writes to stderr but does not throw or abort the operation. If a lock is compromised (the mtime update fails multiple times, causing the lock to appear stale to another process), the current process continues its state mutation while another process may have acquired the same lock. This creates a write-write race condition on STATE.json.
- **Impact:** Severity MEDIUM (but CRITICAL when it occurs). Two concurrent state mutations can interleave, leading to data loss or corruption of STATE.json. This is unlikely in normal single-user operation but can occur if multiple agents or parallel skill invocations run simultaneously.
- **Prevention:** The `onCompromised` callback should throw an error to abort the in-flight mutation. However, this changes error handling semantics for `withStateTransaction`. An alternative is to add a secondary validation step that re-reads and compares state after write to detect concurrent modification (optimistic concurrency).
- **Detection:** Watch for `[RAPID] Lock "state" compromised:` messages in stderr. If they appear, STATE.json may be corrupted.
- **References:** `src/lib/lock.cjs` lines 54-63, [proper-lockfile compromised locks issue](https://github.com/moxystudio/node-proper-lockfile/issues/11)

### Pitfall: `readState` Does Not Acquire Lock
- **Category:** correctness
- **Stack:** state-machine.cjs
- **Description:** The `readState` function (lines 44-64) reads STATE.json without acquiring a lock. While `writeState` and `withStateTransaction` both lock, a pattern like "read state, check condition, call transitionSet" creates a TOCTOU (time-of-check-time-of-use) race. Between the read and the transition, another process could modify the state.
- **Impact:** Severity MEDIUM. In RAPID's typical single-agent-at-a-time operation, this rarely triggers. But with parallel set execution (multiple worktrees, multiple agents), concurrent reads followed by conditional transitions can lead to invalid state.
- **Prevention:** For read-then-mutate patterns, always use `withStateTransaction` which reads inside the lock. Avoid patterns that read state outside a transaction and then make decisions based on the read value.
- **Detection:** Look for code patterns like `const s = await readState(cwd); ... await transitionSet(...)`. The gap between read and transition is the vulnerability window.
- **References:** `src/lib/state-machine.cjs` lines 44-64


## Medium-Risk Pitfalls

### Pitfall: Stale Lock Threshold May Be Too Long for Fast Iterations
- **Category:** performance
- **Stack:** lock.cjs, proper-lockfile 4.1.2
- **Description:** The `STALE_THRESHOLD` is set to 300,000ms (5 minutes). If a process crashes while holding a lock, other processes must wait up to 5 minutes before the lock is detected as stale. The `cleanStaleLocks` function checks PID liveness, but it must be explicitly called -- it doesn't run automatically.
- **Impact:** Severity MEDIUM. After a crash, the user waits up to 5 minutes for the next CLI call to succeed, or must manually invoke stale lock cleanup.
- **Prevention:** Add `cleanStaleLocks()` as a startup step in the main CLI entry point before any lock-requiring operations. Consider reducing the stale threshold to 60 seconds for development use cases.
- **Detection:** If CLI calls hang for several minutes after a crash or Ctrl+C, the stale lock is the cause. Check for `.planning/.locks/*.target.lock` directories.
- **References:** `src/lib/lock.cjs` lines 8, 110-131

### Pitfall: Zod 3.25.x Ships Both Zod 3 and Zod 4 Internally
- **Category:** compatibility
- **Stack:** zod 3.25.76
- **Description:** Starting with Zod 3.25.0, the npm package bundles both Zod 3 and Zod 4 code internally. While this doesn't break the `require('zod')` path (which still returns Zod 3), it nearly doubles the package size and has caused CJS loading issues in some configurations (zod issue #4434). The project uses `require('zod')` which is safe, but any future import from `zod/v4` would change behavior.
- **Impact:** Severity LOW. No functional impact currently, but awareness is needed for future upgrades. If anyone imports from `zod/v4` subpath, the behavior changes significantly (optional properties become nullable by default per issue #4883).
- **Prevention:** Pin to `zod@3.25.76` in package.json. Do not use `zod/v4` subpath imports. When upgrading, test state schema validation thoroughly.
- **Detection:** Check `node -e "console.log(require('zod').z.ZodFirstPartyTypeKind)"` still returns expected values. Run the full state-schemas test suite after any zod version change.
- **References:** [Zod issue #4923](https://github.com/colinhacks/zod/issues/4923), [CJS loading issue #4434](https://github.com/colinhacks/zod/issues/4434), [Optional properties breaking change #4883](https://github.com/colinhacks/zod/issues/4883)

### Pitfall: Display Banner Colors Are Bright Backgrounds, Not Dark Purple
- **Category:** correctness (UX)
- **Stack:** display.cjs
- **Description:** The milestone brief requests changing banner colour from pink to dark purple, but the current implementation uses `\x1b[104m` (bright blue background) for planning stages and `\x1b[101m` (bright red) for review/merge. There is no pink currently -- the "pink" may refer to how `\x1b[101m` (bright red) renders on some terminals, where bright red bg + white text can appear pinkish. Changing to "dark purple" requires using `\x1b[45m` (standard magenta) or a 256-color escape like `\x1b[48;5;53m` (dark purple). The 256-color approach may not work on all terminals.
- **Impact:** Severity LOW. Purely cosmetic, but affects brand consistency.
- **Prevention:** Use standard 16-color ANSI codes for maximum terminal compatibility. `\x1b[45m` (magenta background) is the closest standard option to dark purple. Test on common terminals (iTerm2, Terminal.app, Windows Terminal, Alacritty) before shipping.
- **Detection:** Visual inspection. Run `node "${RAPID_TOOLS}" display banner review` and check the color.
- **References:** `src/lib/display.cjs` lines 50-65

### Pitfall: tool-docs.cjs `plan-check-gate` References Non-Existent Command

## Anti-Patterns for This Stack

### Anti-Pattern: Silently Swallowing CLI Errors with `2>/dev/null || true`
- **What people do:** Append `2>/dev/null || true` to CLI calls to handle "already in state X" re-entry gracefully.
- **Why it fails:** This also silently swallows wrong status names, lock failures, JSON parse errors, and missing state files. The skill proceeds as if the transition succeeded when it did not.
- **Correct approach:** Capture stderr and exit code explicitly. Check for the specific expected error condition (e.g., "already in this state"). Surface all other errors.
- **Example:**
  ```bash
  RESULT=$(node "${RAPID_TOOLS}" state transition set "$M" "$S" discussed 2>&1)
  EXIT=$?
  if [ $EXIT -ne 0 ]; then
    if echo "$RESULT" | grep -q "already in"; then
      echo "Set already discussed, continuing..."
    else
      echo "ERROR: $RESULT"
      exit 1
    fi
  fi
  ```

### Anti-Pattern: Using Filesystem Directory Listing as Source of Truth for Named Entities
- **What people do:** Read `.planning/sets/` directories to enumerate sets, then use array index for numeric resolution.
- **Why it fails:** Directories can be archived, deleted, or out of sync with STATE.json. The numeric index mapping shifts when directories are removed, causing set 6 to resolve to set 8.
- **Correct approach:** Use STATE.json as the authoritative set list. The filesystem is a cache/artifact store, not the index. Resolution should read from `state.milestones[current].sets` and use the set IDs there for both string and numeric lookups.

### Anti-Pattern: Mixing Present-Tense and Past-Tense Status Names
- **What people do:** Documentation, comments, and skill instructions use present-tense ("discussing", "planning", "executing") to describe the current phase, but the state machine uses past-tense ("discussed", "planned", "executed") as status values.
- **Why it fails:** Developers (and AI agents reading the skills) naturally use the present-tense form when writing transition commands, creating mismatches. The skill says "transition to discussing" but the valid target is "discussed".
- **Correct approach:** Choose one convention and apply it everywhere. If the state machine uses past-tense, all documentation, skills, status displays, and transition calls must use past-tense. Alternatively, consider renaming statuses to be unambiguous action verbs or adding validation/aliasing at the CLI layer.

### Anti-Pattern: Referencing Removed Commands in Documentation and Tool Registries
- **What people do:** After removing a CLI command (e.g., `wave-plan`), leave references in SKILL.md files, tool-docs.cjs, and agent role modules.
- **Why it fails:** Agents hallucinate that the command exists because their prompt includes tool docs referencing it. Skills fail at runtime when they call the removed command. The `tool-docs.cjs` registry reports commands that produce "Unknown command" errors.
- **Correct approach:** When removing a CLI command, search all `.md` files and `tool-docs.cjs` for references. The `build-agents` command should validate that all tool doc keys map to existing commands. Add a CI-level check that greps for known-removed command names.


## Version-Specific Gotchas

- **Node.js 18+:** `fs.rmSync` with `{ recursive: true, force: true }` is used in `cleanStaleLocks` (lock.cjs line 122). This API is stable in Node 18+ but throws `ERR_FEATURE_UNAVAILABLE_ON_PLATFORM` on very old Node versions. Not an issue given the Node 18+ requirement, but worth noting for contributors.
- **proper-lockfile 4.1.2:** The `realpath: false` option is used in lock.cjs. This is needed because `.planning/.locks/` may be inside a symlinked directory. Without this option, proper-lockfile resolves symlinks which can break lock targeting.
- **Git 2.30+:** `git worktree add` behavior changed in Git 2.17+ (no longer requires `--checkout`). RAPID assumes the modern behavior. Older Git versions may fail silently with worktree creation.
- **Zod 3.25.76:** The `.default()` method on schema fields (used in state-schemas.cjs for `waves`, `sets`, `jobs`, `status`) works correctly in Zod 3 but has different semantics in Zod 4 regarding `undefined` vs `null`. Stay on the `require('zod')` path.


## Security Pitfalls

1. **Shell injection via set names:** Set IDs are interpolated into bash commands in SKILL.md files (e.g., `"${SET_ID}"`). If a set name contains shell metacharacters, it could cause injection. The Zod schema validates set IDs as `z.string()` with no pattern constraint. Prevention: Add a pattern constraint to SetState.id (e.g., `/^[a-z0-9-]+$/`) or sanitize before shell interpolation.
2. **Lock file PID spoofing:** The `cleanStaleLocks` function reads PID from lock target files and checks `process.kill(pid, 0)`. A malicious actor could write a target file with a valid PID to prevent lock cleanup. Low risk in a single-user development tool.
3. **Arbitrary file read via contract paths:** CONTRACT.json `file` fields reference filesystem paths that are read by skills. If a CONTRACT.json is crafted with paths like `../../.env`, the skill could read and expose sensitive files. Prevention: Validate that contract file paths stay within the project root.


## Performance Traps

1. **`plan.listSets` reads disk on every resolution:** Every call to `resolveSet` triggers `fs.readdirSync` on `.planning/sets/`. In a hot loop (e.g., iterating over all sets to build a status table), this creates unnecessary disk I/O. Mitigation: Cache the listing within a single CLI invocation, or read from STATE.json.
2. **Lock acquisition retry with exponential backoff:** The retry config in lock.cjs uses factor=2 with maxTimeout=2000ms and 10 retries. Total worst-case wait: ~20 seconds (sum of exponential delays). This is fine for serialized operations but can bottleneck parallel agent invocations that all need state access.
3. **`readState` validates via Zod on every read:** Full schema validation (including nested milestone/set/wave/job arrays) runs on every `readState` call. For large state files with many milestones and sets, this adds latency. The current state (8 milestones, 4 sets with no waves/jobs) is trivially fast, but state growth could cause slowdown.
4. **`withStateTransaction` double-validates:** The transaction helper calls `ProjectState.parse(state)` after mutation (line 167), even though `readState` already validated on read (line 59). This is correct for safety but adds overhead. Not a concern at current scale.


## Summary Priority List

1. **Status name mismatch (discussing/discussed, planning/planned, executing/executed)** -- Every skill-to-state transition is broken; root cause of most milestone bugs
2. **Set name resolution uses disk not state** -- Causes wrong set operations after archiving; data corruption risk
3. **`wave-plan` commands do not exist in CLI** -- Contract validation and job listing are completely non-functional in v3
4. **Review skill uses non-existent `reviewing` status** -- Review pipeline cannot start
5. **Discuss-set silently swallows transition errors** -- Masks the status name mismatch, creating downstream confusion
6. **Execute-set uses wrong transition name at start** -- Set execution cannot begin
7. **Compromised lock callback only logs** -- Silent data race under concurrent access
8. **`readState` lacks lock protection** -- TOCTOU race in read-then-mutate patterns
9. **Stale lock 5-minute threshold** -- Poor DX after crashes
10. **Ghost tool-docs entries** -- Agents get hallucination-inducing broken tool references
