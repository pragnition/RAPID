# Pitfalls Research -- Robustness & Fixes (v3.5.0)

## Critical Pitfalls

### Pitfall: Documentation-Code Status Name Mismatch Causes Agent Command Failures
- **Category:** correctness
- **Stack:** RAPID state system (state-schemas.cjs, state-transitions.cjs, CONVENTIONS.md, ARCHITECTURE.md)
- **Description:** The context files that agents read (CONVENTIONS.md, ARCHITECTURE.md) document set statuses as present participles (`discussing`, `planning`, `executing`) while the actual Zod schema and transition map use past tense (`discussed`, `planned`, `executed`). Agents reading the documentation will construct incorrect CLI commands like `state transition set M S discussing` which will fail Zod validation.
- **Trigger:** Any agent reading CONVENTIONS.md line 22 or ARCHITECTURE.md line 42/143 and using those status names in `state transition` commands.
- **Impact:** Critical -- every agent that attempts state transitions based on documented names will fail. This is the root cause of milestone goal #1 (agents use wrong CLI subcommands).
- **Prevention:** Update CONVENTIONS.md and ARCHITECTURE.md to use the actual enum values: `pending`, `discussed`, `planned`, `executed`, `complete`, `merged`. Add a "Status Name Reference" section to core-identity.md so agents always have the correct values.
- **Detection:** `state transition` commands fail with "Unknown status" or "Invalid transition" errors. Search all `.md` files for `discussing|planning` (as status names, not verbs).
- **References:** `src/lib/state-schemas.cjs` line 4: `z.enum(['pending', 'discussed', 'planned', 'executed', 'complete', 'merged'])`; `src/lib/state-transitions.cjs` lines 3-9

### Pitfall: Two-Step Execute Transition Creates Stuck-in-Executed State
- **Category:** correctness
- **Stack:** execute-set skill, state-transitions.cjs, review skill
- **Description:** The execute-set skill performs a two-phase transition: `planned -> executed` (Step 3, before waves run) then `executed -> complete` (Step 6, after verification). If the skill encounters any error between Steps 3 and 6 (wave failure, verification gap, agent timeout, checkpoint), the set remains in `executed` state. The review skill (SKILL.md line 122) requires `complete` status and explicitly rejects `executed`, creating a dead-end where the set can neither be re-executed (no `executed -> executed` self-transition) nor reviewed.
- **Trigger:** Any interruption during execute-set after Step 3 but before Step 6: wave execution failure, agent timeout, user cancellation, checkpoint handoff.
- **Impact:** Critical -- the set becomes stuck. The only recovery is manual STATE.json editing (which violates state management rules) or re-running execute-set which handles the `executed` re-entry case but may re-execute already-completed waves.
- **Prevention:** Either (a) make the review skill accept both `executed` and `complete` statuses, or (b) add a `executed -> executed` self-transition to allow re-entry, or (c) defer the `planned -> executed` transition until after at least one wave completes. Option (a) is the milestone's stated approach.
- **Detection:** `node rapid-tools state get --all` shows sets stuck in `executed` status; review skill outputs "Cannot review set -- current status is 'executed'."
- **References:** `skills/execute-set/SKILL.md` lines 156-162 (Step 3), lines 380-387 (Step 6); `skills/review/SKILL.md` line 122-124

### Pitfall: loadSet() Hard-Fails on Missing DEFINITION.md
- **Category:** correctness
- **Stack:** plan.cjs, worktree.cjs (generateScopedClaudeMd)
- **Description:** `plan.loadSet()` uses `fs.readFileSync()` on `DEFINITION.md` without existence checks (line 202). If DEFINITION.md does not exist in `.planning/sets/{setName}/`, the function throws an unhandled ENOENT error. This cascades to `generateScopedClaudeMd()` (worktree.cjs line 665) which calls `loadSet()`, and to `prepareSetContext()` (execute.cjs line 42). The milestone identifies this as "scoped CLAUDE.md fails due to missing DEFINITION.md."
- **Trigger:** Running `set-init create`, `execute prepare-context`, or `worktree generate-claude-md` before DEFINITION.md has been generated (typically created during plan-set).
- **Impact:** Critical -- blocks execution context preparation entirely. The error message is a raw Node.js ENOENT rather than an actionable RAPID error.
- **Prevention:** Add existence check in `loadSet()` before reading DEFINITION.md. Return a graceful fallback (empty definition string with a warning) rather than throwing. Alternatively, generate a stub DEFINITION.md during `set-init create`.
- **Detection:** Error: `ENOENT: no such file or directory, open '.../DEFINITION.md'` during execution context preparation.
- **References:** `src/lib/plan.cjs` line 202; `src/lib/worktree.cjs` line 665; milestone goal #2

## High-Risk Pitfalls

### Pitfall: Untracked Planning Artifacts on Main Block Merge
- **Category:** correctness
- **Stack:** Git merge, .planning/ directory, merge.cjs
- **Description:** When a set branch is merged into main with `--no-ff`, any `.planning/sets/{setId}/` files that were created on main (e.g., by review, discuss, or plan-set skills running on the main branch context) but never committed will cause git to error with "The following untracked working tree files would be overwritten by merge." This is because the set branch may also contain these files (committed in the worktree), and git refuses to overwrite untracked files.
- **Trigger:** Running `/rapid:review`, `/rapid:discuss-set`, or any skill that writes to `.planning/sets/` while on main, then attempting to merge a set branch that also contains those files.
- **Impact:** High -- merges fail with untracked file errors. Manual intervention required (`git stash -u`, `git clean`, or committing the files).
- **Prevention:** Ensure all planning artifacts are committed in the worktree branch before merge. Add a pre-merge check in the merge skill that detects untracked files in `.planning/sets/{setId}/` on main and either commits or removes them. The merge agent should also `git stash -u` planning files before merge.
- **Detection:** `git merge --no-ff` fails with "untracked working tree files would be overwritten by merge" listing `.planning/` paths.
- **References:** Milestone goal #6; `src/lib/merge.cjs`

### Pitfall: Review File Discovery Fails in Post-Merge Mode
- **Category:** correctness
- **Stack:** review skill, unit-test skill, bug-hunt skill, uat skill
- **Description:** The review skill writes REVIEW-SCOPE.md to `.planning/post-merge/{setId}/` when `--post-merge` is set (SKILL.md line 281). Downstream skills (unit-test, bug-hunt, uat) look for REVIEW-SCOPE.md at `.planning/sets/{setId}/REVIEW-SCOPE.md` in standard mode or `.planning/post-merge/{setId}/REVIEW-SCOPE.md` in post-merge mode. However, if the downstream skill is invoked without `--post-merge` after a post-merge review, it looks in the wrong directory. Additionally, agents spawned by these skills (unit-tester, bug-hunter) may not have the `--post-merge` flag propagated, causing them to search the standard path.
- **Trigger:** Running `/rapid:review {set} --post-merge` then `/rapid:unit-test {set}` (without `--post-merge`), or agents not propagating the flag.
- **Impact:** High -- review pipeline agents can't find their input artifacts. They report "REVIEW-SCOPE.md not found" and halt.
- **Prevention:** Embed the `postMerge` flag inside REVIEW-SCOPE.md metadata (already done via SCOPE-META). Downstream skills should check both paths: first the standard path, then the post-merge path. Alternatively, always write to both locations.
- **Detection:** Unit-test/bug-hunt/uat skills output "REVIEW-SCOPE.md not found for set '{setId}'" even though review was already run.
- **References:** Milestone goal #7; `skills/review/SKILL.md` lines 281-282; `skills/unit-test/SKILL.md` lines 67-68

### Pitfall: AskUserQuestion 4-Option Limit Breaks Discuss Skill Gray Area Selection
- **Category:** correctness
- **Stack:** Claude Code AskUserQuestion tool, discuss-set skill
- **Description:** The AskUserQuestion tool enforces a hard limit of 4 options per question. The discuss-set skill (SKILL.md lines 163-171) presents 5 gray areas as a multi-select with 5 options. This exceeds the tool's constraint and causes the question to either fail silently, truncate options, or error out. The skill also presents per-question options that may exceed 4 if many approaches exist.
- **Trigger:** Every invocation of `/rapid:discuss-set` that identifies 5+ gray areas (common for complex sets).
- **Impact:** High -- the discuss flow either errors out or silently drops the 5th option, leading to incomplete vision capture.
- **Prevention:** Cap gray area selection at 4 options maximum. Either present the top 4 most important gray areas, or batch them into multiple AskUserQuestion calls of 4 options each. Also ensure per-question options within Step 6 never exceed 4.
- **Detection:** AskUserQuestion calls fail or return unexpected results. The discuss skill hangs or produces incomplete CONTEXT.md.
- **References:** Milestone goal #3; [GitHub issue #12420](https://github.com/anthropics/claude-code/issues/12420); `skills/discuss-set/SKILL.md` lines 163-171

### Pitfall: Worktrees Lack Independent node_modules
- **Category:** correctness
- **Stack:** Git worktrees, npm, Node.js, worktree.cjs
- **Description:** When `createWorktree()` (worktree.cjs line 72) creates a new worktree via `git worktree add`, the resulting checkout shares the git repository but gets a fresh working directory. However, `node_modules/` is typically gitignored and not present in the new worktree. No post-creation step runs `npm install` in the worktree. Tests requiring `zod`, `ajv`, or `proper-lockfile` will fail with "Cannot find module" errors.
- **Trigger:** Running tests (`node --test`) in a worktree without first running `npm install` in that worktree.
- **Impact:** High -- test execution fails in worktrees. The executor agent may not realize it needs to install dependencies first, especially since the agent prompt does not mention this requirement.
- **Prevention:** Add an `npm install` step to `createWorktree()` or `setInit()`. Alternatively, symlink node_modules from the project root into the worktree (faster but fragile). The merge agent should also verify that `npm install` has been run before running integration tests.
- **Detection:** Errors like `Error: Cannot find module 'zod'` when running tests in a worktree path.
- **References:** Milestone goal #4; `src/lib/worktree.cjs` lines 72-94

### Pitfall: Zod 3.25.76 Bundles Zod v4 Internals Causing Size and Compatibility Issues
- **Category:** compatibility
- **Stack:** Zod 3.25.76
- **Description:** Zod versions 3.25.68+ include Zod v4 code within the v3 package. This doubles the package size and can cause TypeScript compilation errors from v4 type definitions even when only v3 APIs are used. While this project uses plain JavaScript (not TypeScript), the bundled v4 code increases memory footprint and load time. More importantly, future zod imports may resolve to v4 APIs unexpectedly if `zod/v4` paths are accidentally referenced.
- **Trigger:** Using `zod@^3.25.76` with npm -- the package includes both v3 and v4 code.
- **Impact:** Medium-High -- no immediate breakage in this pure-CJS project, but risk of accidental v4 import and unnecessary bloat. If any agent or code accidentally requires `zod/v4`, behavior diverges silently.
- **Prevention:** Pin zod to `3.25.67` (last version without v4 bundled) or accept the current version and add a lint check to prevent `zod/v4` imports. Document the pinning rationale in package.json.
- **Detection:** Check `node -e "console.log(require('zod/v4'))"` -- if it resolves, v4 is bundled.
- **References:** [GitHub issue #4923](https://github.com/colinhacks/zod/issues/4923); [CJS loading broken #4434](https://github.com/colinhacks/zod/issues/4434)

## Medium-Risk Pitfalls

### Pitfall: proper-lockfile Stale Lock After Agent Crash
- **Category:** data-loss
- **Stack:** proper-lockfile 4.1.2, lock.cjs
- **Description:** If an agent process is killed (SIGKILL, OOM, system crash) while holding a lock, the lock directory (`.target.lock/`) persists. While `cleanStaleLocks()` exists to handle this, it relies on PID checking -- which fails if a new process reuses the dead process's PID. The `onCompromised` handler (lock.cjs line 59) logs but does not abort, meaning a compromised lock allows concurrent writes to STATE.json.
- **Trigger:** Agent crash during `withStateTransaction()`, followed by another agent starting a transaction before stale cleanup runs.
- **Impact:** Medium -- STATE.json corruption is possible if two processes write simultaneously. The 5-minute stale threshold provides a large window where the lock is held by a dead process.
- **Prevention:** Call `cleanStaleLocks()` at the start of every CLI invocation (before any state operation). Reduce the stale threshold from 300s to 60s for faster recovery. Consider making `onCompromised` throw rather than log-and-continue for state locks specifically.
- **Detection:** Multiple agents report "Lock is already being held" errors, or STATE.json contains inconsistent data.
- **References:** [proper-lockfile issue #92](https://github.com/moxystudio/node-proper-lockfile/issues/92); `src/lib/lock.cjs` lines 54-63

### Pitfall: Tool Documentation Registry Diverges From Actual CLI Commands
- **Category:** correctness
- **Stack:** tool-docs.cjs, agent prompt assembly, rapid-tools.cjs
- **Description:** The `TOOL_REGISTRY` in tool-docs.cjs is a static map that must be manually synchronized with the actual CLI commands in rapid-tools.cjs. When new commands are added or signatures change, the tool registry can become stale. Agents receive the tool-docs.cjs descriptions in their prompts, so stale docs cause agents to invoke non-existent subcommands or use wrong argument patterns.
- **Trigger:** Any CLI refactor that changes command names or arguments without updating tool-docs.cjs. For example, `review scope` in tool-docs.cjs takes `<set> <wave>` arguments (line 72) but the actual review scope command may have evolved past wave-level scoping.
- **Impact:** Medium -- agents use wrong command syntax based on stale tool docs. This is a contributing factor to milestone goal #1.
- **Prevention:** Add an automated test that compares TOOL_REGISTRY keys against the actual USAGE string in rapid-tools.cjs. Run this test as part of `build-agents` to catch drift.
- **Detection:** Agents emit "Unknown subcommand" errors from rapid-tools.cjs.
- **References:** `src/lib/tool-docs.cjs` (entire file); `src/bin/rapid-tools.cjs` USAGE string (lines 29-100)

### Pitfall: Solo Mode Config Not Propagated to All Skills
- **Category:** correctness
- **Stack:** config.json, set-init.cjs, skills
- **Description:** The milestone calls for a "project-wide solo mode config" in config.json. Currently, solo mode is per-set via `--solo` flag to `set-init create`. If a global config is added but skills continue to check only the `--solo` CLI flag, the config will be ignored. Skills must be updated to check `config.json` as a fallback when `--solo` is not explicitly passed.
- **Trigger:** Setting `"solo": true` in config.json but running `/rapid:start-set` without `--solo` flag.
- **Impact:** Medium -- sets get created with worktrees when user intended solo mode, wasting disk space and causing confusion when the executor operates on the wrong path.
- **Prevention:** Modify `handleSetInit` in set-init.cjs to read config.json and default to solo if the config specifies it. The `--solo` flag should override config, and `--no-solo` should disable it.
- **Detection:** Worktrees are created for sets when the user expected solo mode.
- **References:** Milestone goal #8; `src/commands/set-init.cjs` line 13

### Pitfall: execSync Git Commands Vulnerable to Path Injection
- **Category:** security
- **Stack:** worktree.cjs (gitExec), child_process
- **Description:** `gitExec()` in worktree.cjs line 22 constructs git commands via string interpolation: `execSync(\`git ${args.join(' ')}\`)`. If any element of `args` contains shell metacharacters (spaces, quotes, semicolons), the command could be misinterpreted or exploited. While set names are validated against whitespace (line 73), other callers of `gitExec()` may pass unvalidated strings.
- **Trigger:** Any `gitExec()` call where args contain shell-special characters, e.g., a branch name with backticks or semicolons.
- **Impact:** Medium -- potential command injection. In practice, set names are validated, but worktree paths with spaces or special characters (e.g., on macOS) could cause failures.
- **Prevention:** Use `execFileSync('git', args)` instead of `execSync(\`git ${args.join(' ')}\`)` to avoid shell interpretation entirely. This is a standard Node.js security best practice.
- **Detection:** Git commands fail with unexpected shell errors when paths contain special characters.
- **References:** `src/lib/worktree.cjs` lines 20-35; Node.js child_process documentation recommends execFileSync for non-shell execution

### Pitfall: Node.js 25 Web Storage Breaks Test Mocks
- **Category:** compatibility
- **Stack:** Node.js v25.8.0, node:test
- **Description:** Node.js 25 introduced Web Storage APIs (localStorage) by default. In test environments, accessing localStorage without a proper origin throws `SecurityError`. While this project uses `node:test` (not Jest), any future test that touches browser-like APIs or uses globalThis properties may encounter unexpected Web Storage interference.
- **Trigger:** Tests that mock or access `globalThis.localStorage` or similar browser globals, or tests that use libraries which feature-detect localStorage.
- **Impact:** Medium -- current tests likely unaffected (pure Node.js/CJS), but adding browser-compat tests or using libraries with localStorage detection could break.
- **Prevention:** If tests need to suppress Web Storage, use `NODE_OPTIONS="--no-experimental-webstorage"`. Document the Node.js 25 requirement in setup instructions.
- **Detection:** Tests fail with `SecurityError: localStorage is not available` or similar.
- **References:** [Node.js 25 release notes](https://nodejs.org/en/blog/release/v25.0.0)

## Anti-Patterns for This Stack

### Anti-Pattern: Documenting Status Names Inconsistently Across Layers
- **What people do:** Write `discussing`, `planning`, `executing` in documentation and agent prompts while the code uses `discussed`, `planned`, `executed`.
- **Why it fails:** Agents construct CLI commands from documentation, so mismatched names cause validation failures. The error messages say "Invalid transition" but do not suggest the correct status name.
- **Correct approach:** Use a single source of truth. Export the status enum values from state-schemas.cjs and reference them in documentation generation. The `build-agents` pipeline should inject the actual status values.
- **Example:** In core-identity.md, add: `Set statuses (exact values): pending, discussed, planned, executed, complete, merged`

### Anti-Pattern: Two-Phase State Transitions Without Rollback
- **What people do:** Transition to an intermediate state early (e.g., `planned -> executed` before execution starts) to "claim" the set, then transition to the final state later.
- **Why it fails:** If the second transition never happens (crash, timeout, error), the entity is stuck in the intermediate state. The transition map has no reverse path (`executed -> planned` is not allowed).
- **Correct approach:** Either (a) use a single atomic transition at the end, (b) add reverse transitions for rollback, or (c) make the intermediate state re-enterable (self-transition). RAPID's approach should be (c) plus making consumers (review skill) tolerant of the intermediate state.
- **Example:** Add `executed: ['complete', 'executed']` to SET_TRANSITIONS to allow re-entry.

### Anti-Pattern: Hardcoding File Paths in Skills Without Cross-Referencing
- **What people do:** Each skill hardcodes paths like `.planning/sets/{setId}/REVIEW-SCOPE.md` independently, without a shared path resolver.
- **Why it fails:** When the path convention changes (e.g., adding post-merge directory), every skill must be updated individually. Missing one causes file-not-found errors.
- **Correct approach:** Centralize path computation in a library function (e.g., `review.getReviewScopePath(setId, { postMerge })`) and have all skills call it. The review CLI command already returns the path -- downstream skills should use it.
- **Example:** `const scopePath = review.getArtifactPath(setId, 'REVIEW-SCOPE.md', { postMerge });`

### Anti-Pattern: Shell String Interpolation for Git Commands
- **What people do:** Build git commands via template strings: `` execSync(`git ${args.join(' ')}`) ``
- **Why it fails:** Shell interpretation of special characters in paths or branch names. Quoted paths inside the interpolated string (line 83: `"${worktreePath}"`) provide some protection but are fragile.
- **Correct approach:** Use `execFileSync('git', args, { cwd })` which bypasses the shell entirely. Arguments are passed as an array, not a string.
- **Example:** `execFileSync('git', ['worktree', 'add', '-b', branch, worktreePath, 'HEAD'], { cwd })`

### Anti-Pattern: Skills Writing Files Without Committing Them
- **What people do:** Skills write planning artifacts (CONTEXT.md, REVIEW-SCOPE.md, etc.) to `.planning/` but leave them uncommitted, expecting a later step to handle the commit.
- **Why it fails:** Untracked files on main block merges. Multiple skills writing to the same set directory create merge conflicts. Artifacts can be lost if the working directory is cleaned.
- **Correct approach:** Every skill that writes to `.planning/` should immediately `git add` and `git commit` the file(s) it creates, with a descriptive commit message. This keeps the working tree clean for subsequent operations.

## Version-Specific Gotchas

- **Node.js v25.8.0:** Non-LTS (Current) release. The `node:test` runner is stable but still evolving. Test isolation behavior may change in minor releases. Pin to a specific Node.js version in CI.
- **Node.js v25.x:** 50+ deprecated APIs removed. If any dependency internally uses `SlowBuffer()` or other removed APIs, it will break at runtime with no deprecation warning.
- **Zod 3.25.76:** The `require('zod')` entry point works for CJS, but `require('zod/v4')` also resolves. Any code completion or AI agent that sees the v4 import path may suggest using it, causing subtle API differences.
- **proper-lockfile 4.1.2:** The library is not actively maintained (last release 2020). Known issues around stale lock detection on systems where PID recycling is fast (containers, CI environments).
- **Git 2.30+:** Worktree pruning behavior changed in recent Git versions. `git worktree prune` may not clean up all expected paths if worktrees were created with absolute paths containing symlinks.

## Security Pitfalls

1. **Shell injection via gitExec():** `execSync` with string interpolation allows shell metacharacter injection. Any user-controlled input reaching `gitExec()` args could execute arbitrary commands. -- Prevention: switch to `execFileSync`.
2. **Lock target file contains PID:** The `.target` file written by `acquireLock()` contains the process PID. While not directly exploitable, it leaks process information. On shared systems, this could aid in targeted attacks. -- Prevention: minimal concern for a local development tool.
3. **STATE.json is world-readable:** No file permission restrictions on `.planning/STATE.json`. On shared development machines, other users could read project state. -- Prevention: not a concern for typical single-user development.
4. **Unvalidated stdin JSON in state commands:** Several CLI commands parse JSON from stdin (e.g., `plan create-set`, `state add-milestone`). Malformed JSON could cause unhandled exceptions. Zod validation catches schema issues but not JSON parse failures. -- Prevention: wrap JSON.parse in try-catch with descriptive error messages.

## Performance Traps

1. **Full STATE.json read for every CLI command:** `readState()` parses and Zod-validates the entire state tree for every command, even those that only need a single set's status. For projects with many milestones and sets, this becomes expensive. -- Mitigation: add a `readStateRaw()` that skips validation for read-only queries, or cache validated state in a process-level variable.
2. **Synchronous git operations block the event loop:** `gitExec()` uses `execSync` with a 30-second timeout. During this time, the Node.js process is completely blocked. Multiple sequential git operations (e.g., conflict detection across all levels) can cause the CLI to appear frozen for minutes. -- Mitigation: use async `execFile` for non-critical git operations.
3. **Lock contention with multiple parallel agents:** When multiple agents modify state concurrently, the exponential retry backoff (100ms to 2000ms, 10 retries) can cause agents to wait up to ~30 seconds for a lock. If many agents are waiting, the total serialization delay compounds. -- Mitigation: reduce transaction scope; batch related state changes into fewer transactions.
4. **generateScopedClaudeMd reads OWNERSHIP.json on every call:** Each invocation re-reads and re-parses OWNERSHIP.json from disk. For execution flows that prepare context for multiple sets, this is redundant I/O. -- Mitigation: pass ownership data as a parameter or cache at the session level.

## Summary Priority List

1. **Documentation-code status name mismatch** -- Agents construct commands with `discussing`/`planning`/`executing` but code expects `discussed`/`planned`/`executed`. Root cause of widespread agent CLI failures.
2. **Stuck-in-executed state** -- Execute-set transitions to `executed` early; if completion step fails, review skill rejects the set. No recovery path without manual intervention.
3. **Missing DEFINITION.md crashes loadSet()** -- Hard ENOENT error with no graceful fallback blocks execution context prep for sets that haven't been fully planned.
4. **AskUserQuestion 4-option limit** -- Discuss skill presents 5 gray area options, exceeding the hard constraint. Breaks the discussion flow.
5. **Untracked planning files block merges** -- Skills write to `.planning/` on main without committing; these untracked files prevent `git merge --no-ff` from the set branch.
6. **Worktrees missing node_modules** -- No `npm install` step after worktree creation; tests fail with missing module errors.
7. **Review file discovery path mismatch** -- Post-merge review writes to a different directory than standard review; downstream skills may look in the wrong location.
8. **Tool docs registry drift** -- Static TOOL_REGISTRY diverges from actual CLI commands, causing agents to use wrong subcommands.
9. **Solo mode config not plumbed through** -- Global config.json solo setting has no effect unless skills explicitly read it.
10. **proper-lockfile stale lock recovery** -- Crashed agent locks persist for 5 minutes; PID reuse can prevent detection.
