# Features Research: Robustness & Fixes (v3.5.0)

## Feature Inventory

### Feature 1: Fix Agent Command Awareness

- **Description:** Agents attempt non-existent CLI subcommands (`state set-status`, `state --help`, `state update-set`). Sweep all agents and skills to ensure they use correct command syntax.
- **Complexity:** moderate
- **Dependencies:** independent

#### Decomposition

1. **Audit agent tool documentation** -- Examine the `core-identity.md` module and all 30 role modules in `src/modules/roles/` for any CLI command references that don't match the actual `rapid-tools.cjs` USAGE string.
2. **Audit all 24 skills** -- Check every `skills/*/SKILL.md` for CLI command invocations against the canonical command list.
3. **Create a CLI command cheat sheet** -- Add a concise command reference to `core-identity.md` so all agents inherit correct syntax at build time.
4. **Rebuild agents** -- Run `build-agents` to propagate the updated core-identity module.

#### Implementation Strategies

**Option 1: Embed command reference in core-identity.md**
- Approach: Add a "## CLI Command Reference" section to `src/modules/core/core-identity.md` listing all valid `state`, `worktree`, `execute`, `merge`, `review`, `plan` subcommands with examples. Since core-identity.md is assembled into every agent, all 26 agents automatically inherit correct syntax.
- Pros: Single source of truth; automatically propagated to all agents via build-agents; low maintenance.
- Cons: Increases agent prompt size (~500 tokens). May need periodic manual updates when new commands are added.
- Libraries/services: None.

**Option 2: Generate tool docs from CLI USAGE at build time**
- Approach: Parse the USAGE string from `rapid-tools.cjs` programmatically during `build-agents` and inject a generated command reference section.
- Pros: Always in sync with actual CLI; zero manual maintenance.
- Cons: Requires new build step; USAGE string format must remain parseable; more complex implementation.
- Libraries/services: None.

**Recommended:** Option 1 for immediate fix, with Option 2 as a future improvement. The agent prompt already includes tool-docs (YAML-based), so a manual command reference in core-identity.md is the fastest path to correctness. The commands rarely change, so manual sync is acceptable.

#### Data Model Impact
- No new entities.
- No state changes.
- Modified files: `src/modules/core/core-identity.md`, potentially `src/lib/tool-docs.cjs`.

#### API Surface
- No new endpoints.
- No existing endpoints modified.

### Feature 2: Fix DEFINITION.md Generation

- **Description:** `start-set` fails with "missing DEFINITION.md" because the init skill writes only `CONTRACT.json` per set but never creates `DEFINITION.md`. The `loadSet()` in `plan.cjs` line 202 reads DEFINITION.md and throws if missing. `generateScopedClaudeMd()` calls `loadSet()`, which cascades the failure into scoped CLAUDE.md generation.
- **Complexity:** moderate
- **Dependencies:** independent

#### Root Cause Analysis

The init skill's Step 9 (roadmap acceptance, lines 618-636) writes:
- a) `ROADMAP.md` via Write tool
- b) `CONTRACT.json` for each set via Write tool
- c) `STATE.json` via Write tool

It does NOT call `plan create-set` or `plan.createSet()`, which is the function that generates `DEFINITION.md` (plan.cjs line 97-133). The roadmapper agent returns `{ roadmap, state, contracts }` but the init skill only writes CONTRACT.json from the contracts array -- it skips creating DEFINITION.md.

Meanwhile, `plan.createSet()` generates DEFINITION.md from a `setDef` object with fields: `name`, `scope`, `ownedFiles`, `tasks`, `acceptance`, `wave`, `parallelWith`, `contract`. But the roadmapper's output does not produce this structure (it outputs sets-only with no wave/task breakdown).

#### Decomposition

1. **Add DEFINITION.md writing to init skill** -- After writing CONTRACT.json, also write a minimal DEFINITION.md for each set using available data from the roadmapper output (set name, scope from ROADMAP.md, contract summary).
2. **Make loadSet() resilient to missing DEFINITION.md** -- Graceful fallback in `plan.cjs` so that generateScopedClaudeMd does not crash.
3. **Add DEFINITION.md validation to set-init create** -- Pre-check that DEFINITION.md exists; if not, auto-generate a minimal one from CONTRACT.json and ROADMAP.md scope.

#### Implementation Strategies

**Option 1: Fix init skill to write DEFINITION.md**
- Approach: Modify `skills/init/SKILL.md` Step 9 to write a minimal `DEFINITION.md` for each set alongside CONTRACT.json. Extract set scope from the roadmapper's roadmap markdown (parsed per-set section). Use a template:
  ```markdown
  # Set: {setId}
  ## Scope
  {scope from roadmap}
  ## File Ownership
  Files this set owns (exclusive write access):
  - (to be determined during planning)
  ## Tasks
  (to be determined during planning)
  ## Interface Contract
  See: CONTRACT.json (adjacent file)
  ```
- Pros: Fixes the root cause; DEFINITION.md exists from initialization onward.
- Cons: Content is minimal until plan-set enriches it.
- Libraries/services: None.

**Option 2: Make loadSet() graceful when DEFINITION.md is missing**
- Approach: Modify `plan.cjs loadSet()` to return a synthetic definition string when DEFINITION.md is absent (derive from CONTRACT.json). Similarly fix `generateScopedClaudeMd()` to handle missing definition gracefully.
- Pros: Defensive; works regardless of init order.
- Cons: Does not fix the upstream problem; DEFINITION.md should exist as a planning artifact.
- Libraries/services: None.

**Recommended:** Both options combined. Option 1 fixes the init flow so DEFINITION.md is always created. Option 2 adds defensive programming so downstream functions never crash even if DEFINITION.md is somehow missing (belt and suspenders).

#### Data Model Impact
- New file per set: `.planning/sets/{setId}/DEFINITION.md` (created during init).
- No schema changes.

#### API Surface
- Modified: `skills/init/SKILL.md` (Step 9 file writes).
- Modified: `src/lib/plan.cjs` (`loadSet()` graceful fallback).

### Feature 4: Worktree Package Management

- **Description:** Worktrees need to be able to run tests independently. Currently, `node_modules` (or equivalent package directory) may not be present in worktrees since git worktrees share the same git objects but have separate working trees. The merge agent should verify packages are installed before running integration tests.
- **Complexity:** moderate
- **Dependencies:** independent

#### Decomposition

1. **Detect package manager in worktree** -- After worktree creation, detect which package manager the project uses (npm, yarn, pnpm, uv, pip, cargo, etc.) via lockfile detection.
2. **Install packages in worktree post-creation** -- Run the appropriate install command after `createWorktree()`.
3. **Add package verification to merge agent** -- Before integration tests in the merge pipeline, verify packages are installed on main.

#### Implementation Strategies

**Option 1: Post-creation hook in set-init**
- Approach: After worktree creation in `worktree.cjs setInit()`, detect the project's package manager by checking for lockfiles (`package-lock.json` -> npm, `yarn.lock` -> yarn, `pnpm-lock.yaml` -> pnpm, `uv.lock` -> uv, `requirements.txt` -> pip, `Cargo.lock` -> cargo). Run the install command (`npm install`, `uv sync`, etc.) in the worktree directory. Add a `packagesInstalled` field to the return object. Also add a `merge integration-test` pre-step that runs package install.
- Pros: Ensures every worktree is test-ready from creation; straightforward detection logic.
- Cons: Adds time to worktree creation; needs to handle install failures gracefully.
- Libraries/services: None (uses child_process.execSync).

**Option 2: Symlink node_modules from project root**
- Approach: After worktree creation, create a symlink from `{worktreePath}/node_modules` -> `{projectRoot}/node_modules`.
- Pros: Instant, no install time. Saves disk space.
- Cons: Only works for npm/yarn/pnpm. Breaks if packages have native bindings with different build dirs. Does not work for Python (uv/pip). Fragile.
- Libraries/services: None.

**Recommended:** Option 1. Symlinking is fragile and language-specific. Running the actual package install is the robust approach. The detection logic is ~30 lines and handles the major ecosystems.

#### Data Model Impact
- New optional field in REGISTRY.json entries: `packagesInstalled: boolean`.
- No state schema changes.

#### API Surface
- Modified: `src/lib/worktree.cjs` (`setInit()`, `setInitSolo()` -- add post-install step).
- Modified: `src/commands/worktree.cjs` (expose install result).
- Modified: `src/lib/merge.cjs` or `skills/merge/SKILL.md` (add pre-test install step).

### Feature 5: Fix Execution-to-Complete State Transition

- **Description:** The executor frequently fails to transition the set from `executed` to `complete` because (a) the shell CWD drifts to the worktree path and state CLI reads the worktree's STATE.json instead of the main project's, and (b) the review agent refuses to operate on sets in `executed` status, only accepting `complete`. The user reports that the `complete` state is "kind of irrelevant" and the review agent should accept `executed`.
- **Complexity:** moderate
- **Dependencies:** independent

#### Root Cause Analysis

Per `rapid-self-todo.md`, the CWD sensitivity issue is well-documented:
1. `state transition set` uses `findProjectRoot()` which resolves from CWD.
2. After executing waves in the worktree, the orchestrator's CWD may be inside `.rapid-worktrees/{set}/`.
3. The worktree has its own copy of `.planning/STATE.json` (from the branch), which may be stale.
4. The `plan.cjs resolveProjectRoot()` function resolves the git-common-dir to find the true project root, but `state-machine.cjs` may not be using it.

Additionally, the review skill (line 122-126) requires `complete` status and rejects `executed`.

#### Decomposition

1. **Fix execute-set skill CWD management** -- Ensure the skill explicitly changes back to project root before running state transitions (Step 3 and Step 6).
2. **Accept `executed` status in review skill** -- Allow review to operate on sets in either `executed` or `complete` status.
3. **Add CWD guard to state CLI** -- Use `resolveProjectRoot()` in state commands to always find the correct STATE.json, regardless of CWD.

#### Implementation Strategies

**Option 1: Fix CWD in execute-set + accept executed in review**
- Approach:
  - In `skills/execute-set/SKILL.md`, add explicit `cd` to project root before state transitions in Steps 3 and 6. Use the env preamble pattern: determine project root at skill start and use absolute paths.
  - In `skills/review/SKILL.md` Step 0c (line 122), change the status check from `must be 'complete'` to `must be 'complete' or 'executed'`.
  - In the state CLI handler, use `resolveProjectRoot()` from plan.cjs instead of raw CWD for finding STATE.json.
- Pros: Fixes both the CWD issue and the review gating issue. Belt and suspenders approach.
- Cons: Requires changes to two skills and one library module.
- Libraries/services: None.

**Option 2: Make state transition CWD-resilient at the library level**
- Approach: Modify `state-machine.cjs` to use `resolveProjectRoot()` for all state operations. This fixes the root cause for all callers, not just execute-set.
- Pros: Universal fix; no per-skill patches needed.
- Cons: Changing a core library function has broader testing implications.
- Libraries/services: None.

**Recommended:** Both. Option 2 fixes the library-level root cause. Option 1 (review accepting `executed`) addresses the user's explicit request to make the review flow more lenient. The execute-set CWD fix in Option 1 is redundant with Option 2 but provides an extra safety net at the skill layer.

#### Data Model Impact
- No data model changes.
- State transition map already allows `executed -> complete` (state-transitions.cjs line 7).

#### API Surface
- Modified: `src/lib/state-machine.cjs` (use resolveProjectRoot).
- Modified: `skills/review/SKILL.md` (accept `executed` status).
- Modified: `skills/execute-set/SKILL.md` (explicit CWD management).

### Feature 6: Fix Merge Untracked File Handling

- **Description:** Planning artifacts (WAVE-COMPLETE.md, PLAN-DIGEST.md) created during execution on both main and the worktree branch cause merge failures ("Untracked files on main would be overwritten by the merge"). Sets should commit planning docs in the worktree before merge.
- **Complexity:** moderate
- **Dependencies:** Feature 5 (relies on state transition fixing)

#### Root Cause Analysis

From `todo-new.md` lines 41-66: During execute-set, the orchestrator creates WAVE-COMPLETE.md and PLAN-DIGEST.md files in `.planning/sets/{setId}/` on main (since the orchestrator runs on main). The same files may also be committed on the worktree branch. When merge runs, git sees these as untracked files on main that conflict with the incoming branch.

The execute-set skill's Step 6 does `git add ".planning/sets/${SET_ID}/WAVE-*-COMPLETE.md"` but this runs on main after execution -- if the CWD is wrong or the files were already created on the worktree branch, the merge encounters conflicts.

#### Decomposition

1. **Commit planning artifacts in the worktree** -- During execute-set, write WAVE-COMPLETE.md markers and PLAN-DIGEST.md into the worktree's branch, not on main.
2. **Clean up untracked planning artifacts before merge** -- In the merge skill/library, detect and remove untracked planning files on main before attempting the merge.
3. **Update execute-set Step 6** -- Commit completion artifacts into the worktree branch instead of main.

#### Implementation Strategies

**Option 1: Write and commit artifacts in worktree**
- Approach: Modify `skills/execute-set/SKILL.md` Steps 4c and 6 to write WAVE-COMPLETE.md, PLAN-DIGEST.md, GAPS.md, and STATE.json commits inside the worktree (`git -C {worktreePath} add ...` and `git -C {worktreePath} commit ...`). The state transition still runs from the project root. Planning artifacts are thus committed on the set's branch and arrive on main cleanly during merge.
- Pros: Clean separation; merge sees artifacts as part of the branch; no untracked file conflicts.
- Cons: Requires careful CWD management in execute-set; state transition must still happen from project root.
- Libraries/services: None.

**Option 2: Pre-merge cleanup of untracked files**
- Approach: Add a pre-merge step in `skills/merge/SKILL.md` (or `src/lib/merge.cjs mergeSet()`) that detects untracked `.planning/` files that would conflict and removes them from main before merging. Run `git clean -f .planning/sets/{setId}/` before the merge.
- Pros: No changes to execute-set; handles the problem at merge time.
- Cons: Destructive operation (removes files); does not fix the upstream cause; cleanup may remove files the user wants to keep.
- Libraries/services: None.

**Recommended:** Option 1 as the primary fix (commit artifacts in worktree). Option 2 as a safety net in the merge pipeline for cases where untracked files still appear (e.g., from manual operations or older runs).

#### Data Model Impact
- No schema changes.
- Artifact file locations remain the same; only the branch they are committed to changes.

#### API Surface
- Modified: `skills/execute-set/SKILL.md` (Steps 4c, 6 -- commit in worktree).
- Modified: `skills/merge/SKILL.md` or `src/lib/merge.cjs` (pre-merge untracked file cleanup).

### Feature 7: Fix Review File Discovery (Post-Merge Mode)

- **Description:** The review skill creates files at `.planning/post-merge/{setId}/REVIEW-SCOPE.md` in post-merge mode, but the downstream skills (unit-test, bug-hunt, uat) cannot find these files because they look in the wrong location or the `--post-merge` flag is not properly propagated.
- **Complexity:** trivial
- **Dependencies:** independent

#### Root Cause Analysis

Examining the skill files:
- `skills/review/SKILL.md` line 281: writes to `.planning/post-merge/{setId}/REVIEW-SCOPE.md` in post-merge mode.
- `skills/unit-test/SKILL.md` lines 67-68: correctly checks both paths based on `POST_MERGE` flag.
- `skills/bug-hunt/SKILL.md` lines 67-68: correctly checks both paths.
- `skills/uat/SKILL.md` lines 67-68: correctly checks both paths.

All three downstream skills document the correct path logic. The likely issue is that the `--post-merge` flag is not being passed when the merge skill suggests running review (line 389: `"/rapid:review {setIndex} --post-merge"`). The user may run downstream skills without `--post-merge`, causing them to look in the standard path.

The fix is to make the downstream skills auto-detect post-merge mode by checking if the standard-path file exists first, and falling back to the post-merge path, rather than requiring an explicit flag.

#### Decomposition

1. **Add auto-detection to unit-test, bug-hunt, uat skills** -- Check both paths and use whichever exists, with explicit flag as override.
2. **Improve post-merge mode messaging** -- Ensure the review completion banner clearly states that downstream commands need `--post-merge`.

#### Implementation Strategies

**Option 1: Auto-detect scope file location**
- Approach: In each of the 3 downstream skills (unit-test, bug-hunt, uat), modify Step 1 to:
  1. First check standard path `.planning/sets/{setId}/REVIEW-SCOPE.md`
  2. If not found, check `.planning/post-merge/{setId}/REVIEW-SCOPE.md`
  3. If found in post-merge path, auto-set `POST_MERGE=true`
  4. If neither exists, show error
  5. If `--post-merge` flag is explicitly set, skip auto-detection and go directly to post-merge path
- Pros: User-friendly; no flag needed; backward compatible.
- Cons: Slightly more complex logic in 3 skills.
- Libraries/services: None.

**Recommended:** Option 1. Auto-detection is a much better UX than requiring the user to remember the `--post-merge` flag.

#### Data Model Impact
- No data model changes.

#### API Surface
- Modified: `skills/unit-test/SKILL.md` (Step 1 auto-detection).
- Modified: `skills/bug-hunt/SKILL.md` (Step 1 auto-detection).
- Modified: `skills/uat/SKILL.md` (Step 1 auto-detection).
