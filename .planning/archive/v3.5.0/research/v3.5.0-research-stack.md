No ESM migration is warranted for this milestone.

### Zod (3.25.76 -> 4.3.6 available)

**Current usage (confirmed via codebase grep):** 5 files import Zod:
- `src/lib/state-schemas.cjs` -- ProjectState hierarchy with `.passthrough()`, `.default()`, `z.enum()`
- `src/lib/returns.cjs` -- RAPID:RETURN marker parsing
- `src/lib/hooks.cjs` -- Hook configuration validation
- `src/lib/review.cjs` -- ReviewIssue schema, `z.record(z.string(), z.string())`
- `src/lib/merge.cjs` -- MergeState schema, `z.record()` with two args

**Zod 4 breaking changes that would affect this codebase:**
1. `.passthrough()` deprecated -- replace with `z.looseObject()` (used 5 times in state-schemas.cjs)
2. `z.record()` requires two arguments -- already compliant (review.cjs and merge.cjs pass 2 args)
3. Error customization APIs changed (`message` -> `error`) -- not used in current codebase
4. `.default()` behavior inside `.optional()` changed -- used in state-schemas.cjs but always at top level, likely safe
5. **CJS compatibility:** Zod 3.25.7 broke CJS, fixed in 3.25.17+. Zod 4 has reported CJS issues (GitHub #4372, #4434). Current 3.25.76 is stable for CJS.

**Recommendation:** Do NOT upgrade to Zod 4 in this milestone. Current 3.25.76 is stable and fully compatible. Zod 4 CJS support has unresolved issues. Upgrade can be considered in a future dedicated milestone.

### proper-lockfile (4.1.2)

**Maintenance status:** Effectively unmaintained. Last npm release was ~5 years ago. 15 open issues, 6 open PRs on GitHub (moxystudio/node-proper-lockfile). Travis CI configuration suggests pre-2020 CI practices.

**Transitive dependencies:**
- graceful-fs 4.2.11 (active)
- retry 0.12.0 (stable, minimal surface)
- signal-exit 3.0.7 (stable)

**Current usage:** Used via `src/lib/lock.cjs` for file-based distributed locking of `STATE.json` and `REGISTRY.json`. Critical for concurrent agent access.

**Risk assessment:** The library works correctly on Node 25 today. No known CVEs. However, if a Node.js core fs API changes break graceful-fs compatibility in a future Node version, there is no active maintainer to issue a fix.

**Recommendation:** Monitor but do not replace in this milestone. The locking behavior is well-tested and functional. If a replacement becomes necessary, `lockfile` (npm) or a custom mkdir-based lock using Node's native `fs.mkdirSync` with `{recursive: false}` (throws on existing) are alternatives.

### ajv (8.18.0) + ajv-formats (3.0.1)

Both are at their latest versions. ajv is actively maintained (Tidelift-backed, last release Feb 2026). ajv-formats has not had a release in ~2 years but is feature-stable. No action needed.

This is a confirmed gap that Goal #4 addresses.

## Stack Risks

Prioritized by relevance to this milestone's goals:

### 1. Worktree node_modules gap (HIGH -- Goal #4)

**Risk:** `createWorktree()` creates a git worktree but does not install npm dependencies. Agents operating in worktrees cannot run tests (`node --test`) because `require('zod')` fails.

**Impact:** Test verification during execution and review phases fails silently or with confusing errors.

**Mitigation:** Add `npm install` or `npm ci` step to `setInit()` / `setInitSolo()`. Merge agent should verify packages are installed before running integration tests. Consider symlinking `node_modules/` from project root to worktree as a faster alternative to full install.

**Stack concern:** `npm install` in a worktree will create a separate `node_modules/` directory. With only 4 production dependencies (total installed size is small), this is acceptable. However, if the target project has many dependencies, `npm ci` could be slow. Symlink approach avoids this.

### 2. Missing `engines` field in package.json (LOW)

**Risk:** No declared Node.js version constraint. Users on unsupported Node versions get cryptic errors.

**Impact:** Low -- setup.sh validates at install time. Runtime is unguarded.

**Mitigation:** Add `"engines": {"node": ">=20"}` to package.json.

### 3. proper-lockfile maintenance (LOW -- monitor)

**Risk:** Unmaintained dependency. If a future Node.js version breaks `graceful-fs` or `fs` internals, no upstream fix will come.

**Impact:** Low today. Would become high if Node breaks compatibility.

**Mitigation:** Monitor. If needed, the locking mechanism can be reimplemented using `fs.mkdirSync({recursive: false})` which throws `EEXIST` atomically.

### 4. Zod 4 eventual migration (LOW -- future)

**Risk:** Zod 3.x will eventually stop receiving updates as Zod 4 matures.

**Impact:** None today. Zod 3.25.76 is stable and complete for current needs.

**Mitigation:** Track Zod 4 CJS stability. Plan migration in a future milestone when Zod 4 CJS support is confirmed stable.

### 5. AskUserQuestion 4-option constraint (MEDIUM -- Goal #3)

**Risk:** This is a Claude Code platform constraint, not a dependency issue, but it has stack implications. The `AskUserQuestion` tool enforces a maximum of 4 options per question. The discuss skill currently templates 5 options (lines 166-171 of `skills/discuss-set/SKILL.md`).

**Impact:** The 5th option silently falls back to text input, breaking the structured decision workflow.

**Mitigation:** Reduce gray area presentation to 4 options maximum. This is a skill template fix, not a code fix. Confirmed: the platform limit is 4 options per question and cannot be changed.
