# Research Summary -- v3.5.0 Robustness & Fixes

## Project Profile
- **Type:** brownfield plugin (Claude Code CLI plugin)
- **Stack:** Node.js (v25.8.0 Current, >=20 recommended) + CommonJS + Git worktrees
- **Scale:** medium (9 features, 24 skills, 26 agents, 21 library modules)
- **Key Constraint:** All fixes are boundary-condition robustness improvements within an architecturally sound 5-layer system -- no structural refactoring required.

## Unified Findings

### Stack and Infrastructure

- **Node.js v25.8.0 (Current) in use; no `engines` field in package.json.** setup.sh enforces Node 18+ at install time but runtime is unguarded. Should add `"engines": {"node": ">=20"}` to package.json. (sources: STACK, OVERSIGHTS)
- **Zod 3.25.76 is stable for CJS; do NOT upgrade to Zod 4 in this milestone.** Zod 4 has unresolved CJS loading issues (#4372, #4434). Zod 3.25.76 bundles Zod v4 internals (doubling package size) but this causes no runtime issues in pure CJS. Consider pinning to 3.25.67 to avoid accidental `zod/v4` imports. (sources: STACK, PITFALLS)
- **proper-lockfile 4.1.2 is unmaintained (~5 years since last release).** Works correctly on Node 25 today. Stale lock recovery relies on PID checking which can fail on PID reuse. The `onCompromised` handler logs but does not abort, allowing concurrent writes. (sources: STACK, PITFALLS)
- **ajv 8.18.0 and ajv-formats 3.0.1 are up-to-date.** No action needed. (source: STACK)
- **No build step, no linting, no CI/CD.** All source is directly executable CJS. Appropriate for this project but increases regression risk. (source: STACK)
- **49 test files (~23,700 lines) using node:test.** Excellent library-level coverage but no integration tests for the full pipeline. Skills (markdown prompt files) have no automated testing. (sources: STACK, OVERSIGHTS)
- **Git 2.53.0 in use; worktree operations require Git 2.30+.** No compatibility issues detected. (source: STACK)

### Feature Landscape

Nine features organized into natural groupings based on file ownership and dependency analysis:

**Critical bug fixes (blocks core workflow):**
1. **F2: DEFINITION.md Generation** -- init skill writes CONTRACT.json but never creates DEFINITION.md; `loadSet()` hard-fails with ENOENT. Blocks `start-set` for all sets. (source: FEATURES)
2. **F5: Execution-to-Complete State Transition** -- Two-step transition creates stuck-in-executed state; review skill rejects `executed` status. CWD sensitivity causes state CLI to read wrong STATE.json. (source: FEATURES)
3. **F1: Agent Command Awareness** -- Agents hallucinate CLI subcommands (`state set-status`, `state --help`). Root cause: core-identity.md lacks a command reference. Contributing factor: documentation uses wrong status names (`discussing` vs `discussed`). (sources: FEATURES, PITFALLS, OVERSIGHTS)

**High-priority fixes (blocks secondary workflows):**
4. **F3: Discuss AskUserQuestion Limit** -- Template lists 5 gray area options; AskUserQuestion hard limit is 4. Breaks discuss-set flow. (source: FEATURES)
5. **F6: Merge Untracked File Handling** -- Planning artifacts committed on main block `git merge --no-ff` from worktree branch. (source: FEATURES)
6. **F7: Review File Discovery** -- Post-merge review writes to `.planning/post-merge/{setId}/` but downstream skills may not detect post-merge mode without explicit flag. (source: FEATURES)

**Medium-priority enhancements:**
7. **F4: Worktree Package Management** -- `createWorktree()` does not install dependencies; tests fail with missing module errors. Needs ecosystem-aware detection (npm/yarn/pnpm/uv/pip/cargo). (source: FEATURES)
8. **F8: Solo Mode Config** -- Add `solo` field to `.planning/config.json` so users do not pass `--solo` on every `/start-set`. (source: FEATURES)

**Low-priority refinement:**
9. **F9: UI/UX Emphasis** -- Strengthen UI/UX gray area guidance in discuss-set; add UI/UX section to plan-set wave templates. (source: FEATURES)

**Feature dependency graph:** All features are independent except F6 which benefits from F5's CWD fix being in place first.

**Third-party services:** None required. All features are internal fixes and configuration changes. (source: FEATURES)

### Architecture Direction

- **Recommended pattern: Defensive middleware with graceful degradation.** All nine issues are boundary-condition failures, not structural deficiencies. Each fix should add a defensive check at the point of failure while preserving the existing 5-layer architecture (Skill -> Agent -> CLI -> Library -> Storage). (source: ARCHITECTURE)
- **No structural changes needed.** The fixes operate within the existing layout. The 5-layer architecture is sound for its purpose. (source: ARCHITECTURE)
- **Transactional state management is a strength.** All mutations flow through `withStateTransaction()` with Zod validation. The state schema hierarchy (ProjectState > milestones > sets > waves > jobs) is well-defined. (source: ARCHITECTURE)
- **CLI as single source of truth is correct.** The "wrong CLI command" problem is a prompt engineering issue, not an architecture issue. (source: ARCHITECTURE)
- **Artifact-based coordination is architecturally correct** but has path resolution gaps in post-merge mode. (source: ARCHITECTURE)
- **Two-branch commit model is a foundational constraint.** State transitions (STATE.json) live on main; code changes live on the worktree branch. Planning artifacts are currently split between both, causing merge conflicts. The fix: commit all `.planning/sets/{SET_ID}/` artifacts in the worktree branch, keep only STATE.json transitions on main. (sources: ARCHITECTURE, OVERSIGHTS)

### Cross-Cutting Requirements

**Must address early (high retrofit cost):**
- **Agent CLI command catalog in core-identity.md** -- Without a systematic mechanism, every CLI command change requires auditing all 26 agents. A one-time sweep will rot within one milestone. The build-agents pipeline should propagate this automatically. (sources: OVERSIGHTS, ARCHITECTURE, PITFALLS)
- **Two-branch commit model for planning artifacts** -- Deciding whether WAVE-COMPLETE.md markers, PLAN-DIGEST.md files live on main or the worktree branch is a foundational choice that every other fix must account for. (sources: OVERSIGHTS, ARCHITECTURE)
- **State transition resilience** -- The `executed` vs `complete` distinction is artificial; understanding this informs multiple fixes. (sources: OVERSIGHTS, ARCHITECTURE, PITFALLS)

**Can address later (low retrofit cost):**
- **Solo mode config** -- Config plumbing; each consuming skill needs a small config check added. (source: OVERSIGHTS)
- **UI/UX emphasis in discuss/plan** -- Prompt engineering in skills, no code changes. (source: OVERSIGHTS)
- **Error recovery strategy** -- Cross-cutting but most fixes are additive (better error messages, retry logic). (source: OVERSIGHTS)
- **NO_COLOR support in display.cjs** -- Terminal accessibility improvement per clig.dev guidelines. (source: UX)
- **Testing alongside each fix** -- Tests should be added with each fix, not as a separate effort. (source: OVERSIGHTS)

### User Experience Direction

- **Domain positioning:** RAPID operates in the emerging specification-driven development (SDD) space alongside Kiro, GitHub Spec Kit, and worktree-workflow. Its unique vocabulary (sets, waves, contracts, DAG) requires onboarding compared to competitors that use simpler terminology. (source: UX)
- **Idempotent commands are a core user expectation.** Users expect safe re-runs of any command. The stuck-in-executed state violates this expectation. RAPID's artifact-based re-entry (WAVE-COMPLETE.md markers) partially addresses this but needs the state transition fix. (source: UX)
- **AskUserQuestion platform constraint (4 options max)** is non-negotiable. The discuss-set template has 5 options. The Key Principles section says "2-5 gray areas" but must be changed to "2-4." (sources: UX, PITFALLS, OVERSIGHTS)
- **CLI command correctness is a trust issue.** When agents reference wrong commands and users read agent output, they cannot reproduce the commands -- this is trust-breaking. (source: UX)
- **Progress indication follows domain conventions.** RAPID uses X-of-Y counters and stage banners, which are appropriate. No changes needed here. (source: UX)
- **Solo mode should be the default for single-developer projects.** Users expect a "simple mode" that avoids unnecessary complexity. (source: UX)
- **Worktree package availability is a basic user expectation.** Users expect `npm test` to work in a worktree immediately after creation. Symlink as default with full install as fallback is the recommended pattern from the worktree-workflow ecosystem. (source: UX)
- **UI/UX gray areas should be explicitly surfaced in discuss-set** for sets with user-facing components. The SDD domain convention surfaces three categories: technical architecture (well-covered), UX/interaction design (gap), and quality/testing (partially covered). (source: UX)
- **Color usage should respect NO_COLOR.** RAPID always outputs ANSI colors with no NO_COLOR environment variable support. The clig.dev guidelines recommend supporting NO_COLOR for accessibility. (source: UX)

## Contradictions

### Contradiction: Worktree Package Strategy (symlink vs. full install)
- **STACK says:** "Consider symlinking `node_modules/` from project root to worktree as a faster alternative to full install."
- **FEATURES says:** "Symlinking is fragile and language-specific. Running the actual package install is the robust approach."
- **UX says:** "Symlink as default with an `npm install` fallback if the worktree adds new dependencies."
- **Resolution:** The research is split. FEATURES recommends full install for robustness; STACK and UX lean toward symlink for speed. The practical resolution is ecosystem-dependent: for Node.js (the primary ecosystem), symlink is fast and works for RAPID's 4 dependencies; for Python/uv, full install is necessary because virtual environments cannot be shared. **Recommend full install as default** (as FEATURES suggests), since it works across all ecosystems, with symlink as an optional optimization flag for Node.js projects.

### Contradiction: Zod Version Pinning
- **STACK says:** "Do NOT upgrade to Zod 4. Current 3.25.76 is stable."
- **PITFALLS says:** "Pin zod to 3.25.67 (last version without v4 bundled) to avoid accidental v4 import and unnecessary bloat."
- **Resolution:** Both agree on not upgrading to Zod 4. The question is whether to downgrade from 3.25.76 to 3.25.67 to remove bundled v4 code. Since RAPID is pure CJS and no v4 imports exist, the risk is low either way. **Recommend keeping 3.25.76** for now (stable, working), with a note to monitor for accidental v4 imports. Pinning to 3.25.67 is a valid alternative if bloat is a concern.

### Contradiction: Solo Mode Config Location
- **FEATURES says:** Add `"solo": boolean` to `.planning/config.json`.
- **OVERSIGHTS says:** There are TWO config files; solo belongs in `.planning/config.json` (planning config, not runtime config).
- **UX says:** Add `"planning.soloMode"` or `"soloMode"` to config.json.
- **Resolution:** All agree it belongs in `.planning/config.json`. The exact field name differs (`solo`, `soloMode`, `defaultSolo`, `execution.solo_mode_default`). **Recommend `solo: boolean`** as the simplest field name, consistent with the existing `--solo` flag naming.

## Recommended Approach

This milestone is a robustness hardening pass on an architecturally sound system. The existing 5-layer architecture (Skill -> Agent -> CLI -> Library -> Storage) is correct and does not need restructuring. All nine goals target boundary-condition failures: missing files, wrong state, incorrect prompts, and untracked artifacts. The recommended pattern is **defensive middleware with graceful degradation** -- adding pre-condition checks and fallback paths at each point of failure.

The most impactful fix is the **agent CLI command awareness** issue, which manifests in multiple ways: agents hallucinating subcommands, documentation using wrong status names, and TOOL_REGISTRY drifting from actual commands. The systemic fix -- embedding a canonical CLI command reference in core-identity.md and propagating it through build-agents -- prevents future drift and eliminates an entire class of user-visible failures. This should be done first because it establishes the correctness foundation for all other agent-driven fixes.

The second critical area is **state transition resilience**. The stuck-in-executed state and untracked-files-blocking-merge issues share a common root cause: the two-branch commit model where STATE.json transitions happen on main while code and planning artifacts live on the worktree branch. The fix requires a clear decision: planning artifacts (WAVE-COMPLETE.md, PLAN-DIGEST.md) should be committed in the worktree branch, and only STATE.json transitions should happen on main. Once this decision is codified, both the merge pipeline and the execute-set flow become predictable.

The third area is **defensive programming in library code**: making loadSet() graceful when DEFINITION.md is missing, making review accept `executed` status, and adding auto-detection of post-merge mode in downstream skills. These are small, localized fixes with outsized impact because they eliminate hard failures that block the pipeline. The approach for each is the same: check preconditions, provide a graceful fallback, and emit a clear warning rather than throwing.

The medium-priority items (worktree package management, solo mode config) and low-priority items (UI/UX emphasis, lockfile improvements, security hardening) can be addressed in a second wave after the critical path is clear. None of these block other work, and they can be implemented independently.

### Suggested Set Boundaries

Based on file ownership clusters, dependency analysis, and parallelizability:

- **Set A (State & Execution Robustness):** Features 5, 6 -- State transition CWD fix, review accepting `executed`, merge untracked file handling. Both touch state-machine.cjs, execute-set/SKILL.md, and merge infrastructure. F6 depends on F5's CWD fix. Rationale: tightly coupled through the execute-merge pipeline.

- **Set B (Agent Correctness & Prompts):** Features 1, 3 -- Agent CLI command awareness sweep, discuss-set 4-option fix. Both involve correcting agent/skill prompt content. Touch core-identity.md, discuss-set/SKILL.md, tool-docs.cjs, and role modules. Includes the status name mismatch fix from PITFALLS. Rationale: both are prompt engineering fixes that require build-agents rebuild.

- **Set C (Init & Configuration):** Features 2, 8 -- DEFINITION.md generation fix, solo mode config. Both involve the init/start-set flow. Touch init/SKILL.md, start-set/SKILL.md, plan.cjs, config.json. Rationale: init skill is the common dependency.

- **Set D (Review Pipeline):** Feature 7 -- Post-merge review file discovery auto-detection. Isolated to unit-test/bug-hunt/uat skill files. Rationale: small scope, no file overlap with other sets.

- **Set E (Worktree Infrastructure):** Feature 4 -- Worktree package management with ecosystem detection. Isolated to worktree.cjs and merge skill. Rationale: independent infrastructure feature.

- **Set F (Planning Refinement):** Feature 9 -- UI/UX emphasis in discuss-set and planner prompts. Isolated to discuss-set/SKILL.md, role-planner.md, plan-set/SKILL.md. Rationale: prompt-only changes with no code dependencies.

Sets A, B, C, D, E, and F can all be worked in parallel. The only sequencing constraint is that F6 (in Set A) benefits from F5's CWD fix. Sets D, E, and F are smallest and can be merged first to reduce integration risk.
