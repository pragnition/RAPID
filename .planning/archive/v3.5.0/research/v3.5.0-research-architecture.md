# Architecture Research: v3.5.0 Robustness & Fixes

## Current Architecture

RAPID is a Claude Code plugin organized in five layers: Skill (24 user-facing command definitions), Agent (26 specialized AI workers built from role modules), CLI (rapid-tools.cjs command router dispatching to handler modules), Library (21 core modules in src/lib/), and Storage (.planning/ state files + .rapid-worktrees/ git worktrees).

### Strengths Relevant to This Milestone

1. **Transactional state management.** All mutations flow through `withStateTransaction()` with Zod validation. The state schema is well-defined with a clear hierarchy: ProjectState > milestones > sets > waves > jobs. This gives a solid foundation for fixing state transition issues (goals 5, 6).

2. **CLI as single source of truth.** All agent-state interaction routes through `rapid-tools.cjs`. Commands are well-documented in the USAGE string with explicit syntax. This makes the "wrong CLI command" problem (goal 1) a prompt engineering issue, not an architecture issue.

3. **Worktree isolation pattern.** Physical isolation via git worktrees prevents cross-set file conflicts. The registry (REGISTRY.json) tracks worktree metadata including solo mode. This isolation model is sound but needs extension for package management (goal 4).

4. **Artifact-based coordination.** Review pipeline uses REVIEW-SCOPE.md as the handoff artifact between `/review` and downstream skills (`/unit-test`, `/bug-hunt`, `/uat`). This loose coupling is architecturally correct but has a path resolution gap in post-merge mode (goal 7).

### Areas Requiring Fix or Extension

1. **Agent prompts lack CLI reference.** The core-identity.md module tells agents to "interact with state exclusively through the rapid-tools.cjs CLI" but does not include the actual command reference. Agents infer commands, leading to hallucinated subcommands like `state set-status`, `state --help`, `state update-set`. The tool-docs.cjs TOOL_REGISTRY is embedded into some agents but not all, and even where present, agents ignore it.

2. **DEFINITION.md generation gap.** `plan.loadSet()` at line 202 of plan.cjs does a hard `fs.readFileSync` on DEFINITION.md, throwing if it does not exist. DEFINITION.md is created by two paths: (a) `plan.createSet()` during init/roadmapper and (b) the `/add-set` skill. However, the `/add-set` skill writes DEFINITION.md via the Write tool (skill-side), while the init flow uses `plan.createSet()`. The problem: the init flow writes ROADMAP.md, CONTRACT.json, and STATE.json but does NOT call `plan.createSet()` -- it writes CONTRACT.json directly. DEFINITION.md is never generated for sets created during init. Downstream, `generateScopedClaudeMd()` calls `plan.loadSet()` which crashes.

3. **State transition rigidity.** The SET_TRANSITIONS map allows `executed -> complete` but the executor skill (Step 6) sometimes fails to make this transition. The review skill (Step 0c) requires status `complete` and rejects `executed`. This is a single-point-of-failure design -- if the executor's final Bash call fails, the set is stuck.

4. **Untracked file collision on merge.** The execute-set skill writes WAVE-COMPLETE.md markers and PLAN-DIGEST.md files to `.planning/sets/{setId}/` on main (not in the worktree). When the set branch also has these files (committed there by the worktree), `git merge` fails because untracked files on main would be overwritten. The architecture correctly isolates source code in worktrees but does not isolate planning artifacts.

5. **Post-merge review file path mismatch.** The `/review` skill writes REVIEW-SCOPE.md to `.planning/post-merge/{setId}/REVIEW-SCOPE.md`. Downstream skills (`/unit-test`, `/bug-hunt`, `/uat`) correctly read from this path when `--post-merge` is set. However, the review skill spawns `rapid-scoper` agent with `worktreePath` set to cwd. The scoper reads files from the worktree path. In post-merge mode, there is no worktree -- all files are on main. This is handled. The actual problem per user feedback is that review creates files on main but the spawned subagent tools (unit-tester, bug-hunter, uat) cannot find them. This suggests the agent working directory or file discovery path is incorrect.

6. **No package management in worktree lifecycle.** `createWorktree()` in worktree.cjs runs `git worktree add` but does not install dependencies. Worktrees share the project's node_modules via symlink in some git configurations, but for languages like Python (uv/pip), each worktree needs its own virtual environment. The merge agent has no post-merge dependency verification step either.

## Recommended Architecture Pattern

- **Pattern:** Defensive middleware with graceful degradation
- **Rationale:** The nine goals in this milestone are all robustness fixes -- hardening existing flows against edge cases, missing files, and incorrect state. Rather than restructuring the architecture, each fix should add a defensive check at the point of failure while preserving the existing 5-layer architecture. Where a hard failure currently occurs (missing DEFINITION.md, wrong state, untracked files), convert to either (a) graceful fallback or (b) explicit pre-condition check with a clear error message.
- **Evidence:** The existing architecture is sound for its purpose. All nine issues are boundary-condition failures, not structural deficiencies. Adding pre-flight checks and fallback paths follows the "make illegal states unrepresentable" principle without requiring layer changes.

## Module Organization

### Module: state-transitions.cjs
- **Responsibility:** Define valid status transitions for sets, waves, and jobs
- **Public interface:** `SET_TRANSITIONS`, `WAVE_TRANSITIONS`, `JOB_TRANSITIONS`, `validateTransition()`
- **Dependencies:** None (pure data + validation)
- **Data ownership:** Transition maps
- **Fix scope:** No change needed here -- the transitions themselves are correct. The fix is in how the review skill interprets status.

### Module: plan.cjs
- **Responsibility:** Set creation, DEFINITION.md + CONTRACT.json generation, set loading, assumptions surfacing
- **Public interface:** `createSet()`, `loadSet()`, `generateDefinition()`, `listSets()`, `surfaceAssumptions()`
- **Dependencies:** contract.cjs, dag.cjs, state-machine.cjs
- **Data ownership:** `.planning/sets/{setName}/DEFINITION.md`, `CONTRACT.json`
- **Fix scope:** `loadSet()` must handle missing DEFINITION.md gracefully (return empty string or generate on-the-fly from ROADMAP.md). The init skill must either call `createSet()` or write DEFINITION.md alongside CONTRACT.json.

### Module: worktree.cjs
- **Responsibility:** Git worktree lifecycle, registry management, scoped CLAUDE.md generation
- **Public interface:** `createWorktree()`, `removeWorktree()`, `generateScopedClaudeMd()`, `readRegistry()`, `withRegistryUpdate()`
- **Dependencies:** lock.cjs, plan.cjs (for loadSet in generateScopedClaudeMd)
- **Data ownership:** `.rapid-worktrees/`, `.planning/worktrees/REGISTRY.json`
- **Fix scope:** (1) `createWorktree()` should detect package manager and install deps. (2) `generateScopedClaudeMd()` depends on `loadSet()` which depends on DEFINITION.md -- fix cascades from plan.cjs fix.

### Module: merge.cjs
- **Responsibility:** 5-level conflict detection, 4-tier resolution, merge execution, integration tests
- **Public interface:** `mergeSet()`, `detectConflicts()`, `resolveConflicts()`, `runIntegrationTests()`
- **Dependencies:** worktree.cjs, contract.cjs, state-machine.cjs
- **Data ownership:** `.planning/sets/{setName}/MERGE-STATE.json`
- **Fix scope:** `mergeSet()` should check for untracked files that would collide and either clean them or commit them before attempting `git merge --no-ff`.

### Module: review.cjs
- **Responsibility:** Review scoping (pre-merge and post-merge), issue management, acceptance criteria extraction
- **Public interface:** `scopeSet()`, `scopeSetPostMerge()`, `serializeReviewScope()`, `parseReviewScope()`, `logIssue()`, `listIssues()`
- **Dependencies:** merge.cjs (for readMergeState), worktree.cjs
- **Data ownership:** `.planning/sets/{setId}/REVIEW-SCOPE.md`, `.planning/post-merge/{setId}/`
- **Fix scope:** The module itself is architecturally correct. The file path logic properly distinguishes standard vs post-merge paths. The issue is in the agent prompts consuming these artifacts.

### Module: tool-docs.cjs
- **Responsibility:** Generate YAML documentation of CLI commands for embedding in agent prompts
- **Public interface:** `TOOL_REGISTRY`, `generateToolDocs()`
- **Dependencies:** None
- **Data ownership:** In-memory tool documentation strings
- **Fix scope:** This is the right mechanism for CLI command awareness. Needs to be embedded more prominently in agent prompts, or a summary needs to go into core-identity.md.

```
  B) plan.loadSet() returns graceful default when DEFINITION.md missing
```

```
  ├── AskUserQuestion constraint: max 4 options
  └── FIX: change to exactly 4 options (remove option 5)
```

### Fix 4: Worktree Package Management (Goal 4)
```
createWorktree() in worktree.cjs
  ├── git worktree add -b branch path HEAD
  ├── MISSING: dependency installation
  └── FIX: detect package manager, run install in new worktree

Detection heuristic:
  package-lock.json exists  → npm install
  yarn.lock exists          → yarn install
  pnpm-lock.yaml exists     → pnpm install
  uv.lock / pyproject.toml  → uv sync / pip install
  Cargo.toml                → (cargo handles per-invocation)
  go.sum                    → go mod download

mergeSet() in merge.cjs
  └── FIX: after successful merge, verify deps still resolve
           (post-merge npm install / uv sync if lockfile changed)
```

### Fix 5: Execution-to-Complete Transition (Goal 5)
```
Current flow:
  /execute-set Step 6
    ├── node rapid-tools state transition set <m> <s> complete
    ├── git commit
    └── IF Bash call fails (timeout, git error): set stuck in "executed"

  /review Step 0c
    ├── Requires: status === 'complete'
    └── Rejects: status === 'executed' (even though all waves are done)

Fix options:
  A) Review skill accepts both "executed" and "complete"
     (review is non-mutating, so "executed" = "all waves done" is sufficient)
  B) Add retry/recovery in execute-set for the transition
  C) Both

RECOMMENDED: (A) -- review should accept "executed" as reviewable
  Additional: (B) -- wrap the transition in try/catch with retry
```

### Fix 6: Merge Untracked File Handling (Goal 6)
```
Problem chain:
  1. /execute-set writes WAVE-*-COMPLETE.md to .planning/sets/{id}/ on main
     (these are untracked on main's working tree)
  2. Set branch rapid/{id} has these same files (committed there by executor)
  3. git merge --no-ff fails: "untracked files would be overwritten"

Two-part fix:
  Part A: Execute-set commits planning artifacts in the worktree branch
    - After each wave, commit WAVE-COMPLETE.md + PLAN-DIGEST.md in the worktree
    - These live on the set branch, not untracked on main
    - On main, only STATE.json transitions are committed

  Part B: Merge pre-flight cleanup
    - Before git merge, check for untracked files matching the set branch
    - If found, either:
      (a) git add + commit them on main first
      (b) remove them (they will arrive via the merge)
    - mergeSet() in merge.cjs should handle this before the merge command
```

### Fix 7: Review File Discovery (Goal 7)
```
Problem:
  /review --post-merge writes to .planning/post-merge/{setId}/REVIEW-SCOPE.md
  /unit-test --post-merge reads from .planning/post-merge/{setId}/REVIEW-SCOPE.md
  Bug: unit-tester/bug-hunter/uat AGENTS spawned by these skills
       receive worktreePath from SCOPE-META but there is no worktree in post-merge mode

Diagnosis: The SCOPE-META JSON block contains `worktreePath` which in post-merge
  mode is set to cwd (project root). Subagent prompts include:
    "Working Directory: {worktreePath}"
  The agents should be finding files on main, which IS the working directory.

Likely issue: The spawned agents look for REVIEW-SCOPE.md in the standard
  (.planning/sets/{id}/) path rather than the post-merge path, because the
  skill passes SET_NAME but the agent does not know about --post-merge context.

Fix: Ensure skills pass the full path to REVIEW-SCOPE.md in the agent prompt
  (not just setId), so agents do not need to reconstruct the path.
  Alternatively, add explicit post-merge path resolution to agent prompts.
```

### Fix 8: Project-Wide Solo Mode Config (Goal 8)
```
Current:
  .planning/config.json contains: { project, model, planning }
  Solo mode is per-set (set during /start-set with --solo flag)

Fix:
  Add to .planning/config.json:
    "execution": { "solo_mode_default": true }
```

## State Management

### Where State Lives
- **STATE.json** (`.planning/STATE.json`): Machine-readable project state with Zod validation. Contains milestones, sets, waves, jobs with status fields.
- **REGISTRY.json** (`.planning/worktrees/REGISTRY.json`): Worktree tracking with solo mode, start commit, phase, branch.
- **config.json** (`.planning/config.json`): Project-level configuration (model, planning options). This is where solo_mode_default belongs.
- **MERGE-STATE.json** (`.planning/sets/{id}/MERGE-STATE.json`): Per-set merge pipeline state.
- **Planning artifacts** (`.planning/sets/{id}/`): DEFINITION.md, CONTRACT.json, CONTEXT.md, wave plans, review scopes.

### How State is Read and Written
- All STATE.json mutations go through `withStateTransaction()` (lock, read, validate, mutate, validate, write, unlock).
- Registry updates go through `withRegistryUpdate()` (similar transaction pattern).
- Planning artifacts are written directly by skills (Write tool) or library functions (`plan.createSet()`).
- config.json is read-only at runtime; modified only by init and install skills.

### Consistency Guarantees
- File-based locking via proper-lockfile prevents concurrent STATE.json mutations.
- Zod schema validation at read and write boundaries catches corruption.
- No cross-file transaction support -- STATE.json and planning artifacts can get out of sync (this is the root cause of goal 5 and goal 6).

## Integration Points

| External System | Protocol | Purpose | Error Handling |
|----------------|----------|---------|---------------|
| Git CLI | execSync/execFileSync | Worktree management, merge, branch ops | gitExec wrapper returns {ok, stdout, stderr} |
| Claude Code Plugin API | SKILL.md + agents/*.md | Agent spawning, skill dispatch | RAPID:RETURN protocol (COMPLETE/CHECKPOINT/BLOCKED) |
| File System | fs.readFileSync/writeFileSync | State persistence, artifact storage | Try-catch with descriptive errors |
| Package Managers (npm/uv/pip) | execSync | Dependency installation (NEW for goal 4) | Detect and run; graceful skip if not found |

## Scaling Considerations

This milestone is focused on robustness, not scaling. However, the fixes improve reliability at the current scale:

- **Agent CLI awareness** (goal 1) reduces wasted agent invocations (fewer retries from CLI errors).
- **Graceful DEFINITION.md** (goal 2) eliminates a blocking failure at worktree creation time.
- **Solo mode config** (goal 8) reduces the number of worktrees needed, saving disk and git overhead.
- **Untracked file handling** (goal 6) prevents merge pipeline stalls that require manual intervention.

## Architecture Risks

1. **Core module change propagation** (goal 1): Editing core-identity.md requires rebuilding all 26 agents via `build-agents`. If the CLI reference embedded in core-identity.md drifts from actual commands, all agents get stale. **Mitigation:** Generate the CLI reference section from TOOL_REGISTRY programmatically during build-agents, ensuring it stays in sync.

2. **DEFINITION.md backward compatibility** (goal 2): Making `loadSet()` tolerant of missing DEFINITION.md could mask genuine errors in new projects where DEFINITION.md should exist. **Mitigation:** Emit a warning when DEFINITION.md is missing, distinguishing "graceful fallback" from "silent failure."

3. **State transition loosening** (goal 5): Accepting "executed" in the review skill could allow review of sets where the final verification step failed. **Mitigation:** The review skill should check not just the set status but also verify that all wave COMPLETE markers exist. If markers are missing, warn but allow review to proceed.

4. **Package manager detection** (goal 4): Auto-detecting and running package managers in worktrees introduces platform-specific failure modes (wrong Python version, missing system deps). **Mitigation:** Make package installation a best-effort step with clear error reporting rather than a blocking failure. Log the install command and output.

5. **Prompt size inflation** (goal 1): Embedding the full CLI reference in core-identity.md increases every agent's prompt size. With 26 agents, each gaining ~2KB of CLI reference, this adds ~52KB total across all built agents. **Mitigation:** Use a condensed format (the TOOL_REGISTRY one-liners are already compact at ~3KB total). Only embed commands relevant to each agent category rather than the full set.

## Recommendations

1. **Fix DEFINITION.md generation in init flow** -- Add DEFINITION.md creation alongside CONTRACT.json in the init skill's roadmap acceptance step. Also make `plan.loadSet()` return a default when DEFINITION.md is missing. **Priority: critical** -- This blocks worktree creation for any set created during init.

2. **Embed CLI command reference in core-identity.md** -- Add a condensed version of TOOL_REGISTRY commands to the core identity module. Build-agents will propagate to all 26 agents. Add explicit instruction: "NEVER invent CLI subcommands not listed above." **Priority: critical** -- Agents failing on wrong commands is the most user-visible bug.

3. **Accept "executed" status in review skill** -- Change review SKILL.md Step 0c to accept both "complete" and "executed" states. Add defensive wave-completion check. **Priority: critical** -- This blocks the review pipeline.

4. **Fix discuss skill to 4 options** -- The discuss-set SKILL.md Step 5 currently lists 5 options but AskUserQuestion supports max 4. Reduce to 4 gray areas or restructure the selection. **Priority: high** -- Causes runtime error in discuss flow.

5. **Commit planning artifacts in worktree before merge** -- Change execute-set to commit WAVE-COMPLETE.md and PLAN-DIGEST.md in the worktree branch (via git -C {worktreePath}), not on main. Add pre-merge cleanup in merge.cjs to handle any lingering untracked files. **Priority: high** -- Blocks merge pipeline.

6. **Fix review file discovery for post-merge mode** -- Ensure the unit-test, bug-hunt, and uat skill prompts pass the full artifact path (not just setId) to spawned agents, and explicitly mark post-merge mode in agent task prompts so agents resolve file paths correctly. **Priority: high** -- Broken post-merge review.

7. **Add worktree dependency installation** -- Add post-create hook in worktree.cjs to detect package manager and run install. Add post-merge verification in merge skill. Make this best-effort with clear logging. **Priority: medium** -- Affects Python/uv projects more than Node.js.

8. **Add solo_mode_default to config.json** -- Extend .planning/config.json schema with `execution.solo_mode_default` boolean. Have /start-set and set-init read this config to auto-set solo mode. **Priority: medium** -- Quality of life improvement.

9. **Strengthen UI/UX emphasis in discuss/plan** -- Add explicit UI/UX gray area requirement to discuss-set Step 5 and UI/UX section to plan-set wave template. **Priority: low** -- Guidance improvement, not a bug fix.
