# /rapid:plan -- Project Decomposition into Parallelizable Sets

You are the RAPID project planner. This skill decomposes a project's work into parallelizable sets -- independent workstreams that can run simultaneously in isolated worktrees. Each set gets interface contracts, file ownership, and dependency edges. Follow these steps IN ORDER. Do not skip steps.

## Step 1: Check for Existing Sets

Run the set listing command to see if decomposition has already happened:

```bash
node "${RAPID_TOOLS:-$HOME/RAPID/src/bin/rapid-tools.cjs}" plan list-sets
```

Parse the JSON output. The response contains a `sets` array.

**If sets already exist** (array is non-empty):
Display the existing sets and present exactly 3 options:

> **Existing sets detected:**
> {list each set name and its wave assignment}
>
> 1. **Re-plan** -- Delete existing `.planning/sets/` and `.planning/contracts/` directories and start fresh
> 2. **View existing** -- Show current set details and exit
> 3. **Cancel** -- Exit without changes

**NEVER proceed without the user's explicit choice.** Wait for their selection.

- If **Re-plan**: Delete the `.planning/sets/` and `.planning/contracts/` directories using `rm -rf`, then continue to Step 2.
- If **View existing**: For each set, run `node "${RAPID_TOOLS:-$HOME/RAPID/src/bin/rapid-tools.cjs}" plan load-set <name>` and display the definition and contract summary. Then **STOP**.
- If **Cancel**: **STOP** immediately.

**If no sets exist** (empty array or command errors):
Continue to Step 2.

## Step 2: Load Project Context

Gather all context the planner subagent needs for decomposition:

**a) Requirements (primary input):**
Read `.planning/REQUIREMENTS.md` using the Read tool. This is the primary input for decomposition -- it contains all the requirements the project must fulfill.

**b) Project overview:**
Read `.planning/PROJECT.md` for the project name, description, core value proposition, and team size.

**c) Codebase scan (brownfield detection):**
Run the brownfield detector to understand existing code structure:

```bash
node "${RAPID_TOOLS:-$HOME/RAPID/src/bin/rapid-tools.cjs}" context detect
```

Parse the JSON output. If `hasSourceCode` is true, store the `manifest` for the subagent. The manifest contains detected languages, frameworks, config files, directory structure, and sample source files. If false, note this is a greenfield project.

**d) Architecture context (if available):**
Check if `.planning/context/ARCHITECTURE.md` exists (generated by `/rapid:context` in Phase 3). If it exists, read it -- this provides architectural patterns the planner should respect.

**e) Conventions context (if available):**
Check if `.planning/context/CONVENTIONS.md` exists. If it exists, read it -- this provides naming patterns and code conventions the planner should follow.

Compile all gathered context into a structured brief for the planner subagent.

## Step 3: Planner Subagent -- Propose Sets

Spawn the planner subagent using the Agent tool. The subagent's job is to analyze the requirements and codebase context and propose a set decomposition.

**Agent tool invocation:**

Provide the subagent with:
1. The role instructions: "You are the RAPID planner. You decompose project work into parallelizable sets -- independent workstreams that different developers or Claude instances can work on simultaneously. Each set goes through its own full discuss/plan/execute lifecycle in an isolated worktree."
2. All context gathered in Step 2 (REQUIREMENTS.md content, PROJECT.md content, codebase manifest, architecture/conventions if available)
3. These decomposition guidelines:

**Set Decomposition Strategy:**
- Start with requirements groups: cluster related requirements into candidate sets
- Validate against codebase structure: does the proposed set boundary align with directory/module boundaries?
- Check for shared resources: identify files needed by multiple sets (package.json, configs, shared types)
- Assign shared resources to the most logical owner; create CONTRIBUTIONS.json entries for others
- Look for natural layering: core/shared types set at wave 1, feature sets at wave 2, integration set at wave 3
- Check set size: minimum 3-5 owned files and 2+ tasks per set. Merge tiny sets into neighbors

**Contract Design Guidance:**
- `exports.functions`: list every function this set will expose that other sets might consume
- `exports.types`: list every data type/shape this set defines that other sets reference
- `imports.fromSets`: list every other set this set depends on and which functions/types it consumes
- `behavioral.invariants`: conditions that must always hold (e.g., "authenticate returns null, never throws, for invalid tokens")
- `behavioral.sideEffects`: observable effects (e.g., "writes session to disk", "emits auth:login event")
- Keep contracts at the interface level -- do not specify implementation details

**Ownership Rules:**
- Every file that could be modified must be assigned to exactly one set
- Shared files (package.json, configs) get explicit single ownership with CONTRIBUTIONS.json for cross-set changes
- Dependencies between sets MUST be explicit edges
- Extract shared types into a "core" or "shared" set if bidirectional dependencies would create cycles

**Output format requirement:** The subagent must return a JSON array of set definition objects:
```json
[
  {
    "name": "set-name-kebab-case",
    "scope": "What this set does in 1-3 sentences",
    "ownedFiles": ["src/module/**", "tests/module/**", "specific-file.js"],
    "tasks": [
      { "description": "Task description", "acceptance": "How to verify it is done" }
    ],
    "acceptance": ["Overall set acceptance criterion 1"],
    "wave": 1,
    "parallelWith": ["other-set-name"],
    "contract": {
      "exports": {
        "functions": [{ "name": "fn", "file": "src/module/index.cjs", "params": [{"name": "p", "type": "string"}], "returns": "object" }],
        "types": [{ "name": "TypeName", "file": "src/module/types.cjs", "shape": { "type": "object", "properties": {} } }]
      },
      "imports": {
        "fromSets": [{ "set": "other-set", "functions": ["otherFn"], "types": ["OtherType"] }]
      },
      "behavioral": {
        "invariants": ["condition that must always hold"],
        "sideEffects": ["observable effect"]
      }
    },
    "contributions": [
      { "file": "package.json", "owner": "core-set", "intent": "Add dependency", "section": "dependencies", "priority": 2 }
    ]
  }
]
```

Wait for the subagent to return the proposal.

## Step 5: Persist Decomposition

When the developer approves the proposal, write the full set of approved set definitions as JSON and pipe to the decompose CLI command:

```bash
echo '<approved setDefs JSON array>' | node "${RAPID_TOOLS:-$HOME/RAPID/src/bin/rapid-tools.cjs}" plan decompose
```

**IMPORTANT:** The JSON must be a single line or properly escaped for the shell. Use a temporary file if the JSON is large:

```bash
# Alternative for large JSON payloads:
cat /tmp/rapid-setdefs.json | node "${RAPID_TOOLS:-$HOME/RAPID/src/bin/rapid-tools.cjs}" plan decompose
```

Parse the JSON result from the decompose command.

**On success**, display:

> **Decomposition Complete**
>
> **Sets created:**
> {list each set directory with its files: DEFINITION.md, CONTRACT.json, contract.test.cjs}
>
> **Artifacts:**
> - DAG: `.planning/sets/DAG.json`
> - Ownership: `.planning/sets/OWNERSHIP.json`
> - Manifest: `.planning/contracts/MANIFEST.json`
> - Gates: `.planning/sets/GATES.json`
>
> **Wave breakdown:**
> - Wave 1: {set names}
> - Wave 2: {set names}
> - ...
>
> **Total:** {N} sets across {M} waves

**On error** (ownership conflict, DAG cycle, validation failure):
Display the error clearly and suggest how to fix it. Common errors:
- **Ownership conflict:** "Two sets claim ownership of {file}. Assign it to one set and use CONTRIBUTIONS.json for the other."
- **DAG cycle:** "Circular dependency detected between {sets}. Extract shared types into a wave-1 'core' set."
- **Contract validation error:** "Contract for set {name} failed schema validation: {details}. Check the exports/imports structure."

If the error is fixable, offer to re-enter Step 3 with adjusted constraints.

## Step 6: Next Steps

After successful decomposition, tell the developer:

> **Next steps:**
> - Run `/rapid:assumptions <set-name>` to review Claude's mental model for any set before execution begins.
> - All sets in Wave 1 must complete planning (each set's own discuss/plan/execute lifecycle) before Wave 1 execution can begin.
> - Run `/rapid:status` to see set status and planning gate progress.
>
> Each set will go through its own full discuss/plan/execute lifecycle in an isolated worktree. The contracts defined above ensure that sets can develop in parallel -- each set codes against the contract interface, not the other set's implementation.

## Important Notes

- **Re-run behavior:** Running `/rapid:plan` when sets already exist shows a gate -- the developer must explicitly choose to re-plan, view, or cancel. This prevents accidental overwrites.
- **Contract stubs:** Contracts serve as stubs for parallel development. If Set A needs Set B's types, Set A codes against the contract definition. Both sets can run in parallel.
- **Ownership is exclusive:** Every modified file belongs to exactly one set. Cross-set changes go through CONTRIBUTIONS.json.
- **Wave gates:** Per-wave planning gates ensure all sets in a wave are planned before any execute. Wave 2 planning can overlap with Wave 1 execution.
- **Agent tool usage:** This skill uses the Agent tool to spawn a planner subagent for set decomposition. The subagent does the analytical work; the skill orchestrates the flow and handles developer interaction.
