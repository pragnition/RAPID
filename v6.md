# Big Feature changes

1. /init -> during the init phase, the init command should ask the user for any spec files that they would like to provide as well, just to serve as confirmation before the research phase starts. Furthermore, the research agent's should not "start fresh". Rather, they should build on specs/findings already found. 

2. /scaffold command -> currently the scaffold command is not an integral part of the flow. Let's change this. After building a huge project with my team, I have realised that the main issue with the way we are divvy up sets is that they often edit the same files, causing the merge to be extremely hard. Thus, here is  my proposed solution to solve this
- depending on the number of developers in the team, we will need to plan the sets in such a way that allows us to split the project into N groups of sets which have a very low likelihood of changing the same files much.

- there is an issue of a lack of observability in the sense that when 1 person is working on one part and the other is working on another part, person A may not be able to sufficiently test their part if they are doing backend work and person B is doing frontend work. therefore, the /scaffold command should now scaffold stubs in as well (eg. a very basic frontend to test, basic stub backend to test the frontend with). /scaffold will go after /init --> /branding --> /scaffold
  - in another vein, the /init command should also plan out a foundational set that helps to fill in these stubs
- all we need for these stubs is to be able to allow oberservable testing between groups of sets


3. /init & /new_version ->
In line with the above changes, /init and /new_version should generate a DAG that shows the sets dependencies. This DAG, through the dependencies will allow the agent to divide the project up into rough groups of sets. The reason we use groups of sets instead of sets now is that a project with 12 sets and 3 developers will likely look like 3 developers doing 4 sets each linearly, rather than 4 sets in parallel.


4. Personas to agents 
giving agents personas can be quite helpful. we should give our agent appropriate personas
for instance, /init might have an expert system architect persona
/executer might be an IOI gold medalist high functioning autism developer with a 30 year experience developing

/verifier might be a cranky senior engineer reviewing an intern's pull request 

/discuss might be an expert systems designer and architect and an overtly curious person 


5. Test taxonomoy

RAPID currently generates test without any concept of test taxonomy. There's no distinction between:

  Throwaway review tests — one-off validation for the review pipeline, not committed
  Permanent unit tests — committed, run in CI, fast, isolated
  Integration tests — committed, run in CI, test cross-component boundaries
  One-off smoke tests — manual verification, may or may not be kept

Tests generated by unit-test, bug-hunt, and uat all end up in the worktree with no guidance on lifecycle, location, or intent. This leaves the user to sort out what to keep, where to put it, and how it fits into the project's existing test strategy.

We should try to generate tests in a manner that can be re-used throughout phases/sets


6. Interactive human review of implementation plans before execution. After plan-set completes, there ought to be a mechanism for the human to interactively review plans before execute-set begins. I think the way we should do this is instead of disrupting the review pipeline, we should add an optional /review-plan skill that shows the use an overview of the plan and allows the user to conversationally suggest changes

7. /init 

/init should discuss and capture meta principles with the user in order to derive high level architectural and style decisions (eg. code style, etc)

8. /discuss 'claude decides' option.

Currently claude decides is analogous to the recommended option. Therefore it is redundant in most cases. We should change 'claude decides' to a 'tell me more' option if the user wishes for claude to tell them more about the option and then pose another AskUserQuestion to surface the final decision

9. DAG.json 

as mentioned above, the DAG should now be a central part of rapid

# Quality of life changes

1. RAPID:planner agent should optimize for wall clock time rather than minimal wave count only. Large sequential waves can take disproportionately long, blocking the entire pipeline. When tasks within a wave are independent (no shared file ownership, no logical dependency), the planner should consider splitting them into parallel waves within the same batch.

2. Agents calling other agents not belonging to rapid. We should explicitly tell agents to ONLY call subagents that belong to rapid with the rapid: prefix

3. Execution agents for the unit-test skill should use foreground parallel dispatch instead of background paralell dispatch so that the orchestrator doesn't sit idle between task notifications

# Bug fixes  

1. /init requirements.md overwrite 
In /rapid:init, Step 4D writes user-approved acceptance criteria to .planning/REQUIREMENTS.md (SKILL.md lines 458-460). Step 5 then runs init scaffold --mode fresh, which unconditionally overwrites REQUIREMENTS.md with a blank stub ("No requirements defined yet").

2. The rapid-tools init scaffold CLI only accepts --desc (src/commands/init.cjs:32), but skills/init/SKILL.md instructs the LLM to use --description (line 481). Line 382 also references --description as the flag name. Any LLM following the init skill instructions will fail at the scaffold step. Not sure if this is real so please check if this is still presetn

3. During rapid:init, the roadmapper agent returns a state JSON with only milestones and currentMilestone. The state-machine schema (state-schemas.cjs) also requires version, projectName, createdAt, and lastUpdatedAt. Step 5 (scaffold) creates a valid STATE.json with those fields, but Step 9 overwrites it entirely with the roadmapper's incomplete output. recalculateDAG then fails with "Cannot recalculate DAG: STATE.json is missing or invalid".
